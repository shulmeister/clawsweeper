## Summary

Keep #66672 open. Current main has adjacent safety controls (tool allow/deny policy, sandboxing, exec approvals, elevated mode, ACP runtime set-mode, and strict-agentic plan-only retries), but it does not implement the requested first-class Ask / Plan / Modify workflow across OpenClaw chat surfaces. The remaining work is a core UX/runtime contract, not a ClawHub/plugin cleanup item.

## Best Possible Solution

Keep the issue open and design a first-class core workflow-mode contract. A good implementation would add an explicit per-run or per-session mode to the gateway/chat protocol, expose it in Control UI/TUI/slash commands, enforce read-only and mutation-gated behavior in the agent tool pipeline, map ACP session modes only where applicable, and document how it composes with existing tool policy, sandboxing, exec approvals, and strict-agentic retries.

## Risks / Open Questions

- Closing as implemented would conflate adjacent controls with the requested product contract. Users still cannot choose a first-class read-only Ask mode, review-and-approve Plan mode, or explicit Modify mode from the normal chat surfaces.
- A proper implementation likely crosses gateway protocol, session/runtime state, tool authorization, UI/TUI commands, ACP mapping, docs, and regression tests, so it needs maintainer product/API judgment rather than conservative cleanup.
