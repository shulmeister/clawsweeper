## Summary

Keep open. Current main has general tool-loop detection, but it is keyed on exact tool parameters and known polling patterns; it does not normalize exec read-only command families, target file paths, or line-window variants. The reported same-file/same-file-pair reread loop remains a distinct missing runtime guard rather than an implemented feature or a clean duplicate of generic repeat hard-abort work.

## Best Possible Solution

Keep this issue open. The best implementation is a narrow core exec/tool-loop detector that classifies obvious read-only exec commands, normalizes cwd plus target file paths while collapsing line-window changes, tracks same-file and same-file-pair counts per run/session, and emits a warning-only synthetic result before executing the next redundant read. Add config/docs and focused tests for `sed -n`, `nl -ba | sed -n`, `cat`, bounded `rg`, alternating file pairs, threshold behavior, and false-positive cases. Coordinate with generic_repeat hard-abort work, but do not rely on exact-params hashing to solve this report.

## Risks / Open Questions

- A low-threshold readback detector can false-positive during legitimate debugging unless it is narrow, warning-first, and limited to obvious read-only command families.
- The right hook point needs care: built-in exec runs locally, in sandbox, or on node hosts, so normalization should use the resolved cwd/command plan without adding owner-specific behavior or bypassing approval contracts.
- This overlaps adjacent generic loop-detection work, but closing as duplicate would lose the requested file-target and line-window normalization requirements.
