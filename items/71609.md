## Summary

Keep #71609 open. Current main still has the stale browser device-token loop shape described in the issue: cached device-token-primary failures are not cleared, `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as terminal, and the checked-in browser tests still preserve reconnecting after a token mismatch in one no-retry path. Main contains useful partial hardening for explicit shared-token precedence and one bounded retry, but not the full stale-token cleanup/non-looping recovery requested here.

## Best Possible Solution

Fix the browser Control UI auth recovery path in `ui/src/ui/gateway.ts`: clear stale cached device auth when a stored-token-primary connect receives `AUTH_DEVICE_TOKEN_MISMATCH`, stop automatic reconnect loops after no safe retry is available or after the one-shot retry fails, preserve explicit shared-token precedence, and update gateway/client-facing mismatch guidance away from rotate/reissue toward stale identity cleanup or reconnecting with current shared auth. Add focused browser tests for the scope-upgrade stale-token case and the non-looping recovery path; keep the older explicit shared-token/signature fixes as-is.
