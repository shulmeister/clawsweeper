## Summary

Keep open. Current `main` still does not carry the linked Telegram `animation` fix: the inbound media detection, file-id resolution, download-path selection, and placeholder logic all still omit `animation`, which matches the root-cause analysis in the linked PR discussion. The reporter’s 60 MB sample would also still fail on the official Telegram Bot API today because `getFile` downloads remain capped at 20 MB, and OpenClaw explicitly treats that response as a skip-to-placeholder path. ([github.com](https://github.com/openclaw/openclaw/pull/56336))

## Evidence

- **Inbound media detection still skips animation:** `hasInboundMedia` and `resolveInboundMediaFileId` handle `video`, `video_note`, `document`, `audio`, `voice`, and `sticker`, but not `animation`. That means Telegram GIF-style video messages can still miss the inbound media pipeline on `main`.

## Risks / Open Questions

- The original repro used a 60 MB attachment. Even with the missing `animation` branches fixed, that exact sample still will not download through the official Telegram Bot API unless the user switches to a local Bot API server, because Telegram still documents a 20 MB `getFile` download cap.
