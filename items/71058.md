## Summary

Keep open. Current main still treats Microsoft Teams as a single configured bot identity: the public config type/schema expose one strict `channels.msteams` object with top-level `appId`, `appPassword`, and `tenantId`, the Teams plugin config adapter always resolves only the default account, and the runtime monitor starts one provider from one credential set. The related title-search hits are different channels or generic multi-account work, not a canonical Teams tracker.

## Best Possible Solution

Keep #71058 open and implement it in the bundled Microsoft Teams plugin. A proper fix should add a backward-compatible `channels.msteams.accounts`/`defaultAccount` contract with per-account credentials and policies, start or dispatch all configured bot identities safely, preserve accountId in inbound/outbound session binding, update Teams docs/config metadata, and add Teams-specific schema and runtime tests.
