## Summary

Issue #52974 is a scope mismatch for OpenClaw core. Current main explicitly routes optional capabilities and community-maintained plugins to external distribution, and memory plugins are already a supported exclusive plugin kind selected via `plugins.slots.memory`. Given the reporter already published an external package, the RFC belongs as a ClawHub/npm community plugin rather than a new bundled `extensions/memory-nexo` addition.

## Evidence

It also says memory is a special plugin slot and OpenClaw plans to converge on one recommended default path over time.

## Close Comment

Thanks for the RFC. I reviewed current `main` in Codex, and this looks out of scope for OpenClaw core rather than something missing in the product. The current repo direction is explicit that optional capability should usually live as plugins, that plugin authors should publish and maintain those plugins in their own repos (`VISION.md:54-64`), and that you do not need to add a plugin to this repository because OpenClaw supports publishing to ClawHub or npm and installing it with `openclaw plugins install <package-name>` (`docs/plugins/building-plugins.md:16-19`, `docs/plugins/community.md:9-22`).

This is also already a supported plugin shape: memory plugins are a first-class exclusive plugin kind selected via `plugins.slots.memory` (`docs/plugins/manifest.md:707`), and current main exposes dedicated memory-plugin registration APIs in `src/plugins/types.ts:2130-2162`. Given that current main already supports external memory plugins and the RFC proposes a community-maintained backend that the issue author has already published externally, this belongs on the ClawHub/npm plugin path rather than as a new bundled `extensions/memory-nexo` addition.
