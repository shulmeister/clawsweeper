## Summary

Keep #30067 open. Current main still lacks the requested proactive first-run/setup heads-up before the TUI/Web/chat handoff that provider initialization or probes may temporarily hit API rate limits. Existing handling is reactive docs/status guidance, not an onboarding notification.

## Best Possible Solution

Keep the issue open. Add a concise first-run note in `src/wizard/setup.finalize.ts` before the TUI/Web/Later hatch prompt, and mirror equivalent guidance in the Crestodian startup/setup handoff if that path can launch normal chat. The copy should explain that first-run provider probes or background initialization may temporarily hit provider rate limits, suggest waiting/retrying or choosing a fallback model, and add focused tests asserting the note appears in the relevant first-run paths.
