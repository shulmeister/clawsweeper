## Summary

Keep open. Current main appears to fix the specific `workspace-*` blacklist ordering for callers that pass explicit `localRoots`, and the image tool now handles workspace-relative inputs, but the PDF tool still does not resolve non-sandbox relative PDF paths against `workspaceDir`. The issue's reproduction uses `pdf="file-in-workspace.pdf"`, and there is no narrower canonical follow-up in the provided context for that remaining PDF workspace-relative path gap.

## Best Possible Solution

Keep the issue open but narrow the maintainer task: mirror the image tool's workspace-relative path handling in `src/agents/tools/pdf-tool.ts` by resolving non-URL, non-absolute PDF inputs against `workspaceDir` or passing `workspaceDir` through to `loadWebMediaRaw`, then add regression coverage for both absolute and relative PDFs under `workspace-<agentId>` plus a guard that non-workspace paths remain rejected under workspace-only policy.
