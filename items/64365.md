## Summary

Keep open. Current main still has no outbound Slack `replyBroadcast`/`reply_broadcast` support, and Slack's Web API/SDK supports the requested chat.postMessage parameter when paired with `thread_ts`. This PR is a plausible implementation candidate, but it is not merge-ready on current main: it targets older Slack message-tool schema wiring, leaves review feedback unresolved around the docs example and upload behavior, and does not fully align with the current `message` tool contract.

## Best Possible Solution

Keep this PR open as a change-request candidate, not as cleanup-close material. The best path is to rebase it on current main, add a Slack-owned `replyBroadcast` schema contribution from `describeSlackMessageTool`, thread the option through `message-action-dispatch`, `action-runtime`, `actions`, and `sendMessageSlack`, emit `reply_broadcast: true` only on the first eligible `chat.postMessage` call when `threadTs` is set, and either remove upload-file support for this flag or implement a clearly documented follow-up chat-post behavior. Fix the docs example to the current `message` tool contract and add focused tests in Slack send, action runtime/dispatch, and message-tool discovery. Leave #63904 open for the broader inbound/automatic mirroring work unless a maintainer deliberately scopes this PR to close only the explicit outbound parameter.
