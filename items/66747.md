## Summary

Keep open. The report is recent, has a concrete macOS-specific reproduction and an actionable follow-up comment, and current main still appears to contain the implicated packaging and chmod behavior rather than an implemented fix or documented workaround.

## Best Possible Solution

Keep the issue open and treat it as an actionable macOS packaging/runtime hardening bug. The likely fix should combine a packaging or launcher change that avoids/detects `com.apple.provenance` EPERM on npm-installed shebang files, a clear doctor/startup diagnostic with workaround text, and a safer task/task-flow SQLite permission helper that tolerates chmod EPERM only when stat/ownership/mode checks prove the existing path is already private enough. Add focused unit coverage for chmod EPERM handling and a release/package guard for the published entrypoint behavior.

The same comment proposes core/package fixes: strip shebang from the published entry path, detect provenance+EPERM, or provide a `.cjs` alternative.
