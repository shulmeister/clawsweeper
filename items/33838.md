## Summary

Keep #33838 open. Current main still treats node commands as declared command-name strings and forwards unknown invoke params/results; it does not provide a protocol, pairing, registry, discovery, validation, or node/plugin API path for nodes to declare per-command input/output schemas. The existing discussion is a valid gateway protocol/API feature request, not a duplicate of the related node file-transfer, rule-checking, or denyCommands items.

## Best Possible Solution

Keep the issue open for an additive gateway protocol/API design. A good implementation would retain `commands: string[]` compatibility while adding optional per-command metadata, carrying it through connect, node.pair.request/persistence, node registry/session state, node.list/node.describe discovery, generated clients, node-host/plugin registration APIs, docs, and focused tests. Validation of node.invoke params/results should be explicit maintainer policy rather than an accidental side effect.
