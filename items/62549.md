## Summary

Keep #62549 open. Current main still does not accept the `baseURL` alias for Ollama provider config, and several production paths still read only `baseUrl`; the PR’s requested behavior is not implemented on main and is not safely superseded by another merged change in the provided context.

## Evidence

- **Discovery path still reads only baseUrl:** `hasMeaningfulExplicitOllamaConfig`, explicit provider injection, and ambient discovery all read `providerConfig.baseUrl`, `explicit.baseUrl`, or `explicit?.baseUrl` directly, so a config containing only `baseURL` still falls back to the default Ollama base URL during discovery.
  - file: [extensions/ollama/src/discovery-shared.ts:66](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/ollama/src/discovery-shared.ts#L66)

  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Embedding path still reads only baseUrl:** The embedding client resolves `rawBaseUrl` from `options.remote?.baseUrl` or `providerConfig?.baseUrl?.trim()`; it does not check `baseURL`. This matches one of the gaps raised in the PR discussion.
  - file: [extensions/ollama/src/embedding-provider.ts:122](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/ollama/src/embedding-provider.ts#L122)
