## Summary

Keep #20950 open. Current main still implements cron persistence as a single `cron.store` definitions file plus a derived `jobs-state.json` sidecar; there is no `cron.storeDir`, `jobs.d`, or per-job JSON directory loader/saver. PR #41206 is the relevant open implementation candidate, but it is unmerged/stale and has review-noted data-integrity concerns, so it should not close this issue as shipped or fully superseded.

## Best Possible Solution

Keep the issue open until core cron storage supports an additive directory-backed definitions store or maintainers explicitly reject that storage shape. The likely implementation path is an additive `cron.storeDir` or store-backend contract, deterministic per-file loading, compatibility with existing `cron.store`, state-sidecar behavior, doctor/config schema/help/docs updates, and focused tests; PR #41206 is the current candidate to review or replace.
