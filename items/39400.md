## Summary

Keep #39400 open. Current main still exposes provider quota in `session_status` as an optional human-readable status line, not as stable structured usage fields with explicit availability, source, or reset metadata. The existing ClawSweeper comment is still accurate on current main; later commits did not touch this surface.

## Best Possible Solution

Keep the issue open. If cron/monitor use of `session_status` is supported, extend the agent-facing status contract with a structured `usage` object in `details` that includes provider/source, explicit availability or unavailable reason, normalized windows with percent/resetAt, and reset explanation where known. Preserve the current human-readable status line and add focused tests for no auth, fetch error, empty window, timeout, and Codex 5h/Day/Week mapping behavior. If maintainers prefer not to expand `session_status`, track an equivalent dedicated usage tool and link #39400 to that implementation path.
