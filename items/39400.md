## Summary

Keep #39400 open. Current main still exposes provider quota in `session_status` as an optional human-readable status line, not stable structured usage fields. The status path suppresses provider usage on errors, empty windows, and timeouts, and the provider usage model still lacks `source`, explicit unavailable metadata, and reset explanations. Related reports cover adjacent token/cost surfaces, but none supersede this provider-window stability request.

## Best Possible Solution

Keep the issue open and implement a stable machine-readable `session_status` usage payload if maintainers want this monitor use case supported. The likely path is to extend the provider usage snapshot/status contract with explicit availability/source/reset metadata, return those fields in `session_status.details`, and keep the existing human-readable status line as presentation only. Add focused tests for error/no-window/timeout behavior and Codex 5h/Day window mapping.
