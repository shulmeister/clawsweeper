## Summary

Keep open. Current `main` does not contain this fix, and the PR as written is not a safe cleanup candidate: the new guard would run before the request-time config hot-reload path, it tells users to set `browser.hosting` even though the repo’s config/docs use `browser.enabled`, and it also carries unrelated committed quality-gate logs.

## Evidence

- **Current main still returns status instead of rejecting disabled hosting:** `extensions/browser/src/browser/routes/basic.ts:68-105` still calls `ctx.state()`, then immediately resolves the profile, and finally returns `enabled: current.resolved.enabled` in the JSON response. There is no explicit disabled-hosting error path on `main`.
