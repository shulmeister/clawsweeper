## Summary

Keep open. Current main has a clearer runtime/startup guard and docs for `hooks.enabled` requiring `hooks.token`, but the requested schema/config-write dependency validation is still missing from the shared config validator, write RPCs, persistence path, and `config.schema.lookup` metadata.

## Best Possible Solution

Keep this issue open. Add a shared cross-field config validator that rejects `hooks.enabled=true` unless `hooks.token` is a non-empty string, and route it through `validateConfigObjectRaw` / `validateConfigObjectWithPlugins` so `config.set`, `config.apply`, `config.patch`, and `writeConfigFile` all reject the invalid config before persistence. If `config.schema.lookup` is meant to guide UI/custom clients, add explicit dependency or warning metadata there too. Keep `resolveHooksConfig` as the startup/reload backstop and add focused regression tests for validator and Gateway RPC write behavior.

## Risks / Open Questions

- Adding only a hooks-specific check inside one RPC handler would leave other write surfaces divergent; the rule should live in shared config validation and be reused by RPC and persistence paths.
- A broad dependency metadata framework could be larger than necessary; a small shared cross-field validator plus optional lookup hint would address the reported outage path with less surface area.
