## Summary

Current main still does not satisfy the request. The shared message CLI formatter hard-caps rendered message rows at 25, the affected read/list-pins/search text paths still use that formatter, and provider-side pagination remains partial rather than a reusable limit/cursor contract.

## Best Possible Solution

Keep this issue open and implement a narrow message action contract fix: pass the requested display limit into formatMessageCliText/renderMessageList, remove or configure the hardcoded 25-row cap, propagate limit and pagination metadata through read/list-pins/channel-list/search where providers support it, expose next-page or cursor metadata consistently, and add regression tests that render more than 25 rows while preserving existing Teams @odata.nextLink behavior.
