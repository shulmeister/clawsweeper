## Summary

Current `main` still handles QQ Bot close code `4009` by clearing session state and forcing a fresh identify path, so the reported behavior is not implemented or disproven. The issue is recent, specific, and already linked to an open PR, so conservative cleanup should keep it open.

## Evidence

- **Current reconnect logic still treats 4009 like 4006/4007:** `ReconnectState.handleClose()` groups `INVALID_SESSION` (4006), `SEQ_OUT_OF_RANGE` (4007), and `SESSION_TIMEOUT` (4009) together, logs `will re-identify`, and returns `clearSession: true` plus `refreshToken: true`. That matches the issue report rather than resolving it.
  - file: [extensions/qqbot/src/engine/gateway/reconnect.ts:133](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/qqbot/src/engine/gateway/reconnect.ts#L133)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/reconnect.ts | sed -n '120,165p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Clearing session state forces IDENTIFY instead of RESUME:** `GatewayConnection.handleClose()` nulls `sessionId` and `lastSeq` when `action.clearSession` is true. `handleHello()` only sends `GatewayOp.RESUME` when both values are still present; otherwise it sends `GatewayOp.IDENTIFY`. On current main, a 4009 close therefore falls back to identify.
  - file: [extensions/qqbot/src/engine/gateway/gateway-connection.ts:313](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/qqbot/src/engine/gateway/gateway-connection.ts#L313)
