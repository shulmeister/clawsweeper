## Summary

Keep #37967 open. Current main supports Feishu sender/topic-scoped group sessions, manual routing bindings, DM-only dynamic agent creation, and explicit thread-bound subagent hooks, but it still does not automatically create and route each Feishu group sender or role to an isolated `__mc_*` subagent with separate workspace, skills/config, memory, and admin controls.

## Best Possible Solution

Keep the issue open for Feishu plugin product/design work. A good implementation would add an explicit opt-in group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, adds admin controls, and covers provisioning/isolation with tests and security docs. If maintainers prefer this outside the bundled Feishu plugin, first define the stable Feishu inbound-routing/plugin API needed for external implementation.

## Risks / Open Questions

- The feature changes Feishu group trust boundaries: per-user memory, workspaces, skills, model/config overrides, and admin controls need explicit opt-in authorization and security documentation.
- A ClawHub/plugin implementation may be viable if it owns the Feishu channel path, but current evidence does not show the requested behavior can be added externally to the bundled Feishu channel without a maintainer API/product decision.
