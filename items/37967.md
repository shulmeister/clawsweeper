## Summary

Keep #37967 open. Current main still does not provide automatic Feishu group-chat per-sender subAgents with isolated workspace, skills, config, and memory. The implemented pieces are adjacent: Feishu can create dynamic agents for direct-user flows, scope group sessions by sender/topic, and bind explicitly spawned subagent sessions back to Feishu threads, but those do not satisfy the requested automatic group provisioning/orchestration behavior.

## Best Possible Solution

Keep this open for Feishu plugin product/design work. A solid implementation would add an explicit opt-in group mode that maps sender or role identity to isolated agent/subagent profiles, safely provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating Feishu group/thread, provides bounded admin controls, documents the trust model, and adds regression coverage for automatic provisioning and isolation. If maintainers prefer this outside the bundled Feishu plugin, first define the needed plugin/API contract clearly.

## Risks / Open Questions

- The requested capability affects trust boundaries: per-user group memory and skills are useful, but the implementation must not imply per-user host authorization inside one shared gateway.
- Manual bindings, sender-scoped group sessions, and thread-bound subagent hooks solve adjacent routing problems and should not be treated as sufficient for automatic per-sender provisioning.
