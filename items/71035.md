## Summary

Current main already solves the linked #70901 user-facing case through config-wide `${VAR_NAME}` substitution before OpenClaw-managed `mcp.servers` config reaches the MCP runtime. The PR’s lower-level `mcp-http.ts` helper is redundant for `openclaw.json` and diverges from existing empty-env handling, which was already called out in review.

## Best Possible Solution

Close this PR as already implemented for OpenClaw-managed MCP config. Keep the existing config-wide substitution path as the source of truth; if raw bundle MCP manifests need placeholder expansion later, add a focused follow-up that reuses `resolveConfigEnvVars` semantics and includes resolver coverage for missing and empty env values.
