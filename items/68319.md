## Summary

The issue describes a private YAML/Markdown project-reconciliation workflow layered on top of OpenClaw, not a shipped OpenClaw core task surface. Current main tracks detached runtime work only, and the project vision/docs direct this kind of custom workflow logic to plugins/skills published through ClawHub.

## Evidence

- **Core tasks are detached-run records, not a project tracker:** The background-tasks docs define tasks as records for ACP runs, subagent spawns, cron executions, and CLI operations. They explicitly say heartbeats and normal chat do not create task records, which does not match the issue's proposed MEMORY/heartbeat-backed project reconciliation model.
  - file: [docs/automation/tasks.md:12](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/automation/tasks.md#L12)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current core task statuses do not model paused projects:** `TaskStatus` on current main is limited to `queued`, `running`, `succeeded`, `failed`, `timed_out`, `cancelled`, and `lost`. The implementation also treats only `queued`/`running` as active. There is no built-in `paused` or `resume_after` project-state contract in the core task registry.
  - file: [src/tasks/task-registry.types.ts:5](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/tasks/task-registry.types.ts#L5)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Task Flow docs route reusable workflow logic to plugins/ClawHub:** The Task Flow docs separate scheduling, orchestration, and reliability layers, and explicitly say reusable workflow packages should be shipped as a skill or plugin and published through ClawHub.
  - file: [docs/automation/taskflow.md:23](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/automation/taskflow.md#L23)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Project vision keeps optional capability out of core:** The repo vision says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub rather than OpenClaw core.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/VISION.md#L52)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Plugin docs explicitly support external packaging:** The plugin guide says plugins do not need to be added to the OpenClaw repository and should be published to ClawHub or npm, which fits this kind of workspace-specific task reconciler.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/plugins/building-plugins.md#L16)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Reported files/surface are not present on main:** Repo-wide search found no shipped references to `tasks/active.yaml`, `tasks/ad-hoc`, `memory/knowledge/projects`, or `Task Reconciler`, which strongly suggests the report targets a private workspace convention rather than an OpenClaw core surface.
  - command: `rg -n "memory/knowledge/projects|tasks/active\.yaml|tasks/ad-hoc|Task Reconciler" docs src extensions test README.md`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)

## Risks / Open Questions

- A maintainer could still choose to bring a generalized paused-project workflow into core later, but current scope docs point the other direction.
- The issue may describe a real pain point in a private OpenClaw-based workflow; closing as `clawhub` does not solve that workflow by itself.
