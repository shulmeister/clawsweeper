## Summary

Keep this issue open. Current main still creates a fresh native subagent session key with `crypto.randomUUID()` for every `sessions_spawn` call, and the public tool/config schemas do not expose a native subagent reuse key, pool, or reuse-main option. The related merged PR #64468 addressed bootstrap-marker persistence after clean `sessions_yield`; it was explicitly related but did not add a warm-reuse path for new `sessions_spawn` delegations.

## Best Possible Solution

Keep this issue open and design an explicit native subagent reuse contract. The likely implementation belongs around `src/agents/subagent-spawn.ts`, `src/agents/tools/sessions-spawn-tool.ts`, and the subagent config schema, with tests covering opt-in reuse semantics, isolation/history tradeoffs, concurrency locking, cleanup/archive interaction, and `sessions_send` follow-up behavior. The merged #64468 work should be treated as a related cache-cost fix, not as the canonical resolution for this issue.

## Risks / Open Questions

- A naive stable-key implementation could mix unrelated delegation history, violate the documented isolated-session default, or race when two parents spawn the same target concurrently.
- ACP has an ACP-only `resumeSessionId` path, but native `runtime="subagent"` does not; treating ACP resume as the fix would leave the reported native subagent behavior unchanged.
