## Summary

The proposal is coherent and useful, but current main explicitly steers optional plugin functionality like this out of the core repo and into the external ClawHub/community-plugin path. The plugin SDK already exposes the extension points this design needs, while current main also already provides built-in auth-status, probe, fallback, and key-rotation primitives for the underlying problem.

## Risks / Open Questions

- If the proposed plugin needs a new SDK seam that external plugins cannot access today, that missing seam would deserve its own narrowly scoped issue instead of bundling the whole plugin into core.
- The issue references an unpublished implementation, so this review can only judge repository scope and current platform support, not the quality of that external code.

## Close Comment

Closing after Codex review: this looks like a valid plugin idea, but it fits the ClawHub/community-plugin path rather than OpenClaw core. Current main is explicit here: `VISION.md:54-69` says optional capability should usually ship as plugins maintained in their own repository, and `docs/plugins/community.md:9-22` plus `docs/plugins/building-plugins.md:16-19` say community plugins should be published to ClawHub or npm instead of being added to this repo. The needed extension points already exist for external plugins in the SDK (`docs/plugins/sdk-overview.md:83-99` covers tools, commands, gateway methods, CLI, and background services). Current main also already includes core auth/fallback primitives such as `openclaw models status --check/--probe` and rate-limit key rotation (`docs/cli/models.md:28-38`, `docs/gateway/authentication.md:104-163`). If `@openclaw/shield` is still something you want to ship, the preferred path is an external plugin published to ClawHub or npm, with a focused follow-up issue only if you hit a missing SDK seam.
