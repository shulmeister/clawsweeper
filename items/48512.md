## Summary

Keep open. Current main does not contain the pending-tool-call hint or adapter this PR proposes, and the PR is still an implementation candidate for the retry-gap/tool-result race. It should not land as-is: the head branch is heavily contaminated with unrelated generated/runtime/UI/hook changes and has unresolved review findings, so the best maintainer path is to salvage a narrow fix or ask for a clean replacement PR.

## Best Possible Solution

Keep this PR open for maintainer handling, but do not land the current branch. The best path is a clean, narrow PR that wires a best-effort `hasPendingToolCalls` adapter from the pi agent state into all embedded cleanup paths, including attempt and compaction, with remaining-timeout accounting and focused regression tests for the overloaded retry gap.
