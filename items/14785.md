## Summary

Keep #14785 open. Current main has partial schema/action scoping for the message tool and documented workarounds like tool policies, but the central request remains: OpenClaw still builds and passes full tool parameter schemas to the model, the message schema still contains broad cross-channel fields, and the issue discussion explicitly records remaining P0/P1/P2 work after partial mitigations.

## Best Possible Solution

Keep #14785 open as the parent performance tracker. The best path is incremental PRs with measurements and tests: remove duplicated tool-list descriptions where appropriate, skip tool definitions when the effective list is empty, prune message parameters by current channel/capability without breaking cross-channel use cases, move tutorial text out of descriptions, and evaluate a compatible lazy/slim schema mode. Close only after current main measurably solves the broad fixed tool-schema overhead problem or a maintainer designates a narrower canonical replacement.
