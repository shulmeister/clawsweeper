## Summary

Keep open. Current main has not implemented PR #71503's LAN address selection behavior: gateway status/control UI still picks only exact en0/eth0 before falling back to the first external IPv4, while pairing setup and the bundled device-pair plugin still pick the first matching private IPv4. The PR is a recent, focused fix for linked issue #71493, and the discussion shows the contributor addressed review comments about CNI overlays and default-route preference while preserving the plugin boundary.

## Best Possible Solution

Keep PR #71503 open and review it as the active fix for #71493. The likely product path is to land or adapt its testable LAN IPv4 selection helper in the gateway path, use it for pairing setup, mirror the behavior in device-pair without violating the plugin boundary, and keep regression coverage for Docker bridge, CNI overlay, fallback-only bridge, and default-route ordering.
