## Summary

Keep this issue open. Current main still has the reported UI/backend mismatch: the Agents view's Set Default action stages `agents.defaultId`, while the config schema only accepts `agents.defaults` and `agents.list`, and the runtime default-agent resolver still uses `agents.list[].default` before falling back to the first list entry.

## Best Possible Solution

Fix the Agents UI Set Default action to update `agents.list` by marking the selected agent `default: true` and clearing any other `default` flags, then add UI/controller coverage proving the serialized `config.set` payload contains valid `agents.list[].default` and no `agents.defaultId`. Keep the runtime resolver's existing `agents.list[].default` fallback behavior unless maintainers decide to add a new schema-backed top-level default field.
