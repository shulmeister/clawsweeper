## Summary

Current main still appears to reproduce the reported bug. `/hooks/agent` dispatch creates an isolated cron job, and isolated cron runs still pass `forceNew: true` into `resolveCronSession`, which bypasses existing `sessionId` reuse even when the hook session key was accepted by policy. The linked PR #70895 directly targets this, but it is open/unmerged in the provided context and the fix is not present on main.

## Best Possible Solution

Keep #70894 open and use #70895 or an equivalent maintainer-reviewed patch to make hook-sourced isolated agent jobs reuse fresh sessions for the same permitted `sessionKey`, while preserving forced fresh sessions for ordinary isolated cron runs. Add a regression test around `runCronIsolatedAgentTurn` or `/hooks/agent` proving webhook-sourced isolated jobs pass `forceNew: false` and normal isolated cron still passes `forceNew: true`.
