## Summary

Keep open. Current `main` still has the reported behavior: `gateway probe` adds the SSH tunnel target plus the configured remote target when their literal URLs differ, and `buildGatewayStatusWarnings` emits `multiple_gateways` solely from `reachable.length > 1` without comparing `self`/presence identity. The issue is recent, has clear reproduction steps, no protected-label or maintainer-author auto-close blocker, and the related ClawSweeper items cover different gateway-probe false-negative/scope-timeout work rather than this false-positive multiple-transport warning.

## Best Possible Solution

Keep the issue open and fix the CLI gateway probe warning classifier. The likely best path is to add a small helper in `src/commands/gateway-status/output.ts` that groups reachable probes by a stable gateway identity from `entry.self` when available, emits `multiple_gateways` only for multiple distinct identities or ambiguous reachable targets, and optionally emits clearer informational copy for one gateway reachable over multiple transports. Add focused tests covering SSH tunnel plus `configRemote` with the same `self`, distinct `self` values, and null/ambiguous identity behavior, then update the gateway probe docs if the warning semantics change.
