## Summary

Keep open. Current `main` still writes configured providers into generated `models.json` without filtering out explicit custom providers whose `models[]` exist but whose auth cannot be resolved, so the core bug described in part 1 is not clearly fixed. Part 2 looks at least partly explained by current documented `/model` allowlist behavior, but the mixed report still contains an unresolved core issue.

## Evidence

- **models.json planner writes every final provider row without an unresolved-provider exclusion step:** `planOpenClawModelsJsonWithDeps()` normalizes, merges, enforces source-managed secrets, then serializes `finalProviders` directly to JSON. There is no filter that drops providers with `models` but no resolved `apiKey`.

- **unresolved custom providers remain unchanged instead of being excluded:** `resolveMissingProviderApiKey()` returns the original provider unchanged when it has `models` but no env/profile/API-key resolver result. `normalizeProviders()` then keeps that provider in the output map.
