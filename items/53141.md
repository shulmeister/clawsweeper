## Summary

Current `main` no longer matches the reported behavior: oversized auto-TTS replies are kept on the speech path by summarize-then-speak or truncate-then-speak fallback, and Telegram + ElevenLabs still route successful speech output as voice notes rather than silently dropping to text-only.

## Evidence

- **Long replies stay on the TTS path:** `maybeApplyTtsToPayload` no longer skips long replies when they exceed the soft TTS limit. If summarization is disabled it truncates and continues; if summarization is enabled it summarizes; if summarization fails it truncates and continues.
  - file: [extensions/speech-core/src/tts.ts:1116](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/speech-core/src/tts.ts#L1116)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1116,1143p'`
- **Successful auto-TTS still attaches audio:** After synthesis succeeds, the reply payload is returned with `mediaUrl`, and native voice-note channels set `audioAsVoice` when the provider says the output is voice-compatible.
  - file: [extensions/speech-core/src/tts.ts:1161](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/speech-core/src/tts.ts#L1161)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1152,1180p'`
- **Telegram voice delivery uses `sendVoice` for voice-compatible audio:** Telegram outbound delivery chooses `sendVoice` for audio replies marked `audioAsVoice`, only falling back to text for Telegram-side `VOICE_MESSAGES_FORBIDDEN` errors or resending captionless voice plus text when captions are too long.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:349](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/telegram/src/bot/delivery.replies.ts#L349)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '349,439p'`
- **ElevenLabs produces native voice-note output for Telegram-style targets:** The ElevenLabs speech provider emits `opus_48000_64`, `.opus`, and `voiceCompatible: true` when the synthesis target is `voice-note`.
  - file: [extensions/elevenlabs/speech-provider.ts:455](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/elevenlabs/speech-provider.ts#L455)
  - command: `nl -ba extensions/elevenlabs/speech-provider.ts | sed -n '455,500p'`
- **User-visible TTS observability exists:** `/tts status` reports the last attempt, text length, whether it was summarized, provider/fallback chain, attempt details, latency, and errors.
  - file: [src/auto-reply/reply/commands-tts.ts:297](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/commands-tts.ts#L297)
  - command: `nl -ba src/auto-reply/reply/commands-tts.ts | sed -n '297,340p'`

## Close Comment

Closing from a Codex review because current `main` already implements the requested behavior well enough. In `extensions/speech-core/src/tts.ts:1116`, over-limit auto-TTS replies no longer silently drop out of the speech path: with summary off they are truncated and still synthesized, with summary on they are summarized, and if summarization fails they are truncated and synthesized instead. Successful TTS replies still attach audio in `extensions/speech-core/src/tts.ts:1161`, Telegram sends voice-compatible audio through `sendVoice` in `extensions/telegram/src/bot/delivery.replies.ts:349`, and ElevenLabs returns native voice-note-compatible Opus output in `extensions/elevenlabs/speech-provider.ts:455`. There is also user-visible observability via `/tts status` in `src/auto-reply/reply/commands-tts.ts:297`, which reports the last attempt, provider/fallback chain, and errors. If this is still reproducible on a newer build, please open a fresh issue with the exact `messages.tts` config, `/tts status` output, and verbose logs so maintainers can chase any remaining provider- or account-specific failure.
