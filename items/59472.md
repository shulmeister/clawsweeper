## Summary

Keep open. Current `main` still lets `channels list --json` fail before emitting JSON when provider-usage loading throws, and PR #59472 is not merge-ready as written because review comments identified unresolved behavior and test-isolation issues.

## Evidence

- **Current main still aborts JSON output on usage-load errors:** `src/commands/channels/list.ts:124-133` still does a direct `await loadProviderUsageSummary(...)` in the JSON branch and only calls `writeRuntimeJson()` afterward. If usage loading throws, JSON output never gets written. By contrast, text mode still degrades gracefully via `loadUsageWithProgress()` at `src/commands/channels/list.ts:91-102`.
  - file: [src/commands/channels/list.ts:124](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/commands/channels/list.ts#L124)
  - command: `nl -ba src/commands/channels/list.ts | sed -n '90,145p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **The usage loader still has real throw paths on main:** `src/infra/provider-usage.load.ts:82-92` still throws when no fetch implementation is available, and `src/infra/provider-usage.load.test.ts:136-149` explicitly asserts that `loadProviderUsageSummary()` rejects with `"fetch is not available"`. That keeps the reported failure mode plausible on current main.
  - file: [src/infra/provider-usage.load.ts:82](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/infra/provider-usage.load.ts#L82)
