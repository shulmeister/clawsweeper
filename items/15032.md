## Summary

Keep #15032 open. Current main still does not expose any per-spawn allow/deny or tool-less selector on `sessions_spawn`: the public tool schema, native spawn params, child session metadata, and subagent policy resolver only support existing global/config-derived subagent policy. The related implementation PR #23506 was closed unmerged, and #23906 remains a narrower open follow-up rather than a shipped replacement.

## Best Possible Solution

Keep #15032 open as the canonical core API/security gap. Implement an additive `sessions_spawn` tool policy parameter such as `tools: { allow?: string[], deny?: string[] }`, optionally including `tools: "none"` if maintainers want to cover #23906's local-model use case. Thread it through the tool schema, `SpawnSubagentParams`, child-session persistence, and `resolveSubagentToolPolicyForSession`; enforce deny-wins and narrow-only semantics; add focused unit/e2e coverage and update the public Sub-agents docs.

## Risks / Open Questions

- No live `sessions_spawn` run or test suite was executed because this review was constrained to read-only inspection; the missing schema, type, wiring, persistence, and docs path is sufficient to keep the feature open.
- The implementation needs explicit maintainer security semantics: per-spawn policy should narrow only, deny should win, and it must not re-grant tools removed by profile, provider, group, sandbox, or hard subagent policy.
