## Summary

Keep open. Current main still lacks per-spawn tool restrictions for `sessions_spawn`: the tool schema and native spawn parameter path do not accept a per-call `tools` allow/deny policy, the subagent policy resolver only reads global `tools.subagents.tools`, and the docs describe only global subagent tool policy. The linked Claw Drive use case and closed-unmerged PR #23506 are relevant evidence that this is a real core API gap, not a cleanup item to close.

## Best Possible Solution

Keep #15032 open. The best path is an additive `sessions_spawn` policy parameter, likely `tools: { allow?: string[], deny?: string[] }`, threaded through `SpawnSubagentParams` into child session/runtime policy and merged with global `tools.subagents.tools` with clear deny-wins semantics, docs, config/schema updates if needed, and focused unit/e2e coverage. PR #23506 and the Claw Drive comment are useful prior art but the implementation needs rebasing/review against current main.

## Risks / Open Questions

- The eventual API shape needs maintainer security judgment: per-spawn allow should not accidentally grant tools filtered out by profile, provider, group, sandbox, or hard subagent deny policy unless explicitly intended.
- If this is implemented, tests should cover both schema/wiring and deny-wins merge semantics for spawned native subagents; ACP/runtime parity should be an explicit product decision.
