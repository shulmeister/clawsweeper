## Summary

Current main still does not implement per-spawn tool restrictions for `sessions_spawn`. The request remains a real core API gap: the schema has no `tools` allow/deny parameter, the native spawn call cannot carry one, child session state stores only depth/role/control metadata, and the subagent policy resolver still reads only global `tools.subagents.tools`. The Claw Drive comment and closed-unmerged PR #23506 are useful prior art, but they do not supersede or ship the feature.

## Best Possible Solution

Keep #15032 open as the canonical core API gap. The best implementation is an additive `sessions_spawn` policy parameter, likely `tools: { allow?: string[], deny?: string[] }`, threaded through the schema, `SpawnSubagentParams`, child session persistence, and `resolveSubagentToolPolicyForSession`, with deny-wins semantics that cannot accidentally re-grant tools removed by profile, provider, group, sandbox, or hard subagent policy. Reuse PR #23506 as prior art, then add focused unit/e2e coverage and update `docs/tools/subagents.md` / public docs.
