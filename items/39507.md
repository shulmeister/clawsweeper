## Summary

Keep open. Current main still computes Feishu `threadReply` from inbound `thread_id`/`root_id` before applying the `replyInThread` setting, then the dispatcher upgrades `threadReply` into `reply_in_thread` delivery. The existing test suite also explicitly expects `replyInThread: true` and `threadReply: true` for `replyInThread: "disabled"` when `thread_id` is present, so PR #39507 is still a live behavior change rather than implemented or obsolete. Related PR #33789 fixed reply target selection, but it did not fully suppress thread replies when the config is disabled; #39507 also has a follow-up commit preserving topic-session context for the review concern.

## Best Possible Solution

Keep #39507 open and resolve the Feishu semantics explicitly. The likely code path is `extensions/feishu/src/bot-content.ts`: make `replyInThread` reflect only the configured value for non-topic groups, suppress `threadReply` for non-topic quote replies when disabled, preserve topic anchoring for `group_topic` and `group_topic_sender`, and update the current regression test that now expects forced thread replies under disabled config. If maintainers prefer the existing follow-existing-thread behavior, rename or document the setting so `disabled` no longer promises fully inline replies.
