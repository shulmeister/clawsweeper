## Summary

Keep this open. Current main still uses the same `abortedLastRun` marker for explicit user stop/no-action aborts and for restart recovery, while recovery paths still synthesize continuation prompts through `callGateway`. The prior ClawSweeper comment remains accurate against current main: this is not ready for conservative closure.

## Best Possible Solution

Keep this issue open. The best fix is to split restart-interruption recovery state from explicit user-abort state: `/stop` and recognized no-action phrases should persist a terminal user-abort or recovery-suppression marker, recovery scans should skip those sessions and clear pending retry state, and focused regression tests should prove `/stop` followed by gateway startup does not call `callGateway` while legitimate gateway-reload recovery still resumes interrupted work.
