## Summary

Keep open. Current main still uses the same hosted docs MCP endpoint and fail-fast command path reported by the issue, and there is no source, docs, test, changelog, or release evidence that OpenClaw has fixed the macOS-specific failure. Live Linux checks now show docs.openclaw.ai TLS is healthy, so the original outage may have been hosted/transient, but that is not enough for a conservative auto-close of a detailed macOS report.

## Best Possible Solution

Keep the issue open until a maintainer or CI lane verifies the current hosted MCP endpoint from macOS. If it now works, close with macOS reproduction proof as cannot-reproduce; if it still fails, either fix the hosted docs TLS configuration or add a resilient `openclaw docs` fallback/diagnostic path in `src/commands/docs.ts` so agents are not blocked by a hosted MCP transport failure.
