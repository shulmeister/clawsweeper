## Summary

Keep #9016 open. Current main has estimated cost reporting and transcript/session cost aggregation, but it still does not expose OpenRouter's provider-reported per-response `usage.cost` / `cost_details` to agent runtime metadata or `session_status`. The existing surfaces compute local estimates from configured model pricing, and the shared OpenAI-compatible stream parser ignores OpenRouter-specific cost fields.

## Best Possible Solution

Keep #9016 open. Capture OpenRouter raw usage at the shared OpenAI-compatible response/stream boundary, preserve provider-reported `usage.cost` separately from OpenClaw's local pricing estimate with source and units, include `cost_details.upstream_inference_cost` and `prompt_tokens_details.cache_write_tokens`, carry those fields through transcript/session usage and `EmbeddedPiAgentMeta`, and expose per-turn plus cumulative values through `session_status` or a dedicated usage/cost tool. Telegram and other reply footers should prefer exact OpenRouter cost when available and fall back to current estimates otherwise.

## Risks / Open Questions

- Closing this now would conflate OpenClaw's estimated cost surfaces with exact OpenRouter provider billing data.
- The eventual implementation needs explicit source/units modeling so OpenRouter credits or provider-reported charges are not mixed silently with local USD estimates.
