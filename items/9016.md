## Summary

Keep #9016 open. Current main has local cost estimates, transcript/session cost aggregation, and per-response usage footers, but it still does not capture OpenRouter's provider-reported `usage.cost`, `usage.cost_details.upstream_inference_cost`, or `prompt_tokens_details.cache_write_tokens` from the final streaming usage chunk and expose that exact per-message cost through agent runtime metadata. The ClawRouter comment is a useful external workaround, and #7006/#9696 are related, but neither supersedes this narrower provider-reported OpenRouter cost request.

## Best Possible Solution

Keep #9016 open. The best implementation is to capture OpenRouter `usage.cost`, `usage.cost_details`, and cache-write token details at the OpenAI-compatible streaming transport boundary, preserve provider-reported cost separately from local estimates in transcript/session usage, and expose it through a structured agent-runtime surface such as `session_status` details or a dedicated usage/cost tool. Telegram and other reply footers can then render exact provider cost when available and fall back to existing local estimates otherwise.
