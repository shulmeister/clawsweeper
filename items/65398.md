## Summary

Keep open. Current main still has the Feishu behaviors this PR is trying to address, and the earlier related PR #56520 is closed/unmerged rather than a canonical shipped fix. However, this PR is not safe to close as implemented because the unresolved review comment identifies a real lifecycle problem with the module-level registration guard: activating reloads clear plugin-registered state, so a process-wide one-shot flag can leave Feishu tools/hooks absent after reload. The right maintainer path is to revise the implementation around registry-load idempotency and Feishu named-account config promotion, then review/land or close based on that updated patch.

## Best Possible Solution

Keep this PR open, but require a revision before merge: preserve Feishu top-level shared policy fields for named-account configs via the current setup-promotion contract, add regression coverage for the legacy migration case, and replace the module-level Feishu registration guard with a registry-load-safe idempotency/replace strategy that still re-registers hooks/tools after activating plugin reloads. If a newer PR takes over both fixes, then this PR can be closed as superseded by that canonical item.
