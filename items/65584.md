## Summary

Current `main` already routes gateway CLI teardown through `stopAndWait()` before resolving cron-related gateway calls, and the same implementation plus regression tests are present in the `v2026.4.23` release commit.

## Evidence

- **Current main waits for teardown:** `stopGatewayClient()` awaits `client.stopAndWait({ timeoutMs: 1_000 })` and only falls back to `client.stop()` on failure, addressing the reported fire-and-forget shutdown path.
  - file: [src/gateway/call.ts:95](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/call.ts#L95)
  - command: `nl -ba src/gateway/call.ts | sed -n '1,180p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Gateway request resolution is gated on teardown:** `executeGatewayRequestWithScopes()` clears the wrapper timer, calls `stopGatewayClient(client)`, and resolves or rejects only in that `.finally(...)` callback, so successful CLI calls no longer return before teardown is finished.
  - file: [src/gateway/call.ts:487](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/call.ts#L487)
