## Summary

Keep #39673 open. Current main still preserves a real Linux gateway install failure path for `systemctl --user unavailable: Failed to connect to bus` / no-medium when the direct user-bus call and machine-scope fallback cannot recover, and the install command still reports that as a raw install failure. Existing coverage only proves the preliminary service probe can be nonfatal; it does not prove the real install path succeeds or emits actionable remediation when `assertSystemdAvailable`, `daemon-reload`, `enable`, or `restart` hit the same bus-unavailable class. The related #63561 is useful adjacent systemd-user context, but it is not an exact duplicate of this Chinese-locale/no-medium install report.

## Best Possible Solution

Keep #39673 open and fix the Linux gateway install path directly: recover when safe by deriving/repairing the user bus environment or consistently targeting the correct user scope, and when recovery is impossible fail with explicit headless/Linux systemd remediation hints in `gateway install`. Add a regression test where the real install path, not just the preliminary `isLoaded` probe, sees `Failed to connect to bus` / no-medium during availability or activation.
