## Summary

Keep open. Current main has partial stale ACP binding cleanup, including Telegram restart cleanup for missing/terminal/error entries and Discord startup reconciliation for missing ACP metadata, but I do not see a high-confidence fix for the exact reported persistent stub shape: an existing session row that still carries deliveryContext/origin after ACP metadata is cleared or the session ends. The linked fix PR #60420 is still open in the provided GitHub context, and current main still appears to preserve non-acp session fields when ACP metadata is removed/reset.

## Evidence

- **Telegram startup cleanup is partial:** Telegram removes stale ACP bindings on reload only when the ACP session entry is absent, terminal failed/killed/timeout, or has acp.state === "error". A present stub row with no acp metadata but retained routing fields is not classified stale by this predicate.
