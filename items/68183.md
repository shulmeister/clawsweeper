## Summary

Keep open. Current main adds a newer Docker setup path and a local-build workaround, but the official Docker release workflow still does not build published images with sandbox Docker CLI support, and sandbox runtime code still depends on invoking `docker` directly. I do not have high-confidence evidence that the reported Windows Docker Desktop sandbox failure is fixed on current main.

## Evidence

- **Published Docker images are not built with sandbox Docker CLI enabled:** Current release workflow builds and pushes the default amd64/arm64 images without any `OPENCLAW_INSTALL_DOCKER_CLI` build arg. The only explicit build arg in this workflow is `OPENCLAW_VARIANT=slim` for the slim image job.
  - file: [.github/workflows/docker-release.yml:153](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/.github/workflows/docker-release.yml#L153)
  - command: `nl -ba .github/workflows/docker-release.yml | sed -n '153,180p'; nl -ba .github/workflows/docker-release.yml | sed -n '270,298p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Docker CLI install is still opt-in at image build time:** The main Dockerfile installs `docker-ce-cli` and `docker-compose-plugin` only when `OPENCLAW_INSTALL_DOCKER_CLI` is set. Current source does not show a baked-in legacy Docker CLI, but it also does not make sandbox CLI support the default for published images.
