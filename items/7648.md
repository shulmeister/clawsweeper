## Summary

Current main still does not implement #7648. The browser Control UI/WebChat path still auto-scrolls to the bottom when following new chat content, only exposes a downward "New messages" affordance, and has no persisted UI setting or gateway config for off/top behavior. The native Swift WebChat view also remains bottom-pinned. The related #44130 is TUI-specific and does not supersede this browser/native WebChat request; the latest issue comment already kept #7648 open for the same gap.

## Best Possible Solution

Keep #7648 open. A conservative implementation should add an explicit chat scroll preference for Control UI and native WebChat, likely with a persisted UI setting and/or documented `gateway.webchat`/`gateway.controlUi` config, then thread bottom-follow, off/top behavior, and optionally a Scroll to Top control through `scheduleChatScroll`, Control UI rendering, and `OpenClawChatView`. Cover send, streaming, refresh, initial load, manual-scroll, and native WebChat states with focused tests.
