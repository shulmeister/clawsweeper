## Summary

Keep this PR open. Current main still has the Windows-sensitive direct `fs.rename` path in memory atomic reindex, so the intended fix is not implemented. The PR targets the right owner file and is linked to concrete bug #64187, but the submitted patch is not mergeable as-is because it breaks template literals, sidecar suffix handling, and backup path uniqueness.

## Best Possible Solution

Keep this PR open for revision, or replace it with a clean maintainer patch. Port the existing cron-store retry/fallback behavior into `extensions/memory-core/src/memory/manager-atomic-reindex.ts`, preserve base/`-wal`/`-shm` suffix handling and the `randomUUID()` backup path, add focused atomic-reindex tests for `EBUSY` retry and Windows copy fallback, and keep #64187 as the canonical bug until a corrected implementation lands.
