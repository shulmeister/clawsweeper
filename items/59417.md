## Summary

Issue #59417 is asking for WeChat-specific voice-message behavior in the external `@tencent-weixin/openclaw-weixin` channel plugin. Current `main` already provides the generic `asVoice`/`audioAsVoice` plumbing to channel plugins, and OpenClaw docs/vision explicitly place channel-specific outbound behavior in plugins rather than core.

## Evidence

- **WeChat is explicitly external to core:** The WeChat docs say OpenClaw connects through Tencent's external `@tencent-weixin/openclaw-weixin` plugin, that the WeChat code does not live in the core repo, and that WeChat-specific runtime concerns are owned by the external plugin.
  - file: [docs/channels/wechat.md:10](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/channels/wechat.md#L10)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Channel plugins own outbound delivery:** The channel plugin SDK guide says plugins own outbound sending to the platform, while core owns the shared `message` tool and dispatch.
  - file: [docs/plugins/sdk-channel-plugins.md:21](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/plugins/sdk-channel-plugins.md#L21)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Core already forwards the voice-send intent:** The shared message tool exposes `asVoice`, the channel outbound contract includes `audioAsVoice`, delivery passes that flag into plugin outbound sends, and tests cover forwarding it through generic plugin media delivery.
  - file: [src/agents/tools/message-tool.ts:119](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/message-tool.ts#L119)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Outbound adapter supports plugin handling:** `ChannelOutboundContext` includes `audioAsVoice`, which is the seam external channel plugins use when deciding whether to send voice-note style media vs a normal attachment.
  - file: [src/channels/plugins/outbound.types.ts:14](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/channels/plugins/outbound.types.ts#L14)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Delivery path proves flag propagation:** `deliverOutboundPayloads` preserves `audioAsVoice` in send overrides for channel delivery, so the missing behavior is not in the shared core transport path.
  - file: [src/infra/outbound/deliver.ts:870](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/infra/outbound/deliver.ts#L870)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Regression test covers generic plugin forwarding:** The outbound delivery test asserts that `audioAsVoice: true` is passed through to a plugin's `sendMedia` implementation.
  - file: [src/infra/outbound/deliver.test.ts:603](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/infra/outbound/deliver.test.ts#L603)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Project vision prefers plugin-owned optional capability:** VISION says optional capability should usually ship as plugins, with discovery/promotion in ClawHub rather than core.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/VISION.md#L52)
