## Summary

Keep open. Current `main` still clears the QQBot session on close code `4009`, so the behavior this PR is trying to fix is not implemented yet; the bug report is also consistent with the current reconnect/resume flow.

## Evidence

- **4009 still forces session clear on current main:** `ReconnectState.handleClose` groups `SESSION_TIMEOUT` with `INVALID_SESSION` and `SEQ_OUT_OF_RANGE`, logs `will re-identify`, and returns `clearSession: true` plus `refreshToken: true` for all three cases.
  - file: [extensions/qqbot/src/engine/gateway/reconnect.ts:133](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/qqbot/src/engine/gateway/reconnect.ts#L133)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/reconnect.ts | sed -n '133,151p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Resume depends on preserving session state:** The live gateway path sends `RESUME` only when both `sessionId` and `lastSeq` are present, and `handleClose` clears both fields whenever `action.clearSession` is true. With the current `4009` close action, reconnect falls back to identify rather than resume.
  - file: [extensions/qqbot/src/engine/gateway/gateway-connection.ts:313](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/qqbot/src/engine/gateway/gateway-connection.ts#L313)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/gateway-connection.ts | sed -n '313,357p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Upstream QQBot contract says 4009 is resumable:** The QQ Bot websocket error-code table says `4009` means the connection expired and the client should reconnect and execute `resume`; its simple handling section also says `4009` can re-initiate resume.
- **Related field report is still open:** Issue `#55631` remains open and reports `4009 Session timed out` followed by session clearing and downstream session-reset symptoms; it also has a corroborating follow-up comment from another user hitting the same pattern.

## Risks / Open Questions

- This draft PR was authored against the older `extensions/qqbot/src/gateway.ts` layout from the PR patch, while current `main` handles this logic in `extensions/qqbot/src/engine/gateway/reconnect.ts`; even if the idea is right, it needs a rebase/port to the current structure.
- I did not find a focused current-main regression test covering `4009` resume semantics in the QQBot gateway path, so the behavior gap is easy to regress.
