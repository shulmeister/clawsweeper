## Summary

Keep open. Current `main` still has the reported lazy-import failure mode in `ui/src/ui/app-render.ts`, so this is not implemented on main, and the PR as written has unresolved correctness issues called out in review.

## Evidence

- **Current main still lacks rejection handling in `createLazy`:** `createLazy()` on current `main` stores only `loader().then(...)` in `s.promise` and never clears or handles a rejected import promise. After one rejection, `s.mod` stays `null` and future calls skip reloading because `s.promise` remains set.
  - file: [ui/src/ui/app-render.ts:161](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/ui/src/ui/app-render.ts#L161)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '155,176p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Null lazy module renders nothing:** `lazyRender()` returns `nothing` whenever the getter returns `null`, which matches the blank-view symptom once a lazy module never resolves.
  - file: [ui/src/ui/app-render.ts:217](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/ui/src/ui/app-render.ts#L217)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '216,220p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Proposed PR fix has unresolved retry-loop risk:** PR commit `86f1baef8dad4b0bb5a04bbef599ca15215b5502` adds a `.catch()` that resets `s.promise = null` and immediately calls `_pendingUpdate()`. The existing review correctly notes this can cause unbounded re-import/re-render retries for persistent failures, so the PR is not strong evidence for safe closure.
  - file: [ui/src/ui/app-render.ts:120](https://github.com/openclaw/openclaw/blob/86f1baef8dad4b0bb5a04bbef599ca15215b5502/ui/src/ui/app-render.ts#L120)
  - sha: [86f1baef8dad](https://github.com/openclaw/openclaw/commit/86f1baef8dad4b0bb5a04bbef599ca15215b5502)
- **Proposed PR fix also papers over a type mismatch:** The same PR returns `null as any` from the catch path even though `LazyState<T>.promise` is `Promise<T> | null`, leaving an acknowledged type-safety gap unresolved.
  - file: [ui/src/ui/app-render.ts:124](https://github.com/openclaw/openclaw/blob/86f1baef8dad4b0bb5a04bbef599ca15215b5502/ui/src/ui/app-render.ts#L124)
  - sha: [86f1baef8dad](https://github.com/openclaw/openclaw/commit/86f1baef8dad4b0bb5a04bbef599ca15215b5502)

## Risks / Open Questions

- The underlying blank Control UI failure mode still appears real on current `main` for any lazy view import that rejects.
- Closing this PR would discard a useful lead while the bug remains unimplemented on `main`.
- Merging the PR unchanged would trade the silent failure for a possible infinite retry/error-log loop on deterministic import failures.
