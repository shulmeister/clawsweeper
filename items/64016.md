## Summary

Keep open. Current main still has the reported behavior shape: the wake layer queues a wake that arrives while a heartbeat handler is already running, then drains it immediately afterward; the runner only applies the not-due window to interval-triggered runs, while targeted/session wakes go straight to runOnce and advance the schedule. Existing duplicate-output suppression happens after a model run, so it does not address the reported extra heartbeat/API/git-checkpoint runs.

## Best Possible Solution

Keep this issue open and fix it in the heartbeat scheduler/wake path with targeted regression coverage. The likely product-safe direction is a per-agent in-window cooldown for automatic external wakes queued during an interval heartbeat, while preserving explicitly requested immediate wakes where docs promise `mode now` behavior. Add a scheduler/wake test that simulates an interval heartbeat in progress, queues an exec/session wake, and proves it does not trigger a second heartbeat run inside the configured interval unless an explicitly supported immediate wake mode requires it.

## Risks / Open Questions

- A broad suppression of all non-interval wakes could regress documented manual `system event --mode now`, cron `--wake now`, exec completion, and notification-event delivery semantics.
- Existing duplicate heartbeat payload suppression is post-model-call behavior, so relying on it would still leave the reported wasted API calls and checkpoint churn.
