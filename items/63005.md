## Summary

Keep open. Current main still has the split-prone boundary described by the report: inbound explicit session initialization preserves and writes raw external keys, while gateway key resolution canonicalizes non-agent raw keys to agent-prefixed store keys and can migrate/prune the raw key as a legacy alias. That is enough to keep this as a real product bug rather than close it as already implemented or covered by the narrower related WebUI UUID issue.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make inbound explicit-key initialization and gateway `sessions.resolve`/`chat.send` use the same store-key contract, or preserve raw external aliases that future inbound traffic still uses. Add a regression test that creates a raw key such as `conversation:pair:user_a::user_b:space:space_123`, resolves/sends through the gateway path that currently canonicalizes it, then verifies a subsequent inbound explicit raw key reuses the original `sessionId` and transcript instead of creating a new one.
