## Summary

Current `main` still has `openclaw nodes list` and `openclaw nodes status` backed by different sources of truth, so the reported mismatch remains plausible. I did not find strong evidence that issue #49719 is fully fixed on `main` or in `v2026.4.22`.

## Evidence

- **`nodes list` still counts only `node.pair.list` entries:** The `list` command calls `node.pair.list`, parses `{ pending, paired }`, and prints `Pending: ... · Paired: ...` from that pairing-store result. It only consults `node.list` when `--connected` or `--last-connected` filters are used.
  - file: [src/cli/nodes-cli/register.status.ts:325](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/nodes-cli/register.status.ts#L325)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '315,420p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **`nodes status` uses `node.list`, not `node.pair.list`:** The `status` command calls `node.list` and computes `Known / Paired / Connected` from that payload, so `status` and `list` are still using different RPC methods on current `main`.
  - file: [src/cli/nodes-cli/register.status.ts:125](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/nodes-cli/register.status.ts#L125)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '114,225p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **`node.list` merges multiple pairing/runtime sources:** Gateway `node.list` builds a catalog from device pairing, node pairing, and live connected nodes before returning results, so it can legitimately know about nodes that are not coming solely from the node-pairing store.
  - file: [src/gateway/server-methods/nodes.ts:706](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/gateway/server-methods/nodes.ts#L706)
  - command: `nl -ba src/gateway/server-methods/nodes.ts | sed -n '697,718p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Catalog marks nodes as paired from device pairing or node pairing:** `createKnownNodeCatalog` sets `paired: Boolean(devicePairing ?? nodePairing)`, confirming that `node.list` can report a paired node even when the node-pairing store is not the only source involved.
  - file: [src/gateway/node-catalog.ts:117](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/gateway/node-catalog.ts#L117)
  - command: `nl -ba src/gateway/node-catalog.ts | sed -n '91,120p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Docs explicitly say the stores are separate:** The docs say `nodes status` marks a node as paired when its device pairing role includes `node`, and separately state that `node.pair.*` is a separate gateway-owned node pairing store that does not gate the WebSocket connect handshake.
  - file: [docs/nodes/index.md:44](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/docs/nodes/index.md#L44)

  - file: [src/cli/nodes-cli/rpc.ts:73](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/nodes-cli/rpc.ts#L73)
