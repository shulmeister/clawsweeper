## Summary

Keep #38597 open. Current main at c7a0d9b1889ca09ab5bfd84cda39a8d50d3d5768 still uses whitespace-only Discord long-line splitting and falls back to a hard 2000-character split for no-whitespace CJK text. The discussion already identifies the root cause and an active fix path in PR #71384, but that PR has not landed on main.

## Best Possible Solution

Keep #38597 open and use PR #71384 as the active fix path. The best implementation belongs in extensions/discord/src/chunk.ts: preserve whitespace priority for Latin and mixed text, add a CJK terminal/closing punctuation fallback, add a safe CJK character-boundary fallback with a minimum-progress guard, and land focused Discord chunk tests for Chinese, Japanese, Korean, mixed-script, punctuation, long-token, preserveWhitespace, newline-mode, and surrogate-pair cases.
