## Summary

Keep #69668 open. Current main supports isolated multi-agent routing and the chat session picker already groups sessions by individual agent, but there is no first-class team/project parent concept, no agent team/group metadata in the public agent config or agents.list protocol, and the Sessions view remains a flat sortable table. The related #45086 work is useful evidence of shipped multi-agent/session switching, but it does not cover the requested team/project hierarchy across multiple agents.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A coherent implementation would add a backwards-compatible agent grouping field such as agents.list[].team/project (or an explicit groups collection), expose it through agents.list and any relevant sessions.list metadata, then update the Control UI agent/session selectors and Sessions view to render collapsible team/project groups while preserving today’s isolated per-agent workspaces and session stores.
