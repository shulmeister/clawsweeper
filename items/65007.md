## Summary

Keep open. Current `main` still shows the reported Google Chat gaps: add-on parsing only handles `chat.messagePayload`, Google Chat access control intentionally preserves `allowlist` when a routed space has no sender allowlist, and reply threading still resolves `[[reply_to_current]]` from the inbound message id while Google Chat sends expect a thread resource name.

## Evidence

- **Add-on parser still only handles messagePayload:** `parseGoogleChatInboundPayload` only rewrites add-on requests when `rawObj.chat?.messagePayload` exists. Valid add-on variants like `addedToSpacePayload` or `removedFromSpacePayload` are not mapped here and would fall through to the later `eventType`/`space` checks.
