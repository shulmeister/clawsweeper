## Summary

Keep this issue open. Current main still has the narrow Google Chat add-on parser, still preserves allowlist sender policy for routed spaces with no sender allowlist, and still has a plausible reply_to_current path that can pass a Google Chat message resource where sendGoogleChatMessage expects a thread resource. The related PR #65058 is open/unmerged and explicitly only partially addresses the payload variant portion.

## Best Possible Solution

Keep the issue open. The best fix belongs in the bundled Google Chat plugin: accept add-on message, added-to-space, and removed-from-space payload variants; remove the Google Chat sender-policy special case so wildcard routed spaces without groupAllowFrom use resolveSenderScopedGroupPolicy; and normalize outbound Google Chat reply threading so [[reply_to_current]] cannot pass a message resource as thread. Add focused regression tests for addedToSpacePayload/removedFromSpacePayload, groups["*"] with requireMention and no groupAllowFrom, and Google Chat reply_to_current/thread-resource delivery.
