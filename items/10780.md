## Summary

Current main already provides a browser-based Control UI plus CLI-generated dashboard links for remote/VPS use, including tokenized deep links and documented SSH/Tailscale access paths.

## Evidence

- **Browser Control UI is already a first-class feature:** The docs describe a browser-based Control UI served by the Gateway on port 18789, with auth and device pairing for browser access.
  - file: [docs/web/control-ui.md:9](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/web/control-ui.md#L9)

- **Remote/VPS access path is documented:** The remote-access guide explicitly covers an always-on Gateway on a VPS/home server and recommends Tailscale Serve for the Control UI or SSH tunneling as fallback.
  - file: [docs/gateway/remote.md:25](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/gateway/remote.md#L25)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)

  - file: [src/commands/dashboard.ts:45](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/commands/dashboard.ts#L45)

## Close Comment

Closing as implemented on current `main`. Codex review found that OpenClaw already ships a browser-based Control UI served by the Gateway (`docs/web/control-ui.md:9`), documents remote VPS/tailnet/SSH access for that UI (`docs/gateway/remote.md:25`), and provides `openclaw dashboard --no-open` to print a dashboard link, including `#token=...` when appropriate (`src/commands/dashboard.ts:45`, `docs/cli/dashboard.md:13`). The browser UI also accepts tokenized deep links and strips the token from the URL after bootstrap (`ui/src/ui/app-settings.ts:187`, covered by `ui/src/ui/app-settings.test.ts:346`). This is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
