## Summary

Current main already implements the conservative provider-registry reuse path that this PR’s final diff attempts to add: runtime provider resolution goes through the generic runtime registry resolver, that resolver returns a compatible active registry before loading plugins, and provider hook resolution is memoized before calling into the loader. The final PR diff is therefore redundant on current main.

## Best Possible Solution

Close this PR and keep the current generic resolver plus provider-hook cache implementation. If #62051-like CPU saturation still reproduces on a current release, it should be reopened as a fresh focused bug with current process counts, logs, config shape, and affected plugin/provider details rather than landing this redundant compatible-registry precheck.
