## Summary

Current main still does not implement per-agent subagent concurrency. The config schema and typed agent entry do not accept `agents.list[].maxConcurrent` or `agents.list[].subagents.maxConcurrent`, the only subagent concurrency resolver reads `agents.defaults.subagents.maxConcurrent`, and native plus ACP `sessions_spawn` continue to dispatch through the shared `subagent` lane. The issue discussion also has a second user confirming the same problem and a prior Codex review keeping it open; the current `d54d2d6b` checkout still matches that conclusion.

## Best Possible Solution

Keep #8061 open. The best core path is to add a validated per-target-agent subagent concurrency setting in `agents.list[]` (likely nested under `subagents.maxConcurrent` to avoid ambiguity with global main-run concurrency), update generated schema/help/docs, and enforce it for both native and ACP `sessions_spawn` scheduling alongside the existing global `CommandLane.Subagent` cap. Add focused tests proving a capped target such as `tts-agent` serializes while uncapped target agents can still run concurrently.

## Risks / Open Questions

- The feature needs a clear product decision on whether the per-agent cap applies to all runs for that agent id or only runs spawned via `sessions_spawn` targeting that id.
- Implementation should cover both native and ACP subagent paths, otherwise users could bypass the cap by selecting `runtime: "acp"`.
