## Summary

Keep #8061 open. Current main still only exposes and enforces a global subagent concurrency cap through `agents.defaults.subagents.maxConcurrent`; strict per-agent config does not accept `agents.list[].maxConcurrent` or `agents.list[].subagents.maxConcurrent`, and native plus ACP `sessions_spawn` both dispatch child runs onto the shared `subagent` command lane.

## Best Possible Solution

Keep #8061 open. Add a validated per-target-agent subagent concurrency setting, preferably `agents.list[].subagents.maxConcurrent`; update types, strict schema, generated config schema/help, and docs; then enforce it in both native and ACP `sessions_spawn` paths alongside the existing global `CommandLane.Subagent` limit. Add focused tests proving a capped target such as `tts-agent` serializes while uncapped targets can still run concurrently.

## Risks / Open Questions

- Maintainers still need to choose the public config shape; `agents.list[].subagents.maxConcurrent` is clearer than a top-level `agents.list[].maxConcurrent` because the requested cap is about spawned subagent runs.
- Runtime semantics should define whether the cap applies only to child runs targeting that agent id or to every run under that agent id.
