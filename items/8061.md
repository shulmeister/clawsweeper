## Summary

Current main still does not implement per-target-agent subagent concurrency. `maxConcurrent` exists only on global defaults, `agents.list[]` remains strict without that field, and both native and ACP subagent spawns still enter the single shared `subagent` lane without a target-agent-specific limiter. The existing maintainer comment keeping #8061 open still matches current `d74f897`.

## Best Possible Solution

Keep #8061 open. The best core path is to add a validated and documented target-agent concurrency setting under `agents.list[]` (using maintainer-approved naming, with clear semantics for whether it caps all runs for that agent id or only subagent target runs), update generated schema/help/docs, and enforce it in native and ACP `sessions_spawn` scheduling alongside the existing global `CommandLane.Subagent` cap. Add focused tests proving one capped target such as `tts-agent` serializes while uncapped targets can still run concurrently.
