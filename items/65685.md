## Summary

Keep open. Current `main` still clears the shared typing controller in the `enqueue-followup` path even when the original run is still active, which matches the reported Discord liveness regression. The latest shipped release still contains the same code, repo docs say typing should remain visible while queued work waits its turn, and I did not find a covering test for this case.

## Risks / Open Questions

- Typing-capable channels such as Discord can show no immediate liveness signal for a queued followup even though the message is retained.
- If the active run later errors or times out, the queued message can appear dropped from the user's perspective.
- The behavior conflicts with current queue/typing docs, so closing as implemented or cannot-reproduce would be inaccurate.
