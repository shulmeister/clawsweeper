## Summary

Keep open. Current main does not have a first-class tool-plugin setup hook for global setup/onboarding: the manifest setup descriptor and setup registry are still provider/CLI-backend shaped, the public plugin API has no tool setup registration method, and the existing generic plugin-config wizard deliberately skips sensitive fields such as API keys.

## Best Possible Solution

Keep the issue open as a plugin-SDK/core setup design item. The likely best path is an additive manifest-first tool-plugin setup descriptor plus a setup-only API/helper that can prompt for SecretInput-backed credentials and write through a documented plugin-owned config path, with docs and contract tests alongside the existing provider and channel setup surfaces.

## Risks / Open Questions

- The current generic `uiHints` plugin-config step overlaps part of the desired UX, but it cannot collect sensitive API keys and does not provide a descriptor-first tool setup contract.
- The eventual design needs maintainer judgment on where credentials should be stored, especially because current provider helpers write provider auth profiles while tool plugins generally use `plugins.entries.<id>.config` or other plugin-owned config paths.
