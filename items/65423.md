## Summary

Keep this PR open. Current main still builds embedded-run auth profile candidates in deterministic order for both top-level and spawned runs, so the thundering-herd problem described by the PR is not implemented on main. The PR is a focused contributor fix with a real code path and a bot review that found the production idea sound, but it needs rework before merge because its test duplicates the private helper and the patch must be rebased onto current provider-preferred profile logic.

## Best Possible Solution

Keep the PR open and ask for a small rebase/revision: add a production-owned, testable shuffle or candidate-order helper, apply it only to automatic spawned runs after preserving locked-profile and provider-owned preference semantics, and replace the copied helper test with regression coverage for the actual embedded-run candidate selection behavior.
