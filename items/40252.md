## Summary

Keep this issue open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends, assigns it to the outbound RPC `text` parameter, and current tests still encode that placeholder as the sent media-only text. The two related fix PRs in the provided GitHub context are closed unmerged, so there is no merged implementation candidate that resolves the report.

## Best Possible Solution

Keep this issue open. The best fix belongs in the iMessage plugin send path: stop generating visible `<media:...>` fallback text for media-only sends, send an empty string plus `file` to the iMessage RPC when media exists, preserve text+media captions and reply IDs, and update echo/self-chat dedupe tests so they no longer depend on user-visible placeholder text.
