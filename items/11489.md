## Summary

Keep open. Current main still does not implement group pairing: `groupPolicy` has no `pairing` mode, the pairing CLI/store remain sender-code oriented, Telegram and WhatsApp group paths drop unauthorized group traffic instead of creating owner approval requests, and docs continue to describe pairing as DM-only. Related #8495 and #42400 are useful context but are open and only cover narrower group-join/never-reply pieces.

## Best Possible Solution

Keep #11489 open for maintainer/product direction or implementation. The likely path is a core/channel contract change: extend the group policy/config schema and docs, add pending group approval storage and CLI approve/reject semantics, teach Telegram/WhatsApp group join or first-seen group paths to notify the owner, and decide whether the requested read-only reader-agent mode should reuse existing group sandbox/tool policies or needs a new runtime route. #8495 and #42400 can inform narrower pieces, but neither currently supersedes the full group pairing workflow.
