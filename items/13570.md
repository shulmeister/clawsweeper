## Summary

Keep #13570 open. Current main at 2f6615d2ee9d0d9db0f4ad1e13f71d1c0df8166d still has no `contextPruning.forcePruneRatio` or equivalent high-usage TTL bypass: the config schema/types omit the field, the runtime settings have no parsed value for it, and the context-pruning extension returns before pruning whenever the cache TTL window is still fresh. The related PR #13572 directly matched the request but the provided GitHub context shows it was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open until the requested behavior is implemented or a maintainer explicitly rejects the product direction. The likely implementation path is to add `agents.defaults.contextPruning.forcePruneRatio` to the config types, Zod/generated schema, effective settings, and docs; estimate current context usage before the fresh-TTL return in `src/agents/pi-hooks/context-pruning/extension.ts`; bypass the TTL only when the configured ratio is met; and add regression coverage showing a fresh TTL still prunes at high usage while normal TTL behavior remains unchanged below the threshold.
