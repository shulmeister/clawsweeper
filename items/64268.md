## Summary

Keep open. Current main still sends `llm_output` once after an embedded attempt with cumulative usage, not once per LLM iteration. The newer `model_call_started` / `model_call_ended` hooks help with sanitized per-call telemetry and likely cover the related latency request, but they do not expose per-call usage, assistant output, or tool-call data requested here. Related PR #45990 remains relevant but is not merged or implemented on main.

## Best Possible Solution

Keep this issue open and implement an additive, documented per-call observability contract. The safest path is either a new `llm_call` hook fired from assistant-message/model-call boundaries with per-call usage, provider, model, run/session/agent ids, and optional tool-call data, or a backwards-compatible `llm_output` expansion that preserves the existing aggregate event with an explicit aggregate marker. Update hook types, runner/subscription threading, docs, and focused tests, and coordinate with the still-open PR #45990 and the related context-window follow-up #64260.

## Risks / Open Questions

- Changing `llm_output` from aggregate to per-call would be a plugin compatibility risk unless existing aggregate semantics are preserved or clearly flagged.
- The current `model_call_ended` hook should not be treated as a full substitute for this request because it intentionally omits raw output and token usage.
