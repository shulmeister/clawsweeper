## Summary

Keep open. Current `main` still does not implement the requested behavior: `session_status` selection logic in `buildStatusMessage` ignores `channels.modelByChannel` when choosing the displayed selected model, even though reply routing still applies channel overrides. The existing test coverage also still misses the divergent-model case described in the PR.

## Evidence

- **Selected model still comes only from per-session override or agent default:** `buildStatusMessage` resolves `selectedProvider`/`selectedModel` from `entry?.providerOverride` / `entry?.modelOverride` or the agent default model. There is no earlier `resolveChannelModelOverride` application before the status card model is chosen.
  - file: [src/status/status-message.ts:478](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/status/status-message.ts#L478)

- **Channel override logic only emits a note after the selected model already matches it:** Later in the same function, `channelModelNote` resolves the channel override but returns `undefined` unless the resolved override already equals `selectedProvider`/`selectedModel`. That means it cannot fix the displayed model when the override differs from the default.
  - file: [src/status/status-message.ts:816](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/status/status-message.ts#L816)
