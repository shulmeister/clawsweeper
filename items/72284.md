## Summary

Keep #72284 open. Current main has a real gap between the documented/user-visible Dreams contract and the REM implementation: a default dreaming config writes separate REM phase reports, while `DREAMS.md` is only created by the later narrative path when a subagent run or request-scoped fallback appends a diary entry. The Control UI reads only `DREAMS.md`/`dreams.md`, so the reported state of `memory/dreaming/rem/YYYY-MM-DD.md` existing while the Dreams tab is empty is still plausible on current main.

## Best Possible Solution

Keep the issue open and add a focused memory-core regression around the default `dreaming.enabled`/separate-report REM path: after a successful REM run that writes `memory/dreaming/rem/YYYY-MM-DD.md`, either `DREAMS.md` should be created/updated with a user-readable diary or phase summary, or the Control UI/docs contract should be changed so the Dreams tab can surface separate REM reports when the diary narrative is unavailable. The likely fix belongs in `extensions/memory-core/src/dreaming-phases.ts`/`dreaming-narrative.ts` plus gateway/UI tests for `doctor.memory.dreamDiary` behavior.
