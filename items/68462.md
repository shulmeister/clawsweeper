## Summary

Keep #68462 open. Current main has newer last-known-good pollution checks, but the PR's specific `.bak` restore guard is still missing from `maybeRecoverSuspiciousConfigRead` and its sync twin. The PR is not obsolete: it targets a real remaining restore path, though it should be rebased onto the current helper shape and address the existing review comments about secret-field and mask coverage.

## Best Possible Solution

Keep the PR or canonical issue active. The best fix is to adapt the PR onto current main by validating parsed `.bak` content in both async and sync `maybeRecoverSuspiciousConfigRead` paths before `copyFile`, using a shared detector with the current last-known-good promotion/recovery checks, then add regression tests for polluted `.bak` refusal and the reviewed sensitive-field/mask edge cases.
