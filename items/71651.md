## Summary

Keep PR #71651 open. Current main still lacks the PR's reload-time schedule comparison/invalidation, and the stale `nextRunAtMs` path remains plausible because force-reload ticks merge `jobs-state.json` runtime state into edited `jobs.json` definitions and then collect due jobs with `skipRecompute: true`. The PR discussion also shows maintainer follow-up addressed the bot review concerns with semantic schedule comparison plus broader regression coverage and docs.

## Best Possible Solution

Keep #71651 open and land this PR or an equivalent main-branch fix after rebase/validation. The right product/code path is to compare the hydrated job against the previous in-memory job after `loadCronStore` merges runtime state, invalidate `nextRunAtMs` only when scheduling inputs semantically change, and keep the regression/docs coverage added by the maintainer fixups.
