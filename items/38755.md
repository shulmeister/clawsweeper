## Summary

Current `main` already changed manual `cron.run` from a long-blocking RPC into an immediate enqueue/ack flow, which addresses the reported false 60s timeout behavior.

## Evidence

- **Gateway now enqueues manual runs:** `cron.run` no longer waits for job completion in the gateway handler; it calls `context.cron.enqueueRun(...)` and responds with that result.
  - file: [src/gateway/server-methods/cron.ts:381](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/gateway/server-methods/cron.ts#L381)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '357,390p'`
- **Service returns immediate ack and runs work in background:** `enqueueRun` schedules the work on the cron command lane and immediately returns `{ ok: true, enqueued: true, runId }` instead of blocking on the run finishing.
  - file: [src/cron/service/ops.ts:697](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cron/service/ops.ts#L697)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '697,732p'`
- **Regression test covers the timeout-shaped case:** Gateway test explicitly asserts that `cron.run` returns within `1_000` ms while isolated cron work continues in the background and later emits the finished event.
  - file: [src/gateway/server.cron.test.ts:896](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/gateway/server.cron.test.ts#L896)
  - command: `nl -ba src/gateway/server.cron.test.ts | sed -n '896,953p'`
- **CLI/docs reflect async completion flow:** Docs now state that `openclaw cron run` returns as soon as the manual run is queued and that callers should use `cron runs` to inspect the eventual outcome.
  - file: [docs/cli/cron.md:41](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/docs/cli/cron.md#L41)
  - command: `nl -ba docs/cli/cron.md | sed -n '35,60p'`
- **Shipped in release notes:** The `2026.3.7` changelog includes: `Cron/manual run enqueue flow: queue cron.run requests ... return immediate { ok: true, enqueued: true, runId } acknowledgements ... and document the asynchronous completion flow.`
  - file: [CHANGELOG.md:2970](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/CHANGELOG.md#L2970)

## Close Comment

Closing as implemented on `main` after Codex review. The reported false-failure path is already addressed by the current manual cron run flow: `src/gateway/server-methods/cron.ts:381` now routes `cron.run` through `context.cron.enqueueRun(...)`, and `src/cron/service/ops.ts:697` immediately returns `{ ok: true, enqueued: true, runId }` while the actual work continues in the background. There is regression coverage for this exact behavior in `src/gateway/server.cron.test.ts:896`, which asserts that `cron.run` returns within 1 second even while the isolated run finishes later. The async behavior was also documented and shipped in the `2026.3.7` changelog (`CHANGELOG.md:2970`) and in `docs/cli/cron.md:41`.
