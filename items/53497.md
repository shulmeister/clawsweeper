## Summary

Keep open. Current `main` appears to have fixed the plugin-resolution half of the report for user-installed plugins, but it still does not expose `resolvePreferredOpenClawTmpDir`, `buildChannelConfigSchema`, or `normalizeAccountId` from `openclaw/plugin-sdk`, so the missing-export failure described in #53497 is still real on current code.

## Evidence

- **Root plugin-sdk surface is intentionally tiny:** The contract test locks the root runtime surface to exactly five exports: `buildMemorySystemPromptAddition`, `delegateCompactionToRuntime`, `emptyPluginConfigSchema`, `onDiagnosticEvent`, and `registerContextEngine`. The reported symbols are not part of that surface.
  - file: [src/plugins/contracts/plugin-sdk-index.test.ts:96](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/contracts/plugin-sdk-index.test.ts#L96)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current root barrel does not export the reported helpers:** `src/plugin-sdk/index.ts` only re-exports `emptyPluginConfigSchema`, `registerContextEngine`, `buildMemorySystemPromptAddition`, `delegateCompactionToRuntime`, and `onDiagnosticEvent` at runtime. There is no root re-export for `resolvePreferredOpenClawTmpDir`, `buildChannelConfigSchema`, or `normalizeAccountId`.
  - file: [src/plugin-sdk/index.ts:116](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugin-sdk/index.ts#L116)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Compat/root-alias path also omits at least part of the requested API:** The legacy compat surface re-exports `channel-config-schema` helpers, but it does not export `normalizeAccountId` or `resolvePreferredOpenClawTmpDir`. Since the loader aliases `openclaw/plugin-sdk` to the root alias/compat path for installed plugins, those names are still unavailable from the monolithic root.
  - file: [src/plugin-sdk/compat.ts:20](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugin-sdk/compat.ts#L20)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Direct runtime probe on current checkout reproduces missing root exports:** A read-only `node -e` probe against `src/plugin-sdk/root-alias.cjs` reported `typeof resolvePreferredOpenClawTmpDir === 'undefined'`, `typeof buildChannelConfigSchema === 'undefined'`, and `typeof normalizeAccountId === 'undefined'`, while `emptyPluginConfigSchema` was a function.
  - command: `node -e "const root=require('./src/plugin-sdk/root-alias.cjs'); for (const k of ['resolvePreferredOpenClawTmpDir','buildChannelConfigSchema','normalizeAccountId','emptyPluginConfigSchema']) console.log(k, typeof root[k]);"`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **User-installed plugin resolution appears fixed separately:** Current tests explicitly cover resolving `openclaw/plugin-sdk` aliases for user-installed plugins via the running `openclaw` argv hint, and loader tests verify legacy plugins can load with `require('openclaw/plugin-sdk').emptyPluginConfigSchema`. That points to the original `Cannot find module 'openclaw/plugin-sdk'` symptom being addressed, but not the missing-export symptom.
  - file: [src/plugins/sdk-alias.test.ts:751](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/sdk-alias.test.ts#L751)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)

## Risks / Open Questions

- The checkout does not contain built `dist/plugin-sdk/index.js`, so this review relies on current source, contract tests, and a direct root-alias runtime probe rather than inspecting compiled release artifacts.
- Docs now tell plugin authors to avoid monolithic `openclaw/plugin-sdk` imports (`docs/plugins/building-plugins.md:248`), so final resolution may require either more compatibility exports in OpenClaw or a plugin-side migration to scoped subpaths.
