## Summary

Keep this issue open. Current main has partial fixes around Telegram dispatcher failures and context-overflow fallback replies, but the cron no-op case is still modeled as “the run must first return NO_REPLY,” so it remains vulnerable to a final model timeout before delivery suppression can mark the run successful. Gateway PID/port mismatch is also still surfaced as a diagnostic/restart action rather than a clear in-place self-heal while the runtime is running.

## Best Possible Solution

Keep this issue open, or split it into narrower follow-ups: one to validate/add a Telegram context-overflow fallback regression using the reported assistantError/terminated path, one to make isolated cron treat explicit no-op business output such as send=false as a successful terminal result without requiring a final NO_REPLY model turn, and one to decide whether gateway status/doctor/restart should actively self-heal non-owner listeners while the service runtime is still running.
