## Summary

Keep open. Current main still rejects remote device-less Control UI connections and has UI/docs/connection-time guidance, but it does not emit the requested server-side startup/runtime warning for operators likely to hit the secure-context failure. The related open PR #56830 is scoped to protocol/UI copy for #32473 and does not supersede this startup-log request.

## Best Possible Solution

Keep this issue open and add a targeted gateway startup warning after runtime config, bind hosts, TLS state, Tailscale mode, and Control UI config are resolved. The likely implementation belongs in src/gateway/server-startup-log.ts or the post-attach startup path, with coverage in src/gateway/server-startup-log.test.ts. The warning should point operators to HTTPS/Tailscale Serve, localhost or SSH tunnel access, reverse proxy setup, and the break-glass gateway.controlUi.dangerouslyDisableDeviceAuth option while preserving the current auth policy.

## Risks / Open Questions

- The eventual warning predicate needs careful wording to avoid false positives when HTTPS is terminated by Tailscale Serve or a reverse proxy while the local gateway listener remains HTTP.
- The server can infer likely exposure from bind mode, TLS state, Tailscale mode, and Control UI config, but it cannot know every browser origin at startup.
