## Summary

Keep open. Current `main` still hardcodes a 30s announce timeout for `sessions_send({ timeoutSeconds: 0 })`, still runs peer-session follow-up through the A2A announce path, and does not expose a `sessions_send`-specific config or parameter to override that timeout.

## Evidence

- **Fire-and-forget still hardcodes 30s announce timeout:** `sessions_send` accepts only `timeoutSeconds`; when that value is `0`, `announceTimeoutMs` is forced to `30_000` instead of being configurable or derived from another setting.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/sessions-send-tool.ts#L37)
  - command: `nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '35,80p'; nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '225,345p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **The async path still depends on that timeout:** For non-owned peer sessions, the tool marks delivery as pending and launches `runSessionsSendA2AFlow(...)` with the computed `announceTimeoutMs`, so fire-and-forget follow-up delivery is still governed by the same 30s value.
  - file: [src/agents/tools/sessions-send-tool.ts:317](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/sessions-send-tool.ts#L317)
  - command: `nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '225,345p'`

- **A2A flow aborts when the wait window expires:** `runSessionsSendA2AFlow` waits for the target run using `Math.min(params.announceTimeoutMs, 60_000)`, and if no reply is available afterward it returns without producing an announce message.
