## Summary

Keep open. Current main still uses the file-backed outbound delivery queue and startup recovery still replays pending `.json` entries without any configurable age/TTL gate. Recent main has reliability improvements for delivered-marker cleanup, active recovery claims, backoff, max retries, and permanent-error quarantine, but it does not implement the requested `messages.delivery.maxAgeMs` / `expireAction` behavior or an equivalent expiry policy.

## Best Possible Solution

Keep tracking this issue until outbound delivery recovery gets a bounded expiry policy. The likely core fix is to add a documented `messages.delivery` config surface, apply an age gate inside `recoverPendingDeliveries` and reconnect drains before delivery, choose a safe terminal action such as moving expired entries to `failed/`, and add regression tests for old `enqueuedAt` entries, default backward compatibility, and configured expiry behavior.
