## Summary

Keep #16555 open. Current main still uses the file-based outbound delivery queue and has max-retry/backoff handling, delivered-marker cleanup, and reconnect drains, but it does not implement a configurable TTL/expiry policy for stale queued outbound messages. The issue discussion's related SQLite/outbox attempts were closed unmerged, and the recent maintainer/Codex keep-open comment still matches current main at 631552c554095b0c520376db9131dac0f5427f94.

## Best Possible Solution

Keep #16555 open until outbound delivery recovery has a bounded, documented expiry policy. The likely core path is to add optional `messages.delivery` schema/types/help/docs, enforce an `enqueuedAt` age check before delivery in both `recoverPendingDeliveries` and `drainPendingDeliveries`, choose a conservative terminal action such as moving expired entries to `failed/`, and add regression tests for default backward compatibility, stale entries, reconnect drains, max-retry/backoff interaction, and delivered-marker cleanup.

## Risks / Open Questions

- Old `retryCount === 0` queue entries with no `lastAttemptAt` can still replay on startup no matter how old their `enqueuedAt` timestamp is.
- Reconnect drains can still replay stale matching entries if a channel reconnects before startup recovery handles them.
- Because the config schema is strict and has no delivery-expiry key, users cannot opt into the requested native TTL policy today.
