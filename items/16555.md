## Summary

Keep #16555 open. Current main still uses the file-backed outbound `delivery-queue` and replays pending entries based on retry count, backoff, and permanent-error handling, but not message age. The issue discussion includes a concrete Telegram production reproduction, and the related SQLite/outbox PRs that proposed TTL/expiry were closed unmerged or superseded without this behavior landing on main.

## Best Possible Solution

Keep the issue open until OpenClaw adds a bounded, documented outbound delivery expiry policy. The likely core change is optional `messages.delivery` schema/types/help/docs, an age check against `enqueuedAt` in both `recoverPendingDeliveries` and `drainPendingDeliveries`, a conservative terminal action such as moving expired entries to `delivery-queue/failed/` with clear logging, and regression tests for default compatibility, stale startup recovery, reconnect drains, retry/backoff interaction, and cleanup behavior.

## Risks / Open Questions

- A future fix must apply expiry consistently to both startup recovery and reconnect drains, otherwise stale entries can still replay through the path that was not updated.
- Default behavior needs explicit product choice: preserving current replay semantics by default is safer for compatibility, but it leaves operators without native stale-message hygiene until they opt in.
