## Summary

Keep #15022 open. Current main still lacks the requested opt-in final/tool-interleaved text coalescing mode: final payload assembly preserves multiple assistant text blocks as separate replies, dispatch sends each reply independently, block replies are still flushed before tool execution, and the hold/resume API proposed in linked PR #16275 is absent from main.

## Best Possible Solution

Keep #15022 open as the focused core reply-pipeline issue. The best implementation is an opt-in coalescing or final-only mode that covers both final embedded-run assistantTexts and block-streamed text around tool execution, likely with hold/defer semantics for sequential and parallel tool calls, config schema/docs, and regression tests for text -> tool -> text -> tool -> text turns. PR #16275 is useful prior art but was closed unmerged and is not present on main.
