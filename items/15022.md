## Summary

Current main still preserves the split-message behavior reported in #15022. The related hold/resume PR was closed unmerged, and the active reply pipeline flushes buffered block replies before tool execution and across assistant message index boundaries rather than coalescing interleaved text/tool/text segments into one outbound message.

## Best Possible Solution

Keep #15022 open as the canonical core reply-pipeline issue. The likely fix belongs in the block reply pipeline/embedded subscribe path: hold or defer block-reply flushing across tool execution, support sequential and parallel tool calls, and add regression tests proving text/tool/text assistant turns can deliver as one outbound message without breaking media, reasoning, and explicit boundary behavior.
