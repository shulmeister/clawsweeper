## Summary

Keep open. Current `main` does not implement the PR’s poll idempotency fix, and the PR also includes at least one incorrect MCP ownership change plus a cron trust change that is a no-op on current code. That mix is not safe for conservative cleanup or auto-close.

## Evidence

- **Poll handler still lacks inflight dedupe:** `poll` only checks `context.dedupe` after completion, then awaits channel resolution and `outbound.sendPoll(...)` before writing the dedupe entry. Two concurrent requests with the same `idempotencyKey` can both miss the cache and both dispatch a poll.
  - file: [src/gateway/server-methods/send.ts:595](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server-methods/send.ts#L595)
  - command: `nl -ba src/gateway/server-methods/send.ts | sed -n '592,684p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Inflight helper exists elsewhere, but poll does not use it:** `resolveGatewayInflightMap` and `runGatewayInflightWork` already provide shared in-flight dedupe for `message.action` and `send`, which makes `poll` the obvious outlier rather than proving the bug obsolete.
  - file: [src/gateway/server-methods/send.ts:63](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server-methods/send.ts#L63)
  - command: `nl -ba src/gateway/server-methods/send.ts | sed -n '63,99p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Main derives MCP ownership from bearer token:** Loopback MCP auth resolves `senderIsOwner` from the owner/non-owner bearer token and `resolveMcpRequestContext(...)` forwards that value directly. Hardcoding `senderIsOwner: false` would change current owner-token behavior, not match it.
  - file: [src/gateway/mcp-http.request.ts:63](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/mcp-http.request.ts#L63)
  - command: `nl -ba src/gateway/mcp-http.request.ts | sed -n '60,177p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Tests on main assert owner-token true and spoofed-header false:** Current tests explicitly cover both cases: owner token -> `senderIsOwner: true`, non-owner token + spoofed header -> `senderIsOwner: false`. That contradicts the PR’s test change that expected `false` for the owner-token path.
  - file: [src/gateway/mcp-http.test.ts:190](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/mcp-http.test.ts#L190)

  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Cron trust default change is a no-op on current code:** `buildGatewayCronService` forwards `trusted: opts?.trusted`, and `enqueueSystemEvent(...)` already stores `trusted: options.trusted !== false`, so `undefined` already means trusted. Changing the caller to `opts?.trusted ?? true` does not fix a real behavior gap on `main`.
