## Summary

Keep #39341 open. Current main still omits top-level `channels.discord.allowFrom` from the Discord native slash-command audit's owner allowlist check, even though runtime authorization, schema/docs, and account resolution treat that top-level field as valid. The related fix PR #39387 was closed unmerged, and the latest release tag has the same audit logic.

## Best Possible Solution

Implement a focused Discord plugin fix in `extensions/discord/src/security-audit.ts`: include normalized top-level `discordCfg.allowFrom` in `ownerAllowFromConfigured` alongside legacy `dm.allowFrom` and the pairing/store allowlist, preserving current account-resolution precedence. Add a regression case in `extensions/discord/src/security-audit.test.ts` showing `channels.discord.commands.native.no_allowlists` is not emitted when top-level `allowFrom` contains a Discord user ID. Keep #39341 as the canonical tracker; #39387 is closed unmerged and #67222 appears to be a duplicate of this issue.
