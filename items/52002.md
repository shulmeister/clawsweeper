## Summary

Current `main` already fixes the Web UI provider-prefix bug by carrying provider-qualified model values through the picker state and sending the selected qualified value unchanged in `sessions.patch`.

## Evidence

- **Picker options are provider-qualified from catalog entries:** `buildQualifiedChatModelValue()` prefixes each catalog entry with its own provider, `buildChatModelOptionFromLookup()` uses that value for the option, and duplicate bare IDs are resolved against the catalog instead of the session default provider.
  - file: [ui/src/ui/chat-model-ref.ts:5](https://github.com/openclaw/openclaw/blob/665cc3d5371c275b32c2c400b54238e5d96f6761/ui/src/ui/chat-model-ref.ts#L5)
  - command: `nl -ba ui/src/ui/chat-model-ref.ts | sed -n '1,220p'`
  - sha: [665cc3d5371c](https://github.com/openclaw/openclaw/commit/665cc3d5371c275b32c2c400b54238e5d96f6761)
- **Select state keeps qualified values in the dropdown:** `buildChatModelOptions()` adds the catalog-derived option values directly, so the Web UI dropdown now carries qualified refs rather than bare model IDs.
  - file: [ui/src/ui/chat-model-select-state.ts:58](https://github.com/openclaw/openclaw/blob/665cc3d5371c275b32c2c400b54238e5d96f6761/ui/src/ui/chat-model-select-state.ts#L58)
  - command: `nl -ba ui/src/ui/chat-model-select-state.ts | sed -n '55,110p'`

- **Model switch sends selected qualified value unchanged:** `switchChatModel()` writes `createChatModelOverride(nextModel)` to local state and sends `model: nextModel` in `sessions.patch`, so the chosen provider prefix is preserved instead of being recomputed from the active session provider.
  - file: [ui/src/ui/chat/session-controls.ts:234](https://github.com/openclaw/openclaw/blob/665cc3d5371c275b32c2c400b54238e5d96f6761/ui/src/ui/chat/session-controls.ts#L234)
