## Summary

Keep open. Current `main` still gates WhatsApp group messages on the raw inbound body before any audio transcription runs, so voice notes that arrive as `<media:audio>` can be skipped before the transcript is available for mention matching.

## Evidence

- **WhatsApp group gating happens before message processing:** The WhatsApp inbound handler calls `applyGroupGating(...)` for group chats and returns immediately when `shouldProcess` is false, before `processForRoute(...)` runs.
  - file: [extensions/whatsapp/src/auto-reply/monitor/on-message.ts:132](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/whatsapp/src/auto-reply/monitor/on-message.ts#L132)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/on-message.ts | sed -n '80,170p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Mention check reads raw `msg.body` and can drop the message:** `applyGroupGating` strips mentions and runs `debugMention` against `params.msg.body`, then skips the message when `mentionDecision.shouldSkip` is true. There is no transcription call in this path.
  - file: [extensions/whatsapp/src/auto-reply/monitor/group-gating.ts:129](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/whatsapp/src/auto-reply/monitor/group-gating.ts#L129)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/group-gating.ts | sed -n '110,230p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **WhatsApp audio bodies are still placeholders:** Inbound WhatsApp audio messages are normalized to the literal placeholder `<media:audio>`. That is the body value the gating code sees unless something rewrites it first.
  - file: [extensions/whatsapp/src/inbound/extract.ts:284](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/whatsapp/src/inbound/extract.ts#L284)
  - command: `nl -ba extensions/whatsapp/src/inbound/extract.ts | sed -n '276,292p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Audio transcription runs later in the generic reply pipeline:** The reply path applies media understanding only after inbound context is finalized in `get-reply.ts`; `src/media-understanding/apply.ts` is where audio transcripts are written back into `CommandBody`/`RawBody`. If group gating returns early, this stage is never reached.
  - file: [src/auto-reply/reply/get-reply.ts:246](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/auto-reply/reply/get-reply.ts#L246)

  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Preflight transcription exists for other channels, not WhatsApp:** `transcribeFirstAudio` is explicitly documented as running before mention checking and is wired into Telegram and Discord, but repo search found no WhatsApp caller for that helper.
  - file: [src/media-understanding/audio-preflight.ts:10](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/media-understanding/audio-preflight.ts#L10)
