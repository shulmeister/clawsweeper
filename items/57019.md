## Summary

Keep open. Current `main` and the latest release still delete the in-memory held-lock entry before async lock-file cleanup completes, `shouldTreatAsOrphanSelfLock()` still keys off `!HELD_LOCKS.has(...)`, and the existing tests do not cover the overlapping release/reacquire interleaving described in the issue.

## Evidence

- **Current release path still removes map entry before async cleanup:** On current `main`, `releaseHeldLock()` still calls `HELD_LOCKS.delete(normalizedSessionFile)` before awaiting `held.handle.close()` and `fs.rm(held.lockPath)`. That leaves a window where the process-scoped map says no lock is held while the old lock file is still being cleaned up.

## Risks / Open Questions

- A concurrent reacquire can still observe `HELD_LOCKS` as empty before the old lock file has been removed, which is the core race the issue reports.
- Background watchdog cleanup increases the chance of this interleaving because it can release a held lock asynchronously without a caller explicitly awaiting `release()` first.
- There is still no regression test proving the old cleanup path cannot remove a newly acquired lock file during overlap.
