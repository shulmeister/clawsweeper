## Summary

Current main already preserves WebChat/TUI session identity across reconnects via canonical main-session normalization, persisted session selection, and reconnect-time history reloads; the same implementation is present in released v2026.4.22.

## Close Comment

Closing as implemented on current OpenClaw. This Codex review checked both `main` and the shipped `v2026.4.22` tag and found that the reconnect/session-continuity path is already in place: the gateway now publishes a canonical `sessionDefaults.mainSessionKey` (`src/gateway/server/health-state.ts:21`), WebChat/Control UI normalizes `main` aliases and preserves the selected `sessionKey` / `lastActiveSessionKey` across reconnects and refreshes (`ui/src/ui/app-gateway.ts:180`, `ui/src/ui/storage.ts:126`, `ui/src/ui/app-settings.ts:333`, `ui/src/ui/controllers/chat.ts:139`), and TUI reconnects by reloading history for the existing `currentSessionKey` instead of starting a fresh one (`src/tui/tui.ts:573`, `src/tui/tui.ts:1046`, `src/tui/tui-session-actions.ts:288`). Gateway regression coverage also explicitly checks the alias/round-trip bug that used to orphan main sessions on reconnect (`src/gateway/session-utils.test.ts:100`, `src/infra/state-migrations.session-roundtrip.test.ts:29`). If you still see a continuity loss on a current build, please open a new issue with the exact surface (Control UI vs macOS app vs TUI), the session key shown before/after reconnect, and gateway logs around the disconnect/reconnect window.
