## Summary

Current main has partial prompt-cache hygiene work, especially stable inbound metadata v2 and cache-change observability, but it does not fully solve this issue. `sessions_send` still injects source-specific agent-to-agent context through `extraSystemPrompt`; the system prompt builder still appends that to the provider-facing system prompt; and the OpenAI Codex Responses transport still sends the resulting full prompt as top-level `instructions` with the same session-based `prompt_cache_key`. The requested tool-set stability is also not proven: current tool policy still filters owner-only tools such as `cron` from the exposed schema based on authorization/runtime context rather than keeping a stable schema and enforcing only at execution time.

## Best Possible Solution

Keep this issue open. The best fix is a targeted prompt-cache stability change: move `sessions_send`/cron source metadata out of `extraSystemPrompt` and into the triggering message or another non-prefix-critical turn structure, then add a regression test that compares direct and inter-session turns for the same target session on the OpenAI Codex Responses path. Separately, decide whether `cron` and other restricted tools should remain in a stable schema with runtime authorization denial, or document and test an explicit cache-impact exception for owner/source-based tool filtering.

## Risks / Open Questions

- The current inbound-metadata v2 work should be preserved; the remaining fix should target A2A/source-specific context placement and tool-schema stability without reintroducing volatile message identifiers into the system prompt.
- A good regression test should compare the same target session reached through a direct turn and through `sessions_send`/cron on the OpenAI Codex Responses path, asserting stable cache-relevant instructions prefix and stable tool names unless a real session configuration changed.
