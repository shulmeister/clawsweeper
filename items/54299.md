## Summary

Keep this issue open. Current main still has the two reported Gemini CLI OAuth behaviors: an env-provided client ID can be used without a matching client secret, token exchange only includes `client_secret` when present, and the redirect URI/local listener are still fixed to `localhost:8085`. The existing manual fallback is useful mitigation for port conflicts, but it is not configurable or dynamic redirect-port support.

## Best Possible Solution

Keep this open for the bundled Google plugin owner. The best fix is to make the Gemini CLI OAuth client-secret contract explicit: either reject or warn when an env client ID is provided without a matching secret, or document and test public-client mode as intentional. Separately, add a supported redirect URI/port configuration path if compatible with Google OAuth registration, or document and test the existing manual paste fallback as the intended recovery for `EADDRINUSE`.
