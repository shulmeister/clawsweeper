## Summary

Keep open. Current main still emits the QQBot error strings called out in the issue: raw access-token JSON, terse network errors, and missing-config messages without env var names or docs/setup links. The existing QQBot docs and manifest have the needed public setup information, but the runtime error paths do not surface it yet. The single issue comment says “Fixed, closed,” but the checked source at current main does not support closing this as implemented.

## Best Possible Solution

Keep #65868 open and implement the guidance in the bundled QQBot plugin runtime paths. A good fix would centralize QQBot setup/troubleshooting text, update token-fetch, API-client network, gateway, and outbound missing-config errors to mention `QQBOT_APP_ID`, `QQBOT_CLIENT_SECRET`, `openclaw configure`, https://docs.openclaw.ai/channels/qqbot, and q.qq.com where relevant, while preserving raw API details in debug logs or causes. Add focused tests around the changed error messages and decide whether `QQBOT_DEBUG` should be documented or kept internal.
