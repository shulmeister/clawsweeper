## Summary

Keep #14874 open. Current main still lacks an optional SIE warn/strict verification gate in the TypeScript skill loader or config surface, and the directly related implementation PR #14893 was closed unmerged. Existing safeguards cover path containment, size limits, allowlists, installer scans, and some ClawHub/plugin archive integrity paths, but they do not classify skills as unsigned/valid-signed/invalid-signed before prompt, slash command, snapshot, sandbox sync, or runtime exposure.

## Best Possible Solution

Keep #14874 open. The best path is a disabled-by-default TypeScript skill-loader verification policy, or a first-class loader-time verification hook/API, that makes deterministic warn/strict decisions for unsigned, valid-signed, and invalid-signed skills before they are exposed to prompts, slash commands, skill snapshots, sandbox sync, or runtime use. The closed PR #14893 is useful prior art, but any implementation should align with the current loader, config schema, docs, tests, and maintainer security design.

## Risks / Open Questions

- This is security/product design work: maintainers still need to decide envelope format, trust roots, revocation/update behavior, deterministic reason codes, warning UX, and whether the best shape is built-in policy or a first-class loader verification hook.
- Adjacent install-time and archive-integrity safeguards should not be treated as equivalent to loader-time SIE verification, because they do not cover all local/workspace/personal/plugin skill roots or post-install tampering before prompt/runtime exposure.
