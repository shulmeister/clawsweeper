## Summary

Keep open. Current main has several contention mitigations for heartbeat scheduling, Feishu startup probing, and subagent lane limits, but the reported high-load Gateway CPU saturation is not proven fixed. Subagents still share the Gateway process, default subagent concurrency is still 8, and there is no source, test, changelog, or related-item evidence that the Feishu multi-account heartbeat/subagent saturation scenario has been resolved.

## Best Possible Solution

Keep the issue open. The next maintainer step should be a current-main reproduction or benchmark for the reported five-agent Feishu WebSocket heartbeat workload, then either tune/serialize heartbeat-triggered subagent work, add a lower heartbeat-specific subagent cap/backpressure policy, or move CPU-heavy subagent execution out of the single Gateway event-loop path. Add regression/perf coverage around heartbeat fanout plus subagent lane pressure before closing.

## Evidence

Defaults are 4 main agent runs and 8 subagent runs, which mitigates unbounded fanout but does not prove the Gateway process will remain responsive under the reported CPU-saturation workload.
