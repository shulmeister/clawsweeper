## Summary

Current `main` already handles Telegram `streaming: "partial"` turns that contain both reasoning/thinking and final text. The dispatcher now buffers a final answer when a reasoning segment for the same turn has been hinted but not yet delivered, then flushes that buffered answer after the reasoning lane is sent, which directly addresses the reported `[thinking, text]` drop. Targeted regression tests on `main` and in the latest release commit cover the exact missing-final-text scenario.

## Close Comment

Closing this as a Codex review because current OpenClaw already implements the missing final-text delivery path for Telegram `streaming: "partial"` turns that contain both reasoning and answer text. In `extensions/telegram/src/bot-message-dispatch.ts`, the dispatcher now buffers a final answer when reasoning for the same step was hinted but not yet delivered, then flushes that buffered answer after the reasoning lane sends (`flushBufferedFinalAnswer()` around lines 781-830 on current `main`). The current tests also cover the reported behavior directly: `extensions/telegram/src/bot-message-dispatch.test.ts:2144` verifies that a final answer still gets delivered after a reasoning preview, and `extensions/telegram/src/bot-message-dispatch.test.ts:2442` verifies mixed thinking+answer final payloads are split and both messages are dispatched. The latest release commit we were given (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, `v2026.4.22`) already contains that same logic and regression coverage, so there is not a remaining maintainer action to take on this issue.
