## Summary

Current main still lacks a configurable URL or hostname deny list for `web_fetch`. The public config types, strict runtime schema, generated schema/help labels, docs, and runtime fetch path expose only existing fetch limits plus `ssrfPolicy.allowRfc2544BenchmarkRange`; the shared SSRF guard has hardcoded internal-host blocking and hostname allowlist support, but no user-configurable deny list for ordinary public domains.

## Best Possible Solution

Keep this issue open and implement a core `web_fetch` hostname deny-list surface if maintainers want this behavior. A solid path is to add a typed config key such as `tools.web.fetch.denyHosts`, validate it in strict runtime and generated schemas, add labels/help/docs, and enforce exact plus `*.domain` matching in the guarded fetch path before DNS and again on redirects. Focused tests should cover config acceptance/rejection, fetch guard behavior, redirect handling, and concise `web_fetch` execution errors.
