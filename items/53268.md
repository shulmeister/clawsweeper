## Summary

Keep open. Current `main` does not implement a configurable `web_fetch` hostname/URL deny list, and the request is still a valid core feature request rather than a stale or mis-scoped item.

## Evidence

- **Hardcoded hostname blocking only:** The shared SSRF layer still hardcodes a small blocked-host set (`localhost`, `localhost.localdomain`, `metadata.google.internal`) plus `.localhost`/`.local`/`.internal` suffix blocking. There is no denylist field in the SSRF policy type.
  - file: [src/infra/net/ssrf.ts:39](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/net/ssrf.ts#L39)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '39,190p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`web_fetch` runtime only forwards one SSRF option:** `web_fetch` narrows its runtime `ssrfPolicy` to `allowRfc2544BenchmarkRange` only, and only forwards that one flag into the guarded fetch path. No configurable hostname deny/allow settings are exposed here.
  - file: [src/agents/tools/web-fetch.ts:274](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/tools/web-fetch.ts#L274)

- **Config schema has no `tools.web.fetch.denyHosts`:** The public config type for `tools.web.fetch` exposes `ssrfPolicy.allowRfc2544BenchmarkRange` only; there is no `denyHosts`/denylist option in the shipped config surface.
  - file: [src/config/types.tools.ts:561](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/config/types.tools.ts#L561)
