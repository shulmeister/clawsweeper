## Summary

Current `main` already fixes this class of WhatsApp outbound-send failure by moving listener/controller state onto a process-global registry and adding regression coverage for duplicate-module and split-runtime boundaries; the changelog says it shipped in `v2026.3.23`.

## Close Comment

Closing as implemented on `main`. This was a Codex review: current WhatsApp runtime no longer relies on a module-scoped active-listener `Map`. Instead, `extensions/whatsapp/src/connection-controller-registry.ts:11` stores the registry on `globalThis` via `Symbol.for("openclaw.whatsapp.connectionControllerRegistry")`, and `extensions/whatsapp/src/active-listener.ts:15` resolves outbound listener access through that shared registry. There is also regression coverage for both duplicate module instances (`extensions/whatsapp/src/connection-controller-registry.test.ts:12`) and split light/heavy runtime boundaries (`src/plugins/runtime-plugin-boundary.whatsapp.test.ts:125`). `CHANGELOG.md:2350` records this fix under `v2026.3.23` as the active-listener registry being pinned to a `globalThis` singleton so split WhatsApp bundle chunks share one listener map. If this still reproduces on a current build, please open a fresh issue with the exact release/build and current logs.
