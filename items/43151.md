## Summary

Keep PR #43151 open. Current main still uses the old exact-fingerprint rule and even has a test asserting that a failed `write` remains as `lastToolError` after a successful `write` to a different target, so the PR's central behavior is not implemented or clearly superseded.

## Best Possible Solution

Keep PR #43151 open for maintainer review. The best path is either to land or rework its same-tool/same-action-type retry clearing with regression tests, or choose an alternate UX that shows the failed attempt plus later success; current main does not yet solve the different-target retry case.
