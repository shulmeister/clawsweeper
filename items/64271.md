## Summary

Keep this issue open. Current main still lacks the requested per-profile launch-failure circuit breaker/backoff for managed Chrome CDP startup failures. The browser stack now has natural-language no-retry hints for persistent dispatcher failures, but the local launch path still clears only concurrent in-flight state and explicitly permits a later `ensureBrowserAvailable()` call to launch again after a failure.

## Best Possible Solution

Keep the issue open and implement the browser-owned fix in `extensions/browser`: add per-profile consecutive launch/readiness failure tracking in the availability layer, short-circuit further managed launches for a cooldown or disabled-until-restart window after a small threshold, reset the state on successful readiness, and add focused tests proving repeated `ensureBrowserAvailable()` calls do not keep spawning Chrome after persistent launch/CDP failures. Keep the existing no-retry model hint as defense in depth, but do not rely on it as the fix.
