## Summary

Keep #70325 open. Current main has improved the specific memory-core dreaming case by moving the managed dreaming cron to an isolated lightweight agent turn and migrating stale managed main-session dreaming jobs, but the broader reported behavior for main-session `systemEvent` cron jobs still exists: non-agent cron jobs retain the 10-minute default timeout, `executeJobCoreWithTimeout` still reports `cron: job execution timed out`, and `timeoutSeconds` is only modeled for `agentTurn` payloads, not main-session `systemEvent` jobs.

## Best Possible Solution

Keep the issue open for maintainer triage unless the reporter confirms it was only the managed memory-core dreaming cron now migrated by the current memory-core implementation. A concrete fix would either document the 10-minute main-session `systemEvent` cron ceiling and recommend isolated `agentTurn` for long maintenance work, or add a supported timeout override/long-running mode for main-session cron jobs while preserving the scheduler safety guard.
