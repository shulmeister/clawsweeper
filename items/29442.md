## Summary

Keep #29442 open. Current main has substantial prompt-injection hardening for external content, cron hook payload wrapping, tool policy, sandbox docs, and plugin hooks, and the ClawMoat link in the discussion is useful defense-in-depth. But the requested cron-integrated dual-LLM worker/manager privilege separation and strict sanitizer/router path is not implemented on main, and the current hook catalog still lacks the proposed post-external-content-wrap sanitizer hook from the closed, unmerged #30046 PR.

## Best Possible Solution

Keep this open as the canonical maintainer/product decision point. The best path is either to design and ship a narrow external-content sanitizer hook that lets a ClawHub plugin implement the worker-model sanitizer safely, or to implement a first-class cron/read-untrusted-content workflow that runs a tool-disabled reader agent and passes only structured, provenance-tagged output to the privileged cron agent.

## Risks / Open Questions

- Current main may be sufficient for many practical prompt-injection mitigations, but closing this would lose the remaining design question: whether OpenClaw should add a first-class cron sanitizer/worker-manager workflow or a narrower plugin API seam.
- An external skill/plugin such as ClawMoat can provide runtime scanning today, but the issue and related PR discuss deeper privilege separation; current plugin hooks do not cleanly implement the proposed pre-agent external-content sanitizer router.
