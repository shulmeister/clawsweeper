## Summary

Keep #29442 open. Current main has meaningful prompt-injection hardening, but it still relies on wrapping, warnings, suspicious-pattern logging, tool policy, sandboxing, and operator guidance rather than the requested default cron reader/manager privilege separation or a plugin seam that can sanitize/block wrapped external content and then continue into the privileged manager turn. The linked ClawMoat plugin and closed PR #30046 are useful related work, but they do not make the core security/product request obsolete on current main.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is to add either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with strict structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built and then continue safely. ClawHub plugins such as ClawMoat can then provide scanner/sanitizer implementations on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate external-content wrapping and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- Closing as clawhub-only would be premature because current plugin hooks do not provide a clean external-content sanitizer hook that can replace or block content and then continue the privileged turn.
- The final design needs maintainer security/product judgment on default cron behavior, fail-open versus fail-closed sanitizer policy, latency/cost, and whether this should be first-class core flow or a narrow plugin API.
