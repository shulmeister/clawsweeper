## Summary

Keep #29442 open. Current main has useful prompt-injection hardening for external content, cron hook payloads, tool allowlists, plugin hook policy, and security docs, but it still sends wrapped untrusted cron/hook content into the normal privileged cron agent turn. I did not find the requested read-only/tool-disabled worker flow, strict sanitizer handoff, or a focused plugin hook that can replace/block wrapped external content before the privileged prompt is submitted. The linked ClawMoat plugin is useful defense-in-depth, and the closed/unmerged #30046 shows a plausible direction, but neither makes the core item obsolete.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is to add either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built. ClawHub plugins such as ClawMoat can then provide scanner/sanitizer implementations on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate marker/special-token wrapping and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- Closing as clawhub would be premature because ClawMoat-style scanners are useful, but the current plugin surface does not expose a narrow post-wrap external-content replacement/blocking seam for cron hook payloads before privileged prompt submission.
- The final fix needs maintainer security/product judgment on fail-open versus fail-closed behavior, latency/cost, default cron/web-tool policy, and whether the core change should be a first-class reader-worker flow or a documented plugin API seam.
