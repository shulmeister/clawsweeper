## Summary

Keep open. The report is recent and has a concrete memory-file example for repeated `[object Object]` after `/new`; current main has a related prompt-body guard, but the OpenAI-compatible streaming path still appends `choice.delta.content` without a runtime string/structured-content normalization guard, so this is not safe to close as implemented.

## Best Possible Solution

Keep the issue open and fix the provider-stream boundary: add a regression test with an OpenAI-compatible/Mistral-style chunk whose `delta.content` is structured or otherwise non-string, normalize/extract text before appending in `src/agents/openai-transport-stream.ts`, and consider a defensive session-memory/dreaming sanitizer or doctor repair for already-contaminated transcripts.

## Evidence

- **Issue discussion is specific enough to preserve:** The reporter supplied version 2026.4.22, local Mistral/Gemma models, tool usage and `/new` as triggers, plus a memory-file excerpt where assistant lines contain repeated `[object Object]` before normal text. The follow-up says the failure is random, not absent.
- **Current stream path can still stringify non-string deltas:** Current main checks only `if (choice.delta.content)` and passes it directly to text handling; the append helper concatenates the value into a string. If a provider/proxy emits structured content here, JavaScript string concatenation can still produce `[object Object]`.
