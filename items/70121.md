## Summary

Keep open. #70121 is a recent, detailed third-party plugin report with concrete timing/log evidence and a plausible missing or unclear core plugin subprocess lifecycle contract. Current main documents plugin services and lifecycle hooks, but I did not find a shipped core/API fix that makes a plugin-owned long-running child process safe from the reported Gateway lifecycle interaction, nor a canonical duplicate that already tracks this exact camofox autoStart failure.

## Best Possible Solution

Keep #70121 open for maintainer/plugin-SDK follow-up. Reproduce with the camofox-browser plugin, decide whether it should migrate to documented `registerService` plus `gateway_start`/`gateway_stop`, and if that still cannot safely host a long-running child server, add a narrow documented plugin subprocess lifecycle API with tests for detached children, stdio handling, shutdown, and health-failure behavior.
