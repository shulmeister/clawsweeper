## Summary

Keep #17424 open. Current main still has only partial dimension plumbing: `outputDimensionality` is accepted and carried into memory provider creation, but the OpenAI/OpenAI-compatible memory embedding provider does not send a `dimensions` field for inline `/embeddings` requests or OpenAI batch JSONL requests. The latest issue comment keeping this open remains accurate against f4e6322649a1ef0a9b47bd9d8b681c143850d669.

## Best Possible Solution

Keep #17424 open. Implement the OpenAI-compatible memory embedding path by deciding whether `dimensions` should be a supported alias or whether docs should standardize on `outputDimensionality`, carrying the resolved value into `RemoteEmbeddingClient`/OpenAI client/runtime cache identity, adding `dimensions` to supported inline and batch `/embeddings` request bodies, and adding tests for defaults, per-agent overrides, inline requests, batch JSONL, and reindex/cache identity.

## Risks / Open Questions

- Provider compatibility needs a clear policy: OpenAI-compatible `dimensions` should be sent only where supported or documented, with validation/fallback behavior for providers that reject it.
- Changing embedding dimensions must affect cache/reindex identity because stored vector dimensions have to remain consistent.
