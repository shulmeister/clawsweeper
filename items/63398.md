## Summary

Keep this issue open. Current main has several OpenAI tool-schema improvements, but they do not fully cover the reported LiteLLM-routed OpenAI + MS365 MCP failure. Bundle MCP tools are materialized from MCP `inputSchema` after the provider-owned tool-schema normalization pass, LiteLLM is a proxy-style `openai-completions` provider without OpenAI strict-tool hooks, and the generic transport normalization still preserves nested `type: ["string", "null"]` arrays unless Gemini-specific cleanup is selected.

## Best Possible Solution

Keep this issue open and fix it with a regression test that feeds a synthetic MCP tool schema matching the suspected MS365 shape through the LiteLLM/openai-completions path. The code fix should either normalize bundle MCP tools after materialization with provider/model context, or add an explicit OpenAI-compatible/LiteLLM schema sanitizer that handles `type` arrays, malformed `required`, missing object fields, and unsupported combinators without breaking non-OpenAI-compatible providers. It would also be useful to preserve or log the full provider error body so the exact failing tool/schema path is visible.
