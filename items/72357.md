## Summary

Keep open. Current main does dispose bundled LSP runtimes at attempt cleanup, but the LSP runtime itself starts stdio servers without a detached process group and disposes them with only LSP shutdown/exit plus direct `child.kill()`. That is not enough to conservatively guarantee nested descendants such as `tsserver` are terminated on Linux.

## Best Possible Solution

Keep the issue open and fix the bundled LSP stdio lifecycle to match the proven MCP stdio cleanup pattern: spawn POSIX LSP servers into a disposable process group, perform graceful LSP shutdown/exit with a bounded wait, then call `killProcessTree` for still-running servers so descendants such as `tsserver` are terminated. Add focused tests for LSP spawn options and dispose fallback, plus a regression case that proves tree cleanup is requested.

## Risks / Open Questions

- A narrow fix should avoid calling Unix group kill on the current non-detached LSP child, because that can target the gateway process group.
- There does not appear to be a dedicated `pi-bundle-lsp-runtime` regression test for descendant cleanup; existing attempt tests mock the LSP runtime instead of exercising its process lifecycle.
