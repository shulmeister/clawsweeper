## Summary

Keep #69196 open. Current main still implements the Control UI microphone dictation button with browser `SpeechRecognition`/`webkitSpeechRecognition`, and its error path silently clears recording state. The already-shipped same-origin `Permissions-Policy` header helps ordinary microphone access, but the reporter specifically narrowed this issue to Web Speech returning `not-allowed` even after microphone permission was granted and after trying a local microphone header change.

## Best Possible Solution

Keep the issue open and address the Control UI dictation path directly: either add a robust fallback from browser audio capture to a gateway/provider STT path, or at minimum detect and surface `not-allowed` with actionable UI/docs. The existing `Permissions-Policy` header should remain, but it is not sufficient evidence to close this specific Web Speech failure report.

## Evidence

- **Reporter narrowed failure to Web Speech denial:** Issue comments report Chrome microphone permission is granted, `window.SpeechRecognition` and `window.webkitSpeechRecognition` exist, but manually starting `webkitSpeechRecognition()` on the OpenClaw page fails with `ERROR not-allowed`; the reporter also said a local gateway header change allowing microphone access did not resolve the deeper Web Speech failure.
- **Dictation still depends on browser SpeechRecognition:** The current STT helper resolves `SpeechRecognition ?? webkitSpeechRecognition`, creates that browser recognizer, and calls `recognition.start()`; there is no MediaRecorder/getUserMedia or gateway/provider transcription fallback in this path.
