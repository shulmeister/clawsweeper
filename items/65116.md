## Summary

Current `main` treats channel-specific outbound behavior as plugin-owned. The shared `message` runner can still report `handledBy: "core"` while delegating the actual send/media work to the selected channel plugin, and the exact `account default not started` error string does not exist in the core tree. This makes the XMPP failure belong with the external XMPP plugin/extension rather than `openclaw/openclaw` core.

## Evidence

- **Vision: channels belong in plugins, not core:** Project direction says optional capabilities that need channels should ship as plugins, with plugin promotion/discovery living in ClawHub.
  - file: [VISION.md:64](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/VISION.md#L64)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Channel plugin contract: plugin owns outbound text/media/polls:** Channel plugin docs say core keeps one shared `message` tool, while the plugin owns config, security, pairing, session grammar, outbound sending, and threading.
  - file: [docs/plugins/sdk-channel-plugins.md:21](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/docs/plugins/sdk-channel-plugins.md#L21)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **`handledBy: core` does not mean built-in channel logic:** `executeSendAction(...)` first tries plugin action handling, then falls back to `sendMessage(...)` and still labels that path `handledBy: "core"`. That label is the shared runner path, not proof that the channel implementation is in core.
  - file: [src/infra/outbound/outbound-send-service.ts:125](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/infra/outbound/outbound-send-service.ts#L125)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Direct sends/media delegate into the selected channel plugin:** `sendMessage(...)` resolves the selected channel plugin and `deliverOutboundPayloads(...)` loads `plugin.outbound`, then calls the plugin's `sendText` / `sendMedia` adapter methods.
  - file: [src/infra/outbound/message.ts:235](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/infra/outbound/message.ts#L235)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Outbound delivery adapter call site:** Current main builds a handler from `plugin.outbound` and routes media sends through `sendMedia(...)` when present.
  - file: [src/infra/outbound/deliver.ts:174](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/infra/outbound/deliver.ts#L174)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
