## Summary

Keep #40016 open. Current main has lower-level Gateway `sessions.patch` label plumbing, but it still lacks the requested first-class `openclaw sessions rename` or `openclaw sessions set-subject` command, and the existing `openclaw sessions` table remains key-centric.

## Best Possible Solution

Keep the issue open. Implement a focused `openclaw sessions rename <session-key> <label>` command, with any alias such as `set-label` or `set-subject` requiring maintainer product judgment. Reuse the existing Gateway `sessions.patch` label validation/RPC path instead of direct store edits, preserve channel-owned `subject` and `displayName` semantics, expose an effective label/name in `openclaw sessions` text and JSON output, and update `docs/cli/sessions.md` plus targeted CLI tests.
