## Summary

Keep open. Current `main` has partial session-lock recovery hardening, but it does not clearly implement the reported same-process gateway-restart case where `HELD_LOCKS` is lost while the process PID and start time stay the same.

## Evidence

- **Gateway startup cleanup only removes locks already classified as stale:** `startGatewaySidecars()` runs `cleanStaleLockFiles()` during startup, but that helper only deletes entries when `lockInfo.stale` is true. It does not have a dedicated self-owned/current-process sweep.
  - file: [src/gateway/server-startup-post-attach.ts:149](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/gateway/server-startup-post-attach.ts#L149)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '130,210p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Fresh same-process locks are not stale by default:** `inspectLockPayload()` marks a lock stale for dead/recycled PID, missing/invalid metadata, or age over `staleMs`. A lock from the current live process with matching `starttime` and recent `createdAt` is not stale.
  - file: [src/agents/session-write-lock.ts:325](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/session-write-lock.ts#L325)
  - command: `nl -ba src/agents/session-write-lock.ts | sed -n '300,420p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Same-PID orphan reclaim only applies when `starttime` is absent:** `shouldTreatAsOrphanSelfLock()` returns true only when the lock PID equals `process.pid` and the payload does not contain a valid `starttime`. If the process survives and loses `HELD_LOCKS`, a lock written by the current process with valid `starttime` will not be reclaimed.
  - file: [src/agents/session-write-lock.ts:403](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/session-write-lock.ts#L403)
  - command: `nl -ba src/agents/session-write-lock.ts | sed -n '300,420p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Current regression test covers only the legacy no-`starttime` variant:** The added test explicitly verifies reclaiming orphan lock files 'without starttime when PID matches current process'. There is no matching test for the reported same-process/current-`starttime` restart scenario.
  - file: [src/agents/session-write-lock.test.ts:432](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/session-write-lock.test.ts#L432)
  - command: `nl -ba src/agents/session-write-lock.test.ts | sed -n '420,560p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **In-process gateway restart remains a real runtime path:** `runGatewayLoop()` can fall back to 'in-process restart', reusing the same PID/start time. That is consistent with the issue's reported restart shape.
  - file: [src/cli/gateway-cli/run-loop.ts:82](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/gateway-cli/run-loop.ts#L82)
