## Summary

Current `main` contains a generic startup-write dedupe for the gateway config reloader that covers the exact write paths named in the report (`gateway.auth.token` generation and `gateway.controlUi.allowedOrigins` seeding), preventing the startup watcher from treating those internal writes as external reloads. The regression is also covered by targeted tests, and the changelog says it shipped in `v2026.4.15` via #67557.

## Evidence

- **Startup now captures the post-normalization config hash:** `src/gateway/server.impl.ts` records `startupInternalWriteHash` only after startup writes complete, and the comment explicitly names plugin auto-enable, auth token generation, and Control UI origin seeding as the writes being guarded.
  - file: [src/gateway/server.impl.ts:285](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server.impl.ts#L285)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '280,350p;856,870p'`
- **Captured startup hash is passed into the managed config reloader:** The managed reloader is initialized with `initialInternalWriteHash: startupInternalWriteHash`, so the reloader can suppress the watcher event for that startup write instead of treating it as an external reload.
  - file: [src/gateway/server.impl.ts:860](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server.impl.ts#L860)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '280,350p;856,870p'`
- **Config reloader skips the first watcher reread when the hash matches an internal write:** `startGatewayConfigReloader` stores `initialInternalWriteHash` in `lastAppliedWriteHash` and returns early when the first reread has the same hash, which is the mechanism that prevents the startup reload cycle.
  - file: [src/gateway/config-reload.ts:123](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/config-reload.ts#L123)
  - command: `nl -ba src/gateway/config-reload.ts | sed -n '120,170p;330,360p;388,405p'`
- **In-process config writes are tracked separately from external edits:** The reloader subscribes to config writes, records the persisted hash, and schedules an immediate in-process apply path instead of falling back to chokidar-driven rereads.
  - file: [src/gateway/config-reload.ts:392](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/config-reload.ts#L392)
  - command: `nl -ba src/gateway/config-reload.ts | sed -n '120,170p;330,360p;388,405p'`
- **Regression test covers the exact startup-write dedupe behavior:** `src/gateway/config-reload.test.ts` has a test named `dedupes the first watcher reread for startup internal writes`, and the adjacent test documents the broken pre-fix behavior when `initialInternalWriteHash` is `null` (`#67436`).
  - file: [src/gateway/config-reload.test.ts:1003](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/config-reload.test.ts#L1003)
  - command: `nl -ba src/gateway/config-reload.test.ts | sed -n '1000,1065p'`
- **Channel startup still happens before the config reloader is attached:** `startGatewayPostAttachRuntime` runs `startChannels()` in the startup sidecars phase, while `server.impl.ts` attaches the config reloader afterward. That ordering means startup channel tasks are launched before config watching begins.
  - file: [src/gateway/server-startup-post-attach.ts:250](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server-startup-post-attach.ts#L250)

  - file: [CHANGELOG.md:552](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/CHANGELOG.md#L552)

## Close Comment

Closing as implemented on current `main` after Codex review. The gateway now captures a post-startup config hash for internal startup writes in `src/gateway/server.impl.ts:333` and passes it into the config reloader at `src/gateway/server.impl.ts:860`, and `src/gateway/config-reload.ts:352` uses that hash to suppress the first watcher reread for the same internal write instead of treating it as an external reload. That guard explicitly covers the startup writes called out here, including auth-token generation and `gateway.controlUi.allowedOrigins` seeding. There is also targeted regression coverage in `src/gateway/config-reload.test.ts:1003`, and the changelog says this startup-write reload-loop fix shipped in `v2026.4.15` via #67557 (`CHANGELOG.md:598`). If this still reproduces on a build at or after `v2026.4.15`, please open a fresh issue with exact version/commit and a current log trace, because the mechanism described in this report is now guarded on `main`.
