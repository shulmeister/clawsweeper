## Summary

Keep open. Current main partially strips provider auth env vars for the default OpenClaw ACP bridge, but it does not implement the issue's core request: ACP agent auth-contract metadata, per-route/provider env scrubbing for Gemini ACP launches, or fail-closed auth/readiness preflight hooks. The current docs also still frame Gemini ACP auth as either CLI auth or API key setup without the mixed-credential warning requested here.

## Best Possible Solution

Keep the issue open and implement a narrow ACP auth-contract design: add typed config/schema/docs for ACP agent or binding auth expectations, propagate an env policy through ACP session initialization into runtime plugins, add Gemini ACP scrub/preflight behavior for the known Gemini/GenAI env vars, fail closed when the declared contract is violated, and cover it with acpx/Gemini-focused unit tests plus mixed-credential host docs.
