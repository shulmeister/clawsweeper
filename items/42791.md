## Summary

The duplicate-image-slot behavior reported in #42791 does not reproduce on current `main`: prompt image loading now parses each `[media attached ...]` bracket as one attachment and ignores the post-`|` URL/path segment for image extraction, with a regression test covering the exact duplicated-path shape.

## Evidence

- **Current media-note formatter still prints the duplicated local path:** `buildInboundMediaNote` still renders `| url` unconditionally, so the prompt note can contain the same local path twice when `MediaUrl === MediaPath`. That means the remaining question is whether prompt image extraction still turns that into two image slots.

## Close Comment

Closing from a Codex review because the reported duplicate image-slot behavior does not reproduce on current `main`. The current prompt-image parser treats each `[media attached ...]` bracket as one attachment and only extracts the attachment path before the `|` separator (`src/agents/pi-embedded-runner/run/images.ts:229`), and there is an explicit regression test for the exact duplicated-path form `[media attached: /cache/IMG_6430.jpeg (image/jpeg) | /cache/IMG_6430.jpeg]` that expects a single detected image ref (`src/agents/pi-embedded-runner/run/images.test.ts:165`). The runner’s actual prompt-to-LLM image loading path goes through that parser (`src/agents/pi-embedded-runner/run/attempt.ts:2111`). The inline media note can still display the same local path twice in `src/auto-reply/media-note.ts:29`, but the duplicate-image-slot bug described here is obsolete on current OpenClaw, so I’m closing this as not reproducible.
