## Summary

Keep #39558 open. Current main still has the Feishu reaction message-id bug surface: reaction events intentionally synthesize `om_*:reaction:EMOJI:uuid` IDs, that raw ID can become the reply target, and Feishu API call sites still pass message IDs directly into SDK path parameters. The PR remains a plausible fix for #34528, but it needs a refresh because current main has newer surfaces such as streaming-card replies and no longer has the PR body's `send-message.ts` file.

## Best Possible Solution

Refresh #39558 or replace it with a current-main Feishu plugin patch that centralizes reaction-suffix stripping at API-facing message-id boundaries while preserving suffixed IDs for internal dedup/session identity. Cover send/reply/get/edit/update, media resource and media replies, reactions, typing indicators, channel thread/action paths, and streaming-card replies, then add regression tests for the helper plus at least one API boundary using a suffixed `om_*:reaction:EMOJI:uuid` ID.
