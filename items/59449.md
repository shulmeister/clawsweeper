## Summary

Current `main` already pre-registers the real primary CLI command before parsing `--help`, which fixes the lazy-placeholder behavior behind `openclaw cron <subcommand> --help`. The same implementation is present in the shipped `v2026.4.23` release.

## Evidence

- **Primary command is loaded before help parsing:** `runCli` now resolves the primary command and explicitly registers the real core/sub-CLI command before Commander parses argv, with an inline comment stating this keeps help and command parsing correct under lazy registration.
  - file: [src/cli/run-main.ts:239](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/cli/run-main.ts#L239)
  - command: `nl -ba src/cli/run-main.ts | sed -n '236,256p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Policy applies to `subcommand --help` invocations:** `shouldRegisterPrimaryCommandOnly(...)` returns true whenever a primary command is present, so invocations like `openclaw cron add --help` take the preregistration path instead of leaving the placeholder command in place.
  - file: [src/cli/command-registration-policy.ts:4](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/cli/command-registration-policy.ts#L4)
