## Summary

Keep #71417 open. Current main still routes `openclaw agent --agent <id>` without `--session-id` to that agent's main session key, reuses the existing session id while it is fresh, and has no `--new-session`/`--fresh` flag or visible resume banner. The delivery channel path also still defaults omitted channel input to `last`, though current docs clarify that channel flags affect delivery, not session routing.

## Best Possible Solution

Keep the issue open and implement an explicit fresh-session control for `openclaw agent` and the Gateway path, such as `--new-session`/`--fresh`, plus a non-JSON stderr banner when an invocation without explicit `--session-id`/`--session-key` reuses an existing session. Update `docs/cli/agent.md` and add targeted tests for `--agent` same-session reuse, fresh-session opt-in, JSON stdout safety, and the distinction between delivery channel selection and session routing.

## Risks / Open Questions

- Changing the default to always create a fresh session could break scripts that intentionally build context in an agent main session; an explicit fresh flag plus a resume banner may be less disruptive.
- Default daily reset mitigates very old sessions, so regression coverage should include both same-reset-window reuse and stale-row rollover behavior.
