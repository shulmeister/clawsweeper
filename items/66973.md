## Summary

Current main no longer has the reported unsafe default: omitted `mode` resolves to `run` unless `thread: true` is explicitly requested, ACP `run` maps to one-shot runtime mode, and `mode: "session"` is rejected unless the caller also requests thread binding. Docs and tool descriptions now describe `mode="run"` as one-shot and persistent/thread-bound behavior as explicit. Thread-bound sessions also surface an intro message and have idle/max-age lifecycle controls, so the prior silent permanent binding footgun is addressed well enough on main.

## Best Possible Solution

Keep the current main behavior: omitted `mode` is one-shot/background `run`, persistent thread-bound sessions require explicit thread-binding intent, and channel-visible intro plus idle/max-age lifecycle controls make bindings discoverable and bounded. Remaining prompt-guidance polish, such as the related #34542 discussion, can stay in its narrower guidance/documentation track.
