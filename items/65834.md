## Summary

Current `main` already routes memory-core's managed dreaming reconciliation through the typed `gateway_start` context `getCron()` live getter, and the gateway/plugin tests cover both live cron-handle updates and the startup-vs-runtime warning split. The fix is also documented as shipped in `v2026.4.23`.

## Evidence

- **Memory-core now resolves cron from typed gateway context:** `resolveCronServiceFromGatewayContext()` reads `context.getCron()`, and `registerShortTermPromotionDreaming()` stores that getter for later runtime reconciliation instead of parsing a startup payload object.
  - file: [extensions/memory-core/src/dreaming.ts:344](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/memory-core/src/dreaming.ts#L344)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Runtime reconciliation uses the live startup getter:** The runtime path keeps calling the stored getter (`resolveStartupCron?.()`) before warning, so reconciliation sees the current cron service instead of a stale one-time snapshot.
  - file: [extensions/memory-core/src/dreaming.ts:707](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/memory-core/src/dreaming.ts#L707)

- **Gateway contract provides a live cron getter:** `gateway_start` passes `getCron: () => params.deps.cron`, and the gateway test proves that the getter returns a reloaded cron handle after `params.deps.cron` changes.
