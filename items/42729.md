## Summary

Keep PR #42729 open. Current main still treats any finite past-due `nextRunAtMs` as runnable before checking `lastRunAtMs`, so the stale already-executed-slot case in the PR is not implemented on main. The existing restart catch-up tests cover future-`nextRunAtMs` missed-slot dedupe, and maintenance recompute can advance already-executed stale slots only when no runnable jobs are found, but the startup/timer hot path still selects the stale due job for execution first. The PR also has concrete user confirmation from Kaspre on v2026.3.31 and addresses a prior Codex review by switching the guard to strict `>` for same-ms retries.

## Best Possible Solution

Keep the PR open and ask maintainers to either land a current-main version of the hot-path stale-slot guard plus regression test, or supersede it with an equivalent fix that makes `collectRunnableJobs` skip already-executed expired `nextRunAtMs` slots while preserving legitimate same-ms retry slots.
