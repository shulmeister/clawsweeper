## Summary

Keep #40464 open. Current main has shipped guardrails that reduce catastrophic config clobbers, but it does not implement the PR's write serialization or the ownerDisplaySecret fresh-read-inside-the-write-critical-section behavior. The PR remains a plausible focused fix for a stale-snapshot startup data-loss race linked to #40410, though it likely needs a current-main rebase and review against the newer config recovery/runtime-snapshot code.

## Best Possible Solution

Keep the item open for maintainer review. The best path is to rebase or rewrite the fix against current main as a narrow config-write transaction for ownerDisplaySecret auto-persist, or a documented module-level serialization helper, while preserving current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin index commit behavior. Add regression coverage that starts the ownerDisplaySecret persist while another config write is pending and proves unrelated gateway/auth fields are not lost.
