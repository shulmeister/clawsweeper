## Summary

Current main routes community plugin discovery through ClawHub/native plugin install flows, and repo policy keeps optional community plugin discoverability out of core unless there is a stronger product/security reason. This PR's hard-coded onboarding catalog entry is better handled as a community plugin published and discovered through ClawHub.

## Close Comment

Closing from a Codex review. Current main points community plugin discovery to ClawHub rather than to new hard-coded core onboarding entries: `VISION.md:54-61` keeps optional capability out of core by default, `docs/plugins/community.md:14-22` says ClawHub is the canonical discovery surface for community plugins, and `docs/tools/clawhub.md:10-43` documents native plugin install/update flows through ClawHub. The shipped core fallback catalog is also intentionally tiny and curated (`scripts/lib/official-external-channel-catalog.json:1-28`), not a general community catalog. This makes the DingTalk discoverability goal better suited to ClawHub/plugin publishing than to a new core onboarding catalog entry. If there is still a concrete onboarding gap after publishing through ClawHub, please open a fresh issue with that gap scoped explicitly.
