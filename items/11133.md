## Summary

Current `main` already provides the requested browser chat image-attachment flow: the UI supports attach/paste/drag-drop with previews, the client sends images through `chat.send.attachments`, the gateway validates/sniffs/offloads images, and chat history reloads/rendering preserve uploaded and assistant-generated images via authenticated media routes.

## Risks / Open Questions

- The implementation does not match the issue's proposed design literally: current main uses direct `chat.send` attachment payloads and managed media routes rather than a separate Canvas-host `POST /assets` upload endpoint.
- `src/gateway/protocol/schema/logs-chat.ts` still types `attachments` as `Type.Array(Type.Unknown())`, so the feature is implemented behaviorally even though one checklist item about formalizing the schema is not fully reflected there.

## Close Comment

Closing as implemented on current `main` after a Codex review. The browser chat already supports image attach/paste/drag-drop with preview in `ui/src/ui/views/chat.ts:200` and `ui/src/ui/views/chat.ts:1199`, sends those images through `chat.send.attachments` in `ui/src/ui/controllers/chat.ts:213`, validates/sniffs/offloads them in `src/gateway/chat-attachments.ts:295`, and reloads/renders uploaded and assistant-generated images through authenticated managed media in `src/gateway/managed-image-attachments.ts:976` and `ui/src/ui/chat/grouped-render.ts:1118`. The changelog also records the shipped user-facing pieces in `CHANGELOG.md:22`, `CHANGELOG.md:53`, and `CHANGELOG.md:507`. This is not the exact `/assets` endpoint design proposed here, but the requested WebChat/Control UI image-attachment flow is already present on `main`.
