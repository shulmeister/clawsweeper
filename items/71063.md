## Summary

Keep this PR open. Current main still has the unguarded initial child stdin write path, and the described Gemini CLI/missing-binary route is real. The proposed patch is focused, but maintainer review should verify whether a `stdin.writable` guard is sufficient because a shell-wrapped missing binary can still report writable before emitting async `EPIPE`.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is a scoped supervisor fix in `src/process/supervisor/adapters/child.ts` that protects the initial `params.input` write/end path from both already-non-writable stdin and async `EPIPE`/closed-pipe errors, plus regression coverage in `src/process/supervisor/adapters/child.test.ts` that models both `writable === false` and writable-then-EPIPE child stdin behavior. The Gemini CLI route should then fail through the existing supervisor/CLI failover path instead of crashing the gateway.

## Risks / Open Questions

- The PR's current one-line `child.stdin.writable` guard may not catch the real shell-wrapper race where stdin is writable at write time but emits async `EPIPE` shortly after.
- Current main has no regression test for initial `params.input` write/end failure handling in `createChildAdapter`.
