## Summary

Current main still has the reported Discord configured ACP collision path: a Discord thread that inherits a parent-channel configured binding is materialized as the parent conversation, and the configured ACP session key is hashed from that materialized conversation id. That means sibling threads under the same configured parent channel can still resolve to the same persistent ACP session key. A paired open PR by the same author, #64322, is the right implementation candidate rather than a reason to close this issue early.

## Best Possible Solution

Keep this issue open and review the paired PR #64322 or a successor patch. The desired implementation should make inherited Discord parent-channel ACP bindings resolve to distinct persistent ACP session keys per Discord thread, with regression coverage proving two thread ids under the same parent do not share the same configured ACP key, while preserving existing direct thread bindings and non-Discord canonical binding behavior.
