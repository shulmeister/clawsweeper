## Summary

Keep open. Current main still has a real docs/UX and runtime-refresh gap for CLI-dependent skills like `github`: the bundled skill is gated on `gh`, binary eligibility is cached until PATH/PATHEXT changes, successful gateway-backed skill dependency installation returns only the installer result, and the session skill snapshot is reused unless an explicit skill/config/watch version changes. The current docs explain skill snapshots and watcher behavior, but they do not give prominent GitHub CLI post-install restart guidance or a clear `openclaw gateway restart` path for this scenario.

## Best Possible Solution

Keep #65958 open. The most complete fix is to make successful `skills.install` dependency installs invalidate binary lookup state, bump the relevant skills snapshot version, and refresh or clearly retire affected current-session snapshots; as a lower-risk companion, update the public Skills/GitHub skill docs and installer/UI success copy to say when to start a new session versus run `openclaw gateway restart`. The docs should include the concrete command `openclaw gateway restart` for managed gateway installs and mention that `gh auth login` alone may not make the `github` skill visible in an already-running session.
