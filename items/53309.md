## Summary

Current main already handles this class of cron/message-tool delivery mismatch: transient tool-error payloads are treated as non-fatal when a later successful payload exists, cron delivery target resolution now resolves/canonicalizes explicit targets, and cron fallback delivery is skipped once the message tool already sent to the resolved target. The fix was shipped in v2026.4.20.

Current main already handles the reported pattern where an early message-tool failure is followed by a later successful send: transient error payloads are treated as non-fatal when a later success exists (`src/cron/isolated-agent/helpers.ts:150`), explicit cron delivery targets are resolved/canonicalized before delivery (`src/cron/isolated-agent/delivery-target.ts:68`, `src/cron/isolated-agent/delivery-target.ts:249`), and isolated cron runs skip duplicate fallback delivery once the message tool already sent to the resolved target (`src/cron/isolated-agent/run.ts:848`, covered by `src/cron/isolated-agent/run.message-tool-policy.test.ts:457`). This was shipped in `v2026.4.20` (`CHANGELOG.md:353`, `CHANGELOG.md:424`). If you still see the old behavior on a newer build, please open a fresh issue with the current version, the cron run JSON, and the delivery trace.
