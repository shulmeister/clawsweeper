## Summary

Keep open. Current main contains a recent memory-lancedb fix that always sends `encoding_format: "float"`; that addresses the older Ollama/base64 dimension-mismatch family, but it does not satisfy this issue's follow-up case where ZhiPu rejects unknown `encoding_format` parameters. The plugin still has no config or request path that can omit the parameter while also avoiding the OpenAI SDK's omitted-format base64 decode behavior.

## Best Possible Solution

Keep this issue open and implement an owner-local memory-lancedb compatibility path: add an explicit embedding encoding/provider mode, and for omit-mode custom endpoints bypass the generated OpenAI SDK `embeddings.create` behavior or otherwise post without `encoding_format`, then normalize returned embeddings by accepting float arrays and decoding only string/base64 responses. Add tests for both forced-float providers and reject-unknown-parameter providers.
