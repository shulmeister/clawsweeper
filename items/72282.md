## Summary

Keep #72282 open. It is a fresh latest-release Mattermost regression report with concrete environment/config details, no protected maintainer override, and no evidence that current main has fixed the channel-only inbound path after v2026.4.24. Current code does parse and dispatch Mattermost `posted` WebSocket frames and route non-DM posts, but the reported symptom remains plausible and overlaps prior kept-open Mattermost channel reports rather than an implemented fix.

## Best Possible Solution

Keep the issue open and reproduce against v2026.4.24/current main with a real Mattermost channel. First eliminate config ambiguity by testing `groupPolicy: "open"` or `groupAllowFrom` set to the sender user id, then add a focused regression test or diagnostic capture for public/private channel `posted` WebSocket events through `extensions/mattermost/src/mattermost/monitor.ts`. If the event never reaches OpenClaw, investigate Mattermost subscription/auth/channel membership; if it reaches OpenClaw and is dropped, fix the policy/dispatch branch and improve verbose diagnostics for dropped channel posts.
