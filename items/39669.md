## Summary

Keep #39669 open. Current main still has no generic queued-followup feedback seam for the busy-session `enqueue-followup` path, and the reported Feishu/Lark path still only wires feedback through reply-start typing callbacks. Some channels now have initial lifecycle status reactions, but that does not close the cross-channel queued-message feedback request or the concrete Feishu gap.

## Best Possible Solution

Keep the issue open. The best fix is a generic, non-blocking queued-followup feedback seam in the active-run enqueue-followup path, such as an onQueued callback or equivalent typed receipt hook in GetReplyOptions and the shared reply dispatcher pipeline. It should fire before typing cleanup, swallow/log callback failures so enqueue reliability is preserved, wire Feishu/Lark to an immediate reaction or receipt, avoid duplicate signaling for Slack/Discord/Telegram status reactions, and add focused tests proving it fires only for busy queued followups.
