## Summary

Keep open. Current main still has per-session TUI visibility: `/new` deliberately moves the TUI into an isolated `tui-<uuid>` session, incoming chat and side-result events are dropped unless their `sessionKey` matches the current TUI session, and there is no CLI/config observe or mirror mode. Existing behavior covers shared-session history refresh, not the requested opt-in cross-channel/all-session TUI feed.

## Best Possible Solution

Keep this open as a core TUI/Gateway UX feature. The best path is an explicit read-only `openclaw tui --observe` or `tui.mirrorChannels` mode that preserves `/new` isolation, uses existing read-scoped Gateway broadcasts plus session listing/preview/history APIs, displays source session/channel labels, and only relaxes the exact-session event filter in that opt-in mode. Add tests for same-agent different-session events, channel-originated events, and read-only behavior before closing.
