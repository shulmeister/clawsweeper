## Summary

Keep open. Current main still has the Feishu empty-content crash path described by #66657: `parseMessageContent()` can return the raw runtime `content` value from its parse-error path, `parseFeishuMessageEvent()` stores that value as `ctx.content`, and both sequential-key/debounce handling still trim it without an optional guard. The linked fix PRs #66698 and #66725 were closed unmerged, and current tests do not cover undefined Feishu message content or @mention-only empty content.

## Best Possible Solution

Keep #66657 open and fix the Feishu plugin on main. The most robust path is to normalize inbound Feishu message content to a string at the parser boundary, then keep local guards at the sequential-key/debounce call sites if needed, and add focused regression tests for a group message with a bot mention and missing/empty `message.content` so both `getFeishuSequentialKey()` and the debounce path stop throwing.
