## Summary

Current `main` already hardens the Windows Scheduled Task stop path: after `schtasks /End`, it resolves the gateway port, kills lingering gateway listener PIDs, waits for port release, force-kills any remaining busy-port listeners, and errors if the port is still occupied. The same implementation is present in release `v2026.4.22`, with regression tests covering the lingering `node.exe` listener cases.

## Evidence

- **Windows stop path now cleans up lingering gateway listeners:** `stopScheduledTask` no longer stops at `schtasks /End`; it resolves the scheduled-task port, terminates scheduled-task-backed gateway listeners, waits for the port to be released, force-kills remaining busy listeners, and throws if the port is still busy.

## Close Comment

Codex review: this looks implemented on current `main`, and the same behavior is already shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The Windows Scheduled Task stop path now does more than `schtasks /End`: it resolves the gateway port, terminates lingering gateway listener PIDs, waits for the port to be released, force-kills any remaining busy-port listeners, and fails if the listener still survives (`src/daemon/schtasks.ts:393`, `src/daemon/schtasks.ts:449`, `src/daemon/schtasks.ts:930`). There is also regression coverage for the exact lingering-listener cases, including Windows `node.exe ... gateway --port 18789` fallback detection (`src/daemon/schtasks.stop.test.ts:111`, `src/daemon/schtasks.stop.test.ts:151`). On that basis I would close this as already fixed on main and released. If someone can still reproduce it on `v2026.4.22` or newer, they should open a fresh issue with the exact install mode and current stop/status output.
