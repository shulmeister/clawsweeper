## Summary

Keep #70050 open. Current main still has the reported core risk: `loadSessionStore` synchronously reads/parses the session store, then performs load-time prune/cap work by default when the store is oversized. `chat.history` still reaches that same synchronous load path. The shipped #71356-era change appears to make configured maintenance settings honored, but it does not defer load-time maintenance or make fresh/default startup responsive for oversized stores.

## Best Possible Solution

Keep the issue open and fix it in core. Preserve the OOM-protection goal from #69404, but move load-time maintenance off the startup-critical path or chunk/yield it, avoid full-store sorting where possible, keep `session.maintenance.mode: "warn"` as a reliable opt-out, and add regression coverage for oversized stores so gateway startup and `chat.history` remain responsive. Align docs/schema help with the actual default and workaround.
