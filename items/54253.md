## Summary

Keep this issue open. Current main still routes OpenRouter/OpenAI-compatible model traffic through the guarded Undici fetch path and still maps generic fetch/connection failures to the reported `LLM request failed: network connection error.` message. The proxy workaround in the discussion is useful, and related #58150 was closed as a duplicate, but I found no shipped RISC-V64-specific fix, diagnostic, fallback transport, or narrower canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64/provider-transport tracker. Maintainers should ask for exact board/distro, Node binary source, `process.arch`, `process.versions`, direct `fetch` and `node:https` results against the same provider, redacted gateway service proxy environment, and logs that preserve the underlying provider failure cause. If stale proxy injection is confirmed, fix doctor/docs or service environment handling; if RISC-V64 Undici/fetch is confirmed, add a guarded transport fallback or explicit compatibility diagnostic in the shared provider transport path.
