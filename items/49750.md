## Summary

Keep open. Current `main` still has the two gaps this PR targets: `resolveGroupSessionKey()` only inspects the first `From` segment, so agent-prefixed keys like `agent:<id>:slack:channel:...` are not parsed as Slack group/channel sessions, and restart-sentinel routing still falls back to stored session/base-session delivery context without any parsed-target override from the session key.

## Evidence

- **Group-key parsing still ignores agent prefix:** `src/config/sessions/group.ts:95-119` still derives `head` from `parts[0]` and `second` from `parts[1]`. For `agent:builder:slack:channel:C123:thread:...`, that means `head` is `agent`, not `slack`, so provider detection still depends on `ctx.Provider` instead of the encoded session key.
  - file: [src/config/sessions/group.ts:95](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/config/sessions/group.ts#L95)
  - command: `git show HEAD:src/config/sessions/group.ts | nl -ba | sed -n '93,120p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Repo already has a canonical agent-prefix-aware parser elsewhere:** `src/sessions/session-key-utils.ts:128-137` explicitly skips the `agent:<id>:` prefix before reading `channel` and `kind`. That makes the missing equivalent handling in `resolveGroupSessionKey()` a real current inconsistency, not an obsolete code path.
  - file: [src/sessions/session-key-utils.ts:128](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/sessions/session-key-utils.ts#L128)
