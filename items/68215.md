## Summary

Current `main` already contains the core-side guard for this class of plugin-config reinjection: raw config writes no longer persist plugin AJV defaults back to disk, `update.run` itself does not mutate config, and there is a regression test covering plugin-owned raw config default injection.

## Evidence

- **Raw config writes avoid persisting plugin AJV defaults:** `writeConfigFile` validates the raw `persistCandidate` and then explicitly writes `persistCandidate` rather than `validated.config`, with an inline note that plugin/channel AJV defaults must not be persisted to disk.
  - file: [src/config/io.ts:1658](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/config/io.ts#L1658)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Regression test for plugin-owned raw config default injection:** Current `main` has a test asserting `validateConfigObjectRawWithPlugins` does not inject plugin AJV defaults into plugin-owned config in raw mode.
  - file: [src/config/validation.channel-metadata.test.ts:169](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/config/validation.channel-metadata.test.ts#L169)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Plugin validation still reports extra plugin config keys as invalid:** Plugin config validation still surfaces schema errors such as `invalid config: must NOT have additional properties`, which matches the reported failure mode if an unexpected key is present.
  - file: [src/config/validation.ts:1197](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/config/validation.ts#L1197)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **`update.run` does not patch config:** The gateway `update.run` handler loads config only to read the update channel, runs the updater, writes a restart sentinel, and schedules restart; it does not write plugin config.
  - file: [src/gateway/server-methods/update.ts:41](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server-methods/update.ts#L41)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Startup config writes are limited to auto-enable persistence:** On startup, OpenClaw writes config only when plugin auto-enable produced changes; that narrows the former reinjection path to config-write validation/persistence, which current `main` now guards.
  - file: [src/gateway/server-startup-config.ts:112](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server-startup-config.ts#L112)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Core config-validation code changed after the latest release:** Diffing `v2026.4.23` (`a9797214`) to current `main` (`c2139635`) shows substantive changes in `src/config/validation.ts`, including the current plugin-validation behavior present on `main`.
  - command: `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..c2139635ff82b367b734ea052487258b1b33f060 -- src/config/validation.ts src/config/validation.channel-metadata.test.ts`
