## Summary

Keep open. Current `main` still converts explicit `deviceIdentity: null` into an auto-loaded identity inside `GatewayClient`, and normal `callGateway` requests still do not wire `onConnectError` into the final surfaced error. Probe/status handling has improved separately, which matches the report that probe and normal RPC diagnostics can diverge. The issue also has an open linked PR (#65641), so conservative cleanup should not auto-close it.

## Evidence

- **`GatewayClient` still drops explicit `null` device identity:** `GatewayClientOptions` explicitly allows `deviceIdentity?: DeviceIdentity | null`, but the constructor rewrites `null` to `undefined` and then falls back to `loadOrCreateDeviceIdentity()`. That means callers still cannot reliably say 'send no device identity'.
  - file: [src/gateway/client.ts:125](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/client.ts#L125)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Top-level call path expects `null` to be honored:** `callGateway` passes `opts.deviceIdentity` through unchanged when it is not `undefined`, and the regression test explicitly asserts that `deviceIdentity: null` should stay `null`. That contract is present in the call layer, but the client constructor still breaks it later.
  - file: [src/gateway/call.ts:515](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/call.ts#L515)

If connect/auth/handshake setup fails before hello, the final error path still resolves from `onClose`/timeout formatting rather than preserving the earlier low-level connect error.
