## Summary

Keep open. Current `main` has only the global Discord `allowBots` control (`false`, `true`, or `"mentions"`), and the inbound preflight still treats webhook messages as ordinary bot-authored messages. The requested per-guild `trustedWebhooks` allowlist and optional webhook-to-user remapping are not implemented on `main`.

## Evidence

- **Discord config surface lacks `trustedWebhooks`:** `DiscordGuildEntry` exposes `slug`, mention/ignore settings, allowlists, and channel overrides only; the account config exposes `allowBots` as the bot-message knob. There is no guild-level `trustedWebhooks` or webhook remap field in the current typed config surface.

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Inbound preflight still drops bot/webhook messages unless `allowBots` permits them:** `preflightDiscordMessage()` resolves `webhookId`, but only uses it to skip PluralKit lookup. The actual bot gate is still `allowBotsMode === "off"` => drop, and `allowBotsMode === "mentions"` => require a mention. There is no webhook allowlist check or sender remap path here.

## Risks / Open Questions

- `allowBots: "mentions"` may already cover a subset of webhook-trigger use cases where the webhook can explicitly mention the bot, but it does not satisfy the issue's requested per-webhook trust and identity-mapping behavior.
- `gh` cross-checks were unavailable in this environment because `GH_TOKEN` was unset, so the review is based on the checked-out `main` tree and local git history only.
