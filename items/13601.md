## Summary

Keep open. Current main has persistent JSONL-backed cron run history plus Gateway/UI status, delivery, query, and sort filters, but meaningful parts of #13601 remain missing: CLI status/time filters, time-range filtering in the protocol/run-log reader, success-rate/list statistics, and a manual history prune command. The prior maintainer discussion also explicitly kept this item open, and related #16799/#16831 cover status UX rather than this cron-history workflow.

## Best Possible Solution

Keep #13601 open until core cron history is completed or a maintainer narrows the product scope. The likely implementation path is to extend `cron.runs` and `readCronRunLogEntriesPage` with time-window filters, wire CLI flags or an explicit `cron logs/history` surface, add success-rate/list aggregation, and either implement manual history pruning or document a deliberate config-only retention decision with tests and docs.
