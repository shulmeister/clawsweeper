## Summary

Keep #13601 open. Current main has durable per-job JSONL cron run history plus status/delivery/query filters, UI filtering, and automatic size/line pruning, but it still lacks the central requested product surface: timestamp range filters, `openclaw cron logs`/`cron history` commands, `--failed`/`--success`/`--since`/`--range`/`--follow` CLI options, success-rate/failure aggregates in `cron list`, and manual prune/vacuum/history-retention-days workflow. The provided related context also shows #16799 and PR #16831 were status-UX work, with #16831 closed unmerged, so they do not supersede this issue.

## Best Possible Solution

Keep #13601 open until maintainers either implement or explicitly narrow the remaining cron history product surface. A complete path would add timestamp filters through ReadCronRunLogPageOptions, CronRunsParamsSchema, the cron.runs handler, UI state, CLI options, and docs; expose a clearer `openclaw cron logs` or `openclaw cron history` command with status/time/follow/export behavior; add success-rate and failure aggregates to `cron list` or a dedicated summary; and decide whether manual prune/vacuum/history-retention-days belongs in core or should be documented out of scope.
