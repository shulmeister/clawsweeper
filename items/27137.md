## Summary

Keep #27137 open. Current main still has shared DM/group policy primitives plus several channel-local ingress authorizers, but it does not implement the requested unified `authorizeIngress()` / `IngressContext` gate across channel, account, event kind, sender identity, chat context, allowlists, command authorization, and standardized reasons. The only issue comment in the provided context is also a maintainer-posted Codex review explicitly keeping this item open.

## Best Possible Solution

Keep #27137 open for maintainer design. The best path is a narrow core/security or SDK-backed ingress authorization contract that accepts normalized adapter inputs for channel, account, event kind, sender identity, chat context, configured/store allowlists, and command surface; centralizes DM/group policy, allowlist merging, pairing-store reads, command authorization, and reason codes; then migrates representative message, reaction, pin, interaction, callback, slash, and voice paths with contract tests proving new ingress surfaces cannot bypass the shared gate.
