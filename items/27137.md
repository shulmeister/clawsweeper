## Summary

Keep #27137 open. Current main still lacks a unified `authorizeIngress()`/`IngressContext` gate, and representative ingress surfaces continue to authorize through shared primitives plus channel- or event-local helpers rather than one central ingress decision point.

## Best Possible Solution

Keep #27137 open for maintainer design. The best path is a narrow core/security or SDK-backed ingress authorization helper that takes normalized adapter inputs for channel, account, event kind, sender identity, and chat context; centralizes DM/group policy, allowlist merging, pairing-store reads, command authorization, and reason codes; then migrates representative message, reaction, pin, interaction, callback, slash, and voice paths with contract tests proving new ingress surfaces cannot bypass the shared gate.
