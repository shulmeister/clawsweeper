## Summary

Close as `clawhub`. The request is a useful orchestration UX idea, but current `main` positions this kind of manager-mediated multi-agent workflow as plugin/skill scope rather than OpenClaw core.

## Evidence

- **Vision keeps optional orchestration out of core:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, new skills should go to ClawHub first, and OpenClaw will not merge agent-hierarchy frameworks or heavy orchestration layers that duplicate existing agent/tool infrastructure.
  - file: [VISION.md:55](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/VISION.md#L55)
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Plugin runtime already exposes subagent orchestration:** Plugins can launch and manage background subagents via `api.runtime.subagent.run(...)` and coordinate durable multi-step flows via `api.runtime.taskFlow`.
  - file: [docs/plugins/sdk-runtime.md:87](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/docs/plugins/sdk-runtime.md#L87)
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Slash-command surface is extensible outside core:** Docs say plugin commands come from `registerCommand()` and user-invocable skills are exposed as slash commands, including direct command dispatch to tools.
  - file: [docs/tools/slash-commands.md:80](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/docs/tools/slash-commands.md#L80)
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Bundled examples already model captain-style delegation:** The OpenProse docs describe a 'captain's chair' pattern where a coordinating agent dispatches specialized subagents, validates, and integrates results.
  - file: [extensions/open-prose/skills/prose/examples/README.md:163](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/open-prose/skills/prose/examples/README.md#L163)
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **No built-in `/callon` implementation exists on current main:** A repository search for `callon` returned no matches in `docs`, `src`, `packages`, or `extensions`, so this issue is not already implemented as a core command.
  - command: `rg -n "callon" docs src packages extensions`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)

## Risks / Open Questions

- If maintainers later decide multi-agent team chat is a core product priority despite `VISION.md`, this closure would be premature.
- OpenProse demonstrates the pattern but is not a drop-in implementation of the exact `/callon` UX requested.

## Close Comment

Closing this as a Codex review because it fits ClawHub/plugin scope rather than OpenClaw core. Current `main` explicitly keeps optional capability in plugins/skills (`VISION.md:55`, `VISION.md:69`) and calls out agent-hierarchy frameworks / heavy orchestration layers as out of scope for core (`VISION.md:106`). The extension seams for this already exist: plugins can register their own slash commands (`docs/tools/slash-commands.md:80`, `docs/plugins/sdk-overview.md:88`), launch subagents and manage flow state (`docs/plugins/sdk-runtime.md:87`, `docs/plugins/sdk-runtime.md:143`), and bundled OpenProse already documents a captain/coordinator pattern over subagents (`extensions/open-prose/skills/prose/examples/README.md:163`). There is also no built-in `/callon` command on current `main` (`rg -n "callon" docs src packages extensions` returned no matches). If someone wants to build this, the right path is a ClawHub skill/plugin that provides the `/callon` UX on top of the existing command + subagent + Task Flow seams.
