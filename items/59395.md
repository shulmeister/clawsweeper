## Summary

Keep open. Current `main` still preserves DNS pinning for `web_fetch` even when `HTTP_PROXY`/`HTTPS_PROXY` is configured, so the Clash fake-IP report remains plausible and is effectively codified by tests.

## Evidence

- **`web_fetch` does not opt into trusted proxy mode:** `runWebFetch` calls the web-tool network guard without any `useEnvProxy` flag and only forwards an optional RFC2544 policy override, so normal `web_fetch` requests stay on the strict path.
  - file: [src/agents/tools/web-fetch.ts:415](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/web-fetch.ts#L415)
  - command: `nl -ba src/agents/tools/web-fetch.ts | sed -n '404,435p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Web-tool guard defaults to strict mode:** `fetchWithWebToolsNetworkGuard` uses `withStrictGuardedFetchMode(...)` unless `useEnvProxy` is explicitly true. The trusted env-proxy path is only wired through `withTrustedWebToolsEndpoint`, not the ordinary `web_fetch` tool path.
  - file: [src/agents/tools/web-guarded-fetch.ts:37](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/web-guarded-fetch.ts#L37)
  - command: `nl -ba src/agents/tools/web-guarded-fetch.ts | sed -n '1,90p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Current tests assert the reported behavior:** `web-tools.fetch.test.ts` explicitly verifies that `web_fetch` keeps DNS pinning even when `HTTP_PROXY` is set. That matches the issue's failure mode rather than disproving it.
  - file: [src/agents/tools/web-tools.fetch.test.ts:256](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/web-tools.fetch.test.ts#L256)
  - command: `nl -ba src/agents/tools/web-tools.fetch.test.ts | sed -n '248,272p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Lower-level support exists but is not used here:** The shared fetch guard has a `TRUSTED_ENV_PROXY` branch that skips pinned DNS and creates an `EnvHttpProxyAgent` when proxy env vars are configured, and its SSRF tests cover that mode. The gap is tool wiring, not absence of lower-level capability.
  - file: [src/infra/net/fetch-guard.ts:357](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/infra/net/fetch-guard.ts#L357)

## Risks / Open Questions

- Any fix here needs to preserve SSRF enforcement when no trustworthy proxy is actually active; the lower-level guard already treats that as a security-sensitive distinction.
- There is an opt-in `allowRfc2544BenchmarkRange` escape hatch, but it only helps when `web_fetch` is configured to allow that range explicitly and does not address the default hostname-to-fake-IP proxy setup described in the issue.
