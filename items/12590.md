## Summary

Keep open. Current main still contains the reported counter synchronization path: a flush-triggered compaction can set memoryFlushCompactionCount to the newly incremented compactionCount, while the flush gate skips when those two values match. The issue discussion also points to unmerged fix PRs, and there is no evidence in current main that the requested behavior is implemented.

## Evidence

- **Current main still writes the post-increment count:** runMemoryFlushIfNeeded initializes memoryFlushCompactionCount from the active compactionCount, calls incrementCompactionCount when the flush turn compacts, then assigns memoryFlushCompactionCount = nextCount before persisting it. That is the counter-locking behavior described in the issue.
