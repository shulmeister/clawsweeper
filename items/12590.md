## Summary

Keep #12590 open. Current main still persists the post-increment compaction count as `memoryFlushCompactionCount` after a memory-flush-triggered compaction, while the flush gate skips whenever `memoryFlushCompactionCount === compactionCount`. The current tests still assert that synchronized state, and the issue discussion points to open/unmerged fix attempts rather than a shipped fix.

## Best Possible Solution

Keep #12590 open and fix the bookkeeping contract in `src/auto-reply/reply/agent-runner-memory.ts`: after a flush-triggered compaction, preserve the pre-increment compaction count as `memoryFlushCompactionCount`, or redesign the dedup state so the next eligible auto-compaction cycle is not treated as already flushed. Add regression coverage proving consecutive eligible auto-compaction cycles run `memoryFlush` each time, and update tests that currently assert synchronized counters. PR #51421 and the closed prior fix attempts are useful references, but the issue should remain open until a fix is on main.
