## Summary

Keep open. Current main has a partial workaround via fully qualified agent session keys or workspace auto-discovery, but the requested native `openclaw tui --agent <id>` flag is still not implemented, and the issue's two-part `--session agent:ollama` example still does not parse as an agent-scoped session key.

## Evidence

- **TUI CLI flags:** `openclaw tui` registers `--local`, gateway auth flags, `--session`, `--deliver`, `--thinking`, `--message`, `--timeout-ms`, and `--history-limit`; it does not register or forward a `--agent` option.
  - file: [src/cli/tui-cli.ts:14](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/cli/tui-cli.ts#L14)
  - command: `nl -ba src/cli/tui-cli.ts | sed -n '14,23p'`
  - sha: [1dfa52d0711d](https://github.com/openclaw/openclaw/commit/1dfa52d0711dfc86e1c04538fe46e3d43961766f)
- **Initial agent resolution:** The TUI derives the current agent only from a parseable agent-prefixed session key, workspace auto-discovery from CWD, or the configured default agent. There is no separate CLI agent override path.
  - file: [src/tui/tui.ts:164](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/tui/tui.ts#L164)
