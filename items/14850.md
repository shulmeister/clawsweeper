## Summary

Keep #14850 open. Current main only supports the narrower `bindings[].session.dmScope` override for direct-message bindings; it still has no binding-level `sessionKey`, `sessionAlias`, or `sessionScope`, and non-direct Discord guild/channel peers continue to resolve to isolated channel session keys.

## Best Possible Solution

Keep the issue open until core routing gains an explicit, documented, opt-in binding-level session target for non-direct peers, or maintainers make a product decision to reject channel-to-main session unification. The likely implementation belongs around `AgentRouteBinding.session`, `resolveAgentRoute`, and `buildAgentPeerSessionKey`, with tests proving Discord guild/channel bindings can intentionally share an agent main or aliased session while preserving isolated defaults.

## Risks / Open Questions

- A generic binding-level session override could intentionally merge otherwise isolated guild/channel traffic into a main session, so the implementation needs explicit product and safety handling rather than an accidental plugin workaround.
- Current plugin hooks can observe or claim messages, but the documented `inbound_claim` result is only `{ handled, reply }`, and plugin conversation binding refuses conversations already bound by core routing; that does not provide a clean external override for this built-in Discord/Telegram binding case.
