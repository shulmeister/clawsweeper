## Summary

Keep #14850 open. Current main implements only the narrower binding-level `session.dmScope` override for direct-message route bindings. It still lacks a binding-level fixed session key, alias, or main-session target that would let a Discord guild/channel binding intentionally share the same agent session as Telegram DMs.

## Best Possible Solution

Keep the issue open until maintainers either add and document an explicit binding-level session target for non-direct route bindings, or make an explicit product decision to reject channel-to-main session unification. The likely implementation path is `AgentRouteBinding`, `RouteBindingSchema`, `resolveAgentRoute`, and `buildAgentPeerSessionKey`, with regression tests proving Discord guild/channel bindings can intentionally share an agent main or aliased session while preserving isolated defaults.

## Risks / Open Questions

- Closing this would drop a real remaining product/API request: opt-in unification of non-direct channel bindings with an agent main or aliased session.
- Any implementation needs explicit opt-in validation, documentation, and regression tests because channel-to-main unification can intentionally merge context across surfaces and could leak context if enabled accidentally.
- The implementation should preserve isolated defaults for guild/channel traffic and prove interactions with `identityLinks`, `lastRoutePolicy`, delivery context, and sandbox/non-main behavior.
