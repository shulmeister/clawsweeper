## Summary

Keep open. Current `main` still has no binding-level session override or equivalent cross-surface session unification for guild/channel bindings. Bindings only choose the target agent; session keys are still derived from channel/peer rules, and non-DM channel peers remain isolated from the agent main/DM session.

## Evidence

- **Route binding type has no session override field:** `AgentRouteBinding` only exposes `type`, `agentId`, `comment`, and `match`; there is no `sessionKey`, `sessionAlias`, or `sessionScope` field on route bindings.
  - file: [src/config/types.agents.ts:47](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/config/types.agents.ts#L47)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Config schema would reject requested binding fields:** `RouteBindingSchema` is `.strict()` and only allows `type`, `agentId`, `comment`, and `match`, so the proposed binding-level session override is not supported in current config validation.
  - file: [src/config/zod-schema.agents.ts:38](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/config/zod-schema.agents.ts#L38)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Resolved route computes session key from peer/channel data, not from binding config:** `resolveAgentRoute()` selects an agent from bindings, then `choose()` always builds the session key from `agentId`, `channel`, `accountId`, `peer`, `dmScope`, and `identityLinks`. The matched binding contributes no session override input.
  - file: [src/routing/resolve-route.ts:657](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/routing/resolve-route.ts#L657)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Cross-channel identity linking only applies to direct messages:** `buildAgentPeerSessionKey()` only considers `identityLinks` inside the `peerKind === "direct"` branch. For non-direct peers it always returns `agent:<agentId>:<channel>:<peerKind>:<peerId>`, which keeps guild/channel traffic in its own session.
  - file: [src/routing/session-key.ts:141](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/routing/session-key.ts#L141)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Current tests lock guild/channel bindings to their own session key:** The routing test for a Discord channel peer binding expects `sessionKey: "agent:chan:discord:channel:c1"`, confirming current main does not collapse that binding into the agent main session.
  - file: [src/routing/resolve-route.test.ts:245](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/routing/resolve-route.test.ts#L245)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Docs scope cross-channel sharing to DMs, not bound channels:** The configuration docs describe `dmScope` as controlling DM grouping and `identityLinks` as cross-channel session sharing for those DMs. The bindings docs describe bindings as pinning inbound traffic to an agent, with no session override surface.
  - file: [docs/gateway/configuration-reference.md:2022](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/gateway/configuration-reference.md#L2022)

  - file: [docs/cli/agents.md:33](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/cli/agents.md#L33)
