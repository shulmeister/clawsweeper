## Summary

Keep #14850 open. Current main has related binding-level `session.dmScope` support for direct-message routing, but there is still no normal route-binding `sessionKey`, `sessionAlias`, or main-session target that can intentionally route a Discord guild/channel binding into the same agent main session used by Telegram DMs. Non-direct peers remain channel-scoped by design, and the related continuity/identity PRs do not supersede the requested channel-to-main session unification.

## Best Possible Solution

Keep the issue open until maintainers either add a documented normal route-binding session target for non-direct peers or explicitly reject channel-to-main session unification. A likely implementation would extend `AgentRouteBinding` and `RouteBindingSchema` with a narrow `session.target`/`sessionKey` style field, thread it through `resolveAgentRoute`, document the privacy tradeoff, and add regression tests proving a Discord channel binding can intentionally share `agent:<agentId>:main` while default channel isolation remains unchanged.

## Risks / Open Questions

- Any implementation must be explicit opt-in and documented because merging a guild/channel into an agent main session intentionally shares context across surfaces.
- The existing ACP conversation-binding path is easy to confuse with this request, but it targets ACP sessions and does not solve normal route-binding continuity with Telegram DMs.
