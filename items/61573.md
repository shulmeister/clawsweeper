## Summary

Current main already gives isolated cron jobs' `payload.model` precedence over stored session model state, and that behavior is present in the latest release. The reported sticky fallback in `sessions.json` reflects persisted runtime model telemetry, not the model-selection precedence used for later cron runs.

## Evidence

- **Cron model selection prefers payload.model:** `resolveCronModelSelection` resolves `payload.model` first and only consults stored cron-session `modelOverride`/`providerOverride` when no per-job model is present.

## Risks / Open Questions
