## Summary

Current main implements the requested WebChat behavior. Busy sends are queued, terminal chat events flush the queue, the dequeued message is appended as a local user turn before `chat.send` returns, history reloads preserve optimistic user/assistant tail messages when the server snapshot is stale or empty, and the gateway emits the accepted user turn to transcript subscribers immediately. Regression coverage exists for the optimistic-history and user-transcript-update paths.

## Best Possible Solution

Close as implemented on main and keep the current queue flush, optimistic-history preservation, deferred history reload, and eager user transcript update behavior. If the issue reappears in a released build, it should be filed as a fresh regression with the release version, queue mode, channel override, and relevant history/session.message events.
