## Summary

Keep #39031 open. Current main still does not implement legacy `imsg rpc` tapback/reaction forwarding to the agent. The iMessage monitor path is normal-message/attachment focused: payload types lack reaction metadata, parsing only accepts `raw.message`, `watch.subscribe` only requests attachments, inbound decisions dispatch text bodies, and config/docs expose `includeAttachments` but no reaction forwarding control. BlueBubbles has reaction system-event support, but that is a separate recommended iMessage path and does not satisfy the requested legacy iMessage channel behavior. The related context also keeps #39031 as the canonical request: #60446 was closed back to this issue, while #39322 remains an open implementation candidate.

## Best Possible Solution

Keep #39031 open as the canonical legacy iMessage `imsg rpc` tapback-to-agent feature request. Maintainers should review, rebase, or supersede #39322 with a typed reaction/system-event implementation that covers subscription, payload parsing, access control, debounce/dedupe semantics, config/docs, and regression tests, or make an explicit product decision that reaction event support is BlueBubbles-only and document legacy `imsg` parity as out of scope.
