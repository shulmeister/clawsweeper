## Summary

Keep #71389 open. Current main still routes Linux/Node image operations through the bundled media-understanding image ops artifact, which imports `sharp` in-process with no CPU-feature preflight, no `OPENCLAW_IMAGE_BACKEND=disabled`/`none` escape hatch, and no pass-through fallback in `resizeToJpeg`, `convertHeicToJpeg`, or `resizeToPng`. The related #47419 report was a Docker release-image regression and does not supersede this runtime sharp-load crash path.

## Best Possible Solution

Keep the issue open and fix the runtime image backend boundary. Add a Linux/x64 CPU capability gate or safe sharp probe before importing the native module, support `OPENCLAW_IMAGE_BACKEND=disabled`/`none` as a documented pass-through mode, and make `resizeToJpeg`, `convertHeicToJpeg`, and `resizeToPng` degrade predictably where pass-through is valid. Add regression coverage for disabled/probe-failed sharp behavior without requiring non-AVX hardware, and document the operational fallback.
