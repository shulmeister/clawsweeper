## Summary

Keep open. Current main has partial mitigations for the incident family, including exec payload framing and visible-history filtering in one HTTP/SSE path, but the heartbeat/cron/exec runner still turns automation into a normal prompt passed through the reply pipeline. I did not find a durable backend invariant that prevents internal heartbeat/cron/exec traffic from becoming transcript/user-turn shaped input.

## Best Possible Solution

Keep the issue open and implement the durable backend invariant: heartbeat, cron, exec-event, and wake traffic should be represented as internal/system events or hidden runtime context, not ordinary visible user turns. The likely fix belongs around the heartbeat runner/reply prompt boundary and transcript/history projection: pass a neutral or empty visible transcript prompt for automation, carry orchestration text through hidden runtime context or a typed internal event, update both chat.history and /sessions history projections, and add regression tests proving heartbeat/cron/exec prompts are not persisted or rendered as role:user while clean user-facing assistant summaries still work.
