## Summary

Keep #69548 open. Current main still does not expose or pass `dreaming.staggerMs` through the memory-core dreaming schedule, while the underlying cron subsystem already supports `staggerMs`. The PR is a focused feature with useful review follow-up, but it needs maintainer review or an update around unset/clear semantics before cleanup closure would be appropriate.

## Best Possible Solution

Keep the PR open and ask for a current-main update: wire `dreaming.staggerMs` through `extensions/memory-core/openclaw.plugin.json`, `src/memory-host-sdk/dreaming.ts`, and `extensions/memory-core/src/dreaming.ts`; add resolver and managed-cron tests for undefined, positive, `0`, blank strings, and removal semantics; then document the operator-facing behavior, including how to clear a previous explicit stagger if cron keeps merge semantics.

## Risks / Open Questions

- Merging the PR as-is may leave confusing unset behavior: removing `dreaming.staggerMs` after setting it would not necessarily return an existing managed cron job to default behavior under current cron patch merge semantics.
