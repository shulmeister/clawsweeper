## Summary

Current `main` already decouples dreaming from the selected memory slot well enough for the reported coexistence case: dreaming config is read from the active memory-slot owner, and the loader keeps bundled `memory-core` loaded as a sidecar when another memory plugin enables dreaming.

## Evidence

- **Loader sidecar logic:** `resolveDreamingSidecarEngineId()` returns `memory-core` as a sidecar engine whenever a non-`memory-core` memory slot owner has dreaming enabled, and the bundled-memory fast path explicitly exempts that sidecar so it loads alongside the selected slot plugin.
  - file: [src/plugins/loader.ts:170](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/plugins/loader.ts#L170)
  - command: `nl -ba src/plugins/loader.ts | sed -n '170,186p;2467,2492p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Selected-slot dreaming config:** Dreaming config is no longer hard-coded to `memory-core`; `resolveMemoryDreamingPluginConfig()` reads config from whatever plugin currently owns `plugins.slots.memory`, falling back to `memory-core` only when no slot override is set.
  - file: [src/memory-host-sdk/dreaming.ts:321](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/memory-host-sdk/dreaming.ts#L321)
  - command: `nl -ba src/memory-host-sdk/dreaming.ts | sed -n '321,352p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Regression test for coexistence:** There is an explicit loader test named `loads dreaming engine alongside a different memory slot plugin when dreaming is enabled`; it asserts both `memory-core` and `memory-lancedb` load together when `memory-lancedb` owns the memory slot and enables dreaming.
  - file: [src/plugins/loader.test.ts:5871](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/plugins/loader.test.ts#L5871)
  - command: `nl -ba src/plugins/loader.test.ts | sed -n '5869,5933p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Non-core memory plugin support:** The bundled `memory-lancedb` manifest now accepts a `dreaming` config block and documents it as "consumed when this plugin owns the memory slot," which matches the decoupled loader/config behavior.
  - file: [extensions/memory-lancedb/openclaw.plugin.json:41](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/memory-lancedb/openclaw.plugin.json#L41)

## Risks / Open Questions

- Some docs still use `plugins.entries.memory-core.config.dreaming` wording, which under-describes the newer selected-slot-owner behavior.
- Third-party memory plugins still need to accept a `dreaming` config block in their own schema if they want validation/UI support comparable to bundled plugins.
