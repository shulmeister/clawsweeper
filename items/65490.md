## Summary

Keep this issue open. The reporter’s related links are useful and several narrower fallback/cron/error-reporting items have since been addressed or documented, but current main still only partially covers the central production-operations gap. Startup primary-model handling is a non-blocking warmup warning, not a live reachability gate before accepting turns; in-agent diagnostics expose session/model status and gateway stability data, but not the requested compact “current model, last error, fallback status” surface; and cron add still lacks a dry-run path and does not generally require an announce delivery target before persisting a job.

## Best Possible Solution

Keep this issue open or convert it into a focused tracking issue for the remaining production discipline gaps: a configurable startup/provider preflight that can live-probe the primary model and surface a visible degraded state before accepting turns; a safe read-only runtime diagnostic tool/RPC for the current session that exposes actual responding model, last provider/fallback error category, and fallback state; and cron creation validation/dry-run behavior for delivery targets. Closed related items such as #56852, #58303, #56996/#57045/#57046, #59457, and #63900 can be referenced as completed subwork, but they do not fully supersede this issue.
