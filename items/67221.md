## Summary

Do not auto-close #67221. Current main still rejects the PR's email-based add-on principal path, and the merged diagnostic work in #71145 does not implement that behavior; it warns that email-shaped appPrincipal values are misconfigured. The PR's secondary status-snapshot concern is already handled on main through Google Chat-specific snapshot extras, but the primary auth behavior remains a plausible, focused change that needs maintainer product/security judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep #67221 open for maintainer review. The likely best path is to split/adjust it: keep the already-shipped channel-specific status snapshot approach, then decide the Google Chat app-url add-on principal contract explicitly. If OpenClaw should support Google Workspace add-on systemIdToken validation by service account email, implement that with focused tests and docs/config help distinguishing email-based add-on service accounts from numeric sub/OAuth-client-ID deployments; if maintainers intentionally require numeric sub only, close with that product/security rationale and document the requirement.
