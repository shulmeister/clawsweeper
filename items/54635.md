## Summary

Keep this issue open. Current main still has the reported warning-policy behavior: mutating tool errors return `showWarning: true` before the recoverable-error suppression path, and tests explicitly expect recoverable mutating errors to produce user-facing failed notifications. The edit recovery wrapper is real partial mitigation for post-write throws, but it does not resolve the broader `lastToolError` warning-policy request, and the latest release tag has the same policy.

## Best Possible Solution

Keep this issue open. The likely maintainer path is a focused change around `src/agents/pi-embedded-runner/run/payloads.ts`, with possible `lastToolError` clearing changes in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, plus regression tests showing recoverable or proven-applied mutating failures do not emit false `failed` notifications while real mutating failures remain visible. Coordinate with #59868 and PR #60664, but do not treat either as shipped replacement work.

## Risks / Open Questions

- A fix must preserve user-visible warnings for genuine mutating failures such as permission errors, failed writes, and disk-full conditions.
- The current recoverable-error keyword check is coarse; maintainers should decide whether mutating suppression should use that same classifier or a narrower proven-applied/post-write recovery signal.
