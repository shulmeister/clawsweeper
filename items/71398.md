## Summary

Current main still has a real QQ Bot SecretRef gap. A literal `secretref:/...` value is accepted as a plain string and can reach the QQ token request as the client secret; structured QQ Bot SecretRefs are schema-accepted, but QQ Bot also lacks the channel secret contract needed for normal runtime materialization and SecretRef tooling coverage.

## Best Possible Solution

Keep this issue open. The best fix is in QQ Bot plus the shared SecretRef policy surface: add a QQ Bot secret contract covering `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, wire runtime assignment/materialization and docs surface entries, then add regression coverage for structured SecretRef resolution and the literal `secretref:/...` case. For the literal marker, either migrate/resolve it precisely if it was supported legacy syntax, or reject it during config validation with clear `openclaw doctor --fix` or structured SecretRef guidance.
