## Summary

Keep this issue open. Current main still resolves unset `agents.*.memorySearch.sync.watch` to `true` without any gateway/headless context, and gateway memory startup passes the same config into memory manager creation. Both builtin and QMD memory managers still create chokidar watchers whenever the resolved watch setting is true. The linked #64549 was a separate auto-indexing bug in the same subsystem and does not supersede this gateway-mode default policy request.

## Best Possible Solution

Implement a narrow runtime-aware memory sync default: thread an explicit gateway/headless execution context into memory sync resolution or manager creation, default `sync.watch` to false only for that context, keep explicit user overrides authoritative, and add resolver plus builtin/QMD manager tests for gateway and non-gateway behavior. Update schema help and public memory config docs if the default changes.

## Risks / Open Questions

- A correct fix should use a precise runtime/headless signal; keying only on `gateway.mode` may conflate local desktop gateway flows with headless multi-agent deployments.
- The fix should preserve explicit `sync.watch: true` and `sync.watch: false` overrides and the existing QMD startup deferral behavior.
