## Summary

Keep #70533 open. Current main still discovers bundled/global plugin candidates and materializes their manifests before applying plugin scope or allowlist filtering. The issue's `tools.allow` wording appears imprecise because current plugin enablement is `plugins.allow`, but the central request for discovery-time plugin filtering remains actionable and is not superseded by the closed startup issues.

## Best Possible Solution

Keep the issue open and implement discovery-time narrowing. Compute the required plugin id set from `plugins.allow`, deny rules, explicit entries, configured bundled-channel exceptions, memory/context slots, configured channels, provider/harness/setup triggers, and then pass that scope before manifest parsing. Reuse the persisted installed-plugin index where possible, pass `pluginIds` into installed-index manifest registry calls when scope is known, and add regression coverage proving skipped plugin manifests are not read, not just that skipped runtime modules are not imported.
