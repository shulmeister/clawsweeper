## Summary

Keep this PR open. Current main still lacks the requested Slack `channels.slack.logLevel` config/schema/type/runtime path, and the linked operator report has concrete evidence for socket-mode pong-timeout log noise. The PR is a focused implementation candidate, but the patch shown still needs review because OpenClaw constructs a custom `SocketModeReceiver`; forwarding `logLevel` only to Bolt `App` may not suppress the socket-mode logger that emits the warning.

## Best Possible Solution

Keep this PR open as the active implementation path for #71531, but require one more pass before merge: add a documented `channels.slack.logLevel` config contract using lowercase Slack logger values or an explicit mapping, update schema/types/generated config docs, and forward the resolved level to the `SocketModeReceiver`/socket-mode client path that owns the pong-timeout logger. Forwarding to Bolt `App` as well is fine, but receiver-side coverage is the important proof. Add focused schema and interop tests for configured `error` behavior and omitted default behavior, then run the changed Slack/config gates.
