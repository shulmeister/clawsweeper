## Summary

Current `main` already makes embedded-agent LLM timeouts configurable through existing config keys, so the reported hardcoded 15s behavior is implemented well enough to close.

## Evidence

- **Config schema exposes embedded LLM idle timeout:** `agents.defaults.llm.idleTimeoutSeconds` is defined in the agent-defaults schema as a supported config field for embedded LLM streaming idle timeout.
  - file: [src/config/zod-schema.agent-defaults.ts:156](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/config/zod-schema.agent-defaults.ts#L156)
- **Embedded runner resolves timeout from config:** `resolveLlmIdleTimeoutMs()` reads `cfg.agents.defaults.llm.idleTimeoutSeconds`, falls back to `agents.defaults.timeoutSeconds`, and only disables the default watchdog for cron when neither timeout is configured.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:22](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts#L22)
- **Each embedded attempt uses the resolved timeout:** The embedded runner computes `idleTimeoutMs` from config and wraps the model stream with `streamWithIdleTimeout(...)` for the actual request path.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1562](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-embedded-runner/run/attempt.ts#L1562)
- **Slug-generator path no longer depends on a fixed 15s when agent timeout is configured:** The slug-generator helper resolves its timeout from `agents.defaults.timeoutSeconds` and only falls back to 15000ms when no agent timeout is configured.
  - file: [src/hooks/llm-slug-generator.ts:25](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/hooks/llm-slug-generator.ts#L25)
- **Regression coverage exists for the configurable behavior:** Tests cover fallback to `agents.defaults.timeoutSeconds`, explicit `agents.defaults.llm.idleTimeoutSeconds`, cron behavior, and slug-generator timeout inheritance.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.test.ts:53](https://github.com/openclaw/openclaw/blob/9153598e659daf5a52c479731f5b4ab8a5bf16b7/src/agents/pi-embedded-runner/run/llm-idle-timeout.test.ts#L53)
- **Behavior is already in the latest release:** A scoped diff against `v2026.4.22` shows these timeout-related files unchanged, so the current implementation shipped in that release.
  - command: `git diff --stat v2026.4.22^{}..HEAD -- src/hooks/llm-slug-generator.ts src/hooks/llm-slug-generator.test.ts src/agents/pi-embedded-runner/run/llm-idle-timeout.ts src/agents/pi-embedded-runner/run/llm-idle-timeout.test.ts src/config/zod-schema.agent-defaults.ts src/config/types.agent-defaults.ts src/agents/pi-embedded-runner/run/attempt.ts src/agents/pi-embedded-runner/run.ts`

## Risks / Open Questions

- The exact proposed key `agents.defaults.embeddedPi.timeoutMs` is not implemented; the supported knobs are `agents.defaults.llm.idleTimeoutSeconds` and `agents.defaults.timeoutSeconds`.
- Later comments in the issue describe separate embedded streaming/SSE failures for some providers; those reports do not show that timeout configurability itself is still missing.

## Close Comment

Codex review: closing this as implemented on current `main`. The embedded runner already takes its LLM timeout from config in `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:22` (`agents.defaults.llm.idleTimeoutSeconds`, with fallback to `agents.defaults.timeoutSeconds`) and wires that into each embedded attempt in `src/agents/pi-embedded-runner/run/attempt.ts:1562`. The slug-generator path also uses the configurable agent timeout in `src/hooks/llm-slug-generator.ts:25` instead of staying fixed at 15s when `agents.defaults.timeoutSeconds` is set, with coverage in `src/agents/pi-embedded-runner/run/llm-idle-timeout.test.ts:53` and `src/hooks/llm-slug-generator.test.ts:41`. I also checked the relevant files against release `v2026.4.22`, and this timeout behavior is already present there. The requested behavior exists today, just under the existing timeout keys rather than a new `agents.defaults.embeddedPi.timeoutMs` key. If there is still a provider-specific embedded streaming bug, that should be tracked separately from this configurability request.
