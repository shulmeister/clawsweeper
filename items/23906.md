## Summary

Keep #23906 open. Current main still lacks a public per-call `sessions_spawn` tool policy field and does not transport `tools`, `toolsAllow`, or `disableTools` through the native subagent or gateway agent RPC path. Runner-level primitives exist, but they are not exposed through `sessions_spawn`; the provided discussion also records a recent keep-open review and plugin/MCP use cases that remain unresolved.

## Best Possible Solution

Keep the issue open. A proper fix should add a backward-compatible `sessions_spawn` tool policy field such as `tools: "none" | string[]` or explicit `disableTools`/`toolsAllow`, validate it in the public tool schema, carry it through `SpawnSubagentParams` and the gateway `agent` RPC, map no-tool spawns to `disableTools: true` and subsets to `toolsAllow`, support plugin/MCP tool ids, document precedence against `tools.subagents.tools`, and add tests for local-model no-tool spawns, core tool subsets, plugin/MCP tools, sandbox/security behavior, and the related #15032 isolation use case.
