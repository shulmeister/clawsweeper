## Summary

Keep this issue open. Current main at 5176dba8a03ca85034f99a6c1a4eb67ca1f55e55 still has the plugin loader logging every invalid plugin config validation directly in both runtime and CLI registry paths, with no invalid-plugin-config dedupe/backoff cache. The referenced fix commit exists as a GitHub commit for this exact symptom, but that code is not present in the checked-out main source, and the current test suite does not include the regression test described by that commit.

## Best Possible Solution

Keep this issue open and land a focused plugin-loader fix. The best fix is either to make every startup/metadata/scoped load validate against the resolved plugin entry config, or at minimum dedupe identical plugin invalid-config logs by plugin id and error text while preserving diagnostics/status. Add a regression test that calls the loader repeatedly with the same invalid plugin config and verifies one log line, diagnostics still present, and dedupe reset on `clearPluginLoaderCache()`.
