## Summary

Keep open. Current main has narrowed this PR: the exec default-host problem is handled through `host=auto` resolving to `gateway` when no sandbox exists, and the cron self-deadlock is handled by remapping nested cron-triggered agent work to the `nested` lane. But the browser-status part remains the canonical open work for #41553: current `openclaw browser status` still lacks the proposed layered Control UI/profile/CDP diagnostics, and browser-control HTTP errors still surface raw HTML body text.

## Best Possible Solution

Keep #42027 open, but narrow maintainer review toward the still-missing browser diagnostics. The best path is to rebase/split it: drop or revalidate the now-superseded exec and cron lane changes, then implement the layered browser status diagnostics and bounded HTML-error summarization under `extensions/browser`, with updated tests for status payload, CLI output, and proxy HTML failures.
