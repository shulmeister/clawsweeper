## Summary

Keep #40236 open. Current main has partial overlapping cron behavior, including `payload.timeoutSeconds`, one-shot `deleteAfterRun`, failure alerts, run-log/session pruning, and plugin hooks that can see `ctx.jobId`, but it does not implement the requested job-level lifecycle contract with timeout/retry/auto-cleanup/hooks. The related lifecycle-hook items (#37855 and PR #45465) are narrower and still open, while the cleanup/self-deletion discussion in #33167 was closed stale rather than shipped as the requested broader policy.

## Best Possible Solution

Keep the issue open, but narrow the maintainer path: define an additive cron lifecycle schema/protocol only if core wants this as first-class scheduler behavior; otherwise split into focused issues for recurring-job auto-cleanup, cron internal event hooks, and timeout semantics. Reuse the existing `timeoutSeconds`, `failureAlert`, run-log/session pruning, and plugin-hook surfaces where they already solve part of the problem.
