## Summary

Keep open. Current main and the latest release already document the correct Slack allowlist nuance, but not the exact warning/troubleshooting copy proposed here. More importantly, the PR's key claim that name-based keys under `groupPolicy: "allowlist"` "will never route successfully" is too broad: current Slack runtime still supports name-based configs via startup resolution and via `channels.slack.dangerouslyAllowNameMatching`. That makes this PR neither clearly obsolete nor safe to close as implemented.

## Evidence

- **Released docs already describe the current nuance:** The Slack docs in `v2026.4.22` already say channel allowlists should use stable IDs, unresolved channel-name entries are ignored for routing by default, and direct name/slug routing requires `channels.slack.dangerouslyAllowNameMatching: true`. That covers the main behavior behind the PR, though without this PR's stronger warning block.
