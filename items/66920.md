## Summary

Keep #66920 open. The report is recent, concrete, and describes a remaining WhatsApp group-only inbound degradation after 408 reconnect cycles. Current main has general socket/listener reconnect handling and group hydration on attach, but I did not find a group-subscription recovery path, a per-group liveness watchdog, or regression coverage for repeated 408 cycles where DMs continue while bound group messages stop arriving.

## Best Possible Solution

Keep the issue open and treat it as a WhatsApp plugin reliability bug. The strongest maintainer path is to add a focused regression harness for repeated connection close/reopen cycles with both DM and group traffic, then either add group-subscription rehydration/liveness recovery or document and fix the Baileys dependency behavior needed for group upserts after reconnect. Coordinate with #63410 for the 408 loop and with the still-open group inbound context from #39608.
