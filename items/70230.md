## Summary

Keep open. Current main still does not expose connection-level health fields or a lightweight ping RPC. The health response is a service/channel/agent/session snapshot, while the WebSocket transport only has an internal tick watchdog and heartbeat event methods that do not provide `connected`, `rttMs`, or `lastHeartbeatAt` on `health`. The provided related items are about channel health mismatches/stale channel sockets, not a canonical tracker for this gateway connection-status API request.

## Best Possible Solution

Keep the issue open and implement an additive core gateway transport status surface: either add a `connection` block to `health` with `connected`, `rttMs`, and `lastHeartbeatAt`, or add a lightweight `gateway.ping`/equivalent RPC. The implementation should live in the gateway protocol/server-method layer, compute RTT from a real request/response path or explicit ping/pong, and update docs, UI types, and focused gateway/client tests.
