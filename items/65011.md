## Summary

Keep open. Current main has the native updater primitives, auto-update config/state, startup update hints, a Control UI update-available banner, and a generic config-screen Update button. It still does not provide the requested first-class Control UI Update Manager with full status, dry run, channel explanations, release notes, persisted last success/failure display, safe-update controls, or post-update health summary. The linked companion auto-update kit is useful adoption evidence, but it is not an OpenClaw core implementation; related #12855, #14526, and #62492 cover overlapping pieces rather than superseding this UI/product-surface request.

## Best Possible Solution

Keep this issue open and implement a first-class Control UI Update Manager, likely under Settings or Infrastructure. Add or extend read RPCs to expose current version, latest version, channel, install kind, last check, last attempt/success/failure, and release-note links; wire auto-update toggles through the existing config schema; add dry-run and update-now actions with visible progress and full ok/skipped/error outcomes; then layer backup, doctor/health, plugin sync, and rollback guidance in coordination with #12855, #14526, and #62492.

## Risks / Open Questions

- The request bundles several scopes: update status UI, auto-update config UX, dry-run/update progress, release-note linking, backup/rollback, and health probes. Maintainers may want to split implementation into narrower issues while keeping this issue as the product umbrella.
- Some safe-update building blocks already exist elsewhere, such as CLI update, doctor, health, and backup commands. The UI work should orchestrate those primitives instead of creating a parallel updater.
