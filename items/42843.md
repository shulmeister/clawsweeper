## Summary

Current main does not implement PR #42843's requested WhatsApp reconnect behavior. The WhatsApp plugin still defaults `web.reconnect.maxAttempts` to `12`, the reconnect controller still returns `action: "stop"` when the cap is reached, the monitor still shuts down and breaks out of the loop on that stop decision, and the current e2e test suite still asserts max-attempt stop behavior. Later main has useful adjacent fixes for reconnect-gap delivery and status reporting, but those do not make the PR obsolete because the bounded retry loop can still permanently stop.

## Best Possible Solution

Keep #42843 open for maintainer/product judgment. The best path is to rebase the idea onto `extensions/whatsapp`, decide whether the default should become unlimited, implement degraded slow retry in `WhatsAppConnectionController` and the monitor without misleading `Retry 0/N` logs, preserve terminal stop behavior for logged-out/session-conflict states, and update regression tests plus config/docs semantics for `maxAttempts: 0`.
