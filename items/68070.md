## Summary

Keep PR #68070 open. Current `main` does not already implement the PR's requested behavior: required-parameter validation still throws a plain `Error`, the tool-definition adapter drops any retryability metadata, and the generic tool-loop detector is disabled by default, so it is not a substitute for marking validation failures non-retryable. The PR remains a plausible focused bug fix, though it should get a regression test and dependency-contract verification before merge.

## Best Possible Solution

Keep the PR open and ask for a small regression test around `wrapToolParamValidation` plus `toToolDefinitions` that asserts missing/malformed required params return a structured tool error with `retryable: false`. Before merging, maintainers should verify the supported pi-agent-core/pi-coding-agent contract for non-retryable tool results and prefer a typed local validation-error helper over an unstructured ad-hoc property if the dependency exposes a clearer API.
