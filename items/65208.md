## Summary

Current `main` and the shipped `v2026.4.23` release already construct managed Chrome launches with `--remote-debugging-port=${profile.cdpPort}`, pass that argv through the macOS spawn path unchanged, and lock the behavior with tests, so the 2026.4.11 report no longer matches current OpenClaw behavior.

## Evidence

- **Launch args include the CDP port on current main:** `buildOpenClawChromeLaunchArgs` prepends `--remote-debugging-port=${profile.cdpPort}` to the managed Chrome argv.
  - file: [extensions/browser/src/browser/chrome.ts:111](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/browser/src/browser/chrome.ts#L111)
  - command: `nl -ba extensions/browser/src/browser/chrome.ts | sed -n '96,140p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Managed launch path passes those args into `spawn(...)`:** `launchOpenClawChrome` builds the argv, runs it through `prepareOomScoreAdjustedSpawn`, and then calls `spawn(preparedSpawn.command, preparedSpawn.args, ...)`, so the remote-debugging flag is part of the spawned command line.
  - file: [extensions/browser/src/browser/chrome.ts:284](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/browser/src/browser/chrome.ts#L284)

  - file: [extensions/browser/src/browser/chrome.test.ts:666](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/browser/src/browser/chrome.test.ts#L666)
