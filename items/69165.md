## Summary

Keep open. Current main has adjacent Telegram protections, including grammY throttling on the live bot instance, retry_after-aware retries for direct Telegram helper calls, and a generic write-ahead delivery queue. Those do not implement the requested Telegram outbound scheduler: same-chat/topic sends are not serialized across cron, agent replies, previews, and message-tool/direct outbound paths; there is no per-chat rate limit or max-depth/drop-oldest config; and best-effort delivery only changes error propagation rather than making submission non-blocking and rate-limited.

## Best Possible Solution

Keep #69165 open. The best implementation is a Telegram-owned outbound scheduler shared by direct outbound, message tools, cron/heartbeat delivery, bot final/block replies, previews, and media sends. Key by Telegram account plus chat/topic, serialize same-key sends, enforce configurable per-chat/group/global pacing, honor Telegram retry_after before retrying, integrate with the existing write-ahead delivery queue for crash recovery, and define bestEffort/maxQueueDepth/drop-oldest behavior in config, docs, and regression tests.

## Risks / Open Questions

- A fix must preserve Telegram send idempotency: widening immediate retries for ambiguous network failures can duplicate visible messages, which the current retry policy intentionally avoids.
- The scheduler key needs explicit account plus chat/topic semantics so forum topics, DMs, and groups are neither over-throttled nor under-throttled.
