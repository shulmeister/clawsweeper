## Summary

Current main already uses the isolated-cron direct delivery path for channel sends, includes explicit WhatsApp coverage for that path, and the same implementation is already present in release `v2026.4.22`.

## Evidence

- **Isolated cron runs dispatch direct delivery:** `runCronIsolatedAgentTurn` finalization calls `dispatchCronDelivery(...)` with the resolved channel/account/target for isolated runs, which is the direct cron delivery path rather than a separate cron-local listener flow.

  - file: [src/cron/isolated-agent.direct-delivery-core-channels.test.ts:247](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/isolated-agent.direct-delivery-core-channels.test.ts#L247)

## Risks / Open Questions

- I did not run a live WhatsApp/gateway reproduction in this read-only review, so this conclusion relies on current code, release contents, and regression coverage rather than an end-to-end runtime check.
- If someone can still reproduce on `v2026.4.22` or newer, the remaining problem is more likely environment-specific (for example multiple gateway processes or stale runtime state) than the original isolated-cron delivery gap.

## Close Comment

Closing after a Codex review of current `main`: isolated cron runs already go through the direct cron delivery path (`src/cron/isolated-agent/run.ts:864`), WhatsApp is explicitly covered in that path’s regression tests (`src/cron/isolated-agent.direct-delivery-core-channels.test.ts:247`), and the WhatsApp sender resolves its active web listener from the shared connection-controller registry (`extensions/whatsapp/src/connection-controller-registry.ts:11`, `extensions/whatsapp/src/send.ts:39`). The related isolated-cron delivery fix is also present in the shipped `v2026.4.22` release (`CHANGELOG.md`, release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with exact repro details from that version, including `openclaw cron show`, `openclaw cron runs`, and `openclaw channels status --probe` output.
