## Summary

Keep open. Current main has useful degradation behavior, including embedding storage in `chunks` and in-process cosine fallback when sqlite-vec is unavailable, but it still does not directly address this report's core failure mode: sqlite-vec can appear to load while not registering usable vec functions/modules. The loader returns `ok: true` after `db.loadExtension`/`sqliteVec.load` without a post-load health probe, and `memory-core` marks vector availability true before separately creating the `vec0` virtual table. The issue also has two independent confirmations on 2026.4.14/2026.4.15, and nearby fixes such as #64776/#67898 address different runtime-loading or warning-noise problems rather than ABI/missing-function validation.

## Best Possible Solution

Keep this open and fix it in `memory-core`/`memory-host-sdk`: after loading sqlite-vec, run a deterministic health probe such as creating/dropping a temporary `vec0` table or preparing known sqlite-vec functions, catch that failure as `vector.available=false`, preserve the existing in-process cosine fallback, surface an actionable load error in status/doctor, and add a regression test for the exact load-success/no-registered-functions case. If the probe confirms an upstream binary ABI mismatch, then update or rebuild the bundled sqlite-vec dependency with release-note coverage.

## Risks / Open Questions

- The reporter's `better-sqlite3` detail is stale relative to current and v2026.4.11 source, which use `node:sqlite`; the fix should verify the actual ABI/load failure rather than simply pinning `better-sqlite3`.
- A dependency upgrade alone may not be enough if OpenClaw still treats `loadExtension` success as sqlite-vec health; add a regression for load-success-but-vec0-unusable behavior.
