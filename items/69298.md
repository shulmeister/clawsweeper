## Summary

Keep #69298 open. Current main still maps Codex app-server `inputTokens` into OpenClaw `usage.input` while also storing `cachedInputTokens` as `usage.cacheRead`; the shared usage, session, cache-stat, and status paths still treat `input + cacheRead + cacheWrite` as additive prompt usage. A current test only proves cumulative `total`-only notifications are ignored; it does not cover the reported case where a fresh Codex usage breakdown's `inputTokens` already includes cached input. The provided maintainer comment also explicitly says this remains unfixed on current main and that related PR #70749 does not prove this issue fixed.

## Best Possible Solution

Keep the issue open and fix the Codex app-server usage adapter. The narrow path is to normalize Codex `inputTokens` to uncached input when paired with `cachedInputTokens`, while preserving the current guard that avoids treating cumulative-only `total` usage as fresh context. Add regression coverage for the issue's observed arithmetic, plus session/status assertions showing `Context` and cache hit rate do not use `input + cacheRead` when `input` already includes cached tokens.
