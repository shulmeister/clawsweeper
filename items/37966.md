## Summary

Keep issue #37966 open. Current main still does not implement the reported LiteLLM OpenAI-compatible Anthropic cache-control path: the bundled LiteLLM provider uses `openai-completions`, cache retention only resolves for custom providers when the model API is `anthropic-messages`, and the only generic `cache_control` payload marker wrapper is scoped to verified OpenRouter routes. The linked fix PR #38221 was closed unmerged, and current tests only prove LiteLLM cache retention for an `anthropic-messages` model, not the documented/default LiteLLM `/v1` OpenAI-compatible path in the bug report.

## Best Possible Solution

Keep the issue open until OpenClaw either implements a conservative LiteLLM Anthropic pass-through path or a maintainer explicitly decides this is unsupported. A likely fix belongs in the LiteLLM/provider-wrapper path or shared proxy wrapper logic: detect an explicitly configured `cacheRetention` for LiteLLM Anthropic model refs on `openai-completions`, inject Anthropic `cache_control` blocks into the OpenAI-compatible payload, and add regression coverage for the documented `litellm/claude-opus-4-6` route. If the current docs are the intended product boundary, the maintainer should say so on the issue rather than auto-close it as stale.
