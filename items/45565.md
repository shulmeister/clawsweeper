## Summary

Keep open. Current main has only partial/noise-reduction work: `messages.suppressToolErrors` can suppress some non-mutating tool warnings, and cron has a separate failure-destination path, but the requested dedicated routing target for gateway lifecycle/internal warnings is not implemented. Mutating tool failures such as edit failures still intentionally surface as user-visible warning payloads, so the central problem described in #45565 remains.

## Evidence

- **Issue scope:** #45565 requests routing gateway lifecycle warnings, then broadens in comments to all internal warning/tool-error/system notices via `internalWarningTarget` or per-category `messageRouting`; the related #28637 is described as a cron-specific variant, not as a complete replacement for this broader routing request.
- **Messages config has suppression only, no routing target:** `MessagesConfig` exposes response/ack/queue/inbound/status/TTS settings and `suppressToolErrors`, but no `lifecycleTarget`, `internalWarningTarget`, `messageRouting`, `toolErrors`, or `systemNotices` routing field.
