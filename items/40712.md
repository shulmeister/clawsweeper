## Summary

Keep open. Current main has important mitigation for ordinary managed restarts: SIGUSR1/config restarts are deferred while active embedded runs, queues, replies, and tasks drain, and startup has a narrow main-session recovery path for stale transcript locks. But the requested durable partial-generation persistence/resume is still not implemented: recovery only resumes when the transcript tail is user/tool/toolResult, and assistant-tail interrupted transcripts are explicitly treated as non-resumable/failed. That leaves the crash/forced-restart and partial assistant generation case from this issue unsolved.

## Best Possible Solution

Keep this open, but narrow the maintainer scope to the remaining gap: durable generation-state recovery for interrupted assistant output after a crash, forced restart, or drain timeout. The likely implementation belongs around SessionManager-safe transcript/checkpoint writes plus startup recovery that can distinguish generating, completed-not-delivered, sent, and interrupted states across main and non-main sessions, with regression tests for mid-assistant text, completed-before-channel-delivery, and provider replay safety.

## Risks / Open Questions

- Existing restart drain/recovery already mitigates normal managed restarts, so any new work should preserve that behavior instead of replacing it wholesale.
- Durable partial assistant-message persistence must respect the SessionManager parentId/DAG transcript contract and provider replay constraints.
- Resuming partial visible replies can duplicate channel output unless it is coordinated with delivery queues, draft previews, and final-message acknowledgements.
