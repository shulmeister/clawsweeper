## Summary

Keep open. Current main still does not provide a supported way to pass arbitrary inbound OpenAI-compatible HTTP request headers, such as `custom-token`, into an agent run or expose them to a Python skill script. The existing code and docs support fixed skill env/API-key configuration and a small set of OpenClaw control headers, while the related provider-header/API-key items are outbound provider transport issues rather than this inbound skill-context request.

## Best Possible Solution

Keep open for maintainer product/security design. The safest implementation path is an explicit allowlist that maps selected inbound OpenAI-compatible HTTP headers into per-run context or per-skill environment with redaction, sandbox semantics, per-skill opt-in, docs, and gateway tests. If dynamic secret passthrough is intentionally unsupported, document that custom skill credentials must be configured statically through `skills.entries.<skill>.env` or `apiKey`.
