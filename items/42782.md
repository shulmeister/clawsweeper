## Summary

Current `main` already implements Telegram polling auto-reconnect via polling liveness tracking, stall watchdog restarts, transport rebuilds, and regression coverage, so the feature request is satisfied.

## Close Comment

Closing as implemented on current `main`. Codex review found that Telegram polling now has its own auto-recovery path: recoverable polling network failures trigger a restart in `extensions/telegram/src/monitor.ts:91`, stalled or silent `getUpdates` activity is detected and restarted by the watchdog in `extensions/telegram/src/polling-session.ts:246` and `extensions/telegram/src/polling-liveness.ts:92`, and there is regression coverage for both the network-failure and watchdog paths in `extensions/telegram/src/monitor.test.ts:546` and `extensions/telegram/src/monitor.test.ts:721`. The behavior is also documented in `docs/channels/telegram.md:260` and shipped in the `2026.4.18` changelog (`CHANGELOG.md:457`, `CHANGELOG.md:481`). If you still hit silent Telegram polling stalls on a newer build, please open a fresh issue with the exact version, config, and logs around `Polling stall detected` or `Restarting polling after unhandled network error`.
