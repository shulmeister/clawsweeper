## Summary

Keep open. Current main still has the code-path mismatch described in #66279: the TUI resolves the local `wss://127.0.0.1:<port>` target but does not hydrate or pass the local TLS fingerprint into `GatewayClient`, while status/probe paths do. `GatewayClient` only enables the custom self-signed fingerprint verification path when `tlsFingerprint` is present, so the reported TUI failure remains plausible on current main.

## Best Possible Solution

Keep #66279 open and fix the TUI path in core. A narrow fix should extend the TUI resolved connection shape with `tlsFingerprint`, load the local gateway TLS runtime when `gateway.tls.enabled === true`, the resolved target is local non-override `wss://`, and then pass that fingerprint into `GatewayClient`. Add regression coverage in `src/tui/gateway-chat.test.ts` proving local TLS TUI connections pass the fingerprint and CLI URL overrides do not accidentally inherit a local pin. Consider a separate explicit `openclaw tui --tls-fingerprint` path only if maintainers want remote pinned TLS support for TUI too.
