## Summary

Keep #70588 open. Current main has Discord webhook creation only as an internal thread-binding helper, not as a shared `message` tool action. The Discord message-tool discovery advertises channel/category/thread actions but no `webhook-create`, the shared message action-name union has no webhook action, and the Discord action dispatcher has no webhook creation route. The existing internal webhook path supports bound thread/persona delivery, but it does not expose a tool action or return a generated webhook URL for orchestration workflows.

## Best Possible Solution

Keep the issue open for a Discord-owned implementation or maintainer product decision. A plausible fix would add a gated `webhook-create` message action in the Discord plugin, extend the shared action-name contract if needed, call Discord `Routes.channelWebhooks` with target channel/name/avatar inputs, return a carefully documented webhook URL or structured id/token response, and add focused tests/docs for secret handling and Manage Webhooks permission failures.

## Risks / Open Questions

- Returning a Discord webhook URL would expose a bearer-style secret, so any implementation needs explicit action gating, redaction/logging care, docs, and tests around permission failures and response shape.
- The internal thread-binding webhook helper proves the Discord REST call exists in main, but exposing it through the shared message tool is a separate product/security surface.
