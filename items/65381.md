## Summary

Keep open. Current main still displays cache usage in `/status` but computes the status cost from `inputTokens`/`outputTokens` only, so the cache-token undercount remains real. This PR is a useful implementation candidate, though it now needs to be ported/rebased onto `src/status/status-message.ts` because `src/auto-reply/status.ts` only re-exports the status builder on current main.

## Best Possible Solution

Keep the PR open as a valid fix candidate. Rebase/port the small production change into `src/status/status-message.ts`, include `cacheRead`/`cacheWrite` in both the `hasUsage` guard and `estimateUsageCost` usage object, and keep or add the focused regression test in `src/auto-reply/status.test.ts` for cache-only/cache-inclusive status cost. Then decide whether this narrower PR supersedes the older draft #52329 or whether to consolidate that PR's broader cost-breakdown ideas separately.
