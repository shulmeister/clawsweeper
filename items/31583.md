## Summary

Keep #31583 open. Current main still does not fully satisfy the reported `skills.entries.*.env` behavior for the `gog` case: `GOG_KEYRING_PASSWORD` is a user-configured password-pattern key, the repo `gog` skill does not declare it in metadata, and the sanitizer only re-allows sensitive keys declared by skill metadata. The provided discussion also points to open PR #50960 as an unmerged candidate fix, not shipped behavior.

## Best Possible Solution

Keep #31583 open until #50960 or an equivalent narrow fix lands. The fix should allow explicitly user-configured `skills.entries.<skill>.env` keys such as `GOG_KEYRING_PASSWORD` to bypass only the sensitive-name sanitizer while preserving the dangerous host-env blocklist, add regression coverage for an undeclared password/token key, and separately either thread applied skill env into sandbox exec or keep the host-only sandbox contract clearly documented.
