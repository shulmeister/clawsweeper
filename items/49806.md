## Summary

Current main already contains loader-side SDK aliasing that resolves `openclaw/plugin-sdk/*` imports for source-loaded bundled plugins, and the same implementation is present in the tagged release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The iMessage runtime still imports `openclaw/plugin-sdk/outbound-runtime`, but OpenClaw now rewrites those package-specifier imports through the plugin loader instead of requiring a literal installed `openclaw` package next to the extension.

## Close Comment

Closing from a Codex review. Current `main` already handles this class of failure in the plugin loader: `src/plugins/sdk-alias.ts` maps both `openclaw/plugin-sdk/*` and `@openclaw/plugin-sdk/*` to real SDK source/dist files, and `src/plugins/loader.ts` routes source `.ts` runtime shims through Jiti so those imports are rewritten instead of requiring a literal installed `openclaw` package next to the plugin. There is regression coverage for that exact source-runtime alias path in `src/plugins/sdk-alias.test.ts`. The same implementation is present in the shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this report looks fixed on supported builds. If you can still reproduce on `v2026.4.22` or newer, please open a new issue with the exact workspace layout, startup command, and full gateway log.
