## Summary

Keep open. Current `main` still requires a persisted `sessionId` to reuse an existing thread session, and the inspected code does not reconstruct missing `sessionId` metadata from transcript files on disk. That matches the reported restart/context-loss failure more than a verified fix.

## Evidence

- **Session reuse still fails open when `sessionId` is missing:** `canReuseExistingEntry` is only true when the stored entry already has a `sessionId` and valid `updatedAt`. Otherwise the code generates a brand-new UUID and marks the turn as a new session, which would drop prior thread context instead of recovering it.
  - file: [src/auto-reply/reply/session.ts:427](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/auto-reply/reply/session.ts#L427)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '424,492p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Gateway history still returns zero messages when `sessionId` is absent:** The chat history path loads local transcript messages only when `entry?.sessionId` exists. If the persisted store entry lacks `sessionId`, `localMessages` becomes `[]`, matching the reported `messages=0` symptom after restart.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/server-methods/chat.ts#L1891)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1880,1910p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **No current-main repair path for missing persisted IDs:** `resolveAndPersistSessionFile()` writes back `sessionId` and `sessionFile` only from the session id already chosen by the caller. It does not scan existing JSONL transcripts or reconstruct a missing/stale `sessionId` from disk before reuse.
  - file: [src/config/sessions/session-file.ts:18](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/config/sessions/session-file.ts#L18)
  - command: `nl -ba src/config/sessions/session-file.ts | sed -n '18,46p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Issue discussion adds a second independent repro surface:** The provided issue timeline includes the original Slack/thread restart report plus April 21 Telegram group-topic comments describing stale/broken `sessionFile` pointers after restart and explicitly noting that #64795 was a different isolated-agent path. That discussion strengthens the case that this is still a real, cross-surface bug rather than an obsolete report.

## Risks / Open Questions

- No live reproduction was run in this read-only review, so this conclusion is based on code inspection and the issue discussion.
- The exact write path causing the bad store entries may be elsewhere, but no fix or recovery mechanism for missing persisted `sessionId` was found on current `main`.
