## Summary

Keep #39232 open. Current main has an opt-in WhatsApp `message_received` observer, but it is not the requested pre-filter raw inbound hook: the typed plugin hook catalog still has no `on_raw_inbound`/`onRawInbound`/`before_channel_filter`, WhatsApp still drops access-denied messages before normalization returns, and group mention/access gating still returns before `processMessage()` can emit `message_received`. The related PR #39240 attempted a WhatsApp-specific implementation but was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open for maintainer/API design. A good implementation would add an observe-only, privacy-aware pre-filter inbound contract to the typed plugin/channel SDK, wire bundled channels before access and mention gating where explicitly enabled, document the privacy model on the public plugin/channel docs, and add contract plus channel-specific ordering tests. If maintainers decide ambient raw observation is not supported, document the policy and supported narrower alternatives such as the current post-gating `message_received` opt-in.

## Risks / Open Questions

- A pre-filter raw inbound observer would expose content and identifiers that current access control and mention gating intentionally suppress from plugins, so it needs explicit opt-in, privacy documentation, and security review.
- The product/API shape is still a maintainer decision: global typed hook, per-channel observe sink, or a documented policy that ambient raw observation is out of scope.
