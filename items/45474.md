## Summary

Keep #45474 open. The current main code has partial WhatsApp runtime health reporting for reconnect/conflict states, but the reported silent inbound-drop path is still plausible: unresolved @lid sender JIDs still resolve to null and direct inbound messages are dropped before dispatch. The already-linked PR #45590 explicitly claims to address #45474, but it is still open/unmerged in the provided context, so current main is not enough evidence to close this bug.

## Evidence

- **Reporter reproduced after restart:** The issue discussion includes a live validation after relink plus full gateway restart where the channel reported linked/running/connected, but a real inbound WhatsApp probe produced no inbound log event and `lastInboundAt: null` (#45474 comment 4058291882 / 4058297398).
