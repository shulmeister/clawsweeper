## Summary

This request is better scoped as plugin/ClawHub work than OpenClaw core. Current docs define skills as tool-using instructions, OpenClaw's scope directs optional capability to plugins/ClawHub, the bundled OpenProse extension already provides mid-execution human-in-the-loop input flows, and current main already has substrate-side `request_user_input` plumbing in the Codex harness.

## Evidence

- **Plugin-first scope:** Project vision says optional capability should usually ship as plugins, prefers bundle-style plugins for skills-like surfaces, and says new skills should go through ClawHub first rather than core.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/VISION.md#L54)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Skills are tool instructions:** Creating-skills docs define skills as teaching the agent how and when to use tools, and note that skills can define tool schemas or ship inside plugins alongside the tools they document.
  - file: [docs/tools/creating-skills.md:9](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/tools/creating-skills.md#L9)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Existing extension already fits the use case:** OpenProse is documented as a plugin that installs a skill pack and `/prose` command, and its example syntax includes `input ...` prompts for interactive workflows.
  - file: [docs/prose.md:10](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/prose.md#L10)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **AskUserQuestion and pause-for-input already exist in extension docs:** The bundled OpenProse skill explicitly tells the agent to use AskUserQuestion, and the OpenProse runtime docs say inputs can appear mid-execution and pause for user input when needed.
  - file: [extensions/open-prose/skills/prose/help.md:17](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/open-prose/skills/prose/help.md#L17)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Current main already has substrate-side user-input plumbing:** Codex harness docs say `request_user_input` prompts are sent back to the originating chat and answered by the next queued follow-up message; dedicated tests cover both the bridge and run-level routing.
  - file: [docs/plugins/codex-harness.md:620](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/plugins/codex-harness.md#L620)

## Risks / Open Questions

- If maintainers want a single documented cross-harness user-input contract for third-party plugins beyond the current Codex/OpenProse patterns, closing this issue could hide that narrower API-design gap.
