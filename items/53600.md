## Summary

Keep this issue open. Current main contains useful mechanics and partial diagnostics for the surfaces named in the report, but it does not contain the requested native-VPS performance analysis or instrumentation plan with per-layer latency estimates, quick-win config guidance, code-level optimization targets, and explicit instrumentation gaps.

## Best Possible Solution

Keep this issue open and scope it into a bounded native-Linux gateway performance investigation. The best maintainer path is to capture one timing trace around context/bootstrap assembly, prompt/token estimation, memory search and query embedding, compaction trigger/model-call duration, lane wait and subagent spawn/announce-back, Telegram debounce/preview/final delivery, bundled hook execution, auth-profile fallback decisions, and process RSS/heap/GC; then publish a VPS/debugging runbook with quick-win config changes and file narrower optimization tickets for measured hot spots.
