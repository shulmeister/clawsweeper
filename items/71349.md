## Summary

Keep #71349 open. Current main does not show a shipped fix for the reported memory-lancedb autoCapture behavior, and the adjacent doctor warning still matches current code. The plugin has an autoCapture hook, but it is selective, mostly silent when no heuristic matches, and the agent runners invoke agent_end hooks fire-and-forget; that is not enough evidence to close a recent report with concrete config, version, and observed DB state.

## Best Possible Solution

Keep the issue open and fix or document the intended memory-lancedb capture contract. If per-turn/session capture is expected, add an awaited or lifecycle-safe capture path across embedded, CLI, cron, and channel turns, add debug/status visibility for skipped capture reasons, make doctor recognize the active memory slot/plugin instead of only the legacy backend resolver, and add regression coverage using the reporter-style memory-lancedb + baseUrl/dimensions config. If high-signal-only capture is intended, update docs/UI copy and doctor output so users do not expect every session to persist entries.
