## Summary

Current main still does not implement broad per-session or per-turn `memory-lancedb` autoCapture. The plugin only captures on `agent_end` after successful runs, only scans user-role text, requires hard-coded trigger matches, applies length and prompt-injection filters, checks duplicates, and processes at most the first three capturable texts from the hook payload. The adjacent doctor warning is also still structurally valid for slot-only `memory-lancedb` because the plugin registers tools/hooks/CLI/service but no memory capability/runtime for doctor/status discovery.

## Best Possible Solution

Keep this issue open. Maintainers should define the intended `memory-lancedb` autoCapture contract, then either implement robust current-turn/session capture with poisoning safeguards, duplicate/history cursor handling, retry semantics, and regression tests, or document the current heuristic-only behavior clearly. The adjacent doctor/status false-negative should be handled by making `memory-lancedb` register the memory capability/runtime if it is meant to satisfy the active memory backend contract, with related PRs such as #42083 and #65066 reviewed as partial implementation candidates rather than treated as shipped fixes.
