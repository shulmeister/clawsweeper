## Summary

Close as `clawhub`: current main does not implement per-user `users/*.md`, but OpenClaw already exposes hook/plugin seams for per-sender bootstrap and prompt customization, and the issue now links an open-source community skill for this exact feature layer. Under `VISION.md`, optional memory/context behavior belongs in plugins/skills/ClawHub rather than new core workspace semantics.

## Evidence

- **Core still has one shared USER.md bootstrap slot:** The canonical workspace bootstrap set includes a single `USER.md`, and `loadWorkspaceBootstrapFiles(...)` only loads the fixed root files (`AGENTS.md`, `SOUL.md`, `TOOLS.md`, `IDENTITY.md`, `USER.md`, `HEARTBEAT.md`, `BOOTSTRAP.md`, `MEMORY.md`). There is no built-in `users/*.md` lookup on current `main`.
  - file: [src/agents/workspace.ts:19](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/workspace.ts#L19)
  - command: `nl -ba src/agents/workspace.ts | sed -n '16,40p;600,650p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Bootstrap file selection is already hookable per session:** Before bootstrap files are injected, `applyBootstrapHookOverrides(...)` sends the mutable bootstrap file list plus `sessionKey`/`agentId` through the internal `agent:bootstrap` hook, so userland code can replace or augment the injected files without changing core workspace semantics.
  - file: [src/agents/bootstrap-hooks.ts:7](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/bootstrap-hooks.ts#L7)

- **Hooks expose sender/session data needed for per-user routing:** Docs and types show `agent:bootstrap` fires before bootstrap injection, `context.bootstrapFiles` is mutable, and message hooks expose provider metadata including `senderId`. That is enough surface to map sender identity to per-user files in a hook or plugin.
  - file: [docs/automation/hooks.md:45](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/automation/hooks.md#L45)

Current docs position this as the supported extension path for custom context behavior.

## Risks / Open Questions

- This does not claim feature parity with the proposed `users/*.md` format; it only shows the behavior belongs on existing extension seams rather than new core workspace semantics.
- Per-user isolation beyond bootstrap/prompt context (for example memory-index or transcript partition policy) may still need separate plugin work or narrower extension requests.
