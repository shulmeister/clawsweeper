## Summary

Keep open. Current `main` still routes Windows exec-tool commands through PowerShell (`-NoProfile -NonInteractive -Command`) before spawning, while only setting `windowsHide: true` on the child process. That means the reported flashing-console regression is not implemented away on `main`, and I do not have Windows-local proof that it is obsolete.

## Evidence

- **Windows shell config still hard-codes PowerShell:** `getShellConfig()` returns PowerShell on Windows and explicitly documents that Windows commands are wrapped as `powershell[.exe] -NoProfile -NonInteractive -Command ...` rather than being spawned directly.
  - file: [src/agents/shell-utils.ts:54](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/shell-utils.ts#L54)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Exec tool host path still wraps command text in that shell:** `bash-tools.exec-runtime` builds `childArgv` as `[shell, ...shellArgs, execCommand]` for non-sandbox execs, so Windows exec-tool calls still go through the PowerShell wrapper instead of direct executable spawning.
  - file: [src/agents/bash-tools.exec-runtime.ts:669](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/bash-tools.exec-runtime.ts#L669)
