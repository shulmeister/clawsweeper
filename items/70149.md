## Summary

Keep open. The related #61825 close covered broad WhatsApp env-proxy support, but this newer report is narrower: current main still gives Baileys an ambient `proxy-agent` instance for the WhatsApp WSS handshake, which is the exact path the reporter says times out while an explicit `HttpsProxyAgent` succeeds. The upload/media `fetchAgent` path is separate and does not fix the WebSocket opening handshake.

## Best Possible Solution

Keep the issue open and fix the WhatsApp WSS login proxy path directly. A likely maintainer path is to add a WhatsApp WebSocket proxy resolver that turns `HTTPS_PROXY`/`https_proxy` or fallback `HTTP_PROXY`/`http_proxy` into an explicit `HttpsProxyAgent` for Baileys `agent`, while preserving `NO_PROXY` behavior and unsupported-proxy handling. Add focused tests that distinguish the WebSocket `agent` from upload `fetchAgent` and prove the WSS path no longer uses the ambient `proxy-agent` object identified in this report.
