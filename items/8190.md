## Summary

Keep #8190 open. Current main still has only per-session reset/soft-reset and per-session CLI auth reuse invalidation. It does not expose a global gateway/API/chat action that refreshes auth for all active channel sessions while preserving conversation context, and the existing issue discussion already identifies this as the canonical remaining tracker.

## Best Possible Solution

Keep #8190 open as the canonical tracker. The best implementation is an additive gateway/session capability, such as `sessions.refreshAuth` or an explicit all-sessions auth-refresh mode, that enumerates active sessions, invalidates auth-backed runtime and CLI caches, preserves transcripts and channel/session context where possible, and adds protocol, gateway, session-store/channel tests plus user-facing docs.

## Risks / Open Questions

- Closing this would drop the canonical tracker for a cross-channel auth/session lifecycle capability that remains unimplemented.
- A naive implementation that loops over current `sessions.reset` would rotate session ids and archive transcripts, conflicting with the issue's explicit context-preservation requirement.
- The remaining design needs tests across gateway protocol, session store/reset behavior, CLI auth bindings, and representative channel sessions.
