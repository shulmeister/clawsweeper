## Summary

Keep #64426 open. Current main has a configurable generic tool-loop detector that can warn for repeated identical `read` calls when `tools.loopDetection.enabled` is true, but it is disabled by default and does not cover the reported same-turn, small-set reread pattern strongly enough: the concrete incident's highest repeated file count was 9, while the default generic warning threshold is reached only after 10 prior identical calls, and multi-file reread ratios/path-set loops are not modeled as a first-class read-specific detector.

## Best Possible Solution

Keep the issue open and either extend `tools.loopDetection` with a read-specific per-run detector or define a narrower follow-up. The likely implementation should track canonical `read` paths within a single run/turn, detect repeated same-path and small-path-set reread ratios before edit/write/test/broadened-evidence steps, emit warning-only diagnostics/tool feedback first, and add tests for the concrete 2-4 file reread pattern described in the issue.
