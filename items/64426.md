## Summary

Keep open. Current main has an opt-in generic tool-loop detector that can warn on repeated identical `read` calls when enabled, but it does not implement the requested default/runtime same-turn reread guard for the first-class `read` path, same-small-file-set loops, or no-edit/no-test progress heuristics. The related #63352 is still open and scoped to exec-based rereads, so it does not supersede this read-tool proposal.

## Best Possible Solution

Keep this issue open and implement a read-specific warning-only layer on top of the existing tool-loop detection path: normalize `read` path/`file_path` identities per run/session, detect repeated same-file and same-small-file-set rereads in one turn, suppress false positives after edit/write/test/broadened-context progress, emit a soft nudge plus diagnostic event, and cover it with focused tests and docs/config guidance.
