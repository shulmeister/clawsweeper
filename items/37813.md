## Summary

Keep #37813 open. Current main has partial fixes for the reported behavior, including configured fallback-chain handling, xAI/Grok forward-compatible model definitions, and fallback visibility in status output. The remaining security-relevant request is still not implemented: tool policy is resolved from the active fallback candidate, not from the originally requested model whose `tools.byProvider` deny list the reporter expects to preserve.

## Best Possible Solution

Keep #37813 open until maintainers either implement requested-model tool-policy preservation across fallback attempts or explicitly decide and document that fallback runs should use only the active fallback model's policy. A concrete fix should carry the selected/requested provider and model through the fallback attempt tool-policy path and add regression coverage showing `tools.byProvider[requestedProvider/requestedModel].deny` remains enforced when the run falls back to another provider/model.
