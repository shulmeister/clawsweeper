## Summary

Keep open. Current main has a useful timeout-recovery compaction mitigation for high-context LLM timeouts, but it does not implement the requested session-level circuit breaker: no configurable max consecutive timeout setting, no persisted per-session timeout counter/degraded state, and no breaker event/webhook path. The issue is recent, concrete, non-maintainer-authored, and has no protected-label override, so conservative cleanup should not close it.

## Evidence

- **Requested behavior:** Issue #45389 specifically asks to stop automatic retries after N consecutive LLM timeouts on the same session, mark the session degraded, and emit a system event/webhook notification. Ryce's comment reinforces that the requested breaker is intended to stop timeout spirals before workspace/session damage continues.

- **Current mitigation is per-run timeout compaction:** Current main detects a timed-out run with high prompt token usage, increments a local timeoutCompactionAttempts counter, runs contextEngine.compact(), and retries the prompt. This is helpful mitigation, but the counter is local to runEmbeddedPiAgent and capped at two compaction attempts, not a persisted session-level circuit breaker.
