## Summary

Keep open. Current main still appears to contain the reported live-session delete path: `current` can persist as `session:<live key>`, one-shot `at` jobs default `deleteAfterRun` to true, custom/session targets are passed into the isolated-agent runner as the actual session key, and direct-delivery cleanup still calls `sessions.delete` on `params.agentSessionKey` for any `deleteAfterRun` job. Related PR #67807 is real prior context, but it was scoped to cleaning up isolated cron direct-delivery sessions and does not add a guard for live `current`/`session:*` bindings.

## Best Possible Solution

Keep the issue open and patch core cron cleanup. The safest direction is to make direct-delivery session deletion apply only to cron-owned isolated run sessions, or to run one-shot `current`/`session:*` reminders in a separate cron-owned working session while preserving the live delivery target. Add regression coverage across normalization, gateway session-key selection, isolated-agent session resolution, and delivery cleanup for `current`/`session:<live-key>` so live DM bindings are never deleted or rebound by `deleteAfterRun`.
