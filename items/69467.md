## Summary

Keep open. Current main still has the reported hazardous combination: one-shot `at` jobs default to `deleteAfterRun: true`, `sessionTarget: "current"` is persisted as `session:<live session key>`, non-isolated `session:*` targets reuse or roll over the supplied live session key, and direct-delivery cleanup deletes `params.agentSessionKey` without checking whether that key is a cron-owned isolated session or a live user DM binding. Existing tests cover cleanup for isolated/direct cron sessions but do not cover the live `current`/`session:*` DM case reported in #69467.

## Best Possible Solution

Keep #69467 open and patch cron cleanup semantics. The safest product direction is to make `deleteAfterRun` destructive cleanup apply only to cron-owned isolated/ephemeral run sessions, and either ignore/disable cleanup for `current` and explicit `session:*` targets or execute those reminders in a separate cron-owned working session while preserving the live delivery route. Add a regression test that creates a non-default live channel session target via `current` or `session:<dm-key>`, runs an `at` agentTurn with `deleteAfterRun: true`, and asserts no `sessions.delete` is issued for the live DM key and the next live session binding is preserved.

## Risks / Open Questions

- Fixing this needs to preserve the #67718/#67807 behavior that `deleteAfterRun` reliably cleans up cron-owned isolated direct-delivery sessions, while preventing cleanup from deleting live or persistent user-bound session keys.
- A narrow fix in only normalization or only delivery cleanup may leave another creation path able to persist `deleteAfterRun: true` with a live `session:*` target.
