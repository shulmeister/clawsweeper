## Summary

Keep this issue open. Current main still has the reported semantic split: direct CLI sends enter the message action runner, plugin send actions can handle sends before the core delivery path, Telegram's plugin action sends directly, hook cancellation still collapses to an empty delivery result, and gateway send still turns that empty result into `No delivery result`. The related implementation PR #63596 is closed unmerged, and #38340 was a closed Discord-specific attempt rather than a shipped canonical outbound send contract.

## Best Possible Solution

Keep this issue open and implement a core canonical outbound send contract. The likely path is a shared send wrapper or delivery-layer discriminated result that preserves agent/session/request identity and returns either delivered or blocked with a denial reason, then make direct CLI `message.send`, gateway `send`, and plugin-owned send actions consume the same contract or an equivalent plugin API. Add regression coverage for no-grant blocked, granted delivery result, and revoked-grant blocked across direct and gateway entrypoints; #63596 is useful prior art but is not merged.
