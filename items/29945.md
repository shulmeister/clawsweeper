## Summary

Current main still has the reported behavior: unknown pairing-mode DM senders can receive a branded OpenClaw pairing reply, and there is no `unpairedResponse`/silent/code-only/branded config surface. The issue discussion is also still live: the linked maintainer PR #39065 directly tracks the requested configurable response modes but is open/unmerged in the provided GitHub context, and current main does not contain its implementation.

## Best Possible Solution

Keep #29945 open until a maintainer lands #39065 or an equivalent core/plugin-channel change that adds an explicit unpaired DM response policy, wires it through shared pairing helpers and channel call sites, and updates config schema, docs, and regression tests. The implementation should also resolve the product decision on whether the default becomes silent or remains backward-compatible branded.
