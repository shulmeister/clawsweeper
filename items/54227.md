## Summary

Keep this issue open. Current main still builds and registers Discord interactive component entries independently, and the interaction handler consumes only the clicked component id, so a sibling button in the same non-reusable component message can still dispatch after the first button is consumed. The broader related #54233 is closed and does not supersede this focused remaining bug.

## Best Possible Solution

Keep this issue open as the focused tracker. Fix the Discord plugin component registry and interaction handler by assigning a shared message or panel consumption group when component entries are registered, then invalidating all sibling entries in that group when any non-reusable entry is consumed before callback dispatch. Add a regression test with two buttons in one non-reusable component message proving the second sibling click returns the expired response and does not dispatch.
