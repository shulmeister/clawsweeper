## Summary

Keep #63208 open. Current `main` still scans only the first level under each memory-wiki page group for compile and query/search paths, and there is no `ingest.recursive` config option covering the requested behavior.

## Best Possible Solution

Keep the issue open and implement a shared recursive markdown-page walker for memory-wiki vault page groups, then use it from both compile and query paths. Add focused tests with nested `sources/<topic>/AGENTS.md` covering compile counts, root/group indexes, agent digest paths, `wiki search`/`wiki get`, and lookup behavior. Decide explicitly whether frontmatter-less nested pages should get path-derived IDs, while preserving existing explicit IDs and flat-page compatibility.

## Evidence

- **Issue request:** #63208 asks `openclaw wiki compile` to recognize markdown under nested vault paths such as `sources/OpenClaw配置文件指南/AGENTS.md`, preserving directory organization for existing note vaults. The issue has no comments and no maintainer/protected labels in the provided GitHub context.
- **Compile scanner is non-recursive:** `collectMarkdownFiles` reads only `fs.readdir(dirPath)` for a page-group directory, filters `entry.isFile() && entry.name.endsWith(".md")`, and maps only direct child filenames. ## Risks / Open Questions

- Changing only `compile.ts` would leave `wiki search`, `wiki get`, memory palace, import insights, and apply lookups inconsistent because they use the separate non-recursive query scanner.
- Recursive support needs deterministic ordering and clear `index.md` handling for nested directories so prompt/digest output remains stable.
- If implicit page IDs are added for frontmatter-less nested pages, maintainers should preserve explicit frontmatter IDs and avoid breaking existing flat-page IDs.
