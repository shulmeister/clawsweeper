## Summary

Keep open. Current main has partial NO_REPLY hardening: auto-reply normalization strips trailing NO_REPLY for some channel delivery paths, and the system prompt now warns not to append it. But the gateway/webchat surfaces still only suppress exact control-token text or leading fragments, and outbound display normalization tests still preserve mixed text ending in NO_REPLY. The issue's gateway-level suffix stripping request is therefore not fully implemented on main.

## Best Possible Solution

Keep the issue open and implement a shared, conservative silent-token cleanup contract for user-visible gateway/chat delivery: exact token suppresses, leading malformed control-token output is suppressed or stripped according to existing lead-fragment rules, and a trailing standalone `NO_REPLY` suffix is stripped from otherwise substantive text. Apply it to `src/gateway/server-chat.ts` final/delta buffering and gateway display payload normalization, then add regression tests for `Task complete. NO_REPLY`, `Git is clean.\nNO_REPLY`, exact `NO_REPLY`, leading malformed `NO_REPLY...`, and legitimate text that should remain visible. Update the exact-token docs once the behavior is intentionally broadened.
