## Summary

Keep open. Current main still fixes the file/path Discord upload side but not the buffer-backed `message.send` case reported here: the public send schema advertises `buffer`/`filename`/`contentType`/`mimeType`, while the shared send runner and Discord action bridge only collect URL/path media and forward `mediaUrl` plus `filename`. Related #41779 and #44555 support that the remaining issue is the shared buffer-send contract, not Discord path uploads generally.

## Best Possible Solution

Keep this issue open and narrow it to direct buffer attachments on Discord `message.send`. The best fix is either to hydrate and deliver `buffer` plus `filename` plus `contentType`/`mimeType` end to end for ordinary send, with Discord regression coverage proving a buffer-backed image reaches the multipart `files` upload path, or to remove/fail-fast those fields from ordinary send if buffers are intentionally unsupported. Related #41779 and #44555 should be considered when deciding whether this belongs in shared send hydration or channel-specific validation.
