## Summary

Keep open. Current main still does not include this PR's semantic-cache store or local-fallback runner wrapper. Main already supports ordinary configured local/hosted model fallbacks, so part of the local fallback use case is covered, but the semantic response cache and the proposed new config surfaces are not shipped. The PR also has unresolved review-blocking correctness comments, so no conservative auto-close reason applies.

## Best Possible Solution

Keep this PR open for maintainer product and architecture judgment. If pursued, split local fallback from semantic cache, reuse the existing model-fallback chain where possible, design response caching against the current memory-search/active-memory architecture, and address the P1 review comments with focused regression tests and public docs before merge.

- Semantic response caching can produce stale or incorrect answers if thresholds, TTLs, storage failures, and observability are not designed alongside the existing memory-search stack.
