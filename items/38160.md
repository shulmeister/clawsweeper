## Summary

Keep PR #38160 open. Current main covers much of the local-fallback user goal through generic `agents.defaults.model.fallbacks`, including hosted-to-local fallback, but it does not implement the PR's dedicated `localModelFallback` config or semantic prompt-to-response cache. The PR also has unresolved correctness review findings, so there is no conservative close reason.

## Best Possible Solution

Keep this PR open for maintainer architecture review, but split it before merge. Prefer documenting or extending the existing `agents.defaults.model.fallbacks` path for hosted-to-local fallback unless a concrete missing API remains. Redesign the semantic response cache as a separate proposal with fail-open behavior, correct vector persistence, provider/base URL/auth handling, cache scope/invalidation/privacy rules, telemetry, docs, and focused tests.

## Risks / Open Questions

- The PR is not merge-ready as-is because cache lookup/storage must fail open, persisted vector decoding needs correction, fallback connection settings must be honored, and telemetry should preserve failed cloud attempts.
- A response-level semantic cache needs explicit cache scope, invalidation, privacy, persistence, and observability design before it is safe to ship in core.
- The PR bundles unrelated work, including an onboarding finalize change and root documentation artifact, which should be split before maintainer review.
