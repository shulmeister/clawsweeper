## Summary

Keep PR #67077 open. Current main at f4e6322649a1ef0a9b47bd9d8b681c143850d669 still has the fatal auth-profile bookkeeping save path that the PR targets: lock-save failures return null, then the mark* fallback paths call saveAuthProfileStore directly without a nonfatal catch. The linked issue #62099 remains a concrete bug report, and the PR discussion shows the contributor addressed the earlier test weakness by covering the direct-save EPERM fallback path.

## Best Possible Solution

Keep PR #67077 open and update or land the narrow fix after maintainer review: make only post-success markAuthProfileGood, markAuthProfileUsed, and markAuthProfileFailure persistence failures nonfatal with sanitized logging, keep credential-critical saves fatal, and keep regression coverage for updateAuthProfileStoreWithLock returning null followed by saveAuthProfileStore throwing EPERM. Close #62099 when that behavior ships.
