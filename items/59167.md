## Summary

Keep open. Current `main` still intentionally rejects `gateway.auth.mode: "trusted-proxy"` when the proxy connects from loopback, the tests lock that behavior in, and the docs still prescribe the same workaround the commenters objected to. There is no evidence in `main` that issue #59167 has been fixed or made obsolete.

## Evidence

- **Auth code still hard-rejects loopback trusted proxies:** `authorizeTrustedProxy` first verifies the source is in `gateway.trustedProxies`, then immediately returns `trusted_proxy_loopback_source` if the remote address is loopback. A localhost proxy therefore cannot succeed in trusted-proxy mode on current `main`.

- **Tests codify the reported failure as expected behavior:** Gateway auth tests explicitly assert that loopback trusted-proxy requests are rejected with `trusted_proxy_loopback_source`, including the exact case of identity headers arriving from `127.0.0.1`.
  - file: [src/gateway/auth.test.ts:959](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/auth.test.ts#L959)
