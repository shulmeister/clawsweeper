## Summary

Keep PR #68112 open. Current main has related cron restart hardening for interrupted `runningAtMs` jobs, but it does not implement the PR's central startup-catch-up failure guard: `start()` still awaits `runMissedJobs()` before arming the timer, and `runMissedJobs()` can still throw from startup planning/outcome persistence before `armTimer(state)` is reached.

## Best Possible Solution

Keep the PR open for maintainer review. The best product/code path is to either land a carefully reviewed version of the `start()` resilience fix in `src/cron/service/ops.ts` with tests that force `runMissedJobs()` persistence/load failures before `armTimer()`, or ask the author to rebase and narrow the patch if maintainers want only the startup catch-up guard without the broader runtime zombie cleanup.
