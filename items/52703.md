## Summary

The request is for an optional subagent supervision/orchestration policy, and current main already exposes plugin/ClawHub primitives to implement it outside OpenClaw core.

## Close Comment

Closing as `clawhub` after Codex review against current `main` (`8866544ffefb42ca234c66c337134fb60743d8d0`). This proposal is a useful supervision policy, but it fits the plugin/ClawHub orchestration layer rather than OpenClaw core. Current main already exposes the needed primitives through `api.runtime.subagent` for run/wait/read/delete control (`docs/plugins/sdk-runtime.md:87`) and `api.runtime.taskFlow` for durable orchestration (`docs/plugins/sdk-runtime.md:143`, `docs/automation/taskflow.md:10`). Core also already ships the baseline guardrails for subagents such as `runTimeoutSeconds`, `maxConcurrent`, `maxChildrenPerAgent`, and auto-archive (`docs/tools/subagents.md:86`), plus detached-work auditing via `openclaw tasks audit` (`docs/automation/tasks.md:189`). Per `VISION.md:57` and `VISION.md:101`, optional capability and heavier orchestration layers that can live outside core should go to plugins/ClawHub first. If someone wants to pursue this, a ClawHub plugin/skill is the right path.
