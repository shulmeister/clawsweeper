## Summary

Keep #39120 open. Current main still implements `bootstrap-extra-files` as a recognized-bootstrap-basename loader, not arbitrary workspace context injection: arbitrary files like `hooks/REGISTRY.yaml` are rejected with `invalid-bootstrap-filename`, docs describe the restriction, and tests still assert that a glob skips non-bootstrap basenames. The linked implementation PR #39174 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open for maintainer/product decision. The direct implementation path is to relax or replace the `VALID_BOOTSTRAP_NAMES` check in `loadExtraBootstrapFilesWithDiagnostics()`, widen the `WorkspaceBootstrapFile` name contract as needed, preserve existing realpath and size-budget guards, update subagent/light-context filtering intentionally, and revise tests/docs for arbitrary workspace file injection. If maintainers prefer this outside core, close only after explicitly documenting a supported hook/plugin pattern or ClawHub package that provides equivalent deterministic bootstrap injection.
