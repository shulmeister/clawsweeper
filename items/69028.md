## Summary

Keep open. Current main still matches the reported failure mode: Feishu inbound media download/save errors are caught and logged inside the media resolver, the resolver returns an empty media list, and message dispatch continues to the agent without any user-facing Feishu reply about the rejected file.

## Evidence

- **Inbound media errors are swallowed:** resolveFeishuMediaList catches download/save failures for normal media and only calls log with `feishu: failed to download ${messageType} media`; it then returns the accumulated media list, which can be empty. There is no error result or notification path for mediaMaxMb failures.
  - file: [extensions/feishu/src/bot-content.ts:429](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/extensions/feishu/src/bot-content.ts#L429)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '280,620p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Dispatch continues after empty media payload:** handleFeishuMessage resolves media with maxBytes, builds a media payload from the returned list, spreads it into the inbound context, and then dispatches to the agent. There is no branch that sends a Feishu message when media resolution fails or returns empty after an attempted media download.
