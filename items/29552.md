## Summary

Keep #29552 open. Current main still has an unresolved local gateway pairing failure path: ordinary local gateway calls can select a stale `OPENCLAW_GATEWAY_TOKEN` ahead of `gateway.auth.token`, send it once, and fail on `AUTH_TOKEN_MISMATCH` without a config-auth fallback. The directly related fallback PR #29676 is closed and unmerged, and the remaining behavior is still codified by current tests/docs.

## Best Possible Solution

Keep this open and narrow the implementation to the remaining local pairing failure: after an `AUTH_TOKEN_MISMATCH` from a local, non-URL-override call with no explicit token/password, retry once with config-first `gateway.auth.token`/`gateway.auth.password`. Add regression coverage in `src/gateway/call.test.ts` for stale `OPENCLAW_GATEWAY_TOKEN`. If env-first local precedence is intentional, document that clearly and split version/update drift into a separate canonical issue.
