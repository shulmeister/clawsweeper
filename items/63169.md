## Summary

Keep open. Current main has fixes for the earlier media-routing/drop-before-adapter class, but this issue reports live WhatsApp false success: OpenClaw logs/returns success with a message id while recipients do not receive attachments. The current code still treats the immediate WhatsApp/Baileys send result as success, and I found no code, docs, tests, or release evidence proving recipient-side media delivery is verified or that this report is obsolete.

## Best Possible Solution

Keep this issue open as a live WhatsApp media false-success tracker unless maintainers explicitly consolidate it with #63816. The best fix path is to either add a reliable WhatsApp media-delivery confirmation/diagnostic path or change success semantics so OpenClaw does not report a definitive delivered attachment when it only has the immediate send result; then cover the CLI/message tool media path with focused tests plus live WhatsApp verification for image, document, video, DM, and group cases.
