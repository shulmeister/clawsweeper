## Summary

Keep open. Current `main` still appears to route Telegram DM-topic follow-up messages onto a `:thread:<chat>:<topic>` session key, but the generic inbound session recreation path only writes origin/delivery metadata, and ACP session lookup still requires ACP metadata on that exact row. I did not find a current-main regression test that covers an ACP-bound Telegram DM topic surviving the next inbound turn with ACP metadata/sessionFile intact.

## Evidence

- **DM-topic follow-ups still switch to a thread-suffixed session key:** Telegram inbound context resolution still appends `:thread:<chatId>:<dmThreadId>` for DM topics after route resolution, and runtime conversation binding still resolves DM-topic bindings via `<chatId>:topic:<replyThreadId>`. That means a bound ACP DM topic can still land on a derived thread row instead of the base ACP session row.
