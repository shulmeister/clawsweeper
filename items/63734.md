## Summary

Keep open. Current main has exec policy modes, explicit allowlists, narrow stdin-only safeBins, and plugin hooks that can add optional policy, but it does not implement the requested built-in L1-L4 command risk classifier with broad read-only auto-approval and dangerous-command auto-deny.

## Best Possible Solution

Keep this issue open for maintainer decision. If pursued, implement it as an explicit opt-in exec policy layer or a carefully scoped extension of safeBins/allowlists, with threat-model docs, config/UI affordances, and regression tests for POSIX and Windows shell behavior. If maintainers prefer plugin-first heuristic command policy, close only after documenting the supported ClawHub/plugin path and confirming the plugin API can safely cover the requested behavior.

## Evidence

- **Requested behavior remains broader than current core behavior:** The issue asks for a built-in classifyRiskLevel(commandText) with L1 auto-approval for commands such as ls, cat, git status, grep, sed, and awk, plus L4 auto-deny for destructive patterns such as rm -rf /, fork bombs, mkfs, dd, and curl|sh. The provided GitHub context shows no comments, protected labels, or maintainer-authored signal changing that scope.
- **Exec approvals expose policy modes, not risk levels:** Current core defines ExecSecurity as deny/allowlist/full and ExecAsk as off/on-miss/always. There is no L1-L4 risk enum or built-in command-text risk classifier in the exec approval contract.
