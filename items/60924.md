## Summary

Current main already guards sessions_send reuse against replaying a prior assistant reply: it snapshots the target session before sending, waits on the specific run, returns only changed assistant output, and skips announce flow when no current reply is available.

## Evidence

- **sessions_send captures a pre-run baseline and returns neutral timeout/error results:** The tool snapshots the latest assistant reply before starting the nested run, passes that baseline into the run wait helper, and on timeout/error returns status without reading or surfacing old assistant text.
  - file: [src/agents/tools/sessions-send-tool.ts:257](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/agents/tools/sessions-send-tool.ts#L257)

- **Run wait filters unchanged previous replies:** waitForAgentRunAndReadUpdatedAssistantReply reads chat history only after the run reports ok and returns replyText only when the latest assistant message fingerprint differs from the baseline.
  - file: [src/agents/run-wait.ts:174](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/agents/run-wait.ts#L174)
