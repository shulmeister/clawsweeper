## Summary

Keep open: current `main` still carries exec-approval followups by `sessionKey` only, while `/new` and `/reset` rebind that same key to a fresh `sessionId`. I did not find an `expectedSessionId`-style guard, and existing tests cover sessionKey-based continuation rather than stale-followup dropping.

## Evidence

- **Followup payload has no sessionId:** `sendExecApprovalFollowup` only accepts `sessionKey`, builds the `agent` call with `sessionKey`, and never includes an original `sessionId` for rebinding protection.
  - file: [src/agents/bash-tools.exec-approval-followup.ts:17](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/bash-tools.exec-approval-followup.ts#L17)
  - command: `nl -ba src/agents/bash-tools.exec-approval-followup.ts | sed -n '15,235p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Shared followup target persists only sessionKey:** `ExecApprovalFollowupTarget` stores `approvalId` plus `sessionKey`/turn-source fields only; `sendExecApprovalFollowupResult` forwards only those fields, so the original session instance is not preserved alongside the pending approval.
  - file: [src/agents/bash-tools.exec-host-shared.ts:84](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/bash-tools.exec-host-shared.ts#L84)
  - command: `nl -ba src/agents/bash-tools.exec-host-shared.ts | sed -n '80,120p;315,330p;400,430p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Gateway agent reset rebinds the key to a fresh sessionId:** The `agent` handler handles `/new` and `/reset` by updating `requestedSessionKey` and `resolvedSessionId`, then later reloads the current entry for that key and uses `entry?.sessionId ?? randomUUID()`. There is no check that an incoming followup still matches the original session instance for that key.
  - file: [src/gateway/server-methods/agent.ts:503](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server-methods/agent.ts#L503)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '500,690p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Approval request schema/handler track sessionKey, not sessionId:** The exec-approval request schema and handler accept/store `sessionKey` but no `sessionId`, so the pending approval record itself lacks the original-session binding the issue proposes.
  - file: [src/gateway/protocol/schema/exec-approvals.ts:96](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/protocol/schema/exec-approvals.ts#L96)

- **Tests assert sessionKey-based continuation, not stale-drop behavior:** Current tests verify that delayed approval followups continue the same session by `sessionKey` and external route, but I found no coverage for dropping a followup after `/new` or `/reset` rebinding.
