## Summary

Current main already records the actual model/provider used for each cron run in `cron.runs`, so the issue’s core claim that there is no way to know which model executed the job is obsolete. The implementation does not add `lastModel` to `cron.state`, but it does expose the runtime-used model through run-history telemetry.

## Evidence

- **Actual runtime model is derived from final agent metadata or fallback selection:** `finalizeCronRun` computes `modelUsed`/`providerUsed` from `finalRunResult.meta.agentMeta` and falls back to `execution.fallbackModel` / `execution.fallbackProvider`, then includes them in cron telemetry.
  - file: [src/cron/isolated-agent/run.ts:724](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/src/cron/isolated-agent/run.ts#L724)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '716,805p'`
- **Cron run history persists model/provider telemetry:** Gateway cron event handling appends `model`, `provider`, and `usage` into the JSONL run log for finished cron runs.
  - file: [src/gateway/server-cron.ts:549](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/src/gateway/server-cron.ts#L549)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '545,573p'`
- **`cron.runs` API exposes model/provider fields:** The RPC schema for cron run-log entries includes optional `model` and `provider` fields, so callers can read them back via `cron.runs`.
  - file: [src/gateway/protocol/schema/cron.ts:345](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/src/gateway/protocol/schema/cron.ts#L345)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '345,375p'`

  - file: [docs/cli/cron.md:146](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/docs/cli/cron.md#L146)

## Risks / Open Questions

- `cron.state` and `cron list` still do not surface the last actual model directly; users must inspect `openclaw cron runs` instead.
- If maintainers still want a state-level shortcut for alerting or `cron list`, that would be a separate enhancement from the original reproducibility claim.

## Close Comment

Closing from a Codex review because current `main` already provides a supported way to see which model actually executed a cron run. `src/cron/isolated-agent/run.ts:724` derives the runtime-used `model`/`provider` from final agent metadata and fallback selection, `src/gateway/server-cron.ts:549` persists those fields into the cron run log, and `src/gateway/protocol/schema/cron.ts:345` exposes them through `cron.runs`. The docs now point users to `openclaw cron runs --id <job-id>` for this history in `docs/cli/cron.md:146` and `docs/automation/cron-jobs.md:335`. So the original report’s “there’s no way to know which model actually executed the job” no longer matches current `main`. If you still want that surfaced specifically on `cron.state` / `cron list`, please open a narrower follow-up enhancement request.
