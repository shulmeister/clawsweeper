## Summary

Keep open. Current `main` still applies the unconditional daily default in `resolveSessionResetPolicy`, and the existing regression test still expects group sessions with only `resetByType.dm` configured to resolve to `daily`, so the requested behavior is not implemented on `main`. The PR also has a credible logic flaw in the provided review context for the same dm-only case.

## Evidence

- **Current main still defaults implicitly to daily:** `resolveSessionResetPolicy` computes `mode` from `typeReset`, `baseReset`, then falls back to `(!hasExplicitReset && legacyIdleMinutes != null ? "idle" : DEFAULT_RESET_MODE)`. There is no group/thread-specific implied idle default on current `main`.

- **Current test suite encodes the old behavior:** The regression test named `does not use dm fallback for group/thread types` still asserts `expect(groupPolicy.mode).toBe("daily")` for a config containing only `resetByType.dm`, which matches the unresolved bug report rather than the PR's intended behavior.
