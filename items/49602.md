## Summary

Current main already handles the reported same-PID orphan session-lock case. `acquireSessionWriteLock` treats a `.lock` owned by `process.pid` with no valid `starttime` and no matching in-memory held lock as `orphan-self-pid`, removes it, and retries acquisition instead of timing out. The exact scenario is covered by a regression test, and the same code is present in release `v2026.4.22`.

## Evidence

## Close Comment

Closing from a Codex review: current `main` already handles this case. In `src/agents/session-write-lock.ts:403`, OpenClaw treats a lock file owned by the current PID with no valid `starttime` and no matching in-memory held lock as an `orphan-self-pid`; in the `EEXIST` path at `src/agents/session-write-lock.ts:555`, it removes that lock and retries acquisition instead of timing out. The exact scenario is covered by `src/agents/session-write-lock.test.ts:429` (`reclaims orphan lock files without starttime when PID matches current process`). This logic is also present in released tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so there is no remaining maintainer action to track here.
