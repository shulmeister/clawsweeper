## Summary

Keep open. Current main has image-aware MIME sniffing around `read` and a separate built-in `pdf` tool, but the core `read` path still does not implement generic binary-document preprocessing/extraction for `.docx`, `.xlsx`, `.pptx`, or PDFs as requested.

## Evidence

- **`read` post-processing is image-only:** The OpenClaw wrapper around `read` only rewrites/sanitizes image results. `normalizeReadImageResult` returns early unless the tool result already contains an `image` block, so there is no parallel path here for PDF/OOXML document extraction or binary-file placeholdering.
  - file: [src/agents/pi-tools.read.ts:292](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-tools.read.ts#L292)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Sandbox MIME detection only feeds image handling:** The sandbox read operations expose `detectImageMimeType`, and they explicitly discard non-`image/*` MIME types. That supports image classification, not general binary-document preprocessing.
  - file: [src/agents/pi-tools.read.ts:702](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-tools.read.ts#L702)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current read-tool coverage is image + plain text only:** The main read-tool test exercises a PNG and a `.txt` file. There is no current OpenClaw test proving `read` extracts or safely summarizes `.pdf`, `.docx`, `.xlsx`, or `.pptx` content before it reaches the model.
  - file: [src/agents/pi-tools.create-openclaw-coding-tools.test.ts:468](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-tools.create-openclaw-coding-tools.test.ts#L468)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **PDF handling exists as a separate tool, not as `read` preprocessing:** OpenClaw registers a distinct `pdf` tool alongside the normal tool set. That is an alternative surface for PDFs, but it does not satisfy the issue's request to make the built-in `read` tool detect and preprocess binary documents.
  - file: [src/agents/openclaw-tools.ts:179](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/openclaw-tools.ts#L179)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Document extraction support is PDF-only today:** The bundled document extractor plugin advertises only `application/pdf`, so there is still no built-in extractor path for `.docx`, `.xlsx`, or `.pptx` on current main.
  - file: [extensions/document-extract/document-extractor.ts:208](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/document-extract/document-extractor.ts#L208)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Related work was linked, but not landed here:** The issue timeline cross-references zeroclaw-labs/zeroclaw PR #5094 (`feat(tools): binary file type detection in file_read`), which is directly relevant, but the equivalent behavior is not evident in the current OpenClaw main checkout.

## Risks / Open Questions

- The vendored checkout does not include the upstream `@mariozechner/pi-coding-agent` source, so I could not inspect `createReadTool` internals directly; if upstream added undocumented non-image binary suppression, that is not reflected in OpenClaw's current tests/docs.
- There is adjacent PDF/attachment work on main, so some user scenarios may now be better served by the separate `pdf` tool, but that is still narrower than the issue's requested `read`-tool behavior.
