## Summary

Keep #10137 open. Current main partially covers the reported Responses compatibility problem by always sending `input` as an array and `stream: true`, and it has related prompt-cache/reasoning/store payload policy, but the requested stable compat layer is still missing top-level non-empty `instructions` for non-Codex `openai-responses` providers and an option to omit `max_output_tokens`. #10140 is the closed duplicate, so #10137 remains the canonical tracker.

## Best Possible Solution

Keep #10137 open as the canonical tracker. A solid fix would add documented, stable model/provider compat options for Responses payload normalization, likely in `ModelCompatConfig`/`ModelCompatSchema` and `OpenAIResponsesPayloadPolicy`, covering top-level fallback `instructions` and optional `max_output_tokens` omission with focused tests for proxy-like `api: "openai-responses"` providers. #10140 should remain treated as the closed duplicate copy.
