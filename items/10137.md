## Summary

Keep #10137 open. Current main still only partially covers the request: Responses payloads already use an `input` array and hardcoded `stream: true`, but generic `api: "openai-responses"` providers still lack stable compatibility controls for fallback top-level `instructions` and omitting `max_output_tokens`. The provided discussion also treats #10140 as the closed duplicate and #10137 as the canonical tracker.

## Best Possible Solution

Keep #10137 open as the canonical tracker and leave #10140 closed as the duplicate. Add opt-in Responses proxy compatibility controls in the transport/payload policy and stable model/provider config, especially generic fallback top-level `instructions` and optional omission of `max_output_tokens`, with focused regression tests and docs for custom `api: "openai-responses"` providers.
