## Summary

Keep #10137 open. Current main partially covers the reported Responses compatibility issue by sending `input` as an array and `stream: true`, and Codex-specific Responses requests now use top-level `instructions`, but generic `api: "openai-responses"` providers still lack a stable compat option for non-empty top-level `instructions` or omitting `max_output_tokens`. #10140 is the closed duplicate, so #10137 remains the canonical tracker.

## Best Possible Solution

Keep #10137 open as the canonical tracker. A solid fix would add documented stable provider/model compat options for Responses payload normalization, likely around `ModelCompatConfig`/`ModelCompatSchema` and `OpenAIResponsesPayloadPolicy`, covering generic top-level fallback `instructions` and optional `max_output_tokens` omission with focused tests for proxy-like `api: "openai-responses"` providers. Treat #10140 as the closed duplicate copy.
