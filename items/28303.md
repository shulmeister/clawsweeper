## Summary

Keep open. Current main at a35d259719cc7aecd1854fdc830ad6390b15f02a still uses `zh-CN` and `zh-TW` as Control UI locale identifiers across the public `Locale` type, lazy registry, persisted browser setting, selector values, generated-locale tooling, docs, workflow matrix, and tests. The docs publish pipeline has some `zh-Hans` Mintlify metadata, but it still maps to `zh-CN` docs paths and does not implement the requested Control UI runtime rename or migration. Related PR #42966 was closed unmerged and docs-scoped; open PR #63621 is a zh-TW glossary improvement, not a canonical implementation of the locale-code rename.

## Best Possible Solution

Keep the issue open for a focused Control UI i18n change: rename the runtime locale contract to `zh-Hans`/`zh-Hant`, update lazy imports and generated-locale metadata, add a saved-setting migration from `zh-CN`/`zh-TW`, adjust navigator fallback and selector language keys, update docs/workflow matrices, and revise tests that currently assert the old IDs. If maintainers decide to retain region-based IDs for compatibility, document that decision on this issue instead of closing it as already implemented.
