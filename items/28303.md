## Summary

Keep #28303 open. Current main still uses `zh-CN` and `zh-TW` as Control UI runtime locale IDs, lazy import keys, generated locale metadata, saved browser-storage values, selector option values, docs, workflow matrix, and tests. The related PRs in the issue context are closed and unmerged, and they do not ship the requested runtime rename or saved-locale migration.

## Best Possible Solution

Keep #28303 open for a focused Control UI i18n contract change: rename runtime locale IDs and generated metadata to `zh-Hans`/`zh-Hant`, update lazy imports, selector language keys, docs, workflow, and tests, and add backwards-compatible aliases or saved-setting migration from `zh-CN`/`zh-TW` for both global and scoped browser settings. If maintainers intentionally want region-coded runtime IDs as a stable compatibility contract, document that product/API decision on the issue before closing.

## Risks / Open Questions

- Renaming locale IDs is compatibility-sensitive because values are stored in both `openclaw.i18n.locale` and scoped `openclaw.control.settings.v1:*` browser settings.
- Docs publishing already has some `zh-Hans` metadata, while the Control UI runtime still uses `zh-CN`/`zh-TW`; a fix should keep those contracts distinct.
