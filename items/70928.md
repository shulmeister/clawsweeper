## Summary

Current main still leaves the reported outbound hook gap open for webchat/chat.send and local TUI internal reply paths. The shared outbound delivery helper fires `message_sending`/`message_sent`, but these default local surfaces bypass it; webchat also builds its own dispatcher without the helper that injects `message_sending`. `agent_end` is now wired in the agent runners, but that does not solve the central `message_sent`/`message_sending` and transcript-write coverage problem reported here. The related PR #12584 is closed unmerged in the provided GitHub context, and the open umbrella #50126 does not make the webchat/TUI-specific acceptance criteria obsolete.

## Best Possible Solution

Keep #70928 open. The maintainable fix is to route webchat/chat.send and local TUI session-only replies through a shared outbound lifecycle observer, or explicitly invoke the same `message_sending`, `message_sent`, and transcript-write hook contracts around those paths. Add focused regression tests for webchat `chat.send` and local TUI/internal delivery, then update the plugin hook docs with either guaranteed coverage or a temporary channel matrix. Link #50126 as the architectural umbrella, but preserve this issue as the webchat/TUI acceptance target until those paths are covered.
