## Summary

Keep open. Current `main` has newer heartbeat routing controls (`agents.defaults.heartbeat.target` and optional `to`), but it does not implement the requested per-recipient channel heartbeat routing model. `channels.<channel>.heartbeat` is still visibility-only, and WhatsApp fallback routing still fans out to multiple authorized recipients when no explicit `to` is set. There is also no built-in split where `HEARTBEAT_OK` goes only to an admin while alert content continues routing to the relevant end user.

## Evidence

- **Channel heartbeat config is visibility-only:** `ChannelHeartbeatVisibilityConfig` only defines `showOk`, `showAlerts`, and `useIndicator`; there is no `target`/`targets` recipient field under channel heartbeat config.
  - file: [src/config/types.channels.ts:3](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/config/types.channels.ts#L3)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Resolved heartbeat visibility has no recipient routing:** Heartbeat visibility resolution reads per-channel and per-account heartbeat config, but only for the three visibility booleans. It does not resolve recipient-specific routing.
  - file: [src/infra/heartbeat-visibility.ts:42](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/infra/heartbeat-visibility.ts#L42)

  - file: [docs/gateway/heartbeat.md:105](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/gateway/heartbeat.md#L105)

## Risks / Open Questions

- There is partial overlap with the request because current main supports a single explicit heartbeat destination via `agents.defaults.heartbeat.to`, so a maintainer might be tempted to close this as implemented. That would overstate the current behavior.
- The missing gap is specifically per-recipient routing semantics at the channel layer, including the requested split where OK/noise goes to admins while alerts can still follow the relevant user/session path.
