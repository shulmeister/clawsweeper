## Summary

Current main already covers the core ACP child-to-parent completion plumbing and resume/progress primitives. What remains in #53264 is higher-level code-review orchestration policy, which OpenClaw's vision directs toward external plugins/skills rather than new core behavior.

## Risks / Open Questions

- There is no built-in `onComplete`/`postComplete` review policy in current core, so ClawHub/plugin authors still need to compose the review loop themselves on top of the existing ACP completion and resume primitives.

## Close Comment

Closing this as a `clawhub` item after a Codex review of current `main` (8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf). The core ACP gap described here is no longer the right place to add this: current `main` already has parent-owned ACP child completion routed back through the internal task-completion path (`docs/tools/acp-agents.md:338`, `src/agents/internal-events.ts:12`, `src/agents/tools/sessions-send-tool.ts:294`), plus ACP orchestration primitives like `resumeSessionId` and `streamTo:"parent"` (`src/agents/tools/sessions-spawn-tool.ts:100`, `docs/tools/acp-agents.md:318`). What remains is higher-level workflow policy: automatically reviewing code output and deciding whether to resend feedback/retry. Per `VISION.md:54-70`, optional capabilities and new skills should live as plugins/ClawHub skills rather than new OpenClaw core behavior. If someone wants to ship this, the right path is a ClawHub skill/plugin built on the existing ACP completion + resume/task primitives.
