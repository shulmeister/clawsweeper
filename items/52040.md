## Summary

Current main already fixes this in the Control UI chat model picker by using provider-qualified model values end-to-end, and the same implementation is present in release v2026.4.22.

## Evidence

- **Chat picker submits the selected option value unchanged:** The chat-header model select renders options from `resolveChatModelSelectState(...)` and on change passes the selected `.value.trim()` straight into `switchChatModel(...)`, which then calls `sessions.patch` with `model: nextModel`.
  - file: [ui/src/ui/chat/session-controls.ts:84](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/ui/src/ui/chat/session-controls.ts#L84)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **UI model values are provider-qualified:** `buildQualifiedChatModelValue(...)` prefixes `provider/model` unless the value is already qualified, which is the exact behavior needed to avoid sending bare model ids from the picker.
  - file: [ui/src/ui/chat-model-ref.ts:5](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/ui/src/ui/chat-model-ref.ts#L5)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **State logic removes bare-id duplicates and preserves qualified refs:** `resolvePreferredServerChatModelValue(...)` normalizes plain ids through the catalog when possible, preserves already-qualified refs, and explicitly avoids bogus double-prefix results from stale providers.
  - file: [ui/src/ui/chat-model-ref.ts:106](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/ui/src/ui/chat-model-ref.ts#L106)

## Close Comment

Closing from a Codex review because current `main` already implements the relevant fix. The Control UI chat model picker now uses provider-qualified model values end-to-end: `ui/src/ui/chat-model-ref.ts:5` builds `provider/model` refs, `ui/src/ui/chat/session-controls.ts:84` renders those option values and sends the selected value straight to `sessions.patch`, and `ui/src/ui/chat/session-controls.test.ts:139` asserts the picker patches `model: "openai/gpt-5-mini"` rather than a bare id. The backend still resolves bare ids against the current/default provider in `src/gateway/sessions-patch.ts:411`, so qualifying the UI payload is the correct fix. The same implementation is already present in release `v2026.4.22` (best proving SHA `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with the exact provider/model pair and the current Control UI path used.
