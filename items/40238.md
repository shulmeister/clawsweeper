## Summary

Keep PR #40238 open. Current main still uses the old inline skill tool-dispatch path: `handleInlineActions()` builds tools with `createOpenClawTools()` and then applies only owner-only filtering, while normal coding sessions use `createOpenClawCodingTools()` and its broader profile/provider/group/subagent policy pipeline. The PR remains a focused, plausible security/policy fix, and the PR discussion shows review feedback was addressed rather than abandoned.

## Best Possible Solution

Keep the PR open for maintainer review. The best implementation direction is the PR's current shape or an equivalent main-branch patch: inline `command-dispatch: tool` should build tools through `createOpenClawCodingTools()` with the same route, sender, group, session, model, and config inputs used by normal command/coding paths; remove the standalone owner-only fallback filter; and keep regression tests that prove non-owner/profile-disallowed tools are unavailable while message route context is preserved.
