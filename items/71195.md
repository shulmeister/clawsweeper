## Summary

Keep this issue open. Current main already has the gateway/browser OpenAI Realtime Talk session path, but the native macOS Talk client still uses the legacy SFSpeechRecognizer -> chat.send -> chat.history -> talk.speak loop and has no runtime wrapper or WebRTC/audio session integration for `talk.realtime.session`. The issue discussion also includes explicit maintainer direction to keep this open and narrow the remaining work to native macOS consuming the existing realtime Talk flow.

## Best Possible Solution

Keep this issue open as the canonical native macOS Talk realtime follow-up. The likely implementation is to add a macOS `GatewayConnection` wrapper for `talk.realtime.session`, build a native realtime audio/WebRTC session modeled on `ui/src/ui/chat/realtime-talk.ts`, wire it into `TalkModeRuntime` and overlay state/cancellation, and preserve the existing SFSpeechRecognizer -> chat.send -> talk.speak path as fallback.
