## Summary

Keep open. Current main still serializes compaction and normal message handling through the same per-session/global run lanes. Preflight compaction is awaited before the reply run continues, in-run timeout/overflow compaction is also awaited before retry, and active-run messages default to queued followups/collect rather than being processed concurrently. The Telegram plugin routes inbound messages through this shared reply pipeline, and I found no background/offloaded compaction path or `blockProcessing`-style config on current main.

## Best Possible Solution

Keep the issue open for maintainer product/design work. A real fix needs an explicit concurrency boundary for compaction, such as snapshot-based background compaction, a documented lightweight acknowledgment/offload mode, or a safe per-channel policy that can accept messages while preserving session transcript ordering and tool side-effect safety. Likely touch points are `src/auto-reply/reply/agent-runner-memory.ts`, `src/agents/pi-embedded-runner/compact.queued.ts`, `src/agents/pi-embedded-runner/run.ts`, active-run queue/steering policy, and Telegram/channel integration tests.
