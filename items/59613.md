## Summary

Keep open. The report is recent, concrete, externally corroborated by another user, and describes a plausible Docker sandbox regression for `workspaceAccess: "none"` where `/workspace` is mounted read-only despite docs describing a per-scope sandbox workspace and file tools in the Docker sandbox. I do not have high-confidence evidence that current `main` fixes it, and related sandbox workspace-access items remain open.

## Evidence

- **Concrete reproduction and logs:** Issue #59613 includes exact Docker setup commands, sandbox config (`mode: all`, `scope: agent`, `workspaceAccess: none`), Docker inspect output showing `/home/node/.openclaw/sandboxes/...:/workspace:ro,z`, and tool failures including `Sandbox path is read-only; cannot create directories: /workspace`.
- **Independent corroboration:** A commenter reports the same issue and says it appeared after upgrading to 2026.4.2, which makes this more than a one-off local configuration report.
