Keep open. Current main still omits the reported `file` alias from the shared tool-display path resolver and read-start diagnostic, regression coverage does not cover `{ file: ... }`, and #55121 is still an open, unmerged implementation candidate that references this issue with closing syntax.

## Best Possible Solution

Keep this issue open and review/land #55121 or an equivalent focused fix. The fix should add `file` to `resolvePathArg()`, align the read-start diagnostic with the same alias handling, add regression coverage for `{ file: "..." }`, and update the shared display manifest if OpenClawKit or other manifest-driven surfaces still depend on it. Maintainers should also confirm whether the current canonical `path` runtime policy changes the intended compatibility contract for `file`, `filePath`, and `file_path`.
