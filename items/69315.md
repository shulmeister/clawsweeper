## Summary

Keep open. Current main still applies auto-TTS inline before reply payloads are routed or queued, and the speech-core runtime awaits provider synthesis before returning a media-bearing payload. Telegram also awaits the whole dispatch path under its sequential middleware, so the reported non-blocking/background delivery behavior is not implemented. Related #42539 covers a different text-plus-voice delivery mode and does not supersede this async queue decoupling request.

## Best Possible Solution

Keep the issue open and implement an explicit async/background TTS delivery path in the shared reply/TTS pipeline. A good fix would deliver visible text first, enqueue bounded-concurrency TTS jobs with durable media cleanup, route completed audio through existing channel delivery hooks, and add targeted tests for Telegram sequential inbound handling, block/final reply ordering, and config/schema/docs if a delivery mode option is added.

## Risks / Open Questions

- Implementing this touches reply ordering, media cleanup lifetime, channel-specific edit/follow-up behavior, and transcript visibility semantics.
- Out-of-order or background voice-note delivery will need explicit failure handling so delayed TTS errors do not block or corrupt the main text reply flow.
