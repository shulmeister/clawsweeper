## Summary

Keep open. Current main still awaits TTS generation before reply delivery in the auto-reply/ACP paths, and the config/docs do not expose an async/background TTS delivery mode. The related ClawSweeper context shows adjacent TTS issues, especially #42539 for Telegram text/voice separation, but none is a canonical duplicate for non-blocking TTS generation and inbound queue unblocking.

## Best Possible Solution

Keep the issue open and implement an explicit async/background TTS delivery mode in the TTS/reply delivery layer. The likely product path is text-first delivery by default for async mode, then background synthesis with bounded concurrency and follow-up media delivery through existing channel outbound routing, plus config/schema/docs/tests covering final, block, tool, Telegram voice-note, and per-session queue behavior.

## Risks / Open Questions

- Implementing this touches reply ordering, hooks/archive semantics, channel media normalization, Telegram voice-note behavior, and per-session queue lifecycle, so it should be designed as an explicit delivery policy rather than a small local TTS tweak.
- #42539 remains related but narrower: it asks for Telegram text plus voice-message separation, while #69315 is about non-blocking generation and inbound queue progress.
