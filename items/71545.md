## Summary

Keep open. Current main still creates a cached MCP client/session per configured server and `callTool` forwards directly through that cached client with no lost streamable-http session detection, transport rebuild, catalog refresh, or one-shot retry. PR #71545 is a focused fix for a current streamable-http MCP runtime gap, and the provided discussion shows review feedback plus a follow-up hardening commit rather than an obsolete or duplicate change.

## Best Possible Solution

Keep PR #71545 open and review it as the likely implementation path for streamable-http MCP session recovery. The eventual fix should live in `src/agents/pi-bundle-mcp-runtime.ts`, preserve the custom MCP JSON schema validator on rebuilt clients, serialize or share concurrent reconnect attempts, avoid reconnecting on ordinary tool-level HTTP 4xx errors, keep failed reconnects retryable, refresh catalog/tool metadata after successful reconnect, and add focused regression tests in `src/agents/pi-bundle-mcp-runtime.test.ts`/the MCP harness for successful recovery, failed recovery, and concurrency/tool-error edge cases.
