## Summary

Current main still intentionally rejects Control UI sessions without device identity from non-localhost insecure HTTP contexts. The issue discussion has valid workarounds and the docs explain the policy, but the reported user-facing behavior remains observable and the related fixes are either open/unmerged or narrower than this report.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The likely resolution is either a deliberately designed safe homelab-friendly path for remote HTTP Control UI device identity, or an explicit policy decision to keep rejecting it after landing clearer UI/server warnings such as #71669 and any copy-only improvements from #56830.
