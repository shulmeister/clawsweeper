## Summary

Keep #32473 open. Current main still intentionally rejects non-localhost plain-HTTP Control UI connections that lack browser device identity, even when `allowInsecureAuth` is set. Docs and issue comments document HTTPS, localhost/SSH tunnel, trusted proxy, Traefik, and break-glass workarounds, but the requested homelab/VPS remote HTTP ergonomics and policy decision remain unresolved; the related #71669 warning and #56830 copy-only PR are narrower follow-ups, not a full replacement for this issue.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection path that preserves device identity where browsers allow it, or to make an explicit policy decision that remote plain HTTP must remain rejected and then land the narrower operator-facing warning/copy improvements such as #71669/#56830 before closing this support-heavy issue.

## Risks / Open Questions

- A remote plain-HTTP bypass would weaken the MitM protection that the current policy is preserving; any solution needs explicit product/security design rather than an incidental config relaxation.
- Closing as duplicate of #71669 would preserve a warning request but drop the broader request for a safe homelab/VPS path or an explicit policy decision.
- Some UI copy still points users toward allowing insecure auth explicitly, while the server and docs make that local-only unless the dangerous device-auth disable flag or trusted-proxy auth is used.
