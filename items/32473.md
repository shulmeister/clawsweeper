## Summary

Keep #32473 open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions that lack browser device identity, even when `gateway.controlUi.allowInsecureAuth` is set. The thread has valid documented workarounds, but the remaining homelab/VPS request to keep device authentication enabled over non-127.0.0.1 HTTP remains a product/security decision rather than an implemented fix or fully superseded report.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection model that preserves browser device identity where feasible, or to record an explicit maintainer policy that remote plain HTTP must remain rejected and then close only after narrower warning/guidance work like #56830/#71669 is resolved.

## Risks / Open Questions

- Changing this behavior could weaken the MitM/device-identity protection that current gateway policy and docs intentionally preserve.
- Closing as implemented would ignore commenters who specifically asked for non-127.0.0.1 homelab access while keeping device authentication enabled.
- Closing as duplicate of #56830 or #71669 would narrow the issue to warning/copy work and leave the broader connection-model decision untracked.
