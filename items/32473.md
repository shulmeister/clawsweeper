## Summary

Keep #32473 open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions that lack browser device identity. The thread documents working HTTPS, localhost/SSH tunnel, Tailscale/Traefik, trusted-proxy, and break-glass workarounds, but the broader homelab/VPS request for a safe non-127.0.0.1 path with device identity preserved is not implemented or fully superseded by the narrower warning/copy follow-ups.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection model that preserves device identity where browsers allow it, or explicitly decide that remote plain HTTP must remain rejected and then land the narrower operator-facing warning/copy improvements in #71669/#56830 before closing this support-heavy issue.
