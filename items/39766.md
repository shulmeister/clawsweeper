## Summary

Keep #39766 open. Current main still requires `models.providers.<id>.models` in the source Zod schema, generated JSON schema, and exported config type, and validation fails before runtime/default normalization can fill provider defaults. The runtime embedding path would honor `models.providers.openai.baseUrl` if the config survived validation, so the PR is still addressing a real generic OpenAI/custom-provider bug tied to open #39589. The shipped Ollama work is provider-specific and does not supersede this generic schema fix.

## Best Possible Solution

Keep this PR or canonical issue #39589 open until main accepts generic `models.providers.<id>` entries that omit `models`, preserves a safe runtime contract for downstream model iteration, regenerates the generated config schema, and adds focused regression coverage for config parsing plus OpenAI embedding `baseUrl` fallback behavior. The Ollama-specific shipped normalization should remain, but it does not replace the generic schema fix requested here.

## Risks / Open Questions

- The PR should update or regenerate `src/config/schema.base.generated.ts` alongside the Zod schema change.
- The exported config type currently requires `ModelProviderConfig.models`, so maintainers should choose between keeping the parsed/runtime contract as `models: []` via `.default([])` or widening the type if using `.optional()`.
- The embedding regression test should avoid real DNS/network dependency by stubbing the SSRF hostname preflight path.
- The PR title says credential-only entries, but current main also requires `baseUrl`; maintainers should decide whether pure credential-only provider entries are in scope or keep this focused on `baseUrl` plus credentials without `models`.
