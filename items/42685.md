## Summary

Current main already skips the Tailscale hostname probe when `gateway.tailscale.mode` is `off`, so the `status --all` ENOENT warning described in #42685 is no longer implemented on main. The old `v2026.3.31` code still called `readTailscaleStatusJson(...)` unconditionally, but commit `72dcf942213845dae33d454cc2762c4980017f45` introduced the guarded bootstrap path and that shipped in `v2026.4.8`.

## Evidence

- **Current main short-circuits the probe when mode is off:** `createStatusScanCoreBootstrap` sets `tailscaleDnsPromise` to `Promise.resolve(null)` when `tailscaleMode === "off"`, so no Tailscale binary probe runs on the `status --all` path anymore.
  - file: [src/commands/status.scan.bootstrap-shared.ts:93](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/commands/status.scan.bootstrap-shared.ts#L93)

## Close Comment

Closing as implemented on current `main` after a Codex review. The old `status --all` path did reproduce this in older builds: `v2026.3.31` still called `readTailscaleStatusJson(...)` unconditionally in `src/commands/status-all.ts:61`, which explains the `spawn tailscale ENOENT` warning when `gateway.tailscale.mode = off`. Current `main` now routes through `src/commands/status.scan.bootstrap-shared.ts:93`, where `tailscaleMode === "off"` short-circuits to `Promise.resolve(null)` instead of probing the binary, and `src/commands/status-all/diagnosis.ts:171` only renders an error block if an error is actually present. This behavior is present in commit `72dcf942213845dae33d454cc2762c4980017f45` and shipped in `v2026.4.8`, so there is nothing left to track in this issue on current main.
