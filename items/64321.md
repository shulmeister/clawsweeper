## Summary

This should close as a duplicate of #57715. The reported slow `sessions.list` behavior is the same remaining gateway performance bug: row construction can still do N+1 transcript usage fallback reads, and `limit` is applied only after full row construction. The adjacent #57534 cleanup covered `sessions.json` cache / bulky store-row fields, but it does not cover the large JSONL transcript-history fallback described here.

## Best Possible Solution

Close this issue as a duplicate and keep #57715 as the canonical performance tracker. The best fix belongs there: pre-filter/sort/apply limit before expensive row hydration where semantics allow, build a request-scoped child-session index, use persisted metadata or a small metadata index for default `sessions.list`, and keep transcript-derived usage/title/message previews behind bounded opt-in hydration with regression coverage for large histories.
