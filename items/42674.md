## Summary

Keep open. Current main still has the Telegram `message.send` buffer mismatch: the shared message tool advertises `buffer`/`filename`/`contentType`, but the Telegram send pipeline only hydrates and forwards URL/path media. PR #42674 is stale against the current `extensions/telegram` layout and still has a review-raised size-guard concern, but it addresses a real remaining bug rather than obsolete work.

## Best Possible Solution

Keep the PR or canonical bug open until Telegram `message.send` supports inline base64 buffers end to end on current main: treat `buffer` as sendable media in the shared runner/hydration path, normalize data URLs for `send`, pass `buffer`/`filename`/`contentType` through `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, add inline-buffer support to `extensions/telegram/src/send.ts` using the existing media sender path, enforce `mediaMaxBytes` before and after decoding, and add regression tests at the runner, action-runtime, and transport layers. If maintainers prefer a fresh PR because this one is stale, close #42674 only after a replacement PR or issue is linked.
