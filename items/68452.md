## Summary

Keep open. Current `main` at `d251932fcfd60e205e80bf7e9abdb4af63582e6f` still does not preserve an exact `NO_REPLY` from the latest successful tool result when the final assistant message is empty; the implemented incomplete-turn logic only recognizes assistant-authored silent replies or committed messaging delivery. The linked PRs are evidence of an active fix path, but the bug is not implemented on the checked-out main.

## Best Possible Solution

Land a narrow embedded-runner fix that, for cron-triggered runs with zero final payloads and no abort/timeout/tool-error blocker, inspects `attempt.messagesSnapshot` for the latest non-error `toolResult` whose text is exact `NO_REPLY`, treats that as a quiet success, and adds focused coverage in `src/agents/pi-embedded-runner/run.incomplete-turn.test.ts` plus cron delivery coverage if the final payload contract changes.

Its attempt type does not include `messagesSnapshot`, so it cannot inspect the latest successful `toolResult` for exact `NO_REPLY`.
