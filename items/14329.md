## Summary

Current main already provides real-time user-turn delivery for operator WebSocket clients through the session transcript stream (`sessions.messages.subscribe` -> `session.message`), which satisfies the issue’s stated bridge/dashboard use case without adding `state:"user"` to `chat` events.

## Evidence

## Risks / Open Questions

- Clients listening only to `chat` events will still not see user turns; they need to use `sessions.messages.subscribe` / `session.message` instead.

## Close Comment

Closing from a Codex review because the underlying capability is already on `main` and shipped, just through the newer session transcript stream rather than `chat.state = "user"`. Operator clients can subscribe with `sessions.messages.subscribe` (`src/gateway/server-methods/sessions.ts:608`) and receive `session.message` transcript events (`src/gateway/server-session-events.ts:83`). `chat.send` also emits the user transcript update eagerly so bridges/watchers do not have to wait for the assistant reply (`src/gateway/server-methods/chat.ts:2460`), with coverage in `src/gateway/server-methods/chat.directive-tags.test.ts:1663`. The public protocol docs in `v2026.4.22` already document `sessions.messages.subscribe` and `session.message` as the transcript/event-stream path (`docs/gateway/protocol.md:384`). So the request is implemented well enough on OpenClaw, but via `session.message` instead of adding a new `chat` state.
