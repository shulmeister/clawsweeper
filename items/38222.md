## Summary

Keep open. Current main still does not enforce length limits on plugin hook-injected prompt context. The hook merger, final legacy-plus-phase join, embedded runner, and CLI runner paths all pass prompt-visible hook strings through without a cap, and current tests assert concatenation rather than truncation. The PR remains a plausible focused core hardening change, not a conservative close candidate.

## Best Possible Solution

Keep the PR open and adapt or land a core implementation that clamps prompt-visible hook strings after per-hook merge and after the final legacy-plus-phase join, preserves model/provider override fields, logs truncation, and adds regression coverage for both `before_prompt_build` and legacy `before_agent_start`. Prefer placing the final clamp in shared prompt-hook resolution or otherwise covering both embedded and CLI runner prompt preparation paths.
