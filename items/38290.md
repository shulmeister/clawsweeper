## Summary

Keep PR #38290 open. Current main still normalizes browser origins with `URL.origin`, so an exact `chrome-extension://<id>` or other opaque-origin allowlist entry becomes `"null"` and cannot match `gateway.controlUi.allowedOrigins`. The PR remains a narrow fix with focused unit and WebSocket handshake coverage, and the 2026-04-25 maintainer comment explicitly says the bug is still valid and scheme wildcards should stay separate under #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in `src/gateway/origin-check.ts` that symmetrically normalizes opaque-origin schemes for incoming `Origin` values and configured `allowedOrigins`. Preserve literal `Origin: null` rejection, exact matching, global `"*"` behavior, host-header fallback semantics, and gateway WebSocket regression coverage. Track scheme wildcard behavior separately in #39739.
