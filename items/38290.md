## Summary

Keep open. Current `main` at `d74f897c1c6b2de756c98eb3bc7de45f411d6a5c` still compares browser origins using `URL.origin`, so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` still collapse to `"null"` and cannot match. PR #38290 remains a narrow, relevant gateway fix with focused unit and WebSocket handshake coverage; the separate wildcard request remains tracked by #39739.

## Best Possible Solution

Keep PR #38290 open and rebase or land this exact-origin normalization, or land an equivalent patch in `src/gateway/origin-check.ts` that normalizes opaque-origin schemes symmetrically for request origins and configured allowlist entries while preserving literal `Origin: null` rejection, exact matching, wildcard `"*"` behavior, host-header fallback semantics, and gateway WebSocket regression coverage. Keep scheme-level wildcard support such as `chrome-extension://*` separate under #39739 because that is a broader product/security policy decision.
