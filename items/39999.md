## Summary

Keep #39999 open. Current main still reports package-manager update verification and post-update doctor failures, but it does not reinstall the previous global package version when the update step succeeded and the post-update health check fails. The PR remains a plausible core update-safety fix, though it needs to be rebased onto the current shared package-update helper and should address the existing review concern about only rolling back after a successful install step.

## Best Possible Solution

Keep the PR open and revise it against the current shared package-update helper. The best implementation is to carry the known `beforeVersion` into package update handling, attempt `${packageName}@${beforeVersion}` rollback only after the final package-manager install step succeeds but install verification or post-update doctor fails, rerun doctor when the restored entrypoint exists, preserve the overall error result, and add focused CLI/helper tests for doctor failure, verify failure, rollback doctor behavior, and failed-install-with-no-rollback.

- The submitted patch targets an older inline update flow; current main has centralized package update behavior in `src/infra/package-update-steps.ts`, so a clean rebase likely needs a helper-level design.
