## Summary

Current `main` no longer matches the silent round-trip data-loss report: write-prep now preserves source-only keys, config writes validate and reject suspicious shrink/clobber payloads before commit, and `openclaw plugins install` fails closed on unrelated invalid config instead of proceeding with writeback. Unknown top-level keys are still invalid under the strict root schema, but they now surface as validation errors rather than being silently stripped on write.

## Evidence

- **Source projection now preserves keys missing from runtime shape:** `projectSourceOntoRuntimeShape()` keeps `source` entries when the runtime snapshot does not contain that key, instead of dropping them during merge-patch write preparation.
  - file: [src/config/io.write-prepare.ts:49](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/config/io.write-prepare.ts#L49)

  - file: [src/config/io.ts:1658](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/config/io.ts#L1658)

## Risks / Open Questions

- Current `main` still intentionally treats unknown top-level config keys as invalid input under the strict root schema; the reporter's proposed root `.passthrough()` behavior is not what shipped.
- `plugins.allow` policy for bundled plugin CLI surfaces is still a separate UX/policy question and is not the mechanism behind the silent-write path reviewed here.
