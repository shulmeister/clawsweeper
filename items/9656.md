## Summary

Keep #9656 open. Current main still treats Signal `editMessage` envelopes as ordinary inbound messages by falling back to `editMessage.dataMessage`, while dropping `editMessage.targetSentTimestamp`. That means edits still get a new envelope timestamp as `MessageSid`, flow through normal reply dispatch, and lack the requested edited marker, original-message dedupe/replacement, and edit-specific reply suppression. The related PR #9701 is closed and unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open until the bundled Signal plugin explicitly handles `editMessage`: type and validate `editMessage.targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, and suppress edit-triggered replies unless the edit explicitly mentions the agent. Add focused Signal plugin tests for edit envelopes, messageId/dedupe behavior, and group mention-gated edit handling.
