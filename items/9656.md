## Summary

Keep #9656 open. Current main still normalizes Signal `editMessage` envelopes by reading only `editMessage.dataMessage`, does not type or preserve `editMessage.targetSentTimestamp`, derives `MessageSid` from the new envelope timestamp, and sends edits through the normal inbound reply pipeline. The related PR #9701 was closed unmerged in the provided GitHub context, and its head SHA is not present in this checkout.

## Best Possible Solution

Implement this in the bundled Signal plugin: type and validate `editMessage.targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, suppress edit-triggered replies unless explicitly mentioned, and add focused Signal tests for edit envelopes, `MessageSid`/dedupe behavior, and mention-gated group edits.
