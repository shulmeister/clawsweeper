## Summary

Current main no longer uses the static env-var map this PR edits, and `bailian` is not a current supported provider id. Auth env resolution is manifest-driven now, with active Alibaba/Qwen support living under `alibaba` and `qwen` plus legacy `modelstudio`/`qwencloud` compatibility, so this PR is obsolete against today’s code.

## Close Comment

Closing this PR after a Codex review against current `main` (`564f820efa2bb81ae49b8f55a93e41492ad31917`). The patch no longer matches the current auth surface: `src/agents/model-auth-env-vars.ts:1` is now just a wrapper, and bundled provider auth env vars come from plugin manifests via `src/secrets/provider-env-vars.ts:76`. On current main, the supported Alibaba-hosted provider ids are `qwen` and `alibaba`, with only `modelstudio`/`qwencloud` kept as compatibility aliases (`src/agents/provider-id.ts:3`, `docs/providers/qwen.md:18`). The bundled manifests already define the active env-var candidates for those providers in `extensions/qwen/openclaw.plugin.json:21` and `extensions/alibaba/openclaw.plugin.json:4`, and there is no current `bailian` provider surface on main. Because of that, this older one-line `bailian` map addition is obsolete rather than mergeable. If we decide to add a new `bailian` compatibility alias in today’s architecture, that should come as a fresh PR against the manifest/alias layer.
