## Summary

Keep open. Current main implements meaningful pieces of the Web UI chat responsiveness request, including optimistic user-message display, run tracking, an empty-stream reading indicator, and a non-blocking `chat.send` acknowledgment. The central issue is broader, though: gateway/config UI responsiveness and end-to-end latency attribution from UI action to gateway/provider/model/first-visible feedback are still not implemented as requested, and related work such as #64004 and #71630 remains open rather than providing a canonical merged fix.

## Best Possible Solution

Keep this open as a UX/performance tracker, or split it into concrete maintainer-owned issues: preserve the shipped optimistic/non-blocking chat path; add UI action-to-gateway-to-provider-to-first-visible timing attribution; isolate slow config/status/background refreshes from chat rendering and input; add per-control pending states for config load/save/test/switch actions; and cross-link narrower work such as #64004 and #71630.
