## Summary

Keep open. Current main still treats `responseUsage` as a per-session field set by `/usage`; Discord thread routing can create a separate thread session with a parent session key, but the new session is not seeded from a global/default config, Discord thread default, or parent session `responseUsage`. The issue is recent, has no comments or protected labels, and the provided related items do not supersede this specific usage-footer default/inheritance request.

## Best Possible Solution

Add a native, documented default/inheritance path rather than closing this: either a generic session default such as `session.defaults.responseUsage`, a Discord-owned `channels.discord.threadDefaults.responseUsage`, or parent-session inheritance in the thread-session creation path. The implementation should keep `/usage off|tokens|full` as the per-session override, add focused regression coverage for Discord thread session creation, and update the public usage/session or Discord docs.
