## Summary

Keep this PR open. Current main still has an unbounded WhatsApp connection wait, and the PR remains a plausible focused fix, but it is not complete for the current controller-based monitor path. The prior review discussion was useful: the latest PR commit preserves QR login behavior by making the helper timeout opt-in, but the remaining implementation should wire an initial startup timeout through the controller path and ensure timed-out sockets are closed.

## Best Possible Solution

Keep this PR open and ask for a rebase/update: add an opt-in initial WhatsApp connection timeout through waitForWaConnection and WhatsAppConnectionController.openConnection, use it from the Gateway monitor startup path and the legacy direct inbound path, preserve unbounded QR login waits unless the login layer supplies its own deadline, close timed-out sockets, clean up listeners/timers, and add focused tests for timeout rejection, listener cleanup, socket close on startup timeout, and QR-login behavior.

## Risks / Open Questions

- Landing the PR as-is would likely leave the current WhatsApp Gateway monitor startup path unbounded because the controller still calls waitForWaConnection without a timeout.
- A global default timeout on waitForWaConnection would regress QR login flows that are expected to remain pending while the user scans or receives refreshed QR codes.
- Any timeout implementation must close or transfer ownership of the Baileys socket on timeout so repeated reconnect attempts do not accumulate orphaned sockets.
