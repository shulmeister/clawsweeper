## Summary

Keep this issue open. Current main still has the unmitigated Race C path: plugin binding approvals are cached before an awaited atomic write, with no serialization around concurrent saves. The two lower-severity findings still match the described mitigated behavior, but the approvals cache coherency bug remains actionable in core.

## Best Possible Solution

Keep this issue open and fix Race C in `src/plugins/conversation-binding.ts` by serializing the persistent approvals read-modify-write path, for example with a per-process promise queue or mutex stored in the global plugin binding state. Add a focused regression test that forces two concurrent allow-always approval writes to complete out of order and asserts that both in-memory cache and `plugin-binding-approvals.json` reflect the latest intended state after restart/reload. Then separately decide whether Race A and Race B need documentation or should remain accepted mitigated behavior.
