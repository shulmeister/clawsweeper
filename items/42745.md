## Summary

Keep open. Current `main` has generic ACP block-text fallback logic, but it still does not bridge same-surface live Discord ACP output into Discord’s draft preview lane. ACP live output goes through normal block delivery, while Discord draft streaming only updates from `onPartialReply`.

## Evidence

- **ACP dispatch still uses the generic delivery coordinator:** `tryDispatchAcpReply` builds the ACP projector with `deliver: delivery.deliver`; there is no Discord-specific `onPartialReply` bridge in the ACP path.
  - file: [src/auto-reply/reply/dispatch-acp.ts:355](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/dispatch-acp.ts#L355)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **ACP live blocks still go to normal block replies:** The ACP delivery coordinator records block text and, on direct delivery, sends block payloads via `dispatcher.sendBlockReply(...)`; it does not call any preview callback.
  - file: [src/auto-reply/reply/dispatch-acp-delivery.ts:273](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/dispatch-acp-delivery.ts#L273)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Discord draft previews only consume partial-reply updates:** Discord’s message handler updates the draft preview from `updateDraftFromPartial(...)` and wires that only to `replyOptions.onPartialReply`; normal block replies are delivered separately.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:689](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/discord/src/monitor/message-handler.process.ts#L689)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Current tests assert Discord ACP visibility via block replies, not draft preview bridging:** The ACP tests on `main` specifically expect direct Discord ACP text to be visible through `sendBlockReply(...)` and not through a draft-preview path.
  - file: [src/auto-reply/reply/dispatch-acp.test.ts:1128](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/dispatch-acp.test.ts#L1128)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **One half of the PR is already covered, but not the draft-preview half:** `finalizeAcpTurnOutput` already synthesizes a final text fallback from accumulated ACP block text when needed, so the remaining missing behavior is the live Discord draft-preview bridge itself.
  - file: [src/auto-reply/reply/dispatch-acp.ts:187](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/dispatch-acp.ts#L187)
