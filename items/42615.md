## Summary

Close as implemented on main. The reported 2026.3.8 failure mode matches an older Discord guild-resolution path that required a full `guild` object and could drop allowlisted guild `MESSAGE_CREATE` events when only `guild_id` was present. Current main, and the shipped `v2026.4.22` release, resolve guilds by `guildId`, still register the `DiscordMessageListener`, and include a regression test for the missing-guild-object case.

## Evidence

## Close Comment

Closing from a Codex maintainer review. This report matches an older Discord guild-resolution path from `v2026.3.8`: at that tag, `src/discord/monitor/allow-list.ts` only resolved configured guilds from a full `guild` object, so a guild `MESSAGE_CREATE` carrying only `guild_id` could fall through to the `Blocked discord guild ... (not in discord.guilds)` path. The current shipped code in `extensions/discord/src/monitor/allow-list.ts:325` resolves by `guildId` first, `extensions/discord/src/monitor/message-handler.preflight.ts:731` passes `params.data.guild_id` into that lookup, and `extensions/discord/src/monitor/message-handler.preflight.test.ts:722` now covers `accepts allowlisted guild messages when guild object is missing`. This was already shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so I’m closing this as implemented on main. If you can still reproduce on `v2026.4.22` or newer, please open a new issue with fresh logs and current `openclaw channels status --probe` output.
