## Summary

Keep #17840 open. Current main still supports reaction notification filtering, but it does not provide an opt-in setting that makes reaction events wake the agent. The prior discussion has concrete Slack, Telegram, and Discord workflows, and the latest maintainer-facing Codex review in the issue already kept #17840 as the canonical tracker over #56708.

## Best Possible Solution

Keep #17840 open as the canonical product request. Implement a documented opt-in wake mode such as `reactionTriggersTurn: "off" | "own" | "all" | "allowlist"` or equivalent after existing reaction authorization/filtering, then enqueue the reaction event and request a scoped heartbeat/agent turn for that session. Update config schemas, generated metadata, channel docs, and focused Slack/Telegram/Discord/Signal/Matrix tests while preserving passive default behavior.

## Risks / Open Questions

- Default behavior should remain passive/off to avoid waking agents on ordinary group-chat emoji noise.
- Any implementation needs to run only after existing reaction notification filters, allowlists, sender authorization, and channel access checks.
- Session routing needs channel-specific care, especially Telegram reactions where the Bot API does not include `message_thread_id`.
