## Summary

Keep #17840 open. Current main still has reaction notification queueing, not an opt-in reaction-triggered wake/agent-turn mode. The issue also has concrete Slack, Telegram, Discord, and personal-assistant use cases, plus a recent maintainer-authored review comment explicitly keeping this item open. Related #56708 overlaps but is newer and not a better canonical tracker than this richer issue.

## Best Possible Solution

Keep #17840 open as the canonical product request. Implement an opt-in per-channel/plugin setting such as reactionTriggersTurn: "off" | "own" | "all" | "allowlist" or a similarly scoped wake mode; apply it only after existing reaction authorization and notification filtering; enqueue the reaction system event; then request a scoped heartbeat/agent turn with the existing wake primitive. Update channel schemas, docs, and focused reaction tests while preserving the passive default. Treat #56708 as an overlapping duplicate candidate, not a reason to close this older, more detailed tracker.

## Risks / Open Questions

- Default behavior should remain passive/off to avoid waking agents on normal group-chat reaction noise.
- Any wake implementation must run after existing channel access controls, reactionNotifications filtering, and allowlists so reactions do not bypass auth.
- Thread/session routing needs channel-specific care, especially for Telegram where reaction updates do not include message_thread_id.
