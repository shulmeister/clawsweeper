## Summary

Keep open. Current `main` still keeps the session run active until the entire embedded reply path finishes. Telegram can deliver a visible reply before that point, but there is no code that releases the active session/lane immediately after successful delivery. The newer followup queue (`collect` default) mitigates message loss by queueing later inbound messages, but those followups still wait for the active run to end, so the reported multi-minute delay after failover remains plausible.

## Evidence

- **Embedded runs still occupy a per-session lane for the full run body:** `runEmbeddedPiAgent` wraps the whole embedded execution in `enqueueSession(() => enqueueGlobal(async () => { ... }))`, so the session lane is held until that async run returns or throws.
  - file: [src/agents/pi-embedded-runner/run.ts:252](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run.ts#L252)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Telegram can deliver visible payloads before the run finishes:** Telegram delivery happens through `deliverReplies(...)` during dispatch, and successful delivery is tracked immediately with `deliveryState.markDelivered()`. That is user-visible delivery, but it is separate from session-run completion.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:647](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-message-dispatch.ts#L647)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Active reply state is only cleared on operation completion/failure:** The reply registry marks a session active when the operation begins and only clears it inside `complete()`/`fail()` via `clearState()`. There is no earlier 'delivered final reply' release path here.
  - file: [src/auto-reply/reply/reply-run-registry.ts:328](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/reply-run-registry.ts#L328)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Reply operations complete only in the agent-runner finally block:** `replyOperation.complete()` is called in `runReplyAgent`'s `finally`, after the full run path exits. That means visible delivery during failover does not itself end the active run.
  - file: [src/auto-reply/reply/agent-runner.ts:1788](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/agent-runner.ts#L1788)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current mitigation is followup queueing, not early release:** Queue mode defaults to `collect`, active runs with `collect`/`followup` become `enqueue-followup`, and `runReplyAgent` enqueues that followup instead of starting immediately. This helps preserve later inbound messages, but they still wait behind the active run.
  - file: [src/auto-reply/reply/queue/settings.ts:7](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/queue/settings.ts#L7)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Docs describe queued followups while the agent is busy:** The queue docs explicitly say `collect` is the default and that queued messages wait for a followup turn after the current run ends, which matches the mitigation above rather than immediate lane release.
  - file: [docs/concepts/queue.md:25](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/concepts/queue.md#L25)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)

## Risks / Open Questions

- User-visible Telegram replies can still be followed by a long busy window if failover/timeout cleanup keeps the embedded run active.
- There is no current code or regression test showing that a successful visible reply immediately releases the active session/lane.
