## Summary

Keep open. The reported object-shaped `delivery.channel` misrouting path still appears plausible on current `main`: current normalization/load code drops non-string delivery channels, and the delivery planner then falls back to `"last"`, which can route to another remembered channel such as Feishu. The referenced fix PR #46907 was closed unmerged and its referenced commit is not present in this checkout.

## Evidence

- **Delivery planner falls back to last when channel normalization fails:** `resolveCronDeliveryPlan()` reads `delivery.channel` through `normalizeChannel()` and, for announce delivery, uses `deliveryChannel ?? "last"`. An object-shaped channel therefore resolves as missing and can use the last remembered route instead of Telegram.
  - file: [src/cron/delivery-plan.ts:21](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/cron/delivery-plan.ts#L21)
  - command: `nl -ba src/cron/delivery-plan.ts | sed -n '1,260p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Channel normalization only accepts strings:** `normalizeOptionalString()` returns a value only when the input is a string; `normalizeOptionalLowercaseString()` builds on that. This means `{ id: "telegram" }` is not accepted as a channel value by the planner or delivery-field parser.
  - file: [src/shared/string-coerce.ts:13](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/shared/string-coerce.ts#L13)
