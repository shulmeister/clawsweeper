## Summary

Current main still does not expose Claude CLI thinking/reasoning on either HTTP surface. The Gateway HTTP handlers serialize only visible assistant text, the Responses streaming schema has no reasoning delta events, docs still say `reasoning` is accepted but ignored, and the existing channel-side reasoning path is deliberately suppressed before generic outbound JSON reaches HTTP callers.

## Best Possible Solution

Keep this issue open as a valid Gateway HTTP parity feature. The best path is a core Gateway implementation that captures agent thinking/reasoning before outbound suppression, gates it by the resolved `reasoningDefault` or an explicit request opt-in, emits structured reasoning output for `/v1/responses` in both streaming and non-streaming modes, decides a compatible `/v1/chat/completions` representation, and adds focused tests in `src/gateway/openresponses-http.test.ts` and `src/gateway/openai-http.test.ts`. PR #56674 should be considered related implementation work but is not enough to close this issue on current main.

## Risks / Open Questions

- Reasoning visibility is sensitive and should stay opt-in, with docs and tests making the privacy/security boundary explicit.
- The exact wire shape for `/v1/chat/completions` reasoning is less standardized than Responses output items and needs maintainer API judgment.
