## Summary

Keep open. Current main still does not implement the PR's requested behavior: cache-TTL context pruning eligibility falls back to plugin overrides, Anthropic-family/Kilocode, and Google prompt-cache models only, with no generic `modelApi === "openai-completions"` branch. The PR is focused and plausible, but the broad all-OpenAI-compatible expansion still needs maintainer judgment because current plugin APIs already support provider-owned opt-ins and docs describe cache behavior as provider-specific.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide whether cache-ttl pruning should be enabled for every `openai-completions` transport, or whether OpenAI/DeepSeek/Xunfei/vLLM-style support should be modeled as provider-owned `isCacheTtlEligible` opt-ins, compatibility metadata, and matching docs/tests.
