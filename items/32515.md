## Summary

Keep #32515 open. Current main still does not implement the requested one-click Anthropic browser OAuth onboarding flow: the bundled Anthropic provider exposes Claude CLI reuse, manual setup-token paste, and API-key auth, while the legacy generic `oauth` choice is normalized or rejected in favor of setup-token/provider-specific OAuth. Existing `sk-ant-oat*` handling and `ANTHROPIC_OAUTH_TOKEN` support are useful narrower support, but they do not satisfy `openclaw onboard` opening a browser for “Sign in with Anthropic,” detecting account type, and configuring auth automatically. The related open #48420 is a narrower runtime OAuth/header bug, not a canonical tracker for this first-run UX request.

## Best Possible Solution

Keep the issue open for maintainer/product handling. If accepted, implement this inside the bundled Anthropic provider as a provider-owned OAuth auth method and manifest auth choice, using the existing provider auth `openUrl`/VPS-aware OAuth handler seams and auth-profile persistence. Update onboarding/configure/docs and add regression coverage for browser OAuth selection, API-key-versus-OAuth token behavior, and any account-type/default-model detection. If Anthropic does not support a sanctioned public browser OAuth flow, maintainers should explicitly retitle or close around the narrower Claude CLI/setup-token/API-key UX direction instead of auto-closing this request.
