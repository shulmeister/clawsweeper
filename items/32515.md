## Summary

Keep #32515 open. Current main has related Anthropic auth support, including API-key onboarding, Claude CLI reuse, manual setup-token handling, env-backed `ANTHROPIC_OAUTH_TOKEN`, and runtime handling for `sk-ant-oat*` credentials, but it still does not implement the requested one-click Anthropic browser OAuth flow from `openclaw onboard`. The March discussion’s incremental prefix-detection idea is only partially present in request/runtime handling, not as an onboarding UX that removes the manual API-key/setup-token/Claude-CLI choice.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The concrete path, if accepted, is to add an Anthropic provider-owned OAuth auth method and manifest auth choice in `extensions/anthropic`, wire it through the existing provider auth `openUrl`/OAuth handler seams, persist the resulting profile through the normal auth-profile path, update onboarding/configure/docs, and add regression coverage for browser OAuth selection plus API-key-versus-OAuth token handling. If Anthropic does not provide a supported public OAuth flow, maintainers should explicitly close or retitle this around the narrower setup-token/Claude CLI UX improvements.
