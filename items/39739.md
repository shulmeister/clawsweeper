## Summary

Current main still does not implement `chrome-extension://*` or any scheme-level wildcard support for `gateway.controlUi.allowedOrigins`. The gateway origin checker still accepts only the global `*` wildcard or exact normalized `URL.origin` values, and Node normalizes `chrome-extension://<id>` origins to `"null"`, so the requested wildcard remains a real core gateway feature/security decision. Related PR #39867 directly attempted this but is closed unmerged; PR #38290 is open but intentionally narrower, covering exact extension origins rather than scheme-level wildcards.

## Best Possible Solution

Keep #39739 open for explicit gateway product/security judgment. If accepted, implement a narrowly scoped `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, preferably restricted to parsed `chrome-extension:` origins rather than generic prefix wildcards. Preserve literal `Origin: null` rejection, existing HTTP(S) exact matches, host-header fallback, and loopback behavior. Add focused unit and WebSocket handshake tests for exact extension origins, `chrome-extension://*`, non-extension negatives, malformed origins, plus docs/config help updates. Coordinate with PR #38290 for exact non-standard origin normalization, but keep this issue as the wildcard tracking item unless a maintainer-approved follow-up replaces it.
