## Summary

Keep open. Current main still appears to let an initial WhatsApp Web connection failure such as ETIMEDOUT propagate out of startup, closing the socket and exiting the channel before the reconnect loop can handle it. The related issues are useful context but do not supersede this Linux immediate-connect failure.

## Evidence

- **Issue scope:** #61788 reports OpenClaw 2026.4.5 on Linux x86_64, successful QR pairing, then immediate WhatsApp WebSocket ETIMEDOUT after gateway startup. The provided related issues cover a WSL2 60-second drop pattern (#58481) and credentials corruption (#60136), not this exact initial-connect timeout path.
- **Initial connection errors still throw:** The current controller creates a WhatsApp socket, awaits waitForWaConnection, then closes the socket and rethrows on failure. That matches the reported startup failure shape rather than handling it as a reconnectable close.
