## Summary

Keep open. Current main still rejects active `memorySearch.multimodal` for the built-in `local` memory embedding provider because the local adapter does not advertise multimodal embedding support. Docs and tests currently codify Gemini-only multimodal memory, so the reported regression/request is not implemented or obsolete.

## Evidence

- **Current config validation rejects multimodal providers without explicit multimodal support:** `resolveMemorySearchConfig` throws when multimodal memory is active and the selected provider adapter does not return true from `supportsMultimodalEmbeddings` for the configured model.
  - file: [src/agents/memory-search.ts:393](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/agents/memory-search.ts#L393)

  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Built-in local adapter has no multimodal support hook:** The built-in `local` adapter defines `id: "local"`, `transport: "local"`, and `create`, but does not define `supportsMultimodalEmbeddings`, so it fails the current validation path when multimodal modalities are active.
  - file: [extensions/memory-core/src/memory/provider-adapters.ts:87](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/extensions/memory-core/src/memory/provider-adapters.ts#L87)
