## Summary

Current `main` does not have the reported 30s global Ollama request timeout path. Ollama requests inherit the shared embedded-runner watchdogs: a default 120s LLM idle timeout and a 30-minute Undici stream timeout. The native Ollama stream code does not apply a separate 30s provider timeout, so the issue’s stated 13-60s cold-start reproduction no longer matches current behavior.

## Risks / Open Questions

- A local model that takes longer than 120s to emit its first chunk could still hit the shared LLM idle timeout and then enter normal fallback handling.
- There is no per-provider timeout override in current config types, so operators who need different watchdogs per provider still have only the global `agents.defaults.llm.idleTimeoutSeconds` control.

## Close Comment

Closing from a Codex review because I could not match the reported reproduction on current `main` (`8866544ffefb42ca234c66c337134fb60743d8d0`). The current Ollama native stream path does not apply a separate 30s provider timeout: it forwards only the shared abort signal in `extensions/ollama/src/stream.ts:654`, the embedded runner wraps model streams with the shared LLM idle watchdog in `src/agents/pi-embedded-runner/run/attempt.ts:1620`, that watchdog defaults to 120s in `src/config/agent-timeout-defaults.ts:1`, and the underlying Undici stream timeout floor is 30 minutes in `src/infra/net/undici-global-dispatcher.ts:6`. The docs also already call out `agents.defaults.llm.idleTimeoutSeconds` as the knob for slow local models in `docs/concepts/agent-loop.md:163`. Given that, the issue’s stated 13-60s Ollama cold-start path no longer reproduces as written on current main.
