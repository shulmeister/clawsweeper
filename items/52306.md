## Summary

Current main no longer lets Telegram-style block streaming stalls hold the embedded run until the old 600s watchdog path. Streamed block replies are decoupled from the run, channel delivery is bounded by a 15s timeout, and coverage explicitly verifies that a timed-out block reply still returns the final payload instead of hanging the run.

## Evidence

- **Streaming replies are enqueued instead of blocking the embedded run:** When block streaming is enabled, the reply delivery handler enqueues the payload into the block-reply pipeline instead of awaiting channel delivery inline.
  - file: [src/auto-reply/reply/reply-delivery.ts:150](https://github.com/openclaw/openclaw/blob/665cc3d5/src/auto-reply/reply/reply-delivery.ts#L150)

  - sha: [665cc3d5](https://github.com/openclaw/openclaw/commit/665cc3d5)
- **Blocked channel delivery is hard-bounded to 15s:** `BLOCK_REPLY_SEND_TIMEOUT_MS` is 15000, and the pipeline wraps each `onBlockReply` send in a timeout. On timeout it aborts the send and stops remaining block replies instead of waiting for the embedded-run watchdog.
  - file: [src/auto-reply/reply/block-reply-pipeline.ts:126](https://github.com/openclaw/openclaw/blob/665cc3d5/src/auto-reply/reply/block-reply-pipeline.ts#L126)

## Close Comment

Closing as implemented on current `main` after a Codex review. The streaming path now routes block replies through a bounded pipeline instead of waiting on channel delivery inline: `src/auto-reply/reply/reply-delivery.ts:150` enqueues block replies, `src/auto-reply/reply/agent-runner.ts:80` sets a 15s block-reply timeout, and `src/auto-reply/reply/block-reply-pipeline.ts:126` aborts timed-out deliveries instead of letting them ride the old embedded-run watchdog window. There is also explicit regression coverage in `src/auto-reply/reply/agent-runner.misc.runreplyagent.test.ts:384` proving a timed-out `onBlockReply` still returns the final payload rather than hanging the run. The embedded attempt cleanup still releases the session write lock in `src/agents/pi-embedded-runner/run/attempt.subscription-cleanup.ts:64`, so the reported "hold the lock until the 600s watchdog fires" behavior is no longer how current `main` behaves.
