## Summary

Keep open. Current main still persists session metadata through filesystem `sessions.json` files and conversation transcripts through per-session JSONL files. The config surface exposes `session.store` only as a file path, not a Redis/Mongo/PostgreSQL-style backend selector, and the runtime/session APIs still call filesystem readers/writers plus `SessionManager.open(sessionFile)`. Related #57497 is a narrower PostgreSQL request and is not a shipped implementation or canonical duplicate for this broader external session-storage feature.

## Best Possible Solution

Keep this issue open for maintainer/product design. The likely path is a core session persistence abstraction that covers both `sessions.json` metadata and transcript storage semantics, preserves the current filesystem default, defines locking/concurrency and maintenance behavior, updates config/schema/docs, and adds integration coverage for multi-instance-safe backends. A plugin-facing backend registration API could follow once the core contract is explicit.

## Risks / Open Questions

- External session storage would touch core Gateway session resolution, locking, maintenance, transcript handling, CLI inspection/cleanup, docs, and plugin SDK helper contracts.
- A metadata-only backend would not solve the reported stateless deployment problem unless transcript JSONL persistence and `SessionManager` file semantics are addressed too.
