## Summary

Keep this issue open. Current main still treats `modelstudio` as a Qwen compatibility alias in both core provider normalization and the bundled Qwen plugin. That means an explicit `models.providers.modelstudio` entry is still reachable through the normalized `qwen` path, and Qwen-specific model suppression can run before inline custom-provider resolution. The one issue comment links a setup workaround, but I did not find a repo-side fix that gives user-defined `modelstudio` provider config precedence over the bundled alias.

## Best Possible Solution

Keep the issue open and fix the provider-precedence contract. A conservative path is to stop treating `modelstudio` as a generic provider-id alias when an explicit `models.providers.modelstudio` entry or `modelstudio/...` model ref is meant to be custom, limit legacy compatibility to deprecated auth choices or to cases with no explicit custom provider, and add a regression test for `modelstudio/qwen3.6-plus` with a custom `openai-completions` provider config on the Coding Plan host. Update the Qwen docs after the compatibility boundary is clarified.
