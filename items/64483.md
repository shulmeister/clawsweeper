## Summary

Current main does not provide enough evidence to close #64483. The Qwen plugin still treats `modelstudio` as a Qwen alias, still reads `models.providers.modelstudio.baseUrl` as a Qwen base URL, and still suppresses `modelstudio/qwen3.6-plus` on Coding Plan hosts before the custom-provider fallback path can resolve the user-defined provider. The issue is recent, unprotected, and actionable as a provider-precedence bug.

## Best Possible Solution

Keep #64483 open and fix provider precedence deliberately. The likely maintainer path is to let an exact user-defined `models.providers.modelstudio` entry win over Qwen's legacy alias before Qwen catalog/suppression hooks run, or to narrow the Qwen `modelstudio` alias to legacy auth/onboarding compatibility only. Add a focused regression test for `modelstudio/qwen3.6-plus` with an explicit `models.providers.modelstudio` Coding Plan/OpenAI-compatible config, plus a separate test that old Qwen `modelstudio/...` aliases still work when no explicit custom provider exists.
