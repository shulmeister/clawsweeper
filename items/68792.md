## Summary

PR #68792 is a narrow allowlist-only fix for the Windows Claude CLI spawn bug, but current main still needs the fuller supervisor `.cmd` handling tracked by #68819 or an equivalent core patch. The PR only changes `claude` to `claude.cmd`; it does not address the direct `.cmd` spawn failure path that current code and the related discussion identify.

## Best Possible Solution

Close #68792 as superseded. Do not merge the allowlist-only patch by itself; keep the remaining work on #68819 or an equivalent core fix that wires supervisor child spawns through the current Windows wrapper resolver, preserves child env PATH/PATHEXT semantics, and adds focused regression coverage for claude-cli and generic CLI backend .cmd wrappers.

## Evidence

- **Current child supervisor still lacks the requested allowlist entry on main:** At current main d54d2d6b9b8aa28fb222d895c0f01f43933f4409, the child adapter resolves Windows shims only for npm, pnpm, yarn, and npx. It then passes the resulting argv into spawnWithFallback, so adding claude here would produce a direct claude.cmd child spawn rather than a resolved executable/Node entrypoint.
  - file: [src/process/supervisor/adapters/child.ts:12](https://github.com/openclaw/openclaw/blob/d54d2d6b9b8aa28fb222d895c0f01f43933f4409/src/process/supervisor/adapters/child.ts#L12)

- **The bundled Claude CLI path reaches the child supervisor as bare claude:** The Anthropic plugin registers the claude-cli backend with command "claude". Claude live sessions build argv from that command and create a supervisor child-mode run, so this is the affected spawn path for the Windows report.
  - file: [extensions/anthropic/cli-backend.ts:30](https://github.com/openclaw/openclaw/blob/d54d2d6b9b8aa28fb222d895c0f01f43933f4409/extensions/anthropic/cli-backend.ts#L30)

- **Current code already treats direct .cmd spawning as a deeper Windows problem:** The shared exec path documents that Node 18.20.2+ rejects direct .cmd/.bat spawning with EINVAL and avoids direct .cmd execution for npm/npx. The newer shared windows-spawn helper resolves wrappers to executable or Node entrypoints and fails closed for unresolved wrappers, but the child supervisor does not use it today.
  - file: [src/process/exec.ts:44](https://github.com/openclaw/openclaw/blob/d54d2d6b9b8aa28fb222d895c0f01f43933f4409/src/process/exec.ts#L44)

It does not add entrypoint resolution, PATH/PATHEXT-aware wrapper handling, or reuse the existing Windows spawn helper.

PR #68819 remains open and explicitly tracks resolving Windows .cmd shims to underlying .exe or node entrypoints before spawn, including the follow-on EINVAL failure after the allowlist-only fix.
