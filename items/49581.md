## Summary

Current `main` already has the preview-finalization path that preserves a streamed Discord draft across assistant/tool boundaries and either edits that preview into the final reply or falls back to a normal send. The shipped changelog for `2026.4.22` also explicitly records the related cross-channel preview-streaming fix.

## Evidence

- **Shipped changelog entry:** `CHANGELOG.md` under `## 2026.4.22` says: `Channels/preview streaming: centralize draft-preview finalization so Slack, Discord, Mattermost, and Matrix no longer flush temporary preview messages for media/error finals, and preserve first-reply threading for normal fallback delivery.`
  - file: [CHANGELOG.md:281](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/CHANGELOG.md#L281)
  - command: `nl -ba CHANGELOG.md | sed -n '240,290p'`
- **Shared finalizer now preserves preview then falls back safely:** `deliverFinalizableDraftPreview` flushes and edits the existing preview for `kind: "final"`; if that cannot be done, it discards pending preview state, delivers normally, and only then clears the preview. That is the core behavior needed to avoid the draft-vanish race.
  - file: [src/channels/draft-preview-finalizer.ts:14](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/channels/draft-preview-finalizer.ts#L14)
  - command: `nl -ba src/channels/draft-preview-finalizer.ts | sed -n '1,260p'`
- **Discord final delivery uses the shared preview finalizer:** Discord final replies with an active draft now go through `deliverFinalizableDraftPreview(...)`, with `buildFinalEdit`, `editFinal`, and `deliverNormally` wired so the preview is finalized in place when possible and otherwise sent through the normal Discord reply path.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:791](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/discord/src/monitor/message-handler.process.ts#L791)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '780,870p'`
- **Discord rotates preview on assistant-message boundaries:** Discord draft streaming calls `draftStream.forceNewMessage()` from `onAssistantMessageStart`, resetting draft state at message boundaries instead of letting the old preview be overwritten across tool/message phases.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:950](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/discord/src/monitor/message-handler.process.ts#L950)

## Close Comment

Closing as implemented on `main` after Codex review. Current OpenClaw already routes Discord draft-stream finals through the shared preview finalizer in `extensions/discord/src/monitor/message-handler.process.ts:791`, which flushes/edits the draft when possible and otherwise falls back to normal delivery instead of dropping the final reply. The boundary handling is also present in `extensions/discord/src/monitor/message-handler.process.ts:950`, where Discord rotates preview messages on `assistant_message_start`, and the shared fallback logic lives in `src/channels/draft-preview-finalizer.ts:14`. This was also shipped in `v2026.4.22`; see the changelog entry at `CHANGELOG.md:281` for the preview-streaming finalization fix covering Discord. If you still see a current repro on `v2026.4.22` or newer, please open a fresh issue with updated logs and exact channel/plugin config.
