## Summary

Keep #60856 open. Current main has made useful Ollama memory-embedding progress, including the bundled Ollama adapter, but the reported timeout condition is still plausible in the current code path: non-local providers still get a hard 120s batch timeout, and the Ollama embedding provider still runs a batch as unbounded parallel single-text /api/embeddings calls. The issue comments are also consistent with that remaining risk: the confirmed workaround discussion specifically called out the Promise.all behavior for large CPU-only Ollama indexes.

## Evidence

- **120s timeout still applies to Ollama batches:** Memory batch embedding wraps provider.embedBatch(texts) in withTimeout using resolveEmbeddingTimeout('batch'); resolveEmbeddingTimeout only treats provider id 'local' as local, so the Ollama provider id remains on the 2 minute remote batch timeout path.
