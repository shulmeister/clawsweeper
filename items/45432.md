## Summary

Current main and the latest release no longer match the reported failure mode: OpenClaw’s write paths do not add a UTF-8 BOM, and the pinned edit-tool dependency strips a leading BOM before exact-text matching.

## Evidence

- **Host write path uses plain UTF-8:** createHostWorkspaceWriteTool delegates writes through createHostWriteOperations; the host write helper creates parents and calls fs.writeFile(resolved, content, "utf-8") without prepending EF BB BF.
  - file: [src/agents/pi-tools.read.ts:751](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/agents/pi-tools.read.ts#L751)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '646,826p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Workspace-only Windows write path has no BOM injection:** writeFileWithinRoot routes win32 to the legacy safe writer, whose temp-file write uses tempHandle.writeFile(params.data, params.encoding ?? "utf8") for string data; there is no BOM prefix step.
  - file: [src/infra/fs-safe.ts:386](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/infra/fs-safe.ts#L386)
