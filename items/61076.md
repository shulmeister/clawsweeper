## Summary

Keep this issue open. Current main has substantially changed the Anthropic streaming path since the reported 2026.4.1 build, but I could not verify a complete fix for the specific bad-control-character JSON parse failure. The current OpenClaw-owned Anthropic SSE parser still calls raw JSON.parse on SSE data, and the existing tests do not cover raw control characters, chunk-split Unicode, or the pure-text Telegram/Anthropic failure pattern from this issue.

## Best Possible Solution

Keep this open and add a focused regression around the OpenClaw Anthropic Messages transport: feed SSE chunks containing long text with emoji/non-ASCII and a raw-control-character-in-string case matching the reported JSON.parse failure, including split UTF-8 chunk boundaries, and verify the run emits a normal assistant message instead of an assistant_error. If the test reproduces, replace raw JSON.parse in parseAnthropicSseBody with a repair-capable parser or equivalent safe SSE event parsing, and confirm the linked #68442 tool-call/UI recovery path is either fixed by the same change or tracked separately.
