## Summary

Keep open. Current main has important partial mitigation for the related MCP child leak: session-scoped bundled MCP runtimes are reused, idle-reaped, cleaned up after one-shot runs, and stdio children are killed as process trees. But #68527 is broader and still actionable: current `openclaw mcp set` does not preflight or dry-run the MCP command, the bundle MCP startup path logs and disposes failed starts without a per-server backoff/circuit-breaker state, the generated systemd unit lacks default `MemoryHigh`/`MemoryMax`/`TasksMax`/`OOMScoreAdjust` guardrails, and status/process accounting for MCP child counts is not present. The related closed #68660 appears to cover duplicate MCP instance accumulation, not all remaining resource-guardrail work in this issue.

## Best Possible Solution

Keep #68527 open, but narrow it around the remaining work now that MCP runtime cleanup has partial coverage. Best maintainer path is to split or land focused changes: add a per-server MCP failure registry with capped exponential backoff and circuit reset on config fingerprint change; add an explicit or default `openclaw mcp set` preflight for stdio/HTTP reachability where safe; add systemd resource defaults or a documented installer drop-in for `MemoryHigh`, `MemoryMax`, `TasksMax`, and `OOMScoreAdjust`; and surface MCP child/process resource accounting in gateway status or stability diagnostics.
