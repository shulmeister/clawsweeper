## Summary

Current main already parses Telegram `:topic:<id>` targets into a base chat target plus thread id, preserves that explicit thread through outbound target resolution, and sends it as Telegram `message_thread_id`. The same implementation and regression tests are already present in shipped release `v2026.4.22`.

## Evidence

- **Telegram explicit target parsing:** Telegram channel parsing returns `to: target.chatId` and `threadId: target.messageThreadId` for explicit targets, including `:topic:` targets.
  - file: [extensions/telegram/src/channel.ts:404](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/telegram/src/channel.ts#L404)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Explicit target thread preserved in session delivery resolution:** Outbound session resolution parses explicit targets with the channel plugin and uses the parsed `threadId` when no separate explicit thread id is supplied.
  - file: [src/infra/outbound/targets-session.ts:125](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/infra/outbound/targets-session.ts#L125)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Telegram send path applies parsed topic as message_thread_id:** `sendMessageTelegram()` parses the recipient target, resolves the base chat id, and builds thread params from `target.messageThreadId` before calling Telegram.
  - file: [extensions/telegram/src/send.ts:1491](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/telegram/src/send.ts#L1491)

## Close Comment

Closing as implemented on main. This Codex review checked current `main` and the shipped `v2026.4.22` release, and the Telegram `:topic:` routing path is already implemented end to end: explicit target parsing returns base `to` plus `threadId` in `extensions/telegram/src/channel.ts:404`, outbound target resolution preserves that parsed thread in `src/infra/outbound/targets-session.ts:125`, and Telegram send applies it as `message_thread_id` in `extensions/telegram/src/send.ts:1491`. The repo also has focused regression coverage for both the target-string case and the “do not override an explicit target topic with the current topic” case in `extensions/telegram/src/send.test.ts:1646` and `extensions/telegram/src/action-threading.test.ts:17`, and those same assertions are present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit this on a newer build, please open a fresh issue with the exact OpenClaw version/commit and the concrete `message` tool payload that misrouted.
