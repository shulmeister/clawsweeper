## Summary

Keep open. Current main still chunks Discord long lines by searching only for whitespace and otherwise hard-splitting at the character limit, so the reported CJK boundary problem remains plausible and directly matches the implementation. The Discord send paths still use this chunker, and current tests cover whitespace/fence behavior but do not cover CJK punctuation-aware breaking.

## Evidence

- **Current chunker still uses whitespace-or-hard-break logic:** In current main, `splitLongLine()` scans backward only for `\s`; if none is found, it sets `breakIdx = limit` and slices there. There is no fallback for CJK punctuation or sentence boundaries.
  - file: [extensions/discord/src/chunk.ts:66](https://github.com/openclaw/openclaw/blob/3a18801343f1fe9cee6cfeadaeafd80cd3f7e9b8/extensions/discord/src/chunk.ts#L66)

- **Discord delivery paths still route replies through this chunker:** Outbound Discord text is built with `chunkDiscordTextWithMode(...)` in the shared send path and reply delivery path, so this logic still governs real message splitting on main.
  - file: [extensions/discord/src/send.shared.ts:259](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/extensions/discord/src/send.shared.ts#L259)
