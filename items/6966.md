## Summary

Current main has important adjacent fixes for rate-limit failover, including model-scoped auth cooldowns and configured fallback routing, but it does not fully implement the requested dynamic model switching policy: no upstream round_robin/model-health strategy, no metrics/header-based downgrade policy, and no automatic upgrade-back probe after a fallback candidate is succeeding.

## Evidence

- **Model-scoped cooldown exists on main:** Current main records and checks model-scoped rate-limit cooldowns: `isProfileInCooldown` allows a profile to bypass a rate-limit cooldown when the requested model differs from the recorded `cooldownModel`, while profile-wide disabled windows still block all models. This addresses part of the related #43879 behavior, but it is cooldown scoping rather than the broader dynamic model switching request.
  - file: [src/agents/auth-profiles/usage-state.ts:42](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/src/agents/auth-profiles/usage-state.ts#L42)
  - command: `nl -ba src/agents/auth-profiles/usage-state.ts | sed -n '1,220p'`
  - sha: [0bcb4c95c185](https://github.com/openclaw/openclaw/commit/0bcb4c95c185909b12d463b04d49b16ead3459d5)
- **Rate-limit failures are tied to model IDs:** `markAuthProfileFailure` accepts `modelId` and stores `cooldownModel` for `rate_limit` failures, widening to profile-wide only when a different or unknown model fails during an active cooldown. This supports model-aware fallback but not a tier downgrade/upgrade controller.
  - file: [src/agents/auth-profiles/usage.ts:634](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/src/agents/auth-profiles/usage.ts#L634)

- **Fallback remains ordered failover, not dynamic switching:** The documented runtime flow is static ordered failover: current session model first, configured `agents.defaults.model.fallbacks` in order, then configured primary at the end when running from an override. That is useful, but it is not the issue's requested automatic downgrade when rate limits spike and gradual upgrade when they subside.
  - file: [docs/concepts/model-failover.md:19](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/docs/concepts/model-failover.md#L19)

- **Fallback loop returns on first success:** `runWithModelFallback` iterates candidates and returns immediately when a candidate succeeds. If the session is already on a fallback and that fallback succeeds, the loop does not continue to probe the stronger primary, so the requested automatic upgrade-back behavior remains missing.
