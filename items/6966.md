## Summary

Keep open. Current main has stronger rate-limit handling than the original report described: auth-profile rotation, ordered model fallback, model-scoped cooldowns, and cooldown probing. It still does not implement the requested dynamic rate-limit-driven model routing policy: no model strategy config, no per-model health store, no round-robin/health-aware selector, and no downgrade/upgrade-back policy. The aashari fork linked in the issue remains useful prior art, not shipped upstream behavior.

## Best Possible Solution

Keep #6966 open as a core model-routing design issue. A good upstream path would define documented routing policies such as failover versus health-aware or round-robin routing, add per-provider/model/profile health state with success and failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema and docs, and add regression tests for rate-limit-driven downgrade and recovery scenarios. The aashari fork linked in the discussion is useful prior art to evaluate, but not a reason to close the upstream issue.
