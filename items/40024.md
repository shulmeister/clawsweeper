## Summary

Keep #40024 open. Current main still does not satisfy the reporter's baseUrl-only local OpenAI-compatible proxy case: a configured provider without an explicit API resolves to `openai-responses`, the embedded runner routes that API to the Responses transport rather than `/v1/chat/completions`, and provider fetches only opt into private-network access when `request.allowPrivateNetwork` is set. The existing docs provide an explicit workaround, but the reported regression remains actionable because the product currently times out instead of inferring the intended transport or failing early with a clear configuration error.

## Best Possible Solution

Keep the issue open. Either infer/select `openai-completions` for custom local OpenAI-compatible `/v1` providers that do not support Responses, or fail before the run with an actionable message requiring `models.providers.<id>.api: "openai-completions"` and `models.providers.<id>.request.allowPrivateNetwork: true` for trusted loopback/LAN endpoints. Add a regression test for a `local-agent-proxy` style provider at `http://127.0.0.1:3000/v1` that verifies the selected transport path and error behavior.
