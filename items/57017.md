## Summary

Keep open. Current `main` still reads cron run logs fully into memory on the `cron.runs` read path, while pruning remains an async write-path behavior whose failures are only warned, so the reported oversized-log/OOM risk is still plausible and not implemented away.

## Evidence

- **Whole-file read remains on current main:** `readCronRunLogEntriesPage` still does a full `fs.readFile(...)`, then parses, filters, sorts, and slices in memory. There is no read-side byte cap, tail-read, or queued defensive prune in this function.
  - file: [src/cron/run-log.ts:371](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cron/run-log.ts#L371)
  - command: `nl -ba src/cron/run-log.ts | sed -n '360,460p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Pruning is still write-path only:** `pruneIfNeeded` exists, but it is only called from `appendCronRunLog`. Reads only drain pending writes and then read the file directly.
  - file: [src/cron/run-log.ts:133](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cron/run-log.ts#L133)
  - command: `nl -ba src/cron/run-log.ts | sed -n '116,210p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Cron runtime tolerates append/prune failure:** `server-cron` fires `appendCronRunLog(...)` without awaiting it and only logs a warning on failure, so a failed append/prune can leave an oversized file behind for later reads.
  - file: [src/gateway/server-cron.ts:553](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-cron.ts#L553)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '540,590p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **User-facing read path still reaches the vulnerable code:** The `cron.runs` gateway method still routes requests to `readCronRunLogEntriesPageAll(...)` or `readCronRunLogEntriesPage(...)`, so current CLI/Gateway history views still depend on these whole-file reads.
  - file: [src/gateway/server-methods/cron.ts:434](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-methods/cron.ts#L434)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '430,470p'`

- **Existing coverage does not show a defensive oversized-read fix:** Current tests cover write-side pruning and draining fire-and-forget writes, but there is no test evidence here for a capped read or other oversized-log read mitigation.
  - file: [src/cron/run-log.test.ts:70](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cron/run-log.test.ts#L70)

- **Issue discussion indicates the naive fix was rejected, not shipped:** The provided issue comment records that PR #57913 tried read-time pruning, but review found a race with `writesByPath`; the commenter explicitly points to a non-mutating capped read as the safer remaining fix path.
