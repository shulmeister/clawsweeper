## Summary

Keep open. Current main still has the DM media routing gap described here: direct download can drop unresolved `msteams://media/...` references, and the Bot Framework fallback is still gated on HTML `<attachment>` ids instead of being attempted for DM media attachments after direct download fails. The linked #40463 is an open implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open until the msteams plugin changes the inbound DM media fallback. The likely fix belongs in `extensions/msteams/src/monitor-handler/inbound-media.ts` and related tests: attempt Bot Framework attachment download for personal DM media after direct download fails even when no HTML `<attachment>` tags are present, while preserving the #58617 guard against mention-only HTML noise. Review #40463 for reusable pieces, but require a regression test for the exact RSC-only DM inline image/file case before closing.

## Evidence

The Graph path is also gated on `hasHtmlFileAttachment` and skips Bot Framework personal IDs, so DM inline media without `<attachment id="...">` still has no fallback after direct download returns empty.
