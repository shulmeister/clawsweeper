## Summary

Keep #69182 open. Current main appears to have addressed the more severe related #70481 class by enforcing a hard timeout, but the original #69182 complaint remains: active-memory is still intentionally a blocking pre-prompt step, with default/recommended timeout behavior that can delay the main reply until the configured deadline.

## Best Possible Solution

Keep the issue open as a focused bundled-plugin performance/product item. The likely fix belongs in extensions/active-memory and the prompt-build integration: either make active-memory non-blocking for first visible output, add a much smaller early fail-open budget, or explicitly decide and document that the blocking latency tradeoff is intentional.

- A non-blocking active-memory design would need to preserve prompt-cache determinism and define what happens to late memory after the main model turn has already started.
