## Summary

Keep open. Current main still deduplicates Feishu inbound processing by `message_id`, and the debounce path only removes repeated entries with the same `message_id`; it does not implement the requested sender/timestamp/content fallback for the reported case where Feishu emits distinct IDs for the same private-chat message.

## Evidence

- **Persistent Feishu dedupe is keyed only by messageId:** `claimUnprocessedFeishuMessage` normalizes `messageId`, checks `hasProcessedFeishuMessage(normalizedMessageId, namespace)`, and begins processing with that same ID. No sender, timestamp, or content hash participates in this claim.
  - file: [extensions/feishu/src/dedup.ts:37](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/extensions/feishu/src/dedup.ts#L37)
  - command: `nl -ba extensions/feishu/src/dedup.ts | sed -n '1,260p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **In-process Feishu processing claims are keyed only by namespace plus messageId:** `resolveEventDedupeKey` returns `${namespace}:${trimmed}` where `trimmed` is the message ID. Distinct Feishu message IDs for the same content will therefore produce distinct in-flight claim keys.
  - file: [extensions/feishu/src/processing-claims.ts:6](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/extensions/feishu/src/processing-claims.ts#L6)

- **Inbound handler only drops exact same messageId before debounce:** The receive handler reads `event.message.message_id` and calls `tryBeginFeishuMessageProcessing(messageId, accountId)`. The duplicate log is for `message ${messageId}`, so the early drop cannot catch the reported different-ID duplicate.

Later `onFlush` filters by `hasProcessedMessage(entry.message.message_id, ...)` and joins fresh texts, so two same-content events with different message IDs are still treated as fresh entries.

## Risks / Open Questions

- A content-based fallback will need a narrow time window and sender/chat scoping so intentional repeated messages are not suppressed.
- The current debounce may reduce some rapid duplicates to one dispatch, but it can still combine duplicate text and does not address non-text or out-of-window duplicates.
