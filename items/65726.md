## Summary

Keep this issue open. Current main has the ACP binding plumbing, including Slack thread routing to configured ACP sessions, but it still lacks the requested `notifyPolicy` config surface and does not provide a conversation/thread-level guard that suppresses another session targeting the same Slack thread.

## Best Possible Solution

Keep this issue open and implement it in core ACP/channel routing rather than closing as already handled. A good fix would add a typed `notifyPolicy` to `AgentRuntimeAcpConfig` and/or `AgentAcpBinding.acp`, validate and document it, propagate it through `ConfiguredAcpBindingSpec` into ACP task creation, and add tests proving `silent` ACP-bound sessions do not emit `Background task done` messages. The duplicate-response side should add a conversation-level active ACP ownership check keyed by channel/account/conversation/thread, with tests for Slack bound threads and clear policy for when separate sessions targeting the same thread are suppressed or warned.
