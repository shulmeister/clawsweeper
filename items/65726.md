## Summary

Keep this issue open. Current main still does not expose `notifyPolicy` on ACP binding or per-agent ACP runtime config, ACP background task records still default to visible `done_only` notifications when delivery is pending, and the active-run guard remains keyed by session key rather than Slack/channel thread ownership.

## Best Possible Solution

Keep this issue open and implement it in the ACP binding/channel routing surface. Add typed `notifyPolicy` support to `AgentRuntimeAcpConfig` and/or `AgentAcpBinding.acp`, update strict Zod schemas, generated help/docs, and ACP binding materialization so configured bound ACP tasks can pass `silent` into task creation. Separately add a conversation ownership guard keyed by channel/account/conversation/thread that can detect an active ACP-bound session before another session delivers to the same Slack thread, with clear policy and regression tests for bound Slack threads.
