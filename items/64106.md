## Summary

Current main already solves PR #64106's central MCP context-budget/tool-policy problem: bundled MCP/LSP tools are materialized, then filtered through the final effective tool-policy pass before they are exposed to prompts, runtime tool registration, and context-engine assembly. The exact nested `mcp.servers.<server>.tools` discovery-time config shape from this PR is not present, but users can now filter MCP tools with the supported general `tools.allow` / `tools.deny` surface using exposed `serverName__toolName` tool IDs or the `bundle-mcp` plugin key.

## Best Possible Solution

Close this PR as implemented by the shipped final effective tool-policy pass. Keep the documented general `tools.allow` / `tools.deny` surface for MCP tools using `serverName__toolName` or `bundle-mcp`; open a new focused proposal only if maintainers want a nested `mcp.servers.<server>.tools` alias or discovery-time optimization.
