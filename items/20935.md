## Summary

Keep #20935 open. Current main has a narrow memory host event journal for recall, promotion, and dreaming lifecycle events, but it still does not provide the requested append-only audit log for every MEMORY.md or memory/*.md mutation with what/when/why/session/user context or tamper-evidence. The issue discussion also supports keeping this as remaining core memory work, with Pinchy cited as useful prior art and a maintainer-authored Codex review already keeping the issue open on 2026-04-26.

## Best Possible Solution

Keep #20935 open. The best path is a first-class memory-change audit layer at the shared memory/core write surfaces, with hooks for host tools, sandbox bridge writes, memory flush, promotion, dreaming, and plugin-owned memory writes. Each record should include timestamp, operation, relative path, changed-content summary or diff/hash, trigger/why, session/run/user context when available, and optional signing or hash chaining. Add regression tests across MEMORY.md and memory/*.md write paths, then document the audit surface in the public memory docs.

## Risks / Open Questions

- Closing this now would leave ordinary write/edit/apply_patch mutations, memory flush appends, sandbox writes, and some memory-core direct writes outside any canonical memory-change audit trail.
- A correct implementation needs a privacy decision for memory content: log full diffs only when safe, otherwise log summaries plus hashes or redacted excerpts.
