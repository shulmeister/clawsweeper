## Summary

Keep open. Current main still does not implement the requested plugin-contributed native Control UI pages: navigation is a closed Tab union/group list, the bootstrap config has no plugin page/import-map payload, and the plugin manifest/registry/API only cover existing runtime surfaces such as HTTP routes, services, gateway methods, tools, channels, and providers. The issue is a real plugin API gap, not a ClawHub-only request, and the discussion includes active concrete use cases such as Openclaw-Quest plus unresolved spec questions.

## Best Possible Solution

Keep #66944 open and treat it as an in-scope plugin API/design issue. The best path is an additive spec and implementation covering manifest metadata for UI page contributions, registry/bootstrap transport, Control UI route/sidebar integration, safe dynamic loading of Custom Elements, auth/scope behavior for plugin-owned backends, docs, and contract/browser tests. The Openclaw-Quest comment should be used as an early validation case.

## Risks / Open Questions

- A native plugin page system would load plugin-supplied browser code into an admin Control UI, so the design needs explicit security, CSP, permission, origin, and versioning decisions.
- The discussion has unresolved product/API choices around framework neutrality and whether plugin tabs may use plugin-owned HTTP/WebSocket backends directly or must route through Gateway RPC.
