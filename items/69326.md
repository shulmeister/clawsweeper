## Summary

Keep #69326 open. Current main still does not route browser WebChat/Control UI `chat.send` replies back to the stored external WeChat delivery context; the current implementation explicitly keeps WebChat sends session-only and blocks external route inheritance for WebChat clients. The issue is recent, has concrete reproduction details, and describes a core WebChat/Gateway routing behavior rather than something that can be closed as stale or already implemented.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. A plausible fix would add an explicit, guarded WebChat-to-external delivery mode for selected channel-scoped sessions, with UI affordance, Gateway permission checks, and regression tests covering plugin channel IDs such as `openclaw-weixin`; alternatively, document that Control UI/WebChat intentionally stays session-only and provide a separate supported workflow for replying back through WeChat.

## Evidence

- **Issue has concrete current reproduction context:** The report names OpenClaw 2026.4.15, external plugin @tencent-weixin/openclaw-weixin@2.1.8, Windows_NT, a channel-scoped session key, and the exact mismatch: WeChat-originated messages deliver back to WeChat, but browser WebChat-originated messages in the same WeChat session only reply in the browser.
- **WebChat UI sends chat.send as session-only:** The Control UI/WebChat controller sends `chat.send` with `deliver: false` for normal chat messages, so the browser path does not request outbound channel delivery even when the selected session has external delivery metadata.

## Risks / Open Questions

- Changing this behavior could re-open the cross-channel leakage class that current WebChat route guards are explicitly preventing, so any fix likely needs an explicit opt-in route/permission model rather than unconditional deliveryContext inheritance.
- The external WeChat plugin is not in this repository, so maintainers may need plugin-side confirmation after any core Gateway/WebChat routing change.
