## Summary

Keep #10737 open. Current main still lacks a legacy `channels.imessage` auto-acknowledgment or native typing-feedback path before agent dispatch. BlueBubbles has typing support and is recommended for new iMessage setups, but it is a separate `channels.bluebubbles` implementation and does not satisfy the requested legacy `imsg` channel behavior. The issue discussion also includes a 2026-04-26 Codex review comment from steipete explicitly keeping this item open.

## Best Possible Solution

Keep #10737 open until maintainers either implement or explicitly decline legacy `channels.imessage` feedback. A concrete implementation should add typed config/schema/docs for `autoAck` or native typing with text fallback, send feedback after access/mention gating but before `dispatchInboundMessage`, honor `dmOnly`, and add regression coverage proving the acknowledgment is not reflected as an inbound user message. Continue recommending BlueBubbles for new iMessage deployments, but track legacy imsg behavior separately.

## Risks / Open Questions

- A text auto-ack would be a real outbound iMessage and must be integrated with sent-message cache/reflection guards so the acknowledgment is not re-ingested as inbound user text.
- If maintainers choose native legacy imsg typing instead of text fallback, upstream imsg behavior should be revalidated before wiring it into the channel lifecycle.
- The older related #4065 is closed, but it does not currently track the remaining legacy `channels.imessage` implementation gap shown in current main.
