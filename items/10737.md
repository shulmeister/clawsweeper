## Summary

Keep open. Current `main` does not implement iMessage auto-ack or native typing for the legacy `imessage` channel, so the feature request is still actionable.

## Evidence

- **Legacy iMessage docs still describe only the `imsg rpc` path:** The channel docs still present `imessage` as the legacy `imsg` integration and recommend BlueBubbles for new setups; there is no documented `autoAck` or typing option for `channels.imessage`.
  - file: [docs/channels/imessage.md:17](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/channels/imessage.md#L17)
  - command: `nl -ba docs/channels/imessage.md | sed -n '1,120p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Inbound iMessage reply pipeline has no typing hook:** The monitor builds the reply pipeline with only `cfg`, `agentId`, `channel`, and `accountId`, then dispatches replies with only `onModelSelected`; no `typingCallbacks`, `onReplyStart`, or auto-ack behavior is wired before agent processing.
  - file: [extensions/imessage/src/monitor/monitor-provider.ts:437](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/monitor/monitor-provider.ts#L437)

  - file: [extensions/imessage/src/send.ts:181](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/send.ts#L181)
