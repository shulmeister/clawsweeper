## Summary

Keep #38212 open. Current main still sends the configured `codex-cli/gpt-5.4` default from the session-memory slug helper directly into `runEmbeddedPiAgent`; the embedded runner still resolves the model before auth/profile alias handling, and the OpenAI plugin does not expose `codex-cli` as a provider/hook alias for the `openai-codex` dynamic model resolver. Adjacent auth alias and GPT-5.4 runtime hardening are present, but they do not close the exact helper-path failure described here.

## Best Possible Solution

Keep the issue open. A maintainer-quality fix should add focused coverage for `generateSlugViaLLM` with `codex-cli/gpt-5.4`, then either route slug-helper runs through the same CLI backend path used by normal replies or explicitly normalize CLI-backed Codex helper runs to the supported `openai-codex` embedded model/auth route.
