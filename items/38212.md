## Summary

Keep #38212 open. Current main has adjacent GPT-5.4/runtime/auth hardening and legacy runtime migration, but I did not find source or regression-test evidence that the reported raw `codex-cli/gpt-5.4` helper/embedded path now resolves and authenticates like `openai-codex/gpt-5.4`. The slug helper still forwards the resolved provider/model into the embedded runner unchanged, and `codex-cli` remains modeled as a CLI backend rather than an OpenAI Codex provider alias for model resolution.

## Best Possible Solution

Keep #38212 open until maintainers choose and prove one product path. A maintainer-quality fix should either normalize legacy `codex-cli/*` helper/embedded runs to the intended `openai-codex/*` Codex OAuth model/auth route and add a focused session-memory slug regression, or explicitly enforce and document that legacy `codex-cli/*` primary refs are unsupported for PI helper hooks and must be migrated to canonical `openai/*` plus runtime configuration, with a regression proving the loaded config cannot reach the helper as raw `codex-cli/gpt-5.4`.
