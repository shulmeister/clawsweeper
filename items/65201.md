## Summary

Keep open. Current `main` still has the false-positive path: `doctor` checks raw `gateway.auth.token` config, detects the SecretRef, but never resolves the active gateway auth SecretRef before deciding `needsToken`, so it can still warn even when gateway startup/runtime can use the token successfully.

## Evidence

- **Doctor health path still warns from unresolved raw config:** `runGatewayAuthHealth` reads `ctx.cfg.gateway?.auth?.token`, records that it is a SecretRef, then calls `resolveGatewayAuth` on the raw auth config. If `auth.token` is absent after that call, `needsToken` stays true and the warning is emitted.
  - file: [src/flows/doctor-health-contributions.ts:118](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/flows/doctor-health-contributions.ts#L118)
  - command: `nl -ba src/flows/doctor-health-contributions.ts | sed -n '100,190p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Gateway auth resolver does not resolve SecretRefs itself:** `resolveGatewayAuth` detects SecretRef inputs with `resolveSecretInputRef`, then intentionally drops those config values by passing `undefined` into credential resolution (`configToken: tokenRef ? undefined : authConfig.token`). That means a SecretRef-backed token stays unavailable on this code path unless another caller materializes it first.
  - file: [src/gateway/auth-resolve.ts:61](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/auth-resolve.ts#L61)
  - command: `nl -ba src/gateway/auth-resolve.ts | sed -n '1,220p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Gateway startup/runtime does materialize auth SecretRefs first:** `ensureGatewayStartupAuth` resolves gateway token/password SecretRefs up front via `resolveGatewayTokenSecretRefValue` / `resolveGatewayPasswordSecretRefValue`, builds an override with the resolved string values, and only then calls `resolveGatewayAuth`. That explains how runtime can succeed while `doctor` still warns.
  - file: [src/gateway/startup-auth.ts:160](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/startup-auth.ts#L160)
  - command: `nl -ba src/gateway/startup-auth.ts | sed -n '130,210p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Current tests cover runtime resolution, not the doctor regression:** The secrets runtime test suite explicitly verifies that an env-backed `gateway.auth.token` SecretRef resolves and produces no warning for active token mode, while the existing doctor e2e assertion only covers the missing-env warning case. There is no landed doctor regression test showing the reported false positive is fixed.
  - file: [src/secrets/runtime-gateway-local-surfaces.test.ts:80](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/secrets/runtime-gateway-local-surfaces.test.ts#L80)
