## Summary

Keep #7540 open. Current main still does not subscribe the WhatsApp/Baileys socket to the `call` event, and there is no WhatsApp call-tracking config, type, docs, manifest, hook, or test surface. The pinned Baileys version exposes `call: WACallEvent[]`, and maintainer discussion already marked the request still valid, so this remains an actionable bundled WhatsApp plugin enhancement rather than a cleanup close.

## Best Possible Solution

Keep the issue open and implement it in the bundled WhatsApp plugin. Add an opt-in channel/account setting if maintainers want the default off, subscribe to Baileys `call`, normalize and sanitize the call payload, apply the same account and access-control boundaries as inbound messages, expose call/contact activity through a documented hook/session-event/tool surface, and add focused config schema, type, docs, and tests.
