## Summary

Keep #7540 open. Current main still only wires WhatsApp/Baileys credential, connection, and message events; it does not subscribe to Baileys `call` events or expose WhatsApp call tracking through config, hooks, tools, docs, or tests. Recent maintainer discussion also explicitly marked the issue valid and WhatsApp-plugin-owned.

## Best Possible Solution

Keep the issue open and implement the feature in the bundled WhatsApp plugin. Add an opt-in channel/account setting if maintainers want the default off, subscribe to Baileys `call`, normalize and sanitize call payloads, apply the same account and access-control boundaries used for inbound messages, expose contact activity through a documented hook/session-event/tool surface, and add focused schema, type, docs, and tests.
