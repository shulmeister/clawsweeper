## Summary

Keep this PR open. Current main still lacks a debouncer-wide drain before WhatsApp socket teardown, so the reported close-path loss is not implemented on main. The PR is a plausible direction, but the provided review discussion correctly flags that the current `flushAll` shape needs quiescence handling for late enqueues and active keyed work before it should merge.

## Best Possible Solution

Keep this PR open for author or maintainer follow-up. The best fix is a debouncer-wide drain/quiescence API that flushes all current buffers, waits for active keyed work, accounts for late enqueues from already-running WhatsApp upsert callbacks, and is called from WhatsApp close after detaching inbound listeners but before closing the socket. Add focused regression coverage that proves pending debounced WhatsApp messages can still send `reply` or `sendMedia` before socket teardown, including late-enqueue and in-flight keyed-task cases.
