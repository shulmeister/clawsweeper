## Summary

Keep this PR open. Current main still does not implement the proposed `registerProviderRuntimeAuthOverride` plugin API or an equivalent built-in-provider runtime auth override seam. The existing code has provider-owned runtime auth hooks and public auth lookup helpers, but those require owning/resolving the provider plugin path rather than letting an external plugin supply per-call auth, base URL, or headers for built-in providers like OpenAI/Anthropic/OpenRouter. The PR discussion also shows active review feedback that was partly addressed and later review concerns that still deserve maintainer review, not cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a revised additive core/plugin-SDK seam for external runtime auth overrides, or explicitly choose a different core seam that lets trusted plugins supply per-call credentials, base URL, and request headers for existing providers without replacing provider implementations. Before merge, resolve the remaining review points and add/keep focused coverage for override ordering, throw logging, mode validation, reentrancy, provider alias normalization, modelId propagation, availability semantics, and side-path fallback behavior.
