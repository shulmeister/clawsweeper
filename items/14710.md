## Summary

Echo Guard is not implemented on current main, but the request fits OpenClaw’s plugin/ClawHub extension model rather than core. Main already provides shared prompt-injection hardening primitives and explicitly reserves optional capability for plugins.

## Risks / Open Questions

- A plugin implementation would still need careful threshold tuning and false-positive handling for code-heavy, multilingual, or formatting-sensitive inputs.
- Closing as plugin-scope means core will not provide a built-in, universally enforced echo-screening layer.

## Close Comment

Closing this as `clawhub` after a Codex review of current `main`.

The request is coherent, but it fits OpenClaw’s plugin/ClawHub extension path rather than core. `VISION.md:54` says optional capability should usually ship as plugins, and current main already exposes the right hook surfaces for this kind of policy: `before_agent_reply` can intercept the cleaned user message and short-circuit with a synthetic/blocking reply (`src/auto-reply/reply/get-reply.ts:567`), while the plugin hook API also exposes `before_prompt_build` and `llm_input` (`src/plugins/hooks.ts:530`, `src/plugins/hook-types.ts:167`).

Core already ships shared prompt-injection hardening primitives in `src/security/external-content.ts:14` and documents the project’s layered model in `docs/gateway/security/index.md:559` and `SECURITY.md:197`. Given the extra latency, tuning, and policy tradeoffs in an echo-check design, this is better as an optional plugin than a built-in core feature.

If someone wants to pursue Echo Guard, the right next step is a ClawHub/plugin implementation on top of those existing hook surfaces.
