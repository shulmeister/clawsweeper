## Summary

Keep open. Current `main` still restarts the Windows Scheduled Task by ending the task, immediately killing gateway listener processes, waiting for the port to clear, and then re-running the task. The graceful drain path exists in the gateway run loop, but the Windows service restart path still races or bypasses it, and the CLI/service restart path does not write a restart sentinel for session recovery.

## Evidence

- **Windows restart path is still forceful:** `restartScheduledTask()` calls `schtasks /End`, then immediately kills scheduled-task gateway listeners and any installed Startup-folder runtime, waits for the port to clear, and only then runs `schtasks /Run`. That is a stop/kill/relaunch sequence, not an idle-drain restart.

- **Tests encode the force-kill behavior on restart:** The Windows Scheduled Task restart tests explicitly expect restart cleanup to kill lingering verified gateway listeners before relaunch.
  - file: [src/daemon/schtasks.stop.test.ts:171](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/daemon/schtasks.stop.test.ts#L171)

This is the graceful mechanism the issue is asking to preserve.

  - file: [src/cli/gateway-cli/run-loop.test.ts:231](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/gateway-cli/run-loop.test.ts#L231)
