## Summary

Keep open. Current main still does not implement context-overflow-triggered model fallback: likely context overflow is explicitly rethrown by the model fallback loop, while the embedded runner handles overflow locally with compaction/tool-result truncation and then returns a terminal context_overflow error telling the user to reset or choose a larger-context model. The docs and tests codify that behavior. The related issue context does not supersede this: #28632 is output-token length, #34005 is graceful Ollama overflow handling, and #29014 covered overloaded/retry fallback rather than selecting a larger configured fallback on context overflow. The latest maintainer/Codex comment on #9986 already reached the same keep-open conclusion, and the only relevant path changes since that review are unrelated diagnostic/cron context edits.

## Best Possible Solution

Keep #9986 open until maintainers either explicitly decide the documented policy is intentional or implement a guarded context-overflow fallback path. The likely implementation belongs around the embedded runner overflow exhaustion branch and model-fallback policy: after compaction and tool-result truncation are exhausted, only replay when safe, select a configured fallback with sufficient context headroom, log the transition, and add docs/tests for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- A naive context-overflow fallback could replay side-effecting turns or completed tool work; implementation needs replay-safety and side-effect gating.
- Selecting a larger fallback requires reliable context-window metadata for both primary and fallback models, plus tests for equal or smaller fallback windows.
