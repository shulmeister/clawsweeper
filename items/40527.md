## Summary

Keep open. Current main improved the original root-cause story because the macOS node now advertises and handles `system.which`/`system.run`, but the reported timeout class is still plausible and not clearly fixed: gateway node connect still starts `refreshRemoteNodeBins()` immediately, the probe still waits up to 15s and logs the same timeout warning, and the Swift gateway client ignores non-response frames while waiting for the connect response before starting its normal listener. That matches the April 2 comment's narrower connect-timing diagnosis.

## Best Possible Solution

Keep #40527 open and fix it in the gateway/macOS node handshake path. The likely product fix is to defer the initial remote skill bin probe until after the node has completed its connect response/listener setup, or make the macOS client queue/handle early `node.invoke.request` frames during connect. A targeted follow-up should also decide whether co-located macOS app nodes should be skipped when the gateway already has local shell access. Add regression coverage around node connect plus remote-bin probing so `system.which` probes cannot be sent before the node can process events.
