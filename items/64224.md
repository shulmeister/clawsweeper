## Summary

Keep open. Current main still treats billing failures as profile-wide disables, not model-scoped cooldowns, so the central LiteLLM/proxy problem remains. Related #66314/#66315 addressed the user-facing message for pure billing cooldown exhaustion, but current source, tests, and docs still intentionally skip same-provider sibling models when the provider profile is disabled for billing.

## Best Possible Solution

Keep this issue open and fix it in the auth/fallback core, likely with a deliberate model-scoped billing/proxy policy rather than more message-only handling. A strong implementation would add either model-scoped billing disable state, a proxy-aware classification path for LiteLLM/OpenAI-compatible upstream 402s, or an explicit `billingBackoffHours: 0`/disable option, then cover a single `litellm` profile with two same-provider models where billing on one model does not block the other. Update `docs/concepts/model-failover.md`, `docs/providers/litellm.md`, and config schema docs if the contract changes.
