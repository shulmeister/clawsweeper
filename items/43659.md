## Summary

Keep open. Current main still has the reported GitHub Copilot token-cache behavior: `resolveCopilotApiToken()` reads one shared `github-copilot.token.json` cache entry and returns any unexpired cached Copilot API token without checking which GitHub auth profile/token produced it. The PR’s concept is still relevant, but it needs to be ported to the current `src/agents/github-copilot-token.ts` / plugin SDK path rather than the older `src/providers` path shown in the PR context.

## Best Possible Solution

Keep the PR open, but ask for a current-main port of the same behavior in `src/agents/github-copilot-token.ts` with focused tests covering same-profile cache hit, different-profile cache miss, and legacy cache miss. A stronger implementation would use per-fingerprint cache slots to avoid evicting valid tokens on every profile alternation, but the minimum fix is to prevent cross-profile reuse of a Copilot API token after rate-limit rotation.

## Evidence

The `githubToken` parameter is only used later in the fetch Authorization header, so a valid cached Copilot token bypasses a rotated profile's GitHub token.
