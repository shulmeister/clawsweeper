## Summary

Current main already handles the user-visible stuck TUI activity problem: normal late terminal chat events with no active run are adopted before final/error/aborted handling, and the merged #67401 watchdog covers lost or late final events that would otherwise leave the TUI streaming indefinitely. The remaining PR fallback would also affect local /btw terminal events, which conflicts with the documented/tested side-result isolation contract.

## Best Possible Solution

Close this PR and keep the shipped current-main behavior: normal late terminal events already reset through the active-run path, #67401 provides the broader watchdog safety net, and any future regression test for late terminal events should preserve the existing /btw isolation instead of adding a blanket `activeChatRunId == null` fallback.

## Evidence

- **Late terminal events reset through current active-run adoption:** When a chat event arrives and no active run is set, current main assigns `evt.runId` to `state.activeChatRunId` unless it is a local BTW run. The final/error/aborted handlers then compute `wasActiveRun` after that assignment and call `finalizeRun`/`terminateRun`, whose existing `wasActiveRun` branch updates the activity status.
