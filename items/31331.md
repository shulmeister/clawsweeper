## Summary

Keep #31331 open. Current main still builds Docker sandbox and sandbox-browser workspace bind mounts from gateway-visible paths and passes them directly to Docker. The Docker install path still maps host storage to `/home/node/.openclaw/...` inside the gateway container, so Docker-out-of-Docker setups can still hit the reported host/container path mismatch unless users manually configure host-path parity. The issue is also the canonical `dedupe:parent` with recent confirmations and closed-unmerged fix PRs.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace and agent workspace bind sources through the gateway container mount table before validation, config hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none`. If maintainers instead want host-path parity as the supported contract, the Docker setup/onboarding path should generate or validate that parity and fail clearly when absent; documentation alone should not close the bug.
