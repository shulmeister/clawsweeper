## Summary

Keep #31331 open. Current main still passes gateway-visible workspace paths directly into Docker sandbox and sandbox-browser bind mounts without remapping them to host Docker paths for Docker-outside-of-Docker. The Docker install flow still mounts host storage at /home/node/.openclaw/workspace and only toggles sandbox mode/scope/workspaceAccess during sandbox bootstrap, while the issue remains canonical with recent same-problem reports and the related fix PRs closed unmerged.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/DooD sandbox workspaceAccess bind issue. Best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace, sandbox workspace, and agent workspace bind sources through the gateway container mount table before validation, hashing, and docker create; add regression coverage for workspaceAccess rw, ro, and none plus browser containers. If maintainers prefer host-path parity as the contract, update Docker setup/onboarding to set host-canonical workspace paths and identical volume maps by default, with docs/tests proving the Docker install sandbox path works.
