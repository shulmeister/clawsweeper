## Summary

Keep open. Current `main` still does not apply the completion-time `MIN_REFIRE_GAP_MS` safeguard to `every` schedules, so the PR's bugfix is not already implemented on `main` and there is not strong evidence for any close reason.

## Evidence

- **Current completion path is still cron-only:** In `applyJobResult`, the minimum refire gap is enforced only when `job.schedule.kind === "cron"`; the `else` branch for non-cron jobs still assigns `job.state.nextRunAtMs = naturalNext` with no lower bound.
  - file: [src/cron/service/timer.ts:561](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/service/timer.ts#L561)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '548,580p'`
- **`every` jobs can still compute a next run equal to `endedAt`:** `computeJobNextRunAtMs` uses `if (nextFromLastRun > nowMs)` for `every` jobs. When the interval lands exactly on `nowMs`, it falls through to anchor-based scheduling, and `computeNextRunAtMs` can return `nowMs` exactly on an interval boundary.
  - file: [src/cron/service/jobs.ts:255](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/service/jobs.ts#L255)
  - command: `sed -n '240,270p' src/cron/service/jobs.ts`
- **Anchor scheduling returns the current boundary, not a future one:** For `every` schedules, `computeNextRunAtMs` calculates `steps = Math.max(1, Math.floor((elapsed + everyMs - 1) / everyMs))`; when `nowMs` is exactly on an anchored boundary, that resolves to the current boundary, so `naturalNext` can equal `endedAt`.
  - file: [src/cron/schedule.ts:95](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/schedule.ts#L95)
  - command: `sed -n '88,104p' src/cron/schedule.ts`
- **Timer floor does not fix the stored schedule:** `armTimer` only floors a zero-delay wakeup to `MIN_REFIRE_GAP_MS`; it does not rewrite `nextRunAtMs`. If an `every` job stored `nextRunAtMs <= now`, it will still be due again on the next wake.
  - file: [src/cron/service/timer.ts:649](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/service/timer.ts#L649)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '646,670p'`
- **Main has regression coverage for cron refire only, not this `every` case:** Current tests include `enforces a minimum refire gap for second-granularity cron schedules (#17821)`, but there is no corresponding `every`-schedule regression test from this PR on `main`.
  - file: [src/cron/service/timer.regression.test.ts:439](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/service/timer.regression.test.ts#L439)
  - command: `sed -n '430,580p' src/cron/service/timer.regression.test.ts`
