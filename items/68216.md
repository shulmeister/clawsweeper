## Summary

Keep open. Current main does not implement the reported Gemini CLI write/edit fix: the bundled google-gemini-cli backend passes --skip-trust but not --yolo or --approval-mode auto_edit, and OpenClaw forwards those args directly to the Gemini subprocess. Upstream Gemini CLI treats --skip-trust as folder trust only; write_file/edit/run_shell_command remain denied/excluded in non-interactive --prompt mode unless a permissive approval mode is selected. The linked fix PR #68366 was relevant and well-scoped, but it was closed unmerged, and no open canonical successor was found in the related context/searches.

## Best Possible Solution

Keep the issue open and resolve it with explicit maintainer product judgment. The conservative technical path is to evaluate --skip-trust plus --approval-mode auto_edit for google-gemini-cli fresh and resume args, because upstream Gemini CLI allows write_file/edit in autoEdit mode without enabling the full YOLO allow-all shell/tool posture. If maintainers intentionally want full CLI parity instead, revive the #68366 approach with updated security documentation and tests.

## Risks / Open Questions

- The product/security tradeoff remains real: full --yolo would restore built-in write/edit/shell tools but also lets the Gemini subprocess run its own tools outside OpenClaw-mediated tool policy, as #68366's Aisle review noted.
- Docs currently drift from source in docs/gateway/cli-backends.md: the docs list google-gemini-cli args without --skip-trust, so whichever final behavior is chosen should update the docs and tests together.
