## Summary

Keep open. Current main has useful mitigations for same-path config clobbers and service-env carryover, but the reported HOME-change update path is still not fixed: update service refresh still runs `gateway install --force` from the current invocation environment, and forced reinstall still filters the previously installed service `HOME` and `OPENCLAW_*` roots instead of preserving, migrating, warning, or blocking.

## Best Possible Solution

Keep this issue open and fix the service refresh/install path directly. Before regenerating an installed gateway service during update, compare the previous effective root (`HOME`, `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`) with the update invocation root; preserve the prior root when safe, or produce an explicit migration/warning/block instead of silently creating a fresh config under the new HOME. Add a regression test for a service installed with `HOME=/home/openclaw` and refreshed by an update process running with `HOME=/root`.

## Evidence

- **Issue discussion identifies the relevant failure mode:** The report describes a service unit changing from `HOME=/home/openclaw` to `HOME=/root`, abandoning `/home/openclaw/.openclaw/openclaw.json`; the first comment independently points to root/HOME switching causing config fork. The later GHCR version-tag comment is rollback feedback and does not supersede the service/config-root bug.
- **Update refresh force-runs gateway install from invocation env:** `refreshGatewayServiceEnv` builds `gateway install --force` and passes only `resolveServiceRefreshEnv(params.env ?? process.env, params.invocationCwd)` to the updated entrypoint. That helper only resolves already-present `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, and `OPENCLAW_CONFIG_PATH`; it does not read the existing service `HOME` or config root.
