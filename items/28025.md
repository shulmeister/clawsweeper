## Summary

Keep #28025 open. Current main still does not implement Slack `thread.replyOnToolUse` or equivalent tool-use-gated reply routing. Slack reply placement is controlled by `replyToMode`, incoming thread state, and explicit reply targets; tool callbacks only update progress/status after the delivery plan is already created. The prior ClawSweeper comment reached the same conclusion on an earlier SHA, and current main remains unchanged on the requested behavior.

## Best Possible Solution

Keep the issue open until Slack gets an explicit, documented tool-use-aware reply routing option or a maintainer decides the product should stay with `replyToMode`/manual thread controls only. A likely implementation would add a Slack config/schema/docs surface, carry a per-turn tool-use signal into Slack delivery before the first final/stream delivery decision, and cover both DM and channel cases with focused Slack monitor/replies tests.
