## Summary

Keep #28025 open. Current main still does not implement `channels.slack.thread.replyOnToolUse` or an equivalent Slack policy that routes replies into a thread only after tool use; Slack reply routing is still driven by `replyToMode`, existing thread state, and streaming/delivery planning made before tool callbacks run.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If accepted, implement a default-false typed `channels.slack.thread.replyOnToolUse` option in the bundled Slack plugin/config schema, carry a reliable per-turn tool-use signal into Slack reply planning before final delivery or streaming commits to a target, preserve explicit thread replies and `thread.inheritParent`, and add Slack tests for DMs, channels, existing threads, `replyToMode` interactions, streaming, and tool/progress callbacks.
