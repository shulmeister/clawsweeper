## Summary

Keep open. Current main has partial delivery diagnostics, but it does not expose normalized delivery failure reasons across cron and messaging/task runs. Cron still stores a coarse execution status plus a four-value delivery status and raw delivery error text; messaging and background-task delivery also surface coarse statuses/errors rather than shared reason codes such as not_configured, channel_unavailable, attempted_failed, or rate_limited. Related #49190 remains open for the adjacent cron execution-vs-delivery warning split, but it does not fully cover this broader reason-normalization request.

## Best Possible Solution

Keep this issue open. The best path is to add an additive normalized delivery outcome model shared by cron run logs, message action results, background task delivery, Gateway protocol schemas, Control UI filters/cards, CLI JSON/text output, and docs. Preserve existing fields for compatibility, add reason codes such as not_configured, channel_unavailable, skipped, attempted_failed, and rate_limited, and include tests that cover cron target-resolution failures, outbound adapter configuration failures, provider/rate-limit errors, best-effort skips, task terminal delivery fallback, and UI/API filtering.
