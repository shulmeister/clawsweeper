## Summary

Keep open. The report is recent, detailed, and not protected/maintainer-authored, but current main does not clearly implement a fix or identify a canonical duplicate for the steady ~12s `openai-codex` OAuth server-side latency. Main has relevant transport/cache/fast-mode controls, but they predate or do not conclusively resolve this report.

## Best Possible Solution

Keep this issue open for maintainer/provider investigation. The next useful path is to reproduce on current main with `openai-codex/gpt-5.4` using explicit `transport` and `fastMode/serviceTier` variants, capture diagnostic model-call first-response/total duration plus provider request IDs or upstream processing headers if available, and document whether standard Codex OAuth has an expected baseline or whether OpenClaw should change defaults, diagnostics, or fallback guidance.
