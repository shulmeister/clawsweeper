## Summary

Keep open. Current `main` does not implement the PR’s loopback-specific handshake timeout, and the PR’s approach is stale against the current gateway timeout layout and still appears semantically risky for proxied deployments because current gateway client-IP resolution is trusted-proxy-aware while the PR classifies by raw socket loopback.

## Evidence

- **Current main still uses one pre-auth handshake timeout for all WS clients:** `src/gateway/handshake-timeouts.ts:1` defines a single `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS = 10_000`, and `getPreauthHandshakeTimeoutMsFromEnv()` only reads env overrides before returning that shared default. `src/gateway/server/ws-connection.ts:366` uses that single value directly for the pre-auth handshake timer.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/handshake-timeouts.ts#L1)
  - command: `nl -ba src/gateway/handshake-timeouts.ts | sed -n '1,80p'; nl -ba src/gateway/server/ws-connection.ts | sed -n '360,400p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)

## Risks / Open Questions

- The original loopback-under-load problem may still be real on current `main`; this review did not run a live reproduction because the task required read-only inspection.
- If maintainers still want this behavior, the fix should be rebased onto `src/gateway/handshake-timeouts.ts` and should derive loopback/local status from the same trusted-proxy-aware client resolution used in `src/gateway/server/ws-connection/message-handler.ts`, not raw `remoteAddr` alone.
