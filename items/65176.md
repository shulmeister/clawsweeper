## Summary

Keep open. Current `main` does not include the SSRF hint, and the PR as opened is not merge-ready: it hardcodes a Telegram channel config path into shared SSRF error text that is also used by provider and browser/web-tool flows, and it also includes an unrelated cron-model change that already drew a P1 review finding.

## Evidence

- **Current main still uses the old generic resolved-IP message:** `BLOCKED_RESOLVED_IP_MESSAGE` on `main` is still exactly `Blocked: resolves to private/internal/special-use IP address`, so the PR is not already implemented.
  - file: [src/infra/net/ssrf.ts:202](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/infra/net/ssrf.ts#L202)

- **The shared SSRF error path is used by provider transport, not just channels:** Provider fetches go through `fetchWithSsrFGuard(...)`, and provider request config can opt into private-network access via `request.allowPrivateNetwork`. That means a hardcoded `channels.<plugin>.network.dangerouslyAllowPrivateNetwork` remediation would be wrong for provider failures.
  - file: [src/agents/provider-transport-fetch.ts:153](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/agents/provider-transport-fetch.ts#L153)
