## Summary

Current main still does not implement config-backed defaults for `openclaw tui --deliver`. The TUI CLI registers only an explicit `--deliver` flag defaulting to false, `runTui` falls back to false, send paths forward that fixed value, and the config type/schema/help surfaces still have no `tui.deliver` key. The related PRs #33198 and #33208 were closed unmerged, while #66991 and the linked workaround remain useful impact evidence rather than a canonical replacement for this persisted-default request.

## Best Possible Solution

Keep #33102 open until a maintainer-approved change adds a documented persisted TUI delivery default, likely `tui.deliver`, wires it through `OpenClawConfig`, strict zod/generated config schema/help, CLI parsing precedence, TUI runtime/settings behavior, and focused tests for config default, explicit CLI override, false fallback, and any `--no-deliver` opt-out semantics. Treat #33198, #33208, #66991, and the linked arcabot workaround as prior art and impact evidence, not as superseding canonical work.

## Risks / Open Questions

- A complete fix needs CLI precedence design for config default vs explicit `--deliver` and likely an explicit `--no-deliver` override.
- The TUI docs currently mention `/deliver on` and a settings delivery toggle, but current code does not expose those controls, so implementation should reconcile docs, settings UI, command handling, and config schema together.
- The linked stuck-spinner report #66991 is related impact evidence; maintainers may want to handle that UX failure in the same patch or a narrower follow-up.
