## Summary

Keep this issue open. Current main contains adjacent Telegram polling watchdog and transport-rebuild hardening, but the exact sticky IPv4 fallback path reported here still exists, maintainers already said the local healthy transport probe does not invalidate the India/macOS report, and there is no current-main affected-host retest showing the failure is gone.

## Best Possible Solution

Keep this issue open and request a current-main retest on the affected host with Telegram diagnostics enabled through a full stall/rebuild cycle. If an in-gateway probe using `resolveTelegramFetch` fails, focus the fix in `extensions/telegram/src/fetch.ts` around fallback diagnostics, dispatcher lifecycle, and error-chain logging; if the probe succeeds while gateway polling still wedges, continue in `extensions/telegram/src/polling-session.ts` around runner ownership and restart behavior.
