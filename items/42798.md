## Summary

Keep open. Current `main` still does the reported back-to-back startup sequence with no retry/backoff around `tailscale status --json`, and verbose exec logging would still emit `Command failed: tailscale status --json` on a transient failure.

## Evidence

- **Gateway startup still chains serve -> status immediately:** `startGatewayTailscaleExposure` enables Tailscale and then immediately calls `getTailnetHostname().catch(() => null)` in the same startup block, with no delay or retry.
  - file: [src/gateway/server-tailscale.ts:21](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/gateway/server-tailscale.ts#L21)
  - command: `nl -ba src/gateway/server-tailscale.ts | sed -n '1,200p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Hostname lookup still uses a single status call per candidate:** `getTailnetHostname` runs `status --json` once for each candidate binary and only records the last error; there is no transient-failure retry/backoff path.
  - file: [src/infra/tailscale.ts:110](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/infra/tailscale.ts#L110)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)

  - file: [src/process/exec.ts:161](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/process/exec.ts#L161)

  - file: [src/infra/tailscale.test.ts:47](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/infra/tailscale.test.ts#L47)
