## Summary

Keep open. Current main does not yet implement this PR's central Telegram-native safeguard: a Telegram-sourced approval request with no concrete account binding can still match every eligible Telegram account. The PR also has unresolved review risk around `mode: "both"`, so it should be rebased/fixed rather than auto-closed.

## Best Possible Solution

Keep #70568 open or replace it with a rebased fix that scopes Telegram native approval handling without regressing `mode: "both"`: treat Telegram-sourced requests with no `turnSourceAccountId` or same-channel session binding as ambiguous in multi-account configs, honor `approvals.exec.targets[].accountId` where target-mode native delivery is involved, and add focused regression tests in `extensions/telegram/src/exec-approvals.test.ts` and `extensions/telegram/src/approval-native.test.ts`.
