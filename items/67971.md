## Summary

Current main still matches the reported Discord startup failure. `fetchDiscordApplicationId()` only falls back to `parseApplicationIdFromToken()` when the fetch throws; a non-OK HTTP response such as 429 returns `undefined`, and the provider immediately throws `Failed to resolve Discord application id`. Existing tests cover token parsing and other Discord 429 paths, but not this application-id lookup fallback.

## Best Possible Solution

Keep the issue open and patch `extensions/discord/src/probe.ts` so `fetchDiscordApplicationId()` falls back to `parseApplicationIdFromToken()` for transient non-OK responses such as 429 and 5xx, while still returning `undefined` for 401/403. Add focused tests for 429 fallback, 5xx fallback, thrown transport fallback, and 401/403 fail-fast behavior, then rely on the existing provider throw when no valid application id can be resolved.

## Risks / Open Questions

- The fix should preserve fail-fast behavior for real credential failures such as 401 and 403 so invalid Discord tokens are not masked by token-derived IDs.
- A regression test should simulate non-OK responses directly around `fetchDiscordApplicationId()` because broader Discord 429 tests currently exercise other REST/send paths, not this startup lookup.
