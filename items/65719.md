## Summary

The reported `/clear` stale-history race does not match current `main`. The Control UI still reloads history after `sessions.reset`, but the gateway now swaps the session entry to a fresh `sessionId`/`sessionFile` and creates the new transcript before the RPC returns; `chat.history` then resolves the refreshed store entry, and gateway tests assert the reset path exposes the new transcript immediately under the store lock.

## Evidence

- **Control UI still reloads history after reset:** `clearChatHistory()` still calls `sessions.reset` and then immediately `loadChatHistory()`, so a stale-history repro on current main would require the gateway to return old transcript data after reset.
