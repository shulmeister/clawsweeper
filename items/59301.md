## Summary

Current `main` partially mitigates the reported loss mode by replaying interrupted recurring cron jobs on startup, but it does not implement the issue's requested restart-loop pause/defer behavior, and cron run logs still only record finished runs. The item should stay open.

## Evidence

- **Startup recovery replays recurring jobs immediately:** `start()` clears stale `runningAtMs` markers, then immediately calls `runMissedJobs(...)` during gateway startup. There is no restart-instability detector, uptime gate, or minimum-stable-window check before cron execution resumes.
  - file: [src/cron/service/ops.ts:87](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cron/service/ops.ts#L87)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '70,130p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Catch-up logic staggers overflow but does not pause for gateway stability:** Startup catch-up selects runnable missed jobs right away, runs up to `maxMissedJobsPerRestart`, and reschedules only the overflow by setting future `nextRunAtMs` offsets. That is load-smoothing, not restart-loop deferral based on repeated restarts or uptime stability.
  - file: [src/cron/service/timer.ts:962](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cron/service/timer.ts#L962)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '860,1125p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Tests cover replay/catch-up semantics, not pause-until-stable behavior:** Regression coverage proves interrupted recurring jobs are replayed on first restart and deferred missed jobs are staggered from the post-catchup clock, but there is no test asserting cron suppression during repeated gateway restart loops or until a stable uptime window elapses.
  - file: [src/cron/service.restart-catchup.test.ts:118](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cron/service.restart-catchup.test.ts#L118)
  - command: `sed -n '1,260p' src/cron/service.restart-catchup.test.ts`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **`cron runs` still depends on finished-event logging:** Gateway cron event handling appends run-log entries only for `action: "finished"`, and the run-log type/parser is defined around finished entries. Interrupted runs still do not produce a durable `cron runs --id` record at start time.
  - file: [src/gateway/server-cron.ts:549](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server-cron.ts#L549)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '520,590p'`

- **Run-log parser explicitly ignores non-finished entries:** The run-log schema only models `action: "finished"`, and the parser test confirms `action: "started"` lines are ignored. That means the issue's alternative 'write started before execution' approach is not present on current main.
  - file: [src/cron/run-log.ts:18](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cron/run-log.ts#L18)
