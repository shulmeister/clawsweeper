## Summary

Keep this issue open. Current main still lacks an enforced DM-level toolsBySender policy for allowlisted DM senders: the shared tool policy path is group/channel scoped, direct session keys are excluded from that resolver, Signal only exposes toolsBySender under groups, and Telegram direct toolsBySender config exists in schema/types but is not wired into effective tool filtering.

## Best Possible Solution

Implement a generic direct-message sender tool-policy contract. The maintainer path should add typed config/schema/docs for a DM-level policy surface, resolve direct sender identity consistently for Signal and Telegram, pass that policy into createOpenClawCodingTools and final bundled-tool filtering alongside global/agent/group policies, and add regression tests proving one allowlisted DM sender can retain tools while another allowlisted sender is denied them.
