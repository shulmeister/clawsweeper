## Summary

Keep #43816 open. Current main still uses a fixed 2-hour stale `runningAtMs` cleanup threshold in the timer/maintenance path, while actual cron execution already uses per-job timeout policy. The PR is a focused, plausible fix for the remaining mismatch and its review discussion addressed the no-timeout fallback concern by keeping the 2-hour fallback for `timeoutSeconds <= 0`.

## Best Possible Solution

Keep the PR open and refine it into a maintainer-ready cron fix: reuse `resolveCronJobTimeoutMs` for stuck-lock cleanup, preserve the 2-hour fallback for no-timeout jobs, clamp overflow-sized thresholds, and add explicit tests for the positive-timeout, no-timeout, and overflow cases before merge.
