## Summary

Keep open. Current main still has the allowlist fallback that treats an unregistered primary CLI token as a plugin id, and a maintainer has explicitly kept #64732 open as the canonical tracker for this class of CLI command/plugin-id confusion.

## Best Possible Solution

Keep #64732 open. The best fix is the maintainer-described command-classification layer before the plugin allowlist fallback: framework/core commands should not suggest `plugins.allow`, manifest aliases should point to the owning plugin id, runtime slash aliases should explain chat-only usage or the owning CLI command, and unknown/stale commands should report that no command was registered. Add focused regression tests for help, tool, run, subagents, and the adjacent `/lossless`/manifest-alias behavior.
