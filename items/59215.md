## Summary

Current `main` still appears to match the reported failure mode: backend exec-tool gateway calls in `gateway.auth.mode="none"` are device-bound by default, but the loopback self-pairing bypass only applies to backend clients using token/password or device-token auth, not auth mode `none`. That means an unpaired backend device can still hit `pairing required` on loopback.

## Evidence

- **Docs still describe auth-none as valid open-loopback mode:** The public config/protocol docs say `gateway.auth.mode: "none"` is an explicit no-auth mode for trusted local loopback and that it skips shared-secret connect auth. That matches the reporter's expected behavior rather than documenting pairing as a required workaround for this path.
  - file: [docs/gateway/configuration-reference.md:308](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/gateway/configuration-reference.md#L308)
  - command: `nl -ba docs/gateway/configuration-reference.md | sed -n '300,320p'; nl -ba docs/gateway/protocol.md | sed -n '508,522p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Agent exec path uses backend gateway client mode:** `callGatewayTool()` sends agent tool RPCs through `callGateway()` with client name `GATEWAY_CLIENT`, display name `agent`, and mode `BACKEND`, which is the same backend path implicated by the issue.
  - file: [src/agents/tools/gateway.ts:165](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/tools/gateway.ts#L165)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Gateway calls still attach a device identity by default:** `callGateway()` creates the client with `deviceIdentity` resolved from local state whenever the caller does not explicitly pass `deviceIdentity: null`. In auth-none mode, the resolved credentials path can be empty, so the backend still connects as a device-bearing client.
  - file: [src/gateway/call.ts:515](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/call.ts#L515)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Local backend self-pairing bypass excludes auth method none:** `shouldSkipLocalBackendSelfPairing()` only returns true for direct-local backend clients when the auth method is token/password with shared auth or `device-token`. It does not include auth method `none`.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.ts:262](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server/ws-connection/handshake-auth-helpers.ts#L262)

- **Unpaired device identities still get forced into pairing when the bypass does not apply:** During websocket connect handling, if a device identity is present and not already paired, the server calls `requirePairing("not-paired")` unless `skipLocalBackendSelfPairing` or the control-UI-specific skip applies. That is the direct source of a `pairing required` close for auth-none backend clients.
  - file: [src/gateway/server/ws-connection/message-handler.ts:1066](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server/ws-connection/message-handler.ts#L1066)

- **Current tests do not cover the reported auth-none backend case:** There is a test proving explicit `mode: "none"` works for a loopback connect with no shared secret and no device, and a separate test proving backend loopback shared-auth connections work without pairing when a token is used. I did not find a current test covering backend loopback plus `auth.mode="none"` plus default device identity, which is the reported failing combination.
