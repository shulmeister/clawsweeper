## Summary

Current `main` does not implement multi-window parallel desktop chat sessions. The macOS desktop chat manager is still a singleton: it keeps one window controller and explicitly closes the existing chat window when `show(sessionKey:)` is called for a different session. OpenClaw's chat/session surface remains session-oriented and non-blocking, so this reads as a real product gap, not a stale, incoherent, or ClawHub-only request.

## Evidence

- **Singleton desktop chat manager:** `WebChatManager` stores a single `windowController`/`windowSessionKey` pair and a single `panelController`/`panelSessionKey` pair, so the desktop app is architected around one chat window plus one panel, not multiple parallel chat windows.
  - file: [apps/macos/Sources/OpenClaw/WebChatManager.swift:29](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/apps/macos/Sources/OpenClaw/WebChatManager.swift#L29)
  - command: `nl -ba apps/macos/Sources/OpenClaw/WebChatManager.swift | sed -n '20,120p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Opening another session replaces the existing window:** `show(sessionKey:)` checks whether an existing window is open, and if the requested session key differs it calls `controller.close()` before creating the new window controller. That matches the reported behavior that a new conversation window displaces the current one instead of allowing parallel windows.
  - file: [apps/macos/Sources/OpenClaw/WebChatManager.swift:41](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/apps/macos/Sources/OpenClaw/WebChatManager.swift#L41)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Sessions are first-class and chat runs are non-blocking:** The Control UI docs still describe session-scoped chat operations (`sessions.list`, `sessions.patch`) and non-blocking `chat.send` runs. That indicates the product already models separate sessions; the missing piece is desktop multi-window UI support, not an out-of-scope plugin feature.
