## Summary

Keep open. Current main offers adjacent primitives for the problem space: one Gateway can already handle multiple agents, remote work can be delegated to connected node hosts, and memory is intentionally plugin-backed with options that can use a dedicated service or remote object storage. But the exact feature requested in #57202 — multiple Gateway nodes connected together with one Gateway dedicated to memory storage/processing — is not implemented on current `main`, and the issue is still a coherent, recent feature request.

## Evidence

- **Single-Gateway model for multi-agent setups:** The docs say most setups should use one Gateway because a single Gateway can handle multiple messaging connections and agents; extra gateways are described as isolated profiles/ports for isolation or rescue-bot redundancy, not a shared clustered control plane.
  - file: [docs/gateway/multiple-gateways.md:9](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/gateway/multiple-gateways.md#L9)
  - command: `nl -ba docs/gateway/multiple-gateways.md | sed -n '1,40p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Remote topology is one Gateway plus nodes, not Gateway-to-Gateway clustering:** The remote-access docs define the supported topology as a single Gateway on a dedicated host, with laptops and nodes connecting to it. They explicitly say one gateway service owns state and channels, nodes are peripherals, and nodes do not run the gateway service.
  - file: [docs/gateway/remote.md:8](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/gateway/remote.md#L8)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Distributed execution exists via node hosts:** `openclaw node` is documented as a headless node host that connects to the Gateway WebSocket and exposes `system.run` / `system.which` on another machine, which covers remote execution distribution but not remote memory ownership by another Gateway.
  - file: [docs/cli/node.md:11](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/cli/node.md#L11)
  - command: `nl -ba docs/cli/node.md | sed -n '1,70p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Memory is intentionally a plugin slot:** The project vision says optional capability should usually ship as plugins and identifies memory as a special plugin slot with multiple memory options, which supports the idea that memory scaling/backends belong on extensible memory surfaces rather than new core-specific distributed-Gateway behavior by default.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '50,80p'`

Those are meaningful adjacent solutions for offloading memory/storage pressure, but they are not the same as the requested Gateway-to-Gateway distributed deployment.
