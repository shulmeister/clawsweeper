## Summary

Current main already addresses the reported Discord regressions: the message tool no longer requires Discord `components`, component+media sends take the classic media path instead of silently dropping attachments when the payload is representable as a normal message, and the Discord docs now describe `file`/`media-gallery` attachment usage.

## Close Comment

Closing from a Codex review. Current `main` already covers the reported regressions well enough: Discord no longer contributes a required `components` field to the merged `message` tool schema (`extensions/discord/src/channel-actions.ts:65`, with coverage in `src/agents/tools/message-tool.test.ts:446`), and classic-compatible component payloads with media now fall back to the normal Discord media send path instead of silently losing the attachment (`extensions/discord/src/send.components.ts:249`, covered in `extensions/discord/src/send.components.test.ts:156`). The Discord docs also now document the supported `file` / `media-gallery` attachment patterns and the `attachment://<filename>` scheme (`docs/channels/discord.md:296`). This was already shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit a different Discord media/rendering problem on a newer build, please open a fresh issue with the exact tool call and resulting Discord render.
