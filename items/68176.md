## Summary

Keep open. Current `main` still resolves CLI commit metadata from the live checkout before baked build metadata, and the build config does not inject a build-time commit define. The PR addresses a real current-main gap rather than duplicating an already-landed fix.

## Evidence

- **Current main still prefers live git over baked metadata:** `resolveCommitHash` checks `GIT_COMMIT`/`GIT_SHA`, then cached results, then `readGitCommit(...)` before `build-info.json` and `package.json`. That means a built `dist/` running inside a checkout can still report the moved live HEAD instead of the commit it was built from.
  - file: [src/infra/git-commit.ts:223](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/infra/git-commit.ts#L223)
  - command: `nl -ba src/infra/git-commit.ts | sed -n '213,257p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Existing test codifies the current problematic priority order:** `git-commit.test.ts` has a current-main test named `prefers live git metadata over stale build info in a real checkout`, asserting live git wins over `build-info.json`. That is the behavior the PR is explicitly trying to change for built binaries.
  - file: [src/infra/git-commit.test.ts:113](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/infra/git-commit.test.ts#L113)
