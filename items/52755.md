## Summary

Keep open. Current `main` does not implement this PR’s Feishu chat account-routing fix: `feishu_chat` still binds to the first enabled account and checks only that account’s chat-tool config, while sibling Feishu tools already use the shared account-routing helper and per-session context.

## Evidence

- **`feishu_chat` still hardcodes the first enabled account:** `extensions/feishu/src/chat.ts:136` selects `const firstAccount = accounts[0]`; `extensions/feishu/src/chat.ts:143-154` builds the client with `createFeishuClient(firstAccount)` inside `execute`, so there is no path using `ctx.agentAccountId` or an explicit `accountId` override.
  - file: [extensions/feishu/src/chat.ts:136](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/feishu/src/chat.ts#L136)
  - command: `nl -ba extensions/feishu/src/chat.ts | sed -n '124,190p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Registration still depends only on the first account’s chat config:** `extensions/feishu/src/chat.ts:137-140` calls `resolveToolsConfig(firstAccount.config.tools)` and returns early if that single account has chat disabled. That matches the PR’s reported bug for multi-account setups where another enabled account has chat turned on.
