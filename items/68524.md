## Summary

Keep open. Current main at d251932fcfd60e205e80bf7e9abdb4af63582e6f still loads iOS chat-composer photo picker bytes directly, rejects raw attachments over 5 MB instead of resizing, stores/decodes the original data on the MainActor, and base64-encodes those original bytes during send with no aggregate budget. The related issues in the provided context are adjacent gateway/history/media paths, but none supersedes this specific iOS composer resize request.

## Best Possible Solution

Keep #68524 open and implement the fix in the shared ChatUI/iOS composer path: load/decode/resize off the MainActor, reuse or extend `JPEGTranscoder`/`PhotoCapture` for aspect-preserving JPEG output, enforce a pixel cap, target a per-image budget with headroom below provider limits, enforce a total per-message attachment budget, and add Swift tests for oversized single images, multiple attachments, rejection/error wording, and send payload sizes. The broader #61041 gateway/server-side request can remain separate because this issue is specifically about iOS client-side composer transport safety.
