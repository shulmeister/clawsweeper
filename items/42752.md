## Summary

Keep open. Current `main` still reproduces the reported behavior: Slack read operations always prefer `userToken` when present, and `userTokenReadOnly === false` only affects write-token fallback. The same behavior is present in the latest release tag `v2026.4.22`, and the test suite/doc text currently encode that behavior rather than a fix.

## Evidence

- **Action runtime read path ignores `userTokenReadOnly`:** `getTokenForOperation("read")` returns `userToken ?? botToken` unconditionally. `allowUserWrites` is computed from `account.config.userTokenReadOnly === false` but only used in the write branch.
  - file: [extensions/slack/src/action-runtime.ts:159](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/slack/src/action-runtime.ts#L159)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Channel-level token selection matches the same logic:** `getTokenForOperation(account, "read")` in the channel plugin also returns `userToken ?? botToken` before consulting `allowUserWrites`, so the read-preference behavior is duplicated outside the action runtime.
  - file: [extensions/slack/src/channel.ts:147](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/slack/src/channel.ts#L147)
