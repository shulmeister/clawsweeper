## Summary

Current main still does not implement the requested `/agent` read-tool guard mapping for `workspaceAccess: "ro"` sandboxed sessions. The workspace-root guard only remaps `containerWorkdir`, the sandboxed read-tool wiring only passes `sandbox.containerWorkdir`, and current regression coverage still expects `/agent/...` to be rejected when workspace-only guarding is enabled. The sandbox mount layer does understand `/agent`, so this remains a real missing wiring/guard behavior rather than stale cleanup.

## Best Possible Solution

Keep this issue open. The likely fix is to make the workspace-root guard mount-aware for sandboxed file tools, or minimally pass an explicit `/agent` mount option when `workspaceAccess: "ro"` so valid read-only agent workspace paths are mapped before `assertSandboxPath`. Keep `/tmp` as a separate policy question unless maintainers decide read should cover container-local temp paths too.
