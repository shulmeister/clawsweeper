## Summary

Current main still implements the reported behavior: `session.parentForkMaxTokens` defaults to 100000, `initSessionState()` skips parent forking when the parent token count exceeds that threshold, emits only a subsystem WARN log, and marks the child as already forked so it will not retry. The related compaction provider and checkpoint work exists, but it is not wired into this parent-fork path.

## Best Possible Solution

Keep #63141 open. The best fix likely belongs in `src/auto-reply/reply/session.ts` and `src/auto-reply/reply/session-fork.ts`: either compact or checkpoint the parent before a guarded fork skip, or surface a one-time visible/operator warning when starting a fresh child, and consider a model-aware default in `resolveParentForkMaxTokens()`. Add focused coverage in `src/auto-reply/reply/session.test.ts` and update the session/config docs to describe the new behavior.

## Risks / Open Questions

- Auto-compacting during session initialization could add latency or inherit compaction timeout failure modes, so it needs bounded timeout/fallback behavior and should account for #43661.
- A channel-visible warning needs dedupe and per-channel delivery policy so it does not spam thread creation or leak internal token details.
- Model-aware auto-scaling needs to use the resolved model catalog/context window safely and avoid creating forks that exceed providers with smaller real limits.
