## Summary

Keep open. Current main does not expose `api.runtime.acp.spawn()` or `api.runtime.acp.prompt()` to plugin runtime, and the proposed `plugins.allowAcpSpawn` config gate is also absent. Main does have ACP spawning through the agent-facing `sessions_spawn({ runtime: "acp" })` tool, but that is not the same plugin runtime API this PR proposes for programmatic plugin-managed dispatch. The linked downstream `task-dispatch` validation and the later tracking issue #65022 support that this is a real plugin use case, not a random drive-by, so this PR still needs maintainer product/API judgment rather than automated cleanup.

## Best Possible Solution

Keep the PR open for maintainer API review. If accepted, implement a narrow, documented, lazily loaded `api.runtime.acp` surface with explicit operator opt-in, tests for disabled/enabled paths and delivery target handling, config schema/docs updates, and a clear relationship to existing `sessions_spawn({ runtime: "acp" })` behavior. If maintainers prefer not to expose direct ACP runtime helpers, close only after recording that product decision and pointing plugin authors to the supported alternative.
