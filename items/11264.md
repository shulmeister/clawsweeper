## Summary

Issue #11264 asks for a new messaging integration, and current main explicitly routes that kind of work to the external plugin ecosystem rather than OpenClaw core. The repo documents channel plugins as the supported extension path and has an explicit maintainer auto-close reason for third-party plugin requests.

## Close Comment

Closing this as a Codex maintainer review because current main routes new messaging integrations like XMTP/Convos to the plugin ecosystem rather than OpenClaw core. `VISION.md:54` says OpenClaw has an extensive plugin API, `VISION.md:55` says optional capability should usually ship as plugins, and `VISION.md:58` says plugin authors should host and maintain them in their own repository. The repo also has a dedicated channel-plugin path: `docs/plugins/building-plugins.md:16` says you do not need to add your plugin to the OpenClaw repository, `docs/plugins/sdk-channel-plugins.md:11` is the guide for building a messaging channel plugin, and `docs/plugins/community.md:9` documents community-maintained channel plugins. This is also the repo’s explicit maintainer close path in `.github/workflows/auto-response.yml:81` (`r: third-party-extension`). If you want to pursue XMTP/Convos on current main, the supported path is an external OpenClaw channel plugin published on ClawHub or npm, with a follow-up community-plugin listing once it is available.
