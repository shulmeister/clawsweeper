## Summary

Keep open. Current main has adjacent rate-limit handling, including provider 429 classification, auth-profile cooldown/failover, gateway/webhook inbound limiters, and Codex account rate-limit readouts, but it still does not implement the issue's requested configurable cross-provider outbound API rate limiting, request/token/day windows, budget enforcement, queue/backpressure, warning thresholds, or `openclaw limits` CLI. The related PR #13686 matched this scope but was closed unmerged, so it is not shipped or superseding.

## Best Possible Solution

Continue tracking #13615 as a core product/design item. The likely implementation belongs at the generic outbound provider/channel request boundaries with an opt-in config/schema/docs contract, tests for RPM/TPM/day windows, queue/backpressure behavior, token budget enforcement, warning/status reporting, and a real `openclaw limits` CLI. Provider/plugin extension points should be added only where current hooks cannot express provider-owned classification or transport wrapping.
