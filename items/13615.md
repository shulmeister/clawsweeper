## Summary

Keep open. Current main has adjacent retry, auth-profile cooldown/failover, inbound webhook limits, provider usage display, and Codex-account rate-limit readouts, but it still does not implement the issue's requested configurable cross-provider outbound rate limiting, request/token budgets, queueing/backpressure, warning thresholds, or `openclaw limits` CLI. The related PR #13686 matched this scope but was closed unmerged, so it is not a shipped or superseding fix.

## Best Possible Solution

Continue tracking #13615 as a core product/design item. The likely path is an opt-in, documented outbound policy layer at the generic provider/channel request boundaries, with strict config/schema/docs, focused tests for RPM/TPM/day windows, queue/backpressure behavior, budget enforcement, and a real `openclaw limits` status/config CLI. Provider/plugin extension points should be added only where current hooks cannot express provider-owned classification or transport wrapping.
