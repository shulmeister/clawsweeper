## Summary

Kapso support is not implemented on current main, but this request fits OpenClaw's external plugin path rather than core: current WhatsApp support is explicitly WhatsApp Web/Baileys-only, and the project vision says optional capabilities, commercial service integrations, and wrapper channels around already supported channels should not be merged into core when they can live as plugins.

## Evidence

- **Vision routes optional capability to plugins:** OpenClaw states that optional capability should usually ship as plugins, preferably hosted and maintained outside the core repo, and that the bar for bundling optional plugins is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/VISION.md#L54)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Vision excludes this kind of core integration:** The roadmap guardrails explicitly say OpenClaw will not merge commercial service integrations that do not clearly fit the model-provider category, and will not merge wrapper channels around already supported channels without a clear capability or security gap.
  - file: [VISION.md:99](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/VISION.md#L99)

## Risks / Open Questions

- Kapso could arguably provide a meaningful reliability/compliance difference versus WhatsApp Web, but the request is still for a specific commercial transport and the repo's current scope guidance points that work to an external plugin, not core.
- If maintainers later decide that official WhatsApp Business API support is strategically core-worthy, this specific scope call could change; current main does not indicate that shift.

## Close Comment

Closing after a Codex review. Current main keeps WhatsApp support as the `@openclaw/whatsapp` channel plugin and documents it as WhatsApp Web/Baileys only (`docs/channels/whatsapp.md:8`, `docs/channels/whatsapp.md:138`; `extensions/whatsapp/package.json:33`). The current WhatsApp schema also has no provider switch for a Kapso-style transport (`src/config/zod-schema.providers-whatsapp.ts:51`). More importantly, the project vision routes optional capability to plugins and explicitly says OpenClaw will not merge commercial service integrations or wrapper channels around already supported channels into core (`VISION.md:54`, `VISION.md:99`). OpenClaw already documents external/community plugins and a dedicated channel-plugin SDK path for this kind of integration (`docs/tools/plugin.md:11`, `docs/plugins/sdk-channel-plugins.md:1`). A Kapso-backed WhatsApp integration fits that external plugin / ClawHub path better than OpenClaw core, so we’re closing this as out of scope for core.
