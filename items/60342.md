## Summary

Keep open. Current main has Telegram media/network mitigations and troubleshooting docs, but it does not conclusively fix the reported `getaddrinfo ENOTFOUND api.telegram.org` media-download path: `ENOTFOUND` is not in the Telegram transport fallback retry code list, and media download fallback only runs when that retry predicate accepts the error.

## Evidence

- **Issue symptom:** The issue reports Telegram image/media downloads failing with `MediaFetchError ... getaddrinfo ENOTFOUND api.telegram.org` on OpenClaw 2026.4.1, with success only after changing host DNS to 8.8.8.8.
  - command: `provided GitHub context for issue #60342`
- **Telegram fallback retry excludes ENOTFOUND:** Telegram transport fallback only recognizes selected codes (`ETIMEDOUT`, `ENETUNREACH`, `EHOSTUNREACH`, `UND_ERR_CONNECT_TIMEOUT`, `UND_ERR_SOCKET`) and requires the known-network-code rule to match; `ENOTFOUND` is absent.
  - file: [extensions/telegram/src/fetch.ts:96](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/extensions/telegram/src/fetch.ts#L96)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '1,760p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Media downloads depend on that retry predicate:** Telegram media download calls `fetchRemoteMedia` with `dispatcherAttempts` and `shouldRetryTelegramTransportFallback`, so fallback attempts for media downloads are gated by the same retry predicate.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '1,460p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Generic media fallback stops when predicate rejects:** `fetchRemoteMedia` only advances to later dispatcher attempts when `shouldRetryFetchError` returns true; otherwise it throws the original fetch failure as `MediaFetchError`.
  - file: [src/media/fetch.ts:153](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/media/fetch.ts#L153)
  - command: `nl -ba src/media/fetch.ts | sed -n '1,270p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Existing mitigations are workarounds, not proof of fix:** Docs now describe proxy, IPv4, RFC2544/fake-IP, and DNS-result-order mitigations, but still tell users persistent Telegram failures point to proxy/DNS/IPv6/TLS egress issues rather than guaranteeing success regardless of system DNS.
  - file: [docs/channels/telegram.md:850](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/docs/channels/telegram.md#L850)
