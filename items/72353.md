## Summary

Keep #72353 open. Current main still only strips the literal `ollama/` prefix before sending native Ollama `/api/chat` requests, while the reported custom-provider case needs the active provider key stripped as well. The earlier #67457 regression coverage proves the default-provider case, but there is no equivalent custom-provider regression and the current resolver can preserve a provider-prefixed inline model id into the Ollama request path.

## Best Possible Solution

Keep the issue open and fix it in the bundled Ollama provider path. The best implementation is to generalize the Ollama wire-model normalization so native `api: "ollama"` requests strip the active provider prefix, not just the literal `ollama/`, while preserving legitimate nested model ids for other providers. Add a regression using a custom provider such as `ollama-remote/qwen3:32b` through `buildOllamaChatRequest` or `createOllamaStreamFn`, and ideally one resolver-level test showing a custom `models.providers.<id>` entry cannot leak `<provider>/` to `/api/chat`.
