## Summary

Keep open. Current main has some owned-Chrome lifecycle handling and the checked-in Compose file uses Docker `init: true`, but the official image still starts `node` directly as the container command and the browser launcher only manages the direct Chromium process. I do not see a core/image fix that would reap Chromium/crashpad descendants adopted by OpenClaw when it is PID 1, so the concrete normal-exit zombie leak remains plausible on current main.

## Best Possible Solution

Keep #64593 open. Add a focused Docker reproduction that launches/stops managed browser sessions with OpenClaw as PID 1 and checks for `<defunct>` Chromium/crashpad descendants, then fix the supported deployment path: either make the official image run under an init/reaper by default and document direct `docker run`/Compose requirements, or add a robust browser process-group/container lifecycle strategy if maintainers want the browser service itself to own descendant cleanup. Keep the existing direct-child state cleanup, but do not treat it as sufficient proof for the PID-limit leak.

## Risks / Open Questions

- Users running through the repo Compose file may already be protected by `init: true`, so the remaining affected surface is direct image/custom orchestrator deployment and should be confirmed with a focused container repro.
- The likely root cause is orphaned Chromium/crashpad descendants adopted by Node-as-PID-1, not just the direct child process exit listener, so the fix may belong partly in Docker image/runtime guidance rather than only `chrome.ts`.
