## Summary

Keep open. The report is recent, detailed, and describes a normal-exit PID-1/container zombie leak that is not covered by the closed startup-failure issue. Current main still launches and stops the managed Chromium process as a direct ChildProcess, tracks only that top-level process, and does not show a browser-specific process-group/subreaper/waitpid strategy or regression test for adopted chromium/chrome_crashpad zombies.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime path. The likely maintainer direction is to make managed Chromium lifecycle container-safe: either run Chrome under a supervised helper/init/subreaper that can reap adopted descendants, or launch it in a disposable process group and stop/wait for the full tree without killing the gateway. Add focused unit coverage for the spawn/stop contract and a Docker/browser smoke that exercises repeated normal browser exits and checks for defunct chromium/chrome_crashpad processes, then update Docker/browser troubleshooting docs with the supported deployment requirements.
