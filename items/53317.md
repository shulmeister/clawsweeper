## Summary

Current main already contains the auth-store reload and recovery logic that addresses this report: OAuth refresh now re-reads auth-profiles from disk under the auth lock before spending a refresh token, recovers from refresh_token_reused by adopting fresher stored credentials, and locked auth-store writers reload from disk instead of writing a stale in-memory snapshot back over newer credentials.

## Close Comment

Closing as implemented on current main after Codex review.

The reported stale-snapshot overwrite path is now guarded in a few places:
- src/agents/auth-profiles/oauth-manager.ts:327 reloads auth-profiles from disk under the auth lock and returns an already-usable OAuth credential instead of refreshing it.
- src/agents/auth-profiles/oauth-manager.ts:520 re-reads stored credentials after refresh failures and recovers when the on-disk token advanced.
- src/agents/auth-profiles/store.ts:136 reloads from disk for locked auth-store writes so a live gateway does not write stale in-memory auth state back over fresher CLI/login credentials.

Current main also has regression coverage for both behaviors in src/agents/auth-profiles/oauth.openai-codex-refresh-fallback.test.ts:416 and src/agents/auth-profiles.markauthprofilefailure.test.ts:76.

I could not verify a released tag carrying this from the local checkout, so I am only marking it implemented on main, not fixed in a known release yet.
