## Summary

Keep open. Current `main` does not implement cron postcondition checks or retry/escalation based on deterministic verification output; cron still treats isolated agent-turn completion as the job outcome and only adds transient-error retries/failure alerts around execution failures.

## Evidence

- **Cron job shape has no postconditions field:** `CronJob`, `CronJobPatch`, and `CronJobState` expose schedule, payload, delivery, failure alerts, and run/delivery status only. There is no `postconditions` field and no stored postcondition result fields.
  - file: [src/cron/types.ts:112](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/cron/types.ts#L112)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Protocol schema also lacks postconditions:** Gateway cron schemas allow `delivery`, `failureAlert`, and `state`, with `additionalProperties: false`; a `postconditions` field would currently be rejected at the API boundary.
  - file: [src/gateway/protocol/schema/cron.ts:231](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/gateway/protocol/schema/cron.ts#L231)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Execution path has no framework verification hook:** `executeJobCore()` routes directly to main-session or detached execution, and `executeDetachedCronJob()` returns the isolated agent result straight back. `applyJobResult()` records run status/duration/error/delivery and schedules retries/backoff from that result only; there is no post-run deterministic checker before state is applied.
  - file: [src/cron/service/timer.ts:391](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/cron/service/timer.ts#L391)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Current documented retries are only transient-error retries:** The config docs describe `cron.retry` as one-shot retries for transient execution errors and `cron.failureAlert` / `cron.failureDestination` as notifications. They do not document any outcome-verification or postcondition facility.
  - file: [docs/gateway/configuration-reference.md:3791](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/gateway/configuration-reference.md#L3791)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Cron docs describe execution, delivery, and failure notification only:** The cron docs cover isolated execution, shared delivery, run history, and failure destinations, but nothing about deterministic post-run checks or retrying an agent turn because a verifier script failed.
  - file: [docs/automation/cron-jobs.md:80](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/automation/cron-jobs.md#L80)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **No matching implementation surfaced in local history search:** `git log --oneline --grep='postcondition\|cron.*verify\|cron.*check\|deterministic verification' --all -- src/cron docs/automation/cron-jobs.md src/gateway/protocol/schema/cron.ts` returned no relevant commits.
