## Summary

Keep #12398 open. Current main has cron retry/backoff, failure alerts, and an interim-acknowledgement retry for isolated agent turns, but it still lacks the requested deterministic post-run postcondition contract, persisted checker state/output, and a cron/plugin API that can turn an otherwise clean isolated agent completion into scheduler-owned retry or escalation based on an external verifier. The prior issue discussion also explicitly kept this open, and the current cron changes since that review are about delivery/session handling rather than postconditions.

## Best Possible Solution

Keep #12398 open for maintainer/product design. The likely core path is an additive cron postconditions contract covering types, gateway protocol/schema, CLI, Control UI, docs, run logs, durable CronJobState, deterministic verifier execution after isolated agent completion and before applyJobResult(), bounded retry prompt injection with checker output, and integration with existing cron retry/backoff/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook/API that can mark a clean run failed and request scheduler-owned retry/escalation.

## Risks / Open Questions

- A shell-backed verifier needs maintainer design for command trust, sandbox/approval policy, timeout and output caps, environment exposure, secret handling, and how verifier failures interact with existing retry/backoff and failure-alert policy.
- A plugin-owned implementation is not equivalent yet because the current plugin hook surface can observe or revise agent turns but cannot deterministically fail a completed cron run and ask the scheduler to own retry/escalation.
