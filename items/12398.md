## Summary

Keep #12398 open. Current main still has no first-class cron postcondition contract, persisted verifier state, deterministic post-run checker, or hook/API that can convert an otherwise clean isolated agent run into cron-owned retry/escalation. Existing retry and failure-alert behavior only applies after the cron outcome is already `error`, and the latest issue discussion already recorded the same keep-open conclusion.

## Best Possible Solution

Keep #12398 open for a maintainer/product implementation. The likely path is an additive cron job contract plus gateway/CLI/schema/docs/UI updates, deterministic verifier execution after isolated agent completion and before applyJobResult, persisted verifier state and run-log/event fields, bounded retry prompt injection with checker output, and integration with cron-owned retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook/API that can fail a run and request cron-owned retry/escalation.
