## Summary

Keep #12398 open. Current main still lacks cron postconditions or an equivalent deterministic post-run verifier. The cron contract, gateway schema, scheduler state update path, run logs, CLI/UI types, and tests still model only execution status, delivery status, transient retry/backoff, and failure alerts. Recent main did add cron `jobId` context to plugin hooks, which is useful for plugin-owned observation, but it still does not provide a cron-owned post-run hook that can fail a run and trigger scheduler retry/escalation.

## Best Possible Solution

Keep #12398 open for maintainer/product implementation. The likely core path is an additive cron contract and gateway schema field, deterministic checker execution after isolated agent completion, persisted postcondition state/run-log evidence, retry prompt injection using checker output, and integration with existing retry/failure-alert delivery. If maintainers prefer a plugin route, the missing prerequisite is a first-class cron post-run hook/API that can fail a cron run and schedule cron-owned retry/escalation.
