## Summary

Keep open. Current `main` still stores workspace setup state at `<workspace>/.openclaw/workspace-state.json`, so the PR’s requested behavior is not implemented on `main`, and the report is still coherent and actionable. The PR also appears to need follow-up before landing because the proposed legacy fallback probes the dot-prefixed path in code paths that currently rethrow non-`ENOENT` I/O errors, which is risky for the same dot-restricted filesystems the PR is trying to support.

## Risks / Open Questions

- The PR’s main idea still addresses a real gap: current `main` requires a dot-prefixed workspace state directory.
- As proposed, legacy fallback reads can still surface non-`ENOENT` errors from `.openclaw/...`, which undermines the TigerFS-style compatibility claim.
- Moving the state file to the workspace root would require coordinated test updates beyond `src/agents/workspace.ts`; current tests still assume the legacy location.
