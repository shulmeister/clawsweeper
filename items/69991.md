## Summary

Close as `clawhub`: current `main` already exposes sandbox-backend registration through the plugin SDK, uses that path for the existing `openshell` backend, and the project's current direction is to keep optional capability in external plugins distributed via ClawHub/npm rather than bundling more optional backends into core.

## Evidence

- **Project direction keeps optional capability in plugins:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, plugin authors should host and maintain plugins in their own repositories, the bar for adding optional plugins to core is intentionally high, and plugin promotion belongs in ClawHub.

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Public plugin docs say external distribution is the default path:** `docs/plugins/building-plugins.md` explicitly says you do not need to add a plugin to the OpenClaw repository; instead publish to ClawHub or npm and install it with `openclaw plugins install <package-name>`.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/plugins/building-plugins.md#L16)

  - file: [src/plugin-sdk/sandbox.ts:24](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/plugin-sdk/sandbox.ts#L24)

- **Bundled OpenShell already follows the plugin route:** `extensions/openshell/index.ts` is a normal plugin entry that calls `registerSandboxBackend("openshell", ...)`, showing the intended architecture for sandbox backends is plugin-owned rather than new core behavior.
  - file: [extensions/openshell/index.ts:19](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/openshell/index.ts#L19)

- **Current tests prove registered custom backends resolve through config:** `src/agents/sandbox.resolveSandboxContext.test.ts` registers a non-docker backend (`test-backend`), sets `agents.defaults.sandbox.backend` to that id, and verifies `resolveSandboxContext(...)` resolves it successfully. That directly supports the issue's question about plugin-registered backend ids on current `main`.

- **Issue body already proposes the right fallback:** The issue asks for a bundled `lima-sandbox` extension but also explicitly says the authors can publish it as a standalone package and link it from docs if bundling is not the right fit. That matches current OpenClaw plugin policy and distribution guidance.

## Risks / Open Questions

- If maintainers want Lima to be an official maintainer-owned bundled backend despite the current plugin-first policy, this close would be premature; I did not find evidence of that exception on current `main`.
