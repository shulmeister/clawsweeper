## Summary

Keep open: current `main` still probes every discovered provider, so the cleanup is not already implemented, and the proposed PR logic appears to regress profile-backed providers such as `openai-codex` that are enabled by auth profiles instead of `models.providers` entries.

## Evidence

- **Current main still probes every discovery provider:** `resolvePluginImplicitProviders(...)` iterates every provider in the selected discovery order and calls `runProviderCatalogWithTimeout(...)` without checking whether the provider appears in `config.models.providers`. That means the PR’s requested behavior is not implemented on current `main`.
  - file: [src/agents/models-config.providers.implicit.ts:199](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/models-config.providers.implicit.ts#L199)
  - command: `nl -ba src/agents/models-config.providers.implicit.ts | sed -n '190,260p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`openai-codex` discovery is profile-backed, not config-backed:** The bundled OpenAI Codex provider declares `catalog.order = "profile"` and returns a provider only when `listProfilesForProvider(authStore, "openai-codex")` finds an auth profile. A blanket `models.providers` gate would skip this provider even when the user is authenticated.
  - file: [extensions/openai/openai-codex-provider.ts:483](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/openai/openai-codex-provider.ts#L483)
  - command: `nl -ba extensions/openai/openai-codex-provider.ts | sed -n '475,505p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **OAuth auth results do not automatically create `models.providers.<id>` entries:** The shared OAuth auth result helper defaults `configPatch` to `agents.defaults.models[...]` only, and the OpenAI Codex auth contract test asserts that shape. So a user can have a valid `openai-codex` OAuth profile without any `models.providers.openai-codex` entry.
  - file: [src/plugin-sdk/provider-auth-result.ts:40](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/plugin-sdk/provider-auth-result.ts#L40)
  - command: `nl -ba src/plugin-sdk/provider-auth-result.ts | sed -n '1,80p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **PR discussion already identified the regression:** The provided PR review context includes a Codex review on head commit `958f9f70c3fd88732eefcfdefb9880bb44a92580` warning that gating discovery on `models.providers` would silently drop profile-based providers like `openai-codex` once any explicit provider is configured. That matches the current code paths above.
  - sha: [958f9f70c3fd](https://github.com/openclaw/openclaw/commit/958f9f70c3fd88732eefcfdefb9880bb44a92580)

- The PR’s current approach appears unsafe for profile-backed providers, so closing it as implemented or obsolete would lose a still-valid problem statement while the proposed patch remains incomplete.
