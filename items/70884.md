## Summary

Keep PR #70884 open. Current main still has the docs gap the PR targets: trusted-proxy docs describe `x-openclaw-scopes` as a trusted-proxy HTTP scope header but do not warn that WebSocket scopes are determined by the WS handshake/device identity path, and protocol docs list device-less Control UI exceptions without documenting the scope-clearing consequence. Current runtime and tests support the PR's central clarification, including the trusted-proxy scope clearing behavior, the local `allowInsecureAuth` exception, and the critical risk of `dangerouslyDisableDeviceAuth`.

## Best Possible Solution

Keep the PR open for maintainer review and likely merge after wording polish. The best product/docs path is to update `docs/gateway/trusted-proxy-auth.md` and `docs/gateway/protocol.md` so users understand that `x-openclaw-scopes` is an HTTP API header, WebSocket scopes come from the Gateway handshake/device identity model, missing device identity can clear requested scopes and cause `missing scope` RPC failures, and `dangerouslyDisableDeviceAuth` is a critical break-glass workaround rather than normal setup guidance.
