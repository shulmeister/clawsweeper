## Summary

Current main has durable queueing and recovery for generic direct outbound delivery, plus limited Telegram direct-send retries, but Telegram interactive final/block reply delivery still bypasses that queue. The current code path matches the supplied 2026.4.22 repro: a generated assistant answer can exist while Telegram `sendMessage` fails and the final reply is not durably retried.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it into #50040. The fix should make Telegram interactive final/block replies durable, either by routing those payloads through the shared outbound delivery queue or by adding a Telegram-owned persistent reply queue that preserves chat/topic/thread/reply/quote/session metadata, classifies permanent 4xx/auth/routing errors separately from safe transient and ambiguous network failures, retries with bounded backoff/TTL, drains on startup and Telegram transport recovery, and adds regression coverage for `deliverReplies`/dispatch after `sendMessage` fails once the assistant answer has already been produced.
