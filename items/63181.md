## Summary

Keep open. Current main still builds the exact `## Current Date & Time` section while only adding `Time zone: ...`; tests explicitly preserve the timezone-only behavior and still expect that heading, and docs still publish the same heading. Related reports in the provided context address adjacent temporal grounding paths (#38694/#38782 startup/message timestamps, #63071 prompt cache churn, #55935 heartbeat timestamp format), but none is an open canonical item for the misleading system-prompt section label itself.

## Best Possible Solution

Keep the issue open and fix the prompt/docs/tests coherently: rename the cache-stable section to `## Time Zone` or add an explicit cache-stable limitation line telling agents that current date/time is not in the system prompt and should come from `session_status` or message timestamps. Do not reintroduce dynamic date/time before the prompt cache boundary unless maintainers intentionally choose that tradeoff.
