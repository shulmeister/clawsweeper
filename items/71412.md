## Summary

Current main still matches the reported stop-timeout failure mode. The stop timeout path returns before clearing the task and abort maps, marks the runtime as still running, and the start path still silently no-ops when a stale task remains. The existing regression test also still asserts the buggy behavior, while the provided GitHub context says the direct fix PR #71456 is closed and unmerged.

## Best Possible Solution

Keep this issue open and land PR #71456 or an equivalent core gateway fix. On the stop-timeout path, clear the stale store.aborts and store.tasks entries before returning, mark the runtime stopped instead of running, and update the regression test to prove a subsequent start registers a fresh task. Treat any openclaw doctor liveness improvement as a separate follow-up unless maintainers choose to bundle it.
