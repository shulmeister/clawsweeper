## Summary

Keep #38671 open. Current main still has no first-class direct shell cron payload: cron payload types and gateway validation only allow `systemEvent` or `agentTurn`, the CLI/UI only build those two modes, service validation rejects other payload kinds, and detached cron execution still runs through the isolated agent/model path. Related PR #62789 is useful evidence of a viable direction, but it is open and described as backend/runtime/server only, so it does not ship the requested end-to-end pure command and command+message hybrid workflows.

## Best Possible Solution

Keep #38671 open until OpenClaw either ships direct command cron support end to end or maintainers explicitly reject the product direction. The likely path is to land or rework the backend/protocol pieces from #62789, add CLI/UI/docs support for pure command and command+message hybrid jobs, and align execution with existing exec security, timeout, output truncation, run-log, and delivery contracts.

## Risks / Open Questions

- Direct shell cron has product and security decisions still unresolved: argv parsing vs shell strings, approval policy, timeout and child termination, output truncation, failure handling, run-log shape, delivery semantics, and command-output injection into optional agent analysis.
- Closing this as ClawHub/plugin work would leave the requested built-in scheduler/protocol/CLI/UI behavior unresolved; current core cron does not expose a payload extension point for plugins to add first-class cron job kinds.
