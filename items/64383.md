## Summary

Keep open. Current main still uses the sandbox browser socat relay and CDP port offset that #64383 asks maintainers to evaluate, and the related #62873 context explicitly left socat removal as follow-up architectural work rather than shipping it. Latest release tag v2026.4.24 shows the same socat-based path.

## Best Possible Solution

Keep #64383 open for maintainer architecture review. The best path is to decide whether the socat relay still owns a necessary security/access-control responsibility; if not, implement direct Chromium CDP routing with equivalent Docker binding, bridge/auth, readiness, docs, config migration/deprecation, and focused sandbox browser tests. If the relay is retained, document why it remains intentional after the fail-fast readiness fixes.

## Risks / Open Questions

- Removing socat is not a mechanical cleanup: it must preserve the current exposure-control and access-control semantics currently represented by host loopback publishing, `cdpSourceRange`, bridge auth, and explicit readiness checks.
- The related #66690 and #49392 context covers adjacent CDP reachability/routing problems, but it does not supersede this narrower architecture decision about whether the socat relay should remain.
