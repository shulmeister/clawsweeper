## Summary

Current main already implements the requested behavior via configurable inbound same-sender debouncing plus collected followup queueing, including Telegram-specific batching logic and regression coverage.

## Risks / Open Questions

- Plain same-sender text batching is configurable via `messages.inbound.debounceMs`; if left at `0`, only the active-run `collect` behavior and Telegram's special forwarded-burst coalescing apply by default.

## Close Comment

Closing from a Codex review because current `main` already has the requested feature. OpenClaw now exposes `messages.inbound.debounceMs` for batching rapid same-sender inbound messages before a turn starts (`docs/concepts/messages.md:37`, `src/config/schema.help.ts:1549`), and Telegram uses that debounce path to merge multiple entries into one synthetic message before processing (`extensions/telegram/src/bot-handlers.runtime.ts:159`). If a run is already in flight, queue mode `collect` coalesces followups into a single turn and is documented as the default (`docs/concepts/queue.md:25`), with queue coverage in `src/auto-reply/reply/queue.collect.test.ts:57`. Telegram also has regression coverage for split-send coalescing in `extensions/telegram/src/bot.media.downloads-media-file-path-no-file-download.e2e.test.ts:421`. This was already present in release `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`. If you still hit a bad case after setting an appropriate `messages.inbound.debounceMs`, open a new issue with the exact Telegram config and a minimal reproduction.
