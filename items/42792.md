## Summary

Keep open. Current `main` still does not implement custom WebSocket headers for `openclaw node run` / `openclaw node install`, so the Cloudflare Zero Trust 302-auth problem described in #42792 remains plausible and unaddressed in the shipped code/docs.

## Evidence

- **`node run` / `node install` CLI still has no header flag:** The registered options include `--host`, `--port`, `--tls`, `--tls-fingerprint`, `--node-id`, and `--display-name` for `run`, plus `--runtime`/`--force` for `install`; there is no `--header` or equivalent repeatable header option.
  - file: [src/cli/node-cli/register.ts:41](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/cli/node-cli/register.ts#L41)
  - command: `nl -ba src/cli/node-cli/register.ts | sed -n '35,90p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Persisted node config has no place for custom gateway headers:** `NodeHostGatewayConfig` only carries `host`, `port`, `tls`, and `tlsFingerprint`. `NodeHostConfig` stores that gateway object unchanged, so install-time persistence also has no header field.
  - file: [src/node-host/config.ts:7](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/node-host/config.ts#L7)
