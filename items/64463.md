## Summary

Keep #64463 open. Current main supports daily/idle session resets, thread-binding max-age, disk maintenance, usage display, and auto-compaction, but it does not expose `session.maxDurationMinutes` or `session.maxTokensPerSession`, and I found no runtime enforcement that stops, resets, or compacts an active runaway session based on cumulative wall-clock or token spend.

## Best Possible Solution

Keep the issue open and implement this as first-party core session budgeting: add schema/types/docs for opt-in `session` budget keys, track per-session wall-clock and cumulative token/tool usage from existing lifecycle and usage-accounting paths, enforce before LLM calls and tool invocations, and define deterministic stop/warn/compact/reset behavior with tests. Coordinate with existing auto-compaction and cron budget work, but do not treat those narrower mechanisms as a substitute for the requested kill switch.

## Risks / Open Questions

- The exact product semantics still need maintainer judgment, especially whether budget exceed should stop, warn, compact, or rotate to a new session.
- Enforcement crosses config validation, session lifecycle, embedded Pi/agent run loops, tool invocation, and possibly ACP/runtime adapters; a partial implementation could report budgets without actually preventing runaway active sessions.
