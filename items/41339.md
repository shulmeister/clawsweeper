## Summary

Current `main` already implements the stale-socket fix for quiet Discord channels by basing health-monitor restarts on provider transport activity instead of inbound app-event freshness, and the changelog says that shipped in `v2026.4.22`.

(#69833)`

## Close Comment

Closing as implemented on main after a Codex review. The issue thread later linked this report to the same stale-socket root cause as #58339, and current `main` now handles that path differently: `src/gateway/channel-health-policy.ts` only treats a channel as `stale-socket` when provider transport activity is stale, with an explicit note that quiet Discord workspaces can be healthy while idle. The shipped changelog entry in `CHANGELOG.md` says this landed in `v2026.4.22` as: `Gateway/channel health: base stale-socket recovery on provider-proven transport activity instead of inbound app-event freshness`, which matches the failure pattern described here. Docs also now expose the health-monitor tuning/disable knobs in `docs/gateway/health.md`. If you still see real Discord disconnects on `v2026.4.22` or newer, please open a new issue with fresh logs from `openclaw health --verbose` / `openclaw channels status --probe`, since that would likely be a different bug than this original stale-socket loop.
