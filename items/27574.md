## Summary

Keep #27574 open. Current main still has browser-control primitives, the bundled browser plugin, and an admin `browser.request` gateway method, but it does not implement the requested Dashboard/Control UI browser preview panel with live screenshots, URL/title, console errors, or manual interaction. The current plugin SDK also does not expose a supported Control UI tab/panel contribution API, so this is not a clean ClawHub-only close. Related #63926 overlaps but is narrower and remains open around login handoff/browser management rather than superseding this broader remote-debugging request.

## Best Possible Solution

Continue tracking #27574 as the broader remote browser-preview request. The best path is either a core Control UI browser panel backed by the existing `browser.request` and browser-control primitives, or first adding a supported Control UI panel contribution API and then implementing the browser preview as a plugin. Coordinate with #63926 for login handoff/browser management, but do not treat it as a complete replacement for the live remote-debugging preview request.

## Risks / Open Questions

- A live browser preview/input panel would expose browser pixels, URLs/titles, console/page errors, and possibly remote-control actions, so implementation needs explicit operator-admin authorization, origin protections, rate limiting, and clear handoff semantics.
- If maintainers want this outside core, the likely prerequisite is a supported Control UI tab/panel contribution API; the current plugin surfaces are not enough to implement the requested Dashboard UX cleanly.
