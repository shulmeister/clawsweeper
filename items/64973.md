## Summary

Keep open. Current `main` still contains the two failure ingredients described in #64973: the gateway rejects stale `baseHash` writes, and the macOS app still falls back to direct file writes or performs direct config-file mutations outside the gateway path. I did not find a current fix or regression test covering the reported stale-`config.set` -> local file clobber -> token mismatch/restart sequence.

## Evidence

- **Gateway still rejects stale config writes:** `config.set` on the gateway still returns `INVALID_REQUEST` with `config changed since last load; re-run config.get and retry` when `baseHash` no longer matches. That matches the first log line in the report and means the stale-write path is still live on current `main`.

  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '80,108p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **macOS local save still bypasses gateway on any config.set failure:** In local mode, `ConfigStore.save` first tries `saveToGateway`, but catches any error and immediately calls `OpenClawConfigFile.saveDict(root)`. A stale-hash `config.set` failure therefore still degrades into a raw file write using the caller's old snapshot, which is consistent with the issue's reported `.bak`/config clobber symptoms.

## Risks / Open Questions

- Closing this would likely hide a real data-loss/auth-rotation bug: current main still permits stale macOS config state to be written directly after a rejected optimistic write.
- The issue's linked related work is still relevant evidence, not superseded: the reporter specifically called out gaps around direct non-gateway writers, and current main still contains those writers.
- I could not run a macOS reproduction in this Linux checkout, so there is no high-confidence basis for `cannot_reproduce`.
