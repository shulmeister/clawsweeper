## Summary

Keep open. Current `main` still blocks provider API calls when local DNS resolves public hostnames into RFC2544 fake-IP space unless the request is routed through `HTTP_PROXY`/`HTTPS_PROXY` env proxy mode. The codebase has a narrow fake-IP opt-in for `tools.web.fetch`, but not for provider request config used by `image_generate` provider calls.

## Evidence

- **Resolved fake-IP answers are still blocked by default:** `resolvePinnedHostnameWithPolicy()` re-checks DNS answers and throws `Blocked: resolves to private/internal/special-use IP address`; RFC2544 is only exempted when `SsrFPolicy.allowRfc2544BenchmarkRange` is set.
  - file: [src/infra/net/ssrf.ts:266](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/net/ssrf.ts#L266)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '260,428p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Provider HTTP auto-upgrade is env-proxy only:** `fetchWithTimeoutGuarded()` only upgrades provider HTTP requests to `trusted_env_proxy` when `shouldUseEnvHttpProxyForUrl()` returns true. The nearby comment explicitly scopes this to `HTTP_PROXY`/`HTTPS_PROXY` proxy environments.
  - file: [src/media-understanding/shared.ts:253](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/media-understanding/shared.ts#L253)
  - command: `nl -ba src/media-understanding/shared.ts | sed -n '229,320p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Env-proxy detection does not cover system DNS/proxy fake-IP mode:** `shouldUseEnvHttpProxyForUrl()` only checks `HTTP_PROXY`/`HTTPS_PROXY` env vars plus `NO_PROXY`; it has no path for system-level DNS/proxy rewriting such as Surge fake-IP without env proxy vars.
  - file: [src/infra/net/proxy-env.ts:57](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/net/proxy-env.ts#L57)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`image_generate` provider calls use the shared provider HTTP path:** Google image generation calls `postJsonRequest()` for `${baseUrl}/models/${model}:generateContent`, and OpenAI image generation uses `postJsonRequest()` / `postMultipartRequest()` for `/images/generations` and `/images/edits`. Those flows therefore inherit the env-proxy-only mitigation above.
  - file: [extensions/google/image-generation-provider.ts:158](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/google/image-generation-provider.ts#L158)

- **Fake-IP opt-in exists for web fetch, not provider request config:** The only explicit RFC2544 fake-IP knob in config is `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`; provider request config exposes `allowPrivateNetwork` but not an RFC2544/fake-IP exception, so there is no equivalent narrow provider-side workaround for this issue on current `main`.
