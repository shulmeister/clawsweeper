## Summary

Keep open. Current main and v2026.4.24 still treat JSON-stringified chat content-block arrays as ordinary string content, so the vLLM double-serialization fix is not implemented or superseded. The PR remains useful, but the current PR discussion identifies a real safety issue with overly broad JSON-array detection that should be revised before merge.

## Best Possible Solution

Keep this PR open and revise it against current main. The best implementation is a conservative shared normalization helper that only parses stringified JSON arrays when they have a recognizable chat content-block shape, extracts text consistently with the real-array path, preserves arbitrary JSON strings, routes relevant string-body callers through the same logic, and adds focused tests in `src/shared/chat-content.test.ts` plus a targeted OpenAI-compatible/vLLM transport or display regression.

## Risks / Open Questions

- Closing would discard a focused PR for a behavior that still exists on current main and the latest release by source inspection.
- Merging the PR as-is could regress arbitrary JSON string content such as `[{"type":"event","payload":"x"}]` by treating it as chat content and returning null.
- The remaining fix needs targeted coverage for vLLM text-only blocks, mixed text/media blocks, invalid JSON fallback, and non-chat typed JSON arrays.
