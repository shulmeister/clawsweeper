## Summary

Current main still has the behavior this issue reports. `fullAccessAvailable` is prompt-gated, but the value is derived from config-only `enabled`/`allowed` state when callers do not provide an explicit runtime signal. The fallback still returns available=true for ambiguous allowed elevation, tests assert that fallback, and provider plugin production code does not assign `fullAccessAvailable` for OpenAI/Anthropic/Google/Codex paths.

## Best Possible Solution

Keep this issue open. The best fix is to add a first-class runtime full-access availability signal before prompt construction, remove the enabled+allowed optimistic fallback so ambiguous input resolves unavailable with an explicit blocked reason, and add regression coverage for ambiguous input, blocked runtime states, auto-reply/command prompt plumbing, and the final system prompt guidance. The implementation may need a core runtime/channel/provider capability seam rather than only provider-local edits, because current provider plugin types do not expose a direct `ExecElevatedDefaults` hook.
