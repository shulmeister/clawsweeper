## Summary

Current `main` already prevents the reported duplicate-reply path for single-entry Feishu debounce flushes by routing the event through `handleFeishuMessage`, which performs persistent dedupe before any downstream processing or reply dispatch.

## Close Comment

Closing as implemented on `main` after Codex review. The current Feishu receive path already routes `im.message.receive_v1` through `handleFeishuMessage` (`extensions/feishu/src/monitor.account.ts:328`, `extensions/feishu/src/monitor.message-handler.ts:183`, `extensions/feishu/src/monitor.message-handler.ts:263`), and `handleFeishuMessage` now performs persistent dedupe before any downstream reply work (`extensions/feishu/src/bot.ts:289`). That entry check goes through `finalizeFeishuMessageProcessing(...)` and `tryRecordMessagePersistent(...)` (`extensions/feishu/src/dedup.ts:63`, `extensions/feishu/src/dedup.ts:108`), so an already-recorded single-entry debounce flush exits early instead of sending a second reply. This PR’s intended behavior is already present on current `main` at `564f820efa2bb81ae49b8f55a93e41492ad31917`.
