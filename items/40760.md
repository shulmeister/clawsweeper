## Summary

Keep open. Current main still does not implement this PR's central behavior: reply setup creates the typing controller with the fixed default TTL instead of a timeout/fallback budget, and timeout failures still flow to the generic "Agent failed before reply" copy. The PR is focused and addresses a real remaining gap, but it should be revised before merge because the review thread flagged concrete defects in the proposed timeout classification and fallback-attempt budgeting.

## Best Possible Solution

Keep the PR open or replace it with a narrow updated patch: compute a bounded typing TTL from the actual fallback candidate budget, including the configured-primary append and transient retry cycle, and add explicit timeout-final messaging using a narrow timeout check such as `isTimeoutError` plus tests that distinguish true timeouts from bare network codes. The related configurable-TTL PRs are adjacent but do not fully supersede this timeout/fallback-budget fix.
