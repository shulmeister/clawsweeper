## Summary

Current main already contains macOS launchd recovery hardening plus non-fatal handling for transient network failures, which together cover the reported VPN-drop -> gateway-stopped / LaunchAgent-unloaded behavior.

## Evidence

- **Release notes shipped the launchd recovery changes:** The 2026.4.5 changelog explicitly says macOS gateway restarts now let launchd KeepAlive own supervised restarts, re-bootstrap a LaunchAgent if restart unloads it, and recover installed-but-unloaded LaunchAgents during `openclaw gateway start` and `restart`.
  - file: [CHANGELOG.md:1243](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/CHANGELOG.md#L1243)
  - command: `nl -ba CHANGELOG.md | sed -n '1228,1255p'`
- **LaunchAgent plist now enforces launchd supervision:** The generated macOS LaunchAgent plist includes `RunAtLoad=true`, `KeepAlive=true`, and `ThrottleInterval=1`, matching the intended auto-respawn behavior.
  - file: [src/daemon/launchd-plist.ts:116](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/daemon/launchd-plist.ts#L116)
  - command: `nl -ba src/daemon/launchd-plist.ts | sed -n '1,260p'`
- **Start/restart paths repair unloaded LaunchAgents:** Current CLI recovery logic re-bootstraps an installed-but-unloaded LaunchAgent, and `runDaemonStart` / `runDaemonRestart` call that recovery path on macOS when the service is not loaded.
  - file: [src/cli/daemon-cli/launchd-recovery.ts:14](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cli/daemon-cli/launchd-recovery.ts#L14)
  - command: `nl -ba src/cli/daemon-cli/launchd-recovery.ts | sed -n '1,220p'; nl -ba src/cli/daemon-cli/lifecycle.ts | sed -n '152,210p'`
- **Transient network failures are non-fatal:** Unhandled transient network errors are explicitly suppressed instead of exiting the process, which addresses the reported VPN-drop outbound failure path.
  - file: [src/infra/unhandled-rejections.ts:373](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/infra/unhandled-rejections.ts#L373)
  - command: `nl -ba src/infra/unhandled-rejections.ts | sed -n '340,383p'`
- **Regression coverage exists for both behaviors:** Tests cover re-bootstrapping unloaded LaunchAgents on macOS start/restart and verify that known transient network errors do not exit the process.
  - file: [src/cli/daemon-cli/lifecycle.test.ts:210](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cli/daemon-cli/lifecycle.test.ts#L210)

## Close Comment

Codex review on current `main` suggests this is already implemented. The macOS gateway path now relies on launchd-supervised recovery with `RunAtLoad=true`, `KeepAlive=true`, and `ThrottleInterval=1` (`src/daemon/launchd-plist.ts:116`), and start/restart explicitly recover installed-but-unloaded LaunchAgents (`src/cli/daemon-cli/launchd-recovery.ts:14`, `src/cli/daemon-cli/lifecycle.ts:152`). The shipped changelog for `2026.4.5` also calls out these exact launchd fixes in `CHANGELOG.md:1243`-`1245`. Current main additionally treats transient network failures as non-fatal instead of exiting the gateway process (`src/infra/unhandled-rejections.ts:373`, covered by `src/infra/unhandled-rejections.fatal-detection.test.ts:117`). That matches the reported VPN-drop -> gateway-stopped / LaunchAgent-unloaded failure mode, so I’m closing this as implemented on main. The separate `dmPolicy="open"` / `allowFrom` cold-start validation issue is a different config problem and should be tracked independently if it still needs follow-up.
