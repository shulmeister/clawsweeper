## Summary

Keep #11326 open. Current main still decodes built-in web_fetch responses with UTF-8/default decoding, exposes no encoding parameter, and does not parse HTTP charset or HTML meta before Readability/basic extraction. A synthetic GBK byte check reproduces the decode gap: Response.text/default TextDecoder produce replacement characters while TextDecoder('gbk') decodes the same bytes correctly.

## Best Possible Solution

Keep the issue open until core web_fetch supports non-UTF-8 response decoding. The likely implementation belongs in src/agents/tools/web-shared.ts or a narrow helper used by web_fetch: detect charset from Content-Type and early HTML meta bytes, decode with supported labels such as gbk/gb2312 when available, preserve byte limits and SSRF behavior, add a manual encoding parameter only if maintainers want that surface, and cover it with focused web_fetch tests and docs/tools/web-fetch.md updates.
