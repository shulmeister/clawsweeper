## Summary

Keep #11326 open. Current main still decodes web_fetch response bytes via UTF-8/default response-text paths before extraction, with no charset detection, no GBK/GB2312 handling, no manual encoding parameter, and no regression coverage for legacy Chinese encodings. The related #56538 work is about Windows exec/process output decoding, not HTTP response decoding in web_fetch.

## Best Possible Solution

Keep the issue open until core web_fetch performs charset-aware byte decoding before extraction. A narrow implementation should live in readResponseText or a helper it calls, preserve SSRF and maxResponseBytes behavior, detect charset from Content-Type and early HTML meta bytes, support labels such as gbk/gb2312 where the runtime can decode them, decide whether a manual encoding parameter/config is warranted, and add focused web_fetch tests plus docs updates.
