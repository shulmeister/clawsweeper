## Summary

Keep #11326 open. Current main still decodes built-in `web_fetch` response bodies through default UTF-8 paths before extraction, with no charset detection, GBK/GB2312 handling, manual encoding parameter, config key, docs entry, or regression coverage. The prior April 26 review comment is consistent with current main at 82262387651ab270b565f71533ce4937c9c00dc6.

## Best Possible Solution

Keep the issue open until built-in `web_fetch` performs charset-aware bounded byte decoding before extraction. The likely fix belongs in the shared response read path used by `web_fetch`: choose an encoding from HTTP Content-Type and early HTML meta charset where possible, support GBK/GB2312 through the runtime or an approved dependency, decide whether a manual `encoding` parameter/config key is needed, and add focused regression tests plus docs/config help updates while preserving SSRF, redirect, provider fallback, and max-response-byte behavior.
