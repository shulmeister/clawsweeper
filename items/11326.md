## Summary

Keep open. Current `main` does not implement GBK/GB2312-aware `web_fetch` decoding: the tool schema exposes only `url`, `extractMode`, and `maxChars`, and the response body is decoded with the default `TextDecoder()` path rather than charset detection or a caller-specified encoding.

## Evidence

- **Tool parameters do not include encoding:** `WebFetchSchema` defines only `url`, `extractMode`, and `maxChars`; there is no manual encoding parameter or charset option.
  - file: [src/agents/tools/web-fetch.ts:53](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/tools/web-fetch.ts#L53)
  - command: `nl -ba src/agents/tools/web-fetch.ts | sed -n '45,75p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Runtime execute path accepts no encoding override:** `createWebFetchTool()` reads only `url`, `extractMode`, and `maxChars` before calling `runWebFetch`, so callers cannot request GBK/GB2312 decoding.
  - file: [src/agents/tools/web-fetch.ts:633](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/tools/web-fetch.ts#L633)
  - command: `nl -ba src/agents/tools/web-fetch.ts | sed -n '628,648p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Response text decoding is UTF-8-default only:** `readResponseText()` constructs `new TextDecoder()` with no charset detection and no response-header/meta-tag handling, so GBK/GB2312 pages are not specially supported.
  - file: [src/agents/tools/web-shared.ts:97](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/tools/web-shared.ts#L97)

  - file: [docs/tools/web-fetch.md:26](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/tools/web-fetch.md#L26)
