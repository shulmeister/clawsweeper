## Summary

Keep #11326 open. Current main still decodes built-in web_fetch response bodies through the default UTF-8 fetch/TextDecoder path, has no tool parameter or config for response encoding, and does not inspect HTTP charset or HTML meta charset before Readability/basic extraction. The existing GBK support in the repo is for Windows process output, not web_fetch.

## Best Possible Solution

Keep the issue open until core web_fetch decodes response bytes with charset awareness. The likely fix belongs in `src/agents/tools/web-shared.ts` or a narrow helper used by `web_fetch`: preserve SSRF and maxResponseBytes behavior, detect charset from `Content-Type` and early HTML meta bytes, support labels such as `gbk`/`gb2312` when available, optionally add a manual encoding parameter if maintainers want that surface, and cover it with focused web_fetch tests plus `docs/tools/web-fetch.md` updates.
