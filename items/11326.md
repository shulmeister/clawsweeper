## Summary

Current main still does not implement GBK/GB2312 decoding for the built-in web_fetch path. The tool reads response bodies through UTF-8/default decoding, has no charset/meta detection or manual encoding parameter, and lacks the proposed encoding conversion dependencies. Provider plugins can work around selected fetches, but the default core behavior still matches the reported problem.

## Evidence

- **Built-in response decoding is UTF-8/default only:** readResponseText uses new TextDecoder() without an encoding label for streamed bodies and falls back to res.text(); there is no charset parsing from Content-Type or HTML meta before decoding.
  - file: [src/agents/tools/web-shared.ts:97](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/agents/tools/web-shared.ts#L97)
