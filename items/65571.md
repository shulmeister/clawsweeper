## Summary

Current `main` already contains the loopback-CDP/SSRF fix for this Linux browser-start regression, with matching regression tests, docs, and a changelog entry showing it shipped in `v2026.4.14`.

## Evidence

- **Availability probes now use loopback-aware CDP reachability policy:** `createProfileAvailability()` routes both HTTP and WebSocket CDP probes through `resolveCdpReachabilityPolicy()`, so the startup/status path no longer applies the raw browser SSRF policy directly to managed loopback CDP checks.
  - file: [extensions/browser/src/browser/server-context.availability.ts:72](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/browser/src/browser/server-context.availability.ts#L72)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '1,260p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Managed loopback `openclaw` profile explicitly bypasses browser SSRF control-plane blocking:** `resolveCdpReachabilityPolicy()` returns `undefined` for non-remote loopback profiles with `driver === "openclaw"`, with an inline comment explaining that browser SSRF policy should not self-block OpenClaw's own local CDP control plane.
  - file: [extensions/browser/src/browser/cdp-reachability-policy.ts:19](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/browser/src/browser/cdp-reachability-policy.ts#L19)
