## Summary

Current `main` has partial env-proxy support for ambient/trusted fetch paths, but the memory-search remote embedding path still uses guarded fetch in its default strict mode, so `HTTP_PROXY`/`HTTPS_PROXY` are not clearly respected there. Evidence is not strong enough to close #53007 as fixed or obsolete.

## Evidence

- **Memory remote HTTP path still uses default guarded fetch:** The memory host remote client calls `fetchWithSsrFGuard` with only `url`, `init`, `policy`, and `auditContext`; it does not opt into trusted env-proxy mode or pass an env-proxy dispatcher policy.
  - file: [packages/memory-host-sdk/src/host/remote-http.ts:29](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/packages/memory-host-sdk/src/host/remote-http.ts#L29)
  - command: `nl -ba packages/memory-host-sdk/src/host/remote-http.ts | sed -n '1,120p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Guarded fetch defaults to strict mode:** If no mode is provided, `resolveGuardedFetchMode` returns `GUARDED_FETCH_MODE.STRICT`.
  - file: [src/infra/net/fetch-guard.ts:113](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/infra/net/fetch-guard.ts#L113)
