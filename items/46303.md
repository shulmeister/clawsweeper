## Summary

Keep open. Current main still lacks the core behavior this PR targets: inbound debounce buffers are closure-local timer state with no process-wide flush, followup queues have no restart drain waiter, and the SIGUSR1 restart path only waits active command/embedded work before closing the server. The PR is not obsolete, but it still needs maintainer review and likely rework because the review context identifies real contract and teardown-order risks.

## Best Possible Solution

Keep the PR open and continue it as a targeted restart-drain fix: add a compatible internal/global inbound debounce drain primitive, wire SIGUSR1 restart to flush buffered inbound work and then wait for active turns plus followup queues, preserve the public SDK contract, and finish channel teardown/test coverage before merge.

## Risks / Open Questions

- Merging the PR as-is may break third-party plugin typings if the public `flushKey` return type changes instead of adding a separate internal delivery-count API.
- The global debouncer registry design needs careful channel lifecycle ownership so debouncers are not unregistered before ingress is fully closed and late inbound messages cannot enter an untracked buffer.
- The followup drain waiter should mirror the real drain predicate, including `droppedCount`, and should be ordered around active-turn draining so busy-session followups can actually run.
