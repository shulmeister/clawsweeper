## Summary

Keep this PR open. Current main still lacks a WhatsApp channel/account `readOnly` config surface and still has WhatsApp transport paths that can emit outbound presence, read receipts, pairing replies, messages, reactions, polls, and typing indicators without a transport-level read-only gate. The PR remains a plausible implementation candidate, but the review discussion identifies concrete fixes needed before merge.

## Best Possible Solution

Keep this PR open and finish the WhatsApp transport-level read-only implementation here or in a direct replacement patch. The shipped path should add channel/account `readOnly` config, resolve it per account, pass it into the WhatsApp inbound monitor, hard-gate outbound messages, media, polls, reactions, composing/presence, read receipts, pairing replies, and inbound/IPC send closures, while still creating pairing requests for operator approval without sending WhatsApp replies. Address the triaged review follow-ups before merge.
