## Summary

Current main still appears to have the reported raw JSON inconsistency: the file transport adds a local-offset top-level `time`, but it serializes the original tslog object unchanged, so `_meta.date` is not normalized. The linked PR #48717 correctly identified the likely root cause, but it was closed unmerged in the provided GitHub context and the current release tag still contains the same transport code.

## Best Possible Solution

Keep the issue open. Patch `src/logging/logger.ts` so the file transport computes one timestamp string and writes both top-level `time` and `_meta.date` to that same value in the serialized JSON payload, preserving the rest of `_meta`. Add a regression test in `src/logging/logger-timestamp.test.ts` that asserts `_meta.date === time` and neither uses a trailing `Z` under a deterministic timezone. Keep docs aligned with current semantics: `userTimezone` is prompt context, while file logs use host-local timestamp formatting unless product owners choose a broader logging timezone config.
