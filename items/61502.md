## Summary

Keep open. Current main still appears to have the reported Slack thread-routing bug: block action payloads preserve threadTs in the event text, but the session-key resolution path still omits it, while normal Slack thread messages are routed to thread-scoped session keys.

## Evidence

- **Reported root cause still matches current code:** The issue comment reports that openclaw 2026.4.14 still routed Slack button clicks to the base session because threadTs was parsed but not forwarded to session-key resolution. Current main has the same shape.
- **Block action handler omits threadTs when resolving session key:** enqueueSlackBlockActionEvent includes threadTs in the event payload, then calls resolveSlackSystemEventSessionKey with only channelId, channelType, and senderId before enqueueing the system event.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:695](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/slack/src/monitor/events/interactions.block-actions.ts#L695)
  - command: `nl -ba extensions/slack/src/monitor/events/interactions.block-actions.ts | sed -n '600,730p'`
  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Resolver contract cannot accept threadTs:** SlackMonitorContext.resolveSlackSystemEventSessionKey only accepts channelId, channelType, and senderId, so the interaction path has no way to derive the thread-suffixed target session key through this API.
  - file: [extensions/slack/src/monitor/context.ts:79](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/slack/src/monitor/context.ts#L79)
  - command: `nl -ba extensions/slack/src/monitor/context.ts | sed -n '70,105p'`
  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Normal Slack thread messages are thread-scoped:** Regular Slack message routing derives canonicalThreadId from incoming thread_ts and feeds it to resolveThreadSessionKeys; the colocated test asserts thread replies contain :thread:<ts>. That makes the current interaction session key mismatch meaningful.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:65](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L65)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '42,120p'`
  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Current tests codify the missing argument:** The block action test with container.thread_ts expects resolveSlackSystemEventSessionKey to be called without threadTs, so current automated coverage would not catch the reported mismatch.
  - file: [extensions/slack/src/monitor/events/interactions.test.ts:367](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/slack/src/monitor/events/interactions.test.ts#L367)
