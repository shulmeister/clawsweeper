## Summary

Current `main` still appears to have the reported auth gap for Control UI `[embed]` iframes. The UI renders embed previews from raw `/__openclaw__/canvas/...` URLs, canvas HTTP auth still requires a bearer header or scoped canvas capability, and the only read-path with query-token fallback is `assistant-media`, not canvas embeds.

## Evidence

- **Embed preview iframe uses the resolved canvas URL directly:** Control UI preview rendering passes `resolveCanvasIframeUrl(preview.url, ...)` straight into the iframe `src`; there is no auth-token injection at the iframe call site.
  - file: [ui/src/ui/chat/tool-cards.ts:290](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/chat/tool-cards.ts#L290)
  - command: `nl -ba ui/src/ui/chat/tool-cards.ts | sed -n '280,305p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Canvas URL resolver preserves path/scoped capability only:** `resolveCanvasIframeUrl()` accepts hosted `/__openclaw__/canvas` or `/__openclaw__/a2ui` paths and optionally rewrites them through a scoped capability host path, but it does not append any auth query parameter or otherwise carry bearer auth for normal Control UI embeds.
  - file: [ui/src/ui/canvas-url.ts:39](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/canvas-url.ts#L39)
  - command: `nl -ba ui/src/ui/canvas-url.ts | sed -n '1,80p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Canvas HTTP auth still requires bearer auth or canvas capability:** `authorizeCanvasRequest()` checks `getBearerToken(req)` first and otherwise only allows a live scoped canvas capability. A plain iframe request to `/__openclaw__/canvas/...` without a bearer header remains unauthorized.
  - file: [src/gateway/server/http-auth.ts:70](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/server/http-auth.ts#L70)
  - command: `nl -ba src/gateway/server/http-auth.ts | sed -n '45,85p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Query-token fallback exists for assistant media, not canvas embeds:** Control UI read auth can accept `?token=` only when `allowQueryToken` is enabled, and `handleControlUiAssistantMediaRequest()` explicitly enables that path. I did not find an equivalent canvas-read path.
  - file: [src/gateway/control-ui.ts:210](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/control-ui.ts#L210)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Tests reinforce the asymmetry:** UI tests expect `assistant-media` URLs to include `token=session-token`, while gateway canvas auth tests still assert bare canvas HTTP requests return `401` without bearer auth or capability.
  - file: [ui/src/ui/chat/grouped-render.test.ts:660](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/chat/grouped-render.test.ts#L660)

## Risks / Open Questions
