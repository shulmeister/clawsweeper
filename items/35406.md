## Summary

Keep #35406 open. Current main still does not guard the generic `read` tool against binary document formats before delegating to the pinned upstream read implementation, which still treats any non-image as UTF-8 text. OpenClaw now has a separate `pdf` tool, but that does not solve the reported `read(path="document.docx")` / `.xlsx` / `.pptx` token-waste behavior, and the related fix PRs have not landed on main.

## Best Possible Solution

Keep this issue open until a small core read-tool guard lands. The likely fix belongs in `src/agents/pi-tools.read.ts`: detect known binary document extensions/MIMEs before `executeReadWithAdaptivePaging`, return a concise structured fallback that points PDFs to the `pdf` tool where available and tells users DOCX/XLSX/PPTX need a document parser/conversion tool, and add host plus sandbox regression tests for at least `.docx`, `.xlsx`, `.pptx`, and `.pdf`. PR #57600 can remain the active implementation vehicle if maintainers want that patch revived.
