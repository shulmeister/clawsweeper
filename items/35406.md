## Summary

Keep #35406 open. Current main still lacks a PDF/Office binary-document guard in the generic `read` tool: the OpenClaw wrapper validates params, delegates to the upstream reader, then only performs truncation cleanup and image normalization/sanitization. Existing MIME/document infrastructure and the separate `pdf` tool help adjacent workflows, but they do not prevent `read(path="*.docx")`, `read(path="*.xlsx")`, `read(path="*.pptx")`, or `read(path="*.pdf")` from entering the normal read path. The provided related fix PR #35494 is closed unmerged, and #57600 remains an open draft candidate rather than shipped behavior.

## Best Possible Solution

Keep #35406 open and land #57600 or a maintainer-owned equivalent. The fix should add a small read-tool preflight before `executeReadWithAdaptivePaging`, reuse `src/media/mime.ts` for PDF and Office detection, cover both host and sandbox read paths, return a concise fallback that points PDFs to the `pdf` tool and Office files to parser/conversion tooling, and add regression tests for `.pdf`, `.docx`, `.xlsx`, and `.pptx`.
