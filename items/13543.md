## Summary

Keep #13543 open. Current main still supports session/agent sandbox modes, sandbox backend plugins, sandbox-browser routing when the whole session is sandboxed, sandbox tool allow/deny filtering, and exec-specific host routing, but it does not implement the requested per-tool execution placement policy that can force `web_fetch`/`browser` into a sandbox while keeping `exec`/`message`/`cron` on the host in the same session. The related merged Docker sandbox work (#29974) is useful setup/backend context, but it only partially addresses this issue and does not add selective per-tool routing.

## Best Possible Solution

Keep this as a core sandbox/tool-routing enhancement. The best implementation would add a documented, schema-backed per-tool execution policy, route selected tools such as `web_fetch` and `browser` through a sandbox backend while keeping tools such as `exec`, `message`, `cron`, and `gateway` on the host in the same session, and add tests/docs that clearly distinguish execution placement from `tools.sandbox.tools` availability filtering, `tools.exec.host`, and browser `target` selection.

## Risks / Open Questions

- Closing now would conflate sandbox tool allow/deny filtering, exec-only host routing, and sandbox-browser support with the requested cross-tool execution placement policy.
- A real fix likely needs coordinated schema, typed config, runtime sandbox resolution, tool construction, docs, and regression tests so `web_fetch`/`browser` can be sandboxed while host-only tools remain on the host in the same session.
