## Summary

Keep #13543 open. Current main still has session/agent-level sandboxing, sandbox-only tool availability filtering, browser target selection, and exec host routing, but not a schema-backed per-tool placement policy that can force web_fetch/browser into a sandbox while keeping exec/message/cron/gateway on host in the same session. The existing issue discussion also already called out this distinction and recommended keeping the item open.

## Best Possible Solution

Keep this open as a core sandbox/tool-routing enhancement. The best path is a documented, schema-backed per-tool execution placement policy that can route selected tools such as web_fetch and browser through a sandbox backend while selected host tools such as exec, message, cron, and gateway remain on host in the same session, with tests distinguishing placement from tools.sandbox.tools availability filtering, browser target selection, and tools.exec.host/elevated behavior.

## Risks / Open Questions

- Closing this now would conflate sandbox tool allow/deny, browser target selection, or exec host routing with the requested per-tool execution placement policy.
- A correct implementation likely needs coordinated schema, docs, runtime routing, tests, and plugin-tool context changes so selective placement does not weaken sandbox or elevated boundaries.
