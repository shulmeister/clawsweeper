## Summary

Current main already prevents the reported health-monitor restart loop for intentionally skipped/disabled account entries such as `accounts.default`, and the behavior is present in the latest release tag.

## Evidence

- **Skipped accounts are recorded disabled before provider startup:** When a channel account resolves but `plugin.config.isEnabled(account, cfg)` is false, startup records the runtime as `enabled: false`, `configured: true`, `running: false`, then returns without starting the provider.
  - file: [src/gateway/server-channels.ts:351](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/gateway/server-channels.ts#L351)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '340,410p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Runtime snapshots preserve the disabled state used by the monitor:** `getRuntimeSnapshot()` recomputes each listed account through the plugin resolver and assigns `next.enabled = enabled`, so a `default` account whose plugin `isEnabled()` returns false is visible to the health monitor as disabled.
  - file: [src/gateway/server-channels.ts:645](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/gateway/server-channels.ts#L645)

- **Health policy treats disabled/unconfigured accounts as unmanaged healthy:** `evaluateChannelHealth()` returns `{ healthy: true, reason: "unmanaged" }` before the `not-running` branch when `enabled === false` or `configured === false`, which directly covers skipped account entries.
  - file: [src/gateway/channel-health-policy.ts:48](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/gateway/channel-health-policy.ts#L48)
