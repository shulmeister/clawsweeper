## Summary

Keep #68264 open. Current main has some related canvas-preview and media paths, but it does not implement the reported expectation that `canvas` present/navigate or browser navigation reliably produces a visible webpage/canvas confirmation in chat. The issue is recent, has no maintainer discussion, and the related ClawSweeper matches are adjacent rather than canonical duplicates.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The likely fix should either clarify the intended boundary and improve errors/guidance for `canvas` without a paired node, or add a first-class visual confirmation path: for example, have canvas present/navigate optionally return or trigger a snapshot/media artifact, and/or have browser navigate guide or chain into screenshot/snapshot when the user asks for visual confirmation. Add focused tests covering the Control UI rendering path and the malformed/local-model tool-call failure mode.
