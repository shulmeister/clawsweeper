## Summary

Keep open. Current main still matches the reported contract gap: `sessions.list`/`sessions_list` is backed by the persisted session store, the health monitor restart path only stops and starts the channel account, and Discord reconnect status updates do not recreate session-store rows. Discord session rows are still materialized by inbound message/session recording, so a quiet workspace after reconnect can remain absent from `sessions_list` while channel health/status is OK. The provided maintainer-style comment already identified this as a by-design gap and left the docs-vs-session-materialization direction to the channel-health owner; no current docs/code change resolves that choice, and related ClawSweeper items are adjacent rather than canonical duplicates.

## Best Possible Solution

Keep #70420 open for the channel-health owner. The conservative path is to document the contract clearly: `sessions_list` is a persisted conversation/session-store view, not a Discord/channel socket-liveness check, and heartbeat health checks should use channel status/readiness/health snapshots instead. If maintainers want `sessions_list` to become a liveness signal, implement an explicit synthetic channel heartbeat/session row on successful channel start/reconnect and cover it with Discord reconnect regression tests.
