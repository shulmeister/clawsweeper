## Summary

Keep open. Current `main` still has the Linux-only fnm gap described in the issue: `resolveLinuxUserBinDirs` adds only `current/bin`, while the Darwin resolver already uses `aliases/default/bin`, and the Linux tests still do not cover the modern fnm path. There is already one open fix PR (#68180) and one closed-but-unmerged PR (#68219) that both implement the same change, which reinforces that this remains a real unresolved bug rather than stale noise.

## Evidence

- **Current Linux resolver still only checks fnm current/bin:** On current `main` (`c2139635ff82b367b734ea052487258b1b33f060`), `resolveLinuxUserBinDirs` adds `appendSubdir(env?.FNM_DIR, "current/bin")` and `${home}/.fnm/current/bin`, but no `aliases/default/bin` entries.
  - file: [src/daemon/service-env.ts:167](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/daemon/service-env.ts#L167)

  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Darwin resolver already uses aliases/default/bin:** The same file's macOS resolver explicitly uses `appendSubdir(env?.FNM_DIR, "aliases/default/bin")` and includes both `~/Library/Application Support/fnm/aliases/default/bin` and `~/.fnm/aliases/default/bin`, matching the issue's asymmetry claim.
  - file: [src/daemon/service-env.ts:145](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/daemon/service-env.ts#L145)
