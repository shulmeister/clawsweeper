## Summary

Keep #71170 open. Current main still decides the reminder-guard note only from core `successfulCronAdds` plus existing OpenClaw cron coverage, and it has no `reminderGuard` config key or equivalent external-plugin scheduling signal. The linked issue #70666 remains a real core-runtime/config seam problem rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep this PR open or replace it with a narrow successor that adds `agents.defaults.reminderGuard` plus a per-agent override, wires it into `runReplyAgent`, updates config types/zod/generated help/docs, and adds unit plus reminder-guard integration tests. If maintainers prefer plugin tools to signal scheduled events instead, track that as a separate plugin SDK/runtime contract proposal; current main does not provide enough external-plugin API surface to close this as ClawHub-only.

## Risks / Open Questions

- A config opt-out can hide real missed core reminders for opted-out agents, so docs should clearly scope it to agents backed by a trusted external scheduler.
- A richer plugin-tool scheduling signal may be the better long-term contract, but that is a separate SDK/runtime design decision and should not be required for this narrow false-positive fix.
- Before merge, maintainers still need normal CI/gates for the PR branch, especially config schema/docs generated baseline checks and agent runtime tests.
