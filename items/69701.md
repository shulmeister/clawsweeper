## Summary

Keep PR #69701 open. Current main still does not contain the PR's Windows-specific listener host guard: `resolveGatewayListenHosts("127.0.0.1")` still probes `::1` and returns both loopback hosts when IPv6 is bindable, and gateway startup then listens on every resolved host. The related #69674 has an adjacent health-probe reservation fix on main, but that does not supersede this PR's claimed Windows dual-bind fix.

## Best Possible Solution

Keep the PR open for maintainer review or land an equivalent fix on main: add a Windows-only early return in `resolveGatewayListenHosts` for loopback binds, keep non-Windows dual-loopback behavior, and retain focused tests for both the Windows skip and non-Windows `::1` path. The existing health-probe reservation should remain as separate adjacent hardening, not as a replacement for this listener binding fix.
