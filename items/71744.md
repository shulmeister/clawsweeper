## Summary

Keep #71744 open. Current main has useful per-attempt fallback data in `FallbackSummaryError` and `model_fallback_decision` diagnostic logs, plus lifecycle events when a fallback succeeds, but it does not emit the requested `runtime_event`/`fallback_step` record for each fallback transition or for chain exhaustion. In the mixed-cause failure path, non-verbose operator-facing replies are still intentionally reduced to a generic failure, so the reporter's observability gap remains actionable. Related #62684 is closed and #71710 is a separate root incident, not a canonical fix for this fallback-event gap.

## Best Possible Solution

Keep the issue open and implement a core-owned, sanitized fallback-step observability event on the existing agent/lifecycle or plugin runtime event surface. Emit it from the model fallback path when moving from one candidate to the next and when the chain is exhausted, with provider/model, reason, truncated detail, chain position, next candidate, and final outcome. Add focused tests for primary failure followed by fallback failure, and ensure Telegram/operator-facing or documented runtime-event consumers can see the primary failure without enabling verbose user replies.
