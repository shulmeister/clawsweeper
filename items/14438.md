## Summary

Keep #14438 open. Current main still lacks a supported plugin runtime hot-reload, file watcher that reloads plugin code in-process, or user-facing jiti cache invalidation command. The existing implementation has internal cache/reset paths and a gateway watch mode that restarts the process, but the documented and implemented plugin developer path still requires restarting the live Gateway after plugin code/config changes.

## Best Possible Solution

Keep the issue open as core plugin loader and Gateway lifecycle work. A complete solution should add a supported developer reload path, likely a Gateway RPC plus `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, that invalidates plugin registry, jiti/module state, and plugin-owned registrations, reloads the selected plugin safely, refreshes active Gateway methods/tools/hooks, and documents container/wrapper process behavior. If native plugin code hot reload is intentionally unsupported, maintainers should record that product decision explicitly in docs and CLI guidance before closing.
