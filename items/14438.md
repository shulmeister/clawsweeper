## Summary

Keep #14438 open. Current main still does not provide a supported plugin runtime hot-reload, selected-plugin reload RPC/CLI, file watcher that reloads plugin code in-process, or user-facing jiti cache clear command. The existing paths cover config reload planning, process-level dev watching, and internal/test cache resets, but the documented plugin developer path still tells users to restart the Gateway after plugin code changes.

## Best Possible Solution

Keep the issue open as core plugin loader and Gateway lifecycle work. A complete solution should add a supported developer reload path, likely a Gateway RPC plus `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, that invalidates plugin registry, jiti/module state, plugin-owned registrations, active tools/methods/hooks/services, and documents container/wrapper process behavior. If native plugin code hot reload is intentionally out of scope, maintainers should record that decision explicitly in docs and CLI guidance before closing.
