## Summary

Keep #8321 open. Current main still has no per-cron-job cache/context-pruning override: cron job types, gateway schemas, CLI/UI surfaces, persistence, and the isolated runner only expose existing payload options such as lightContext, while embedded pruning is still driven by agents.defaults.contextPruning. The closed, unmerged PR #8341 attempted the right shape, and #24597's direct script/no-LLM idea remains complementary rather than a replacement for agent-run cron job pruning control.

## Best Possible Solution

Keep #8321 open. Implement an additive top-level cron job override such as contextPruning?: AgentContextPruningConfig using the existing mode: "off" | "cache-ttl" contract, thread it through cron create/update/list schemas, normalization, persistence, CLI, Control UI, docs, and protocol clients, then merge it into cfgWithAgentDefaults before isolated cron invokes the embedded runner. Add regression coverage proving a job-level mode: "off" overrides global or per-agent agents.defaults.contextPruning.mode: "cache-ttl"; track direct script/no-LLM cron execution separately.

## Risks / Open Questions

- The implementation crosses public cron types, persistence, gateway validation, CLI, Control UI, docs, and isolated-run behavior; accepting a field without feeding it into the embedded runner would not solve the token-cost problem.
- Direct script/no-LLM cron execution can reduce cost for deterministic jobs, but it does not replace a per-job pruning/cache override for agent-run cron jobs.
