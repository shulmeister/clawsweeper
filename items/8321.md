## Summary

Keep #8321 open. Current main still has no per-job `disableCache` or `contextPruning` field in cron job types, Gateway protocol schemas, CLI/UI cron surfaces, or the agent cron tool. Isolated cron execution still builds pruning behavior from global/per-agent agent defaults, and the embedded runner's pruning/cache-TTL hooks read only `config.agents.defaults.contextPruning`. The related implementation PR #8341 was closed unmerged, while #24597 covered a broader direct-script/no-LLM cron path rather than this per-job cache/pruning override.

## Best Possible Solution

Keep #8321 open. Implement an additive per-job cron `contextPruning?: AgentContextPruningConfig` override using the existing `mode: "off" | "cache-ttl"` contract, then thread it through cron create/update/list schemas, normalization/store, CLI/UI, agent cron tool schemas, isolated-run config construction, and docs. Add regression tests proving a job-level `mode: "off"` overrides global/per-agent `agents.defaults.contextPruning.mode: "cache-ttl"`. Keep direct script/no-LLM execution tracked separately from this cache/pruning-control request.
