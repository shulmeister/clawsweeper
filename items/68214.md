## Summary

Keep open. Current main appears to address the motivating WhatsApp allowlist privacy leak by making pairing replies occur only on the `dmPolicy="pairing"` path and by testing silent allowlist rejection, but the issue also asks for an unauthorized/access-denied hook with suppress-default behavior, and that API is not present.

## Best Possible Solution

Keep the shipped WhatsApp behavior that only sends pairing replies for `dmPolicy="pairing"`, then decide the product/API shape for a typed unauthorized/access-denied hook at the shared channel access-control boundary. The hook should expose channel/account/sender identity with safe redaction, deny reason/reason code, default action, and an explicit suppress/replace result, with docs and channel contract tests. Do not add `suppressPairingReply` unless maintainers still want a config shortcut after the hook/API design is settled.
