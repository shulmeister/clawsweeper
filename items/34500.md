## Summary

Close #34500 as superseded by the broader open tracker #48779 and active implementation PR #48851. Current main still does not implement the requested model/API call count, but the remaining work is now tracked in one canonical issue/PR pair covering the same usage-footer ambiguity and the adjacent /status surface.

## Best Possible Solution

Close this older standalone issue as superseded, keep the remaining product/code work tracked in #48779 and #48851, and land a complete implementation there that tracks per-turn API/model calls through the embedded runner, persists the count where needed, and renders it in the response usage footer and /status without changing single-call output unnecessarily.
