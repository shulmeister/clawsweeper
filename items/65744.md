## Summary

Current `main` already contains reply-routing and session-delivery hardening that keeps Feishu DM replies pinned to the originating DM instead of falling back to another surface, with direct Feishu regression coverage added in commit `6d271762ab50fb4376b8859bcf55eefb2d3814ea`. The latest tagged release `v2026.4.23` predates that commit, so this appears fixed on `main` but not in a shipped tag I could verify.

## Evidence

- **Cross-surface reply routing now pins to originating Feishu target:** `dispatchReplyFromConfig` resolves one effective reply route from `OriginatingChannel` / `OriginatingTo` and uses `routeReplyToOriginating(...)` for routed payload delivery. The paired regression test covers `Provider=webchat`, `OriginatingChannel=feishu`, `OriginatingTo=ou_feishu_direct_123` and expects routing back to that Feishu DM target.
