## Summary

Keep #39610 open. Current main at 9089e6b595ba33b4afc27b99c56ace80c4fbbc7b still supports only global `agents.defaults.userTimezone`; it does not accept `agents.list[].userTimezone`, does not expose an agent-aware timezone resolver, and key prompt, heartbeat, envelope, gateway timestamp, cron/current-time, compaction, and system-event paths still resolve from defaults only. The provided PR context shows #39610 is the canonical unmerged implementation for closed #39579 and has addressed review comments, so the work remains useful rather than obsolete.

## Best Possible Solution

Keep #39610 open as the canonical implementation path for per-agent `userTimezone`. The maintainer-ready solution is to add `agents.list[].userTimezone` to typed config, strict runtime schema, generated schema, and docs; centralize resolution as per-agent override -> `agents.defaults.userTimezone` -> host timezone; thread the resolved agent id through prompt params, session status/current-time helpers, heartbeat active-hours and prompt injection, envelope/system-event timestamps, gateway/chat timestamp injection, cron, memory flush, post-compaction context, and plugin inbound-envelope surfaces; validate IANA timezone strings consistently; then rebase and run the relevant changed gates.
