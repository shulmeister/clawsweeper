## Summary

Keep PR #39610 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still implements only global `agents.defaults.userTimezone`; it does not define, validate, document, or consistently resolve `agents.list[].userTimezone`. The provided PR and related issue context show #39610 is the canonical unmerged implementation for closed issue #39579, not obsolete cleanup.

## Best Possible Solution

Keep #39610 open as the canonical implementation path for per-agent `userTimezone`. The maintainer-ready path is to rebase it onto current main, add `agents.list[].userTimezone` to config types, runtime schema, generated schema/help, and docs; centralize resolution as per-agent override -> `agents.defaults.userTimezone` -> host timezone; thread the resolved active agent id through system prompt params, `session_status`, current-time helpers, heartbeat active-hours and heartbeat prompt injection, envelope/system-event timestamps, gateway/chat timestamp injection, cron, memory flush, post-compaction context, and plugin inbound-envelope surfaces; validate IANA timezone strings consistently; then run the appropriate changed gates.
