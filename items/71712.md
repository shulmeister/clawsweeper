## Summary

Keep open. Current main has an owner-only cron agent tool and gateway `cron.add`, but it does not implement the RFC’s hardened agent-facing scheduling contract: no playbook allowlist, no required recurring `activeUntil`, no gateway-injected non-forgeable `createdBy` provenance, no per-agent schedule quota, and no audit-ledger entry for schedule creation/modification. The related items do not supersede this: #71711 only tracks expiration, #28584 is closed cron chaining, and #71937 was a closed design-only harness RFC that cross-referenced this as a remaining gap.

## Best Possible Solution

Keep the RFC open as a core security/API design item. A suitable implementation would add or design a gateway-mediated scheduling path that only accepts registered playbook IDs plus schema-validated params, injects immutable creator/session provenance at the gateway, requires an expiration for recurring schedules, enforces per-agent quotas, and records schedule mutations in an audit surface. Coordinate the expiration part with #71711 and provenance semantics with existing input-provenance/harness work, but do not treat those as superseding this issue.

## Risks / Open Questions

- Closing this as implemented would conflate the existing owner-only cron control plane with the narrower non-forgeable agent scheduling contract requested here.
- The best fix touches scheduler schema, gateway authorization/provenance, cron persistence, task/audit surfaces, and docs, so it needs maintainer product/security review rather than automated cleanup.
