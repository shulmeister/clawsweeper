## Summary

Current `main` already supports per-agent exec policy overrides via `agents.list[].tools.exec`, and per-agent exec allowlists already exist in the host-local approvals store. That covers the reported least-privilege use case well enough, even though the allowlist lives in `~/.openclaw/exec-approvals.json` rather than directly under `openclaw.json` agent entries.

## Risks / Open Questions

- The implementation does not place allowlists inside `agents.list[].tools.exec`; operators must still manage allowlist entries in `~/.openclaw/exec-approvals.json` or the UI/CLI approvals flow.
- This appears to be on current `main` after the latest tagged release `v2026.4.22`, so there may not be a shipped release containing it yet.

## Close Comment

Closing from a Codex review because current `main` already covers this request well enough. Per-agent exec overrides are now supported under `agents.list[].tools.exec` for fields including `security`, `ask`, `host`, and `strictInlineEval` (`src/config/zod-schema.agent-runtime.ts:474`, `src/auto-reply/reply/get-reply-directives.ts:587`, `src/infra/exec-approvals-policy.test.ts:535`). The remaining allowlist piece is already agent-scoped in the host approvals store at `~/.openclaw/exec-approvals.json` via `agents.<id>.allowlist`, documented in `docs/tools/exec-approvals-advanced.md:118`. The implementation evidence on `main` is commit `38caa6832d4e1d344736d635b0c5fdad37e4430b`. I’m closing this as implemented on `main`; if you specifically want agent allowlists moved into `openclaw.json` instead of the approvals store, please open a narrower follow-up request for that config-surface change.
