## Summary

Keep open. Current `main` does not implement `api.executeTool`, and the PR as described still appears to conflict with existing plugin-tool semantics: factory-registered tools often have no stored `names`, optional-tool allowlist enforcement currently lives in the normal resolution path, and normal tool execution also carries hook/policy context that the proposed direct call path would skip.

## Evidence

- **`executeTool` is not on current main:** `OpenClawPluginApi` on `main` ends with `registerMemoryEmbeddingProvider`, `resolvePath`, and `on`; there is no `executeTool` method in the public type or the API builder wiring. A `git grep` against release `v2026.4.23` also returned no `executeTool` matches, so this is not already implemented on main or in the latest release.
  - file: [src/plugins/types.ts:2160](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/plugins/types.ts#L2160)
  - command: `nl -ba src/plugins/types.ts | sed -n '2148,2190p'; nl -ba src/plugins/api-builder.ts | sed -n '45,200p'; git grep -n "executeTool" v2026.4.23 -- src/plugins src/plugin-sdk test/helpers extensions/lobster`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Current tool registration does not persist factory tool names by default:** `registerTool()` stores `names` only from `opts.name`/`opts.names`, plus `tool.name` only for non-factory tool objects. Factory registrations without explicit opts therefore land in `registry.tools` with `names: []`. That means the PR's proposed `entry.names.includes(toolName)` lookup would miss valid existing registrations.
  - file: [src/plugins/registry.ts:332](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/plugins/registry.ts#L332)
  - command: `nl -ba src/plugins/registry.ts | sed -n '332,360p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Bundled plugins already use the factory-without-explicit-name pattern:** The bundled browser plugin registers its tool as a factory without passing `opts.name`, and the browser bundled-plugin fixture does the same. This is exactly the common pattern that would not be found by a lookup limited to stored `entry.names`.
  - file: [extensions/browser/plugin-registration.ts:28](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/plugin-registration.ts#L28)
  - command: `nl -ba extensions/browser/plugin-registration.ts | sed -n '20,40p'; nl -ba test/helpers/browser-bundled-plugin-fixture.ts | sed -n '18,45p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Optional-tool policy is enforced in the normal resolution path, not at raw registry lookup:** `resolvePluginTools()` resolves factory output and filters optional tools by actual `tool.name` plus plugin/tool allowlist rules. The plugin docs also state optional tools require explicit user allowlisting. A direct `registry.tools` -> `tool.execute()` path would bypass that existing gate unless it reuses the same policy path.
  - file: [src/plugins/tools.ts:149](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/plugins/tools.ts#L149)
  - command: `nl -ba src/plugins/tools.ts | sed -n '120,175p'; nl -ba docs/plugins/building-plugins.md | sed -n '208,242p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Normal tool execution carries richer context and runs pre-tool hooks:** Plugin tool factories can depend on `fsPolicy`, `workspaceDir`, `sessionKey`, `senderIsOwner`, and `sandboxed` in `OpenClawPluginToolContext`. The normal adapter path also runs `before_tool_call` before invoking `tool.execute()`. That supports the PR discussion concerns about a minimal `{ config }` context and a direct execution path skipping established hook/policy behavior.
  - file: [src/plugins/tool-types.ts:7](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/plugins/tool-types.ts#L7)

## Risks / Open Questions

- If merged as described in the PR body, `executeTool` would fail for valid factory-registered tools that do not declare `opts.name`/`opts.names`.
- A direct self-execution path would bypass current optional-tool allowlist enforcement unless it explicitly reuses `resolvePluginTools` semantics.
- Some tools may behave differently or unsafely when created without the normal runtime context fields and without `before_tool_call` processing.
