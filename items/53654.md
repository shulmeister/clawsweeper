## Summary

Current main still does not implement Discord edit-to-reprocess or delete-to-cancel. The bundled Discord plugin registers message-create, reaction, thread-update, and optional presence listeners only; no Discord message update/delete listener is present, no source Discord message ID to active run mapping exists, and the generic chat abort surface is not wired from Discord message deletes. Related #38505 remains open and covers only the edit half, so it does not supersede this broader request.

## Best Possible Solution

Keep this open and implement it in the bundled Discord plugin. Add Discord MessageUpdateListener and MessageDeleteListener handling in extensions/discord/src/monitor/listeners.ts, register them from extensions/discord/src/monitor/provider.startup.ts, route edited content through the existing preflight/message processing path with mention/content-change gating and edit-aware replay keys, and track the triggering Discord message ID to the accepted run or pending job so delete events can abort only matching active work through the existing chat abort/run abort surface. Coordinate with #38505 for the edit-only half while keeping this broader issue as the delete-to-cancel tracker too.

## Risks / Open Questions

- Edit handling must adjust replay/dedupe behavior because the current replay key is account/channel/message-id based and would otherwise suppress the edited payload.
- Delete-to-cancel needs a Discord source-message-id to active run/session mapping precise enough to avoid aborting a newer unrelated run in the same Discord channel session.
