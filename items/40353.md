## Summary

Keep open. Current main has useful diagnostic and OpenTelemetry trace primitives, but it does not implement the issue's central request: request-scoped trace IDs generated at gateway entry and automatically attached to relevant gateway/channel/MCP/provider log lines. The author's linked implementation notes and fork commit are useful prior art, but that commit is not present in this checkout's main history.

## Best Possible Solution

Keep the issue open and implement it by extending the existing diagnostic trace primitives: create or parse a trace context at HTTP and WS gateway entry, carry it through `GatewayRequestContext` and plugin request scope, make subsystem/file/diagnostic logs attach the active trace automatically, propagate safe `traceparent` or configured headers to provider/MCP calls where appropriate, and add focused tests for gateway request logs, channel webhook/message diagnostics, model/provider calls, and OTEL log/span correlation.
