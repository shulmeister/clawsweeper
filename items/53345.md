## Summary

Keep this issue open. Current main already has the Korean Control UI locale, docs, and locale tests, and that UI portion is present in the v2026.4.24 release tag. The remaining requested behavior, agent-side primary-language persistence across tool-heavy turns, is still not implemented in the default agent system prompt, and the paired PR #53346 was closed unmerged.

## Best Possible Solution

Keep this issue open but narrow it to the remaining agent-side work. Preserve the shipped Korean Control UI locale, then design and test primary-language persistence in the agent prompt or conversation policy with explicit precedence for code, logs, paths, quotes, and user overrides. Add Korean search localization only through provider-aware tool/config behavior rather than assuming one global `search_lang` field works everywhere.

## Risks / Open Questions

- A broad language-persistence prompt needs clear precedence so explicit user instructions, code, logs, file paths, and quoted tool output are not translated incorrectly.
- Search localization should be provider-aware because Brave, Perplexity, SearXNG, native Codex/OpenAI search, and other providers expose different language controls.
