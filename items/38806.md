## Summary

Keep #38806 open. Current `main` at c7a0d9b1889ca09ab5bfd84cda39a8d50d3d5768 still preserves Feishu `root_id` and `thread_id` on inbound events, but the P2P/direct reply delivery path still gates thread-aware reply behavior behind group-chat logic. A P2P thread message can still lose both the thread reply flag and reply target before reaching the Feishu send layer, matching the reported standalone-message symptom. The provided related context also shows PR #38808 remains the targeted open fix candidate, while related PRs #38814 and #38996 were closed unmerged.

## Best Possible Solution

Keep #38806 open and review or land a targeted Feishu fix like PR #38808: detect P2P thread messages with `isDirect && Boolean(ctx.threadId)`, preserve P2P quote-only behavior when only `root_id` is present, pass the thread root plus `reply_in_thread` semantics through the dispatcher for P2P threads, and add regression coverage for both P2P thread replies and P2P plain quote replies.
