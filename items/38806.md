## Summary

Keep #38806 open. Current main at 67b9167b808f2ff065c91665d6f36256a5c23ef2 still preserves Feishu P2P thread fields on inbound events, but the reply-delivery path keeps thread-aware behavior behind group-chat checks. A P2P thread message can still lose reply_in_thread semantics and reply-target metadata before the Feishu send layer, matching the reported standalone-message symptom. The targeted fix candidate PR #38808 is still open and unmerged; related PRs #38814 and #38996 are closed unmerged.

## Best Possible Solution

Keep #38806 open and fix it through PR #38808 or an equivalent targeted Feishu patch: detect P2P thread messages with isDirect && Boolean(ctx.threadId), use the P2P thread root as the reply target, set replyInThread/threadReply for that case, avoid skipReplyToInMessages for direct threads, preserve root_id-only P2P quote behavior, and add regression coverage for both P2P thread replies and P2P plain quote replies.
