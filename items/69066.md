## Summary

Keep #69066 open. Current main has tactical direct-loopback backend shared-auth behavior, but it does not implement the RFC's separate service identity axis, service token config, env injection, or rotation path. Trusted-proxy loopback auth still intentionally fails closed, and the public config/schema expose only the existing gateway auth modes.

## Best Possible Solution

Keep the RFC open for maintainer gateway/auth review. The concrete path is to decide whether OpenClaw wants a first-class service identity axis, then add the config/schema/docs, service token persistence/rotation, GatewayClient injection, and migration of internal callers behind tests. If maintainers instead choose the current shared-auth backend exception as the long-term design, close only after that decision is recorded in the issue with rationale.
