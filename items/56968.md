## Summary

Current `main` still has the reported status bug. The status snapshot path unconditionally depends on built-in `agents.*.memorySearch` config, so third-party memory-slot plugins can still resolve to `null` and render as `enabled (plugin …) · unavailable` even when a plugin runtime is active. Latest release `v2026.4.23` shows no change in this area versus current `main`, and the existing tests only cover `memory-core` / explicit `memorySearch` cases, not third-party memory-slot plugins.

## Evidence

- **Status scan still gates on built-in memorySearch config:** `src/commands/status.scan.shared.ts:184` returns early unless `resolveMemoryConfig(cfg, agentId)` succeeds, then also requires either explicit memorySearch config or an existing built-in store path before it ever asks the active memory plugin runtime for status. That means a third-party memory-slot plugin still falls through to `null` when built-in `memorySearch` is disabled.
  - file: [src/commands/status.scan.shared.ts:184](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/commands/status.scan.shared.ts#L184)
