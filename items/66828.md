## Summary

Keep #66828 open. Current main still contains the reported cron bottleneck: isolated cron agent runs pass `lane: "cron"`, `runEmbeddedPiAgent` resolves that through `resolveGlobalLane`, and `resolveGlobalLane` still maps `CommandLane.Cron` to `CommandLane.Nested` while gateway concurrency is configured for `cron`, `main`, and `subagent`, not `nested`. The linked PR #69240 is a relevant partial fix path for Bug A, but it is still open/unmerged and only covers the cron-lane half of this issue; the subagent cleanup packaging claim still needs separate verification or a concrete repro.

## Best Possible Solution

Keep the issue open until the cron-lane fix lands on main and the subagent cleanup packaging report is verified. The best product path is to land or rework #69240 or an equivalent change so top-level cron dispatch stays on the cron lane while only true inner operations use nested/per-session nested lanes, with regression tests. Then either split Bug B into a narrower issue with npm tarball/runtime-module proof, or close that part only after source, package contents, and cleanup/finalize behavior prove it is no longer present.
