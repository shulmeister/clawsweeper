## Summary

Keep this issue open. Current main has added Telegram inbound reaction handling, but it exposes reactions as human-readable session system events, not as the stable structured hook/event contract the report asks for. There is still no typed plugin/internal hook carrying reaction fields such as emoji, Telegram message id, account/session context, and sender metadata for downstream feedback integrations.

## Best Possible Solution

Keep the issue open and implement an additive structured channel-reaction hook/API seam. The likely product path is a generic plugin/internal hook such as `message_reaction` or `channel_event` with typed fields for channel/account, session/conversation, Telegram chat id, target message id, emoji/reaction operation, sender id/name/username, and raw metadata where needed. Telegram should emit that hook from the existing `message_reaction` handler, while feedback recording itself should live in a plugin or managed hook built on the new seam. Add focused Telegram handler tests, plugin hook contract tests, and docs under the public plugin/channel hook docs plus Telegram reaction docs.

## Risks / Open Questions

- Current main already improves the 2026.4.8 behavior by receiving Telegram reactions and surfacing them to the agent as system events, so the remaining scope should be kept narrow: a structured downstream hook/event contract, not basic reaction receipt.
- System events may be enough for agent-visible reaction awareness, but they are not a stable structured integration surface for feedback-recording hooks or plugins.
