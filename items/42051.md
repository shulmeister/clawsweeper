## Summary

PR #42051 should close as superseded by open PR #57524. The submitted fix is review-blocked because it uses `os.userInfo({ username })` for a named-user lookup, which Node does not do, and the same sudo/systemd install problem is now tracked by a broader follow-up PR.

## Best Possible Solution

Close #42051 as superseded and continue the remaining product/code decision in #57524 or a successor PR: either reject sudo gateway installs at the CLI with clear guidance, or implement a correct named-user home lookup plus ownership/scope handling with regression tests.

## Evidence

- **Broken lookup in this PR:** PR #42051 adds `resolveSudoUserHome()` using `os.userInfo({ username: sudoUser })`; review comments correctly note that `os.userInfo()` returns the current effective user, so under sudo it still resolves to root rather than `SUDO_USER`. A local Node check returned the current username even when a bogus `username` option was passed.
  - file: [src/daemon/paths.ts:14](https://github.com/openclaw/openclaw/blob/a57331eaac2b44a4b6a9142c7c70f6b90e6fb39b/src/daemon/paths.ts#L14)
  - command: `node -e "const os=require('node:os'); console.log(process.version); const current=os.userInfo().username; const bogus=os.userInfo({username:'definitely-not-'+current}).username; console.log(JSON.stringify({current,bogus,same:current===bogus}));"`
  - sha: [a57331eaac2b](https://github.com/openclaw/openclaw/commit/a57331eaac2b44a4b6a9142c7c70f6b90e6fb39b)
- **Current main still derives unit path from HOME:** Current main resolves daemon home from `HOME`/`USERPROFILE` only, and the systemd unit path is built from that resolved home. This confirms #42051 was not independently implemented on main.
  - file: [src/daemon/paths.ts:8](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/daemon/paths.ts#L8)
  - command: `nl -ba src/daemon/paths.ts | sed -n '1,120p'`
  - sha: [a57d681db9b2](https://github.com/openclaw/openclaw/commit/a57d681db9b237348c5d446c8d371b2c1057f016)
- **Systemctl scope and write path can still diverge under sudo:** Current main targets `SUDO_USER` via `systemctl --machine <user>@ --user`, but `writeSystemdUnit()` writes to `resolveSystemdUnitPath(env)`, which still uses `HOME`. That is the mismatch this PR tried to address.
  - file: [src/daemon/systemd.ts:367](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/daemon/systemd.ts#L367)
