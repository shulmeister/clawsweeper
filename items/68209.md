## Summary

Keep #68209 open. Current main has real Codex hardening around routing, auth-profile forwarding, and inflated usage reporting, but the specific combined regression in this issue is not proven fixed: switching from the OpenAI Codex OAuth/PI route to the native Codex app-server route allegedly caused runaway context growth, off-task workspace contamination, and auth-mode drift, and the related merged seam PR explicitly treated #68209 as adjacent rather than closed.

## Best Possible Solution

Keep this issue open as a focused child bug under the broader Codex harness stability work. Maintainers should validate the exact route switch on a current release with a clean session, capture `/status` and Codex app-server auth/runtime events before and after the switch, and then either close it with live proof or narrow it into specific fixes for native Codex context accounting, stale thread binding, or auth-profile/CLI login isolation.

## Evidence

- **Issue discussion shows concrete recent signal from two users:** The reporter described a stable `openai-codex/gpt-5.4` baseline, then severe instability after switching to `codex/gpt-5.4`, including a lost screenshot showing `466.9k / 200k`, context overflow/failed compaction logs, and OpenAI API-key mismatch warnings. A second commenter independently reported Codex login changing from ChatGPT OAuth to API-key auth after enabling Codex in OpenClaw.

That helps configuration clarity, but it is not evidence that the reported native-route instability is resolved.
