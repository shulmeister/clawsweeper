## Summary

Keep open. Current main has advanced since the earlier review by pinning the bundled ACPX plugin to acpx 0.6.1, and upstream ACPX now pools persistent clients across turns, which likely reduces the original respawn pressure. That is not enough to conservatively close this report: the OpenClaw plugin-tools bridge is still a stdio MCP server injected into ACPX session bootstrap, the server has no reconnect/rebind lifecycle handling, and the tests I found cover MCP config injection plus the separate `openclaw acp` bridge-safe path rather than Claude Code ACP `sessions_send` or session-rebind disconnects.

## Best Possible Solution

Keep the issue open and add a focused ACPX/Claude Code regression path with `pluginToolsMcpBridge: true` that exercises both `sessions_send` follow-up runs and `/acp spawn claude --bind here` style rebinds, then proves `bridge_*` tools remain in the Claude Code tool manifest. The fix likely belongs in ACPX/OpenClaw bridge lifecycle: either preserve the nested MCP stdio transport across session loads/rebinds or add explicit clean teardown/reconnect semantics for the old `plugin-tools-serve.js` process tree.
