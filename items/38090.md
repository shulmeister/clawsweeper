## Summary

Keep open. Current main still does not prove the reported Control UI/WebChat reconnect problem is fixed: the browser client auto-reconnects after recoverable WebSocket closes, the gateway removes session-event subscriptions on close, and the UI still clears active `chatRunId`/stream state after reconnect because in-flight terminal events may have been missed. Recent adjacent fixes improve Stop-button aborts across reconnects and channel stale-socket health checks, but they do not implement preservation/recovery of an active Control UI chat run across a WebSocket reconnect.

## Best Possible Solution

Implement reconnect-safe Control UI chat recovery: preserve or re-associate active run state across WebSocket reconnects using stable session/run identity, resubscribe and replay or reload missed events/history without orphaning the stream, and add a regression test for mid-run disconnect/reconnect. If maintainers prefer one tracker, consolidate #38090 with the richer #38091 manually rather than auto-closing this item now.
