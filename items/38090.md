## Summary

Keep open. Current main still does not implement the requested behavior of preserving an in-flight Control UI/WebChat run across a WebSocket reconnect. There are related mitigations for queued Stop-button aborts and channel stale-socket health checks, but the UI reconnect path still clears active chat run/stream state, and the gateway drops the old session event subscription on socket close.

## Best Possible Solution

Keep the issue open until Control UI/Gateway can reconcile active chat runs after reconnect. A solid fix would resubscribe to the active session, recover or replay missed terminal state from server-side run/session state or history, avoid clearing chatRunId/chatStream until terminal state is known, and add UI plus gateway regression tests for reconnect during an active WebChat run. If maintainers decide active stream resumption is out of scope, document the intended reconnect behavior and close only with that product decision.
