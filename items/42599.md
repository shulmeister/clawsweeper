## Summary

Current `main` no longer creates or manages a separate `openclaw` service user in the Podman setup flow. The reported `useradd`/`adduser` duplication is gone because `setup-podman.sh` now delegates to a rootless current-user setup script, and the shipped docs/changelog explicitly describe that newer model.

## Close Comment

Closing after a Codex review. Current `main` already removed the dedicated Podman service-user path that this suggestion was trying to refactor: `setup-podman.sh` is now just a wrapper, and the active flow in `scripts/podman/setup.sh` runs as the current non-root user (`OPENCLAW_USER="$(id -un)"`), refuses root, writes state under that user's `~/.openclaw`, and installs an optional `systemctl --user` Quadlet. The docs now describe `openclaw --container <name> ...` as the management path instead of a separate `openclaw` service user (`docs/install/podman.md:8`, `docs/install/podman.md:15`), and this shipped in `v2026.4.22` (`CHANGELOG.md:1908`, release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Because the old `useradd`/`adduser` branch no longer exists on the maintained Podman path, there is no remaining cleanup item here to track separately.
