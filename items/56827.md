## Summary

Keep open. Current `main` has preventive session-store maintenance (entry cap, `sessions.json` rotation, optional disk-budget cleanup, orphan transcript archival), but the doctor path still loads/parses the full `sessions.json` before any cleanup and the cleanup code does not target `sessions.json.*.tmp` artifacts. I do not see strong evidence that issue #56827 is implemented on `main` or safely auto-closeable.

## Evidence

- **Doctor still loads the entire session store before maintenance:** `loadSessionStore()` reads `sessions.json` with `fs.readFileSync`, parses it with `JSON.parse`, then only applies maintenance afterward. A pathological large store can still fail before prune/cap logic helps.
  - file: [src/config/sessions/store-load.ts:76](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/store-load.ts#L76)
  - command: `nl -ba src/config/sessions/store-load.ts | sed -n '65,150p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Doctor state-integrity cleanup only handles missing transcripts and orphan `.jsonl` files:** The doctor flow warns about missing transcripts and orphan transcript files, and archives only orphan `.jsonl` files by renaming them to `.deleted.<timestamp>`. It does not inspect or remove `sessions.json.<uuid>.tmp` files.
  - file: [src/commands/doctor-state-integrity.ts:841](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/commands/doctor-state-integrity.ts#L841)

- **Disk-budget cleanup excludes stale `sessions.json.*.tmp` artifacts:** `enforceSessionDiskBudget()` counts all files in the sessions dir toward total bytes, but its removable queue only includes archive artifacts and orphan primary `.jsonl` transcripts. Stale `sessions.json.*.tmp` files would still consume budget without being selected for cleanup.
  - file: [src/config/sessions/disk-budget.ts:208](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/disk-budget.ts#L208)

- **Current main has preventive write-side maintenance, but that is not the reported doctor repair:** Write-side session maintenance now rotates oversized `sessions.json` files and can enforce a sessions-directory disk budget; docs also describe `maxEntries`, `rotateBytes`, `maxDiskBytes`, and `highWaterBytes`. This helps future growth, but it is not evidence that doctor now survives and cleans up existing `sessions.json.*.tmp` buildup.

- **Issue discussion only provided a manual workaround:** The lone comment suggests manually deleting `sessions.json.*.tmp` files with `find ... -delete`; no linked PR, shipped release note, or maintainer comment in the issue thread shows that OpenClaw core already fixed this.

## Risks / Open Questions

- I did not reproduce the exact V8 deserialization crash against a user-sized 1.2 GB session directory, so the precise failure mechanism remains unverified.
- Current write-side maintenance may reduce future growth on updated installs, but existing pathological stores can still hit the unbounded read/parse path during doctor.
- Because stale tmp files are counted toward disk usage but not chosen for cleanup, enabling disk-budget cleanup could evict real session entries/transcripts before removing the temporary artifacts the issue calls out.
