## Summary

Keep open. Current main still keeps a module-level cron `storeLocks` Map, writes one entry per `storePath`, and has no cleanup path after the queued operation resolves. I found no current `locked` regression test, changelog entry, or source-side replacement that would make PR #43390 obsolete; the PR discussion also points to the same leak and supports the identity-check cleanup approach.

## Best Possible Solution

Keep PR #43390 open for maintainer review. The best path is to merge or adapt the `finally` identity-check cleanup in `src/cron/service/locked.ts` and add focused cron lock cleanup tests, preserving the existing serialization guarantees for queued same-store operations.
