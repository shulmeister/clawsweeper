## Summary

Keep open. The report is recent, concrete, and still maps to unresolved current-main behavior: native Windows Gateway is in documented scope, model prewarm still runs before channel startup without the requested skip env var, gateway network bootstrap is only an env-proxy dispatcher bootstrap, and the existing Telegram/Discord mitigations are narrower than the reported process-wide HTTP failure. Related #56733 is useful context for gateway-wide timeout/freeze symptoms, but it remains open and does not supersede this Windows-specific reproduction.

## Best Possible Solution

Keep this open for a targeted maintainer fix. The best path is to reproduce on native Windows 10/Node 22 with multiple Discord and Telegram accounts, instrument gateway-process fetch/undici dispatcher state and libuv handles, then add a bounded model-prewarm workaround such as OPENCLAW_SKIP_MODEL_PREWARM or a prewarm timeout while separately fixing the process-wide HTTP failure in the gateway network/runtime path. Regression coverage should prove channel startup is not blocked indefinitely and that Telegram, Discord REST, and model-catalog HTTP calls remain healthy in the same gateway process.
