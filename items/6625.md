## Summary

Keep #6625 open. Current main still has hard sub-agent run/wait timeouts plus after-the-fact timeout handling and partial-output announcement, but no configurable pre-timeout grace period or injected checkpoint warning. The existing manual steer/control path can restart a child with a message, yet nothing schedules that path before runTimeoutSeconds expires. The 2026-04-26 maintainer/Codex comment in the issue already separated this request from partial-progress persistence, and current main still matches that assessment.

## Best Possible Solution

Keep #6625 open for core sub-agent lifecycle work. A good implementation would add a documented agents.defaults.subagents grace/warning setting, schedule a warning before runTimeoutSeconds, deliver it through the existing child-session message/control path for native and ACP sub-agents where feasible, and add focused tests/docs while preserving the current partial-progress timeout fallback.

## Risks / Open Questions

- Existing partial-progress extraction and timeout announcements reduce data loss, but they do not give the child agent a chance to checkpoint before termination.
- Manual subagent steer/control shows a possible delivery mechanism, but using it safely as an automatic pre-timeout warning needs core lifecycle design to avoid racing timeout, cleanup, announce, and ACP/native runtime differences.
