## Summary

Keep #6625 open. Current main still has hard `runTimeoutSeconds` handling and post-timeout partial-progress reporting, but no configurable pre-timeout grace window or injected checkpoint warning for native or ACP sub-agents. The latest provided issue discussion also explicitly kept this item open after separating it from partial-result persistence.

## Best Possible Solution

Keep #6625 open for core sub-agent lifecycle work. A good fix would add a documented `agents.defaults.subagents` grace/warning setting, schedule a warning before `runTimeoutSeconds`, deliver it through the existing child-session control/message path where feasible for native and ACP sub-agents, and add focused tests/docs while preserving the current partial-progress timeout fallback.

## Risks / Open Questions

- Existing partial-progress extraction reduces loss after timeout, but it does not satisfy the requested chance for the child to checkpoint before termination.
- The manual sub-agent steer/restart path is a plausible delivery mechanism, but automatic pre-timeout steering would need careful lifecycle design across native and ACP runs to avoid racing timeout, cleanup, announce, and queue handling.
