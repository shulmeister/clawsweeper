## Summary

Keep open. Current main still implements sub-agent timeouts as hard run/wait timeouts with after-the-fact timeout handling and partial-progress extraction; it does not provide a configurable pre-timeout warning or injected checkpoint message for the child agent.

## Best Possible Solution

Keep #6625 open for core sub-agent lifecycle design. A good fix would add a documented `agents.defaults.subagents` grace-period setting, schedule a warning before `runTimeoutSeconds`, deliver it through the existing child-session messaging path for native and ACP sub-agents where feasible, and cover the timeout/warning behavior with focused tests and docs while preserving the existing partial-progress fallback.
