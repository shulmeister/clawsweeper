## Summary

Current main already implements the duplicate-send guard for the reported Telegram/provider-timeout replay path. The active changelog explicitly names this issue as duplicate validation for the replay-unsafe provider/runtime failure fix, and the source now prevents fallback/retry after visible block output while committing inbound dedupe keys after replay-unsafe failures.

## Best Possible Solution

Keep the current main-branch implementation and close this issue as implemented. Any remaining duplicate Telegram delivery should be tracked only with fresh logs from a build containing 82b928232e072d06ff2e867f57ebbdaed909c709, because the reported provider-timeout replay path now has explicit replay-unsafe dedupe and fallback guards.
