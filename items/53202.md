## Summary

Current `main` already avoids the reported cron/subagent self-deadlock by steering completion announces into the active requester session instead of waiting on a gateway `agent` call timeout; the same behavior is present in release `v2026.4.22`.

## Evidence

## Risks / Open Questions

- If an active requester session cannot be woken (`queueEmbeddedPiMessage` returns false), the code still reports a direct-path failure and may rely on later fallback handling; that is a different failure mode from the reported repeated 90s self-timeout loop.

## Close Comment

Closing from a Codex review: current `main` already implements the reported fix. In `src/agents/subagent-announce-delivery.ts:492`, completion announces first detect whether the requester session is already active; if it is, OpenClaw injects the completion with `queueEmbeddedPiMessage(...)` and returns `path: "steered"` instead of issuing another gateway `agent` call that can sit for the full timeout. The direct gateway path only runs when the requester is not active. This is covered by `src/agents/subagent-announce-delivery.test.ts:119`, which verifies that active completion requesters are handled without any gateway call, and the same logic is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
