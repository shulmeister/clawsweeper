## Summary

Keep open. Current main has inbound/channel audio transcription and built-in voice surfaces, but it still lacks a documented canonical custom-client voice ingress contract. `/v1/responses` supports text, image, and file parts but no `input_audio`; default `input_file` handling excludes audio MIME types; and `chat.send` attachments remain loosely typed while the parser/UI path is image-focused and drops non-image payloads. The related closed issues cover channel transcription or native Control UI voice, not this custom mobile/WebChat audio-ingress contract.

## Best Possible Solution

Keep #70007 open as a focused Gateway/API design item. The best path is to define one additive, documented server-side audio ingress contract, likely either `input_audio` on `/v1/responses` or a typed `chat.send.attachments`/media-upload reference flow, then route accepted audio through the existing `tools.media.audio` media-understanding pipeline with protocol docs and regression tests.

## Risks / Open Questions

- A supported server-side audio ingress path needs an additive protocol/API contract, auth and size-limit behavior, transcript persistence semantics, and docs/client follow-through.
- Leaving this open is preferable to closing as ClawHub because the missing piece is the core Gateway/OpenResponses/chat.send contract that custom clients would rely on.
