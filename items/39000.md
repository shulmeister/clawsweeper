## Summary

Keep #39000 open. Current main still supports plugin gateway method registration, but does not expose an injected `api.callGatewayMethod()` or equivalent in-process plugin runtime seam for invoking another plugin's registered gateway method. The linked metabolism/stability issue is a concrete cross-plugin use case, and this is a plugin SDK/runtime capability gap rather than ClawHub cleanup.

## Best Possible Solution

Keep the issue open for maintainer product/API design. The likely path is an additive plugin SDK/runtime seam that lets a plugin invoke a registered plugin-owned method with explicit scope/context propagation and documented lifecycle/error behavior. If maintainers decide gateway RPC should not be used for inter-plugin calls, document the supported alternative, such as a typed service registry or request/response event seam, and update plugin SDK docs accordingly.

## Risks / Open Questions

- A future implementation needs maintainer design for authorization/scope propagation, startup availability, recursion/reentrancy, error typing, and whether gateway RPC is the right abstraction versus a typed service registry or event/request bus.
- The existing `GatewayClient` export is a possible low-level workaround for code that can manage a websocket connection and credentials, but treating it as the answer would preserve the coupling and auth burden this issue is asking core to solve.
