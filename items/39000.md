## Summary

Keep #39000 open. Current main still has gateway method registration and external gateway dispatch, but no plugin-facing `api.callGatewayMethod()` or equivalent generic inter-plugin request/response seam.

## Best Possible Solution

Keep #39000 open for maintainer product/API design at the plugin SDK/runtime boundary. The best fix is a documented inter-plugin request/response seam, with contract tests covering plugin isolation, caller identity and scopes, plugin-registered handler dispatch, error propagation, lifecycle/deferred-load behavior, and plugin author docs.

## Risks / Open Questions

- The public `plugin-sdk/gateway-runtime` GatewayClient helper is not an equivalent in-process inter-plugin seam; using it from a loaded plugin would be a gateway-client workaround with separate auth, connection, and request-context semantics.
- Maintainer API design is still needed for caller identity, scopes, missing-method behavior, error propagation, recursion, deferred startup/load timing, and whether the final shape should be `api.callGatewayMethod()` or a typed service/RPC abstraction.
