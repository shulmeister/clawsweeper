## Summary

Keep #39000 open. Current main still has plugin-side registration for Gateway RPC methods and external Gateway dispatch, but no plugin-facing `api.callGatewayMethod()` or equivalent in-process request/response seam. The linked metabolism/stability use case is a real Plugin SDK/runtime API gap, not a stale or ClawHub-only request.

## Best Possible Solution

Keep #39000 open for Plugin SDK/runtime design. The best path is a documented inter-plugin request/response seam, whether named `api.callGatewayMethod()` or expressed as typed services/capabilities, with contract tests for plugin isolation, caller identity/scopes, handler dispatch, errors, recursion/lifecycle behavior, deferred-load timing, and plugin-author docs.

## Risks / Open Questions

- Adding this API still needs maintainer design for caller identity, scopes/authorization, missing-method errors, recursion/reentrancy, lifecycle/deferred-load timing, and whether Gateway RPC or typed services/capabilities should be the stable abstraction.
- Treating the existing external `GatewayClient`/`callGateway` helpers as the answer would likely miss SDK semantics for plugin isolation and in-process dispatch.
