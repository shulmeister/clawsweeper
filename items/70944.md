## Summary

Keep #70944 open. Current main still only uses `session.identityLinks` for session-key canonicalization; it does not resolve a Discord sender to a trusted canonical principal, does not thread that value through Discord preflight/process/native command contexts, and does not render `sender_principal` in inbound meta. The PR is recent and its review context shows the author addressed the bot review concerns, including the PluralKit sender lookup issue, so the remaining work is still useful rather than obsolete.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a focused implementation that resolves Discord sender IDs through `session.identityLinks`, threads a sanitized canonical principal through Discord preflight/process/native command contexts, and renders it as trusted inbound metadata. If the shared identity-link resolver becomes plugin-facing, align the SDK export with the plugin SDK boundary and add the corresponding contract/docs checks.
