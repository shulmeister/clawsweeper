## Summary

Keep open. Current main still does not implement this PR's Telegram partial-mode coalescing behavior: answer preview boundaries still rotate through `forceNewMessage()`, the PR's `revive()` / `shouldSplitPreviewMessages` mechanism is absent, and current tests explicitly assert separate preview messages in `streamMode: "partial"`. The maintainer discussion also asked to hold the PR for rebase/alignment and unresolved boundary regressions.

## Best Possible Solution

Keep #39213 open as the canonical maintainer-triaged thread for Telegram partial-mode coalescing. A viable path is to rebase or redesign the fix on current main, explicitly account for #38906 and the current lane lifecycle design, address the retained/fallback/media-only boundary regressions from review, and align docs plus regression tests with the chosen partial-mode semantics.
