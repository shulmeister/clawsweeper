## Summary

Keep #39213 open. Current main still forces/rotates Telegram answer previews across assistant-message boundaries in partial mode, lacks the PR's `revive()`/`shouldSplitPreviewMessages` mechanism, and has tests that explicitly expect `forceNewMessage()` in `streamMode: "partial"`. The PR also has an explicit maintainer hold comment requesting rebase/alignment and unresolved boundary-regression work.

## Best Possible Solution

Keep #39213 open as the canonical Telegram partial-preview coalescing work. Rebase or redesign it against current `extensions/telegram` main, explicitly account for #38906 and the current lane lifecycle, resolve the early-partial and media-only boundary review comments, then update regression tests so Telegram partial mode matches the documented single-preview behavior.
