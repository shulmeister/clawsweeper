## Summary

Current `main` does not appear to implement this request for normal Discord thread sessions. Parent-channel `modelByChannel` fallback depends on `ParentSessionKey`, but Discord only sets `ParentSessionKey` for threads when `channels.discord.thread.inheritParent` is enabled, which is unrelated to model override inheritance and defaults to `false`.

## Evidence

- **Model override resolution only falls back through parent session context:** `getReply` passes `parentSessionKey` into `resolveChannelModelOverride`, and there is no `ThreadParentId` fallback in that call path.
  - file: [src/auto-reply/reply/get-reply.ts:315](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/get-reply.ts#L315)

  - file: [src/channels/model-overrides.test.ts:51](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/channels/model-overrides.test.ts#L51)

## Risks / Open Questions

- Docs currently say Discord threads inherit parent channel config unless overridden (`docs/channels/discord.md`), but the runtime evidence above suggests `modelByChannel` inheritance is still conditional on `thread.inheritParent`, so closing as implemented would risk masking a real bug.
- I did not run runtime tests because this review is read-only and must not create artifacts; the conclusion is based on current source and existing tests.
