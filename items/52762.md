## Summary

Keep open. Current `main` still runs the expensive prune path on every new dedupe insertion, and Discord inbound replay still uses that cache with a `5 * 60_000` ms / `5000` entry window. This PR is not implemented on `main`, and the review thread raises unresolved concerns about saturated-cache behavior and `NaN` limit handling, so there is not strong evidence to close it conservatively.

## Risks / Open Questions

- If this PR is closed now, `main` still retains the same `DedupeCache` hot path that scans the cache on each new insert.
- The proposed patch appears incomplete for high-throughput Discord installs because once the dedupe window stays above `5000`, it still re-enters the full prune path repeatedly.
- The `map-size.ts` change needs explicit validation for invalid runtime inputs like `NaN`; current in-repo tests do not cover that case.
