## Summary

Keep this PR open. Current main still does not hydrate media/path/file URL hints into buffer/contentType/filename for plugin-handled send actions, and the latest PR discussion points toward a real guarded opt-in design rather than obsolete work. Related #41779/#42674 and #57335 overlap on send attachment behavior, but they do not fully supersede this plugin-channel media hydration path.

## Best Possible Solution

Keep the PR open for maintainer review or contributor rework. The best path is a guarded public plugin action contract, such as preloadSendMedia, wired from runMessageAction into hydrateAttachmentParamsForAction, with opt-in only for send handlers that directly consume buffer/contentType/filename. Add regression coverage for both opt-in hydration and non-opt-in URL/path forwarding, and coordinate with #41779/#42674 and #57335 without closing this PR as implemented or duplicate.

## Risks / Open Questions

- A blanket version of this change would be risky because existing Google Chat and Matrix send handlers already fetch media from URL/path hints; core preloading can double-fetch or fail earlier for short-lived URLs and gateway-backed sends.
- The final PR patch shown in the provided context adds a preloadSendMedia opt-in but does not show any bundled channel opting in, so maintainers should confirm a real handler consumes hydrated buffer/contentType/filename before merging.
