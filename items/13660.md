## Summary

Current `main` already supports manual cron runs that bypass schedule checks. The implementation uses force-run as the default behavior for `openclaw cron run <job-id>`, with `--due` preserving the older schedule-respecting path, and this behavior is documented and shipped in `v2026.4.22`.

## Evidence

- **CLI defaults manual runs to force mode:** `openclaw cron run <id>` maps to `mode: "force"`; `--due` switches back to due-only behavior.
  - file: [src/cli/cron-cli/register.cron-simple.ts:162](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cli/cron-cli/register.cron-simple.ts#L162)

  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Gateway API defaults cron.run to force:** `cron.run` accepts `mode?: "due" | "force"` and calls `enqueueRun(jobId, p.mode ?? "force")`.
  - file: [src/gateway/server-methods/cron.ts:369](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/server-methods/cron.ts#L369)

## Risks / Open Questions

- The implementation chose a stronger UX change than the issue text proposed: force-run is the default and `--due` is the opt-in compatibility flag, so there is no literal `--force` flag.
- The exact `triggeredBy: "manual-force"` run-log schema from the proposal was not verified in current main; closure is based on the core requested capability being implemented.
