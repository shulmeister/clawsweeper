## Summary

Keep open. Current `main` already renders inline audio players for assistant audio attachments, but webchat uploads are still image-only end to end, so the requested audio upload support is not implemented and the issue remains materially valid.

## Evidence

- **Inline playback already exists for assistant audio attachments:** `renderMessageGroup` renders `<audio controls preload="metadata">` when a normalized attachment has `kind === "audio"`, so part of the request is present on current `main`.
  - file: [ui/src/ui/chat/grouped-render.ts:1177](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/chat/grouped-render.ts#L1177)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Playback is covered by a UI test:** The chat render test asserts that an assistant message containing `MEDIA:https://example.com/voice.ogg` produces an `audio` element and a `Voice note` badge.
  - file: [ui/src/ui/chat/grouped-render.test.ts:610](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/chat/grouped-render.test.ts#L610)

  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/chat/attachment-support.ts#L1)

  - file: [ui/src/ui/views/chat.ts:244](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/ui/src/ui/views/chat.ts#L244)
