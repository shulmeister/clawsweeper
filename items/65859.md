## Summary

Keep this issue open. Current main partially implements the request: WebChat now advertises `audio/*` in the picker, serializes non-image uploads as files, persists non-video attachments, and renders assistant audio attachments with `<audio controls>`. The remaining gap is the ASR/transcription workflow: uploaded audio is offloaded and appended to the message as a `media://inbound/...` text marker, but the live dispatch context does not carry the offloaded audio as `MediaPath`/`MediaTypes`, so the media-understanding audio path is not clearly invoked for the current turn. There is also still inconsistent `audio/webm` support.

## Best Possible Solution

Keep the issue open and narrow the remaining work to end-to-end WebChat audio upload semantics: populate `MediaPath`/`MediaPaths` and `MediaType`/`MediaTypes` for offloaded audio before live dispatch so `tools.media.audio` can transcribe the current turn, add focused WebChat/Gateway/media-understanding regression coverage for MP3/M4A/OGG/WAV upload, and either include `audio/webm` support here or explicitly land the shared WebM MIME mapping first. Preserve the existing assistant `<audio>` rendering as already implemented.
