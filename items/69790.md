## Summary

Keep #69790 open. Current main has useful mitigations: Telegram DM voice notes are now transcribed before dispatch, normal Telegram bot updates go through grammY sequentialization, and embedded attempts release their session lock in cleanup. But I did not find a source, test, changelog entry, or related canonical issue showing that the reported same-session synthetic inbound path now reuses or waits on a live gateway-held session write lock. In fact, the current lock contract remains non-reentrant by default, including same-process acquisitions, unless callers explicitly pass `allowReentrant: true`; the embedded run acquisition path does not pass that option.

## Best Possible Solution

Keep the issue open and handle it as a product/runtime contract question. The best fix is likely a supported Gateway-owned synthetic inbound or session-send API for Telegram that routes through the existing per-session queue and active-run/followup policy, rather than importing built `dist/extensions/telegram/*` and calling `bot.handleUpdate` in a separate process. If maintainers decide external synthetic updates are unsupported, document that boundary and point users at the native Telegram audio preflight transcription path or a supported message/session injection API.
