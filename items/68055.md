## Summary

Current `main` still blocks cron webhooks to private-network targets by default, has no `cron.webhookAllowPrivateNetwork` config surface, and documents/tests only the locked-down behavior. The PR is coherent, scoped to existing SSRF policy machinery, and not already implemented on `main`, so there is not strong evidence for any conservative auto-close reason.

## Evidence

- **Cron webhook fetch on main has no private-network opt-in:** `postCronWebhook` calls `fetchWithSsrFGuard` without a `policy` parameter, and the current failure-alert / finished-run / failure-destination call sites only pass webhook URL, token, payload, and log strings. There is no main-branch wiring for a cron-specific SSRF relaxation knob.
