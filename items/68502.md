## Summary

Keep PR #68502 open. Current main has first-class Microsoft Foundry provider code, but it still lacks an equivalent user-facing Foundry provider guide or provider-directory entry, and the PR is a concrete docs contribution rather than a cleanup-close candidate. The PR head also addressed the bot review's two findings by adding the provider index entry and documenting both endpoint URL shapes. It likely needs revision before merge because current main now exposes the bundled provider as `microsoft-foundry` with `openai-completions`/`openai-responses` behavior, while the PR text is framed around manual `anthropic-messages` and `azure-openai-responses` adapter paths.

## Best Possible Solution

Keep the PR open and revise it into a current Microsoft/Azure Foundry provider guide: add or update a provider docs page, link it from `docs/providers/index.md`, document `openclaw onboard` for Entra ID and API-key auth, explain endpoint/deployment naming, and clearly separate shipped `microsoft-foundry` behavior from any advanced manual adapter configuration.
