## Summary

Keep #38248 open. Current main still has cost metadata, usage/cost reporting, failover error classification, and external spend-limit workarounds, but it does not implement the requested native request-level rolling spend ceiling for model fallback chains.

## Best Possible Solution

Keep the issue open until OpenClaw either adds a native cost guard in the model request/failover path or maintainers intentionally design a blocking plugin/API seam for rolling spend enforcement. A strong implementation would add documented config, shared rolling hourly/daily accounting, pre-attempt estimated-cost checks, post-response reconciliation from provider usage/cost, pause/reject behavior, owner notification, and regression coverage around fallback chains.

## Risks / Open Questions

- Closing as implemented would hide a still-missing financial safety gap: expensive successful fallback calls can proceed repeatedly without a native rolling spend ceiling.
- A ClawHub-only close is not high-confidence because the requested protection needs a blocking request-path enforcement point or an explicit plugin/API seam with cross-provider spend context.
- External mitigations such as vibe-billing, Kalibr routing, OpenRouter/LiteLLM budgets, or provider-side limits are useful but remain opt-in workarounds outside OpenClaw's native failover path.
