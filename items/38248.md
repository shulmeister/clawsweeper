## Summary

Keep #38248 open. Current main still has cost estimation, usage summaries, spend-limit error classification, cooldown/failover behavior, and external LiteLLM/vibe-billing-style mitigations, but it does not implement a native request-level rolling spend ceiling such as `agents.defaults.costGuard.maxSpendPerHour` or `maxSpendPerDay`.

## Best Possible Solution

Keep the issue open until OpenClaw either adds a native cost guard in the model request/failover path or maintainers intentionally design a blocking plugin/API seam for rolling spend enforcement. A strong implementation would add a strict documented config surface, shared rolling hourly/daily cost accounting, pre-attempt estimated-cost checks, post-response reconciliation from provider usage/cost, pause/reject behavior, owner notification, and regression coverage around fallback chains.

## Risks / Open Questions

- Closing as implemented would hide a still-missing financial safety gap: expensive fallback calls can still succeed repeatedly without a native rolling spend ceiling.
- A ClawHub-only close is not high-confidence because current plugin hooks do not provide a documented blocking pre-provider-call budget API with cross-provider spend context.
- External mitigations such as vibe-billing, Kalibr routing, OpenRouter limits, or LiteLLM virtual-key budgets are useful but remain opt-in workarounds outside OpenClaw’s native failover path.
