## Summary

Current `main` already implements the Telegram polling restart fix described in #53338: polling now reuses a shared Telegram transport so sticky IPv4 fallback state survives clean polling restarts, and dirty/error restarts rebuild transport explicitly. The shipped code is also covered by regression tests for stalled polling restarts and unhandled polling network rejections.

## Close Comment

Closing as implemented on `main` based on this Codex review. The current Telegram monitor now preserves sticky IPv4 fallback state across polling restarts by creating a shared transport before starting polling and reusing it on clean/conflict restarts (`extensions/telegram/src/monitor.ts:209`). The sticky fallback state itself is held inside the transport closure (`extensions/telegram/src/fetch.ts:619`), and restart handling only rebuilds transport for dirty/error cycles (`extensions/telegram/src/polling-transport-state.ts:22`). There is also regression coverage for stalled restart and polling-network-error restart paths in `extensions/telegram/src/monitor.test.ts:564` and `extensions/telegram/src/polling-session.test.ts:505`. This is already present in the released `v2026.4.22` tree (commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so there is no remaining maintainer action on this issue.
