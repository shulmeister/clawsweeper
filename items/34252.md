## Summary

Keep #34252 open. Current main has partial Ollama-provider work, including provider-specific baseUrl lookup when the Ollama provider hook is invoked directly, but it still does not provide documented, end-to-end support for multiple independently named Ollama provider ids such as `ollama-jetson` or `ollama2`. The issue discussion and related ClawSweeper reports show this remains the canonical tracking item for multi-host/multi-instance Ollama support.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw either implements provider-family ownership for configured `api: "ollama"` provider ids or documents a supported multi-Ollama configuration. The implementation should cover runtime hook activation, synthetic/local auth, provider-specific baseUrl selection, model selection, and targeted tests for a secondary id such as `ollama-jetson`, including the memory/heartbeat routing surfaces mentioned in the issue.

This is useful partial implementation, but it only helps once the Ollama provider hook has already been selected for that provider id.
