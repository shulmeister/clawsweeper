## Summary

Keep #34252 open. Current main still does not provide end-to-end support for multiple native Ollama provider ids with separate base URLs. The Ollama plugin has partial direct-hook plumbing for a custom provider id, but normal provider ownership, runtime hook resolution, generic transport fallback, synthetic auth, and memory embedding paths remain keyed to the canonical `ollama` provider.

## Best Possible Solution

Keep #34252 open as the canonical tracker. The best implementation is an explicit provider-family or activation contract for configured providers whose native API is Ollama, so custom ids can be owned by the Ollama plugin without hard-coding every id. That should cover runtime stream hooks, synthetic/local auth, provider-qualified model selection, heartbeat and memory-search routing, tests for custom ids such as `ollama2`/`ollama-jetson`, and user-facing docs for the supported multi-host config shape.

## Risks / Open Questions

- A direct Ollama `createStreamFn` unit test can make the feature look implemented, but it bypasses the provider resolution path used by normal model calls.
- Generic local custom-provider auth may avoid an auth error for some local configs, but it does not register a native Ollama stream for `api: "ollama"` when the provider id is custom.
