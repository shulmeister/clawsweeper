## Summary

Keep #34252 open. Current main still does not provide end-to-end native multi-Ollama provider support for custom provider ids such as `ollama-gpu`, `ollama-cpu`, or `ollama-jetson`; it has only partial low-level baseUrl handling if the Ollama hook is invoked directly. Runtime provider activation and hook lookup remain centered on the canonical `ollama` provider id, and the generic transport fallback still treats `api: "ollama"` as unsupported.

## Best Possible Solution

Keep #34252 open as the canonical product/implementation tracker. The best fix is to add an explicit provider-family or activation contract for configured providers whose native API is Ollama, make the Ollama runtime hook/auth/synthetic-local behavior work for custom ids such as `ollama-cpu` and `ollama-gpu`, cover provider-qualified `heartbeat.model` and model selection, and document/test the supported config shape including how memory search should target a separate Ollama host.
