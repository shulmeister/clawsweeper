## Summary

Keep this issue open. Current main still routes `openclaw models set` through the generic config replacement path, while the config writer still rejects more-than-half size drops as `CONFIG_WRITE_REJECTED` unless a caller explicitly opts into destructive writes. The existing models tests still mock the write layer, so they do not cover this failure path.

## Best Possible Solution

Keep this open. Add a regression test that exercises a real `openclaw models set` write against a valid shrink-prone config, then fix the models command/config mutation path or add a narrowly scoped config-writer policy that accepts legitimate `agents.defaults.model.primary` updates while preserving the size-drop guard for true clobbers.
