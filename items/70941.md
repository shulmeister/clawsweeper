## Summary

Keep #70941 open. The issue is recent and low-detail, but the title points at a concrete path: `openclaw models set` still writes through the shared config mutation path, and current `main` still has a protective `size-drop:` rejection in `writeConfigFile`. I found no issue discussion, comments, timeline, or current regression coverage proving this specific `models set` false-positive is already fixed or superseded.

## Best Possible Solution

Keep the issue open. A maintainer should reproduce with the reporter's large config shape, then make `openclaw models set` preserve the authored source shape or route the narrow model-primary mutation through a safe config mutation path that cannot look like a destructive size drop. Add a regression test covering `models set` against the triggering large config while keeping the existing destructive-write guard green.
