## Summary

Keep #7359 open. Current main still keeps Slack DMs/main and Slack channel conversations in separate session buckets. Outbound message-tool sends are mirrored into the destination Slack session, but default session-tool visibility remains tree-scoped, so a DM/main session still cannot see an unrelated Slack channel session by default. The related closed Slack issues address thread context/routing, not this same-agent cross-context self-activity recall request.

## Best Possible Solution

Keep #7359 open. Implement an explicit same-agent Slack self-activity recall path, likely a scoped outbound activity ledger or session-history bridge keyed by agent, Slack account, destination channel/thread, and sent message id. Expose that from DM context without granting broad cross-session history, and add regression coverage for sending from a DM to a Slack channel, then asking from the DM whether the agent sent it while unrelated channel history remains gated.

## Risks / Open Questions

- A broad fix could leak unrelated Slack channel or user history into DMs; the design needs a scoped same-agent self-activity path rather than blanket cross-session visibility.
- Existing workarounds such as tools.sessions.visibility=agent/all or manually querying a channel session are not equivalent to safe default recall of the agent's own outbound Slack activity.
