## Summary

Keep open. Current `main` still models Discord native commands with a single default `description` string, builds Carbon command objects without any localization payload, and redeploys those descriptors on startup. The reporter's proposed `description_localizations` direction matches a real gap in the current Discord native-command path.

## Evidence

- **Command schema lacks localization fields:** `ChatCommandDefinition` and `NativeCommandSpec` only expose a single `description: string`; there is no `descriptionLocalizations` or similar field in the current command model.
  - file: [src/auto-reply/commands-registry.types.ts:56](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/commands-registry.types.ts#L56)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Registry forwards only the default description:** `toNativeCommandSpec()` converts chat commands into native command specs by copying only `command.description`, so the native-command registry has no path for localized description metadata.
  - file: [src/auto-reply/commands-registry.ts:85](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/commands-registry.ts#L85)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Discord command creation sets only `description`:** `createDiscordNativeCommand()` builds the Carbon `Command` with `name`, truncated `description`, and `options`; no command- or option-localization fields are attached before deploy.
  - file: [extensions/discord/src/monitor/native-command.ts:634](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/native-command.ts#L634)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Startup still reconciles and redeploys native commands:** Discord provider startup still goes through the Carbon reconcile/deploy path and calls `handleDeployRequest()`, which means packaged descriptors are republished after restart/update.
  - file: [extensions/discord/src/monitor/provider.ts:378](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/provider.ts#L378)
