## Summary

Issue #53223 no longer describes one actionable OpenClaw core bug. The thread mixes several contradictory root-cause claims across comments: binding mismatch, peer-id normalization, health-monitor restarts, channel-manager auto-restarts, external plugin crash loops, and version-regression claims. Current main shows the cited restart behavior is expected when a channel plugin's `gateway.startAccount()` promise resolves, and configured binding routing for plugin-owned channels depends on plugin-provided binding adapters. The report needs a fresh, minimal, single-symptom repro before maintainer work can proceed.

## Close Comment

Closing this as a Codex review because the thread no longer points to one actionable OpenClaw core bug. The comments move between several incompatible explanations: binding config mismatch, peer-id normalization, health-monitor restarts, channel-manager auto-restarts, and an external plugin crash loop. Current `main` also shows that the reported restart log pattern is expected whenever a channel plugin's `gateway.startAccount()` promise resolves (`src/gateway/server-channels.ts:431-470`), and the SDK examples explicitly document that plugin accounts must stay pending with `waitUntilAbort(...)` or they will restart (`extensions/synology-chat/src/channel.ts:271-274`). On top of that, configured binding routing for plugin-owned channels only works when the plugin implements the binding adapter hooks (`src/channels/plugins/configured-binding-compiler.ts:55-63`). If this is still happening, please open a new issue focused on one symptom with a minimal repro on current `main`: exact external plugin version, reduced config, the inbound conversation/account ids seen by the plugin, and the specific logs showing where routing diverges.
