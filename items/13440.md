## Summary

Keep open. Current main still models `session.store` as a filesystem path and routes session load/save/update through the disk-backed `sessions.json` implementation, with no driver object or backend adapter registration seam. The related PostgreSQL/MongoDB discussion is useful evidence of demand for pluggable persistence, but those items target memory backends rather than the session-store backend API requested here, and the existing maintainer/Codex comment on #13440 already reached the same keep-open conclusion.

## Best Possible Solution

Keep #13440 open until maintainers either implement a core `SessionStoreAdapter` contract or intentionally add a plugin API that lets external SQLite, Redis, Convex, PostgreSQL, or similar backends own session persistence. A complete solution should update `session.store` schema/types/help/docs, preserve the current file backend as the default, route load/save/update/readUpdatedAt and discovery through the abstraction, and add contract tests for the file adapter plus a mocked or real non-file adapter path.
