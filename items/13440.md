## Summary

Keep #13440 open. Current main still treats `session.store` as a filesystem path, resolves it to `sessions.json`, loads/saves JSON via direct filesystem IO, and serializes writes through the existing lock-file path. Plugin-facing runtime surfaces expose those file helpers, but there is still no session-store driver/adapter registration API for Redis, SQLite, Convex, PostgreSQL, or custom backends.

## Best Possible Solution

Keep #13440 open until maintainers add either a core `SessionStoreAdapter` contract or a stable plugin API that can own session persistence end to end. The conservative path is to keep the file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/discovery/maintenance through the abstraction, and cover the file backend plus a mocked non-file backend with contract tests.
