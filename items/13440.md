## Summary

Current main still does not implement pluggable session-store backends. The session store remains a file-path string backed by `sessions.json`, synchronous filesystem JSON reads, locked writes, and `.lock` files. Existing plugin runtime surfaces expose session-store read/write helpers but not a backend adapter registration seam, and the related PostgreSQL/MongoDB discussions are memory-backend work rather than the session-store backend API requested here.

## Best Possible Solution

Keep this issue open until maintainers either implement a core session-store adapter contract or explicitly define a plugin API that lets external SQLite, Redis, Convex, or similar backends own session persistence. A complete solution should update `session.store` config schema/types/docs, route load/save/update/readUpdatedAt through the adapter abstraction, preserve the current file backend as default, and add contract tests for at least the file adapter plus one non-file or mocked adapter path.
