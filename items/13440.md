## Summary

Keep #13440 open. Current main still models `session.store` as a filesystem path and routes session load/save/update through `sessions.json` helpers plus filesystem lock files. There is no driver-shaped session-store config, `SessionStoreAdapter`, registry, or plugin hook that would let Redis, SQLite, Convex, PostgreSQL, or a custom module own session persistence. The supplied discussion also includes a same-day maintainer/Codex review explicitly keeping the issue open, and the related memory-backend PRs/issues are adjacent demand signals rather than a shipped session-store backend abstraction.

## Best Possible Solution

Keep #13440 open until maintainers either add a core `SessionStoreAdapter` contract or expose a stable plugin API that lets external backends own session persistence. The best path is to keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/session discovery through the abstraction, and add contract tests for the file adapter plus a mocked non-file adapter.
