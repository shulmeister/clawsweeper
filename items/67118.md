## Summary

Current main does not yet close the reported cron-specific gap. The shared model fallback runner can classify successful-looking returned results as failed attempts, and user/follow-up agent paths wire that classifier, but isolated cron `agentTurn` still calls `runWithModelFallback(...)` without `classifyResult` and returns `runEmbeddedPiAgent(...)` results directly. `runResult.meta?.error` is only consumed after fallback selection has already completed, so the reported `meta.error` return shape can still bypass candidate advancement.

## Best Possible Solution

Keep the issue open and implement the cron-specific normalization path. The likely fix belongs in `src/cron/isolated-agent/run-executor.ts`: pass the shared embedded-result classifier, or a cron-specific wrapper around it, into `runWithModelFallback(...)` so failover-eligible returned `meta.error` results become classified fallback errors before the candidate is accepted. Add a focused isolated-cron regression where primary `runEmbeddedPiAgent` returns a failover-eligible `meta.error` result and `payload.fallbacks` advances to the fallback provider/model, while preserving non-fallback terminal errors such as context overflow, role ordering, and image-size failures.
