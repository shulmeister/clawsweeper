## Summary

Current `main` now threads the raw activation-source config through the status/diagnostic plugin loader and has regression tests for the exact false-positive allowlist/provenance warning pattern reported here, so this issue appears fixed on `main`. I could not tie that fix to a released tag; it looks newer than `v2026.4.23`.

## Evidence

- **Status path now preserves raw trust/provenance source config:** `buildPluginReport()` resolves both the auto-enabled runtime config and the original `rawConfig`, then passes `activationSourceConfig: rawConfig` into both the diagnostics loader and metadata snapshot loader. That matches the split-context failure mode described in the issue comments and prevents status from evaluating trust against the wrong config snapshot.
  - file: [src/plugins/status.ts:183](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/plugins/status.ts#L183)
  - command: `nl -ba src/plugins/status.ts | sed -n '142,202p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Loader merges trust lists from the activation-source snapshot:** `resolvePluginLoadCacheContext()` builds `activationSourceConfig`, then `mergeTrustPluginConfigFromActivationSource()` merges allow/deny/loadPaths from that source snapshot before `warnWhenAllowlistIsOpen()` and `warnAboutUntrackedLoadedPlugins()` run. This is the core implementation needed to avoid false `plugins.allow is empty` and provenance warnings when runtime config differs from the source snapshot.
  - file: [src/plugins/loader.ts:1018](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/plugins/loader.ts#L1018)
  - command: `nl -ba src/plugins/loader.ts | sed -n '1018,1115p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Regression test covers the reported warning pattern:** The loader test `uses the source runtime snapshot allowlist for plugin trust checks` sets a runtime config and a different source config, then asserts that a plugin trusted only in the source snapshot loads, an untrusted plugin stays disabled, and no false `plugins.allow is empty` or `loaded without install/load-path provenance` warnings are emitted.
  - file: [src/plugins/loader.test.ts:6552](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/plugins/loader.test.ts#L6552)
  - command: `nl -ba src/plugins/loader.test.ts | sed -n '6552,6608p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Status tests verify raw activation-source plumbing:** `status.test.ts` explicitly checks that snapshot/diagnostic status loads use the auto-enabled config for runtime loading while still passing `activationSourceConfig: rawConfig`, which is the status-side wiring needed for this issue.
  - file: [src/plugins/status.test.ts:421](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/plugins/status.test.ts#L421)
  - command: `nl -ba src/plugins/status.test.ts | sed -n '406,430p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Issue discussion correctly identified the split-context root cause:** Commenters traced the bug to status/CLI paths seeing a different config context from the real runtime/source snapshot. Current `main` now carries an explicit raw activation-source config through those paths, so the discussion aligns with the implemented fix.
