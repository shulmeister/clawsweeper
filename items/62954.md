## Summary

Keep open. Current main still persists the just-used heartbeat provider/model into the session entry, and gateway/session display paths still resolve UI model identity from those persisted runtime fields. The prior direct fix attempt #62987 is closed unmerged, while the related cost snapshot work for #69347 did not add a heartbeat guard for model/modelProvider persistence.

## Best Possible Solution

Keep this open and implement the narrow persistence fix: thread `isHeartbeat` through `persistRunSessionUsage` / `persistSessionUsageUpdate` from both `runReplyAgent` and `createFollowupRunner`; when true, leave `entry.model` and `entry.modelProvider` unchanged in both persistence branches while still recording usage/context fields as intended. Add a focused regression test with default model != heartbeat model proving the heartbeat run uses the heartbeat model but the persisted session row/model display remains the normal session model.
