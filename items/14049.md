## Summary

Current main treats alternative memory backends as plugin-owned surfaces, not OpenClaw core work. A Qdrant memory backend fits the existing memory plugin slot and community plugin distribution path, so this request is out of scope for core and should live as a plugin/ClawHub package instead.

## Close Comment

Closing this as a `clawhub`-scope request after a Codex review of current `main`. The current architecture puts alternate memory backends behind the memory plugin slot rather than in core: `VISION.md:54-64` says optional capability should usually ship as plugins and that memory is a special plugin slot, `docs/plugins/manifest.md:676` routes memory backends through `plugins.slots.memory`, and `docs/tools/plugin.md:109-111` already treats `memory-core` and `memory-lancedb` as swappable memory plugins. Current docs also already cover semantic memory search, active recall, dreaming, and plugin-based cross-session memory (`docs/concepts/memory.md`, `docs/concepts/memory-honcho.md`). Given that scope, a Qdrant backend is better handled as an external memory plugin published via the normal plugin path/ClawHub instead of a core OpenClaw issue.
