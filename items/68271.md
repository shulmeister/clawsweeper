## Summary

Current main does not have a first-class per-agent write/edit path allow/deny schema, but it already exposes an in-process plugin hook surface that can inspect tool params plus agent identity and fail-closed block write/edit calls. Under the repo vision, that makes this a ClawHub/plugin concern rather than new OpenClaw core config.

## Evidence

- **Plugin hook can block tool calls with agent-aware context:** `before_tool_call` is documented to receive `event.toolName`, `event.params`, and `ctx.agentId`/`ctx.sessionKey`, and can return `block`/`blockReason`, which is enough to enforce per-agent write/edit path rules in a plugin.
  - file: [docs/plugins/hooks.md:105](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/plugins/hooks.md#L105)
  - command: `nl -ba docs/plugins/hooks.md | sed -n '70,150p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Hook path is in-process, not shell-out:** The automation docs explicitly describe plugin hooks as typed, in-process hooks and point users to them for `before_tool_call` use cases, addressing the issue body's concern that hooks would need to shell out.
  - file: [docs/automation/hooks.md:208](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/automation/hooks.md#L208)
  - command: `nl -ba docs/automation/hooks.md | sed -n '206,212p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Security posture is fail-closed:** The global hook runner config marks `before_tool_call` as `fail-closed`, which is the right behavior for a write/edit guard plugin: if the hook layer fails, the tool call is blocked instead of allowed through.
  - file: [src/plugins/hook-runner-global.ts:42](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/plugins/hook-runner-global.ts#L42)
  - command: `nl -ba src/plugins/hook-runner-global.ts | sed -n '36,50p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Plugins have their own config namespace:** OpenClaw documents plugin-specific config under `plugins.entries.<id>.config`, and the SDK exposes that data as `api.pluginConfig`, so an allow/deny glob policy can live in plugin config without adding new top-level core schema.
  - file: [docs/tools/plugin.md:43](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/tools/plugin.md#L43)
  - command: `nl -ba docs/tools/plugin.md | sed -n '36,46p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current core FS policy is intentionally coarse:** Current main only exposes `tools.fs.workspaceOnly` for filesystem path hardening; there is no built-in write/edit path allow/deny schema in the runtime types/schema. This is a capability gap, but one that can already be filled by the hookable plugin surface.
  - file: [src/config/types.tools.ts:296](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/config/types.tools.ts#L296)

- **Project vision pushes optional runtime capabilities to plugins/ClawHub:** The repo vision says core should stay lean, optional capabilities should usually ship as plugins, and capabilities needing runtime hooks belong on the plugin path with discovery/promotion through ClawHub.
