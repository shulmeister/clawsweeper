## Summary

Keep open. Current main has fixed the active config-path resolution part in 5d3168c3, but it has not implemented this PR's additive `redactPatterns` behavior. `resolvePatterns` still chooses configured patterns instead of defaults when any user patterns are present, and `redactToolDetail` still passes resolved config back through `redactSensitiveText`. The PR may need rebasing onto the newer config-path fix, but the remaining additive/default-redaction issue is real and unshipped.

## Best Possible Solution

Keep the PR open and adapt it onto current main: merge configured `logging.redactPatterns` with built-in defaults for config-derived redaction paths, keep explicit caller-supplied options semantics deliberate and tested, add regression coverage for config custom pattern plus default API-key redaction and `redactToolDetail`, then align user docs/schema help with the chosen additive contract.
