## Summary

Current main already handles the reported plugin-config reload crash path: plugin config changes are classified as gateway restarts, restart requests are coalesced and deferred until active work drains, SIGUSR1 goes through the managed restart loop, shutdown closes WebSocket/HTTP listeners with bounded waits, and loopback WebSocket connection failures are handled instead of becoming unhandled `ws` errors. The linked PR #64255 was a valid related attempt but was closed unmerged; main now covers the observable failure path through broader restart/shutdown hardening.

## Best Possible Solution

Close this issue as implemented on main. Keep the current managed restart/shutdown implementation and use any future reports to add a focused Docker reload regression if a new reproducible crash appears on a current release.
