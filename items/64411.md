## Summary

Keep open. Current main has partial MCP resilience, mainly a per-server connection timeout and startup error logging, but the central requested behavior is not implemented: bundle MCP catalog materialization still runs on the embedded Pi attempt setup path, MCP tool calls are awaited without an OpenClaw per-tool budget or circuit breaker, and plugin hooks can still block until they settle. The current docs/config expose only connection timeout and idle TTL controls, not degraded health state, failure thresholds, or fast optional-tool downgrade semantics.

## Best Possible Solution

Keep this open and implement a core resilience layer around external plugin/MCP work: configurable short catalog and tool-call budgets, per-server/tool failure counters with cooldown/circuit state, empty-tool fast degrade for optional/unhealthy MCP capability, explicit health diagnostics in status/logs, and tests covering slow catalog, slow callTool, repeated failure cooldown, and plugin hook timeout behavior. Preserve explicit tool allowlists and required tool calls so intentionally requested external tools can still fail visibly.
