## Summary

Keep #70934 open. Current main has faster QuickStart defaults and a post-setup hatch path, but the first-run flow is still setup-first rather than value-first: users configure auth/model, workspace, gateway, channels/search/skills, service/health, and then choose TUI/Web/later. The macOS app similarly runs connection and gateway wizard pages before the onboarding chat, and the mobile apps are pairing/permissions-first. This is a real product UX request, not a duplicate, not a ClawHub/plugin item, and not already implemented on main.

## Best Possible Solution

Keep the issue open for maintainer product/design work. A conservative path would add a clearly scoped value-first mode to CLI and macOS onboarding: present a few safe example tasks or templates, let the user run a low-risk first chat with sensible defaults, then defer optional persona/name/tone/channel/skill customization until after that first success while preserving explicit security/auth gates.
