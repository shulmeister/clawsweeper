## Summary

Keep open. Current main still has the llm-task model-routing bug described here: the tool derives provider and model as separate raw strings, validates the raw provider/model pair, and passes those raw values to the embedded runner. A model value such as `gemini-flash` or `google/gemini-3-flash-preview` is not resolved through the shared model alias/provider-ref path before dispatch, so the selected provider can remain the session/default provider. The related discussion remains useful: #48576 fixed a related resolver-family bug on another surface, while #26018/#26175 were haiku-alias-specific and do not fix llm-task bypassing alias resolution.

## Best Possible Solution

Keep this issue open. The best fix is to resolve llm-task's requested model through the shared alias/provider-ref path before allowlist validation and before calling `runEmbeddedPiAgent`, then pass the resolved provider plus bare model id. If the bundled plugin boundary cannot currently access that resolver, add a narrow runtime/plugin-facing resolver seam or change the embedded runner API to accept and resolve a model ref explicitly. Add focused regression tests for `model: "gemini-flash"`, `model: "google/gemini-3-flash-preview"`, and an Anthropic alias.
