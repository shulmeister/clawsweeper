## Summary

Current main still permits the reported overlap: `memorySearch` can index both workspace memory files and session transcript files, while the bundled `session-memory` hook still writes recent transcript content into `<workspace>/memory/*.md`. Search/result merging deduplicates by chunk identity, and chunk identity includes source/path, so a transcript chunk and a hook-generated memory summary remain separate searchable entries. The related PR #40618 is still only an open partial mitigation, not shipped on main.

## Best Possible Solution

Keep this issue open. The lowest-risk maintainer fix is a public docs warning in memory search/config and hooks docs: use `sources: ["memory"]` when relying on the bundled `session-memory` hook, or disable/redirect the hook when indexing transcripts directly. A code fix should add explicit provenance or a cross-source dedupe key so it avoids duplicate embedding work without hiding legitimate long-term memory notes.
