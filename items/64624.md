## Summary

Keep open. Current main still unconditionally queues WhatsApp connected/disconnected lifecycle notices as system events, then drains those queued events into agent-visible `System:` prompt lines. I found no current `statusEvents`, `notifyAfterMs`, or equivalent verbosity/throttle config, and the default no-peer route can still target the main session.

## Best Possible Solution

Keep this issue open and implement a tested channel lifecycle notification policy. A practical path is to add a typed WhatsApp/channel config such as `channels.<channel>.statusEvents` and/or `notifyAfterMs`, coalesce reconnects that recover before the threshold, preserve serious stop/logged-out/conflict notices, and add regression tests around the WhatsApp monitor plus `system-events` draining behavior. If generalized beyond WhatsApp, expose it as an additive channel/plugin SDK helper rather than hard-coding channel IDs in core.

## Risks / Open Questions

- Suppressing lifecycle notices must not hide durable logged-out, conflict, or stopped-channel states that require operator action.
- A generic core policy may affect other event producers; the narrowest safe first step is likely WhatsApp/channel lifecycle event filtering with shared helpers only if a second channel needs the same behavior.
