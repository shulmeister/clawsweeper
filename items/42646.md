## Summary

Keep open. Current `main` does not implement the requested SQLite-backed canonical memory record model. OpenClaw still treats Markdown memory files as the durable source of truth, and the builtin SQLite schema is only a search/index layer with `meta`, `files`, `chunks`, optional `embedding_cache`, and FTS/vector tables.

## Evidence

- **Memory source of truth is still Markdown files:** `docs/concepts/memory.md` says OpenClaw remembers things by writing plain Markdown files in the agent workspace, with durable memory in `MEMORY.md` and daily notes in `memory/YYYY-MM-DD.md`.
  - file: [docs/concepts/memory.md:9](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/docs/concepts/memory.md#L9)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Builtin SQLite is documented as an index, not canonical memory storage:** `docs/concepts/memory-builtin.md` describes the builtin engine as storing a per-agent SQLite memory index and says it indexes `MEMORY.md` and `memory/*.md` into chunks.
  - file: [docs/concepts/memory-builtin.md:9](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/docs/concepts/memory-builtin.md#L9)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Current SQLite schema lacks requested record/revision/entity tables:** `src/memory-host-sdk/host/memory-schema.ts` creates only `meta`, `files`, `chunks`, and optional `embedding_cache`, plus FTS. It does not define `memories`, `memory_revisions`, `memory_entities`, or `memory_links`, nor typed canonical records/status transitions.
  - file: [src/memory-host-sdk/host/memory-schema.ts:12](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/memory-host-sdk/host/memory-schema.ts#L12)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Memory core uses that index schema directly:** `extensions/memory-core/src/memory/manager-sync-ops.ts` calls `ensureMemoryIndexSchema(...)` for the active memory database, confirming the shipped implementation is the chunk index schema above.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:348](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-core/src/memory/manager-sync-ops.ts#L348)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Indexed writes are chunk-oriented, not canonical memory-record oriented:** `extensions/memory-core/src/memory/manager-embedding-ops.ts` writes chunk rows with `path`, `start_line`, `end_line`, `hash`, `model`, `text`, `embedding`, and `updated_at`, which is materially different from the issue's requested typed memory record model.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:534](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-core/src/memory/manager-embedding-ops.ts#L534)
