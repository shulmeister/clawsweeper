## Summary

Keep #71593 open. Current main supports SecretRef primitives for built-in env/file/exec providers and manifest-declared plugin config SecretInput paths, but it still does not expose a plugin-defined SecretRef provider contract or universal SecretInput coverage for the requested core schema surfaces. The linked AWS Secrets Manager PR #43643 was closed unmerged, and no equivalent AWS secret provider is present on current main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either explicitly document env/file/exec, especially exec, as the supported SecretRef extensibility model, or add a versioned plugin SecretRef-provider contract with aligned core types, Zod schemas, plugin SDK/runtime registration, Gateway protocol schema, resolver dispatch, docs, and tests. If pursuing the broader coverage ask, include remaining secret-bearing schema surfaces such as skills.entries.*.env.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would alter public config, plugin SDK, runtime resolver, and Gateway protocol contracts, so it needs explicit maintainer design and security review.
- The existing exec provider may be the intended extensibility answer, but that is not equivalent to the requested plugin-defined provider contract and would need maintainer confirmation plus clearer docs if this issue is to be closed later.
