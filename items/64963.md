## Summary

Keep open. Current `main` still only skips heartbeat execution when the main lane or resolved session lane has queued work; it does not check the active reply-run registry, so this PR's guard is not already implemented on `main` and the reported race remains plausible.

## Evidence

- **Current heartbeat guards on main do not include active reply-run state:** `runHeartbeatOnce` returns `requests-in-flight` for a busy main lane or busy resolved session lane, but there is no `resolveActiveReplyRunSessionId(sessionKey)` check between preflight and the session-lane guard.
  - file: [src/infra/heartbeat-runner.ts:759](https://github.com/openclaw/openclaw/blob/713807b55d0beb17c0b96d0490a8abae695bb960/src/infra/heartbeat-runner.ts#L759)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '754,796p'`
  - sha: [713807b55d0b](https://github.com/openclaw/openclaw/commit/713807b55d0beb17c0b96d0490a8abae695bb960)
- **Main still exposes the exact registry seam the PR wants to use:** `createReplyOperation` registers active runs by `sessionKey`, and `resolveActiveReplyRunSessionId(sessionKey)` is exported on current `main`, so the proposed fix targets a real existing contract rather than dead code.
  - file: [src/auto-reply/reply/reply-run-registry.ts:191](https://github.com/openclaw/openclaw/blob/713807b55d0beb17c0b96d0490a8abae695bb960/src/auto-reply/reply/reply-run-registry.ts#L191)
