## Summary

Keep open. Current main has adjacent mitigations for reserve-floor/context-window pressure, and the reporter later said their own loop stopped after raising reserveTokensFloor and restarting. But the exact requested retry-path cleanup is not fully implemented: successful overflow compaction still retries the same prompt, docs still say post-compaction retries rebuild context, and the hard-reset message still gives the generic reserveTokensFloor advice that this issue called misleading.

## Best Possible Solution

Keep this open unless maintainers intentionally consolidate it into #66126. The best fix is to make overflow-compaction retries resume the same logical turn with explicit retry metadata instead of replaying the original prompt/bootstrap as fresh input, make the hard-reset guidance context-aware when reserveTokensFloor is already high or the model contextWindow is the limiting factor, and add regression coverage for the high-reserve/bootstrap-pressure loop described here.
