## Summary

Keep open. Current `main` still rebuilds a reduced Teams conversation object for proactive sends instead of threading the stored conversation payload through unchanged, and the latest release `v2026.4.23` still contains that behavior. Existing regression coverage on `main` only locks in `tenantId`/`aadObjectId` forwarding, not the full conversation-payload preservation this PR adds.

## Evidence

- **Current proactive send path still reconstructs a reduced conversation object:** `createSendContext` still accepts `conversationId` and `conversationType` separately, then creates `conversation: { id: params.conversationId, conversationType: params.conversationType ?? "personal", ...(tenantId) }` for the outbound activity. That is the exact behavior the PR is trying to replace with forwarding the stored conversation payload unchanged.
  - file: [extensions/msteams/src/sdk.ts:277](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/msteams/src/sdk.ts#L277)

- **Adapter still drops stored conversation fields before send:** `continueConversation` on `main` extracts `reference.conversation?.id` and `reference.conversation?.conversationType`, then calls `createSendContext` with only those derived fields plus tenant/recipient data. Any additional stored conversation metadata is not preserved into the proactive send path.
