## Summary

Keep open. Current `main` does not contain the PR's same-provider embedded-model normalization or the related runtime-id propagation, so the reported fallback bug is still plausible and this PR is not already implemented on `main`.

## Evidence

- **No generic same-provider normalization on main:** `normalizeStaticProviderModelId()` only special-cases Anthropic, Hugging Face, Google, OpenRouter, xAI, and Vercel AI Gateway, then returns the original model string. A ref like `polza/gpt-4o-mini` is therefore not stripped on current `main`.
  - file: [src/agents/model-ref-shared.ts:59](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/model-ref-shared.ts#L59)
  - command: `nl -ba src/agents/model-ref-shared.ts | sed -n '56,82p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Configured fallback still exact-matches the requested id:** `resolveConfiguredFallbackModel()` looks up `providerConfig.models` only with `candidate.id === modelId`, and then builds the synthetic fallback model with `id: modelId`. If config stores bare `gpt-4o-mini` and the request is `polza/gpt-4o-mini`, model-level overrides are skipped and the prefixed id is preserved.
  - file: [src/agents/pi-embedded-runner/model.ts:549](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/pi-embedded-runner/model.ts#L549)

- **Resolver has no stripped-id retry path:** `resolveModelWithRegistry()` normalizes through `normalizeStaticProviderModelId()`, then tries explicit lookup, plugin dynamic lookup, and exact configured fallback. There is no second-pass retry with a same-provider prefix removed, which is the core change proposed by PR #49566.
