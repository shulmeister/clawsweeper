## Summary

Current `main` already addresses the issue as a docs/tooling gap: it now steers logged-in browser work toward the isolated `openclaw` browser or explicit `profile="user"` use, teaches agents to rely on visible UI state and report manual blockers, and surfaces Chrome existing-session prerequisites via browser doctor instead of leaving them implicit.

## Evidence

- **Browser login docs now recommend manual host-browser login and limit `profile="user"` use:** `docs/tools/browser-login.md` tells users to sign in manually in the host `openclaw` browser, not hand credentials to the model, and to use `profile="user"` only when existing logged-in sessions matter and the user is present.
  - file: [docs/tools/browser-login.md:13](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/docs/tools/browser-login.md#L13)

  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Browser docs now make the isolated browser the default and call out manual blockers:** `docs/tools/browser.md` says the default is the isolated `openclaw` browser, reserves `profile="user"` for explicit logged-in-session cases, and documents that login/2FA/captcha/camera/microphone blockers should be treated as manual action instead of guessed through.
  - file: [docs/tools/browser.md:72](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/docs/tools/browser.md#L72)
