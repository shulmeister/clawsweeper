## Summary

Keep #69450 open. Current main has agent-scoped store loading for explicit session keys, but the reported per-message gateway hot path is still plausible: normal agent runs emit file-only transcript updates without a sessionKey, and the gateway broadcast handler resolves the session key before checking that there is no message to broadcast. That resolution loads the combined gateway session store and discovers/loads stores across all agent session directories, so the item is not safe to close as implemented.

## Best Possible Solution

Keep the issue open and fix the remaining hot path directly: in the transcript update broadcast handler, return before resolving a session key when update.message is absent, and prefer carrying sessionKey on transcript events that do need broadcast. Add a focused regression test proving a file-only transcript update does not call the combined all-agent store lookup, while message-bearing updates with a sessionKey stay scoped to that session.
