## Summary

Current `main` and the shipped `v2026.4.22` release already contain the OpenAI-compatible usage fixes that match this report: streaming completions now always request usage metadata, Kimi/Moonshot cached-token fields are normalized, and `/status`/`session_status` fall back to transcript usage when stored session totals are zero.

## Evidence

- **Shipped fix for 0% usage on OpenAI-compatible streaming backends:** `CHANGELOG.md` documents a fix in `2026.4.19-beta.2`: `Agents/openai-completions: always send stream_options.include_usage on streaming requests, so local and custom OpenAI-compatible backends report real context usage instead of showing 0%.` That is the reported symptom here.
  - file: [CHANGELOG.md:441](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/CHANGELOG.md#L441)

- **Latest stable release contains the request-side usage fix:** The `v2026.4.22` tag (`a94763ee224e858e29dd177cd6f7f4daa438c2f1`) includes `stream_options: { include_usage: true }` in the OpenAI-completions streaming request builder, so compatible backends are asked to return usage instead of defaulting to zero.
  - file: [src/agents/openai-transport-stream.ts:1692](https://github.com/openclaw/openclaw/blob/a94763ee224e858e29dd177cd6f7f4daa438c2f1/src/agents/openai-transport-stream.ts#L1692)
