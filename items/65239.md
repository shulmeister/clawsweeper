## Summary

Keep open. Current main has useful SecretRef/audit/scrub machinery, including external file/exec providers, but it does not implement the requested built-in encrypted-at-rest configuration core or transparent `ENC:` values. Plaintext remains supported and still defaults in onboarding for API keys, so the central security feature remains unresolved.

## Best Possible Solution

Keep this open for maintainer product/security design. The likely path is to extend the existing SecretRef system with a first-party encrypted/keyring-backed provider or equivalent encrypted local vault, update onboarding/config writes to prefer that storage where appropriate, preserve fail-fast and last-known-good runtime behavior, and document/audit the migration from plaintext `.env` and auth-profile values. Avoid closing this as ClawHub because built-in encryption at rest is core security hardening, not just an optional third-party integration.

## Risks / Open Questions

- A future implementation should build on the existing SecretRef runtime snapshot, audit, apply, and active-surface semantics instead of adding a parallel secret path that bypasses current validation.
- Transparent decryption into assistant-readable context could worsen exfiltration risk unless the design also defines redaction, logging, prompt-context, and write-policy boundaries.
