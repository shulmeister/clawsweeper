## Summary

Current main still persists auto-enable-derived `enabled: true` values during gateway startup and doctor repair. A separate shipped fix appears to suppress one self-write restart-loop path, but the issue's requested runtime-only/non-persistent behavior remains missing.

## Best Possible Solution

Keep the issue open and fix the remaining config-boundary bug by separating runtime activation from persisted normalization: gateway startup and runtime loaders should use an in-memory auto-enabled snapshot, while `openclaw.json` and `doctor --fix` should not write auto-enable-derived `enabled` flags. Add regression coverage for missing `channels.whatsapp.enabled`, `channels.telegram.enabled`, and `plugins.entries.openai.enabled` staying absent after startup/doctor while the effective runtime still activates the configured components.
