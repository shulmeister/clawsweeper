## Summary

Keep #8969 open. Current `main` has implemented the first quick win as `openclaw skills check`, but the two remaining central requests are still not implemented: manifest-registry diagnostics are not exposed through the `status` surfaces, and the non-tool skill command path still rewrites the request body without an operator-visible warning. The existing maintainer/Codex discussion also explicitly says to keep this item open for those remaining gaps.

## Best Possible Solution

Keep #8969 open or split it into two focused follow-ups. Preserve the shipped `openclaw skills check` behavior, then add manifest-registry diagnostics to `openclaw status`, `openclaw status --json`, and `openclaw status --all`; add an operator-visible warning or structured log when a non-tool skill command rewrites the request for model-directed execution. Cover the status payload/text surfaces and the skill-command rewrite branch with focused tests, and update the public status/skills docs.
