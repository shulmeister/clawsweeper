## Summary

Keep open. Current main still creates Docker sandbox containers with `sleep infinity` as the long-lived container command and I found no sandbox `--init`/tini/dumb-init/reaper path, so the reported PID 1 zombie-reaping problem remains plausible and unimplemented.

## Evidence

- **Default sandbox image uses sleep as container command:** The default Docker sandbox image ends with `CMD ["sleep", "infinity"]`, matching the reporter's later observation that PID 1 is not an init/reaper process.
  - file: [Dockerfile.sandbox:24](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/Dockerfile.sandbox#L24)
  - command: `nl -ba Dockerfile.sandbox | sed -n '1,120p'`
  - sha: [0bcb4c95c185](https://github.com/openclaw/openclaw/commit/0bcb4c95c185909b12d463b04d49b16ead3459d5)
- **Runtime create path also starts sandbox with sleep infinity:** `createSandboxContainer` appends the configured image plus `sleep infinity` to the `docker create` arguments, then starts that container. This leaves no OpenClaw-side init/reaper command in the sandbox container startup path.
  - file: [src/agents/sandbox/docker.ts:499](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/src/agents/sandbox/docker.ts#L499)
  - command: `nl -ba src/agents/sandbox/docker.ts | sed -n '312,620p'`
  - sha: [0bcb4c95c185](https://github.com/openclaw/openclaw/commit/0bcb4c95c185909b12d463b04d49b16ead3459d5)
- **No current sandbox init/reaper implementation found:** A targeted search for `--init`, `tini`, `dumb-init`, `reaper`, `SIGCHLD`, and related terms in the Docker sandbox code/docs only found the `sleep infinity` entries, not a reaping implementation.
  - command: `rg -n -e "--init" -e "init: true" -e "tini" -e "dumb-init" -e "reaper" -e "SIGCHLD" -e "wait -n" -e '"sleep", "infinity"' Dockerfile.sandbox Dockerfile.sandbox-common src/agents/sandbox src/agents/bash-tools.shared.ts src/commands/doctor-sandbox.ts docs/gateway/sandboxing.md docs/cli/sandbox.md`
  - sha: [0bcb4c95c185](https://github.com/openclaw/openclaw/commit/0bcb4c95c185909b12d463b04d49b16ead3459d5)
- **pids limit risk is a real configured surface:** The Docker create args include `--pids-limit` when `sandbox.docker.pidsLimit` is configured, so unreaped zombies can consume the exact process-budget surface described by the issue.
  - file: [src/agents/sandbox/docker.ts:433](https://github.com/openclaw/openclaw/blob/0bcb4c95c185909b12d463b04d49b16ead3459d5/src/agents/sandbox/docker.ts#L433)
  - command: `nl -ba src/agents/sandbox/docker.ts | sed -n '312,620p'`
  - sha: [0bcb4c95c185](https://github.com/openclaw/openclaw/commit/0bcb4c95c185909b12d463b04d49b16ead3459d5)
- **Issue discussion narrows but does not obsolete the report:** The maintainer/commenter negative datapoint was limited to quiet gateway uptime, while the reporter's follow-up narrowed the issue to orphaned children under PID 1 after shell-heavy/subagent-adjacent workloads and observed PID 1 as `sleep infinity` (issue comment https://github.com/openclaw/openclaw/issues/68691#issuecomment-4320580821).
