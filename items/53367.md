## Summary

Keep open. Current `main` still lacks any runtime auth-snapshot freshness check, so this PR is not implemented on `main` and the reported startup overwrite path remains plausible from code inspection.

## Evidence

- **Runtime snapshot layer has no freshness metadata or stale invalidation on main:** `src/agents/auth-profiles/runtime-snapshots.ts:4-18` stores snapshots in a plain `Map` and returns clones directly; `:37-57` only clear/set entries. There is no mtime or load-timestamp tracking anywhere in this file.
  - file: [src/agents/auth-profiles/runtime-snapshots.ts:4](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles/runtime-snapshots.ts#L4)

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)

If `auth-profiles.json` changed after snapshot activation, current main would still serve the cached runtime copy.
  - file: [src/agents/auth-profiles/store.ts:60](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles/store.ts#L60)
