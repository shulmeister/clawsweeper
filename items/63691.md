## Summary

Keep open. Current main has beta-channel and rollback documentation, but the concrete hang described in the discussion is still plausible in the active Gateway run path: embedded runs still serialize by session lane, then read/parse the full session JSONL before `SessionManager.open`, with no large-file guard like the CLI history path has. A closed related Windows gateway report is not a canonical replacement for this large-transcript bootstrap freeze.

## Best Possible Solution

Keep this issue open and fix the active Gateway transcript-load path. The best implementation would add bounded large-session handling before any full-file repair or `SessionManager.open` on embedded run and compaction paths, either by rotating/compacting safely, using a streaming/yielding repair/load path, or failing fast with an actionable `sessions cleanup`, `/reset`, or `/compact` remediation. Add regression coverage with an oversized JSONL transcript that proves Gateway health/channel loops are not starved, then document the operator recovery path in the public troubleshooting/session docs.

## Evidence

- **Reporter discussion identifies a concrete root cause:** The issue comment reports the Gateway becoming silent after acquiring `agent:main:main`, with a large 89 MB / ~40,000-line JSONL session file causing event-loop lockup and freezing Telegram, WeChat, and Feishu handlers. That turns the original broad stability complaint into a specific large-transcript bootstrap/load failure to fix rather than a stale general support request.
- **Current run path still serializes by session lane before the attempt:** `runEmbeddedPiAgent` resolves a per-session lane from the session key/session id and enqueues the run on that lane. This matches the reported `agent:main:main` queue behavior; it does not isolate expensive transcript parsing away from the Gateway event loop.
