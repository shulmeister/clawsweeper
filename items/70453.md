## Summary

Keep open. Current main has better Gemini transport error propagation and a documented model-provider `request.proxy` override, but the reported ambient `http_proxy`/`https_proxy` workflow is not implemented for LLM provider guarded fetches: without a configured `models.providers.google.request.proxy.mode: "env-proxy"`, `buildGuardedModelFetch` still calls `fetchWithSsrFGuard` in strict/default mode and DNS-pins the target, so a Fake-IP private/special-use DNS answer can still hit the SSRF block instead of routing through the proxy.

## Best Possible Solution

Keep the issue open and either implement a safe LLM provider auto-upgrade to trusted env-proxy mode when `HTTP_PROXY`/`HTTPS_PROXY` applies and `NO_PROXY` does not, or make a maintainer product decision that model-provider proxying must remain explicit and then update Google/provider docs plus CLI/WebUI diagnostics to point affected users at `models.providers.google.request.proxy.mode: "env-proxy"` or `explicit-proxy`. Add a regression test covering Google `google-generative-ai` with proxy env plus Fake-IP/private DNS behavior and user-visible error propagation.
