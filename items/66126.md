## Summary

Keep #66126 open. Current main has useful replay/liveness hardening for compaction pressure, but the specific cron/system prompt compaction-retry problem is not implemented or safely superseded: overflow and timeout recovery still rebuild the prompt from `params.prompt` and `continue` into another attempt after compaction, without the requested durable retry marker, original-message ledger, or turn-attempt identity. Related #59618 and #63216 remain relevant but are broader/adjacent, not a clean canonical duplicate for this cron/tool-error reinjection loop.

## Best Possible Solution

Keep this open and fix it in the embedded runner/session lifecycle: carry durable turn identity through compaction retries (`retryOfTurnId`, original message ids, attempt count, or equivalent), avoid presenting cron/system/runtime prompts as fresh user input after compaction, and add a regression covering an isolated cron agent turn with a deterministic failing `exec`/tool call that triggers compaction and must not re-run indefinitely. #59618/#63216 can inform the design, but #66126 should remain as the concrete cron/tool-error retry-loop bug.
