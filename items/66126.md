## Summary

Keep this issue open. Current main still retries overflow compaction by rebuilding and resubmitting the same prompt, without a durable turn/retry identity such as retryOfTurnId/originalMessageIds/turnAttempt. Recent cron denial handling reduces one exec-failure trigger, but it does not fix the generic compaction-resume contract or the cron/system re-injection behavior described here.

## Best Possible Solution

Keep the issue open and fix the Pi embedded runner/subscriber compaction retry contract: preserve a stable current-turn identity across compaction, carry retry attempt/original-message metadata into the resumed attempt, avoid treating cron/system prompt reconstruction as a fresh user turn, and add regression coverage showing a failed cron/tool cascade does not re-execute as new input after compaction. Related #59618 and #63216 should remain cross-linked but should not replace this narrower report.
