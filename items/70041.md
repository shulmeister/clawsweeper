## Summary

Current `main` still appears to route streamed Google Chat follow-up chunks through `currentMessageId`, and for Google Chat that value is the message resource (`spaces/.../messages/...`) rather than the thread resource (`spaces/.../threads/...`). The issue matches the present code path and should stay open.

## Evidence

- **Inbound Google Chat context separates message id from thread id:** The Google Chat monitor populates `MessageSid`/`MessageSidFull` from `message.name`, while `ReplyToId`/`ReplyToIdFull` come from `message.thread?.name`. That matches the reporter's distinction between message resource and thread resource.
