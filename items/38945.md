## Summary

Keep #38945 open. Current main still has the Unicode tokenizer gaps the PR is trying to fix, now under `extensions/memory-core`: MMR still tokenizes broad text with an ASCII-only regex plus CJK special handling, FTS query construction still omits `\p{M}` and NFC normalization, and the shared FTS search planner has the same mark/normalization gap. The merged CJK chunking work is related but does not supersede this PR's accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining-mark, or NFC/NFD behavior.

## Best Possible Solution

Keep the PR open and port the narrow tokenizer fix to current `extensions/memory-core`: update MMR tokenization, `buildFtsQuery()`, and shared FTS search token planning to use NFC-normalized Unicode letter/mark/number tokens while preserving shipped CJK and trigram behavior. Add focused tests for ASCII, CJK, Finnish/accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining marks, and NFC/NFD equivalence.
