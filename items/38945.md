## Summary

Keep #38945 open. Current main still has the Unicode tokenizer gaps the PR targets: MMR tokenization only extracts ASCII plus CJK-specific tokens, FTS query construction omits combining marks and NFC normalization, and the shared keyword search planner uses the same marks-missing regex. The merged CJK/trigram work improves Chinese/Japanese/Korean paths but does not cover accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining marks, or NFC/NFD equivalence.

## Best Possible Solution

Keep the PR open and port the narrow tokenizer fix to current `extensions/memory-core`: update MMR tokenization, `buildFtsQuery()`, and shared keyword-search planning to use NFC-normalized Unicode letter/mark/number tokens while preserving current CJK bigram/unigram and trigram fallback behavior. Add focused regression tests for ASCII, CJK, Finnish/accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining marks, and NFC/NFD equivalence.
