## Summary

Keep PR #38945 open. Current main still lacks the PR’s central Unicode tokenizer fix in the current `extensions/memory-core` paths: MMR remains ASCII-plus-CJK rather than general Unicode, and FTS query tokenization still omits combining marks and NFC normalization. The related CJK work does not supersede this because it covers CJK-specific behavior, while this PR still addresses non-CJK scripts and combining-mark scripts.

## Best Possible Solution

Keep this PR or a rebased equivalent open. The best implementation path is to add NFC normalization and `\p{L}\p{M}\p{N}_` token handling in the current memory-core MMR/FTS paths, reconcile the duplicate search-token regex in `manager-search.ts`, and add focused regression tests for accented Latin, Arabic/Hebrew, Thai, Hindi, and NFC/NFD behavior without regressing existing CJK tests.

## Evidence

It does not tokenize Arabic, Hebrew, Thai, Hindi, Cyrillic, or accented Latin as whole Unicode words and does not normalize NFC.
