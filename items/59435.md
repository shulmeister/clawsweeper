## Summary

Keep open. Current `main` still refreshes the Gateway service after `openclaw update` by running `gateway install --force` with the caller's ambient `process.env`, and the forced reinstall path explicitly does not reuse existing `OPENCLAW_*` values from the installed service. That means the reviewed code still has the exact failure mode described: a default LaunchAgent can be rewritten with whatever `OPENCLAW_CONFIG_PATH` the update process inherited, instead of preserving the config path already stored in the existing plist.

## Evidence

- **Update flow rewrites service env from ambient process env:** After a successful update, the restart path sets `refreshGatewayServiceEnv = true` when a service is loaded, then `refreshGatewayServiceEnv()` runs `gateway install --force` using `resolveServiceRefreshEnv(process.env, invocationCwd)`. There is no readback of the currently installed plist before reinstall.
  - file: [src/cli/update-cli/update-command.ts:256](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/update-cli/update-command.ts#L256)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '170,290p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Forced reinstall intentionally drops existing OPENCLAW_* service variables:** `runDaemonInstall()` merges some existing service env, but explicitly skips `HOME`, `PATH`, `TMPDIR`, and every `OPENCLAW_*` variable. So an existing service's `OPENCLAW_CONFIG_PATH` is not preserved during `--force` reinstall.
  - file: [src/cli/daemon-cli/install.ts:27](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/daemon-cli/install.ts#L27)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Test locks in non-preservation of existing config path:** The install test asserts that a forced reinstall leaves `env.OPENCLAW_CONFIG_PATH` undefined even when the currently loaded service command contained one, confirming the code is designed not to reuse the installed service's config path.
  - file: [src/cli/daemon-cli/install.test.ts:417](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/daemon-cli/install.test.ts#L417)
  - command: `nl -ba src/cli/daemon-cli/install.test.ts | sed -n '416,452p'`

- **Service env writes config path from invocation env, label from profile only:** `buildServiceEnvironment()` copies `OPENCLAW_CONFIG_PATH` directly from the invocation env into the managed service environment, while the LaunchAgent label is derived separately from `OPENCLAW_PROFILE`. With no profile set, the label remains `ai.openclaw.gateway` even if `OPENCLAW_CONFIG_PATH` points elsewhere.
  - file: [src/daemon/service-env.ts:253](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/daemon/service-env.ts#L253)

- **Docs show multi-gateway support expects separate profiles/service names:** The docs describe supported multi-gateway isolation as separate profiles with separate managed service names, and the default macOS label remains `ai.openclaw.gateway` only for the default profile. That makes overwriting the default plist with a different config path especially risky for mixed default/custom-env setups.
