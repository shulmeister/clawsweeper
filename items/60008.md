## Summary

Keep open. Current main still has the reported diagnostic gap: `handleToolExecutionStart` only treats `path` and `file_path` as read paths, so `file`/`filePath` are not implemented there. Current read-tool validation also appears to have moved toward canonical `path` only, so this needs maintainer alignment rather than cleanup closure.

## Best Possible Solution

Keep this issue open until maintainers choose a coherent direction: either support `path`, `file_path`, `file`, and `filePath` end-to-end in the read tool schema/validation/normalization plus diagnostic tests, or explicitly document/enforce canonical `path` only and close the alias request with that rationale. Do not close as implemented on current main.
