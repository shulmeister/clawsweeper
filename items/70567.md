## Summary

Keep #70567 open. Current main retries short-lived embedding 5xx failures and avoids swapping a failed full reindex into place, but it still does not persist per-file embedding failures or derive `Dirty` from source/index coverage. A status-only manager reports clean whenever index metadata exists, and the CLI status path only prints the indexed/total ratio without turning a partial store into a dirty or warning state. The related #26772 remains adjacent but not canonical for this per-embedding-failure/status-visibility bug.

## Best Possible Solution

Keep the issue open and fix the built-in memory backend/status contract. The best path is to add durable failed-or-missing source coverage detection: either persist failed file paths/attempt state during indexing and include it in `status().dirty`, or have `memory status` compare scanned source files with indexed `files` rows and emit an explicit warning/dirty signal when coverage is incomplete. Add a regression test that simulates a remote `/v1/embeddings` 500 during indexing, verifies the failed file is not silently considered clean, and verifies a subsequent non-force `openclaw memory index` or `memory status --index` repairs the missing files.
