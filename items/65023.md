## Summary

Keep open. Current main does not implement the proposed `google-genai` provider or `google-genai-sdk` auth contract, and this PR is a plausible focused implementation candidate rather than conservative cleanup-close material. Main has existing `google` / `google-vertex` support and uses `@google/genai` in the bundled Google plugin for some media surfaces, but not the requested unified text provider/auth path; the provided latest review context also still identifies behavior gaps to resolve before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is either to revise it so the new Google GenAI provider fits the current bundled Google plugin/provider architecture, covers all keyless-auth callers, forwards tools/system/options/finish metadata, and adds focused tests/docs, or to supersede it with a narrower PR that fixes the remaining `google-vertex` ADC/proxy behavior in the existing provider path.

## Risks / Open Questions

- The PR changes shared model-provider config and auth contracts, so it needs maintainer review for whether a new provider id is preferable to extending the existing bundled Google plugin.
- The proposed keyless auth mode must be handled consistently across embedded runs, `/btw`, tools, compaction, and any other `requireApiKey` caller before merge.
- The existing Google/Vertex issues referenced by the PR overlap but are not identical; closing this PR now would discard a possible implementation path while some related Vertex auth/proxy reports remain open.
