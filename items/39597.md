## Summary

Keep open. Current main already distinguishes browser Control UI and native macOS clients in Gateway connect metadata, but it still does not expose a trusted client type in the agent-facing WebChat runtime context; chat turns only carry the client id as sender metadata/scopes, and the runtime line still renders channel/capabilities without client fields.

## Best Possible Solution

Keep #39597 open and implement a narrow trusted Gateway client-context path: define sanitized client metadata, propagate connect.client through chat.send and agent run setup into RuntimeInfoInput or trusted inbound metadata, update buildRuntimeLine and prompt tests to expose fields such as client=openclaw-macos or client_mode=ui, and document which WebChat/native client fields are trusted and privacy-safe.
