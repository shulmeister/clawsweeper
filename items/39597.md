## Summary

Keep open. Current main already carries distinguishable Gateway client metadata for browser Control UI and native macOS clients, but the agent-facing runtime/inbound context still does not expose a trusted client type such as `client=macapp`; it only exposes `channel=webchat` plus channel capabilities, while the Gateway caller ID is mapped into sender metadata.

## Best Possible Solution

Keep the issue open and implement a narrow trusted client-context path: define a sanitized Gateway client descriptor for agent runtime context, propagate it from `client.connect.client` through `chat.send`/agent run setup into `RuntimeInfoInput` or trusted inbound metadata, update `buildRuntimeLine`/prompt tests, and document which fields are trusted and privacy-safe for WebChat/native clients.

## Risks / Open Questions

- Agents can sometimes infer the caller from untrusted sender metadata today, but that does not satisfy the requested trusted runtime/client context.
- A future fix should avoid exposing raw `userAgent` to agents without an explicit privacy/product decision; `id`, `mode`, and sanitized platform are safer first fields.
