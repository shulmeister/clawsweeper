## Summary

Keep open. Current main has implemented important parts of the request, but the remaining hardening is still actionable: OAuth-producing login paths still persist through the unlocked auth-profile upsert path, and duplicate-enrollment protection is still scoped to provider/profileId rather than the same OAuth enrollment copied under different profile IDs, stores, or hosts.

## Best Possible Solution

Keep this issue open and narrow it to the remaining hardening: route OAuth-producing auth mutations such as models auth login and writeOAuthCredentials through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, add a practical duplicate-enrollment guard or stable identity strategy where feasible, and document the single-writer boundary for auth-profiles.json and Codex CLI coexistence. Preserve the already-shipped atomic writes, same-profile refresh serialization, refresh-token-reused recovery, and Codex bootstrap-only behavior.

## Risks / Open Questions

- Closing now would lose tracking for the remaining unlocked OAuth credential mutation paths.
- The current refresh lock is keyed by provider/profileId, not by a stable enrollment identity, so copied OAuth chains can still race when they live under different profile IDs, stores, or hosts.
- Docs describe the token-sink and Codex CLI bootstrap boundary, but they do not yet document or enforce the single-writer/duplicate-enrollment boundary requested by downstream operators.
