## Summary

Current `main` still treats channel reactions as queued system events, not inbound turns. The documented/configured reaction modes remain notification-only, and there is no `reactionAutoRespond`/`reactionTriggersTurn`-style surface on `main`, so #11747 is not implemented and should stay open.

## Evidence

- **Discord docs still describe reaction notifications as system events only:** The Discord channel docs say reaction events are turned into system events and attached to the routed session; they do not describe waking or starting an agent turn.
  - file: [docs/channels/discord.md:709](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/channels/discord.md#L709)

- **Discord reaction handler only enqueues a system event:** The Discord listener resolves the agent route and calls `enqueueSystemEvent(...)` for reactions. There is no follow-up wake/request/reply call in this path.
  - file: [extensions/discord/src/monitor/listeners.ts:645](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/discord/src/monitor/listeners.ts#L645)
