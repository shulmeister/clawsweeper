## Summary

Current main satisfies the issue's requested stronger status-truth path for Discord: `connected` is now driven by gateway READY/reconnect state, WebSocket frame activity is surfaced as `lastTransportActivityAt`, and health/readiness logic treats disconnected or stale transport state as actionable. The activity timestamps themselves remain process-local, but the issue explicitly allowed a reliable runtime/probe connection signal as an acceptable fix.
