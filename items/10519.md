## Summary

Keep open. Current main still renders Telegram fenced code blocks as plain `<pre><code>` without a `language-*` class, and the Telegram tests currently assert that behavior. The related WebChat copy-button issue is a separate surface and does not supersede the Telegram request.

## Evidence

- **Telegram renderer omits language class:** `renderTelegramHtml` maps `code_block` to the fixed marker pair `<pre><code>` / `</code></pre>`, so fence language is not emitted in Telegram HTML.
  - file: [extensions/telegram/src/format.ts:66](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/telegram/src/format.ts#L66)
  - command: `nl -ba extensions/telegram/src/format.ts | sed -n '48,90p'`
  - sha: [1dfa52d0711d](https://github.com/openclaw/openclaw/commit/1dfa52d0711dfc86e1c04538fe46e3d43961766f)
- **Regression test preserves current missing behavior:** The fenced-code test passes ` ```js ` input and expects `<pre><code>const x = 1;\n</code></pre>`, with no `class="language-js"`.
  - file: [extensions/telegram/src/format.test.ts:55](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/telegram/src/format.test.ts#L55)
