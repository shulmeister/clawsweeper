## Summary

Keep #10519 open. Current main at 82262387651ab270b565f71533ce4937c9c00dc6 still renders Telegram fenced code blocks as static `<pre><code>` HTML, drops the markdown fence language before rendering, and has tests that explicitly expect no `language-js` class. The related WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram by retaining optional fence language metadata through the shared Markdown IR/render/chunking path, or by adding a narrowly scoped Telegram fence-aware renderer if maintainers do not want to widen the shared IR contract. Add Telegram formatter and chunking tests that expect `class="language-js"` for language fences while preserving unlabeled/indented code-block behavior, escaping, file-reference wrapping, and chunk-limit handling.
