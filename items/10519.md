## Summary

Keep #10519 open. Current main still renders Telegram fenced code blocks as static `<pre><code>` without preserving the fence language, and the same-day maintainer/Codex comment already identified this as unsolved and separate from WebChat copy-button work.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram by carrying optional code-block language metadata through the shared Markdown IR/render path, or by adding a Telegram-specific fence-aware renderer if maintainers prefer not to widen the IR. Update Telegram format, wrapping/chunking, and outbound tests to expect `class="language-js"` while preserving current escaping and chunking behavior.
