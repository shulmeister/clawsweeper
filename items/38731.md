## Summary

Keep PR #38731 open. Current main still does not implement the requested non-blocking high-frequency `--every <30m` cron advisory, and the PR is a plausible focused fix, but it needs rework before merge because it covers only add paths and the review comments identify concrete logger/output issues.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation. Centralize a small high-frequency `every` schedule helper and apply it consistently in CLI `cron add`, CLI `cron edit --every`, gateway `cron.add`, and gateway `cron.update`. Keep sub-30-minute schedules valid, emit/log the advisory non-fatally through the repo's warning channel and `SubsystemLogger.warn(message, meta)` order, and add focused regression tests for add and update paths.

## Risks / Open Questions

- Merging the PR as-is would leave `cron edit --every` and direct `cron.update` clients without the requested advisory.
- The PR's gateway warning call uses the wrong `SubsystemLogger.warn(message, meta)` argument order and can break the warning path when console warning output is enabled.
- The CLI patch emits a non-fatal advisory through `defaultRuntime.error`, which is inconsistent with warning-style CLI output and may confuse scripts that treat stderr as failure.
