## Summary

Keep PR #38731 open. Current main has shipped cron run-session retention/reaping, which mitigates the linked accumulation class, but it still does not implement this PR's specific high-frequency `--every <30m` advisory in CLI or gateway create/update paths. The PR remains a plausible focused guardrail, but it needs rework before merge because the current patch only covers add/create, uses the CLI error channel for a non-fatal warning, and calls the gateway logger with the wrong argument order.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation. The best path is to centralize a small `isHighFrequencyEverySchedule`/warning helper and apply it consistently in CLI `cron add`, CLI `cron edit --every`, gateway `cron.add`, and gateway `cron.update`. Keep sub-30-minute schedules valid, emit/log the warning non-fatally using the proper CLI warning channel and `SubsystemLogger.warn(message, meta)` order, and add focused regression tests for add and update paths.

## Risks / Open Questions
