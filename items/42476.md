## Summary

Keep open. Current main still serves `/media/:id` with only `X-Content-Type-Options: nosniff`; it does not set the `Cache-Control`, `Pragma`, or `Expires` directives proposed by PR #42476, and the media tests do not assert those cache headers. The related #42606 hardens a separate noVNC surface and does not supersede this media endpoint change.

## Best Possible Solution

Keep the PR open and refresh it against current main: set the anti-cache headers immediately after a `/media/:id` route match and before all validation/read branches, then add regression coverage for success, missing media, and preferably other error/HEAD responses through the shared media test helper.

## Evidence

- **Current media handler lacks anti-cache headers:** At current main, the `/media/:id` handler sets `X-Content-Type-Options` before validation and response branches, but there are no `Cache-Control`, `Pragma`, or `Expires` setters in the media server path.
  - file: [src/media/server.ts:137](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/media/server.ts#L137)

  - sha: [a57d681db9b2](https://github.com/openclaw/openclaw/commit/a57d681db9b237348c5d446c8d371b2c1057f016)
- **Tests only cover nosniff/status behavior:** The shared media response assertion only checks status and optional `x-content-type-options`; the missing-media case likewise only asks for `expectedNoSniff`, so current tests would not catch absent no-store/no-cache headers.
