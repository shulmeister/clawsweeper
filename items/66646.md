## Summary

Current main partially fixes the classification piece by giving session lock timeouts a typed error and suppressing timeout failover classification. However, `runWithModelFallback()` still deliberately continues fallback on unrecognized errors while candidates remain, and a session lock timeout becomes exactly that non-failover/unknown error. The issue’s central request, fail fast or queue/back off instead of trying other models, is still not implemented.

## Best Possible Solution

Keep the issue open. The narrow fix should make `SessionWriteLockTimeoutError` or `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` a non-fallback infrastructure error in `runWithModelFallback()` before candidate recording/continuation, then add a regression test with multiple configured candidates proving only one run attempt occurs and no model substitution/fallback decision is emitted. Queue/backoff behavior should be implemented at the session/gateway scheduling layer if maintainers want automatic retry after the fail-fast path.
