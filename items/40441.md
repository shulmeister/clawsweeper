## Summary

Keep open. Current main still has the stale config reload behavior this PR targets, but #40441's one-line fix resets `configFormDirty` inside the shared `loadConfig()` path and would also discard unsaved edits during passive refresh paths such as tab refreshes and reconnects. The best path is an explicit-reload-only discard mechanism, likely using the related #40443 approach or a similar scoped option.

## Best Possible Solution

Do not merge #40441 unchanged. Implement or follow a scoped explicit-reload solution, such as a `loadConfig` option or separate reload handler that discards pending config edits only for user-initiated Reload actions, while preserving dirty form state for passive refreshes; keep regression tests for both behaviors. Related PR #40443 appears closer to that product shape and should be treated as the likely canonical follow-up if maintainers choose one.
