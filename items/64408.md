## Summary

Keep open. Current main still contains the reported `read tool called without path` warning path, and the warning still lacks the requested run/session correlation. Main does now validate canonical `path` at read execution time and has shipped heartbeat path guidance, but that does not fully resolve this issue’s malformed embedded caller/recovery-loop concern, especially the Feishu/reconnect/timeout context.

## Best Possible Solution

Keep this issue open as a diagnostics/root-cause task. Add structured correlation to the `read tool called without path` warning or adjacent event (`runId`, `sessionId`, `sessionKey`, trigger, channel/plugin/provider where available), then reproduce the Feishu timeout/reconnect/failover path. If the caller is heartbeat, cron, or another embedded task, fix that owner to construct canonical `read({ path })`; if malformed model/provider output is expected, reject or normalize it before the embedded start handler so this warning becomes actionable rather than anonymous noise.
