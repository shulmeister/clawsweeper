## Summary

Keep this issue open. Current main has useful mitigations for auto-sourced session overrides, backup creation, session UI/RPC deletion, and some cron/session durability work, but the central upgrade-safety request remains only partially addressed: updates still do not automatically snapshot state/workspace/session data, rollback is documented as manual pinning rather than a first-class restore path, `openclaw sessions` still lacks targeted reset/delete tooling for one poisoned session key, and memory/status diagnostics still do not clearly flag a missing `MEMORY.md` when other daily memory files exist. The reporter's follow-up discussion also has a second user confirming repeated webchat session resets on 2026.4.11, so this is not a low-signal single-host report.

## Best Possible Solution

Keep this issue open and split or implement it as concrete upgrade-safety work: wire an opt-out pre-update backup through the existing backup planner, design a supported backup restore or rollback command, add safe CLI tooling to reset/delete or clear overrides for one session key, and add doctor/migration diagnostics for stale model/provider/auth overrides plus missing `MEMORY.md`/workspace binding checks. Existing fixes for auto-sourced overrides and compaction bookkeeping should be retained as partial mitigation, not treated as complete resolution.
