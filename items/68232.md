## Summary

Keep open. Current `main` still treats any `channels.telegram.*` edit as a hot-reload action that restarts the Telegram channel, and Telegram group history is held in per-bot in-memory maps that are recreated on start. I did not find a fix on `main`, and the current docs still describe channel config as hot-applied without restart, so this does not look implemented or obsolete.

## Evidence

- **Telegram plugin marks the whole config subtree as reload-sensitive:** The bundled Telegram channel declares `reload: { configPrefixes: ["channels.telegram"] }`, so any change under `channels.telegram` is eligible for channel reload handling.
  - file: [extensions/telegram/src/shared.ts:161](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/telegram/src/shared.ts#L161)
  - command: `nl -ba extensions/telegram/src/shared.ts | sed -n '150,165p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Gateway reload planning turns channel config changes into restart-channel actions:** `buildGatewayReloadPlan` converts each channel plugin `configPrefixes` entry into a hot rule with `actions: ["restart-channel:<id>"]`. That means a matching Telegram config edit does not just swap config in place; it schedules a Telegram channel restart.
  - file: [src/gateway/config-reload-plan.ts:140](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/config-reload-plan.ts#L140)

- **Hot reload handler actually stops and starts the channel:** When `plan.restartChannels` is non-empty, the handler logs `restarting <name> channel`, then calls `stopChannel(name)` followed by `startChannel(name)` for each entry.
  - file: [src/gateway/server-reload-handlers.ts:175](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server-reload-handlers.ts#L175)

  - file: [src/gateway/config-reload.test.ts:202](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/config-reload.test.ts#L202)

## Risks / Open Questions

- I did not run a live Telegram reproduction in this read-only pass, so the exact user-visible impact on cache-ttl/session state was inferred less directly than the channel restart and `groupHistories` loss.
- The docs evidence is about user-facing restart expectations; it strengthens that the issue is not clearly resolved, but it does not by itself prove the runtime bug.
