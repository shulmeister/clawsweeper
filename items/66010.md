## Summary

Keep open. Current main has adjacent guardrails for subagent spawning and related retry safety, but it still lacks the requested native per-subagent consecutive-failure circuit breaker with cooldown, reset/override behavior, notification, config schema, CLI, and UI visibility. The same-feature #65817 is already closed and is not an active canonical tracker; #20341 was a closed unmerged heartbeat PR, and #60266 addressed model fallback classification rather than this spawn-level breaker.

## Best Possible Solution

Keep this issue open and implement a native subagent breaker in the core spawn and registry path. The likely path is to persist breaker state keyed by requester plus target agent and a normalized task signature, update it from subagent completion outcomes, reset on success, check it before the gateway agent call, support explicit override/manual reset, add strict config/schema/docs, and expose state through subagent status, tasks/agents CLI, and Control UI. Provider-level cost/rate-limit guardrails should stay as coordinated follow-up scope unless maintainers intentionally combine them.

## Risks / Open Questions

- Breaker keying needs design: target agent only may block unrelated tasks, while task-signature only may miss repeated bad agent configuration.
- Provider-level billing/rate-limit cooldowns should remain separate from this subagent failure breaker, even though the reported incident touches both surfaces.
- Defaults and override semantics need care so transient failures do not permanently block legitimate recovery attempts.
