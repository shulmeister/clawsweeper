## Summary

Current main already treats HTTP 503 as a retryable failover condition and advances to configured model fallbacks; the same behavior is present in release v2026.4.22.

## Close Comment

Codex review: current `main` already implements this. HTTP 503 is classified as a retryable failover reason in `src/agents/pi-embedded-helpers/errors.ts:661-665`, and the model fallback loop in `src/agents/model-fallback.ts:848-906` continues to the next configured candidate for recognized failover errors instead of stopping on the first provider. The regression coverage in `src/agents/failover-error.test.ts:78-82` and `src/agents/pi-embedded-helpers.isbillingerrormessage.test.ts:1113-1117` explicitly covers `503`/`service unavailable` as retryable timeout-style failures. This behavior is also already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so I’m closing this as already implemented.
