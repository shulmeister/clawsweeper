## Summary

Keep open. Current main still appends the unscheduled-reminder note whenever reply text looks like a reminder, `successfulCronAdds` is zero, and no enabled cron job exists for the same session. The Pi tool telemetry still increments `successfulCronAdds` only for the native `cron` tool add action, not for successful `exec`/`bash` commands that run `openclaw cron add`. The linked fix attempt in #53024 is closed unmerged, and #47589 targets a related regex false-positive issue rather than this cron scheduling path.

## Best Possible Solution

Keep this issue open and implement #53024 or an equivalent current-main fix: count successful `openclaw cron add` executions through `exec`/`bash` as the same turn-local `successfulCronAdds` signal, or ensure assistant-created reminders use the native `cron.add` tool consistently. Add regression coverage for successful exec-based cron add, failed exec-based cron add, and final auto-reply/Telegram text that confirms a reminder without appending the false note.

## Risks / Open Questions

- A broad suppression based on any existing cron job would hide real unbacked reminder promises; the fix should record a successful scheduler action in the current turn or route assistant-created reminders through the native cron tool path.
