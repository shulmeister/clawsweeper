## Summary

Keep open. Current `main` still accepts Slack bot-authored room messages whenever `allowBots` is enabled, and there is no `requireOwnerPresent` config surface or owner-membership gate in the inbound authorization path.

## Evidence

- **Bot messages pass auth when allowBots=true:** `authorizeSlackInboundMessage` drops bot messages only when `allowBots` is false, then returns `senderId` and `allowFromLower` without any owner-membership check for channel traffic.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:178](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/slack/src/monitor/message-handler/prepare.ts#L178)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.ts | sed -n '168,253p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Owner allowlist only gates control commands, not normal routing:** For room messages, `channelUserAuthorized` is the routing gate; if `channelConfig.users` is unset it defaults to allow. `ownerAuthorized` is computed separately and only fed into `resolveControlCommandGate` for control-command blocking.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:356](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/slack/src/monitor/message-handler/prepare.ts#L356)
