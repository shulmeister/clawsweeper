## Summary

Keep open. Current main has optional sandbox-browser isolation when sandboxing is explicitly enabled, but the normal browser tool path still defaults to the host browser and omitted profile resolves to the shared configured `openclaw` profile. That does not fully implement the issue’s requested automatic per-agent Chrome process/profile isolation.

## Evidence

- **Browser tool defaults to host unless a sandbox bridge is provided:** `createBrowserTool` sets `targetDefault` to `host` when there is no `sandboxBridgeUrl`, reads `profile` only from tool params, and uses that profile/baseUrl for calls; it does not derive a per-agent browser profile from the session or agent id.
  - file: [extensions/browser/src/browser-tool.ts:419](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/browser/src/browser-tool.ts#L419)

  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Default browser profiles are global config profiles:** Browser config always ensures a single `openclaw` managed profile and a single `user` existing-session profile, then chooses `openclaw` as the default when no explicit default exists. There is no per-agent default profile expansion here.
  - file: [extensions/browser/src/browser/config.ts:301](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/extensions/browser/src/browser/config.ts#L301)
