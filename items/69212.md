## Summary

Keep open. Current main does not implement this PR's central behavior: OpenAI-completions reasoning replay still stores only the reasoning field name as `thinkingSignature` and does not capture/replay `encrypted_content`, and the HTTP `openai-responses` / `openai-codex-responses` path still has no `previous_response_id` continuation planning. Existing `previous_response_id` support is limited to the WebSocket path, so it does not supersede this PR.

## Best Possible Solution

Keep the PR open. The best maintainer path is to rebase it onto current main, integrate `encrypted_content` capture/replay into the current OpenAI-completions reasoning model, add HTTP Responses/Codex `previous_response_id` continuation with route capability gating, and add focused tests for encrypted-only/multi-chunk replay plus final outbound HTTP payload shape.
