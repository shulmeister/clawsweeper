## Summary

Keep open. Current main still does not implement the PR's central behavior: an isolated cron run with `finalRunResult.meta?.error` but no `isError` payload is still classified from `hasFatalErrorPayload` alone, so it can report `status: "ok"`. The related fallback-advancement issue #67118 is complementary, not a superseding fix for this run-status bug.

## Best Possible Solution

Keep the PR open or rework it into a current-main patch. The best implementation is to make the cron payload outcome expose a combined fatal flag for payload-level errors plus run-level `meta.error`, format string/object run-level errors clearly, use that combined flag in `finalizeCronRun()` including the delivery-result override gate, and add focused regression tests for meta.error-only empty/non-error payloads plus the existing payload-error precedence case.
