## Summary

Keep this issue open. The linked discussion and comment say the OpenAI SDK timeout was fixed, but current main still does not pass a `timeout` option to the OpenAI/Azure SDK clients or per-request calls for the OpenAI-compatible streaming transports, while the pinned OpenAI SDK still defaults to 600000 ms when no timeout is supplied. The secondary exec-timeout note appears to have been aligned around `timeoutSec`, but the issue title's central OpenAI SDK timeout bug remains unresolved.

## Best Possible Solution

Keep the issue open and implement the narrow timeout propagation in the owner code path: derive the effective HTTP timeout from the resolved agent run timeout, or introduce an explicit `agents.defaults.llm.httpTimeoutSeconds`/provider transport timeout if maintainers prefer a separate knob, then pass it as `timeout` to the `OpenAI` and `AzureOpenAI` client constructors or per-request options for `openai-responses`, `openai-completions`, and `azure-openai-responses`. Add focused regression tests that mock the SDK client/options and prove a configured timeout above 600 seconds reaches the SDK for local OpenAI-compatible providers.
