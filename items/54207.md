## Summary

Keep this issue open. Current main still traces the reported Docker Desktop failure: the containerized gateway resolves sandbox workspaces under its own `/home/node/.openclaw` state path, then passes that container-visible path directly to the host Docker daemon as a bind source. The newer docs warn users about DooD host-path parity, but the Docker setup/runtime does not translate `OPENCLAW_CONFIG_DIR` to host-visible sandbox paths or configure a default parity mount for the sandbox root.

## Best Possible Solution

Keep the issue open and fix the Docker gateway sandbox path contract. The likely maintainer path is to translate gateway-visible state/workspace paths back to the host `OPENCLAW_CONFIG_DIR`/`OPENCLAW_WORKSPACE_DIR` before `docker create`, or update Docker setup to configure a host-visible `agents.defaults.sandbox.workspaceRoot` with an identical gateway-visible parity mount. Add regression coverage for the Docker gateway sandbox flow so `/home/node/.openclaw/sandboxes/...` is not emitted as a host Docker bind source on Docker Desktop macOS.
