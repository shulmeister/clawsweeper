## Summary

Current `main` still implements Telegram forum topics and DM topics as isolated sessions. The codebase preserves topic/thread metadata on messages and replies, but there is no Telegram `sessionMode: "shared"`-style config or routing path that collapses multiple topics/threads into one main session.

## Evidence

- **Docs still describe isolated topic/thread sessions:** Telegram docs say forum topics append `:topic:<threadId>` to keep topics isolated, and DM messages with `message_thread_id` use thread-aware session keys.
  - file: [docs/channels/telegram.md:257](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/telegram.md#L257)

  - file: [src/config/types.telegram.ts:223](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/config/types.telegram.ts#L223)

## Risks / Open Questions

- There is partial overlap with the request because topic/thread metadata is already surfaced in the inbound context, so future maintainers may think this is done when the shared-session routing piece is still missing.
- Telegram ACP/thread binding features provide alternative exact-topic routing, but they do not satisfy the requested single shared main session across multiple topics/threads.
