## Summary

Keep #31396 open. Current main has documented and tested safeguards for explicit `/model`, `session_status(model=...)`, and retry-bounded live model switching, but the report still does not say which path froze, which provider/model was involved, or whether this was provider connection latency. The related provider connect-timeout PR #41936 is still open/unmerged, so closing as implemented or cannot-reproduce would be premature.

## Best Possible Solution

Keep #31396 open and ask for a concrete reproduction: exact command or prompt, channel/UI, provider/model, current/default model, whether a run was active, OpenClaw version/config, and logs around the freeze. If it reproduces in `/model`, `session_status(model=...)`, or `sessions.patch`, add a targeted live-switch regression. If it is a provider connection hang, handle it through a scoped provider/request timeout design, likely tracking or superseding PR #41936.
