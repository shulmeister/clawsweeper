## Summary

Keep open. Current main still uses the transient in-memory cron active-job set as the sole cron backing-session check, and maintenance still marks stale active cron tasks `lost` when that set does not contain the job. The linked PR #71040 is open follow-up work, not shipped behavior on main.

## Best Possible Solution

Keep the issue open. The best fix is to make cron task lost reconciliation use durable cron execution state instead of transient `activeJobIds` alone, likely by registering a cron-owned `tryRecoverTaskBeforeMarkLost` hook or by wiring task maintenance to a durable cron run/running-state lookup. Add regression coverage for a gateway restart/startup catch-up or long manual cron run where the task ledger must not mark the row `lost` without durable evidence the run vanished.

## Risks / Open Questions

- PR #71040 may partially fix startup catch-up and manual-run false positives if merged, but it does not appear on current main and should be reviewed for whether it fully covers the restart/durable-state concern.
- Docs currently describe cron task reconciliation as runtime-owned active tracking, so the final fix needs a product decision: durable cron-state reconciliation, cron-specific recovery hook, or deliberately skipping cron lost-marking when durable evidence is unavailable.
