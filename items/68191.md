## Summary

Keep open. Current `main` still uses transient in-memory cron active-job state as the sole backing check for cron task reconciliation, and the existing regression test explicitly expects stale cron tasks to be marked `lost` when that runtime state is absent. The linked PR #71040 appears to be ongoing work, not a shipped fix.

## Evidence

- **Cron lost-check still depends only on active-job membership:** `hasBackingSession` special-cases `runtime === "cron"` and returns only `isCronJobActive(jobId)`. `shouldMarkLost` then marks active tasks lost once grace expires and `hasBackingSession` is false.
  - file: [src/tasks/task-registry.maintenance.ts:124](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/tasks/task-registry.maintenance.ts#L124)
  - command: `nl -ba src/tasks/task-registry.maintenance.ts | sed -n '118,175p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Backing cron state is transient in-memory process state:** `activeJobIds` is a process-global `Set<string>` with `mark`, `clear`, and `has` helpers only. There is no durable restore path in this module, so a gateway restart clears the backing signal used by task reconciliation.
  - file: [src/cron/active-jobs.ts:3](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/cron/active-jobs.ts#L3)
  - command: `nl -ba src/cron/active-jobs.ts | sed -n '1,80p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current tests encode the reported false-positive behavior:** `task-registry maintenance issue #60299` includes a test named `marks stale cron tasks lost once the runtime no longer tracks the job as active`, and it expects `status: "lost"` for a cron task whose job is no longer present in the active-job set.
  - file: [src/tasks/task-registry.maintenance.issue-60299.test.ts:133](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/tasks/task-registry.maintenance.issue-60299.test.ts#L133)

That supports that maintainers are treating this as real unresolved work rather than something already fixed on `main`.
