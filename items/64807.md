## Summary

Keep open. Current main still has the reported divergence path: the gateway Dashboard banner is driven from persisted/throttled update-check state, while `openclaw update status` uses a live registry-backed status path. The code also still persists a fresh `lastCheckedAt` and returns when the npm channel lookup produces no version, so an empty effective state can be trusted for the normal gateway interval.

## Best Possible Solution

Keep this issue open and fix it in the gateway update-check owner path. The likely implementation should make Dashboard availability use the same effective registry result as `openclaw update status` when the persisted state is stale or empty, avoid treating an unresolved npm channel lookup as a successful full-interval check, add a near-term retry/backoff for empty registry results, and add regression coverage for stale-cache and fresh-empty-cache cases in `src/infra/update-startup.test.ts`.

## Evidence

This means the gateway can keep serving cached banner state without consulting the registry.
