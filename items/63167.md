## Summary

Keep open. Current `main` still has the misleading QMD availability path: the low-level probe spawns with the configured `cwd` without validating it, and `doctor` still reports every failed QMD probe as a binary-start failure with install/path guidance. The PR remains a focused fix for linked issue #63158, though it likely needs rebasing to the current package/test layout and should address the review note with a typed failure reason rather than string-prefix parsing.

## Best Possible Solution

Keep the PR open and update it against current `main`: validate the QMD probe cwd before spawning, return a structured unavailable reason such as `workspace-cwd` versus `binary`, branch doctor guidance on that reason, and add regression coverage in the current QMD process test location plus `src/commands/doctor-memory-search.test.ts`. The existing memory-core workspace-creation behavior should remain intact.
