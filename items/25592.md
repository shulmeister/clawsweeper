## Summary

Keep #25592 open. Current main has meaningful commentary/final_answer hardening, but the broader requested behavior is not implemented: ordinary unphased assistant text can still enter block-reply buffering, be flushed at tool start, and be used as final reply payload input. The issue discussion also includes recent reproduction reports and a maintainer review explicitly keeping this as the broader canonical bug.

## Best Possible Solution

Keep #25592 open as the canonical product bug until OpenClaw implements or explicitly rejects a channel-safe policy such as final-only channel delivery, configurable enforceFinalTag for channel sessions, or an opt-in replyMode/tool-only explicit-send mode. Preserve the shipped commentary/final_answer hardening, then add source changes and regression tests for ordinary unphased pre-tool/intermediate assistant text in both block-reply delivery and final payload assembly.
