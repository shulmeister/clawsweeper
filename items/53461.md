## Summary

Current main already has the channel-registry pinning and cache-invalidation behavior this PR proposes, with broader regression coverage and shutdown/error-path handling, so the PR is redundant.

## Evidence

- **Channel registry is a first-class pinned runtime surface:** `src/plugins/runtime.ts` now maintains a dedicated tracked channel registry with pin/release/get/require helpers so channel lookups can stay bound to a stable registry across later `setActivePluginRegistry(...)` swaps.
  - file: [src/plugins/runtime.ts:151](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/runtime.ts#L151)
- **Gateway startup and teardown wire the channel pin:** `createGatewayRuntimeState(...)` pins the channel registry on startup and releases it on both normal shutdown and startup error paths, alongside the HTTP-route registry.
  - file: [src/gateway/server-runtime-state.ts:108](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-runtime-state.ts#L108)
- **Channel lookup cache invalidates on registry ref/version changes:** Loaded channel lookup now rebuilds when either the channel-registry version changes or the backing registry object changes, preventing stale cached channel-plugin results after registry replacement.
  - file: [src/channels/plugins/registry-loaded.ts:57](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/channels/plugins/registry-loaded.ts#L57)
- **Regression tests cover pinned-registry survival and cache refresh:** Tests explicitly cover preserving a pinned channel registry across active-registry replacement, re-pin cache invalidation, and outbound adapter resolution after post-boot registry replacement.
  - file: [src/plugins/runtime.channel-pin.test.ts:99](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/runtime.channel-pin.test.ts#L99)
- **Helpers cover the deferred-load fallback case:** Tests verify that when the pinned channel registry is empty, channel resolution falls back to the active registry, matching the deferred/full-load scenario called out in the PR discussion.
  - file: [src/channels/registry.helpers.test.ts:69](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/channels/registry.helpers.test.ts#L69)
- **Shutdown releases the pin only after channel stop iteration:** Gateway shutdown iterates `listChannelPlugins()` before the pinned registries are released in the final cleanup block, addressing the stale-shutdown ordering concern raised on the PR.
  - file: [src/gateway/server-close.ts:137](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-close.ts#L137)
