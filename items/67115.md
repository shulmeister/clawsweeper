## Summary

Keep open. Current main still has the broad SSH/no-display guard before the macOS `open` branch, so macOS with `SSH_*` set and no `DISPLAY`/`WAYLAND_DISPLAY` still resolves to `ssh-no-display`. The PR directly addresses the linked #67088 behavior and is not obsolete or superseded by main.

## Best Possible Solution

Keep #67115 open and treat it as a plausible focused bug fix for #67088. The best path is to land the darwin exception in `resolveBrowserOpenCommand`, ideally with a regression test that stubs `process.platform` as `darwin`, sets `SSH_CLIENT`/`SSH_TTY` or `SSH_CONNECTION`, clears `DISPLAY`/`WAYLAND_DISPLAY`, and verifies the resolver reaches the macOS `open` path instead of returning `ssh-no-display`.

## Evidence
