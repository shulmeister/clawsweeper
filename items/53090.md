## Summary

Current OpenClaw already implements the relevant gateway behavior: shared-secret WebSocket auth is carried in `connect.params.auth.token/password`, non-CLI loopback clients with valid shared-secret auth are treated as local for silent pairing, and `auth=none` does not bypass pairing for generic gateway agents. The current `openclaw devices` CLI also includes a local loopback fallback for pairing list/approve. These gateway code paths are unchanged between `v2026.4.22` and current `main`, so the behavior is already shipped.

## Evidence

- **WS connect auth is in the protocol frame, not custom headers:** `connect.params.auth` accepts `token`, `bootstrapToken`, `deviceToken`, and `password`; the server resolves shared auth from that object.
  - file: [src/gateway/protocol/schema/frames.ts:55](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/protocol/schema/frames.ts#L55)
- **Protocol docs match the frame contract:** Docs state shared-secret gateway auth uses `connect.params.auth.token` or `connect.params.auth.password`, and that `gateway.auth.mode: "none"` skips only shared-secret connect auth.
  - file: [docs/gateway/protocol.md:509](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/gateway/protocol.md#L509)
- **Loopback shared-secret clients are treated as local for pairing:** Current tests cover a non-CLI loopback client with shared-secret auth resolving to `shared_secret_loopback_local`, and silent local pairing is allowed for `not-paired` / role / scope upgrades.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.test.ts:445](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server/ws-connection/handshake-auth-helpers.test.ts#L445)
- **Silent pairing is wired into the live handshake path:** On connect, the server computes `allowSilentLocalPairing`, marks the pairing request `silent`, and auto-approves it when that path applies.
  - file: [src/gateway/server/ws-connection/message-handler.ts:919](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server/ws-connection/message-handler.ts#L919)
- **`auth=none` pairing bypass is intentionally limited:** Current policy skips pairing for `authMode === "none"` only for Control UI operator sessions, not generic gateway agents like Paperclip adapters.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:46](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server/ws-connection/connect-policy.ts#L46)
- **CLI has a local loopback pairing fallback:** `openclaw devices` falls back to local pairing files for list/approve when the error is pairing-related and no explicit `--url` was passed.
  - file: [src/cli/devices-cli.ts:122](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cli/devices-cli.ts#L122)
- **Behavior shipped in latest release:** Comparing `v2026.4.22` (`00bd2cf7`) to current `main` for the relevant gateway/CLI code files produced no code diffs; only docs changed after the release.
  - command: `git diff --name-only 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5..8866544ffefb42ca234c66c337134fb60743d8d0 -- src/cli/devices-cli.ts src/gateway/server/ws-connection/connect-policy.ts src/gateway/server/ws-connection/handshake-auth-helpers.ts src/gateway/server/ws-connection/message-handler.ts src/gateway/client.ts`

## Risks / Open Questions

- This review did not execute Paperclip itself; if Paperclip still omits `connect.params.auth.token` or rotates device identity on every connect, that would remain an external client-side interoperability problem rather than an OpenClaw core regression.
- The clearer protocol/troubleshooting docs on current `main` postdate `v2026.4.22`, so reporter confusion may still have been real on older docs even though the gateway behavior is already present.

## Close Comment

Closing as implemented on current OpenClaw. Codex review checked the current `main` gateway handshake and found that OpenClaw already expects shared-secret WebSocket auth in `connect.params.auth.token` / `connect.params.auth.password` (`src/gateway/protocol/schema/frames.ts:55`, `docs/gateway/protocol.md:509`), not custom WS headers. It also already treats non-CLI loopback clients with valid shared-secret auth as local for pairing (`src/gateway/server/ws-connection/handshake-auth-helpers.test.ts:445`) and auto-approves that silent local pairing path in the live handshake (`src/gateway/server/ws-connection/message-handler.ts:919`). The `auth=none` bypass is intentionally limited to Control UI operator sessions, not generic gateway agents (`src/gateway/server/ws-connection/connect-policy.ts:46`), so seeing pairing still matter for a Paperclip-style agent in `auth=none` is expected. On the CLI side, `openclaw devices` now has a local loopback fallback for list/approve when pairing scope is unavailable (`src/cli/devices-cli.ts:122`). I also compared these gateway code paths against release `v2026.4.22` and found no code diffs, so this behavior is already shipped there.
