## Summary

Keep #42675 open. Current main still catches Discord `fetchUser("@me")` startup failures and continues with an undefined `botUserId`, and the latest release tag has the same behavior. The PR is directionally useful but incomplete and stale against the moved Discord plugin path; the linked security bug #42219 remains the canonical issue to fix, not evidence that this PR is obsolete.

## Best Possible Solution

Keep the PR open or replace it with a maintainer-owned fix for #42219. The complete fix should live in `extensions/discord/src/monitor/provider.startup.ts`, fail startup when `fetchUser("@me")` throws or returns no valid `id`, and add focused tests for thrown fetch failure and missing bot id. Maintainers should also decide whether to remove the extra `botId &&` preflight guard or otherwise keep `requireMention` strict when bot identity is unavailable.

## Risks / Open Questions

- The PR patch targets the old `src/discord/monitor/provider.ts` path, while current main has the Discord implementation under `extensions/discord/src/monitor/provider.startup.ts`, so it needs a rebase before it can land.
- Linked issue #42219 is still open and has a protected `security` label, so the underlying bug should remain under maintainer/security handling until a complete fix and regression test land.
