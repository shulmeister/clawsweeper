## Summary

Keep #10659 open. Current main has useful partial hardening through SecretRefs, host exec env sanitization, redaction settings, and plugin hooks that support the linked user-isolation and credential-vault mitigation paths. It still does not ship the native masked-secrets feature requested here: no `{{secret:NAME}}` command substitution, no `secrets.mask` or `maskedSecrets` config contract, no exact secret mask list for arbitrary exec output, and no generic policy that prevents an agent from reading otherwise-readable env files or variables. The latest maintainer-associated comment also explicitly kept this issue open.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secret or credential-provider path for agent-executed commands and process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue acknowledging `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as useful mitigations that can run today, while treating the native core security boundary as unresolved.
