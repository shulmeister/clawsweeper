## Summary

Keep open for maintainer/contributor follow-up rather than auto-close. Current main still performs exact provider/model allowlist matching, while #43211's unconditional hyphen-stripped expansion has unresolved authorization-regression review comments because hyphenated and non-hyphenated provider IDs can be distinct.

## Best Possible Solution

Do not merge #43211 as-is. Keep it open for maintainer/contributor follow-up: first prove the specific runtime path that legitimately treats a hyphenated provider and stripped provider as the same provider, then implement a narrower alias-aware fix limited to declared provider aliases or a documented migration path, keeping `allowedKeys` and `allowedCatalog` consistent and adding regression tests for both the intended alias case and the distinct-provider authorization case.
