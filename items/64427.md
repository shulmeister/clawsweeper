## Summary

Current main still appears to miss the reported Slack 1:1 DM history hydration. Slack `dmHistoryLimit` is accepted and documented, but the Slack monitor computes and carries only the group/channel `historyLimit`; inbound Slack history context is then gated on `isRoomish`, which excludes `channel_type === "im"`. Existing Slack API history reads cover message-action readback, preview edit readback, or thread replies, not top-level DM session backfill. The related #7359 report is adjacent but does not supersede this narrower `dmHistoryLimit` implementation gap.

## Best Possible Solution

Keep this issue open and implement the behavior in `extensions/slack`: resolve per-DM history limits from `channels.slack.dms.<user>.historyLimit` and `channels.slack.dmHistoryLimit`, fetch recent top-level DM messages with Slack `conversations.history` when a new DM session starts, format them into the inbound context without rehydrating every turn, and add focused Slack monitor tests for `channel_type: "im"` plus docs updates if the final behavior differs from the current documented contract.

## Risks / Open Questions

- A fix should not confuse Pi transcript truncation (`getHistoryLimitFromSessionKey`) with Slack API backfill; transcript limiting alone does not restore context for a fresh Slack DM session.
- The Slack implementation should hydrate only when a DM session is new, exclude the current inbound event, handle pagination/scopes/errors, and apply the existing sender-visibility/security rules deliberately.
