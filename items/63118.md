## Summary

Keep open. Current main has useful partial session-stop plumbing (`sessions.abort`, `/sessions/:sessionKey/kill`, chat `/stop`/Stop button), but #63118 asks for user-facing session lifecycle management that is still missing from the documented CLI and Sessions UI: `openclaw sessions` is list-plus-cleanup only, the Sessions page exposes delete/metadata controls but no stop/pause/resume actions, and no per-session resource-limit control surface is present there.

## Best Possible Solution

Keep #63118 open but narrow implementation into phases: first add a documented `openclaw sessions stop` or `abort` command and Sessions UI stop action backed by the existing `sessions.abort`/`chat.abort` path; then decide whether pause/resume and per-session resource limits need new gateway protocol fields, persistence, and policy docs. Coordinate with the related open session-management gaps for REST endpoints (#20934) and CLI delete/clear commands (#35682) instead of closing this broad lifecycle request as already implemented.
