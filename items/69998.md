## Summary

Keep #69998 open. Current main still returns the updated `skills.update` config object directly, so `apiKey` and secret-like `env` values can be echoed in the RPC response. The linked bug #66769 remains open, and the related PRs #66928 and #66986 were closed unmerged rather than superseding this PR.

## Best Possible Solution

Keep the PR open and ask for the redaction fix to land with regression coverage. The safest maintainer direction is to either return only non-sensitive confirmation metadata from `skills.update`, or redact the response and add sentinel-restore/ignore handling for `apiKey` and secret-like `env` fields before writes, mirroring the config endpoints' round-trip contract.

## Risks / Open Questions

- The PR as written likely needs one more design decision: if `skills.update` returns `__OPENCLAW_REDACTED__`, the write side should avoid persisting that sentinel on future round-trips or should stop returning the config object entirely.
- Until some version of this fix lands, `skills.update` can still leak plaintext skill credentials in gateway RPC responses on current main.
