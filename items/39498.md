## Summary

Keep #39498 open. Current main has partial ByteString hardening for model/provider API-key paths, but the shared `SecretInput` boundary remains trim-only and at least one header-bound plugin path can still pass a non-Latin1 secret into `Headers`. The PR is also not safe to merge as written because globally switching `normalizeSecretInputString` would mutate valid non-header secrets and alter presence checks.

## Best Possible Solution

Keep the PR open, but do not merge the shared-normalizer change as-is. The best fix is a scoped header-bound secret normalization path: use `normalizeSecretInput` or a configurable normalizer only where values are destined for HTTP headers, add regression coverage for those callsites such as Comfy `X-API-Key`, and preserve trim-only semantics for generic `SecretInput`, non-header passwords/tokens, and presence detection.
