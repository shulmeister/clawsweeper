## Summary

Keep open. The issue is a recent, concrete UX request for user-visible transient/retrying feedback during rate-limit or timeout recovery, and current main evidence I could inspect shows fallback/cooldown handling and structured observability, but not the requested in-progress yellow/retrying UI state or countdown/progress feedback. Related recent fallback reports also show user-facing confusion around rate-limit fallback remains plausible.

## Evidence

- **Issue request is concrete and recent:** Issue #60470 asks to distinguish fatal errors from transient/retrying states, including a yellow warning such as `System busy, retrying... (1/3)`, and was updated with concrete log examples on 2026-04-25. It is not older than 60 days and has no protected-label/maintainer-author auto-close reason.
  - command: `GitHub issue page/context inspection`
- **Current docs describe fallback mechanics, not the requested live retry UI:** The model failover docs say OpenClaw advances fallback on rate limits/timeouts and records per-attempt details for logs and cooldown messaging, but the documented user-facing message is tied to `FallbackSummaryError` when every candidate fails, not an in-progress retrying/progress state while recovery is still underway.
