## Summary

Keep this issue open. Current main still does not provide the requested dedicated Slack user DM flow: setup/docs expose only bot event subscriptions, the monitor is bootstrapped around bot/app tokens, monitor context and reply delivery remain bot-token centered, and `userTokenReadOnly: false` only enables user-token writes as a fallback when no bot token is available. The earlier Codex review comment is still accurate against the current main SHA.

## Best Possible Solution

Implement this in the bundled Slack plugin: add explicit `user_events` setup/docs for Slack events on behalf of users, carry the configured user token and user identity through monitor context, detect eligible user-level `message.im` events separately from bot-level events, route those replies through the xoxp token when `userTokenReadOnly: false`, and add focused tests for inbound DM routing, token selection, self-loop prevention, setup manifest output, and docs.
