## Summary

Keep open. Current main still has the semantics this issue asks to change: `workspaceAccess: "rw"` resolves `/workspace` to the agent workspace, `none`/`ro` use the sandbox workspace but keep it read-only for mutations, and split workspaces can still expose the agent workspace at `/agent`. The issue is concrete, recent, unprotected, and not superseded by the narrower related reports.

## Best Possible Solution

Keep the issue open for maintainer product direction and implementation. A complete fix would change sandbox layout so `/workspace` consistently represents the per-scope sandbox workspace, remove implicit `/agent` exposure from normal sandbox modes, make write/edit/apply_patch availability follow the resolved mount writability, update Docker/SSH/OpenShell/browser mount paths together, and revise the sandboxing docs plus mount/fs bridge tests.
