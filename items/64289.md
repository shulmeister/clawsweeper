## Summary

Current main now loads the relevant gateway startup plugin set before WebSocket handlers are attached/listening, includes configured channels, startup sidecars, and selected/default memory plugins in that startup set, and reuses the active gateway-bindable registry for runtime/session tool resolution instead of re-registering plugins on first interaction after idle.

## Best Possible Solution

Close this as implemented on main. Keep the current startup-scoped plugin registry behavior: gateway startup should prepare the configured channel, sidecar, and memory plugin set before webchat sockets attach, and session/runtime paths should reuse the active gateway-bindable registry. Any new post-idle flash on a current build should be triaged as a separate reconnect regression with current-version gateway logs.

## Evidence

- **Gateway startup loads plugins before webchat listen:** `prepareGatewayPluginBootstrap` runs during startup, then deferred configured channel plugins are reloaded and `runtimeState.gatewayMethods` is updated before `attachGatewayWsHandlers` and `startListening` are called. This means webchat clients connect after the startup plugin registry is prepared, not before a later first-session interaction.
