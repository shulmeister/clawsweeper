## Summary

Keep open. Current `main` makes a literal hidden cron job less likely, but it does not provide high-confidence proof that this exact autonomous image/`exec` loop is fixed; the report remains a plausible unresolved runtime bug or misrouted automation path.

## Evidence

- **Cron definitions are persisted, not ephemeral:** The user-facing docs say cron job definitions live in `~/.openclaw/cron/jobs.json` and runtime state lives in `~/.openclaw/cron/jobs-state.json`, so an empty store argues against a persisted 'ghost cron' definition but does not explain other autonomous activity.
  - file: [docs/automation/cron-jobs.md:35](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/automation/cron-jobs.md#L35)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '20,50p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **`cron list` reads the loaded cron store:** `status()` reports `jobs: state.store?.jobs.length` and `list()` filters/sorts `state.store?.jobs`, so current main does not synthesize hidden cron jobs outside the persisted store for normal listing/status paths.
  - file: [src/cron/service/ops.ts:148](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/cron/service/ops.ts#L148)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '146,170p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Cron executions are tracked as tasks:** The docs say every cron execution creates a task record, which means missing cron definitions still would not rule out a different detached/background runtime; the report did not include task evidence.
  - file: [docs/automation/tasks.md:302](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/automation/tasks.md#L302)
  - command: `nl -ba docs/automation/tasks.md | sed -n '280,330p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Always-on guard is only for unknown tool names:** Current main has an always-on unknown-tool guard, but the code explicitly scopes it to hallucinated or removed tool names that are not registered in the run. The reported log snippet shows a valid tool path (`exec`) invoking a bad command (`nano-banana`), which is not the same failure mode.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:351](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/pi-embedded-runner/run/attempt.ts#L351)

That leaves no high-confidence proof on current main that a repeated `exec`/image no-progress loop like the report is prevented out of the box.
