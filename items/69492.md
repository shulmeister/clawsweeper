## Summary

Keep open. Current main has useful transcript-hygiene work, but it does not implement the issue's core event-audience primitive: system events still have only text/routing/trust metadata, exec completion still enqueues without consulting trigger/audience intent, and heartbeat user relay is still controlled by delivery target plus showAlerts rather than per-event classification.

## Best Possible Solution

Keep this as the canonical core design issue. The best fix is an additive `SystemEventAudience`/`completionAudience` contract on the core system-event path, derived from exec trigger where safe, honored by heartbeat relay and transcript/history projections, and covered by focused tests for exec, cron, heartbeat, chat.history, sessions.get, and sessions history HTTP behavior.

## Risks / Open Questions

- Defaulting too many events to internal could hide expected user-facing exec completion notifications; a fix needs explicit compatibility defaults and regression tests.
- This is core event, heartbeat, transcript, and plugin-SDK plumbing rather than a ClawHub-only plugin idea; closing it externally would leave the missing core API unresolved.
