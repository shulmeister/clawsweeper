## Summary

Keep #40540 open. Current main still runs the npm global self-update from the active OpenClaw CLI process before any gateway/service handoff, and the current failure-hint/test coverage does not cover the reported Windows `EBUSY` rename failure. The related #40724 was closed unmerged and only proposed guidance, not a self-update lock fix; the provided #44614 ClawSweeper context says the detached-helper work remains open and not implemented on main.

## Best Possible Solution

Keep the bug open until OpenClaw ships a Windows-safe package self-update path. The likely fix belongs in the update CLI/package-update flow: stop or hand off the running gateway/CLI before npm renames the global package directory, or run the npm update from a detached helper/temp install path, then add regression coverage for `EBUSY` rename output and document the recovery path.

## Evidence

There is no detached updater or pre-update process handoff in this path.
