## Summary

Keep open. Current `main` still routes Feishu outbound sends through `resolveReceiveIdType`, which falls back to `"user_id"` for otherwise-unrecognized targets and is forwarded directly into `client.im.message.create` without any guard. The issue is not implemented on `main`, and the existing tests still encode that fallback behavior.
