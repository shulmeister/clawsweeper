## Summary

Keep #39554 open. Current main still has the reported Google Chat API-boundary bug: `sendGoogleChatMessage` trusts any supplied thread string and several outbound paths can still forward raw `threadId`/`replyToId` values into `body.thread.name`. The exact validation PR #46519 was closed unmerged, while #67055/#42510 cover the narrower `replyToMode: "off"` propagation behavior and #69422 covers broader streamed-reply metadata, not a shipped provider-local thread-resource validation fix.

## Best Possible Solution

Keep #39554 open until Google Chat validates or normalizes thread identifiers at the plugin boundary. The likely fix is a shared Google Chat thread helper used by `sendGoogleChatMessage` callers: preserve valid `spaces/{space}/threads/{thread}` values, omit raw message/session IDs for flat sends, and add focused regression tests for API sends, attached-results text/media, message actions, monitor reply delivery, and the `replyToMode: "off"` path without breaking explicit reply tags. Keep #67055/#42510 linked for the narrower off-mode behavior and #69422 linked for broader streamed-reply metadata work.

## Evidence

- **Upstream Google Chat thread contract:** Google Chat's Message resource defines message names as `spaces/{space}/messages/{message}` and Thread.name as a thread resource name with example `spaces/{space}/threads/{thread}`; message creation uses `thread.name`/`threadKey` and `messageReplyOption` for threaded sends. 
  - command: `web.open Google Chat REST reference for spaces.messages and spaces.messages.create`
- **API helper still trusts thread input:** `sendGoogleChatMessage` accepts `thread?: string`; when it is truthy, the helper writes `body.thread = { name: thread }` and adds `messageReplyOption=REPLY_MESSAGE_FALLBACK_TO_NEW_THREAD` without validating, normalizing, or dropping non-Google-Chat thread resource names.
