## Summary

Keep #39554 open. Current main still forwards arbitrary Google Chat thread strings into `thread.name` without validating the required resource format, and multiple outbound paths can still supply raw `threadId`, `replyTo`, or `replyToId` values. The exact validation PR #46519 was closed unmerged, while #67055 and #69422 are related but do not supersede this provider-boundary validation bug.

## Best Possible Solution

Keep #39554 open until the Google Chat plugin validates or normalizes thread identifiers at the provider boundary. The best path is a shared helper used by every `sendGoogleChatMessage` caller: preserve valid `spaces/{space}/threads/{thread}` resources, omit invalid raw message/session ids for flat sends, respect `replyToMode: "off"`, and add regression tests for API sends, message actions, attached results text/media, monitor reply delivery, and streamed/block reply paths.
