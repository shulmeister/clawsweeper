## Summary

Keep open. Current main still forwards unvalidated Google Chat thread values into send paths, and the exact validation PR was closed unmerged while the related replyToMode PR is open and narrower. The tests cover valid thread sends and no-thread sends, but they do not prove invalid thread names are dropped before the API call.

## Best Possible Solution

Keep this issue open until Google Chat has provider-local thread resource validation/normalization on every outbound path. The likely fix belongs in `extensions/googlechat` near `sendGoogleChatMessage` or the Google Chat threading/outbound adapter: preserve valid `spaces/{space}/threads/{thread}` values, drop invalid raw message/session ids for flat sends, and add focused regression tests for API sends, message actions, attached results, and direct reply delivery.

## Evidence
