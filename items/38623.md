## Summary

Keep open. Current `main` has partial video support through the media-understanding pipeline, including a bundled Qwen video-description provider, but it does not implement the issue’s requested direct model-level video input support or `input: ["text", "image", "video"]` catalog/config path.

## Evidence

- **Configured model schema still rejects video input:** The core config schema only allows model `input` values `"text"` and `"image"`, and the runtime model definition type matches that restriction.
  - file: [src/config/zod-schema.core.ts:307](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/config/zod-schema.core.ts#L307)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Model catalog types exclude video:** Model catalog types still define `ModelInputType` as `"text" | "image" | "document"`, so shipped/runtime catalog entries do not have first-class video input capability.
  - file: [src/agents/model-catalog.types.ts:1](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/agents/model-catalog.types.ts#L1)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Configured provider catalog normalization drops video:** Configured provider catalog entries are normalized to `"text" | "image" | "document"` only, so even a user-supplied `input: ["video"]` model config would not become a first-class catalog capability.
  - file: [src/plugin-sdk/provider-catalog-shared.ts:20](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/plugin-sdk/provider-catalog-shared.ts#L20)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Inbound multimodal replay path handles images, not videos:** The OpenAI WS message conversion path checks only `supportsImageInput()` and converts only image parts into model input blocks; there is no equivalent video-content handling in this path.
  - file: [src/agents/openai-ws-message-conversion.ts:47](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/agents/openai-ws-message-conversion.ts#L47)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Main does already have a separate video-understanding feature:** Docs and plugin code show current `main` supports inbound video understanding through the media-understanding pipeline, with bundled provider fallback order `Google → Qwen → Moonshot`, and the Qwen plugin registers `describeVideo` support. That makes the report coherent but only partially addressed, not implemented as requested.
  - file: [docs/nodes/media-understanding.md:145](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/docs/nodes/media-understanding.md#L145)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Qwen bundled plugin provides video understanding outside the model catalog:** The Qwen plugin exposes media-understanding capabilities `['image', 'video']` and a `describeVideo` implementation using `qwen-vl-max-latest`, separate from the chat-model catalog.
  - file: [extensions/qwen/media-understanding-provider.ts:95](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/extensions/qwen/media-understanding-provider.ts#L95)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Bundled Qwen chat catalog still marks models as text/image only:** The shipped Qwen model catalog marks `qwen3.5-plus` and related chat refs as `input: ['text', 'image']`, with no first-class `video` catalog entry.
  - file: [extensions/qwen/models.ts:27](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/extensions/qwen/models.ts#L27)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)

## Risks / Open Questions

- Part of the issue body is now outdated: current main already includes Qwen video understanding in the separate media-understanding path, so future maintainer triage should distinguish 'video summaries before reply' from 'direct video-capable chat model input'.
- Closing as implemented would be too aggressive because the requested schema/runtime contract change for model `input: ["video"]` is still absent in current main.
