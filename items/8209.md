## Summary

Keep #8209 open. Current main has internal cron started/finished events, completion-only webhook delivery, completion-only run logs, failure destinations, and a gateway_start plugin context that can inspect cron jobs, but it still lacks the requested native start+completion webhook/telemetry/plugin lifecycle surface for external Mission Control-style integrations. The linked PR #8855 was closed unmerged and does not supersede or ship the requested behavior.

## Best Possible Solution

Keep #8209 open for explicit maintainer product/API direction. The conservative implementation path is a generic, typed cron lifecycle event surface, preferably a plugin hook or subscription API that emits start and finish events for both agentTurn and systemEvent jobs. If core webhooks are chosen instead, keep them generic, schema-validated, SSRF-guarded, documented, and covered by focused tests for start, success, failure, timeout, disabled jobs, and misconfigured destinations.

## Risks / Open Questions

- Closing this now would hide a real remaining integration gap: external systems still cannot receive documented event-driven cron start and completion records without polling, completion-only webhooks, or agent/plugin workarounds.
- The eventual implementation needs maintainer product/API direction because a generic plugin lifecycle hook or subscription API is likely safer than a one-off Mission Control-specific core feature.
