## Summary

Keep open: current `main` does not implement timezone-aware session resets or an `idleGuard` AND mode, and the submitted PR still has unrebutted correctness findings from review.

## Evidence

- **Current main still uses host-local daily reset and OR logic:** `resolveDailyResetAtMs()` still uses `new Date(now).setHours(...)` with the gateway host timezone, and `evaluateSessionFreshness()` still expires sessions when `staleDaily || staleIdle` is true. That is the pre-PR behavior, not timezone-aware reset or daily+idle AND mode.
  - file: [src/config/sessions/reset-policy.ts:23](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/config/sessions/reset-policy.ts#L23)
  - command: `nl -ba src/config/sessions/reset-policy.ts | sed -n '20,92p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Config/schema surface on main does not expose the requested fields:** `SessionResetConfig` only has `mode`, `atHour`, and `idleMinutes`, and the session Zod schema likewise omits `timezone` and `idleGuard`. Main therefore does not currently accept the feature this PR proposes.
  - file: [src/config/types.base.ts:125](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/config/types.base.ts#L125)
  - command: `nl -ba src/config/types.base.ts | sed -n '120,160p'; nl -ba src/config/zod-schema.session.ts | sed -n '17,23p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Session reset call sites do not thread through a user timezone on main:** The session resolution paths in reply handling, command resolution, and isolated cron sessions call `resolveSessionResetPolicy(...)` without a user-timezone parameter, so there is no implemented fallback from `agents.defaults.userTimezone` into session reset behavior.
  - file: [src/auto-reply/reply/session.ts:422](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/auto-reply/reply/session.ts#L422)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '422,426p'; nl -ba src/agents/command/session.ts | sed -n '227,236p'; nl -ba src/cron/isolated-agent/session.ts | sed -n '31,37p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **User-facing docs still describe the old behavior:** The docs still say daily reset happens at 4:00 AM local time on the gateway host, and that when daily and idle resets are both configured, whichever expires first wins. That matches current main and contradicts the PR's proposed timezone-aware reset plus optional AND guard.
  - file: [docs/concepts/session.md:61](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/concepts/session.md#L61)

The provided timeline shows no follow-up commit after those reviews.

## Risks / Open Questions

- Users still cannot configure session reset boundaries in a timezone different from the gateway host timezone.
- Users who want daily reset plus an inactivity guard still only have the existing whichever-expires-first behavior on `main`.
- The open PR is not merge-ready as submitted because the review-identified runtime and correctness issues remain unaddressed in the provided discussion.
