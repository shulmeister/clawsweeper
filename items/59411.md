## Summary

Keep PR #59411 open. Current main does not implement `session.reset.timezone` or `idleGuard`; daily reset still uses the gateway host local timezone and daily+idle still uses OR semantics. The AND-mode part overlaps with open PR #71400, but the timezone-aware daily boundary request remains a meaningful unsuperseded capability. The PR should not merge as-is because review comments identify real correctness blockers, but that is not a cleanup close reason.

## Best Possible Solution

Keep the PR open or ask for rework rather than closing it. The best maintainer path is to let #71400 carry the daily+idle AND semantics, then either split or update #59411 so the timezone-aware daily reset lands with strict IANA validation, robust offset/DST tests including UTC+13/+14 zones, config type/schema/help/docs updates, and all reset call sites wired consistently.
