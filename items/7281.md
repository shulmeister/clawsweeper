## Summary

Current main still does not implement configurable fallback delivery for hook mappings. The hook config/schema, mapping normalization, hook-to-cron bridge, cron delivery model, and direct delivery dispatch all carry only one delivery channel/target; the existing fallback behavior is limited to resolving `last`/session targets or separate failure notifications, not retrying an alternate channel after primary hook delivery fails.

## Best Possible Solution

Keep #7281 open until maintainers either implement the feature or make an explicit product decision not to. A proper implementation would add typed `fallbackChannel`/`fallbackChannels` support to hook mapping config and validation, propagate it through hook mapping resolution and `dispatchAgentHook` into cron/direct delivery, retry ordered alternate channels only after primary delivery failure, document the behavior at https://docs.openclaw.ai/gateway/configuration-reference, and add focused tests that distinguish it from `channel: "last"`, best-effort delivery, and cron failure notifications.
