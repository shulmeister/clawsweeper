## Summary

Keep #7281 open. Current main still has only a single hook delivery target (`channel`/`to`) from config through mapping resolution, HTTP dispatch, cron job creation, and direct outbound delivery. The proposed `fallbackChannel`/`fallbackChannels` fields are not accepted by the strict hook mapping schema and do not reach the delivery path. The prior steipete/Codex review in the issue already kept this open, and current main does not add contradictory support; the related closed Slack issue #54029 is a similar delivery-failure surface, not a canonical fallback-channel implementation.

## Best Possible Solution

Keep #7281 open for maintainer/product handling. The best implementation is typed `fallbackChannel` or ordered `fallbackChannels` support on hook mappings, carried through schema/help/docs, transform outputs, dispatch payloads, cron delivery planning, and outbound delivery. Fallbacks should be attempted only after primary delivery failure, with tests for primary-fails/fallback-succeeds, all-targets-failed reporting, and separation from `channel="last"`, `bestEffort`, and `failureDestination`.
