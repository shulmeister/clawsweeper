## Summary

Current main still lacks configurable `fallbackChannel`/`fallbackChannels` support for hook mappings. Hook config, schema validation, mapping normalization, request replay scope, dispatch into cron delivery, and direct outbound delivery all carry one primary `channel`/`to` target. Existing fallback helpers only choose an initial channel when `channel="last"` or tool context is missing; they do not retry an alternate target after primary send failure.

## Best Possible Solution

Keep #7281 open for maintainer/product handling. A complete fix should add typed `fallbackChannel` or `fallbackChannels` support to hook mapping config and validation, propagate ordered fallback targets through hook normalization, replay keys, dispatch payloads, cron delivery, target resolution, and direct outbound delivery, retry alternates only after primary delivery failure, document how this differs from `channel="last"`, `bestEffortDeliver`, and `failureDestination`, and add focused tests for primary failure followed by fallback success.
