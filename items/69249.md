## Summary

Keep #69249 open. Current main has partial restart-aborted main-session recovery, but that path resumes internally with `deliver: false` and does not post a Discord-visible recovery/apology/resume message. The Discord inbound path also still returns silently on lifecycle aborts, while the visible restart-continuation machinery is limited to explicit restart-sentinel payloads and is not used for this generic interrupted-turn recovery gap.

## Best Possible Solution

Keep the issue open and implement channel-visible restart interruption recovery. The likely path is to extend `src/agents/main-session-restart-recovery.ts` to preserve or resolve the interrupted session's delivery context, then either enqueue a bounded Discord-visible recovery notice via shared outbound/session-delivery infrastructure or safely resume with channel delivery enabled. Add a targeted Discord/main-session regression proving a gateway restart produces either a visible recovery message or a resumed final reply without duplicate delivery.

- A fix needs duplicate-notice protection when the final Discord reply was delivered just before shutdown or when a restart sentinel already handled the session.
- Recovery should preserve Discord account, channel/thread, and reply-target metadata so the notice or resumed reply lands in the original conversation.
