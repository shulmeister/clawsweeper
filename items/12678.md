## Summary

Keep open. Current main has narrower controls: global/per-agent/per-channel tool allow/deny, Gateway HTTP default-deny for selected dangerous tool names, and plugin hooks that can block or require approval at call time. It still does not implement the requested manifest-declared capability permissions with default-deny grants/revocations and immutable high-risk invocation audit semantics.

## Best Possible Solution

Continue tracking this as a core/plugin-API security hardening item. A suitable implementation would add a manifest-declared capability schema, enforce default-deny grants before risky tools are exposed or invoked, add grant/revoke UI/CLI flows, and record high-risk capability invocations in a durable audit log. The linked @vainplex/openclaw-governance plugin remains useful prior art for call-time policy, but it does not replace the missing API-layer authorization model.
