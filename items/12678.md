## Summary

Keep open. Current main has useful but narrower controls: tool allow/deny policies, per-sender/group restrictions, plugin before_tool_call hooks, and Gateway HTTP default-deny for selected dangerous tool names. It still does not implement the requested manifest-declared capability permissions with default-deny grants/revocations, pre-exposure enforcement, and immutable high-risk invocation audit semantics.

## Best Possible Solution

Continue tracking this as a core/plugin-API security hardening item. A suitable implementation should add a manifest-declared capability schema, enforce default-deny grants before risky tools are exposed or invoked, provide grant/revoke UI and CLI flows, and record high-risk capability invocations in a durable audit log. The linked @vainplex/openclaw-governance plugin is valuable prior art for call-time policy, but it does not replace the missing API-layer authorization model.
