## Summary

Keep #12678 open. Current main has adjacent hardening, including tool profiles/allowlists, group/sender policy, Gateway HTTP hard-denies, sandbox and exec approval controls, and plugin before_tool_call governance, but it still does not implement the requested manifest-declared, persisted, default-deny capability authorization model for skills/tools.

## Best Possible Solution

Keep #12678 open as a core/plugin-API hardening item. The best path is to design a manifest-declared capability schema, persist grant/revoke state, expose CLI/UI consent management, enforce default-deny before high-risk tools are listed or invoked across agent/Gateway/HTTP/MCP/channel surfaces, and write durable audit records for high-risk capability use. Treat @vainplex/openclaw-governance as useful before_tool_call prior art, not as a replacement for API-layer authorization.
