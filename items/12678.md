## Summary

Keep #12678 open. Current main has tool-name policies, per-provider/per-agent/per-sender filtering, owner-only gates, plugin before_tool_call blocking/approval hooks, exec approvals, diagnostics, and skill env hardening, but it still does not implement the requested manifest-declared capability permission model with persisted grants/revocations, default-deny before risky tools are exposed, or durable high-risk capability audit logging.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. The best path is to design a manifest-declared capability schema, persist grant/revoke state with CLI/UI management, default-deny high-risk capabilities before tools are listed or invoked across agent/HTTP/MCP surfaces, and record durable high-risk capability audit entries. Treat @vainplex/openclaw-governance as useful before_tool_call governance prior art, not a replacement for API-layer capability authorization.
