## Summary

Keep open. Current `main` does not ship `heartbeat.maxCostPerRun`, and the PR’s proposed guard is not proven equivalent to current runtime behavior because heartbeat runs still price against the effective per-agent/session model and usually include existing session history unless `isolatedSession` is enabled.

## Risks / Open Questions

- The feature request is not implemented on `main`, so closing as `implemented_on_main` would be incorrect.
- Current docs and types show heartbeat cost control is still indirect (`isolatedSession`, cheaper model), so the underlying problem remains relevant.
- The PR as described still has correctness risk: normal heartbeats can include prior session history unless `isolatedSession` is enabled, so a cap based only on prompt-body length would under-estimate real spend.
