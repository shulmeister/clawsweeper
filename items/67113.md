## Summary

Keep #67113 open. Current main has partial improvements: QMD embeds use `memory.qmd.update.embedTimeoutMs`, `embedInterval` is documented, and `embedInterval <= 0` suppresses the dedicated periodic embed timer. But the reported ARM/WSL failure is not fully resolved because startup still calls `runUpdate("boot", true)`, forced updates still run `qmd embed`, and tests explicitly assert that search-mode QMD still runs embeds. The linked #59234 is related but narrower: it tracks the `qmd status` vector probe in `searchMode: "search"`, not the boot/forced `qmd embed` timeout loop reported here.

## Best Possible Solution

Keep #67113 open and treat it as a real QMD-on-CPU/ARM follow-up. The best path is a maintainer decision in `extensions/memory-core/src/memory/qmd-manager.ts`: either add an explicit BM25-only/no-embed mode, make `embedInterval: "0"` also suppress boot/forced embeds unless explicitly forced by a user command, or document the current behavior and provide a supported ARM workaround. Coordinate only the `qmd status` vector-probe portion with #59234; do not close this issue until the boot/forced `qmd embed` behavior is resolved or intentionally rejected with docs.
