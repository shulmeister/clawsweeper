## Summary

Keep #69242 open. Current main has the Linux child `oom_score_adj` mitigation from #70419, but that shipped fix is scoped to gateway OOM victim selection and explicitly does not resolve this issue's user-facing SIGKILL observability/resource-diagnostics request. The current exec path still collapses an unforced `SIGKILL` into a generic `Command aborted by signal SIGKILL` result without enough cause metadata to distinguish OS kill, supervisor kill, timeout, cancellation, RSS pressure, or a log/run pointer for broad search failures.

## Best Possible Solution

Keep this issue open, preferably as a Linux-specific exec SIGKILL diagnostics/resource-governance bug linked to #70419 and #72240. The best fix is in the exec/supervisor path: preserve and expose termination reason, whether OpenClaw initiated the kill, timeout/no-output status, exit signal, output tail, run/log pointer, and any feasible resource/process-tree observations; then add targeted tests for OS-signal, supervisor-timeout, manual-cancel, and no-output-timeout result payloads. If the broad-search guardrail is pursued, add bounded warnings or safer defaults for recursive scans over known-large OpenClaw/workspace state rather than hiding the SIGKILL behind generic failure text.
