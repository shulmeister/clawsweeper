## Summary

Keep open. Current main still preserves canonical conversation IDs in `toPluginMessageContext`, while channel resolvers and tests intentionally use typed `user:`/`channel:` conversation IDs for inbound claim contexts. That matches the review comments that this PR's normalization would create a message-vs-claim mismatch rather than a safe cleanup.

## Evidence

- **Current main does not implement the PR's normalization:** `toPluginMessageContext` on `main` still assigns `conversationId: canonical.conversationId` directly; there is no shared normalization helper in this file.

  - file: [src/hooks/message-hook-mappers.ts:261](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/hooks/message-hook-mappers.ts#L261)
