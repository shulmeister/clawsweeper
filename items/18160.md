## Summary

Keep #18160 open. Current main still does not implement a direct `exec`/`shell` cron payload: the core cron types, Gateway schema, validation, runtime, CLI, agent tool schema, Control UI, docs, and tests remain built around `systemEvent` and `agentTurn`. The issue also has concrete production-use comments, a closed/unmerged implementation PR (#51276), and an existing maintainer/Codex comment explicitly keeping #18160 open as the substantive tracker.

## Best Possible Solution

Keep #18160 open as the active core cron tracker unless maintainers explicitly consolidate it. The concrete path is to add a versioned `exec`/`shell` payload contract across `src/cron/types.ts`, Gateway schemas/normalization, cron validation/runtime, CLI add/edit, agent tool schema, Control UI, docs, and tests; implement a controlled process runner with timeout/abort, stdout/stderr capture, run-log/state updates, retry/failure behavior, channel/webhook delivery, and security behavior aligned with existing exec approvals and sandbox policy.

## Risks / Open Questions

- A direct cron exec mode would schedule host command execution outside the model loop, so the eventual implementation needs explicit security policy, timeout/abort behavior, stdout/stderr capture, run-log/state semantics, retry/failure handling, and delivery rules.
- Closing #18160 now would remove the strongest active tracker for a still-missing core cron capability; related open issues are narrower or broader variants rather than a clearly accepted canonical replacement.
