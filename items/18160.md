## Summary

Keep #18160 open. Current main at 82262387651ab270b565f71533ce4937c9c00dc6 still has no direct exec/shell cron payload: the type contract, Gateway schema, validation, runtime dispatch, CLI, Control UI, docs, and tests remain centered on `systemEvent` and `agentTurn`. The issue discussion also shows multiple production users asking for the same zero-LLM scheduled shell execution, and the only direct implementation PR (#51276) was closed unmerged, leaving #18160 as the canonical core tracker.

## Best Possible Solution

Keep #18160 open as the canonical core cron direct-exec tracker. The best implementation is an additive `exec` or `shell` cron payload contract across `src/cron/types.ts`, Gateway schema/validators, normalization, runtime dispatch, `openclaw cron add/edit`, Control UI, docs, and focused tests. It should use a controlled process runner integrated with OpenClaw's existing exec security/approval model, timeouts, stdout/stderr capture, retry/backoff, run history, failure alerting, and channel/webhook delivery. Broader conditional pipelines or direct tool invocation should stay separate unless maintainers intentionally fold them into the design.

## Risks / Open Questions

- A native direct exec cron path is privileged host-command execution and needs explicit security, approval/sandbox, timeout, stdout/stderr capture, retry, logging, run-history, and delivery semantics before implementation.
- Closing this now would discard the apparent canonical tracker while duplicate requests and the closed unmerged implementation PR still point to the same remaining core work.
- Broader direct-tool or conditional-pipeline proposals overlap, but they do not replace this narrower direct shell/exec cron payload request.
