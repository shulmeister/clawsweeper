## Summary

Keep open. Current `main` does not implement the requested bounded memory write tool or hard durable-memory character limits. The issue is recent, coherent, and still matches the shipped memory model: read-side tools plus append-style promotion into `MEMORY.md`.

## Evidence

- **Bundled memory tool surface:** The bundled `memory-core` plugin registers only `memory_search` and `memory_get`. There is no built-in mutating memory tool for add/replace/remove with hard-cap enforcement.
  - file: [extensions/memory-core/index.ts:42](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-core/index.ts#L42)
  - command: `nl -ba extensions/memory-core/index.ts | sed -n '1,120p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Current tools are recall/read only:** `memory_search` is described as semantic recall, and `memory_get` as a bounded exact excerpt read. The current tool code/docs do not expose a write API resembling the issue proposal.
  - file: [extensions/memory-core/src/tools.ts:188](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-core/src/tools.ts#L188)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **`MEMORY.md` is still injected each turn:** Docs for current `main` still say `MEMORY.md` is injected into the context window on every turn, with bootstrap truncation caps. That bounds prompt injection size, but it is not the requested hard limit on persistent memory contents.
  - file: [docs/concepts/system-prompt.md:110](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/docs/concepts/system-prompt.md#L110)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Durable writes are still append-style promotion:** Current docs say long-term promotion writes only to `MEMORY.md`, and `memory promote --apply` appends selected candidates. This is materially different from the proposed bounded store that rejects overflow and supports replace/remove.
  - file: [docs/cli/memory.md:75](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/docs/cli/memory.md#L75)

That suggests implementation scope may belong in a memory plugin, but it is not strong enough evidence to close this feature request outright.

## Risks / Open Questions

- The eventual solution may be redirected into a memory plugin or external plugin path rather than `memory-core`, but current `main` still lacks the requested behavior.
- Bootstrap truncation partially addresses token cost, so maintainers could narrow the issue to durable-write semantics rather than prompt-size concerns.
