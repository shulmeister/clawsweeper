## Summary

Current main still appears to have the reported same-target media-only suppression path. A successful media-only message tool call can commit target and media metadata while leaving `messagingToolSentTexts` empty; `buildReplyPayloads` still checks same-target suppression because media/target metadata exists, and then drops all final session payloads when the target matches. The provided related PR #37900 described the right narrow fix, but it was closed unmerged, and the current tests do not cover same-target media-only sends preserving session text.

## Best Possible Solution

Keep #37841 open and implement the narrow reply-payload fix: same-target reply suppression should only drop final/follow-up session text when the message tool actually sent text to that same origin target, while media URL dedupe should still strip duplicate media. Add a focused regression test for same-target WhatsApp or generic `message` media-only sends preserving final session text, plus a follow-up-delivery check if that path can receive the same metadata shape.
