## Summary

Keep open. Current main still exposes `sessions_yield` to coding agents, the tool intentionally ends the current turn whenever called, and the Kimi Coding provider has no special guard that prevents a spurious Kimi-emitted `sessions_yield` call from stopping long-running work. Related Kimi tool-loop reports do not provide a high-confidence implemented fix or a canonical sessions_yield-specific superseding issue.

## Best Possible Solution

Keep the issue open. The likely maintainer path is to add a Kimi/session-yield regression and then either make `sessions_yield` context-aware so it only yields when the requester is actually waiting for spawned child completions, or add provider/tool-policy prompting/guardrails that prevent Kimi Code from using `sessions_yield` as a generic pause mechanism. If maintainers want consolidation, link it to an open Kimi tool-loop tracker only after confirming that tracker explicitly covers spurious valid `sessions_yield` calls and manual-resume behavior.
