## Summary

Keep open. The report misstates one part of the current contract (`channels.signal.groupAllowFrom` is documented as a sender allowlist, not a group-id allowlist), but current `main` still appears to have a real docs/code mismatch: Signal group ingress checks only sender allowlists and do not enforce `channels.signal.groups` as the documented group-id allowlist. The `requireMention` default-true behavior is intentional and documented, so that sub-claim is not a close-worthy fix-on-main.

## Evidence

- **Signal docs define sender vs group allowlists separately:** Signal docs say `channels.signal.groupAllowFrom` is a `group sender allowlist`, while `channels.signal.groups` is keyed by Signal group id.
  - file: [docs/channels/signal.md:316](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/channels/signal.md#L316)

- **Shared group docs say specific groups belong under `groups`, not `groupAllowFrom`:** The generic groups guide says `groups: { "<group-id>": { ... } }` is the way to allow only specific groups, while `groupAllowFrom` is for limiting which people can trigger inside groups.
  - file: [docs/channels/groups.md:67](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/channels/groups.md#L67)

## Risks / Open Questions

- The issue body's proposed fix for `groupAllowFrom` is inconsistent with current docs and shared group-policy semantics, so the eventual fix likely needs reframing rather than applying the report verbatim.
- Because `requireMention` defaulting to `true` is documented behavior, closing on that basis would discard the still-plausible Signal-specific bug that `channels.signal.groups` is not consulted for access control.
