## Summary

Keep #39138 open. Current main still starts outbound delivery recovery on gateway activation and replays pending outbound queue entries ordered by enqueuedAt, with retry/backoff, a wall-clock recovery budget, and max-retry pruning, but no age-based expiry, max replay count, or user-facing recovery policy that would stop 15-hour-old retryCount=0 entries from being resent.

## Best Possible Solution

Implement an outbound delivery recovery policy in src/infra/outbound that expires or fails stale queued deliveries before replay, likely with a conservative default max age plus focused tests for retryCount=0 stale entries, maxRecoveryMs interaction, and restart recovery. If maintainers want operator control, add a documented config/schema surface for max age and possibly max replay count while preserving recovery for recent crash-window sends.
