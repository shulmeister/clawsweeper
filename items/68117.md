## Summary

Current main already implements the browser/CDP fix for this exact false-negative startup path: managed local loopback CDP health checks bypass browser SSRF enforcement, the startup/status code uses that policy, regression coverage exists, and the changelog says the "not reachable after start" misclassification shipped in v2026.4.14. The report was filed against older v2026.4.12.

## Evidence

- **Shipped fix in changelog:** The v2026.4.14 changelog explicitly says managed local Chrome readiness/status probes and managed loopback CDP control now bypass browser SSRF policy so OpenClaw no longer misclassifies a healthy child browser as "not reachable after start."
