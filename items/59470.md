## Summary

Keep open. The current gateway/plugin contract still requires plugin RPC handlers to call `respond(...)`; `main` does not auto-convert a handler's return value into an RPC reply, and unanswered requests still time out. That makes the report plausible on current `main`, while the public plugin docs remain too implicit to safely conclude this was only reporter misuse.

## Evidence

- **Plugin gateway handler contract is callback-based:** `GatewayRequestHandlerOptions` exposes a mandatory `respond` callback, and `GatewayRequestHandler` returns `Promise<void> | void` rather than a response payload.
  - file: [src/gateway/server-methods/shared-types.ts:119](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server-methods/shared-types.ts#L119)
  - command: `nl -ba src/gateway/server-methods/shared-types.ts | sed -n '110,132p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Gateway dispatch does not synthesize a reply from the handler return value:** `handleGatewayRequest(...)` passes `respond` into the plugin/core handler and only awaits the handler. There is no post-handler fallback that sends a reply when the handler returns normally without calling `respond`.
  - file: [src/gateway/server-methods.ts:164](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/server-methods.ts#L164)
  - command: `nl -ba src/gateway/server-methods.ts | sed -n '155,176p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Unanswered RPC requests still time out:** `GatewayClient.request()` keeps the request pending until a response frame arrives, then rejects with `gateway request timeout for <method>` when the timer expires. The CLI wrapper separately formats that as `gateway timeout after <ms>`.
  - file: [src/gateway/client.ts:930](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/client.ts#L930)
  - command: `nl -ba src/gateway/client.ts | sed -n '917,955p'; nl -ba src/gateway/call.ts | sed -n '430,435p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **In-tree plugin implementations explicitly call `respond`:** Bundled plugin gateway methods, such as `wiki.status`, follow the callback pattern and call `respond(...)` directly instead of returning a payload.
  - file: [extensions/memory-wiki/src/gateway.ts:103](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/memory-wiki/src/gateway.ts#L103)
  - command: `nl -ba extensions/memory-wiki/src/gateway.ts | sed -n '103,129p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Public docs are too implicit to prove reporter misuse:** The plugin SDK docs list `api.registerGatewayMethod(name, handler)` but do not explain on that page that the handler must answer through `respond(...)`, so the report could still reflect a real SDK/docs mismatch rather than a pure user mistake.
  - file: [docs/plugins/sdk-overview.md:96](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/plugins/sdk-overview.md#L96)

  - command: `git log --since='2026-04-02' --oneline -- src/gateway/server-methods.ts src/gateway/server-plugins.ts src/plugins/registry.ts src/gateway/server-methods/shared-types.ts`

## Risks / Open Questions

- The reporter's plugin may simply be returning a value without calling `respond(...)`, which would make this plugin misuse rather than a core runtime bug.
- The issue includes no plugin source, comments, or runnable reproducer against current `main`, so the exact failure mode cannot be proven from the thread alone.
