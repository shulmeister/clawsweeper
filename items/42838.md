## Summary

Current main treats this as expected Control UI secure-context/device-identity enforcement, not an unexplained frontend 4008 bug, and now surfaces actionable guidance for the old `4008 / connect failed` symptom.

## Evidence

## Close Comment

Closing from a Codex review. Current `main` does not point to an unresolved generic frontend `4008` bug here: the gateway intentionally rejects remote/non-secure Control UI connections that lack device identity (`src/gateway/server/ws-connection/connect-policy.ts:129`, `src/gateway/server/ws-connection/message-handler.ts:653`), and the Control UI now maps the old `4008 / connect failed` wrapper to actionable guidance (`ui/src/ui/connect-error.ts:74`, covered by `ui/src/ui/app-gateway.node.test.ts:369`). The docs also now state that `http://<lan-ip>` / `http://<tailscale-ip>` is blocked by design and that `gateway.controlUi.allowInsecureAuth` only helps localhost, not remote LAN access (`docs/web/control-ui.md:254`). Given that the original report did not include the exact URL, browser/version, or structured connect error details, this no longer reproduces as an actionable current-main bug report. If this still reproduces on a supported `https://...` or localhost setup, please open a new issue with the exact URL pattern, browser/version, and the full structured connect error details from the current build.
