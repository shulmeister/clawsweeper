## Summary

Keep open. Current main still accepts any `tools.web.search.provider` string in the config schema and plugin-aware config validation does not reject provider typos, while runtime code preserves fallback/warning behavior for unknown configured providers. The issue remains actionable core/plugin-boundary work, not cleanup-close material.

## Best Possible Solution

Keep this issue open. Add plugin-aware config validation for `tools.web.search.provider`, deriving accepted provider IDs from bundled and installed plugin `contracts.webSearchProviders` metadata, rejecting or clearly reporting typos while preserving a warning-only path for stale missing-plugin configs. Cover built-in providers, manifest-declared third-party providers, invalid typos, and missing-plugin compatibility with focused regression tests.

## Risks / Open Questions

- Typos in `tools.web.search.provider` can still pass config validation and change runtime behavior through auto-detect or fallback selection.
- A fix should preserve the requested warning-only compatibility path for stale configs where a previously installed provider plugin is missing.
- Validation should derive accepted IDs from manifest/plugin metadata, not reintroduce a hardcoded enum that blocks third-party providers.
