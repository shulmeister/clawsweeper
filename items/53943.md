## Summary

Current main still does not implement the requested Slack `message` read by `messageId`. The prior ClawSweeper comment correctly identified the missing wiring, and the same gap remains at `9626ef274ae2346b3d519f99c31856be98aab570`: Slack read forwards `threadId` but never reads, forwards, or filters by `messageId`.

## Best Possible Solution

Keep this issue open. The fix belongs in the Slack plugin read path: add `messageId` to the Slack read action mapping, pass it through `readMessages`, extend `readSlackMessages` to return an exact thread reply when `messageId + threadId` are provided and an exact channel message when `messageId` alone is provided, and add focused regression tests for both cases. If Slack URL parsing feeds this tool path, add coverage for `pMESSAGE_TS?thread_ts=THREAD_TS` normalization as part of the same change.
