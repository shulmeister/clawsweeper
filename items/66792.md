## Summary

Keep #66792 open. The report is recent and concrete, and current main still sends active-memory through the normal embedded runner setup path before the model call. `bootstrapContextMode: "lightweight"` currently filters bootstrap files, but it does not bypass runtime plugin loading, model/auth resolution, sandbox/session/tool setup, or system-prompt assembly, and I found no current source, test, changelog, or release evidence that the reported 6s pre-model bootstrap gap has been profiled or fixed.

## Best Possible Solution

Keep the issue open for maintainer performance work. The best next step is to add phase timing around active-memory `runEmbeddedPiAgent` startup through first model call, reproduce the reported Docker/fast-model path, and then optimize or cache the implicated setup work. If the intended product contract is that `recent` active-memory needs roughly 15s, update the issue with that explicit expectation; otherwise add a regression/diagnostic test or benchmark proving lightweight embedded bootstrap no longer dominates blocking plugin latency.
