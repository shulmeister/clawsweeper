## Summary

Keep open. Current main still runs Active Memory as a blocking pre-reply embedded agent call, and the current implementation only mitigates timeout/fail-open behavior; it does not prove or implement a sub-1s `bootstrapContextMode: "lightweight"` bootstrap path for blocking plugins.

## Best Possible Solution

Keep this issue open and treat it as a focused embedded-runner cold-path profiling task for blocking plugin calls. Add stage timings around lane wait, runtime plugin load, hook/model/harness/auth/profile resolution, session lock/session manager setup, context-engine/bootstrap resolution, tool materialization, bundle runtime setup, and prompt start; then cache or bypass safe per-run invariants for lightweight blocking runs. Preserve the Active Memory hard timeout/fail-open behavior and update the Active Memory docs with realistic latency and timeout guidance once measured.
