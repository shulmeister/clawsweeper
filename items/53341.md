## Summary

Current `main` already provides the requested async inbound gating capability via the public `before_dispatch` hook, which is awaited before model dispatch and can short-circuit with an optional reply. The implementation does not add a blocking mode to `message_received`, but it covers the feature request well enough through a dedicated pre-dispatch hook.

## Evidence

- **Public async pre-dispatch hook exists:** `before_dispatch` is part of the public hook name set and its handler can return `PluginHookBeforeDispatchResult`, while `message_received` remains observer-only (`Promise<void> | void`).
  - file: [src/plugins/hook-types.ts:695](https://github.com/openclaw/openclaw/blob/0a3da5cd8a6e712de6418008e01b11fb2afd7aea/src/plugins/hook-types.ts#L695)

## Risks / Open Questions

- The shipped API is `before_dispatch`, not `message_received` with `mode: "blocking"`, so any external discussion of the older proposed shape is now stale.

## Close Comment

Closing based on a Codex review of current `main`. The requested capability is now present through the public `before_dispatch` hook rather than a blocking mode on `message_received`: `src/plugins/hook-types.ts:695` exposes an async result-bearing hook, `src/plugins/hooks.ts:701` runs it as a claiming pre-dispatch hook, and `src/auto-reply/reply/dispatch-from-config.ts:682` awaits it before model dispatch and returns early with an optional terminal reply. Coverage in `src/auto-reply/reply/dispatch-from-config.test.ts:3301` verifies that a handled result skips the normal dispatch path and can send a blocking reply. `message_received` itself remains observer-only at `src/plugins/hook-types.ts:703`, but the underlying feature request is implemented on `main` via this dedicated hook. This appears to land in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`; I could not confirm a tagged release containing it yet.
