## Summary

Keep this issue open. Current main still lacks the requested first-class Control UI Settings surface for persisted TTS enable/disable, provider selection, provider-filtered default voice selection, and provider-backed sample playback. Backend TTS RPCs, provider voice metadata, config/docs, and a per-message browser-native Read aloud button exist, but the discoverable Settings UX requested here has not been implemented.

## Best Possible Solution

Keep this issue open. Implement a first-class Text-to-Speech panel in Control UI Settings that reads tts.status and tts.providers, calls tts.enable/disable and tts.setProvider, clearly marks configured versus unavailable providers, exposes provider-filtered voice choices using existing provider voice metadata/listVoices or a new safe bounded RPC where needed, persists the selected default voice through the agreed TTS preference/config contract, and uses tts.convert for a short Play sample flow. Add focused UI and gateway tests for the settings workflow.
