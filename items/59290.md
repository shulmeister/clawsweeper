## Summary

Keep open. Current `main` at `78a431ed224b6f5b9d58672346551a5b411ad200` still lacks the PR's two URL guards, so the reported Discord voice-note failure path is not implemented on `main` and remains plausible from the live code paths.

## Evidence

- **Attachment resolution still passes possibly-undefined URLs into the fetch path and fallback payload:** `appendResolvedMediaFromAttachments` still calls `fetchDiscordMedia` with `attachment.url` and, on error, unconditionally pushes `path: attachment.url`. There is no `if (!url) continue` guard in current `main`.
  - file: [extensions/discord/src/monitor/message-utils.ts:368](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/discord/src/monitor/message-utils.ts#L368)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '368,414p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Preflight audio collection still treats URL-less audio attachments as audio attachments:** `collectAudioAttachments` still filters only on `content_type?.startsWith("audio/")`. The later `audioUrls` array is filtered separately, so `hasAudioAttachment` and the preflight branch can still be driven by an attachment that has no usable URL.
  - file: [extensions/discord/src/monitor/preflight-audio.ts:18](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/discord/src/monitor/preflight-audio.ts#L18)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Downstream media normalization drops entries without a path or URL:** `buildMediaPayload` mirrors `media.path` into `MediaPath`/`MediaUrl`/`MediaPaths`/`MediaUrls`, and `normalizeAttachments` later filters out entries that have neither a normalized path nor URL. That means a fallback entry with `path: undefined` does not become a usable media attachment for media-understanding.
  - file: [src/channels/plugins/media-payload.ts:15](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/channels/plugins/media-payload.ts#L15)

- **Normalization behavior confirms URL-less media entries are discarded:** When `MediaPaths` are present, `normalizeAttachments` maps each entry and filters with `Boolean(entry.path ?? normalizeOptionalString(entry.url))`; URL-less / path-less entries are removed before audio transcription/media-understanding can use them.
