## Summary

Keep open. Current main still routes Discord inbound traffic through the generic binding resolver and falls back to the configured default agent when no binding matches; I did not find an implemented Discord accountId-to-agentId fallback. The related fix PR #39485 is open/unmerged in the provided GitHub context, so the issue is still the right tracker.

## Best Possible Solution

Keep #39428 open until PR #39485 or an equivalent maintainer-approved change lands. The likely implementation belongs in the routing path after explicit binding tiers and before default fallback, should guard the literal default account from surprising overrides, and should include regression tests for Discord accountId matching and default-account behavior.
