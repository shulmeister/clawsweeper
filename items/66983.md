## Summary

Keep #66983 open. Current main partially has canvas commands available through the `unknown` platform fallback, but it still does not model `web` as a first-class node platform and it intentionally keeps token/trusted-proxy/device-auth bypasses scoped away from node-role browser sessions. The requested web browser A2UI canvas-node pattern remains a real core gateway/protocol/design item, not something already implemented or safely closeable as ClawHub.

## Best Possible Solution

Keep the issue open as a core gateway/protocol enhancement. A good implementation would add an explicit `web` platform command profile for canvas/A2UI, define the supported browser-node enrollment model, and update gateway connect/pairing policy only after deciding whether secure browser device identity plus normal pairing is enough or whether a new narrowly scoped node-role trusted-proxy/token path is acceptable. It should include focused tests in `node-command-policy`, `connect-policy`, node pairing/auto-approval coverage, and public docs for the web canvas-node pattern.

## Risks / Open Questions

- The auth half of this request is security-sensitive: current code and tests deliberately avoid device-less node-role registration, so a fix needs maintainer product/security judgment rather than a small allowlist tweak.
- The command-policy half is partially mitigated by the `unknown` canvas fallback, so the remaining product decision is whether explicit `web` platform semantics are needed and what commands a web canvas node should receive by default.
