## Summary

Keep open. Current main has partial mitigations that match the discussion, including channel-start fault isolation, Feishu sequential startup probes, Telegram control-plane timeouts, and readiness checks. But the reported user problem is not fully solved: channel tasks are still handed off before provider recovery is complete, Feishu identity recovery can still enter a 30s timeout plus background retry path, Telegram webhook cleanup remains a polling startup prerequisite, and there is no clear per-channel startup/recovery timing summary that distinguishes gateway-ready from channels-fully-recovered.

## Best Possible Solution

Keep this open unless maintainers split it into narrower canonical issues. The best path is a channel startup/recovery observability pass: report gateway listener readiness separately from per-channel recovered/degraded state, emit per-channel startup timing and recovery summaries, make Feishu identity recovery visibly degraded without ambiguous startup delay, and surface Telegram cleanup/network fallback delays as channel-specific recovery state.
