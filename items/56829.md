## Summary

Current `main` already covers the requested behavior through the existing per-agent skill allowlist, `metadata.openclaw.always: true` eligibility pinning, and next-turn skill snapshot refresh logic; the exact `agents.list[].skills.always` config shape does not exist, but the underlying capability is already implemented.

## Evidence

- **Per-agent skill set already exists:** The public skills docs define `agents.defaults.skills` plus `agents.list[].skills`, and state that a non-empty `agents.list[].skills` list is the final set for that agent and is applied across prompt building and skill snapshots.
  - file: [docs/tools/skills.md:51](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/tools/skills.md#L51)

- **Skill metadata already supports always-include:** The skills docs document `metadata.openclaw.always: true` as 'always include the skill (skip other gates)'.
  - file: [docs/tools/skills.md:193](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/tools/skills.md#L193)

  - file: [src/agents/skills/config.ts:73](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/skills/config.ts#L73)

## Risks / Open Questions

- The exact `agents.list[].skills.always` syntax requested in the issue is not present; this close depends on treating the existing allowlist plus `metadata.openclaw.always` as behaviorally sufficient.
- OpenClaw still lets the model decide when to read a SKILL.md from `<available_skills>`; core does not force a skill file to be read on every turn.
