## Summary

Keep open. Current main still resolves PDF tool models through the registry-only helper, so the config-defined custom-provider failure described by the linked bug remains unimplemented on main. The PR is a plausible focused fix, but it should address the existing provider-alias review concern before landing.

## Best Possible Solution

Keep this PR open or have a maintainer take it over. The best fix is a PDF/media model resolver that preserves the current normalized registry lookup, falls back to `cfg.models.providers` using normalized provider/model matching like `findInlineModelMatch`, and adds regression coverage for both the registry-miss custom provider and a provider-alias custom provider.
