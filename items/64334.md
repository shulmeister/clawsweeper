## Summary

Keep open. Current main has a `/status` cost line and a separate `/usage cost` command, but `/status` does not yet show the requested cumulative session cost from transcript `usage.cost.total`. The status card cost is derived from the current session entry's input/output tokens and only falls back to transcript token totals when stored values are missing; the transcript cost total is available elsewhere but is not wired into `/status`.

## Best Possible Solution

Keep this issue open and implement the feature in the chat status/status-message path. Reuse the existing transcript-backed session cost summary or `usage.cost.total` aggregation, add a distinct cumulative `Session cost` line to `/status` and `session_status`, keep API-key/OAuth visibility semantics, and cover it with focused status-card tests for multi-turn transcript totals and non-API-key hiding.

## Risks / Open Questions

- The implementation should preserve existing OAuth/token behavior that hides dollar costs.
- Avoid double-counting delivery mirrors, compaction artifacts, or repeated persist calls; the existing session-cost summary path is the safest source to reuse.
- The existing `💵 Cost:` label may be confused with cumulative session cost, so the UI should use a distinct label such as `Session cost`.
- Related API-call-count work in #48851 is a different status observability request and does not supersede this cost request.
