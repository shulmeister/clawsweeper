## Summary

Keep open. Current main has implemented the cron run-session retention/reaper path and documents `cron.sessionRetention`, so the unbounded-retention portion is partially addressed. But the issue also asks for visible guardrails and warnings; current main still accepts high-frequency `--every` cron jobs without a CLI or gateway warning, and the related PR #38731 that proposes that warning is still open/unmerged in the provided context.

## Best Possible Solution

Keep #38730 open until the remaining user-visible guardrails are implemented or explicitly split into a canonical maintainer-tracked follow-up. The shipped cron session reaper and `cron.sessionRetention` docs should stay; the missing work is a non-blocking warning/health guardrail for high-frequency cron schedules and/or session/context pressure, likely in the cron CLI, gateway `cron.add` path, and focused tests.
