## Summary

Keep open. Current main has strong partial mitigation for cron run-session accumulation via `cron.sessionRetention` and an automatic cron run-session reaper, plus some context-usage visibility. But the requested guardrails are not fully shipped: `cron add` still creates high-frequency `--every` jobs without a CLI or gateway warning, `sessions cleanup` does not explicitly invoke the cron run-session reaper, and the linked warning PR #38731 remains open and unmerged in the provided GitHub context.

## Best Possible Solution

Keep #38730 open until maintainers either land a conservative equivalent of #38731's CLI and gateway warning for high-frequency `--every` schedules or explicitly decide not to warn. A complete cleanup would also decide whether `openclaw sessions cleanup` should invoke the cron run-session reaper or clearly document that cron retention is handled by the scheduler-owned `cron.sessionRetention` path.
