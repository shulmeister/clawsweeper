## Summary

Keep open. Current main has a real cron run-session retention/reaper mitigation, but it still does not implement the issue’s creation-time guardrail: neither CLI `cron add`, gateway `cron.add`, the Control UI cron add path, nor the agent cron tool warns on high-frequency `every` schedules. Manual `openclaw sessions cleanup` also still uses generic session maintenance and does not invoke the cron run-session reaper directly. The linked PR #38731 is still open/unmerged in the provided GitHub context, so this remains actionable.

## Best Possible Solution

Keep #38730 open until maintainers land PR #38731 or an equivalent fix. The most direct path is a shared high-frequency `every` schedule warning for CLI, gateway RPC, Control UI, and agent-tool-created cron jobs, plus a maintainer decision on whether `openclaw sessions cleanup` should call the cron run-session reaper, expose a cron-specific cleanup command, or document cron cleanup as scheduler-owned maintenance only.
