## Summary

Current main partially mitigates #38730 with cron run-session retention and automatic reaping, but the issue is not fully implemented. The requested high-frequency cron creation guardrail is still absent from both CLI and gateway `cron.add`, `openclaw sessions cleanup` still runs generic session maintenance instead of the cron run-session reaper, and the directly linked PR #38731 remains open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep #38730 open until maintainers either land PR #38731 or add an equivalent conservative warning in both `src/cli/cron-cli/register.cron-add.ts` and `src/gateway/server-methods/cron.ts` for high-frequency `schedule.kind: "every"` jobs. Then decide whether `openclaw sessions cleanup` should invoke the cron run-session reaper, expose a cron-specific cleanup path, or document that cron retention is scheduler-owned through `cron.sessionRetention`.
