## Summary

Current `main` has a real mitigation for cron-session accumulation via `cron.sessionRetention` and the cron session reaper, but the issue asks for broader guardrails/warnings. I found the retention cleanup on `main`, yet I did not find a high-frequency schedule warning in cron creation/edit flows or a user-facing context-pressure warning path tied to cron buildup, so the evidence is not strong enough to close conservatively.

## Evidence

- **Cron run sessions are now pruned by retention policy:** `sweepCronRunSessions` deletes expired `...:cron:<jobId>:run:<uuid>` entries and archives transcript files, with a default 24h retention window.
  - file: [src/cron/session-reaper.ts:54](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cron/session-reaper.ts#L54)
- **Reaper runs even when cron timer work fails early:** `onTimer` invokes the cron session reaper from the `finally` path so stale isolated run sessions still get pruned after timer/job failures.
  - file: [src/cron/service/timer.ts:816](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cron/service/timer.ts#L816)
- **This mitigation was shipped and documented:** The changelog entry for release `2026.2.9` says sessions now prune cron run sessions after retention windows; current docs still document `cron.sessionRetention` as the control for completed isolated run sessions.
  - file: [CHANGELOG.md:5233](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/CHANGELOG.md#L5233)
- **Docs expose retention, not creation-time guardrails:** Current cron CLI docs document retention pruning (`cron.sessionRetention`) but do not describe warnings or a minimum recommended interval on job creation.
  - file: [docs/cli/cron.md:80](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/docs/cli/cron.md#L80)
- **CLI cron creation still accepts any positive interval without a frequency warning:** `cron add` parses schedule options and validates payload/session combinations, but there is no `<30m` warning/guardrail in the creation path.
  - file: [src/cli/cron-cli/register.cron-add.ts:66](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/cli/cron-cli/register.cron-add.ts#L66)
- **Web UI cron form also lacks a high-frequency warning check:** Cron form validation checks only that `everyAmount` is positive; it does not warn on aggressive cadences such as every 15m or every 30m.
  - file: [ui/src/ui/controllers/cron.ts:111](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/ui/src/ui/controllers/cron.ts#L111)
