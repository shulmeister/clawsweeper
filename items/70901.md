## Summary

Current main already resolves `${ENV_VAR}` in `openclaw.json` before MCP server configs reach the bundled MCP HTTP transport, so `mcp.servers.*.headers.Authorization: "Bearer ${SUPABASE_PAT}"` is no longer sent literally when the env var is present. The related PR #71035 identified the same symptom, but the shipped/current path handles it at config-load time and then passes the resolved headers into the MCP SDK transports.

## Best Possible Solution

Keep the current config-loader based implementation: resolve `${ENV_VAR}` once when reading OpenClaw config, pass the resolved `mcp.servers` object through the bundled MCP registry, and keep the docs pointing users to `${VAR_NAME}` config substitution for header tokens. PR #71035 can be closed or redirected as redundant unless maintainers want an extra raw-helper fallback for non-config callers.
