## Summary

Keep #71711 open. Current main does not implement an `activeUntil`/`expiresAt` expiration contract for recurring cron jobs: the job model and gateway schemas only expose existing fields such as `deleteAfterRun`, the CLI has no `--active-until` option, and the scheduler still re-arms successful recurring jobs instead of expiring/removing/disabling them. The related #71712 is an open sister RFC for agent-facing scheduling and depends on this idea rather than superseding it.

## Best Possible Solution

Keep #71711 open. The best maintainer path is a core cron enhancement: add optional `activeUntil`/`onExpire` fields to cron job types, normalization, gateway schemas, CLI add/edit/list/show, scheduler due-run handling, persistence, docs, and tests. Coordinate naming and security defaults with the open #71712 scheduling API RFC, but do not close this narrower recurring-job expiration request as implemented or superseded.

## Risks / Open Questions

- Implementation will need a backward-compatible persisted job schema/RPC contract change plus CLI, list/show, docs, and regression tests.
- Expiration behavior needs careful handling for due, missed, disabled, running, and startup-catchup jobs so expired recurring jobs cannot run once more by accident.
