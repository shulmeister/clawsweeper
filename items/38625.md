## Summary

Keep open. Current `main` has a WebSocket `update.run` control-plane method and a separate `status.runtimeVersion`, but it does not implement the requested `version` field on the HTTP `GET /health` response, does not expose a `POST /admin/update` HTTP route, and the existing update RPC does not accept a target version pin.

## Evidence

- **HTTP /health is still a shallow liveness response:** `/health` and `/healthz` are mapped to the same live probe path, and the handler returns `JSON.stringify({ ok: true, status })` for live probes rather than a payload with a version field.
  - file: [src/gateway/server-http.ts:187](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/gateway/server-http.ts#L187)

  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Probe test locks in the shallow health payload:** The current test suite asserts `/healthz` returns exactly `{"ok":true,"status":"live"}`. That is the same code path as `/health`, so current main intentionally does not include `version` there.
  - file: [src/gateway/server-http.probe.test.ts:246](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/gateway/server-http.probe.test.ts#L246)

## Risks / Open Questions

- The issue is partially obsolete because maintainers already shipped a WS-level `update.run` path and `status.runtimeVersion`, but closing it would lose the still-missing HTTP `/health` version field and `/admin/update` transport requested by the reporter.
- The existing update flow also lacks version pinning, so even treating `update.run` as a substitute would not satisfy the full requested contract.
