## Summary

Current `main` already provides a browser-side queued-input flow for active runs and drains queued messages as normal follow-up chat turns; soft-injected/heartbeat-like behavior is now an explicit separate `Steer` action.

## Close Comment

Closing from a Codex review because current `main` already implements the requested Control UI queue flow. While a run is active, webchat now exposes a `Queue` action (`ui/src/ui/chat/run-controls.ts:43`), busy sends are appended to `chatQueue` instead of being soft-injected (`ui/src/ui/app-chat.ts:386`), and terminal chat events drain that queue through `sendChatMessageNow(...)` (`ui/src/ui/app-gateway.ts:400`, `ui/src/ui/app-chat.ts:275`). That send path creates a normal user turn and starts a fresh tracked run (`ui/src/ui/controllers/chat.ts:325`). The only soft-injected path is the explicit queued-message `Steer` action, which is separately labeled as `Steered` in the UI (`ui/src/ui/views/chat.ts:1142`). I also found tests covering the active-run queue button and the steer-vs-queued distinction (`ui/src/ui/chat/run-controls.test.ts:84`, `ui/src/ui/views/chat.test.ts:72`). Reviewed against `main` at `60f7a59f5ed581839cfd1d52583449c5aa1f03a4`.
