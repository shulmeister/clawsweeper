## Summary

Keep PR #42208 open. Current main still does not implement the central hard-enforcement behavior: there is no shipped tool-call guard for external hook sessions, `web_search` wrapping still omits the long security warning, and the documented/current protection model remains prompt wrapping plus operator-configured tool policy, exec approvals, sandboxing, and allowlists. The PR is not ready as-is because the review discussion correctly identified that its new guard is not wired into runtime dispatch, but the underlying core security hardening remains a real, unsuperseded item.

## Best Possible Solution

Keep the PR open for maintainer review, but require changes before merge. The best implementation is a core enforcement path at the tool-execution boundary, likely near the existing `before_tool_call`/tool wrapping path, that uses session context to detect external hook sessions, respects the explicit unsafe-content bypass, and requires human/plugin approval for exec, file mutation, outbound messaging, and subagent-spawn tools. If the web_search warning behavior is still desired, update the source, tests, and any docs that describe warning/banner behavior together.
