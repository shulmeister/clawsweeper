## Summary

Current `main` already distinguishes the configured model from the actual fallback model in `/status`, persists fallback state into the session, and this behavior was already shipped in `v2026.4.23`.

## Evidence

- **Current `/status` output includes the active fallback model:** `buildStatusMessage()` keeps the configured model on the `Model:` line and adds a separate `↪️ Fallback:` line with the actual active model and fallback reason when the runtime model differs.

- **Automatic fallback state is persisted for later `/status` reads:** After a run, `agent-runner` stores `fallbackNoticeSelectedModel`, `fallbackNoticeActiveModel`, and `fallbackNoticeReason` into the session entry and persisted session store, which is the state consumed by the status renderer.
  - file: [src/auto-reply/reply/agent-runner.ts:1288](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/auto-reply/reply/agent-runner.ts#L1288)
