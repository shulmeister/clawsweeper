## Summary

Current `main` treats Tailscale Serve identity as a Control UI/WebSocket auth mechanism, not as a device-pairing bypass, and the current device-pairing flow creates/surfaces pending requests through `openclaw devices list`. The issue's expectation and CLI path do not match current `main` behavior.

## Close Comment

Codex review: I could not confirm a current-`main` bug here. On current `main`, Tailscale Serve identity satisfies the Control UI/WebSocket auth check, but it does not bypass device pairing (`src/gateway/auth.ts:461`, `src/gateway/server/ws-connection/connect-policy.ts:43`). The docs explicitly say a new browser/device still needs a one-time pairing approval even with `gateway.auth.allowTailscale: true` (`docs/web/control-ui.md:38`). Current `main` also routes these approvals through `openclaw devices list` / `openclaw devices approve`, while `openclaw pairing ...` is the separate channel-DM pairing CLI (`docs/cli/devices.md:15`, `docs/cli/pairing.md:10`). For unpaired devices, the handshake still creates a device-pairing request and includes `requestId` in the error details (`src/gateway/server/ws-connection/message-handler.ts:939`, `src/gateway/protocol/connect-error-details.ts:296`). Based on that, I’m closing this as not reproducible on current `main` as filed. If someone can still reproduce on current `main` using `openclaw devices list` and there is truly no pending request, please open a fresh issue with the exact `error.details`, request URL/mode, and `openclaw devices list --json` output.
