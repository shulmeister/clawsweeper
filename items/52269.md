## Summary

Current main already handles bundled/installed plugin runtime entry resolution for Feishu/Lark-style plugins: bundled metadata is rewritten to built `.js` entrypoints, package entry validation is anchored to the plugin directory, and discovery tests cover installed package runtime loading instead of rejecting `./index.js` as escaping.

## Close Comment

Closing from a Codex review because current `main` already carries the packaged-plugin repair path for this failure. `scripts/copy-bundled-plugin-metadata.mjs:23` and `scripts/copy-bundled-plugin-metadata.mjs:295` rewrite bundled plugin `openclaw.extensions` / `setupEntry` metadata to built `.js` runtime paths, `src/plugins/copy-bundled-plugin-metadata.test.ts:88` asserts that rewrite to `./index.js`, and `src/plugins/discovery.ts:556` resolves entries relative to the plugin package directory instead of treating `./index.js` as escaping. The installed-package discovery coverage in `src/plugins/discovery.test.ts:411` also verifies runtime `dist/index.js` and `dist/setup-entry.js` loading. This is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), whose changelog also calls out the Feishu/Lark packaged-install repair in `CHANGELOG.md:243`. If you still hit this on `v2026.4.22` or newer, please open a new issue with the exact generated plugin `package.json` and install layout so maintainers can check for a new regression.
