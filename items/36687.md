## Summary

Keep #36687 open. Current main has guardrails and regression coverage for the WebChat/UI cross-channel leakage symptom, but it still does not prove or implement a fix for the stronger upgrade/reset claim that `session.dmScope` was changed or effectively lost during the 2026.3.2 update path. Runtime, outbound, routing, doctor, and docs still treat an unset `session.dmScope` as `main`, while onboarding only writes `per-channel-peer` for local setup when the value is unset and preserves explicit `main`.

## Best Possible Solution

Keep the issue open until maintainers either verify the 2026.3.2 update/reset path and add a targeted regression/repair if needed, or decide explicitly that unset `session.dmScope` should continue to mean `main` and close with that product decision. The shipped WebChat/UI leakage guardrails should remain; the remaining work is specifically the config/update preservation or defaulting question.
