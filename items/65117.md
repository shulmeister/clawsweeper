## Summary

Current `main` already routes `message`-tool attachment inputs (`media`, `mediaUrl`, `path`, `filePath`, `fileUrl`) into outbound media delivery, and the WhatsApp outbound path now canonicalizes media URLs before logging/sending. The reported silent-drop behavior matches older builds, but the code and tests on `main` now cover this path.

## Evidence

- **Message tool now accepts `filePath`/`path` as media inputs for send:** `handleSendAction` reads `media`, `mediaUrl`, `path`, `filePath`, and `fileUrl`, normalizes them, and assigns the first resolved media path back to `params.media` before delivery.
  - file: [src/infra/outbound/message-action-runner.ts:481](https://github.com/openclaw/openclaw/blob/019ef71fe84d25b4a7ed196dff27a29e3c567292/src/infra/outbound/message-action-runner.ts#L481)
  - command: `nl -ba src/infra/outbound/message-action-runner.ts | sed -n '476,545p'`
  - sha: [019ef71fe84d](https://github.com/openclaw/openclaw/commit/019ef71fe84d25b4a7ed196dff27a29e3c567292)
- **Regression coverage for in-sandbox send media rewrites:** Current tests cover dry-send rewriting for `media`, `mediaUrl`, `fileUrl`, and `MEDIA:` directives so outbound sends keep a resolved attachment path instead of dropping media.
  - file: [src/infra/outbound/message-action-runner.media.test.ts:723](https://github.com/openclaw/openclaw/blob/019ef71fe84d25b4a7ed196dff27a29e3c567292/src/infra/outbound/message-action-runner.media.test.ts#L723)

- **WhatsApp send path logs and sends using resolved primary media:** `sendMessageWhatsApp` now derives `primaryMediaUrl` from the canonical media URL list, loads the media buffer when present, and logs `hasMedia` from that resolved value before sending.
  - file: [extensions/whatsapp/src/send.ts:81](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/extensions/whatsapp/src/send.ts#L81)
