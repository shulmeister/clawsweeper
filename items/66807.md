## Summary

Keep #66807 open. Current main still does not do peer-aware outbound account selection in `runMessageAction`; it only honors an explicit/default account and then falls back to an agent-keyed binding lookup. The PR addresses a plausible real bug with empirical Slack reproduction, but the submitted patch has unresolved correctness concerns and should be revised rather than closed or merged as-is.

## Best Possible Solution

Keep the PR open, but require a revision. The best path is to implement peer-aware outbound account selection using the existing route-binding normalization surface, preferably `listRouteBindings` or `resolveFirstBoundAccountId`, while preserving agent scoping and handling duplicate peer bindings deterministically. Add a focused regression test under `src/infra/outbound/message-action-runner*.test.ts` that reproduces the multi-account Slack-style case: default-agent binding points at one account, target peer binding points at another, and `runMessageAction` selects the peer-specific account only when no explicit `--account`/`accountId` is supplied.

There is no lookup against `params.target`, `params.to`, `params.channelId`, or `bindings[].match.peer.id`.
