## Summary

Keep #71783 open. Current main still has the unresolved QMD conflict path described by the issue: OpenClaw can rebind a different-name collection only when `qmd collection list` exposes path+pattern metadata, but it does not parse the conflicting name from an add error such as `Name: life-main (qmd://life-main/)`. If metadata is unavailable, current tests still expect a warning/skip, and search continues to target the configured managed name.

## Best Possible Solution

Keep #71783 open and add a narrow memory-core repair. `extensions/memory-core/src/memory/qmd-manager.ts` should parse the conflicting collection name from QMD add errors like `A collection already exists for this path and pattern` plus `Name: life-main (qmd://life-main/)`, then safely remove/re-add under the managed name only when the attempted add path and pattern match the configured collection. Add regression coverage for `life`/`life-main` and for missing-collection search retry. If automatic rebinding is judged too destructive, add a targeted diagnostic and `openclaw doctor --fix` repair instead.
