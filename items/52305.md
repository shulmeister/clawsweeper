## Summary

Current `main` narrows the original theory but does not fully close the reported gap. Internal exec/node completion plumbing is now session-scoped, but the external `openclaw system event` / `wake` path still lacks an explicit session target and still routes through the wake RPC that only accepts `mode` and `text`, matching the remaining end-to-end failure described in the issue.

## Evidence

- **CLI surface still has no session targeting:** `openclaw system event` only accepts `--text`, `--mode`, and `--json`, then calls gateway method `wake` with `{ mode, text }`. There is no `--session` / `--session-key` flag or payload field.
  - file: [src/cli/system-cli.ts:52](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cli/system-cli.ts#L52)

  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Official docs still describe main-session-only behavior:** The CLI docs explicitly say `system event` enqueues on the main session.
  - file: [docs/cli/system.md:34](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/docs/cli/system.md#L34)

  - file: [src/gateway/server-methods/system.ts:54](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/server-methods/system.ts#L54)
