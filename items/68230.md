## Summary

Keep open. Current `main` still exposes only `contextPruning.mode: "off" | "cache-ttl"`, and the docs explicitly describe session pruning as in-memory only, with compaction as the separate persistent mechanism. The report's specific diagnosis that pruning TTL collapses to gateway uptime is weakened by current code that persists and reloads cache-touch timestamps from session entries, so this is not a safe cleanup close; it needs maintainer/product judgment as an active feature request or a narrower bug report.

## Risks / Open Questions

- The issue body mixes two separate concerns: OpenClaw's pruning cadence and upstream provider prompt-cache retention across restarts. Current code addresses the former by persisting cache-touch timestamps, but that does not automatically prove or disprove the latter.
- Compaction may already cover part of the user's persistence goal, but it is behaviorally different from a disk-backed pruning mode because it rewrites future context via a persisted summary rather than replaying trimmed raw tool output.
