## Summary

Current main already solves the restart/TTL problem behind this request. Cache-TTL runs now persist an `openclaw.cache-ttl` custom session entry with timestamp/provider/model, reload the latest matching timestamp when the embedded runner installs context pruning, and gate pruning from that persisted `lastCacheTouchAt` instead of from gateway process uptime. The docs also clarify that session pruning is in-memory only and does not rewrite the on-disk transcript, so full conversation history is preserved across restarts.

## Best Possible Solution

Keep the shipped cache-TTL persistence behavior in core. If someone wants a broader disk-backed alternate context policy beyond cache-TTL bookkeeping, that should be pursued as a context-engine plugin or a focused plugin API discussion rather than another legacy `contextPruning.mode`.
