## Summary

Keep open. Current main has meaningful cron reliability work, including startup missed-job catch-up, delivery diagnostics, retries, and failure alerts, but the report is a recent multi-symptom Telegram/macOS regression with no concrete reproduction or logs. Related #18160 and #29907 cover only the direct-exec cron subset, not the full Telegram cron delivery, approval lifecycle, OOM, and fallback behavior described here.

## Best Possible Solution

Keep this issue open and ask the reporter for sanitized logs, exact cron job definitions, failing commands, approval IDs/timestamps, current OpenClaw version, and whether the failure is main-session or isolated cron. Then split or link sub-issues as needed: use #18160/#29907 for direct shell/exec cron, keep Telegram approval/delivery fixes in the Telegram plugin and shared approval runtime, and keep scheduler catch-up/failure-reporting changes in `src/cron` only when current-main repro shows a core cron defect.
