## Summary

Keep open. Current `main` does not implement a watcher or periodic reload for an empty cron store, so the reported behavior still exists when the scheduler starts with zero jobs and `jobs.json` is edited externally after startup.

## Evidence

- **Startup only arms the existing scheduler state:** `start()` loads the store once, recomputes runs, then calls `armTimer()` based on the in-memory jobs loaded at startup.
  - file: [src/cron/service/ops.ts:87](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/ops.ts#L87)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **No timer when there are zero enabled jobs:** `armTimer()` returns without scheduling any recheck when `nextWakeAtMs` is missing and `enabledCount` is 0, logging `armTimer skipped - no jobs with nextRunAtMs`. That means a gateway started with an empty store will not wake back up to notice later file edits.
  - file: [src/cron/service/timer.ts:618](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/timer.ts#L618)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Reload happens only during active timer/manual paths:** `onTimer()` does force-reload the store, but only after a timer tick occurs. With no timer armed, that reload path is never reached for the empty-store case.
  - file: [src/cron/service/timer.ts:687](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/timer.ts#L687)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **No cron file watcher exists on main:** Repository search found no `fs.watch`, `watchFile`, or similar watcher implementation under `src/cron`; the only watcher references are unrelated gateway/config and Gmail watcher code.
  - command: `rg -n "fs\.watch|watchFile|watch\(|watcher|jobs\.json" src/cron src/gateway docs test -g '!node_modules'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Store fast-path still caches in memory unless explicitly forced:** `ensureLoaded()` returns immediately from the in-memory store unless `forceReload` is requested, so there is no passive background reread of `jobs.json`.
  - file: [src/cron/service/store.ts:20](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/store.ts#L20)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)

## Risks / Open Questions

- If at least one enabled job is already scheduled, the 60s timer path can force-reload and partially mask the bug; the gap is specifically the zero-jobs-at-startup case described in the issue.
- This was a code inspection review only. I did not run the gateway or tests because the task required the checkout to remain byte-for-byte clean.
