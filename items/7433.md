## Summary

Keep #7433 open. Current main still initializes the bundled WhatsApp/Baileys socket without `getMessage` or `cachedGroupMetadata`, still lacks the requested group and LID event wiring, and only has narrower on-demand group metadata and LID lookup behavior. The prior maintainer/Codex discussion keeping this open still matches current main at d54d2d6b9b8aa28fb222d895c0f01f43933f4409.

## Best Possible Solution

Implement the reliability work in the bundled `@openclaw/whatsapp` plugin: add bounded retry message storage and pass `getMessage`, share a group metadata cache through `cachedGroupMetadata`, update it from hydration plus `groups.upsert`/`groups.update`/participant events, and decide whether current Baileys `signalRepository.lidMapping` persistence is sufficient or an explicit `lid-mapping.update` handler is still needed. Add focused WhatsApp plugin tests covering socket options and cache/event update behavior.
