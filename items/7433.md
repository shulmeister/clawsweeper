## Summary

Keep #7433 open. Current main still does not pass Baileys `getMessage` or `cachedGroupMetadata`, does not wire the requested group/LID event handlers, and only has narrower inbound metadata/LID lookup behavior. The existing maintainer review comment keeping this open still matches current main at 7a85c1a8226eb7d3a96906d2721a189768d3034a.

## Best Possible Solution

Keep #7433 open and implement the missing reliability work in the bundled `@openclaw/whatsapp` plugin: add bounded retry message storage and pass `getMessage`, share a group metadata cache through `cachedGroupMetadata`, update it from hydration plus `groups.upsert`/`groups.update`/participant events, and decide whether current Baileys `signalRepository.lidMapping` persistence is sufficient or an explicit `lid-mapping.update` handler is still needed. Add focused WhatsApp plugin tests for the socket options and cache update behavior.
