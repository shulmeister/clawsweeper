## Summary

Keep open. Current main still does not implement the central WhatsApp/Baileys group reliability requests: the WhatsApp socket does not pass Baileys `getMessage` or `cachedGroupMetadata`, production source has no `groups.upsert`, `groups.update`, `group-participants.update`, or `lid-mapping.update` handlers, and the existing caches only cover narrower inbound metadata/LID lookup behavior. The existing issue discussion already contains a maintainer Codex review keeping this open, and current main still matches that assessment.

## Best Possible Solution

Keep #7433 open and implement the missing reliability work in the bundled `@openclaw/whatsapp` plugin: add a bounded message-content store and pass a Baileys `getMessage` callback, create a shared group metadata cache passed as `cachedGroupMetadata`, seed/update it from group hydration plus `groups.upsert`/`groups.update`/participant events, and decide whether current Baileys `signalRepository.lidMapping` persistence is sufficient or whether an explicit `lid-mapping.update` listener is still required. Add focused WhatsApp plugin tests for the socket options and cache update behavior.
