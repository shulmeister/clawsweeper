## Summary

Keep #7433 open. Current main still lacks the requested Baileys reliability hooks for the bundled WhatsApp plugin: no socket-level `getMessage`, no socket `cachedGroupMetadata`, no proactive group cache event listeners, and no `lid-mapping.update` persistence path. The related closed WhatsApp stability item addressed quiet-socket reconnect behavior, not this retry/group metadata work.

## Best Possible Solution

Keep the issue open and implement this in the bundled `@openclaw/whatsapp` plugin: add bounded per-account message retry storage and pass it through Baileys `getMessage`; create a shared group metadata cache, wire it through `cachedGroupMetadata`, seed it from connect hydration, and maintain it from `groups.upsert`, `groups.update`, and `group-participants.update`; then decide whether current `signalRepository.lidMapping` lookup is enough or add `lid-mapping.update` persistence. Cover the behavior with focused WhatsApp plugin tests for socket options, retry lookup, group cache updates, participant updates, and LID mapping.
