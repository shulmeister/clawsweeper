## Summary

Issue #49530 matches an older 3-second gateway pre-auth handshake timeout that was explicitly raised in release 2026.3.22. Current main uses a 10-second shared handshake timeout on both the client and server handshake paths, which addresses the reported `handshake timeout` / `gateway closed (1000)` symptom from 2026.3.13.

## Evidence

- **Released fix entry:** The 2026.3.22 changelog includes: "Gateway/WS handshake: raise the default pre-auth handshake timeout to 10 seconds ... so busy local gateways stop dropping healthy CLI connections at 3 seconds." That is the same failure mode described in the issue.
  - file: [CHANGELOG.md:2446](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/CHANGELOG.md#L2446)
  - command: `nl -ba CHANGELOG.md | sed -n '2200,2450p'`
- **Current default timeout:** Current main defines the shared pre-auth handshake timeout as 10,000 ms and exposes the runtime override env noted in the changelog.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/gateway/handshake-timeouts.ts#L1)
  - command: `nl -ba src/gateway/handshake-timeouts.ts | sed -n '1,220p'`
- **Server uses shared handshake timeout:** The WebSocket server handshake timer now reads the shared pre-auth timeout before closing idle unauthenticated sockets.
  - file: [src/gateway/server/ws-connection.ts:366](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/gateway/server/ws-connection.ts#L366)

## Close Comment

Closing as implemented on current `main`. This was a Codex review of the current code and release history. The report was against `2026.3.13`, and `CHANGELOG.md:2446` shows that `2026.3.22` raised the gateway pre-auth WebSocket handshake timeout from 3 seconds to 10 seconds specifically so busy local gateways stop dropping healthy CLI connections. Current `main` has that 10-second shared timeout in `src/gateway/handshake-timeouts.ts:1`, and both the server and client handshake paths use it in `src/gateway/server/ws-connection.ts:366` and `src/gateway/client.ts:819`. One follow-up detail from the thread does not line up with the original repro: `nodes ... --json` disables CLI progress in `src/cli/nodes-cli/rpc.runtime.ts:12`, so the spinner theory is not needed to explain this report. If someone still sees the same failure on `2026.3.22+` or current `main`, please open a fresh issue with new logs and the exact current version.
