## Summary

Keep open. The report has concrete measurements for a native Ollama `gemma4:26b` first-heavy-turn latency problem, and current `main` does not include the proposed Ollama `keep_alive`/warm-retention mitigation from linked PR #66511. Main has an opt-in `localModelLean` workaround for reducing tool prompt weight, but it is explicitly experimental and off by default, so it does not resolve the reported default-path bug.

## Best Possible Solution

Keep #64541 open and use PR #66511 as the current implementation path unless maintainers choose a different design. The best fix belongs in the bundled Ollama plugin runtime: add a narrow native `/api/chat` warm-retention or `keep_alive` request option, cover the outgoing request shape in `extensions/ollama/src/stream-runtime.test.ts`, and document any user-facing config only if the behavior becomes configurable rather than a safe default.

## Evidence

- **Issue provides concrete reproduction and diagnosis context:** The issue body reports OpenClaw 2026.4.9 native `api: "ollama"` runs where first-turn latency scales from about 49s to 160s or an idle timeout as workspace/bootstrap and tool schema size increase, while later turns in the same session return to about 4.5-7.5s. The sole comment agrees the data points to first-turn prompt/tool-schema overhead, not a generic broken Ollama model.
- **Native Ollama request builder has no keep_alive field on main:** `buildOllamaChatRequest` returns only `model`, `messages`, `stream`, optional `tools`, and optional `options`. The `OllamaChatRequest` interface includes `think` but no `keep_alive` or equivalent warm-retention field.
