## Summary

Keep #70529 open. Current main still defaults the Chrome MCP existing-session transport to spawning `npx chrome-devtools-mcp@latest`, and the built-in `user` profile still routes through that existing-session path. That leaves the Desktop packaged-app failure (`spawn npx ENOENT` when the bundle lacks `npx`) unresolved. The issue also has a maintainer triage comment explicitly asking to keep it open as the Desktop-specific variant, separate from #62288 and #63926.

## Best Possible Solution

Keep this open and fix the Desktop/browser runtime so `profile="user"` existing-session attach does not depend on ambient `npx` in the packaged app. The likely product/code direction is to bundle or explicitly resolve the Chrome MCP driver for Desktop, and add a structured diagnostic/fallback to the managed `openclaw` profile when existing-session attach cannot be started. Keep #62288 for broader existing-session brittleness and #63926 for the separate visible login handoff UI.
