## Summary

Keep open. The issue is recent, concrete, non-protected, and current main still matches the reported Gateway/embedded-lock failure shape: `openclaw agent` waits for the final Gateway `agent` response with an explicit timeout, falls back to embedded execution on Gateway failure, and the embedded runner still takes the per-session file lock with a 10s default acquisition timeout. The docs mention fallback but do not document daemon-vs-embedded same-state concurrency as unsupported or guarded.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The best fix is to make `openclaw agent` handle the fast accepted `runId` path explicitly, likely by separating dispatch from completion waiting via `agent.wait`, and to stop falling back into same-state embedded mode when a daemon is already running. The embedded path should get a clear daemon/state-dir guard, separate state-dir support, or a refusal/error that explains the limitation, plus docs in `docs/cli/agent.md` and regression coverage for the 60s Gateway timeout and 10s session-lock fallback behavior.
