## Summary

Current main still preserves the reported integration failure shape: `openclaw agent` waits for the final Gateway `agent` response, falls back to same-state embedded execution on Gateway failure, and embedded runs still take the per-session transcript lock with a 10s default wait. The related #62981 tracks the lock-contention subset, but it does not cover the broader fresh-session Gateway dispatch timeout and CLI/Gateway behavior reported here.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The likely product fix is to split `openclaw agent` into fast Gateway `agent` acceptance plus explicit `agent.wait` final waiting, then either prevent same-state embedded fallback while a running Gateway owns that state directory or add an explicit alternate state-dir/local isolation path. Update the public `openclaw agent` and agent-send docs, and add regression coverage for Gateway timeout handling plus embedded fallback/session-lock contention. Coordinate with #62981 for the narrower lock-contention subset rather than closing this broader report.
