## Summary

Keep open. Current main still logs MCP startup failures at the bundle-MCP runtime layer and silently omits the failed server's tools from the materialized tool set; it does not notify the agent, block required servers, or preserve a failed-vs-expected tool diff for prompt/channel delivery.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostic path: preserve failed server records in the session MCP catalog, expose redacted diagnostics to the embedded runner as a system/developer message or equivalent prompt context, add typed `mcp.servers.<name>.required` semantics that can fail agent startup and notify the active channel, and cover failed stdio/HTTP startup with focused runtime and runner tests plus config/docs updates.
