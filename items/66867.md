## Summary

Keep #66867 open. Current `main` still has the reported race behavior: `tryRepairWritableBits` returns `false` when the repair-time stat observes already-tightened `0o700` permissions, so the invalid fallback path can still throw `Unsafe fallback OpenClaw temp dir`. The follow-up comment claiming the option-2 fix is present does not match the checked-out source or the latest release tag.

## Best Possible Solution

Keep the issue open and land a focused fix in `src/infra/tmp-openclaw-dir.ts`: when `tryRepairWritableBits` sees a trusted directory with no group/other write bits, re-run `resolveDirState(candidatePath) === "available"` instead of returning false. Add a regression test that simulates the race by making the first fallback or preferred stat look invalid and the repair-time stat look already secure.
