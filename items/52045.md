## Summary

Current `main` does not provide strong evidence that issue #52045 is fixed. The browser status route now reports an explicit `enabled` field, but the gateway-side `browser.request` path still auto-routes to a connected browser-capable node by default and returns node/browser-proxy failures without a local fallback. That still leaves a plausible path for the reporter's post-pairing mismatch, so this should stay open pending a verified repro or a landed fix.
