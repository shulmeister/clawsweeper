## Summary

Keep open. Current `main` does not implement this PR, and the existing Slack extension intentionally still prefers the user token for reads. There are also unresolved correctness/security questions with the proposed behavior change because Slack read actions accept caller-supplied targets and do not add a retry or narrower authorization check before using a broader bot token.

## Evidence

## Risks / Open Questions

- Merging as-is can regress installations where both tokens are configured but only the user token can read a given private channel or file, because current read paths do not retry with the user token after a bot-token denial.
- Closing as implemented would be incorrect: current `main` code, tests, and docs all still reflect the pre-PR behavior.
