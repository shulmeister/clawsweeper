## Summary

Keep #42847 open. Current main still prefers the Slack user token for read operations whenever it is configured, so the PR’s central bug is not implemented on main. The linked issue #42752 was closed as superseded by this PR, making this the canonical remaining item, but the PR needs maintainer judgment because bot-token reads raise security and fallback-policy questions and the patch targets the older pre-plugin file path.

## Best Possible Solution

Keep the PR open for maintainer review. If OpenClaw wants userTokenReadOnly:false to make bot token primary for reads, rebase the fix into extensions/slack/src/action-runtime.ts, add regression tests for the userTokenReadOnly:false read path and fallback behavior, and update Slack token-model docs. If maintainers prefer the current least-privilege model, close the PR manually with that product/security rationale and consider documenting the intended workaround or adding a separate explicit config for bot-token read preference.

## Risks / Open Questions

- The proposed bot-token read preference may broaden Slack data access compared with least-privilege user-token reads, as noted in the security review comment.
- A simple preference flip can regress channels where a configured bot token exists but lacks channel membership or read scope while the user token could read.
- The PR patch targets the older src/agents/tools/slack-actions.ts path; current Slack action runtime lives under extensions/slack/src/action-runtime.ts, so the work needs rebasing before it can land.
