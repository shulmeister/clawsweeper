## Summary

Current main already solves the central Slack attachment-content problem through the shared media-understanding file extraction path: Slack downloads inbound files into media, passes paths/types into context, and the generic media pass extracts text-like/PDF attachments into untrusted file blocks that are refinalized into `BodyForAgent`. The stale PR’s Slack-specific implementation is no longer the best path and had unresolved review issues around double downloads, auth behavior, HTML pollution, and misleading skip reasons. Office OpenXML extraction is not shipped, but that should be a narrow document-extractor/media-understanding follow-up rather than reviving this Slack-local PR.

## Best Possible Solution

Close the stale Slack-specific PR and keep the shipped generic media-understanding attachment extraction path. Any remaining Office OpenXML support should be implemented as a narrow document extractor or media-understanding enhancement with tests, rather than adding a separate Slack-only extraction pipeline.
