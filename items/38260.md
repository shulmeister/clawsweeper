## Summary

Keep #38260 open. Current main still routes WebChat image flows through in-process sharp/libvips on Linux/Node, and I did not find code, tests, docs, changelog, or related-item evidence proving the reported libvips-cpp SIGILL crash is fixed, reproduced as gone, or superseded.

## Best Possible Solution

Keep the issue open. A maintainer should reproduce or collect a core/backtrace on Linux npm-global Node 22, identify whether sharp's bundled libvips, system libvips selection, CPU feature dispatch, or OpenClaw's image optimization path is responsible, then ship a known-safe dependency/runtime fallback, a documented kill switch, or isolate native image processing outside the gateway process with a WebChat image-flow regression smoke.
