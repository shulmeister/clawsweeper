## Summary

Keep open. Current main still does not support `workspace` on normal route bindings: the type, zod schema, resolved route, docs, and `getReplyFromConfig` all only use agent/default workspace for route-based replies. The PR is also not obsolete because the related current capability is ACP-only (`bindings[].acp.cwd`), and the PR discussion still identifies unresolved correctness risks in the PR head around full binding scope and agent-id normalization.

## Best Possible Solution

Keep the PR open for maintainer/product review and request changes. The robust implementation should carry the matched binding's workspace through the already-resolved route, or otherwise pass the matched route context into reply execution, instead of re-deriving a binding from `sessionKey`. It should update the route binding type, strict zod schema, config help/generated schema, and docs, then add focused tests for peer, channel/account scope, non-peer binding tiers if supported, normalized agent IDs, and default agent workspace fallback.
