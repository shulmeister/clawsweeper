## Summary

Current main has related transcript safeguards, but the requested first-party reopen/startup repair is not fully implemented. Tool-call pairing repair is applied to replay/history arrays and can synthesize results in memory, and live writes flush pending synthetic tool results before later non-tool messages, but startup session-file repair does not append tail-only synthetic toolResult rows. Orphan trailing user handling still merges the orphan into the next prompt and moves the active leaf, rather than preserving the persisted tail with an explicit assistant recovery marker.

## Best Possible Solution

Keep this issue open. The best fix is a narrow first-party reopen/startup compensation seam near session-file open/load that inspects only the active transcript tail, appends synthetic error toolResult rows for unresolved eligible tail tool calls, appends one explicit assistant recovery marker for a persisted orphan-user tail, never rewrites or deletes existing rows, is idempotent across repeated reopen, and adds focused tests for missing tail result, partial tail results, orphan-user tail, healthy no-op, and repeated reopen no-op.

## Risks / Open Questions

- There are existing in-memory and live-write safeguards; a future fix should avoid double-inserting synthetic tool results when those paths already repaired a turn.
- Aborted or errored assistant turns intentionally skip synthetic missing tool results today, so the requested tail repair needs to preserve that provider-safety distinction or make the eligibility rules explicit.
