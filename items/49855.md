## Summary

Keep open. Current `main` still downloads non-image Feishu message resources with `type=file`, including inbound `video` and `media` messages, so the reported 502 fallback gap is not implemented.

## Evidence

- **Resource downloader only accepts image|file:** `downloadMessageResourceFeishu` on current `main` only accepts `type: "image" | "file"` and passes that value directly to `client.im.messageResource.get(...)`; there is no `media` mode or retry/fallback path.
  - file: [extensions/feishu/src/media.ts:276](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/feishu/src/media.ts#L276)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '276,301p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Inbound routing maps video/media to file:** `toMessageResourceType` returns `"file"` for every non-image message type, and `resolveFeishuMediaList` uses that helper for inbound downloads. That means `video`, `media`, and `file` messages all still request `type=file`.
  - file: [extensions/feishu/src/bot-content.ts:307](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/feishu/src/bot-content.ts#L307)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '307,433p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Tests lock in current behavior:** Current tests explicitly assert that `toMessageResourceType("video") === "file"` and that inbound `video`/`media` downloads call `downloadMessageResourceFeishu(..., type: "file")`. This is the opposite of the requested fix.
  - file: [extensions/feishu/src/bot.helpers.test.ts:34](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/feishu/src/bot.helpers.test.ts#L34)
