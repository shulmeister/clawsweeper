## Summary

Keep PR #69363 open. Current main still has the exact parser behavior the PR targets: `parseCliJsonl` preserves the first session id it sees, while the JSON and streaming parsers still accept any later session id. There is no hook-aware guard or regression coverage for Claude CLI `hook_started`/`hook_response` records. The already-shipped #65028 mitigation only classifies `No conversation found` as `session_expired`; it does not prevent persisting a transient hook id in the first place.

## Best Possible Solution

Land PR #69363 or an equivalent maintainer-owned patch: add a shared hook-aware session-id selection helper used by `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`, preserve hook ids only as a no-real-id fallback, and add regression tests for hook-before-init and hook-after-init ordering before running the changed agents test/typecheck lanes.
