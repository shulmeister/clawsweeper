## Summary

Current `main` already contains code and regression tests that cover the reported Docker loopback/native-approval failure path: internal native approval handlers connect as a backend `gateway-client`, direct-local backend self-pairing is explicitly bypassed, explicit `gateway.bind: "loopback"` is honored inside containers, Docker-style loopback/private-host-header pairing is regression-tested, and pending approval replay no longer blocks startup.

## Evidence

- **Native approval handlers use backend gateway client:** The approval bootstrap creates native approval handlers with the `Telegram Native Approvals (...)` display name, and the shared operator-approvals client connects as `gateway-client` in `BACKEND` mode with `operator.approvals` scope.
  - file: [src/infra/approval-handler-bootstrap.ts:64](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/infra/approval-handler-bootstrap.ts#L64)

  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Direct-local backend self-pairing is skipped:** Current handshake auth logic explicitly skips self-pairing for direct-local `gateway-client`/`BACKEND` connections using shared-secret auth or device-token auth, which matches the internal native-approval client path instead of creating repair requests.
  - file: [src/gateway/server/ws-connection/handshake-auth-helpers.ts:249](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/gateway/server/ws-connection/handshake-auth-helpers.ts#L249)
