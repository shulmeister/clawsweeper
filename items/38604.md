## Summary

Keep open. Current `main` still does not apply any default `pidsLimit` for sandbox Docker config, so sandbox container creation omits `--pids-limit` unless the user explicitly configures one. The docs show `pidsLimit: 256` only as an example, not as an enforced default.

## Evidence

- **Resolved sandbox config leaves `pidsLimit` unset by default:** `resolveSandboxDockerConfig` falls back for `image`, `readOnlyRoot`, `tmpfs`, `network`, and `capDrop`, but `pidsLimit` is only `agentDocker?.pidsLimit ?? globalDocker?.pidsLimit` with no final default.
  - file: [src/agents/sandbox/config.ts:115](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/src/agents/sandbox/config.ts#L115)
  - command: `nl -ba src/agents/sandbox/config.ts | sed -n '90,130p'`
  - sha: [ac063568d354](https://github.com/openclaw/openclaw/commit/ac063568d354c14e0a0a43bcffb9725bb3fca651)
- **Docker run args only add `--pids-limit` when configured:** Container creation adds `--pids-limit` only when `typeof params.cfg.pidsLimit === "number" && params.cfg.pidsLimit > 0`; otherwise no PID cap is passed at all.
  - file: [src/agents/sandbox/docker.ts:433](https://github.com/openclaw/openclaw/blob/ac063568d354c14e0a0a43bcffb9725bb3fca651/src/agents/sandbox/docker.ts#L433)

## Risks / Open Questions

- Sandbox Docker containers on current `main` have no enforced PID ceiling unless operators explicitly set `agents.defaults.sandbox.docker.pidsLimit` or a per-agent override.
- Because docs show an example `pidsLimit` value without code enforcing a default, maintainers could mistakenly assume this hardening is already on by default.
