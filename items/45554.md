## Summary

Keep open. Current main still lacks the requested dmScope-back-to-main reconciliation/retirement path for stale peer-keyed Telegram direct sessions. The code routes new DMs to the main key under `session.dmScope: "main"`, but existing peer-keyed rows are not archived, marked inactive, merged, or specifically cleaned up; the available cleanup command is generic stale/count/missing-transcript maintenance rather than a dmScope migration tool.

## Evidence

- **Issue discussion identifies missing lifecycle behavior:** The provided issue comments explicitly say `dmScope` controls routing going forward but there is no reconciliation pass when switching back to `main`, and recommend marking stale peer-keyed direct sessions inactive/archived rather than manual store surgery.
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Routing collapses direct DMs to main when dmScope is main:** `buildAgentPeerSessionKey` returns `buildAgentMainSessionKey` for direct peers unless dmScope is one of the per-peer/per-channel modes. That proves new DMs can route to `agent:<agent>:main`, but it does not reconcile existing peer-keyed direct rows.
