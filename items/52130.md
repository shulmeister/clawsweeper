## Summary

Keep open. Current main still rejects boolean `channels.telegram.retry.jitter`, and Telegram doctor/status inspection still treats non-`env` SecretRefs as unavailable in read-only paths, which matches the report's two main user-visible failures. There is some partial mitigation on restart storms via systemd rate limits, but the issue is not implemented well enough on `main` to close conservatively.

## Evidence

- **Telegram retry config still requires a numeric jitter:** `TelegramAccountSchemaBase` still uses shared `RetryConfigSchema`, and that schema still defines `jitter` as `z.number().min(0).max(1).optional()`. A boolean `channels.telegram.retry.jitter: true` would still fail validation on current `main`.

  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Telegram channel still consumes the strict retry schema:** Telegram config wiring still points `retry` at `RetryConfigSchema`, so there is no Telegram-specific coercion layer for legacy boolean jitter values.
  - file: [src/config/zod-schema.providers-core.ts:245](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/config/zod-schema.providers-core.ts#L245)
