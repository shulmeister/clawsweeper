## Summary

Keep open. Current main mitigates the empty-payload symptom by surfacing an incomplete-turn error, but it still does not implement the issue's requested Anthropic/tool_result behavior: preserving a final assistant reply or synthesizing a user-visible reply from a successful tool result. A prior commenter also reported the same empty assistant-message class in WebChat/Control UI, so this remains a real product bug rather than cleanup-ready stale noise.

## Evidence

- **Run path disables inline tool-result fallback:** The main embedded-agent result path calls buildEmbeddedRunPayloads with inlineToolResultsAllowed: false, so successful tool metadata is not converted into a user-visible fallback reply in the normal command path.
  - file: [src/agents/pi-embedded-runner/run.ts:1789](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/agents/pi-embedded-runner/run.ts#L1789)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1760,1865p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Payload builder only emits assistant text or enabled inline tool summaries:** buildEmbeddedRunPayloads only formats toolMetas when inlineToolResultsAllowed is true, then otherwise chooses non-empty assistantTexts or extracted assistant-visible text. Empty assistant content therefore still produces no successful tool-result-derived reply.
