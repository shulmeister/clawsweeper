## Summary

Keep open. Current `main` does not implement the requested recurring cron auto-retry behavior for low-frequency schedules: on recurring-job failure it computes a backoff, but then schedules the next run to the later of the natural schedule and the backoff window, so a daily cron still waits until the next scheduled day. The requested per-job CLI flags are also not present.

## Evidence

- **Recurring error path keeps the later schedule, not an earlier retry:** In the recurring-job error branch, `backoffNext` is computed, then `nextRunAtMs` is set via `resolveCronNextRunWithLowerBound(...)` for `cron` jobs or `Math.max(normalNext, backoffNext)` for other recurring jobs. That preserves the normal cadence instead of scheduling an earlier retry after failure.
  - file: [src/cron/service/timer.ts:510](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cron/service/timer.ts#L510)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '510,538p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Cron helper explicitly chooses the later timestamp:** `resolveCronNextRunWithLowerBound` returns `Math.max(params.naturalNext, params.lowerBoundMs)`, which means a daily cron whose natural next run is tomorrow will not retry after a short backoff today.
  - file: [src/cron/service/timer.ts:223](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cron/service/timer.ts#L223)

## Risks / Open Questions

- `docs/cli/cron.md:39` and `docs/automation/cron-jobs.md:384` currently describe recurring retry behavior that the scheduler logic does not deliver for low-frequency recurring jobs.
- Because minute-level schedules can replay after a missed slot, the gap is easiest to miss on daily/weekly crons, which is exactly the user report.
