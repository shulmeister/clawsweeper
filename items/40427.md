## Summary

Keep open. Current main partially reduces the original gateway spam through runtime snapshots and config-reloader hash checks, but it has not implemented this PR's config-warning dedupe: `createConfigIO().loadConfig()` still emits `Config warnings:` unconditionally whenever validation warnings are present. The PR also has an unresolved review edge case about clearing dedupe state on non-warning load paths, so the best maintainer path is a revised, tested patch rather than auto-closing it.

## Best Possible Solution

Keep PR #40427 open for maintainer review and request a revised implementation that centralizes config-warning fingerprinting per config path, clears state on every non-warning load path, re-logs when raw config or warning details change, and adds regression coverage for repeated loads plus warning -> invalid/missing/no-warning -> same-warning transitions. If maintainers decide runtime snapshotting is the intended product fix instead, close only with a canonical issue/PR that explicitly documents that boundary.
