## Summary

Keep #67116 open. Current `main` has `--local-time` support and a partial auto-local path when `TZ` is explicitly set, but the CLI still preserves UTC as the default timestamp rendering path for normal text output. The docs also continue to tell users to pass `--local-time`, so the requested default behavior is not fully implemented.

## Best Possible Solution

Keep the issue open and change the `openclaw logs` text-mode default to render host-local timestamps consistently, ideally with an explicit UTC opt-out for operators who depend on UTC. Update `docs/cli/logs.md` and `docs/logging.md`, and add regression coverage for `openclaw logs --follow` default output so Beijing/Asia-Shanghai timestamps no longer appear eight hours behind unless UTC is explicitly requested.

## Risks / Open Questions

- Changing the default text timestamp mode may affect users or scripts that currently expect UTC `Z` output from non-JSON `openclaw logs`; a maintainer-friendly fix should preserve a clear UTC opt-out or keep `--json` raw and stable.
- The current partial `TZ` auto-local behavior may mask the bug on some hosts, so regression coverage should include both explicit `TZ=Asia/Shanghai` and an OS-resolved timezone path without relying solely on `process.env.TZ`.
