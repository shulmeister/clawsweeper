## Summary

Keep open. Current main still generates synthetic Feishu reaction message IDs with a `:reaction:<emoji>:<uuid>` suffix and still routes those IDs into Feishu reply and typing API path parameters without normalization. The linked bug #34528 remains real, and PR #34565 is still useful rather than stale cleanup, though it likely needs a refresh for newer reply paths on main.

## Best Possible Solution

Keep this PR or a canonical successor open until Feishu reaction synthetic IDs are normalized at every API-facing boundary. The best implementation is a small tested helper that strips only the exact `:reaction:<emoji>:<uuid>` suffix before Feishu API calls, while leaving internal dedupe/session IDs unchanged, and coordinating with #34528 and the similar #39558 PR so maintainers choose one final patch.

## Risks / Open Questions

- The submitted patch strips one main reply-target computation, but current main has additional API-facing paths such as ACP init failure replies, streaming cards, and media replies that should be audited before merge.
- The fix should preserve synthetic suffixed IDs for internal dedupe and session tracking while stripping only Feishu API-facing `message_id` values.
- No live Feishu credentials were used in this read-only review; the source path plus linked 400 log are enough to keep the PR open, but live verification would still be useful before landing.
