## Summary

Keep open. Current main still has the same observable bug in the current GitHub Copilot plugin path: implicit Copilot catalog discovery resolves a GitHub token by listing stored profiles and selecting `profileIds[0]`, without using `resolveAuthProfileOrder` or `auth.order`. The related PRs/comments are useful context, especially open PR #48682, but the fix is not present in current main.

## Evidence

- **Copilot discovery calls first-token helper:** The bundled GitHub Copilot provider catalog path calls `resolveFirstGithubToken` before exchanging the GitHub token for the Copilot API token/base URL.
  - file: [extensions/github-copilot/index.ts:123](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/extensions/github-copilot/index.ts#L123)
  - command: `nl -ba extensions/github-copilot/index.ts | sed -n '112,149p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **First stored profile still wins:** `resolveFirstGithubToken` lists profiles with `listProfilesForProvider(authStore, PROVIDER_ID)` and then selects `profileIds[0]`; this is raw profile/object order and ignores `auth.order`.
  - file: [extensions/github-copilot/auth.ts:21](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/extensions/github-copilot/auth.ts#L21)
