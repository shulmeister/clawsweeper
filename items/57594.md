## Summary

Current main still appears to reject Zalo proactive outbound targets that are non-numeric hex-like chat IDs before reaching the Zalo send API. The Zalo plugin wires `targetResolver.looksLikeId` to the shared numeric-only helper, while the outbound target resolver only treats id-like inputs as direct normalized targets. A non-numeric chat ID falls through to directory lookup/plugin fallback and can produce the reported `Unknown target ... Hint: <chatId>` error. The send layer itself accepts string `chat_id`, so the remaining problem is target resolution/validation, matching the issue.

## Best Possible Solution

Keep the issue open and fix the Zalo plugin-owned messaging target contract. The narrow product/code direction is to add a Zalo-specific `looksLikeId` predicate, or a `targetResolver.resolveTarget` fallback, that accepts valid Bot Platform string `chat_id` values such as hex-like IDs while preserving prefix normalization and directory behavior. Add a focused regression test for `openclaw message send --channel zalo --target 3becaa50ae12474c1e03` resolving as a normalized direct Zalo target, and update `docs/channels/zalo.md` to document non-numeric chat IDs.
