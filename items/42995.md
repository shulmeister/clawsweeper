## Summary

Current main already supports cross-channel DM memory through existing session routing: direct messages share one session by default, and per-user cross-channel sharing is supported with `session.dmScope: "per-peer"` plus `session.identityLinks`. The PR's `crossChannelMemory` / `--share-memory` surface is therefore redundant rather than filling a missing capability.

## Risks / Open Questions

- The PR also proposes a convenience per-agent UX; closing it treats that UX as redundant rather than preserving it as a separate product choice.
- The docs around `dmScope` and `identityLinks` may still be easy to miss, so similar requests may recur even though the capability already exists.

## Close Comment

Closing after Codex review. Current OpenClaw already ships this capability through the existing session-routing model: direct messages share one session by default (`docs/concepts/session.md:16`, `src/routing/resolve-route.test.ts:101`), and the supported way to keep cross-channel sharing while isolating by user is `session.dmScope: "per-peer"` plus `session.identityLinks` (`docs/concepts/session.md:45`, `src/routing/session-key.ts:142`, `src/routing/resolve-route.test.ts:181`). I also verified the same behavior is present in the released `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Since the requested behavior is already implemented on main and shipped, this PR’s `crossChannelMemory` / `--share-memory` surface is redundant, so I’m closing it as implemented on main.
