## Summary

Current `main` does not fully implement the requested fix. `src/agents/model-catalog.ts` still builds the catalog from PI discovery plus provider-plugin augment hooks only, and never performs a general merge of arbitrary `models.providers.*.models` entries for non-plugin custom providers. Telegram `/models` can now surface a custom provider indirectly when it is already referenced by defaults/allowlist config, but the underlying catalog gap remains and still affects code paths that rely on `loadModelCatalog()` directly.

## Evidence

- **Global model catalog still skips arbitrary configured custom providers:** `loadModelCatalog()` reads PI registry entries, then only calls `augmentModelCatalogWithProviderPlugins(...)`. There is no general merge over `cfg.models.providers` for non-plugin custom providers.
  - file: [src/agents/model-catalog.ts:162](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/agents/model-catalog.ts#L162)
  - command: `nl -ba src/agents/model-catalog.ts | sed -n '150,210p'`
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Provider-plugin augmentation is plugin-hook only:** `augmentModelCatalogWithProviderPlugins()` just iterates resolved provider plugins and calls each plugin's `augmentModelCatalog` hook. A custom OpenAI-compatible provider with no owning plugin never enters this path.
  - file: [src/plugins/provider-runtime.ts:867](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/plugins/provider-runtime.ts#L867)

- **Configured-provider helper is scoped to named provider plugins:** `readConfiguredProviderCatalogEntries()` can read `models.providers.<id>.models`, but only for a specific `providerId` passed by a provider plugin. That helps bundled/plugin-backed providers, not arbitrary custom provider ids.
  - file: [src/plugin-sdk/provider-catalog-shared.ts:61](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/plugin-sdk/provider-catalog-shared.ts#L61)

That can make a custom provider appear if it is already referenced there, even though it is still absent from the global model catalog.

## Risks / Open Questions

- Closing this as implemented would hide a still-open catalog consistency bug for non-plugin custom providers.
- The original Telegram symptom may appear fixed only when the custom provider is also referenced in defaults/allowlist config; providers declared only under `models.providers` are still not generally merged into the catalog.
- Capability-sensitive flows that rely on `loadModelCatalog()` can still mis-handle configured custom models because their metadata never enters the catalog.
