## Summary

Current main still does not preserve Slack CodedError structured fields in delivery queue recovery. The queue recovery path formats caught errors through the generic infra formatter, which only uses Error.message and cause-chain text, so Slack err.data.needed and err.data.response_metadata.scopes remain unavailable in lastError and recovery logs.

## Evidence

- **Recovery still uses generic error formatting:** recoverPendingDeliveries delegates failures to drainQueuedEntry, which converts the caught value with formatErrorMessage(err) before logging and persisting the failure. There is no Slack-specific structured field extraction in this path.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:210](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/infra/outbound/delivery-queue-recovery.ts#L210)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '1,470p'`
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **Generic formatter drops Slack data fields:** formatErrorMessage returns err.message for Error instances and only appends nested cause messages. It does not inspect extra object fields such as data.needed, data.response_metadata.scopes, acceptedScopes, or provided.
  - file: [src/infra/errors.ts:68](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/infra/errors.ts#L68)
