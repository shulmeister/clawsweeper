## Summary

Keep open. Current `main` has websocket cleanup and shutdown-hardening code, but the available evidence does not show that issue #56215's reported long-running `CLOSE_WAIT`/`FIN_WAIT_2` accumulation on Windows is fixed. The report is also too recent for `stale_insufficient_info`, and a follow-up comment says the latest version still shows `CLOSE_WAIT`, so conservative maintainer cleanup should not auto-close it.

## Evidence

- **Per-connection websocket close path exists on main:** The gateway websocket connection handler removes the client from the tracked set, releases the preauth budget, and calls `socket.close(...)` both from the local `close()` helper and again from the socket `close` event path. This shows explicit connection-lifecycle cleanup exists, but it is code inspection only, not proof that long-running Windows sockets cannot still accumulate in `CLOSE_WAIT`/`FIN_WAIT_2`.
