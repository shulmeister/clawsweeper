## Summary

Keep open. Current main has related subagent announce and stale-run fixes, but the specific gateway-unresponsive/completion-announcement timeout path still exists: completion announces still try a direct `agent` gateway call first, treat `gateway timeout` as transient, retry it, and tests still assert repeated 120s timeout retries. The provided related context also does not identify a merged/open canonical PR that closes this exact Windows alive-but-unresponsive gateway failure.

## Best Possible Solution

Keep this issue open as the focused Windows alive-but-unresponsive gateway/subagent completion timeout report. The best maintainer path is to reproduce on a current release or add a stress regression around concurrent subagent completion announces, then decide whether completion delivery should fail over faster, avoid direct parent-agent calls when the gateway is saturated, expose an unresponsive-gateway health watchdog for supervised Windows installs, or reduce lane/provider starvation. Cross-link #47975 for stale subagent/session state and #71699 for the separate Windows hard-crash path, but do not close this issue until this exact timeout/unresponsive behavior is fixed or intentionally superseded by a narrower canonical tracker.
