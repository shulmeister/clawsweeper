## Summary

Keep #37804 open. Current main has strong partial fixes for configured provider models, including runtime fallback resolution and `models list` visibility, but the broader RFC is still not fully implemented as one shared resolved model descriptor consumed consistently by status, list, runtime, availability, capability gates, and docs. The latest provided issue comment already reached the same conclusion, and current main at d74f897c still supports that keep-open decision.

## Best Possible Solution

Keep #37804 open as the canonical RFC until maintainers either finish the remaining shared resolution work or explicitly narrow the product direction. The likely path is a general resolved model descriptor carrying identity, transport, capabilities, provenance, and availability state, followed by migration of `models status`, `models list`, runtime execution, model picker/auto-reply, and capability gates to consume it consistently.

## Evidence

- **Issue discussion scopes this as broader than one fix:** The provided comments frame the desired invariant as a single canonical resolution result shared by runtime, `models status`, `models list --all`, and capability gates. who96's implementation-plan comment explicitly described phased work, with consumer migration after the initial shared fact-source extraction.
- **Related PR #38173 was phase 1 only:** The provided PR context says #38173 was limited to the `src/agents` fact-source layer and explicitly deferred CLI / auto-reply / picker consumer migration, so it does not close the whole RFC.
- **Runtime fallback exists for configured provider models:** `resolveConfiguredFallbackModel` builds a runtime model from `models.providers` entries, and `resolveModelWithRegistry` falls back to it after explicit registry and plugin dynamic resolution miss. This is meaningful progress, but it is one runtime path rather than a shared cross-surface descriptor.
