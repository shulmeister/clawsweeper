## Summary

Keep #37804 open. Current main has substantial partial support for config-injected and forward-compatible models, but the RFC’s central invariant is still not implemented as one shared resolved model descriptor consumed consistently by status, list, runtime, capability gates, and docs.

## Best Possible Solution

Keep #37804 open as the umbrella design tracker until maintainers either narrow the accepted invariant or land/link canonical follow-up issues for a shared resolved model descriptor with identity, transport, capabilities, provenance, availability, status/list consumers, capability gates, and docs.

## Evidence

- **RFC scope and related discussion:** The issue body asks for config-injected provider models to behave as first-class forward-compat models across status, list, runtime, availability/auth heuristics, thinking gates, and docs. The follow-up comment and PR #38173 frame the implementation as phased work, with CLI/auto-reply/picker migration deferred rather than complete.
- **Runtime has config-backed fallback resolution:** Runtime resolution can synthesize a fallback model from models.providers metadata when registry/plugin lookup does not resolve the model, preserving configured reasoning/input/token metadata through normalizeResolvedModel. This is useful partial implementation, but it returns a runtime model rather than a cross-surface descriptor with provenance/availability state.
