## Summary

Keep #12429 open. Current main has minimal single-pod Kubernetes deployment support and CIDR-based first-time node auto-approval, but it still lacks the requested Kubernetes ServiceAccount Trust authentication path for autoscaled worker/node-host pods. The linked implementation PR #12434 was closed unmerged, and the broader distributed runtime RFC #42026 does not specifically replace this K8s SA Trust work.

## Best Possible Solution

Keep the issue open until OpenClaw implements an opt-in Kubernetes ServiceAccount Trust path across gateway config/schema, WebSocket connect protocol, GatewayClient/node-host credential handling, TokenReview validation, tests, docs, and deployment assets, or until maintainers explicitly mark a narrower canonical distributed-runtime/K8s-auth issue as the replacement. #42026 can inform the architecture, but it should not silently close this specific SA Trust request.

## Risks / Open Questions

- This is security-sensitive gateway authentication and node-pairing bypass work; any implementation should fail closed and receive maintainer/security review.
- Treating this as ClawHub-only would be premature because the current plugin surface exposes paired node runtime APIs, plugin HTTP routes, providers, channels, and tools, but not a safe replacement for connect-time gateway protocol authentication.
- The existing CIDR auto-approval feature may help controlled private networks but is not equivalent to Kubernetes ServiceAccount identity and TokenReview validation for autoscaled pods.
