## Summary

Keep #12429 open. Current main still lacks an opt-in Kubernetes ServiceAccount Trust authentication path for autoscaled node/worker pods: gateway auth config/schema, WebSocket connect auth, handshake authorization, node-host runtime/CLI, and K8s deployment assets still use existing token/password/device-pairing or trusted-CIDR paths. The linked implementation PR #12434 is closed unmerged, and the broader distributed runtime RFC #42026 does not ship or supersede this specific TokenReview-based pairing bypass request.

## Best Possible Solution

Keep #12429 open until OpenClaw either implements an opt-in Kubernetes ServiceAccount Trust path or maintainers nominate a narrower canonical issue. A complete solution likely belongs in core gateway config/types/schema, the gateway protocol connect auth payload, GatewayClient/node-host credential handling, fail-closed TokenReview validation with namespace/service account/audience allowlisting, explicit pairing-skip policy for validated node identities, regression tests, docs, and deployment assets. Use #12434 as prior implementation evidence and #42026 as architecture input, but neither closes the current gap.

## Risks / Open Questions

- This is security-sensitive because a valid implementation would bypass normal node device pairing; it needs fail-closed TokenReview validation, namespace/service-account/audience allowlisting, and maintainer/security review.
- The existing trusted-CIDR node auto-approval path is not equivalent to Kubernetes ServiceAccount Trust and does not solve headless autoscaling across dynamic pods.
- The prior PR #12434 is useful prior art, but it was closed unmerged and its review context raised security/operational concerns, so it should inform a new design rather than be treated as shipped.
