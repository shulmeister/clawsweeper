## Summary

Current main still does not implement the requested Kubernetes ServiceAccount Trust pairing path. OpenClaw has a minimal Kustomize gateway deployment and an optional trusted-CIDR node auto-approval path, but no TokenReview validator, no K8s service-account token field in the gateway connect protocol, no node-host service-account-token reader or `--k8s-trust` flag, and no worker Deployment/Helm chart for autoscaled node hosts. The linked implementation PR #12434 was closed unmerged, and the broader open RFC #42026 is related but does not precisely supersede this narrow auth/pairing request.

## Best Possible Solution

Keep this issue open until OpenClaw either implements a secure headless Kubernetes worker pairing path in the gateway/node protocol, config, node-host CLI/runtime, docs, and deployment assets, or a maintainer explicitly redirects it to a narrower canonical design issue. If plugin ownership is desired, core likely needs a documented gateway handshake/auth extension point first; current plugin surfaces cover HTTP routes/RPC/capabilities, not WS connect authentication replacement.
