## Summary

Keep open. Current `main` still has an unbounded Chrome MCP bootstrap path for existing-session profiles, so a hung `npx chrome-devtools-mcp` startup can still block browser-control reachability and tab routes indefinitely rather than failing fast and cleaning up.

## Evidence

- **Chrome MCP bootstrap still has no connect timeout:** `createRealSession()` uses raw `StdioClientTransport` and awaits `client.connect(transport)` inside `ready` with no timeout wrapper. `getSession()` then awaits `session.ready` directly, so a stalled MCP initialize path can hang the request path indefinitely.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:225](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/browser/src/browser/chrome-mcp.ts#L225)
  - command: `nl -ba extensions/browser/src/browser/chrome-mcp.ts | sed -n '220,380p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Reachability probes ignore their timeout for Chrome MCP:** `isReachable(timeoutMs)` accepts a timeout, but for Chrome MCP it just calls `listChromeMcpTabs(profile.name, profile.userDataDir)` and returns true on success. The timeout parameter is not applied anywhere on that path.
  - file: [extensions/browser/src/browser/server-context.availability.ts:74](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/browser/src/browser/server-context.availability.ts#L74)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '70,100p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **HTTP routes rely on short reachability probes that the Chrome MCP path cannot honor:** `GET /` calls `isHttpReachable(300)` and `isReachable(600)`, and `/tabs` calls `isReachable(300)`, but those limits do not constrain the Chrome MCP attach path because of the availability code above. That matches the report’s symptom of loopback routes hanging until the outer HTTP client times out.
  - file: [extensions/browser/src/browser/routes/basic.ts:82](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/browser/src/browser/routes/basic.ts#L82)
  - command: `nl -ba extensions/browser/src/browser/routes/basic.ts | sed -n '70,160p'; nl -ba extensions/browser/src/browser/routes/tabs.ts | sed -n '68,164p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Browser code bypasses OpenClaw's safer stdio transport:** The repo already has `OpenClawStdioClientTransport`, which spawns in a disposable process group and kills the process tree on close timeout, but the browser Chrome MCP path imports the SDK transport directly instead of using that wrapper.
  - file: [src/agents/mcp-stdio-transport.ts:42](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/mcp-stdio-transport.ts#L42)

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Existing tests cover post-connect tool hangs, not hung session bootstrap:** Current Chrome MCP tests verify reconnect after a stuck `click` or `navigate_page` tool call, but there is no coverage for a stall during initial `client.connect()` / session startup.
  - file: [extensions/browser/src/browser/chrome-mcp.test.ts:430](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/browser/src/browser/chrome-mcp.test.ts#L430)

## Risks / Open Questions

- The issue report pins the hang to session startup inside Gateway; current code analysis matches that failure mode, but I did not run a live reproduction because this review is read-only.
- There is a recent adjacent fix in `CHANGELOG.md:2162` for waiting until attached tabs become usable after Chrome MCP attach, but that does not address a bootstrap path that never returns from `client.connect()`.
- `gh` inspection beyond the provided issue context was blocked here because `GH_TOKEN` is not configured, so I relied on the supplied GitHub context plus local code/history.
