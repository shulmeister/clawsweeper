## Summary

Keep open. Current main has mitigations for Chrome MCP existing-session hangs, including a 30s handshake guard, short status probes, tool-call timeout handling for some actions, and a documented mcpCommand/mcpArgs escape hatch. It still does not fully close the reported default-path hang and orphan-cleanup concern: existing-session still defaults to spawning npx chrome-devtools-mcp@latest through the SDK StdioClientTransport, /tabs calls isReachable(300) but the Chrome MCP branch drops that timeout, and adjacent tests do not prove a hung default spawn/connect path is bounded and cleaned up without orphaned subprocesses. The Windows Defender comments are useful platform-specific context, but they do not supersede the original macOS/Node 25 report.

## Best Possible Solution

Keep this open and implement a focused browser-plugin fix: propagate route timeout and AbortSignal through Chrome MCP availability/list calls, use or adapt the process-tree-aware OpenClaw stdio transport for browser Chrome MCP subprocesses, close/kill failed pending sessions reliably, and add a regression that simulates a hung npx/client.connect/list_pages path through /tabs proving bounded response time and subprocess cleanup. Keep the mcpCommand/mcpArgs escape hatch, and document Windows Defender-specific guidance separately if maintainers confirm that platform path.
