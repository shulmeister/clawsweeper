## Summary

Keep PR #66931 open. Current main still has the exact behavior the PR targets: non-batch memory indexing falls back to the hardcoded concurrency of 4, so a configured `agents.defaults.memorySearch.remote.batch.concurrency` is ignored when the provider has no batch API, including the bundled Ollama embedding adapter. The PR is a plausible focused fix for open bug #66822, though it should get regression coverage and maintainer review of the unbounded-concurrency concern before merge.

## Best Possible Solution

Keep the PR open and refine it into a small, tested fix: make `getIndexConcurrency()` honor explicitly configured concurrency when provider batch mode is unavailable, add coverage for the no-`batchEmbed` path, and resolve the maximum-concurrency policy before merge. If maintainers prefer clearer config semantics, the alternative is a dedicated memory indexing concurrency setting that covers inline embeddings while preserving `remote.batch.concurrency` for provider batch APIs.
