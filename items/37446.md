## Summary

Keep open. Current main has useful partial protections, but the core `sessions_spawn` retry/idempotency gap remains: each call creates a fresh child session and fresh gateway `agent` idempotency key, so a parent retry after an ambiguous timeout can still spawn duplicate work. Gateway agent dedupe works only when the same idempotency key is reused, and `sessions_spawn` currently exposes no caller-supplied or deterministic recovery key. The linked ClawHub multi-agent deployment skill is a useful operational workaround, but it cannot close the remaining core sub-agent recovery semantics gap.

## Best Possible Solution

Keep this issue open until core sub-agent orchestration has an idempotent recovery contract. The likely product/code path is to add a stable caller-supplied or deterministic `sessions_spawn` recovery key, reuse or return the existing child session/run/task when the same key is replayed, and add regression coverage for lost-ack/parent-timeout retries. The ClawHub skill can remain recommended operational guidance, but it should not replace the missing core recovery API.
