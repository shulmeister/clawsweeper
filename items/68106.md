## Summary

Keep open. Current `main` still swallows workflow-path resolution errors and falls back to inline pipeline parsing, so the reported bug is still present. This PR also has an unresolved regression risk: its suffix-only heuristic would treat valid single-step pipelines that merely end in `*.json`/`*.lobster`/`*.yaml`/`*.yml` as workflow paths and throw path errors instead of executing them.

## Evidence

- **Current main still masks workflow path errors:** `detectWorkflowFile()` catches every `resolveWorkflowFile()` failure and returns `null`, which sends missing relative workflow paths down the inline `pipeline` execution path instead of surfacing the original file/path error.
  - file: [extensions/lobster/src/lobster-runner.ts:199](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/lobster/src/lobster-runner.ts#L199)

That matches the bug report's expectation that a missing `.lobster` path should fail as a path/file problem, not as an inline command parse.
  - file: [docs/tools/lobster.md:266](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/tools/lobster.md#L266)
