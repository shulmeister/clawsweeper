## Summary

Current main already intercepts bare bundled plugin ids like `acpx` before any ClawHub lookup, and the shipped ACPX plugin is bundled/enabled by default. The fix is present in the current released tree as well.

## Close Comment

Closing from a Codex review because current OpenClaw already implements this on `main`, and the same fix is present in `v2026.4.22`.

Concrete evidence:
- `src/cli/plugin-install-plan.ts:48` resolves bare install specs like `acpx` to a bundled plugin source before npm/ClawHub fallback.
- `src/cli/plugins-install-command.ts:484` runs that bundled resolution before `parseClawHubPluginSpec(raw)`, so `openclaw plugins install acpx` no longer has to fall through to `clawhub:acpx`.
- `src/plugins/bundled-dir.ts:111` now resolves bundled plugins from the running installed CLI package root and recovers from stale bundled-dir overrides; `src/plugins/bundled-dir.test.ts:299` covers those packaged-install scenarios.
- The bundled ACPX plugin still exists and is enabled by default in `extensions/acpx/openclaw.plugin.json:2`.

Because the current released tree already contains that behavior, there is nothing left to track in this issue. If you still see the skill redirect on a newer build, please open a fresh issue with the exact `openclaw --version` output and any relevant plugin-path env overrides.
