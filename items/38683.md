## Summary

Keep #38683 open. Current main still enforces the explicit Telegram contract that `dmPolicy="open"` requires an effective `allowFrom` wildcard, so the PR's startup/config-load auto-heal behavior is not implemented or superseded. The PR remains a real product/security decision because the discussion already identified account-scoped widening risks that need maintainer judgment.

## Best Possible Solution

Keep the PR open for maintainer judgment. Maintainers should decide whether OpenClaw should keep the explicit `dmPolicy="open"` plus `allowFrom: ["*"]` contract and rely on setup/doctor guidance, or accept a narrow Telegram startup/config-load normalization that mutates only the intended account, emits a testable warning, and includes tests, docs, and changelog.

## Risks / Open Questions

- Auto-healing `allowFrom` is security-sensitive: if implemented at the wrong inheritance level, it can broaden access for sibling Telegram accounts that inherit the parent allowlist.
- The PR changes an explicit documented validation contract, so it needs maintainer product approval plus docs/changelog updates if accepted.
- Regression coverage should include missing, empty, and non-wildcard top-level `allowFrom`, account-scoped `dmPolicy="open"`, inherited allowFrom behavior, and no sibling-account widening.
