## Summary

Keep open. Current `main` still renders the Control UI context badge from `session.totalTokens` over `session.contextTokens`, labels it as current `context used`, and only suppresses the badge when `totalTokensFresh === false`, so the issue's stale/unknown-freshness concern is not clearly fixed.

## Evidence

- **Issue report is concrete and recent:** The provided issue body and follow-up comment document an observed `100% context used 468.2k / 48k` badge on OpenClaw `2026.4.1`, with the reporter explicitly tying the larger number to historical accumulated token totals rather than the live context window.
- **Current UI still mixes totalTokens with contextTokens under a context-used label:** `getContextNoticeViewModel` reads `session.totalTokens` as `used`, reads `session.contextTokens` as `limit`, and renders `${model.pct}% context used` plus the `used / limit` detail. That is the exact display path the issue describes.

  - command: `nl -ba ui/src/ui/chat/context-notice.ts | sed -n '44,118p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Freshness gate is still permissive for unknown freshness:** The badge is hidden only when `session.totalTokensFresh === false`. Sessions with `totalTokensFresh` unset still pass through, which matches the report's concern about unknown/stale totals being rendered as live context usage.
  - file: [ui/src/ui/chat/context-notice.ts:53](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/ui/src/ui/chat/context-notice.ts#L53)
