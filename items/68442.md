## Summary

Keep #68442 open. Current main has adjacent recovery paths for chat error events and some malformed tool-argument repair, but I did not find source, tests, changelog, or history proving that direct Anthropic `edit` tool calls with non-ASCII `oldText`/`newText` no longer trigger the reported JSON parse failure. The related #61076 remains a related-but-not-equivalent open report rather than a clear canonical duplicate.

## Best Possible Solution

Keep the issue open and add a targeted fix plus regression coverage: exercise direct Anthropic `input_json_delta` streaming for `edit` arguments containing umlauts, arrows, emoji, and box-drawing characters; ensure valid UTF-8 never causes a parse failure; and ensure any malformed provider frame produces a terminal chat error/final event that clears Control UI run state without requiring a Gateway restart.
