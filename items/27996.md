## Summary

Keep #27996 open. Current main recomputes `nextRunAtMs` for normal `openclaw cron edit` / `cron.update` schedule patches, but the direct `jobs.json` edit path described in the issue comments is still not fixed: startup/load preserves still-future runtime state and does not detect schedule or timezone drift against the persisted sidecar state.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for external cron definition edits. A conservative fix would persist a schedule fingerprint derived from schedule kind/expression/timezone/every/at/stagger, or otherwise compare the stored next run against the current schedule on startup and force reloads, then recompute and persist when the schedule changed. The regression test should cover `jobs.json` schedule edits with an existing `jobs-state.json` future `nextRunAtMs`, plus normal `cron.update` behavior to avoid regressing the already-correct CLI/API path. The related watcher/reload work is adjacent but not sufficient by itself unless it also handles sidecar state drift.
