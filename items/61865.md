## Summary

Current main does not satisfy the issue. The shared `/status` renderer still trusts `entry.totalTokens` or falls back to `inputTokens + outputTokens` without checking `totalTokensFresh`, and its transcript fallback only replaces the displayed total when the transcript-derived value is larger. That leaves the reported stale-pre-compaction shape possible for already-compacted sessions.

## Evidence

- **Status display ignores freshness:** `buildStatusMessage` initializes `totalTokens` from `entry.totalTokens ?? inputTokens + outputTokens` with no `totalTokensFresh` guard, so stale session-store totals can still be rendered as current context usage.
  - file: [src/status/status-message.ts:551](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/status/status-message.ts#L551)
  - command: `nl -ba src/status/status-message.ts | sed -n '468,760p'`
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **Transcript fallback cannot lower stale totals:** When transcript usage is available, `/status` only assigns it if the transcript candidate is greater than the current `totalTokens`. A smaller post-compaction prompt total cannot replace a larger stale pre-compaction total.
  - file: [src/status/status-message.ts:564](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/status/status-message.ts#L564)

- **Stale value is rendered directly:** The context line formats the computed `totalTokens` unconditionally and displays it beside the compaction count, so a stale total can still appear as a current `Context:` value even when `Compactions` is nonzero.
  - file: [src/status/status-message.ts:727](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/status/status-message.ts#L727)
