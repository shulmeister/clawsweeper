## Summary

Keep open. Current `main` has multiple launchd hardening changes and recovery paths, but I did not find strong enough evidence that this exact report is obsolete. The code and docs say the macOS Gateway LaunchAgent should stay alive under launchd, yet there is no direct coverage for the reported reproduction of sending an external `SIGTERM` to the running gateway and verifying automatic respawn instead of `on-demand-only` state.

## Evidence

- **Current plist generation expects always-on launchd behavior:** The generated LaunchAgent plist still sets both `RunAtLoad` and `KeepAlive` to `true`, which matches the intended always-on behavior described in the issue.
  - file: [src/daemon/launchd-plist.ts:106](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/launchd-plist.ts#L106)
  - command: `sed -n '1,260p' src/daemon/launchd-plist.ts`
- **Current launchd management explicitly clears disabled state on repair/install/restart:** `bootstrapLaunchAgentOrThrow()` runs `launchctl enable` before `bootstrap`, `installLaunchAgent()` uses that path, and `restartLaunchAgent()` also re-enables the service before `kickstart`. That is relevant because the issue symptom (`on-demand-only mode`) is consistent with a disabled LaunchAgent.
  - file: [src/daemon/launchd.ts:188](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/launchd.ts#L188)
  - command: `sed -n '260,920p' src/daemon/launchd.ts`
- **Current CLI has explicit recovery for installed-but-unloaded LaunchAgents:** `recoverInstalledLaunchAgent()` re-bootstraps an installed LaunchAgent on macOS during `gateway start` and `gateway restart`, showing maintainers have already addressed related unloaded-launchd states.
  - file: [src/cli/daemon-cli/launchd-recovery.ts:1](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/daemon-cli/launchd-recovery.ts#L1)
  - command: `sed -n '1,220p' src/cli/daemon-cli/launchd-recovery.ts`
- **Release notes show related launchd fixes shipped after the reported version:** The changelog for `2026.4.10` mentions keeping `gateway stop` persistent while re-enabling the LaunchAgent on explicit restart or repair, and `2026.4.5` mentions letting launchd `KeepAlive` own supervised restarts again plus re-bootstrapping unloaded LaunchAgents.
  - file: [CHANGELOG.md:988](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/CHANGELOG.md#L988)
  - command: `nl -ba CHANGELOG.md | sed -n '984,992p' && nl -ba CHANGELOG.md | sed -n '1267,1272p'`
- **No direct regression test for the reported SIGTERM path:** The launchd integration test covers install/restart/stop/start flows, but it does not kill the running launchd-managed process with `SIGTERM` and verify automatic respawn. That gap makes closure too speculative.
  - file: [src/daemon/launchd.integration.e2e.test.ts:1](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/launchd.integration.e2e.test.ts#L1)
  - command: `sed -n '1,260p' src/daemon/launchd.integration.e2e.test.ts`

## Risks / Open Questions

- The issue's exact symptom (`pending spawn, domain in on-demand-only mode`) strongly suggests a persisted disabled-state edge case, but I could not prove the reporter's reproduction is fully covered by the current code paths.
- This review environment is Linux and `pnpm` is unavailable, so I could not run the macOS reproduction path or execute the existing launchd integration test.
- Closing as implemented would rely on inference from adjacent fixes rather than a direct proof that external `SIGTERM` respawns correctly on current `main`.
