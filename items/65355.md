## Summary

Keep open. Current main still matches the reported bug: `gateway probe --timeout <large>` routes active `configRemote` targets through `resolveProbeBudgetMs`, and any non-loopback target is still capped to `Math.min(1500, overallMs)`. The existing tests explicitly preserve that non-local cap, and the latest release tag still contains the same behavior.

## Best Possible Solution

Keep this issue open and fix the gateway probe budget policy for active remote targets. The likely maintainer path is to let active `configRemote` and explicit remote URLs honor the caller's `--timeout` budget, or introduce a much larger documented remote cap, while keeping the intentional short cap for inactive local loopback probes. Add regression coverage that `gateway.mode="remote"` with a configured remote URL passes the expected timeout to `probeGateway`, and update CLI/docs output if per-target effective budgets remain different from the overall budget.
