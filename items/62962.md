## Summary

Keep open. Current main still lacks the requested schema-driven tool security classification properties on the core tool type, plugin tool API, adapter layer, and conformance snapshot. Existing behavior covers related safety concerns through allow/deny policy, owner-only tools, plugin before_tool_call approvals, mutation heuristics, and provider-level parallel_tool_calls, but those are not the standardized per-tool contract requested here.

## Best Possible Solution

Keep this issue open. The best path is a backwards-compatible core/public SDK tool metadata contract: add the classification type to the core tool surface, preserve it through wrappers, plugin/channel tool APIs, adapters, and catalogs, annotate core tools, extend TOOL_POLICY_CONFORMANCE and tests, document the contract, then incrementally consume the metadata for approval wording, interrupt behavior, and scheduling while retaining current heuristics as conservative fallbacks.
