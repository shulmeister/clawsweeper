## Summary

Keep open. Current main has session-tool visibility guards, but the requested cron-side per-agent isolation is still missing: cron list/update/remove/run/runs do not receive or enforce the calling agent id, and the service still operates on the full cron job store by job id.

## Evidence

- **Cron tool does not scope non-add operations:** createCronTool builds gateway options without caller agent identity, sends cron.list with only includeDisabled, and sends update/remove/run/runs with id/patch/mode only. The caller session is only used to stamp agentId/sessionKey during add.
  - file: [src/agents/tools/cron-tool.ts:526](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/agents/tools/cron-tool.ts#L526)
  - command: `nl -ba src/agents/tools/cron-tool.ts | sed -n '430,730p'`
  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Gateway cron.list returns unscoped cron page:** The cron.list handler validates request params, calls context.cron.listPage with pagination/query/enabled/sort options, then returns page jobs and delivery previews. It does not derive a caller agentId or apply an agent ownership filter.
  - file: [src/gateway/server-methods/cron.ts:137](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/gateway/server-methods/cron.ts#L137)
