## Summary

Keep open. Current `main` does not give strong evidence that WebChat cross-channel message refresh is fully fixed: chat refresh still depends on `session.message` events whose `sessionKey` must exactly equal the UI's active `sessionKey`, and there is still no idle `chat.history` polling fallback.

## Evidence

- **No chat polling fallback:** The Control UI polling module only defines polling for nodes, logs, and debug. There is no chat polling path analogous to logs polling.
  - file: [ui/src/ui/app-polling.ts:8](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/ui/src/ui/app-polling.ts#L8)
  - command: `nl -ba ui/src/ui/app-polling.ts | sed -n '1,120p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Session message reload requires exact key equality:** `handleSessionMessageGatewayEvent` returns unless `payload.sessionKey === host.sessionKey`, so refresh still hinges on exact session-key matching.
  - file: [ui/src/ui/app-gateway.ts:481](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/ui/src/ui/app-gateway.ts#L481)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)

  - file: [src/gateway/session-message-events.test.ts:213](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/gateway/session-message-events.test.ts#L213)
