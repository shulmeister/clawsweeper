## Summary

Current `main` no longer silently ignores `attachAs.mountPath` for `runtime:"subagent"`; it documents the field as a hint, surfaces the real attachment location, and returns attachment receipt metadata to the caller.

## Evidence

- **Schema now documents hint-only semantics:** `attachAs.mountPath` is explicitly described as a hint and not a real mount target.

## Risks / Open Questions

- Current main clarifies and exposes hint-only behavior; it still does not implement true custom in-child mount-path materialization.
- I did not identify a released tag after `v2026.4.22` that shipped this behavior change.

## Close Comment

Closing from a Codex review because current `main` already resolves the silent-behavior part of this report. `attachAs.mountPath` is now documented as a hint rather than a real mount target in `src/agents/tools/sessions-spawn-tool.ts:146`, the subagent spawn path only forwards it as `mountPathHint` in `src/agents/subagent-spawn.ts:837`, and attachment materialization tells the child the real workspace-relative path while also echoing the requested hint in `src/agents/subagent-attachments.ts:230`. Successful `sessions_spawn` results also now include `attachments.relDir` in `src/agents/subagent-spawn.ts:1151`, so callers get the actual attachment location instead of a silent success with no usable path. I could not tie that change to a released tag newer than `v2026.4.22`, so this is fixed on `main` but I’m leaving `fixedRelease` unset.
