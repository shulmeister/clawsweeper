## Summary

Keep open. Current `main` and the latest release still strip the `user:` / `channel:` prefix in Discord explicit-target parsing, and the surrounding outbound flow shows that this value is consumed directly for routing and chat-type inference. The PR's one-line fix is not already implemented on `main`, the reported failure mode is plausible from current code, and the discussion adds credible regression-test coverage rather than weakening the case.

## Evidence

- **Current main still drops the normalized prefix:** `parseDiscordExplicitTarget` still returns `to: target.id`, and both `messaging.parseExplicitTarget` and `messaging.inferTargetChatType` call that helper on current `main`. That means explicit Discord targets are still reduced to bare snowflakes before downstream routing.
  - file: [extensions/discord/src/channel.ts:349](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/discord/src/channel.ts#L349)

  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Explicit target output is used directly by outbound session resolution:** Core outbound session targeting replaces the raw explicit destination with `parsedExplicitTarget.to` when a channel plugin provides one, with no renormalization step afterward.
  - file: [src/infra/outbound/targets-session.ts:126](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/infra/outbound/targets-session.ts#L126)
