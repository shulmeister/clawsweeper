## Summary

Keep PR #39350 open. Current main still has the gateway final-message gap when a run ends after only a partial `NO_REPLY` fragment: the live delta path suppresses the fragment, but the final path resolves the same buffered text with lead-fragment suppression disabled and would emit `NO` as a final assistant message. The PR is not merge-ready unchanged because review comments correctly identify that its `isSilentReplyPrefixText` final guard would also drop a legitimate uppercase `NO` answer, but the bug and regression test remain useful.

## Best Possible Solution

Keep the PR open and ask for a revised gateway final-path fix. The best direction is to preserve provenance that the buffered text came from a suppressed control-token lead fragment, then suppress only that final payload while preserving an intentional uppercase `NO` answer. Add regression coverage for lifecycle end after only `NO`/`NO_` is buffered, plus coverage that an intentional final uppercase `NO` remains visible.
