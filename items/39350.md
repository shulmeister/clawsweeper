## Summary

Keep PR #39350 open. Current main still does not cover the narrow gateway final-message gap: a buffered partial control-token fragment such as `NO` is suppressed on delta/flush paths, but `emitChatFinal` deliberately resolves the final payload with lead-fragment suppression disabled and will emit `NO` as a final assistant message. The PR also should not be merged unchanged because the proposed `isSilentReplyPrefixText` guard would suppress legitimate uppercase `NO` final replies, matching the existing review comments.

## Best Possible Solution

Keep this PR open, but do not merge the current one-line guard unchanged. The best fix belongs in the gateway final path and should distinguish buffered control-token fragments from legitimate final replies, with regression coverage for lifecycle end after only `NO`/`NO_` is buffered plus coverage preserving a deliberate uppercase `NO` final answer.
