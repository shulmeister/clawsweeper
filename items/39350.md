## Summary

Keep PR #39350 open. Current main still has the gateway final-message gap for a lifecycle that ends after only a partial `NO_REPLY` fragment: the live delta path suppresses the fragment, but `emitChatFinal` resolves the buffered text with lead-fragment suppression disabled and would emit `NO` as a final assistant message. The PR is not ready to merge unchanged because its proposed `isSilentReplyPrefixText` guard would also suppress a legitimate uppercase `NO` final reply, but the bug and added regression test remain useful.

## Best Possible Solution

Keep the PR open and ask for a revised gateway final-path fix. The best direction is to distinguish buffered control-token provenance from a deliberate final `NO` answer, then add regression coverage for lifecycle end after only `NO`/`NO_` is buffered plus coverage preserving an intentional uppercase `NO` final response.
