## Summary

Keep this PR open. Current main still drops `ReplyToId` and `ReplyToBody` before the `inbound_claim` plugin event, while the source fields exist and are populated upstream. The PR remains a narrow additive implementation candidate rather than obsolete, duplicated, or ClawHub-only work.

## Best Possible Solution

Keep this PR open and review it as a small plugin hook API fix. The best path is to add optional `replyToId` and `replyToBody` to the canonical inbound hook context, map them from `FinalizedMsgContext`, expose them in `inbound_claim` metadata, and add a focused mapper test that proves reply metadata reaches plugin handlers.
