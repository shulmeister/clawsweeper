## Summary

Keep #39683 open. Current main still accepts Feishu/Lark announce cron delivery without a raw `delivery.to` at create/update validation time, but the submitted PR is not safe enough to close or land as-is because it validates only the raw field and would also run on unrelated updates to legacy jobs. The linked canonical bug #39599 remains open, and the broader #40031 is not a merged or clearly canonical replacement for this focused fix.

## Best Possible Solution

Keep the PR open but request rework. The fix should validate unresolved Feishu/Lark announce targets through the existing delivery-resolution seam, accept explicit `delivery.to` and deterministic session/plugin resolution, avoid blocking unrelated updates to legacy jobs, keep Feishu-specific target hints in the Feishu plugin surface, and add focused create/update regression coverage. If #40031 or another PR lands that resolved-target path first, this PR can then close as superseded.

## Risks / Open Questions

- The submitted raw `delivery.to` check can reject jobs whose target is deterministically resolvable from session or plugin routing.
- Because validation runs after any `applyJobPatch`, the submitted guard can block remediation updates such as disabling legacy Feishu/Lark jobs that lack `delivery.to`.
- Closing this PR now would drop a focused implementation thread while the canonical bug #39599 remains open and no narrower merged fix is present on main.
