## Summary

Keep open. The issue is recent, externally authored, has no protected labels, and the current main code still treats a mixed `command` + `url` + `transport: "streamable-http"` server as stdio because stdio resolution wins before HTTP resolution. Current main does not implement a managed streamable-http child-process lifecycle, and it also does not reject the mixed configuration early despite docs implying that mixed `command`/`url` entries should be rejected.

## Best Possible Solution

Keep the issue open. The best fix is either to add an explicit managed streamable-http child-process mode with process startup, port/readiness probing, HTTP client connection, and lifecycle cleanup, or to reject mixed `command` + `url`/HTTP transport entries before runtime with an actionable error. Add regression coverage for the mixed config and keep URL-only streamable HTTP coverage intact.

## Evidence
