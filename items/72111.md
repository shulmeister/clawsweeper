## Summary

Keep open. Current main still treats a mixed `command` + `url` + `transport: "streamable-http"` MCP server as stdio because stdio resolution returns first when `command` is present. That can produce the reported 30000ms startup timeout, and #66940 only covered the streamable-HTTP Accept header path, not this managed-child/mixed-config behavior.

## Best Possible Solution

Keep #72111 open. Maintainers should either implement a first-class managed streamable-http child-process mode with readiness probing, HTTP client connection, lifecycle cleanup, and regression tests, or reject mixed `command` + `url`/HTTP transport configs early with a clear diagnostic and docs/test alignment. The current silent stdio precedence should not remain the user-visible timeout path.
