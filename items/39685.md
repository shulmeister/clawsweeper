## Summary

Keep #39685 open. Current main still has only lower-level SSRF hostname allowlist support plus a narrow Codex-native `tools.web.search.openaiCodex.allowedDomains` setting. It does not expose or enforce the requested global/per-agent `allowedDomains` and `denyDomains` egress firewall for OpenClaw-managed `web_fetch` or managed/provider `web_search`.

## Best Possible Solution

Keep the issue open and implement a core web-tool network policy: add typed/schema/docs support for global and per-agent `allowedDomains` and `denyDomains`, resolve it consistently with other tool policies, map allowlists into `SsrFPolicy.hostnameAllowlist` for `web_fetch`, add denylist precedence, and explicitly define managed/provider-native `web_search` behavior and limitations. Add focused tests for schema validation, allow/deny precedence, redirects, cache keys, wildcard matching, and provider limitations.

## Risks / Open Questions

- Closing this now would hide a real core security/product gap: OpenClaw still cannot configure a domain walled garden for managed `web_fetch`, and managed/provider `web_search` behavior remains undefined.
- The existing Codex-native `allowedDomains` knob is easy to confuse with this request, but it only patches one native Codex search payload and provides no general deny precedence or per-agent policy.
