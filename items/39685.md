## Summary

Keep #39685 open. Current main still has lower-level SSRF hostname allowlist support and a Codex-native `tools.web.search.openaiCodex.allowedDomains` option, but it does not expose or enforce the requested global/per-agent `allowedDomains`/`denyDomains` egress firewall for `web_fetch` and managed `web_search`.

## Best Possible Solution

Keep the issue open and implement a core web tool network policy: add typed/schema/docs support for global and per-agent `allowedDomains`/`denyDomains`, resolve it consistently with other tool policies, map allowlists into `SsrFPolicy.hostnameAllowlist` for `web_fetch`, add deny precedence, and define enforceable behavior for managed/native `web_search` providers. If some provider-native search cannot be constrained locally, document that limitation and fail closed where feasible.

## Risks / Open Questions

- Closing this would lose a real security/product gap: OpenClaw still cannot configure a domain walled garden for local `web_fetch` or managed `web_search`.
- Provider-native and server-side search providers may require explicit fail-closed semantics or documented limitations because OpenClaw may not control every outbound retrieval path.
- Deny precedence, wildcard matching, redirect handling, cache keys, and interaction with existing SSRF exceptions need focused regression tests before shipping.
