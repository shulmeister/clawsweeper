## Summary

Keep open. The report matches current `main` and the latest release: `OpenClawChannelBridge` retains unbounded `pendingClaudePermissions` and `pendingApprovals` entries with no TTL/sweeper and no close-time clearing, and current tests do not cover those stale-entry paths.

## Evidence

- **Maps persist across bridge lifetime and are not cleared on close:** `OpenClawChannelBridge` owns `pendingClaudePermissions` and `pendingApprovals` at lines 50-51. `close()` clears only `pendingWaiters` and then stops the gateway; it does not clear either pending map.
  - file: [src/mcp/channel-bridge.ts:50](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/mcp/channel-bridge.ts#L50)
  - command: `nl -ba src/mcp/channel-bridge.ts | sed -n '1,240p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Claude permission entries only delete on one matched reply path:** `handleClaudePermissionRequest()` unconditionally inserts into `pendingClaudePermissions` at lines 273-294. The only deletion path is `handleSessionMessageEvent()` lines 455-459 after a reply matching `CLAUDE_PERMISSION_REPLY_RE`.
  - file: [src/mcp/channel-bridge.ts:273](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/mcp/channel-bridge.ts#L273)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Permission request schema and reply matcher are broader/narrower than each other:** The MCP notification schema accepts any string `request_id`, but the reply matcher only accepts `yes|no` plus a 5-character `[a-km-z]{5}` token. That means the bridge can store permission requests it cannot later match through the reply parser.
  - file: [src/mcp/channel-shared.ts:121](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/mcp/channel-shared.ts#L121)

There is no timer/sweeper in this bridge using `expiresAtMs`.
