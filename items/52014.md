## Summary

Current `main` does not have a code path where a node-host `PAIRING_REQUIRED` connect failure crash-loops the gateway. The gateway and node host are separate supervised services, the node-host runner only logs connect/close failures and then stays alive, and the shared `GatewayClient` explicitly pauses reconnect on `PAIRING_REQUIRED` instead of hot-looping or throwing out of the process.

## Close Comment

Closing from a Codex review because this does not reproduce against current `main`. The current code treats the gateway and node host as separate supervised services (`src/daemon/constants.ts:4`, `src/daemon/node-service.ts:44`), and the node-host runner handles gateway auth/pairing failures by logging them and staying alive rather than exiting (`src/node-host/runner.ts:211`, `src/node-host/runner.ts:226`). On the shared client path, `PAIRING_REQUIRED` is explicitly treated as a terminal auth state that pauses reconnect instead of hot-looping or crashing (`src/gateway/client.ts:302`, `src/gateway/client.ts:608`), with coverage in `src/gateway/reconnect-gating.test.ts:48`. The maintenance concern from the April comment also no longer matches current code: `openclaw gateway restart` now restarts via service/PID inspection instead of requiring a healthy gateway RPC session (`src/cli/daemon-cli/lifecycle.ts:124`, `src/cli/daemon-cli/lifecycle.ts:186`). If anyone still sees this on a newer build, please open a fresh issue with the exact release/commit, whether the failing process is `ai.openclaw.gateway` or `ai.openclaw.node`, and the corresponding launchd log excerpt.
