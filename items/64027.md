## Summary

Keep this PR open. Current main does not implement the PR's central Feishu behavior: carrying an update-card payload in card action envelopes, parsing the clicked card's message id, and patching the original card after confirm/cancel/quick-action clicks. The PR is still an implementation candidate, although it should be narrowed and retested before landing.

## Best Possible Solution

Keep this PR open as a candidate implementation. The best maintainer path is to review or ask for a focused rebase that only wires Feishu card update payloads through card-interaction/card-ux-shared, parses the callback message_id in monitor.account, calls updateCardFeishu from card-action with failure isolation, and adds targeted unit/lifecycle tests for confirm, cancel, quick action, malformed uc, and missing message_id behavior. If maintainers prefer not to carry the contributor branch, close only after opening or landing a narrower replacement PR for the same Feishu card-update behavior.
