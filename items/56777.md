## Summary

Keep open. Current `main` still has the reported gap: Telegram native slash command auth resolves `topicConfig`, but it evaluates group policy with topic/group overrides explicitly disabled, so a topic-level `groupPolicy: "disabled"` will not participate in command gating even though normal Telegram group-message handling does honor topic overrides and the docs say topic entries inherit `groupPolicy`.

## Evidence

- **Policy helper only applies topic/group overrides when enabled:** `evaluateTelegramGroupPolicyAccess` uses `topicConfig.groupPolicy` and `groupConfig.groupPolicy` only when `useTopicAndGroupOverrides` is `true`; otherwise it falls back to provider/default policy.
  - file: [extensions/telegram/src/group-access.ts:146](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/group-access.ts#L146)
  - command: `nl -ba extensions/telegram/src/group-access.ts | sed -n '138,153p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Native Telegram command auth disables topic/group policy overrides:** `resolveTelegramCommandAuth` passes `topicConfig` and `groupConfig` into `evaluateTelegramGroupPolicyAccess`, but sets `useTopicAndGroupOverrides: false`, which bypasses topic-level `groupPolicy` for native slash commands.
  - file: [extensions/telegram/src/bot-native-commands.ts:450](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-native-commands.ts#L450)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.ts | sed -n '450,467p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Normal Telegram group-message handling does honor topic/group overrides:** The standard Telegram inbound group-message path calls the same policy helper with `useTopicAndGroupOverrides: true`, so command handling and message handling currently diverge.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:594](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-handlers.runtime.ts#L594)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '594,607p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Docs say Telegram topics inherit groupPolicy:** Telegram docs state topic entries inherit group settings including `groupPolicy`, so topic-level command bypass contradicts documented behavior.
  - file: [docs/channels/telegram.md:509](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/channels/telegram.md#L509)
  - command: `nl -ba docs/channels/telegram.md | sed -n '496,510p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Native-command tests cover group-level policy but not topic-level groupPolicy:** Current native-command auth regression coverage asserts account/group-level `groupPolicy: "disabled"` behavior, but there is no corresponding topic-level `groupPolicy` case in the native-command tests.
  - file: [extensions/telegram/src/bot-native-commands.group-auth.test.ts:99](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot-native-commands.group-auth.test.ts#L99)
