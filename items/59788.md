## Summary

Keep open. The request is a plausible feature gap for outbound-first routed sessions: current main has the generic CLI transcript mirroring path, but the inspected code still appears to anchor CLI sends to the default agent rather than resolving the agent from the outbound channel/account binding, which is the specific non-default-agent WhatsApp scenario reported in #59788.

## Evidence

- **Issue is actionable and not stale:** #59788 was opened 2026-04-02, has only the enhancement label, author association NONE, and describes a concrete WhatsApp account binding to non-default agent `realtor`, expected transcript persistence, actual missing opener context, and example CLI command.
- **Current CLI send path passes default agent id:** The current `messageCommand` path calls `runMessageAction` with `agentId: resolveDefaultAgentId(cfg)`, which supports generic mirroring but does not by itself resolve the routed agent from the supplied channel/account binding requested by the issue.
  - file: [src/commands/message.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/src/commands/message.ts)
  - command: `read-only raw source inspection`
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **Outbound action runner only maps agent to account, not account to agent:** `runMessageAction` derives `resolvedAgentId` from `input.agentId` or session key. Its binding lookup chooses an account for an already-resolved agent when no accountId is supplied; it does not appear to reverse-resolve a non-default agent from an explicit accountId such as `realtor-whatsapp`.
  - file: [src/infra/outbound/message-action-runner.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/src/infra/outbound/message-action-runner.ts)
  - command: `read-only raw source inspection`

- **WhatsApp outbound session key support exists but depends on correct agent id:** The outbound session resolver has a WhatsApp resolver that builds the session key from `agentId`, channel, accountId, and normalized WhatsApp peer. That means the feature hinges on the caller resolving the right non-default agent before mirroring.
  - file: [src/infra/outbound/outbound-session.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/src/infra/outbound/outbound-session.ts)

That is useful related history, but #59788 asks for routed non-default agent resolution from outbound account binding, which is not clearly covered by that fix.
