## Summary

Current main already addresses the reported upgrade-triggered Telegram native-approvals regression: Telegram exec approvals no longer auto-enable just because approvers/allowlists resolve, and OpenClaw now documents/surfaces the paired-device scope-upgrade approval path. The infinite retry path still exists for explicitly enabled native approvals, but the specific post-upgrade implicit-enable failure described here is implemented on main and shipped in v2026.4.23.

## Evidence

## Risks / Open Questions

- `docs/channels/telegram.md` still says Telegram exec approvals auto-enable when approvers are resolvable; that doc text appears stale relative to current code/tests and could confuse users.
- If an operator explicitly sets `channels.telegram.execApprovals.enabled` to `true` or `"auto"` without approving the broader device scopes, the bootstrap retry loop can still happen; this close only covers the upgrade-triggered implicit-enable regression described here.
