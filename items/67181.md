## Summary

Keep #67181 open. Current main still prepends the exact user-visible resume-fallback banner when an exec approval followup cannot resume the session but can send directly, and the regression test currently asserts that behavior for a successful `code 0` completion. The issue is actionable and not already implemented away.

## Best Possible Solution

Keep the issue open and fix the exec approval followup fallback in `src/agents/bash-tools.exec-approval-followup.ts`: when the command result itself succeeded, direct delivery should omit the internal resume-failure banner or use neutral copy, with the resume failure kept to logs. Update `src/agents/bash-tools.exec-approval-followup.test.ts` so successful `code 0` fallback delivery expects only the sanitized command result, while denied or failed-command paths keep appropriate user-facing wording.
