## Summary

Keep #72286 open. Current main still documents and implements Control UI Talk as OpenAI-only browser WebRTC, while Google Live is limited to backend realtime bridges. The PR is a focused draft adding a missing provider/browser transport capability, and it appears to require core/protocol/UI contract work that cannot be dismissed as already implemented or purely ClawHub-only.

## Best Possible Solution

Keep the PR open for maintainer review. The likely best path is to continue the shared contract work: add an explicit realtime browser transport discriminator, implement Google Live browser session minting in the Google provider, teach Control UI Talk to handle Google Live WebSocket sessions alongside OpenAI WebRTC, and keep the provider docs/tests aligned. If maintainers want this externalized later, they should first define the missing plugin/browser transport API rather than closing this as already handled.
