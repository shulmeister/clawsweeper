## Summary

Keep #43317 open for maintainer judgment. Current main has already shipped a different sandbox backend registry and plugin SDK surface, so the PR's proposed provider interface is partly superseded. However, current main does not implement the PR's gVisor/runsc backend or Firecracker chain, and the PR carries security/core sandbox isolation work plus unresolved review concerns, so it is not a conservative auto-close candidate.

## Best Possible Solution

Keep the PR open or have a maintainer explicitly ask the contributor to rebase/split it: retain the current `SandboxBackendHandle`/`registerSandboxBackend` architecture, extract any still-desired gVisor/runsc provider as either a small core backend or plugin-backed backend, and address the existing review findings with focused tests for workspace mounts, lifecycle, network policy, registry/prune behavior, browser sessions, and setupCommand.
