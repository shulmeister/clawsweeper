## Summary

Keep open. Current main has already picked up the related LINE file-message handling, but the PR's central sandbox-storage fix is not implemented: LINE downloads still write to the preferred temp directory via buildRandomTempFilePath/fs.writeFile, tests still assert that temp-file behavior, and the handler does not pass the original LINE filename into downloadLineMedia.

## Best Possible Solution

Keep the PR open and ask for a current-main port: update extensions/line/src/download.ts to use getMessageContentWithHttpInfo, save through openclaw/plugin-sdk/media-store saveMediaBuffer(..., "inbound", maxBytes, originalFilename), pass message.fileName for LINE file messages from bot-handlers, and update LINE/media-store tests to cover sandbox-accessible media/inbound storage and filename-based MIME fallback.
