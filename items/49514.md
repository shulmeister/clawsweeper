## Summary

Keep open. Current `main` does not contain the PR's singleton fix or an equivalent alternative. `extensions/slack/src/http/registry.ts` still uses a module-local `Map`, while registration and request handling are wired from different Slack entry surfaces, so there is not strong evidence the reported split-chunk 404 is resolved on `main`.

## Evidence

- **Registry is still module-local:** `slackHttpRoutes` is still declared as `new Map(...)` at module scope, with no `globalThis`/`Symbol.for()` singleton or other cross-chunk sharing mechanism.
  - file: [extensions/slack/src/http/registry.ts:18](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/slack/src/http/registry.ts#L18)

If bundling duplicates `registry.ts`, the write and read sides can still diverge exactly as described in the PR.
  - file: [extensions/slack/src/monitor/provider.ts:321](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/slack/src/monitor/provider.ts#L321)
