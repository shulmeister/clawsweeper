## Summary

Keep #71697 open. Current main still supports direct Ollama Cloud/native `api: "ollama"` configuration, but the agent/Gateway replay path does not add an Ollama-native turn-normalization policy before sending converted history to `/api/chat`. The issue is distinct from #66505, which addressed direct Cloud reachability/auth rather than native Ollama replay ordering.

## Best Possible Solution

Keep the issue open and fix the native Ollama provider path. The likely maintainer path is an Ollama-owned replay policy or native converter normalization for `api: "ollama"` that merges/reshapes non-alternating user and assistant turns before `/api/chat`, while preserving supported Ollama tool-call/tool-result behavior. Add regression tests that feed agent/Gateway-style non-alternating history through the native Ollama path and assert the request body is accepted-shaped rather than falling through to model fallback.
