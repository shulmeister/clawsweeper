## Summary

Keep this issue open. Current main preserves established external delivery routes in some narrower dashboard/subagent cases, but it still does not implement the requested force-origin-channel or immutable persisted-origin behavior for Telegram/Discord/WhatsApp-style replies when WebChat/dashboard is involved. The linked implementation PR #57802 is useful prior art, but it was closed unmerged.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is a narrow reply-routing design at the session/reply boundary, either an explicit opt-in force-origin setting or an immutable persisted-origin route based on the first external user turn, with regression tests for Telegram/Discord/WhatsApp-style inbound sessions while WebChat/dashboard is active and safeguards against stale-route or wrong-channel delivery.

## Risks / Open Questions

- A broad force-origin default could misdeliver replies into stale or unintended external conversations, especially for shared sessions or dashboard-initiated turns.
- Any fix needs to preserve the existing route-safety behavior that prevents WebChat/control-ui clients from silently inheriting stale external delivery routes.
- The prior PR #57802 is useful design input, but it needs maintainer review and updated tests against current routing safeguards before landing.
