## Summary

Keep #12508 open. Current main still implements trusted, mutable, in-process plugin hook chains with priority/registration ordering, result merging, and only coarse prompt/conversation hook policy. I did not find host-owned hook-chain integrity verification, per-handler modification audit records, or per-hook manifest permissions that would satisfy the request.

## Best Possible Solution

Keep #12508 open for core plugin-runtime security design. The best implementation likely belongs across the hook runner, plugin registry/config or manifest policy, audit/provenance storage, docs, and focused tests for sequential mutation provenance, per-handler audit records, integrity verification, and per-hook registration permissions. Related message-hook issues track hook availability/context, not this chain-integrity guarantee.

## Risks / Open Questions

- Naive checksum enforcement could break legitimate documented hook rewrites; the design needs to distinguish authorized mutation, provenance, auditability, and tamper detection.
- This should not be moved to ClawHub as-is because an external plugin cannot enforce integrity across the full host-owned hook chain without new core runtime support.
