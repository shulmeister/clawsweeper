## Summary

Keep open. Current main still executes plugin hooks in ordered in-process chains where handlers can intentionally rewrite messages, tool params, and transcript records. Core has narrower trust controls such as prompt-injection and conversation-access gates, but there is no hook-chain checksum/signature verification, per-stage tamper detection, or audit log of each hook modification. The existing issue discussion also already reached the same keep-open conclusion, and the current main SHA does not materially change that result.

## Best Possible Solution

Keep #12508 open for maintainer/product security design. The likely implementation belongs around core hook registration/execution in src/plugins/hooks.ts and src/plugins/registry.ts, with config/schema/docs/tests covering per-hook permission declarations, deterministic modification auditing, and host-enforced integrity/tamper detection for modifying hooks. A ClawHub plugin alone cannot enforce this between other plugins' hook handlers.
