## Summary

Keep open. Current main still runs plugin hooks as ordered in-process chains where modifying hooks can rewrite prompts, tool params, message content, and transcript-adjacent records. Existing controls cover narrower prompt-injection and conversation-access categories, but there is no hook-chain checksum/signature verification, per-stage tamper detection, manifest hook allow-list, or audit log of every hook modification.

## Best Possible Solution

Keep #12508 open for maintainer security/product design. The likely path is a core/plugin-runtime change around `src/plugins/hooks.ts`, `src/plugins/registry.ts`, config schema/help, plugin docs, and contract tests to define per-hook permissions, deterministic modification auditing, and host-enforced integrity/tamper checks for modifying hook chains. This should not be closed as ClawHub-only because external plugins cannot enforce integrity between other plugins' handlers in the core runner.
