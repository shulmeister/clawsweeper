## Summary

Keep open. Current main accepts Slack `thread_broadcast` events and documents them as normal inbound messages, but it still does not expose broadcast visibility to agents or provide an outbound `reply_broadcast`/`replyBroadcast` option on the Slack send paths. Related Slack thread items cover `thread_ts` support, not this broadcast mirroring behavior.

## Best Possible Solution

Keep this issue open and implement it as a scoped Slack plugin change: add a typed `replyBroadcast` option through `sendMessageSlack`, `sendSlackMessage`, `handleSlackAction`, and any generic Slack message-action surface that should expose it; pass `reply_broadcast` to `chat.postMessage` only when sending a threaded reply; surface inbound `subtype: "thread_broadcast"` or equivalent broadcast visibility in agent-visible Slack context; then add regression tests and update Slack docs/skill examples.
