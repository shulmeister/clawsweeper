## Summary

Keep open. The report's `script`-payload premise no longer matches current `main`, but current docs still do not explain the existing cron timeout behavior for `agentTurn` jobs well enough to justify closure.

## Evidence

- **Current cron payloads do not include `script`:** `CronPayload` is only `systemEvent | agentTurn`, and the public gateway schema likewise defines only `systemEvent` and `agentTurn` payload variants. That makes the issue's `script`-job premise obsolete on current `main`, but not resolved in a way that cleanly closes the docs gap.
  - file: [src/cron/types.ts:112](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/types.ts#L112)
  - command: `nl -ba src/cron/types.ts | sed -n '98,132p' ; nl -ba src/gateway/protocol/schema/cron.ts | sed -n '1,40p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Timeout policy is still undocumented in the docs section reviewed:** The cron docs section I inspected lists `timeoutSeconds` for isolated agent turns, but does not document the default safety timeout or the `0 = no timeout` behavior.
  - file: [docs/automation/cron-jobs.md:241](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/docs/automation/cron-jobs.md#L241)

  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Current code still has important timeout semantics worth documenting:** `resolveCronJobTimeoutMs` uses a 10-minute default for non-agent jobs, a 60-minute default for `agentTurn`, and only honors `payload.timeoutSeconds` for `agentTurn` jobs. Those behaviors are current code, not already documented as part of this issue.
  - file: [src/cron/service/timeout-policy.ts:8](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/service/timeout-policy.ts#L8)

That reinforces that there is still a real documentation gap around current behavior.
