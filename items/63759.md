## Summary

Keep this issue open. Current main sends Telegram typing through the normal reply typing callback path, and run-start typing now happens before auto-reply preflight compaction for instant-mode runs, but the Telegram inbound path still does not fire `sendChatAction("typing")` immediately after the message is accepted/normalized as requested. There are still pre-dispatch and queue paths where no early Telegram typing cue is sent.

## Best Possible Solution

Implement an extension-owned Telegram fix that starts the native typing cue at the earliest safe point after access, routing, mention/body normalization, and thread resolution succeed, but before dispatch-heavy work, queue handling, compaction, or model/tool execution. Add a regression test that makes a preflight/queue step intentionally delayed and asserts `sendChatAction("typing")` is called first for accepted Telegram messages, while still asserting no typing for ignored/unauthorized messages.
