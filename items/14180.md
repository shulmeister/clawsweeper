## Summary

Current `main` already implements the core architecture requested in #14180: isolated background sub-agents with control/steering, lane-aware queueing so follow-up user messages do not blindly replace active work, durable background task and Task Flow state, and heartbeat/task wake integration for non-blocking ongoing work.

## Evidence

- **Lane-aware queueing prevents active-work clobbering:** The command queue is lane-aware, enqueues by session key, supports separate global lanes, and documents `steer`, `followup`, `collect`, and `steer-backlog` modes so new inbound messages can steer or queue instead of replacing active work. See `docs/concepts/queue.md` lines 17-23, 25-39, 78-84 and `src/process/command-queue.ts` lines 31-35, 45-52, 155-170.
  - file: [docs/concepts/queue.md:17](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/concepts/queue.md#L17)

- **Sub-agents run independently and can be controlled:** Current docs describe sub-agents as background runs in their own session with push-based completion, `/subagents list|kill|info|send|steer`, and persistent thread-bound `sessions_spawn(..., thread: true, mode: "session")`. The tool implementation supports list/kill/steer actions. See `docs/tools/subagents.md` lines 10-11, 40-63, 109-123, 162-212 and `src/agents/tools/subagents-tool.ts` lines 33-48, 57-75, 135-175.
