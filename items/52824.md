## Summary

Keep open. Current `main` partially covers thread preservation for direct announces, but the bound completion routing path still keys Telegram requester conversations by `threadId` before the chat id from `to`, so it does not match Telegram's canonical bound conversation ids and does not clearly implement this PR.

## Evidence

- **Bound completion routing still prioritizes requester `threadId`:** `resolveSubagentCompletionOrigin` builds `conversationId` as `threadId || resolveConversationIdFromTargets([to])`, then uses that for `createBoundDeliveryRouter().resolveDestination(...)`. For Telegram DM topics, that means the requester key is just the topic id instead of the chat/topic conversation id.
  - file: [src/agents/subagent-announce-delivery.ts:230](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/subagent-announce-delivery.ts#L230)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '222,260p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Telegram bindings are canonicalized as chat-plus-topic, not bare topic id:** Telegram's binding resolver canonicalizes threaded conversations through `resolveTelegramInboundConversation`, and Telegram binding tests assert stored conversation ids like `-100200300:topic:77`. That does not line up with a requester conversation id of only `77` or `42`.
  - file: [extensions/telegram/src/channel.ts:623](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/telegram/src/channel.ts#L623)

That leaves the PR's bound-route case unproven on `main`.
