## Summary

Keep #69451 open. Current main has session-store maintenance that bounds some growth, but the core memory complaint remains: `sessions.json` is still read and parsed as one full JSON object, saves still stringify the whole store, and history reads still load a whole transcript before applying cursor/limit pagination. The open related issue #71227 is useful corroboration for the same storage hotspot, but it is framed as the latency/run-budget angle rather than a canonical replacement for this memory/RSS issue.

## Best Possible Solution

Keep the issue open and fix the storage/read path rather than closing on maintenance mitigations. The best path is to add a shard or indexed metadata store for `sessions.json` entries, avoid retaining both parsed and serialized large stores when possible, and replace `chat.history`/HTTP history full-transcript reads with a tail/cursor-aware JSONL reader that can return bounded pages without loading the whole transcript. #71227 can remain the related latency-focused report, but this issue should track the memory/RSS and streaming/pagination work unless maintainers create a narrower canonical storage redesign issue.
