## Summary

Keep #16121 open. Current main still lacks first-class inline Telegram custom emoji support for Markdown-rendered Telegram messages, captions, action sends, edits, replies, and streaming delivery. Existing code only covers adjacent forum-topic icon IDs and reaction metadata, and narrower related items such as #40848 and #47562 do not supersede this broader message-formatting request.

## Best Possible Solution

Keep #16121 open and implement inline Telegram custom emoji in the bundled Telegram plugin. A conservative path is to extend the Markdown/text IR or add a safe explicit syntax that carries `emoji-id` plus fallback emoji text, then render it consistently to Telegram HTML or entities for messages, captions, edits, replies/quotes, preview streaming, and action sends. Coordinate with narrower open work like #40848 for entity passthrough and #47562 for reaction-specific custom emoji handling, but do not close this broader issue as superseded by those narrower items.

## Risks / Open Questions

- A trusted `textMode: "html"` path can manually pass caller-owned raw HTML, so the remaining gap should be framed as first-class Markdown/action/caption/reply/preview support, not absolute impossibility.
- A naive allowlist for `<tg-emoji>` could weaken current escaping and parse-fallback behavior; implementation should preserve safe fallback text and Telegram parse-error recovery.
- Telegram custom emoji support has account eligibility and fallback-emoji constraints, so docs and tests should cover unsupported accounts, malformed IDs, and fallback display.
