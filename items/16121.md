## Summary

Keep #16121 open. Current main still lacks first-class Telegram custom emoji support for normal outbound markdown/model output, captions, replies, and preview streaming; the only current custom-emoji handling I found is unrelated forum-topic/reaction metadata.

## Best Possible Solution

Keep #16121 open and implement it in the bundled Telegram plugin formatting path: add a documented custom emoji syntax or structured markdown entity, validate and escape `emoji-id`, render Telegram HTML as `<tg-emoji emoji-id="...">fallback emoji</tg-emoji>`, preserve plain-text fallback, and cover messages, captions, replies, chunking, parse-fallback behavior, preview streaming, and docs/tests under `extensions/telegram` and `docs/channels/telegram.md`.

## Risks / Open Questions

- A naive raw-HTML allowlist would weaken the formatter's deliberate raw HTML escaping; this needs a validated structured syntax or entity path instead.
- Telegram custom emoji has account eligibility and fallback-display constraints, so docs and tests should cover unsupported/non-premium behavior rather than promising universal rendering.
