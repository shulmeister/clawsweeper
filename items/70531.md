## Summary

Keep #70531 open. Current main still persists cron job definitions as a single `jobs.json` array and does not discover, migrate, add, edit, or delete individual per-job definition files under `~/.openclaw/cron/jobs/`. Main has already split runtime state into `jobs-state.json` and per-job run logs, but that only partially addresses file churn and does not implement the requested per-job config layout.

## Best Possible Solution

Keep this open as a core cron storage enhancement. If maintainers accept the direction, implement a versioned cron store that loads legacy `jobs.json`, discovers deterministic per-job definition files under `~/.openclaw/cron/jobs/`, writes add/edit/delete operations atomically per job, preserves the existing `jobs-state.json` runtime-state split and per-job run logs, and updates doctor/config docs plus cron store/service/Gateway tests.

## Risks / Open Questions

- The existing `jobs-state.json` split partially addresses runtime-state churn and should be preserved or deliberately migrated if per-job definition files are added.
- A full implementation needs deterministic file naming and load order, atomic per-job add/edit/delete, legacy `jobs.json` fallback/migration, doctor repair coverage, and docs/test updates.
