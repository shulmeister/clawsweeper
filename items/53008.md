## Summary

Current main does not yet implement the requested memoryFlush lane isolation. The memoryFlush embedded run is still dispatched without a lane override, so runEmbeddedPiAgent resolves it onto the shared main global lane; user-triggered embedded runs also use that default path. The prior implementation attempt, #53154, was closed unmerged, and current tests cover memoryFlush forwarding but not non-main lane routing.

## Best Possible Solution

Keep this issue open and implement focused memoryFlush isolation. The likely core fix is to pass an explicit non-main lane, or introduce a dedicated compaction lane, when runMemoryFlushIfNeeded invokes runEmbeddedPiAgent; add regression coverage in src/auto-reply/reply/agent-runner-memory.test.ts that asserts the memoryFlush embedded run is not dispatched on the default main lane. Maintainers should also make an explicit product call on whether background memoryFlush may bypass per-session ordering.

## Risks / Open Questions

- The default main-lane concurrency is now 4, which may reduce the blast radius versus the reported v2026.3.x configuration, but it does not provide the requested dedicated/background compaction lane and remains configurable back to 1.
- A global lane override alone would not answer the same-session ordering question because runEmbeddedPiAgent also wraps execution in a session lane; maintainers need to decide whether memoryFlush may bypass same-session serialization or should only avoid cross-session/global starvation.
