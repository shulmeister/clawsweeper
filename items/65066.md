## Summary

Keep open. Current `main` still does not register a memory runtime/capability for `memory-lancedb`, while doctor/status still resolve the active memory backend only through that runtime path. The linked bug report and PR discussion match the current code, so there is not strong evidence this PR is obsolete or already implemented elsewhere on `main`.

## Evidence

- **Current memory-lancedb plugin still lacks runtime registration:** In current `main`, `extensions/memory-lancedb/index.ts` registers tools and hooks in `register(...)`, but there is still no `api.registerMemoryCapability(...)` or `api.registerMemoryRuntime(...)` call anywhere in the file.
  - file: [extensions/memory-lancedb/index.ts:304](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/memory-lancedb/index.ts#L304)
  - command: `rg -n "registerMemoryCapability|registerMemoryRuntime" extensions/memory-lancedb/index.ts && nl -ba extensions/memory-lancedb/index.ts | sed -n '297,520p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Doctor/status still depend on the memory runtime path:** `src/plugins/memory-runtime.ts` calls `getMemoryRuntime()` and returns `manager: null` / `backendConfig: null` when no runtime is registered. `src/plugins/memory-state.ts` confirms that only `registerMemoryCapability(...{ runtime })` or legacy `registerMemoryRuntime(...)` populate that slot.
  - file: [src/plugins/memory-runtime.ts:9](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/plugins/memory-runtime.ts#L9)
  - command: `nl -ba src/plugins/memory-runtime.ts | sed -n '1,40p' && nl -ba src/plugins/memory-state.ts | sed -n '252,259p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Current user-facing false negative is still present:** `openclaw doctor` still emits `No active memory plugin is registered for the current config.` when `resolveActiveMemoryBackendConfig(...)` returns null, and gateway doctor status still reports embeddings unavailable when `getActiveMemorySearchManager(...)` returns no manager.
  - file: [src/commands/doctor-memory-search.ts:332](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/commands/doctor-memory-search.ts#L332)
  - command: `nl -ba src/commands/doctor-memory-search.ts | sed -n '320,336p' && nl -ba src/gateway/server-methods/doctor.ts | sed -n '785,800p'`

- **Built-in memory-core already uses the required registration path:** `extensions/memory-core/index.ts` calls `api.registerMemoryCapability({ ... runtime: memoryRuntime ... })`, which matches the SDK docs describing `registerMemoryCapability` as the preferred exclusive memory-plugin API. `memory-lancedb` does not currently do the same.
  - file: [extensions/memory-core/index.ts:29](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/memory-core/index.ts#L29)
