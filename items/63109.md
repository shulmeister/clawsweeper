## Summary

Keep open. Current main still treats `/acp ...` slash commands as non-bypass candidates for ACP dispatch, so the PR’s central behavior is not implemented. The PR also is not cleanly mergeable as-is because the posted review comments identify a real ordering regression: a generic `/` catch-all placed before `isResetCommandCandidate` would change `/new` and `/reset` behavior.

## Best Possible Solution

Keep the PR open, but require a corrected patch. The likely product/code direction is to preserve the `/new` and `/reset` bypass first, then add a generic slash/control-command bypass for ACP/local chat commands where native Discord thread slash commands must not be sent as ACP prompt text. Add focused regression tests for `/acp cancel` or `/acp status` in an ACP-bound Discord thread/native command context, plus `/new` and `/reset` with `commands.text=false` to prevent the ordering regression. Review the Discord channel lookup timeout as a separate change or clearly justify it in this PR.
