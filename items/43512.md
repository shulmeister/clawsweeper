## Summary

Keep open. Current main only gives Feishu abort-like text such as `/stop` a separate `:control` sequential lane; `/status`, `/new`, `/reset`, `/model`, and `/help` still use the normal per-chat lane and can remain queued behind active same-chat work. The PR is not cleanly implemented on main, and the discussion contains useful review work around debounce flushing, destructive-command drops, and already-queued chat tasks that would need maintainer judgment before any replacement lands.

## Best Possible Solution

Keep the PR open or replace it with a focused maintainer-approved Feishu queue policy change. A good path is to add explicit tests for active-run ordering of `/stop`, read-only status commands, and destructive session commands, then decide whether Feishu should mirror Telegram's abort/status control-lane behavior or introduce a safe direct-dispatch/drop policy for `/new` and `/reset` that handles debounced and already queued work without losing unauthorized user text.
