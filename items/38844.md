## Summary

Keep #38844 open. Current main has meaningful code mitigations for atomic upload and dispatcher timeout wording, and the related #44934 selector/batch work helps avoid stale refs. The remaining product gap is still real: public browser-control docs still describe `upload` as a split arming call before the triggering click, and the bundled browser automation skill gives stale-ref recovery guidance but no upload/file-chooser atomic workflow guidance.

## Best Possible Solution

Keep the issue open until maintainers either explicitly accept the code mitigations as sufficient or land the missing guidance and coverage. The concrete path is to update `docs/tools/browser-control.md` and `extensions/browser/skills/browser-automation/SKILL.md` to prefer single-call `upload` with `ref` or direct `inputRef` on dynamic pages, clarify selector support versus existing-session limits, and add a focused contract test that locks the atomic `/hooks/file-chooser` upload-plus-click behavior.

## Risks / Open Questions

- Closing now would overstate the fix because the current user-facing API docs can still lead callers into the split upload-then-click workflow described in the issue.
- There is route-level coverage for hooks, but the focused contract still does not clearly assert the upload-with-ref path performs the arm and triggering click as a single intended workflow.
- Docs are inconsistent around selector support: current source and tests support selector actions for managed profiles, while `docs/tools/browser-control.md` still says CSS selectors are intentionally unsupported for actions.
