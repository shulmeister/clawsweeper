## Summary

Keep #7722 open. Current main still implements only the narrower `tools.fs.workspaceOnly` guard plus sandbox backends; it does not accept the requested `tools.fileAccess` config, does not expose `mode`/`allowedPaths`/`denyPaths`, and does not pass a filesystem allow/deny policy into `exec` command execution. The latest issue discussion also explicitly preserved this remaining scope.

## Best Possible Solution

Keep #7722 open until OpenClaw either implements a versioned filesystem access policy surface, likely under `tools.fs` or an equivalent config contract, with allow/deny path rules enforced across read/write/edit/apply_patch and a defensible exec strategy, or a maintainer explicitly narrows or rejects that product direction. Treat shipped `tools.fs.workspaceOnly` and sandbox backends as useful partial hardening, not completion of this issue.
