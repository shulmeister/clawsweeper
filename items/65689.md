## Summary

Keep this PR open. Current main still does not decode or render `errorMessage` on persisted OpenClawKit chat history messages, while the backend contract explicitly keeps raw provider failure text in the peer `errorMessage` field instead of replayed `content`. The PR is therefore still addressing a real missing client path, but it should be revised before merge to gate fallback rendering to true assistant error turns because stale `errorMessage` on successful turns is a known regression class.

## Best Possible Solution

Keep this PR open and ask for a small revision: add `errorMessage` to `OpenClawChatMessage`, preserve it through copy/reconcile paths, render it only for assistant error turns with no visible text, and add tests for both the desired history-refresh error case and the stale-success/tool-only `errorMessage` cases raised in review. Android/native parity may need the same history-decoding treatment if maintainers want the behavior across all app chat clients.

## Risks / Open Questions

- Merging the PR as submitted could reintroduce stale provider-error text on successful or tool-only assistant turns unless fallback rendering is gated by assistant role and `stopReason == "error"`.
- Closing the PR now would leave OpenClawKit history refresh unable to show the backend-owned provider error text that current backend code intentionally stores outside replayed assistant content.
