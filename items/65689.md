## Summary

Current `main` does not implement this PR’s requested behavior: chat history decoding still drops per-message `errorMessage`, and the chat view only shows provider failures via transient `errorText` banners. The PR also has valid unresolved review feedback: its fallback would need to be gated to true assistant error turns to avoid rewriting successful messages that carry stale/background `errorMessage` values.

## Evidence

- **Current chat history model still omits per-message error text:** `OpenClawChatMessage` on `main` has `role`, `content`, `timestamp`, `toolCallId`, `toolName`, `usage`, and `stopReason`, but no `errorMessage` field. Its decoder falls back from array content to string content, then otherwise sets `content = []`, so an empty assistant error turn with only `errorMessage` is still decoded as empty history content.

## Risks / Open Questions

- As written in the PR discussion, a naive `errorMessage` fallback would mislabel successful assistant turns that carry stale/background `errorMessage` values unless it is gated on `stopReason == "error"` and likely assistant role.
- Leaving this PR closed would lose a still-missing user-facing fix: current `main` can refresh history after an error run yet still show no assistant transcript entry for the stored provider failure.
