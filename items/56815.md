## Summary

Keep open. Current `main` still deletes the temp index on any failed full reindex, only retries embedding failures that match rate-limit/5xx-style patterns, and still logs memory sync failures with `String(err)`. The reported `TypeError: fetch failed` plus total progress loss is therefore not clearly fixed on current or shipped code, despite the related PR cross-reference in the timeline.

## Evidence

- **Atomic reindex still discards partial progress on failure:** `runMemoryAtomicReindex` still removes the temp index files in its catch path before rethrowing, so a failed long-running rebuild loses the staged sqlite state instead of preserving it for resume/debug.

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current test suite expects temp cleanup after failed reindex:** The atomic reindex test still asserts that a failed reindex keeps the old index and that the temp index path is gone afterward, which matches the reporter's progress-loss complaint rather than disproving it.
  - file: [extensions/memory-core/src/memory/manager.atomic-reindex.test.ts:44](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/memory-core/src/memory/manager.atomic-reindex.test.ts#L44)
