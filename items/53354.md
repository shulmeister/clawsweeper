## Summary

Current main already routes WebSocket chat traffic through `chat.send`, intercepts slash commands before any agent run, and has a regression test covering that behavior. The report's `agentTurn` observation matches a different RPC surface (`agent`/isolated assistant task), not the supported chat command path.

## Close Comment

Closing from a Codex review. Current `main` already implements this on the supported Gateway WebSocket chat surface: `chat.send` routes slash commands through the shared inbound dispatcher, and there is now a regression test proving a WS `chat.send` slash command (`/context list`) completes without starting an agent run (`src/gateway/server.chat.gateway-server-chat.test.ts:612`). The Gateway protocol docs also explicitly describe slash commands being handled through `chat.send` (`docs/gateway/protocol.md:222`), and `/model` is still registered as a built-in command (`src/auto-reply/commands-registry.shared.ts:833`). The `agent`/`agentTurn` path is a separate isolated-assistant RPC surface, not the normal chat command lane (`docs/web/control-ui.md:107`). This is present on current `main` at commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`, so I’m closing this as implemented on main. If you still see a problem on a fresh build, please open a new issue with the exact WS method being used (`chat.send` vs `agent`) plus a minimal request/response sample.
