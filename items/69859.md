## Summary

Keep PR #69859 open. Current main still does not accept or propagate `threadId` on `POST /hooks/agent`, so the requested capability is not implemented. The downstream cron delivery stack already supports explicit `delivery.threadId`, which means the right fix is narrower than this PR: parse `threadId` at the hook boundary and place it on the Cron job delivery object, with tests and docs. The submitted PR also needs revision because the provided context shows an accidental backup file and unrelated model-selection change bundled with the feature.

## Best Possible Solution

Keep the PR open for author/maintainer revision, or replace it with a narrow fix tracked by #69856: add `threadId?: string | number` to the hook agent payload, normalize it with the existing thread-value helper, include it in the hook replay/idempotency scope, set `delivery.threadId` in `dispatchAgentHook`, and add targeted hook/server tests plus docs for `POST /hooks/agent`. Do not add threadId to `CronAgentTurnPayloadFields`; current main intentionally normalizes thread routing through `CronDelivery`.
