## Summary

Keep open. Current main has partial mitigations: config writes validate before persisting, startup/hot reload can restore last-known-good config, and doctor can strip unknown keys during explicit repair. But the requested configurable strict/tolerant validation mode is not implemented, and normal read/startup validation still fails closed rather than stripping unknown keys and continuing. The linked tolerant-load PR #46213 remains open and unmerged.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A complete solution would add an explicit config validation policy, wire it through read/startup and write/control-plane paths, preserve strict write defaults, define logging for stripped unknown keys, and add regression tests covering root and nested unknown keys plus config.patch/config.apply behavior. The open #46213 work is relevant but should be reviewed against current last-known-good and doctor recovery semantics before landing.
