## Summary

Current main has some exact SecretRef deduping inside a single resolver call, but the reported startup duplicate-resolution class still looks real. Gateway startup can prepare runtime secrets twice when active gateway auth SecretRefs exist, and each prepare call creates a fresh resolver cache. The batch resolver also dedupes only refs passed in that one call; it does not consult/populate the per-ref cache for later resolver calls in the same context.

## Best Possible Solution

Keep the issue open. The likely maintainer fix is to make SecretRef resolution memoization truly startup/pass scoped: share a SecretRefResolveCache across gateway startup preflight and final activation where safe, make resolveSecretRefValues consult and populate resolvedByRefKey for batch calls, and consider narrowing gateway-auth preflight so it only materializes gateway auth refs instead of resolving the full runtime config twice. Provider-internal 1Password vault/item lookup caching should live in the 1Password exec/provider implementation unless OpenClaw adds a first-class provider API for those lookup stages.
