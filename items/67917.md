## Summary

Keep open. Current main fixes the stale Codex Responses backend route part of the report, but the OAuth TLS precheck complaint is still a concrete recent bug path: browser OAuth still hard-fails any TLS-cert-classified preflight error before OAuth starts, and I found no current test or implementation proof that the reporter's custom NODE_EXTRA_CA_CERTS setup is honored by that preflight/runtime path.

## Best Possible Solution

Keep the issue open for a targeted maintainer fix. The best path is to verify the OAuth preflight and runtime request path under the same macOS/custom-CA startup environment, add regression coverage for NODE_EXTRA_CA_CERTS/custom CA handling or an explicit diagnostic when it was not loaded before Node startup, and keep the already-shipped Codex Responses route canonicalization to /backend-api/codex/responses.

## Evidence

- **Issue evidence:** #67917 is a recent non-maintainer bug/regression report against 2026.4.15 with concrete commands and logs: OAuth precheck fails with UNABLE_TO_GET_ISSUER_CERT_LOCALLY and runtime openai-codex/gpt-5.4 requests fail with rawError=fetch failed despite the reporter's manual Node TLS probes succeeding with NODE_EXTRA_CA_CERTS.
- **OAuth browser login still blocks on TLS preflight failure:** The browser OAuth flow bootstraps the env proxy dispatcher, runs runOpenAIOAuthTlsPreflight(), and throws immediately when the result is kind "tls-cert".
