## Summary

Keep open. Current `main` still has the status-path mismatch the report describes: `status` builds plugin compatibility notices from the resolved config only, while plugin trust warnings are suppressed only when the original source config is supplied as the activation source. The issue comments also confirm reproduction on `v2026.4.23`, and I did not find a current-main change wiring `sourceConfig` into this path.

## Evidence

- **`status` still drops `sourceConfig` when checking plugin compatibility:** `scanStatus()` calls `buildPluginCompatibilitySnapshotNotices({ config: overview.cfg })` and does not pass `overview.sourceConfig`, even though the overview already carries both values.
  - file: [src/commands/status.scan.ts:66](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/commands/status.scan.ts#L66)
  - command: `nl -ba src/commands/status.scan.ts | sed -n '1,120p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Overview loader already has both resolved and source config:** `collectStatusScanOverview()` loads `sourceConfig` and `resolvedConfig: cfg` separately via `loadStatusScanCommandConfig(...)`, so the status command has the original config available but does not forward it into plugin compatibility loading.
  - file: [src/commands/status.scan-overview.ts:156](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/commands/status.scan-overview.ts#L156)
  - command: `nl -ba src/commands/status.scan-overview.ts | sed -n '140,230p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Plugin compatibility snapshot path only forwards `config`:** `buildPluginCompatibilitySnapshotNotices()` builds a snapshot report from `buildPluginSnapshotReport(params)` and only forwards the same params object; there is no separate `sourceConfig` input in this API.
  - file: [src/plugins/status.ts:413](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/plugins/status.ts#L413)
  - command: `nl -ba src/plugins/status.ts | sed -n '393,423p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Trust warnings depend on activation-source trust data:** `resolvePluginActivationSourceConfig()` only falls back to the source snapshot when the runtime snapshot was explicitly set; otherwise it returns `params.config`. In the status scan path, no runtime snapshot is set, so passing only resolved config means trust checks use resolved config instead of the raw source config.
  - file: [src/plugins/activation-source-config.ts:7](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/plugins/activation-source-config.ts#L7)
  - command: `nl -ba src/plugins/activation-source-config.ts | sed -n '1,40p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Loader warnings are keyed off trust lists merged from activation source:** `mergeTrustPluginConfigFromActivationSource()` merges `allow`, `deny`, and `loadPaths` from the activation-source config, and the warning emitters use the merged allowlist/provenance state. If `status` does not pass the original source config, the false warning remains plausible on current `main`.
  - file: [src/plugins/loader.ts:1083](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/plugins/loader.ts#L1083)

- **Reporter confirmations extend through latest shipped release:** Issue comments report the same two warnings on `v2026.4.23` with `lossless-claw 0.9.2`, and another commenter narrowed it specifically to `openclaw status` while `status --all` stayed clean. That matches the current unresolved status-only control flow above.
