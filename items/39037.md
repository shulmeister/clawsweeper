## Summary

Keep #39037 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still has no WhatsApp CTWA/referral extraction or propagation path: the inbound extractor can locate Baileys contextInfo, but current consumers only model mentions, replies, location/contact/media, and the only structured-context population path is for WhatsApp contact cards. The issue is contributor-authored, has no protected labels in the provided GitHub context, and the prior Apr 26 ClawSweeper keep-open review remains accurate against the current SHA.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin rather than closing it. Add a focused extractor for Baileys CTWA/referral data, including contextInfo.externalAdReply/ctwaClid-style fields and any Cloud-API referral-shaped inbound payload OpenClaw receives. Normalize ctwaClid/ctwa_clid, sourceUrl, sourceType, sourceId, headline/title, and related ad metadata into an untrusted structured context block and, if product wants durable attribution, an explicit session/hook metadata field. Add targeted unit tests for extraction, queueing into WebInboundMessage.untrustedStructuredContext, prompt rendering, and hook/session metadata behavior.
