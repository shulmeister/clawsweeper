## Summary

Keep open. Current main still permits host exec with no approval in the default gateway/node posture (`security="full"`, `ask="off"`), and docs explicitly state that this mode has no extra heuristic command-obfuscation or script-preflight approval layer. I did not find a package-name, self-uninstall, or destructive-command hard guard for `npm uninstall -g openclaw`; the only matching uninstall references are installer/uninstall docs and scripts, not exec safety enforcement. Related ClawSweeper context for #54416 also kept open the adjacent problem that model-initiated mutating OpenClaw commands through `exec` lack a hard confirmation boundary.

## Best Possible Solution

Keep this issue open for a maintainer-owned exec safety decision. The concrete product path is to add an execution-time confirmation or denial boundary for high-risk host commands, especially package-manager removal of OpenClaw itself (`npm uninstall -g openclaw`, equivalent package-manager forms, and confusing near-name targets), independent of model prompt compliance. The likely implementation belongs in the exec approval/analysis path with regression tests for full/off, allowlist/on-miss, and sandbox/gateway/node behavior, or a maintainer should explicitly document that trusted-operator YOLO mode accepts this risk.
