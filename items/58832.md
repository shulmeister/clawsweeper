## Summary

Keep open. Current main still appears to preserve the reported DM auto-thread session behavior when Slack `replyToMode` is `all`, rather than implementing the channel-style top-level-message exception for DMs.

## Evidence

- **Current Slack routing keeps top-level DMs thread-scoped when replyToMode=all:** `resolveSlackRoutingContext` computes `autoThreadId` from the message timestamp for non-thread replies when `replyToMode === "all"`, then uses it for non-roomish conversations; the inline comment says this preserves existing auto-thread behavior for DMs.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:3](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/extensions/slack/src/monitor/message-handler/prepare.ts#L3)
  - command: `web open raw.githubusercontent.com/openclaw/openclaw/main/extensions/slack/src/monitor/message-handler/prepare.ts`
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **Slack thread context returns message ts as MessageThreadId for top-level replyToMode=all:** `resolveSlackThreadContext` sets `messageThreadId` to `messageTs` when the message is not a thread reply and `replyToMode === "all"`, matching the reported `:thread:<message ts>` route metadata.
  - file: [extensions/slack/src/threading.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/extensions/slack/src/threading.ts)
