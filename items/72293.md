## Summary

Keep #72293 open. Current main has partial handling for lifecycle `stopReason: "aborted"` and for empty subagent announce output, but I did not find a source/test path proving the reported case is fixed: final assistant `content: []` with `stopReason: "aborted"` must be propagated from transcript/run result into subagent lifecycle status and parent-visible failure delivery.

## Best Possible Solution

Keep the issue open and add a focused regression around a subagent run whose final assistant message has `content: []`, `stopReason: "aborted"`, and an abort error message. The fix should propagate that terminal stop reason into run meta/lifecycle/session status (`abortedLastRun` or an explicit aborted/failed status) and ensure the parent receives a failure/completion announcement instead of relying on generic empty-output handling.
