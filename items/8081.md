## Summary

Keep #8081 open. Current main still documents OpenClaw as a one-trusted-operator/personal-assistant gateway, not a hostile multi-tenant or per-user RBAC boundary. The code has useful partial controls such as operator scopes, trusted-proxy user allowlists, config redaction, per-agent auth stores, and tool policy isolation, but it does not implement the requested admin/user/guest role model, per-user secret visibility, per-user credential isolation within one shared gateway, or audit model. The issue discussion also contains an explicit maintainer Codex review from steipete on 2026-04-26 keeping this issue open; external wrappers such as Pinchy, teamclaw, and openclaw-mission-control are useful related implementations, but they do not make the OpenClaw core request obsolete.

## Best Possible Solution

Keep #8081 open as the broad product/security tracker. A complete core direction should define identity for gateway/API/channel callers, role- or scope-based authorization for sensitive config and credentials, per-user or per-tenant secret isolation, audit logging, channel/group identity mapping, docs for the trust boundary, and regression tests across gateway auth, config reads/writes, HTTP APIs, and channel command surfaces. If maintainers choose an external-wrapper or ClawHub-first direction, document that decision and explicitly list the plugin/API surfaces required to build it outside core.
