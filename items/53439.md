## Summary

Keep open. Current `main` still implements the exact behavior the issue asks to change: the Synology Chat webhook handler returns `405` for any non-`POST` request, and its ACK helper still sends `204 No Content`. The corresponding tests on `main` also still expect `204` for successful webhook handling and `405` for non-`POST` methods, so there is no evidence this was already fixed on `main`.
