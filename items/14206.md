## Summary

Keep open. Current `main` does not implement a user-configurable outbound Feishu message rate limit or chunk pacing. The channel docs expose `streaming`, `textChunkLimit`, `typingIndicator`, and `resolveSenderNames`, but no minimum-interval setting, and the send path still emits chunked replies back-to-back with no delay between chunks.

## Evidence

- **Feishu docs expose workaround knobs, not rate limiting:** The channel docs say `streaming: false` sends the complete reply in one message and list quota optimizations as `typingIndicator` and `resolveSenderNames`, but they do not document any outbound rate-limit or minimum-interval option.
  - file: [docs/channels/feishu.md:251](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/channels/feishu.md#L251)
  - command: `nl -ba docs/channels/feishu.md | sed -n '228,268p'`
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Config table has no min-interval key:** The Feishu config reference lists `textChunkLimit`, `streaming`, `blockStreaming`, `typingIndicator`, and `resolveSenderNames`; there is no `messageInterval`, `rateLimit`, or similar outbound pacing setting.
  - file: [docs/channels/feishu.md:396](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/channels/feishu.md#L396)
  - command: `nl -ba docs/channels/feishu.md | sed -n '377,401p'`

- **Chunked replies are sent sequentially with no throttle:** `sendChunkedTextReply` iterates chunks and immediately awaits `sendChunk` for each one. The non-streaming text path then calls `sendMessageFeishu` per chunk, with no delay or rate-limiter between sends.
  - file: [extensions/feishu/src/reply-dispatcher.ts:322](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/extensions/feishu/src/reply-dispatcher.ts#L322)

- **Only streaming-card updates have a fixed internal throttle:** The streaming-card session hard-codes `updateThrottleMs = 100` and suppresses updates inside that window, but this is internal card-update throttling, not a user-configurable message send interval.

## Risks / Open Questions

- Users can partially mitigate API pressure with `streaming: false` or by disabling typing indicators, but long replies above `textChunkLimit` still split into multiple outbound sends with no pacing.
- There may be Feishu server-side behavior changes or undocumented SDK retries outside this checkout, but nothing in current `main` shows an explicit configurable outbound rate limiter.
