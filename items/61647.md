## Summary

Keep open. Current main has partial durable allow-always support, but the specific requested capability is still missing: interpreter-like commands can still fall back to exact command hashes, and Windows interpreter allow-always entries are intentionally argv-bound, so slight Python/script argument changes can still prompt again under on-miss policy.

## Evidence

- **allow-always falls back to exact command hash when no broader pattern can be derived:** Gateway approval handling persists derived allow-always patterns when possible, but if no pattern is produced it calls addDurableCommandApproval, which creates an =command:<hash> entry for the exact command text.
  - file: [src/agents/bash-tools.exec-host-gateway.ts:324](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/agents/bash-tools.exec-host-gateway.ts#L324)
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **exact command durable approvals are still implemented as command hashes:** addDurableCommandApproval hashes the normalized command text via buildDurableCommandApprovalPattern and stores it as source=allow-always.
  - file: [src/infra/exec-approvals.ts:810](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/infra/exec-approvals.ts#L810)
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **default interpreter-like allow-always does not persist executable-wide trust:** resolveAllowAlwaysPatternEntries returns no pattern for interpreter-like executables unless strictInlineEval is true and the invocation is not inline eval; the colocated test explicitly expects interpreter-like allow-always persistence to be empty.
  - file: [src/infra/exec-approvals-allowlist.ts:965](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/infra/exec-approvals-allowlist.ts#L965)
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **Windows interpreter durable approvals are argv-bound, not executable-wide:** The test for Windows strict-inline-eval interpreter approvals expects an argPattern on the executable entry and verifies a different argv no longer matches, which is the opposite of the requested trust-this-executable behavior.
  - file: [src/infra/exec-approvals-allow-always.test.ts:254](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/src/infra/exec-approvals-allow-always.test.ts#L254)
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
- **Docs describe the current limitation for inline interpreter eval:** The exec approvals docs state strictInlineEval makes inline interpreter eval forms require explicit approval and that allow-always does not persist new allowlist entries for them automatically.
  - file: [docs/tools/exec-approvals.md:229](https://github.com/openclaw/openclaw/blob/20c3177281d292d3db208b1e875e4b51a14fec94/docs/tools/exec-approvals.md#L229)
  - sha: [20c3177281d2](https://github.com/openclaw/openclaw/commit/20c3177281d292d3db208b1e875e4b51a14fec94)
