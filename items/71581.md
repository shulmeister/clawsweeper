## Summary

Keep open. Current main still appears to have the reported gap: exec-event and cron-event heartbeat wakes preserve the forced per-channel session key, while `session.dmScope=main` single-owner pinning is only applied on inbound DM routing. The related ClawSweeper items do not supersede this specific dmScope/heartbeat queue behavior.

## Best Possible Solution

Keep the issue open and implement a core heartbeat/session helper for `session.dmScope=main` plus the same single-owner allowlist precondition used by inbound DM routing. Apply it consistently to exec completion and cron-event enqueue/wake handling, either by re-keying system events to the main session at enqueue time or by carrying the original queue key while running the heartbeat in the main session. Add regression tests for background exec completion and cron wake with a per-channel DM session key under dmScope main, while asserting delivery stays pinned to the original destination.
