## Summary

Keep this issue open. Current main still has the reported Control UI auth-loop behavior when no cached device-token retry is possible, and `chat.history` still synchronously reads and parses the full transcript before applying response limits. The prior Codex review comment identified the same remaining work, and the current main SHA still matches that finding.

## Best Possible Solution

Fix this in core Control UI and Gateway history handling. Make AUTH_TOKEN_MISMATCH terminal when no trusted cached-device-token retry can be attempted, and after the single retry is exhausted. Separately, make chat.history avoid synchronous full-transcript read/parse work on the event loop before limits are applied, then add regression coverage for the no-retry auth-loop case and large-history responsiveness while channel timers remain healthy.
