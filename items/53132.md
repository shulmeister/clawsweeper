## Summary

The reported multi-account Discord startup regression from v2026.3.22 is implemented/fixed on current `main`, and the reporter later confirmed a clean upgrade to v2026.3.24 resolved the user-facing behavior.

## Evidence

- **Reporter confirmed fix in shipped release:** In the issue timeline, the reporter stated on 2026-03-25 that `v2026.3.24` resolved the problem on a clean install: all 4 Discord bot accounts were functional and responding, with only the pre-ready log line still appearing transiently.
- **Changelog shipped Discord gateway fix in 2026.3.24:** `CHANGELOG.md` for `2026.3.24` includes `Discord/gateway supervision`, describing the shipped fix that centralizes gateway error handling so startup/teardown gateway failures no longer surface as process-killing crashes.

## Risks / Open Questions

- The `awaiting gateway readiness` log text is still easy to misread, so separate Discord gateway issues can still be reported against the same message even though this specific multi-account v2026.3.22 regression was fixed.

## Close Comment

Closing as implemented on `main` based on this Codex review. The issue-specific regression was reported resolved by the original reporter in `v2026.3.24`, and the shipped changelog for that release includes the Discord gateway supervision fix (`CHANGELOG.md:2085`). Current `main` still contains that supervision/lifecycle plumbing in `extensions/discord/src/monitor/provider.ts:814` and `extensions/discord/src/monitor/provider.lifecycle.ts:143`. The `awaiting gateway readiness` message is also still intentionally a pre-READY status string (`extensions/discord/src/monitor/startup-status.ts:1`), so seeing that line briefly is not itself evidence of the old hang. If there is a new Discord gateway failure on a current release, please open a fresh issue with the exact version, account count, and logs.
