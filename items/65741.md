## Summary

Keep this PR open. Current main does not contain the PR's observable changes: `normalizeChatType()` still does not accept `supergroup`, Telegram group allowlist rejection still happens before any explicit binding bypass, and channel hot reload still restarts immediately after `stopChannel()`. The PR is not ready to close as implemented or obsolete, but it should be rebased and narrowed against the current `extensions/telegram` layout, with tests. The provided PR discussion is useful: Greptile's only P2 concern about applying the delay to every channel was addressed by the author's second commit, which scoped the delay to Telegram.

## Best Possible Solution

Keep this PR open and ask for a current-main rebase. The best implementation path is to add focused regression coverage first, then update current files: add `supergroup` normalization in `src/channels/chat-type.ts` if still needed by a real binding/config path; make Telegram group allowlist gating binding-aware in `extensions/telegram/src/bot-handlers.runtime.ts` or a small Telegram-owned helper without weakening disabled/sender authorization; and only add a Telegram-scoped hot-reload pause in `src/gateway/server-reload-handlers.ts` if it remains necessary after the existing polling 409 recovery. The PR should also include tests for bound-but-not-listed Telegram groups and the restart-delay scoping.
