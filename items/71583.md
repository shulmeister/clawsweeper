## Summary

Keep this issue open. The report is recent, specific, and still matches current main: QQ Bot token acquisition is still a bundled plugin path that posts directly to `bots.qq.com` without an explicit timeout or proxy/dispatcher-aware helper, and gateway connection waits on that token before opening the QQ WebSocket.

## Best Possible Solution

Keep this open as a bundled QQ Bot plugin reliability and troubleshooting issue. Add a bounded timeout and structured error metadata around `TokenManager.doFetchToken()`, route token acquisition through an existing proxy/dispatcher-aware helper or a new generic plugin transport seam if appropriate, add focused regression tests for timeout/abort behavior, and update `https://docs.openclaw.ai/channels/qqbot` plus channel troubleshooting with `bots.qq.com` token endpoint diagnostics and expected timeout signatures. Keep core generic unless the plugin fix proves a missing SDK seam is needed.
