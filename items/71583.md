## Summary

Keep open. The report is recent, concrete, and tied to the bundled QQ Bot plugin's token-fetch stage. Current main still fetches `https://bots.qq.com/app/getAppAccessToken` from the QQ Bot plugin, the gateway connection cannot proceed until that succeeds, and QQ Bot does not currently expose a `channels status --probe` credential/token probe or docs for token-endpoint routing/proxy diagnostics. The related ClawSweeper matches are generic timeout reports, not a canonical QQ token endpoint duplicate.

## Best Possible Solution

Keep the issue open for QQ Bot maintainer triage. The best repo path is a narrow QQ Bot diagnostic improvement: add a bounded `probeAccount` token-endpoint check, surface timeout/network/proxy hints in `channels status --probe`, and document QQ token endpoint troubleshooting including gateway `HTTPS_PROXY`/`HTTP_PROXY` behavior. If maintainers confirm it is purely Tencent/egress policy with no OpenClaw diagnostic gap, it can then be closed as external support with that evidence.

- **QQ token endpoint is still the active implementation:** Current main hardcodes the QQ token endpoint as `https://bots.qq.com/app/getAppAccessToken` and posts `appId`/`clientSecret` via `fetch`; network failures are surfaced as `Network error getting access_token`.
