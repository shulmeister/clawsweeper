## Summary

Current `main` already preserves custom model `compat` during inline provider resolution and now suppresses WebSocket `store:false` when `compat.supportsStore` is `false`, matching the issue request.

## Close Comment

Closing from a Codex review because current `main` already implements the requested behavior. For custom `models.providers.<id>.models[]` entries, the inline-provider path now preserves the configured model object directly (`src/agents/pi-embedded-runner/model.inline-provider.ts:155`), and model resolution uses that inline match as the resolved model (`src/agents/pi-embedded-runner/model.ts:425`). Compat normalization also preserves existing compat keys by spreading `...compat` before adding defaults (`src/plugins/provider-model-compat.ts:94-130`), so a user-set `compat.supportsStore: false` is retained instead of being dropped. On the WebSocket path, `store: false` is now gated through `supportsResponsesStoreField`, which explicitly turns off when `compat.supportsStore` is `false` (`src/agents/provider-attribution.ts:681-684`, `src/agents/openai-ws-request.ts:104-116`), and there is coverage for omitting `store` in that case (`src/agents/openai-ws-stream.test.ts:1843-1862`). This is present on current `main`; the relevant implementation traces to commit `38caa6832d4e1d344736d635b0c5fdad37e4430b`. I could not tie it to a shipped release tag, so `fixedRelease` is left unset.
