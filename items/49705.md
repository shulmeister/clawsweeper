## Summary

Current main already handles non-default Chromium profile attachment for `existing-session` via `browser.profiles.<name>.userDataDir` and Chrome DevTools MCP `--userDataDir`; the reported `cdpUrl`/`cdpPort` path is no longer the supported configuration for this driver.

## Risks / Open Questions

- If a reporter can still reproduce this on current main or v2026.4.22 with `driver: "existing-session"` plus the correct `browser.profiles.<name>.userDataDir` pointing at the real Chrome profile directory, that would be a separate bug worth a fresh report.
- The schema still accepts `cdpUrl` on profile objects generally, so users can write obsolete config even though the docs now say not to use it for `existing-session`.

## Close Comment

Closing from a Codex review because current `main` already covers this path. `existing-session` now attaches through Chrome DevTools MCP with explicit `browser.profiles.<name>.userDataDir` support, not through per-profile `cdpUrl`/`cdpPort`. The current implementation resolves `userDataDir` for `existing-session` profiles in `extensions/browser/src/browser/config.ts:316`, passes it to Chrome MCP as `--userDataDir` in `extensions/browser/src/browser/chrome-mcp.ts:214`, and has tests for that wiring in `extensions/browser/src/browser/chrome-mcp.test.ts:128` and `extensions/browser/src/browser/chrome-mcp.test.ts:366`. The docs also now explicitly say not to set `cdpUrl` for `driver: "existing-session"` and to use `browser.profiles.<name>.userDataDir` for non-default Chrome/Brave/Edge/Chromium profiles in `docs/tools/browser.md:163` and `docs/tools/browser.md:408`. This support is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
