## Summary

Current `main` intentionally scopes provider-specific TTS transport work to plugins rather than core. The repo vision says optional capability should usually ship as plugins/ClawHub, the architecture rules keep provider-specific behavior extension-owned, and the existing TTS stack already resolves speech providers from plugin capability contracts. This issue asks for a generic built-in adapter for arbitrary/custom protocols, which cuts against that direction rather than exposing a missing core seam.

## Risks / Open Questions

- This closes a real usability request for non-developers; the plugin path still requires someone to build and publish the adapter.
- If maintainers want a future no-code 'custom speech transport' builder, that would need a new generic plugin/SDK contract proposal rather than this broad core-native feature request.
