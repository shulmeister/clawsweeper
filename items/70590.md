## Summary

Keep open. Current main has real Control UI i18n infrastructure, supported locale bundles, a language picker, docs, and structural locale tests, but the specific request is about residual hard-coded English in the web UI. Source inspection found many current user-facing strings still rendered directly in UI components across chat, agents/tools, debug, nodes, skills, and cron surfaces, so the central problem is not implemented on main. The provided related ClawSweeper title matches are unrelated to web UI localization and do not supersede this issue.

## Best Possible Solution

Keep the issue open as a Control UI localization cleanup. The best path is to use the existing i18n pipeline: move remaining user-facing literals from web UI components into ui/src/i18n/locales/en.ts, preserve intentional product/protocol tokens, run the locale sync pipeline to regenerate non-English bundles and .i18n metadata, and add or extend a guard that catches new hard-coded user-facing English in UI templates where practical. Docs should continue pointing users to https://docs.openclaw.ai/web/control-ui for the language picker behavior.
