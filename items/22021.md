## Summary

Keep #22021 open. Current main still does not expose `X-Actual-Model` or an equivalent actual runtime provider/model on the OpenAI-compatible HTTP response surfaces. The prior maintainer/Codex review comment correctly identified the missing Gateway HTTP contract, and the current code still returns request-side agent-target model ids while the actual selected model callback remains inside the reply pipeline.

## Best Possible Solution

Keep the issue open. Define a Gateway HTTP contract for actual runtime provider/model metadata, thread selected provider/model from the fallback execution path into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`, choose explicit non-streaming and SSE semantics, document it on the Gateway HTTP API docs, and add focused tests for both compatibility endpoints.

## Risks / Open Questions

- Streaming semantics remain the main design risk: SSE headers are flushed before model fallback selection is observable, so the fix may need an explicit metadata event, trailer strategy, or delayed-header contract.
- The OpenAI-compatible `model` response field currently has documented agent-target meaning; any implementation should keep that separate from actual backend provider/model metadata.
