## Summary

Current main still has the reported fire-and-forget path for `sessionTarget: "main"` + `wakeMode: "next-heartbeat"`: it enqueues the system event, requests a heartbeat, and records `ok` without awaiting a heartbeat result. I did not find the requested ghost-run warning, configurable threshold, run-log warning flag, or a merged/superseding PR.

## Best Possible Solution

Keep the issue open. The best product/code path is a tested main-session cron change that either records these next-heartbeat handoffs as a distinct queued/handed-off state or adds the requested configurable ghost-run warning for `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `wakeMode: "next-heartbeat"`, with structured fields surfaced in cron logs/run history without changing isolated or wake-now semantics.
