## Summary

Keep open. Current main still has the reported Control UI failure mode: the UI treats any non-null `chatRunId` as busy, queues new messages behind it, and only drains that queue after the busy state clears from a terminal chat event or reconnect reset. Backend `agent.wait` already supports the recovery signal described in the issue, but the Control UI code does not call it or run a stale-run watchdog. The linked PR #42258 directly targets this gap, but it is open/unmerged in the provided context.

## Best Possible Solution

Keep #42213 open and land or rework #42258, or implement the same behavior directly: add a Control UI inactivity watchdog for the active `chatRunId`, call `agent.wait`, clear stale run/stream state and drain the queue only when the gateway reports terminal completion, and cover finished-run recovery plus still-active timeout behavior in focused UI tests.
