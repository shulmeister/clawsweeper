## Summary

Keep this issue open. Current main still rejects cross-agent session keys before checking spawned-session lineage, so a parent reading its own ACP child such as `agent:codex:acp:*` remains blocked under default `tree` visibility. The related fix PRs in the provided context are closed and unmerged, and current tests still do not cover the reported cross-agent spawned-child shape.

## Best Possible Solution

Keep this issue open and implement the narrow guard fix in `src/plugin-sdk/session-visibility.ts`: in `tree` mode, allow the requester session and sessions present in the spawned set before applying generic cross-agent rejection, while preserving agent-to-agent enforcement for non-spawned cross-agent targets. Add RED/GREEN coverage for `agent:main:main` reading a spawned `agent:codex:acp:<id>` child with default/tree visibility and no broad allowlist, ideally at both guard level and `sessions_history` tool level. The closed PRs #55983, #56019, and #56028 are useful references but are not merged.

## Risks / Open Questions

- Closing now would leave ACP orchestration unable to retrieve parent-owned child histories when the child session key uses a different agent id.
- The eventual fix must only bypass generic cross-agent policy for proven spawned-tree lineage; non-spawned cross-agent targets should remain governed by `tools.sessions.visibility` and `tools.agentToAgent`.
