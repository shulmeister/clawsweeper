## Summary

Current main still does not fix the reported same-command exec loop. Exec outcomes still fall through to the generic outcome hash that includes raw details, while exec results contain volatile fields such as durationMs, pid, and startedAt. Because the no-progress detectors require identical result hashes, repeated successful exec calls can still evade critical blocking; the remaining fix is already represented by the open PR #41502, but it has not shipped on main.

## Best Possible Solution

Keep #34574 open until main lands an exec-specific stable outcome hash for exec results, plus regression tests for repeated successful exec calls whose only changing fields are durationMs, pid, startedAt, sessionId, or cwd. After that ships, close this as implemented and track the broader result-similarity detector separately if maintainers want that larger behavior change.
