## Summary

Keep #71040 open. Current main still lacks `activeJobIds` mark/clear coverage for startup catch-up and manual cron runs, while task maintenance still uses `isCronJobActive(jobId)` as the authoritative in-process backing signal after the 5-minute grace. The later durable recovery logic can recover completed cron runs from run logs or stored job terminal state, but it does not protect an in-flight long-running manual or startup catch-up run before completion. Related #68191 remains broader durable reconciliation work; this PR is a focused, still-useful fix rather than a duplicate or already-shipped change.

## Best Possible Solution

Keep the PR open for maintainer review and likely merge, or land an equivalent patch that adds `markCronJobActive`/`clearCronJobActive` with `try/finally` coverage to startup catch-up and manual run execution, plus the proposed producer-side regression tests. Treat #68191 as the canonical broader follow-up for durable post-restart cron task reconciliation rather than as a superseding replacement for this focused live-run fix.
