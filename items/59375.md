## Summary

Current main already covers the reported workflow: agents can discover other sessions, target them by label plus optional agent id instead of only exact session keys, and use the existing `sessions_send` A2A follow-up path to deliver a completion announce back to the visible channel or thread. The same relevant implementation is already present in the shipped `v2026.4.23` release.

## Evidence

- **Session discovery already exposes channel metadata:** `sessions_list` is documented to return each session's key, agentId, kind, and channel, which addresses the issue's claimed lack of a discovery mechanism for finding the right session.
  - file: [docs/concepts/session-tool.md:27](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/docs/concepts/session-tool.md#L27)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **`sessions_send` supports label-based targeting:** The current tool schema accepts `label` and optional `agentId`, then resolves that to a session with `sessions.resolve`; it no longer requires only an exact `sessionKey`. Cross-agent sends are explicitly gated through `tools.agentToAgent`.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/tools/sessions-send-tool.ts#L37)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Cross-agent visibility is a supported configuration path:** The config help states `tools.sessions.visibility="all"` allows targeting any session, with cross-agent access additionally gated by `tools.agentToAgent`.
  - file: [src/config/schema.help.ts:705](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/config/schema.help.ts#L705)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Completion announce-back is already part of peer session flow:** The ACP/session docs state that `sessions_send` can target another session, wait for its reply, optionally run bounded follow-up turns, then ask the target to produce an announce message and deliver that announce to the visible channel or thread.
  - file: [docs/tools/acp-agents.md:357](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/tools/acp-agents.md#L357)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Tests cover label lookup and announce routing:** Gateway and tool tests verify both finding a target by label and the peer-session ping-pong/announce flow, including direct delivery back to the target chat context.
  - file: [src/gateway/server.sessions-send.test.ts:161](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/gateway/server.sessions-send.test.ts#L161)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Announce target hydration reuses stored channel/thread routing:** The announce-target resolver reads `sessions.list` delivery context and last-known channel/thread metadata so completion notices can route back to the originating chat or thread instead of needing a new callback system.
  - file: [src/agents/tools/sessions-announce-target.ts:34](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/tools/sessions-announce-target.ts#L34)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Shipped release matches current main on the relevant session-tool implementation:** A release-to-main diff over the core session-send/list/announce-target code and session-tool docs is empty, so the current behavior is already shipped in `v2026.4.23`.

## Risks / Open Questions

- This evidence covers cross-session and cross-agent messaging within one OpenClaw gateway/session store; it does not prove cross-gateway notifications between completely separate deployments.
- There is still no first-class `sessions_send(channel: ...)` shortcut; the supported path is discovery via `sessions_list` and/or targeting by session label.
