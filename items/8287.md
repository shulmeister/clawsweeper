## Summary

Keep #8287 open. Current main still does not implement node-registered agent tools: nodes can declare caps/commands and receive `node.invoke`, and plugins can wrap nodes through `api.runtime.nodes`, but connected nodes cannot advertise first-class LLM tools with schemas, lifecycle, catalog entries, or node provenance.

## Best Possible Solution

Keep #8287 open for a maintainer-approved additive Gateway protocol and runtime design: define a validated node tool-advertisement schema, store advertised tool lifecycle on connected `NodeRegistry` sessions, synthesize dynamic agent tools plus `tools.catalog`/`tools.effective` entries with node provenance, route calls through `node.invoke`, and document naming, conflicts, pairing, allowlists, and policy behavior. Keep `api.runtime.nodes` and openclaw-tescmd-style plugins as current workarounds, not the final zero-plugin model.

## Risks / Open Questions

- Dynamic node-advertised tools need explicit security and policy design so they cannot bypass pairing, declared command allowlists, node-local approvals, or `tools.allow`/`tools.deny`.
- The remaining design needs stable naming, conflict handling, reconnect idempotency, catalog/effective-inventory representation, and user-facing policy semantics.
