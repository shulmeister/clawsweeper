## Summary

Current main still does not implement the requested node-advertised agent-tool protocol/lifecycle. Nodes can advertise command names and plugins can call `api.runtime.nodes`, but there is no `tools` field in the node connect contract, no node tool definitions stored in `NodeRegistry`, no synthesized per-node agent tools, and no node source in the tool catalog/effective inventory. The existing openclaw-tescmd workaround and `api.runtime.nodes` are valid current workarounds, not replacements for the requested zero-plugin node registration model.

## Best Possible Solution

Keep #8287 open for a maintainer-approved additive gateway protocol/runtime design: define a validated node tool-advertisement schema, store advertised tools on `NodeRegistry` with connect/disconnect lifecycle, synthesize agent tools plus catalog/effective-inventory entries from connected nodes, route calls through `node.invoke`, and document naming plus policy behavior. Keep `api.runtime.nodes` and openclaw-tescmd as current plugin workarounds, not as the final implementation.

## Risks / Open Questions

- Dynamic node tools would need explicit security review so advertised tools cannot bypass pairing, command allowlists, node-local approval, or `tools.allow`/`tools.deny` behavior.
- The remaining design needs stable naming, conflict handling, reconnect idempotency, catalog display, and policy matching for node-provided tools.
