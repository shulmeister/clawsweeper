## Summary

Current `main` still matches #36754: `discoverInDirectory` scans extension roots with `fs.readdirSync(..., { withFileTypes: true })`, handles regular files, then skips anything where `entry.isDirectory()` is false. A symlinked plugin directory therefore still falls through before package, bundle, or index discovery. The documented global/workspace extension roots use this scanner, while the `plugins.load.paths` workaround remains a separate `fs.statSync()` path that follows symlinks. The April 15 confirmation and the closed-unmerged fix PRs in the provided GitHub context remain relevant.

## Best Possible Solution

Keep #36754 open. The best fix is a narrow change in `discoverInDirectory` that resolves symlink Dirents safely before the directory/file decision, deliberately skips or diagnoses broken symlinks, reuses the existing package/runtime entry safety checks, and adds a regression test proving a symlinked plugin directory under `~/.openclaw/extensions` or `<workspace>/.openclaw/extensions` is discovered.
