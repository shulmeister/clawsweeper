## Summary

Keep #10944 open. Current main still lacks a supported `channels.telegram.parseMode` or per-account `parseMode` config, while Telegram sends, media captions, edits, delivery helpers, and streaming previews continue to use hardcoded HTML formatting with plain-text fallback only after parse/render failures. The issue discussion already corrected the original Markdown-vs-HTML detail and the April 26 Codex review also kept this broader configurable parse-mode request open; closed #10758 is narrower custom-emoji/raw-HTML work, not a canonical replacement.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface. A conservative path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, then route it through account resolution and every Telegram outbound path. Update strict schema, exported config types, generated config metadata/help, public Telegram docs, and regression tests for current HTML behavior plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji support as explicit maintainer/security decisions or narrower follow-up items.
