## Summary

Keep #10944 open. Current main still lacks a `channels.telegram.parseMode` or per-account parse-mode config, while Telegram outbound sends, captions, edits, draft previews, and lower-level delivery helpers remain HTML-first with plain-text fallback only. The provided issue discussion also includes a same-day maintainer/Codex comment explicitly keeping this open, and related #10758 is a narrower closed custom-emoji/raw-HTML request rather than a canonical tracker for this config work.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface. A conservative path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired through normal sends, media captions, edits, draft previews, and delivery helpers. Update the strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and focused tests for HTML plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji as an explicit maintainer/security choice or a separate follow-up if needed.
