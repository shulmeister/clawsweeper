## Summary

Keep open. Current main still makes executeWithApiKeyRotation retry only while rotating across multiple configured API keys, and the default retry predicate is still limited to rate-limit/quota patterns. The media understanding audio/video call sites do not pass a transient-error shouldRetry override, so the requested 5xx/timeout/network retry with a single key remains missing.

## Evidence

- **Default retry condition remains rate-limit-only:** executeWithApiKeyRotation formats the caught error, then uses params.shouldRetry when provided, otherwise falls back to isApiKeyRateLimitError(message). It also stops when attempt + 1 >= keys.length, so a single configured key gets no retry attempt.
