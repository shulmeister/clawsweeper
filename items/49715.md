## Summary

Keep open. Current `main` still has the reported Slack `replyToMode="first"` block-streaming gap by inspection: block replies still consume `replyPlan.nextThreadTs()` without reusing an already-established reply thread, and the regression-test/helper changes from this PR are not present in the checked-out code.

## Evidence

- **Normal deliveries still exhaust the first-mode planner:** `deliverNormally` still computes `replyThreadTs` as `forcedThreadTs ?? replyPlan.nextThreadTs()`. In `replyToMode="first"`, once the first threaded send consumes the planner, later normal/block deliveries fall back to `undefined` instead of reusing an established thread.
