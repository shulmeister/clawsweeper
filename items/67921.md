## Summary

Keep open. Current `main` already tolerates individual channel startup failures and continues starting later channels, so the PR’s core problem statement is partly obsolete. The remaining change would alter the documented/runtime ordering contract by letting `gateway:startup` hooks run before channel startup finishes, which matches the existing review concern and needs maintainer judgment rather than conservative cleanup closure.

## Risks / Open Questions

- The PR description assumes channel failures currently block later channel startups, but current `main` already catches those failures and keeps starting later plugins.
- Detaching `startChannels()` would introduce a race between gateway readiness/startup hooks and channel readiness, which can silently break startup automations that depend on channel state.
- The PR does not show updated docs or regression tests for the changed startup ordering, so merging as-is would weaken an existing contract without proof.
