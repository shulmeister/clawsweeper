## Summary

Keep open. Current main still scopes explicit custom QMD collection names when the path resolves inside the agent workspace, while preserving explicit names only for shared paths outside the workspace. The PR targets a real remaining behavior gap, but it is not safe to close or merge as-is because the runtime compatibility path does not yet handle scoped-to-unscoped renames for existing collections.

## Best Possible Solution

Keep the PR open and ask for a compatibility-safe version: preserve explicit custom names as the target behavior, add a scoped-to-unscoped migration/rebind path for existing QMD collections across JSON, text, and name-only `collection list` outputs, cover the upgrade case in `extensions/memory-core` tests, rename the misleading tests, and update the multi-agent/QMD docs before merge.

## Risks / Open Questions

- Merging the PR as-is could strand existing in-workspace explicit custom collections named `<name>-<agent>` while searches start targeting `<name>`, especially with older qmd output that lacks path metadata.
- Using the same explicit name across multiple agents for different workspace-relative paths could collapse distinct managed collections into one QMD collection unless the product contract explicitly accepts that tradeoff.
- Tests and docs need to be updated together so the new rule is clear: explicit names are authoritative, while unnamed custom collections and default memory collections remain generated/scoped.
