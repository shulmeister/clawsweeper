## Summary

Keep #47335 open. Current main has mitigation around run-level block pipeline flushing, but the specific compaction-retry race described in the issue is still present in the lifecycle ordering: compaction retry wait is resolved before the async block/channel flush is awaited, and the attempt path still has no post-wait flush. Related PR #62157 remains open and tracks a related final-payload suppression problem, not a merged fix for this compaction rollover race.

## Evidence

- **Lifecycle resolves compaction retry before awaited channel flush:** handleAgentEnd starts ctx.flushBlockReplyBuffer(), immediately calls finalizeAgentEnd(), and finalizeAgentEnd resolves ctx.resolveCompactionRetry() when pendingCompactionRetry > 0. The later flushPendingMediaAndChannel/onBlockReplyFlush promise is awaited only after the compaction wait has already been resolved.

## Risks / Open Questions

- A top-level runReplyAgent blockReplyPipeline.flush() may mitigate executions where the retry block reply has already been enqueued, but it does not prove the lifecycle race is gone because flushBlockReplyBuffer/onBlockReply can still be asynchronous before enqueueing completes.
- No live compaction rollover reproduction was run because this review was constrained to read-only inspection; the current source and test ordering are sufficient to avoid an automatic close.
