## Summary

Current main still implements `before_tool_call` and `after_tool_call` as plugin hook surfaces, not as managed/internal `HOOK.md` hook events. The internal hook event type, dispatcher, loader, docs, and tests do not provide the blocking/param-rewrite bridge requested by #13364, and the provided discussion identifies this issue as the canonical open request after #17008 was closed as its duplicate.

## Best Possible Solution

Keep #13364 open as the canonical product/API request unless maintainers explicitly decide that tool-call interception should remain plugin-only. A complete fix would add typed internal `tool_call` before/after events, define how managed handler return values block or rewrite params, bridge tool execution without double-firing plugin hooks, and update `docs/automation/hooks.md`, `docs/plugins/hooks.md`, and focused managed `HOOK.md` tests.
