## Summary

Keep #13364 open. Current main still exposes `before_tool_call` and `after_tool_call` through the plugin hook runner, but managed/internal `HOOK.md` hooks remain a void-returning internal event system with no `tool_call` event type, blocking result contract, parameter rewrite contract, or tool-execution bridge. The provided discussion also has fresh maintainer signal keeping #13364 open as the canonical tracker, with #17008 closed as duplicate and #29590 closed unmerged as only a partial after-tool-call attempt.

## Best Possible Solution

Keep #13364 open for maintainer product/API design. A complete fix should add typed managed/internal before/after tool-call events, define blocking and parameter-rewrite return semantics, reject or warn on unsupported HOOK.md event names, bridge the relevant tool execution paths without double-firing existing plugin hooks, and add focused internal-hook tests plus public docs. If maintainers later decide tool-call interception should stay plugin-only, close with an explicit product decision and point users to plugin hooks or ClawHub packaging.
