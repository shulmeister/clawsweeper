## Summary

Keep open. Current main still exposes `before_tool_call` and `after_tool_call` as plugin hooks, but not through the `HOOK.md` internal/managed hook system requested by #13364. The internal hook dispatcher remains void-only and typed around command/session/agent/gateway/message events, while tool execution calls the global plugin hook runner directly. The provided discussion also supports this as the canonical open request: #17008 was closed as a duplicate of #13364, commenters added independent security/audit/TTS/workflow use cases, and a maintainer already posted a keep-open review on April 26, 2026.

## Best Possible Solution

Keep #13364 open as the canonical product/API request unless maintainers explicitly decide that tool-call interception must remain plugin-only. A complete implementation would add a typed internal `tool_call` before/after event contract, define how internal handler return values can block or rewrite params, bridge tool execution to that contract without double-firing plugin hooks, and update `docs/automation/hooks.md`, `docs/plugins/hooks.md`, and focused tests for managed `HOOK.md` behavior. If maintainers reject the internal-hook bridge, the issue should be closed only with an explicit product decision and docs that clearly explain the plugin-only alternative.
