## Summary

Keep this issue open. Current main has related startup-performance work, setup-runtime loading, and post-listen sidecar startup, but it still does not implement the requested lazy channel connect behavior: there is no consumed `channels.lazyConnect` option, configured channel plugins are still loaded for gateway startup, and `startChannels()` still starts every registered channel rather than connecting on first inbound/outbound use.

## Best Possible Solution

Keep the issue open and scope an explicit lazy channel lifecycle design. A maintainer-quality implementation should add an additive config or successor to `channels.lazyConnect`, a plugin/manifest capability for channels that can safely defer `startAccount`, startup behavior that registers routes/status without connecting eligible channels, first outbound/inbound connection where feasible, clear `registered` versus `connected/running` status semantics, readiness behavior that matches the selected policy, and regression coverage for configured multi-account channels plus channels that must remain eager.

## Risks / Open Questions

- The existing setup-runtime and post-listen sidecar work can be mistaken for this feature, but it does not provide the requested global config, first-message connect semantics, or registered-vs-connected lifecycle split.
- A real implementation needs per-channel safety declarations because webhook, polling, socket, outbound-only, health-monitor, and native-command channels may not all support lazy connection the same way.
