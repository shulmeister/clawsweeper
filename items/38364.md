## Summary

Keep open. Current main still lacks an inbound Control UI reply-to/parent metadata path into `chat.send` and the agent `MsgContext`. The no-output/silent-token side is partially handled by current UI/Gateway filtering, but that does not satisfy the central reply metadata request.

## Best Possible Solution

Keep the issue open and implement a first-class, backward-compatible reply metadata contract for `chat.send`: add typed request fields for the replied-to message id and optional summary/content, collect that state in Control UI reply interactions, validate it in the Gateway protocol, populate `MsgContext.ReplyToId`/`ReplyToBody` or a documented successor, and add focused protocol, gateway, and UI tests. Keep the existing silent-token/no-output filtering as shipped behavior while adding tests or UX wording only for any remaining non-`NO_REPLY` empty-completion cases.
