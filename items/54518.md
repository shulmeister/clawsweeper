## Summary

Keep this issue open. Current main still treats an agent workspace as the default working directory, not a hard filesystem boundary. The code preserves `tools.fs.workspaceOnly` as an opt-in guard, tests assert host read/write/edit can access outside the workspace when it is unset or false, and exec/process remain separate host/sandbox policy surfaces.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A complete fix would add a per-agent confinement profile or onboarding option that sets `tools.fs.workspaceOnly: true`, denies or sandboxes runtime tools like `exec` and `process`, and updates skill guidance to avoid fixed `~/.openclaw/workspace` paths. If maintainers want hard workspace boundaries by default, that needs an explicit migration plan and regression coverage across file tools, exec workdir handling, sandbox modes, plugin/MCP tools, media paths, and per-agent overrides.

## Risks / Open Questions

- Changing the default workspace boundary would be a compatibility and product-policy change because current code, tests, changelog history, and docs intentionally preserve host-file access unless `tools.fs.workspaceOnly` or sandbox/tool policy is configured.
- `tools.fs.workspaceOnly` alone is not full confinement: exec/process, elevated mode, plugin tools, MCP tools, and channel/provider-specific file paths also need explicit sandboxing or tool-policy coverage.
- Skill instructions that hard-code `~/.openclaw/workspace` can still steer an agent toward the wrong workspace unless skills are updated to refer to the active workspace or use workspace-relative paths.
