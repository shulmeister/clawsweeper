## Summary

Keep open. Current `main` does not already implement the startup conflict-detection feature, and the PR as described is not a safe/obviously-obsolete cleanup: key checks are either already enforced differently on `main` or would be incomplete against current agent-specific config behavior.

## Evidence

- **No equivalent startup hook on main:** Current `main` has no `detectConfigConflicts`/`logConfigConflicts` surface, and `startGatewayServer()` goes through `resolveGatewayRuntimeConfig(...)` into runtime setup without any config-conflict logging hook.
  - file: [src/gateway/server.impl.ts:376](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/server.impl.ts#L376)
  - command: `rg -n "detectConfigConflicts|logConfigConflicts|config/conflicts" src docs; nl -ba src/gateway/server.impl.ts | sed -n '320,470p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **One advertised PR warning is already handled as a hard runtime guard:** `main` already refuses non-loopback binds without shared-secret auth: `resolveGatewayRuntimeConfig()` throws when the effective bind host is not loopback and auth is neither token/password nor trusted-proxy. That means the PR's `gateway.bind exposed + auth.mode=none` startup warning is not evidence of a missing mainline implementation and would need to align with effective runtime overrides, not just raw config.
  - file: [src/gateway/server-runtime-config.ts:141](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/server-runtime-config.ts#L141)
  - command: `nl -ba src/gateway/server-runtime-config.ts | sed -n '40,170p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Proposed detector pattern can introduce new startup failures:** `resolveSandboxConfigForAgent()` always resolves SSH sandbox config, and `normalizeRemoteRoot()` throws if `sandbox.ssh.workspaceRoot` is not an absolute POSIX path. A warning-only startup detector that calls this unconditionally can turn a non-fatal config quirk into a boot-time failure.
  - file: [src/agents/sandbox/config.ts:177](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/sandbox/config.ts#L177)
  - command: `nl -ba src/agents/sandbox/config.ts | sed -n '170,270p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Current runtime behavior is agent-specific, so default-only conflict checks would miss real cases:** `resolveExecConfig()` merges per-agent `agents.list[].tools.exec` overrides over global `tools.exec`, and `resolveSandboxConfigForAgent()` accepts an `agentId` to apply per-agent sandbox overrides. A detector that inspects only the default/global view would produce false negatives for exactly the agent-specific conflicts this PR is meant to surface.
  - file: [src/agents/pi-tools.ts:194](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/pi-tools.ts#L194)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '194,260p'; nl -ba src/agents/sandbox/config.ts | sed -n '220,260p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
