## Summary

Keep open. Current `main` has partial fixes for threaded Slack announce routing, but the issue is not fully implemented: internal A2A turns still enter the target session as `webchat`, `resolveGroupSessionKey()` still cannot recover `agent:<id>:slack:...` from that context, and `sessions_list` still prefers persisted `origin.provider` over `lastChannel`, so a new Slack thread target can still surface as `webchat` in session metadata.

## Evidence

- **Internal A2A delivery still enters the target session as webchat:** `src/agents/tools/sessions-send-tool.ts:275-280` sends `sessions_send` via `deliver: false` with `channel: INTERNAL_MESSAGE_CHANNEL`, and `src/utils/message-channel-constants.ts:1` defines that channel as `webchat`. `src/gateway/server-methods/chat.ts:2297-2307` then builds the target `MsgContext` with `Provider: INTERNAL_MESSAGE_CHANNEL` and `Surface: INTERNAL_MESSAGE_CHANNEL` for that turn.
  - file: [src/agents/tools/sessions-send-tool.ts:275](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/tools/sessions-send-tool.ts#L275)
  - command: `nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '270,288p'; nl -ba src/gateway/server-methods/chat.ts | sed -n '2297,2307p'; nl -ba src/utils/message-channel-constants.ts | sed -n '1,10p'`
