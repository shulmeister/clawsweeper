## Summary

Keep this PR open. Current main has adjacent mitigations and documentation for restrictive `plugins.allow`, but the live advisory/error surfaces named by the PR still do not consistently warn that a non-empty `plugins.allow` is exclusive. The PR is not cleanly obsolete; it should be rebased against current main, especially the split from `audit-extra.async.ts` into `audit-plugins-trust.ts`, and should add focused assertions for the updated user-facing messages.

## Best Possible Solution

Keep the PR open as a useful implementation candidate, but ask for a rebase against current main and small test updates that assert the exclusivity warning in the loader, security audit remediation, CLI missing-plugin messages, and config validation warning. Avoid resurrecting the earlier reviewed generated allowlist suggestions; the safer direction is explanatory text plus existing auto-enable/allowlist preservation behavior.
