## Summary

Keep #6792 open. Current main still lacks manifest-level install-time `configPatch`: the native manifest type/parser/docs do not expose it, install result/persistence paths cannot carry or apply it, and the related implementation PR #43421 was closed unmerged. The request is a concrete core plugin install/API seam, not a ClawHub-only plugin idea.

## Best Possible Solution

Keep #6792 open for a maintainer design/implementation decision. A complete fix should add a typed manifest-level config intent or equivalent install-time config API, parse and sanitize it from native manifests, propagate it through linked path, copied path/archive, npm, marketplace, ClawHub, and bundled install paths, apply a validated merge during install persistence with explicit array/conflict semantics and visible diff/skip output, add uninstall messaging if patches were applied, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/tools/plugin.

## Risks / Open Questions

- Manifest-declared config mutation is trust-sensitive and would need sanitization, prototype-pollution protection, validation before write, conflict handling, and clear diff/skip UX.
- Array semantics remain unresolved: the original issue proposes replacement, while the tools.alsoAllow use case likely needs additive union behavior to avoid clobbering another plugin’s entries.
- The related closed bug #31251 shows plugin install/uninstall config writes can be fragile, so any implementation should prove unrelated config keys are preserved.
