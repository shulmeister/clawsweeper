## Summary

Keep open. Current main still does not implement manifest-level install-time `configPatch`; the only shipped `configPatch` path is provider-auth scoped, and the closed related PR #43421 was not merged. The issue discussion remains actionable because service/tool plugins still need a core install/config contract to avoid manual `openclaw.json` edits.

## Best Possible Solution

Keep this issue open for a core plugin install/API decision. A complete fix should add a documented typed patch contract, parse and sanitize it from plugin metadata, propagate it through all install sources, apply a validated merge in the install persistence path, handle allowlist-style arrays deliberately, show the applied diff or skip reason, and cover path/link/archive/npm/ClawHub/marketplace/bundled plus invalid-patch cases in tests.

## Risks / Open Questions

- A future implementation needs deliberate merge semantics, especially arrays such as `tools.alsoAllow`, to avoid plugin installs clobbering each other.
- A manifest-level patch is a trust-sensitive install-time config mutation, so validation, diff output, skip/override UX, and prototype-pollution guards should be part of the design.
