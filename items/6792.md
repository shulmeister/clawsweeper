## Summary

Keep #6792 open. Current main still does not implement manifest-level install-time `configPatch`; the existing `configPatch` support is limited to provider-auth onboarding flows, not plugin manifests or `openclaw plugins install`. The related implementation PR #43421 was closed unmerged, and the latest issue discussion already contains an explicit keep-open review.

## Best Possible Solution

Keep #6792 open for maintainer design and core implementation. The best path is a typed manifest-level config intent or equivalent install-time plugin config API that is parsed and sanitized from `openclaw.plugin.json`, validated after merge, propagated through local/link/archive/npm/marketplace/ClawHub/bundled install flows, documents conflict and array semantics, shows a user-visible diff with a skip/no-patch option, gives uninstall guidance, and updates tests plus public docs such as `docs/plugins/manifest.md` and `docs/plugins/building-plugins.md`.

## Risks / Open Questions

- Install-time config mutation is trust-sensitive and still needs design for sanitization, validation, conflict handling, visible diffs, and a skip path before it should ship.
- Array semantics remain unresolved: the original proposal suggests replace-on-array, while the `tools.alsoAllow` use case in the issue comments likely needs additive/union behavior to avoid clobbering other plugin entries.
- Plugin install/uninstall config writes have related history (#31251) around unrelated config drift, so a future implementation needs regression coverage proving unrelated keys and includes are preserved.
