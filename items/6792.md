## Summary

Current main still does not implement manifest-level install-time `configPatch`. The manifest type/loader omit it, install results do not carry it, all plugin install routes persist only install metadata and enablement, and the user-facing docs still tell users to configure plugins manually after install. The related implementation PR #43421 was closed unmerged, and the existing `configPatch` path remains provider-auth scoped.

## Best Possible Solution

Keep this issue open for a core plugin install/API decision. A complete fix should add a typed manifest-level `configPatch` contract, parse and sanitize it from plugin metadata, propagate it through every install source, apply a validated merge in config persistence with explicit array semantics, show the applied diff or skip reason, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/tools/plugin.

## Risks / Open Questions

- Manifest-declared config mutation is trust-sensitive; a future implementation needs validation, sanitization, prototype-pollution guards, and clear diff/skip UX.
- Array semantics need an explicit contract so additive arrays such as `tools.alsoAllow` do not clobber other plugins' entries.
- Coverage should include local path, link, archive, npm, ClawHub, marketplace, bundled fallback, invalid patch, and config include/write-through cases.
