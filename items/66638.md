## Summary

Keep #66638 open. Current main has useful pieces from the related notification work: subagent/ACP runs are tracked as background tasks, completion announces can go through direct channel delivery with idempotency and a queue fallback, and plugins have subagent spawn/end hooks. It still does not implement the issue’s core request: per-run immediate spawn/progress/completion/failure/cancel user-facing events, a durable announce queue, or a delivery path fully decoupled from the requester session lane. The docs still describe one final completion update and explicitly say pending announce-back work is lost on gateway restart.

## Best Possible Solution

Keep the issue open as a core architecture/product item. The best path is to design a first-class run lifecycle notification pipeline with durable per-run event storage, stable idempotency keys, per-run ordering, immediate framework spawn ACKs, progress/completion/failure/cancel events, and safe channel-direct delivery that does not depend on the busy requester session lane. It should coordinate with #22358 for extension hooks and #54488 for lane-starvation behavior.

## Risks / Open Questions

- Implementing this should reuse the existing completion direct-delivery, idempotency, task registry, and plugin hook machinery to avoid duplicate notifications.
- This is core delivery architecture, not a ClawHub-only plugin request, because the requested guarantees involve runtime event creation, persistence, session-lane bypass, and channel delivery semantics.
