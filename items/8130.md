## Summary

Keep #8130 open. Current main still has internal fallback attempt tracking, lifecycle events, Control UI fallback status, and verbose-only final reply notices, but it does not implement proactive active-channel retry/failover notifications for normal users or the requested notification config knobs. Related duplicate issues were closed while #8130 remains the canonical remaining UX gap, and the latest issue discussion explicitly says to keep it open.

## Best Possible Solution

Keep #8130 open and implement this in the core reply-runner/model-fallback path, not as a cleanup close. Add a configurable retry/failover notice policy, wire failed-attempt/fallback transitions to the existing active-channel block reply pipeline, sanitize provider failure summaries, and add tests proving non-verbose users receive delayed retry/fallback notices without duplicating verbose or Control UI observability.
