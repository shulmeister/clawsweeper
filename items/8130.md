## Summary

Keep #8130 open. Current main has fallback observability in logs/lifecycle events, session fallback state, status output, and a final verbose-only fallback prefix, but it still does not proactively notify the active channel during retry/failover when verbose mode is off. The requested `failoverNotify`/`retryNotifyAfterMs` or equivalent model-fallback notification config is also absent. The issue discussion, including the recent maintainer/Codex review comment, identifies this as the canonical remaining UX gap; the related issues are closed cross-references, not an open replacement.

## Best Possible Solution

Keep #8130 open as the core gateway/reply-runner UX gap. Implement a configurable proactive notification path near `runWithModelFallback`, likely by wiring sanitized candidate-failure and successful-fallback events through existing block reply delivery/onBlockReply after an optional delay threshold, with regression tests proving visible retry/failover notices work when verbose mode is off and do not duplicate the current final verbose notice.
