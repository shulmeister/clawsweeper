## Summary

Current `main` still does not implement proactive active-channel retry/failover notifications for model fallback. OpenClaw now tracks fallback state, emits lifecycle events, and can prepend a final verbose-only fallback notice, but non-verbose users still receive no channel notice during provider retry/failover and the requested notification config surface is absent.

## Best Possible Solution

Keep #8130 open and implement this in the core reply-runner/model-fallback path. Add a configurable retry/failover notice policy, wire failed-attempt and successful-fallback transitions into active-channel reply delivery, sanitize provider error summaries, support delayed retry notices, and add tests proving non-verbose users receive retry/fallback notices without duplicating existing verbose or lifecycle observability.

## Risks / Open Questions

- Provider/API error summaries must be sanitized before posting them to user channels.
- Notice delivery needs to avoid duplicate messages with existing verbose fallback output, lifecycle events, session/status indicators, and all-models-failed error payloads.
- The implementation needs careful threading/streaming behavior across block replies, delayed retry notices, silent/system events, and delivery failures.
