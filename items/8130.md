## Summary

Keep #8130 open. Current main has partial fallback observability, but it still does not implement proactive active-channel retry/failover notifications during the silent window. The shipped behavior is lifecycle events plus a final verbose-only "Model Fallback" prefix; verbose-off users still get no in-channel fallback notice, and the proposed failover notification config is absent.

## Best Possible Solution

Keep #8130 open as the canonical core UX gap. Implement a gateway/reply-runner notification path near `runWithModelFallback`, likely by wiring sanitized candidate-failure and successful-fallback events through the existing reply/block delivery channel, making the behavior configurable with a clear default, and adding regression tests proving visible retry/failover notices work with verbose off without duplicating the existing final verbose notice.
