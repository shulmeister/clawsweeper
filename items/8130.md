## Summary

#8130 should remain open. Current main still has fallback logs, lifecycle/status state, and a final verbose-only fallback notice, but it does not proactively notify the active user channel during retry/failover with verbose mode off, and there is no model-failover notification config.

## Best Possible Solution

Keep #8130 open as the canonical core gateway/reply-runner UX gap. Implement a configurable proactive notification path near `runWithModelFallback`, likely by wiring sanitized candidate-failure and successful-fallback events through existing active-channel block reply delivery after an optional delay threshold, with regression tests proving visible retry/failover notices work when verbose mode is off and do not duplicate current verbose notices.

## Risks / Open Questions

- Implementation should sanitize provider/API error text before sending it to channels.
- New proactive notices should avoid duplicating the existing verbose-only final fallback prefix, lifecycle events, and status fallback state.
- Delivery should respect existing block streaming, threading, silent/system-event, and reply failure policies.
