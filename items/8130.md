## Summary

Keep open. Current main has partial model-fallback visibility through verbose-mode final reply notices, persisted fallback state, and lifecycle events, but it does not implement the requested proactive active-channel notification during retry/failover. The feature remains meaningful, especially because verbose defaults to off and the existing notice is emitted only after a successful fallback run is ready to return a response.

## Evidence

- **Requested behavior:** Issue #8130 asks for active-channel API error/retry, failover, and optional recovery notifications; the follow-up comment specifically says a native gateway-level notification would be more reliable because it would catch failover before the fallback model runs.
