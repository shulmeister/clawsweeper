## Summary

Keep PR #69084 open. Current main still gates live Gateway/WebChat heartbeat chat suppression on `showOk`, while the config contract and heartbeat runner treat `showOk` and `showAlerts` as separate controls. The PR identifies a plausible mismatch for `showAlerts=false`, but the one-line `showAlerts` swap is not safe enough to close or supersede because it can suppress OK-only heartbeat acknowledgments when `showOk=true`.

## Best Possible Solution

Keep the PR open, but ask for a content-aware Gateway/WebChat fix instead of a single flag swap: suppress or buffer heartbeat streaming safely, then decide the final chat payload by classifying the heartbeat text. OK-only output should follow `showOk`; non-OK alert content should follow `showAlerts`. Add focused server-chat tests for the default case, `showOk=true/showAlerts=false` with `HEARTBEAT_OK`, `showOk=true/showAlerts=false` with alert content, and `showOk=false/showAlerts=true` with alert content. If the broader #69492 event-audience work lands first, close this only after it proves the same live WebChat cases.

## Risks / Open Questions

- Closing now could drop a plausible live WebChat bug for `showOk=true` and `showAlerts=false`, where alert-like heartbeat output may still be broadcast because current main only checks `showOk`.

- The current Gateway chat tests lack a visibility matrix for OK-only versus alert heartbeat output, so either close or merge would be under-supported without new targeted coverage.
