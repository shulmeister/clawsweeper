## Summary

Current main does not provide enough evidence to close this as fixed. The exec path still turns timeouts into SIGKILL and reports generic timeout/signal failures, with no crontab-specific macOS handling or clearer diagnostic for the reported case. The follow-up CLI concern is also not fully resolved: cron RPCs still require admin operator scope, scope-upgrade pairings remain manual, and tests explicitly preserve that CLI clients do not use the backend self-pairing bypass even when shared-token auth succeeds.

## Best Possible Solution

Keep this issue open. Best path is to reproduce `crontab -l` and `echo ... | crontab -` through exec on current macOS LaunchAgent, then either fix exec handling/diagnostics for noninteractive crontab failures or document the supported alternative. Separately, decide whether same-host CLI cron commands with valid shared-token auth should bypass stale device scope baselines, provide a terminal approval flow, or keep requiring explicit Control UI approval for admin scope upgrades.
