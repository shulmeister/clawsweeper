## Summary

The proposal is coherent, but on current `main` it fits the existing plugin/context-engine extension model rather than OpenClaw core. OpenClaw already exposes dedicated slots and SDK seams for alternate cross-session recall and memory-adjacent knowledge layers, and project vision explicitly keeps optional capability out of core and on the plugin/ClawHub path.

## Risks / Open Questions

- Closing as `clawhub` may disappoint users who wanted a blessed built-in default rather than an extension path.
- The issue body frames this as an architectural proposal, so maintainers could still choose to keep it open as product-direction input despite the strong plugin fit.

## Close Comment

Thanks for the proposal. This Codex review checked current `main`, and the feature fits OpenClaw’s existing plugin architecture much better than a new core behavior. Current `main` already has a dedicated context-engine slot for alternative cross-session recall/assembly behavior (`docs/concepts/context-engine.md:10`, `docs/plugins/manifest.md:707`, `src/plugins/types.ts:2111`), plus the bundled `memory-wiki` plugin for a compiled disk-backed knowledge layer with machine-readable digests and shared search integration (`docs/plugins/memory-wiki.md:10`, `extensions/memory-wiki/index.ts:23`, `extensions/memory-wiki/README.md:145`). There is also already a `/new` / `/reset` lifecycle hook that persists session context into memory (`src/hooks/bundled/session-memory/handler.ts:51`). Given `VISION.md:54`, where optional capability is expected to ship as plugins and not core by default, this is best pursued as a plugin/context-engine published through the normal ClawHub/plugin path rather than an OpenClaw core issue.
