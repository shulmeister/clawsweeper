## Summary

Keep PR #38991 open. Current main still does not backfill finite non-negative `createdAtMs` and `updatedAtMs` for legacy cron jobs before `cron.list` returns them, while the Gateway schema and macOS model still require both fields. The PR’s core change remains useful, though its unrelated `.secrets.baseline` and Fastfile churn should be removed or split before merge.

## Best Possible Solution

Keep #38991 open or replace it with an equivalent scoped patch. The best implementation is to backfill finite non-negative `createdAtMs` and `updatedAtMs` during cron store/service load, persist only when legacy storage is actually mutated, and extend regression coverage so both `cron.list` and the persisted store expose numeric timestamps for legacy jobs. Drop the unrelated baseline/Fastfile churn before merge.
