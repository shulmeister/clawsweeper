## Summary

Keep open. Current `main` has agent-session sandboxing and tool-policy controls, but it does not implement the requested unified built-in plugin/skill sandbox with per-plugin process isolation, manifest-declared fs/net/env/resource caps, multi-platform native backends for plugins, or tiered sandbox presets.

## Evidence

- **Plugin runtime is still in-process:** The plugin architecture doc says native OpenClaw plugins are loaded in-process via jiti and register capabilities into a central registry, which does not match the requested separate-process plugin sandbox model.
  - file: [docs/plugins/architecture.md:132](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/plugins/architecture.md#L132)
  - command: `nl -ba docs/plugins/architecture.md | sed -n '120,150p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Loader executes plugin code directly:** The loader imports plugin modules with `getJiti(...)(...)` and then calls `runPluginRegisterSync(register, api)` in-process. The same loader also mutates `process.env.NODE_PATH` for plugin runtime deps, reinforcing that plugins share host process state.
  - file: [src/plugins/loader.ts:2405](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/plugins/loader.ts#L2405)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2390,2420p'; nl -ba src/plugins/loader.ts | sed -n '2695,2718p'; nl -ba src/plugins/loader.ts | sed -n '492,500p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Manifest contract has no sandbox capability block:** `PluginManifest` includes config, auth, activation, setup, QA, skills, contracts, and channel config metadata, but no per-plugin sandbox declaration for env/fs/net/resources/process isolation like the issue proposes.
  - file: [src/plugins/manifest.ts:157](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/plugins/manifest.ts#L157)
  - command: `nl -ba src/plugins/manifest.ts | sed -n '157,231p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Current sandboxing is agent-session scoped:** The sandbox docs and config reference define `agents.defaults.sandbox.mode`, `backend`, and `scope` for agent sessions, with backends `docker`, `ssh`, and `openshell`. They do not describe plugin-level isolation or tiered presets such as `standard`/`strict`/`paranoid`.
  - file: [docs/gateway/sandboxing.md:39](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/gateway/sandboxing.md#L39)

- **Security model still treats plugins as trusted code:** `SECURITY.md` explicitly says plugins/extensions are part of the trusted computing base and that enabling a plugin grants it the same trust level as local code on the gateway host. That is the opposite of the requested default isolated-plugin model.
  - file: [SECURITY.md:123](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/SECURITY.md#L123)
