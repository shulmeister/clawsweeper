## Summary

Keep #37816 open. Current main still lacks an opt-in Telegram mode that mirrors Web UI agent activity as durable, separate messages. The bundled Telegram plugin supports edited answer/reasoning previews, short tool/progress lines inside that edited preview, and default tool-progress messages when preview streaming is off, but not persistent per-activity messages for reasoning, tools, command output, memory, and other agent events.

## Best Possible Solution

Keep #37816 open as bundled Telegram plugin work. Add an explicit opt-in channels.telegram activity visibility mode or flag, with schema, config UI hints, docs, and Telegram tests. Reuse existing replyOptions and agent-event streams where sufficient; if reasoning, memory, or command-output details are not exposed cleanly enough, add a narrow additive channel/plugin event contract. Preserve current preview-edit behavior by default and include NO_REPLY/internal-run suppression, rate-limit controls, and sensitive-output safeguards.

## Risks / Open Questions

- A durable per-activity Telegram mode could be noisy and may hit Telegram Bot API rate limits, especially in groups or tool-heavy turns.
- Reasoning, memory, command output, and tool result snippets can expose sensitive data, so the feature should be explicit opt-in with filtering/redaction controls.
- Internal housekeeping turns such as NO_REPLY and memory flushes need deliberate suppression so background activity does not leak into user-facing chats.
