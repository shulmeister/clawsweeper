## Summary

Keep #20934 open. Current main still lacks REST-native `/v1/sessions/reset`, `/v1/sessions/status`, `/v1/sessions/compact`, `/v1/sessions/model`, or `/v1/sessions/reasoning` routes for HTTP `/v1/chat/completions` clients. The requested operations exist on the Gateway RPC/control plane, and there are narrow HTTP kill/history routes, but that does not satisfy the issue’s REST session-management request. The related discussion also treats this as unresolved: #23941 was the direct REST attempt and is closed unmerged, while #29985 was scoped to plugin SDK reset and explicitly left REST endpoints as follow-up.

## Best Possible Solution

Keep #20934 open as the canonical REST session-management tracker. The best implementation is an authenticated Gateway HTTP surface for reset, status/list or usage, compact, and model/reasoning patch operations that reuses the existing `sessions.reset`, `sessions.list`, `sessions.compact`, and `sessions.patch` semantics, with route/auth/scope tests and public docs under the OpenAI-compatible HTTP API.
