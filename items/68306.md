## Summary

Keep open. Current main still accepts zero and negative Telegram reply-to message IDs in the shared normalizer, and the send-time call sites forward any non-null normalized value to Telegram parameters. Existing regression coverage only covers non-numeric/NaN inputs, while PR #68306 adds the missing positive-value guard and focused tests.

## Best Possible Solution

Keep #68306 open and either merge it after maintainer review or apply the same `> 0` validation in `normalizeTelegramReplyToMessageId`, with regression tests for numeric and string zero/negative values while leaving `parseTelegramThreadId` semantics unchanged.

## Evidence

- **Current normalizer permits 0 and negatives:** `parseIntegerId` accepts an optional leading minus sign, the numeric branch returns any finite truncated number, and the string branch returns parsed integers without a `> 0` check.
