## Summary

Current `main` still accepts `0` and negative Telegram `replyToMessageId` values in `extensions/telegram/src/outbound-params.ts`, and multiple send paths forward those normalized values to Telegram. The PR addresses a real validation gap and adds missing tests, so this should remain open for maintainer review rather than conservative cleanup.

## Evidence

- **Current parser still allows zero and negative reply IDs:** `normalizeTelegramReplyToMessageId` truncates any finite number and returns any parsed integer string without enforcing `> 0`, so `0`, `-1`, and `"-5"` are currently treated as valid reply targets.
  - file: [extensions/telegram/src/outbound-params.ts:9](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/telegram/src/outbound-params.ts#L9)
  - command: `nl -ba extensions/telegram/src/outbound-params.ts | sed -n '1,220p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current send paths forward normalized reply IDs:** Outbound send helpers attach `reply_to_message_id` whenever the normalized value is not `null`/`undefined`, so negative values from the current parser can reach Telegram API calls on `main`.
