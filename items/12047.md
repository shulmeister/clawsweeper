## Summary

Keep open. Current main no longer has the original placeholder-only behavior for every Telegram document: Telegram now saves inbound document files into `MediaPaths`, and reply processing runs generic media/file understanding that can inline text-like attachments and route PDFs through a dedicated extraction path. But the request explicitly called out `.docx` support, and current file extraction still classifies `application/vnd.*` Office-style documents as binary and skips them. That makes this a partial implementation, not strong enough evidence for closure.

## Evidence

- **Telegram documents are downloaded and stored as local media:** `resolveMedia()` fetches the Telegram file, saves it, and returns the saved path plus MIME type, even for document messages.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:404](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/telegram/src/bot/delivery.resolve-media.ts#L404)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Telegram forwards saved attachment paths into agent context:** Inbound Telegram context writes `MediaPath`, `MediaPaths`, and `MediaTypes`, so downstream processing sees the saved document attachment instead of only the placeholder text.
  - file: [extensions/telegram/src/bot-message-context.session.ts:366](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/telegram/src/bot-message-context.session.ts#L366)

## Risks / Open Questions

- The issue body explicitly asks for `.docx` support, and current main still blocks Office vendor MIME types from file extraction.
- Behavior is partially improved already, so any future maintainer follow-up should likely narrow the scope to the remaining unsupported document formats instead of re-triaging it as fully unresolved.
