## Summary

Keep #12047 open. Current main partially addresses the original Telegram placeholder-only problem for plain text, Markdown, JSON/CSV/HTML, and PDF attachments through the shared media-understanding file path, but the issue explicitly asks for .docx/Office-style document extraction. Current main still treats application/vnd.* Office documents as binary and has a regression test asserting those attachments are skipped, so the requested capability remains meaningfully unresolved.

## Best Possible Solution

Keep tracking this as an OpenClaw issue. The best implementation is to extend the shared media/document extraction path so supported Office/document MIME types such as .docx can be routed through a safe document-extractor contract, with size limits, untrusted-content wrapping, tests, and docs. Telegram should continue to pass downloaded MediaPath/MediaType into that shared path rather than owning document parsing itself.

## Risks / Open Questions

- Current main already improves the original report for PDF and text-style attachments, so a future fix should extend the shared media/document extraction path rather than adding Telegram-only extraction.
- The SDK has a document-extractor subpath, but the current inbound attachment pipeline only routes PDF through document extractors and skips Office MIME types before extraction, so a .docx extractor alone would not satisfy this issue without core routing changes.
