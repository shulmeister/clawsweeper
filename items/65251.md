## Summary

Keep open. Current main has bundle-backed LSP support, but it is explicit/static: enabled Claude bundles can contribute `.lsp.json` or manifest-declared `lspServers`, and the runtime can spawn configured stdio servers for hover, definition, and references tools. I did not find project file/package scanning, PATH discovery of installed language servers, install/fix prompts, per-language auto-configuration, or a status/dashboard flow. The related #65252 does not supersede this because that item is about real-time diagnostics, while this issue is about auto-detection and setup.

## Best Possible Solution

Keep open for product/API design. A complete path would add a safe project detection layer, discover installed language servers without surprising installs, expose explicit opt-in install/fix actions, support user overrides, and surface LSP status through CLI/UI diagnostics. If maintainers want this externalized, the core task should be a documented plugin or bundle API for dynamic per-workspace LSP server configuration.
