## Summary

Keep PR #68046 open. Current `main` still normalizes MCP HTTP `tools/call` result blocks down to `{ type, text }`, so the underlying image/audio/resource data-loss bug is still present. The PR is useful, but it also has unresolved review feedback about validating known MCP block shapes before pass-through, so it needs maintainer/reviewer follow-up rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open and have the contributor or maintainer revise `normalizeToolCallContent` to preserve valid MCP content blocks for `text`, `image`, `audio`, `resource`, and `resource_link`, while falling back to safe text normalization for strings, unknown types, and malformed typed objects. Add focused tests in the MCP HTTP gateway coverage for each valid block type plus malformed fallback cases, then land it as the canonical fix for issue #68037.
