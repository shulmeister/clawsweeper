## Summary

Keep this issue open. Current main has a post-run path that can persist a freshly resolved runtime context window, but the `/status` rendering path can still prefer an existing stale `sessionEntry.contextTokens` over the context window for the active `channels.modelByChannel` model, and the status command itself only builds a reply without clearing or updating that persisted field.

## Best Possible Solution

Teach `/status` context resolution to detect the effective `channels.modelByChannel` model for the target session and ignore or clear stale `entry.contextTokens` when there is no user session model override and the persisted value belongs to a prior model. Add a regression test with a session entry carrying `contextTokens: 1048576`, a channel override to a smaller MiniMax-style model, and an expected `/status` context line using the active model's configured window.
