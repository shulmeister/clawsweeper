## Summary

Current `main` still has the mismatch the issue describes. `openclaw health --json` builds Telegram health from config inspection plus probe results only, then feeds that partial snapshot into the Telegram summary helper, which defaults missing lifecycle fields to `running=false`, `lastStartAt=null`, `lastStopAt=null`, and `tokenSource="none"`. By contrast, `channels.status`/`status --deep` passes the live runtime snapshot through the account-status adapter, so the two surfaces can still disagree exactly as reported.

## Evidence

- **`health` path drops runtime lifecycle state:** `getHealthSnapshot()` constructs a per-account snapshot with only `accountId`, `enabled`, `configured`, optional `probe`, and `lastProbeAt`, then calls `plugin.status.buildChannelSummary(...)`. It does not include runtime fields such as `running`, `lastStartAt`, `lastStopAt`, or credential metadata.
