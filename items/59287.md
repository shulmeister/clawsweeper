## Summary

Current main still matches the reported bug. `openclaw health --json` is built from config/probe-only channel snapshots, while `channels.status` / `status --deep` uses the live gateway channel runtime snapshot. For Telegram, the reduced health snapshot can still omit `running`, `lastStartAt`, `tokenSource`, and related lifecycle fields; the shared summary helper then defaults them to `false`, `null`, and `none`. The issue is recent, was reaffirmed for Telegram on 2026-04-18 against 2026.4.15, and a 2026-04-26 comment reports the same class for Slack on 2026.4.24.

## Best Possible Solution

Keep #59287 open as the canonical narrow Telegram health JSON bug, while recognizing that PR #42586 or an equivalent fix is the likely implementation path. The best fix is to make gateway health summaries use the same runtime-backed account snapshot path as `channels.status`, including per-account runtime lifecycle and credential-source fields, then add regression coverage for Telegram and at least one passive-token-source channel such as Slack.
