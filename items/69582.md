## Summary

Keep #69582 open. The report is recent, non-maintainer-authored, and has no protected labels, but current main does not clearly implement the requested protection: malformed `process(action={})` still flows into the process tool as a runtime value, the process schema advertises `action` only as a broad string rather than a flat enum, loop detection is opt-in/disabled by default, and existing parameter validation does not sanitize trailing `{}` on otherwise string values.

## Best Possible Solution

Keep the issue open and fix it in core/tool-runtime behavior. A conservative path is to harden OpenClaw-owned tool schemas and runtime validation: make `process.action` a flat string enum using the existing `stringEnum` helper, reject non-string or unknown process actions with a clear non-retryable tool result before requiring `sessionId`, and add regression coverage for `{ action: {} }` plus trailing `{}` contamination on file/edit string fields. If maintainers choose sanitization, keep it narrow and schema-backed so valid user text is not silently rewritten.

## Evidence

- **Issue reports concrete malformed tool args on GLM/local model:** The provided issue body identifies `local/glm-5` on OpenClaw 2026.3.24 with corrupted values such as `edit.newText`/`edit.path` receiving trailing `{}` and `process.action` receiving `{}`; the follow-up comment says the reporter reproduced across March and April versions.
- **Process schema does not constrain action to valid commands:** `processSchema` defines `action` as `Type.String({ description: "Process action" })`; there is no flat enum for `list`, `poll`, `log`, `write`, `send-keys`, `submit`, `paste`, `kill`, `clear`, or `remove`, so the schema does not give providers the strongest available contract.
