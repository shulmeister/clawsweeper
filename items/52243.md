## Summary

Current main already exposes inbound attachment file paths to agent runs: inbound media is surfaced as `MediaPath`/`MediaPaths`, staged into `media/inbound/*` for sandboxed sessions, injected into the native agent prompt as `[media attached: ...]`, and preserved in inbound claim metadata. The shipped v2026.4.22 tree already contains this behavior.

## Close Comment

Closing as implemented on `main` after Codex review. OpenClaw already exposes inbound attachment paths to agent runs: the docs explicitly advertise `{{MediaPath}}` as a local temp file path (`docs/start/openclaw.md:181`), sandboxed sessions stage inbound files into `media/inbound/*` and rewrite `MediaPath`/`MediaPaths` accordingly (`src/auto-reply/reply/stage-sandbox-media.ts:22`, covered by `src/auto-reply/reply.triggers.trigger-handling.stages-inbound-media-into-sandbox-workspace.test.ts:168`), and the native reply runner injects those attachment paths into the agent prompt via `[media attached: ...]` notes (`src/auto-reply/reply/prompt-prelude.ts:33`, with coverage in `src/auto-reply/reply.media-note.test.ts:6`). This behavior is already present in the shipped `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
