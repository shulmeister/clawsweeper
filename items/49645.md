## Summary

Current `main` already handles assistant-side 429/rate-limit failures inside the embedded runner retry loop before terminal completion, and the same logic is present in shipped release `v2026.4.22` (`00bd2cf7`). The report matches older `2026.3.13-1` behavior, not current OpenClaw.

## Evidence

- **Assistant-side 429 is classified as retryable before completion:** `resolveRunFailoverDecision()` treats assistant-side failover signals, including `rate_limit`, as `rotate_profile` on the first pass instead of letting the run complete normally.
  - file: [src/agents/pi-embedded-runner/run/failover-policy.ts:88](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/pi-embedded-runner/run/failover-policy.ts#L88)

- **Rate-limit assistant failures return an inline retry outcome:** `handleAssistantFailover()` escalates `rate_limit` through the failover path and, when profile rotation succeeds, returns `{ action: "retry" }` rather than allowing terminal lifecycle completion.
  - file: [src/agents/pi-embedded-runner/run/assistant-failover.ts:145](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/agents/pi-embedded-runner/run/assistant-failover.ts#L145)

## Close Comment

Closing from a Codex review because current OpenClaw already implements the requested behavior. Assistant-side rate-limit failures are treated as retry/failover candidates in `src/agents/pi-embedded-runner/run/failover-policy.ts:88`, `src/agents/pi-embedded-runner/run/assistant-failover.ts:145`, and `src/agents/pi-embedded-runner/run.ts:1641`, and the runner loops on that retry path before it reaches `embedded run done` at `src/agents/pi-embedded-runner/run.ts:2148`. There is also regression coverage for assistant-side 429 classification in `src/agents/pi-embedded-runner/run/failover-policy.test.ts:64`. I also checked the shipped `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and the same retry-before-completion logic is present there. This looks like older `2026.3.13-1` behavior rather than a current-main bug. If you can still reproduce it on `v2026.4.22` or newer, please open a new issue with fresh logs and exact provider/profile configuration.
