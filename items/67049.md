## Summary

Keep #67049 open. The bug is still real on current main, and the PR is a plausible direction, but it is incomplete: current main still only clears followup/command queues during Gateway reset, chat `/new` still reaches prompt construction where queued system events are drained into the next user prompt, and the PR discussion already identifies required fixes before merge.

## Best Possible Solution

Keep the PR open and ask for an update: drain stale system events for every deliberate fresh-session rollover that preserves the same session key, including Gateway `sessions.reset`/delete cleanup and chat `initSessionState()` resets. Use `clearSessionQueues()`'s normalized returned keys, drain only after successful reset cleanup/mutation, and add focused regression tests for both Gateway RPC reset and chat `/new`/`/reset` so #66864 can close with confidence.

## Risks / Open Questions

- Merging the PR as provided would cover only the Gateway RPC reset surface and leave the normal chat `/new`/`/reset` path leaking stale system events.
- The raw drain loop proposed in the PR can diverge from `clearSessionQueues()` normalization and may throw on empty or whitespace keys.
- Draining before reset cleanup is known to have a data-loss edge case if the reset later fails, so the final fix should drain only after the reset is committed or otherwise known to proceed.
