## Summary

Keep #23451 open. Current main still has exec approval and plugin-level `before_tool_call.requireApproval` primitives, but it does not implement the requested built-in `tools.confirmation` risk-tier policy or an assistant-text-aware pause before the first same-response tool executes. The issue discussion already contains a 2026-04-26 maintainer-side Codex review reaching the same keep-open conclusion, and current source at `631552c554095b0c520376db9131dac0f5427f94` still supports that.

## Best Possible Solution

Keep #23451 open for maintainer/product work at the agent tool-execution boundary. The likely path is a documented `tools.confirmation` config/schema/help/docs surface, deterministic tool risk classification, access to same-response assistant text before first tool execution, user-visible flush before holding tools, approval/timeout handling, and clear execute/skip transcript semantics while preserving plugin `before_tool_call.requireApproval` as the lower-level custom policy primitive.

## Risks / Open Questions

- The eventual fix needs careful transcript and channel UX semantics so held, approved, denied, or timed-out tool calls are visible and deterministic across chat channels.
