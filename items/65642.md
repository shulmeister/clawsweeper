## Summary

Keep this PR open. Current main still does not implement the central behavior: the OpenResponses streaming handler only reacts to assistant and lifecycle events, the terminal response snapshot only includes the assistant message, and the output schema still lacks `function_call_output` as an output item. Internal tool start/result events already exist elsewhere, so this PR remains a plausible focused implementation candidate rather than cleanup fodder.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this PR or an equivalent core gateway change that listens to internal `stream: "tool"` events in OpenResponses streaming, emits stable `function_call` and `function_call_output` output items, includes them in terminal response snapshots, safely preserves raw HTTP tool output without leaking it on the shared event bus, normalizes response-shaped next-turn inputs, and adds focused gateway/agent-event regression coverage.
