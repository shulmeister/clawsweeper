## Summary

Keep open. The most recent maintainer/Codex review in the issue already found the bundled Google Chat plugin still lacks user OAuth support, and current `main` at `c3833f7729163a8f6cdc47d3b0be86fd5fcab160` still appears to have the same service-account-only config, schema, credential resolution, auth scope, and API token path. This is not a conservative cleanup close candidate.

## Best Possible Solution

Keep the issue open until the bundled Google Chat plugin grows an explicit user OAuth credential model, schema/config/docs, safe secret handling, token selection for reactions/uploads/proactive DMs, preserved service-account behavior for bot/webhook operations, and tests for fallback when user OAuth is absent. If maintainers decide this should move outside the bundled plugin, that product/API decision should be made explicitly rather than handled by automated cleanup.
