## Summary

Keep #70905 open. Current main carries `agentId` through several upstream outbound paths, but the public `ChannelOutboundContext` handed to channel adapters still does not expose it. This is a concrete public plugin SDK gap, not a stale report, duplicate, or ClawHub-only plugin request.

## Best Possible Solution

Add optional `agentId?: string` to `ChannelOutboundContext`, thread it into `createChannelOutboundContextBase` from the existing delivery session/mirror agent scope, and cover all adapter callback paths (`sendText`, `sendMedia`, `sendPayload`, formatted text/media) with focused tests. Update the plugin SDK docs/API baseline so third-party channel plugins can rely on the new optional field.

## Risks / Open Questions

- The requested change is an additive public plugin SDK contract change, so it should include API baseline/export checks, plugin docs, and direct delivery tests for text, media, formatted, and payload adapter paths.
- Care is needed to define the exact fallback source, likely `params.session?.agentId ?? params.mirror?.agentId`, and to keep the field optional for human/system sends without an agent in scope.
