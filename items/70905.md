## Summary

Keep this issue open. Current main still does not expose a stable `agentId` on `ChannelOutboundContext` or pass one through `createChannelOutboundContextBase`, even though the upstream outbound tool/send paths already carry `agentId` before direct channel delivery. This remains a public channel plugin SDK plumbing gap, not a ClawHub-only plugin request or duplicate of the title-search related items.

## Best Possible Solution

Add an optional documented `agentId?: string` to `ChannelOutboundContext`, thread a single resolved value such as `params.session?.agentId ?? params.mirror?.agentId` through `ChannelHandlerParams` and `createChannelOutboundContextBase`, and verify it reaches `sendText`, `sendMedia`, `sendPayload`, `sendFormattedText`, `sendFormattedMedia`, and `renderPresentation`. Add focused tests for inbound reply, sessions_send/message-tool, heartbeat/autonomous, media, and no-agent/system cases, plus channel plugin docs clarifying stable agent attribution versus display `identity`.

## Risks / Open Questions

- The exact precedence should be specified and tested when both `session.agentId` and `mirror.agentId` are present and differ.
- Cron announce delivery currently has at least one direct `deliverOutboundPayloads` call without a `session`, so implementation should audit all delivery call sites rather than only the message tool path.
- Because this changes a public plugin SDK context type, it needs contract tests and docs updates in addition to internal plumbing.
