## Summary

Keep open. Current `main` still has the two message-tool regressions described for default-valued poll and Discord `components` inputs, although the workspace-media portion appears to be at least partially addressed on current `main` via agent-scoped media roots.

## Evidence

- **Poll guard still treats non-zero defaults as poll intent:** `hasPollCreationParams()` still returns `true` for any non-zero numeric poll value. A default-like `pollDurationHours: 24` would therefore still count as poll creation intent even without a `pollQuestion`.
  - file: [src/poll-params.ts:49](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/poll-params.ts#L49)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Send path still throws the reported poll error:** The message runner still throws `Poll fields require action "poll"...` whenever `action === "send"` and `hasPollCreationParams(params)` is true, so the numeric-default case above still maps to the reported failure.
  - file: [src/infra/outbound/message-action-runner.ts:927](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/infra/outbound/message-action-runner.ts#L927)

- **Discord components parser still rejects empty-default modal payloads:** `readDiscordComponentSpec()` still parses any non-null `components` object and throws if `components.modal.fields` is missing or empty. There is no early return that collapses all-empty default component payloads back to `null`.
  - file: [extensions/discord/src/components.ts:434](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/components.ts#L434)

That means default-filled `components` objects still risk routing into the components code path.

Docs also now say workspace-only mode allows agent workspace paths. That suggests the issue's third sub-bug is at least partially addressed already.
