## Summary

Keep open. Current main has related reload deferral and durable generic/restart-continuation queues, but the reported Telegram same-surface auto-reply path is still in-process and direct to Telegram. sendChatAction failures are treated as typing-indicator failures, not as durable delivery journal entries or hard visible turn failures.

## Best Possible Solution

Keep this issue open. Add a focused Telegram regression that simulates a live Telegram turn during channel hot reload with sendChatAction and/or Bot API network failure, then either journal normal same-surface Telegram replies before Bot API delivery or fail the turn visibly in Telegram and UI with clear retry guidance.
