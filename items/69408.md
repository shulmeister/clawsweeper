## Summary

Current main still does not implement #69408. Loop-detection warnings are emitted at warn level based only on detector thresholds, the tool hook context has no cron session-target, schedule interval, or run-lifetime metadata to apply the requested downgrade, and isolated cron execution currently passes a fresh sessionId but the base cron sessionKey into the embedded runner. Related PR #40217 is adjacent stale-history work, but it does not supersede this cron-isolated warning-to-info request.

## Best Possible Solution

Keep #69408 open. The best maintainer path is a narrow cron-aware loop-warning policy: pass explicit run context into the tool hook or detector, including trigger, sessionTarget, run start time, and schedule interval when available; downgrade only warning-level detections for bounded isolated cron runs to info-level audit logs/events; preserve all critical blockers and non-isolated warning behavior; and add focused tests around `runBeforeToolCallHook` plus the embedded cron path. Maintainers should also decide whether diagnostic loop state for isolated cron should key by per-run session id/session key instead of the base cron session key.
