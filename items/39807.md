## Summary

Keep #39807 open. Current main still does not give inline `models.providers.<id>.apiKey` credentials a durable auth-profile or provider cooldown identity, so billing backoff and fallback skipping remain profile-store based. The related fix PR #39836 proposed the synthetic-profile approach but is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open and implement a stable cooldown identity for inline `models.providers.<id>.apiKey` credentials. The likely fix is to return a synthetic profile id or provider-level cooldown key for inline API-key resolution, persist billing/auth failure state for that identity, and add regression coverage proving later turns skip the disabled inline provider and proceed to configured fallbacks.
