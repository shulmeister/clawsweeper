## Summary

Keep open. Current main does not implement this PR’s write-time operational-alert filter: `resolveMirroredTranscriptText` still returns any non-empty trimmed text, and the append path persists it as an `openclaw`/`delivery-mirror` assistant transcript message. Main does have a broader replay-time filter that prevents delivery-mirror and gateway-injected assistant records from being replayed to providers, so part of the contamination concern is mitigated, but the PR’s stated goal of keeping stale operational alerts out of the transcript mirror remains unimplemented. The PR discussion also shows the initial false-positive concern was addressed by narrowing the Spanish `Error en` pattern and adding a regression case, so the branch still looks like a plausible focused fix rather than obsolete drive-by churn.

## Best Possible Solution

Keep the PR open for maintainer review. If the desired behavior is to prevent stale operational status from appearing in session transcript records, land a narrow write-time filter with regression tests, preserving the branch’s fix for the earlier false-positive review (`Error en ...` without the warning prefix should remain visible). If maintainers decide replay-time filtering is the actual boundary, close only after documenting that decision and adding coverage proving alert records cannot affect user-visible session/history flows.
