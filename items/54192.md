## Summary

Keep this issue open. Current main still reuses stored CLI sessions on fallback retry and sends the retry prompt again; only the embedded runner has explicit orphaned trailing user repair. The prior Codex review comment remains consistent with current main at 9626ef274ae2346b3d519f99c31856be98aab570.

## Best Possible Solution

Keep this issue open. The best fix is a CLI-owned fallback retry repair in `src/agents/command/attempt-execution.ts` and/or `src/agents/cli-runner.ts`: when `isFallbackRetry` would reuse a stored CLI session after rate-limit or timeout, either repair/branch the backend transcript where supported or deliberately clear/reseed a fresh CLI session instead of resuming and submitting the same prompt again. Add a focused regression test in `src/agents/command/attempt-execution.cli.test.ts` or `src/agents/cli-runner.reliability.test.ts` for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not append a duplicate user turn.

## Risks / Open Questions

- A naive prompt-skip could lose the user's task if the backend cannot continue from an already-appended orphaned user turn; the regression should model failure after prompt submission, not just before process spawn.
- Different CLI backends may need different repair strategies because not all expose branch/delete/continue semantics for backend transcripts.
