## Summary

Current main has useful partial mitigations for large-session pressure, but it does not yet solve the core crash-restart doomloop described here. The agent run path still repairs and opens the full transcript file before pre-prompt overflow recovery can run, and the compacted successor transcript rotation that would bound future loads is opt-in with a documented default of false. The issue discussion also identifies this as a critical recovery problem rather than just a normal context-overflow failure.

## Best Possible Solution

Keep this issue open and implement a core recovery path before any full transcript materialization: stat and classify oversized or repeatedly failing session files before repair/SessionManager.open, rotate or quarantine them into a bounded successor/degraded session, cap or dedupe bootstrap-context markers, and add regression coverage for a large transcript restart loop with media refs and context-engine compaction. The existing opt-in truncateAfterCompaction and media offload work can be reused, but the default gateway startup/run path needs the guard.
