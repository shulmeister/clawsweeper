## Summary

Keep open. Current main partially fixes the behavioral dead end because `models: []` with a configured Ollama `baseUrl` now goes through discovery against that configured host, but the exact schema/docs contradiction reported here remains: `models.providers.*.models` is still required by the config schema and type surface, while the Ollama custom-base-url docs still show a provider block without `models`. The related PRs that tried to make `models` optional were closed unmerged, so there is no canonical landed fix to close this against.

## Best Possible Solution

Keep the issue open and narrow the maintainer path: either make `models.providers.*.models` optional in the schema/types with regression coverage for an Ollama baseUrl-only provider block discovering via `/api/tags`, or keep `models` required and update the Ollama docs to show the supported `models: []` remote autodiscovery form or a full manual `models` array. Also clarify whether `openclaw models list --local` should include LAN/self-hosted Ollama hosts or whether users should use `--provider ollama` for remote Ollama.
