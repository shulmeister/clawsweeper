## Summary

Keep open. Current main still uses a coarse file-wide heuristic for `potential-exfiltration`, so the reported false positive remains plausible, and the requested source/sink confidence metadata is not implemented.

## Evidence

- **Heuristic is still broad and file-wide:** `potential-exfiltration` fires when the full source matches `/readFileSync|readFile/` and also matches a network-send context regex. This is not path-sensitive and can match identifiers like `readFileSegment`, not just actual `fs.readFile*` calls.
  - file: [src/security/skill-scanner.ts:180](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/skill-scanner.ts#L180)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Finding location comes from first pattern hit, not a proven sink:** For source rules, the scanner reports the first line matching the source pattern as the finding line/evidence. That means a report at `engine.ts:829` can point at a local helper name while the network context exists elsewhere in the file.
  - file: [src/security/skill-scanner.ts:280](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/skill-scanner.ts#L280)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Test coverage only proves positive detection, not the reported false-positive case:** The current tests cover a simple `fs.readFileSync` plus `fetch(...post...)` case, but there is no regression test for local read-only parsing helpers sharing a file with unrelated network code.
  - file: [src/security/skill-scanner.test.ts:180](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/skill-scanner.test.ts#L180)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Audit output does not expose source/sink confidence metadata:** `SkillScanFinding` carries only `ruleId`, `severity`, `file`, `line`, `message`, and `evidence`, and top-level audit findings carry only `checkId`, `severity`, `title`, `detail`, and optional `remediation`. There is no implemented source/sink-confidence structure matching the request.
  - file: [src/security/skill-scanner.ts:12](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/skill-scanner.ts#L12)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Deep audit surfaces generic code-safety findings only:** Plugin code-safety findings are formatted as `[ruleId] message (path:line)` under `plugins.code_safety`, with no richer suppression or confidence metadata.
  - file: [src/security/audit-extra.async.ts:153](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/audit-extra.async.ts#L153)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
