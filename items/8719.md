## Summary

Keep #8719 open. Current main has related hardening, including workspace-only filesystem guards, default workspace-contained apply_patch, config self-modification guards, sandbox/network controls in some surfaces, and useful external prior art from ClawMoat, but it still does not implement the requested native data-centric security profile. The core filesystem policy remains a single workspaceOnly boolean, unrestricted host filesystem behavior is still preserved when that flag is false or unset, and docs still track data classification, URL allowlisting for web_fetch, skill sandboxing, audit/config integrity, signing, and sensitive-action validation as remaining work.

## Best Possible Solution

Keep #8719 open as a product/security umbrella until maintainers either define and implement a native Security Profile or split it into canonical narrower issues. Track filesystem PathGuard separately (#39672/#52621/#60981), acknowledge ClawMoat as complementary external prior art, and create/keep follow-ups for data classification, web_fetch/network egress policy, sensitive-action audit logging, TTL elevation, session isolation, config integrity, skill/plugin verification, and cross-tool policy enforcement.

## Risks / Open Questions

- Closing as duplicate_or_superseded would lose the umbrella for non-filesystem work because the related PathGuard items cover only a subset of filesystem policy and explicitly leave exec/network/data governance out of scope.
- Closing as clawhub would be premature: ClawMoat is useful external prior art, but the issue asks for native cross-tool defaults, approval, audit, sandbox, session isolation, and policy semantics that remain product/security design work for OpenClaw.
