## Summary

Keep #8719 open. Current main has useful related hardening, but it still does not implement the requested native data-centric Security Profile v1.1 with data levels, default protected folders, cross-tool data policy, network egress by sensitivity, sensitive-action audit log, TTL elevation, or a secure-by-default profile. The ClawMoat comment is useful external prior art, not a reason to close the OpenClaw core product/security umbrella.

## Best Possible Solution

Keep #8719 open as the product/security umbrella until maintainers either define and implement a native Security Profile or split it into canonical narrower issues. Treat ClawMoat as complementary external prior art, track PathGuard/filesystem work separately, and create/keep follow-ups for data classification, protected OS presets, network egress policy, sensitive-action audit logging, TTL elevation, session isolation, config integrity, plugin verification, and cross-tool policy enforcement.

## Risks / Open Questions

- Closing as implemented would conflate narrower workspace-only/apply_patch/config/SSRF/plugin-install hardening with the broader native data-centric profile requested here.
- Closing as duplicate_or_superseded would lose the umbrella for non-filesystem work; related PathGuard items cover only a subset and #60981 explicitly excludes exec sandboxing and remote-source governance.
- Closing as clawhub would be premature: ClawMoat is useful prior art, but native secure defaults, core config policy, approval/elevation semantics, audit events, and cross-tool enforcement remain OpenClaw product/security design work.
