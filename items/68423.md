## Summary

Keep open. Current main has redacted-placeholder guards for the newer last-known-good promotion/recovery path, but `maybeRecoverSuspiciousConfigRead` and its sync variant still parse `.bak` and copy it over the live config without checking whether the backup contains redacted secret placeholders. `maintainConfigBackups` also still copies the current config directly into `.bak`, so the pollution-spread part of the report remains actionable. The related PR #68462 directly targets this issue, but the provided context shows it is still open and unmerged.

## Best Possible Solution

Keep #68423 open. The best maintainer path is to land or adapt PR #68462, or an equivalent patch, so both async and sync `maybeRecoverSuspiciousConfigRead` validate `backupParsed` with the existing polluted-placeholder detector before copying `.bak` over the live config. A follow-up should separately decide whether `maintainConfigBackups` should skip creating `.bak` from polluted current config and whether CLI validation should scan config backups.
