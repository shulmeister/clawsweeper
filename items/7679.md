## Summary

Keep #7679 open. Current main still defaults Telegram DMs to pairing, still sends pairing challenges to unknown DM senders, and does not implement token-verified owner Telegram ID capture during onboarding. The issue discussion also has a recent maintainer/Codex comment explicitly keeping it open and a prior warning that capture must verify the intended sender rather than accepting the first inbound message.

## Best Possible Solution

Keep #7679 open. The product fix should add a Telegram onboarding path that generates a short verification token, asks the intended owner to send that exact token to the bot, captures that sender's numeric Telegram user ID, writes dmPolicy="allowlist" plus allowFrom for the correct default or named account, and updates schema/runtime/docs/tests with a clear backcompat path for existing pairing-mode configurations.

## Risks / Open Questions

- A schema-only default change would be insufficient because allowlist mode requires a populated allowFrom list.
- Blindly capturing the first inbound Telegram message would preserve the race condition identified in the issue discussion; the setup flow needs a token or equivalent sender verification.
- Existing pairing-mode installs need an explicit compatibility/migration story so they are not unexpectedly locked out.
