## Summary

Keep #7679 open. Current main still defaults Telegram DMs to pairing in schema, runtime, setup, docs, and tests; unknown senders still receive pairing challenges; and onboarding still asks for manual numeric allowFrom entry rather than token-verified owner ID capture. The issue discussion's warning about verifying the sender during capture remains unresolved.

## Best Possible Solution

Keep #7679 open. The maintainer path should add Telegram onboarding that generates a short verification token, asks the intended owner to send that exact token to the bot, captures the matching numeric Telegram sender ID, writes dmPolicy="allowlist" plus allowFrom for that owner on fresh setup, and updates schema/runtime/docs/tests while preserving compatibility for existing explicit pairing-mode configs.

## Risks / Open Questions

- Changing only the default to allowlist would fail validation or lock out fresh users unless onboarding first captures and writes a verified non-empty allowFrom owner ID.
- Blind first-message capture would preserve the race condition raised in the issue discussion.
- Existing pairing-mode installations need compatibility handling so users are not unexpectedly locked out by a safer fresh-setup default.
