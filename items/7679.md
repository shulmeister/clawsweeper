## Summary

Keep #7679 open. Current main still defaults Telegram DM access to pairing in schema and runtime, still issues pairing challenges to unknown DM senders, and does not implement token-verified owner-ID capture during onboarding. The recent maintainer/Codex issue comment also explicitly kept this open, and the earlier discussion's race-condition concern remains unresolved.

## Best Possible Solution

Keep #7679 open. The product fix should add a Telegram onboarding flow that generates a short verification token, asks the intended owner to send that exact token to the bot, captures that sender's numeric Telegram user ID, writes explicit dmPolicy="allowlist" plus allowFrom for the correct account, and updates schema/runtime/docs/tests with a clear migration/backcompat path for existing pairing-mode installs.

## Risks / Open Questions

- Naively changing only the schema/runtime default to allowlist would fail validation or block all Telegram DMs unless onboarding also captures and writes a valid owner Telegram user ID.
- Blindly capturing the first inbound Telegram message would leave the race condition described in the issue discussion; the owner capture needs a displayed random token or equivalent sender verification.
- Docs currently contain mixed messaging: the QuickStart page says Telegram DMs default to allowlist, while the CLI reference and current implementation still show pairing as the default.
