## Summary

Keep open. Current main still defaults Telegram DMs to `dmPolicy: "pairing"` in schema, runtime, tests, and docs. QuickStart can force a manual allowlist prompt, but it does not auto-capture the owner Telegram ID during onboarding or implement the token-verified sender capture raised in the issue discussion.

## Best Possible Solution

Keep #7679 open. The best product path is to implement a Telegram onboarding flow that generates a short verification token, asks the intended owner to send that exact token to the bot, captures that sender's numeric Telegram user ID, writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, and updates schema/runtime/docs/tests so new Telegram setups no longer present pairing as the safer default. Existing configured installs need a clear migration/backcompat story.

## Risks / Open Questions

- Changing the schema default naively to `allowlist` without a captured owner ID would currently fail validation or block all DMs, so the fix needs a deliberate onboarding and backcompat path.
- Blindly capturing the first inbound Telegram message would introduce the race described in the issue comments; owner capture should require a displayed random token or equivalent sender verification.
