## Summary

Current main records `livenessState` in `meta` and emits it on agent lifecycle events, but the outbound reply payload and delivery paths still do not surface that state to user-facing channel payloads. I did not find any current gateway/UI/channel consumer that reads `livenessState`, so the issue remains materially open.

## Evidence

- **Reply payload shape still has no liveness field:** `EmbeddedPiRunResult.payloads` still only carries text/media/reply flags. `livenessState` remains on `meta`, so payload-only consumers cannot receive it from the result payload shape itself.
  - file: [src/agents/pi-embedded-runner/types.ts:136](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/agents/pi-embedded-runner/types.ts#L136)
  - command: `nl -ba src/agents/pi-embedded-runner/types.ts | sed -n '136,146p'`
  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Channel reply normalization ignores liveness metadata:** `normalizeAgentCommandReplyPayloads` transforms only `payloads` plus reply-prefix/model context. It does not read `result.meta.livenessState` or append any user-visible footer/state marker before delivery.
  - file: [src/agents/command/delivery.ts:105](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/agents/command/delivery.ts#L105)
  - command: `nl -ba src/agents/command/delivery.ts | sed -n '105,173p'`
  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Outbound delivery only preserves meta for JSON envelope:** The delivery path includes `meta` only in the JSON envelope/log return path. Non-JSON delivery still logs `No reply from agent.` on empty payloads and otherwise sends normalized payloads, with no liveness-aware rendering step.
  - file: [src/agents/command/delivery.ts:322](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/agents/command/delivery.ts#L322)
  - command: `nl -ba src/agents/command/delivery.ts | sed -n '322,343p'`
  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Current main surfaces liveness on lifecycle events, not outbound payloads:** `handleAgentEnd` emits `livenessState` and `replayInvalid` on `lifecycle` `end`/`error` events. That is useful progress, but it is a separate event stream from the outbound reply payload path described in the issue.
  - file: [src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:113](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/agents/pi-embedded-subscribe.handlers.lifecycle.ts#L113)

- **No current gateway/UI/channel consumer references livenessState:** A repo search over `src/gateway`, `ui/src`, `src/channels`, and `extensions` found no `livenessState` references, which is consistent with the issue's claim that downstream user-facing surfaces still do not render it.
  - command: `rg -n "livenessState" src/gateway ui/src src/channels extensions || true`

## Risks / Open Questions

- Outbound channel users can still see a terminal reply without any explicit blocked/abandoned signal because the delivery path remains payload-only.
- Web/gateway surfaces may receive lifecycle events internally, but there is no current evidence of a user-facing renderer consuming `livenessState`, so the acceptance criteria are not met.
