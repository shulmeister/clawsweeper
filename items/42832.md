## Summary

Keep open. Current `main` still has the core problems this PR is trying to fix: Nostr gateway startup returns immediately instead of staying alive until abort, Nostr outbound still has no `sendMedia` path so media delivery falls back to dropped/failed attachments, and the current DM subscription call still passes an array where `nostr-tools` 2.23.3 documents a single `Filter`. The PR is therefore not implemented on `main`, though it still needs maintainer review for its raw-URL media fallback behavior.

## Evidence

- **Gateway task still resolves immediately on main:** `startNostrGatewayAccount()` returns a `{ stop() }` object immediately after starting the bus, with no wait on `ctx.abortSignal`. In the gateway runner, account tasks are treated as the long-lived channel process; once the promise resolves, runtime is marked stopped and restart handling kicks in.
  - file: [extensions/nostr/src/gateway.ts:238](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/nostr/src/gateway.ts#L238)

  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Server expects `startAccount` to be long-lived:** `server-channels` records the `startAccount()` promise as the running task and marks the channel stopped in `.finally()` when that promise settles. A plugin that resolves immediately does not match this contract.
  - file: [src/gateway/server-channels.ts:418](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/gateway/server-channels.ts#L418)

## Risks / Open Questions

- PR head `6836683b2158cff57f2843d7be2ec23ffa0a9fd7` appears to fix real mainline bugs, so closing it would discard useful work that is not yet on `main`.
