## Summary

Current `main` only shows a human-friendly device name when the client already supplies `displayName`; the Control UI and CLI pairing paths do not supply one, and there is no user-facing device rename RPC/CLI command. The request is therefore still unimplemented for the reported web UI / `openclaw devices list` case.

## Evidence

- **Control UI pairing does not send a display name:** The Control UI connect payload includes `id`, `version`, `platform`, `mode`, and `instanceId`, but no `displayName`, so browser-paired devices still arrive without a friendly label.
  - file: [ui/src/ui/gateway.ts:377](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/ui/src/ui/gateway.ts#L377)

- **CLI pairing does not send a display name:** `openclaw devices` calls `callGateway` with `clientName` and `mode` only. The generic gateway client only transmits a client display name from `opts.clientDisplayName`, which is absent here.
  - file: [src/cli/devices-cli.ts:103](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/cli/devices-cli.ts#L103)
