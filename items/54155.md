## Summary

Keep this issue open. Current main has partial mitigations for long-running Gateway memory growth, especially seqByRun cleanup and session-store maintenance, but it still has the session-store structuredClone/cache path, whole-transcript materialization in Gateway history paths, and opt-in transcript rotation rather than proof that Gateway RSS now plateaus under the reported multi-day workload.

## Best Possible Solution

Keep this issue open as the canonical broad Gateway RSS/session-accumulation tracker. The best maintainer path is to land or supersede the focused #45438 structuredClone/cache fix, reduce retained dual serialized/object session-store state, stream or cap transcript reads used by Gateway history/sequencing paths, decide whether transcript rotation should become a safer default or automatic policy, and add targeted regression or soak evidence showing long-running Gateway RSS plateaus under cron, channel traffic, and subagent sessions.
