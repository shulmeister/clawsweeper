## Summary

Keep open. Current main still has the exact immediate `loadChatHistory()` call after `/clear` sends `sessions.reset`, and `loadChatHistory()` still writes returned `chat.history` messages back into `chatMessages`. The PR is therefore not implemented on main and remains a plausible focused fix; it likely needs maintainer re-review/rebase and a regression test rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer re-review/rebase. The best implementation is to remove the immediate post-reset history reload from the active `/clear` path, add a regression test that simulates a stale `chat.history` response after `sessions.reset`, and either fix or remove the parallel `app-render.ts` clear handler if it is still reachable.

## Evidence

- **Current main still reloads history after /clear reset:** `handleSendChat()` routes the local `clear` slash command to `clearChatHistory()`, which calls `sessions.reset`, zeros local chat state, and then immediately awaits `loadChatHistory(host as unknown as ChatState)`. This is the exact call the PR proposes to remove.
