## Summary

Keep #37943 open. Current main still matches the reported split behavior: the installed node service runs `openclaw node run`, the global invalid-config guard still does not allow `node`, and the node runner still calls strict `loadConfig()` before connecting. Gateway startup has its own best-effort and last-known-good recovery path, so the reporter's “gateway works, node service remains stuck” scenario is not implemented on main.

## Best Possible Solution

Keep the issue open. A maintainer should decide whether `openclaw node run` should share the gateway's best-effort/last-known-good recovery for node startup, or remain strict but emit node-specific diagnostics and repair guidance. Likely implementation surfaces are `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, `src/cli/node-cli/register.ts`, and `src/daemon/program-args.ts`, with regression coverage for node startup under invalid effective config.
