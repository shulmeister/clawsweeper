## Summary

Keep #37943 open. Current main still has the same unresolved node-host startup gap described in the issue and in the prior ClawSweeper comment: the installed macOS node service launches `openclaw node run`, the CLI config guard does not allow `node` to continue through an invalid effective main config, and `runNodeHost()` still calls strict `loadConfig()` before gateway TLS/auth and plugin registry setup. Gateway startup has a separate best-effort/last-known-good recovery path, so the reporter's observation that the gateway can recover while the node service fails remains plausible on current main.

## Best Possible Solution

Keep the issue open. A maintainer should decide the intended product behavior for `openclaw node run` when the main config is JSON-valid but schema-invalid: either share gateway startup's best-effort/last-known-good recovery where safe, or remain strict but emit node-specific schema diagnostics. The likely implementation surface is `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, and the node service install/run path, with a focused regression test for the node command and docs/logging updates for the chosen behavior.
