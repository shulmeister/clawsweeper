## Summary

Keep #37943 open. Current main still has no node-service recovery path comparable to gateway startup: the node service is installed as `openclaw node run`, that path is not allowlisted to continue through invalid main config, and the node runner still uses strict `loadConfig()` for gateway TLS, plugin registry, and auth resolution. The gateway can recover from invalid effective config via best-effort and last-known-good logic, so the reporter's observation that gateway can run while the node service fails remains plausible rather than obsolete.

## Best Possible Solution

Keep the issue open. A maintainer should decide whether node host startup should intentionally share gateway's best-effort/last-known-good recovery path, or remain strict but emit precise schema diagnostics for the node service. The likely implementation surface is `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, and the node service install/run path, with a regression test for JSON-valid schema-invalid config and docs/logging updates for the intended behavior.
