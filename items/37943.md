## Summary

Keep open. Current main still has the reported node service path using `openclaw node run`, and that path still calls strict `loadConfig()` on the main `openclaw.json`. Gateway startup uses a separate best-effort/recovery path, so the reporter's observation that the gateway can run does not prove the node service parse problem is fixed. The related PR #38017 was closed unmerged and, per the provided context, did not change code or tests.

## Best Possible Solution

Keep #37943 open. A maintainer should either make the node host startup path tolerate/recover from the same config states the gateway can recover from, or explicitly fail with precise schema-validation diagnostics if node hosts must require a fully valid OpenClaw config. The likely code surface is `src/node-host/runner.ts` and the node CLI/service path, with a focused regression test for `openclaw node run` against JSON-valid but schema-invalid config and documentation/logging updates if strict validation is the intended product behavior.
