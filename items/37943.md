## Summary

Keep open. Current main still routes the macOS node service through `openclaw node run`, and the node host still reads the main config through the strict `loadConfig()` path while `gateway run` uses a separate best-effort/recovery path. The provided related PR #38017 was closed unmerged and only added issue metadata, so there is no high-confidence close reason.

## Best Possible Solution

Keep the issue open. Maintainers should request a redacted config plus `openclaw config validate`/node logs, then add a regression around the macOS node service path. If the config is schema-valid, make the node host/service share the gateway's best-effort/recovery behavior or otherwise avoid failing on recoverable config snapshots; if it is schema-invalid, improve node-service diagnostics so the user sees the specific schema issue rather than a generic parse/invalid-config symptom.
