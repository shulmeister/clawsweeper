## Summary

Current main still does not implement #8813. The Skills prompt builder still hardcodes read-based SKILL.md loading, even though the agent system prompt already normalizes available tool names and preserves caller casing. Tests and docs still assert/document the read path, and the provided issue discussion already reached the same keep-open conclusion. This remains a core prompt-composition enhancement, not a ClawHub/plugin-only cleanup item.

## Best Possible Solution

Keep #8813 open. The likely fix belongs in core prompt composition: pass the normalized available-tool set and casing resolver into buildSkillsSection, prefer invoking a registered Skill tool when present, keep the existing read-based fallback when absent, and update system-prompt tests plus public Skills/system-prompt docs to cover both paths.
