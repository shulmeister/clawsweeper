## Summary

Current main still does not implement #8813. The system prompt Skills section still hardcodes `read`-based SKILL.md loading, even though the prompt builder already normalizes the available tool set and preserves caller casing. Tests and docs still assert/document the read path, and the recent issue comment reached the same keep-open conclusion. No related item in the provided context supersedes this specific Skill-tool preference request.

## Best Possible Solution

Keep #8813 open. The likely fix belongs in core prompt composition: pass the normalized available-tool set plus casing resolver into `buildSkillsSection`, prefer invoking a registered Skill tool when present, keep the existing read-based fallback when absent, and update system-prompt tests plus the public system-prompt/skills docs for both paths.
