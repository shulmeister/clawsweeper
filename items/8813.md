## Summary

Current main at d54d2d6b9b8aa28fb222d895c0f01f43933f4409 still does not implement #8813. The Skills prompt builder always instructs the model to load SKILL.md with the read tool, while available tool names are normalized nearby but not passed into the skills section. Tests and docs still lock in the read-based behavior, and the existing issue discussion already reached the same keep-open conclusion.

## Best Possible Solution

Keep #8813 open. The likely fix belongs in core prompt composition: pass the normalized available-tool set and casing resolver into buildSkillsSection, prefer invoking a registered Skill tool when present, keep the existing read-based fallback when absent, and update system-prompt tests plus public Skills/system-prompt docs for both paths.
