## Summary

Keep #38721 open. Current main has related shutdown hardening for WebSocket close, MCP runtime disposal, stdio MCP process-tree cleanup, service control-group cleanup, and lane reset after restart, but I did not find source or test evidence that the specific child-process/pipe shutdown blocker is fully fixed for generic exec/tool or other supervisor-owned subprocesses. The maintainer discussion also explicitly says #61565 fixed a different websocket shutdown hang and should not supersede this child-process-handle report.

## Best Possible Solution

Keep the issue open until OpenClaw either implements a targeted gateway-shutdown path that identifies and cancels/bounds active supervisor-owned exec/tool children, or a maintainer/live repro proves the remaining child-process handle was only an MCP/websocket/service-unit case already fixed on main. A good fix should add a regression around gateway shutdown with an active child-process handle and document the intended shutdown behavior for background exec/process sessions.
