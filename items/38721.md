## Summary

Keep #38721 open. Current main still has adjacent bounded shutdown fixes, but I did not find a gateway shutdown path that cancels, detaches, or drains arbitrary active process-supervisor child handles before the shutdown watchdog. The issue discussion explicitly separates #61565’s websocket-close fix from this child-process/pipe-handle report, and current code still matches that distinction.

## Best Possible Solution

Keep #38721 as the canonical tracker for gateway shutdown blocked by active child-process/pipe handles. The best fix is a gateway shutdown policy for supervisor-managed exec/process sessions: stop accepting new process work, cancel or detach active runs with a bounded grace window, expose the necessary process-supervisor shutdown API, and add regression coverage proving active child handles cannot trip the shutdown watchdog. Treat #61565 and the current websocket/HTTP/MCP bounds as adjacent mitigations, not closure evidence for this issue.
