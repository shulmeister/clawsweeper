## Summary

Keep #38721 open. Current main has adjacent shutdown hardening for websocket, HTTP, MCP runtimes, systemd control-group cleanup, and process-tree kill behavior, but I did not find a gateway shutdown path that enumerates, cancels, or bounded-drains active ProcessSupervisor child/PTY runs. The issue discussion also explicitly says #61565 fixed a websocket-close hang and did not supersede this child-process/pipe-handle report.

## Best Possible Solution

Keep #38721 as the canonical tracker for gateway shutdown blocked by active child-process or pipe handles. The best fix is to add an explicit ProcessSupervisor shutdown contract: stop accepting new supervisor spawns during gateway shutdown, expose active-run enumeration with command/pid diagnostics, add cancel-all or bounded-drain APIs, invoke that path from gateway shutdown before the watchdog expires, and add regression coverage proving active supervisor child/PTY handles cannot trip the shutdown watchdog. Treat #61565 and current websocket/MCP/HTTP/systemd hardening as adjacent mitigations, not closure evidence.
