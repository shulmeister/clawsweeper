## Summary

Current `main` already prevents restart catch-up from replaying a cron slot when that slot already ran, and it includes a dedicated regression test for the duplicate-after-restart case described in #42640.

## Evidence

- **Restart catch-up now checks prior cron slot against `lastRunAtMs`:** `isRunnableJob` only replays a missed cron slot on restart when `computeJobPreviousRunAtMs(job, nowMs) > job.state.lastRunAtMs`; otherwise it skips replay, which is the missing guard described in the issue.
  - file: [src/cron/service/timer.ts:903](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/cron/service/timer.ts#L903)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '850,935p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Regression test covers the no-duplicate restart scenario:** The test seeds a cron job whose `nextRunAtMs` is already advanced to the future slot, with `lastRunAtMs` equal to the latest scheduled slot before restart, then asserts restart catch-up does not enqueue or heartbeat the job again.
  - file: [src/cron/service.restart-catchup.test.ts:230](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/cron/service.restart-catchup.test.ts#L230)

## Close Comment

Closing as implemented on `main` based on this Codex review. In current `main` (`564f820efa2bb81ae49b8f55a93e41492ad31917`), restart catch-up no longer replays a cron slot just because the scheduler restarted: `src/cron/service/timer.ts:903` computes the previous scheduled cron slot and only replays it when that slot is newer than `state.lastRunAtMs`. There is also an explicit regression test at `src/cron/service.restart-catchup.test.ts:230` that models the reported case (`nextRunAtMs` already advanced, `lastRunAtMs` already recorded for the latest slot) and asserts no duplicate execution happens after restart.
