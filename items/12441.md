## Summary

Keep open. Current main has partial Control UI HTTP bearer support for protected read/bootstrap requests, but it still does not solve the reported reverse-proxy shared-token flow: the browser WebSocket connect path uses only stored/manual token or password, the server-side WebSocket auth path reads shared-secret auth from connect params rather than the upgrade Authorization header, and the bootstrap config does not issue or hydrate a UI session token.

## Best Possible Solution

Keep tracking this in core. The best fix is a deliberate Gateway/Control UI auth design for proxy-injected shared secrets: either accept `Authorization`/`X-OpenClaw-Token` on the WebSocket upgrade and merge it into the existing shared-secret auth path, or validate the header on the initial same-origin HTTP request and issue a short-lived UI session/bootstrap credential for the browser WebSocket. Add focused server/UI tests and update the public dashboard/auth docs. Keep `trusted-proxy` identity mode as a separate documented option, not a substitute for this shared-token flow.
