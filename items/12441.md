## Summary

Keep #12441 open. Current main accepts bearer credentials for some authenticated Control UI HTTP read endpoints after the browser already has a credential, and it has separate identity-header modes for Tailscale/trusted-proxy auth. It still does not map a reverse-proxy-injected Authorization or X-OpenClaw-Token header into shared-secret Control UI WebSocket auth, nor does bootstrap JSON mint a browser session credential for the WebSocket. The related ClawSweeper items cover token persistence/drift, not this header bootstrap path.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best path is either to validate Authorization: Bearer and/or X-OpenClaw-Token on the Control UI WebSocket upgrade and feed it into the existing shared-secret auth path after the usual origin/proxy checks, or to validate the header on a same-origin HTTP bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. Add focused gateway WS/auth tests, Control UI bootstrap tests, and public dashboard/control UI docs.

## Risks / Open Questions

- A fix needs careful proxy-trust, browser origin, rate-limit, credential-leakage, and session lifetime design before accepting injected headers for browser WebSocket auth.
- Existing HTTP bearer support is close enough to be confusing, but closing this would leave shared-secret reverse-proxy setups dependent on URL fragment/sessionStorage/manual entry or a separate identity-bearing auth mode.
