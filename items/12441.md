## Summary

Keep open. Current `main` only supports Control UI shared-secret auth for the WebSocket via `connect.params.auth.token` / `connect.params.auth.password` or identity-header modes; it does not bootstrap a browser session from an injected `Authorization` header alone. There is partial HTTP-side support for bearer auth on the Control UI bootstrap/config route, but that does not solve the reverse-proxy case described in #12441 because the WebSocket auth path still relies on explicit connect payload credentials.

## Risks / Open Questions

- There may be an acceptable workaround via `gateway.auth.mode: "trusted-proxy"`, but that is a different auth mode and does not implement the requested bearer-token bootstrap for token/password deployments.
- The issue title could sound partially satisfied because HTTP Control UI routes already honor bearer auth, but closing on that basis would miss the actual reverse-proxy browser/WebSocket gap described in the body.
