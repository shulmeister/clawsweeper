## Summary

Keep open. Current main has partial bearer support for Control UI HTTP/bootstrap/read requests, but it still does not solve the reverse-proxy shared-token flow: the browser WebSocket connects with stored/manual credentials, server-side shared-secret WebSocket auth is derived from `connect.params.auth`, and the bootstrap JSON does not issue or hydrate a UI session credential.

## Best Possible Solution

Keep tracking this in core. The best fix is a deliberate Gateway/Control UI auth design for proxy-injected shared secrets: either extract `Authorization: Bearer ...` / `X-OpenClaw-Token` from the WebSocket upgrade and map it into the existing shared-secret auth path, or validate the header on an initial same-origin HTTP request and issue a short-lived UI session/bootstrap credential for the browser WebSocket. Add focused server/UI tests and update https://docs.openclaw.ai/web/dashboard.
