## Summary

Keep open. Current main has partial Control UI HTTP bearer support and documented trusted-proxy/Tailscale identity modes, but it still does not accept a gateway shared token from the WebSocket upgrade Authorization header or map a proxy-injected bearer token into the browser WebSocket connect payload. The specific requested capability remains missing.

## Evidence

- **Browser WS connect still uses stored/manual credentials:** Control UI constructs GatewayBrowserClient with only `host.settings.token` and in-memory `host.password`; there is no header-derived token or session populated from the page/bootstrap request before connecting.
  - file: [ui/src/ui/app-gateway.ts:286](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/ui/src/ui/app-gateway.ts#L286)

- **WS auth state ignores Authorization bearer for token mode:** The WebSocket handshake auth state resolves shared-secret auth from `connectParams.auth` (`params.connectAuth`) and passes that to `authorizeWsControlUiGatewayConnect`; the upgrade request is available, but the bearer header is not converted into shared token auth.
  - file: [src/gateway/server/ws-connection/auth-context.ts:80](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/gateway/server/ws-connection/auth-context.ts#L80)

  - file: [src/gateway/auth.ts:481](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/gateway/auth.ts#L481)
