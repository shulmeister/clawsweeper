## Summary

Issue #12441 should stay open. Current main supports Control UI HTTP bootstrap reads with a bearer header and supports explicit WebSocket auth through `connect.params.auth`, Tailscale identity, or trusted-proxy identity, but it still does not convert a reverse-proxy-injected `Authorization: Bearer ...` or `X-OpenClaw-Token` header into the browser WebSocket shared-secret auth path or a short-lived UI bootstrap credential.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The likely implementation should either validate `Authorization: Bearer` and/or a documented OpenClaw token header on the WebSocket upgrade and feed it into the existing shared-secret auth path after normal origin/proxy checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for WebSocket connection. Add focused gateway WS/auth tests, Control UI bootstrap tests, and public Control UI/reverse-proxy docs.

## Risks / Open Questions

- Closing this would leave shared-secret reverse-proxy deployments dependent on URL fragment bootstrap, manual token entry, cached sessionStorage, Tailscale Serve identity, or full trusted-proxy identity mode rather than the requested injected bearer/header flow.
- A real fix needs security design around trusted proxy boundaries, browser origin checks, credential lifetime, rate limiting, and whether `X-OpenClaw-Token` should become a documented cross-surface gateway auth header.
