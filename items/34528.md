## Summary

Current main still appears to carry Feishu synthetic reaction message IDs into API-facing paths. The issue has useful linked PRs, but they are proposed/unmerged fixes rather than shipped behavior on main.

## Best Possible Solution

Keep #34528 open until a Feishu plugin fix lands. The likely fix is to add a narrow message-id normalizer for OpenClaw's synthetic `:reaction:<emoji>:<uuid>` suffix and apply it at Feishu API boundaries or at the single reply-target handoff, while preserving the suffixed ID for internal dedup/session identity. Add regression tests covering reply, typing/streaming, media, and reaction API paths as applicable.
