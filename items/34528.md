## Summary

Keep #34528 open. Current main still creates Feishu synthetic reaction message IDs with the `:reaction:<emoji>:<uuid>` suffix and still passes the resulting `replyTargetMessageId` into Feishu API boundary paths without a shipped normalizer. The related fix PR #39558 is open and unmerged, so the issue is not implemented on main.

## Best Possible Solution

Keep the issue open until a Feishu plugin fix lands on main. The best path is a narrow message-ID normalizer for OpenClaw's `:reaction:<emoji>:<uuid>` suffix, applied at Feishu API boundaries for replies, typing/reaction APIs, media/resource calls, streaming cards, and message edit/get paths as applicable, with regression tests proving clean IDs are sent externally while synthetic IDs remain available internally. PR #39558 is the current candidate to review or supersede.
