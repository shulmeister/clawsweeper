## Summary

Current `main` already implements WebChat/Control UI image attachments end-to-end: the UI collects pasted/selected images, sends them as `chat.send.attachments` over the existing WebSocket RPC path, and the gateway parses, persists, and records them in transcript media fields. The report's expectation of a separate POST/fetch/XHR upload does not match the current implementation.

## Close Comment

Closing after Codex review because current `main` already implements this path. In the Control UI, pasted/selected images are collected in `ui/src/ui/views/chat.ts:206` and serialized into `chat.send.attachments` in `ui/src/ui/controllers/chat.ts:205`. Those requests are sent over the existing WebSocket RPC transport in `ui/src/ui/gateway.ts:621`, so there is not expected to be a separate POST/fetch/XHR upload. On the gateway side, `chat.send` accepts attachment-bearing messages, persists inbound images, and writes `MediaPath`/`MediaPaths` into transcript history in `src/gateway/server-methods/chat.ts:2113`, `src/gateway/server-methods/chat.ts:680`, and `src/gateway/server-methods/chat.ts:755`. The current test suite also covers both model dispatch and transcript persistence for these attachments in `src/gateway/server-methods/chat.directive-tags.test.ts:1679` and `src/gateway/server-methods/chat.directive-tags.test.ts:2074`. Based on the current codebase, this is already implemented on `main`.
