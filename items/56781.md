## Summary

Keep open. Current `main` still treats compaction as a single-model override, not a `{ primary, fallbacks }` chain, and the compaction execution path does not use the normal `runWithModelFallback()` flow that agent chat uses. The `lossless-claw` half also depends on an external plugin, so there is no strong basis to close this from the OpenClaw checkout alone.

## Evidence

- **Compaction config surface documents only one override model:** The user-facing config docs for `agents.defaults.compaction` list a single `model` field and describe it as an optional `provider/model-id` override. There is no documented `fallbacks` companion field under compaction.
  - file: [docs/gateway/config-agents.md:536](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/gateway/config-agents.md#L536)
  - command: `nl -ba docs/gateway/config-agents.md | sed -n '536,580p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Compaction schema accepts only `model: string`:** The runtime Zod schema for `agents.defaults.compaction` defines `model: z.string().optional()` and does not define any `fallbacks` field, so the requested config shape is not implemented on `main`.
  - file: [src/config/zod-schema.agent-defaults.ts:171](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/zod-schema.agent-defaults.ts#L171)
  - command: `nl -ba src/config/zod-schema.agent-defaults.ts | sed -n '171,200p'`

- **Compaction target resolver reads one string override:** `resolveEmbeddedCompactionTarget()` reads `config?.agents?.defaults?.compaction?.model?.trim()` and returns a single provider/model target. It has no logic for `fallbacks` arrays or fallback candidate expansion.
  - file: [src/agents/pi-embedded-runner/compaction-runtime-context.ts:34](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/compaction-runtime-context.ts#L34)
