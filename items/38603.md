## Summary

Current main still has the real gap reported in #38603: Telegram's send adapter lacks a send-level exact `NO_REPLY` guard, while Slack has one. Adjacent Telegram delivery/streaming logic handles some silent-reply cases, and PR #68304 is an open fix attempt, but the requested behavior is not implemented on main.

## Best Possible Solution

Keep #38603 open and review or replace PR #68304 with a focused Telegram send-adapter change: import/use the SDK `isSilentReplyText` helper in `extensions/telegram/src/send.ts`, return a synthetic suppressed result before resolving/sending when `text` is exactly a silent reply and there is no media or inline keyboard, and add `extensions/telegram/src/send.test.ts` regression coverage proving the Telegram API is not called. Track the larger pre-sentinel streaming-buffering behavior separately if maintainers want to eliminate flashes from mixed visible text plus trailing `NO_REPLY`.
