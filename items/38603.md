## Summary

Keep open. Current main still lacks the requested send-level Telegram `NO_REPLY` guard, while Slack has the parity guard and the provided related PR #68304 is still the active fix path rather than shipped behavior.

## Best Possible Solution

Keep the issue open and review or replace PR #68304 with a focused change in `extensions/telegram/src/send.ts`: import the SDK silent-reply helper, return a synthetic suppressed result before Telegram API calls when `text.trim()` is exactly `NO_REPLY` and there is no media or inline keyboard, and add regression coverage in `extensions/telegram/src/send.test.ts`. If maintainers want to solve the broader pre-sentinel preview flash, track that as a separate streaming-buffering change rather than closing this guard request prematurely.
