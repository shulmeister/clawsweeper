## Summary

Keep this issue open. Current main still documents the QMD symlink-loop limitation as a workaround-only state, and the runtime path still delegates configured collections to `qmd update` without an OpenClaw-side no-follow, exclusion, or realpath-cycle policy. The issue discussion also has maintainer comments explicitly keeping the runtime traversal problem open after the docs correction.

## Best Possible Solution

Keep this issue open as a focused QMD traversal robustness item. Add the controlled symlink-loop repro as a regression, then make traversal cycle-safe/no-follow upstream or add an OpenClaw-side exclusion/pruning policy before `qmd update` can recurse. The current docs workaround should remain, but it is not the product fix.
