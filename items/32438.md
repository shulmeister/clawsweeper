## Summary

Current main still does not implement #32438. The logger resolves and caches only level, file, and maxFileBytes, writes JSONL only to logging.file, and log-tail/Control UI/docs continue to treat the durable file-log surface as JSONL. The existing `openclaw logs` command can format JSONL for stdout, including a local fallback in one pairing-required case, but it does not create the requested persistent sibling `-formatted` file for direct `tail`/`grep` when the Gateway is down. PR #32442 attempted this exact feature but is closed and unmerged in the provided context, while #29378 is also closed, so this issue remains the active tracker.

## Best Possible Solution

Keep this issue open until a maintainer either implements the core logger behavior or explicitly rejects the product direction. A complete implementation should derive a stable sibling path from `logging.file`, append one human-readable line whenever the JSONL file is written, preserve the existing JSONL contract for `logs.tail`/Control UI/scripts, define size-cap behavior, add focused logger/log-tail tests, and update the public logging docs. If maintainers prefer the CLI formatter instead, document that as the supported workflow before closing.
