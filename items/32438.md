## Summary

Current main still does not implement the requested derived human-readable log file. The gateway logger resolves only level, file, and maxFileBytes, writes JSONL only to logging.file, and docs/UI still present the file-log surface as JSONL. The existing CLI can format JSONL output for `openclaw logs`, but that is not the requested persistent sibling file and it does not make Control UI usable when the Gateway is down. PR #32442 attempted the exact change but is closed and unmerged in the provided GitHub context, while #29378 is also closed, so this issue remains the active tracker.

## Best Possible Solution

Keep this issue open until a maintainer either implements the core logger behavior or explicitly rejects the product direction. A complete implementation would derive a stable sibling path from `logging.file`, append one human-readable line whenever the JSONL file is written, preserve the existing JSONL file for `logs.tail`/Control UI/scripts, add focused logger and size-cap tests, and update the logging docs. If maintainers prefer the existing CLI formatter instead, document that as the supported down-Gateway workflow and close only with that explicit product decision.
