## Summary

Keep open. Current main still has the reported Control UI recovery gap: Stop visibility is tied to volatile browser-local `chatRunId`, reconnect clears that state, disconnected abort without a local run id is a no-op, and the gateway still lacks WS protocol ping evidence. Backend session-scoped abort and `/stop` support are useful partial fixes, but they do not make the Stop affordance survive reload/reconnect or prove the long-task connection drop is fixed.

## Best Possible Solution

Keep this issue open as the canonical Control UI/Gateway long-task recovery regression. A maintainer-quality fix should expose or recover active run state for the selected session after reconnect/reload, keep a session-level Stop affordance available while the gateway has abortable work, preserve visible `/stop` feedback after a fresh reload, add WS protocol ping/pong or equivalent proven keepalive, and add regression coverage for active run disconnect/reconnect/reload.
