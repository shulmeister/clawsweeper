## Summary

Keep open. Current main has partial abort improvements, but the reported loss-of-control path is not fully fixed: the Control UI still derives Stop visibility from local `chatRunId`, reconnect clears that state, and a reload/reconnect can still leave no Stop button for a server-side active run. The connection-drop side is also not clearly obsolete because Gateway still uses JSON `tick` broadcasts rather than a WebSocket protocol ping/pong keepalive.

## Best Possible Solution

Keep #70991 open as the canonical issue for the remaining product bug. The likely fix is to expose or recover active abortable run state from the Gateway after reconnect/reload, make the Control UI Stop affordance depend on that server-side state instead of only local `chatRunId`, preserve `/stop` as a session-scoped fallback for chat and agent runs, and address the transport stability side with #56668 or an equivalent protocol-level keepalive plus focused regression coverage.
