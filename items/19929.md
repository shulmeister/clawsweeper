## Summary

Keep #19929 open. Current main still lacks selective shared-session routing for chosen group/channel/thread chats: non-direct peers still resolve to isolated per-channel/per-peer keys, config/types expose only DM scoping and identity links, and docs still describe groups/channels as isolated. The related PR #20078 was closed unmerged, and the related #7524 tracker is narrower than this issue's cross-channel/thread grouping and trust-boundary request.

## Best Possible Solution

Keep #19929 open as the concrete tracker for selective shared sessions across chosen group/channel/thread chats. A good implementation should add a backward-compatible core routing/config surface, thread it through buildAgentSessionKey, buildAgentPeerSessionKey, resolveAgentRoute, outbound/base-session routing, thread suffix or inheritance paths, and runtime policy derivation; update strict Zod schemas, exported config types, generated help/docs; and add regression tests for grouped routing, ungrouped compatibility, originating-channel reply delivery, queue serialization, optional thread inheritance, and sandbox/trust-boundary behavior. Treat #20078 as prior art and #7524 as related but narrower.
