## Summary

Current main still implements and documents the Control UI/device-token fallback that this issue reports. Some surrounding behavior has changed since the reported version, but the central token-rotation boundary is not implemented as requested, and there is no canonical related item in the provided context that supersedes it.

## Best Possible Solution

Keep this issue open. The best fix is to define the intended security contract for shared-token rotation, then update the Control UI and gateway auth path accordingly: either bind/revoke browser device tokens when gateway auth rotates or stop advertising/using retry_with_device_token after AUTH_TOKEN_MISMATCH for shared-token sessions. Add a visible Control UI logout that clears the sessionStorage gateway token, persisted device auth, and in-memory client state, then cover old-token refresh and logout-after-refresh behavior with focused regression tests.
