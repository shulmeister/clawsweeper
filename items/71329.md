## Summary

Keep #71329 open. Current main still ties Slack lifecycle status reactions to the ack-reaction gate, so `messages.statusReactions.enabled: true` alone does not activate the controller in the scenario this PR targets. The PR is a plausible narrow fix, with one test-hardening note from the existing bot review.

## Best Possible Solution

Keep the PR open and move it toward merge with the narrow Slack prepare activation change: treat `messages.statusReactions.enabled === true` as sufficient to create the status-reaction promise when a Slack message timestamp exists, preserve `enabled: false` and the existing plain ack fallback, and add regression tests for both explicit-enable-without-ack and explicit-disable-with-ack paths before running the Slack extension changed gate.
