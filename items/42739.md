## Summary

#42739 is a useful idea, but current main clearly scopes alternate memory backends to the memory-plugin/community-plugin path rather than OpenClaw core. Memvid should be pursued as a third-party memory plugin published via ClawHub or npm, not kept open as a core issue.

## Close Comment

Closing this as a `clawhub` item after Codex review of current `main` (`564f820efa2bb81ae49b8f55a93e41492ad31917`). The project currently treats alternate memory systems as plugin-scoped work, not core roadmap work: `VISION.md:52-64` explicitly keeps optional capability in plugins and calls memory a dedicated plugin slot, `docs/plugins/sdk-entrypoints.md:78-80` and `docs/plugins/sdk-overview.md:184-188` expose a first-class `kind: "memory"` / `registerMemoryCapability(...)` path, and `docs/tools/plugin.md:109-124` plus `extensions/memory-lancedb/package.json:18-31` show install-on-demand memory plugins already ship through the plugin ecosystem. Given that scope, a Memvid-backed memory layer is better pursued as a community memory plugin published on ClawHub or npm rather than tracked as an OpenClaw core issue. If a concrete core seam is missing for that plugin path, please open a focused follow-up issue for that specific SDK/core gap.
