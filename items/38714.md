## Summary

Keep #38714 open. Current main still detects Discord reaction add/remove events and routes them as session-scoped system events, but there is no first-class internal Hooks or typed plugin-hook reaction event for automation. The issue has only the non-protected `stale` label, author association is `NONE`, and the provided related context does not show a duplicate, superseding PR, maintainer decision, or posted plugin workaround that can observe these Discord reaction events through todayâ€™s core APIs.

## Best Possible Solution

Keep this issue open until OpenClaw has a first-class reaction automation surface. The likely maintainer path is an additive, channel-generic reaction event for internal Hooks and/or typed plugin hooks, with a structured payload for action, emoji, reactor, channel/account/conversation, target message id, and thread/session correlation; then bridge the Discord reaction listener after its existing authorization/notification checks and update docs plus focused Discord/hook tests.
