## Summary

Keep #38714 open. Current main still supports Discord reaction ingress, but it only queues reactions as routed system events; it does not expose Discord reaction add/remove events through the internal Hooks event surface or the typed plugin hook catalog/message_received bridge. Related #11747 tracks waking/auto-responding to reaction system events, while the closed #52658 and #60805 reports overlap but do not provide a canonical shipped replacement for this hook/API gap.

## Best Possible Solution

Keep #38714 open until maintainers choose and implement a reaction event contract. The likely fix belongs in the Discord reaction handling path plus internal/plugin hook surfaces: either add dedicated typed reaction add/remove hooks or emit structured synthetic message_received/internal message:received events containing emoji, action, reactor, target message, route/session, account/guild/channel/thread metadata, and removal state. Add Discord listener tests, plugin/internal hook tests, and docs. Coordinate agent wake/auto-response behavior separately with #11747 rather than closing this hook/API request as a duplicate.
