## Summary

Keep open. Current main still does not implement the PR’s central recovery for a numeric Telegram chat ID that lands in `delivery.channel`; it only treats `jobPayload.to` as the explicit destination. The PR remains a plausible focused bug fix for #70758, but it also has an unresolved security-review concern about inferred explicit targets bypassing implicit allowFrom handling, so maintainer review is still needed rather than auto-closing it.

## Best Possible Solution

Keep PR #70814 open. The best path is to finish the narrow cron resolver fix with regression coverage for both `delivery.to` and misplaced numeric `delivery.channel`, while explicitly resolving the allowFrom/security concern by either preserving implicit authorization checks for inferred targets or rejecting ambiguous malformed channel values.
