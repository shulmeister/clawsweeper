## Summary

Keep open. Current `main` still drops the explicit Discord target kind in `parseDiscordExplicitTarget`, and the downstream outbound normalization then coerces the bare numeric id to `channel:<id>`, which matches the PR’s reported failure mode. The PR’s one-line fix and focused tests are still relevant; this is not a candidate for conservative auto-close.

## Evidence

- **Current main still strips the target kind:** `parseDiscordExplicitTarget(raw)` currently returns `to: target.id` instead of the normalized `kind:id` form, so `user:<id>` becomes a bare id before downstream delivery resolution.
  - file: [extensions/discord/src/channel.ts:356](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/discord/src/channel.ts#L356)

  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Normalized target contract includes the kind prefix:** `MessagingTarget` carries both `id` and `normalized`, and `buildMessagingTarget` defines `normalized` as `${kind}:${id}`. Returning `id` discards information the parser already computed.
  - file: [src/channels/targets.ts:24](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/channels/targets.ts#L24)
