## Summary

Keep #14376 open. Current main has adjacent pieces: cron records lastErrorReason, one-shot jobs have transient retry config, and recurring jobs use consecutive-error backoff. The requested reason-aware recurring cron guardrail is still missing: no billing/auth/format circuit breaker, no provider Retry-After scheduling, no job-level cooldown/disabled-until state, and no configurable reason-specific policy.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely implementation should span src/cron/service/timer.ts, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify terminal run errors once, persist/display reason and cooldown state, respect Retry-After where exposed, auto-pause or circuit-break recurring jobs for billing/auth/format with safe configurable defaults and one alert, and keep timeout/network/rate-limit jobs enabled with bounded backoff coordinated with existing auth-profile cooldowns.
