## Summary

Current `main` only partially covers this request. Cron now records `lastErrorReason` and one-shot (`at`) jobs can disable immediately on permanent errors, but recurring cron jobs still apply the same generic exponential backoff regardless of whether the failure was `billing`, `auth`, `rate_limit`, `timeout`, or `format`, and there is no job-level circuit breaker or Retry-After-aware scheduling for recurring jobs.

## Evidence

## Risks / Open Questions

- Maintainers may mistake `lastErrorReason` support for full resolution; the core scheduler still lacks reason-aware mitigation for recurring cron jobs.
- Real retry-storm reports in the issue remain plausible on current `main` because recurring jobs are not auto-disabled or globally cooled down based on failover reason.
- I could not inspect GitHub PR metadata with `gh` in this environment because `GH_TOKEN` was not configured, so this review is based on current `main` code/docs/tests only.
