## Summary

Keep open. Current main has partial supporting work, but it still does not implement the requested reason-aware recurring cron guard policy. Cron records `lastErrorReason`, has one-shot transient retry handling, generic recurring backoff, failure alerts, and model/auth-profile cooldowns, but recurring jobs do not circuit-break on billing/auth/format, do not consume provider `Retry-After`, and do not provide a cron-level provider/global circuit breaker for the retry-storm scenarios described in the issue comments.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling applies an explicit reason-aware guard policy. The likely implementation belongs around `src/cron/service/timer.ts` and cron state/config types: classify the terminal run error once, persist the reason, use provider `Retry-After` where available for rate limits, circuit-break or auto-pause recurring jobs for billing/auth/format according to safe configurable defaults, and add tests/docs for recurring jobs plus interaction with existing auth-profile cooldowns.
