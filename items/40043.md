## Summary

Keep #40043 open. Current main still does not implement arbitrary recursive skill discovery and does not include the PR's generic startup warning for a selected provider with no usable auth. The Anthropic-default symptom from #39903 is partly reduced on main, and #69663 is a focused open PR for the recursive-scan subset, but this combined PR is not fully implemented or fully superseded. It also should not merge as-is because the existing review comments identify real correctness blockers.

## Best Possible Solution

Keep #40043 open for maintainer rework or contributor split. The recursive skill discovery part should move to or align with the focused #69663 path, with bounded traversal, containment and symlink-loop protection, candidate budgeting, watcher/docs alignment, and regression tests for grouped managed/workspace/extra skill roots. The startup auth warning should be a separate best-effort diagnostic using the same runtime auth contract as `hasAvailableAuthForProvider`, through a read-only/non-throwing path, or explicitly dropped if maintainers decide the current Anthropic auth-signal defaulting fully resolves #39903.

## Risks / Open Questions

- Merging #40043 as written could put write-capable auth-store I/O on the gateway startup path and turn a diagnostic warning into a startup failure.
- Merging the auth warning as written could produce false warnings for valid auth sources such as custom provider API keys, local synthetic auth, Bedrock SDK auth, or broader runtime profile resolution.
- Merging the recursive scanner as written could traverse large non-skill trees without preserving the existing `maxCandidatesPerRoot` candidate budget.
- Closing #40043 before #69663 or equivalent recursive-scan work lands would leave the nested skill-layout request unresolved unless maintainers intentionally decide the narrow `dir/skills/*` support is sufficient.
