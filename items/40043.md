## Summary

Keep #40043 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still lacks the PR's two central behaviors: gateway startup does not warn when the selected provider lacks usable auth, and skill discovery still does not support arbitrary recursive nested skill layouts. The PR discussion also has unresolved correctness blockers, so it should be reworked or split rather than closed as implemented or merged as-is.

## Best Possible Solution

Keep #40043 open for maintainer rework or ask the contributor to split it. The recursive skill discovery change should be implemented in the core skill loader with max-candidate budgeting, root-containment and symlink-loop protections, tests for nested bundle layouts, and a deliberate watcher/docs update. The startup auth warning should be a separate best-effort, read-only diagnostic based on hasAvailableAuthForProvider() or the same runtime auth contract, and it must never block gateway startup or warn for valid auth sources. If #69663 lands first, use it as the canonical path for the recursive-scan portion and keep/replace only the auth-warning portion here.

## Risks / Open Questions

- Merging the PR as written could put write-capable auth-store I/O on the gateway startup path and turn a diagnostic warning into a startup failure.
- Merging the auth warning as written could warn even when requests would succeed through supported config, plugin, local, Bedrock, or runtime auth sources.
- Merging the recursive scanner as written could traverse large non-skill trees without enforcing the existing candidate budget.
- A newer open PR (#69663) appears to track the recursive-scan half, but it does not cover the startup auth warning; closing #40043 as fully superseded would drop a remaining requested behavior unless maintainers explicitly split the work.
