## Summary

Current `main` no longer matches the exact silent-exit report for `openclaw agent`: both gateway and embedded paths now reject missing session selectors, and CLI runtime errors exit nonzero. But the item is not strong enough to close conservatively because `docs/tools/agent-send.md` still advertises selectorless `openclaw agent --message ...`, and this read-only review did not produce high-confidence proof that the reported ~30s headless gateway/discovery stall is gone on current `main`. The `lobster`/`llm-task` command expectations also look like a docs/expectation mismatch rather than a confirmed core regression: current docs and plugin registration treat both as optional plugin tools, not top-level CLI commands.

## Evidence

- **Current agent paths reject missing selectors instead of silently returning success:** Gateway CLI path throws `Pass --to <E.164>, --session-id, or --agent to choose a session` when no selector is provided, and the embedded path enforces the same guard.
  - file: [src/commands/agent-via-gateway.ts:92](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/commands/agent-via-gateway.ts#L92)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '87,198p'`
- **Embedded agent path has the same validation:** The local/embedded agent command also throws on missing `--to`, `--session-id`, `--sessionKey`, or `--agentId`.
  - file: [src/agents/agent-command.ts:251](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/agent-command.ts#L251)

  - file: [src/cli/cli-utils.ts:34](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/cli-utils.ts#L34)

## Risks / Open Questions

- The reported ~30s headless Linux delay was not live-reproduced in this read-only review, so closing would be under-evidenced if that regression still exists in some CLI path outside the current obvious discovery defaults.
- `docs/tools/agent-send.md` still shows selectorless `openclaw agent --message ...`, so user confusion around the required selector remains present on `main` even though the code now validates it.
- There is no pinpointed fix commit or release note here for the exact `v2026.3.31 exits 0 with no output` report, only current-source evidence that the behavior has changed.
