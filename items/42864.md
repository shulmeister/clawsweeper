## Summary

Current main already addresses the reported oversized-session timeout loop: embedded runs now force timeout-recovery compaction before retrying high-context LLM timeouts, and unrecoverable overflow/compaction-failure cases reset the session instead of letting the same poisoned session keep retrying. The changelog records this as shipped in v2026.3.28.

## Risks / Open Questions

- This is not the exact configurable `circuitBreaker` API proposed in the issue; it is a different implementation that addresses the reported oversized-context timeout loop directly.
- If a remaining loop still exists on current releases, it would likely be a narrower provider-specific timeout path rather than the general session-bloat failure described here.

## Close Comment

Closing after Codex review. Current `main` already addresses the failure described here: `src/agents/pi-embedded-runner/run.ts:962` now forces timeout-recovery compaction before retrying high-context LLM timeouts, specifically to stop repeated oversized-request loops, and `src/auto-reply/reply/agent-runner-execution.ts:1316` resets the session when overflow/compaction failure still escapes. If you still see a repeat loop on `v2026.4.22` or newer, please open a fresh issue with the provider/model, logs, and repro path so maintainers can check for a remaining provider-specific gap.
