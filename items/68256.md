## Summary

Current main already resolves this non-default-agent OpenRouter path through the correct per-agent auth store. Telegram-bound sessions resolve the active agent id and agentDir, the embedded runner passes that agentDir into model resolution and auth selection, and Pi discovery builds runtime credentials from that agent's auth-profiles.json. The behavior is also present in the shipped v2026.4.24 release.

## Best Possible Solution

Close this issue and keep the shipped per-agent auth-profile runtime path. If a v2026.4.24-or-newer user still sees empty OpenRouter replies, the next best action is a new issue with current gateway logs, agent id, model ref, auth profile id/order, and whether the per-agent auth-profiles.json contains an openrouter api_key profile.
