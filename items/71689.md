## Summary

Keep open. Current main still restores the task registry by calling the SQLite-backed store and catching any failure as a warning only; there is no malformed-database detection, bad DB snapshot/rebuild, reopened recovered store, or task-registry audit/operator finding. The provided related items are title-overlap only and do not supersede corrupted `tasks/runs.sqlite` recovery.

## Best Possible Solution

Keep this issue open. Implement task-registry-owned SQLite corruption recovery in `src/tasks/task-registry.store.sqlite.ts` and expose restore/audit state through `src/tasks/task-registry.ts` and `src/tasks/task-registry.audit.ts`: detect malformed/corrupt SQLite errors during open, PRAGMA/schema setup, statement preparation, and load; close the handle; snapshot `runs.sqlite` and sidecars; attempt safe recovery/rebuild or fall back to an explicit empty registry only with an operator-facing audit/event; and add regression tests for malformed database restore.

## Risks / Open Questions

- A naive recovery path could silently lose durable task state unless it preserves `runs.sqlite` plus `-wal` and `-shm` before replacing or rebuilding anything.
- Malformed SQLite can fail during database open, PRAGMA/schema migration, statement preparation, or SELECT, so the fix should classify corruption across the full restore path rather than adding a broad empty-registry fallback.
