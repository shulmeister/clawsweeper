## Summary

Keep this issue open. Current main has PWA/Web Push infrastructure, `chat.inject`, `openclaw system event`, `openclaw nodes notify`, and the generic `openclaw gateway call` escape hatch, but it still lacks a first-class `openclaw notify <message>` command or production Gateway/UI notification broadcast for visible connected-session notifications without an agent turn.

## Best Possible Solution

Keep this issue open. The best implementation path is a production-facing `openclaw notify <message>` wrapper backed by an explicit admin-scoped Gateway method such as `notifications.broadcast` or `ui.notify`, plus a typed Control UI event/toast handler for connected sessions. It should define scope and UX policy, optionally fan out through Web Push for subscribed browsers, and add focused CLI, Gateway, and UI tests plus CLI/docs coverage. If maintainers want this plugin-first, first add a documented plugin/UI notification API; the current plugin surface is not enough to close the request.

## Risks / Open Questions

- The existing `openclaw gateway call push.web.test --params ...` path may satisfy a narrow browser-push-only workflow, but it remains low-level, test-named, and subscription-based rather than a connected-session UI notification contract.
- A production notify surface needs explicit maintainer decisions for operator scope, rate limiting or spam UX, in-app toast behavior, and whether Web Push should be reused as a fallback.
