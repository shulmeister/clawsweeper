## Summary

Keep open. Current `main` still undercalculates `/status` cost because the status message passes only `input` and `output` into the cost estimator, even though the same code path tracks and displays `cacheRead`/`cacheWrite`, and the shared estimator supports cache-token pricing.

## Evidence

- **`/status` tracks cache token usage:** `buildStatusMessage` loads `cacheRead` and `cacheWrite` from the session entry and transcript fallback before rendering status output.
  - file: [src/status/status-message.ts:547](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/status/status-message.ts#L547)
  - command: `nl -ba src/status/status-message.ts | sed -n '535,605p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **`/status` omits cache tokens from cost estimation:** The `/status` cost calculation calls `estimateUsageCost` with only `input` and `output`; `cacheRead` and `cacheWrite` are not passed.
  - file: [src/status/status-message.ts:806](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/status/status-message.ts#L806)

  - file: [src/status/status-message.ts:887](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/status/status-message.ts#L887)

## Risks / Open Questions

- `/status` underreports billed cost whenever cached prompt tokens are present; on current `main`, both `cacheRead` and `cacheWrite` are excluded from the estimate.
- Because cache usage is displayed separately from cost, users can trust the token counts while still getting a materially wrong dollar estimate.
