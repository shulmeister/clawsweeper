## Summary

Current main already routes `tools.exec.security` into the exec runtime, regression coverage exists, and the changelog shows the host-exec default-policy fix shipped after the reported `2026.3.31` version.

## Evidence

- **Session/agent/global exec policy is propagated into runtime defaults:** `resolveExecDefaults` prefers session, then agent, then global `tools.exec.security` / `tools.exec.ask`, and returns those values as the exec defaults consumed by agent tooling.
  - file: [src/agents/exec-defaults.ts:121](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/exec-defaults.ts#L121)
  - command: `nl -ba src/agents/exec-defaults.ts | sed -n '91,140p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Chat/tool surfaces pass resolved exec policy into the exec tool:** `createOpenClawCodingTools` forwards the resolved `security` and `ask` values into `createLazyExecTool`, so web/chat surfaces use the configured exec policy instead of ignoring it.
  - file: [src/agents/pi-tools.ts:500](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/pi-tools.ts#L500)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '499,505p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Exec runtime enforces configured security, and only throws deny on effective deny:** At execution time, `configuredSecurity` comes from the resolved defaults before host execution. The `exec denied: host=... security=deny` error is thrown only after merging requested policy with the stricter effective host policy.
  - file: [src/agents/bash-tools.exec.ts:1480](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/bash-tools.exec.ts#L1480)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '1478,1492p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Host deny error path is conditional, not unconditional:** `resolveExecHostApprovalContext` computes `hostSecurity = minSecurity(params.security, approvals.agent.security)` and throws only when that merged value is `deny`, which is inconsistent with the report that config is categorically ignored.
  - file: [src/agents/bash-tools.exec-host-shared.ts:198](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/bash-tools.exec-host-shared.ts#L198)

- **Regression coverage exercises the `security: full` gateway path:** Tests assert gateway defaults resolve to `security: "full"` / `ask: "off"`, and a tool configured with `tools.exec.security: "full"` completes successfully.
  - file: [src/agents/exec-defaults.test.ts:87](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/exec-defaults.test.ts#L87)

## Risks / Open Questions

- A local `~/.openclaw/exec-approvals.json` policy or persisted session-level exec override can still tighten effective policy back to `deny`; the issue did not include those artifacts, so the original environment may have had an external override rather than a current-main core bug.
