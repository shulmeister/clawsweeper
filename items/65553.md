## Summary

Keep open. Current main contains several partial QMD mitigations, but it does not prove the Raspberry Pi/arm64 standalone memory CLI hang is fixed. The current `openclaw memory search` path can still create a full QMD manager, start boot update/embed work, and then wait for that pending work during manager close; that matches the reported long-lived CLI process and ARM QMD embed-timeout discussion.

## Best Possible Solution

Keep this issue open. The best fix is likely in the memory-core QMD CLI/manager boundary: add a one-shot CLI mode that does not start boot update/embed maintenance for `openclaw memory search`, or make CLI close cancel/detach pending boot maintenance safely instead of awaiting it. Also add regression coverage where QMD update/embed hangs and `openclaw memory search` exits promptly; keep the existing status-only path and configurable QMD timeouts as supporting mitigations.
