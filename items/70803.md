## Summary

Keep #70803 open. Current main still has the exact normalization gap the PR fixes: `normalizeWhatsAppTarget`/`isWhatsAppUserTarget` only accept `@lid`, while the runtime JID-to-E.164 path already accepts `@hosted.lid`. The PR is a focused contributor bug fix with matching regression tests, and the Greptile comment positively reviewed the one-line regex change as isolated and safe.

## Best Possible Solution

Keep #70803 open and review it as a narrow WhatsApp plugin bug fix. The likely best implementation is the PR's small regex update plus the lower/upper-case `@hosted.lid` regression cases in `extensions/whatsapp/src/resolve-target.test.ts`; before landing, run the WhatsApp targeted test/lint or the appropriate changed extension gate. A small follow-up could centralize the accepted LID suffix pattern so `normalize-target.ts`, `targets-runtime.ts`, and `identity.ts` do not drift again.
