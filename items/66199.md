## Summary

Keep open. Current main still sends iMessage inbound text to the agent through raw `BodyForAgent`, so the hardening request is not implemented. The submitted PR should not be merged as-is, because it wraps only the legacy envelope `Body` path while the reply pipeline prefers `BodyForAgent` for prompt assembly; the existing review comments correctly flag that mismatch.

## Best Possible Solution

Keep the PR open but require changes before merge. A safe fix should update the actual iMessage model-bound path, not only `formatInboundEnvelope`: either wrap/sanitize `BodyForAgent` while preserving raw `CommandBody` for command parsing, or introduce structured `UntrustedContext`/user-context handling that avoids duplicating raw message text as trusted prompt content. Add focused tests for `buildIMessageInboundContext` and reply prompt assembly proving the iMessage message body reaches the agent with the intended untrusted-content boundary.

## Risks / Open Questions

- Merging the PR as submitted would create a false sense of security: legacy `Body` consumers would see wrappers, but normal agent prompt assembly would still use raw `BodyForAgent`.
- A correct fix needs explicit maintainer/product judgment on whether current channel messages should be wrapped as untrusted prompt content, or whether only supplemental context/history should be marked untrusted while authorized sender text remains ordinary user input.
