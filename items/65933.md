## Summary

Keep open. Current main still has the reported behavior: `exec` accepts the model/tool-call `security` argument, normalizes it, and applies `minSecurity(configuredSecurity, requestedSecurity)`, so configured `full` plus requested `allowlist` still becomes `allowlist`. The PR discussion also converged on ignoring model-supplied security entirely after review caught both downgrade and escalation variants, and that end-state is not on main.

## Best Possible Solution

Keep this PR open or land a refined version of its final approach: in `src/agents/bash-tools.exec.ts`, make normal exec use `let security = configuredSecurity` and ignore model-supplied `params.security`, while preserving the explicit elevated `full` override. Add focused regression tests for configured `full` plus model `allowlist`, configured `allowlist` plus model `deny`, and configured `deny` plus model `full`, and avoid adding an unused `maxSecurity` helper unless a real call site needs it.

## Risks / Open Questions

- A partial floor-only or `maxSecurity` implementation can introduce the opposite bug: model-supplied `security:"full"` could escalate an operator-configured `deny` or `allowlist` policy.
- The final patch should preserve the explicit elevated full override, because the PR discussion notes it is operator-granted and reachable in production.
- Docs/schema may need a small follow-up if `security` remains in the model-facing exec schema but is ignored for normal model tool calls.
