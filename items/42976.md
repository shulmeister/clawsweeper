## Summary

Keep open. Current `main` does not implement the PR’s two core changes: the sample custom redaction regex is still rejected by the safe-regex guard, and session persistence still does not redact message content or thinking blocks before writing `session.jsonl`.

## Evidence

- **Persistence wrapper still only applies provenance:** `guardSessionManager()` passes `transformMessageForPersistence` that only calls `applyInputProvenanceToUserMessage(...)`; there is no `redactSensitiveText` import or redaction step in the current file.
  - file: [src/agents/session-tool-result-guard-wrapper.ts:74](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/agents/session-tool-result-guard-wrapper.ts#L74)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **All persisted messages flow through that transform:** `installSessionToolResultGuard()` defines `persistMessage()` from `transformMessageForPersistence` and uses it for normal messages and tool results before `originalAppend(...)`, so the missing redaction step means persisted transcript entries are not sanitized here.
  - file: [src/agents/session-tool-result-guard.ts:116](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/agents/session-tool-result-guard.ts#L116)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Custom redact patterns still compile through safe-regex:** `logging.redactPatterns` are parsed with `compileConfigRegex(...)`, which delegates to `compileSafeRegexDetailed(...)`, so the safe-regex behavior directly controls whether the PR’s sample pattern is accepted.
  - file: [src/logging/redact.ts:57](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/logging/redact.ts#L57)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Tokenizer bug is still present on main:** In `tokenizePattern()`, backslashes are handled before the `inCharClass` branch. That is the exact ordering the PR changes; escaped sequences inside `[...]` are still tokenized as standalone simple tokens on current `main`.
  - file: [src/security/safe-regex.ts:143](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/security/safe-regex.ts#L143)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **PR sample regex still fails on current main:** Read-only reproduction on `main` with Node 22 strip-types: `node --experimental-strip-types --input-type=module -e 'import("./src/security/safe-regex.ts").then((m) => { const pattern = String.raw`([\\w]|[-.])+@([\\w]|[-.])+\\.\\w+`; const result = m.compileSafeRegexDetailed(pattern); console.log(JSON.stringify({ pattern, reason: result.reason, hasRegex: Boolean(result.regex) })); })'` returned `{"pattern":"([\\\\w]|[-.])+@([\\\\w]|[-.])+\\\\.\\\\w+","reason":"unsafe-nested-repetition","hasRegex":false}`.
  - command: `node --experimental-strip-types --input-type=module -e 'import("./src/security/safe-regex.ts").then((m) => { const pattern = String.raw`([\\w]|[-.])+@([\\w]|[-.])+\\.\\w+`; const result = m.compileSafeRegexDetailed(pattern); console.log(JSON.stringify({ pattern, reason: result.reason, hasRegex: Boolean(result.regex) })); })'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Current tests do not cover the escaped-char-class case:** `safe-regex.test.ts` covers nested repetition and a few simple safe patterns, but there is no regression test for patterns like `([\\w]|[-.])+...`, so the reported bug is neither fixed nor covered on `main`.
  - file: [src/security/safe-regex.test.ts:19](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/security/safe-regex.test.ts#L19)
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)

## Risks / Open Questions

- The PR as described remains needed: users configuring `logging.redactPatterns` with escaped character classes can still have valid patterns rejected on current `main`.
- Session transcript persistence still lacks a built-in redaction step in `guardSessionManager()`, so the PR’s claimed thinking/session-log redaction behavior is not already implemented.
- The proposed PR also appears incomplete from the provided review context because it would redact `text`/`thinking` blocks but not `toolCall.arguments`, which could still leave sensitive tool inputs in `session.jsonl`.
