## Summary

Current `main` does not contain this PR’s health-refresh logic. `refreshGatewayHealthSnapshot()` still dedupes to a single in-flight promise with no queued follow-up state, so newer probe intent can still be dropped, and `getHealthSnapshot()` still has no `runtimeSnapshot` plumbing. This PR is not implemented on `main` or in release `v2026.4.22`.

## Evidence

- **No pending probe/runtime follow-up logic on main:** `refreshGatewayHealthSnapshot()` only starts one refresh and returns the existing `healthRefresh` promise for concurrent callers. There is no `pendingProbe`, `pendingRuntimeSnapshot`, or runtime snapshot getter on current `main`.

- **Cached health path still triggers background probe:false refresh:** The health RPC responds from cache, then fires `refreshHealthSnapshot({ probe: false })` in the background.
