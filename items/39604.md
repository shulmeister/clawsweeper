## Summary

Current main still does not implement the requested web_fetch private-network opt-in. The public config schema and type only expose `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, and the runtime only forwards that RFC2544 policy to the guarded fetch path. Localhost, LAN, and private DNS results remain blocked by current tests and docs. The lower-level SSRF guard already supports `allowPrivateNetwork`/`dangerouslyAllowPrivateNetwork`, so the remaining work is the web_fetch config/runtime/docs/test wiring plus maintainer security judgment. The linked PR #39630 is the active implementation path, but it is open and unmerged, so the issue is not obsolete or shipped.

## Best Possible Solution

Keep #39604 open until maintainers either land an explicit web_fetch private-network opt-in or decide not to support it. The likely path is to supersede or update PR #39630 against current main, wiring a deliberately named default-off config through `src/config/zod-schema.agent-runtime.ts`, `src/config/types.tools.ts`, generated config metadata/help, `src/agents/tools/web-fetch.ts`, SSRF/cache tests, and `docs/tools/web-fetch.md`. Given current naming, `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` may fit the existing security vocabulary better than the originally proposed flat key, but the user problem remains unsolved on main.

## Risks / Open Questions

- A general private-network web_fetch opt-in expands SSRF reach when enabled, so the final config shape should remain explicit, default-off, and covered by regression tests and docs.
- Do not treat `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` as satisfying this issue; it only covers the 198.18.0.0/15 fake-IP proxy range, not localhost, 10.x, 192.168.x, or 172.16-31.x generally.
- Closing the issue now in favor of open PR #39630 would be premature because the PR is unmerged and may be revised or rejected.
