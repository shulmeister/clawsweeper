## Summary

Keep this issue open. Current main still does not expose a general `web_fetch` private-network opt-in for localhost, LAN, or internal/private address ranges. It only supports the narrow `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` exception for 198.18.0.0/15 fake-IP proxy compatibility. The same-author implementation PR #39630 is still open and unmerged in the provided GitHub context, so it remains an implementation candidate rather than a reason to close this issue.

## Best Possible Solution

Keep this issue open until maintainers either land an explicit default-off private-network opt-in for `web_fetch` or make a product/security decision not to support it. The likely path is under `tools.web.fetch.ssrfPolicy`, with a deliberately named option such as `dangerouslyAllowPrivateNetwork`, schema/types/generated help, runtime guard policy wiring, cache isolation, docs, and SSRF tests, while preserving the existing RFC2544-only option.

## Risks / Open Questions

- A broad private-network opt-in expands SSRF reach when enabled, so any implementation needs explicit default-off semantics, clear naming, docs, cache-key isolation, and regression tests.
- The current RFC2544 setting only solves the 198.18.0.0/15 fake-IP proxy case; it does not solve localhost, LAN, 10.x, 172.16-31.x, or 192.168.x workflows described here.
- PR #39630 likely needs review or adjustment against the current `ssrfPolicy` shape before landing, rather than closing this issue as already handled.
