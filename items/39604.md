## Summary

Keep #39604 open. Current main at e672b614 still does not expose a general private-network opt-in for `web_fetch`: schema/types/docs/runtime/tests only support the narrow `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` exception for 198.18.0.0/15 fake-IP proxy compatibility. The linked implementation PR #39630 is still open/unmerged in the provided GitHub context, and #43700 was closed unmerged, so the requested localhost/LAN/private/internal behavior is not shipped or safely superseded.

## Best Possible Solution

Keep #39604 open until maintainers either land an explicit default-off `web_fetch` private-network opt-in or make a product decision not to support it. The likely implementation path is under `tools.web.fetch.ssrfPolicy`, preferably with a deliberately named option such as `dangerouslyAllowPrivateNetwork`, plus schema/types/generated help, runtime guarded-fetch policy wiring, cache-key isolation, docs, and SSRF tests while preserving the existing RFC2544-only option.

## Risks / Open Questions

- A broad private-network opt-in expands SSRF reach when enabled, so implementation should use explicit dangerous naming or equally clear default-off semantics, cache-key isolation, generated config metadata, docs, and regression tests.
- The existing RFC2544 setting only solves the 198.18.0.0/15 fake-IP proxy case; treating it as a complete fix would leave localhost, LAN, and internal-service fetch workflows unresolved.
- The open PR #39630 may still need rebase/design adjustment against the current `ssrfPolicy` shape before it is safe to land.
