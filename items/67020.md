## Summary

Current main still matches the reported Slack bug. Slack config and docs continue to accept/advertise `dmHistoryLimit`, but the Slack monitor only derives and carries `historyLimit`; 1:1 DMs are excluded from the room/history gates used for pending-history recording, prompt-body history injection, structured `InboundHistory`, and history clearing. The related Slack fix PR is closed and unmerged, so it does not supersede this issue.

## Best Possible Solution

Keep this issue open and land a focused Slack monitor fix: derive `account.config.dmHistoryLimit`, carry it through `SlackMonitorContext`, compute an effective history limit for 1:1 DMs versus room/group contexts, record pending DM history, use that limit for `combinedBody`, `InboundHistory`, and history clearing, and add a regression test for a non-threaded 1:1 Slack DM with `dmHistoryLimit > 0`. After Slack is fixed, decide whether the broader cross-channel `dmHistoryLimit` parity concern should become a separate shared-abstraction issue.
