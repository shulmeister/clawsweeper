## Summary

The request is a valid idea, but current main’s architecture points it away from OpenClaw core and toward a provider/plugin implementation distributed through the plugin ecosystem or ClawHub. Core owns generic model failover; provider/runtime hooks already exist for provider-specific local-backend behavior.

## Risks / Open Questions

- A generic core-level script hook for model failures would expand the trusted execution surface in config and cut across the current provider/plugin boundary.
- A plugin solution still requires someone to build and maintain the Ollama/LM Studio startup behavior; closing here does not mean the capability already exists out of the box.

## Close Comment

Closing this as a Codex review because it fits the plugin/ClawHub boundary better than OpenClaw core. Current main keeps optional capability out of core (`VISION.md:54`), and the provider architecture explicitly says core owns generic failover while provider plugins own provider-specific runtime behavior (`docs/plugins/architecture.md:651`). There is also already a provider seam for this kind of work: the bundled LM Studio plugin registers `wrapStreamFn` (`extensions/lmstudio/index.ts:117`) and uses it to do local backend side effects before inference (`extensions/lmstudio/src/stream.ts:160`). A local-server starter for Ollama/LM Studio is better implemented as a provider/plugin extension rather than a new global `openclaw.json` script-hook surface.
