## Summary

Keep #42245 open. Current main still allows `cron add --message` without `--agent`, sends `agentId: undefined`, and later resolves missing cron job agent IDs to the configured default agent. I found no current warning path or regression tests for the PR’s behavior, and linked issue #42196 still describes the same user-facing confusion.

## Best Possible Solution

Keep the PR open and update it against current main. The best implementation is a small CLI-only warning for `payload.kind === "agentTurn"` when no explicit `--agent` was provided, emitted on stderr, with tests for the reviewed edge cases and wording that reflects the configured default agent rather than assuming `main`.

## Risks / Open Questions

- The PR should not hardcode `main` as the default agent in warning text unless it actually resolves the configured default; current main supports non-`main` defaults via `agents.list[].default` or the first configured agent.
- The warning should stay on stderr or otherwise avoid corrupting stdout JSON output from `cron add`.
- Tests should cover no warning for `--system-event`, warning with `--session-key` but no `--agent`, and no warning when `--agent` is explicitly provided.
