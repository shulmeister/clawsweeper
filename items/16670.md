## Summary

Keep #16670 open. Current main supports memory search configuration, provider auto-detection, docs, diagnostics, and `openclaw memory status --deep`, but the requested onboarding/configure step for memory embedding provider, model, API key or SecretRef, and verification is still missing. The issue also has a fresh maintainer/Codex discussion comment explicitly keeping this gap open.

## Best Possible Solution

Keep #16670 open until OpenClaw adds a dedicated memory/embedding setup path or a maintainer explicitly decides not to make it mandatory. The likely implementation belongs in onboarding/configure: prompt for provider, model, and API key or SecretRef under `agents.defaults.memorySearch`, use the existing embedding provider registry/config schema, verify readiness with a small embedding probe or `openclaw memory status --deep`, and update public docs for API-key, Codex OAuth, and daemon environment requirements.

## Risks / Open Questions

- Making memory/embedding setup mandatory is still a product decision because some installs may intentionally rely on keyless local providers, Bedrock or GitHub Copilot credentials, lexical fallback, or minimal onboarding.
- A fix should handle daemon and LaunchAgent environment visibility and SecretRef-backed credentials so the gateway can resolve embedding keys, not only the interactive shell.
