## Summary

Current main still treats memory-wiki as one gateway-level plugin config, not a per-agent vault config. The issue discussion remains consistent with the code: agent config has no plugin override slot, memory-wiki resolves `api.pluginConfig` once at plugin registration, `wiki_apply` writes through the captured `config.vault.path`, and `wiki_search`/`wiki_get` use `agentId` only for shared-memory fallback, not for selecting a wiki vault. Open PR #67584 appears to be an implementation candidate, but it is still open and unmerged, so this issue should remain open.

## Best Possible Solution

Keep this issue open as the canonical narrow memory-wiki isolation request. The best path is to review and land PR #67584 or an equivalent implementation that resolves the memory-wiki vault per active agent/context, then close this issue as implemented once the fix is on main. The shipped fix should cover `wiki_search`, `wiki_get`, `wiki_apply`, gateway/wiki surfaces where applicable, docs, and tests proving separate agents cannot read or write each other's wiki vaults while preserving the current single-vault default.
