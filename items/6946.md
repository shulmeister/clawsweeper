## Summary

Keep #6946 open. Current main still lacks the requested Telegram standalone processing-indicator message lifecycle: no accepted processingIndicator/deleteOnReply/failure-edit config, no inbound placeholder message ID carried through Telegram context, and no delete/edit cleanup around final replies. Existing Telegram feedback remains adjacent typing, ack/status reactions, preview streaming, and fallback replies. Related implementation PRs #43021 and #43720 were closed unmerged; open PR #40421 is related waiting-tip work but is unmerged and not a shipped replacement.

## Best Possible Solution

Keep #6946 open as the canonical Telegram feature request. Best implementation path: add an opt-in bundled Telegram plugin feature with typed config/schema/help/generated metadata/docs, send a standalone placeholder after an eligible inbound Telegram message is accepted and before dispatch starts, carry the placeholder message ID through Telegram context/dispatch, delete it after successful final delivery, and edit or replace it on dispatch failure. Add focused tests for success, error fallback, cleanup failure, preview streaming interaction, group/forum topic handling, and superseded turns.
