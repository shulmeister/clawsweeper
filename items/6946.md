## Summary

Current main still does not implement the requested configurable Telegram processing-indicator message lifecycle. Telegram has adjacent preview-stream tool progress, native-command progress placeholders, and ack/status reactions, but none sends a standalone visible message immediately after an inbound user message, then deletes it on final reply or edits it on failure. The related implementation PRs #43021 and #43720 were closed unmerged, and the issue discussion explicitly treats typing TTL work as complementary rather than a replacement.

## Best Possible Solution

Keep the issue open. The best path is an opt-in implementation in the bundled Telegram plugin: add schema/types/docs/generated metadata, send a standalone placeholder immediately after an eligible inbound message is accepted, store the message ID through dispatch, delete it after successful final delivery, and edit or replace it on dispatch failure. Use the native-command placeholder path and closed PRs #43021/#43720 as prior art, but implement fresh against current main with config and Telegram dispatch tests.
