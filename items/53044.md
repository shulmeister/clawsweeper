## Summary

Current main no longer matches the PR’s reported failure mode. Remote plain-HTTP Control UI sessions without device identity are rejected during handshake, and the localhost insecure-auth compatibility path preserves scopes so approval-capable UI sessions still retain operator access. The proposed client-id shortcut is also inconsistent with current approval routing, which still requires approval scopes for exec approval methods/events.

## Close Comment

Closing from a Codex review because the failure mode described here no longer reproduces on current `main`. Device-less Control UI over plain HTTP is now rejected for non-local clients during the WebSocket handshake (`src/gateway/server/ws-connection/connect-policy.ts:129`, `src/gateway/server/ws-connection/message-handler.ts:653`), and the localhost compatibility path for `gateway.controlUi.allowInsecureAuth=true` explicitly preserves scopes instead of stripping them (`src/gateway/server/ws-connection/message-handler.ts:616`). The current suite also proves that a localhost Control UI connection without device identity can still complete an admin RPC on that path (`src/gateway/server.auth.control-ui.suite.ts:105`, `src/gateway/server.auth.control-ui.suite.ts:318`).

The proposed client-id shortcut would also be out of sync with the remaining approval transport, which is still scope-gated: `hasExecApprovalClients()` only counts `operator.admin`/`operator.approvals` (`src/gateway/server-request-context.ts:92`), approval events are broadcast only to `operator.approvals` clients (`src/gateway/server-broadcast.ts:28`), and `exec.approval.*` methods still sit behind the approvals scope (`src/gateway/method-scopes.ts:43`). On current `main`, the original `no-approval-route` report is obsolete rather than something this PR still needs to fix.
