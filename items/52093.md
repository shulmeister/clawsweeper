## Summary

Current `main` already handles the reported Windows `EPERM` rename failure in the shared atomic JSON writer used for `devices/pending.json`, with a regression test covering that path, and the same implementation is present in release `v2026.4.22`.

## Evidence

- **Shared atomic writer now falls back on Windows rename failure:** `writeTextAtomic()` routes the final replace step through `replaceFileWithWindowsFallback()`, which catches Windows `EPERM`/`EEXIST` from `fs.rename()` and falls back to `fs.copyFile()` plus cleanup instead of surfacing the rename failure.
  - file: [src/infra/json-files.ts:9](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/infra/json-files.ts#L9)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **`devices/pending.json` writes use that helper:** Device pairing persistence writes `pendingPath` via `writeJsonAtomic(pendingPath, state.pendingById)`, so the reported `devices/pending.json` path goes through the guarded writer on current `main`.
  - file: [src/infra/device-pairing.ts:157](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/infra/device-pairing.ts#L157)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Regression test covers Windows `EPERM` path:** There is a focused test asserting that a Windows `EPERM` from `fs.rename()` falls back to copy-on-replace and still leaves the target file updated.
  - file: [src/infra/json-files.test.ts:75](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/infra/json-files.test.ts#L75)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)

## Close Comment

Closing from a Codex review: current `main` already implements this fix. The shared atomic writer now routes the replace step through `replaceFileWithWindowsFallback()` in `src/infra/json-files.ts:9`, and `writeTextAtomic()` uses it at `src/infra/json-files.ts:79`, so a Windows `EPERM` during rename no longer aborts the write and instead falls back to `copyFile`. The reported `devices/pending.json` path goes through that helper via `src/infra/device-pairing.ts:157`. There is also a focused regression test for the Windows `EPERM` path in `src/infra/json-files.test.ts:75`. I also verified the same implementation is present in release `v2026.4.22`. If you still see a different Windows locking failure on a current build, please open a fresh issue with the exact error code and reproduction details.
