## Summary

Keep open. Current `main` has targeted auto-reset paths for specific failure classes, but it does not implement a general circuit breaker that counts consecutive session aborts and resets the session after N aborts.

## Evidence

- **Abort state is a single boolean, not a counter:** User-triggered abort persistence sets `abortedLastRun = true` on the session entry. There is no per-session consecutive-abort counter or threshold stored here.
  - file: [src/auto-reply/reply/abort.ts:307](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/auto-reply/reply/abort.ts#L307)
  - command: `nl -ba src/auto-reply/reply/abort.ts | sed -n '300,330p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Next turn clears the abort flag immediately:** The next reply only prepends a caution hint for one aborted run, then clears `abortedLastRun` back to `false`. That behavior prevents tracking consecutive aborts across turns.
  - file: [src/auto-reply/reply/body.ts:23](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/auto-reply/reply/body.ts#L23)
  - command: `nl -ba src/auto-reply/reply/body.ts | sed -n '12,45p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Gateway abort tracking is per-run TTL bookkeeping:** Gateway chat abort handling records aborted run IDs in `chatAbortedRuns`, and maintenance later prunes that map by TTL. This is run-level cleanup state, not a per-session circuit breaker.
  - file: [src/gateway/chat-abort.ts:32](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/chat-abort.ts#L32)
  - command: `nl -ba src/gateway/chat-abort.ts | sed -n '28,95p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Existing auto-reset logic is narrow and failure-specific:** `main` auto-resets sessions for compaction overflow, role-ordering conflicts, and Gemini session corruption, via explicit reset helpers. That is materially narrower than the issue's requested generic 'N consecutive aborts' behavior.
  - file: [src/auto-reply/reply/agent-runner.ts:1183](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/auto-reply/reply/agent-runner.ts#L1183)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1180,1206p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Gateway docs expose abort/reset as manual controls:** The protocol docs list `sessions.abort` and `sessions.reset` as separate maintenance operations, with no documented automatic abort-threshold reset behavior.
  - file: [docs/gateway/protocol.md:347](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/gateway/protocol.md#L347)

## Risks / Open Questions

- Some existing targeted auto-recovery paths may partially mask the problem for specific providers or failure modes, but they do not satisfy the requested generic consecutive-abort circuit breaker.
- The issue body leaves threshold/config details open, so implementation design is still needed even though the gap on `main` is clear.
