## Summary

Current main already fixes the observable runaway dreaming-narrative problem by moving managed dreaming to an isolated cron agent turn and gating heartbeat handling on a pending managed cron event. PR #68534's file-based cooldown store is not the shipped path, and current docs now define dreaming as one scheduled full sweep rather than separate phase cooldowns.

## Best Possible Solution

Close PR #68534 as implemented on main. Keep the shipped #70737 scheduler-boundary fix, which prevents normal heartbeat polls from spawning dreaming narratives and aligns with the documented single-sweep `dreaming.frequency` model. If per-phase cadence or cooldown policy is desired later, open a focused product issue for explicit phase-specific config and tests instead of merging this stale PR.
