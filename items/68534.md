## Summary

Current main already solves the runaway dreaming-narrative heartbeat problem, but with a scheduler-boundary fix rather than PR #68534's per-phase cooldown store. Managed dreaming now runs as an isolated cron agent turn with no delivery, the heartbeat hook only accepts a live pending managed cron event, and docs/release notes describe the shipped single full-sweep dreaming model.

## Best Possible Solution

Close PR #68534 as implemented on main. Keep the shipped #70737 scheduler-boundary approach, which prevents normal heartbeat polls from spawning dreaming narratives and preserves the documented single `dreaming.frequency` full-sweep model. If maintainers later want explicit per-phase cadence or cooldown controls, open a focused product/API issue with phase-specific config and tests rather than merging this stale cooldown-store PR.

## Evidence

This removes the old main-session system-event shape that let normal heartbeat polls replay the dreaming trigger.
