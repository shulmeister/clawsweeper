## Summary

Keep open. The current `main` code still stages the systemd unit and then does a single `systemctl --user daemon-reload`, single `enable`, and single `restart` with no retry or unit-visibility check, so the reported migrated `root + systemd --user` race is not clearly fixed. The issue is also recent and includes a concrete workaround from the reporter, so `stale_insufficient_info` does not apply.

## Evidence

- **Current install path still has no retry after daemon-reload:** `activateSystemdService` on current `main` immediately runs `systemctl --user daemon-reload`, then `enable`, then `restart`, and throws on the first enable failure. There is no retry, backoff, second reload, or explicit check that the new unit became visible before enabling it.
