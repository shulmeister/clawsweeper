## Summary

Keep #70330 open. Current main still has the relevant product behavior: WebChat New Session and slash reset flow through generic chat/agent send paths, reset triggers still rotate the same session key to a new sessionId, and reset suffixes can be stripped so the resulting transcript starts with an ordinary-looking post-reset message. Related #23755 fixed label preservation only; the restart/reconnect silent-rotation/audit/discoverability problem remains unresolved.

## Best Possible Solution

Keep the issue open and fix it in the WebChat/gateway session boundary: move New Session/reset UI actions to an explicit RPC or explicit allowSessionControl path with trigger metadata, prevent generic chat.send replay from silently invoking reset, persist a visible/auditable session-rotation event with old/new session ids, and add regression coverage for reconnect plus post-reconnect normal send preserving agent:main:main unless an explicit reset was requested.
