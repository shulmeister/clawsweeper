## Summary

Keep this issue open. Current main still has the WebChat/session-continuity gap: the UI New Session action still routes through `/new` on generic `chat.send`, authorized reset text can still rotate the active session id and archive the prior transcript, and the UI/protocol still lacks first-class reset provenance or reset-archive recovery for the active WebChat row.

## Best Possible Solution

Keep this in OpenClaw core. Wire WebChat New Session to an explicit `sessions.create` or `sessions.reset` flow with confirmed UI provenance, require an explicit session-control flag for WebChat `/new` and `/reset` over generic `chat.send`, persist visible reset/rotation audit data with old/new session ids and trigger, clear queued reset/draft state across reconnect, and expose reset archives/checkpoints or a restore path in the Sessions/WebChat UI. Add regression coverage for gateway restart/reconnect followed by a normal visible send so `agent:main:main` does not silently rotate without provenance.
