## Summary

Keep open. Current `main` has `delivery.mode="none"` and Telegram-specific fallback now sends only the final assistant-visible text, but there is still no generic cron `deliveryPayloads`/`last` policy, and core-channel tests explicitly preserve multi-payload announce delivery for Slack, Discord, WhatsApp, and iMessage.

## Evidence

- **Cron delivery config still has no payload-selection field:** `CronDelivery` exposes only `mode`, `channel`, `to`, `threadId`, `accountId`, `bestEffort`, and `failureDestination`. There is no per-job `deliveryPayloads` or equivalent `full|last|none` selector on current main.

- **Current core-channel behavior still sends multiple payloads:** The regression path is still represented in tests: for Slack, Discord, WhatsApp, and iMessage, cron announce delivery preserves multiple text payloads even when `finalAssistantVisibleText` exists.
  - file: [src/cron/isolated-agent.direct-delivery-core-channels.test.ts:286](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cron/isolated-agent.direct-delivery-core-channels.test.ts#L286)

The default policy is false, and Telegram is the visible built-in channel opting into it.

## Risks / Open Questions

- There is a docs/code mismatch: user-facing docs and CLI wording describe generic final-text fallback, but current tests show multi-payload announce is still preserved for several channels.
- Telegram-specific workflows may no longer reproduce the report, but closing the issue would overstate the state of cross-channel support and the requested configurable policy.
