## Summary

Keep this issue open. Current main still exposes the ambiguous `running`/`done`/`failed`/`killed`/`timeout` session status contract without an additive `resumable`, `taskStatus`, or explicit agent-facing note. Related closed lifecycle work added timing/state handling, but it still maps lifecycle outcomes to the same status names and does not resolve the orchestrator guidance problem described here.

## Best Possible Solution

Keep the issue open and fix it as an agent-facing contract/API clarity problem. Add a backward-compatible `resumable` and/or distinct `lifecycleStatus` / `lastTurnStatus` field to `sessions.list` rows, document the semantics on the public session tools page, and update the default tool descriptions or system prompt so orchestrators prefer `sessions_send` for known sessions unless the session is deleted or invisible. Add focused tests for the row shape and prompt/tool-description guidance; defer any status rename to a versioned protocol migration.
