## Summary

Current `main` still shows the reported hook-context drift. `message_received` uses the canonical `conversationId`, but `inbound_claim` recomputes it through channel-specific normalization, and agent-phase hook contexts still omit `accountId` (and any conversation identifier), so there is not strong evidence to close #49746.

## Evidence

- **Canonical message context keeps the original conversationId:** `deriveInboundMessageHookContext` sets `conversationId` from `OriginatingTo ?? To ?? From`, and `toPluginMessageContext` forwards `canonical.conversationId` unchanged for message hooks.
  - file: [src/hooks/message-hook-mappers.ts:90](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/hooks/message-hook-mappers.ts#L90)

- **Inbound-claim context still rewrites conversationId:** `toPluginInboundClaimContext` does not use `canonical.conversationId`; it calls `resolveInboundConversation`, which strips prefixes or asks the channel plugin to remap the identifier before exposing it to `inbound_claim`.
