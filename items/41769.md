## Summary

The issue targets the legacy embedded macOS gateway runtime from AutoClaw v0.2.14. Current `main` no longer uses an embedded app-bundled gateway for macOS; it installs and runs the external `openclaw` CLI/runtime instead, so the reported `/Applications/AutoClaw.app/Contents/Resources/gateway/.../sharp` load path is no longer part of the current product.

## Evidence

- **Changelog shipped the architecture change:** `CHANGELOG.md` states that macOS now installs the global `openclaw` CLI via `install-cli.sh` and uses the external launchd/CLI path instead of the embedded gateway runtime.
  - file: [CHANGELOG.md:6528](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/CHANGELOG.md#L6528)
  - command: `nl -ba CHANGELOG.md | sed -n '6500,6545p'`
- **Current macOS app requires external CLI/runtime:** `GatewayEnvironment.check()` looks for `openclaw` on PATH or a local checkout entrypoint and reports `openclaw CLI not found in PATH; install the CLI.` when neither exists, which matches the post-embedded-runtime design.
  - file: [apps/macos/Sources/OpenClaw/GatewayEnvironment.swift:130](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/apps/macos/Sources/OpenClaw/GatewayEnvironment.swift#L130)
  - command: `nl -ba apps/macos/Sources/OpenClaw/GatewayEnvironment.swift | sed -n '119,170p'`
- **macOS app installs CLI into managed prefix:** `CLIInstaller` runs `install-cli.sh` with `--prefix ~/.openclaw`, confirming the app's supported runtime path is the separately installed CLI rather than an embedded gateway bundle.
  - file: [apps/macos/Sources/OpenClaw/CLIInstaller.swift:37](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/apps/macos/Sources/OpenClaw/CLIInstaller.swift#L37)
  - command: `nl -ba apps/macos/Sources/OpenClaw/CLIInstaller.swift | sed -n '37,77p'`
- **Current app packager does not stage an embedded gateway tree:** `scripts/package-mac-app.sh` copies app resources and signs the bundle, but the packaged resources section shown here has no embedded `gateway/openclaw/node_modules` staging step, consistent with the changelog and runtime resolver.
  - file: [scripts/package-mac-app.sh:257](https://github.com/openclaw/openclaw/blob/a53fea39050beba21ac5468d876c37323cfdd2e6/scripts/package-mac-app.sh#L257)
  - command: `nl -ba scripts/package-mac-app.sh | sed -n '257,326p'`

## Risks / Open Questions

- A separate `sharp` problem could still exist in the standalone CLI install path, but that would be a different bug than the embedded-app-bundle failure described in this issue.

## Close Comment

Closing from a Codex review against current `main`. This report is about the old embedded macOS app gateway at `/Applications/AutoClaw.app/Contents/Resources/gateway/...`, but that runtime path is no longer how the current app works. `CHANGELOG.md:6528` shows that in `2026.1.11` the macOS app switched to installing and using the external `openclaw` CLI via `install-cli.sh` instead of the embedded gateway runtime, and the current code enforces that flow in `apps/macos/Sources/OpenClaw/GatewayEnvironment.swift:130` and `apps/macos/Sources/OpenClaw/CLIInstaller.swift:37`. The current packager also no longer stages an embedded gateway tree into the app bundle (`scripts/package-mac-app.sh:257`). Because the failing surface from AutoClaw v0.2.14 is already gone on current `main`, I’m closing this as implemented on main. If you still hit a `sharp` load failure on a current OpenClaw release through the standalone CLI install path, please open a new issue with the exact `openclaw version`, install method, Node version, and current stack trace.
