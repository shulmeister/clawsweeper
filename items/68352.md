## Summary

Keep open. Current main still lacks the specific channel-id-to-plugin-id diagnostic requested in #68352, and maintainer discussion explicitly kept this issue open as a real UX/design gap distinct from already-fixed false-positive provenance/status warning bugs.

## Best Possible Solution

Keep #68352 open and either land/supersede PR #68389 with a diagnostic-only fix or implement an equivalent loader/config-validation warning that uses discovered manifest channel metadata to say: the allowlist entry is a channel id, the discovered owning plugin id is X, and plugins.allow must use plugin ids. Avoid changing allow/deny trust semantics unless a maintainer explicitly decides channel-id aliases are desired.
