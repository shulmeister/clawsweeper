## Summary

Current main still makes `/v1/models` agent-first: it returns `openclaw`, `openclaw/default`, and `openclaw/<agentId>` entries, and the HTTP router only treats `/v1/models` as that compatibility endpoint. The runtime provider model catalog does exist through the Gateway `models.list` RPC and CLI surfaces, but there is no OpenAI-compatible HTTP model-discovery endpoint returning provider/model IDs. The related reports in the provided context are Control UI/provider-picker or provider-specific model issues, not a canonical duplicate for this HTTP API request.

## Best Possible Solution

Keep this open for maintainer product/API design. The likely implementation path is an authenticated, read-scoped HTTP discovery surface backed by the existing `loadGatewayModelCatalog()` plus `buildAllowedModelSet()` behavior, while preserving the documented agent-first `/v1/models` contract unless maintainers intentionally choose a breaking or extension-field change. The implementation should also decide whether “available” means configured/allowed, discovered, or live-probed reachable.

## Risks / Open Questions

- Changing `/v1/models` itself could break clients that rely on the documented agent-target contract, so an additive endpoint or explicit extension fields may be safer than replacing the list outright.
- The issue asks for models that are actually reachable; the existing catalog/allowlist path can show configured or discovered entries, but live reachability semantics would need a deliberate probe/caching design.
