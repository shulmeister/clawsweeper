## Summary

Current main already implements busy-session follow-up queueing in the Control UI, flushes queued messages after terminal chat events, and the embedded runner serializes runs per session lane. This matches the issue’s requested behavior closely enough to close as already implemented.

## Close Comment

Closing from a Codex review: current `main` already implements the busy-session handling described here. In the Control UI, new follow-ups are queued instead of being sent immediately when a run is active (`ui/src/ui/app-chat.ts:386`), those queued items are resumed after `final`/`error`/`aborted` events (`ui/src/ui/app-gateway.ts:401`), and the queued/Steer UX is documented and tested (`docs/web/control-ui.md:174`, `ui/src/ui/chat/run-controls.test.ts:84`, `ui/src/ui/views/chat.ts:1137`). On the backend, embedded runs are also serialized per session lane (`src/agents/pi-embedded-runner/run.ts:249`). This behavior is present on current `main` and in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so there is no remaining maintainer action on this specific report. If you still hit dropped sends on a newer build, please open a fresh issue with the exact client surface, build version, and gateway logs around the affected `runId`.
