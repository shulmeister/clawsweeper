## Summary

Current `main` does not implement the Feishu per-topic sequential-lane behavior from PR #64920. The existing code still serializes all messages in a Feishu chat onto one queue key, while Feishu topic-session routing already distinguishes topic-scoped conversations by `root_id`/`thread_id`. The PR therefore addresses a live gap on `main`, and the review discussion surfaced real edge cases rather than obsoleting the work. This is not a conservative auto-close candidate.

## Evidence

- **Current main still uses one Feishu sequential key per chat:** `getFeishuSequentialKey()` builds `feishu:${accountId}:${chatId}`, only special-cases `/stop` and `/btw`, and has no `root_id`/`thread_id` or topic-scope handling on `main`. Different topics in one Feishu group therefore still collapse to the same queue key.
