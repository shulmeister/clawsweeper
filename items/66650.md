## Summary

Keep PR #66650 open. Current main still exposes `sessions_history.limit` without a 1000 maximum and forwards finite caller-provided limits directly to `chat.history`, while the gateway protocol still rejects `chat.history.limit > 1000` during validation. The PR remains a focused fix for the linked bug #66573; the related PR #66615 is an open alternative and does not supersede this one.

## Best Possible Solution

Keep the PR or its bugfix path alive. The best maintainer outcome is to clamp `sessions_history` outgoing `limit` to 1000, add `maximum: 1000` to its tool schema, and add the runtime-clamp regression test suggested in review so direct `execute` calls with `limit: 2000` are covered. PR #66615 should not be treated as canonical unless maintainers deliberately choose to change the gateway `chat.history` public cap instead.
