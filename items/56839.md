## Summary

Keep open. Current `main` still has a plausible cross-channel misrouting path for cron announce/failure delivery: when `delivery.channel` is omitted it resolves through `last`, and the explicit `delivery.to` value is parsed with that fallback channel's target parser. In the WhatsApp path, `telegram:<digits>` is still normalizable into a WhatsApp E.164 target instead of being rejected, so the reported Telegram-ID-to-WhatsApp-number leak is not high-confidence fixed on `main`.

## Evidence

- **Cron delivery still defaults missing channel to `last` and forwards explicit `to` into session target resolution:** `resolveDeliveryTarget()` sets `requestedChannel` to `"last"` when `jobPayload.channel` is absent, then passes `explicitTo` into `resolveSessionDeliveryTarget()` with `allowMismatchedLastTo` enabled for `last` routes.
  - file: [src/cron/isolated-agent/delivery-target.ts:81](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cron/isolated-agent/delivery-target.ts#L81)
  - command: `nl -ba src/cron/isolated-agent/delivery-target.ts | sed -n '70,180p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Explicit cron target is parsed by the active/fallback channel plugin, not by any embedded provider prefix:** `resolveSessionDeliveryTarget()` calls `parseExplicitTargetForLoadedChannel(provider, raw)` where `provider` is the resolved channel or fallback `lastChannel`. If `lastChannel` is WhatsApp, a raw target like `telegram:1234567890` is handed to the WhatsApp parser.
  - file: [src/infra/outbound/targets-session.ts:32](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/outbound/targets-session.ts#L32)

- **WhatsApp normalization still accepts arbitrary non-WhatsApp prefixed numeric input:** `normalizeWhatsAppTarget()` strips only `whatsapp:` explicitly, but otherwise falls through to `normalizeE164(candidate)` for any non-`@` input. That means a foreign provider string can still be treated as a phone-like target instead of being rejected.
  - file: [extensions/whatsapp/src/normalize-target.ts:57](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/whatsapp/src/normalize-target.ts#L57)
