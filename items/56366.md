## Summary

Keep open. Current `main` still loads `~/.agents/skills` unconditionally, the public config schema has no `skills.load` toggle or ignore-list for that root, and docs/tests explicitly describe and assert the personal-skill scan behavior. Existing agent allowlists and prompt-char limits mitigate visibility/token cost but do not disable the scan or prevent same-name conflicts at the source.

## Evidence

- **Personal `~/.agents/skills` scan is unconditional in current loader:** `buildWorkspaceSkillEntries` resolves the home directory, always constructs `~/.agents/skills`, loads it as `agents-skills-personal`, and merges it ahead of managed skills.

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current config schema exposes no disable/ignore knob for built-in personal skill roots:** `SkillsLoadConfig` only defines `extraDirs`, `watch`, and `watchDebounceMs`, and the zod schema matches that shape. There is no `scanAgentsSkills`, `ignoreDirs`, or equivalent built-in-root opt-out.
  - file: [src/config/types.skills.ts:10](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/types.skills.ts#L10)

## Risks / Open Questions

- OpenClaw does already provide partial mitigations via `agents.defaults.skills` / `agents.list[].skills` filtering and `skills.limits.maxSkillsPromptChars`, so any future triage should distinguish prompt-budget concerns from the separate built-in-root scanning concern.
- This review is source-based and read-only; I did not run a live mixed OpenClaw/Claude-Code reproduction against a real home directory, but current code, docs, and tests are aligned that the reported behavior still exists.
