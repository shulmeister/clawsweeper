## Summary

Keep open. Current `main` still has a real `health --json` summary bug for token-backed channels like Telegram: the health path builds a snapshot with only `accountId`/`enabled`/`configured`, then passes it into the plugin status summary builder, which defaults missing `tokenSource` to `"none"` and missing `running` to `false`. That matches the issue's reported Telegram misreport. Separately, some gateway-probe behavior from the report was already improved after `2026.3.13` (false-negative timeout handling in `2026.3.23`), and plain `openclaw status` is documented as shallow unless `--deep`, so the issue is mixed, but it is not fully implemented/fixed on `main`.

## Evidence

- **Health snapshot drops runtime/token fields before channel summary:** `getHealthSnapshot()` constructs `snapshot` with only `accountId`, `enabled`, `configured`, plus optional `probe`/`lastProbeAt`, then calls `plugin.status.buildChannelSummary(...)`. No `tokenSource`, `running`, or other runtime fields are populated on this path.

- **Token channel summaries default missing token source to none:** `buildTokenChannelStatusSummary()` sets `tokenSource: snapshot.tokenSource ?? "none"`. Its base summary also normalizes missing `running` to `false`, so a health snapshot that omits those fields will synthesize exactly the bad state reported in the issue.
