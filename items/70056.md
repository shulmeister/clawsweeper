## Summary

Keep #70056 open. Current main still has the reported stop-timeout behavior: the timeout branch sets `running: true`, returns before store cleanup, and the existing test still asserts that a second start is blocked. The PR is a plausible fix for #70024, but review comments identify a valid race in the proposed patch because the old tracked promise can still run its unguarded cleanup after a replacement start.

## Best Possible Solution

Revise #70056 rather than close it: keep the timeout fix direction, but add a task identity or generation guard around all old-task runtime mutations and restart scheduling before allowing a replacement start. Add regression coverage for both never-settling timeout and eventually-settling-after-restart cases, then validate the gateway changed lane.
