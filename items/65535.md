## Summary

Keep open. Current `main` still treats Telegram preview streaming as direct `editMessageText` updates/final-in-place edits, while plugin `message_sending` only runs on the normal reply-delivery path. The current hook types also have no `isStreamingEdit`/`isFinal` metadata, and preview-finalized Telegram deliveries emit only the internal `message:sent` hook, not the plugin `message_sent` hook. This matches the reported gap rather than an already-fixed or ClawHub-scoped request.

## Risks / Open Questions

- A future generalized delivery-hook change could supersede this specific Telegram issue, but the current `main` code still leaves the reported Telegram streaming gap in place.
- `reply_dispatch` can let a plugin take over full dispatch, but that is a heavier custom-delivery workaround, not evidence that the requested streaming interception seam already exists.
