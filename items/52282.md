## Summary

Keep open. Current `main` still routes Windows `.cmd`/`.bat` launchers through `shell: true` in `scripts/ui.js` without quoting the resolved command path, so a pnpm path like `D:\Program Files\...\pnpm.cmd` can still be split by the shell exactly as reported.

## Evidence

- **UI wrapper still shells unresolved path without quoting:** `createSpawnOptions` enables `shell: true` for Windows shell launchers, and both `run`/`runSync` pass `cmd` directly to `spawn`/`spawnSync`. There is no quoting or `cmd.exe /c` command-line construction in this file.
  - file: [scripts/ui.js:79](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/scripts/ui.js#L79)
  - command: `nl -ba scripts/ui.js | sed -n '79,126p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Repo already has a safe Windows cmd.exe quoting helper, but `scripts/ui.js` does not use it:** `buildCmdExeCommandLine` explicitly quotes commands containing spaces for `cmd.exe`, and other runners such as `scripts/pnpm-runner.mjs` use it for Windows. `scripts/ui.js` bypasses that helper.
  - file: [scripts/windows-cmd-helpers.mjs:7](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/scripts/windows-cmd-helpers.mjs#L7)
  - command: `nl -ba scripts/windows-cmd-helpers.mjs | sed -n '1,20p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Current regression test does not cover the reported failure mode:** `test/scripts/ui.test.ts` only checks extension detection and unsafe forwarded args. It never asserts quoting of a command path containing spaces or the actual Windows spawn invocation.
  - file: [test/scripts/ui.test.ts:4](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/test/scripts/ui.test.ts#L4)
  - command: `nl -ba test/scripts/ui.test.ts | sed -n '1,35p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **`ui:build` still goes through the affected wrapper on current main:** The root script remains `node scripts/ui.js build`, so Windows users still hit this code path for `pnpm ui:build`.
  - file: [package.json:1574](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/package.json#L1574)
  - command: `nl -ba package.json | sed -n '1570,1578p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Shell behavior is consistent with the report:** A direct Node check on this checkout's Node (`v24.14.1`) showed `spawnSync("echo foo", [], { shell: true })` executes `echo` with `foo` as a separate token rather than treating `echo foo` as one quoted command, which matches the reported `D:\Program` split behavior when the command path contains spaces.
  - command: `node -e 'const {spawnSync}=require("node:child_process"); const r=spawnSync("echo foo",[],{shell:true,encoding:"utf8"}); console.log(JSON.stringify({status:r.status,stdout:r.stdout,stderr:r.stderr}));'`
