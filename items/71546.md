## Summary

Keep #71546 open. The report is fresh, concrete, unprotected, and describes a current Discord ingest-lag surface that is not clearly implemented, obsolete, or fully duplicated by #38596. Current main has Discord gateway startup/reconnect hardening and listener/worker timeout knobs, but I found no shipped reconnect catch-up/backfill path keyed by last seen Discord message ID, and the existing code still depends on Carbon delivering MESSAGE_CREATE after RESUME.

## Best Possible Solution

Keep #71546 open and treat it as the narrow current Discord ingest-latency/catch-up tracker. The best fix path is in the Discord plugin/lifecycle: add targeted telemetry around Carbon RESUMED, MESSAGE_CREATE dispatch age, EventQueue age/drop metrics, and OpenClaw enqueue time; then either preserve/verify listener delivery across reconnects or add a guarded REST catch-up keyed by last seen message ID after successful reconnect/RESUMED. Coordinate with #38596 for shared reconnect-health work, but do not close this issue until one canonical item explicitly covers the delayed-ingest/backfill requirement.
