## Summary

Current main already has the stable/beta/dev release mechanism requested here: users can stay on stable, beta candidates are separate, dev tracks main, stable releases use CalVer tags and npm dist-tags, and maintainer release docs/workflows define beta-first validation, release branches, preflight, publish, and promotion. The exact monthly calendar/SLA is not promised, but the core user problem of having a tested stable lane separate from rapid development builds is implemented.

## Best Possible Solution

Keep the shipped stable/beta/dev release mechanism and release policy as the core answer here. If maintainers want a hard monthly calendar or support-SLA promise, that should be a narrower maintainer policy decision; app/gateway compatibility grace-period work should stay tracked separately in #65095.
