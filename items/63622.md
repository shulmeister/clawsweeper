## Summary

Current main already solves the observable WhatsApp user problem through the shared assistant-visible text sanitizer. Model reply text is sanitized before reply payload construction, and the sanitizer has broader coverage than this WhatsApp-local PR, including closed, dangling, mismatched, and Qwen-style tool-call XML blocks. The PR body is an unfilled template; the only substantive review comment supports the patch shape but notes a PR-local logging mismatch rather than a remaining product gap.

## Best Possible Solution

Close this PR as obsolete and keep the shared assistant-visible sanitizer as the canonical fix. If a future report shows a non-agent WhatsApp resolver or custom reply path can still inject raw internal XML directly into delivery, add a focused delivery-boundary sanitizer with a reproduction for that path rather than merging this stale WhatsApp-only patch.
