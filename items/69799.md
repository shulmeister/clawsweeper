## Summary

Keep this issue open. Current main has useful mitigations: app-owned TCC surfaces for some macOS node/PeekabooBridge work, an install-cli local-prefix Node path, and new service wrapper support. But it still does not ship or require an OpenClaw-owned bundled/dedicated Node runtime for the macOS Gateway/FDA path described here. The documented Gateway path remains external-CLI/Node-backed, and legacy iMessage still requires Full Disk Access/Automation for the process context running OpenClaw/imsg.

## Best Possible Solution

Keep this as core macOS security/packaging architecture work. The strongest path is to make the supported macOS Gateway/FDA runtime use a stable OpenClaw-owned identity by default, either a signed app-owned Gateway/runtime or a dedicated installed Node binary with documented migration/doctor/onboarding support, then coordinate it with #7227, the docs warning from #69561, and the stable-path lessons from #22179. Treat current install-cli local Node and --wrapper support as mitigations until maintainers explicitly define and wire them as the supported TCC boundary.
