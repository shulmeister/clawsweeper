## Summary

Keep open. Current main still lacks the per-reply Feishu streaming-card closed-state guard that this draft PR proposes. The linked bug #71977 is open and current main only suppresses exact duplicate final text after a streaming card closes; a later distinct final can still start another FeishuStreamingSession/card.

## Best Possible Solution

Keep the PR open and continue reviewing it as the focused Feishu fix for #71977. The best product/code path is to preserve one streaming card per reply after card close, suppress later distinct text finals that are already represented by the card, keep media delivery intact, and add regression coverage for the after-idle distinct-final sequence against the real dispatcher ordering.
