## Summary

Keep open. Current main still only suppresses exact duplicate Feishu final text after a streaming card closes; a distinct late final in card streaming mode can still start a new streaming card. This PR targets that remaining behavior from #71977 and adds focused regression coverage, although the async idle/error lifecycle concerns raised in review should be resolved before landing.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for the remaining Feishu card-mode bug. The desired fix belongs in extensions/feishu/src/reply-dispatcher.ts with regression coverage in extensions/feishu/src/reply-dispatcher.test.ts: suppress distinct late text finals only after a successful streaming-card close for the same reply, preserve media delivery, and guard the state against async idle/error closeout races from src/auto-reply/reply/reply-dispatcher.ts.
