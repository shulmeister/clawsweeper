## Summary

Keep open. Current `main` still hardcodes the gateway systemd unit during install/uninstall, while node-service install paths set `OPENCLAW_SYSTEMD_UNIT` to `openclaw-node`, so the PR addresses a live inconsistency rather than an already-fixed or obsolete report.

## Evidence

- **Current main still uses gateway-only helper in activate:** `activateSystemdService` resolves the unit name with `resolveGatewaySystemdServiceName(params.env.OPENCLAW_PROFILE)` instead of the override-aware `resolveSystemdServiceName(env)`.
  - file: [src/daemon/systemd.ts:541](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/daemon/systemd.ts#L541)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '538,596p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current main still uses gateway-only helper in uninstall:** `uninstallSystemdService` also resolves the unit name with `resolveGatewaySystemdServiceName(env.OPENCLAW_PROFILE)`, so disable/remove targets can diverge from the staged unit path when `OPENCLAW_SYSTEMD_UNIT` is set.
  - file: [src/daemon/systemd.ts:586](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/daemon/systemd.ts#L586)

  - file: [src/daemon/systemd.ts:52](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/daemon/systemd.ts#L52)
