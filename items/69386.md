## Summary

Keep #69386 open. Current main has partial improvements around abort handling and safer denied-exec followup text, but the central denial-cascade problem is not fully implemented: ordinary denied async exec completions still trigger an agent followup turn, there is no source-level evidence of an auto-stop/consecutive-denial threshold on main, and the exact reported Chinese stop phrases are not recognized in the current abort trigger set. The linked PR #69387 was closed unmerged, and the open related issue #69570 overlaps on async completion cascades but does not fully supersede this denial-specific report.

## Best Possible Solution

Keep this open or split it into a canonical remaining bug that covers denied async exec cascades specifically. The best fix likely belongs in the exec approval followup/session queue path: suppress or collapse denied completion followups for user-denied approvals, add a per-session or per-flow consecutive-denial guard that cancels/enters wait state after a small threshold, ensure stop/abort commands preempt pending exec followups and queued messages, and add regression coverage for Control UI/webchat including the exact reported stop phrases '停下来' and '暂停'. Coordinate with #69570 for the broader approved-async-completion cascade so the two fixes share the same queue-priority model.
