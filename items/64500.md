## Summary

Keep this issue open. Current main still treats the global breaker as a same-tool/same-signature no-progress breaker and ping-pong critical blocking still requires stable result hashes on both sides, so the reported variable-result memory_search <-> memory_get ping-pong gap remains plausible and unimplemented.

## Best Possible Solution

Keep this issue open. The maintainer fix should make tool-loop protection pair-aware: either allow sustained A/B alternation to escalate without identical per-side result hashes, add a pair-scoped threshold before the per-tool global breaker, or persist short-lived session-level blocks for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the offending pair.
