## Summary

#64500 should remain open. Current main still requires stable identical result hashes before ping-pong can escalate to a critical block, and the before-tool-call enforcement only blocks the current attempted tool. The martingarramon issue comment correctly identifies both the primary `noProgressEvidence` gate and the secondary unused `pairedToolName` enforcement gap.

## Best Possible Solution

Keep #64500 open and fix it in the agent loop-detection/enforcement path. A maintainer-quality fix should add pair-aware regression coverage for sustained `memory_search` <-> `memory_get` alternation with varying results, decide the intended critical threshold semantics for sustained ping-pong loops, and either make the ping-pong detector escalate without requiring identical result hashes after enough alternations or add pair-level circuit-breaker state so a critical/global breaker suppresses both tools in the detected pair. Update `docs/tools/loop-detection.md` and `docs/gateway/config-tools.md` if the threshold semantics change.
