## Summary

Keep #64500 open. Current main still has the reported product gap: the global circuit breaker is computed for the current tool/signature, ping-pong only becomes critical when both sides have stable identical outcomes, and enforcement blocks only the current critical call while merely logging the paired tool name.

## Best Possible Solution

Keep the issue open. A maintainer fix should make ping-pong protection pair-aware: either let sustained A/B alternation escalate without requiring identical result hashes, add a pair-scoped threshold before the per-tool global breaker, or store a short-lived session-level block for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get style alternation and for blocking both sides of the pair.
