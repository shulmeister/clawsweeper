## Summary

The request is a reasonable direction, but current main scope puts this kind of multi-instance orchestration outside OpenClaw core. The repo vision is plugin-first for optional capability and explicitly excludes heavy orchestration layers that duplicate existing agent/tool infrastructure; current core already provides lighter remote-node execution primitives instead.

## Close Comment

Thanks for the detailed write-up and the prototype updates. I reviewed this against current `main` in a Codex maintainer pass, and I’m closing it as a `clawhub`-type request rather than a core issue.

The main reason is scope: `VISION.md:55` says optional capability should usually ship as plugins, `VISION.md:69` says new skills should go to ClawHub first, and `VISION.md:106`-`107` explicitly calls out agent-hierarchy frameworks and heavy orchestration layers as things OpenClaw will not merge into core for now. Your proposed task ledger / artifact-handoff coordination layer fits that bucket.

Current `main` already has lighter primitives for remote specialist execution, like node-host routing in `docs/nodes/index.md:55`-`166` and remote skill/node eligibility in `src/infra/skills-remote.ts:319`-`346`. That makes an external plugin/orchestrator the better fit than expanding core into multi-instance orchestration.

Given the external prototype you linked, this looks like useful work, just not OpenClaw-core work under the current project direction. If you keep pushing it forward as a ClawHub/community plugin or adjacent orchestrator, that matches the repo’s current architecture much better.
