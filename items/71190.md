## Summary

Keep #71190 open. Current main still prepends the full orphan leaf text into the next prompt, and the duplicate guard only checks whether that entire orphan text is already present in the new prompt. There is no stripping of prior queued markers or abort hints, and existing tests cover only single-merge/dedupe behavior, not repeated post-abort snowballing.

## Best Possible Solution

Keep the issue open and fix `mergeOrphanedTrailingUserPrompt()` in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts` so it canonicalizes orphan text before merging: strip prior queued markers and abort-note prefixes or otherwise bound/dedupe merged orphan blocks, while keeping `removeLeaf` semantics that preserve real cross-channel orphan content. Add a regression test that simulates two or more consecutive repairs and asserts the prompt contains only one queued marker and only the latest intended orphan content.
