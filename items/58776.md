## Summary

Current main does not implement the requested additive lifecycle phases. The embedded lifecycle path still emits `start`, `end`, and `error`; the subagent registry still ignores any lifecycle phase other than `start`, `end`, or `error`; and repo-wide searches found no `first-progress` or `startup-failed` contract. Some adjacent fixes exist for prompt-error persistence, abort-wrapped prompt-error failover, and delayed timeout cleanup, but they do not solve the reported parent-visible startup-silence distinction.

## Best Possible Solution

Keep this issue open and implement a narrow core lifecycle enhancement in the embedded runner/subagent event path: emit a first-progress signal on first visible assistant text or qualifying tool activity, emit a typed startup-failed signal for pre-progress abort/timeout/auth/empty-output cases, and add registry plus tests proving parent-visible status no longer collapses startup abort into a generic timeout or running hang.
