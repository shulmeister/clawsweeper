## Summary

Keep #26370 open. Current main still lacks a trusted per-agent or per-session ownership boundary for cron list/update/remove/run/runs; recent isolated-cron fixes address run execution details, not scheduled-job visibility or mutation isolation.

## Best Possible Solution

Keep #26370 open. The best product/code path is to pass a trusted caller scope from createCronTool through the Gateway cron RPC boundary, enforce agentId/sessionKey ownership for list, update, remove, run, and runs in Gateway/service code, preserve an explicit admin/operator all-jobs path, safely handle existing global jobs/run logs, and add regression tests plus docs for multi-agent and multi-user cron isolation.
