## Summary

Current main still leaves cron jobs in a global scheduler surface without trusted per-agent/session ownership enforcement for list, update, remove, manual run, or run-history reads. The related reports in the supplied context cover adjacent isolated-cron execution/model/introspection bugs, but none supersedes the remaining ownership/isolation request in #26370.

## Best Possible Solution

Keep #26370 open. The best path is to pass a trusted caller scope from createCronTool through the Gateway cron RPC boundary, enforce agentId/sessionKey ownership for list/update/remove/run/runs in Gateway/service code, preserve an explicit admin/operator all-jobs path, safely migrate or backfill existing global jobs, and add regression tests plus docs for multi-agent and multi-user cron isolation.
