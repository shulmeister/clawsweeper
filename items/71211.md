## Summary

Keep open. Current main still returns foreground exec stdout/stderr-derived text directly to the tool result and keeps the raw aggregate in result details. Existing redaction utilities cover logs, approval display, and some UI/support paths, but I did not find a core pass that redacts generic exec output before it becomes model-visible tool-result content.

## Best Possible Solution

Keep the issue open for core security hardening. The best path is to redact exec stdout/stderr-derived text before model-visible tool results are emitted, preferably by applying the existing bounded `redactSensitiveText(..., { mode: "tools" })` style logic at the exec outcome/result boundary and ensuring `details.aggregated`, foreground failures, `process` poll/log, and node-host exec all receive consistent treatment. Add regression coverage for env-style assignments and known token formats printed by commands, plus a deliberate product decision on whether any explicit raw-output opt-in is allowed.

## Risks / Open Questions

- A naive regex-only fix may miss secret formats or over-redact legitimate diagnostics; it should use the existing bounded redaction helpers and add focused regression tests.
- Exec output appears in foreground, background process, node-host, persistence/details, and provider transport paths, so a partial fix at only one layer could leave another model-visible route open.
