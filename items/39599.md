## Summary

Keep #39599 open. Current main still accepts Feishu announce cron jobs without `delivery.to` through the protocol, gateway, cron service, and CLI creation path, and the missing target can still surface later during outbound delivery. The related PRs #39602, #39683, and #40031 are open candidate fixes, not shipped replacements.

## Best Possible Solution

Keep #39599 open and land one focused canonical fix. Add create/update validation at the gateway/service boundary for Feishu/Lark announce cron delivery when neither explicit `delivery.to` nor a deterministic resolved target is available, then add regression coverage for cron service, gateway behavior, and CLI fail-fast behavior if the CLI layer changes. Use #39602, #39683, and #40031 as reference material, but do not close this issue until a fix lands on main.
