## Summary

Keep #39599 open. Current main still accepts an isolated Feishu announce cron job without `delivery.to` at creation/update time, while target resolution can still fail later with the Feishu missing-target error. The two related fix PRs in the provided GitHub context, #39602 and #39683, are open/unmerged and should be treated as candidate fixes rather than shipped behavior.

## Best Possible Solution

Keep #39599 open and land a gateway-side validation fix in the cron creation/update path, with CLI/UI fail-fast validation as a usability layer. The fix should reject `delivery.mode === "announce"` with `delivery.channel === "feishu"` or `"lark"` when no explicit target or deterministic resolved target exists, and add focused regression tests in `src/cron/service.jobs.test.ts` plus CLI coverage if the CLI changes.
