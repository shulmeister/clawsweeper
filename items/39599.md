## Summary

Keep #39599 open. Current main at 7e13f3f5140168e31027b39a9855ade3fadf24d5 still does not reject Feishu/Lark announce cron jobs without `delivery.to` at creation or update time. The supplied related PRs #39602, #39683, and #40031 are open/unmerged candidate fixes, so the issue is not implemented or superseded on main.

## Best Possible Solution

Keep #39599 open and land a canonical fix, likely through one of the open related PRs. The fix should add gateway/service validation for cron creation and update that rejects Feishu/Lark announce delivery when neither explicit `delivery.to` nor a deterministic resolved target is available, and should add focused regression coverage in gateway cron validation, cron service jobs, and CLI fail-fast tests if the CLI layer is changed.
