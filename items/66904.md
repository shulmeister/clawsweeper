## Summary

Keep open. Current main only has partial WhatsApp reconnect handling through `messages.upsert` `append` events. It does not listen for Baileys `messaging-history.set`, does not expose the proposed reconnect-history config surface, and current tests intentionally suppress stale/offline history instead of making it available to agent context. Related prior history-limit work covers `historyLimit`/`dmHistoryLimit`, but not this reconnect history sync request.

## Best Possible Solution

Keep the issue open and implement this in the WhatsApp plugin runtime: add a typed `messaging-history.set` handler, gate it behind explicit reconnect-sync config, normalize/dedupe messages through the existing inbound path, and store eligible messages in the same DM/group history surfaces without unintentionally auto-replying to stale backlog. Update the WhatsApp config schema/types/docs and add regression tests for sync type filtering, stale backlog behavior, dedupe, and history-limit interaction.
