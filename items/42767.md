## Summary

Keep open. Current main still keys the TUI/chat "running" state off `chatRunId`, and that local state is only cleared by chat terminal events (`final`/`aborted`/`error`) or a reconnect reset. Gateway session lifecycle updates do mark the session `done` and broadcast `sessions.changed`, but the UI handler for `sessions.changed` only reloads sessions and does not reconcile `chatRunId`. That leaves the reported stuck-"running" state plausible on current `main` whenever a long-running run misses its terminal chat event even though backend/session status is already ready/done.
