## Summary

Keep open. The report is recent, concrete, unprotected, and not fixed on current main. The linked PR #71274 was closed unmerged after review because its proposed guard did not match the code path well enough and could regress valid Kimi tagged tool calls. Current main still has the Kimi tagged-tool wrapper behavior under discussion, and I did not find a shipped Kimi-specific regression test or implementation that prevents the reported repeated read/exec loop.

## Best Possible Solution

Keep #71273 open and ask for or add a focused reproduction: a captured Kimi anthropic-messages transcript showing the assistant/tool-result sequence that repeats read/exec with identical arguments. The eventual fix should target the verified failure path in extensions/kimi-coding or the provider replay/session loop boundary, and add a regression test that proves final post-tool-result answers end with stop while valid tagged tool-call turns still become toolUse.
