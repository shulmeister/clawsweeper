## Summary

Current `main` still leaves the full memory UUID only in tool `details`, while the user-visible/tool-result `content` either omits IDs entirely (`memory_recall`) or truncates them to 8 chars (`memory_forget` query mode). The main agent and MCP serialization paths forward only `content`, so an agent still cannot reliably obtain a usable full `memoryId` for deletion from these tool results.

## Evidence

- **`memory_recall` hides IDs in content:** The recall tool's returned text lists category/text/score only; the full UUID exists only in `details.memories[].id`.
  - file: [extensions/memory-lancedb/index.ts:372](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-lancedb/index.ts#L372)

  - file: [extensions/memory-lancedb/index.ts:499](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/memory-lancedb/index.ts#L499)
