## Summary

Keep PR #38170 open. Current main still does not implement the requested Telegram `download-file` action, and inbound Telegram media placeholders still do not embed `file_id` attributes. The PR is stale against the current `extensions/telegram` plugin layout and needs rework, but the underlying capability remains missing and useful.

## Best Possible Solution

Keep this PR or a refreshed follow-up open until the Telegram plugin implements the capability on current main: expose `download-file` from `extensions/telegram/src/channel-actions.ts`, handle it in `extensions/telegram/src/action-runtime.ts` through the existing proxy/apiRoot-aware Telegram transport, expose inbound media file identifiers in a parser-safe body or placeholder format, update audio-placeholder consumers, and add Telegram action/body/docs coverage.

## Risks / Open Questions

- The PR patch targets older paths such as `src/agents/tools/telegram-actions.ts` and `src/telegram/...`; current main uses `extensions/telegram/src`, so it needs a real rebase.

- If placeholders gain attributes, audio placeholder consumers that currently match exact bare `<media:audio>` forms need coordinated updates and tests.
- The existing PR review also flagged a deterministic filename collision risk from truncating `fileId` in the proposed download path.
