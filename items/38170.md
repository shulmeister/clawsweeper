## Summary

Keep PR #38170 open. Current main still lacks the requested Telegram `download-file` action and still does not expose Telegram media `file_id` values as parser-safe media-placeholder attributes. The PR is stale against the current `extensions/telegram` plugin layout and needs rework, but it addresses a real remaining Telegram media capability gap from #8843 and the prior `tg-media-resolve` workaround discussion.

## Best Possible Solution

Keep this PR, or a refreshed follow-up, open until the current Telegram plugin implements the capability in `extensions/telegram`: expose `download-file` from `channel-actions.ts`, handle it in `action-runtime.ts`, use the existing Telegram getFile/file-download transport with proxy/apiRoot/SSRF/local-root behavior, expose inbound media identifiers in a parser-safe placeholder or body format, and update the Telegram body/action tests and docs.

## Risks / Open Questions

- The submitted PR cannot merge as-is because it targets the pre-plugin Telegram file layout.
- The PR review comments identify real integration risks for a refresh: proxy/apiRoot-aware fetching, deterministic output-path collisions, and audio placeholder consumers that currently use exact `<media:audio>` matching.
- Current main has a partial `[file_id:...]` fallback for some failed-download caption paths, so the implementation should define one consistent parser-safe body/placeholder contract rather than adding a second ambiguous format.
