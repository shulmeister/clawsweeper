## Summary

Current main addresses this by converting descendant-settle follow-up into a fresh wake run for the yielded orchestrator session, instead of continuing inside the announce callback lifecycle. The new path is implemented and covered by regression tests, but it does not appear to be in the latest tagged release yet.

## Evidence

## Close Comment

Closing from a Codex review. Current `main` now handles this by re-waking the yielded orchestrator with a fresh run after descendant subagents settle, rather than leaving resumed work tied to the `announce:v1` callback lifecycle. The implementation is in `src/agents/subagent-announce.ts:157` and `src/agents/subagent-registry-lifecycle.ts:501`, and regression coverage in `src/agents/subagent-announce.format.e2e.test.ts:2603` asserts that the parent run is replaced with a new run id after child completion results arrive. That matches the bug reported here, so this is implemented on `main` at `9f8b4006307d8c840814610e0d664155178aab12`. I could not tie it to a released tag yet, so there is no shipped release number to cite.
