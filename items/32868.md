## Summary

Keep #32868 open. Current main still matches the reported root cause: same-channel block replies enqueue through ReplyDispatcher without awaiting actual channel delivery, while the tool-start flush only waits for the block pipeline callback boundary. The newer #71664 is duplicate evidence of the same remaining bug, and the linked fix PRs #61974, #63051, and #65761 were closed unmerged.

## Best Possible Solution

Keep this issue open and fix the core same-channel delivery contract. The likely path is to make ReplyDispatcher.sendBlockReply return false or a delivery Promise, await that in dispatch-from-config for same-channel block replies, update ACP/SDK types and mocks, and add a regression test proving text_end block delivery completes before tool_execution_start continues.
