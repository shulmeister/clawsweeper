## Summary

Keep #32868 open. Current main still has the same-channel block streaming delivery gap: the cross-channel path awaits delivery, but the same-channel path only enqueues through ReplyDispatcher, whose sendBlockReply contract still returns a boolean rather than a delivery promise. The reported behavior is supported by multi-channel issue comments and closed unmerged fix PRs, not by stale or missing reproduction data.

## Best Possible Solution

Keep this issue open and fix the core same-channel delivery contract. The likely maintainer path is to make ReplyDispatcher.sendBlockReply return a delivery promise or add an awaited block-delivery method, await it in dispatch-from-config.ts for same-channel block replies with abort/timeout behavior, update ACP/plugin SDK types and mocks, and add a regression test proving text_end block delivery completes before tool_execution_start continues. The WeCom text-parse workaround is useful plugin defense, but it should not replace the core ordering fix.

## Risks / Open Questions

- A fix should await same-channel delivery with abort and timeout handling so a hung channel send cannot block tool execution indefinitely.
- Changing sendBlockReply's return contract will touch core mocks, plugin/runtime SDK types, and channel tests; those should be updated together with a regression test for text_end delivery before tool_execution_start.
