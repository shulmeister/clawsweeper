## Summary

Keep #32868 open. Current main still matches the reported root cause: same-channel block replies are queued through ReplyDispatcher fire-and-forget, while tool-start flushing only waits for the block pipeline callback to resolve, not for actual channel delivery. The newer #71664 is duplicate evidence of the same remaining bug, and the prior fix PRs in the provided context were closed unmerged.

## Best Possible Solution

Keep this issue open and fix the core same-channel delivery contract. The likely path is to make ReplyDispatcher.sendBlockReply return false or a delivery Promise, await that in dispatch-from-config for same-channel block replies, update ACP/SDK types and mocks, and add a regression test proving text_end block delivery completes before tool_execution_start continues.
