## Summary

Keep #32868 open. Current main still has the same-channel block-streaming delivery gap: the block flush path can wait for the block-reply callback, but the same-channel callback only enqueues dispatcher delivery and returns before actual channel delivery completes. The WeCom community workaround noted in the discussion is useful defensive plugin behavior, but it does not replace the missing core delivery contract.

## Best Possible Solution

Keep #32868 open as the canonical core bug. The maintainer path is to make same-channel block delivery awaitable, wait for it before tool_execution_start continues when blockStreamingBreak is "text_end", update dispatcher/SDK/plugin mocks and ACP call sites consistently, and add a regression test proving actual channel delivery completes before tool execution proceeds.
