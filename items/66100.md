## Summary

Current main already resolves the reported Android bootstrap-token loop: new setup-code connects clear stale gateway/device auth, stored paired-device auth wins over bootstrap auth, invalid bootstrap auth pauses reconnect instead of retrying forever, and the reported private-LAN plain ws:// route no longer matches current Android behavior because manual/scanned cleartext is loopback-only as of 2026.4.23. The ACP note also does not appear to affect Android online/offline state; Android consumes gateway session defaults derived from agents.list, not acp.defaultAgent.

## Best Possible Solution

Close this issue as implemented on main. Keep the shipped Android auth-state handling and loopback-only cleartext policy; if maintainers want to reintroduce private-LAN cleartext pairing, that should be a new explicit product/security decision rather than reopening this stale bootstrap-token-loop report.
