## Summary

Keep open. Current `main` clearly includes the loopback-CDP/SSRF fixes that match the managed-browser half of this report, but the `user` / Chrome MCP attach half is only better diagnosed, not clearly fixed. Since the issue bundles both paths, the evidence is not strong enough for an `implemented_on_main` close.

## Evidence

- **Managed loopback CDP now bypasses strict browser SSRF for its own control plane:** `fetchCdpChecked()` special-cases loopback CDP hosts and allowlists the parsed hostname instead of applying the default strict private-network block. That directly addresses the reported `Chrome CDP websocket for profile "openclaw" is not reachable after start` failure mode when Chrome was already listening on localhost.
