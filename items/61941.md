## Summary

Current main supports the Opik-style typed hook path: plugins register `llm_input`, `llm_output`, and `agent_end` through `api.on`, the CLI/Gateway agent path dispatches those hooks, and non-bundled conversation hooks now have a documented, schema-valid `hooks.allowConversationAccess` opt-in. The missing opt-in explains the reproduction as written; with it enabled, the core hook path is implemented on main.

## Evidence

- **Conversation hook gate and registration:** The loader accepts typed hooks but intentionally blocks `llm_input`, `llm_output`, and `agent_end` for non-bundled plugins unless `plugins.entries.<id>.hooks.allowConversationAccess=true`; accepted hooks are pushed into `registry.typedHooks`.
