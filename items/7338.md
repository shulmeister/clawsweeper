## Summary

Keep #7338 open. Current main still lacks an opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary external HTTP requests from built-in web_fetch, skills, or tools. Existing header support is limited to basic web_fetch headers, model-provider request headers, and provider-scoped attribution/hooks, which do not solve the issue's requested signed/static agent attestation for external services.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether attestation belongs in core or a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported external HTTP paths consistently, document privacy defaults, and add tests proving allowed-domain injection plus non-leakage to other domains.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, but that does not provide a shared OpenClaw trust root, opt-in domain policy, or consistent external HTTP middleware for built-in web_fetch, skills, and arbitrary tools.
- Any implementation needs explicit privacy/security design around tracking, signing keys, expiry, audience/domain scoping, and which outgoing request paths participate.
