## Summary

Keep #7338 open. Current `main` still has provider-scoped attribution/custom-header support, but no opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary external HTTP requests, `web_fetch`, or skill-driven requests. The existing maintainer/Codex discussion also explicitly kept this issue open and treated #11137 as adjacent rather than a replacement.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether attestation belongs in core or a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other domains. Treat #11137/provider-header work as adjacent, not a replacement.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, but that does not solve the reported cross-cutting OpenClaw trust-root problem for built-in web_fetch, skills, and arbitrary tool HTTP clients.
- Any eventual implementation needs explicit security/privacy design for signing keys, verifier trust, replay expiry, audience/domain scoping, opt-in defaults, and cross-service tracking risk.
