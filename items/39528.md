## Summary

Keep open. Current main has useful hardening since the 2026.3.2 report, including owner-only gateway tools, protected config-path guards, config-flow prompt guidance, and docs. It still does not fully satisfy the reported expectation that the agent cannot modify config or restart itself without explicit user consent: the owner-facing model tool can still schedule a gateway restart when `commands.restart` is enabled, and allowed config paths can still be patched/applied by the tool without a separate consent token or confirmation state. The context-loss part also remains unproven as fixed by source/tests/docs.

## Best Possible Solution

Keep tracking this as a product/security-behavior bug. The best fix is a hard, test-backed explicit-consent path for agent-initiated `gateway restart`, `config.patch`, `config.apply`, and `update.run` actions, rather than prompt-only guidance. That likely means adding a confirmation/approval token or recent user-intent check in the gateway tool/runtime, preserving the existing protected-path allowlist, and adding regression coverage for long sessions/compaction where SOUL/USER/MEMORY and user red-line instructions remain available or are safely re-read before config/restart actions.
