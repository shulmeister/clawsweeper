## Summary

Keep #11487 open. Current main at 82262387651ab270b565f71533ce4937c9c00dc6 still has no optional human-readable `name`, `label`, or `description` field on Telegram or representative channel group/channel config entries; strict schemas, TypeScript types, generated metadata, and public docs remain behavior-only. The existing issue discussion also already kept this open, and the diff since that review did not touch the relevant config contract files.

## Best Possible Solution

Keep the issue open. The best implementation is a core config-contract change: choose one canonical optional string field, add it consistently to channel group/channel entry TypeScript types and strict Zod schemas, preserve it through config load/write while ignoring it at runtime, regenerate bundled channel config metadata, update public group/channel docs, and add validation tests proving the field is accepted without changing group behavior.
