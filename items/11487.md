## Summary

Keep #11487 open. Current main still does not accept or document a canonical optional human-readable `name`, `label`, or `description` field for channel group config entries. Representative core and bundled plugin group schemas remain strict behavior-only objects, generated config metadata still forbids extra group-entry fields, docs only list behavior fields, and tests still enforce unknown group-entry rejection.

## Best Possible Solution

Keep the issue open. The best implementation is to choose one canonical optional string metadata field, add it consistently to core and bundled channel group-entry types and Zod schemas, preserve it while ignoring it functionally at runtime, regenerate config metadata/schema, update public group/channel docs, and add focused validation tests proving the field is accepted without affecting group behavior.
