## Summary

Current `main` does not have a Web UI-side cached version path. The version shown in the sidebar and update banner is populated live from the running gateway process, so the reported mismatch is most consistent with an older OpenClaw process or install path still serving `127.0.0.1:18789`, not a UI regression.

## Close Comment

Closing after a Codex review because I could not substantiate a current Web UI version-display bug on `main`. The sidebar version is rendered from `hello.server.version` in `ui/src/ui/app-render.ts:1364`, and both the websocket hello (`src/gateway/server/ws-connection/message-handler.ts:1278`) and the bootstrap JSON (`src/gateway/control-ui.ts:750`) source that value from `resolveRuntimeServiceVersion(process.env)`. The update banner's `currentVersion` also comes from the running runtime version in `src/infra/update-startup.ts:168`.

That means a page showing `2026.2.19-2` while you expected `2026.3.8` points much more strongly to an older OpenClaw process or install path still serving `127.0.0.1:18789` than to cached Web UI assets. If this still reproduces on a current release with only one OpenClaw install path active, please open a new issue with the active listener/process details (`command -v openclaw`, `openclaw --version`, `lsof -nP -iTCP:18789 -sTCP:LISTEN`).
