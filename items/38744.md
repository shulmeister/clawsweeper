## Summary

Keep PR #38744 open. Current main still uses Apple SFSpeechRecognizer for native iOS Talk Mode continuous and push-to-talk capture, and it does not include the PR's iOS ElevenLabs realtime WebSocket STT client, WAV encoder, batch fallback, or STT language/model settings. Main does have ElevenLabs Scribe v2 Realtime as a gateway/plugin realtime transcription provider, but docs and code scope that to Voice Call streaming STT, plus browser Talk uses a separate WebRTC realtime-voice session surface. The PR discussion also contains unresolved reliability review findings, so it is not safe to close as implemented or superseded.

## Best Possible Solution

Keep the PR open for maintainer/product judgment. If the feature proceeds, rebase it onto current main and decide whether native iOS Talk Mode should get an ElevenLabs realtime STT client directly or consume a new gateway/native realtime transcription seam. Before merge, address WebSocket readiness, committed-transcript waiting for PTT, Apple Speech fallback permissions/recovery, iOS settings persistence, docs, and focused tests.
