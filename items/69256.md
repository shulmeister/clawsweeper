## Summary

Keep #69256 open. Current main still has the premature direct-cleanup behavior the PR targets, but the PR is not safely obsolete or ready to close: the proposed guard is useful, yet reviewer comments identify a real follow-up requirement because an early return from direct cleanup has no evident later retry once descendants drain.

## Best Possible Solution

Keep the PR open and ask for an updated fix: make direct deleteAfterRun cleanup subagent-aware without permanently skipping cleanup, likely by deferring or re-attempting session deletion after descendant runs drain, and add a regression test in src/cron/isolated-agent/delivery-dispatch.double-announce.test.ts for structured/threaded direct delivery with active descendants.
