## Summary

Keep open. Current `main` does not implement this Mattermost slash-command hardening: registration still proceeds with derived non-HTTPS callback URLs, and callback auth still accepts any registered token without binding it to a specific slash trigger. The PR is addressing an unfixed security issue rather than something already landed or obsolete.

## Evidence

- **Current callback registration only warns, does not fail closed:** `registerMattermostMonitorSlashCommands` derives `slashCallbackUrl`, calls `warnOnSuspiciousCallbackUrl(...)`, and then immediately registers commands with that URL. There is no HTTPS enforcement or early return before registration.

  - command: `nl -ba extensions/mattermost/src/mattermost/monitor-slash.ts | sed -n '156,210p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Current HTTP handler validates only token membership:** `createSlashCommandHttpHandler` checks whether `payload.token` is present in `commandTokens`, then proceeds to trust `payload.command` with no token-to-trigger binding. That leaves the cross-trigger replay scenario described in the PR body unaddressed on `main`.
