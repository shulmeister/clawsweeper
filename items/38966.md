## Summary

Keep #38966 open. Current main still has the gateway-side sessions.changed notification path, but the TUI does not subscribe to session lifecycle events and does not dispatch any sessions.changed/session reset handler that would reload history, clear token/run state, or show a reset notice. The linked PR #40472 remains the active proposed fix in the provided GitHub context, but it is not merged into the checked-out main SHA.

## Best Possible Solution

Keep #38966 open until the missing TUI behavior lands. The likely product/code path is to subscribe the TUI gateway client to session lifecycle changes, dispatch sessions.changed reset/new events for the current session, reload history/session info, clear stale active run/tool stream state, and cover wrong-session and stale-run cases in src/tui/tui-event-handlers.test.ts. PR #40472 appears to be the active candidate fix and should remain the canonical implementation path unless maintainers choose a different event contract.
