## Summary

Current main already prevents the reported Telegram-visible `NO_REPLY`/`NO` leak through the existing delivery pipeline: message-tool duplicate suppression prevents the mirrored tool text from being resent, Telegram delivery normalizes exact silent payloads before transport, and current tests cover Telegram direct/group silent handling plus streamed control-token lead-fragment suppression. The PR's specific simplification is superseded and should not land as-is because it would preserve raw `NO_REPLY` into assistant stream events, matching the review concern on the PR.

## Best Possible Solution

Close this PR and keep the current main approach: preserve message-tool transcript mirroring, suppress duplicate block replies after successful message-tool sends, and normalize silent payloads at the outbound/Telegram delivery boundary. Any remaining gap should be filed as a narrow follow-up against the specific stream or delivery surface, not by landing this PR's raw-token-preserving helper change.
