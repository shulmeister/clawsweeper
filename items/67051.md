## Summary

Current main already resolves the reported WebUI `skills.update` failure. The Skills UI sends `skills.update`, the gateway handler patches only `skills.entries.<skillKey>`, and persistence now goes through the shared config writer. That writer reads a snapshot after legacy `tools.web.search.*` provider-owned config has been migrated into `plugins.entries.<plugin>.config.webSearch`, projects the skill-only mutation onto that migrated source, and validates the migrated persist candidate rather than stale legacy provider config. The related same-surface issue #69554 is closed, and the v2026.4.24 release tag contains the relevant config read/write path.

## Best Possible Solution

Close as implemented and keep the shared config read/write migration path as the fix, rather than adding a WebUI-only workaround. If this resurfaces on a release newer than v2026.4.24, it should be reopened as a fresh regression with sanitized config shape and gateway logs from `skills.update`.
