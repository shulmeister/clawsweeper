## Summary

Keep open. Current `main` still lacks the Control UI/TUI stripping changes this PR proposes, so the request is not implemented on main, and the PR discussion also surfaced non-trivial follow-up risks that need maintainer judgment rather than auto-close cleanup.

## Evidence

- **Grouped chat renderer still bypasses the shared text-extraction contract:** On `main`, `renderGroupedMessage` rebuilds text by joining `normalizeMessage(message).content` directly instead of using `extractText`/`extractTextCached`, which is exactly the leak path described in the PR for assistant/user scaffolding and metadata.
  - file: [ui/src/ui/chat/grouped-render.ts:1379](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/chat/grouped-render.ts#L1379)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '1337,1445p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Shared extractor already has the intended stripping contract, but it is not reused here:** `extractText()` on `main` routes assistant text through `extractSharedAssistantVisibleText()` and then `processMessageText()`, but `processMessageText` remains private and the grouped renderer does not call this path.
  - file: [ui/src/ui/chat/message-extract.ts:10](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/chat/message-extract.ts#L10)
  - command: `nl -ba ui/src/ui/chat/message-extract.ts | sed -n '1,60p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Inbound metadata stripper on main does not handle per-line system-event/internal-instruction leaks:** `SENTINEL_FAST_RE` only covers structured JSON metadata sentinels plus the untrusted-context header, and `stripInboundMetadata()` only strips those block forms. There is no `System:`/`System (untrusted):` or `Current time:`/async-command line handling on `main`.
  - file: [src/auto-reply/reply/strip-inbound-meta.ts:36](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/auto-reply/reply/strip-inbound-meta.ts#L36)
  - command: `nl -ba src/auto-reply/reply/strip-inbound-meta.ts | sed -n '1,320p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **TUI still uses the narrower leading-only stripper:** `stripLeadingInboundMetadata()` on `main` only removes leading structured metadata blocks, and `src/tui/tui-formatters.ts` still applies that function to user/command turns. That matches the PR's follow-up claim that TUI history can still leak these lines.
  - file: [src/tui/tui-formatters.ts:353](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/tui/tui-formatters.ts#L353)
