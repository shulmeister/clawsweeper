## Summary

Keep this issue open. Current main still exposes the generic hook metadata escape hatch, and Telegram already derives some reply/forward context for agent prompts, but the received-hook path still does not pass Telegram forward, via-bot, or reply provenance into `MessageReceivedHookEvent.context.metadata`. The prior Codex review comment remains accurate against current main.

## Best Possible Solution

Keep this issue open and implement a Telegram-owned additive fix. Add a provenance metadata path to the canonical received-hook context, map normal dispatch and silent-ingest emits to `metadata.forward`, `metadata.viaBot`, and bounded `metadata.replyTo`, reuse existing Telegram reply/forward visibility logic where appropriate, and add focused mapper plus Telegram emit-path tests.
