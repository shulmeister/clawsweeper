## Summary

Keep open. Current main still lacks a scoped private-network opt-in for cron webhook egress: cron webhook sends all go through the SSRF-guarded fetch helper with no caller policy, while config, protocol schemas, CLI/UI surfaces, docs, and tests expose only URL/token/delivery controls. This is an actionable bundled cron/security design issue, not a ClawHub integration request.

## Best Possible Solution

Keep the issue open until maintainers either implement a scoped cron webhook private-network opt-in or explicitly decide cron webhook destinations must always block intranet targets. A likely fix belongs in the cron config and/or per-job delivery contract, passes an explicit SSRF policy into postCronWebhook for finished, failure-alert, and failureDestination webhook sends, preserves secure defaults, and updates strict schema, generated config help/docs, Control UI/CLI/API surfaces, and regression coverage.
