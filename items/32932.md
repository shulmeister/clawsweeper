## Summary

Keep open. Current main still has no scoped cron webhook private-network opt-in: cron completion webhooks, failure alerts, and failure destinations all post through the strict SSRF-guarded fetch path without passing a policy. The request targets an existing core cron delivery surface after GHSA SSRF hardening, so it is not a conservative ClawHub close or duplicate of the GitHub webhook ingress/plugin item in the related context.

## Best Possible Solution

Keep this issue open until OpenClaw either implements a scoped cron webhook private-network opt-in or a maintainer explicitly decides cron webhook destinations must continue to block intranet targets. A likely implementation belongs in the cron config/job contract, should pass an explicit SSRF policy into postCronWebhook for completion, failure alert, and failureDestination webhooks, and should update strict schema, generated config help/docs, security docs, and regression tests for both default-blocked and opted-in private-network behavior.
