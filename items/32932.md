## Summary

Keep open. Current main still blocks private/LAN cron webhook destinations with no cron-level opt-in, and the existing tests explicitly assert that behavior. This is a concrete recent feature request on an existing core cron delivery path, not a ClawHub-only cleanup item.

## Best Possible Solution

Keep tracking this issue until OpenClaw either implements a scoped cron webhook private-network opt-in or a maintainer explicitly decides the product should continue blocking intranet webhook destinations. A likely implementation would add a clearly named cron config field, thread it through postCronWebhook for completion, failure alert, and failure destination webhooks, and update strict schema, generated config docs/help, security docs, and tests for both default-blocked and opted-in behavior.
