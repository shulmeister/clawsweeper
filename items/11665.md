## Summary

Keep open. Current `main` still forces `/hooks/agent` runs onto `sessionTarget: "isolated"`, which in turn passes `forceNew: true` into `resolveCronSession()`. That bypasses the fresh-session reuse path and still creates a new session ID for repeated hook calls with the same `sessionKey`, so the issue is not implemented on current `main`.

## Evidence

- **Hook requests still create isolated cron jobs:** `dispatchAgentHook()` hard-codes `sessionTarget: "isolated"` for hook agent runs, even when a stable `sessionKey` is supplied.
  - file: [src/gateway/server/hooks.ts:64](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/gateway/server/hooks.ts#L64)
  - command: `nl -ba src/gateway/server/hooks.ts | sed -n '40,110p'`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Isolated hook jobs force a new cron session:** `runCronIsolatedAgentTurn()` calls `resolveCronSession()` with `forceNew: input.job.sessionTarget === "isolated"`. For hook jobs from `dispatchAgentHook()`, this evaluates to `true`.
  - file: [src/cron/isolated-agent/run.ts:486](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/cron/isolated-agent/run.ts#L486)
