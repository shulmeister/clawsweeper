## Summary

Keep #11665 open. Current main still routes direct `/hooks/agent` and mapped agent hooks through an isolated cron-style turn, so a stable hook `sessionKey` cannot reuse the existing transcript. The lower-level session resolver can reuse only when `forceNew` is false, but hook dispatch hard-codes `sessionTarget: "isolated"`, and the runner turns that into `forceNew: true`. The hook payload, mapping, transform result, config type, and strict schema still have `sessionKey` but no `sessionMode`, `reuseSession`, or equivalent opt-in. The provided GitHub context also shows prior fix attempts #15084, #27123, and #27193 closed unmerged, plus a recent maintainer/Codex review that kept this exact issue open.

## Best Possible Solution

Keep the issue open and implement a backward-compatible persistent hook-session option in core, likely `sessionMode: "isolated" | "persistent"` or an equivalent reuse flag. Thread it through direct `/hooks/agent` payloads, mapped hook actions, transform results, config types/schema/help, docs, dispatch, and regression tests, then pass `forceNew: false` only for the persistent hook path while preserving the current isolated default.
