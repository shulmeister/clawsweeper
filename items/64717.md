## Summary

Keep this PR open. Current main still hardcodes 4000 for Telegram send-time HTML/plain-text chunk planning, while this PR targets that exact remaining gap by using the configured Telegram text chunk limit with Telegram's 4096 hard cap. The bot review comment is consistent with the local code read: this is a narrow, plausible bug fix, not obsolete or already implemented.

## Best Possible Solution

Review and land this PR, or an equivalent Telegram-owned patch, after rebasing onto current main. The fix should compute the send-time text chunk limit with `resolveTextChunkLimit(cfg, "telegram", account.accountId, { fallbackLimit: 4000 })`, clamp it to Telegram's 4096 API maximum, use it for all HTML/plain-text branches in `buildChunkedTextPlan`, and keep focused regression coverage for both the 4096 cap case and a lower configured limit.
