## Summary

Strict reply-format enforcement is better handled as a plugin/ClawHub package, not OpenClaw core. Current main already exposes plugin hooks that can rewrite outgoing message content or short-circuit replies, and the project vision explicitly keeps optional capability in plugins.

## Risks / Open Questions

- A plugin-based formatter still needs to be authored and distributed; there is no turnkey built-in formatter package cited here.
- If maintainers later want a first-party UX for selecting formatter plugins, that would be separate product work.

## Close Comment

Closing this as ClawHub/plugin scope after a Codex review. Current main already exposes the seams needed for strict reply-format enforcement without adding new core behavior: the project vision keeps optional capability in plugins (VISION.md:54-59), plugin hooks can claim a turn before the LLM (`before_agent_reply`) and can rewrite outbound message content via `message_sending` (docs/concepts/agent-loop.md:91-113, src/plugins/hook-message.types.ts:42-53, src/plugins/hooks.ts:733-755), and outbound delivery actually runs that hook before sending each payload (src/infra/outbound/deliver.ts:850-866). Also, imported Claude `outputStyles` are explicitly detect-only today, not something core executes (docs/plugins/bundles.md:196-201). Because of that, a reply-format enforcer fits better as a plugin distributed through the community/ClawHub path than as new OpenClaw core scope.
