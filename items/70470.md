## Summary

Keep #70470 open. The reported WhatsApp behavior is recent, concrete, and still plausibly actionable on current main: accepted inbound WhatsApp messages still await `sock.readMessages()` before enrichment and enqueueing. Current main catches rejected read-receipt calls, and blocked/unauthorized messages are already filtered before read receipts, so the triage comment’s exact `shouldMarkRead` explanation is only partly supported. However, there is no regression coverage proving read-receipt failure or delay cannot affect processing, and the accepted-message path still has read receipt delivery ahead of the agent/calendar work.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin by decoupling read-receipt delivery from inbound message processing: either move the read-receipt side effect after enqueueing or make it bounded/fire-and-forget with logging, preserve `sendReadReceipts` and self-chat behavior, consider using `access.shouldMarkRead` for clarity, and add a regression test proving `sock.readMessages` rejection or delay cannot prevent `onMessage` and downstream calendar/tool work.
