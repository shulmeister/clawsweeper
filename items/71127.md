## Summary

Keep #71127 open. Current main still implements stuck-session diagnostics as warn/event telemetry only; there is no config key, heartbeat callback, or runtime recovery hook that aborts or drains a processing session after the stuck threshold. The related #71455 cross-reference is a specific DeepSeek reasoning-history failure and does not supersede this broader gateway self-recovery request.

## Best Possible Solution

Keep the issue open and implement a runtime-owned stuck-session recovery path: add a documented/configured abort threshold or policy, have the diagnostic heartbeat or a sibling watchdog call an injected recovery hook, use existing session abort and queue cleanup primitives, emit explicit recovery telemetry, and add regression coverage for a processing session that exceeds the threshold and then accepts subsequent messages. Also keep #71455 separate unless a maintainer chooses to track provider-specific poisoning under a narrower follow-up.

## Risks / Open Questions

- The report has no deterministic local reproduction yet, so a fix should include either a synthetic stuck-run harness or narrowly scoped integration coverage proving the watchdog can abort and release a session without corrupting normal long-running work.
- Diagnostics currently lives in logging/telemetry code; wiring it directly to mutation would need careful ownership boundaries, likely through an injected runtime recovery hook rather than making the logger import gateway/session internals.
