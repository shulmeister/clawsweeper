## Summary

Current main partially addresses this report: Google Chat app-url auth uses the add-on principal binding, logs actionable appPrincipal warnings, and accepts Workspace Add-on MESSAGE payloads with body-carried systemIdToken. The issue should stay open because the requested compatibility is not complete: docs/setup still do not document or prompt for appPrincipal, and the native Add-ons wrapper parser only unwraps chat.messagePayload, while the issue/comment workaround explicitly covered other Add-ons payload variants too.

## Best Possible Solution

Keep this open and finish the Google Chat Add-ons compatibility work in the Google Chat plugin: document app-url plus numeric appPrincipal discovery in the public Google Chat docs, add setup/status guidance or prompts for appPrincipal, and broaden Add-ons wrapper normalization to all relevant Chat payload union variants or intentionally 200-ignore unsupported non-message add-on events with tests. Preserve the current strict principal binding unless a maintainer explicitly accepts auto-discovery/trust changes.
