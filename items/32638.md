## Summary

Keep #32638 open. Current main still does not correctly map provider/model-native reasoning values: Groq handling is hardcoded to one model and maps to `default`, but the resolver only accepts OpenAI-style efforts and can fall back to `low`; LM Studio discovery still collapses `allowed_options: ["off", "on"]` to a boolean without preserving the accepted request vocabulary. The failover-chain concern from the discussion is partly addressed, but the invalid reasoning-parameter problem remains.

## Best Possible Solution

Keep this open and fix reasoning compatibility at the provider/model contract level. A solid maintainer path would preserve discovered native reasoning vocabularies, add typed compat for request-value maps, make Groq `none/default` handling apply to all affected Groq reasoning models, wire LM Studio/Qwen `on/off` discovery into request shaping, and add regression tests for both Groq and LM Studio OpenAI-compatible completions.
