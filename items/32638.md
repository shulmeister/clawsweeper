## Summary

Keep #32638 open. Current main still cannot represent provider-native reasoning values such as Groq `default` or LM Studio `on`/`off`: Groq handling is limited to one exact model id, mapped native values are filtered back through the OpenAI effort enum, and LM Studio discovery still collapses `allowed_options` to a boolean.

## Best Possible Solution

Keep the issue open and fix reasoning compatibility at the provider/model contract level: preserve provider-native request values outside the OpenAI effort enum, apply Groq `none`/`default` handling across affected Groq reasoning models, retain LM Studio `allowed_options` such as `on`/`off` during discovery, expose any needed compat/config surface intentionally, and add request-building tests for Groq and LM Studio OpenAI-compatible completions.
