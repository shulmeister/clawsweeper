## Summary

Keep open. Current main still has only a narrow Groq `qwen/qwen3-32b` reasoning-effort mapping, while the OpenAI-compatible completions path can still send generic `low`/`medium`/`high` values to other provider/model combinations. The April LM Studio/Qwen `on`/`off` reports are still supported by current source and tests: LM Studio discovery reads `allowed_options` only to compute a boolean and drops the native reasoning vocabulary.

## Best Possible Solution

Keep this issue open and implement a provider/model-native reasoning compatibility path. A strong fix would preserve discovered vocabularies such as LM Studio `on/off`, expose or type the compat metadata needed for provider-specific reasoning maps, broaden Groq handling beyond one hardcoded model id if Groq still requires `none/default`, and add regression tests for `buildOpenAICompletionsParams` with Groq and LM Studio/Qwen-style models.
