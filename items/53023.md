## Summary

Keep open. Current main still serializes each session lane, exposes only global and subagent concurrency config, keeps the lane wait warning default hardcoded unless callers pass an option, and has no config-backed auto-yield policy. The related PR #63864 is open and may cover part of the concurrency request, but it is unmerged and does not cover auto-yield or warning-threshold configurability.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review or rework PR #63864 for the session-lane concurrency portion, wire any accepted setting through AgentDefaults types, strict schema, defaults, generated metadata/docs, startup/reload application, and queue/runner tests, then separately design whether auto-yield belongs in the embedded runner/tool loop or queue policy. Configurable lane-wait warnings can be added on the same config surface or split into a narrower canonical follow-up if maintainers want to scope the work.

## Risks / Open Questions

- Increasing per-session concurrency may violate current transcript/session mutation assumptions unless the implementation accounts for the session write lock and runner lifecycle.
- Auto-yield is a separate runtime/product design problem because forced interruption across tool execution differs from the existing explicit sessions_yield flow.
- PR #63864 may supersede part of this once reviewed and merged, but current main does not include it.
