## Summary

Current `main` still lacks the requested `openclaw system event --agent-id/--session-key` surface, and the PR’s proposed wiring still has an unresolved `sessionKey`-only routing bug in the wake path. This is a real core gap, not safe to auto-close.

## Evidence

- **CLI on main still only sends `mode` and `text`:** `registerSystemCli()` exposes only `--text`, `--mode`, and `--json`, then calls `callGatewayFromCli("wake", opts, { mode, text }, ...)`; there is no `--agent-id` or `--session-key` support on current `main`.
  - file: [src/cli/system-cli.ts:52](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/system-cli.ts#L52)
  - command: `nl -ba src/cli/system-cli.ts | sed -n '42,72p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **User-facing docs still describe main-session-only behavior:** The CLI docs say `system event` enqueues a system event on the `main` session and document only `--text`, `--mode`, and `--json`. That matches current shipped behavior, so this request is not already implemented on `main`.
  - file: [docs/cli/system.md:34](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/cli/system.md#L34)

- **Gateway wake entrypoints on main still ignore target fields:** `cronHandlers["wake"]` casts params to `{ mode, text }` and calls `context.cron.wake({ mode, text })`, while `CronService.wake()` and `timer.wake()` also only accept `{ mode, text }`. The end-to-end wake flow on `main` is not wired for agent/session targeting yet.
  - file: [src/gateway/server-methods/cron.ts:130](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-methods/cron.ts#L130)

- **Current cron runtime still shows the PR review’s `sessionKey`-only mismatch risk:** In `buildGatewayCronService`, `enqueueSystemEvent` resolves the target agent from `opts?.agentId` only, while `requestHeartbeatNow` uses `resolveCronWakeTarget(opts)`, which can derive the agent from `sessionKey`. That means a `sessionKey`-only wake can still enqueue to the default agent while waking a different agent, matching the prior review concern.
  - file: [src/gateway/server-cron.ts:281](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-cron.ts#L281)

That concern remains consistent with current `main`’s cron wiring.

## Risks / Open Questions

- Closing this PR would drop a still-valid core feature gap: current `main` cannot target `openclaw system event` to a specific agent/session.
- Merging or closing based on the current patch without fixing `sessionKey`-only routing risks misdirecting the system event queue and the heartbeat to different targets.
