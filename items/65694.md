## Summary

Current `main` contains the MCP leak fix: stdio MCP transports now tear down lingering process trees on close, and session/cron cleanup paths retire per-session bundled MCP runtimes on reset/delete/rollover. The changelog and appcast record the same fix in unreleased `2026.4.24`, so this is implemented on `main` but not yet shipped in tagged release `v2026.4.23`.

## Evidence

- **Stdio MCP transport now kills lingering process trees:** `OpenClawStdioClientTransport.close()` ends stdin, waits briefly, and calls `killProcessTree(processToClose.pid)` if the stdio server has not exited, which directly addresses orphaned stdio MCP subprocesses.
  - file: [src/agents/mcp-stdio-transport.ts:112](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/mcp-stdio-transport.ts#L112)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Regression tests cover forced teardown behavior:** Unit tests verify both the forced process-tree kill path when graceful close hangs and the no-kill path when the stdio server exits cleanly.
  - file: [src/agents/mcp-stdio-transport.test.ts:79](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/mcp-stdio-transport.test.ts#L79)

- **Gateway session delete/reset retires session MCP runtimes:** Gateway session cleanup now calls `retireSessionMcpRuntime()` after the embedded run ends, so deleting or resetting a session disposes that session's bundled MCP runtime.

- **Reply-session rollover retires the previous MCP runtime:** When a reply session rolls over, OpenClaw archives the prior transcript and retires the previous session's bundled MCP runtime before continuing.
  - file: [src/auto-reply/reply/session.ts:794](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/auto-reply/reply/session.ts#L794)
