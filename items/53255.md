## Summary

Current main already implements the issue's accepted no-copy/paste flow: `openclaw dashboard` resolves the gateway token from config/env, opens or copies a tokenized Control UI URL by default, and the Control UI auto-imports `#token=` and strips it from the URL. The exact browser-side config/env autoload proposed in the title was not adopted, but the body explicitly accepted the tokenized-URL / `--open` alternative and that is already shipped.

## Risks / Open Questions

- The exact title request ('browser auto-load from config/env/.token') is not implemented; the supported solution is the accepted tokenized-URL bootstrap path instead.
- SecretRef-managed tokens intentionally do not auto-tokenize the URL, so those setups may still require manual auth if the external secret is not exposed to the CLI shell.

## Close Comment

Closing from a Codex review: current `main` already covers the no-manual-paste flow through the accepted tokenized-URL bootstrap path. `src/commands/dashboard.ts:27-47` resolves `gateway.auth.token` with env fallback and builds a `#token=...` dashboard URL, and `src/commands/dashboard.ts:67-80` opens it by default unless `--no-open` is used. On the UI side, `ui/src/ui/app-settings.ts:179-220` auto-imports `#token=` (and legacy `?token=`), applies it, and strips it from the URL; `ui/src/ui/storage.ts:154-193` keeps that token in sessionStorage for the current gateway/tab scope. This is also documented in `docs/web/dashboard.md:31-49` and covered by `src/commands/dashboard.links.test.ts:84-93` plus `ui/src/ui/app-settings.test.ts:301-325`. The issue title asked for browser-side config/env autoload specifically, but the body also proposed the tokenized-URL / `--open` path as an acceptable fix, and that behavior is already shipped in `v2026.4.22`.
