## Summary

Keep open. Current main still loads dotenv values literally: `src/infra/dotenv.ts` calls `dotenv.parse(...)`, stores each parsed value unchanged, and assigns that unchanged value into `process.env` for workspace, runtime, and global fallback dotenv paths. The current dotenv dependency does not perform variable expansion itself, the local tests cover precedence/blocking but not `$VAR` or `${VAR}` expansion in `.env`, and the related fix PR #38254 was closed unmerged.

## Best Possible Solution

Keep #38259 open and implement dotenv reference expansion in the shared dotenv loading path, with explicit tests for same-file, chained, shell-backed, workspace/global fallback, and precedence-preserving expansion. The fix should not weaken the existing workspace dotenv blocklist or shell-env precedence guarantees.
