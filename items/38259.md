## Summary

Keep open. Current main still parses loaded .env files with dotenv.parse and assigns parsed strings directly to process.env for workspace, runtime, and global fallback paths, so values such as OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY} remain literal. The only related fix PR in the provided context, #38254, was closed unmerged, so it does not supersede or ship the requested behavior.

## Best Possible Solution

Implement dotenv-level expansion in src/infra/dotenv.ts before assigning values to process.env, supporting ${VAR} and $VAR aliases from the existing process env plus earlier loaded dotenv entries while preserving non-overwrite precedence. Add focused tests in src/infra/dotenv.test.ts for shell-backed aliases, same-file aliases, chained references, missing references, workspace .env, and global/state-dir fallback behavior. Reuse the intent from closed PR #38254, but re-check the workspace .env security constraints before landing.
