## Summary

Keep open. Current main still parses .env files with dotenv.parse and assigns parsed strings directly to process.env, so $VAR and ${VAR} aliases remain literal. The related fix PR #38254 was closed unmerged, and current docs/tests cover precedence and config-level ${VAR} substitution but not dotenv-file expansion.

## Best Possible Solution

Implement dotenv-level expansion in src/infra/dotenv.ts before assigning values to process.env. Support ${VAR} and $VAR aliases from the existing process environment plus earlier loaded dotenv entries, while preserving non-overwrite precedence. Add focused tests in src/infra/dotenv.test.ts for shell-backed aliases, same-file aliases, chained references, missing references, workspace .env, global/state-dir fallback, and gateway.env fallback behavior. PR #38254 is useful prior work but should be rechecked against the current workspace .env security constraints before landing.
