## Summary

Keep open. Current main still loads `.env` entries literally: `src/infra/dotenv.ts` parses with `dotenv.parse(...)` and assigns each parsed value directly to `process.env` for workspace, runtime, and global fallback dotenv loading. I found config-value `${VAR}` substitution support, but that is a separate config resolver and does not expand aliases inside loaded `.env` variables such as `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}`.

## Best Possible Solution

Implement dotenv-level expansion in `src/infra/dotenv.ts` before values are assigned to `process.env`, preserving existing shell/process precedence and CWD-over-global precedence, and add focused regression tests in `src/infra/dotenv.test.ts` for shell-backed aliases, same-file aliases, chained references, missing references, and both workspace and global `.env` paths. Reuse or adapt the intent from closed PR #38254, but re-review the security interaction with workspace `.env` blocklists before landing.
