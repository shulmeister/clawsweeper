## Summary

Keep open. Current `main` added a doctor-side check/repair for gateway service entrypoint drift, but the reported `openclaw browser status` failure path is not directly handled there: the browser CLI still blindly calls `browser.request`, the status surface only explains config-path mismatches, and the current docs for `openclaw browser` still point mainly at `plugins.allow`. The issue discussion also shows live, distinct failure modes beyond the original macOS LaunchAgent case (`plugins.entries` omission and Linux/systemd PATH drift), so the thread is still describing actionable current behavior rather than an obsolete report.
