## Summary

Keep #13362 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still does not expose `openclaw doctor --dry-run` or a doctor-wide no-write validation mode, and the current doctor flow still has config, service, plugin registry, bundled dependency, and runtime repair paths that can mutate state. The open global agent dry-run issue #41418 is complementary and does not cover doctor/config validation.

## Best Possible Solution

Keep #13362 open until OpenClaw either implements or explicitly rejects a doctor-specific dry-run mode. The likely implementation is to add `openclaw doctor --dry-run`, thread `dryRun` through `DoctorOptions`, `resolveDoctorRepairMode`, the prompter, config flow, and every doctor contribution, make repair/write paths emit planned changes without mutating config/state/services/runtime dependencies, define exit-code behavior for findings, update https://docs.openclaw.ai/cli/doctor, and add focused tests proving dry-run does not mutate config, plugin registry state, bundled plugin deps, service state, cron/state stores, or plugin/channel-owned doctor repairs.

## Risks / Open Questions

- A shallow CLI-only `--dry-run` would be misleading unless every doctor side-effect path observes the same no-write contract, including config writes, gateway service repair, plugin registry refresh/migration, bundled plugin runtime dependency repair, cron/state/session repairs, and plugin/channel-owned doctor repair hooks.
- The requested non-zero exit behavior for findings still needs explicit product semantics so automation can distinguish validation findings from command failures.
