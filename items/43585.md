## Summary

Keep open. Current main has not implemented this PR's core hardening: config include resolution still normalizes the raw include path before any explicit NUL-byte or include-path length validation, and the tests/docs still only cover generic malformed-path behavior without the proposed deterministic ConfigIncludeError or 4096-character policy.

## Best Possible Solution

Keep the PR open or replace it with an equivalent main-branch patch that rejects NUL bytes before path resolution, enforces a byte-based include path and resolved path limit with clear ConfigIncludeError messages, and updates config include tests plus the public gateway configuration docs.
