## Summary

Keep #67055 open. Current main still has the Google Chat DM threading problem the PR is trying to fix: the monitor copies `message.thread?.name` into inbound reply context and sends the typing placeholder in that same thread without checking `replyToMode`. The PR is relevant, but it should not be closed as implemented or landed as-is because its delivery-time guards would drop explicit reply tags/directives that the shared reply pipeline intentionally preserves when `replyToMode` is `off`.

## Best Possible Solution

Revise #67055 rather than close it: suppress only automatic Google Chat monitor thread propagation for `replyToMode: "off"` (`ReplyToId`/`ReplyToIdFull` and the typing indicator), but do not blanket-clear `payload.replyToId` in delivery because that can represent explicit reply tags preserved by the shared reply pipeline. Add focused Google Chat monitor/reply tests covering flat DM replies, typing indicator behavior, text/media sends, and explicit reply tag preservation. Keep #52475/#39554 linked as the user-facing bug context.
