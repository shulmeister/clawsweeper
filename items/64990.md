## Summary

Keep open. Current `main` still derives the session-memory daily filename from `toISOString()`, so the reported UTC/local-date mismatch is still present. The same UTC-derived date is also reused in the markdown session title, and existing tests do not cover timezone-sensitive filename behavior.

## Evidence

- **Current handler still uses UTC date for daily filename:** `saveSessionToMemory` builds `dateStr` with `now.toISOString().split("T")[0]`, then uses that value in the output filename. That matches the report's root cause and means users ahead of UTC can still get the previous day's filename after local midnight.
  - file: [src/hooks/bundled/session-memory/handler.ts:85](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/hooks/bundled/session-memory/handler.ts#L85)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '72,210p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Session heading still reuses the UTC-derived date:** The same handler formats `timeStr` from `toISOString()` and emits `# Session: ${dateStr} ${timeStr} UTC`, so the issue's second symptom is also still present on `main`.
  - file: [src/hooks/bundled/session-memory/handler.ts:185](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/hooks/bundled/session-memory/handler.ts#L185)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '176,190p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Repo already has local-date patterns elsewhere:** `resolveDefaultRollingLogFile` uses `getFullYear/getMonth/getDate` for local-date log rotation, and `formatDayKey` uses `toLocaleDateString("en-CA", { timeZone: Intl.DateTimeFormat().resolvedOptions().timeZone })`. That supports the issue's claim that session-memory is the outlier.
  - file: [src/logging/log-file-path.ts:27](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/logging/log-file-path.ts#L27)
  - command: `nl -ba src/logging/log-file-path.ts | sed -n '27,36p' && nl -ba src/infra/session-cost-usage.ts | sed -n '165,166p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Reasonable reproduction still shows the mismatch:** A direct Node repro with `new Date('2026-04-11T19:53:00Z')` prints `2026-04-12` for `Intl.DateTimeFormat('en-CA', { timeZone: 'Asia/Shanghai' })`, while `toISOString().split('T')[0]` prints `2026-04-11`. That is the exact off-by-one-day behavior described in the issue.
  - command: `node -e "const d=new Date('2026-04-11T19:53:00Z'); console.log(new Intl.DateTimeFormat('en-CA',{timeZone:'Asia/Shanghai'}).format(d)); console.log(d.toISOString().split('T')[0]);"`
- **Existing tests miss timezone/date assertions:** The session-memory tests verify file creation and transcript content for `/new` and `/reset`, but these cases do not assert the generated date string or timezone-sensitive filename behavior. There is no coverage that would catch this regression today.
  - file: [src/hooks/bundled/session-memory/handler.test.ts:217](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/hooks/bundled/session-memory/handler.test.ts#L217)
