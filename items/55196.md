## Summary

Keep open. Current main still lacks the requested `brave.http` diagnostics flag: Brave sends API requests and uses the shared web-search cache, but there is no Brave-specific diagnostics flag gate, subsystem logger, request/status/timing log, or cache hit/miss log. The existing diagnostics flag machinery and Telegram pattern make this actionable inside the bundled Brave plugin without a new core API.

## Best Possible Solution

Implement the feature in `extensions/brave`: pass full config or a precomputed diagnostics state into the Brave runtime, gate logs on `brave.http` via `openclaw/plugin-sdk/diagnostic-runtime`, log sanitized request URL/query/params plus HTTP status and duration, log cache hit/miss around the existing cache path, add focused Brave provider tests, and document the flag in `docs/diagnostics/flags.md` and `docs/tools/brave-search.md`.
