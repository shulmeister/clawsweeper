## Summary

Current main, and the v2026.4.24 release tree, already retire bundled MCP runtimes for completed nested `sessions_send`/A2A steps. The direct gateway flag proposal from linked PRs was not the landed shape, but the observable leak path is covered by the shipped runtime-retirement path and regression tests.

## Best Possible Solution

Keep the shipped v2026.4.24 runtime-retirement implementation for completed `sessions_send` steps. If operators still see leaked MCP cohorts on 2026.4.24 or newer after the A2A completion flow has finished, track that as a narrower follow-up with process counts, timeout/error status, and the exact channel/plugin path.

## Risks / Open Questions

- This close is scoped to the completed `sessions_send` leak reported here; pending or timed-out nested runs intentionally are not retired while they may still be active.
- The gateway handler still does not default `cleanupBundleMcpOnRunEnd` from the lane by itself, so a future non-`sessions_send` nested gateway entrypoint should get separate lifecycle coverage if it leaks.
