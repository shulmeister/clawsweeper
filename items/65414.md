## Summary

Keep open. Current main still routes ACTION_ASSIST launches through the Chat destination, and there is no one-shot Voice-tab mic auto-activation state for assistant launches. The reporter's file-level analysis matches the current Android code; the related local title-search items are unrelated TTS/native voice-message reports, not a canonical duplicate for this Android assistant-entrypoint behavior.

## Best Possible Solution

Implement this in the Android app: keep ASK_OPENCLAW/text-prompt launches on Chat, but route promptless source="assist" launches to HomeDestination.Voice, add a one-shot pending voice auto-activation state consumed by VoiceTabScreen, and extend the mic permission flow so permission grant enables the mic exactly once for that launch. Add focused Android unit or Robolectric coverage for the routing and one-shot consumption behavior.
