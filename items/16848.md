## Summary

Keep open. Current main still does not solve BlueBubbles auto-TTS arriving as regular MP3 attachments. The lower BlueBubbles attachment layer can send iMessage voice memos when `asVoice` is set, and shared outbound delivery now preserves `audioAsVoice`, but speech-core still excludes BlueBubbles from native voice-note TTS and the BlueBubbles reply/media paths still do not pass `asVoice` into `sendBlueBubblesMedia`. The reporter’s related PR #19522 directly addressed this gap but was closed unmerged per the provided GitHub context.

## Best Possible Solution

Keep this issue open until BlueBubbles auto-TTS is implemented end to end: add BlueBubbles to the intended voice-note TTS capability path or a channel capability-driven replacement, ensure provider output/codec handling is compatible with iMessage voice memos, pass `payload.audioAsVoice` through both BlueBubbles outbound `sendMedia` and monitor-processing media replies into `sendBlueBubblesMedia({ asVoice: true })`, and add focused speech-core plus BlueBubbles regression tests. The prior #19522 discussion is useful implementation context, but it should be reworked with the command-injection concern resolved before landing.
