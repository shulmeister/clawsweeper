## Summary

Keep open. Current main still does not implement the requested BlueBubbles auto-TTS native voice-message path. Speech-core still excludes BlueBubbles from native voice-note TTS, the BlueBubbles outbound and monitor media paths still do not forward `audioAsVoice` as `asVoice`, and the related fixes in #16885/#19522 were closed unmerged. The lower BlueBubbles attachment layer can send voice memos when `asVoice` is explicitly routed, so this remains an implementation gap rather than a stale or unsupported request.

## Best Possible Solution

Keep this issue open until BlueBubbles auto-TTS is implemented end to end: make BlueBubbles part of the native voice-note TTS capability path or replace the hard-coded channel set with a channel capability contract, ensure provider output/codec handling is compatible with iMessage voice memos, pass `audioAsVoice` through BlueBubbles outbound and monitor media delivery as `sendBlueBubblesMedia({ asVoice: true })`, and add focused speech-core plus BlueBubbles regression tests. The prior #19522 discussion should be treated as useful design and testing context, not as a landable patch without security rework.
