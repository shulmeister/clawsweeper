## Summary

Keep open. Current main still does not implement BlueBubbles auto-TTS as native iMessage voice messages: speech-core excludes BlueBubbles from native voice-note TTS, BlueBubbles outbound delivery does not consume `audioAsVoice`, and the monitor reply path still calls `sendBlueBubblesMedia` without `asVoice`. The existing BlueBubbles attachment/tool path can send voice memos when `asVoice` is explicitly provided, so this remains a real implementation gap rather than a stale request.

## Best Possible Solution

Keep the issue open until BlueBubbles auto-TTS is implemented end to end. The best path is to move BlueBubbles into the native voice-note TTS capability path or replace the hard-coded channel set with a channel capability contract, ensure provider output/codec handling produces iMessage-compatible voice memo media, propagate `audioAsVoice` through BlueBubbles outbound and monitor delivery as `sendBlueBubblesMedia({ asVoice: true })`, preserve paired text where needed, and add focused speech-core plus BlueBubbles regression tests. Treat #19522 as useful design/testing context, not a landable patch as-is.
