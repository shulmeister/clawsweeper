## Summary

Current `main` does not match the reported restart failure. Discord thread messages are routed by the thread channel id into a stable session key, that key is persisted on inbound handling, and session reuse logic reopens the existing session entry when it is still fresh. Current code still lacks Slack-style `ThreadHistoryBody` bootstrap for genuinely new Discord thread sessions, but the specific report that a gateway restart creates a brand-new session for the same thread is obsolete against `main`.

## Evidence

- **Discord routes by thread conversation id:** Preflight builds the Discord route from `messageChannelId`, so a thread message resolves against the thread's own conversation id rather than a per-process key.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:626](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/discord/src/monitor/message-handler.preflight.ts#L626)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Resolved route shape is stable `agent:<agent>:discord:channel:<id>`:** `buildDiscordRoutePeer` uses the conversation id as the peer id for channel traffic, and `resolveDiscordConversationRoute` delegates to the normal route/session-key builder.
  - file: [extensions/discord/src/monitor/route-resolution.ts:10](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/discord/src/monitor/route-resolution.ts#L10)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Inbound Discord processing persists the same session key for thread traffic:** `processDiscordMessage` keeps `useSuffix: false` and writes `SessionKey: boundSessionKey ?? autoThreadContext?.SessionKey ?? threadKeys.sessionKey`, so thread traffic stays on the resolved thread session key instead of minting a restart-specific replacement.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:431](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/discord/src/monitor/message-handler.process.ts#L431)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Session engine reuses an existing fresh entry instead of creating a new session:** When the stored entry has a session id and is still fresh, the reply session logic reuses `entry.sessionId`; it only creates a new UUID when the entry is missing or stale.
  - file: [src/auto-reply/reply/session.ts:427](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/auto-reply/reply/session.ts#L427)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Fresh-thread resets happen on archive events, not on gateway restart:** Discord's explicit thread-session reset path is wired to `THREAD_UPDATE` archive handling, which calls `closeDiscordThreadSessions()` to mark matching thread sessions stale. I found no restart-path code doing the same.
  - file: [extensions/discord/src/monitor/listeners.ts:791](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/discord/src/monitor/listeners.ts#L791)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Slack still has extra bootstrap history that Discord does not:** Slack reads `thread.initialHistoryLimit`, checks whether the thread session already exists, and fills `threadHistoryBody` from fetched thread replies on a genuinely new session.
  - file: [extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:151](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/slack/src/monitor/message-handler/prepare-thread-context.ts#L151)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)

## Risks / Open Questions

- Discord still lacks Slack-style `ThreadHistoryBody` bootstrap for genuinely new thread sessions, so this closure only covers the reported gateway-restart reproduction path, not that broader parity gap.
- I did not run a live Discord restart smoke in this read-only review; the conclusion is based on current `main` code paths and tests.

## Close Comment

Closing after a Codex review of current `main`. I could not substantiate the reported restart behavior on the current codebase: Discord thread routing now resolves off the thread conversation id (`extensions/discord/src/monitor/message-handler.preflight.ts:626`, `extensions/discord/src/monitor/route-resolution.ts:10`), inbound handling persists that same session key for the thread (`extensions/discord/src/monitor/message-handler.process.ts:431`), and the session layer reuses the existing fresh session entry instead of minting a new one (`src/auto-reply/reply/session.ts:427`). The only built-in path I found that intentionally forces a fresh Discord thread session is thread archive handling, not gateway restart (`extensions/discord/src/monitor/listeners.ts:791`). Discord still does not have Slack-style `ThreadHistoryBody` bootstrap parity (`extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:151`), but the specific report that a gateway restart creates a brand-new session for the same thread no longer matches current `main`. If you still hit this on a current build, please open a new issue with exact restart steps, current version/commit, session store config, and the before/after session keys or transcript files you see.
