## Summary

Current main already fixes the duplicate attachment-backed webchat send path: `chat.send` now returns `in_flight` for same-idempotency duplicate attachment sends even when the duplicate wins the attachment parsing/model-catalog race, preventing a second agent dispatch and duplicate persisted image turn. The Control UI also sends a generated run ID as the `idempotencyKey`, and Gateway applies attachment size/offload/sanitization handling before provider submission.

## Best Possible Solution

Close this issue as implemented on main and keep the shipped server-side idempotency/race fix. If users later reproduce duplicates with different idempotency keys, track that as a new narrower Control UI client retry/content-dedupe issue.
