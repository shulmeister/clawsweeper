## Summary

Keep open. Current `main` still has a plausible code-path match for the report: attach-only loopback CDP profiles are treated as the loopback `/json/list` path, that path still reads only `id` from tab listings, and `browser.open` can still succeed independently via `Target.createTarget`, which matches the issueâ€™s `open`-works / `tabs`-empty / `snapshot`-tab-not-found symptom.

## Evidence

- **Loopback attach-only profiles still bypass the remote Playwright tab path:** Capability selection keys only on `profile.cdpIsLoopback`; loopback profiles stay in `local-managed` mode with `supportsPerTabWs`/`supportsJsonTabEndpoints`, while only non-loopback CDP profiles use persistent Playwright. An attach-only localhost Browserless-style profile therefore still goes through `/json/list`-based tab discovery.
  - file: [extensions/browser/src/browser/profile-capabilities.ts:33](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/src/browser/profile-capabilities.ts#L33)
  - command: `nl -ba extensions/browser/src/browser/profile-capabilities.ts | sed -n '17,56p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **`/json/list` tab parsing still requires `id`, not `targetId`:** The loopback tab-listing path maps `targetId: t.id ?? ""` and filters out empty ids. If an attach-only external CDP endpoint returns page entries with `targetId` instead of `id` as reported in #59424, `browser.tabs` will still come back empty on current `main`.
  - file: [extensions/browser/src/browser/server-context.tab-ops.ts:97](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/src/browser/server-context.tab-ops.ts#L97)
  - command: `nl -ba extensions/browser/src/browser/server-context.tab-ops.ts | sed -n '90,120p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Open-tab creation is independent and can still return a target id:** `createTargetViaCdp()` returns the `Target.createTarget` result directly, and `openTab()` returns that `targetId` even if subsequent polling never rediscovers the tab in `listTabs()`. That matches the reported state where `browser.open` succeeds first but later tab resolution fails.
  - file: [extensions/browser/src/browser/cdp.ts:230](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/src/browser/cdp.ts#L230)
  - command: `nl -ba extensions/browser/src/browser/cdp.ts | sed -n '176,280p'`

- **Current attach-only loopback regression coverage only exercises `id`-shaped tab payloads:** The existing attach-only loopback tabs test mocks `/json/list` with `{ id: "PAGE-1", ... }` and does not cover the Browserless-style `{ targetId: ... }` shape from the issue report, so there is no regression proof on `main` that this case was fixed.
