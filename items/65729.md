## Summary

Keep this issue open. Current main still deliberately keeps Mattermost DMs non-threaded, disables direct-message history, has no WebSocket ping/pong keepalive, lacks Mattermost thread-history backfill, and still drops mention-only messages after normalization. Related #51104 and #53186 cover parts of the discussion, but this issue includes broader and partly unique Mattermost threading reliability work that is not fully superseded.

## Best Possible Solution

Keep this open and implement it as Mattermost-plugin-owned reliability work in extensions/mattermost. The likely product path is to coordinate #51104 and #53186, preserve the current documented flat-DM default, add an explicit DM/per-chat-type threading option, add WebSocket ping/pong keepalive configuration, backfill thread history from Mattermost when an empty thread-scoped session is resumed, handle mention-only invocations, and cover the behavior with Mattermost plugin tests plus docs/config metadata updates.

## Risks / Open Questions

- Changing DM threading semantics can regress users relying on the documented flat-DM behavior; keep the default non-threaded and add an explicit per-chat-type or DM-specific opt-in if this is implemented.
- Thread-history backfill must respect Mattermost sender allowlists and context-visibility policy so it does not inject unauthorized or unexpected thread content into model context.
- WebSocket ping/pong should clear timers on close/error/abort and account for test fakes or ws-compatible clients that may not expose ping/pong methods.
