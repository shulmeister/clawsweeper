## Summary

Keep open. Current main still leaves the bundled `memory-lancedb` plugin on pure vector retrieval: it creates/open a LanceDB table without an FTS index, `MemoryDB.search` accepts only an embedding vector and calls `vectorSearch`, the plugin config/manifest have no `hybrid` options, and recall/forget/CLI/hooks do not pass lexical query text into the DB layer. The related implementation PR #7636 was closed unmerged, and the recent issue discussion already contains a maintainer-posted Codex review keeping this open; current main at `d54d2d6b9b8aa28fb222d895c0f01f43933f4409` still matches that finding.

## Best Possible Solution

Keep #7629 open as a focused bundled plugin parity enhancement. The best path is to implement this inside `extensions/memory-lancedb`: verify the current LanceDB JS APIs, add typed `hybrid` config and manifest/docs entries, create or maintain an FTS index on the `text` column during initialization, pass original query text through recall/forget/CLI/hooks, use LanceDB hybrid search or a verified fallback/reranker when enabled, preserve vector-only fallback, and add targeted tests for config validation, FTS setup, exact-keyword recall, fallback behavior, and query text propagation.
