## Summary

Keep #63257 open. The current Feishu startup path still performs a blocking startup preflight with a 30,000 ms default timeout before each account monitor is started, and the tests only cover “continues after timeout,” not “does not delay startup.” The related ClawSweeper matches are Windows gateway/WebSocket handshake issues, not this Feishu API preflight timeout.

## Best Possible Solution

Keep the issue open and fix the Feishu plugin startup path directly: make the bot identity/API probe non-blocking or much shorter by default, wire any timeout control through documented channel config, preserve background bot identity recovery, and add regression coverage that gateway startup does not wait 30 seconds when Feishu's token/probe request stalls on Windows.
