## Summary

Current `main` still does not let MiniMax `web_search` authenticate with `MINIMAX_OAUTH_TOKEN`. The runtime, plugin metadata, tests, and docs all continue to require a Coding Plan key or API key fallback, so this issue remains actionable and should stay open.

## Evidence

- **Runtime credential lookup excludes OAuth token:** `resolveMiniMaxApiKey()` only reads configured `tools.web.search.apiKey`, `MINIMAX_CODE_PLAN_KEY`, `MINIMAX_CODING_API_KEY`, and `MINIMAX_API_KEY`. `MINIMAX_OAUTH_TOKEN` is not in the lookup chain.
  - file: [extensions/minimax/src/minimax-web-search-provider.runtime.ts:51](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/minimax/src/minimax-web-search-provider.runtime.ts#L51)
  - command: `nl -ba extensions/minimax/src/minimax-web-search-provider.runtime.ts | sed -n '35,65p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Web search plugin contract advertises Coding Plan creds only:** The MiniMax web-search provider exposes `credentialLabel: "MiniMax Coding Plan key"` and `envVars` limited to `MINIMAX_CODE_PLAN_KEY` and `MINIMAX_CODING_API_KEY`.
  - file: [extensions/minimax/src/minimax-web-search-provider.ts:31](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/minimax/src/minimax-web-search-provider.ts#L31)

## Risks / Open Questions

- OAuth-only MiniMax users on current `main` still need a separate Coding Plan key to use `web_search`.
- Because docs and tests currently match the code, closing this as fixed would be incorrect; any cleanup should come from an actual code+docs change or an explicit product decision that OAuth is intentionally unsupported for search.
