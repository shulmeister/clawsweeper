## Summary

Keep open. The issue is recent, author is not protected, labels are not protected, and the report gives concrete reproduction details across builtin local, Ollama, and QMD backends. I did not find high-confidence evidence that current main specifically fixed the EMFILE watcher crash, and the current memory-core path still has watcher-backed indexing behavior, so conservative cleanup should not close it.

## Evidence

- **Concrete recent reproduction:** Issue #59101 reports OpenClaw 2026.3.31 (213a704) on macOS with Node v24.14.1, including a clean-room profile/workspace with only two small memory files where `openclaw memory index --force` still crashes with `EMFILE: too many open files, watch`.
- **Current main still uses a memory watcher path:** Current main's memory-core manager imports `FSWatcher` from `chokidar`, stores watcher/timer state, and constructs the watcher for non-status memory managers before indexing state is initialized. That keeps the reported failure path plausible on main rather than obviously obsolete.
