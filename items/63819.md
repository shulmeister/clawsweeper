## Summary

Keep open. The issue is recent, has a second-user confirmation, and the reporter posted a concrete workaround aimed at the gateway terminal-lifecycle error path. Current main has adjacent timeout/lifecycle hardening, but `pendingTerminalLifecycleErrors` still stores only a timer and later non-error events still clear a pending terminal lifecycle error without finalizing it, so I do not see source/test/release evidence that this specific v2026.4.9 stuck-running timeout regression is fixed.

## Best Possible Solution

Add a focused regression test in `src/gateway/server-chat.agent-events.test.ts` for lifecycle start/error scheduling pending terminal cleanup, then a later non-error/non-terminal event arriving without lifecycle end, and require the run/session to still finalize and clear state. Then update `src/gateway/server-chat.ts` so pending terminal lifecycle errors retain the event/options and only suppress finalization for intentional retry/fallback paths, preserving the existing fallback-success tests.

## Evidence

- **Issue discussion includes concrete workaround and second confirmation:** The provided issue comments include another v2026.4.9 gateway user confirming sessions re-stick after request timeouts, and the reporter's workaround changes `clearPendingTerminalLifecycleError` to retain `{ timer, evt, opts }` and optionally finalize the pending lifecycle error when clearing it. That maps directly to the current gateway code path below.
- **Pending lifecycle errors still store only a timer:** Current main declares `pendingTerminalLifecycleErrors` as `Map<string, NodeJS.Timeout>` and `clearPendingTerminalLifecycleError` only clears/deletes that timer.
