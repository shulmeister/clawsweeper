## Summary

Current main already supports tool-level streaming via the existing `onUpdate` callback path: tools can emit incremental partial `AgentToolResult` updates during execution, those updates are propagated through the runtime as `tool_execution_update`/`tool_call_update`, and the behavior is documented and tested. The exact return shape requested in the issue (`AsyncGenerator`/`AsyncIterable`/`ReadableStream`) is not the contract, but the requested capability is already implemented well enough on main and shipped in the latest release.

## Risks / Open Questions

- The issue asks for `AsyncGenerator`/`AsyncIterable`/`ReadableStream` return values specifically; current main covers the streaming capability through `onUpdate` instead, so the closure assumes callback-based streaming is sufficient for the reported use case.

## Close Comment

Codex review of current `main` found that tool-level streaming is already implemented on the existing callback-based path. Tools can stream partial results through `onUpdate` (`src/agents/tools/common.ts:21`), the adapter forwards that callback into tool execution (`src/agents/pi-tool-definition-adapter.ts:226`), long-running tools like `exec` emit incremental updates (`src/agents/bash-tools.exec-runtime.ts:569`), and OpenClaw surfaces those as `tool_call_update` events (`src/acp/translator.ts:893`; documented in `docs/cli/acp.md:50`). This behavior is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Closing as already implemented. If the remaining ask is specifically for `AsyncIterable`/`ReadableStream` return types instead of the existing `onUpdate` callback contract, please open a new request focused on that API shape.
