## Summary

Keep #69118 open. Current main has a partial fix for volatile inbound metadata via `extraSystemPromptStatic`, but the specific reported `groupIntro` drift is still present: first group turns include `groupIntro` in the static hash input, later turns usually omit it, and `resolveCliSessionReuse` still invalidates on any stored/current `extraSystemPromptHash` mismatch.

## Best Possible Solution

Fix the CLI session reuse hash input for group replies by excluding `groupIntro` from `extraSystemPromptStaticParts` or otherwise separating activation-only prompt text from stable session identity. Add a regression test that exercises two consecutive group turns: first turn with `shouldInjectGroupIntro` true, follow-up with it false, and assert the CLI session is reused instead of returning `reason=system-prompt`. Consider a tolerant path for legacy bindings with missing prompt hash metadata if maintainers still want upgrade smoothness.
