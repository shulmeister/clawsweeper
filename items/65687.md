## Summary

Current main still leaves the reported Discord voice/Deepgram failure path open. The media capability resolver can return a partial active media-understanding registry before compat loading, Discord voice still uses that shared audio transcription path, and no related item in the provided context supersedes this specific provider-resolution bug.

## Best Possible Solution

Keep this issue open. Fix `src/plugins/capability-provider-runtime.ts` so configured media-understanding providers from compat loading are merged into a partial active registry, without pulling unrelated bundled providers. Add a regression test where active mediaUnderstandingProviders contains OpenAI/MiniMax, `cfg.tools.media.audio.models` requests Deepgram, and the resolved provider set includes Deepgram.
