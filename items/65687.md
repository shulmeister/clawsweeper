## Summary

Keep open. Current main still appears to have the reported provider-resolution failure mode: Discord voice STT uses the shared media-understanding audio path, but `resolvePluginCapabilityProviders` can return a partially active non-speech provider registry before loading bundled compat providers like Deepgram. The issue is recent, has no protected maintainer/owner signal, and the related ClawSweeper matches are older Discord voice/message items rather than a canonical tracker for this Deepgram registry bug.

## Best Possible Solution

Keep the issue open and fix the provider-resolution contract. The likely patch is in `src/plugins/capability-provider-runtime.ts`: for `mediaUnderstandingProviders`, merge active providers with bundled compat providers requested by `tools.media.audio.models` or otherwise avoid the broad early return when config requests a missing bundled provider. Add a focused regression test with an active registry containing only OpenAI/MiniMax media providers and a config requesting Deepgram, then verify the Discord voice STT path through the shared `transcribeAudioFile` runtime.
