## Summary

Keep #70879 open. Current main does not implement retry/backoff for Feishu outbound message sends: text/card sends make a single SDK call and throw on nonzero API responses, and the Feishu tests explicitly expect a rate-limit-like SDK error to be rethrown without retry. The existing Feishu backoff handling is scoped to typing indicator reactions, not normal message delivery.

## Best Possible Solution

Keep the issue open and implement bounded retry/backoff in the Feishu plugin-owned outbound send layer. The fix should wrap message create/reply/patch and streaming-card send/update paths as appropriate, classify retryable thrown SDK/Axios errors plus nonzero Feishu response codes such as the reported 11232/2200/429 family after checking Feishu docs, respect Retry-After when available, and add focused tests proving a retryable failure succeeds on a later attempt while non-retryable errors still fail promptly.
