## Summary

Keep open. Current `main` does not implement this cleanup fix on the legacy fallback path, and the PR as proposed still appears incomplete for restored thread/topic sessions. Normal sessions usually clean up via the persisted `sessionFile`, but the fallback delete candidate still omits thread/topic context.

## Evidence

- **Reset cleanup still drops thread/topic suffix on fallback candidate:** `resetReplyRunSession` resolves the stored `sessionFile`, then also adds a fallback candidate with `resolveSessionTranscriptPath(prevSessionId, agentId)` and no thread/topic argument. That means the legacy/stale-entry fallback still targets `<sessionId>.jsonl`, not `<sessionId>-topic-<id>.jsonl`.
  - file: [src/auto-reply/reply/agent-runner-session-reset.ts:117](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/agent-runner-session-reset.ts#L117)
  - command: `nl -ba src/auto-reply/reply/agent-runner-session-reset.ts | sed -n '107,124p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current reset call only forwards string thread ids:** `runReplyAgent` passes `messageThreadId` into the reset helper only when `sessionCtx.MessageThreadId` is a string. Channel contexts allow `string | number`, so numeric topic ids are dropped before cleanup fallback resolution.
  - file: [src/auto-reply/reply/agent-runner.ts:1173](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/agent-runner.ts#L1173)

## Risks / Open Questions

- The reported leak may not reproduce for freshly persisted sessions because `prevEntry.sessionFile` is usually present and resolved first, so the bug is concentrated in legacy/stale-entry fallback cases.
- Closing as implemented would be unsafe: current `main` still lacks a topic-aware fallback delete path, and the open PR likely needs revision to use restored thread/topic context rather than only the live `MessageThreadId`.
