## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal plugin downloads the selected GitHub release asset through raw `node:https` redirect recursion and streams it directly to disk without the guarded fetch helper, an explicit timeout, or byte-limit enforcement.

## Best Possible Solution

Fix this in the bundled Signal plugin. Replace the private raw `node:https` `downloadToFile` implementation in `extensions/signal/src/install-signal-cli.ts` with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, wire an explicit install-download timeout and redirect limit, choose trusted env-proxy mode only if maintainers want proxy support for this trusted GitHub release download flow, enforce both declared and streamed byte caps, clean partial temp files, and add focused Signal installer tests for timeout wiring, private-network redirect blocking, proxy mode selection, oversized downloads, streamed over-limit abort, and cleanup.

## Risks / Open Questions

- The user-triggered Signal install path can still hang on a stalled download, follow redirects without the shared SSRF guard, and write an unbounded response body to disk.
- The fix needs a conservative archive size cap and cleanup on content-length rejection, streamed over-limit abort, fetch failure, and extraction failure.
- The implementation should stay within the bundled plugin boundary by using `openclaw/plugin-sdk/ssrf-runtime` rather than core internals.
