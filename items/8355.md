## Summary

Current main still lacks the requested sentence-level LLM -> TTS -> telephony audio pipeline. The classic voice-call path waits for a completed agent text response, then a completed telephony TTS buffer, then chunks that buffer to Twilio. Realtime voice support is separate full-duplex provider mode, and the provided maintainer comments identify #8355 as the canonical remaining issue after #42187 closed unmerged and #42159 was closed as the narrower duplicate.

## Best Possible Solution

Keep #8355 open as the canonical provider-generic voice-call latency feature. The best implementation is an additive streaming telephony TTS capability with buffered fallback, wiring embedded-agent partial replies or sentence chunks into that capability, streaming synthesized audio chunks through Twilio and supported Telnyx media paths, and preserving queue clearing, abort, timeout, fallback, transcript, and barge-in semantics with focused contract and voice-call tests.
