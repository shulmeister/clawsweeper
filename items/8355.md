## Summary

Keep open. Current main still batches the classic voice-call STT -> LLM -> TTS path: it waits for one completed text response, runs TTS as one full-buffer synthesis call, and only then chunks audio to Twilio. The newer realtime voice path is a separate full-duplex architecture, and the provided discussion already identifies #8355 as the canonical provider-generic sentence-level streaming TTS request after related PRs closed unmerged.

## Best Possible Solution

Keep #8355 open as the canonical feature request. The best product/code path is a provider-generic sentence-level pipeline in the voice-call plugin: feed embedded-agent partial replies into a sentence splitter, add a streaming telephony TTS contract that yields audio chunks, wire Twilio/Telnyx media playback with abort and barge-in support, and retain the current full-buffer path as fallback for non-streaming speech providers.

## Evidence

- **Classic response generation returns one completed text result:** `VoiceResponseResult` is `{ text: string | null }`; `generateVoiceResponse` awaits `agentRuntime.runEmbeddedPiAgent(...)`, extracts final payload text, and returns `{ text }`. The voice-call path does not pass partial-reply callbacks or expose an async text iterator here.

  - file: [extensions/voice-call/src/webhook.ts:830](https://github.com/openclaw/openclaw/blob/d74f897c1c6b2de756c98eb3bc7de45f411d6a5c/extensions/voice-call/src/webhook.ts#L830)

  - file: [extensions/voice-call/src/telephony-tts.ts:22](https://github.com/openclaw/openclaw/blob/d74f897c1c6b2de756c98eb3bc7de45f411d6a5c/extensions/voice-call/src/telephony-tts.ts#L22)
