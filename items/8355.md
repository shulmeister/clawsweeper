## Summary

Keep open. Current `main` adds a separate realtime voice path, but the classic voice-call STT -> LLM -> TTS flow is still batched end-to-end, so the requested sentence-level streaming TTS pipeline is not implemented on `main`.

## Evidence

- **Classic voice response still resolves to one final text payload:** `generateVoiceResponse` returns `Promise<VoiceResponseResult>` where the result shape is `{ text: string | null }`, and the webhook awaits that full result before calling `manager.speak(callId, result.text)`.
  - file: [extensions/voice-call/src/response-generator.ts:31](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/voice-call/src/response-generator.ts#L31)
  - command: `nl -ba extensions/voice-call/src/response-generator.ts | sed -n '1,260p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Webhook waits for full generated text before playback:** The auto-response path does `const result = await generateVoiceResponse(...)` and only then, if `result.text` exists, calls `await this.manager.speak(callId, result.text)`. That is still a batched text stage, not streamed sentence output.
  - file: [extensions/voice-call/src/webhook.ts:830](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/voice-call/src/webhook.ts#L830)
  - command: `nl -ba extensions/voice-call/src/webhook.ts | sed -n '810,860p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Telephony TTS contract is still full-buffer:** `TelephonyTtsProvider.synthesizeForTelephony` is typed as `(text: string) => Promise<Buffer>`, and `createTelephonyTtsProvider` converts one completed PCM buffer to one mu-law buffer before returning it. There is no streaming iterator/chunk callback in this contract.
  - file: [extensions/voice-call/src/telephony-tts.ts:22](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/voice-call/src/telephony-tts.ts#L22)
  - command: `nl -ba extensions/voice-call/src/telephony-tts.ts | sed -n '1,220p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Twilio stream playback still waits for the full synthesized audio buffer:** `playTtsViaStream(text, streamSid)` calls `ttsProvider.synthesizeForTelephony(text)`, awaits the completed `muLawAudio` buffer via `Promise.race`, and only afterward iterates `chunkAudio(muLawAudio, CHUNK_SIZE)` to send 20 ms frames. That matches the issue's reported bottleneck.
  - file: [extensions/voice-call/src/providers/twilio.ts:619](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/voice-call/src/providers/twilio.ts#L619)

- **Realtime voice is documented and validated as a separate architecture:** Current docs say `realtime` is 'separate from `streaming`', and config validation rejects `realtime.enabled` together with `streaming.enabled`. That confirms the newer low-latency realtime path does not implement the requested sentence-level streaming upgrade for the classic pipeline.
  - file: [docs/plugins/voice-call.md:191](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/docs/plugins/voice-call.md#L191)

- **Issue discussion already distinguishes realtime support from this still-open feature:** The April 25, 2026 maintainer comment says current `main` has `realtime.enabled` for full-duplex voice providers, but that it is a separate architecture that bypasses classic STT -> LLM -> TTS; it also notes PR #42187 was closed for rework and keeps this issue open as the canonical provider-generic sentence-level streaming request. The Feb. 4 comment also broadens scope to provider-agnostic support by calling out Telnyx parity needs.
