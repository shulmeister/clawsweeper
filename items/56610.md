## Summary

Keep open. Current `main` has a generic outbound write-ahead delivery queue and Discord REST retries, but it does not implement the requested Discord reconnect-drain parity with WhatsApp. The existing evidence is not strong enough to say the report is obsolete or already fixed.

## Evidence

- **Discord delivery retry is still request-level, not reconnect-aware:** `withDiscordDeliveryRetry()` only retries HTTP-style failures based on status codes (`429` and `>=500`). There is no Discord-specific reconnect queue/drain logic here.
  - file: [extensions/discord/src/delivery-retry.ts:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/delivery-retry.ts#L16)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Docs still describe Discord retry narrowly:** The public retry docs still say Discord retries only on rate-limit errors (`HTTP 429`), which matches the issue's claim that reconnect handling is separate from the documented retry policy.
  - file: [docs/concepts/retry.md:38](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/concepts/retry.md#L38)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Generic outbound queue exists, but recovery is startup-oriented:** Outbound payloads are persisted before send via the write-ahead `delivery-queue`, and gateway startup runs `recoverPendingDeliveries()`. That is useful background, but it is not the same as a Discord reconnect-triggered drain.
  - file: [src/infra/outbound/deliver.ts:762](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/outbound/deliver.ts#L762)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway startup recovery is wired centrally:** `recoverPendingDeliveries()` is invoked from gateway runtime services during startup, confirming queue recovery exists globally.
  - file: [src/gateway/server-runtime-services.ts:55](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-runtime-services.ts#L55)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Reconnect-drain parity exists for WhatsApp, not Discord:** WhatsApp explicitly calls `drainPendingDeliveries()` on reconnect. A repo search found no corresponding `drainPendingDeliveries()` call under `extensions/discord`, so the requested parity is not present on current `main`.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:392](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/whatsapp/src/auto-reply/monitor.ts#L392)
  - command: `rg -n "drainPendingDeliveries\(" extensions/discord extensions/whatsapp src/plugin-sdk`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Discord outbound sends are REST/token based:** Discord outbound message sending constructs a REST client from the configured bot token and uses that client in `sendMessageDiscord()`. That means the reported message loss may involve a higher-level delivery path, but the requested reconnect queue itself is still not implemented.
  - file: [extensions/discord/src/client.ts:105](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/client.ts#L105)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
