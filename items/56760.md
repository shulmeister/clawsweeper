## Summary

Current `main` already routes per-channel-peer DM sessions as real session-scoped conversations and passes those session keys through the same `before_agent_start` and `agent_end` hook paths used by other runs. The follow-up comment also reports failure with `dmScope: "main"`, which weakens the original claim that this was a per-channel-peer-only core bug.

## Evidence

- **Per-channel-peer routing is canonical and session-scoped:** Routing tests expect `dmScope: "per-channel-peer"` to resolve direct messages to keys like `agent:main:whatsapp:direct:+15551234567` and mark them `lastRoutePolicy: "session"`, not collapse them into `agent:main:main`.
  - file: [src/routing/resolve-route.test.ts:118](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/routing/resolve-route.test.ts#L118)
  - command: `nl -ba src/routing/resolve-route.test.ts | sed -n '112,138p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Runner now backfills session keys before hook consumers:** `runEmbeddedPiAgent()` resolves `sessionKey` early specifically so downstream consumers including hooks receive a non-null key.
  - file: [src/agents/pi-embedded-runner/run.ts:241](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run.ts#L241)

  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2196](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/attempt.ts#L2196)
