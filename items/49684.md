## Summary

Keep open. Current `main` does not provide strong evidence that #49684 is fixed or obsolete. The gateway server still only binds its own configured host/port, but Ollama discovery remains active on startup and can probe the local Ollama host with `/api/tags` plus bounded `/api/show` requests. I found no code path that changes OS/system proxy settings, so the reported symptom is still unexplained rather than disproven.

## Evidence

- **Gateway listens only on its own bind host/port:** The gateway HTTP server calls `httpServer.listen(port, bindHost)` and reports conflicts only for that socket. There is no neighboring logic here that touches Ollama's default port or rewrites localhost routing.
  - file: [src/gateway/server/http-listen.ts:18](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/gateway/server/http-listen.ts#L18)
  - command: `nl -ba src/gateway/server/http-listen.ts | sed -n '1,120p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **CLI proxy bootstrap is process-local Undici setup:** CLI startup calls `ensureGlobalUndiciEnvProxyDispatcher()`, which only swaps the current process's Undici global dispatcher to `EnvHttpProxyAgent` when proxy env vars are set. This is not OS proxy mutation and does not explain direct `curl` traffic by itself.
  - file: [src/cli/run-main.ts:182](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/run-main.ts#L182)
  - command: `nl -ba src/cli/run-main.ts | sed -n '165,190p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Guarded fetch uses env proxy only in explicit trusted mode:** `fetchWithSsrFGuard` uses env proxy dispatch only when `mode === TRUSTED_ENV_PROXY`; otherwise it pins direct/policy dispatchers. That means the current Ollama plugin code is not implicitly routing its own guarded requests through ambient proxy env.
  - file: [src/infra/net/fetch-guard.ts:357](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/infra/net/fetch-guard.ts#L357)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '350,390p'`

- **Ollama discovery still runs on startup and can hit the local host repeatedly:** When discovery is enabled or ambient discovery applies, `resolveOllamaDiscoveryResult` calls `buildProvider`. That code queries `/api/tags`, then enriches up to 200 models via `/api/show` at concurrency 8. This leaves a plausible startup-load interaction with a local Ollama instance, and I found no test proving gateway startup leaves Ollama fully unaffected.
