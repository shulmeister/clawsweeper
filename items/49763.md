## Summary

Keep open. Current `main` still has code paths that can drop the TUI busy spinner during a queued follow-up handoff and when `Ctrl+O` is pressed, so there is not strong evidence that issue #49763 is already fixed.

## Evidence

- **Queued follow-up still relies on a pending flag, not an active run binding:** Sending another message during an active run sets `pendingOptimisticUserMessage = true` and switches the footer to `sending`/`waiting`, but it does not bind the queued turn to a new `activeChatRunId` until a later chat event arrives.
  - file: [src/tui/tui-command-handlers.ts:589](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/tui/tui-command-handlers.ts#L589)
  - command: `nl -ba src/tui/tui-command-handlers.ts | sed -n '573,605p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **When the current active run finishes, the handler still resets to idle even if a queued optimistic message is pending:** `handleChatEvent()` computes `wasActiveRun` from the finishing run and `finalizeRun()` unconditionally calls `setActivityStatus('idle'|'error')` for that active run. There is no guard for `pendingOptimisticUserMessage`, so the spinner can disappear between the old run finishing and the queued run emitting its first event.
  - file: [src/tui/tui-event-handlers.ts:329](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/tui/tui-event-handlers.ts#L329)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '197,212p'; nl -ba src/tui/tui-event-handlers.ts | sed -n '327,379p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Ctrl+O still forces a non-busy status and stops the loader:** `editor.onCtrlO` calls `setActivityStatus('tools expanded'|'tools collapsed')`. `renderStatus()` only keeps the animated loader for `sending|waiting|streaming|running`, and otherwise stops the timers/loader and renders static text. That matches the reporter's note that `Ctrl+O` can freeze the spinner during an active run.
  - file: [src/tui/tui.ts:585](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/tui/tui.ts#L585)
  - command: `nl -ba src/tui/tui.ts | sed -n '585,727p'; nl -ba src/tui/tui.ts | sed -n '999,1003p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **No targeted regression coverage for this exact case:** The current tests cover some concurrent-run final/delta handling, but I did not find a test for preserving the spinner across queued follow-up handoff or `Ctrl+O` during a busy state.
  - file: [src/tui/tui-event-handlers.test.ts:507](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/tui/tui-event-handlers.test.ts#L507)
  - command: `rg -n 'onCtrlO|ctrl\+o|tools expanded|tools collapsed' src/tui/*.test.ts src/tui/components/*.test.ts; nl -ba src/tui/tui-event-handlers.test.ts | sed -n '507,553p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
