## Summary

Keep #70133 open. Current main still has a Feishu per-chat sequential queue whose entries only release when the queued task settles. Normal P2P messages use the base chat lane and await the full reply dispatch path before the queue advances; there is no Feishu-level timeout/release guard for a stuck dispatch. Related items in the provided context are adjacent, but they do not supersede this specific timeout/release bug.

## Best Possible Solution

Keep #70133 open. The likely fix belongs in the Feishu channel/plugin dispatch path or a shared channel reply-dispatch seam: add a bounded timeout/release guard for per-chat sequential queue tasks, propagate an abort/idle signal where feasible, and add regression coverage showing a never-settling first same-chat P2P dispatch cannot block later same-chat messages forever while normal completed dispatches still preserve ordering.

## Evidence

- **Issue has concrete reproduction and logs:** The issue body reports OpenClaw 2026.4.20 Feishu websocket P2P runs where one message logged `dispatching to agent`, later same-chat messages logged `queued`, and `dispatch complete` did not appear for several minutes or not before observation ended.
- **Sequential queue only releases on task settlement:** `createSequentialQueue` chains same-key work with `previous.then(task, task)`, stores the resulting promise, and only deletes the queue entry in `next.then(cleanup, cleanup)`. There is no timeout, abort, or forced release path in this queue.
