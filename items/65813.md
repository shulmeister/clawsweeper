## Summary

Current main implements the observable two-profile Codex rotation/cooldown behavior: failed profiles are marked into cooldown before rotation, rotation skips cooldowned profiles instead of bouncing back to the exhausted primary, successful rotated profiles are marked good/used, Codex OAuth cooldowns use usage-window-aware backoff, and expired cooldowns are cleared back into the eligible pool. The related merged #70743 is useful adjacent Codex/GPT-5.4 runtime hardening context, but the close is based on the current auth cooldown/rotation implementation and tests.

## Best Possible Solution

Keep the current shipped cooldown/rotation behavior: failed Codex profiles enter a meaningful cooldown, cooldowned profiles are skipped until expiry, healthy rotated profiles continue normally, and operators who intentionally want more same-provider attempts can tune `auth.cooldowns.rateLimitedProfileRotations` or auth order rather than adding a separate core `profile_preferred` option for this report.

## Risks / Open Questions

- The default same-provider rate-limit cap is still one rotation, so installations with more than two Codex profiles will not try every profile before cross-provider fallback unless `auth.cooldowns.rateLimitedProfileRotations` is raised. That is now documented and tunable behavior, and it does not leave the reported two-profile cooldown loop unresolved.
