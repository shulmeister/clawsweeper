## Summary

Keep open. Current main still implements a rate-limit profile-rotation cap before model fallback, and regression tests explicitly assert that a provider with three OpenAI profiles falls back after only two OpenAI attempts. That is the opposite of #65813's requested behavior of escalating only after all eligible profiles are independently in cooldown. The cross-referenced #70743 is useful context for GPT-5.4/OpenAI-Codex hardening and auth alias fixes, but it does not supersede this specific rotation-policy request.

## Best Possible Solution

Keep #65813 open for maintainer/product judgment. A concrete fix would likely change the rate-limit policy so OpenAI-Codex auto-profile rotation tries each eligible non-cooldown profile before model fallback, or adds a provider/config mode for that behavior, with a regression test covering two OpenAI-Codex OAuth profiles plus an OpenRouter fallback where the second profile remains healthy.
