## Summary

Keep open. Current main now documents the Tailscale Serve trust boundary and how to disable Tailscale header auth, but it still does not implement a server-side secondary-auth requirement that combines verified Tailscale/trusted-proxy identity with token/password.

## Evidence

- **Serve docs describe tokenless auth and only offer disable-or-use-shared-secret fallback:** The Tailscale docs say Control UI/WebSocket auth can use Tailscale identity headers without a token/password when `gateway.auth.allowTailscale` is true, and the documented hardening path is to disable `gateway.auth.allowTailscale` and use token/password instead. That is documentation and a mode switch, not a combined second factor.

## Risks / Open Questions

- The issue body conflates Tailscale Serve header auth with `gateway.auth.mode: "trusted-proxy"`; any future implementation should clarify whether the ask is limited to Serve, trusted-proxy, or both.
- Documentation has improved on current main, so the remaining work is narrower than the issue title suggests: it is now primarily a feature request for defense-in-depth, not an undisclosed behavior bug.
