## Summary

Keep open. Current main still parses `chat_id:` through `Number.parseInt(value, 10)` without first proving the whole value is decimal-only, so a hex-like identifier beginning with a digit still becomes a numeric chat id. The iMessage helper targeted by the PR now re-exports the shared plugin parser, so the PR needs rebasing to `src/channels/plugins/chat-target-prefixes.ts`, but the underlying bug and the proposed regression coverage remain useful. The linked #17362 is closed as stale in the provided context, not superseded by a shipped fix.

## Best Possible Solution

Keep PR #53316 open, but rebase/port it to the current shared parser in `src/channels/plugins/chat-target-prefixes.ts` rather than the old iMessage helper body. Add regression tests that prove non-decimal `chat_id:` values throw on explicit targets and do not match numeric `chatId` values in allowlists; consider covering the shared plugin-SDK parser so iMessage and BlueBubbles cannot drift.
