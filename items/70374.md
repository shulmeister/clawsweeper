## Summary

Keep open. Current main still does not implement PR #70374's central behavior: plugin-owned `message.send` and `message.poll` go through `executeSendAction`/`executePollAction`, whose plugin path locally dispatches actions without consulting `resolveExecutionMode` or calling gateway `message.action`. Telegram also has no current-main `send` gateway opt-in. The PR is still relevant, but the posted fallback/security review comments mean it needs maintainer review and likely revisions rather than cleanup closure.

## Best Possible Solution

Keep PR #70374 open. The best path is to implement plugin-owned `send`/`poll` gateway routing in the shared outbound send service and have Telegram opt `send` into it, while revising the PR so fallback is limited to explicit gateway-unavailable transport failures or otherwise fails closed, errors are visible, idempotency prevents duplicate sends, and focused outbound/Telegram/gateway tests cover both gateway and local paths.

## Risks / Open Questions

- The PR's broad gateway-error fallback can duplicate sends or bypass gateway validation if the gateway processed a request but the client timed out or received a non-transient error.
- Gateway URL, auth, idempotency, and least-privilege behavior should be reviewed against the existing gateway call hardening before landing this user-visible Telegram path.
