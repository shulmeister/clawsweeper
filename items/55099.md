## Summary

Keep this issue open. Current main still has the Telegram transport behavior described in the report: qualifying fetch failures can arm a sticky IPv4-only dispatcher, the fallback policy forces `family: 4`/`dnsResultOrder: "ipv4first"`, and existing tests assert sticky IPv4 reuse rather than the requested loopback-ECONNREFUSED plus IPv6-override exception. The prior Codex review remains current for the relevant files.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin transport layer. Add a focused regression test for an aggregate `fetch failed` that includes loopback IPv4 `ECONNREFUSED` and IPv6 timeout under `dnsResultOrder: "verbatim"`, then adjust `extensions/telegram/src/fetch.ts` so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when IPv6 is the configured or previously healthy route. The fix should also decide, with coverage, whether the reported `UND_ERR_SOCKET` case should still trigger sticky fallback.
