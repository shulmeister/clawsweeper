## Summary

Close as `clawhub`. Current `main` already exposes the core primitives for health checks, diagnostics, channel auto-restarts, and service restart/install/repair. The remaining proposal is an environment-specific operator watchdog around `systemctl`, disk cleanup, memory thresholds, and notification routing, which fits the repo vision for optional capability to live outside core as a ClawHub skill/plugin or external service.

## Close Comment

Thanks for the proposal. Codex review: closing this as `clawhub` rather than a core OpenClaw item.

Current `main` already exposes the core pieces an external recovery monitor would need: auth-free liveness/readiness probes in `src/gateway/server-http.ts:187`, CLI health/status/stability surfaces in `docs/cli/gateway.md:92`, built-in channel health-monitor restarts in `docs/gateway/health.md:32` and `src/gateway/server-runtime-services.ts:28`, plus doctor-driven gateway service install/start/restart repair in `src/commands/doctor-gateway-daemon-flow.ts:89`. Linux service guidance already uses `Restart=always` in `docs/platforms/linux.md:64`.

What remains in this issue is operator-specific automation around `systemctl`, sudoers, disk cleanup, memory thresholds, and notification routing (for example Discord). That is a better fit for an external watchdog, plugin, or ClawHub skill than additional OpenClaw core behavior. This also matches the current scope guidance in `VISION.md:54` and `VISION.md:68`, which says optional capability should usually ship as plugins and new skills should go to ClawHub first.

If someone wants to pursue this, the better path is a ClawHub skill/plugin or external service that consumes the existing health/status endpoints and restarts the service under the operator’s own policy.
