## Summary

Keep open. Current main has strong mitigations for cron duplicate execution and session-store growth, but one meaningful part of this issue remains: startup still reads and parses the full sessions.json before any load-time capping, so the reported unbootable gateway/OOM recovery problem is not proven fixed on main.

## Evidence

- **Startup still materializes the whole session store before maintenance:** loadSessionStore reads sessions.json with fs.readFileSync and JSON.parse before applying prune/cap maintenance. It then returns structuredClone(store), adding another full object copy. This does not satisfy the issue’s startup/OOM requirement to avoid loading the full file before recovery.
  - file: [src/config/sessions/store-load.ts:102](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/src/config/sessions/store-load.ts#L102)
  - command: `nl -ba src/config/sessions/store-load.ts | sed -n '1,260p'`
  - sha: [4069c81b1590](https://github.com/openclaw/openclaw/commit/4069c81b159087cb71f6b64676d533cf0d897d68)
- **Load-time cap exists but runs only after full parse:** The oversized-store maintenance path prunes and caps only after the file has already been read and parsed into memory, so it helps normal oversized stores but is not a streaming/lazy recovery path for a crash-looping sessions.json.
  - file: [src/config/sessions/store-load.ts:131](https://github.com/openclaw/openclaw/blob/4069c81b159087cb71f6b64676d533cf0d897d68/src/config/sessions/store-load.ts#L131)
