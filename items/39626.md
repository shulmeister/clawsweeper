## Summary

Keep open. Current main has a useful but narrow OpenAI-compatible mitigation: proxy/local `openai-completions` tool requests now default `tool_choice` to `"auto"`, which addresses one way providers emit plain-text tool directives. That does not prove the reported bare `exec{"command": ...}` leak is prevented across the shell and Telegram delivery surfaces. The shared visible-text sanitizer, final payload path, and block streaming path still cover XML-style tool-call tags and bracketed `[Tool Call: ...]` scaffolding, not bare tool-name-plus-JSON pseudo calls.

## Best Possible Solution

Keep tracking this as a behavior bug. The best maintainer path is to preserve the `tool_choice: "auto"` transport fix, then add either provider-specific root-cause handling for the reporter's model or a conservative assistant-visible sanitizer for bare tool-name-plus-JSON pseudo calls such as `exec{...}`. Add focused regression coverage for `extractAssistantVisibleText`, final reply payload construction, and block streaming used by shell/Telegram delivery.
