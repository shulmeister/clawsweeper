## Summary

Keep PR #42350 open. Current main still renders iOS/macOS assistant text only through the shared Swift markdown path and has no Adaptive Card marker parser, SwiftUI renderer, or AdaptiveCard source files wired into the app. The related standalone plugin context is useful, but the missing piece here is a native app rendering surface that cannot be satisfied by a community plugin listing alone.

## Best Possible Solution

Keep the PR open for maintainer/product review. If OpenClaw wants native Adaptive Cards in companion app chat, the work should land in the shared OpenClawKit chat UI with source-list/package wiring, focused parser/renderer tests, handling for numeric column widths and invalid image URLs, and clear fallback behavior so markers from @vikrantsingh01/openclaw-adaptive-cards render cleanly instead of leaking raw JSON. If maintainers decide Adaptive Cards should stay outside core, the missing app-renderer extension/API should be discussed explicitly rather than closing this as already handled.
