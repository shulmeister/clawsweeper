## Summary

Keep #8122 open. Current main still only supports global, per-type, and per-channel session reset policy, while the requested `channels.telegram.groups.<id>.session.reset` shape is neither accepted by Telegram group config/schema nor applied during session initialization. The related implementation PR #8134 was closed unmerged, and the broader #20888 lifecycle issue does not supersede this specific per-Telegram-group configuration request.

## Best Possible Solution

Keep the issue open. The best path is an additive, account-aware implementation that lets Telegram group or topic config override reset policy, or a generic channel group reset resolver if maintainers want this beyond Telegram. Update Telegram config types, Zod schema, generated metadata/help, public docs, and focused tests for precedence such as group/topic over channel over type/global, including multi-account behavior.

## Risks / Open Questions

- A naive implementation could repeat PR #8134's multi-account risk by resolving a group reset policy without tying it to the inbound Telegram account context.
- A Telegram-only core shortcut would weaken the plugin boundary; a generic channel group/topic reset resolver may be the cleaner product/API direction.
