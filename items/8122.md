## Summary

Keep #8122 open. Current main still only supports session reset policy at global, chat-type, or channel scope; Telegram group/topic config has no session reset field, the strict Telegram schema would reject the proposed shape, and session initialization only passes a channel-level reset override into the reset resolver. The related implementation PR #8134 was closed unmerged, and the broader #20888 lifecycle issue does not supersede this specific per-Telegram-group configuration request.

## Best Possible Solution

Keep the issue open. Implement an additive, account-aware reset policy override for Telegram group/topic config, or introduce a generic channel group/topic reset resolver if maintainers want this across plugins. Update Telegram types, Zod schema, generated config metadata/help, public docs, and focused precedence tests such as topic/group over channel over type/global, including multi-account behavior.

## Risks / Open Questions

- A naive implementation could repeat PR #8134's multi-account risk if the reset policy is resolved only by group id instead of the inbound Telegram account plus group/topic context.
- A Telegram-only implementation may be acceptable, but maintainers may prefer a generic channel group/topic reset resolver so external plugins can implement the same behavior without core-specific bypasses.
