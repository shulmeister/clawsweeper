## Summary

Keep #8122 open. Current main still does not implement per-Telegram-group or per-topic session reset configuration. Session reset policy remains scoped to global, chat type, and channel overrides; Telegram group/topic config types and strict schemas do not accept the proposed `session.reset` block; and the runtime resolver still receives only a channel-level override. The related PR #8134 was closed unmerged, and its multi-account Telegram concern remains relevant to any future implementation.

## Best Possible Solution

Keep #8122 open. Implement an additive, account-aware conversation-scoped reset override for Telegram group/topic config, or introduce a generic channel/plugin reset policy resolver if maintainers want this across plugins. Update Telegram config types, strict Zod schemas, generated config labels/help, public docs, and focused precedence tests covering topic/group over channel over type/global, including multi-account Telegram cases.

## Risks / Open Questions

- A naive implementation keyed only by Telegram group id could apply the wrong reset policy in multi-account Telegram setups; the resolver should include account or route context.
- Maintainers still need to choose whether the product surface should be Telegram-specific `groups.<id>.session.reset` and `topics.<id>.session.reset`, or a generic channel/plugin conversation-scoped reset resolver.
