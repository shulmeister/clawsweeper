## Summary

Keep open. Current main has automatic cache-TTL pruning that can soft-trim or hard-clear old tool results after a TTL, plus informational `/context` reporting, but it does not provide the requested manual per-tool-result purge, agent-accessible clear function, or `clearAfterUse` tool-call flag. The linked #45522 is related background about state/checkpointing, not a canonical tracker for this specific manual clearing request.

## Evidence

- **Cache-TTL gate remains automatic:** The context-pruning extension only prunes when `mode === "cache-ttl"` and the TTL has elapsed since `lastCacheTouchAt`; otherwise it returns without changing messages. This does not offer immediate/manual clearing after a result has been consumed.

  - sha: [28497515fe1e](https://github.com/openclaw/openclaw/commit/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071)
- **Config has no manual-clear mode or flag:** The public context pruning config exposes only `mode?: "off" | "cache-ttl"`, TTL, ratios, size thresholds, soft-trim/hard-clear settings, and tool allow/deny matching. There is no per-result id purge, `clearAfterUse`, recover/archive knob, or agent-callable clear setting.

## Risks / Open Questions

- Automatic cache-TTL pruning is a meaningful mitigation, so implementation design should account for how manual clearing interacts with cache checkpoints, recovery, and compaction. It still does not satisfy the explicit manual/immediate purge request.
