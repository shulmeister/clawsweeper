## Summary

Keep this issue open. Current main includes the partial #65365 startup-ordering fix, but cron activation and hot reload still do not wait for connected/healthy channel readiness, so the reporter's SIGUSR1/config.patch and Discord WS readiness concern remains actionable.

## Best Possible Solution

Implement a bounded cron startup and reload readiness gate at the gateway/cron boundary. It should delay automatic cron timer execution and startup catch-up until configured delivery channels are connected/healthy, a configurable wait such as cron.channelReadyWaitMs expires, or readiness fails with clear diagnostics. Cover full startup, SIGUSR1/config.patch hot reload, Discord-style async readiness, and existing /readyz/channel-health behavior with targeted gateway and cron tests.

## Evidence

- **Reporter narrowed the bug:** The provided comments refine the original session-store theory into two readiness problems: cron can fire before Discord WS is connected, and SIGUSR1/config.patch restarts can re-arm cron before the full-restart timing that masked the problem. The requested fix is a channel-readiness gate or configurable wait before cron execution.
- **Startup only gates scheduled services on post-attach sidecars:** Current main awaits post-attach runtime, marks startup ready, then calls activateGatewayScheduledServices. This preserves the partial #65365 behavior but does not itself prove provider-level channel readiness.
  - file: [src/gateway/server.impl.ts:830](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/src/gateway/server.impl.ts#L830)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)
- **Cron activation has no readiness dependency:** activateGatewayScheduledServices starts the heartbeat runner, calls startGatewayCronWithLogging immediately, and starts delivery recovery without taking a ReadinessChecker, channel manager status, or configurable channel-ready wait parameter.
  - file: [src/gateway/server-runtime-services.ts:116](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/src/gateway/server-runtime-services.ts#L116)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)
- **Cron start can execute startup catch-up immediately:** CronService.start runs missed jobs before arming the regular timer, so starting cron is itself an execution point, not just passive scheduling.
  - file: [src/cron/service/ops.ts:136](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/src/cron/service/ops.ts#L136)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)
- **Channel startup is a handoff, not connected readiness:** startChannelInternal marks an account running, hands off startAccount asynchronously, and resolves the startup gate in finally. startChannels therefore can complete before a provider reports connected/READY.
  - file: [src/gateway/server-channels.ts:432](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/src/gateway/server-channels.ts#L432)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)
- **Discord connected readiness is later provider state:** Discord readiness waits until gateway.isConnected becomes true and then pushes the connected status patch, which is separate from the core startChannels handoff.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:288](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/extensions/discord/src/monitor/provider.lifecycle.ts#L288)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)

  - file: [src/gateway/server-reload-handlers.ts:245](https://github.com/openclaw/openclaw/blob/9626ef274ae2346b3d519f99c31856be98aab570/src/gateway/server-reload-handlers.ts#L245)
  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)

  - sha: [9626ef274ae2](https://github.com/openclaw/openclaw/commit/9626ef274ae2346b3d519f99c31856be98aab570)

## Risks / Open Questions

- The exact multi-minute Discord timing is environment-dependent, but the current source still permits cron to start before connected channel readiness.
- A fix needs bounded semantics so webhook-only or no-channel cron jobs are not blocked indefinitely by optional channel health.
