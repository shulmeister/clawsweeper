## Summary

Keep #10253 open. Current main still has no hook/webhook-specific session lifecycle policy: hook agent dispatch creates isolated cron-style runs under hook session keys, the hooks config/schema/payload surfaces do not expose cleanup/archive/transcript-retention controls, and existing cleanup mechanisms cover subagents, broad session maintenance, or canonical cron run-session aliases rather than `hook:gmail:*` sessions. The prior steipete/Codex review also explicitly kept this open, and the relevant files have not changed since that review SHA.

## Best Possible Solution

Keep #10253 open. The best maintainer path is a core hook-session lifecycle policy under `hooks`, with config types/schema/help/docs, dispatch/finalization wiring for `/hooks/agent` plus mapped/Gmail hooks, explicit archive/delete/transcript-retention semantics, and focused tests for Gmail per-message keys, delayed cleanup, sessions_list visibility, and interaction with existing session and cron retention settings.
