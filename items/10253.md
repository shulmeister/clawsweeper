## Summary

Keep #10253 open. Current main still does not provide a hook/webhook-specific lifecycle policy for on-complete archive/delete, delayed cleanup, transcript retention, or completed-hook visibility. The existing maintainer/Codex comment also kept it open, and the current code still routes hook agent work through isolated cron-style execution while persisting non-cron hook keys as ordinary agent-scoped session rows.

## Best Possible Solution

Keep #10253 open. The best path is a core hooks.sessionCleanup policy with typed config, strict schema/help/docs, and runtime finalization for direct /hooks/agent, mapped hooks, and Gmail preset runs. It should define archive/delete/no-op modes, delayed cleanup, keepTranscript behavior, completed-hook visibility in sessions_list, generated hook:* and agent:<id>:hook:gmail:* key handling, and focused tests around persistence, transcript rename/delete behavior, and precedence with session.maintenance and cron.sessionRetention.

## Risks / Open Questions

- Generic session.maintenance can eventually prune old session rows by age/count, and cron.sessionRetention handles cron run-session aliases, but neither defines hook-specific on-complete/delayed behavior or transcript-retention semantics.
- A fix needs explicit precedence with session.maintenance and cron.sessionRetention so hook cleanup cannot accidentally remove unrelated sessions or audit transcripts.
