## Summary

Keep open. Current main still lacks a configurable lifecycle policy for hook/webhook sessions. The hook config/schema has no cleanup/archive/transcript-retention surface, hook dispatch creates one-shot isolated cron runs without `deleteAfterRun` or another cleanup policy, isolated hook runs still force fresh persisted sessions, and the existing `archiveAfterMinutes` behavior remains scoped to subagents.

## Best Possible Solution

Keep this issue open. The product fix belongs in the core hook/cron session lifecycle: add a maintainer-approved `hooks` cleanup policy surface, validate and document it, wire it through hook dispatch and isolated cron finalization, decide transcript retention semantics, and add focused tests for Gmail/per-hook session keys, delayed cleanup, `deleteAfterRun` interaction, and `sessions.list` visibility/status behavior.
