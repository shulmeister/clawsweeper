## Summary

Keep #7406 open. Current main only partially addresses Telegram topic names: it learns and persists them for inbound context, prompt metadata, and hooks, but does not feed them into session label/displayName, sessions.list rows, or the Control UI session dropdown. The implementation PRs referenced by the issue are closed and unmerged, while duplicate reports were closed back to #7406 as the canonical tracker.

## Best Possible Solution

Keep #7406 open as the canonical implementation tracker. The best fix belongs in the bundled Telegram/session metadata path: map cached Telegram TopicName into persisted SessionEntry display metadata, likely via ThreadLabel, displayName, label, or a dedicated topicName field during inbound session metadata recording; ensure sessions.list and the Control UI session selector render stable human-readable group/topic labels with key/id fallback; optionally add a schema-backed per-topic name fallback; and add regression coverage for Telegram context extraction, session metadata persistence, gateway session rows, and Control UI label resolution.
