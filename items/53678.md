## Summary

Keep open. Current main still preserves and exposes the source-channel timestamp, but it does not provide a generic Gateway-received timestamp in the shared inbound context, plugin SDK inbound helpers, hook payloads, or agent-visible inbound metadata. Telegram has a local `receivedAtMs` path for buffering/debug timing, and QQ has command-local timing, but those are not the cross-channel metadata field requested here.

## Best Possible Solution

Add an additive typed receipt timestamp, preferably `GatewayReceivedAtMs` or `ReceivedAtMs`, to the shared inbound context while preserving `Timestamp` as the source-channel timestamp. Set it once at channel ingress or in shared inbound dispatch helpers, propagate it through public SDK helpers such as `direct-dm`, `inbound-envelope`, and reply dispatch paths, expose it to hooks if hooks should measure latency, render it in agent-visible inbound metadata as a stable ISO timestamp, and add focused tests for the metadata builder plus at least one long-poll/debounce-style channel path.
