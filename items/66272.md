## Summary

Keep open. Current main has not implemented the PR's user-visible fix: the security audit still emits `gateway.trusted_proxies_missing` specifically for `gateway.bind: "loopback"`, and the current test suite still asserts that behavior. The PR is also not ready as-is because the review discussion correctly points out that `bind !== "loopback"` is only a mode check and can misclassify effective loopback binds such as `custom` with `customBindHost: "127.0.0.1"` and some `auto` resolutions.

## Best Possible Solution

Keep the PR open and ask for a revised implementation that bases `gateway.trusted_proxies_missing` on effective gateway exposure, not only the raw `gateway.bind` mode. The fix should stop warning for local-only loopback-resolving binds, still warn for genuinely exposed reverse-proxy cases, and add regression tests for plain loopback, `lan`, custom loopback, and an exposed custom/auto path.
