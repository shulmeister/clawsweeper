## Summary

Keep open. Current main still appears to drop Feishu message-tool `replyTo` on ordinary `send`: generic outbound code resolves `replyTo`, and lower Feishu send primitives can perform replies, but the Feishu plugin action handler only assigns `replyToMessageId` for `thread-reply`, not `send`.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin owner module. `extensions/feishu/src/channel.ts` should treat explicit `replyTo`/`reply_to` on `action: "send"` as a normal inline `replyToMessageId`, pass it through text, card, and media sends, and preserve `thread-reply` as the explicit `reply_in_thread` path. Add focused Feishu channel tests for message-tool `send` with explicit `replyTo` across text, presentation/card, and media sends, plus a guard that media-only and text sends do not require a card payload.

## Evidence

- **Issue context:** The reporter describes Feishu WebSocket replies using the message tool `replyTo` sometimes not rendering quoted replies after OpenClaw 2026.3.23-2, and another user corroborates that the behavior is unstable outside private chats.
