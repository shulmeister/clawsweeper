## Summary

Keep open. Current `main` still has separate tracing/task/event shapes rather than one canonical run-trace schema spanning agent runs, ACP relay, subagents, and task flows. The issue is also only 8 days old and contains a concrete, actionable design request, so it is not a stale-cleanup candidate.

## Evidence

- **Agent trace base is minimal and lacks parent/step/task linkage:** `AgentTraceBase` only carries `runId`, session identifiers, provider/model, and workspace metadata. It does not define the requested cross-flow fields such as `parentRunId`, `taskId`, `stepId`, `eventType`, or `handoffType`.
  - file: [src/agents/trace-base.ts:1](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/trace-base.ts#L1)
  - command: `nl -ba src/agents/trace-base.ts | sed -n '1,40p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Embedded cache tracing uses its own stage-oriented schema:** `CacheTraceEvent` is a cache/session/prompt JSONL format keyed by `stage`; it is not a canonical run timeline schema and is limited to cache-related stages rather than shared agent/ACP/task flow events.
  - file: [src/agents/cache-trace.ts:13](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/cache-trace.ts#L13)
  - command: `nl -ba src/agents/cache-trace.ts | sed -n '13,60p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **ACP parent-child relay logs use a different ad-hoc event shape:** ACP spawn relay logging writes JSON lines with `runId`, `parentSessionKey`, `childSessionKey`, `agentId`, and `kind`. That is a separate schema from both diagnostic events and task records, and it still does not provide the requested canonical linkage fields.
  - file: [src/agents/acp-spawn-parent-stream.ts:167](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/acp-spawn-parent-stream.ts#L167)

- **Task execution state is modeled separately from tracing:** Detached task/task-flow state lives in `TaskRecord` and `TaskEventRecord` with fields such as `taskId`, `parentTaskId`, `runId`, `status`, and summaries. This is useful runtime state, but it is not the unified trace event schema requested in the issue.

That confirms tracing exists, but remains fragmented across subsystems.
