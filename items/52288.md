## Summary

Keep open. Current `main` still stores hosted client tool calls as a single value, converts that to at most one `pendingToolCalls` entry, and both `/v1/responses` handlers emit only `pendingToolCalls[0]`. I found no local evidence that PR #52287 or an equivalent fix landed.

## Evidence

- **Single detected client tool call is overwritten:** `src/agents/pi-embedded-runner/run/attempt.ts` uses one mutable `clientToolCallDetected` variable and overwrites it on every `onClientToolCall` callback, so only the last call survives the attempt.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1148](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/pi-embedded-runner/run/attempt.ts#L1148)

- **Attempt result type is singular:** `src/agents/pi-embedded-runner/run/types.ts` still exposes a singular `clientToolCall?: { ... }` field rather than an array, matching the issue's reported bottleneck.
  - file: [src/agents/pi-embedded-runner/run/types.ts:101](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/pi-embedded-runner/run/types.ts#L101)
