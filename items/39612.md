## Summary

Keep open. Current main still intentionally exposes sanitized provider/runner failure details to external direct chats when `/verbose` is `on` or `full`, which conflicts with the issue’s central request to never send raw LLM/API failure details through WhatsApp, Telegram, or similar user-facing channels. The newer thinking-block transcript hygiene helps part of the reported Anthropic failure class, but it does not resolve the external-channel failure-detail policy.

## Best Possible Solution

Keep this open for an explicit maintainer decision on external-channel failure policy. A likely core fix is to keep generic default direct-chat copy and silent group/channel behavior, but stop sending provider/runner exception details to external chat users even under `/verbose`; reserve details for logs, Control UI, or operator-only surfaces. Add regression coverage for WhatsApp and Telegram direct verbose LLM failures. Treat multi-agent escalation and deeper session migration/recovery as separate follow-ups if maintainers want those parts in scope.
