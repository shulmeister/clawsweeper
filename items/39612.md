## Summary

Keep open. Current main only partially addresses the issue: normal external direct-chat failures get generic copy and group/channel failures can be silenced, but external direct chats with `/verbose on` or `/verbose full` still intentionally receive provider/runner error details. The issue asks for a framework-level policy to never expose raw LLM/API errors to users, and current docs/tests preserve the remaining verbose-detail behavior.

## Best Possible Solution

Keep the issue open for an explicit maintainer product/security decision on external-channel failure policy. A likely implementation is to keep generic default and silent group/channel behavior, but stop forwarding provider/runner exception detail to external chat users even under `/verbose`, or restrict verbose details to operator/internal surfaces. Add regression coverage for WhatsApp/Telegram direct verbose failures; track multi-agent auto-escalation and session migration as separate follow-ups if they remain in scope.
