## Summary

Current main already supports Telegram user interrupts for in-flight runs: Telegram routes `/stop` and standalone abort phrases onto a separate control lane, the shared abort handler cancels the targeted session and clears queued followups, and there are tests covering native Telegram stop targeting plus docs telling users how to cancel active work.

## Evidence

- **Telegram abort requests bypass the normal per-chat queue:** `getTelegramSequentialKey()` sends `/stop` and natural-language abort requests to `telegram:<chat>:control` instead of the normal session lane, which is the mechanism needed to avoid waiting behind the active long-running turn.

## Close Comment

Closing as implemented on main. Codex review checked current OpenClaw code and found that Telegram already routes `/stop` and standalone abort phrases onto a separate control lane instead of the normal per-chat queue (`extensions/telegram/src/sequential-key.ts:66`, `extensions/telegram/src/sequential-key.test.ts:58`). From there, the shared stop handler aborts the targeted session and clears queued followups (`src/auto-reply/reply/commands-session-abort.ts:129`), and there is explicit coverage for a native Telegram `/stop` aborting the target session and draining queued followups (`src/auto-reply/reply.triggers.trigger-handling.targets-active-session-native-stop.e2e.test.ts:575`). User docs also already document the cancel flow (`docs/help/faq.md:1883`, `docs/web/control-ui.md:170`). This support is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
