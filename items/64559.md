## Summary

Keep open. Current main has related hook emission and WhatsApp opt-in support, but the PR is not obsolete with high confidence: the author’s latest PR comment says the fix was incomplete, the linked bug remains open, and current source/tests do not prove the requested queued/in-flight inbound behavior is fully covered.

## Best Possible Solution

Keep the PR open or supersede it with a maintainer-owned/canonical fix for #64525. The best path is a shared ingestion-layer hook helper used before any dispatch-vs-enqueue or fast-drain bypass, while preserving channel privacy policy such as WhatsApp’s explicit opt-in, plus focused regression tests proving the hook fires for queued/in-flight messages and before the reply resolver/enqueue decision.

## Evidence

- **PR discussion says this fix is incomplete:** The latest author follow-up on PR #64559 reports real WhatsApp testing still missed a significant portion of inbound `message_received` / `message:received` hooks and says they were investigating `followup-runner.ts` / `enqueueFollowupRun` for a complementary fix.
- **Current generic hook still lives inside dispatchReplyFromConfig:** Current main emits the plugin `message_received` hook and internal `message:received` hook from `dispatchReplyFromConfig`, after plugin-owned binding early returns and before the reply resolver. That is related progress, but it is still dispatch-layer coupling rather than a clearly shared ingestion-layer helper.
