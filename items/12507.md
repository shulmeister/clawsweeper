## Summary

Keep #12507 open. Current main still does not implement native SKILL.md integrity/signature verification: ClawHub skill detail/version metadata lacks trusted digest/signature fields, native skill install downloads and extracts archives without comparing against trusted metadata, persisted .clawhub origin/lock records only version/time, and runtime skill loading/snapshots/status carry no verification state or unsigned capability tier. Existing plugin/npm integrity handling is adjacent but does not cover the separate `openclaw skills install` path.

## Best Possible Solution

Keep #12507 open as the canonical native skill integrity/signature work. A complete solution should add trusted digest/signature/publisher metadata to ClawHub skill/version APIs, verify native skill archives before extraction, persist expected integrity/signature state in `.clawhub` origin and lock metadata, reverify SKILL.md or full skill contents during skill snapshot/load, expose verification state in CLI/UI/status surfaces, and document/test the unsigned or unverified skill policy and capability restrictions.

## Risks / Open Questions

- Unsigned/degraded-mode behavior needs explicit maintainer security/product policy because it affects skill visibility, capability access, and operator trust semantics.
- Adjacent plugin/npm integrity paths may look related, but they do not cover the separate native `openclaw skills install` flow.
