## Summary

Current main already routes isolated cron runs through the shared model-fallback path, and the shipped v2026.4.22 release explicitly includes the timeout-to-fallback fix for transport failures. The reported missing fallback behavior is implemented on current main.

## Risks / Open Questions

- This was a read-only code/history review; I did not perform a live provider repro against OpenRouter, so the conclusion is based on the current code path and shipped release notes rather than an end-to-end network run.
- The original report also speculated about why no request reached OpenRouter. This review only found strong evidence that the missing fallback handoff is implemented; it does not independently prove the underlying upstream timeout cause is gone in every environment.

## Close Comment

Closing from a Codex review because current `main` already implements the missing fallback handoff for isolated cron runs. Isolated cron execution now goes through the shared model fallback wrapper in `src/cron/isolated-agent/run-executor.ts:109`, and that fallback loop continues past timeout/transport failures to the next configured candidate in `src/agents/model-fallback.ts:865`. There is direct regression coverage for a timeout on the first model falling through to the next model in `src/agents/model-fallback.test.ts:1443`, and the shipped `v2026.4.22` changelog explicitly calls out the timeout-classification fix in `CHANGELOG.md:160`. If this still reproduces on `v2026.4.22` or newer, please open a new issue with fresh logs and exact provider/model details, because that would be a new bug beyond the missing fallback behavior reviewed here.
