## Summary

Keep open. Current `main` still does not implement the Mattermost thread-context propagation and implicit thread inheritance this PR adds, and I did not find an equivalent fix elsewhere in the outbound runner or plugin adapter.

## Evidence

- **Mattermost send action still ignores tool context:** `extensions/mattermost/src/channel.ts:113` defines `handleAction` without `toolContext`, and `extensions/mattermost/src/channel.ts:187-201` only derives `replyToId` from explicit `params.replyToId` / `params.replyTo` before calling `sendMessageMattermost`. There is no fallback to session thread context on current `main`.
  - file: [extensions/mattermost/src/channel.ts:113](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/mattermost/src/channel.ts#L113)
  - command: `nl -ba extensions/mattermost/src/channel.ts | sed -n '100,230p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Mattermost threading adapter lacks the needed hook:** `extensions/mattermost/src/channel.ts:412-430` exposes `scopedAccountReplyToMode` and `resolveReplyTransport`, but no `threading.buildToolContext` and no `resolveAutoThreadId`. That means the generic runner never receives Mattermost-specific `currentThreadTs` / `replyToMode` context.
  - file: [extensions/mattermost/src/channel.ts:412](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/mattermost/src/channel.ts#L412)
  - command: `nl -ba extensions/mattermost/src/channel.ts | sed -n '404,455p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Generic fallback drops thread metadata when a plugin has no buildToolContext:** `src/auto-reply/reply/agent-runner-utils.ts:114-120` falls back to a minimal tool context containing `currentChannelId`, `currentChannelProvider`, `currentMessageId`, and `hasRepliedRef` when `threading.buildToolContext` is absent. It does not populate `currentThreadTs` or `replyToMode`.
  - file: [src/auto-reply/reply/agent-runner-utils.ts:114](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/auto-reply/reply/agent-runner-utils.ts#L114)
  - command: `nl -ba src/auto-reply/reply/agent-runner-utils.ts | sed -n '100,150p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Outbound threading machinery needs reply mode/thread context that Mattermost does not supply:** `src/infra/outbound/message-action-threading.ts:65-112` only auto-injects inherited `replyTo` when `toolContext.replyToMode` allows it, and the shared tests in `src/infra/outbound/message-action-runner.threading.test.ts:169-247` cover that behavior using `currentMessageId` plus `replyToMode`. Current Mattermost never supplies that context, so this generic path does not make the PR obsolete.
  - file: [src/infra/outbound/message-action-threading.ts:65](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/infra/outbound/message-action-threading.ts#L65)

The gap is specifically that current `main` does not propagate or consume it for message-tool sends.
