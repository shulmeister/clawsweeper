## Summary

Current main already addresses the reported failure mode: subagent completion announce cleanup is detached from the run-completion path, and announce delivery now treats `Unknown channel`-style failures as permanent instead of retrying them through the long transient backoff loop. The same implementation is present in the shipped `v2026.4.22` tree.

## Evidence

- **Announce cleanup is no longer awaited in the completion path:** `completeSubagentRun()` triggers `startSubagentAnnounceCleanupFlow()` and does not await announce delivery; `startSubagentAnnounceCleanupFlow()` launches `runSubagentAnnounceFlow(...)` with `void ... .then(...).catch(...)`, so the parent session lane is not held open by announce retries.
  - file: [src/agents/subagent-registry-lifecycle.ts:552](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/subagent-registry-lifecycle.ts#L552)
  - command: `nl -ba src/agents/subagent-registry-lifecycle.ts | sed -n '552,616p'; nl -ba src/agents/subagent-registry-lifecycle.ts | sed -n '717,736p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`Unknown channel` is classified as permanent for announce delivery:** The permanent announce-delivery error patterns include `unknown channel`, and the retry loop only continues when the error is classified as transient. That prevents the old retry/backoff path from running on this class of channel failure.

## Close Comment

Closing from a Codex review: current main already implements this well enough. The completion cleanup path now starts subagent announce delivery in the background instead of awaiting it in the parent session lane (`src/agents/subagent-registry-lifecycle.ts:552`, `src/agents/subagent-registry-lifecycle.ts:586`, `src/agents/subagent-registry-lifecycle.ts:735`). Announce delivery also treats `Unknown channel` as a permanent error instead of retrying it through the long transient backoff loop (`src/agents/subagent-announce-delivery.ts:137`, `src/agents/subagent-announce-delivery.ts:200`). The same implementation is present in the shipped `v2026.4.22` tree. The later issue follow-up also reports that the underlying Telegram channel-resolution failure stopped reproducing after upgrading to `2026.3.24`. If this still reproduces on a newer build, please open a fresh issue with current logs and exact channel/plugin state.
