## Summary

Keep #13948 open. Current main still enforces tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed tools whose `action` is read only inside each tool's `execute` handler after policy filtering. The provided thread context also has a same-day maintainer/Codex comment explicitly keeping this as the canonical tracker, with related action-level gateway reports #54260 and #46684 closed back toward this issue.

## Best Possible Solution

Keep #13948 open as the canonical tracker. The best implementation path is to add an action-aware policy contract to config types/schema/docs, extend the shared policy pipeline so decisions can inspect multiplexed tool actions before execution, define precedence across existing policy scopes, and add regression tests proving `message:send` or `gateway:config.patch` can be denied while read-only actions such as `message:read` and `gateway:config.schema.lookup` remain available. The related gateway action-level requests should continue to point here rather than replacing it.
