## Summary

Keep #13948 open. Current main still models tool policy at whole-tool granularity, while the shared `message` tool exposes many actions through one tool name. Existing plugin hooks can block calls by inspecting params, but the requested core/config-level action deny syntax is not implemented, and the Lobster/openclaw.invoke safety use case remains a meaningful core policy request rather than obsolete cleanup.

## Evidence

- **Tool policy schema has only whole-tool allow/deny fields:** The shared config fields for tool policy are `profile`, `allow`, `alsoAllow`, `deny`, and `byProvider`; there is no `denyActions`, `allowActions`, or action-qualified policy field. The message-specific config covers cross-context and broadcast settings, not send/read/search action policy.
