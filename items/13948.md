## Summary

Keep #13948 open. Current main still implements tool policy as whole-tool allow/deny, while `message` remains one multiplexed tool whose `action` parameter includes `send`, `read`, `react`, `search`, `member-info`, and other actions. The issue is also the canonical tracker referenced by related ClawSweeper reports, and the existing maintainer-thread comment explicitly kept it open.

## Best Possible Solution

Keep #13948 open as the canonical action-level tool-policy tracker. The best implementation path is to extend the core config schema/types/docs and policy evaluator with action-aware matching for multiplexed tools such as `message` and `gateway`, define deny precedence and `/tools/invoke`/Lobster behavior explicitly, and add regression tests proving `message:send` can be denied while `message:read` remains available.
