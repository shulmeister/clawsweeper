## Summary

Keep #13948 open. Current main still implements tool policy at whole-tool granularity only; `message` and `gateway` are still multiplexed tools whose `action` is read inside `execute` after the shared policy pipeline has filtered only by tool name. The provided related-item context also shows #46684 and #54260 were closed back toward #13948, so this remains the canonical tracker rather than a duplicate.

## Best Possible Solution

Keep #13948 open as the canonical core action-level tool policy tracker. The best path is to add a typed and documented action-aware policy contract, enforce it in the shared runtime before multiplexed tool actions execute, define precedence and bypass/approval semantics, and add regression coverage for cases such as `message:send` denied while `message:read` remains available and `gateway:config.patch` denied while `gateway:config.schema.lookup` remains available.

## Risks / Open Questions

- Maintainer design is still needed for action grammar, wildcard behavior, deny/allow precedence across global/provider/agent/group/subagent scopes, and whether privileged approval workflows such as Lobster may bypass action denies.
- Plugin `before_tool_call` hooks can inspect params and provide custom stopgaps, but they are not the typed, documented config-level action policy requested here.
