## Summary

Keep #13948 open. Current main still exposes tool policy as whole-tool allow/deny, while the shared `message` tool multiplexes `send`, `read`, `search`, `react`, `member-info`, and other actions through one tool name. The requested core/config action-level deny behavior is not implemented, and the Lobster `openclaw.invoke` use case remains governed by the same whole-tool policy path.

## Best Possible Solution

Keep the issue open as the canonical action-level tool-policy tracker. The strongest implementation path is to extend the core config/schema/types/docs and policy evaluator with action-aware matching for multiplexed tools such as `message` and `gateway`, define deny precedence and `/tools/invoke`/Lobster behavior explicitly, and add regression tests showing `message:send` can be denied while `message:read` remains available. If maintainers choose a plugin-only direction instead, document the supported hook/context contract and any privileged workflow distinction needed for Lobster-style approvals.
