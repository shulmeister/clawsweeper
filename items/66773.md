## Summary

Keep open. Current main still returns the freshly rotated plaintext device token from `device.token.rotate`, and existing tests, Control UI code, CLI behavior, and docs still treat that token as the expected response payload. The linked fix PR #66794 was closed unmerged, so it does not supersede or implement the issue.

## Best Possible Solution

Fix the gateway response contract so normal `device.token.rotate` does not return the raw bearer token, and update the dependent Control UI, CLI, protocol docs, and regression tests together. If token display is still required for provisioning, move it to an explicit one-time bootstrap/export path with clear handling expectations and tests proving the normal RPC response cannot echo `entry.token`.

## Risks / Open Questions

- A one-line redaction would break or corrupt current clients: the Control UI would store/prompt the replacement string as if it were a usable token unless the client contract changes too.
- The public CLI/docs currently promise token output, so the fix needs a deliberate replacement flow or contract update rather than only changing the gateway payload.
