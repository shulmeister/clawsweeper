## Summary

Keep open. Current main proves only the lower-level unwrapped FS tool path can read outside the workspace when `tools.fs.workspaceOnly=false`; it does not prove the direct-chat/session tool-host mismatch from #70795 is fixed. The active source still centers read path validation on a single workspace/sandbox root, and there is no direct-session regression test for an approved non-workspace root such as config or installed skill directories.

## Best Possible Solution

Keep #70795 open. The best path is to trace the topmost direct-session `read` provider/tool-host path, make it consume the same effective FS policy as OpenClaw's runtime, define any approved readonly roots explicitly, and add a regression test that exercises a direct-session `read` against an approved non-workspace root while preserving workspace-only denial behavior.
