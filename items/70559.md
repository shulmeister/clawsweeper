## Summary

Keep #70559 open. Current main still has the reported unsafe reindex/vector-table failure shape: `runUnsafeReindex()` resets the index, `resetIndex()` drops `chunks_vec` while leaving `vector.available` intact, and the subsequent sync setup prepares `DELETE FROM chunks_vec ...` before any later vector-table recreation path can run. The related #71611 is an open safe-path Windows EBUSY PR and does not supersede this exact `runUnsafeReindex`/`sqlite-vec` crash on main.

## Best Possible Solution

Keep the issue open. The likely maintainer path is a focused memory-core fix that keeps `chunks_vec` schema-valid during `resetIndex()` or defensively handles missing vector tables before preparing stale-row vector DELETE statements, plus a regression test for unsafe full reindex with vector enabled/available. Track #71611 separately for the Windows safe-path EBUSY fallback; it should not be treated as the canonical fix for this exact unsafe sqlite-vec failure unless it lands and maintainers intentionally retire/support-gate the unsafe path.
