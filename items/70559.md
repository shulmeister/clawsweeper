## Summary

Keep #70559 open. Current main still has the source-level failure path described in the issue: the unsafe full reindex can run after sqlite-vec has been marked available, then `resetIndex()` drops `chunks_vec` without clearing `vector.available`, and the next sync prepares `DELETE FROM chunks_vec ...` before any vector table recreation. The related open PR #71611 addresses the safe atomic-reindex Windows rename/EBUSY path, not this unsafe sqlite-vec table-missing path on main.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-core`: make unsafe full reindex leave vector state and the `chunks_vec` schema consistent, or reset vector availability so sync does not prepare SQL against a dropped table. Add a focused regression test that simulates sqlite-vec being available with an existing vector table, runs the unsafe full reindex path, and verifies it does not throw before vectors are recreated. Coordinate with #71611 only for the separate Windows safe-reindex EBUSY problem.
