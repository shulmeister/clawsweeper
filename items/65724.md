## Summary

Current main already routes the shared Windows child-process execution path away from Node's deprecated `spawn(..., args, { shell: true })` form. The shared helper returns `false` for shell spawning, wraps Windows `.cmd`/`.bat` shims through explicit `cmd.exe` argv instead, and the changelog records the matching Windows shell-invocation fix in `v2026.2.14`. A later duplicate DEP0190 report, #69142, was already reviewed and closed on the same implemented-on-main basis.

## Best Possible Solution

Close this as already implemented. Keep the current shared Windows exec behavior; if DEP0190 still appears, the next actionable path is a new narrow report with `node --trace-deprecation`, the exact OpenClaw command, and the log source so maintainers can identify any non-shared process path still using shell mode.

## Evidence

- **Shared Windows exec path disables shell mode:** `resolveChildProcessInvocation()` rewrites Windows shim execution to `cmd.exe /d /s /c ...`, while `shouldSpawnWithShell()` is hard-coded to return `false`; `runCommandWithTimeout()` only adds `{ shell: true }` if that helper returns true.
