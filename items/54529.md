## Summary

Keep open. Current main exposes Telegram `apiRoot`, but the reported migration bug remains: update offsets are still account/bot scoped rather than endpoint scoped, lower local Bot API update IDs can still be skipped before dispatch, and `deleteWebhook` 400 `Logged out` is still treated as fatal rather than a narrow migration/cooldown condition.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin: persist a normalized `apiRoot` or endpoint fingerprint with update-offset state, ignore/reset stored offsets when the endpoint changes, handle Telegram `400: Logged out` during local Bot API migration without crash-looping, and add focused regression tests plus local Bot API migration docs.
