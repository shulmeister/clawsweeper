## Summary

Keep open. Current main still has close-blocking evidence for the reported false-positive plugin config validation path: plugin config is documented and typed under `plugins.entries.<id>.config`, but the loader still validates `entry?.config` after converting missing config to `{}` and logs `[plugins] <id> invalid config` from both normal and CLI metadata paths. The related #70811 work is an open implementation candidate, not merged, and the provided #70658 context explicitly treats this issue as related but separate.

## Best Possible Solution

Keep this issue open. The best path is a targeted loader fix, or an expanded/merged #70811, that defers schema validation only for non-activating paths where plugin config is genuinely unavailable, while keeping real activation and explicit config validation strict. Add a regression test with a strict-schema plugin configured under `plugins.entries.<id>.config` that loads without repeated false `invalid config` logs.
