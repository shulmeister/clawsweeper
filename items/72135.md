## Summary

Keep open. Current main still classifies a trailing `isError` payload as fatal when no later non-error text exists, and that directly records the cron run as `status: "error"`. I found no `DONE_OK`/`CRON_OK` handling in current source, docs, or resolver tests, so this draft PR is not implemented, duplicated, superseded, or ClawHub-scoped.

## Best Possible Solution

Keep the PR open for maintainer review. The likely path is to implement sentinel-aware classification in `resolveCronPayloadOutcome`, preserve `runLevelError` precedence, add regression coverage for final-text and payload-text sentinels plus no-sentinel/run-level-error cases, and decide whether the token should remain `===DONE_OK===`, be renamed, or be exposed as an observable outcome flag.
