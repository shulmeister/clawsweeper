## Summary

Keep this PR open. Current main still reads the first existing transcript candidate and extracts usage without validating that the transcript header id matches the requested session id. The PR remains a focused implementation candidate, with only a minor test fixture style nit noted in review context.

## Best Possible Solution

Keep this PR open and review or land an equivalent fix on main: readLatestSessionUsageFromTranscript should return null when a discovered transcript has a parseable { type: "session", id } header that differs from the requested session id, while preserving legacy/custom transcripts without a parseable session header. Add the regression test with current fixture style.
