## Summary

Keep PR #69954 open. Current main still treats a structurally valid but missing persisted `sessionFile` as authoritative during `sessions cleanup --fix-missing`, so it can prune the store entry without checking the canonical `<sessionId>.jsonl` path. The PR addresses a real remaining bug from #50248, but the existing bot review flags a small JSON summary accuracy issue (`wouldMutate` for repair-only cleanup) that should be handled before merge rather than closing the PR as obsolete.

## Best Possible Solution

Keep the PR open and ask for a small follow-up on the existing branch: have the cleanup helper report both pruned entries and repaired transcript paths, update `wouldMutate`/JSON summary behavior for repair-only dry-runs and applied runs, and keep the stale-sessionFile canonical fallback regression tests. This is better than closing because current main still has the #50248 cleanup safety bug.
