## Summary

Keep open. Current main appears to fix the related #60299 stale-active-task path, but not this issue's narrower terminal-state mismatch. A cron task already marked `lost` is still reported as an audit error, and task maintenance only stamps/prunes terminal rows; it does not reconcile a `lost` cron row against cron run history showing `finished` + `ok`.

## Best Possible Solution

Keep the issue open and implement a focused cron/task-ledger repair path. The safest direction is to make cron task lifecycle and cron run history share a durable run identity, then add maintenance/audit logic that can reconcile or prune stale terminal `lost` cron rows only when the matching cron run is conclusively `finished` + `ok`. Add regression coverage for the exact case: a `runtime: "cron"` task in `lost` with `backing session missing`, matching successful cron run history for the same execution, and `openclaw tasks maintenance --apply` clearing the audit error without masking genuine failed or still-unknown runs.
