## Summary

Keep #64625 open. Current main still matches the reported bug: Slack’s primary plugin registers a custom invoke path, and `createSlackActions` forwards the raw `toolContext` to that custom invoke without merging `ctx.mediaLocalRoots` or `ctx.mediaReadFile`. The fallback runtime path does inject those fields, but the primary Slack plugin does not use that fallback. The related PR #64632 was closed unmerged in the provided GitHub context, and the current checkout does not contain its referenced fix or regression test.

## Best Possible Solution

Keep the issue open and land the narrow Slack adapter fix: in `extensions/slack/src/channel-actions.ts`, merge `ctx.mediaLocalRoots` and `ctx.mediaReadFile` into the context passed to `options.invoke`, matching the fallback runtime path. Add a focused regression test for `createSlackActions` with a custom invoke and `upload-file`/media action context so the primary Slack plugin path cannot drop agent-scoped media roots again.
