## Summary

Keep #8508 open. Current main still supports one configured ack reaction value per global/channel/account path, plus separate lifecycle status reactions. It does not provide an ack reaction array, contextual selector, random choice, or model-driven per-message ack decision, and the existing bundled channel paths consume a single resolved value.

## Best Possible Solution

Keep #8508 open for a backwards-compatible core/channel design. Preserve current `ackReaction` string behavior, add a clear array or selector contract such as `messages.ackReactions`, define whether inbound message context is available for selection, and update exported config types, zod schemas, generated metadata/help, public docs, resolver tests, SDK/channel-feedback surfaces if needed, and focused bundled-channel tests.

## Risks / Open Questions

- Product direction is still needed for deterministic, random, rule-based, or model-driven selection, especially because ack reactions happen before the normal reply turn and should stay low latency.
- Channel validation differs: Telegram expects supported unicode reactions, Slack uses shortcode names, Discord supports unicode/custom names, WhatsApp has its own `channels.whatsapp.ackReaction` object, and BlueBubbles tapbacks are constrained.
