## Summary

Keep #8508 open. Current main still exposes and resolves one configured ack reaction string per account/channel/global config, then sends that single value in Telegram, Discord, and Slack. There is no configured emoji array, contextual selector, or model-driven per-message ack selection on the current code path, and the existing issue discussion explicitly reached the same keep-open conclusion earlier today.

## Best Possible Solution

Keep the issue open for a backwards-compatible core/channel behavior design. A strong implementation would preserve existing ackReaction semantics, add a clear configured-array or selector contract such as messages.ackReactions or channel/account equivalents, pass enough inbound context into the resolver if contextual selection is desired, implement the shared path in supported bundled channels, and update config schema/help/docs plus focused tests for resolver behavior and at least Telegram/Discord/Slack channel handling.
