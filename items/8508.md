## Summary

Keep #8508 open. Current main still exposes ack reactions as a single configured string resolved before processing, with no `ackReactions` array, selector, rotation strategy, contextual chooser, or model-driven pre-ack hook. The prior Codex review comment kept this open for the same reason, and the current checkout at `82262387651ab270b565f71533ce4937c9c00dc6` still matches that assessment.

## Best Possible Solution

Keep #8508 open for a backward-compatible core/channel design. Preserve the existing `ackReaction` string contract, then add a clearly named `messages.ackReactions` array or selector/strategy contract after maintainers decide whether selection is deterministic, random, rule-based, or model-driven. The implementation likely needs config types, zod/generated schemas, help/metadata, public docs, shared channel-feedback APIs, SDK exports if third-party channels need them, and bundled-channel tests for Slack, Discord, Telegram, Matrix/BlueBubbles/WhatsApp fallback behavior where applicable.

## Risks / Open Questions

- Product/API direction remains unresolved: deterministic rotation, random selection, rule-based selection, and model-driven selection have different latency and predictability tradeoffs because ack reactions are sent before normal reply generation.
- Channel emoji contracts differ across Slack shortcodes, Discord unicode/custom emoji, Telegram supported reactions, WhatsApp config objects, Matrix reactions, and BlueBubbles tapbacks, so the eventual design needs clear validation and fallback behavior.
