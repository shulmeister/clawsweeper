## Summary

Keep #8508 open. Current main still resolves a single `ackReaction` string from account/channel/global/identity config and sends that one value through bundled channel paths. There is no configured emoji array, contextual selector, or model-driven per-message ack selection, and the existing issue discussion already reached the same keep-open conclusion.

## Best Possible Solution

Keep the issue open for a backwards-compatible core/channel behavior design. Preserve existing `ackReaction` string semantics, add a clear configured-array or selector contract such as `messages.ackReactions` with channel/account overrides, define how inbound message context is passed into selection if contextual behavior is desired, implement the shared path in supported bundled channels, and update config schema/help/docs plus resolver and channel tests.
