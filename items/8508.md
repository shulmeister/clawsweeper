## Summary

Keep #8508 open. Current main still implements a single configured ack reaction string plus lifecycle status reactions, but not the requested configured emoji array or context/model-selected ack reaction per inbound message. The existing same-day Codex/maintainer comment also explicitly kept the issue open for this missing behavior, and current main has not changed the relevant implementation since then.

## Best Possible Solution

Keep the issue open for maintainer product direction. A good implementation would add a backwards-compatible config contract such as messages.ackReactions or channel/account equivalents, preserve existing ackReaction semantics, define selection behavior, and update the resolver, bundled channel ack paths, config schema/help/docs, and targeted tests across at least Telegram plus a shared contract test.
