## Summary

Keep open. Current `main` still lacks the proposed iMessage `chat.db` fallback that hydrates inbound `reply_to_*` fields when `imsg` omits them, so this PR is neither implemented-on-main nor obsolete.

## Evidence

- **Current monitor path has no lookup:** `handleMessage` only parses the RPC payload and immediately enqueues it. There is no `chat.db` read or reply-context hydration step before dispatch.
  - file: [extensions/imessage/src/monitor/monitor-provider.ts:502](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/imessage/src/monitor/monitor-provider.ts#L502)
  - command: `nl -ba extensions/imessage/src/monitor/monitor-provider.ts | sed -n '430,520p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Reply context still depends entirely on inbound payload fields:** `describeReplyContext` builds reply context only from `message.reply_to_id`, `message.reply_to_text`, and `message.reply_to_sender`. If `imsg` omits those fields, current main produces no reply context.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:60](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/imessage/src/monitor/inbound-processing.ts#L60)
  - command: `nl -ba extensions/imessage/src/monitor/inbound-processing.ts | sed -n '1,160p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **No lookup helper or regression tests exist on main:** The monitor directory contains no `reply-context-lookup.ts` helper, and the existing gating tests only cover cases where `reply_to_*` is already present on the payload.
  - file: [extensions/imessage/src/monitor.gating.test.ts:162](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/imessage/src/monitor.gating.test.ts#L162)
