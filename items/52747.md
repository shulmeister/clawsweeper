## Summary

Keep open. Current `main` does not implement ACP session-lane task timeouts, so there is no strong basis to close this as already fixed or obsolete. The PR also appears to introduce a real serialization risk: its timeout wrapper rejects early while the underlying per-session operation keeps running, which would let the keyed actor queue start the next task for the same session before the timed-out task actually finishes.

## Risks / Open Questions

- Closing this PR would leave the reported stuck-session-lane problem unaddressed on current `main`.
- Merging the PR as written risks per-session actor races and late runtime/cache mutation after timeout because the timed-out operation can continue in the background while the next queued task starts.
