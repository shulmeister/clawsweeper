## Summary

Current main already covers the request’s two core asks: gateway config writes are validated and rejected before persistence, and invalid startup config is auto-recovered from last-known-good state with a warning to avoid repeating the bad write. The same behavior is present in release v2026.4.22.

## Risks / Open Questions

- The current implementation rejects bad writes and restores last-known-good config; it does not attempt to guess and rewrite an invalid enum to a new 'correct' enum value automatically. That is a narrower but safer interpretation of the request.

## Close Comment

Closing from a Codex review: current OpenClaw already implements the safety behavior requested here. Gateway config writes validate the full post-change config and reject invalid patches/applies before persisting them (`src/gateway/server-methods/config.ts:471`, `src/config/io.ts:1805`). If a direct edit still leaves the config invalid at startup, the gateway restores the last-known-good config before continuing (`src/gateway/server-startup-config.ts:71`), preserves the rejected file as `.clobbered.*`, and warns the main agent not to write the bad config again (`src/gateway/config-recovery-notice.ts:8`). The shipped docs also describe this exact rejected/auto-restored flow and recommend `config.schema.lookup` plus `config.patch` for safe partial edits (`docs/help/faq.md:804`, `docs/gateway/configuration.md:560`). This behavior is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce a concrete bad write path on a current build, open a new issue with the exact tool call/config payload and logs.
