## Summary

Keep open. Current `main` still starts Bonjour/mDNS advertising without checking whether the gateway is loopback-only, and the bundled Bonjour advertiser still contains the watchdog re-advertise/recreate behavior that matches the reported `probing`/handshake-timeout pattern. The issue discussion’s `OPENCLAW_DISABLE_BONJOUR=1` workaround is still consistent with current docs, but there is no source-backed evidence on `main` that the loopback-bind case was fixed or auto-disabled.

## Evidence

- **Gateway discovery startup does not gate local mDNS on bind mode:** `startGatewayDiscovery()` enables local discovery when `discovery.mdns.mode` is not `off` and `OPENCLAW_DISABLE_BONJOUR` is not set, then unconditionally calls each registered discovery service’s `advertise(...)`. There is no bind-mode or bind-host check in this function.
  - file: [src/gateway/server-discovery-runtime.ts:27](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/gateway/server-discovery-runtime.ts#L27)
  - command: `nl -ba src/gateway/server-discovery-runtime.ts | sed -n '1,120p'`
  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Early gateway runtime never passes loopback-vs-LAN context into discovery startup:** `startGatewayEarlyRuntime()` calls `startGatewayDiscovery(...)` with port, TLS, tailscale, and mDNS settings, but not the resolved gateway bind mode/host. That means loopback-only binding is not part of the decision to start Bonjour advertising.
  - file: [src/gateway/server-startup-early.ts:61](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/gateway/server-startup-early.ts#L61)

  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Bundled Bonjour plugin still watchdog-retries and recreates unhealthy services:** The advertiser watchdog logs `watchdog detected non-announced service; attempting re-advertise`, retries `svc.advertise()`, and recreates the advertiser when a service remains unhealthy/stuck. That matches the behavior described in the report rather than showing a fix on `main`.
  - file: [extensions/bonjour/src/advertiser.ts:373](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/extensions/bonjour/src/advertiser.ts#L373)

- **Current tests cover the same unhealthy Bonjour churn path:** `advertiser.test.ts` has explicit tests for watchdog retries after advertise failures and for probing/announcing unhealthy windows that trigger advertiser recreation. The current test suite therefore still expects this retry/restart machinery to be active.
