## Summary

Issue #42870 is a real upstream WAF false-positive scenario, but the requested mitigation is an endpoint-specific request/body rewrite for custom OpenAI-compatible proxies. Current main exposes that as provider-plugin surface, and the repo vision says optional capability should usually live in plugins rather than core.

## Close Comment

Closing this as a Codex maintainer review because it is a better fit for a ClawHub/community provider plugin than OpenClaw core.

The current main branch already treats generic OpenAI-compatible `/v1` endpoints as proxy routes rather than native OpenAI (`docs/providers/openai.md:764`). For those proxy/provider-specific cases, the public provider SDK already exposes the right seams: `api.registerTextTransforms(...)` for rewriting text before transport (`docs/plugins/sdk-provider-plugins.md:175`) and `wrapStreamFn` for request-header/body modifications and request-body rewrites (`docs/plugins/sdk-provider-plugins.md:366`, `src/plugins/types.ts:1280`). The repo vision also says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high (`VISION.md:54`).

Because the reported 403 is an upstream Cloudflare/hosted-WAF false positive on a custom base URL, this looks like provider/deployment-specific transport policy rather than a core OpenClaw defect. The right path here is a dedicated provider plugin for the affected proxy/baseUrl, published via the community plugin path/ClawHub, where it can own the opt-in body rewrite without teaching core about one WAF workaround.
