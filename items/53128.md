## Summary

Current `main` no longer matches the reported failure mode. Local onboarding still writes `gateway.auth.token` without mirroring `gateway.remote.token`, but local gateway probe/status auth now uses the local auth surface (`gateway.auth.token`) for loopback targets, so `openclaw gateway status` is not expected to fail with a remote-token mismatch after `onboard --install-daemon`. The local/remote split is documented and covered by current tests.
