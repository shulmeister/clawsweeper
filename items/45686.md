## Summary

Keep open. Current main has partial mitigations for compaction pressure, but the requested compaction circuit breaker is not implemented: compaction can be statically configured to use a different model, but there is no automatic fallback model after repeated compaction failures, no force-truncate of oldest conversation context after repeated failed compactions, and compaction timeouts are explicitly excluded from the normal assistant timeout fallback path.

## Evidence

- **Issue discussion requests an architectural circuit breaker:** The issue asks for automatic fallback compaction model, force-truncate after repeated failures, transport isolation for Telegram polling, and gateway status health. The only comment agrees all four proposals are sound and calls the death spiral a real architectural gap.
- **Compaction model is a static override, not failure-triggered fallback:** The config surface exposes `agents.defaults.compaction.model` as an optional model override used instead of the primary model when set; there is no `compactionFallbackModel` or failure-count based fallback field nearby.
