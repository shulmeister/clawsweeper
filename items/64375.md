## Summary

Keep open. Current main still gates cron self-deletion to `schedule.kind === "at"`, so the `every` and `cron` behavior targeted by this PR and linked issue #63770 remains unfixed. The latest release tag still has the same guard, and current tests cover one-shot delete-after-run behavior but not the recurring schedule regression added by this PR.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best maintainer path is to review or rebase it, then land an equivalent fix that removes the `schedule.kind === "at"` restriction and keeps regression coverage for successful `every` and `cron` deletion plus the failure case where jobs must remain.
