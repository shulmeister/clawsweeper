## Summary

Current main still does not implement the central request. Active memory indexing still sends Markdown through a size-based `chunkMarkdown(content, { tokens, overlap })` path with no heading or section strategy, no config field for heading-aware chunking, and no reindex metadata that would rebuild when such a strategy changes. MMR and memory-wiki bridge/source-page support are adjacent improvements, but they do not prevent adjacent `##`/`###` memory sections from being blended into one active-memory search chunk, and the related heading-aware PRs in the provided context are closed unmerged while the remaining open chunking PRs are only implementation candidates.

## Best Possible Solution

Keep this issue open. The best maintainer path is to implement heading-aware or section-aware chunking in the active memory indexing path, add the config/schema/help surface, persist a strategy field in reindex metadata so indexes rebuild when the strategy changes, and add focused tests proving adjacent `##`/`###` sections do not blend into one embedding/FTS chunk. Treat entity extraction and graph memory as a separate memory-core or memory-wiki design track unless maintainers intentionally keep this issue as the umbrella.
