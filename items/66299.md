## Summary

Keep open. Current main still has the reported short-circuit in `resolveAcpTargetSessionKey`: an unresolved non-empty token returns `Unable to resolve session target` before the thread-bound lookup can run. The related PR #66338 proposed the right shape of fix but was closed unmerged, and the latest release tag still contains the same resolver behavior.

## Best Possible Solution

Keep the issue open and implement the PR #66338 direction on current main: in `src/auto-reply/reply/commands-acp/targets.ts`, return immediately only when token resolution succeeds, then try `resolveBoundAcpThreadSessionKey`, and only preserve `Unable to resolve session target: <token>` after no thread binding is found. Add a focused regression test for a bound-thread `/acp close <unresolved-thread-id>` path, with the existing no-token bound-thread tests left intact.
