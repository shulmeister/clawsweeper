## Summary

Keep #29384 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still does not expose Docker sandbox `--gpus all`, `--ipc=host`, `capAdd`, `--device`, `--privileged`, or raw extra Docker create-arg support. The existing OpenShell `gpu` option is a separate backend/plugin path, and the related #53108 capAdd-only PR is open and would not satisfy the central Docker GPU/IPC request even if merged.

## Best Possible Solution

Keep the issue open until a maintainer-approved Docker sandbox extension lands. A good implementation path is to add explicit, security-reviewed config fields such as `agents.defaults.sandbox.docker.gpus` and `agents.defaults.sandbox.docker.ipc`, or a constrained extra-create-args mechanism, then update the TypeScript config type, strict Zod schema, resolver/hash behavior, runtime validation/audit messaging, `buildSandboxCreateArgs`, generated schema/help/docs, and focused regression tests. #53108 can remain related capAdd prior work, but it is not enough to close this GPU/IPC request.

## Risks / Open Questions

- The requested GPU, host IPC, device, privileged, cap-add, or raw Docker argument support weakens sandbox isolation and needs explicit maintainer security/product review before implementation.
- A capAdd-only implementation would still leave the reporter's stated `--gpus all` and `--ipc=host` workflow unresolved.
- OpenShell GPU support may be a useful workaround for some users, but it is a different backend and should not be treated as implementing Docker sandbox parameters.
