## Summary

Keep #69307 open. Current main still has no modifying hook between final reply generation and TTS directive parsing. The dispatch path calls `maybeApplyTtsToReplyPayload()` before routing/delivery, `message_sending` runs only as dispatcher `beforeDeliver`, `llm_output` is observational, `before_agent_finalize` can only request another model pass/finalize, and `reply_dispatch` requires owning the dispatch path. This is a real missing plugin API seam, not a ClawHub-closeable plugin implementation request yet.

## Best Possible Solution

Keep the issue open as a plugin API design request. The best implementation is an additive, typed modifying hook, likely named `before_tts` or broader `reply_transform`, invoked after reply text is produced and before every `maybeApplyTtsToReplyPayload()` path. It should expose text/payload plus stable context such as session key, channel, account/conversation target, run id, agent id, TTS kind, and current TTS mode, and return text/payload changes plus an explicit skip-TTS decision. Add hook runner/types/docs and focused tests proving injected directives affect TTS voice routing and visible text is still cleaned normally.

## Risks / Open Questions

- Adding this hook would expose raw assistant text and routing metadata to plugins, so it should reuse existing conversation-access and prompt-injection permission patterns where appropriate.
- The hook must be wired consistently for final replies, block-streamed TTS accumulation, tool/block replies when `messages.tts.mode: "all"`, and TTS-only synthetic final audio.
- A `skip` result needs clear precedence with `messages.tts.auto`, session TTS overrides, directive stripping, and existing delivery cancellation semantics.
