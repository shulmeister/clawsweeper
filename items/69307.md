## Summary

Keep open. Current main does not expose a modifying plugin hook between model reply generation and TTS directive parsing/synthesis. The issue is a concrete, recent plugin-API gap rather than a stale support request or ClawHub-only plugin idea: a plugin can observe `llm_output`, claim all dispatch via `reply_dispatch`, or rewrite final outbound content via `message_sending`, but there is no narrow post-LLM/pre-TTS text transform hook for injecting `[[tts:...]]` directives before `maybeApplyTtsToPayload()` runs.

## Best Possible Solution

Keep the issue open for an additive plugin SDK design. The likely path is a narrow modifying hook such as `before_tts` or a more general reply-payload transform in `src/plugins/hook-types.ts` and `src/plugins/hooks.ts`, invoked from `src/auto-reply/reply/dispatch-from-config.ts` immediately before each `maybeApplyTtsToReplyPayload()` call, with docs/tests covering directive injection, skip-TTS behavior, routing fields, and conversation-access policy.

## Risks / Open Questions

- The exact contract needs maintainer API design: whether the hook should cover final replies only or also tool, block, and synthetic TTS-only replies.
- The hook would expose raw post-LLM assistant text to plugins, so it should align with existing conversation-access trust policy and hook documentation.
