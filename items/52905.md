## Summary

Keep open. Current main appears to have mitigated the specific `openclaw status` path by switching status scans to a presence-only gateway probe, but the broader enhancement request is not fully implemented: the dedicated gateway probe/status path still defaults to a full read probe, still includes the expensive `status` RPC, and still uses a 3000 ms default budget.

## Evidence

- **`openclaw status` now uses presence-only probes:** `resolveGatewayProbeSnapshot()` defaults status scans to `detailLevel: "presence"`, so normal status surfaces no longer request full gateway detail RPCs.
  - file: [src/commands/status.scan.shared.ts:115](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/commands/status.scan.shared.ts#L115)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '109,117p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Presence-only mode skips the expensive `status` RPC:** `probeGateway()` treats `detailLevel: "presence"` specially and only requests `system-presence`; the full mode still does `health`, `status`, `system-presence`, and `config.get` together.
  - file: [src/gateway/probe.ts:297](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/probe.ts#L297)
