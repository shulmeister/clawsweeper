## Summary

Keep this issue open. Current main still enforces conversation-content hook access from per-plugin host config only, while the native plugin manifest type, parser, manifest registry record, loader policy merge, tests, and docs do not support a manifest-owned `permissions.conversationAccess` field. The existing maintainer/Codex discussion is still consistent with current main at 9626ef274ae2346b3d519f99c31856be98aab570.

## Best Possible Solution

Keep the issue open. Add an additive native manifest contract, likely `permissions.conversationAccess`, in `src/plugins/manifest.ts`; normalize it in `loadPluginManifest`; carry it through `PluginManifestRecord`; merge it into loader hook policy with clearly documented host-config precedence; update hooks/config/manifest docs; and add tests for manifest-only allow, config-only backward compatibility, both-present precedence, and default rejection.
