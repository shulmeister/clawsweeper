## Summary

Keep open. Current main still preserves the reported DAVE-disabled pre-Ready join-abort shape: Discord voice passes `daveEncryption: false` through to `@discordjs/voice`, waits 15 seconds for `VoiceConnectionStatus.Ready`, destroys the connection on failure, and only records an OpenClaw voice session after Ready. The current DAVE recovery path is post-Ready receive/decrypt recovery, not a fix for a voice WebSocket close after Hello and before UDP/Ready. Related #67453 is the separate post-join TTS problem, and related #39825 remains open rather than providing a shipped canonical fix.

## Best Possible Solution

Keep this issue open until maintainers decide the intended non-DAVE behavior. The likely fix belongs in `extensions/discord/src/voice/manager.ts`: add explicit pre-Ready close/abort diagnostics and choose a bounded retry, fallback to DAVE-enabled join, or a clear rejection/deprecation path for `daveEncryption: false` if Discord now requires DAVE. Add regression coverage for the initial Ready failure path and update Discord voice docs/config hints after that product decision.
