## Summary

Keep open. Current main supports Node/Gateway `agent.request` image attachments as agent input, and supports media delivery when the agent reply contains `mediaUrl`, `mediaUrls`, or `MEDIA:` directives. It does not implement an `auto_forward_attachments` flag or any fallback that relays the original Node attachments to the IM recipient when the agent returns only text or no payload.

## Best Possible Solution

Keep the issue open for maintainer product judgment and implementation. A concrete path would add an explicit opt-in field to the `agent.request`/Gateway agent schema, preserve accepted inline/offloaded attachment refs through the agent command boundary, and inject them into outbound reply payloads only for `deliver: true` when the agent reply contains no media. Add focused tests for the Node event path, the Gateway agent RPC path if supported there, no-agent-payload behavior, and at least one channel outbound media path; update user-facing docs to distinguish this from the existing multimodal-agent-input support.

## Risks / Open Questions

- Auto-forwarding Node attachments would change the current agent-controlled delivery model; any implementation should be explicit opt-in and reuse existing outbound media access and channel policy checks.
- Offloaded inbound media lifetimes and cleanup need a clear contract if refs are retained past prompt parsing for later outbound delivery.
