## Summary

Keep open. Current main still rejects the root-level `paperclip` object on `agent` params: `AgentParamsSchema` remains strict and has no `paperclip` or generic metadata field, the gateway compiles that schema with `removeAdditional: false`, and the `agent` handler returns `INVALID_REQUEST` before dispatch on validation failure. The PR’s linked issue and production corroboration describe the same schema-boundary failure, so the work is still actionable even if maintainers may want a more extension-agnostic field name than `paperclip`.

## Best Possible Solution

Keep this PR open as the actionable fix path, but have a maintainer decide the protocol shape before merge: either land the narrow compatibility field with the regression test, or introduce a generic opaque external-adapter metadata field and update Paperclip docs/adapter guidance. Do not close until current main accepts the documented Paperclip `agent` payload or the adapter is changed to stop sending unsupported root-level metadata.
