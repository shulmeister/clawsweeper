## Summary

Current main already supports Discord channel-per-agent routing through peer-scoped bindings and documents the config path; the only remaining gap described in the issue is CLI ergonomics for creating those bindings, which is a separate enhancement.

## Evidence

  - file: [src/routing/resolve-route.ts:726](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/routing/resolve-route.ts#L726)

  - file: [extensions/discord/src/monitor/route-resolution.ts:10](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/extensions/discord/src/monitor/route-resolution.ts#L10)

## Close Comment

Codex review against current `main` (`564f820efa2bb81ae49b8f55a93e41492ad31917`) found that the researched capability is already implemented on main. Peer-scoped routing is a first-class match tier in `src/routing/resolve-route.ts:726`, Discord inbound routing builds `peer.kind: "channel"` from the conversation id in `extensions/discord/src/monitor/route-resolution.ts:10`, and the supported config shape is documented in `docs/gateway/config-agents.md:978` plus the Discord example in `docs/channels/discord.md:663`. The remaining gap called out here is still true: `openclaw agents bind` only parses `--bind <channel[:accountId]>` (`src/commands/agents.bindings.ts:281`, `docs/cli/agents.md:124`), so peer bindings still need direct config edits. That looks like a separate follow-up enhancement rather than an unresolved problem with the researched Discord channel-per-agent binding itself, so closing this research issue based on current main.
