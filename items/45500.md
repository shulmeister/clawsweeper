## Summary

Keep open. Current main has adjacent duplicate-delivery safeguards, but I do not see a per-turn idempotency guard for the reported case: a transient LLM/provider retry after a Discord-visible reply has already been delivered. The existing suppression is exact-content based, so a retry that produces slightly different wording can still be delivered as a second reply.

## Evidence

- **Transient provider retry is still unconditional on prior visible delivery:** The first transient HTTP/provider error path sets a one-shot retry flag, logs "Transient HTTP provider error before reply", waits, and continues the whole runWithModelFallback cycle. This branch does not check whether onBlockReply/direct channel delivery already sent user-visible text before retrying.
