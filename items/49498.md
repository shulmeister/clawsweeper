## Summary

Current main no longer relies on copying repo-root `src/` or `apps/` into the Docker runtime image for bundled plugin startup. Dockerized installs resolve bundled plugins from compiled `dist/extensions`, WhatsApp now imports Plugin SDK helpers instead of `src/channels/...`, and the tool-display runtime path uses `src/agents/tool-display-config.ts` rather than runtime-loading the shared app JSON.

## Evidence

- **Docker runtime now prefers compiled bundled plugins:** Bundled plugin resolution prefers `dist-runtime/extensions` or `dist/extensions` and only falls back to source `extensions/` for source checkouts. In an installed Docker image, that means compiled `dist/extensions` is used, not `/app/extensions` source entries.
  - file: [src/plugins/bundled-dir.ts:79](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/plugins/bundled-dir.ts#L79)
  - command: `nl -ba src/plugins/bundled-dir.ts | sed -n '79,104p'`
- **Release notes explicitly call out the Docker fix:** The changelog for `2026.4.7` says Docker runtime images stopped forcing bundled plugin discovery to `/app/extensions` so packaged installs use compiled `dist/extensions` artifacts again.
  - file: [CHANGELOG.md:1144](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/CHANGELOG.md#L1144)
  - command: `nl -ba CHANGELOG.md | sed -n '1072,1145p'`
- **WhatsApp no longer deep-imports core `src/` helpers:** Current WhatsApp shared runtime imports `openclaw/plugin-sdk/account-helpers` and related SDK paths, not `../../../src/channels/plugins/account-helpers.js`. That removes the reported `/app/src` runtime dependency.
  - file: [extensions/whatsapp/src/shared.ts:1](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/whatsapp/src/shared.ts#L1)
  - command: `nl -ba extensions/whatsapp/src/shared.ts | sed -n '1,8p'`
- **Server runtime tool display no longer depends on app JSON:** Core runtime tool display reads `TOOL_DISPLAY_CONFIG` from `src/agents/tool-display-config.ts`; the `apps/shared/.../tool-display.json` import is only in Control UI source.
  - file: [src/agents/tool-display.ts:1](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/agents/tool-display.ts#L1)
  - command: `nl -ba src/agents/tool-display.ts | sed -n '1,20p'`
- **Runtime image still does not copy `src/` or `apps/`, by design:** The Dockerfile runtime stage copies `dist`, `node_modules`, bundled plugin tree, docs, QA, and skills, but not repo-root `src/` or `apps/`. Combined with the code above, that shows current main fixed the bug by removing those runtime source-tree dependencies rather than adding the reporter's workaround COPY lines.

## Close Comment

Closing as implemented on main after a Codex review. Current Docker/runtime behavior no longer depends on copying repo-root `src/` or `apps/` into the final image: bundled plugin resolution now prefers compiled `dist/extensions` in packaged installs (`src/plugins/bundled-dir.ts:79`), and the `2026.4.7` changelog explicitly calls out the Docker fix to stop forcing `/app/extensions` source paths in runtime images (`CHANGELOG.md:1144`). The two concrete failures from this report are also addressed in current code: WhatsApp now imports `openclaw/plugin-sdk/account-helpers` instead of deep-importing `src/channels/...` (`extensions/whatsapp/src/shared.ts:1`), and the core runtime tool-display path uses `src/agents/tool-display-config.ts` rather than needing `apps/shared/.../tool-display.json` at runtime (`src/agents/tool-display.ts:1`). The runtime Docker stage still does not copy `/app/src` or `/app/apps` (`Dockerfile:173`), which is the point: main fixed the underlying packaged-runtime dependency path instead of masking it with extra COPY lines.
