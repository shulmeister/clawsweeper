## Summary

Keep open. Current main still does not canonicalize Telegram forum-topic requester conversation IDs during subagent completion routing. `resolveSubagentCompletionOrigin` passes the raw `threadId` as the requester conversation ID, while Telegram bindings are canonicalized as `<chatId>:topic:<topicId>`. The bound-delivery router only exact-matches conversation IDs, so topic-bound completion routing can still miss the intended binding on main.

## Evidence

- **Current completion routing uses raw threadId:** `resolveSubagentCompletionOrigin` derives `conversationId` as `threadId || resolveConversationIdFromTargets(...)`. For a Telegram topic request like `to: "123", threadId: 42`, that becomes `"42"`, not the canonical topic conversation id.
  - file: [src/agents/subagent-announce-delivery.ts:226](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/subagent-announce-delivery.ts#L226)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '214,255p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Bound router requires exact requester conversation match:** `resolveBindingForRequester` only treats a binding as a requester match when the normalized binding `conversationId` exactly equals `requester.conversationId`; otherwise it falls back only when there is exactly one matching binding.
  - file: [src/infra/outbound/bound-delivery-router.ts:30](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/infra/outbound/bound-delivery-router.ts#L30)

That does not match the raw requester `threadId` used above.

## Risks / Open Questions

- Telegram subagent completions bound to forum topics can miss requester-match when a child session has multiple active Telegram topic bindings.
- When requester-match fails, delivery can fall back away from the canonical bound topic target, leaving the original routing bug unresolved on current main.
