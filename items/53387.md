## Summary

Keep open. Current main still reads state-dir `.env` values as durable service env, merges them into the gateway install plan, and hands that full environment to the LaunchAgent plist writer. The plist writer serializes every non-empty environment entry under `EnvironmentVariables`, while runtime dotenv loading still refuses to override existing process env values. The stale-plist precedence failure described in this issue is therefore still present on current main.

## Best Possible Solution

Keep this issue open and fix the macOS LaunchAgent install path. The likely direction is to stop copying user-managed state-dir `.env` secrets and env-backed auth/profile refs into LaunchAgent `EnvironmentVariables`, keep only internal service metadata and explicitly supported service-level env, add LaunchAgent/install-plan regression coverage that `.env` keys are excluded, and document or implement repair behavior for already-installed plists.
