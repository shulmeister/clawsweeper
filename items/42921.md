## Summary

Current `main` already resolves the default cron store path lazily at call time, so the reported import-time `OPENCLAW_STATE_DIR` bug is no longer present. The same implementation is also present in release `v2026.4.22`, which makes this PR redundant on top of current `main`.

## Evidence

- **Cron store path is resolved lazily at runtime:** `resolveCronStorePath()` falls back to `resolveDefaultCronStorePath()` only when called, and `resolveDefaultCronStorePath()` computes the path through `resolveDefaultCronDir()` instead of reading a module-level constant.
  - file: [src/cron/store.ts:26](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/cron/store.ts#L26)
  - command: `nl -ba src/cron/store.ts | sed -n '20,80p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **State-dir override is read dynamically:** `resolveConfigDir()` reads `env.OPENCLAW_STATE_DIR` inside the function body, so a later `loadDotEnv()` update to `process.env` is honored when the cron path is resolved.
  - file: [src/utils.ts:136](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/utils.ts#L136)
  - command: `nl -ba src/utils.ts | sed -n '135,165p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Latest release already contains the lazy resolution:** The `v2026.4.22` tag shows the same `resolveDefaultCronDir()` / `resolveDefaultCronStorePath()` runtime path resolution in `src/cron/store.ts`, so this behavior has already shipped.
  - file: [src/cron/store.ts:26](https://github.com/openclaw/openclaw/blob/a94763ee224e858e29dd177cd6f7f4daa438c2f1/src/cron/store.ts#L26)
  - command: `git show v2026.4.22:src/cron/store.ts | nl -ba | sed -n '20,80p'`
  - sha: [a94763ee224e](https://github.com/openclaw/openclaw/commit/a94763ee224e858e29dd177cd6f7f4daa438c2f1)

## Risks / Open Questions

- Current main solves this via `resolveConfigDir()` rather than the PR's `resolveStateDir()` approach, so legacy-state-dir semantics differ from the proposed patch, but the reported `OPENCLAW_STATE_DIR` runtime-resolution bug itself is already addressed.

## Close Comment

Closing as implemented on `main` after Codex review. `src/cron/store.ts:26`-`32` now resolves the default cron directory/store path lazily at runtime, and `src/cron/store.ts:72`-`80` uses that runtime resolver from `resolveCronStorePath()`. The underlying state-dir override is also read dynamically in `src/utils.ts:136`-`157` via `resolveConfigDir()`, so `OPENCLAW_STATE_DIR` loaded later from dotenv is respected when the cron path is actually resolved. This behavior is already present in release `v2026.4.22`, so there is no remaining main-branch gap for this PR to land.
