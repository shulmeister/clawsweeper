## Summary

Keep open. Current main has model-fallback observability through subsystem logs, lifecycle events, session-store fallback notice state, and raw trace payloads, but I did not find a persisted session JSONL custom event that records the primary failure before the `model-snapshot` record. The related user-visible fallback-notice work does not satisfy this issue's forensic audit-trail request.

## Best Possible Solution

Keep this open and add a narrow transcript audit event for fallback activation. The best implementation is a generic fallback audit hook or transcript writer passed into the fallback/reply-runner boundary, writing a sanitized `custom` record before the fallback model snapshot/retry with primary provider/model, fallback provider/model, failure class, status/code when available, duration if available, and hashed or preview-limited error details. Add focused tests that assert JSONL ordering and redaction, while keeping the existing subsystem logs, lifecycle events, and user-visible fallback notices.

## Risks / Open Questions

- A transcript-level audit event must sanitize provider errors to avoid persisting secrets, request payloads, or raw stack traces in long-lived session files.
- The implementation needs to preserve JSONL ordering around fallback retry and `model-snapshot` writes, especially under concurrent session updates and transcript rotation.
