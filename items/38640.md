## Summary

Keep PR #38640 open. Current main still does not implement the PR's prompt-level exponential-backoff retry wrapper around `activeSession.prompt(...)`; it has adjacent SDK retry-wait caps, auth-profile rotation, cooldowns, and model fallback instead. The PR is plausible but not merge-ready because review comments still point at retry-boundary and abort/status parsing concerns, so it needs maintainer design/review rather than automated cleanup.

## Best Possible Solution

Keep the PR open for maintainer judgment unless maintainers explicitly choose a canonical retry design such as the open configurable LLM/provider retry work. A best implementation would place retry at the intended provider/runner boundary, honor abort signals during backoff, parse numeric and string status/statusCode consistently with `failover-error.ts`, respect provider SDK retry behavior, and add focused unit plus runner/failover coverage.
