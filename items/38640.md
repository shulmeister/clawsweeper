## Summary

Keep PR #38640 open. Current main has adjacent provider SDK retry capping, auth-profile rotation, cooldowns, and model fallback, but it still does not implement this PR's prompt-level LLM retry wrapper or an equivalent configurable same-request retry loop for 429/5xx failures. The PR discussion also still contains unresolved correctness concerns, so this should not be auto-closed as implemented or merely obsolete.

## Best Possible Solution

Keep the PR open until a maintainer decides the product direction for LLM retry. The likely best path is to implement retry at the provider-transport or runner boundary with abort-aware backoff, robust status/Retry-After parsing, and tests for 429/5xx, rather than landing the current stale wrapper unchanged. Related open trackers #49376 and #61443 should inform the final design, but they do not make current main implement this PR yet.
