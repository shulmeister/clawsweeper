## Summary

Keep open. On current `main` (`8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`), docs still promise a Discord elevated fallback from `channels.discord.allowFrom`, but the elevated runtime only consults a plugin `allowFromFallback` hook and there is no implementation for that hook anywhere in `src/` or `extensions/`. `sandbox explain` also only checks explicit `tools.elevated.allowFrom.<channel>` entries, so the diagnostics inconsistency in the report is still present.

## Evidence

- **Docs still promise Discord fallback:** `docs/tools/elevated.md` says: if `tools.elevated.allowFrom.discord` is omitted, `channels.discord.allowFrom` is used as fallback.
  - file: [docs/tools/elevated.md:90](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/tools/elevated.md#L90)
  - command: `nl -ba docs/tools/elevated.md | sed -n '80,110p'`

- **Elevated runtime depends on plugin fallback hook:** `resolveElevatedPermissions` asks the channel plugin for `elevated.allowFromFallback` and passes that into sender authorization; without a plugin implementation, only explicit `tools.elevated.allowFrom.<provider>` is effective.
  - file: [src/auto-reply/reply/reply-elevated.ts:202](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/reply-elevated.ts#L202)

  - command: `rg -n "allowFromFallback" extensions src --glob '*.ts'`
