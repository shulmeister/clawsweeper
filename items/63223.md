## Summary

Keep open. Current main has adjacent Discord gateway lifecycle fixes, but #63223’s core asks are not fully satisfied: a healthy READY transition can still have no later `[discord] logged in...` line if the startup status was emitted before `gateway.isConnected`, and cert-expiry failures are still handled by in-process Discord/channel lifecycle restarts rather than an explicit full process/service respawn or documented system-CA-rotation path. The issue discussion also has a second reporter confirming the same `certificate has expired` loop on Debian, which strengthens that this is not just one macOS launchd incident.

## Best Possible Solution

Keep #63223 open and address it as a Discord gateway stability/observability bug. Best next implementation is to log a READY/healthy line from the actual Carbon READY listener or lifecycle transition, add tests for initialized-then-ready logging, and add a cert-expiry escalation path that either requests a full gateway process/service restart or documents and exposes an operator hook for that restart. The existing startup race and stale-socket fixes should be treated as partial adjacent work, not the canonical resolution.

## Risks / Open Questions

- Closing this now would leave a user-visible observability gap: operators can still see only `discord client initialized...; awaiting gateway readiness` on a healthy delayed READY path.
- The current in-process channel restart path may not refresh process-local Node TLS trust state, which is the specific failure mode reported for system CA rotation and repeated cert-expiry loops.
