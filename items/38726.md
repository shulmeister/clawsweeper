## Summary

Current `main` already ships a first-party dashboard usage view with token-focused model/session breakdowns, and local Ollama usage is recorded from `prompt_eval_count`/`eval_count` and rolled into the same session-based aggregates.

## Risks / Open Questions

- The implementation is transcript/session-backed rather than the exact `~/.openclaw/usage/usage.jsonl` + `GET /api/usage` shape proposed in the issue body, so this closes on user-facing capability rather than identical architecture.

## Close Comment

Closing as implemented based on a Codex review of current `main`. OpenClaw already ships a built-in Web UI token usage dashboard (`CHANGELOG.md:5295-5310`, released in `v2026.2.6`), backed by `sessions.usage`, `sessions.usage.timeseries`, and `sessions.usage.logs` (`docs/gateway/protocol.md:281-288`, `src/gateway/server-methods/usage.ts:409`). The current dashboard is explicitly token/usage-focused with top-model summaries, daily token charts, session drilldowns, and even cron-session filtering (`ui/src/i18n/locales/en.ts:468-485`, `ui/src/ui/views/usage-render-overview.ts:541-565`, `ui/src/ui/views/usageTypes.ts:47-57`). Local Ollama runs are included because the Ollama provider records `prompt_eval_count` / `eval_count` into normalized usage (`extensions/ollama/src/stream.ts:545-552`), and the session aggregator reads provider/model/usage from transcript entries before rolling them into dashboard aggregates (`src/infra/session-cost-usage.ts:122-158`).
