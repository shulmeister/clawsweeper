## Summary

Keep open. Current `main` still has the Discord presence cache and exposes presence through the `memberInfo` guild action, but it does not thread sender client status into inbound conversation metadata. The PR’s requested behavior is therefore not implemented on `main`, and the only review feedback in the discussion is minor cleanup/edge-case guidance rather than a blocker.

## Risks / Open Questions

- If this PR is revived as-is, it should add coverage for the new conversation metadata path; current tests cover `sender_id` and related conversation info fields but nothing for `sender_client_status`.
- Greptile’s empty-object concern is credible in this codebase because an existing Discord test already uses `client_status: {}` (`extensions/discord/src/actions/runtime.test.ts:576`), so a revived patch should avoid emitting `sender_client_status: {}` when no platform key is present.
