## Summary

Current main still does not implement native Feishu post `at` elements for outbound mentions. The send path prefixes pseudo-XML `<at ...>` text and then serializes the whole message as a single `md` post element, so PR #49850 is not already implemented on `main` and should remain open.

## Evidence

- **Post payload still emits one markdown element:** `buildFeishuPostMessagePayload()` always returns `zh_cn.content` as `[[{ tag: "md", text: messageText }]]` with no branch that emits native Feishu `at` elements.
  - file: [extensions/feishu/src/send.ts:432](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/send.ts#L432)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Mention formatting is still pseudo-XML text:** `formatMentionForText()` still formats mentions as literal `<at user_id="...">name</at>` strings rather than structured post elements.
  - file: [extensions/feishu/src/mention.ts:78](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/mention.ts#L78)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Outbound send path still round-trips through that text:** When `mentions` are present, `sendMessageFeishu()` calls `buildMentionedMessage()` and then passes the resulting string into `buildFeishuPostMessagePayload()`, so mentions are still embedded inside the single `md` element on current main.
  - file: [extensions/feishu/src/send.ts:454](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/send.ts#L454)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Literal `<at>` markup already exists in normal Feishu text flows:** `normalizeMentions()` rewrites inbound mention placeholders into literal `<at user_id="...">...</at>` text, which matches the PR review concern that any eventual parser must only tokenize intentional outbound mentions, not arbitrary body text.
  - file: [extensions/feishu/src/bot-content.ts:260](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/bot-content.ts#L260)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)

## Risks / Open Questions

- The underlying Feishu mention-rendering bug is still present on current main, so closing as `implemented_on_main` or `cannot_reproduce` would be incorrect.
- The PR appears directionally useful, but its review comments about restricting parsing scope still matter because this plugin already uses literal `<at ...>` text in ordinary normalized message bodies.
