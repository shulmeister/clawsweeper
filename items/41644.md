## Summary

Current `main` already avoids sending image payloads to text-only models and documents/tests the text-only attachment fallback path, so the reported behavior is implemented/fixed on `main` rather than still actionable in core.

## Close Comment

Closing from a Codex review because current `main` already has the text-only image-handling path this report was missing. In `src/gateway/chat-attachments.ts:274`, text-only runs (`supportsImages: false`) offload inbound images to `media://inbound/*` refs instead of sending inline image blocks, and `src/agents/openai-transport-stream.ts:256` filters `input_image` parts unless the selected model advertises image input. That behavior is covered by `src/gateway/chat-attachments.test.ts:146` and `src/gateway/server-methods/chat.directive-tags.test.ts:2074`, and the current docs now describe the same fallback in `docs/nodes/media-understanding.md:139`. The DingTalk integration referenced here is also documented as an external community plugin in `docs/plugins/community.md:52`, so any remaining DingTalk-specific adapter bug would need plugin-side logs/version details. If this still reproduces on a current OpenClaw build, please open a new issue with the current OpenClaw version, the DingTalk plugin version, exact model ref, and a failing request log snippet.
