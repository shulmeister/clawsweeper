## Summary

Current main already solves PR #63586 / issue #63575 by applying `plugins.allow` during plugin activation planning before validation or runtime import. Off-allowlist plugins become disabled records with `not in allowlist`, so they do not become `status === "error"` entries that strict loading can throw on. The same behavior is present in the v2026.4.24 release tag.

## Best Possible Solution

Close this PR as already implemented on main. Keep the current activation-first design where `plugins.allow` gates plugin eligibility before validation/import, and open a narrower follow-up only if someone finds a current strict-load path that still creates `status === "error"` for a plugin policy says should be excluded.
