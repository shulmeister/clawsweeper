## Summary

Current main has improved approval registration and client-side pending-approval replay, plus broad restart recovery for interrupted main sessions, but I did not find the requested stale `approval-pending` transcript invalidation or re-registration after a gateway process restart. The gateway still creates a fresh in-memory `ExecApprovalManager` per process, `resolve`/`get` return `INVALID_REQUEST` for ids absent from that manager, and persisted approval-pending exec tool results still carry old approval ids in transcript content even though runtime `details` are stripped before model replay.

## Best Possible Solution

Keep this issue open. The likely fix should live in the agent replay/session repair path or the approval manager contract: on restart/resume, detect exec tool results with `status: "approval-pending"` whose id is not present in the live approval manager and rewrite them to an expired/error tool result so the agent re-issues the exec call, or durably persist and re-register pending approval records. Add a focused regression test that simulates a gateway restart with an approval-pending transcript tail and proves the resumed turn does not attempt to resolve the stale id in a loop.
