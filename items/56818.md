## Summary

Current `main` still routes Feishu card-action callbacks through a synthetic `message_id` and then reuses that synthetic ID for streaming replies. There is a generic start-failure fallback, but no Feishu-specific fix for callback `open_message_id` handling, so the reported bug is still plausible and not safe to auto-close.

## Evidence

- **Card actions synthesize a fake message ID:** The Feishu card-action handler builds a synthetic inbound event with `message.message_id = \`card-action-${event.token}\`` instead of preserving any original Feishu message anchor from the callback payload.
  - file: [extensions/feishu/src/card-action.ts:119](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/card-action.ts#L119)
  - command: `nl -ba extensions/feishu/src/card-action.ts | sed -n '105,159p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Reply target still comes from the synthetic inbound message:** Normal Feishu dispatch derives `replyTargetMessageId` from `ctx.messageId` for non-topic replies. For a synthetic card-action event, that means downstream replies target the synthetic `card-action-*` ID.
  - file: [extensions/feishu/src/bot.ts:1039](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/bot.ts#L1039)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '1036,1108p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Streaming path ignores the DM reply suppression and still replies to the target ID:** `createFeishuReplyDispatcher()` computes `sendReplyToMessageId` so normal non-streaming messages can skip reply metadata, but `startStreaming()` still passes the raw `replyToMessageId` into `streaming.start()`, which then calls `im.message.reply` whenever a reply target exists.
  - file: [extensions/feishu/src/reply-dispatcher.ts:341](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/reply-dispatcher.ts#L341)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '317,355p' && nl -ba extensions/feishu/src/streaming-card.ts | sed -n '265,280p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Reply fallback does not cover invalid callback IDs:** Feishu reply fallback only recognizes withdrawn/not-found targets (`230011`, `231003`, `withdrawn`, `not found`). Non-matching reply failures still throw, and the tests explicitly assert that non-withdrawn reply failures are not downgraded to direct-send fallback.
  - file: [extensions/feishu/src/send.ts:17](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/send.ts#L17)
  - command: `nl -ba extensions/feishu/src/send.ts | sed -n '17,60p' && nl -ba extensions/feishu/src/send.reply-fallback.test.ts | sed -n '109,120p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **No code reads callback `open_message_id`:** A repository search under the Feishu extension found no handler code that reads `context.open_message_id` or `open_message_id`, so the issue's proposed source for the original message anchor is not implemented on current `main`.
  - command: `rg -n "context\.open_message_id|open_message_id" extensions/feishu/src`
- **Current tests only cover generic streaming-start fallback:** The added reply-dispatcher coverage exercises a generic HTTP 400 `start()` failure and verifies backoff to a non-streaming card, but there is no card-action-specific test proving that callback replies use a valid original Feishu message ID.
  - file: [extensions/feishu/src/reply-dispatcher.test.ts:966](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/reply-dispatcher.test.ts#L966)

## Risks / Open Questions

- The current generic fallback may mask the first streaming failure in some DM cases, but it does not prove the callback reply target is correct.
- Group or thread callback flows are higher risk because reply-to fallback is intentionally restricted there.
