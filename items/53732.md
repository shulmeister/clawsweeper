## Summary

Keep this issue open. Current main still embeds BOOT.md verbatim in the boot agent prompt and suppresses only the normal final delivery path with `deliver: false`; I did not find a boot-run marker, prompt-echo detector, or message-tool delivery guard that would prevent a fallback model from sending echoed BOOT.md content to a user. The related merged fallback retry work preserved the original prompt for fallback attempts, which improves task continuity but does not address prompt echo suppression.

## Best Possible Solution

Keep this issue open and add a narrow internal boot-run guard. Mark BOOT.md runs as internal, suppress or reject substantial literal echoes of the generated boot prompt before both final reply projection and message-tool delivery, and add regression coverage where the primary model fails, a fallback model returns or sends BOOT.md text verbatim, and no user-visible payload is emitted while legitimate BOOT.md-requested message sends still work.

## Risks / Open Questions

- The reported behavior is model-dependent, so the best regression should use a deterministic simulated fallback response/tool call rather than live Gemini behavior.
- The normal final-reply leak path appears suppressed by `deliver: false`; the remaining actionable risk is message-tool or other outbound delivery during BOOT.md fallback runs.
