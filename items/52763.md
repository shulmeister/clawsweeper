## Summary

Current main does not show a WhatsApp groupPolicy bypass here. The reported config still authorizes group senders via the documented `groupAllowFrom <- allowFrom` fallback, and current debounce code with `debounceMs: 0` does not batch messages into a later flush.

## Close Comment

Thanks for the report. I reviewed current `main` with Codex and could not verify a present WhatsApp groupPolicy bypass. On current `main`, WhatsApp falls back from `groupAllowFrom` to `allowFrom` when `groupAllowFrom` is unset (`extensions/whatsapp/src/inbound-policy.ts:88`), and `groupPolicy: "allowlist"` intentionally allows group traffic through when that sender-level allowlist exists (`src/config/group-policy.ts:352`). The docs match that behavior: if `channels.whatsapp.groups` is omitted, all groups are eligible, and `groupAllowFrom` falls back to `allowFrom` (`docs/channels/whatsapp.md:181`). Current tests also cover the fail-closed case only when both group/group-sender allowlists are empty (`extensions/whatsapp/src/monitor-inbox.blocks-messages-from-unauthorized-senders-not-allowfrom.test-support.ts:330`). Also, current debounce logic only batches when `debounceMs > 0`, so the reported `debounceMs: 0` batched flush does not match the current code path (`src/auto-reply/inbound-debounce.ts:166`). Closing this as not reproducible on current `main`. If you can still reproduce it on a newer build, please open a fresh issue with the exact config in use at that time, whether `channels.whatsapp.groups` is set, and fresh logs around the triggering message.
