## Summary

Keep open. Current main still has the reported parser gap: `splitMediaFromOutput()` only shields fenced code spans, then trims leading whitespace before matching `MEDIA:`, so a 4-space or tab-indented code-block line can still be treated as a delivery directive. The linked fix PR #69312 is open/unmerged and should remain the active path.

## Best Possible Solution

Keep #69311 open and use PR #69312, or an equivalent maintainer-shaped patch, as the active fix path. The best fix belongs in `src/media/parse.ts` with focused regression tests in `src/media/parse.test.ts`, and should make MEDIA extraction code-block-aware before trimming leading whitespace; the related `parseFenceSpans()` and whitespace-preservation cleanups can be handled in the same PR if maintainers accept that scope.

## Evidence

- **Parser only skips fenced spans:** `splitMediaFromOutput()` builds `fenceSpans` from `parseFenceSpans()` and only bypasses extraction when the current line offset is inside one of those fenced spans; there is no CommonMark indented-code-block guard in the current loop.
  - file: [src/media/parse.ts:501](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/media/parse.ts#L501)
  - command: `nl -ba src/media/parse.ts | sed -n '450,575p'`
  - sha: [a57d681db9b2](https://github.com/openclaw/openclaw/commit/a57d681db9b237348c5d446c8d371b2c1057f016)
- **Indented MEDIA lines still enter directive parsing:** The parser calls `line.trimStart().toUpperCase().startsWith("MEDIA:")`, so leading spaces or tabs are removed before directive detection. A line like `    MEDIA:/tmp/example.png` then reaches `MEDIA_TOKEN_RE`, and local paths are accepted by `isValidMedia()`.
  - file: [src/media/parse.ts:519](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/media/parse.ts#L519)
  - command: `nl -ba src/media/parse.ts | sed -n '554,642p'`
  - sha: [a57d681db9b2](https://github.com/openclaw/openclaw/commit/a57d681db9b237348c5d446c8d371b2c1057f016)
- **Fence parser does not model indented code blocks:** `parseFenceSpans()` recognizes only up-to-three-space indented backtick/tilde fences. It does not return spans for 4-space or tab-indented CommonMark code blocks, matching the issue's root-cause claim.
  - file: [src/markdown/fences.ts:28](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/markdown/fences.ts#L28)

  - sha: [a57d681db9b2](https://github.com/openclaw/openclaw/commit/a57d681db9b237348c5d446c8d371b2c1057f016)
- **Existing tests do not cover this regression:** Current parser tests assert ordinary indented `MEDIA:` directives are accepted and only cover ignoring `MEDIA:` inside fenced blocks. There is no test asserting 4-space/tab indented code-block `MEDIA:` lines are preserved as text with no media extraction.
  - file: [src/media/parse.test.ts:47](https://github.com/openclaw/openclaw/blob/a57d681db9b237348c5d446c8d371b2c1057f016/src/media/parse.test.ts#L47)
