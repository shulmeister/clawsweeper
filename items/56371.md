## Summary

Current `main` now persists Control UI deleted-message state per session and filters those groups on every render, so the reported WebUI symptom of deleted messages reappearing after a new message or refresh is already addressed on `main`.

## Evidence

- **Per-session deleted-message persistence store:** `DeletedMessages` loads and saves deleted chat-group keys under `openclaw:deleted:<sessionKey>` in localStorage, which is the persistence layer for Control UI message deletion state.
  - file: [ui/src/ui/chat/deleted-messages.ts:3](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/ui/src/ui/chat/deleted-messages.ts#L3)

  - sha: [fc5920fb5134](https://github.com/openclaw/openclaw/commit/fc5920fb51346a682b269fdbeeec0971ae4ae723)
- **Chat view reads persisted delete state and records deletes:** `renderChat` creates a per-session `DeletedMessages` instance, skips rendering any group whose key is marked deleted, and the delete handler writes the group key before rerendering.
  - file: [ui/src/ui/views/chat.ts:144](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/ui/src/ui/views/chat.ts#L144)
