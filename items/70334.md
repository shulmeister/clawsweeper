## Summary

Keep this issue open. Current main still has the same overflow-compaction-success retry path and the same active-run/session-state cleanup model as the prior review. The only relevant code change since that review is cron failure-signal propagation, not a recovery path for a post-compaction retry that leaves the session in `processing` with `queueDepth=0`.

## Best Possible Solution

Keep this issue open as the canonical tracker for the post-overflow-compaction retry/session-lock freeze. The best fix is an integration regression that exercises overflow compaction followed by retry success, retry error, and a hung/lost retry terminalization path, then asserts active-run cleanup, diagnostic state leaving `processing` or entering a bounded abort state, queue depth draining, and follow-up messages proceeding. If the current timeout paths are intended to solve this, add source-backed and test-backed proof tying those paths to the reported diagnostics before closing.
