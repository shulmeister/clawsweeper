## Summary

Keep #70334 open. The report is recent, concrete, corroborated by another reporter, and #70591 was already closed as a duplicate/superseded ClawX variant of this same compaction/session-freeze problem. Current main has relevant mitigations, including an aggregate timeout for lost compaction-retry resolution and final cleanup paths, but I did not find source, test, changelog, or history evidence proving the exact `processing` + `queueDepth=0` stuck state after successful overflow compaction and prompt retry is fixed.

## Best Possible Solution

Keep #70334 open as the canonical tracker. Add a focused regression around the embedded overflow-compaction retry lifecycle: simulate successful explicit overflow compaction followed by retry success, retry error, and lost retry/lifecycle resolution, then assert the diagnostic session state returns to idle or the run is boundedly aborted, the active run registry clears, and queued followups can drain. If the root cause is stale active-run state rather than compaction wait resolution, add a conservative recovery path tied to existing abort/cleanup primitives instead of leaving `session.stuck` warn-only.
