## Summary

Keep #65923 open. Current main still implements the reported behavior: an unset `session.reset` resolves to a daily reset at 4am, the session path uses that policy to roll sessions, and tests explicitly lock the default. Docs now mention the daily default in some places, but config/help surfaces are still mixed and there is no `none`/manual-only mode or startup warning. Related #61238 overlaps and remains open, but the provided context does not establish it as a clear canonical duplicate of this narrower unset-default issue.

## Best Possible Solution

Keep this open as a focused product/UX bug. Best path is to decide the intended contract for omitted `session.reset`: either add a true manual-only/no-reset policy for unset or an explicit `mode: "none"`, with schema/types/tests/docs/config UI updated, or deliberately retain daily reset and make it highly visible in config help, FAQ, `/config` or status output, and startup/operator warnings. Coordinate wording and safety expectations with the broader open #61238 daily-reset issue rather than closing this narrower report prematurely.
