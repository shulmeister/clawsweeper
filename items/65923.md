## Summary

Current main still matches the reported behavior: omitted `session.reset` resolves to `daily` at 4am, the runtime applies that policy to roll sessions, and tests explicitly lock the default. Docs mention the daily default in some places, but config/help surfaces are mixed and there is still no supported no-reset/manual-only mode. The related broader daily-reset issue overlaps but is not enough to close this narrower unset-default report as a duplicate with high confidence.

## Best Possible Solution

Keep this open as a focused product/UX bug. Maintainers should decide the intended contract for omitted `session.reset`: either add a true manual-only/no-reset policy such as an explicit `none` mode and update schema/types/tests/docs/config UI, or deliberately retain daily reset and make it highly visible in config help, FAQ, status/setup output, and operator warnings. Coordinate with the broader open daily-reset issue rather than closing this narrower report prematurely.
