## Summary

Current main already bakes in the workaround from the issue comments: stale OpenRouter `https://openrouter.ai/v1` configs are normalized back to `https://openrouter.ai/api/v1` across provider config, resolved model metadata, and transport resolution, with regression tests and changelog evidence showing this was shipped.

## Evidence

- **Issue comments identified the root cause:** Commenters traced the failure to stale OpenRouter `baseUrl` values in `.openclaw/agents/<agent>/agent/models.json` using `https://openrouter.ai/v1` instead of `https://openrouter.ai/api/v1`, and another reporter confirmed that changing the URL fixed the bug.
- **OpenRouter catalog now canonicalizes stale `/v1` URLs:** `OPENROUTER_BASE_URL` is the canonical `https://openrouter.ai/api/v1`, and `normalizeOpenRouterBaseUrl()` maps both the canonical and legacy `https://openrouter.ai/v1` values back to the canonical URL.
  - file: [extensions/openrouter/provider-catalog.ts:3](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/openrouter/provider-catalog.ts#L3)
  - command: `nl -ba extensions/openrouter/provider-catalog.ts | sed -n '1,34p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Runtime normalization covers config, discovered models, and transport:** The bundled OpenRouter plugin applies that normalization at three layers: provider config (`normalizeConfig`), resolved model metadata (`normalizeResolvedModel`), and request transport (`normalizeTransport`). That matches the stale-`models.json` workaround from the issue thread.
