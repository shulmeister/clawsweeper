## Summary

Current `main` already implements the requested overload failover controls in core, under `auth.cooldowns.*`, and persists/translates overload failures into cooldown-based cross-request fallback behavior with probe throttling.

## Evidence

- **Config knobs exist on main:** `auth.cooldowns.overloadedProfileRotations`, `auth.cooldowns.overloadedBackoffMs`, and `auth.cooldowns.rateLimitedProfileRotations` are part of the public config type and schema help, matching the request for configurable overload retry/failover tuning.
  - file: [src/config/types.auth.ts:39](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/config/types.auth.ts#L39)
  - command: `git blame -L 17,52 -- src/config/types.auth.ts`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Overload failover defaults are already conservative:** The embedded runner resolves overload handling from config with defaults of `0 ms` backoff and `1` same-provider rotation, which is materially different from the issue's reported 30-40s retry tax.
  - file: [src/agents/pi-embedded-runner/run/helpers.ts:32](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-embedded-runner/run/helpers.ts#L32)
  - command: `git blame -L 32,47 -- src/agents/pi-embedded-runner/run/helpers.ts`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Runner escalates overloaded providers to model fallback:** On overloaded errors, the assistant failover path increments an overload rotation counter and throws a `FailoverError` to advance to model fallback once the configured cap is exceeded.
  - file: [src/agents/pi-embedded-runner/run/assistant-failover.ts:116](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-embedded-runner/run/assistant-failover.ts#L116)
  - command: `git blame -L 116,157 -- src/agents/pi-embedded-runner/run/assistant-failover.ts`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Cross-request cooldown/probe throttle exists:** Model fallback keeps probe-throttle state, skips cooldowned providers when probing is not warranted, and only probes the primary near expiry or on a limited transient probe path. That is the cross-request circuit-breaker behavior the issue asks for, even though the config names differ.
  - file: [src/agents/model-fallback.ts:537](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/agents/model-fallback.ts#L537)

- **Docs and tests describe and verify persisted overload cooldown:** The user-facing failover doc says overloaded errors get one same-provider retry and then switch to fallback without waiting, and the e2e tests verify overloaded cooldown persistence across turns with fallback behavior.
  - file: [docs/concepts/model-failover.md:219](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/concepts/model-failover.md#L219)
