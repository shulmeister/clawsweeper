## Summary

Issue #53003 is obsolete. The reported break was real for `@larksuite/openclaw-lark` v2026.3.18, but the current published plugin has already migrated off the removed root `openclaw/plugin-sdk` helpers, so the specific reported incompatibility no longer matches the current package state.

## Evidence

- **Root SDK no longer exposes broad legacy helpers:** `src/plugin-sdk/index.ts` is intentionally tiny and directs legacy consumers to the compat surface instead of the root barrel; it does not export `resolveSenderCommandAuthorization` or `isNormalizedSenderAllowed`.
  - file: [src/plugin-sdk/index.ts:1](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugin-sdk/index.ts#L1)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current migration policy is documented on main:** `docs/plugins/sdk-migration.md` documents that external plugins should move to focused `openclaw/plugin-sdk/<subpath>` imports and that compatibility work is handled through adapters, diagnostics, docs, and a deprecation window.
  - file: [docs/plugins/sdk-migration.md:21](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/plugins/sdk-migration.md#L21)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current OpenClaw install flow fails closed on declared incompatibility:** `src/plugins/clawhub.ts` rejects plugin installs when advertised `pluginApiRange` or `minGatewayVersion` is incompatible, matching `docs/tools/clawhub.md` guidance that incompatible hosts fail closed early.
  - file: [src/plugins/clawhub.ts:663](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugins/clawhub.ts#L663)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Reported old plugin version used the broken root imports:** Inspection of the published `@larksuite/openclaw-lark@2026.3.18` tarball shows calls to `plugin_sdk_1.resolveSenderCommandAuthorization(...)` and `plugin_sdk_1.isNormalizedSenderAllowed(...)` from the root `openclaw/plugin-sdk` import, matching the issue report.

## Close Comment

Closing as `cannot_reproduce` based on a Codex review of current `main` and the current published Lark plugin state. The original report was accurate for `@larksuite/openclaw-lark` v2026.3.18: that package imported legacy helpers like `resolveSenderCommandAuthorization` and `isNormalizedSenderAllowed` from the root `openclaw/plugin-sdk` surface, while current OpenClaw keeps the root SDK intentionally narrow (`src/plugin-sdk/index.ts:1`) and documents focused subpath imports plus migration guidance in `docs/plugins/sdk-migration.md:21`.

The specific reported incompatibility now appears obsolete. The current published `@larksuite/openclaw-lark` release is v2026.4.8, its package metadata declares `peerDependencies.openclaw >=2026.3.22`, and its published code now imports the supported subpaths (`openclaw/plugin-sdk/zalouser` and `openclaw/plugin-sdk/allow-from`) instead of the broken root helpers. Current OpenClaw also has fail-closed ClawHub compatibility checks in `src/plugins/clawhub.ts:663` for declared plugin API / minimum gateway ranges.

If this still reproduces with current OpenClaw and a current `@larksuite/openclaw-lark` release, open a fresh issue with exact versions, install source, and the current load/runtime logs.
