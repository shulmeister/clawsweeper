## Summary

Keep open. Current main does not implement `chat.history({ runId })`: the protocol schema still accepts only `sessionKey`, `limit`, and `maxChars`, and the handler returns bounded history without run scoping. The PR targets a real Gateway protocol gap, but the latest PR discussion still identifies a matcher overreach for punctuation/symbol run IDs, and protocol-boundary guidance requires docs/generated client artifacts to stay in sync for new payload fields.

## Best Possible Solution

Keep the PR open and ask for a revision: make run matching exact enough for arbitrary `NonEmptyString` run IDs, apply the filter before history budget trimming as already discussed, add production-path regression coverage for punctuation/symbol and escaped IDs, and update the Gateway protocol docs plus generated client artifacts required by the protocol boundary.
