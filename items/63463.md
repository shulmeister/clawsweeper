## Summary

Keep open. Current main appears to handle the suspected provider/model normalization path for slash-containing model IDs, and the reported config likely needs the OpenAI-compatible `/v1` base URL plus `api: "openai-completions"`; however, this is a recent concrete MLX report with no exact MLX-specific regression test or maintainer response, so there is not a conservative close basis yet.

## Best Possible Solution

Keep this open for maintainer triage. First ask the reporter to retry with `models.providers.mlx.baseUrl` set to `http://127.0.0.1:8080/v1` and `api: "openai-completions"`; if it still fails, add a focused regression around an MLX-like custom provider with a slash-containing model ID and confirm whether the outbound request uses `/v1/chat/completions` with model `mlx-community/Qwen3-30B-A3B-6bit`. If MLX support is desired beyond custom config, document it as a local OpenAI-compatible provider or move provider-specific automation into a plugin.
