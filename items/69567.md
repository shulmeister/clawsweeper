## Summary

Keep PR #69567 open. Current main still has custom checkout blocks in `.github/workflows/ci.yml` that derive a base64 `auth_header` from `github.token` and pass it as `AUTHORIZATION: basic ${auth_header}` without registering it with `::add-mask::`. The PR is a focused CI/security hardening, has positive bot/author evidence, and is not duplicated or implemented on main. It likely needs maintainer/security review and possibly one more workflow update because `.github/workflows/ci-check-testbox.yml` has the same derived-header pattern outside the PR's touched file.

## Best Possible Solution

Keep the PR open and ask for maintainer/security review. The best fix is to register the derived `auth_header` with `::add-mask::` immediately after construction and before any `git` invocation in every custom checkout block that uses it, including `.github/workflows/ci.yml` and likely `.github/workflows/ci-check-testbox.yml`, or explicitly document why the Testbox workflow is intentionally out of scope. A follow-up cleanup could reduce the duplicated checkout shell block so this hardening is not repeated manually across workflows.
