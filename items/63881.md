## Summary

Keep open. Current main still does not enforce a default dangerous-tool deny list on the plugin-tools MCP bridge; it only filters owner-only tools and wraps before_tool_call hooks. The PR is a plausible security hardening candidate, but it should be rebased/adapted to the current MCP handler split and should keep the dependency/release-age cleanup separate if that is still relevant.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, implement the universal dangerous-tool deny list in the current plugin-tools MCP path while preserving ownerOnly filtering and before_tool_call hooks, add focused tests for listTools/callTool default denial, gateway.tools allow/deny precedence, and HTTP-only tool scoping such as whatsapp_login, and split out any unrelated dependency release-age changes.

## Risks / Open Questions

- The PR patch is stale relative to current main’s MCP refactor: plugin-tools serving now goes through createToolsMcpServer/createPluginToolsMcpHandlers, so the fix should be rebased into that structure rather than applied mechanically.
- Changing default MCP tool exposure can break operators or plugins that intentionally expose a tool with one of the denied names; the implementation should preserve a clear gateway.tools.allow/gateway.tools.deny override contract and test listTools plus callTool behavior.
- The supplied review thread flagged the blanket axios release-age exclusion. Current main already contains axios and axios@1.15.0 in pnpm-workspace.yaml, but any dependency-policy cleanup should be separated from the MCP security fix unless a maintainer explicitly wants it bundled.
