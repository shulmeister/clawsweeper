## Summary

Keep open. Current main still does not implement the requested gateway capability fallback: `resolveGatewayModelSupportsImages()` only checks the loaded gateway model catalog and returns false on a catalog miss, while its callers do not pass the runtime config needed to inspect `models.providers[provider].models[].input`. The runner has separate configured-model metadata handling, but the specific gateway image-support decision described in this issue remains unresolved.

## Best Possible Solution

Keep this issue open and fix the gateway capability check directly. The maintainer-grade path is to let `resolveGatewayModelSupportsImages()` inspect normalized configured model metadata, likely via `buildConfiguredModelCatalog()` or a narrower shared helper, when the catalog lookup misses. Wire `cfg` through chat, agent, and node-event callers, and add regression tests for explicit provider, inferred provider, catalog miss plus config hit, and config `input: ["text", "image"]` causing attachments to remain image-capable.
