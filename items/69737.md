## Summary

Keep #69737 open. Current main still has both reported behaviors: the lifecycle error path can still derive user-facing error text from `lastAssistant.errorMessage`, and the visible assistant text extraction still joins multiple text blocks with newlines. The existing issue comment correctly notes that Bug 1 and Bug 2 have different blast radii; Bug 2 in particular is shared extractor policy, not a narrow Slack/Telegram-only change.

## Best Possible Solution

Keep the issue open. The best maintainer path is to split the work: remove or strictly genericize the raw `lastAssistant.errorMessage` fallback in the lifecycle/channel error path with a regression test for OpenAI-style 5xx prose, then handle multi-text assistant output as a separate shared extractor/delivery policy decision, likely with targeted tests for Slack/Telegram delivery plus shared extractor callers.
