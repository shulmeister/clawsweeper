## Summary

Keep open. Current main has redaction and some friendly error classification, but the requested delivery policy is not implemented: unknown assistant `stopReason: "error"` text can still be derived from `lastAssistant.errorMessage`, and assistant visible text still concatenates multiple matching text blocks before channel block-reply delivery. The related #42243 context is overlapping but not canonical for this item because it tracks cron announce all-error delivery and does not cover the multi-text assistant-message policy.

## Best Possible Solution

Keep #69737 open. Implement a narrow delivery-layer fix for unknown `stopReason: "error"` messages, likely removing raw `lastAssistant.errorMessage` from user-facing fallback while keeping redacted diagnostics in logs. Handle the multi-text case as a separate policy decision or config-backed delivery option, with tests covering Slack/Telegram block replies and any shared extractor surfaces intentionally changed.
