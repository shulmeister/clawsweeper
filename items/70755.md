## Summary

Current main still uses the BlueBubbles server/API password as the BlueBubbles webhook authentication token, and the linked fix PR #70757 is still open rather than merged. The issue should remain open until a dedicated webhook secret/config surface lands and the runtime, schema, setup, docs, and tests stop relying on `channels.bluebubbles.password` for inbound webhook auth.

## Best Possible Solution

Keep #70755 open and use PR #70757, or an equivalent maintainer-reviewed patch, to add a separate BlueBubbles webhook secret with account/top-level schema support, runtime auth against that secret only, setup/docs updates that stop publishing the API password, SecretRef registration, validation, and regression tests for the old inherited-password path.
