## Summary

Current main already addresses the reported subcommand-help slowdown by using primary-only command/subcli registration for help and by deferring the expensive status/agent runtime behind dynamic imports. The exact implementation differs from the issue's proposed route-first patch, but the eager pi-embedded/jiti bootstrap path described in #59135 is no longer how builtin subcommand help is wired.

## Evidence

## Risks / Open Questions

- I did not run live timing benchmarks, because this review had to keep the checkout read-only and avoid commands that could write caches or state.
- `tryRouteCli()` still returns false for help in `src/cli/route.ts:41`, so the fix is architectural laziness in the fallback path rather than the exact route-first change proposed in the issue.
- Some plugin-provided or non-builtin commands can still have their own cold-start costs, but the specific eager pi-embedded/jiti help path described in #59135 is no longer how builtin subcommand help is wired on main.
