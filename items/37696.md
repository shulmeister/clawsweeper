## Summary

Keep open. Current main still has the reported strict-mode data-loss behavior: an unclosed reasoning tag leaves the parser in `inThinking`, so strict mode omits the remaining assistant text. Existing tests still assert that drop behavior, and shared/channel delivery paths still use strict stripping. Related PR #51915 is a direct open fix candidate, while earlier fix PRs #37703, #37861, and #46205 were closed unmerged, so the issue is not implemented or superseded by shipped code.

## Best Possible Solution

Keep #37696 open until OpenClaw lands a source-level fix, likely by updating `stripReasoningTagsFromText` to recover visible text after malformed unclosed reasoning tags and adding regression coverage for shared extraction plus affected streaming paths. PR #51915 or a narrower successor is the natural canonical implementation path; if maintainers intentionally want strict-mode discard, document that decision and adjust channel behavior so users do not see silent empty responses.
