## Summary

Keep #37696 open. Current main still implements the reported strict-mode behavior: an unclosed reasoning tag leaves the parser inside a thinking block, so strict mode drops the trailing assistant text. Checked-in tests still assert that loss, assistant extraction still sanitizes through the strict delivery profile, and message/channel delivery paths still skip empty sanitized text. The linked fix attempts in the provided context remain unshipped: #37703, #37861, and #46205 are closed unmerged, while #51915 is still open.

## Best Possible Solution

Keep #37696 open until a focused fix lands. The likely path is to update `stripReasoningTagsFromText` strict-mode behavior and regression tests so malformed or unclosed reasoning tags cannot erase all deliverable assistant text, then verify assistant extraction plus message-end and channel preview delivery paths. #51915 is the current open fix candidate to merge, revise, or supersede; if maintainers intentionally want strict-mode discard for privacy, document that contract and add a delivery-level fallback so users do not get silent non-delivery.
