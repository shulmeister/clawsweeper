## Summary

Current main still matches the reported bug. `stripReasoningTagsFromText` leaves `inThinking=true` for an unclosed `<think>` tag and only appends the trailing text when `!inThinking` or `mode === "preserve"`, so strict mode can still erase the deliverable response. Checked-in tests still assert that strict/unclosed reasoning tags collapse to a prefix or empty output, and shared delivery/channel paths still use strict reasoning stripping. The linked fix attempts #37703, #37861, and #46205 are closed unmerged; #51915 remains an open fix candidate, not shipped behavior.

## Best Possible Solution

Keep #37696 open until a focused fix lands. The likely path is to update `stripReasoningTagsFromText` so malformed/unclosed reasoning tags cannot erase all deliverable assistant text, adjust strict-mode and `extractAssistantText` tests, and either merge or supersede #51915 with the chosen safety semantics. If maintainers prefer a different tradeoff, capture it explicitly in the sanitizer contract and channel tests.
