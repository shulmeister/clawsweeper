## Summary

Keep this issue open. Current main still feeds startup memory into bare `/new` reset agent runs, and the bundled session-memory hook can persist prior assistant content into those dated memory files. Existing tests affirm that reset/startup memory is prepended to the agent prompt; I did not find a source or regression test that prevents a `[note self-improvement (before reset): ...]` memory artifact from being repeated in the first Telegram-visible reply.

## Best Possible Solution

Add a focused regression test for Telegram/native bare `/new` with startup memory containing `[note self-improvement (before reset): ...]`, then change the reset startup path so internal memory or hook notes cannot become the first user-visible reply. Likely areas are `src/auto-reply/reply/get-reply-run.ts`, `src/auto-reply/reply/startup-context.ts`, and the session-memory filtering/prompt contract.
