## Summary

Keep #40247 open. Current main still returns only the primary non-loopback bind host from `resolveGatewayListenHosts()`, while local gateway client details still target `127.0.0.1`; that combination leaves the reported tailnet/custom-bind self-connection failure plausible. The PR is a focused gateway fix with concrete linked reproductions and tests, and the related open #50751 overlaps CLI host resolution but does not supersede this server-side dual-bind approach.

## Best Possible Solution

Keep the PR open for maintainer review. The best product path is either to land a current-main version of this server-side dual-bind fix with the PR's tailnet/custom regression tests, or explicitly choose the related client-side host-resolution path in #50751 and close this only after that shipped path demonstrably fixes internal calls, CLI probes, and `sessions_spawn` for `gateway.bind=tailnet` and `gateway.bind=custom`.
