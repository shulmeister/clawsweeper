## Summary

Keep #69401 open. Current main has structured untrusted metadata blocks and a shared external-content wrapper, and Discord has a partial guild-context wrapper, but the generic inbound channel body still reaches the agent as raw BodyForAgent for Slack, Telegram, WhatsApp, and other channels. The memory/dreaming ingestion path strips channel metadata and keeps plain user text without channel provenance, so the requested cross-channel body wrapping and actor-aware memory attribution are not implemented on main.

## Best Possible Solution

Keep the issue open. The best path is a shared inbound-body trust boundary in the core reply pipeline or finalized channel context contract, reusing `wrapExternalContent`-style boundaries for agent-facing message bodies while preserving raw command parsing/display fields. Add cross-channel tests for Slack, Telegram, WhatsApp, Discord, and memory/dreaming ingestion, including provenance fields for channel, sender, timestamp, and session filtering.

## Risks / Open Questions

- A correct fix will touch shared prompt construction, channel context contracts, command parsing, transcript/UI stripping, and memory ingestion; it needs targeted regression tests rather than a channel-only patch.
- Existing external-content wrappers sanitize special tokens and boundary spoofing, so a new channel-body wrapper should reuse that helper or preserve equivalent guarantees.
- Care is needed to keep `BodyForCommands` and raw UI display behavior stable while changing the agent-facing `BodyForAgent` prompt shape.
