## Summary

Keep #69401 open. Current main has external-content wrapping primitives and wraps some channel metadata, but representative inbound channel paths still pass the primary user message body into the agent prompt as raw user-role text after only newline/system-tag sanitization. The memory/session export path strips OpenClaw metadata and keeps the user text, while channel ingress does not attach per-channel external-user provenance for downstream filtering. The commenter’s nonce/tag-boundary and memory-provenance points remain valid design concerns, not shipped behavior.

## Best Possible Solution

Keep this open and implement it in the shared inbound prompt path, not as a per-channel workaround. The likely direction is to wrap only the model-facing current message body with the existing external-content boundary mechanism or a channel-specific variant with nonces, keep BodyForCommands/RawBody clean for command detection, attach InputProvenance with sourceChannel/sourceSessionKey for external channel turns, and update memory/session export or retrieval so untrusted-channel provenance can be filtered or downgraded. Add regression tests for Slack/Telegram/direct plugin ingress, prompt text shape, tag-boundary spoofing, and memory/session export behavior; then align the security docs with the shipped behavior.

- A fix needs care to preserve raw command parsing and transcript readability while wrapping the model-facing body and carrying provenance into memory/retrieval.
