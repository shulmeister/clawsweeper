## Summary

Keep this issue open. Current main still models `plugins.slots.memory` and `plugins.slots.contextEngine` as bare string IDs in public config types, runtime schema validation, normalization, activation, install/write, uninstall/update reset paths, tests, and docs. The RFC asks for a core plugin/config contract change with `{ owner, claimed_at, claimed_by_version }` read/write support plus takeover invariants; that remains unimplemented and is not a duplicate or ClawHub-only request.

## Best Possible Solution

Keep the RFC open for maintainer design review. A concrete implementation path would add a backwards-compatible slot ownership union, normalize legacy string slots to an owner object on read, update install/register/write paths to persist canonical ownership metadata, add explicit conflict and force semantics for slot takeover, and cover validation, generated schema/help, docs, install/update/uninstall, activation, default slots, and `"none"` behavior with focused tests.
