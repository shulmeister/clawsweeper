## Summary

Keep #70823 open. Current main has an exclusive plugin-slot mechanism, but it still models `plugins.slots.memory` and `plugins.slots.contextEngine` as bare plugin-id strings and lacks the proposed object read shape, provenance fields, and overwrite invariant. This is also not a ClawHub-only request: the current core schema rejects the proposed object form, and the requested behavior requires core config/schema/resolver/write-path changes.

## Best Possible Solution

Keep the RFC open for maintainer API/product judgment. A good implementation path would add a typed slot-owner union or normalized internal slot-owner shape, accept legacy strings on read, update Zod/schema/help/docs, teach validators/resolvers/install-selection/uninstall paths to use `owner`, and add focused regression tests before considering provenance fields and overwrite enforcement.

## Risks / Open Questions

- The proposal changes a public config contract, so it needs backward-compatible schema, generated config docs/help updates, and careful handling of existing string values.
- The overwrite invariant needs maintainer design for manual config edits, plugin install/update/uninstall flows, default inferred slots, and any future force/allow_override semantics.
