## Summary

Keep #64448 open. The report is recent, concrete, and describes Feishu image duplicate delivery with logs. Current main still sends Feishu media through direct plugin delivery without carrying a stable outbound idempotency key into the Feishu message create/reply call. The existing 120s media HTTP timeout may reduce one timeout class, but it is not a duplicate-send/idempotency fix, and I did not find a regression test covering this reported reconnect/retry behavior. The only issue comment is unrelated setup-guide spam and does not change triage.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin/direct outbound path. The likely durable direction is to propagate a stable outbound idempotency key into direct plugin sends, then set Feishu `im.message.create`/`im.message.reply` `data.uuid` for media and text/card paths. Add a Feishu regression test that proves duplicate/retried media delivery reuses the same UUID or is otherwise suppressed, plus a dependency-backed check for the exact Lark SDK field.

## Risks / Open Questions

- I did not live-test against Feishu credentials, so the exact WebSocket reconnect duplicate was not reproduced against the service.
- The issue body names `idempotent_id`, but the current Lark SDK type for IM message send/reply uses `uuid`; a fix should verify Feishu's current API semantics before implementation.
- A message `uuid` would dedupe final message creation/reply, but it would not by itself dedupe repeated image uploads; tests should cover the observable duplicate chat delivery.
