## Summary

Keep this issue open. The report has concrete Feishu image-duplicate symptoms, and current main still routes Feishu media sends through upload-plus-message-create without a provider idempotency token or a Feishu-specific in-flight guard.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin media path. Thread a stable per-send idempotency value from `openclaw message send`/message actions into Feishu media delivery, apply the supported Feishu/Lark API dedupe field on the final image/file message send after verifying the SDK types/docs, and add a regression test that simulates duplicate/retried media send behavior without suppressing intentional later sends.

## Risks / Open Questions

- The exact Feishu/Lark SDK/API request field for outbound message idempotency must be verified before implementing; the issue names `idempotent_id`, while SDK/API versions may expose a different field such as a UUID-style dedupe token.
- A local in-flight map keyed only by media URL could incorrectly suppress intentional repeated sends; a per-send provider idempotency token or explicit command idempotency propagation is safer.
