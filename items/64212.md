## Summary

Keep this issue open. Current main does implement the adjacent custom-provider image routing path, but I did not find source, test, or release evidence that the NVIDIA NIM Kimi K2.5 image-tool abort itself is fixed. The current image path still runs generic OpenAI-compatible image completion with fixed timeouts that can surface `Request was aborted`, the bundled NVIDIA Kimi catalog remains text-only, and related NVIDIA/Kimi tool-compatibility reports remain open rather than proving this image case is resolved.

## Best Possible Solution

Keep the issue open and add a focused maintainer reproduction/regression for `agents.defaults.imageModel.primary: "nvidia-vision/moonshotai/kimi-k2.5"` with `input: ["text", "image"]`. Then fix the observed failure in the owner path: either mark/support NVIDIA Kimi vision in the NVIDIA provider, add any needed OpenAI-compatible payload/compat handling for NVIDIA NIM, and/or make the image tool honor configured timeout settings for slow vision models.
