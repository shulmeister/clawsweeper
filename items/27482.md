## Summary

Keep #27482 open. Current main still has separate video understanding/generation surfaces, but the requested direct chat/message video upload path remains image-only: the Control UI picker/paste/drop guards only accept images, UI previews assume `<img>`, chat.send serialization emits every attachment as `type: "image"`, the Gateway attachment parser only returns `ChatImageContent[]`, and model dispatch exposes `images` rather than a video attachment/content contract. The prior April 26 Codex review comment also explicitly kept this open for the same product gap, and the relevant files did not change between that reviewed SHA and current main.

## Best Possible Solution

Keep #27482 open until OpenClaw adds first-class direct video chat attachments in the core Control UI and Gateway path: accept and preview MP4/WebM/MOV, add a video-capable attachment/content model, route to provider-native video input or the existing media-understanding video runtime with capability and size checks, and cover UI, Gateway, provider, and docs behavior with focused tests.
