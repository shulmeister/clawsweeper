## Summary

Keep this issue open. Current main still rejects `plugins.entries.exa.config.webSearch.baseUrl`, the Exa runtime still only reads `apiKey`, requests still go to the hard-coded Exa endpoint, and the related implementation PR #54939 is open and unmerged.

## Best Possible Solution

Keep this issue open and review PR #54939 as the implementation candidate. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json` and UI hints, plumb it through `extensions/exa/src/exa-web-search-provider.runtime.ts`, include the endpoint in cache-key behavior, document it on https://docs.openclaw.ai/tools/exa-search and https://docs.openclaw.ai/tools/web, and add Exa plugin coverage proving config validation accepts `baseUrl` and runtime requests use it. If maintainers decide Exa should intentionally not support custom endpoints, document that explicitly and improve provider-specific validation guidance instead.
