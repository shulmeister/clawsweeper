## Summary

Keep open. Current main still lacks a cron-scoped `tools.cron.tools.allow/deny` config surface and does not make cron sessions inherit `tools.subagents.tools`; cron only has per-job `payload.toolsAllow` plus delivery-mode-specific message-tool handling. The existing Apr 26, 2026 issue comment reached the same conclusion and remains consistent with current source inspection.

## Best Possible Solution

Keep #12736 open for maintainer product direction. A complete implementation should add a cron-scoped policy such as `tools.cron.tools.allow/deny` or deliberately document and implement inheritance from `tools.subagents.tools`, define precedence with existing tool-policy scopes and per-job `toolsAllow`, reconcile message-tool behavior across cron delivery modes, then update schema/types/help/generated config metadata, docs, and regression tests.

## Risks / Open Questions

- A future fix needs an explicit precedence model against global, provider, agent, group, sandbox, subagent, and per-job `payload.toolsAllow` policy so cron restrictions do not silently widen or over-narrow tool access.
- Cron delivery-mode message-tool handling is already special-cased, so a cron deny policy must reconcile `announce`, `webhook`, and `none` delivery behavior rather than only adding schema fields.
