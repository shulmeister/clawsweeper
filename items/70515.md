## Summary

Keep #70515 open. Current main still swallows backup restore copy failures in both async and sync suspicious-read recovery, still logs the success message unconditionally, still writes `valid: true`, and the observe audit schema has no restore error fields. The PR is a focused fix with regression-test intent, and the provided review discussion shows the only bot P2 follow-ups were addressed rather than making the PR obsolete.

## Best Possible Solution

Keep the PR open or land an equivalent maintainer patch. The right product/code direction is to capture async and sync backup-copy failures, log a distinct failure warning, set observe-audit `valid` from actual restore success, persist nullable restore error details, default those fields on non-recovery observe events, and keep both async and sync regression tests.
