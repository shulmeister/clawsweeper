## Summary

Keep PR #70515 open. Current main still has the exact false-success bug the PR fixes: both suspicious config read recovery paths swallow backup copy failures, emit the success warning, and write observe audit records as valid. The PR is focused, has regression-test coverage in its diff, and the provided review discussion says the only P2 follow-ups were addressed in 4da2add. Related PR #70676 is a separate gateway revocation silent-failure fix, not a replacement for this config recovery change.

## Best Possible Solution

Keep the PR open and have maintainers review or land a rebased version of the same fix: capture async and sync backup-copy errors in `src/config/io.observe-recovery.ts`, log a distinct restore-failed warning, mark the observe audit record invalid when restore fails, add nullable restore error fields to the audit record, and keep the async plus sync regression tests from the PR.
