## Summary

Current main does not fully fix #66822. The user-configured `remote.batch.concurrency` is parsed, but memory indexing only uses it while provider batch mode is active; when batch mode is disabled or unavailable, indexing falls back to a hard-coded concurrency of 4. The directly linked fix PR #66931 is still open/unmerged in the provided GitHub context, so this remains a real bug rather than cleanup-closeable work.

## Best Possible Solution

Keep #66822 open until a focused fix lands. The best path is to make non-batch/inline memory indexing honor a user-configured positive concurrency value, or add a dedicated documented `memorySearch.sync` indexing concurrency setting, then add regression coverage for `batch.enabled: false` and providers without `runtime.batchEmbed` such as Ollama/OpenAI-compatible local endpoints.
