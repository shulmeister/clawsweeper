## Summary

Keep open. Current main preserves and reports provider `stopReason: "length"`, but it does not implement the requested configurable `onLengthTruncation` behavior for `mode: "run"` subagents. A partial visible response can still take the normal success path, and subagent completion currently marks any `agent.wait` status `ok` as an `ok` outcome. The related #63210 is complementary main-session work, not a canonical replacement for this subagent-mode request.

## Best Possible Solution

Keep the issue open. Implement a subagent-specific runtime policy under the existing agent/subagent config surface: first add `fail` behavior that turns `stopReason: "length"` in `mode: "run"` into an error lifecycle/task outcome surfaced to the requester, with regression tests for partial visible output and empty reasoning-only output. Then add optional bounded `continue` handling with `maxContinuations`, side-effect/replay guards, schema/docs updates, and alignment with the complementary main-session tracking in #63210.

## Risks / Open Questions

- The implementation should coordinate with #63210 so main-session and subagent truncation handling share provider-stop-reason semantics without conflating the two user-visible workflows.
- A naive continue mode could duplicate side effects or repeat partial file writes; the first low-risk step is likely a hard fail/signal for `mode: "run"` length truncation, followed by bounded continuation only where replay is safe.
