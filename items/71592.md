## Summary

Keep #71592 open. Current main still advertises shared `/status` and `/compact` in the TUI slash list, but the TUI command handler has no local `status` or `compact` cases. In local mode, the default branch sends the raw slash text through `sendMessage`, which records it as a user message and calls the embedded backend; that backend starts a local `agentCommandFromIngress` turn rather than routing through the gateway chat command handlers. The earlier Clawsweeper comment is still accurate against current main SHA 9089e6b595ba33b4afc27b99c56ace80c4fbbc7b.

## Best Possible Solution

Keep the issue open and fix the TUI local embedded command boundary. Recognized shared commands such as `/status` and `/compact` should either execute through a shared/local command adapter or return an explicit local unsupported/system response in `opts.local` mode, but they should not fall through to `sendMessage(raw)`. Add focused regression tests proving local `/status` and `/compact` do not reach `sendChat` / `EmbeddedTuiBackend` / `agentCommandFromIngress`, while preserving intentional gateway-connected forwarding for commands handled by `chat.send`.

## Risks / Open Questions

- A broad change to block all unknown slash commands in `handleCommand()` could regress gateway-connected TUI forwarding for shared commands such as `/context`; the fix should be local-mode-aware or use a shared command execution adapter.
- There is a product choice between implementing local TUI handlers for `/status` and `/compact` versus returning an explicit local unsupported response, but the current fallthrough-to-model behavior is still a real bug.
