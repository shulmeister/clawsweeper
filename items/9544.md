## Summary

Keep open. Current main still imports and calls Baileys `useMultiFileAuthState` for WhatsApp auth, then wraps `state.keys` with `makeCacheableSignalKeyStore`; only `creds.json` gets OpenClaw's atomic persistence mitigation. Tests and config/docs still model `authDir` as Baileys multi-file auth state, and the related PR #66863 is closed unmerged per the provided GitHub context, so it is prior art rather than a shipped fix.

## Best Possible Solution

Keep #9544 open until the WhatsApp plugin replaces `useMultiFileAuthState` with a production auth store for credentials and signal keys, adds migration from the existing Baileys multi-file layout, preserves or deliberately supersedes the current atomic `creds.json` safety behavior, and covers the path with focused WhatsApp plugin tests. Treat PR #66863 as prior art, not as the canonical shipped fix.
