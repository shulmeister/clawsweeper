## Summary

Keep open. Current main still uses Baileys `useMultiFileAuthState` for the WhatsApp socket and only adds a narrower atomic `creds.json` persistence mitigation; it has not moved credentials plus signal keys to a SQL/consolidated production auth store. Related PR #66863 is useful prior art, but it was closed after review and supersession discussion, and current main does not contain its `useAtomicAuthState` path. ([github.com](https://github.com/openclaw/openclaw/pull/66863))

## Best Possible Solution

Keep #9544 open until the WhatsApp plugin replaces `useMultiFileAuthState` with a production auth store for credentials and signal keys, adds migration from the existing Baileys multi-file layout, preserves or deliberately supersedes the current atomic `creds.json` safety behavior, updates auth presence/logout/read helpers, and covers fresh install plus migration with focused WhatsApp plugin tests.
