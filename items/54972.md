## Summary

Keep open. Current main still sends Feishu streaming replies as CardKit `card_id` interactive messages, while the quote/thread read paths do not request raw CardKit content and the parser has no `json_card` or `card_id` resolution path. The related implementation PR #55466 was closed unmerged, so the reported `[Interactive Card]` behavior remains actionable.

## Best Possible Solution

Keep this issue open and rework or revive the #55466 approach: validate the current Feishu/Lark API contract, request raw CardKit content from both quoted-message and thread-history reads, parse `json_card` summary/body text before legacy element/post fallback, and add regression tests for quoted-message plus thread-history reads of streaming CardKit cards.
