## Summary

Keep open. Current `main` still returns `sessions_send.status = "timeout"` immediately when the first `agent.wait` call times out, with no post-timeout compensation to distinguish "accepted but still running" from an actual delivery failure. The gateway/docs contract still says `agent.wait` timeout is wait-only and does not stop the run, so the reported misclassification remains plausible on current code.

## Evidence

- **`sessions_send` still hard-returns timeout on first wait timeout:** After starting the target run, `createSessionsSendTool()` calls `waitForAgentRunAndReadUpdatedAssistantReply(...)`. If that helper returns `status === "timeout"`, the tool immediately returns `{ status: "timeout" }` and does not inspect session history or convert to an accepted/pending delivery state.
