## Summary

Keep this issue open. The report is recent, hardware-specific, and describes a visible macOS chat-window compositor problem with concrete measurements. Current main has a newer native SwiftUI chat implementation, but I did not find source, tests, docs, or release history proving that the WindowServer CPU regression is fixed; the current macOS chat surface still uses transparent/non-opaque windows, NSVisualEffectView/material backgrounds, shadows, scroll animations, and an animated typing indicator that are plausible compositor stressors on an older Intel Retina GPU.

## Best Possible Solution

Keep the issue open and treat it as a macOS chat rendering performance investigation. The likely path is to profile the native chat window on affected Intel Retina hardware, compare normal versus opaque/no-material/no-shadow/no-animation variants, then ship a targeted fix such as a low-compositor chat mode, reduced material usage, or honoring Reduce Motion/older Intel GPU heuristics for chat animations and translucent surfaces. If the native SwiftUI rewrite is intended to supersede the old chat UI, verify it with WindowServer measurements before closing as implemented.
