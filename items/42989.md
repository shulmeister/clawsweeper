## Summary

Current `main` has moved new provider integrations behind the plugin/provider-plugin system rather than adding more provider-specific wiring in core. This PR implements Model Hub by editing legacy core registries (`src/agents/*`, `src/commands/*`, `src/media-understanding/*`, `src/providers/*`), but current maintainer guidance says provider auth/onboarding/catalog behavior should live in a provider plugin or external plugin package instead.

## Close Comment

Thanks for the contribution. I reviewed this against current `main` as a Codex maintainer pass, and I’m closing it because Model Hub now fits the provider-plugin / ClawHub path rather than additional core wiring.

The key evidence on current `main` is:
- `VISION.md:52-59` says optional capability should usually ship as plugins, with a high bar for adding more to core.
- `extensions/AGENTS.md:39-41` says provider auth, onboarding, catalog selection, and vendor-only behavior should stay in the provider plugin, not core.
- `docs/plugins/sdk-provider-plugins.md:11-13` and `:31-95` document the supported way to add an OpenAI-compatible provider through a manifest-driven provider plugin.
- Current setup/env-var discovery is plugin-manifest driven in `src/commands/auth-choice-options.ts:26-56` and `src/secrets/provider-env-vars.ts:76-117`, so the PR’s edits to static core provider/auth registries are targeting an older integration shape.
- A current bundled example is `extensions/kilocode/openclaw.plugin.json` plus `extensions/kilocode/index.ts:10-33`, which is the pattern this provider would need to follow now.

If Model Hub is still worth shipping, the next step would be to reintroduce it as a provider plugin (bundled or external/ClawHub), not as more provider-specific logic in `src/agents/*`, `src/commands/*`, and other core registries.
