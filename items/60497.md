## Summary

Keep open. Current main does classify shell exit 127 as a failed exec result, but the embedded reply path still suppresses ordinary non-timeout exec tool errors unless verbose detail is enabled, and it can preserve assistant text after a tool failure. Existing tests explicitly cover suppressing `Command not found` exec errors at default/off verbosity, so the reported user-visible failure mode is not resolved on main. The related open PR #60507 is a companion HIL/destructive-command catalog, not a closing implementation for this agent-loop truthfulness bug.

## Best Possible Solution

Keep this issue open and fix the embedded agent loop/reply policy for exec-like mutating tool failures. A good fix should surface the exact failed tool result for default local runs, prevent success-looking assistant text from being delivered when the matching exec action failed, and add regression coverage for `python: command not found`/exit 127 in local embedded mode with default verbosity. The sandbox-side Python symlink workaround can remain upstream; OpenClaw core still needs fail-closed user-facing behavior after tool failure.
