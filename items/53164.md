## Summary

Current `main` does not contain the ClawHub marketplace moderation pipeline described in the report, and the OpenClaw skill install/search paths in this checkout do not surface or enforce a ClawHub-side `"Suspicious - Medium Confidence"` flag for skills. This reads as a ClawHub service/backend moderation issue rather than a reproducible bug in `openclaw/openclaw` current `main`.

## Evidence

- **Native skill installs bypass the in-repo installer-metadata scanner path:** Docs say gateway-backed `skills.install` uses the built-in dangerous-code scanner, but `openclaw skills install <slug>` is different: it downloads a ClawHub skill folder and does not use that installer-metadata path.

## Close Comment

Closing from a Codex review of current `main` because I could not reproduce this as an `openclaw/openclaw` codebase issue. The in-repo skill paths here either proxy ClawHub skill metadata/downloads (`src/gateway/server-methods/skills.ts:167`, `src/agents/skills-clawhub.ts:228`, `src/agents/skills-clawhub.ts:288`) or run the local dangerous-code scanner for gateway-backed installer metadata, and the docs explicitly note that native `openclaw skills install <slug>` is a different direct-download flow (`docs/tools/skills.md:136`). The ClawHub client types on current `main` expose verification fields for package endpoints, but not skill moderation/trust fields (`src/infra/clawhub.ts:26`, `src/infra/clawhub.ts:59`, `src/infra/clawhub.ts:114`, `src/infra/clawhub.ts:538`). Repo docs also describe the moderation logic as ClawHub-side (`docs/security/THREAT-MODEL-ATLAS.md:317`, `docs/security/THREAT-MODEL-ATLAS.md:456`). In other words, there is nothing in this checkout that matches or fixes the reported marketplace `"Suspicious - Medium Confidence"` flag, so this does not map to a reproducible current-main bug in the OpenClaw repo.
