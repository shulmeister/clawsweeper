## Summary

Keep #71142 open. Current main supports Control UI image uploads, but the requested configurable Control UI upload limit is not implemented: `chat.send` still passes a hardcoded 5,000,000-byte cap into attachment parsing, oversized attachments are rejected there, and `gateway.controlUi` has no schema/type/docs surface for `maxUploadSizeMb` or per-media `mediaLimits`.

## Best Possible Solution

Add a first-class Control UI/WebChat upload limit configuration, likely under `gateway.controlUi`, with a conservative default preserving current behavior. Plumb it into the `chat.send` attachment parser, advertise or fetch the effective limit for the Control UI so users get pre-send validation, document the new field in the public Control UI/config docs, and add tests for default behavior, configured larger limits, schema validation, and oversized rejection.

## Risks / Open Questions

- Raising the cap needs server-side enforcement and client feedback; relying only on browser-side checks would leave other `chat.send` clients able to exceed the intended policy.
- The WebSocket policy is currently 25 MB max payload, so any configurable limit above that needs a transport-aware design rather than just increasing the attachment parser cap.
