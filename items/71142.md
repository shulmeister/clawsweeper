## Summary

Keep this issue open. The prior Codex review still matches current main: Control UI serializes browser-selected files into `chat.send` attachments, and the server-side ingestion path still applies a hardcoded 5,000,000 byte cap. The `gateway.controlUi` config contract has no upload-size field and remains strict, while `tools.media.*.maxBytes` is a later media-understanding policy, not the Control UI upload-ingestion limit.

## Best Possible Solution

Implement a documented upload policy rather than closing this. A conservative path is to add either `gateway.controlUi.maxUploadSizeMb` or per-kind `gateway.controlUi.mediaLimits`, wire it through config types, zod schema, generated schema/help metadata, Control UI runtime/error display, and the `chat.send` attachment parser. Add coverage for the default 5 MB behavior, raised limits, oversized rejection, and any WebSocket/base64 or media-store constraints for larger uploads.

## Risks / Open Questions

- Raising only the parser cap may still fail for larger raw files because browser uploads are base64-encoded over the Gateway WebSocket, whose post-auth max payload is 25 MiB.
- The product/API decision remains whether this should be `gateway.controlUi`-specific or a shared Gateway `chat.send` attachment policy used by other clients too.
