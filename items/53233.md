## Summary

Current `main` already scopes cooldown handling to the candidate provider's own auth profiles, so an Anthropic cooldown does not propagate to an independent fallback provider like Google Vertex, especially when Vertex has no `google-vertex:*` auth profile. The fallback loop only applies cooldown skipping/probing when `resolveAuthProfileOrder()` returns profiles for that candidate provider, and current tests explicitly cover cross-provider fallback from an Anthropic rate-limit state and no-profile fallback providers.

## Close Comment

Closing as `implemented_on_main` based on this Codex review. Current `main` scopes fallback cooldown handling to the candidate provider's own auth profiles, so an Anthropic cooldown does not propagate to an independent fallback provider. The relevant guard is in `src/agents/model-fallback.ts:755-764`, where cooldown logic only runs when the candidate provider has matching profiles, and `src/agents/auth-profiles/order.ts:34-40` rejects cross-provider profile reuse as `provider_mismatch`. Current regression coverage also includes cross-provider fallback after an Anthropic rate-limit and no-profile fallback providers in `src/agents/model-fallback.test.ts:1713-1815`. The behavior is present on `main` in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`, but I could not tie it to a released tag yet, so `fixedRelease` is left null.
