## Summary

Keep #42259 open. Current main still does not implement configurable image-tool compression: the strict agent defaults config omits `agents.defaults.imageCompressionQuality` and `agents.defaults.imageCompressionMaxSide`, the image tool passes only `maxBytes` into media loading, and JPEG optimization still uses fixed side/quality ladders. The PR is a plausible implementation attempt for the linked #41771 request, but it needs revision before merge because the review discussion identifies constraint fallback bugs and the branch includes unrelated docs/gateway changes.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up. The best path is a focused image-tool config change that adds the two strict config fields plus labels/help/docs, threads `minQuality` and `maxSide` from `createImageTool` to `loadWebMedia` and `optimizeImageToJpeg`, makes out-of-ladder configured values honor the promised floor/cap, and adds regression tests for values like quality 95/85 and maxSide 600 while dropping unrelated docs/gateway commits from the branch.
