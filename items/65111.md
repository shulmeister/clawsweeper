## Summary

Keep open. The underlying UX report looks real: `commands` is a built-in chat command, not a core CLI primary. But this PR's guard is unsafe because it would also treat ordinary plugin ids without explicit `plugins.entries` config as built-ins, regressing the existing `plugins.allow` diagnostic for bundled plugin commands like `browser`.

## Evidence

- **Built-in chat command registry contains `commands`:** The built-in chat command registry defines `commands` as a native slash/text command, which supports the report that `openclaw commands` is not backed by a plugin CLI registration path.
  - file: [src/auto-reply/commands-registry.shared.ts:139](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/auto-reply/commands-registry.shared.ts#L139)

  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Core CLI does not expose a `commands` primary:** Core CLI descriptors list the built-in primary commands and do not include `commands`, so the current CLI has a real mismatch between chat-command naming and CLI/plugin resolution.
  - file: [src/cli/program/core-command-descriptors.ts:6](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/cli/program/core-command-descriptors.ts#L6)

  - file: [src/cli/run-main.test.ts:95](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/cli/run-main.test.ts#L95)
