## Summary

Keep this PR open. It targets a real UX bug: `commands` is a built-in chat command, while current `main` still routes an unknown primary command through the `plugins.allow` diagnostic and would produce the misleading add-`commands` message. The submitted patch is not safe as-is because the review comments are right: using `plugins.entries.<id>` as the plugin discriminator would suppress valid allowlist diagnostics for real plugin commands that have no explicit config entry.

## Best Possible Solution

Keep the PR open for changes or replace it with a safer superseding PR. The fix should distinguish known plugin ids from core chat commands using the manifest/registry surface rather than `plugins.entries` presence, preserve the existing blocked-plugin diagnostic for `browser`, and add focused tests for both `openclaw commands` under a restrictive `plugins.allow` and a real plugin command excluded by the allowlist.
