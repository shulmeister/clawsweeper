## Summary

Keep open: current `main` still turns a missing `node:sqlite` runtime failure into the generic `embedding/provider error` warning and `Check embedding provider configuration` action, so the reported UX bug is still present.

## Evidence

- **Generic unavailable mapping is still unchanged:** `buildMemorySearchUnavailableResult` treats every non-quota failure as an embedding/provider problem and returns the same generic warning/action text.
  - file: [extensions/memory-core/src/tools.shared.ts:120](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/memory-core/src/tools.shared.ts#L120)

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **memory_search still uses that mapping for manager-init failures:** When memory manager creation returns `{ error }`, `memory_search` immediately returns `buildMemorySearchUnavailableResult(memory.error)`, so a missing `node:sqlite` error is surfaced through the generic embedding/provider payload.
  - file: [extensions/memory-core/src/tools.ts:207](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/memory-core/src/tools.ts#L207)
