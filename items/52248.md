## Summary

Current `main` does not support the reported "agent.wait response routes to the wrong websocket connection" path. The websocket response callback is bound to the original request socket, `agent.wait` only resolves through that callback, and current webchat coverage exercises lifecycle completion over a websocket client without any alternate-connection routing.

## Close Comment

Closing from a Codex review because this report does not match current `main`. The websocket RPC response path is still tied to the original request socket: the request handler creates a per-request `respond` closure that captures that socket's `send` function and `connId` (`src/gateway/server/ws-connection/message-handler.ts:1459`), and `agent.wait` only resolves by calling that same `respond(...)` callback (`src/gateway/server-methods/agent.ts:1033`). Current coverage also exercises `agent.wait` from a real webchat websocket and verifies lifecycle completion resolves back through that websocket path (`src/gateway/server.chat.gateway-server-chat.test.ts:1135`). In addition, long-running `agent.wait` RPCs were hardened so pending waits own their timeout instead of being dropped by the gateway client watchdog (`src/gateway/client.ts:860`, `CHANGELOG.md:220`). If you can still reproduce this on a current build, please open a fresh issue with the exact current version plus paired gateway/client logs showing the stuck request on current `main`.
