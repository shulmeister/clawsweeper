## Summary

Keep #71699 open. The report is recent, detailed, and describes a hard Windows process exit during Mattermost streaming that current main does not clearly fix or supersede. Current Mattermost streaming still creates/edits preview posts through the normal draft lifecycle, the Windows scheduled-task launcher does not add Node fatal-error report flags, and repo history/search shows no targeted #71699 / 0xC0000409 / Mattermost crash fix on main. The related items in the provided context remain distinct: #64253 is an alive-but-unresponsive gateway wedge, and #56215 is a broader WebSocket leak/crash report.

## Best Possible Solution

Keep the issue open and narrow it into an actionable Windows stability fix: add supervised Windows gateway Node fatal-error report collection, audit Mattermost/ws streaming under backpressure and embedded Ollama runs, verify whether bootstrap truncation is only noisy or actually implicated, and add a Windows-focused regression/smoke path once the failing frame is known. Related #64253 and #56215 can inform investigation, but they should not supersede this hard-crash report unless a canonical maintainer issue is created for the same Windows crash signature.
