## Summary

Current main already ships the requested reliability pieces: a fast `/status` control path that bypasses agent execution, built-in stale-channel watchdog restarts for channels including WhatsApp, documented Linux service restart policy, and automatic stale gateway-lock cleanup.

## Close Comment

Closing after Codex review: current `main` already covers this request well enough. `docs/gateway/health.md:15-21` documents a fast `/status` path for WhatsApp/WebChat that replies without invoking the agent, and `src/auto-reply/reply.triggers.trigger-handling.filters-usage-summary-current-model-provider.cases.ts:57-79` verifies that the agent is not called. The built-in stale-channel watchdog is documented in `docs/gateway/configuration.md:228-254` and implemented in `src/gateway/channel-health-monitor.ts:76-156`, with WhatsApp included in the supported monitored channels listed in `docs/gateway/health.md:34-39`. For remote Linux hosts, `docs/platforms/linux.md:64-92` now documents the supported systemd service with `Restart=always`, and stale gateway locks are reclaimed automatically in `src/infra/gateway-lock.ts:299-333` instead of requiring manual lock-file cleanup. I also verified these pieces are present in the shipped `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce a current stuck/no-reply case on a recent release, please open a new bug with `openclaw status --all`, `openclaw health --verbose`, and a `openclaw gateway diagnostics export` bundle.
