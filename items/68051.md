## Summary

The requested claim-verification layer fits current plugin/ClawHub scope on `main`: OpenClaw’s vision explicitly keeps optional policy features out of core when plugins can express them, and current hook surfaces already expose the correlation and delivery controls needed to implement this as a plugin.

## Evidence

- **Vision scope favors plugins over new core optional policy:** `VISION.md` says OpenClaw keeps core lean, optional capability should usually ship as plugins, and if a feature cannot be built as a plugin the preferred direction is extending plugin APIs rather than adding one-off core behavior.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/VISION.md#L54)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Public plugin hooks already cover tool observation and outbound delivery control:** `docs/plugins/hooks.md` documents `llm_output` for provider output observation, `after_tool_call` for tool result/error observation, `before_agent_reply` for synthetic reply short-circuiting, and `message_sending` for rewrite/cancel of outbound delivery.
  - file: [docs/plugins/hooks.md:10](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/plugins/hooks.md#L10)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Hook contracts carry the IDs needed for per-run claim verification:** `PluginHookLlmOutputEvent` includes `runId` and assistant text; `PluginHookAfterToolCallEvent` includes `runId`, `toolCallId`, params, result/error; `PluginHookMessageContext` carries `runId`; `PluginHookMessageSendingResult` can rewrite content or cancel delivery.
  - file: [src/plugins/hook-types.ts:190](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/plugins/hook-types.ts#L190)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Message hook types support delivery gating:** `src/plugins/hook-message.types.ts` defines `PluginHookMessageContext.runId` and `PluginHookMessageSendingResult { content?, cancel? }`, which is the core delivery gate a warn/strict policy needs.
  - file: [src/plugins/hook-message.types.ts:4](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/plugins/hook-message.types.ts#L4)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **No built-in claim-verification feature exists on current main:** Search for `claimVerification|claim verification|success-claim|unsupported claim|tool-call ledger` across `src`, `docs`, `extensions`, and `VISION.md` returned no matches on current `main`, so this is not already implemented in core.
  - command: `rg -n "claimVerification|claim verification|success-claim|unsupported claim|tool-call ledger" src docs extensions VISION.md`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Issue discussion already points toward plugin packaging:** Issue comment `4284201786` describes a local implementation and explicitly says it would be structured as a plugin with clean configuration points if upstream receptivity existed, which aligns with current plugin-first project scope.

## Risks / Open Questions

- If maintainers later want this enabled by default across all installs, a core-owned default plugin or additional hook ergonomics may still be worth discussing.
- A plugin implementation may still reveal missing helper APIs for richer tool-intent matching, but that is an API-extension question rather than justification for a new core feature.
