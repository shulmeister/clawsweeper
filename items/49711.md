## Summary

Current `main` already loads `node-llama-cpp` through an async `import()` bridge, and the same implementation is present in the shipped `v2026.4.22` release, so the reported `require()` vs ESM incompatibility is already addressed.

## Close Comment

Closing from a Codex review because the reported ESM/CommonJS boundary is already handled in current OpenClaw. On `main`, `src/memory-host-sdk/host/node-llama.ts:29` loads `node-llama-cpp` with `import("node-llama-cpp")`, and `src/memory-host-sdk/host/embeddings.ts:31` uses that async loader for the local embedding provider instead of `require()`. I also checked the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and it contains the same dynamic-import path, so the specific incompatibility described here is already implemented in released code. If local embeddings are still unavailable on a current build, please open a fresh issue with `openclaw memory status --deep`, the exact runtime error, Node version, install method, and platform details so maintainers can chase the remaining failure mode.
