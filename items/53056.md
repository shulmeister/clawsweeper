## Summary

Keep this issue open. Current main still performs Slack mrkdwn ampersand escaping on normal outbound sends and preview final edits, tests still assert that behavior, and the two linked fix attempts were closed unmerged after discussion identified Slack contract and regression risks. The remaining maintainer work is to reproduce where the second entity pass is introduced and fix that boundary without removing required Slack control-character escaping globally.

## Best Possible Solution

Keep this open. Add focused Slack regression coverage for raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` through normal send, slash/reply delivery, draft-preview final edit, native streaming `markdown_text`, and block fallback paths. Then fix the specific layer that applies a second entity pass while preserving exactly-one Slack mrkdwn escaping for `&`, `<`, and `>`.

## Risks / Open Questions

- The visible `&#38;` may be introduced only on a specific live delivery path, readback/display normalization path, native Slack streaming path, middleware layer, or with already-escaped upstream text; current tests do not isolate that boundary.
- Removing ampersand escaping globally would conflict with Slack mrkdwn control-character rules and can turn literal entity-like text such as `&lt;@U123&gt;` into parsed Slack mentions or links.
