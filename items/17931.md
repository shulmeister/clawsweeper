## Summary

Keep #17931 open. Current main still does not implement automatic read-only per-skill bind mounts for sandboxed sessions: non-`rw` modes still copy eligible skills into the sandbox workspace, and `rw` mode still exposes workspace skills through the normal writable `/workspace` mount unless the operator manually configures overlapping read-only custom binds.

## Best Possible Solution

Keep #17931 open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that sandbox-hardening direction. A robust fix should derive eligible skill source directories, add deterministic read-only overlays after broader workspace mounts for `rw`, avoid writable copied skills where the backend can mount source directories directly, update sandbox config hashing/recreate behavior and fs-bridge mount resolution as needed, and cover the behavior in tests and sandboxing docs. Treat #22669 as a useful manual workaround, not the canonical fix.

## Risks / Open Questions

- Sandboxed agents can still mutate mirrored skill copies in `workspaceAccess: "ro"` and `"none"` because those copies live under the writable sandbox workspace.
- In `workspaceAccess: "rw"`, workspace-local skills remain writable through the normal `/workspace` mount unless an operator manually supplies overlapping read-only custom binds.
- Closing this as implemented would conflate the #22669 custom-bind ordering fix with the requested default sandbox hardening.
