## Summary

Keep open. Current main still does not implement automatic read-only per-skill mounts for sandbox containers. Non-rw Docker modes now mount the sandbox workspace read-only, and #22669 provides a manual custom-bind workaround, but current code still mirrors skills by copy for non-rw modes and exposes workspace skills through the writable /workspace mount in workspaceAccess "rw". There is no automatic per-skill overlay, config-hash input, or filesystem-bridge mount source for immutable skill directories.

## Best Possible Solution

Keep #17931 open until OpenClaw implements or explicitly rejects automatic read-only skill-directory handling in core sandboxing. A robust fix should derive eligible SkillEntry source directories deterministically, add read-only Docker overlays after workspace mounts for workspaceAccess "rw", avoid relying on writable skill copies where a backend can mount sources directly, update config hashing and recreate behavior, teach fs-bridge/remote bridge about skill mounts, and add tests/docs. Treat #22669 as a useful manual workaround, not the canonical fix.
