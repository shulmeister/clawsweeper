## Summary

Keep #38846 open. Current main does not implement the proposed exported Windows argument validator, null/control-byte checks, or PowerShell-pattern hardening, and the PR diff still has unresolved correctness and secret-exposure blockers rather than being obsolete or safely mergeable.

## Best Possible Solution

Keep the PR open for rework. A safe version should harden the Windows wrapper boundary without rejecting ordinary Windows paths, slash switches, scoped package names, or `=` flags; redact or avoid full argv values in errors; and add focused regression tests for `.cmd`/`.bat` wrapper paths, npm/pnpm fallback, scoped packages, equals flags, and null/control-byte handling.

## Risks / Open Questions

- Merging the PR as written would likely regress Windows `.cmd`/`.bat` execution because normal Windows command paths contain backslashes.
- The proposed `/` and `=` blocking would reject valid Windows switches, scoped package names, and npm flags used by current code paths.
- The proposed error messages include full argument values, which could expose tokens or credentials in logs and stack traces.
- Closing now would drop an unimplemented security-hardening thread without a canonical replacement item.
