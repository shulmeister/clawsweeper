## Summary

Keep PR #38846 open. Current main has not implemented the proposed Windows argument validator/null/control-character hardening, and the PR as written is still behaviorally unsafe because it rejects normal Windows path separators, slash switches/scoped package specs, and equals-style flags used by OpenClaw command paths. No canonical replacement issue or shipped fix was found in the provided context or local checkout.

## Best Possible Solution

Keep the PR open for rework, or first create a narrower canonical Windows command-hardening issue before closing it as superseded. A safe implementation should harden the cmd.exe wrapper boundary without rejecting ordinary paths, slash switches, scoped package names, or equals flags; keep PowerShell-specific checks scoped to actual PowerShell execution; redact argument values in errors; and add focused Windows regression tests for `.cmd`/`.bat` wrapping, npm/pnpm fallback, scoped package specs, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely break normal Windows `.cmd`/`.bat` execution because resolved Windows paths contain backslashes.
- The PR would reject legitimate Windows switches, scoped package names, and npm-style equals flags already used by OpenClaw.
- The PR's proposed validation errors include full argument values, which can expose secrets in logs or stack traces.
- Closing now would drop an unimplemented security-hardening thread without a narrower canonical replacement issue or safe successor PR.
