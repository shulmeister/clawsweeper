## Summary

Current main still uses live Kilo model discovery at startup, and the live Kilo gateway endpoint now returns hundreds of models instead of only the static `kilo/auto` fallback, so the reported one-model-only behavior is not reproducible anymore.

## Evidence

- **Kilo provider uses live discovery for normal catalog builds:** The bundled `kilocode` provider is wired so normal catalog resolution uses `buildKilocodeProviderWithDiscovery`, while the one-model `buildKilocodeProvider` path is only the explicit static fallback.
  - file: [extensions/kilocode/index.ts:30](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/kilocode/index.ts#L30)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Discovery fetches the live gateway catalog and only prepends the static fallback if needed:** `discoverKilocodeModels()` fetches `https://api.kilo.ai/api/gateway/models`, parses all returned entries, and only prepends the static `kilo/auto` model when the live response does not already contain it or discovery fails.
  - file: [extensions/kilocode/provider-models.ts:124](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/extensions/kilocode/provider-models.ts#L124)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Startup implicit-provider resolution calls the live provider catalog with real env/auth:** Current startup model discovery uses `runProviderCatalogWithTimeout(...)` inside `resolveImplicitProviders(...)`, passing through the actual config, env, and auth resolvers, so Kilo startup discovery goes through the live catalog path rather than the static one-model path.
  - file: [src/agents/models-config.providers.implicit.ts:246](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/models-config.providers.implicit.ts#L246)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Live upstream endpoint no longer matches the reported one-model result:** A direct request during this review to `https://api.kilo.ai/api/gateway/models` returned HTTP 200 and a `data` array of 339 models; the first ids included `kilo-auto/frontier`, `kilo-auto/balanced`, `kilo-auto/free`, and non-Kilo vendor models, not just a single `kilo/auto` entry.
  - command: `curl -sS https://api.kilo.ai/api/gateway/models | jq '.data | length, (.[0:5] | map(.id))'`
- **Docs and code both describe startup as dynamic Kilo discovery plus static fallback:** The provider docs say startup queries the Kilo gateway models endpoint and merges discovered models ahead of the static fallback catalog, which matches the current implementation.
  - file: [docs/providers/kilocode.md:60](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/providers/kilocode.md#L60)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)

## Risks / Open Questions

- I could not run a full end-to-end OpenClaw CLI reproduction in this environment because `pnpm` is unavailable here, so the verification is based on current code paths plus the live Kilo gateway response.
- This path depends on an upstream Kilo gateway catalog; if Kilo changes that endpoint behavior again, the symptom could reappear without a local OpenClaw code change.
