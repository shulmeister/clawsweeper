## Summary

Keep open. The issue is recent, non-maintainer-authored, and describes a plausible runaway tool loop on Feishu/Qwen with token waste. Current main has relevant loop-detection machinery, but source, docs, and tests show the no-progress tool-loop guard is opt-in and explicitly disabled by default, so I cannot say current main already fixes the reported default behavior.

## Best Possible Solution

Keep #71369 open and ask for a sanitized session transcript or diagnostics export showing the repeated tool calls, tool outputs, config, and Feishu/Qwen route. If reproduced, add a regression around empty/no-progress tool results in the embedded runner/before-tool-call path and decide whether this class of no-progress loop should be guarded by default, enabled for Feishu/Qwen routes, or documented as a required tools.loopDetection setting for high-risk deployments.
