## Summary

Keep open. Current `main` still does not implement Slack `groupPolicy: "private"`: the shared `GroupPolicy` contract and config schema reject it, Slack channel/slash routing only evaluates `open | disabled | allowlist`, and the Slack docs/tests still document and cover only those three policies. The prior issue discussion already reached the same conclusion, and the current checkout at `04e08cea62983b2c766d15f61f15b3ffc90f9baa` has not changed that surface.

## Best Possible Solution

Implement this in the bundled Slack plugin and aligned config contract: add a supported `private` groupPolicy mode, preserve or fetch Slack `is_private` for inbound channel/slash routing, allow private channels while still permitting explicit public-channel allowlist entries, keep `disabled` and explicit `allow: false` behavior intact, update security-policy warnings/audit severity handling, and add Slack schema, runtime gating, slash routing, docs, and tests covering private channels, public allowlist fallback, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` to the shared `GroupPolicy` union globally could expose the mode to channels that cannot classify public/private rooms; the implementation should either make the semantics generic and supported everywhere or scope the extra value carefully to Slack.
- Slack already records `isPrivate` in channel list resolution, but message/slash routing currently does not carry that metadata into policy evaluation, so the fix needs a runtime path for reliable private/public classification and a documented Slack Connect caveat.
