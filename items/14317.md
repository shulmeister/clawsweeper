## Summary

Keep #14317 open. Current `main` still does not implement Slack `groupPolicy: "private"`: the shared config type/schema reject that value, the Slack runtime policy only evaluates `open`, `allowlist`, and `disabled` without private-channel metadata, and Slack docs/tests still describe only the existing three modes. The prior issue discussion keeping this open remains consistent with the current checkout at `631552c554095b0c520376db9131dac0f5427f94`.

## Best Possible Solution

Implement this in the bundled Slack plugin and aligned config/security contract: add a supported Slack `groupPolicy: "private"` mode, carry or fetch Slack private-channel metadata for message and slash routing, allow invited private channels while preserving explicit public-channel allowlist entries and explicit denies, update security warnings/audit behavior so `private` is not treated like `open`, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats. Scope the config/API change so other channels are not accidentally assigned Slack-only semantics.

## Risks / Open Questions

- Extending the shared `GroupPolicy` union directly would affect non-Slack channels unless the new semantics are scoped carefully or made generic with explicit contracts.
- Slack private channels and Slack Connect/shared channels can include external participants, so the implementation and docs should avoid treating `is_private` as equivalent to internal-only trust.
