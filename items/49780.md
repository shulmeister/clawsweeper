## Summary

Current `main` does not show a cron serialization bug for nested payload objects. The cron payload contract is flat and validated as such; nested payload keys are rejected by schema, and store persistence uses ordinary `JSON.stringify` on the full job/state objects rather than a lossy custom serializer.

## Evidence

## Close Comment

Closing from a Codex review against current `main` (`9ece33c50506ba031df382f69bde65dfda19bea5`). I could not confirm a cron serialization bug here. The current cron payload contract is flat, not nested: `agentTurn` payloads only allow the documented top-level fields and reject extra nested keys via schema validation in `src/gateway/protocol/schema/cron.ts:4` and `src/gateway/protocol/schema/cron.ts:130`, and `cron.add`/`cron.update` enforce that validation in `src/gateway/server-methods/cron.ts:189`. The persistence path in `src/cron/store.ts:264` uses ordinary `JSON.stringify` on the whole job/state objects, so there is no custom serializer here that would selectively drop nested properties. The only place values are currently removed is input coercion in `src/cron/normalize.ts:147`, where invalid typed fields like a non-string `payload.model` are intentionally deleted. If you still have a repro on a current build, please open a new issue with the exact `cron.add` or `cron.update` payload and the returned validation/error response.
