## Summary

Keep #11703 open. Current main still debounces WhatsApp inbound messages by receive-time windows keyed on account/chat/sender, not by Baileys `messageTimestamp`. The timestamp is preserved on the inbound message, but there is no timestamp bucket, coalescing window, config option, docs entry, or regression test for delayed same-timestamp WhatsApp deliveries. The linked #19716 context does not supersede this; it was a closed, unmerged PR about surfacing timestamps to agent context, not batching behavior.

## Best Possible Solution

Keep the issue open. Implement the fix in the WhatsApp-owned inbound path, or add an additive shared inbound debouncer capability if maintainers want it reusable: group messages by account/chat/sender plus same or near `messageTimestamp`, tolerate Baileys delivery delays beyond the receive-time debounce, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any new config option if exposed.
