## Summary

Keep #11703 open. Current main still uses WhatsApp's shared inbound debouncer keyed by account/conversation/sender and a receive-time timer. `messageTimestamp` is preserved as metadata/timestamp input, but it is not used as a batching key, bucket, late-arrival coalescing window, or exposed config option. The related #19716 was closed unmerged and concerned timestamp visibility in agent context, not WhatsApp batching.

## Best Possible Solution

Keep the issue open. Implement the fix in the WhatsApp-owned inbound path, or add an additive shared debouncer capability if maintainers want reuse: group by account/conversation/sender plus same-or-near WhatsApp `messageTimestamp`, use a bounded late-arrival/coalescing policy that tolerates Baileys delivery gaps without excessive latency, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any exposed config.
