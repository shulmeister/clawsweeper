## Summary

Keep #11703 open. Current main still implements WhatsApp batching as a receive-time debounce for the same account/conversation/sender, while `messageTimestamp` is only carried into the inbound message timestamp field. There is no timestamp-coalescing key/window or config surface matching the requested behavior, and the related #19716 context is about exposing timestamps to agent prompts, not batching WhatsApp messages.

## Evidence

- **WhatsApp debounce key is account/conversation/sender only:** `attachWebInboxToSocket` creates a shared inbound debouncer with `debounceMs: options.debounceMs ?? 0`; its `buildKey` returns `${accountId}:${conversationKey}:${senderKey}` and does not include `messageTimestamp` or any timestamp bucket.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:176](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/whatsapp/src/inbound/monitor.ts#L176)

  - sha: [1dfa52d0711d](https://github.com/openclaw/openclaw/commit/1dfa52d0711dfc86e1c04538fe46e3d43961766f)
- **WhatsApp server timestamp is preserved but not used for coalescing:** The monitor parses `msg.messageTimestamp` into `messageTimestampMs`, then assigns it to `WebInboundMessage.timestamp`; the debounce key and flush scheduling are already defined earlier and do not reference this value.

- **Shared inbound debouncer is receive/enqueue-time based:** `createInboundDebouncer` schedules `setTimeout(..., buffer.debounceMs)` when an item is enqueued and only extends the current buffer when another same-key item arrives before that timeout; it has no event-timestamp-aware waiting or late coalescing path.
