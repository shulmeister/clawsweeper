## Summary

Keep #11703 open. Current main still batches WhatsApp inbound messages only with the shared receive/enqueue-time debounce keyed by account, conversation, and sender. `messageTimestamp` is parsed and preserved on the inbound message, but it is not part of the debounce key, flush timing, or config surface. The related #19716 context is about surfacing timestamps to the agent prompt and does not supersede WhatsApp timestamp-based batching.

## Best Possible Solution

Keep the issue open and implement a WhatsApp-owned timestamp-aware coalescing path, or an additive shared debouncer capability if maintainers want it reusable. The fix should include a regression test with same chat/sender messages sharing `messageTimestamp` but arriving after the normal `debounceMs` window, plus config/docs if a new option is exposed.
