## Summary

Keep #11703 open. Current main still debounces WhatsApp inbound messages by receive/enqueue timing, keyed by account, conversation, and sender. `messageTimestamp` is parsed and preserved on the inbound message, but it is not used to hold or merge late Baileys deliveries that share the same WhatsApp server timestamp. The related #19716 context is about surfacing timestamps to the agent, not timestamp-based batching, and it does not supersede this issue.

## Best Possible Solution

Keep the issue open. Implement a WhatsApp-owned timestamp-aware coalescing path, or an additive shared inbound debouncer capability if maintainers want it reusable. The fix should group same chat/sender messages with the same or near `messageTimestamp` even when Baileys delivers them after the normal receive-time debounce window, and should add regression coverage plus config/docs if a new option is exposed.
