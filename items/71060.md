## Summary

Keep #71060 open. Current main has hardened the macOS `gateway restart` path, including kickstart-first restart, not-loaded bootstrap recovery, and detached handoff when invoked from the managed LaunchAgent. But the `gateway stop` path still matches the core reported failure mode: it waits only 10 x 100ms for launchd to report stopped, then falls back to `launchctl bootout` with warnings that explicitly say the service is left unloaded. Existing tests also assert that degraded bootout behavior.

## Best Possible Solution

Keep the issue open and narrow the maintainer work to macOS LaunchAgent stop semantics. The likely fix belongs in `src/daemon/launchd.ts`: replace the hard-coded 1s stop confirmation window with a drain-aware timeout, avoid booting out a loaded LaunchAgent merely because graceful shutdown exceeds that window, and if a bootout fallback remains necessary, re-bootstrap or return a clearly recoverable state. Add regression coverage for slow-but-eventually-stopped services and update Gateway lifecycle docs if stop semantics change.
