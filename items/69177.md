## Summary

Current main already solves the user-visible bug without this PR's exact internal forwarding change: `session_status` passes the resolved session entry into the shared status renderer, and the renderer now reads `sessionEntry.contextTokens` directly before falling back to model defaults. The linked bug #69151 is already closed, and the latest release tag contains the same persisted-context-token behavior.

## Best Possible Solution

Close this PR as redundant. Keep the current main implementation that reads persisted `sessionEntry.contextTokens` in the shared status renderer; if maintainers want extra guard coverage later, add a behavior-level `session_status` status-output test rather than asserting the internal `buildStatusText` parameter shape.
