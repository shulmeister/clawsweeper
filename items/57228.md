## Summary

Current main already avoids corpus-wide embedding requests during memory indexing. The memory-core indexer processes files individually, splits oversize chunks down to the provider token budget before embedding, and then batches embedding calls under an 8000-token cap. The same implementation is present in release v2026.4.23.

## Evidence

- **Per-file indexing, not whole-corpus embedding:** Memory sync builds one task per discovered file and calls `indexFile(entry, { source: ... })` for each memory/session file, so indexing is scoped to individual files rather than one giant corpus request.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:718](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/memory-core/src/memory/manager-sync-ops.ts#L718)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Oversize chunks are split before embedding:** `indexFile` chunks markdown first, then runs `enforceEmbeddingMaxInputTokens(...)` before any embedding call. That helper splits text chunks to the provider/hard max token budget and rewrites hashes/embedding inputs for the smaller pieces.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:623](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/memory-core/src/memory/manager-embedding-ops.ts#L623)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Token-limit enforcement helper:** `enforceEmbeddingMaxInputTokens` compares each chunk against the resolved provider max input tokens and uses `splitTextToUtf8ByteLimit(...)` when a chunk is too large.
  - file: [src/memory-host-sdk/host/embedding-chunk-limits.ts:7](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/memory-host-sdk/host/embedding-chunk-limits.ts#L7)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Embedding requests are batched under a size cap:** Non-batch fallback embedding groups missing chunks with `buildMemoryEmbeddingBatches(missingChunks, EMBEDDING_BATCH_MAX_TOKENS)`, and the batch builder starts a new request before accumulated estimated bytes exceed the configured limit.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:101](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/memory-core/src/memory/manager-embedding-ops.ts#L101)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Regression coverage for long input splitting:** Tests cover chunk splitting for overly long lines and token-budget-sensitive text, which is the guardrail needed to avoid oversized embedding inputs during indexing.
  - file: [src/memory-host-sdk/host/internal.test.ts:362](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/memory-host-sdk/host/internal.test.ts#L362)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)

## Risks / Open Questions

- I did not run a live Ollama reproduction in this read-only review, so this decision relies on current code paths and shipped release contents rather than runtime verification against `mxbai-embed-large`.
- The Ollama adapter itself does not advertise a model-specific `maxInputTokens`; the fix relies on the generic pre-splitting and per-request batching paths now present in memory-core.
