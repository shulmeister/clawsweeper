## Summary

Keep open. Current main still skips browser device identity when `crypto.subtle` is unavailable and still fingerprints the browser public key with WebCrypto, so the main LAN HTTP/reverse-proxy request is not implemented. The secondary bypass and operator scope-normalization pieces are fixed, but they do not resolve the Secure Context dependency.

## Best Possible Solution

Keep this issue open unless a maintainer explicitly consolidates it into #32473. The concrete path is to decide whether browser device identity should be supported over non-secure HTTP, then remove the `crypto.subtle` gate, replace the WebCrypto-only fingerprint with an approved pure-JS hash path or direct dependency, and update gateway policy tests plus Control UI/security docs. A refreshed version of the closed, unmerged PR #53276 could be used as a starting point while preserving the already-fixed bypass and scope-normalization behavior.
