## Summary

Keep open. Current main fixes the original direct HTTP 500/status path and the JSON `server_error` Codex path, but the latest concrete reproduction in the issue is a raw OpenAI Codex message variant (`The server had an error processing your request...`) that is still not classified as failover-worthy. The generic fallback loop now continues past thrown unrecognized errors, but assistant-side `stopReason: "error"` only enters model fallback when the error is classified, timed out, or otherwise marked failover-worthy, so the latest reported path is not safely implemented yet.

## Best Possible Solution

Add a focused regression for the April 20 raw OpenAI Codex message (`The server had an error processing your request...`) with `stopReason: "error"` and configured fallbacks, then either add that wording to the server-error classifier or normalize this assistant-side unclassified provider failure into the existing failover path without replaying side-effectful turns.
