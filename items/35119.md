## Summary

Keep #35119 open. Current main fixes the original structured HTTP 500/status path, but the latest concrete reproduction is a raw OpenAI Codex message variant (`The server had an error processing your request...`) that still is not classified as failover-worthy. Assistant-side fallback still requires a failover signal or timeout, so that remaining path can still stop instead of advancing the configured fallback chain.

## Best Possible Solution

Add a focused regression for the April 20 raw OpenAI Codex message with `stopReason: "error"` and configured model fallbacks, then either add that wording to the server-error classifier or normalize assistant-side provider server failures into the existing failover path without replaying turns that had side effects.
