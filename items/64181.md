## Summary

Keep this PR open. Current main still has the reported slug handling behavior: `generateSlugViaLLM` accepts any first payload text, slugifies it directly, strips edge dashes before truncation, and returns that value to the session-memory filename path. The already-merged #66455 timeout fix is present but does not address error-payload rejection or post-truncation dash stripping, and the open #38161 is broader/complementary rather than a shipped replacement for this focused bug fix.

## Best Possible Solution

Review and land this focused PR, or apply the same behavior on main: extract/test a pure slugification helper, reject non-slug-like LLM payload text before filename construction, and strip leading/trailing dashes after truncation. Do not close it as superseded by #66455 or #38161 unless another merged change first ships these exact safeguards.

There is no max-word/max-char/newline sanity check before slugifying provider error text, and the dash strip still happens before truncation.
