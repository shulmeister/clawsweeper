## Summary

Keep #68497 open. Current main partially improves the original symptom by injecting `window.openclawSendUserAction`, but browser-originated A2UI actions still do not route back to the Gateway: the helper is native-bridge-only, the canvas WebSocket is live-reload-only, and there is no `/__openclaw__/canvas/action` POST path. Related #66983 is broader web canvas-node work, but this remains a narrower actionable core canvas/gateway transport request.

## Best Possible Solution

Keep the issue open and implement the browser fallback in core canvas/gateway code. A good fix would define a typed A2UI user-action payload, route it over either an authenticated `/__openclaw__/canvas/action` endpoint or an explicit `/__openclaw__/ws` message, dispatch `openclaw:a2ui-action-status` back to the page, update the A2UI bundle/host to use `openclawSendUserAction` consistently, and add gateway/canvas tests for normal browser behavior plus auth failure cases. Coordinate with #66983 only if maintainers choose the broader web-canvas-node model instead of a narrow action bridge.

## Risks / Open Questions

- Browser-originated canvas actions would let same-origin canvas JavaScript ask the Gateway to handle user actions, so the implementation needs explicit auth/capability, payload validation, and status semantics.
- The remaining work overlaps #66983, but #66983 is broader web canvas-node registration; maintainers should decide whether to solve this as a small authenticated action endpoint/WebSocket message or fold it into the larger web-node design.
