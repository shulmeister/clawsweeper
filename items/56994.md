## Summary

Keep open. Current `main` appears to have resolved the `saveJsonFile` and restart-sentinel parts of the report, but the issue is not fully implemented: shell-profile completion install still writes in place, and the exec-approvals writer still uses a narrower temp+rename path instead of the stronger crash-safe atomic helper pattern requested.

## Evidence

- **`saveJsonFile` is now atomic on main:** `saveJsonFile` now writes to a temp file, fsyncs it, renames it into place, chmods, and best-effort fsyncs the parent directory instead of writing the destination directly.
  - file: [src/infra/json-file.ts:109](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/json-file.ts#L109)
  - command: `nl -ba src/infra/json-file.ts | sed -n '100,130p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Restart sentinel now uses `writeJsonAtomic`:** `writeRestartSentinel` calls `writeJsonAtomic(..., { trailingNewline: true, ensureDirMode: 0o700 })`, matching the issue's async-path fix request.
  - file: [src/infra/restart-sentinel.ts:80](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/restart-sentinel.ts#L80)

- **`saveJsonFile` has tests asserting temp-file replacement behavior:** The current tests explicitly assert that `saveJsonFile` writes through a sibling temp file before replacing the destination.
  - file: [src/infra/json-file.test.ts:117](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/json-file.test.ts#L117)
  - command: `nl -ba src/infra/json-file.test.ts | sed -n '70,130p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)

It does not use the shared atomic writer machinery with fsync/dir-sync semantics that the issue asked to standardize on.
  - file: [src/infra/exec-approvals.ts:487](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/exec-approvals.ts#L487)

- **Shell profile writes are still in-place:** `installCompletion` still creates and updates the user's shell profile with direct `fs.writeFile(...)` calls, so the shell-profile part of the report remains unresolved on `main`.
  - file: [src/cli/completion-runtime.ts:233](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/completion-runtime.ts#L233)

Current `main` confirms the restart-sentinel fix and `saveJsonFile` fix, but not the shell-profile change.

## Risks / Open Questions

- Interrupting `openclaw completion --install` can still truncate or corrupt `~/.zshrc`, `~/.bashrc`, or `~/.bash_profile` because the code still overwrites those files directly.
- `saveExecApprovals` is safer than the original report described, but it still does not provide the same fsync and parent-directory sync guarantees as the shared atomic JSON/text writers used elsewhere.
