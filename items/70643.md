## Summary

Keep #70643 open. The linked fix PR #70655 directly proposed serializing config writes with `withFileLock`, but that PR is closed unmerged, and current `main` still does not wrap the config read/merge/write path in an advisory lock. Current code has temp-file-and-rename writes plus last-known-good recovery, but that does not linearize concurrent config mutations or prove the reported race is fixed.

## Best Possible Solution

Keep the issue open and implement a config-wide advisory lock around the entire read/merge/validate/write transaction, not just the final rename. The fix should cover `writeConfigFile`, `replaceConfigFile`, `mutateConfigFile`, gateway `config.patch`/`config.set`, doctor/configure/model writers, and include a regression test where two concurrent config-touching writers preserve both changes and leave valid JSON. A separate recovery improvement for concatenated top-level JSON can be added, but it would not replace write serialization.
