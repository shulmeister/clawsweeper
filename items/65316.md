## Summary

Keep open. Current main still has the reported mismatch for sandboxed sessions with default workspace access: sandbox config defaults to `workspaceAccess: "none"`, the Docker workspace mount is read-only unless access is `"rw"`, but tool construction only suppresses write/edit/apply_patch for `"ro"`, leaving the `"none"` case exposed even though the filesystem bridge rejects writes.

## Best Possible Solution

Keep this issue open and fix the sandbox tool-surface contract. The likely code path is `src/agents/pi-tools.ts`: derive write/edit/apply_patch availability from an effective writable sandbox target, with `workspaceAccess: "none"` and `"ro"` suppressing write-capable file tools unless a supported writable mount is intentionally available and the bridge can actually write there. Add regression tests for default `mode: "all"`/`workspaceAccess: "none"`, `"ro"`, `"rw"`, and any intended writable bind-mount case; update sandbox diagnostics if the effective writable target is absent.
