## Summary

Keep open. Current main has some correct agent propagation for ordinary message-tool runs, but the outbound mirror code still passes `mirror.agentId` from the current request/resolved agent instead of deriving it from the outbound route/session key. Gateway coverage even preserves explicit request-agent precedence over a provided session key, so the reported multi-account wrong-agent transcript class is not proven fixed.

## Best Possible Solution

Keep #66953 open. Add a focused regression around Feishu or the generic outbound route with `session.dmScope: "per-account-channel-peer"`, a writer account/agent binding, and a delivery mirror append; then either derive the mirror transcript agent/store from `outboundRoute.sessionKey` or ensure every caller propagates the writer agent before route construction. Update the gateway/message-action precedence tests to encode the chosen contract.

## Evidence

- **Issue discussion:** Issue #66953 reports a concrete Feishu multi-account, multi-agent reproduction on OpenClaw 2026.4.14, and a later commenter added that they also hit the issue. The report is recent and has no protected labels, but the discussion describes a real routing/transcript integrity bug rather than a stale or incoherent request.
- **Message action mirror still uses request agent:** `runMessageAction` builds the delivery mirror with `sessionKey: outboundRoute.sessionKey` but `agentId` from the current resolved request agent.
