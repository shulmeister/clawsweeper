## Summary

Keep open. Current `main` still appears vulnerable to WebChat runs that never receive a terminal chat event: the browser UI enters streaming mode immediately, only clears that state on `final`/`aborted`/`error`, and defers history reloads while the run is marked active. OpenClaw has shipped narrower related fixes, but I did not find strong evidence that this general intermittent WebChat stall is fully resolved on `main`.

## Evidence

- **WebChat enters streaming state as soon as a send starts:** `sendChatMessage` assigns a fresh `chatRunId`, initializes `chatStream` to an empty string, and records `chatStreamStartedAt` before waiting for the gateway call to complete.
  - file: [ui/src/ui/controllers/chat.ts:336](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/controllers/chat.ts#L336)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '330,339p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Only terminal chat events clear the active-run state:** `handleChatEvent` clears `chatRunId` / `chatStream` / `chatStreamStartedAt` only in the `final`, `aborted`, and `error` branches. There is no browser-side timeout or watchdog branch here.
  - file: [ui/src/ui/controllers/chat.ts:451](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/controllers/chat.ts#L451)

If the terminal event is lost, the UI stays stuck and history reload does not replace the streaming state.
