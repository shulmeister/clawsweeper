## Summary

Keep this issue open. The report has a concrete WhatsApp reproduction and follow-up analysis, and current main does not appear to handle the exact leaked `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant-text format. WhatsApp now suppresses text-only tool lifecycle payloads, but final assistant text is still delivered after normalization, and the shared visible-text sanitizer only covers XML-style tool-call blocks and `[Tool Call: ...]` downgraded blocks, not the all-caps bracket syntax from this issue.

## Best Possible Solution

Keep the issue open and fix it in the shared agent/runtime delivery path, not only in WhatsApp rendering. A good fix would add a regression test for the exact `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant-text shape, then either convert that provider-emitted pseudo call into a structured tool call before final delivery or strip/suppress it from user-visible replies when it is only leaked scaffolding. Also verify the WhatsApp auto-reply path with a channel-level test so text-only tool lifecycle suppression and final answer delivery still behave correctly.
