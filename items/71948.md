## Summary

Current main still has the reported Discord inbound worker timeout behavior and still lacks the requested diagnostic details in the user-facing fallback reply, timeout log, run/session tracking, and queue/status surface. The prior ClawSweeper comment remains accurate for current SHA 9626ef274ae2346b3d519f99c31856be98aab570; the focused Discord worker, process, queue, test, and docs files have no diff from that earlier review SHA.

## Best Possible Solution

Keep this open as Discord plugin runtime diagnostics and hardening work. The likely fix belongs in `extensions/discord/src/monitor/inbound-worker.ts` and `extensions/discord/src/monitor/message-handler.process.ts`: capture safe account/session/timeout-source details, wire `onAgentRunStart` so the timeout path can include a run id when available, track best-known progress from existing model/tool/plan/compaction callbacks, enrich the timeout log and fallback reply, and add focused queue tests. Add a small generic queue-depth/status seam only if maintainers decide queue depth is required beyond the Discord plugin.
