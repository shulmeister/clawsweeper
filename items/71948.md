## Summary

Keep open. Current main still has the reported Discord inbound worker timeout surface: the timeout is configurable and tested, but the timeout log only carries channel/message context and the Discord fallback reply remains exactly `Discord inbound worker timed out.`. The prior ClawSweeper comment and related closed reports point to narrower listener, websocket, and attachment timeout fixes; none cover this newer diagnostics/unblockability request against v2026.4.24.

## Best Possible Solution

Keep this open as Discord plugin diagnostics/runtime hardening. The likely fix belongs around `extensions/discord/src/monitor/inbound-worker.ts`, `extensions/discord/src/monitor/message-handler.process.ts`, and the queue/status helpers: capture safe account/session/queue/timeout-source details, wire `onAgentRunStart` so the timeout path can report a run id when available, track best-known stage/progress, enrich the log and fallback reply, and add focused queue tests for the enriched timeout output. Add a small production queue-depth/status API only if maintainers want queue depth in the timeout report.
