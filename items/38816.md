## Summary

Keep #38816 open. Current main still preserves an explicit providerless `agents.defaults.imageModel.primary` and then resolves bare image-model candidates with `DEFAULT_PROVIDER`, so `moondream` can still become `anthropic/moondream`. The recent configured-provider/image-runtime work covers provider-prefixed/configured rows, but not the reported providerless `imageModel.primary`/fallback case.

## Best Possible Solution

Keep the issue open until OpenClaw either infers a unique configured provider for bare `agents.defaults.imageModel.primary` and fallback entries using the same configured-model evidence as default chat model resolution, or explicitly rejects providerless image model refs with a helpful validation/runtime error and documentation. Add a regression test for a bare Ollama/LiteLLM-style image model such as `moondream` or `qwen2.5vl:7b` configured under `models.providers`.

## Evidence

- **Explicit imageModel bypasses auto provider selection:** `resolveImageModelConfigForTool` returns `agents.defaults.imageModel` as soon as it has a primary/fallback, before the provider-aware auto-selection path runs. A bare `primary: "moondream"` therefore remains bare.
  - file: [src/agents/tools/image-tool.ts:119](https://github.com/openclaw/openclaw/blob/e672b61417af5c45b0431df6d9109a1f4b618ef5/src/agents/tools/image-tool.ts#L119)
  - sha: [e672b61417af](https://github.com/openclaw/openclaw/commit/e672b61417af5c45b0431df6d9109a1f4b618ef5)
- **Image fallback default provider still comes from DEFAULT_PROVIDER for bare imageModel primary:** `resolveImageFallbackDefaultProvider` parses the configured image primary with `defaultProvider: DEFAULT_PROVIDER`; for a non-aliased bare id this resolves to the default provider rather than inferring Ollama/LiteLLM from configured provider model rows.
  - file: [src/agents/model-fallback.ts:441](https://github.com/openclaw/openclaw/blob/e672b61417af5c45b0431df6d9109a1f4b618ef5/src/agents/model-fallback.ts#L441)
