## Summary

Keep open. Current `main` still ships `openclaw backup` only as `create`/`verify`, while the reset/uninstall docs still describe backups as a way to get a restorable snapshot. The linked `openclaw-snapshot` skill is a useful workaround, but it does not implement the requested core CLI command on `main` or make this PR clearly out of scope for core.

## Risks / Open Questions

- This PR touches disaster-recovery behavior and was reviewed as `size: XL`; earlier review discussion raised substantive safety and integrity concerns around forced overwrite rollback, archive validation, restore-target trust, and edge cases like empty directories, so maintainer review is still needed before merge.
- Closing as `implemented_on_main` would be incorrect because current `main` and current docs still omit `openclaw backup restore`.
- Closing as `clawhub` would be premature: the linked `openclaw-snapshot` skill is only an external workaround, while the existing core backup/reset/uninstall UX still frames backup as part of the built-in local recovery flow.
