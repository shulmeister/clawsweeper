## Summary

Current `main` already covers the requested behavior through existing generic routing/config surfaces: per-Telegram-group agent selection via top-level `bindings[].match.peer`, and per-Telegram-group model selection via `channels.modelByChannel.telegram["<group-id>"]`. The Telegram group schema still does not accept `agentId`/`model`, but the underlying use case is already implemented and shipped.

## Evidence

- **Telegram topic schema has `agentId`, group schema does not:** Current config types and Zod schema still allow `agentId` on topics while omitting it from `TelegramGroupConfig`/`TelegramGroupSchema`, which matches the reporter's schema observation rather than contradicting it.

- **Supported per-group agent routing uses top-level peer bindings:** Current docs say routing prefers exact peer matches first, and show a Telegram group example using `bindings` with `peer.kind: "group"` and the Telegram chat id. Multi-agent docs also state that routing a single DM/group to a different agent should use `match.peer`, and that peer matches win over channel-wide rules.

## Risks / Open Questions

- The requested Telegram-specific schema shape (`channels.telegram.groups.<id>.agentId` / `model`) is still unsupported; the supported solution is the generic `bindings` + `channels.modelByChannel` path, which is less discoverable if a user only inspects the Telegram group schema.
