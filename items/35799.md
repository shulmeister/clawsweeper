## Summary

Keep #35799 open. Current main has useful partial mitigation: heartbeat now uses an internal stable per-agent phase and cron expressions support deterministic top-of-hour staggering. The requested public scheduler surface is still missing: no prime cadence mode, no heartbeat phase-jitter controls or status exposure, and cron `every` remains a fixed interval that rejects/prunes stagger settings.

## Best Possible Solution

Keep the issue open and narrow it into a deliberate scheduler/API change. Preserve the shipped internal heartbeat stable phase, then add backward-compatible public controls for heartbeat phase/jitter and optional prime cadence if maintainers want that product behavior. Extend cron `every` types, normalizer, CLI, UI/status/docs, and tests with explicit jitter/stagger semantics, and expose/log resolved schedule details for operators.
