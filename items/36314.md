## Summary

Keep #36314 open. Current main has several useful robustness pieces, but they are partial and do not implement the issue's central Gateway-level watchdog/error-recovery/health/degraded-startup surface. The existing mechanisms are mostly channel/account health monitoring, supervisor defaults, fatal logging/exit handling, diagnostics bundles, and last-known-good config recovery.

## Best Possible Solution

Keep the umbrella issue open or split it into focused maintainer-owned tasks: define a first-class gateway.watchdog config/schema/docs surface, specify process restart-loop detection with auto-stop/notification above platform supervisors, decide whether configurable global error recovery strategies are wanted, enrich Gateway health/RPC/probe output with safe process and error counters, and design degraded startup behavior that composes with last-known-good config recovery and current channel health monitoring.

## Risks / Open Questions

- The issue bundles multiple product decisions; maintainers may want to split it into Gateway watchdog, configurable global error recovery, health metrics, and degraded-startup tasks.
- Any implementation should reuse the existing channel health monitor, supervisor service defaults, fatal-error hooks, stability bundles, config recovery, and diagnostics rather than creating a parallel reliability stack.
- Richer health counters need a privacy-safe and low-cardinality contract before exposing process/error metrics through public or authenticated health surfaces.
