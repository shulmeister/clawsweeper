## Summary

Keep open. Current `main` still does not add a hard fail-open timeout around `memory_search`; it only converts thrown errors into an unavailable payload. A hung manager init/sync/search can still block the tool result path and wedge the turn.

## Evidence

- **Tool handler still awaits manager + search directly:** `createMemorySearchTool` awaits `getMemoryManagerContext(...)` and then `memory.manager.search(...)` directly. The only fail-open path is the surrounding `catch`, which handles rejections but not a promise that never resolves.
  - file: [extensions/memory-core/src/tools.ts:207](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/memory-core/src/tools.ts#L207)
  - command: `nl -ba extensions/memory-core/src/tools.ts | sed -n '180,330p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Builtin search path still blocks on synchronous bootstrap/init:** Builtin `MemoryIndexManager.search()` still awaits `this.sync({ reason: "search", force: true })` on first search and then awaits `this.ensureProviderInitialized()`. `sync()` itself reuses the in-flight promise and has no timeout guard, so a stalled sync/init can block the search call indefinitely.
  - file: [extensions/memory-core/src/memory/manager.ts:301](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/memory-core/src/memory/manager.ts#L301)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '289,360p'; nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '561,580p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **QMD path also still awaits sync inline:** QMD search awaits `maybeSyncDirtySearchState()`, and that helper awaits `this.sync({ reason: "search" })` directly. There is a timeout for pending updates and some QMD subprocess calls, but not for the overall fail-open tool-result contract requested in this issue.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:1071](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/memory-core/src/memory/qmd-manager.ts#L1071)

- **Watcher setup still lacks an error handler:** The builtin watcher for workspace memory plus `extraPaths` wires `add/change/unlink` handlers only. There is no watcher `error` handler here, so the reported extra-path watcher instability (`EINTR`) remains plausible from the current code.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:406](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/memory-core/src/memory/manager-sync-ops.ts#L406)
