## Summary

Keep open. Current main still does not expose `--session-key` on `openclaw agent`, while the resolver, embedded agent command type, and Gateway protocol/handler already support explicit session keys. The PR addresses a real core CLI parity gap with downstream demand, so no conservative close reason applies.

## Best Possible Solution

Keep the PR open. The best path is a scoped core CLI change that adds `--session-key` to `openclaw agent`, threads it through `AgentCliOpts`, the selector guard, `resolveSessionKeyForRequest()`, the Gateway request, and embedded fallback, then adds docs/help and focused CLI tests. If maintainers do not want raw session keys exposed, that should be an explicit product decision rather than an automated stale close.
