## Summary

Keep open. Current main still does not expose `--session-key` on `openclaw agent`, while lower-level agent and Gateway paths already accept `sessionKey`; the PR addresses a real CLI surface gap with concrete downstream demand in the discussion.

## Best Possible Solution

Keep the PR open. The best path is a scoped core CLI change that registers `--session-key` on `openclaw agent`, adds it to `AgentCliOpts`, forwards it into `resolveSessionKeyForRequest()` and embedded fallback, aligns Gateway/local validation, and updates docs/help plus focused tests. If maintainers do not want raw session keys exposed, that should be an explicit product decision rather than automated cleanup.
