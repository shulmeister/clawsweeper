## Summary

Keep open. Current main still has the reported failure path: explicit `gateway.bind="loopback"` calls `resolveGatewayBindHost("loopback")`, falls back to `0.0.0.0` when the loopback preflight fails, and `resolveGatewayRuntimeConfig` then throws the exact `gateway bind=loopback resolved to non-loopback host ...` error. The linked targeted PR #65635 was closed unmerged, and I did not find equivalent landed code or changelog evidence on current main.

## Best Possible Solution

Review or recreate the targeted fix from #65635: make explicit `gateway.bind="loopback"` stay pinned to `127.0.0.1` instead of downgrading to `0.0.0.0` after a failed preflight, while preserving container `auto` behavior and non-loopback auth/origin guards. Add focused coverage for the failed loopback-preflight path in `src/gateway/net.test.ts` or `src/gateway/server-runtime-config.test.ts`, then update the changelog with the reporter/PR author attribution if it ships.
