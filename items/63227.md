## Summary

Keep open. Current main has partial startup-route mitigations for `openclaw status` and `openclaw gateway status --no-probe`, but I did not find high-confidence evidence that the broader full-initialization/event-loop hang class is fixed. The linked follow-up #64216 remains open and reports the same post-output hang for browser/plugin CLI commands beyond native Windows.

## Best Possible Solution

Keep this issue open until maintainers either reproduce and fix the remaining full-initialization shutdown handle on Windows/plugin CLI paths, or deliberately consolidate it into a canonical open CLI-hang issue. A good fix would add a targeted shutdown/handle regression for routed and plugin-owned CLI commands, especially `openclaw browser status/tabs/snapshot`, and verify the process exits promptly after output on Windows and WSL/Linux.
