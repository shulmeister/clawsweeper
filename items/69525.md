## Summary

Current main still appears to contain the reported bug: `runMessageAction` resolves the target channel, then unconditionally falls back to `input.defaultAccountId`, and that value is passed through the core send path into the target channel outbound context. Cross-provider policy only permits or blocks the send; it does not clear a source-provider account id. The linked fix PR #69529 is open/unmerged, so the issue should remain open as the canonical bug report.

## Best Possible Solution

Keep #69525 open as the canonical bug. Land a focused core fix in `src/infra/outbound/message-action-runner.ts` that detects `allowAcrossProviders && input.toolContext?.currentChannelProvider !== channel`, preserves explicitly supplied `accountId`, and otherwise avoids applying the source session `defaultAccountId` to the target provider. Add regression coverage for cross-provider direct sends, including the `sendText`/`sendMedia` or plugin dispatch context receiving `undefined` unless an account id was explicitly provided. PR #69529 is relevant but should be narrowed if unrelated fixes are bundled.
