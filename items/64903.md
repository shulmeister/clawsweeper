## Summary

Keep open. Current main still uses `NodeForegroundService` as a persistent `dataSync` foreground service on targetSdk 36, with no guard for exhausted Android 16 foreground-service quota and no timeout/error fallback. The current tests and docs confirm the `dataSync` behavior rather than showing a shipped mitigation, and the provided related-item context does not identify a canonical duplicate or merged fix.

## Best Possible Solution

Keep this issue open and fix it in the Android app. The likely path is to stop using `dataSync` for the always-on gateway connection on targetSdk 36/Android 16, choose a foreground-service type or lifecycle pattern that matches Android policy, add timeout/start-failure handling around `NodeForegroundService`, and cover the behavior with a focused unit test plus device or emulator proof where possible.
