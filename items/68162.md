## Summary

Keep open. Current main's nominal Control UI reconnect path does not obviously reset sessions, but the issue has a recent independent reproduction on v2026.4.21 and I did not find a targeted fix, release note, or regression test proving the reported `action=new` / `.reset.*` takeover is gone. The observed `commands.log action=new` points at an actual reset path, so this needs a focused reconnect reproduction before maintainer cleanup can safely close it.

## Best Possible Solution

Add a focused regression around Control UI WebSocket reconnect/page refresh: seed an existing `sessionKey` and transcript, connect as `openclaw-control-ui`, force close/reconnect with the same settings, then call `chat.history` and a normal `chat.send`. Assert the same `sessionId` is retained, no `sessions.reset` or command hook with `action=new` fires, and no `.jsonl.reset.*` archive is created. If it fails, fix the frontend reconnect/session-selection path or the gateway reset trigger that fires for Control UI; if it passes, document the exact conditions that distinguish reconnect from an explicit New Session/reset.
