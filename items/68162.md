## Summary

Keep open. Current `main` still does not show an explicit Control UI websocket reconnect session-resume path, and the existing reconnect coverage does not prove that an active webchat session is preserved across socket drops or refreshes. Server-side session reuse still falls back to creating a new `sessionId` when the stored entry is absent or considered stale, so there is not enough evidence to conservatively close this as fixed or obsolete.

## Evidence

- **Websocket connect payload has no session binding:** `GatewayBrowserClient` reconnects by opening a fresh socket and sending only protocol/client/auth/device metadata in `connect`; there is no `sessionKey`/`sessionId` in the connect payload, so reconnect itself does not explicitly rebind the active chat session.
  - file: [ui/src/ui/gateway.ts:388](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/gateway.ts#L388)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Control UI hello path reloads UI state but does not resume a prior chat session:** After reconnect, `connectGateway` clears transient UI state, subscribes to sessions, reloads identity/agents/health, and refreshes the active tab. It does not issue a dedicated session-resume call for the previously active chat session.
  - file: [ui/src/ui/app-gateway.ts:292](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/app-gateway.ts#L292)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Chat history/send use the local `sessionKey` after connect:** The Control UI chat controller requests `chat.history` and `chat.send` with the current `sessionKey`, but this is just normal chat traffic after reconnect, not a websocket-level resume handshake.
  - file: [ui/src/ui/controllers/chat.ts:152](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/controllers/chat.ts#L152)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Server still creates a fresh session when the stored entry is not reusable:** `initSessionState` reuses an existing session only when the prior entry exists and is considered fresh; otherwise it assigns `sessionId = crypto.randomUUID()` and marks the turn as a new session.
  - file: [src/auto-reply/reply/session.ts:427](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/auto-reply/reply/session.ts#L427)
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Reconnect tests do not cover chat-session continuity:** Current Control UI reconnect tests cover stale callback suppression, pending approvals, shutdown messages, and queued abort handling, but they do not assert that the same webchat session is resumed after websocket reconnect or page refresh.
  - file: [ui/src/ui/app-gateway.node.test.ts:215](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/app-gateway.node.test.ts#L215)
