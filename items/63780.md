## Summary

Keep this PR open. Current main has the uncaught-exception handler registry, but it does not implement this PR's requested behavior: transient uncaught network errors still exit through the CLI/global uncaught path, and the Telegram monitor only registers an unhandled-rejection restart handler, not an uncaught-exception handler. The linked production symptom and #60515 are still a plausible current bug path, so this PR remains a useful implementation candidate rather than cleanup-close material.

## Best Possible Solution

Keep this PR open and have a maintainer review or rebase it against current main. The best implementation is to preserve the current fatal-hook behavior, add transient uncaught network suppression in the top-level CLI/global handlers, register a Telegram uncaught-exception handler that reuses the polling restart path, avoid duplicate legacy-handler logging if practical, and keep the regression coverage for CLI ENETUNREACH suppression plus Telegram uncaught polling HttpError restart.
