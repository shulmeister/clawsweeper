## Summary

Keep open. Current `main` still hardcodes the MiniMax usage polling endpoint to the CN host, so the PR's reported bug is not implemented on `main`. The proposed PR also has an unresolved behavioral concern from review: its `baseUrl` selection order can diverge from `resolveUsageAuth` token selection when both `minimax` and `minimax-portal` are configured, and it adds no URL-routing test coverage.

## Evidence

- **Current main still ignores provider baseUrl when fetching MiniMax usage:** The bundled MiniMax provider's `fetchUsageSnapshot` passes only `ctx.token`, `ctx.timeoutMs`, and `ctx.fetchFn` into `fetchMinimaxUsage`; it does not read configured provider `baseUrl` from `ctx.config`.

## Risks / Open Questions

- Global MiniMax users on `api.minimax.io` still appear affected on current `main` because usage polling remains pinned to the CN host.
- Closing this PR would discard a plausible bug fix, but merging it as-is risks mismatching the selected auth token and usage host when both MiniMax provider configs exist.
- The touched surface still lacks targeted tests for usage-host routing, so regressions would be easy to miss.
