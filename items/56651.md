## Summary

Keep open. Current `main` has partial mitigation via readiness probes and channel health monitoring, but it does not implement configurable gateway announce/connection/plugin-init timeout settings, and gateway channel startup still awaits plugin/channel startup without a timeout wrapper.

## Evidence

- **Gateway config surface lacks the requested timeout knobs:** `GatewayConfig` exposes `channelHealthCheckMinutes`, `channelStaleEventThresholdMinutes`, and `channelMaxRestartsPerHour`, but no `announceTimeoutMs`, `connectionTimeoutMs`, or `pluginInitTimeoutMs` fields under `gateway`.
  - file: [src/config/types.gateway.ts:453](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/types.gateway.ts#L453)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Schema/docs only cover health-monitor settings, not startup/connect timeout config:** The gateway schema validates only the health-monitor fields in this area, and the public docs/reference describe the same three settings. There is no shipped gateway config contract for configurable announce/connection/plugin-init timeouts.
  - file: [src/config/zod-schema.ts:785](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/zod-schema.ts#L785)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway startup still waits on channel startup without a timeout:** `startGatewaySidecars()` directly awaits `params.startChannels()` inside the startup path. If a channel/plugin startup hangs instead of resolving or rejecting, this path has no local timeout/fail-fast guard.
  - file: [src/gateway/server-startup-post-attach.ts:258](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-startup-post-attach.ts#L258)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Per-channel startup hands plugin `startAccount` through without timeout wrapping:** `startChannelInternal()` creates the task by calling the channel plugin's `startAccount(...)` and tracks it, but does not bound that promise with a startup timeout. That means the requested configurable startup/connect timeout behavior is not implemented in core.
  - file: [src/gateway/server-channels.ts:418](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-channels.ts#L418)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Main does provide partial observability and recovery, but not the requested feature:** The current docs and tests show `/ready`/`/readyz` returning not-ready while startup sidecars are pending, and the gateway health monitor can restart stale/disconnected channels. Those are useful mitigations, but they are not configurable startup/connect/plugin-init timeout controls.
  - file: [docs/cli/gateway.md:98](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/cli/gateway.md#L98)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
