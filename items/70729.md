## Summary

Keep #70729 open. Current main authenticates trusted-proxy WebSocket connections and extracts the configured user header, but that identity is not persisted on the WebSocket client record and `chat.send` still builds the agent `MsgContext` sender fields from the Control UI client id/displayName rather than the trusted-proxy user. The issue is recent, concrete, unprotected, and not covered by the related title-search reports.

## Best Possible Solution

Implement this in the gateway WebSocket path: persist the authenticated trusted-proxy user from `authResult.user` on `GatewayWsClient` or an equivalent request context, thread it into server-method client types, and have webchat `chat.send` populate agent-visible identity (`SenderId`/`SenderName`, or an explicitly designed authenticated-user field) from that trusted value. Add regression coverage for trusted-proxy Control UI `chat.send` proving the agent `MsgContext` receives the email, plus a token/password webchat test proving no spoofable sender is accepted from params.

## Risks / Open Questions

- This is a security-sensitive auth boundary: the implementation should only use identity captured from a successful trusted-proxy handshake, not any browser-supplied sender field.
- Sender semantics need a maintainer choice: using `SenderId`/`SenderName` matches other channels, but a distinct authenticated-user field may be clearer if downstream code must distinguish proxy identity from channel-native sender identity.
