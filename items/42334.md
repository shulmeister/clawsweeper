## Summary

Current main already solves the PR's central bug by routing native OpenAI WebSocket only for the default or api.openai.com Responses endpoint. Explicit local/custom OpenAI-compatible baseUrls stay out of the native WebSocket path and use the HTTP streaming transport, while an omitted baseUrl is still treated as direct OpenAI.

## Best Possible Solution

Close this stale PR and keep the shipped endpoint-classifier based transport selection. It solves the user-visible bug without taking the PR's unsafe wrapper change; future regressions should add focused coverage around `shouldUseOpenAIWebSocketTransport` and the HTTP transport stream rather than reviving this patch as-is.
