## Summary

Keep open. Current `main` at `60f7a59f5ed581839cfd1d52583449c5aa1f03a4` still implements the exact brittle respawn path described in the issue: detached gateway respawn reuses `process.execArgv` plus `process.argv.slice(1)` and the gateway loop exits after that handoff. The focused tests still assert that behavior. The linked PR `#52319` was closed unmerged (`merged_at: null`), so there is no evidence the proposed stable-wrapper fix landed on `main` or in `v2026.4.22`.
