## Summary

Current `main` already routes Slack `read`/`readMessages` through the action runtime, which prefers `userToken` for read operations, passes that token into `readSlackMessages`, and has matching regression coverage and docs. The same implementation is present in the shipped `v2026.4.23` release.

## Evidence

- **Message tool read path goes through Slack action runtime:** Generic Slack `read` actions are translated to `readMessages`, and the channel action adapter invokes `handleSlackAction` for message-tool requests.
  - file: [extensions/slack/src/message-action-dispatch.ts:107](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/slack/src/message-action-dispatch.ts#L107)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Read operations prefer `userToken` over `botToken`:** `handleSlackAction` resolves `userToken`/`botToken`, chooses `userToken ?? botToken` for `read`, builds `readOpts`, and passes those opts into `readSlackMessages`. That directly addresses the reported DM readback path.
  - file: [extensions/slack/src/action-runtime.ts:157](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/slack/src/action-runtime.ts#L157)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Regression test asserts read-time user-token preference:** `resolveReadToken()` drives a `readMessages` action and the test verifies that a configured `xoxp` token is the one forwarded on reads; the next test confirms fallback behavior when no user token exists.
  - file: [extensions/slack/src/action-runtime.test.ts:653](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/slack/src/action-runtime.test.ts#L653)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Docs match the implemented token model:** Slack docs now state that actions/directory reads can prefer the configured user token, while writes remain bot-token-first unless explicitly allowed otherwise.
  - file: [docs/channels/slack.md:422](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/channels/slack.md#L422)
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Latest release already contains the same fix:** The shipped `v2026.4.23` tag contains the same `getTokenForOperation("read") => userToken ?? botToken` logic in `extensions/slack/src/action-runtime.ts`, so this is not just an unreleased `main` change.
  - command: `git show v2026.4.23:extensions/slack/src/action-runtime.ts | nl -ba | sed -n '157,183p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

## Risks / Open Questions

- The low-level helper in `extensions/slack/src/actions.ts` still defaults to bot-token resolution when called directly without an explicit token, but the current user-facing message/read tool path does not use it that way; it injects the resolved read token through `action-runtime.ts`.
