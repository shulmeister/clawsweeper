## Summary

Keep open. The requested stable cron job id capability is still missing on current main: cron creation always assigns a UUID, the create type does not accept an id, and the Gateway/CLI add surfaces do not expose or accept an id. The PR addresses a real core cron/API gap, but it should be rebased and narrowed before merge because the branch is currently non-mergeable and still carries review-sensitive edge cases plus unrelated fixture/secret-baseline churn.

## Best Possible Solution

Keep the PR open and converge it into a focused core cron contract patch: add optional stable `id` support in `CronJobCreate` and `createJob`, preserve/normalize it through Gateway validation, CLI `--id`, cron-tool flat-param recovery, and Swift protocol models, reject duplicate ids case-insensitively under the existing add lock with a non-string guard, document the CLI/API behavior, and keep the regression tests. Drop unrelated secret-scanning and talk fixture churn unless handled in a separate maintainer-reviewed change.
