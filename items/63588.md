## Summary

Keep this PR open. Current main still does not implement the PR's central Slack-owned CDN redirect behavior: Slack media downloads start with an Authorization header, but the shared guarded fetch path strips sensitive headers, including Authorization, whenever the redirect crosses origins. The existing tests on main also assert cross-origin Slack redirect auth is stripped, so this PR is not obsolete even though the linked issue is closed.

## Best Possible Solution

Keep this PR open and ask for an update or maintainer patch that preserves Authorization only for Slack-owned HTTPS media redirect targets while keeping SSRF checks and token stripping for non-Slack redirects. The regression test should cover `files.slack.com` redirecting to Slack CDN with Authorization retained, and a non-Slack redirect with Authorization removed; the PR also needs its existing review comments resolved before merge.
