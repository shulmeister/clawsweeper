## Summary

Keep open. Current main has generic mitigations for silent error turns and UI error surfacing, but there is no strong evidence that Qianfan-specific API error payloads are parsed and preserved. The Qianfan plugin is still metadata-only, and the shared OpenAI transport still just forwards `Error.message`, so a Baidu-shaped body like `error_msg`/`error_code` would still depend on upstream SDK behavior rather than repo code.

## Evidence

- **Qianfan plugin has no runtime error adapter:** `extensions/qianfan` only defines provider metadata, onboarding, and catalog wiring. There is no Qianfan-specific transport or error-normalization code in the plugin.
  - file: [extensions/qianfan/index.ts:1](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/qianfan/index.ts#L1)
  - command: `rg -n "qianfan|error_msg|error_code|Baidu|千帆" src extensions packages ui docs test`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Shared OpenAI transport only forwards thrown error text:** On transport failure, the OpenAI path sets `output.errorMessage` from `error.message` (or `JSON.stringify(error)`) and does not inspect provider-specific response fields such as `error_msg` or `error_code`. If the SDK drops the Qianfan body, OpenClaw does not reconstruct it here.

## Risks / Open Questions

- The upstream `openai` SDK may already preserve some non-standard Qianfan bodies in `Error.message`, but that behavior is not asserted anywhere in this repo.
- Generic silent-error retry reduces user impact, so the original exact symptom may be harder to observe even if vendor error details are still lost.
- There is no Qianfan regression test on main proving that a Baidu-style error body is surfaced to the user.
