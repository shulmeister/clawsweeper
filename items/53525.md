## Summary

Keep open. Current main has generic empty-error rendering and provider HTTP helpers, but Qianfan still routes through the generic OpenAI-compatible completions path and I did not find Qianfan/Baidu error-shape normalization for top-level `error_msg`/`error_code` or streamed error objects without `choices`. The reported `stopReason: "error"`, `error: null`, `content: []` failure can still lose provider-specific detail before the UI fallback runs.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider plugin adapter or shared OpenAI-compatible error-normalization seam that extracts Baidu/Qianfan top-level `code`/`message`, legacy `error_code`/`error_msg`, and request identifiers from non-2xx responses and streamed error events into assistant `errorMessage`, with regression coverage for Qianfan-shaped payloads. The current TUI unknown-error fallback should remain only a fallback, not the primary fix.
