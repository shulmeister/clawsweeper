## Summary

Keep #72121 open. Current main still rejects `sk-ant-api03-...` through `openclaw models auth paste-token --provider anthropic`: the shared validator only accepts `sk-ant-oat01-...`, and the paste-token command uses that validator before storing the auth profile. Anthropic API-key auth exists as a separate provider method, but the reported paste-token behavior remains unfixed; the related fix PR #72124 is still open/unmerged in the provided context.

## Best Possible Solution

Keep the issue open until #72124 or an equivalent fix lands. The best code path is to widen the Anthropic paste-token validator to accept both `sk-ant-oat01-...` setup tokens and `sk-ant-api03-...` standard API keys, add focused validator and CLI regression tests for both prefixes and error messages, and clarify CLI/docs wording if paste-token is intended to remain token-only.
