## Summary

Keep #13239 open. Current main has provider setup docs, provider-owned model catalogs, runtime auth probes, and partial catalog status metadata, but it still does not publish the requested provider x model-family x capability compatibility matrix, deprecation policy, generated docs page, or doctor/status lifecycle output. The related Matrix plugin report #52899 is useful motivation, but it was a closed plugin-specific regression and does not supersede this broader provider policy proposal.

## Best Possible Solution

Keep the issue open until maintainers approve and ship a provider support-lifecycle policy plus a machine-readable registry, either by extending the existing model catalog/manifest/provider-index surfaces or adding a clearly separate lifecycle registry. The registry should cover provider, model family, capabilities, lifecycle state, deadline, replacement, maintainer ownership, and canonical issue link, then generate public docs and concise CLI output while keeping live auth/probe health separate from support lifecycle state.
