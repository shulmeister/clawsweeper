## Summary

Keep open. Current `main` still implements the Control UI chat speaker button with browser `speechSynthesis` only and does not appear to read `messages.tts` or the configured Microsoft/Edge voice, so the report remains plausible and unaddressed rather than obsolete.

## Evidence

- **Chat speaker button uses local browser TTS:** The chat UI `Read aloud` button directly calls `speakText(text)` from the click handler and does not fetch gateway TTS settings or provider output.
  - file: [ui/src/ui/chat/grouped-render.ts:632](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/ui/src/ui/chat/grouped-render.ts#L632)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '600,690p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Browser speech path has no OpenClaw voice selection:** `speakText` constructs a `SpeechSynthesisUtterance`, sets only generic `rate` and `pitch`, and calls `speechSynthesis.speak(utterance)` without selecting a configured voice or consulting OpenClaw config.
  - file: [ui/src/ui/chat/speech.ts:136](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/ui/src/ui/chat/speech.ts#L136)

- **OpenClaw docs scope `messages.tts` to runtime/provider-driven speech surfaces:** The runtime/plugin docs say `api.runtime.tts` uses core `messages.tts` configuration and provider selection, while the config help describes `messages.tts` as policy for supported voice or audio surfaces.
