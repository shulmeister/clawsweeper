## Summary

Keep open. Current main has improved WhatsApp watchdog behavior for quiet-but-healthy sessions, but it does not implement the shorter message-flow health check requested here. The current controller treats transport inactivity after 30 minutes as stale and only treats application silence as stale after a longer 4x safety window, so a connected socket with live frames but no emitted inbound messages can still exceed the reported 10-17 minute loss window.

## Best Possible Solution

Keep this issue open as the remaining WhatsApp message-loss/stale-message-flow bug. The best path is a WhatsApp-owned stale-pipeline detector or configurable app-silence timeout in the extension controller/config/docs, with regression coverage for a connected socket that keeps transport frames alive but emits no inbound messages for the reported 10-17 minute class of failure, while preserving the quiet-session fix from #70678 and coordinating any config surface with #63939.

## Evidence
