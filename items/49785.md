## Summary

Current `main` does not contain the PR's fix, and the open bug remains plausible. The current Slack dispatcher still consumes `replyPlan.nextThreadTs()` only once for normal deliveries in `replyToMode="first"`, while draft-preview threading still latches `usedReplyThreadTs` before send success. At the same time, the PR's fallback approach would change established `"first"` semantics by reusing the first thread for later normal payloads in the same turn, so this is not a conservative close.

## Evidence

- **Normal Slack delivery still uses single-use planner state:** `deliverNormally` computes `replyThreadTs` as `forcedThreadTs ?? replyPlan.nextThreadTs()`. On current `main`, there is no `usedReplyThreadTs` fallback here, so a second non-forced block/final delivery in `replyToMode="first"` still loses the thread target.
