## Summary

Keep open. Current `main` still resets `sessionId` while reusing the previous `sessionFile`, so the stale-transcript bug described in this PR is still present. The PR remains relevant, and the discussion shows the custom-name regression concern was addressed in the follow-up commit.

## Evidence

- **Reset path still forwards the old sessionFile unchanged:** In `performGatewaySessionReset`, `nextSessionId` is regenerated, but `resolveSessionFilePath` still receives `currentEntry.sessionFile` directly. There is no rewrite step before the new entry is stored.
  - file: [src/gateway/session-reset-service.ts:541](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/session-reset-service.ts#L541)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '500,620p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Path resolver preserves an existing valid sessionFile:** `resolveSessionFilePath` returns the provided `entry.sessionFile` when it is valid within the sessions directory, and only falls back to a new `<sessionId>.jsonl` path when that hint is absent or invalid. That means reset can keep pointing at the old transcript file.
