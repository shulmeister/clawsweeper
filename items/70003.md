## Summary

Keep open. Current main has useful related controls: immediate `chat.send` acceptance, `/stop`/`chat.abort`, `sessions.steer`, queue modes, and async subagent spawn acceptance. But it still does not implement the central request for concurrent same-session user turns during long-running tool execution: per-session agent runs are intentionally serialized, default follow-up messages are collected for a later turn, steer only injects at streaming/tool boundaries, and interrupt aborts then waits for the active run to end before continuing.

## Best Possible Solution

Keep #70003 open as the canonical product/design issue for true same-session preemption or concurrency during long-running tool execution. The best path is an explicit gateway/session design that either supports cancellable active turns with fast replacement, a separate background-result event channel, or carefully scoped concurrent turn lanes, while preserving existing `/stop`, `chat.abort`, `sessions.steer`, queue modes, and transcript consistency.

## Risks / Open Questions

- A real implementation must preserve transcript/session-file ordering guarantees that current per-session lanes provide.
- Changing default behavior from `collect` to automatic interrupt/steer would be product-visible and needs maintainer judgment.
