## Summary

Keep #37487 open. Current main still has named browser profiles and a global `browser.extraArgs` proxy workaround, but it does not implement the requested first-class `browser.profiles.<name>.proxy` config, per-agent browser profile defaults, or proxy diagnostics. The linked `m13v/browser-lock` project is useful prior art for per-agent browser isolation, and PR #42806 targeted this exact feature, but that PR is closed unmerged and its head commit is not present in this checkout.

## Best Possible Solution

Keep the issue open and implement a first-class browser/agent contract: add secret-aware `browser.profiles.<name>.proxy` for local managed profiles, apply it during managed browser launch, expose redacted diagnostics such as `proxyConfigured`, add `agents.defaults.browser.profile` and `agents.list[].browser.profile` with clear subagent inheritance semantics, and cover concurrent isolated multi-profile use in docs and tests. Treat PR #42806 and `m13v/browser-lock` as prior art rather than shipped fixes.

## Risks / Open Questions

- `browser.extraArgs` can route a managed browser through a proxy today, but it is global and cannot provide different proxy egress per profile or per agent during concurrent runs.
- Existing named profiles and separate CDP ports solve part of cookie/storage isolation, but the requested ergonomic agent binding, secret-aware proxy config, redacted diagnostics, and documentation remain missing.
- The external `https://github.com/m13v/browser-lock` layer is useful prior art and workaround evidence, but it is not an OpenClaw core/browser-plugin implementation of the requested contract.
