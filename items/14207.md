## Summary

Current main still does not implement configurable non-batch memory indexing concurrency. The memory indexer uses a hard-coded worker concurrency of 4 unless provider batch APIs are enabled, and the only exposed config knob remains `memorySearch.remote.batch.concurrency`, which is separate from the inline/custom-proxy path described in #14207. Related #68812/#68822 are open follow-up work, not a shipped or clearly canonical replacement for this older, specific report.

## Best Possible Solution

Keep #14207 open until OpenClaw adds an agent-scoped non-batch memory indexing concurrency setting, wires it through `MemorySearchConfig`, `ResolvedMemorySearchConfig`, Zod schema/help/generated metadata, docs, and focused tests, and uses it in `getIndexConcurrency()` without changing the existing provider batch API meaning of `remote.batch.concurrency`.
