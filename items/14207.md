## Summary

Keep #14207 open. Current main still fixes non-batch memory indexing concurrency at 4, while the exposed `remote.batch.concurrency` setting only applies to provider Batch API jobs. The config types, resolver, strict schema, docs, and focused tests still lack an agent-scoped inline indexing worker concurrency option. Related #68812/#68822 are adjacent retry/concurrency work, but they are not shipped or clearly canonical for closing this issue.

## Best Possible Solution

Keep #14207 open until OpenClaw adds an agent-scoped inline/non-batch memory indexing concurrency setting, wires it through `MemorySearchConfig`, `ResolvedMemorySearchConfig`, strict Zod schema, generated labels/help, public docs, and focused tests, then uses it in `getIndexConcurrency()` only for non-Batch-API indexing. Preserve the existing meaning of `remote.batch.concurrency` for provider Batch API jobs and coordinate naming with #68812/#68822 if that broader retry/concurrency work continues.
