## Summary

Keep open. Current main still uses a fixed non-batch memory indexing worker concurrency of 4, and the only configurable concurrency exposed today is `remote.batch.concurrency`, which applies when batch embedding APIs are enabled. The requested non-batch indexing concurrency knob remains missing.

## Evidence

- **Fixed non-batch index concurrency:** `EMBEDDING_INDEX_CONCURRENCY` is hard-coded to 4, and `getIndexConcurrency()` returns that fixed value unless batch mode is enabled.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:46](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/memory-core/src/memory/manager-embedding-ops.ts#L46)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '340,700p'`
  - sha: [1dfa52d0711d](https://github.com/openclaw/openclaw/commit/1dfa52d0711dfc86e1c04538fe46e3d43961766f)
- **Indexing workers use getIndexConcurrency:** Memory file entry building and per-file indexing both fan out via `runWithConcurrency(..., this.getIndexConcurrency())`, so the hard-coded non-batch value controls the reported worker pool.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:691](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/memory-core/src/memory/manager-sync-ops.ts#L691)

- **Batch concurrency is the only exposed knob:** Config resolution only reads `memorySearch.remote.batch.concurrency`, defaulting it to 2; there is no `memorySearch.indexing.concurrency` or equivalent non-batch indexing setting.
  - file: [src/agents/memory-search.ts:173](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/src/agents/memory-search.ts#L173)
