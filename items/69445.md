## Summary

Current main does not implement per-agent status card field suppression. `/status` and `session_status` still build and render provider usage when available, and the per-agent config type/schema has no `status`, `hideUsage`, `hideSections`, or equivalent presentation policy. The related #10005 context overlaps, but it tracks hiding the status card more broadly; this issue's per-agent usage-only privacy request remains concrete enough to keep open.

## Best Possible Solution

Keep the issue open and implement this as a core status presentation policy, likely under `agents.list[].status` and optionally `agents.defaults.status`. The implementation should update the config type, strict zod schema, generated config help/docs, `buildStatusText`, `buildStatusMessage`, `/status`, and `session_status` tests, and should avoid fetching provider usage when usage is hidden for the resolved agent.

## Risks / Open Questions

- A display-only hide could still fetch provider usage/billing data; the best fix should skip usage resolution when the active agent suppresses that section.
- The product shape needs a small decision: a narrow `hideUsage` boolean versus a future-proof `hideSections` or `visible` allowlist that can cover usage, cache, context, and cost lines consistently.
