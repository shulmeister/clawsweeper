## Summary

Keep open. Current main still uses a direct rename-based memory index swap with no EBUSY/EPERM/EACCES retry or fallback, the bug has multiple Windows reproductions in the discussion, and open PR #71611 explicitly proposes the fix with closing syntax. The issue should not be closed before that PR is reviewed and merged or rejected.

## Best Possible Solution

Keep this issue open and use #71611 as the current implementation candidate. The best path is to review or revise that PR so memory-core's atomic reindex move gets bounded retry/backoff for Windows lock errors, a carefully scoped fallback where appropriate, and regression tests for EBUSY/EPERM/EACCES behavior before merge.
