## Summary

Keep open. Current `main` still accepts numeric `allowFrom` entries at the shared schema layer, the dashboard submit path still coerces string values in `string|number` unions to numbers, and BlueBubbles still compares normalized handles as strings without re-adding a lost `+` prefix. That matches the reported failure mode rather than showing an implemented fix or an obsolete report.

## Evidence

- **Shared allowFrom schema still permits numbers:** `AllowFromEntrySchema` on current `main` is still `z.union([z.string(), z.number()])`, so BlueBubbles `allowFrom` and `groupAllowFrom` continue to accept numeric entries through the config schema.
  - file: [src/channels/plugins/config-schema.ts:19](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/channels/plugins/config-schema.ts#L19)
  - command: `nl -ba src/channels/plugins/config-schema.ts | sed -n '1,80p'`
  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Dashboard submit path still coerces union strings to numbers:** Form submission runs `coerceFormValues(...)` before serialization. In `anyOf`/`oneOf` unions, string inputs are tried against number variants and returned as numbers when `Number(value)` is finite. JavaScript converts `"+19175971855"` to `19175971855`, reproducing the reported prefix loss during save.
  - file: [ui/src/ui/controllers/config/form-coerce.ts:55](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/ui/src/ui/controllers/config/form-coerce.ts#L55)

  - sha: [9ece33c50506](https://github.com/openclaw/openclaw/commit/9ece33c50506ba031df382f69bde65dfda19bea5)
- **Config form save path uses that coercion on submit:** `serializeFormForSubmit` calls `coerceFormValues(state.configForm, schema)` before JSON serialization, so the coercion above is in the active dashboard save path, not dead code.
  - file: [ui/src/ui/controllers/config.ts:123](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/ui/src/ui/controllers/config.ts#L123)

A saved numeric entry becomes `"19175971855"`, and nothing in the BlueBubbles normalization path adds the missing `+` back.
  - file: [extensions/bluebubbles/src/targets.ts:116](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/bluebubbles/src/targets.ts#L116)

The DM allowlist test passes with `allowFrom: ["+15551234567"]`, and the channel docs describe `allowFrom` as accepting E.164 numbers and show direct-handle examples like `+15555550123`.
