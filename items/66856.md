## Summary

Current main already fixes the reported `openclaw status`/`openclaw doctor` false `plugins.allow is empty` warning. The memory-search config hot path no longer cold-loads plugin capability providers, plugin trust checks merge the activation/source config allowlist before warning evaluation, and regression coverage asserts that an allowlisted discovered plugin does not emit either the open-allowlist or untracked-provenance warning. The implementation is present in the v2026.4.24 release tag.

## Best Possible Solution

Close this issue as fixed. Keep the current main implementation that avoids cold plugin capability loads during memory-search config resolution and preserves source-config allowlist trust for warning evaluation. The linked PR #63848 was a useful diagnosis, but the remaining cleanup for that PR is to supersede or close it against the shipped main behavior rather than track this issue separately.

## Evidence

- **Memory config path avoids cold plugin loads:** `resolveMemorySearchConfig` imports `getMemoryEmbeddingProvider` from the registered in-process memory provider registry, and the current code explicitly keeps startup/doctor config resolution limited to adapters already registered by the active runtime instead of cold-loading plugins. That removes the config-less capability-provider fallback diagnosed in the issue comments.
