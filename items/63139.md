## Summary

Keep open. Current main still runs `before_model_resolve` inside `runEmbeddedPiAgent`, and `runWithModelFallback` still invokes its `run` callback once per candidate, so an unconditional routing hook can still rewrite every fallback attempt. Main has a partial mitigation because hook context now carries `runId` plus current candidate `modelProviderId`/`modelId`, but it does not carry `attempt` or `isFallbackRetry`, and the public hook docstring still describes the prompt as being for "this run." The related #41487 remains open and is adjacent, not a clear canonical replacement for this concrete failure mode.

## Best Possible Solution

Keep the issue open and fix the core/plugin contract deliberately: either document the current per-candidate contract and add `attempt`/`isFallbackRetry` to `PluginHookAgentContext` with tests for an unconditional router plugin, or hoist model-resolution hooks to the logical-run boundary with compatibility handling for plugins that observe fallback candidates. Coordinate with #41487 because it covers the adjacent under-application side of the same hook/fallback boundary.

## Risks / Open Questions

- Hoisting `before_model_resolve` outside the fallback loop could remove current per-candidate visibility through `modelProviderId` and `modelId`; this needs explicit compatibility design.
- Adding `attempt` and `isFallbackRetry` is smaller, but all fallback callers should be checked so plugin context is consistent across user, auto-reply, memory, follow-up, and cron paths.
