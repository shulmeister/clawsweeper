## Summary

Keep open. Current main still runs `before_model_resolve` from inside each embedded fallback candidate attempt, so an unconditional routing hook can still override every candidate back to the router-selected model. Main has the partial #54265-style mitigation (`ctx.runId`, plus current candidate model fields), but it does not expose `attempt`/`isFallbackRetry`, does not document the hook as per-candidate, and does not implement the issue's single-shot logical-run behavior.

## Best Possible Solution

Keep this issue open until OpenClaw chooses and ships a clear hook contract. The strongest product fix is to run `before_model_resolve` once at the logical run boundary and feed that choice into `runWithModelFallback` as the primary candidate. A smaller compatible fix would expose `attempt` and `isFallbackRetry` on `PluginHookAgentContext`, wire them from the fallback loop through `runAgentAttempt` into `runEmbeddedPiAgent`, update the public hooks docs/types to state the hook is per fallback candidate, and add regression coverage showing a routing plugin can preserve fallback failover intentionally.
