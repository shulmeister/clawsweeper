## Summary

The reported `2026.4.1` cron `toolsAllow` validator mismatch is fixed on current `main` and is already present in released `v2026.4.23`: the CLI still writes `payload.toolsAllow`, the gateway cron schema now accepts that field for both add and update payloads, and regression tests cover the add/edit/update paths plus runtime forwarding.

## Evidence

- **`cron add` serializes `--tools` into the agentTurn payload:** The current CLI keeps `--tools` on cron creation and assigns the parsed allow-list to `payload.toolsAllow`, which is the shape the issue expected once the validator accepted it.
  - file: [src/cli/cron-cli/register.cron-add.ts:156](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/cron-cli/register.cron-add.ts#L156)
  - command: `nl -ba src/cli/cron-cli/register.cron-add.ts | sed -n '90,190p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **`cron edit` supports setting and clearing `toolsAllow` in payload patches:** The edit command still exposes `--tools` / `--clear-tools` and writes `patch.payload.toolsAllow` or `null`, so the previously failing CLI path is intentionally supported rather than removed.
  - file: [src/cli/cron-cli/register.cron-edit.ts:225](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/cron-cli/register.cron-edit.ts#L225)

- **Gateway cron schema now accepts `toolsAllow` on both add and update payloads:** `cronAgentTurnPayloadSchema` includes optional `toolsAllow`, and both `CronPayloadSchema` and `CronPayloadPatchSchema` instantiate that field for create/update validation, removing the Ajv rejection described in the report.
  - file: [src/gateway/protocol/schema/cron.ts:15](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/gateway/protocol/schema/cron.ts#L15)
