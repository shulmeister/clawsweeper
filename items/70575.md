## Summary

Keep #70575 open. Current main still builds the agent session patch from the earlier `loadSessionEntry` result for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`, then merges that stale patch into the fresh store entry inside `updateSessionStore`. The related PR #19328 remains open in the provided context, and current tests do not contain the proposed stale-cache/fresh-store regression coverage for these residual fields.

## Best Possible Solution

Keep the issue open and patch `src/gateway/server-methods/agent.ts` so the locked `updateSessionStore` mutator either builds the session entry from the fresh store entry or conditionally includes only delivery/channel fields derived from the current request/effective delivery context. Add focused regression tests in `src/gateway/server-methods/agent.test.ts` for stale cached entries preserving fresh `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`.

## Risks / Open Questions

- The fix should preserve intentional request-supplied delivery updates while omitting stale fallback fields from the patch when the current run has no effective delivery value.
- Regression coverage should model a stale cached `entry` plus a fresh locked store entry for both delivery fields and `channel`, because the generic store merge behavior is expected and should not be changed broadly.
