## Summary

Current main still does not implement the requested behavior. `x-openclaw-agent-id` can select the agent, but `/v1/chat/completions` still validates the request body `model` as an OpenClaw agent-target id and returns `invalid_request_error` for generic OpenAI SDK values such as `gpt-4`. The April 2026 issue comments also show the reporter says this remains actively needed.

## Best Possible Solution

Keep #30381 open. The likely fix belongs in `src/gateway/http-utils.ts` and `src/gateway/openai-http.ts`: when `x-openclaw-agent-id` selects a valid agent, ignore or bypass validation of the OpenAI body `model` for chat completions while still using the selected agent's configured model and still validating any explicit `x-openclaw-model` backend override. Add a regression test with `model: "gpt-4"` plus `x-openclaw-agent-id` to prove dispatch succeeds and routes to the header-selected agent.

## Evidence

There is no header-aware bypass before the rejection.

## Risks / Open Questions

- Changing this path should preserve the `x-openclaw-model` allowlist and the documented agent-target contract for requests that do not include `x-openclaw-agent-id`.
- The same helper is also used by `/v1/responses`, so maintainers should decide whether the compatibility bypass should apply there too or stay scoped to chat completions.
