## Summary

Keep this issue open. Current main still has a real mismatch: sandboxed `write`/`edit` tool paths require Python in the helper runtime, while ordinary shell writes can succeed without it. The checked-in default sandbox Dockerfile installs `python3` when built through the documented setup script, and SimSef's workaround confirms that adding Python fixes the symptom, but current runtime provisioning can still tag a bare `debian:bookworm-slim` image as the default sandbox image, which would preserve the reported failure mode.

## Best Possible Solution

Keep the issue open and fix the product/runtime contract. The most direct path is to stop auto-tagging bare `debian:bookworm-slim` as the default sandbox image, ensure default image creation/doctor repair produces an image with `python3`, detect and warn/repair stale default images that lack the helper dependency, and add a regression test around missing/default-image provisioning. A broader fix could also remove the Python dependency from the pinned mutation helper or provide a precise runtime diagnostic that distinguishes shell write success from helper runtime requirements.
