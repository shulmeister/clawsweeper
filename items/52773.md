## Summary

Current main already carries the WhatsApp listener/status refactor that addresses the reported symptoms: listener readiness is account-scoped, reconnect handoff keeps the prior controller live until a replacement listener is ready, and status reporting now surfaces stale/reconnecting/stopped/logged-out listener states instead of only coarse connected/running flags.

## Close Comment

Closing from a Codex review because current OpenClaw already implements the relevant WhatsApp listener/status fixes. The older v2026.3.13-1 code used an unscoped active-listener check in `extensions/whatsapp/src/channel.ts:320`, which fits the multi-account false-positive `status --probe` symptom in this report. Current main moved that logic to account-scoped readiness checks in `extensions/whatsapp/src/heartbeat.ts:6`, keeps the previous registered controller alive until a replacement listener is ready in `extensions/whatsapp/src/connection-controller.ts:339`, and reports stale/reconnecting/stopped/logged-out listener states through `extensions/whatsapp/src/status-issues.ts:43` and the WhatsApp status adapter in `extensions/whatsapp/src/channel.ts:187`. Those paths are already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with current gateway logs, exact account ids involved, and the `status --probe` output for the affected account.
