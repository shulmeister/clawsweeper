## Summary

Keep #20802 open. Current main still implements `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the remaining `after_tool_call` result-mutation request is not shipped or fully superseded. Related PR #61485 appears to track only the `llm_input`/`llm_output` portion, so this PR still contains unique product/API work that needs maintainer judgment.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the broader direction or lands a canonical replacement. Let #61485 or an equivalent focused PR cover mutable `llm_input`/`llm_output`, then make a separate API decision on whether adapter-side `after_tool_call` should become result-mutating or whether `tool_result_persist` remains the intended tool-result mutation surface.
