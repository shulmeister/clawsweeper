## Summary

Keep #20802 open. Current main still keeps `llm_input`, `llm_output`, and `after_tool_call` as void/observation hooks, and the embedded attempt plus tool-adapter paths do not apply returned mutations. Related PR #61485 overlaps the `llm_input`/`llm_output` part, but it does not supersede #20802's adapter-side `after_tool_call` result-mutation proposal.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the broader direction or lands a canonical replacement. Prefer reviewing or landing the focused `llm_input`/`llm_output` work through #61485 or another narrow PR, then split adapter-side `after_tool_call` result mutation into a separate plugin API decision that accounts for current exactly-once hook dispatch and `tool_result_persist` behavior.
