## Summary

Keep open. Current main has meaningful partial mitigation: `chat.history` and session-history snapshots now hide default heartbeat prompts, `HEARTBEAT_OK`, and marked internal runtime-context blocks. But the reported memory-system action block is still not covered by the visible-history projection, inbound metadata stripper, or internal-runtime-context stripper. The issue comment’s direction remains valid: this needs a customer-safe transcript/history boundary fix, ideally structural provenance rather than WebChat-only rendering logic.

## Best Possible Solution

Keep the issue open. The best maintainer path is to make memory and heartbeat system-injection rows structurally internal before they reach visible history, or extend `projectChatDisplayMessages`/session-history SSE sanitization with narrowly tested known-block handling. Add regression tests for string content and text-block content covering the reported memory action block, default heartbeat prompt with workspace-file suffix, normal user text that mentions those phrases, `chat.history`, and session-history SSE. Update public WebChat/Control UI docs once the boundary behavior is guaranteed.

## Risks / Open Questions

- A naive text-only filter for `Read HEARTBEAT.md` or `## Memory system` could hide legitimate user-authored text; provenance or explicit hidden/internal transcript metadata would be safer where feasible.
- External memory plugins may produce similar prompt blocks with different wording, so a durable fix should provide a clear plugin/core mechanism for agent-only prompt context instead of relying only on literal strings.
