## Summary

Current `main` and the shipped `v2026.4.23` Control UI do not have a code path that would render `0 */6 * * *` as `3:00`: Cron Jobs render raw cron expressions, while Dreaming phase badges render `nextRunAtMs` timestamps supplied separately by the gateway.

## Evidence

- **Cron schedule formatter returns raw cron text:** `formatCronSchedule(job)` only parses `schedule.at`; for `schedule.kind === "cron"` it returns `Cron ${s.expr}${s.tz ? ... : ""}`. A job with `expr: "0 */6 * * *"` would display as raw cron text, not as `3:00`.
  - file: [ui/src/ui/presenter.ts:59](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/presenter.ts#L59)
  - command: `nl -ba ui/src/ui/presenter.ts | sed -n '55,72p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Cron Jobs view uses that formatter directly:** The Cron Jobs list subtitle is rendered with `formatCronSchedule(job)`, so the Cron Jobs panel follows the raw-expression behavior above.
  - file: [ui/src/ui/views/cron.ts:1484](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/views/cron.ts#L1484)
  - command: `nl -ba ui/src/ui/views/cron.ts | sed -n '1480,1488p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Dreaming phase badges render timestamps, not cron strings:** The Dreaming UI formats `phase.nextRunAtMs` with `formatPhaseNextRun`, and the phase row prints that value as the status. It does not parse `phase.cron` for display.
  - file: [ui/src/ui/views/dreaming.ts:362](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/views/dreaming.ts#L362)
  - command: `nl -ba ui/src/ui/views/dreaming.ts | sed -n '360,368p'; nl -ba ui/src/ui/views/dreaming.ts | sed -n '432,444p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Gateway status keeps cron and next-run time as separate fields:** `doctor.memory.status` builds phase payloads with a raw `cron` string, then separately derives `nextRunAtMs` from managed cron job state. That matches the Dreaming UI contract and avoids parsing cron strings into display times.
  - file: [src/gateway/server-methods/doctor.ts:226](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/server-methods/doctor.ts#L226)
  - command: `nl -ba src/gateway/server-methods/doctor.ts | sed -n '220,248p'; nl -ba src/gateway/server-methods/doctor.ts | sed -n '699,717p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Tests preserve raw `*/6` cron data in dreaming status:** The Dreaming controller test fixture includes `light.cron: "0 */6 * * *"` alongside a separate `nextRunAtMs`, reinforcing that current UI state carries the cron expression through without converting it to a wall-clock label.
  - file: [ui/src/ui/controllers/dreaming.test.ts:127](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/ui/src/ui/controllers/dreaming.test.ts#L127)

- **Same Cron formatter shipped in latest release:** The latest release tag `v2026.4.23` already contains the same `formatCronSchedule` implementation that renders cron expressions as raw text, so the current shipped Control UI also does not match the reported `*/6` -> `3:00` path.
  - file: [ui/src/ui/presenter.ts:59](https://github.com/openclaw/openclaw/blob/899e927dab9ef2d6164004d34a30410b3506b6f5/ui/src/ui/presenter.ts#L59)
