## Summary

Keep open. Current main still does not implement auto-mirroring for existing local `<qqmedia>` files outside OpenClaw-managed media roots. The current QQ Bot code allows `~/.openclaw/media` and remaps some missing legacy QQ Bot workspace paths, but an existing file like `~/.openclaw/workspace/projects/foo/test.pptx` still falls through to the same storage-boundary error. The security concern raised in the issue discussion is valid, so this needs a constrained QQ Bot fix rather than closure.

## Best Possible Solution

Keep this issue open and implement a QQ Bot-owned constrained staging path: pass or reuse core `mediaLocalRoots`/`mediaAccess` for `<qqmedia>` delivery, realpath-check files against agent/workspace/media allowlists, copy only allowed files into managed QQ Bot media storage or stage them through the existing outbound attachment store, then keep the current size/type/symlink boundary checks. Update QQ Bot media docs/skill guidance and add regression tests for allowed workspace files, rejected outside-root files, symlink escapes, and the original `<qqmedia>` PPTX-style flow.

## Risks / Open Questions

- A naive implementation that copies any existing absolute path into QQ Bot media storage could let a user or model send arbitrary host files.
- Fixing only generic reply payload media staging will not cover `<qqmedia>` tags because QQ Bot handles those tags before the plain reply path.
- The QQ Bot skill text currently tells agents they can send local files with absolute `<qqmedia>` paths, which conflicts with the storage-boundary behavior and may keep producing this UX failure.
