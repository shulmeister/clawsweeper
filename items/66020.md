## Summary

Keep open. Current main still rejects existing local files outside the guarded QQ Bot/OpenClaw media roots, including the workspace-style path from the report, and the requested auto-mirror behavior would relax an intentionally security-sensitive outbound media boundary.

## Best Possible Solution

Keep the issue open and resolve it with an explicit QQ Bot media policy decision. A safe implementation would likely allow only OpenClaw-produced/generated artifacts with trusted provenance, or add an approval-gated export/mirror path into `~/.openclaw/media/qqbot` with size, type, realpath, and symlink-race checks. If maintainers decide not to relax the boundary, update the QQ Bot media skill/docs so agents save sendable files under OpenClaw-managed media storage instead of arbitrary workspace project paths.
