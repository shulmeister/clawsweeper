## Summary

Keep open. Current `main` contains plausible post-report fixes for Control UI blanking/auth scope handling and for oversized session-store cleanup, but the issue report does not include the console/network evidence needed to map #49776 to either path with high confidence, and I could not perform a live reproduction in this checkout environment.

## Evidence

- **Report is too ambiguous to map to one fix:** The issue body says the dashboard loads HTML/CSS/JS and then freezes, but it includes no browser console error, no failing RPC/WebSocket frame, no screenshot, and no config details beyond `windows11`, npm install, and a Bailian model. The follow-up comments are speculative; the reporter's later 'Recovery Steps' mention checking port occupancy and session file sizes, but do not confirm the root cause.
  - command: `GitHub context provided in the task`
- **There is a likely-but-not-proven auth blanking fix after the reported version:** `CHANGELOG.md` documents a 2026.3.23 fix: 'Control UI/auth: preserve operator scopes through the device-auth bypass path, ignore cached under-scoped operator tokens, and show a clear `operator.read` fallback message when a connection really lacks read scope, so operator sessions stop failing or blanking on read-backed pages.' Current code matches that behavior: `ui/src/ui/gateway.ts` refuses cached operator tokens without read/write/admin scope, `ui/src/ui/gateway.node.test.ts` covers that case, and `ui/src/ui/controllers/sessions.ts` now converts missing `operator.read` failures into a visible error instead of leaving the page blank. That is relevant, but it does not prove #49776 was this auth-scope bug.
