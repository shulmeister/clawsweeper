## Summary

Current main still does not implement the requested SKILL.md instruction-isolation boundary. It mitigates startup exposure by injecting an escaped skills catalog, but selected skills are still read and followed as model-facing instructions, with no role locking, behavior monitoring, or enforceable per-skill capability permission model. The existing Apr 26 maintainer/Codex comment also kept this open, and the relevant surfaces have not changed since that review.

## Best Possible Solution

Keep #12512 open as core security hardening. A good fix belongs around core skill prompt assembly and SKILL.md consumption: define a documented instruction-isolation model, add enforceable skill capability/permission metadata if skills can influence role, memory, or tools, and add tests/docs showing malicious skill text cannot cross that boundary. This should not be closed as ClawHub work because the missing piece is the core runtime boundary that all skills depend on.
