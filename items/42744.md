## Summary

Current main already has the requested behavior under `agents.defaults.imageModel` rather than a separate `visionModel`: OpenClaw documents automatic image fallback/routing when the primary model cannot accept images, the media-understanding runtime prefers the configured image model automatically, and regression tests cover that path. Changelog entries show the image-model config shipped in 2026.1.5 and implicit auto image routing was fixed in 2026.2.17, both before this issue was filed.

## Risks / Open Questions

- The implementation uses the existing `imageModel` name rather than the issue's proposed `visionModel` key, so the request is implemented functionally but not with the exact config shape suggested in the report.
- The automatic path appears to route image handling through media-understanding/image-model resolution; if the reporter expected the entire turn to switch session models rather than using the existing image fallback pipeline, that narrower expectation is not implemented as a distinct config key.

## Close Comment

Closing as implemented on current `main`. Codex review found that OpenClaw already supports this behavior via `agents.defaults.imageModel` rather than a separate `visionModel` key: `docs/concepts/models.md:26` says `imageModel` is used when the primary model cannot accept images, `docs/gateway/config-agents.md:339-341` documents fallback routing for image input, and `src/media-understanding/runner.ts:580-602` plus `src/media-understanding/runner.vision-skip.test.ts:195-212` show the runtime automatically preferring the configured image model for image handling. The changelog also shows this shipped in `2026.1.5` with the implicit auto-routing fix in `2026.2.17` (`CHANGELOG.md:6800-6801`, `CHANGELOG.md:4665`). If you still hit a current-main case where inbound images are not handled correctly, please open a new bug with the exact provider/model pair, channel, and reproduction steps.
