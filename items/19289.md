## Summary

Current main still does not provide the requested third-party browser backend/profile-driver registration interface. The browser plugin is replaceable as a whole, but the requested seam is narrower and deeper: profile driver selection remains closed over built-in modes, profile creation rejects unknown drivers, resolved runtime profiles only carry built-in driver values, and the plugin SDK exposes no browser-driver registration capability. The existing issue discussion already kept this open for the same reason, and current main does not show a newer implementation that changes that conclusion.

## Best Possible Solution

Keep #19289 open for maintainer/product design. The likely implementation is an additive browser backend capability in core and the plugin SDK: allow plugins to register driver ids and adapters, update browser profile config/schema/docs to accept those ids, route profile resolution and creation through the registered driver registry, and add contract tests proving third-party MCP/custom adapters can bind without replacing the entire bundled browser plugin.
