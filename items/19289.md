## Summary

Keep #19289 open. Current main still does not expose a third-party browser backend/profile-driver registration interface. Browser profiles, config validation, runtime profile resolution, CLI/profile creation, and the public plugin API remain closed over built-in `openclaw`/CDP and `existing-session`/Chrome MCP paths, so this is a missing core/plugin-SDK capability rather than work that can be closed as implemented or moved entirely to ClawHub.

## Best Possible Solution

Keep #19289 open for maintainer/product design. The best path is an additive browser backend capability in core and the plugin SDK: allow plugins to register driver ids and adapters, update browser profile config/schema/docs and create-profile handling to accept registered ids, route profile resolution through the registry, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale.
