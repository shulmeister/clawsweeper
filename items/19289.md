## Summary

Keep #19289 open. Current main still lacks the requested additive browser backend/profile-driver registration seam: browser profile config, validation, resolution, control routes, CLI, client/status types, and Plugin SDK registration all remain closed around OpenClaw-managed CDP plus existing-session Chrome MCP. Whole-browser plugin replacement is documented, and the related Patchright request (#52190) fits that ClawHub path, but that does not satisfy this issue's request for third-party drivers selectable through browser profiles without replacing the bundled browser plugin.

## Best Possible Solution

Keep #19289 open for maintainer/product design. The best implementation path is an additive browser backend capability in core and the Plugin SDK: define a typed browser driver/adapter contract, let plugins register driver ids, broaden browser profile config/schema/client/CLI/control routes to accept registered ids, route profile resolution and runtime dispatch through that registry, document the contract at the public plugin/browser docs level, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. The related #52190 Patchright ClawHub outcome remains useful as a whole-plugin or optional-backend path, but it should not supersede this core API request.
