## Summary

Keep #19289 open. Current main still lacks an additive browser backend/plugin registration interface for profile-selectable third-party MCP/custom drivers. The shipped surfaces support the bundled browser plugin, whole-tool replacement, managed CDP profiles, remote/attach-only CDP profiles, and the built-in Chrome DevTools MCP existing-session path, but config, validation, client/control types, runtime resolution, SDK registration, docs, and tests remain constrained to built-in browser modes.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is an additive browser driver/adapter extension point in core and the Plugin SDK: define a typed browser backend contract, let plugins register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 as useful backend-specific prior art, not a canonical tracker for this API gap.
