## Summary

Current main still does not expose the requested third-party browser backend/profile-driver registration interface. The browser config/schema, runtime profile resolver, CLI/control create-profile paths, client types, and public plugin API remain closed over the built-in `openclaw`/CDP and `existing-session`/Chrome MCP paths. The related Patchright item is distinguishable: replacing the whole browser plugin can be ClawHub work, but this issue asks for the missing core/plugin-SDK capability that would let third-party drivers bind through profiles without replacing the bundled browser plugin wholesale.

## Best Possible Solution

Keep #19289 open for maintainer/product design. The best path is an additive browser backend capability in core and the Plugin SDK: define a typed browser driver/adapter contract, let plugins register driver ids, update browser profile config/schema/docs and create-profile handling to accept registered ids, route profile resolution and runtime dispatch through the registry, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale.
