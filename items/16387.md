## Summary

Keep #16387 open. Current main still treats configured provider headers as literal/static config, the Fireworks plugin does not add `x-session-affinity`, and the provider-owned hook APIs do not provide arbitrary `models.providers.*.headers` / `request.headers` template interpolation for custom providers. The related implementation PRs #16448 and #18213 were closed unmerged, while the latest maintainer comment also explicitly kept this issue open.

## Best Possible Solution

Keep the issue open. The best maintainer path is a narrow, documented runtime header-template resolver for `models.providers.*.headers` and `models.providers.*.request.headers`, evaluated at provider request dispatch with explicit placeholders for session key, agent identity, and per-turn/run identity. Add regression coverage for Fireworks `x-session-affinity`, LiteLLM/Langfuse headers, main turns, compaction, retries, static headers, SecretRef-backed values, and OpenAI-compatible transports. If maintainers reject config templating, document the provider-plugin/API path that replaces it and ensure it covers custom providers, not only bundled provider plugins.

## Risks / Open Questions

- A Fireworks-specific provider hook could partially solve the original Fireworks caching case, but it would not solve the LiteLLM/Langfuse and other custom-provider header-template use cases described in the issue comments.
- The exact placeholder set and precedence need maintainer design, especially around SecretRef-backed headers, retries, compaction, WebSocket transports, and whether to expose `run.id` versus transport `turnId`.
