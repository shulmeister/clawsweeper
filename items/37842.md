## Summary

Keep #37842 open. Current main at c7a0d9b1889ca09ab5bfd84cda39a8d50d3d5768 still has source-only InputProvenance and pairwise sessions_send A2A limiting; it does not carry a multi-agent chain, reject A->B->C->A cycles, or include regression coverage for graph-aware loop detection. The linked PR #37945 is useful prior work but remains closed/unmerged, and #51832 tracks a different Slack gateway error-cascade mechanism rather than this agent-initiated sessions_send chain guard.

## Best Possible Solution

Keep #37842 open for a core sessions_send fix. The likely path is an additive InputProvenance/gateway-schema chain or trace field, propagation through createSessionsSendTool, createOpenClawTools/pi-tools, nested runAgentStep/A2A follow-up calls, and a clear block/error when the target agent/session already appears in the chain. Add regression coverage for A->B->C->A while preserving the existing direct maxPingPongTurns behavior. Do not close as duplicate of #51832 because that issue covers Slack gateway-posted error cascades, not agent-initiated sessions_send provenance chains.
