## Summary

Keep #23014 open. Current main at f4e6322649a1ef0a9b47bd9d8b681c143850d669 still lacks extension-only `.m2a` audio support: the canonical audio extension set and MIME extension mapping omit `.m2a`, attachment kind fallback still depends on `isAudioFileName`, and Telegram audio preflight/context still depends on `audio/*` content types or that resolved audio classification. The linked PR #23260 was draft, closed, and unmerged, so it does not supersede the issue; the prior Codex/maintainer discussion correctly kept this open and the current checkout still matches that conclusion.

## Best Possible Solution

Keep the issue open until `.m2a` support is implemented or explicitly rejected by a maintainer. The likely fix belongs in the shared media pipeline: add `.m2a` as an audio alias in `src/media/mime.ts`, ensure generic-MIME saved media can retain or infer `.m2a` from the original filename where appropriate, and add focused tests for `detectMime` filename fallback, `isAudioFileName`, `resolveAttachmentKind`, and Telegram generic-MIME `.m2a` audio preflight/context behavior. Maintainers should also decide whether secondary allowlists such as webchat audio embedding, memory multimodal audio indexing, and voice-message compatibility should include `.m2a` for consistent media handling.
