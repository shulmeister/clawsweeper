## Summary

Keep #23014 open. Current main at 6a688e33f66f3965968343922e874cc0e9440998 still does not implement extension-only `.m2a` audio handling: the canonical audio extension set omits `.m2a`, `isAudioFileName` depends on that set, media attachment kind fallback depends on `isAudioFileName`, and Telegram preflight/content paths still rely on MIME or that resolved audio classification. The related PR #23260 was draft, closed, and unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open until `.m2a` support is implemented or explicitly rejected by a maintainer. The likely fix belongs in `src/media/mime.ts` by adding `.m2a` to the canonical audio extension and MIME mapping as appropriate, plus focused regression tests for `isAudioFileName`, `detectMime` filename fallback, and `resolveAttachmentKind`; maintainers should also decide whether secondary audio extension allowlists such as memory multimodal and webchat audio embedding should include `.m2a` for consistent media handling.
