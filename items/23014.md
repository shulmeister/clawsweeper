## Summary

Keep #23014 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still lacks `.m2a` in the shared MIME/extension map and audio filename classifier, and the Telegram inbound path still depends on Telegram message type or an `audio/*` content type rather than `.m2a` filename inference. Related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers implement or explicitly reject `.m2a` support. The likely fix belongs in shared media handling: add a `.m2a` audio MIME alias to `src/media/mime.ts`, make inbound save/detection preserve or use original filenames for generic-MIME Telegram documents, and add focused tests for `detectMime`, `isAudioFileName`, Telegram context/preflight behavior, and adjacent WebChat/UI audio allowlists if consistent support is intended.
