## Summary

Keep this issue open. Current main still does not implement Telegram inline query support: Telegram can request inline-query update delivery and report BotFather inline capability, but there is no runtime handler, no answerInlineQuery path, and no config/docs surface for enabling inline query UX. The related PR #51963 is useful context but was closed unmerged, not shipped.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely implementation belongs in the bundled Telegram plugin: add an explicit inline-query config/schema/docs surface, intentionally request and handle inline_query and chosen_inline_result updates, authorize senders conservatively, route bounded inline requests through the agent flow, answer with Telegram InlineQueryResultArticle results, and add focused Telegram plugin tests. Treat PR #51963 as reference material, not shipped behavior.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram chats, so implementation needs explicit authorization, privacy, and session/conversation design before routing them through the agent flow.
- The existing inlineQueries status flag can confuse operators because it reflects BotFather capability, not OpenClaw inline-query support.
