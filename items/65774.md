## Summary

Current main partially addresses persisted running cron jobs on startup, but the central safety request remains unresolved: missed cron slots can still replay immediately outside the current wall-clock window, cron removal does not abort in-flight runs, gateway shutdown only stops the scheduler timer, and no emergency outbound halt surface exists.

## Best Possible Solution

Keep this open and fix it in cron/gateway core. The likely path is to add a service-level registry for active cron run AbortControllers, make cron.remove/disable and gateway shutdown able to abort matching in-flight runs, add an explicit emergency stop or pause-all outbound path, and add regression coverage for an America/Guatemala out-of-hours reconnect plus deleting or aborting an active WhatsApp-style isolated delivery. If #55160 becomes the canonical maintenance-window/deferred-replay design, link this issue there but do not close it before the relevant safety behavior ships.

## Evidence
