## Summary

Keep open. Current `main` still has the two core behaviors behind this report: restart catch-up will immediately run a recurring cron from stale overdue `nextRunAtMs` state without re-validating the current cron window, and delete/stop paths still do not have an in-flight cancellation mechanism for an executing cron job.

## Evidence

- **Startup catch-up still treats stale overdue `nextRunAtMs` as runnable:** `isRunnableJob()` returns `true` as soon as `nowMs >= nextRunAtMs`, and `planStartupCatchup()` uses that predicate directly for restart catch-up. There is no extra check that the current wall clock still falls inside the cron expression's allowed hours before the job is started.
  - file: [src/cron/service/timer.ts:894](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cron/service/timer.ts#L894)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '857,1023p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Service startup still replays interrupted recurring jobs from persisted state:** On startup, `start()` clears stale `runningAtMs` markers, persists, and immediately calls `runMissedJobs()` before recomputing schedules. That matches the issue comment's restart/catch-up path and means an interrupted recurring job is replayed from stored scheduler state.
  - file: [src/cron/service/ops.ts:93](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cron/service/ops.ts#L93)

  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Current tests on `main` intentionally lock in restart replay behavior:** `service.restart-catchup.test.ts` explicitly asserts that an overdue recurring cron executes immediately on start and that an interrupted recurring cron is replayed on first restart. There is no adjacent regression proving out-of-window cron catch-up is rejected.
  - file: [src/cron/service.restart-catchup.test.ts:79](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cron/service.restart-catchup.test.ts#L79)

- **`cron remove` still does not abort an active execution:** `remove()` only filters the job out of the store, persists, and re-arms the timer. There is no hook from removal into the currently executing job or its abort signal, so deleting a running cron still is not a real stop path.
  - file: [src/cron/service/ops.ts:332](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cron/service/ops.ts#L332)

That leaves delete and shutdown paths without a handle to cancel in-flight cron work.

That halts future ticks, but it does not abort a job already inside execution.
