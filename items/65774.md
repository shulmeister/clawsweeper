## Summary

Keep #65774 open. Current main has a partial startup fix for persisted `runningAtMs` jobs, but the broader reported behavior is still not fully solved: startup catch-up still treats any past `nextRunAtMs` as runnable without re-checking whether the current time is inside the cron expression/window, and `cron.remove` still only removes persisted jobs without aborting an in-flight execution.

## Best Possible Solution

Keep the issue open and implement a focused cron safety fix: add catch-up gating for cron schedules using the job timezone/expression and a conservative replay window, add regression tests for an America/Guatemala 1 AM replay against a 9-12/14-17 schedule, and wire running cron executions to abortable state so `cron remove` or an explicit emergency stop can cancel in-flight isolated runs and outbound delivery attempts.

## Evidence

The follow-up root-cause comment specifically calls out startup catch-up ignoring cron schedule windows, lack of delete-time abort, and lack of an emergency kill switch.
