## Summary

Keep #68438 open. Current main has a first-time Android TLS trust prompt, but the reported certificate-renewal case still remains: once a fingerprint is stored, Android reconnects with that old fingerprint as the required expected pin, a renewed reverse-proxy certificate fails with the same hard mismatch error, and there is no fingerprint-changed re-trust path.

## Best Possible Solution

Add a deliberate Android certificate-rotation flow for previously trusted wss gateways, likely in NodeRuntime/GatewayTls error handling: detect a stored-pin mismatch, probe/present the new SHA-256 fingerprint with clear changed-certificate copy, update the stored pin only after user approval, and add regression tests for stored-pin mismatch, approval, decline, and reverse-proxy/setup-code paths. Preserve the existing fail-closed rule that unauthenticated discovery TXT must never replace a stored pin silently.
