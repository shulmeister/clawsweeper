## Summary

Keep this PR open. Current main still lacks the fallback `inbound_claim` broadcast this PR is trying to add, and the related discussion points to a real plugin use case rather than obsolete work. The patch still needs maintainer review and a narrow follow-up because the proposed handled broadcast path can drop synthetic replies and bypass the existing inbound dedupe cleanup tail.

## Best Possible Solution

Keep this PR open and request a narrow follow-up patch: add the fallback broadcast guarded by `pluginFallbackReason`, deliver `broadcastResult.reply` with the same semantics as the targeted plugin-bound handled path, explicitly clean up inbound dedupe before any new terminal return, and add regression tests for missing-plugin/no-handler broadcast, handled-with-reply, and dedupe cleanup.
