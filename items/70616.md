## Summary

Keep #70616 open. Current main partially normalizes a dreaming execution shape, which supports the commenterâ€™s observation, but the shipped memory-core manifest schema still rejects `dreaming.execution` and per-phase `execution`, fallback models are not modeled, and the narrative subagent path still runs with the default runtime model rather than passing the configured model override.

## Best Possible Solution

Implement this in `memory-core`: extend `openclaw.plugin.json` to accept global and per-phase execution config, add resolver/tests for the chosen `execution.model`/fallback shape, wire the resolved model/provider/fallback into dreaming narrative subagent runs through the existing plugin subagent override API, and update the Dreaming plus memory config docs with the supported keys and required `plugins.entries.memory-core.subagent.allowModelOverride` trust policy.

## Risks / Open Questions

- Adding only the manifest schema would make config validate but still not honor cheaper/fallback models unless the resolved execution config is wired into the subagent run path.
- The final config shape needs a maintainer choice because current code reads `dreaming.execution.defaults.model`, while the issue proposes `dreaming.execution.model`. Backward-compatible acceptance of both may be appropriate.
