## Summary

Keep #70616 open. Current main has partial internal dreaming execution normalization, but it does not satisfy the requested user-facing config: the strict `memory-core` plugin manifest still omits `dreaming.execution`, docs only expose `enabled` and `frequency`, the Dream Diary docs say narrative subagent runs use the default runtime model, and the memory-core dreaming path drops execution settings before scheduling/running the sweep. The requested per-dreaming model plus fallback-model support remains actionable in this repo.

## Best Possible Solution

Keep the issue open and implement a first-class, documented memory-core dreaming execution contract. The conservative path is to add `plugins.entries.memory-core.config.dreaming.execution` to the plugin manifest/schema/docs/tests, decide the canonical shape for primary plus fallbacks (prefer aligning with existing `{ primary, fallbacks }` model config rather than the issue's singular `fallback` array), plumb it through `resolveMemoryDreamingConfig` and memory-core dreaming/narrative subagent runs, and use the existing plugin subagent override trust/allowlist mechanism safely for provider/model overrides.
