## Summary

Keep open. Current main still requires `openclaw cron runs --id <id>` and has no positional `[id]` argument for `cron runs`, while nearby cron commands already use positional job IDs. The PR remains a focused, relevant CLI UX fix for #42300, though it should address the review nits before merge.

## Best Possible Solution

Keep the PR open and have it rebased or refreshed against current main, preserving `--id` compatibility, adding `cron runs <jobId>` parsing, rejecting conflicting positional/flag IDs, and covering positional, flag-only, conflict, and missing-ID cases in `src/cli/cron-cli.test.ts`. Update cron CLI docs if maintainers want to advertise the positional alias.
