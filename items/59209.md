## Summary

Keep open. Current `main` has stronger generic remote-CDP diagnostics and dedicated WSL2 remote-CDP troubleshooting docs, but it does not implement the issue’s requested Windows-specific detection for a stale `netsh portproxy` self-loop or a non-Chrome PID owning the CDP port.

## Evidence

- **Remote CDP availability errors stay generic:** The browser availability path only distinguishes generic remote-CDP cases like `Remote CDP ... is not reachable` and `Remote CDP websocket ... is not reachable`; it does not inspect Windows port ownership or portproxy state before failing.
  - file: [extensions/browser/src/browser/server-context.availability.ts:240](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/src/browser/server-context.availability.ts#L240)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '88,310p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Diagnostic categories do not cover port-owner or portproxy misroutes:** `diagnoseChromeCdp()` classifies failures as buckets such as `http_unreachable`, `http_status_failed`, `invalid_json`, and websocket handshake/health failures. There is no category for `svchost`/`iphlpsvc`, stale `portproxy`, or `chrome.exe` absence behind a listening port.
  - file: [extensions/browser/src/browser/chrome.diagnostics.ts:17](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/browser/src/browser/chrome.diagnostics.ts#L17)
