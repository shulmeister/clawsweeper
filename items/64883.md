## Summary

Keep open. Current `main` still treats almost any non-`update_plan` tool activity as sufficient progress, with only a narrow retry-safe exception for `read`/`search`-style tools, so a dummy tool call like `bash`/`exec` running `sleep` can still suppress the planning-only retry path the issue describes.

## Evidence

- **Non-plan tool activity still short-circuits planning-only retry:** `resolvePlanningOnlyRetryInstruction` returns `null` when `hasNonPlanToolActivity(...)` is true, unless the turn matches a narrow single-action narrative bypass. `hasNonPlanToolActivity` itself is just `toolName !== "update_plan"`, so arbitrary tools still count as progress.
  - file: [src/agents/pi-embedded-runner/run/incomplete-turn.ts:501](https://github.com/openclaw/openclaw/blob/d99d9eda371adc343e7e414830ec4430824c05ee/src/agents/pi-embedded-runner/run/incomplete-turn.ts#L501)
  - command: `nl -ba src/agents/pi-embedded-runner/run/incomplete-turn.ts | sed -n '493,612p'`
  - sha: [d99d9eda371a](https://github.com/openclaw/openclaw/commit/d99d9eda371adc343e7e414830ec4430824c05ee)
- **Whitelist is narrow and does not cover dummy shell tools:** The only retry-safe single-tool names are `read`, `search`, `find`, `grep`, `glob`, and `ls`. Tools such as `bash`/`exec` are not in that set, so a no-op shell call is still treated as real progress rather than planning-only.
