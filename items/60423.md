## Summary

Keep open. Current main still does not wrap the getReplyFromConfig media-understanding call in a try/catch, so an unexpected rejection from the media-understanding layer would still abort the normal reply pipeline instead of falling back to a text/raw-media reply.

## Evidence

- **Normal reply path still awaits media understanding without an error boundary:** getReplyFromConfig finalizes inbound context and then directly awaits applyMediaUnderstandingIfNeeded before link understanding and hooks; there is no surrounding try/catch in this path.
  - file: [src/auto-reply/reply/get-reply.ts:246](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/auto-reply/reply/get-reply.ts#L246)

- **Helper does not catch applyMediaUnderstanding failures:** applyMediaUnderstandingIfNeeded only checks for inbound media, imports the runtime, awaits applyMediaUnderstanding, and returns true. Any rejection from applyMediaUnderstanding propagates to getReplyFromConfig.
  - file: [src/auto-reply/reply/get-reply.ts:131](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/auto-reply/reply/get-reply.ts#L131)
