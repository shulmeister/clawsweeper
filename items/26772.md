## Summary

Keep #26772 open. Current main still uses the builtin memory engine by default and still performs full/forced builtin reindexes by building a fresh temp SQLite DB, writing metadata only after all memory/session sources finish, then swapping the temp DB into place. I found progress callbacks and an embedding cache, but no durable checkpoint cursor, resumable temp DB discovery, resume-on-restart config, or regression coverage for interrupted full-reindex resume. The related #70567 remains adjacent, not canonical, because it covers transient embedding failure dirty-state reporting rather than process-crash checkpoint/resume.

## Best Possible Solution

Keep #26772 open until memory-core has a durable full-reindex checkpoint/resume design. A good fix should persist generation/checkpoint state or safely reuse/discard incomplete temp DBs, resume from committed file/chunk progress after process restart, prevent search from serving incomplete generations as clean results, expose status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader reliability project intentionally covers both process-crash resume and transient embedding-failure dirty-state reporting.
