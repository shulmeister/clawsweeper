## Summary

Current main implements the timestamp-ordering fix for cron-spawned tasks. The cron path still passes a pre-captured startedAt into task creation, but task registry creation/update/restore now normalizes task lifecycle timestamps before audit-visible records are stored or exposed, and regression tests assert no inconsistent_timestamps finding for earlier startedAt values.

## Best Possible Solution

Close #65987 as implemented on main and keep the shipped task-registry normalization. Any remaining cleanup should focus on superseding older duplicate PRs/issues that proposed the same timestamp clamp.
