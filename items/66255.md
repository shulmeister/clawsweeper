## Summary

Keep open. Current main still has startup paths importing `@mariozechner/pi-coding-agent`, the pinned pi package still has optional native clipboard support, and upstream `pi-coding-agent@0.70.2` still evaluates clipboard-native loading when `DISPLAY` or `WAYLAND_DISPLAY` is present. The issue discussion also reports the crash on Fedora/KDE/Wayland and confirms the `TERMUX_VERSION=1` workaround, so this is still a real OpenClaw gateway dependency hardening item rather than stale headless-only noise.

## Best Possible Solution

Keep the issue open. The best fix is an OpenClaw-owned hardening path: either consume an upstream pi release with an explicit clipboard disable flag and set it before any pi import in gateway/service startup, or change OpenClaw/pi integration so gateway startup imports only session APIs without evaluating clipboard utilities. Add a regression test around importing the gateway/pi session path with `WAYLAND_DISPLAY` or `DISPLAY` set and a disabled-clipboard mode that proves `@mariozechner/clipboard` is not required, then document the supported service workaround instead of relying on `TERMUX_VERSION=1`.
