## Summary

Keep #39248 open. The report has concrete reproduction details and current main still does not contain a source, test, or live-verification proof that native `sessions_spawn` children initialize and make progress under `agents.defaults.sandbox.mode: "non-main"`; the accepted response still happens before asynchronous execution, and the child run still enters the sandbox-resolution path.

## Best Possible Solution

Keep #39248 open until maintainers add a targeted regression proof or live verification for native `sessions_spawn` under `agents.defaults.sandbox.mode: "non-main"`. The best fix path is a focused integration or high-fidelity mocked test that verifies the child run starts, creates transcript/message state, sets `systemSent` and token state, and surfaces sandbox initialization or dispatch failures instead of silently timing out; if the repro still fails, patch the Docker-backed `resolveSandboxContext`/dispatch path.

## Evidence

- **Child sessions are non-main sandbox candidates:** `spawnSubagentDirect` creates children as `agent:<targetAgentId>:subagent:<uuid>` and immediately resolves requester and child sandbox runtime status before dispatch.
