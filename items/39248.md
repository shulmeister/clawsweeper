## Summary

Keep #39248 open. The issue has concrete reproduction details, no protected author/label blocker, and current main still does not prove that a Docker-backed native `sessions_spawn` child under `agents.defaults.sandbox.mode: "non-main"` actually initializes, writes transcript/message state, updates `systemSent`/tokens, or reports sandbox startup failures instead of timing out.

## Best Possible Solution

Keep #39248 open until maintainers add a targeted regression proof or live verification for native `sessions_spawn` under `agents.defaults.sandbox.mode: "non-main"`. The best fix path is a focused integration or high-fidelity mocked test that verifies the child run starts, creates transcript/message state, sets `systemSent`/token state, and surfaces sandbox initialization/dispatch failures instead of silently timing out. If the repro still fails, patch the Docker-backed `resolveSandboxContext`/dispatch path; if it now passes live, close only with that source/test/manual evidence.
