## Summary

Keep PR #19328 open. Current main still has the stale-writeback pattern described by the PR: the agent handler reads a cache-backed session entry, copies fields such as modelOverride/providerOverride/sendPolicy/skillsSnapshot/CLI bindings from that possibly stale entry into nextEntryPatch, and then merges that patch over a fresh store read inside updateSessionStore. Newer subagent runtime-model persistence may mask part of the original first-run model symptom, but it does not remove this stale-cache overwrite class or preserve patch-only fields.

## Best Possible Solution

Continue tracking PR #19328 or land an equivalent focused fix that stops putting cache-derived session fields into nextEntryPatch before merging over the fresh store. The safest implementation is to include only request-derived fields in the patch, preserve fresh store-only fields through mergeSessionEntry, and add a regression test that simulates stale loadSessionEntry data versus a fresh updateSessionStore store for modelOverride/providerOverride and the broader session-field class.
