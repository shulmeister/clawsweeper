## Summary

Keep PR #19328 open. Current main at 04e08cea62983b2c766d15f61f15b3ffc90f9baa still has the stale cached-entry writeback pattern the PR targets: the agent handler copies modelOverride/providerOverride and other cache-derived fields into nextEntryPatch, then merges that patch over a fresh store read, so a concurrent sessions.patch write can still be clobbered.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop putting cache-derived session fields into nextEntryPatch before merging over the fresh store, preserve fresh store-only fields through mergeSessionEntry, and add regression coverage that simulates stale loadSessionEntry data against a fresh updateSessionStore store for modelOverride/providerOverride and the broader patch-field class.

## Evidence

Because loadSessionEntry can use the cache, these patch values can be stale.
