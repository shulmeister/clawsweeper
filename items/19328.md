## Summary

Keep PR #19328 open. Current main still builds the agent session store patch from the cached loadSessionEntry result, including modelOverride/providerOverride and other store-owned fields, then merges that stale patch over the fresh store row inside updateSessionStore. That matches the PR's reported stale-cache race, and current tests do not appear to contain the proposed #5369 regression coverage.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix in src/gateway/server-methods/agent.ts: stop including cache-derived store-owned fields such as modelOverride, providerOverride, policy/thinking/skill fields, spawn metadata, and CLI bindings in nextEntryPatch, or construct those values from the fresh store row inside updateSessionStore. Add a regression in src/gateway/server-methods/agent.test.ts that simulates stale loadSessionEntry data racing with a fresh store row containing modelOverride. Leave the narrower lastChannel/channel residual to #70575.
