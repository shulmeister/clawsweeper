## Summary

Keep PR #19328 open. Current main still builds the agent session patch from a potentially cached `loadSessionEntry()` result and then merges that patch over a fresh `updateSessionStore()` read, so stale `modelOverride`/`providerOverride` values can still overwrite fresh values. The latest provided maintainer/Codex comment also explicitly kept this PR open for the same code-backed reason.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop copying cache-derived store-only fields such as `modelOverride`, `providerOverride`, send policy, thinking/session metadata, and CLI bindings into `nextEntryPatch` before merging over the fresh store row, and to add regression coverage that simulates stale `loadSessionEntry` data against fresh `updateSessionStore` data. Keep the narrower lastChannel/channel residual work tracked in #70575.
