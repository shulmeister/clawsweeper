## Summary

Keep open. Current main has partial inline media support in Control UI chat, including assistant `MEDIA:` image/audio/video attachments and managed/generated image blocks, but the full request is not implemented. The useful issue discussion correctly identifies this as multi-layer work: web/link previews are still absent, remote http(s) image loading is blocked by the Control UI CSP, generated audio persists as `{ type: "audio", source: ... }` blocks that the UI normalizer does not convert into renderable attachments, and related tool-result/remote-image gaps remain tracked separately in #50779/#52025 and #41025. The closed #58543-style attachment work is useful progress, not enough to close this broader issue.

## Best Possible Solution

Keep this issue open as the umbrella unless maintainers create narrower canonical follow-ups. Preserve the shipped managed-image and `MEDIA:` attachment rendering work, then explicitly track the remaining core UI work: server-side link preview/unfurling with a CSP/privacy model, normalization/rendering for persisted audio/video content blocks, direct tool-result image block rendering aligned with #50779/#52025, and a deliberate remote-image policy aligned with #41025.
