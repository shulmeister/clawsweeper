## Summary

Do not auto-close #70451 yet. Current main appears to have addressed part of the referenced #64306 class by making CLI plugin-registry preload opt-in and keeping status/health/gateway status off plugin loading, but the exact affected `hooks enable/list/check/info` path still builds a module-aware plugin diagnostics report during command execution. That path can still broadly load plugins, so there is not enough source-backed evidence that the Windows timeout/SIGKILL report is fixed for the hook commands in this issue.

## Best Possible Solution

Keep the issue open and narrow the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` use cold hook/plugin metadata by default, or load only the selected hook's owning package, instead of calling `buildPluginDiagnosticsReport()` and broad `loadOpenClawPlugins()` during hook management. Add a regression test that `hooks enable <standalone-hook>` and `hooks list --json` do not call the CLI plugin registry preload or broad module-loading diagnostics, and document the config-only fallback under `hooks.internal.entries.<id>.enabled` if that is supported for the reporter's case.
