## Summary

Current main still does not show that the Windows hooks CLI timeout is fixed. The broad startup preload path is now conservative by default, but the hooks CLI still builds every list/info/check/toggle report through plugin diagnostics, and that diagnostics report explicitly loads plugin modules. The issue should stay open and remain scoped to the hooks CLI path rather than the already-closed broader #64306 startup class.

## Best Possible Solution

Keep this issue open and fix the hooks CLI path directly. The likely maintainer direction is to make `openclaw hooks list/info/check/enable/disable` derive plugin-managed hook visibility from manifest/config/workspace snapshot data, or add a bounded hook-specific plugin metadata path, with regression coverage proving these commands do not import full plugin runtime modules while still showing plugin-managed hooks as promised in the docs.
