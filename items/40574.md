## Summary

Keep PR #40574 open. Current main still does not implement a general `append: true` mode for the model-facing `write` tool; it only has a narrow append-only wrapper for memory-triggered flush runs. The PR is still a relevant candidate for the linked #40001 data-loss problem, though the review thread shows it needs maintainer review and hardening before merge.

## Best Possible Solution

Keep the PR open for maintainer review and rework it against current main. The best implementation is a backwards-compatible `write` append mode or equivalent append primitive that advertises `append` in `tool.parameters`, validates canonical `path` and `content`, preserves existing host/workspace path semantics, uses safe append handles for workspace-contained writes, gives explicit sandbox behavior, returns structured tool errors, and adds targeted host/sandbox/cron regression tests. Track the user-facing need with open issue #40001.
