## Summary

Keep open. Current `main` still starts iMessage monitors from the raw configured account-id list, so the duplicate-monitor bug is not implemented on main. The proposed PR is also not ready to close conservatively: its filtering change lives only in `extensions/imessage/src/accounts.ts`, while startup still uses `plugin.config.listAccountIds(cfg)`, and the review thread identified two credible regressions around disabled named accounts and case-insensitive `default` keys.

## Evidence

- **Current main has no iMessage fallback-monitor dedupe:** `listEnabledIMessageAccounts()` on `main` is just `listIMessageAccountIds(cfg).map(...).filter(enabled)` with no special handling for `default`, so the duplicate-monitor behavior described in the PR is not already fixed on `main`.

That means a fix limited to `listEnabledIMessageAccounts()` would not change the actual monitor-start path.
