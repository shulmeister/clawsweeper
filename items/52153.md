## Summary

Keep open. Current main does not implement a fix for proactive WeChat sends from `BOOT.md`. OpenClaw core still treats WeChat as an external plugin and explicitly assigns context-token/runtime behavior to that plugin, while the `boot-md` path only runs a generic agent turn and does not create WeChat session context. The currently published `@tencent-weixin/openclaw-weixin` package still documents `context_token` as required for `sendMessage`, and no `dmPolicy`/proactive-send support was found in the inspected surfaces.

## Evidence

- **WeChat remains external-plugin-owned on main:** Docs say the WeChat code does not live in core and that WeChat login, Tencent iLink API calls, media upload/download, context tokens, and account monitoring are owned by the external plugin.
  - file: [docs/channels/wechat.md:27](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/docs/channels/wechat.md#L27)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **BOOT hook is generic startup execution only:** Bundled hook docs say `boot-md` runs `BOOT.md` when the gateway starts; it does not describe any channel-specific session bootstrap or proactive-send workaround.
  - file: [src/hooks/bundled/README.md:49](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/hooks/bundled/README.md#L49)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **BOOT runtime does not synthesize channel context:** `runBootOnce` builds a generic prompt telling the agent to use the shared message tool, then calls `agentCommand` with `deliver: false`. No WeChat-specific session/context-token creation path appears here.
  - file: [src/gateway/boot.ts:43](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/boot.ts#L43)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Core/plugin boundary keeps outbound execution in the plugin:** Plugin architecture docs say core owns one shared `message` tool, while channel plugins own channel-specific discovery and final action execution.
  - file: [docs/plugins/architecture.md:178](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/docs/plugins/architecture.md#L178)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Published WeChat plugin still requires context_token:** The current published `@tencent-weixin/openclaw-weixin` README (version 2.1.9) documents `sendMessage` with required `context_token`, and the inspected README showed no `dmPolicy` or proactive-send surface.
  - command: `npm view @tencent-weixin/openclaw-weixin version readme --json`

## Risks / Open Questions

- The external WeChat plugin source is not vendored in this checkout, so runtime verification is limited to OpenClaw main docs/boundaries and the published package metadata/README.
- A fix could exist in unpublished plugin code or another repository, but there is no strong evidence of that in current main or the published package surface.
