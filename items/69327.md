## Summary

Keep #69327 open. The reported root-cause comment still matches current main: subagent spawn only checks whether the child runtime is sandboxed, then creates an independent child `agent:<id>:subagent:<uuid>` run. The spawned context carries workspace/group metadata but no parent Docker env or resolved sandbox context, and sandbox resolution still rebuilds Docker config from global/target-agent config only. Current main has general Docker env/setupCommand/recreate support, but I did not find source or tests proving parent sandbox.docker.env/bootstrap state is inherited across `sessions_spawn(..., sandbox="inherit")`.

## Best Possible Solution

Keep the issue open and fix it in core with a regression test around `sessions_spawn` from a sandboxed parent. The likely maintainer path is to define `sandbox="inherit"` precisely, then either thread the parent's resolved Docker env/config into child sandbox resolution or document and test target-agent-only semantics. Also add coverage for recreate/setupCommand applying to the child runtime scope that `sessions_spawn` actually uses.
