## Summary

Current main already separates filesystem boundary enforcement from `sandbox.mode` and defaults `tools.fs.workspaceOnly` to `false`, so host `write`/`edit` no longer hit `sandbox root` boundary checks when that FS guard is unset. The released tag `v2026.4.22` already contains this behavior and regression coverage for writes outside the workspace.

## Evidence

## Risks / Open Questions

- The reporter's expected model ('sandbox.mode=off disables every FS guard') is not how current main is designed; `tools.fs.workspaceOnly=true` will still intentionally block out-of-workspace and escaping-symlink paths.

## Close Comment

Closing as implemented on main. This Codex review checked current `main` and the released `v2026.4.22` code, and the filesystem boundary behavior has already moved away from the older `sandbox root` rejection in the unrestricted host case. `tools.fs.workspaceOnly` is now the separate switch for workspace containment and is documented as defaulting to `false` (`src/config/schema.help.ts:703`). With that flag unset/false, `write` and `edit` use direct host `fs` operations instead of the boundary-path guard (`src/agents/pi-tools.ts:426`, `src/agents/pi-tools.read.ts:757`), and regression coverage explicitly allows writes/edits outside the workspace in that mode (`src/agents/pi-tools.workspace-only-false.test.ts:82`). The released tag `v2026.4.22` already contains that implementation (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this specific bug is no longer current on supported builds. If you still see the same failure on a newer version, please open a fresh issue with the exact `tools.fs.workspaceOnly` setting and a minimal symlink repro.
