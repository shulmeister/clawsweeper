## Summary

Keep #30828 open. Current main has reactive rate-limit handling, including auth-profile cooldowns, model-scoped cooldown metadata, Retry-After bypass, same-provider rotation caps, fallback models, provider usage/status snapshots, and a Codex WHAM usage probe. It still does not implement the requested manually configurable per-model tokens-per-interval or requests-per-interval limiter, and the OpenAI x-ratelimit header idea from the comment remains unimplemented as request scheduling.

## Best Possible Solution

Keep the issue open for maintainer product/design work. A complete fix likely belongs in the model/provider request path plus config schema: define per-provider or per-model request and token budgets with interval windows, document the config, add scheduler/failover tests, and separately decide whether provider headers such as OpenAI x-ratelimit remaining/reset should auto-adjust those budgets.
