## Summary

Keep #30828 open. Current main has reactive rate-limit handling, including auth-profile cooldowns, model-scoped cooldown metadata, same-provider rotation caps, model fallback, and Retry-After handling. It still does not implement the requested proactive, manually configurable per-model tokens-per-interval or requests-per-interval limiter, and the OpenAI x-ratelimit request/token header idea from the comment remains unimplemented.

## Best Possible Solution

Keep the issue open for maintainer product/design work. A complete fix likely belongs in the model/provider request path and config schema: define per-provider or per-model request and token budgets with interval windows, document the config, add scheduler tests, and separately decide whether OpenAI x-ratelimit remaining/reset headers should auto-adjust those budgets.
