## Summary

Current main still has the same credible ACP bridge failure path: the default `openclaw acp client` session becomes an ACP-shaped Gateway session key without persisted ACP metadata, and ACP reply dispatch treats that as `ACP_SESSION_INIT_FAILED` instead of falling through to normal assistant generation.

## Best Possible Solution

Keep #38907 open until the ACP bridge/runtime boundary is fixed and covered. The likely fix is to either stop default Gateway-backed bridge sessions from using ACP-shaped `acp:<uuid>` keys that the ACP runtime claims, or persist valid `SessionAcpMeta` for those sessions before first prompt. Add a focused regression test for default `openclaw acp client` / `AcpGatewayAgent.newSession` first prompt producing `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
