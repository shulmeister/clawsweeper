## Summary

Current main still appears to contain the reported ACP bridge failure path. The default `openclaw acp client` creates an ACP bridge session with no `_meta` session key/label and no ACP runtime metadata, the bridge falls back to an `acp:<uuid>` Gateway key, Gateway canonicalizes that to an agent-scoped ACP key, and ACP reply dispatch treats ACP-shaped keys without `SessionEntry.acp` as `ACP_SESSION_INIT_FAILED` before normal assistant generation can proceed.

## Best Possible Solution

Keep #38907 open until the ACP bridge/runtime boundary is fixed and covered. The likely fix is to either stop default Gateway-backed bridge sessions from using ACP-shaped `acp:<uuid>` keys that the ACP runtime claims, or persist valid `SessionAcpMeta` for those sessions before the first prompt. Add a focused regression test for default `openclaw acp client` / `AcpGatewayAgent.newSession` first prompt behavior proving assistant `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
