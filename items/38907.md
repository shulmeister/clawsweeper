## Summary

Keep open. Current main still has a credible source-level path matching the reported `openclaw acp client` failure: the debug client creates a default ACP session with no explicit Gateway session key, the bridge maps that to an `acp:<uuid>` fallback, Gateway canonicalizes that bare key into an ACP-looking agent session key, and the ACPX reply_dispatch path still treats ACP-shaped sessions without stored ACP metadata as `ACP_SESSION_INIT_FAILED` instead of proving normal assistant chunk streaming.

## Best Possible Solution

Keep the issue open and add a focused fix at the ACP bridge versus ACPX boundary. A good fix would make default bridge-created sessions avoid ACPX runtime classification, or explicitly mark bridge-originated `chat.send` turns so `reply_dispatch` declines them unless they are real ACPX-managed sessions. Add regression coverage for the default `openclaw acp client` newSession -> prompt path proving assistant chunks are emitted and no `ACP_SESSION_INIT_FAILED` is recorded.
