## Summary

Keep #38907 open. Current main still matches the reported default ACP debug-client failure path: `openclaw acp client` creates a bridge session without ACP session metadata, the bridge falls back to an `acp:<uuid>` key, Gateway canonicalizes that key into an agent-scoped ACP-looking session, and the ACP manager treats ACP-shaped sessions without `SessionAcpMeta` as stale `ACP_SESSION_INIT_FAILED` instead of running the normal Gateway-backed assistant path.

## Best Possible Solution

Keep the issue open until the ACP bridge/runtime boundary is fixed and covered. The likely fix is to stop default Gateway-backed `openclaw acp client` sessions from using ACP-shaped fallback keys that the ACP runtime claims, or persist valid `SessionAcpMeta` before the first prompt. Add focused regression coverage for the default `createAcpClient` / `AcpGatewayAgent.newSession` first-prompt path proving `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`. The related closed ACP reports cover thread-bound or runtime-recovery paths, not this default debug-client bridge path.
