## Summary

Keep open. Current main still has a credible unresolved failure path matching #38907: the ACP debug bridge creates isolated `acp:<uuid>` Gateway session keys, `chat.send` canonicalizes them into `agent:<default>:acp:<uuid>`, and the ACPX reply_dispatch path treats ACP-shaped sessions with no stored `acp` metadata as `ACP_SESSION_INIT_FAILED` rather than falling through to normal Gateway model dispatch. The issue discussion already confirmed the same first-prompt failure mode, and I did not find a shipped main-branch change or regression test proving `openclaw acp client` now streams assistant chunks for default bridge sessions.

## Best Possible Solution

Keep #38907 open until the ACP bridge and ACP runtime dispatch boundary is fixed and covered. A likely fix is to prevent Gateway-backed bridge sessions that lack `SessionAcpMeta` from being classified as ACPX runtime sessions, either by using a non-ACPX default bridge session key or by making reply_dispatch decline ACP-looking bridge sessions without stored ACP metadata. If the intended design is for bridge-created sessions to be real ACP runtime sessions, then `openclaw acp` should persist valid `entry.acp` metadata during session creation. Add a focused regression test for default `openclaw acp client` / `AcpGatewayAgent.newSession` first prompt producing assistant chunks instead of `ACP_SESSION_INIT_FAILED`.
