## Summary

Current `main` does not match the reported initialization-timing bug. Plugin `register()` runs before registry activation, typed hooks are stored on the live registry, `hasHooks()` checks that live registry, and `agent_end` is still wired into the agent harness. The issue also omitted the current opt-in required for non-bundled conversation hooks like `agent_end`.

## Evidence

- **Plugin registration completes before hook runner activation:** `runPluginRegisterSync(register, api)` executes during plugin loading, and only after loading finishes does `loadOpenClawPlugins()` call `activatePluginRegistry(...)`, which then calls `initializeGlobalHookRunner(registry)`. That contradicts the report's claim that the runner snapshots an empty registry before `api.on()` registrations.

## Close Comment

Closing from a Codex review because I could not support the reported bug on current `main`. The current loader runs plugin `register()` before activation (`src/plugins/loader.ts:2733`) and only initializes the global hook runner afterward (`src/plugins/loader.ts:2828`, `src/plugins/loader.ts:1820`). The hook runner also checks the live `registry.typedHooks` array rather than an init-time snapshot (`src/plugins/hooks.ts:1101`), and `agent_end` is still dispatched from the agent harness (`src/agents/harness/lifecycle-hook-helpers.ts:38`). One important detail: for non-bundled plugins, conversation hooks like `agent_end` now require `plugins.entries.<id>.hooks.allowConversationAccess=true`; that behavior is enforced in `src/plugins/registry.ts:1247` and covered by `src/plugins/loader.test.ts:5360` and `src/plugins/loader.test.ts:5388`. `before_agent_start` also remains intentionally supported as a legacy compatibility hook (`docs/plugins/architecture.md:95`). Based on current `main`, this issue no longer reproduces as filed.
