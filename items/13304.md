## Summary

Keep #13304 open. Current main still only supports static binding-route fields, strict config/schema validation rejects the proposed `bindings[].match.content` object, and route resolution cannot inspect or strip inbound message text before choosing an agent. The related #10969/#11220 reports are model-routing context, not a canonical replacement for this agent-routing request, and the existing maintainer-side review comment explicitly kept #13304 open.

## Best Possible Solution

Keep #13304 open. The best path is a core binding/pre-routing design that can choose `agentId` from inbound message content and optionally transform the `BodyForAgent` before dispatch. A concrete change should update `AgentBindingMatch`, strict Zod validation, generated schema/help/docs, route resolver inputs/cache keys/precedence, representative channel call sites, and tests. Treat #10969 and #11220 as related model-routing context only, not superseding trackers.

## Risks / Open Questions

- A future implementation needs deterministic precedence for content matchers relative to existing peer, parent, guild, role, team, account, and channel tiers.
- Resolver caching must include content-dependent inputs or avoid static cache hits when bindings can vary by message text.
- Regex and prefix stripping need validation, safety limits, docs, and representative channel regression coverage before shipping.
