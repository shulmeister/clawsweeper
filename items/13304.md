## Summary

Keep #13304 open. Current main still does not implement content-based agent routing in bindings: config types/schema/docs only expose static channel/account/peer/guild/team/roles match fields, route resolution has no message text input or transformed-body output, and existing hooks can short-circuit or override models but cannot declaratively choose a normal target agent and strip a prefix before dispatch. The existing maintainer-side Codex comment also explicitly kept this open, and the linked #10969/#11220 are model-routing items rather than canonical trackers for agent routing.

## Best Possible Solution

Keep #13304 open. Implement either bindings[].match.content with deterministic prefix/regex matching plus stripMatch behavior, or add a documented pre-routing plugin hook that can return agentId, session behavior, and transformed BodyForAgent before normal dispatch. Update config types, strict zod schema, generated config help/docs, resolver cache keys and precedence, representative channel call sites, and route/channel regression tests.

## Risks / Open Questions

- First-class regex/prefix matching would need deterministic precedence, safe regex validation, and clear interaction with existing peer/guild/team/account tiers.
- A hook-based alternative would need a new pre-routing contract that can return agentId, session behavior, and transformed body text; existing hooks only observe, short-circuit, or override model selection.
