## Summary

Keep open. Current main still has `doctor.memory.status` await a live embedding probe with no cache or server-side short budget, and the built-in memory manager still implements that probe by embedding `["ping"]` through the batch embedding path that can use 120s remote or 600s local timeouts. Related memory probe/status issues are adjacent but do not fully supersede this latency/control-plane report.

## Best Possible Solution

Keep #71568 open and fix the gateway memory-status contract: return fast status from manager state plus cached embedding readiness with `lastProbeAt` / `lastProbeOk`, put the expensive live embedding probe behind a deep/explicit path or a short server-side timeout, and add regression tests proving repeated `doctor.memory.status` calls do not wait on the full embedding batch timeout. The adjacent false-warning/status-flapping issues can inform the same design, but this issue should remain as the latency/control-plane tracker.
