## Summary

Keep open. Current main still has `doctor.memory.status` await a live embedding readiness probe, and the built-in memory manager still implements that probe by running `embedBatchWithRetry(["ping"])` through the normal batch timeout path. The provided GitHub context also shows open PR #72443 uses closing syntax for this issue and is unmerged, so this issue should remain open until that implementation candidate is reviewed and merged or rejected.

## Best Possible Solution

Keep this issue open and review PR #72443 as the current implementation candidate. The likely fix belongs in the memory-core readiness probe or the doctor/status gateway path: default status should return cached or strictly bounded embedding readiness, while any long live embedding probe should be behind an explicit deep check, with focused regression tests covering cache hits, TTL expiry, probe failures, and the doctor/status response path.
