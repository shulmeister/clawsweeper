## Summary

Current main already solves the reported status-card context-window problem through the newer runtime-aware status path, while preserving the intentional empty-provider sentinel that the PR review flagged. The PR's exact normalization change is no longer the best patch to merge.

## Best Possible Solution

Close this PR without merging its exact patch. Keep the current runtime-aware status implementation and existing tests; if maintainers want extra coverage, add a narrower regression that proves `providerOverride: ""` plus a known `modelProvider` still renders the provider-specific context window without changing the provider-less sentinel behavior.

## Evidence

- **Current status tool only injects empty providerOverride for provider-less runtime models:** `session_status` computes `hasExplicitModelOverride`, reads the runtime provider/model, and only creates `{ ...resolved.entry, providerOverride: "" }` when there is a runtime model but no runtime provider. A known runtime provider such as `openai-codex` is passed through instead of being converted to an empty override.
