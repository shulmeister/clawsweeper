## Summary

Current `main` already fixes the reported listed-but-not-found cron lookup mismatch by canonicalizing legacy persisted `jobId` values into `id` before the shared cron store is used by `cron.list`, `cron.update`, and `cron.remove`. Regression coverage and the changelog both confirm this is already shipped.

## Evidence

- **Legacy cron IDs are normalized on load:** `ensureLoaded()` now calls `normalizeCronJobIdentityFields()` for each persisted job, copying legacy `jobId` into canonical `id` and removing the alias before the job enters the in-memory store.
  - file: [src/cron/service/store.ts:39](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/store.ts#L39)
  - command: `nl -ba src/cron/service/store.ts | sed -n '19,80p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Normalization logic explicitly promotes `jobId` to `id`:** `normalizeCronJobIdentityFields()` sets `raw.id` from `raw.jobId` when `id` is missing, which directly addresses the class of `unknown cron job id` failures described in the issue.
  - file: [src/cron/normalize-job-identity.ts:3](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/normalize-job-identity.ts#L3)

- **List, update, and remove now operate on the same canonical store:** `list()` reads `state.store.jobs`, `update()` resolves the job from that same store via `findJobOrThrow()`, and `remove()` deletes from that same `state.store.jobs` array, so current main does not have separate lookup sources for these operations.
  - file: [src/cron/service/ops.ts:160](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/ops.ts#L160)

  - file: [src/cron/service/store.test.ts:101](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/store.test.ts#L101)
