## Summary

Keep #65437 open. Current main still drops Slack room/group `/elevated` text commands at the inbound control-command gate before `tools.elevated.allowFrom.slack` can be evaluated, and the docs still do not describe or implement a Slack fallback. The related Discord/tiered-auth reports do not supersede this Slack-specific bug.

## Best Possible Solution

Keep this open and fix the Slack room/group `/elevated` path narrowly. The likely product-safe fix is to let Slack evaluate elevated-specific authorization for `/elevated`-bearing messages before the generic control-command early drop, or add an elevated-only authorizer to the inbound command gate, without broadening unrelated control commands. Add regression coverage for a Slack channel/group sender listed only in `tools.elevated.allowFrom.slack`, with no `channels.slack.allowFrom` and no per-channel `users`, then update the public elevated and Slack docs for the resulting behavior.
