## Summary

Keep open. Current `main` still persists webchat audio replies as inline base64 `audio` blocks and the embedded-runner char estimator still falls unknown/audio blocks through `estimateUnknownChars()`, so the reported token inflation path remains present. The related PR #65688 proposed a fix, but GitHub shows it was closed unmerged, so there is no evidence this bug is implemented on `main`.

## Evidence

- **Current estimator still serializes non-text/non-image blocks:** `estimateContentBlockChars()` only special-cases text and image blocks, then falls through to `estimateUnknownChars(block)`, which uses `JSON.stringify(value).length`. An `audio` block would still be counted by serialized blob size here.
