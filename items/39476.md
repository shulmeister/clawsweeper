## Summary

Keep #39476 open. Current main still exposes requester session identity to A2A target/reply/announce prompts, still runs the normal bounded ping-pong plus announce lifecycle for peer sessions, and still has no per-run guard that blocks `sessions_send` during those A2A turns. The recent issue comments report continued reproduction, and the related closed PR attempts were not merged; #44309 is adjacent dispatch-mode work, not an equivalent duplicate-delivery guard.

## Best Possible Solution

Keep the issue open and fix the core `sessions_send` A2A path. The likely maintainer path is a framework-level per-run guard that prevents A2A target/reply/announce turns from calling `sessions_send` back to the requester, plus narrowing or removing requester-session prompt exposure where feasible. Add regression coverage for duplicate visible delivery and keep #44309 separate as the broader dispatch-only/handoff semantics request.
