## Summary

Keep open. Current main still matches the reported failure shape: `sessions_send` injects the requester session key into the target A2A context, then the normal peer A2A flow runs reply-back turns and an announce step without a code-level guard that disables `sessions_send` for the target run. The related fix PRs were closed unmerged, #44309 is a broader dispatch-mode request rather than a complete duplicate, and the issue discussion includes a fresh April 24, 2026 report that this still reproduced on a new install.

## Best Possible Solution

Keep this issue open and fix it in the A2A sessions_send path. The likely maintainer path is a code-level guard that prevents A2A target runs from calling `sessions_send` back to the requester, plus removal or narrowing of requester-session prompt exposure where possible, with regression coverage for duplicate visible delivery. Keep #44309 as the broader dispatch-only feature request; it does not fully supersede this reverse-call bug.
