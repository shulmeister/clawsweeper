## Summary

Keep open. Current main has partially addressed the linked lock-timeout problem by introducing a structured timeout error and preventing session lock timeouts from being treated as model timeouts, but it has not implemented this PR’s main wiring change: the embedded attempt still does not pass the agent `--timeout` budget into `acquireSessionWriteLock`. The SDK export concern from the PR discussion also remains unresolved for plugin consumers.

## Best Possible Solution

Keep the PR open, but ask for a rebase/rework against current main: preserve the existing `SessionWriteLockTimeoutError` code-based contract, decide whether lock wait should use full `--timeout` or a bounded/session-specific wait budget, add a regression test proving `runEmbeddedAttempt` forwards the intended lock timeout, and export the lock timeout error or predicate through `openclaw/plugin-sdk/agent-harness` if plugin consumers are expected to branch on it. Broader lock-scope/configuration work should continue under #13744.

## Risks / Open Questions

- The PR is based on an older proposed `SessionWriteLockError` shape; a current-main version should use or deliberately evolve `SessionWriteLockTimeoutError` instead of adding a parallel error type.
- Passing the full run timeout into the lock wait may extend total CLI wall-clock time before the actual attempt timeout starts, and can interact poorly with model fallback if each fallback attempt waits on the same session lock.
- The SDK export should be treated as a public surface decision because `plugin-sdk/agent-harness` is an experimental trusted-plugin API, not just an internal barrel.
