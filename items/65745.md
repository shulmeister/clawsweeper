## Summary

Keep this issue open. Current main still builds the `gateway.nodes.denyCommands` audit vocabulary from `resolveNodeCommandAllowlist()` with denies cleared, while the dangerous command IDs named in the issue are deliberately excluded from default allowlists. The existing tests cover pattern-like and unknown deny entries, but do not cover valid dangerous deny entries being accepted without an ineffective warning.

## Best Possible Solution

Fix the audit vocabulary in `src/security/audit-extra.sync.ts` so valid dangerous node commands are recognized when evaluating `gateway.nodes.denyCommands`, preferably by deriving from a full node command catalog or at least unioning `DEFAULT_DANGEROUS_NODE_COMMANDS`. Add a regression case in `src/security/audit-node-command-findings.test.ts` proving that denying the default dangerous commands does not emit `gateway.nodes.deny_commands_ineffective`, while typo and pattern-like warnings still fire.
