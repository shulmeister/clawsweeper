## Summary

Keep open. Current `main` still uses persistent per-scope sandbox workspaces under `~/.openclaw/sandboxes`, reuses them across runs, and does not provide a first-class ephemeral/tmpfs-backed workspace mode for isolated sessions.

## Evidence

- **Sandbox workspace is a persistent per-scope directory:** `resolveSandboxContext` derives `sandboxWorkspaceDir` from `cfg.workspaceRoot` plus the scope key, then uses that directory whenever `workspaceAccess != "rw"`.
  - file: [src/agents/sandbox/context.ts:38](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/agents/sandbox/context.ts#L38)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '30,70p'`
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Workspace setup reuses existing contents instead of resetting each run:** `ensureSandboxWorkspace` only `mkdir`s the target and seeds bootstrap files when the destination file does not already exist; it does not clear the workspace before reuse.
  - file: [src/agents/sandbox/workspace.ts:22](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/agents/sandbox/workspace.ts#L22)
  - command: `nl -ba src/agents/sandbox/workspace.ts | sed -n '1,70p'`
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Docker sandbox removal deletes the container, not the workspace directory:** Docker `removeRuntime` runs `docker rm -f` only, and `removeSandboxContainer` just delegates to backend removal plus registry cleanup. There is no host-side sandbox workspace deletion in the Docker path.
  - file: [src/agents/sandbox/docker-backend.ts:143](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/src/agents/sandbox/docker-backend.ts#L143)
  - command: `nl -ba src/agents/sandbox/docker-backend.ts | sed -n '143,149p'; nl -ba src/agents/sandbox/manage.ts | sed -n '91,103p'`
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Docs describe persistent per-scope workspaces, not ephemeral ones:** The config reference documents `workspaceRoot: "~/.openclaw/sandboxes"`, `tmpfs: ["/tmp", "/var/tmp", "/run"]`, and `scope: "session"` as a per-session container plus workspace. No ephemeral/tmpfs-backed workspace option is documented.
  - file: [docs/gateway/configuration-reference.md:1538](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/gateway/configuration-reference.md#L1538)
  - command: `nl -ba docs/gateway/configuration-reference.md | sed -n '1538,1548p'; nl -ba docs/gateway/configuration-reference.md | sed -n '1661,1669p'`
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)

## Risks / Open Questions

- Users can get per-session isolation, but not guaranteed fresh-on-every-run ephemeral storage; workspace changes can persist for reused session keys.
- There may be external operator workarounds (unique session keys, external tmpfs mounts, manual cleanup), but there is no built-in core option matching the issue request.
