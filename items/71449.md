## Summary

Keep PR #71449 open. Current main does not implement the PR's central startup-clarity behavior: the gateway still logs plain `ready (...)`, there is no `sidecars settled` channel health summary, Telegram polling still blocks/retries webhook cleanup before polling without the proposed startup budget/background retry behavior, and Feishu only handles timeout/abort probe failures. The PR discussion is also active and useful: review comments were P2-only and the author followed up on the timeout and deterministic-summary concerns.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a cleaned version, or split it if maintainers prefer narrower risk, that preserves the legacy `ready (` marker for tooling while clarifying gateway-listening semantics, adds a deterministic sidecars-settled channel health summary using existing runtime snapshot and health policy data, bounds Telegram webhook cleanup without delaying usable gateway startup on recoverable network failures, and extends Feishu startup handling for transient TLS/socket failures with targeted tests. Do not close it as implemented or superseded because current main lacks the central behavior.
