## Summary

Keep this issue open. Current main has partial adjacent work, but the central doctor/status behavior is not implemented: doctor still gates daemon repair on a `health` RPC, still uses a 3000ms timeout in non-interactive mode, and the status probe can still mark a gateway unreachable after a successful WebSocket accept if the follow-up detail RPC times out. The comment pointing to #64511 identified the likely fix direction, but the provided PR context says that PR was closed unmerged, and the current source still does not contain its proposed doctor Phase A `status` RPC change.

## Best Possible Solution

Keep the issue open and implement the narrow core fix: make doctor Phase A use a lightweight liveness RPC such as `status` with an appropriate timeout before offering restart repair; adjust status probing so a successful local WebSocket accept or provably live listener is not reported as unreachable solely because a follow-up detail RPC timed out; add regression tests that a slow `health` RPC does not trigger `service.restart` and that status preserves reachable/degraded state for post-connect timeouts.
