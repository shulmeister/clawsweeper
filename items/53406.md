## Summary

Current main has substantial related ACP and sub-agent control work, but the specific request is still not implemented for ACP-bound chat threads: ordinary messages are routed as queued ACP prompt turns, not delivered as live input to the active ACP runtime turn.

## Best Possible Solution

Keep the issue open. The best fix is a real active-turn input path for bound chat sessions: add an ACP manager/runtime capability outside the per-session actor queue, expose it in `src/acp/runtime/types.ts`, have `src/auto-reply/reply/dispatch-acp.ts` use it for ordinary bound-thread replies when a turn is active, and add Discord/thread-bound regression coverage proving a second message reaches the active ACP turn before the first `runTurn` completes. For native sub-agent sessions, maintainers should make the product decision explicit: ordinary bound-thread replies either default to live steering or continue to respect `messages.queue`, with docs/tests matching that behavior.

## Evidence

- **Issue scope:** The issue asks for ordinary messages sent in a bound Discord/chat thread to reach a running ACP or sub-agent session while it is still working, not after the current turn finishes. The existing ClawSweeper comment also identified this exact remaining ACP gap after reviewing related ACP session-control work.
- **Bound ACP replies are routed through ACP dispatch before model fallback:** Current dispatch resolves a bound ACP session key, runs the `reply_dispatch` hook, and returns immediately when the hook handles the message.
