## Summary

Keep open. Current main has thread/conversation binding plus explicit `/acp steer` and `/subagents steer` controls, but it does not implement the requested behavior where ordinary chat messages sent into a bound Discord thread are delivered into an already-running ACP or sub-agent session mid-turn. The ACP control plane still serializes same-session turns, and sub-agent steering is implemented as abort/restart control rather than live in-thread input injection.

## Evidence

- **ACP docs separate bound follow-ups from explicit steer control:** The ACP docs say bound conversations route follow-up messages to the session, but list `/acp steer` as a separate control specifically for sending steer instructions to a running session.
  - file: [docs/tools/acp-agents.md:69](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/tools/acp-agents.md#L69)
  - command: `nl -ba docs/tools/acp-agents.md | sed -n '60,85p' && nl -ba docs/tools/acp-agents.md | sed -n '486,500p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **ACP manager queues a second turn until the first finishes:** The control-plane test starts one `runTurn`, submits a second `runTurn` for the same ACP session, confirms only one runtime turn starts before the first is released, then completes both. That matches queued serial turns, not mid-turn input delivery.
  - file: [src/acp/control-plane/manager.test.ts:410](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/acp/control-plane/manager.test.ts#L410)

- **ACP steer is a dedicated command path:** `/acp steer` calls `acpManager.runTurn` with `mode: "steer"`; this is an explicit command handler, not automatic routing of ordinary thread messages into a running session.
  - file: [src/auto-reply/reply/commands-acp/lifecycle.ts:740](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/commands-acp/lifecycle.ts#L740)

- **Sub-agent control is also command-based, and steer restarts the run:** Sub-agent docs expose `/subagents send` and `/subagents steer` as separate controls. In code, `steerControlledSubagentRun` aborts the current run, clears queues, waits for settle, and starts a replacement run; `sendControlledSubagentMessage` sends a message and waits for a later reply. Neither path is ordinary bound-thread message injection into a live run.
  - file: [src/agents/subagent-control.ts:443](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/subagent-control.ts#L443)

## Risks / Open Questions

- There is partial overlap with existing controls: maintainers could view `/acp steer` or `/subagents steer` as an acceptable workaround, but those are explicit commands rather than plain in-thread message steering.
- Some native Codex bound-chat controls exist elsewhere, but this issue explicitly targets ACP and sub-agent sessions; the current evidence for those paths still points to missing live chat-surface input.
