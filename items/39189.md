## Summary

Current main still does not implement either central fix from PR #39189. Anthropic "specified API usage limits" wording is still covered by the generic rate-limit matcher rather than a billing matcher, and final-reply dedupe is still gated on same-target suppression, which does not prove email and phone handles are aliases for iMessage or BlueBubbles users. The PR should stay open, but its current patch needs narrower rework because review comments correctly identify false-positive risks in both the broad billing matcher and unconditional text dedupe.

## Best Possible Solution

Keep PR #39189 open, but split or rework it into two narrower fixes: add a high-confidence Anthropic billing matcher and ordering/tests for the exact "specified API usage limits" plus "regain access" wording while preserving periodic usage-window rate-limit tests; and fix multi-handle duplicate suppression with identity-aware iMessage/BlueBubbles target matching or a channel-owned alias/suppression hook, plus positive email/phone same-person tests and negative cross-target same-text tests.

## Risks / Open Questions

- Anthropic billing-limit wording like "You have reached your specified API usage limits. You will regain access on ..." can still be treated as rate_limit on current main, causing cooldown/retry behavior instead of billing failover.
- The reported multi-handle duplicate final reply can still occur when the message tool sends to one identifier and the originating conversation uses another identifier for the same iMessage/BlueBubbles user.
- Merging the PR as written would risk suppressing legitimate same-text replies sent to a different target and could move transient usage-window limits into the longer billing-disable path.
