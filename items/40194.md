## Summary

Keep #40194 open. Current main has improved Feishu multi-account handling and passes `accountId` through monitor, routing, inbound context, and reply dispatch, but the core group/channel session-key builder still omits account/app identity for non-DM peers. For the reported case of multiple Feishu bots in the same group routed to the same agent, the session key can still collapse to `agent:<agent>:feishu:group:<chatId>` rather than being account/app scoped.

## Best Possible Solution

Keep the issue open and implement account-aware Feishu group session isolation. A conservative path is to add a Feishu-specific or generic opt-in account-scoped group session mode, use a stable normalized accountId rather than raw app secret data in the session key, preserve legacy session continuity where possible, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct `SessionKey` values and correct `AccountId`/reply account.

## Risks / Open Questions

- Account-scoped route bindings can work around some deployments by routing each Feishu account to a different agent, but they do not solve same-agent session isolation for multiple bot accounts in one group.
- Changing group session key shape globally would affect existing session continuity for other group-capable channels; a Feishu-specific or opt-in migration path may be safer.
