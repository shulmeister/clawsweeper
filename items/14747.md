## Summary

Keep #14747 open. Current main still does not expose a configurable lane wait diagnostic threshold: the queue fallback remains hardcoded, cron manual-run enqueue still passes a fixed warning threshold, and the config schema/types/labels/docs only expose unrelated diagnostics fields such as `stuckSessionWarnMs`. The related PR #16366 targeted this request but is closed unmerged and its head commit is not present in the local checkout.

## Best Possible Solution

Keep the issue open until a maintainer-approved implementation adds a configurable lane wait diagnostic threshold. A focused fix likely belongs in the config contract (`src/config/types.base.ts`, `src/config/zod-schema.ts`, generated schema/help/labels, Control UI metadata, and public docs), then should wire `src/process/command-queue.ts` so per-call `warnAfterMs` still wins while the default can come from config. The cron enqueue path should either inherit that default or expose an explicitly documented cron-specific threshold, with regression coverage for the configured default and the long-running cron queue scenario.
