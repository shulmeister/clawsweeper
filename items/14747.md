## Summary

Current main still does not implement a configurable lane wait diagnostic threshold. The queue default remains hardcoded at 2 seconds, the manual cron enqueue path still uses a fixed 5-second warning threshold, the diagnostics config type/schema/generated metadata/docs expose only existing diagnostics fields such as `stuckSessionWarnMs`, and the related implementation PR #16366 is closed unmerged in the provided GitHub context. The latest maintainer/Codex comment on the issue also explicitly kept it open for the same remaining gap.

## Best Possible Solution

Keep #14747 open. A focused fix should add a documented configurable lane wait warning threshold, likely `diagnostics.laneWaitWarnMs`, across exported config types, strict zod schema, generated schema/help/labels, public docs, reload planning if needed, and regression tests. Wire the queue so explicit per-call `warnAfterMs` still wins while the default comes from config, and make a maintainer decision on whether cronâ€™s fixed manual-run threshold should inherit that default or have a documented cron-specific override.
