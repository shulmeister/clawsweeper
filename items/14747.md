## Summary

Keep #14747 open. Current main still has only per-call lane wait thresholds, with a hardcoded 2,000 ms default in the command queue and a hardcoded 5,000 ms cron enqueue threshold. The diagnostics config contract, schema/help labels, and public configuration docs still do not expose `diagnostics.laneWaitWarnMs` or an equivalent setting. The related PR #16366 was closed unmerged, and its head commit is not present in this checkout.

## Best Possible Solution

Keep #14747 open until a focused maintainer-approved patch adds a documented configurable lane wait diagnostic threshold. A likely implementation is `diagnostics.laneWaitWarnMs` in the config types, strict zod schema, labels/help/generated metadata, and public docs, then wiring `src/process/command-queue.ts` so explicit per-call `warnAfterMs` still wins while the default comes from config. The cron enqueue path should inherit that configured default or expose a documented cron-specific threshold, with regression coverage for configured defaults and long-running cron queue waits.
