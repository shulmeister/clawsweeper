## Summary

Keep open. This is a recent Windows reply-latency regression report, and current main only shows adjacent mitigations rather than a source-backed fix for the broad slowdown the reporter described. Active Memory disabled state is now honored and the enabled path has a hard timeout, but Active Memory is still a blocking before-prompt hook when enabled, QMD/session-lock related latency has separate open trackers, and the issue lacks the trace needed to map it conclusively to a fixed or duplicate item.

## Best Possible Solution

Keep this issue open as a Windows reply-latency investigation. Ask the reporter to retest on the latest release/main and attach one slow-turn trace with Windows version, OpenClaw version, channel/UI, model/provider, Active Memory entry/config enabled values, /status output, and gateway logs showing hook timing, model-call timing, QMD/memory startup, and any session-write-lock warnings. If that trace matches a narrower open tracker such as #70857, #66792, #69182, or #72015, maintainers can then close or retitle this as a duplicate with high confidence; otherwise fix the implicated owner path directly.
