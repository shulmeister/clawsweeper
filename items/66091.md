## Summary

The issue reports broad Windows reply latency regression after 2026.4.12 and includes an Active Memory config snippet, but current evidence is not strong enough for an automatic close. Current main has Active Memory disable/timeout safeguards and later performance-related fixes, yet I did not find a source-backed fix that clearly resolves the reported overall AGENT response slowdown.

## Best Possible Solution

Keep the issue open as a performance investigation. Ask the reporter to retest on the latest release/main and attach a minimal latency trace: Windows version, OpenClaw version, channel/UI used, model/provider, whether `plugins.entries.active-memory.enabled` or `plugins.entries.active-memory.config.enabled` is set, `/status` output, and timing/debug logs for one slow turn. If current releases still reproduce, the likely maintainer path is a focused startup/turn-latency profile around plugin hooks, Active Memory gating, provider streaming, and Windows gateway/client transport rather than a broad feature request close.
