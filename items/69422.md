## Summary

Keep #69422 open. Current main still does not carry Google Chat thread identity as a first-class reply/follow-up field, and the relevant code paths still rely on `replyToId` or `MessageThreadId` in ways that do not preserve `message.thread.name` across streamed blocks and queued follow-ups. The later independent comments in the issue match current source, and the related closed #70041 is narrower than this remaining root-cause issue.

## Best Possible Solution

Keep #69422 open as the canonical root-cause bug. The likely fix belongs in the Google Chat plugin plus the generic reply/follow-up contracts: populate a Google Chat thread resource into `MessageThreadId` or a documented channel-specific reply field, make block coalescing/dedup thread-aware, ensure all chunks and queued follow-ups route with the same thread, add Google Chat regression tests for streamed chunks and queued follow-ups, and add `googlechat` to the queue by-channel schema if maintainers want the mitigation.
