## Summary

Keep open. Current main still matches the reported failure shape: Google Chat thread identity is not carried as a first-class outbound/follow-up field, block streaming still keys and coalesces only on replyToId, and Google Chat inbound only stores the thread in ReplyToId while queued followups read MessageThreadId. The comment-proposed mitigation for messages.queue.byChannel.googlechat is also still missing from the zod schema.

## Best Possible Solution

Keep #69422 open as the canonical Google Chat streamed/followup thread-routing bug. A good fix should preserve the Google Chat thread at ingress, carry it through the reply payload or MessageThreadId contract, make block coalescing thread-aware, route queued followups with that thread id, add googlechat to messages.queue.byChannel schema if the workaround is intended, and add regression coverage for streamed chunks plus concurrent queued followups.

## Risks / Open Questions

- A narrow Google Chat send adapter change could make long chunk sends look fixed while still leaving block coalescing and queued followups capable of losing thread routing.
- Changing replyToId semantics directly may affect other channels; the safer path is an explicit thread-routing contract or a deliberate Google Chat mapping into MessageThreadId with focused regression tests.
