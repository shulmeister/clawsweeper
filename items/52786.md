## Summary

Keep open. Current `main` still builds the agent session patch from a pre-lock cached `entry`, copies `modelOverride`/`providerOverride` from that stale object, then merges that patch into the fresh store inside `updateSessionStore`; because `mergeSessionEntry` uses object spread, `undefined` values from the stale patch overwrite freshly patched overrides. I did not find a regression test covering this stale-cache interleaving on `main`.

## Evidence

- **Agent handler still snapshots override fields before the locked store reread:** `loadSessionEntry(requestedSessionKey)` happens before the write lock, `nextEntryPatch` copies `modelOverride: entry?.modelOverride` and `providerOverride: entry?.providerOverride`, then `updateSessionStore(...)` merges that prebuilt patch later.
