## Summary

Current main bounds the reported Codex harness initialize/startup hang. The original report correctly identified that a missing initialize timeout made app-server bootstrap failures appear as an indefinite hang; current Codex run, model-discovery, shared-client, and isolated-client paths now wrap startup/initialize with timeouts, clear the shared client, and close the child so the failure surfaces as an explicit timeout instead of hanging forever. The reporter's #61741 comparison was useful adjacent lifecycle evidence, but the Codex-specific startup path now has its own guardrails and regression tests.

## Best Possible Solution

Close this as implemented on main. Keep the shipped timeout and teardown behavior, and use the broader #66251 tracker or a new focused report only if a current release still produces a concrete Codex timeout/error with fresh logs and config.
