## Summary

Keep open. Current `main` has partial cleanup for duplicate gateway listeners after restart, but the reported false-healthy inbound gap is not clearly fixed for Feishu: `gateway status` still documents that it only proves service/connect/auth capability, and Feishu channel status still probes a Feishu ping endpoint rather than end-to-end inbound delivery or agent dispatch.

## Evidence

- **Gateway status is still scope-limited by design:** The CLI docs explicitly say default `gateway status` only proves service state, WebSocket connect, and handshake-visible auth capability; it does not prove downstream operations.
  - file: [docs/cli/gateway.md:190](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/docs/cli/gateway.md#L190)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Feishu status path only wires in probeFeishu:** The Feishu channel status adapter defines `probeAccount`, but no Feishu-specific audit path or status-issue collector for stalled inbound delivery is present in this block.
  - file: [extensions/feishu/src/channel.ts:1182](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/channel.ts#L1182)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Feishu probe validates credentials, not message flow:** `probeFeishu()` calls `/open-apis/bot/v1/openclaw_bot/ping` and comments describe it as validating credentials / bot registration. Static inspection shows no end-to-end inbound-message or dispatch verification here.
  - file: [extensions/feishu/src/probe.ts:83](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/extensions/feishu/src/probe.ts#L83)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Inbound timestamps only surface when populated:** Gateway `channels.status` backfills `lastInboundAt` from generic channel activity, and the human formatter only prints `in:<age>` when that timestamp exists.
  - file: [src/gateway/server-methods/channels.ts:240](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/gateway/server-methods/channels.ts#L240)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Channels status output is conditional on lastInboundAt:** The formatter adds inbound/outbound age bits only when `lastInboundAt` / `lastOutboundAt` are numeric.
  - file: [src/commands/channels/status.ts:57](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/commands/channels/status.ts#L57)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Feishu does not record generic channel activity in current main:** Repository search for `recordChannelActivity(` under `extensions/feishu/src` returned no matches, so the generic `lastInboundAt` fallback currently has nothing Feishu-specific to display.
  - command: `rg -n "recordChannelActivity\(" extensions/feishu/src`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Restart hygiene is improved, but only for the duplicate-process side of the report:** Unmanaged restart now errors when multiple gateway listeners are present, and service-managed restart health can detect stale gateway PIDs, kill them, and retry.
  - file: [src/cli/daemon-cli/lifecycle.ts:130](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/cli/daemon-cli/lifecycle.ts#L130)
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
