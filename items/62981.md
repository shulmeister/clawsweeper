## Summary

Current main still has the reported timeout-fallback/session-lock failure shape. The CLI waits for the final Gateway `agent` response, times out client-side, then falls back to embedded execution with the same original session selectors. The Gateway run can remain active and hold the same transcript lock, while the embedded path resolves back to the same session file and can still fail with `session file locked`.

## Best Possible Solution

Keep this issue open as the focused tracker for the timeout-fallback/session-lock bug. Fix the CLI/Gateway fallback contract in `src/commands/agent-via-gateway.ts`: after a Gateway `agent` request has been accepted and final waiting times out, either continue waiting through `agent.wait`, explicitly abort the Gateway run and wait for lock release before retrying, or start embedded under a clearly fresh user-visible session. Add regression coverage in `src/commands/agent-via-gateway.test.ts` and update `docs/cli/agent.md` or `docs/tools/agent-send.md` if fallback semantics change. Coordinate with #71605 for the broader Gateway dispatch-timeout work rather than closing this focused report.
