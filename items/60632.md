## Summary

Current main still allows the reported inversion: task creation stamps `createdAt` from `Date.now()` while preserving a caller-supplied earlier `startedAt`, and cron task creation passes a pre-captured `startedAt`/`lastEventAt`. The audit code still flags `startedAt < createdAt`, so the user-visible bug remains until a fix such as the related open PRs lands.

## Evidence

- **Task registry still preserves earlier startedAt:** `createTaskRecord()` captures `now = Date.now()`, assigns `createdAt: now`, and assigns `startedAt: params.startedAt` without clamping or normalization.
  - file: [src/tasks/task-registry.ts:1440](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/tasks/task-registry.ts#L1440)
  - command: `nl -ba src/tasks/task-registry.ts | sed -n '1360,1495p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Cron creates task records with a pre-captured startedAt:** Cron captures `startedAt` before task record insertion, then passes that value as both `startedAt` and `lastEventAt` to `createRunningTaskRun()`, which can make it earlier than the registry-created `createdAt`.
  - file: [src/cron/service/timer.ts:135](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/cron/service/timer.ts#L135)

  - file: [src/cron/service/ops.ts:489](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/cron/service/ops.ts#L489)
