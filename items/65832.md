## Summary

Keep open. Current `main` still resolves image-tool models through `ModelRegistry.find(...)`, and the bundled `@mariozechner/pi-coding-agent` version still drops all custom-model loading when any non-built-in provider with custom models fails validation. OpenClaw has partial coverage for configured custom providers, but I did not find a regression proof for the reported image-tool execution path or the mixed-provider failure mode.

## Evidence

- **Image tool still throws on registry miss:** `resolveImageRuntime()` still prepares `models.json`, calls `discoverModels(...)`, then throws `Unknown model` if `modelRegistry.find(...)` returns null. The reported failure surface is still present on current `main`.
  - file: [src/media-understanding/image.ts:107](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/media-understanding/image.ts#L107)
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Current dependency still fails custom-model loading all-or-nothing:** OpenClaw still depends on `@mariozechner/pi-coding-agent` `0.70.2`. In that version, `loadCustomModels()` returns an empty custom-model result on any validation error, and `validateConfig()` still requires `apiKey` for any non-built-in provider that defines custom models. That means one invalid provider entry can still suppress all custom providers and provider overrides for the registry used by the image tool.
  - file: [package.json:1597](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/package.json#L1597)
  - command: `curl -L --max-time 20 https://unpkg.com/@mariozechner/pi-coding-agent@0.70.2/dist/core/model-registry.js | nl -ba | sed -n '260,390p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **OpenClaw does not generically backfill missing auth for every models.json provider with models:** `resolveMissingProviderApiKey()` only auto-fills missing `apiKey` from resolver/env/profile paths, with special handling for `aws-sdk`; it does not provide a general fallback for every provider that has `models` but no `apiKey`. So the validation-triggering class of failure is still plausible in current code depending on how a provider entry is produced.
  - file: [src/agents/models-config.providers.secret-helpers.ts:271](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/models-config.providers.secret-helpers.ts#L271)

- **OAuth markers are opt-in, not automatic:** OAuth-backed providers only get a synthetic `apiKey` marker when a caller explicitly passes `oauthMarker` into `createProviderAuthResolver()`. That helps some bundled provider catalogs, but it is not a blanket fix for every provider entry that can reach `models.json`.
  - file: [src/agents/models-config.providers.secrets.ts:90](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/models-config.providers.secrets.ts#L90)
