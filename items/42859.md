## Summary

Keep open. Current `main` still injects gateway client identity into `chat.send`, but the proposed unconditional removal would also erase the caller identity that current command/binding ownership checks depend on.

## Evidence

- **Current main still injects sender metadata:** `chat.send` on `main` still copies gateway `clientInfo.id` and `clientInfo.displayName` into `SenderId`, `SenderName`, and `SenderUsername` before dispatch.
  - file: [src/gateway/server-methods/chat.ts:2315](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/gateway/server-methods/chat.ts#L2315)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2290,2345p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Command auth propagates SenderId into command identity:** `resolveCommandAuthorization` derives `senderId` from `ctx.SenderId` and returns it as `command.senderId`; `operator.admin` scope only affects owner authorization, not the per-caller identity value.
  - file: [src/auto-reply/command-auth.ts:669](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/command-auth.ts#L669)
  - command: `nl -ba src/auto-reply/command-auth.ts | sed -n '640,735p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Lifecycle updates rely on a non-empty senderId for ownership enforcement:** `/session idle` and `/session max-age` only reject a different caller when `senderId` is present; removing `SenderId` from gateway `chat.send` would bypass this check for gateway callers.
  - file: [src/auto-reply/reply/commands-session.ts:575](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/commands-session.ts#L575)
  - command: `nl -ba src/auto-reply/reply/commands-session.ts | sed -n '520,620p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **Same senderId pattern protects focus/unfocus and ACP rebinding:** Current ownership checks for `/focus`, `/unfocus`, and ACP rebinding all gate on `senderId && senderId !== boundBy`, so an empty sender id weakens those protections too.
  - file: [src/auto-reply/reply/commands-subagents/action-unfocus.ts:30](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/commands-subagents/action-unfocus.ts#L30)
  - command: `nl -ba src/auto-reply/reply/commands-subagents/action-unfocus.ts | sed -n '1,60p'`
  - sha: [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2bb81ae49b8f55a93e41492ad31917)
- **The prompt-metadata concern is real, but separate from command identity:** Inbound prompt assembly still emits a `Sender (untrusted metadata)` block from `SenderId`/`SenderName`/`SenderUsername`, so the underlying regression likely needs a narrower fix than deleting sender identity from the dispatch context.
  - file: [src/auto-reply/reply/inbound-meta.ts:242](https://github.com/openclaw/openclaw/blob/564f820efa2bb81ae49b8f55a93e41492ad31917/src/auto-reply/reply/inbound-meta.ts#L242)
  - command: `nl -ba src/auto-reply/reply/inbound-meta.ts | sed -n '242,260p'`

## Risks / Open Questions

- The user-facing prompt-metadata regression appears plausible on `main`; keeping this PR open means the underlying issue may still need a narrower fix.
- A correct fix likely needs separate fields or prompt filtering so internal gateway callers stay identifiable for authorization/binding ownership without leaking control-ui identity into model-facing sender metadata.
