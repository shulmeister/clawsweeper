## Summary

Keep #42859 open. Current main still injects gateway client identity into `chat.send` sender metadata, and prompt assembly still renders those fields into `Sender (untrusted metadata)`, so the central bug is not implemented on main. The PR is not safe to close as solved or merge as-is: the review comment is backed by current code because sender identity is also used by command/lifecycle ownership checks. The right fix is to separate trusted gateway requester identity from prompt-visible sender metadata.

## Best Possible Solution

Keep the PR open for maintainer follow-up, but require an updated patch. The best fix is to stop exposing Control UI gateway identity as prompt-visible `Sender*` metadata while preserving a trusted internal requester identity for command authorization, lifecycle binding ownership, `/whoami`, and related gateway-originated command paths. A regression test should cover both sides: no `Sender (untrusted metadata)` block for Control UI `chat.send`, and no loss of requester identity for focused-session lifecycle/refocus authorization.
