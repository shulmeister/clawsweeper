## Summary

Keep #39889 open. Current main has related browser-bundle hardening for the `tmp-openclaw-dir`/`fs.constants.W_OK` crash, but that does not prove the reported Chromium/Vivaldi `openclaw-app` registration hang is fixed. The served Control UI bootstrap shape still relies on side-effect Lit decorator registration, and current browser tests do not load the real served `ui/index.html` path and assert root custom-element registration.

## Best Possible Solution

Keep #39889 open until maintainers add or run a served-Control-UI Chromium regression that loads the real `ui/index.html` or built bundle, captures module/CSP/resource/service-worker errors, asserts `customElements.get('openclaw-app')`, and verifies Chat renders after backend RPC success. If #48930/#48062 is believed to be the fix, link that evidence back to #39889 with a targeted regression or reporter confirmation.
