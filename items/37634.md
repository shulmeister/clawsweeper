## Summary

Keep #37634 open. Current main still makes `workspaceAccess: "none"` use an isolated sandbox workspace, but the Docker/browser mount layer, host fs bridge, remote fs bridge, and regression tests continue to treat that workspace as read-only. The related fix PRs discussed in the issue were useful, but they are closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep #37634 open as the canonical tracker. The best fix is to preserve `workspaceAccess: "none"` isolation from the agent workspace while making the isolated sandbox workspace writable consistently across Docker/browser mounts, host fs bridge, remote fs bridge, tool/write gating, memory-flush behavior if applicable, regression tests, and docs.
