## Summary

Keep open. Current `main` does not implement the proposed `skill.yaml` permission manifest or an equivalent install-time permission review for skills. Skills are still defined as `SKILL.md` bundles with limited frontmatter metadata, `openclaw skills install` still downloads and installs ClawHub skill folders directly, and the threat-model docs still describe malicious skill installation as a current critical risk rather than a solved problem.

## Evidence

- **Skills are still defined by `SKILL.md`, not `skill.yaml`:** The skills docs define the format as a directory with `SKILL.md`, then document only current `metadata.openclaw` gating fields such as `os`, `requires.*`, `primaryEnv`, and `install`.
  - file: [docs/tools/skills.md:142](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/tools/skills.md#L142)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Current skill metadata type has no permission-manifest fields:** `OpenClawSkillMetadata` only includes `always`, `skillKey`, `primaryEnv`, `emoji`, `homepage`, `os`, `requires`, and `install`; there is no schema for declared network/filesystem/shell/credential permissions.
  - file: [src/agents/skills/types.ts:19](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/skills/types.ts#L19)

  - file: [src/agents/skills/frontmatter.ts:187](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/skills/frontmatter.ts#L187)

## Risks / Open Questions

- The proposal is still actionable, but it may need a design update because OpenClaw skills are instruction bundles centered on `SKILL.md`, not standalone executables with a preexisting manifest contract.
- Some runtime-enforcement discussion in the issue is partly outdated because `before_tool_call` now exists for plugins, so a future implementation should separate install-time skill declarations from runtime policy enforcement.
