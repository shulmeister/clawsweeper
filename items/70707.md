## Summary

Keep PR #70707 open. Current main does not contain the proposed synchronous invalidation path: device removal/token rotate/token revoke still respond first and only schedule `disconnectClientsForDevice` in a microtask, while per-RPC dispatch only rechecks shared gateway auth generation and has no equivalent device-token invalidation guard. The PR remains a focused gateway hardening change with relevant tests in the provided diff, so it is not conservative maintainer cleanup to close it.

## Best Possible Solution

Keep the PR open. The best path is for maintainers to review or independently land the same synchronous device-client invalidation design: add an in-memory invalidated flag to gateway clients, mark matching device-token clients before remove/rotate/revoke responses, reject further RPC dispatch for invalidated clients, and keep the ordering covered by focused gateway tests.
