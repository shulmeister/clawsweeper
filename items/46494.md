## Summary

Keep open. Current main still has the reported health/status divergence: the health path builds Telegram summaries from config/probe-only data, while channels.status reads the gateway runtime snapshot and passes it into the channel status builder. The linked PR #46527 appears to track the right fix, but it is open/unmerged in the provided context and the fix is not present in this checkout.

## Evidence

- **health snapshot lacks runtime state:** getHealthSnapshot constructs a ChannelAccountSnapshot with only accountId, enabled, configured, probe, and lastProbeAt, then passes that minimal snapshot to plugin.status.buildChannelSummary. It does not read or pass the gateway runtime snapshot, so running/lastStartAt/mode/tokenSource are absent on this path.
  - file: [src/commands/health.ts:324](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/commands/health.ts#L324)

- **missing fields default to reported false/none/null values:** buildTokenChannelStatusSummary defaults missing runtime fields through buildBaseChannelStatusSummary and then defaults tokenSource to "none" and mode to null, matching the issue's health JSON symptom when health passes a minimal snapshot.
  - file: [src/plugin-sdk/status-helpers.ts:321](https://github.com/openclaw/openclaw/blob/28497515fe1e6fa7ab5d51e64d2f3fe76ea39071/src/plugin-sdk/status-helpers.ts#L321)
