## Summary

Keep open. Current `main` still only surfaces chat disconnection when the browser WebSocket actually closes, while `chat.send` requests stay pending indefinitely on an open socket and the chat UI has no separate queued/retrying state for degraded backend transport. That matches the core trust/UX concern in #68303 even though I did not run a live stale-channel reproduction.

## Evidence

- **Composer gating only checks `connected`:** `renderChat` is called with `connected`/`canSend` derived directly from `state.connected`, and the composer textarea/buttons are disabled only when `!props.connected`. There is no separate stale/reconnecting/degraded state in the chat surface.
  - file: [ui/src/ui/app-render.ts:2258](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/app-render.ts#L2258)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '2210,2295p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Chat input disables only on full disconnect:** The chat composer textarea and attachment/talk controls use `?disabled=${!props.connected}`. A half-open backend path that leaves the WebSocket open would not trigger any UI disable/banner from this code path.
  - file: [ui/src/ui/views/chat.ts:1253](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/views/chat.ts#L1253)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '1230,1355p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Gateway client requests have no timeout or queued state:** `GatewayBrowserClient.request()` sends a WS frame, stores the promise in `pending`, and returns it. That promise resolves only on a response frame or rejects when the socket closes; there is no request timeout, retry timer, or degraded-transport signal for stuck requests on an open socket.
  - file: [ui/src/ui/gateway.ts:621](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/gateway.ts#L621)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '560,690p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Chat send path lacks retrying/queued UX for stalled sends:** `sendChatMessage()` optimistically appends the user message and then awaits `chat.send`; the only explicit recovery path is the `catch` when the request rejects. There is no intermediate queued/retrying status attached to the sent message if the request simply hangs.
  - file: [ui/src/ui/controllers/chat.ts:325](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/ui/src/ui/controllers/chat.ts#L325)

- **Documented queue UX is only for active-run follow-ups:** The user-facing Control UI docs document queued follow-ups only while a run is already active (`Steer` on queued messages). They do not describe any queued/retrying UX for degraded dashboard connectivity.
  - file: [docs/web/control-ui.md:178](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/docs/web/control-ui.md#L178)
