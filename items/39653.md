## Summary

Keep PR #39653 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and current gateway/client/onboarding tests still encode hostname acceptance. The linked issue #37177 is closed stale and earlier PR #37883 was closed unmerged, so this PR is not implemented or superseded.

## Best Possible Solution

Keep PR #39653 open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the intended contract. If private DNS hostname support is desired, track that separately with an explicit security design.
