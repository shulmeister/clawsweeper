## Summary

Keep PR #39653 open. Current main still accepts arbitrary `ws://` hostnames whenever `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` is set, and multiple current tests still assert that hostname-allowing behavior. The linked issue #37177 is closed stale and prior PR #37883 was closed unmerged, so neither supersedes this PR.

## Best Possible Solution

Keep PR #39653 open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the intended contract. If private DNS hostnames should be supported, track that separately with an explicit allowlist or DNS-resolution design.
