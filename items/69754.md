## Summary

Keep #69754 open. Current main still has a Feishu approval UX gap: Feishu accepts a generic capabilities array and advertises message-tool presentation, but its approval capability is only same-chat approval authorization, not a native/rendered approval UI. The pending exec approval text still tells the user to reply with `/approve ...` and even says foreground interactive approvals are available, without explaining that Feishu buttons/native approval cards are unavailable or that this is the intended text-command fallback. The related #58298 remains a separate Feishu interactive-card request for agent-authored `message` sends, not a canonical tracker for built-in approval fallback clarity.

## Best Possible Solution

Keep the issue open and fix it in the built-in approval path. The likely maintainer-grade fix is to make pending approval copy channel-aware when the initiating/forwarded surface has no approval render/native adapter: say Feishu approval is using text-command fallback, show the exact `/approve <id> <decision>` commands, state that completion can happen in the same Feishu chat when authorized, and avoid wording like “interactive approvals available” when no Feishu buttons will render. If Feishu buttons are intended, add a real approval render/native adapter or outbound `sendPayload` support, then document the Feishu approval behavior on the public Feishu and exec approvals docs. Keep #58298 separate unless maintainers broaden it to cover built-in approval UX.
