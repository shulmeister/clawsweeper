## Summary

Current `main` already covers the DeepInfra report through the generic OpenAI-compatible provider path: non-OpenAI `/v1` endpoints are treated as proxy-style `openai-completions`, `reasoning_content` is parsed, and slash-bearing model IDs now resolve correctly in selection and allowlist flows.

## Evidence

- **Custom OpenAI-compatible provider path is documented:** The provider docs describe configuring custom `models.providers.<id>` entries with `api: "openai-completions"` and explicitly call out non-native OpenAI-compatible endpoints.
  - file: [docs/concepts/model-providers.md:603](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/docs/concepts/model-providers.md#L603)
  - command: `nl -ba docs/concepts/model-providers.md | sed -n '600,620p'`
- **Non-native endpoints disable unsupported developer-role behavior:** OpenClaw forces `supportsDeveloperRole: false` for non-default/non-public OpenAI-compatible endpoints, matching the kind of 4xx compatibility failure described in the issue.
  - file: [src/agents/openai-completions-compat.ts:98](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/openai-completions-compat.ts#L98)
  - command: `nl -ba src/agents/openai-completions-compat.ts | sed -n '50,105p'`
- **`reasoning_content` is consumed by the OpenAI-compatible transport:** The streaming transport explicitly reads `reasoning_content`, `reasoning`, and `reasoning_text`, which covers the DeepInfra response shape shown in the issue body.
  - file: [src/agents/openai-transport-stream.ts:1436](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/openai-transport-stream.ts#L1436)

  - command: `nl -ba src/agents/model-selection-shared.ts | sed -n '45,110p'`

  - file: [src/agents/model-selection-shared.ts:610](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/agents/model-selection-shared.ts#L610)

## Risks / Open Questions

- No live DeepInfra request was run in this read-only review; the conclusion is based on current code, docs, and tests.
- There is still no bundled DeepInfra-specific provider plugin or provider doc page, so this close assumes the generic OpenAI-compatible path is sufficient for the original request.

## Close Comment

Closing from a Codex review because current `main` already implements this well enough through the generic OpenAI-compatible provider path. OpenClaw now documents custom `models.providers.<id>` entries for `api: "openai-completions"` and treats non-OpenAI base URLs as proxy-style endpoints (`docs/concepts/model-providers.md:603`, `docs/gateway/configuration-reference.md:2607`). The transport also handles DeepInfra-style `reasoning_content` fields (`src/agents/openai-transport-stream.ts:1436`), and the model-selection/allowlist path now resolves slash-bearing model IDs instead of rejecting them as not allowed (`src/agents/model-selection-shared.ts:52`, `src/agents/model-selection-shared.ts:409`, `src/agents/model-selection-shared.ts:610`). I could not tie that implementation to a shipped release after `v2026.4.22`, so `fixedRelease` is left unset, but on current `main` this issue is already covered.
