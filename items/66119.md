## Summary

Keep #66119 open. The report is recent, detailed, non-maintainer-authored, and has no protected labels, but current `main` does not appear to implement the requested guard/backup/preflight around `signal-cli` account data. OpenClaw delegates startup to `signal-cli daemon` and waits for the HTTP daemon; the Signal config surface and tests cover daemon startup/readiness, not detection of a `registered:false` account file or prevention of a risky restart loop.

## Best Possible Solution

Keep the issue open for maintainer investigation. The best path is a Signal plugin hardening change: confirm the relevant `signal-cli` account-store contract upstream, then add a conservative startup diagnostic/preflight that detects a configured account whose local signal-cli state is `registered:false`, stops before repeatedly starting the daemon, surfaces a clear recovery warning, and optionally documents or implements a safe backup strategy without silently mutating signal-cli state.
