## Summary

Keep open. Current main still exposes Discord `channels.discord.actions.messages` as a single boolean and still uses that one gate for Discord message read/fetch, send, edit, and delete paths. The issue is concrete, not protected by author/label rules, and the related items in the provided context do not supersede this specific Discord permission-granularity request.

## Best Possible Solution

Continue tracking this issue until Discord supports a backward-compatible granular message gate, likely `channels.discord.actions.messages: boolean | { send?: boolean; read?: boolean; edit?: boolean; delete?: boolean }`, with aligned config types, zod/generated schema metadata, account action-gate handling, discovery/runtime checks, docs, and focused tests. If maintainers prefer a different authorization model, record that product decision here rather than closing the request as cleanup.
