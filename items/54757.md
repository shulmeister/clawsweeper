## Summary

Keep this issue open. Current main still suppresses only WhatsApp text-only lifecycle tool payloads, while final warning payloads are returned for delivery and sent through `deliverWebReply`. The embedded runner still builds `⚠️ ... failed` payloads for mutating tool failures unless the stronger per-run `suppressToolErrorWarnings` option is set, and the WhatsApp inbound auto-reply path does not pass that option.

## Best Possible Solution

Implement an external-channel-safe tool-error policy for WhatsApp and similar external recipient channels. The narrow path is to pass `suppressToolErrorWarnings` or a channel-scoped equivalent into WhatsApp inbound auto-reply runs, then add regression tests proving mutating read/edit/write/message failure warnings are not delivered as final WhatsApp replies while normal final replies and intentional media/tool artifacts still deliver.
