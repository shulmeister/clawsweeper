## Summary

Current `main` still enumerates sibling home directories for `.openclaw`, still prints those paths in `doctor`, and still documents that cross-home-directory scan. This remains an actionable core bug, not a stale or already-fixed report.

## Evidence

- **Current code still scans every top-level home dir entry:** `findOtherStateDirs` resolves roots as `/home` on Linux and `/Users` on macOS, iterates every non-hidden directory under that root, and checks `<root>/<entry>/.openclaw`. There is no current-user restriction, backup-name exclusion, or opt-out.

  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Doctor still emits every discovered path in warning output:** The `doctor` state-integrity warning adds all results from `findOtherStateDirs` into `extraStateDirs` and renders each directory path under `- Multiple state directories detected. This can split session history.`

## Risks / Open Questions

- Cross-user state-directory enumeration still appears possible on shared Linux or macOS hosts when sibling homes are readable.
- Backup or snapshot directories under `/home` or `/Users` that contain `.openclaw` still look eligible for false-positive warnings.
- There is no inspected evidence of a mitigation flag or config escape hatch in current source or docs.
