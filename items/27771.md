## Summary

Keep PR #27771 open. The requested `channels.signal.configPath` behavior is still missing on current `main`: Signal config validation remains strict without that key, the bundled Signal plugin does not propagate a config path into daemon startup, and daemon argument construction cannot emit `signal-cli --config <path>`. The PR is stale because Signal now lives under `extensions/signal`, but the feature remains concrete and unsuperseded; the contributor explicitly said maintainers may take over.

## Best Possible Solution

Keep the PR open for maintainer takeover or replacement. Port the feature to the current bundled Signal plugin by adding optional `channels.signal.configPath` to the Signal config type/schema and generated metadata/UI hints, documenting it, treating nonblank values as configured account data, propagating it through `monitorSignalProvider` to `spawnSignalDaemon`, and emitting `--config <trimmed path>` before `daemon` with regression coverage for set, unset, and blank values.
