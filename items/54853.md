## Summary

This issue should remain open. Current main still lacks an agent-level or generic target-channel binding contract that lets a DingTalk-originated agent select the intended Feishu account for Feishu tools; the existing Feishu resolver only considers an explicit tool accountId, a same-id contextual Feishu account, and the Feishu default account.

## Best Possible Solution

Keep this issue open. The likely fix is a generic, documented target-channel account resolution contract for plugin tools, such as agents.list[].channelBindings or reuse of route binding data keyed by agentId plus target channel. Feishu tool resolution should then prefer explicit tool accountId, then the current agent's bound Feishu account, then a valid same-id contextual Feishu account, then channels.feishu.defaultAccount, with regression coverage for a DingTalk-originated agent whose DingTalk account id differs from the intended Feishu account id.
