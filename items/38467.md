## Summary

Keep #38467 open. Current main still exposes WhatsApp peer directory entries from configured `allowFrom` values, not from Baileys `contacts.upsert`, and there is no contact persistence or contact-name outbound resolution path. The generic directory CLI and target resolver can already consume directory entries, so the missing work is in the WhatsApp plugin implementation. Related PR #33924 was closed unmerged and addressed outbound `allowSendTo` gating, not persisted phone-contact lookup.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin: add an account-scoped, local contact store populated from Baileys `contacts.upsert`; add privacy/config handling such as an opt-in `storeContacts` flag if maintainers want it; merge stored contacts into `directory.listPeers`/`listPeersLive` with names/handles; allow the existing outbound target resolver to resolve unique contact names for message sends; and cover it with WhatsApp directory, socket-event, schema/docs, and message-action resolution tests.
