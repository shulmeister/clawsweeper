## Summary

Current main does not match the reported failure mode. `gateway:startup` still fires during startup, but Telegram message-tool sends now go through a direct send path that returns success only after Telegram returns a real `message_id`, rather than depending on the polling adapter becoming "connected."

## Evidence

- **Startup hook timing:** `gateway:startup` is dispatched after `startChannels()` plus a short timer in `src/gateway/server-startup-post-attach.ts:268-282`. This means the hook is still an early-start lifecycle event, not a post-readiness event.
  - file: [src/gateway/server-startup-post-attach.ts:268](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/gateway/server-startup-post-attach.ts#L268)

  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)

So adapter readiness is decoupled from startup completion.
