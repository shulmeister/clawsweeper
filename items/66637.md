## Summary

Keep open. Current main still has a real macOS-app-specific gap: the macOS app node advertises `browser.proxy` when browser control is enabled, but that command forwards to a local HTTP browser control listener on `127.0.0.1:<gatewayPort+2>` and the remote-mode app path shown in code does not start that listener. The separate CLI node-host path is different and does start an in-process browser control service, which matches the reporter's comparison that CLI node-host works on the same Mac while the app node does not.

## Best Possible Solution

Keep the issue open and fix the macOS app node boundary. The likely product/code direction is to make the macOS app node either start/use the same in-process browser control service as the CLI node-host path, or stop advertising `browser.proxy` unless a reachable local browser control service is actually available. Add a macOS regression test for remote mode app-node browser proxy readiness, plus routing coverage showing CLI node-host and app-node behavior are intentionally aligned.
