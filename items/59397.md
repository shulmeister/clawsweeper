## Summary

Keep open. Current `main` still does not validate `nodes` CLI transport timeouts, but this PR should not be closed as implemented or obsolete: its proposed switch to `parseTimeoutMsWithFallback` would change existing numeric timeout semantics (`Number("1e3") === 1000`, helper path parses via `parseInt`), and the patch is also stale against the current runtime split.

## Evidence

- **Current main still uses raw Number parsing for nodes CLI transport timeout:** The live `nodes` CLI runtime still passes `timeoutMs: callOpts?.transportTimeoutMs ?? Number(opts.timeout ?? 10_000)` into `callGateway`, so the PR's intended validation is not present on `main`.
  - file: [src/cli/nodes-cli/rpc.runtime.ts:24](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/nodes-cli/rpc.runtime.ts#L24)
  - command: `nl -ba src/cli/nodes-cli/rpc.runtime.ts | sed -n '1,200p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Shared timeout helper uses parseInt semantics:** `parseTimeoutMsWithFallback` parses string input with `Number.parseInt`, so inputs such as `1e3` would be reinterpreted as `1` rather than preserving the current `Number(...)` behavior.
  - file: [src/cli/parse-timeout.ts:49](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/parse-timeout.ts#L49)
  - command: `nl -ba src/cli/parse-timeout.ts | sed -n '1,220p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Other CLI gateway timeout paths still use Number(...):** The gateway CLI runtime still uses `Number(opts.timeout ?? 10_000)`, which matches the current nodes behavior. Changing only nodes to the parseInt-based helper would make timeout parsing inconsistent across CLI surfaces.
  - file: [src/cli/gateway-rpc.runtime.ts:26](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/gateway-rpc.runtime.ts#L26)
  - command: `nl -ba src/cli/gateway-rpc.runtime.ts | sed -n '1,80p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)
- **Review discussion already identified the regression risk:** The PR's Codex review on commit `017bef7a4d3231f2abaef457f5173adcca71d582` called out that switching this path to the helper would turn `--timeout 1e3` from `1000` ms into `1` ms unless the format is explicitly rejected first.
  - sha: [017bef7a4d32](https://github.com/openclaw/openclaw/commit/017bef7a4d3231f2abaef457f5173adcca71d582)
- **The submitted patch is stale against current file layout:** The PR body and diff target `src/cli/nodes-cli/rpc.ts`, but the transport-timeout implementation now lives in `src/cli/nodes-cli/rpc.runtime.ts`; `rpc.ts` is only a lazy-loading wrapper on current `main`.
  - file: [src/cli/nodes-cli/rpc.ts:30](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/src/cli/nodes-cli/rpc.ts#L30)
  - command: `nl -ba src/cli/nodes-cli/rpc.ts | sed -n '1,200p'`
  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)

## Risks / Open Questions

- Merging the PR as written would risk a behavioral regression for scientific-notation timeout inputs and create inconsistent timeout parsing between CLI commands.
- Closing the PR would drop a still-relevant cleanup: `main` does not yet validate invalid `nodes` CLI timeout values on this path.
