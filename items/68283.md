## Summary

Keep open. Current `main` still uses a fixed 1s retry for approval-handler bootstrap failures and still starts an already-registered context through the throwing path, so this PR's backoff/guarded-bootstrap behavior is not implemented on `main`. A later landed runtime change hardens pending-approval replay after gateway auth, but that is a different seam and does not replace the bootstrap retry changes proposed here.

## Evidence

- **Bootstrap still retries every 1000ms:** `APPROVAL_HANDLER_BOOTSTRAP_RETRY_MS` is still hard-coded to `1_000`, and `scheduleRetryForContext()` always requeues with that constant. There is no exponential backoff, no 300s pairing-required branch, and no connect-error detail handling in current `main`.
