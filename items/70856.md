## Summary

Keep open. Current main has several WhatsApp reconnect/auth hardening paths, but the issue reports a recent Windows npm/Scheduled Task regression with repeated 408/428/499 disconnects, missed inbound/outbound delivery, and a libsignal Bad MAC stack. I could not verify that current main fully covers that exact failure mode; in particular, the crypto-error recovery filter appears narrower than the reporter's stack and reconnect gaps can still leave no active listener while sends are attempted.

## Best Possible Solution

Keep this issue open and ask for maintainer follow-up against current v2026.4.24/main on the same Windows npm Scheduled Task setup. The likely product fix belongs in the WhatsApp plugin runtime: broaden/retest crypto-error recovery for libsignal Bad MAC stacks, add regression coverage for sends during reconnect gaps, and verify the health-monitor/reconnect controller does not unregister the usable listener or burn delivery retries while replacing the socket.
