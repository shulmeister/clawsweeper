## Summary

Keep open. Current main has partial heartbeat improvements, including lightweight bootstrap filtering and exec-event prompt bodies, but the central request is not fully resolved: `system heartbeat last` still reads an in-memory singleton heartbeat event, heartbeat runs can still emit events with `reason: "exec-event"`, and there is no durable heartbeat receipt surface for the CLI to distinguish scheduled/manual heartbeat receipts from async exec-event wakes. The related #66382 appears narrower and closed, while #68386 remains open for a related async-routing path rather than superseding this broader issue.

## Best Possible Solution

Keep this issue open but narrow it. The best path is to either implement a durable heartbeat receipt surface for actual scheduled/manual heartbeat runs and make `system heartbeat last` read that, or explicitly rename/split the current in-memory event endpoint so exec-event wakes cannot be mistaken for canonical heartbeat cadence evidence. Separately, track the async completion routing work under #68386 and treat `heartbeat-state.json` as a docs/product-contract clarification unless maintainers decide the runtime should own that file.
