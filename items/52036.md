## Summary

Keep open. Current `main` does not already implement this serialization fix, and the proposed PR still appears to introduce a behavioral regression by making guarded `appendMessage` effectively async while current main has synchronous callers that consume the returned `messageId` immediately.

## Evidence

- **Current guard is still unsynchronized:** `guardedAppend` directly reads and mutates shared pending state, calls `flushPendingToolResults()` inline, calls `originalAppend(...)`, and only then records new tool calls with `pendingState.trackToolCalls(toolCalls)`. There is no mutex/queue on current `main`, so the PR's claimed fix is not already implemented.

## Risks / Open Questions

- Closing as implemented would be incorrect: current `main` still has the original non-serialized guard path.
- Treating the PR as safe-to-close would also ignore a likely regression: converting guarded `appendMessage` into an always-async path conflicts with current synchronous `messageId` consumers.
