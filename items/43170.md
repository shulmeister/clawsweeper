## Summary

Keep open. Current main has superseded part of this PR: Feishu configured ACP bindings now use the generic configured-binding registry and Feishu routes inbound messages through `resolveConfiguredBindingRoute`/`ensureConfiguredBindingRouteReady`, with tests and docs. But the PR is not fully implemented or safely closeable: `acp.defaultChannels` is absent from current config types/schema, ACP bindings still require `match.peer.id`, and the bundled `qqbot` channel currently has no configured-binding provider or ACP route/readiness integration. The remaining QQ Bot/defaultChannels scope is real enough that this PR should either be replaced by a smaller current-architecture PR or closed manually by a maintainer, not auto-closed as implemented.

## Best Possible Solution

Replace this PR with a narrow current-main implementation if the remaining scope is desired: add QQ Bot `ChannelConfiguredBindingProvider` support and inbound configured-binding route/readiness handling in `extensions/qqbot`, with tests and docs. Treat `acp.defaultChannels` and catch-all ACP bindings as a separate product/config decision, because current validation still requires explicit `match.peer.id`.
