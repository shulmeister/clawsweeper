## Summary

DingTalk support is already positioned on current main as a community channel plugin, and the repo’s current scope says optional channel integrations should live as external plugins rather than be merged into OpenClaw core.

## Evidence

- **Vision scopes optional integrations to plugins:** Current project vision says optional capability should usually ship as plugins, plugin authors should host and maintain them in their own repository, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '40,115p'`
- **Plugin guide says not to add plugins to this repo:** The main plugin guide explicitly says you do not need to add your plugin to the OpenClaw repository and should publish it to ClawHub or npm instead. The same page lists channel plugins as a supported plugin type.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/plugins/building-plugins.md#L16)
  - command: `nl -ba docs/plugins/building-plugins.md | sed -n '1,60p'`

  - file: [docs/plugins/community.md:52](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/plugins/community.md#L52)

## Risks / Open Questions

- This closes the request as out-of-scope for core bundling; it does not assess the external DingTalk plugin’s code quality or maintenance health.
- If maintainers later decide DingTalk is a strategic bundled channel, they may still want a fresh core-tracking issue despite the current plugin-first policy.

## Close Comment

Closing as `clawhub` after a Codex review. Current main already treats DingTalk as a community plugin rather than a core-bundled channel: `docs/plugins/community.md:52` lists the DingTalk plugin (`@largezhou/ddingtalk` / `largezhou/openclaw-dingtalk`) and documents installing it with `openclaw plugins install @largezhou/ddingtalk`. That matches the repo’s current scope: `VISION.md:54` says optional capability should usually ship as plugins, be hosted in the author’s own repository, and clear a high bar before being added to core, and `docs/plugins/building-plugins.md:16` explicitly says plugins do not need to be added to the OpenClaw repository. So this looks like a valid OpenClaw integration, but the right home is the external plugin / ClawHub path rather than OpenClaw core.
