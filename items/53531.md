## Summary

Keep this issue open. Current main still does not implement same-provider auth-profile failover inside compaction: the compaction path resolves one target auth profile/API key and performs one compaction request, while the normal run path has separate profile-candidate rotation machinery. The discussion's idempotency concern remains important because compaction retries need to apply only one summary to one stable transcript snapshot.

## Best Possible Solution

Keep this issue open. The best fix is to extract/reuse the auth-profile order, cooldown, and failure-classification logic in a compaction-safe helper, snapshot the transcript once, try eligible same-provider profiles against that snapshot, mark transient profile failures, and commit exactly one successful summary/checkpoint/transcript rotation. Add regression coverage for manual compaction, overflow-triggered compaction, timeout-triggered compaction, and a partial-failure/idempotency case where profile A fails and profile B succeeds.

## Risks / Open Questions

- A naive outer-run retry after a compaction failure can replay work or double-compact; the fix needs a stable compaction snapshot and single-commit boundary.
- Manual `/compact`, overflow recovery, timeout recovery, and context-engine delegated compaction enter the path differently and need separate coverage.
