## Summary

DingTalk support is now positioned as a community plugin rather than a bundled OpenClaw core extension. Current main documents both the plugin-first scope and an installable DingTalk plugin.

## Evidence

- **Plugin-first project scope:** OpenClaw's vision says core stays lean, optional capability should usually ship as plugins, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/VISION.md#L54)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **Canonical community-plugin path:** The community plugins doc says third-party plugins are the path for new channels/tools/providers, that ClawHub is the canonical discovery surface, and that `openclaw plugins install` checks ClawHub first and falls back to npm.
  - file: [docs/plugins/community.md:9](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/plugins/community.md#L9)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)
- **DingTalk plugin already documented on main:** Current main explicitly lists a DingTalk plugin with package `@largezhou/ddingtalk` and install command `openclaw plugins install @largezhou/ddingtalk`.
  - file: [docs/plugins/community.md:52](https://github.com/openclaw/openclaw/blob/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e/docs/plugins/community.md#L52)
  - sha: [272bd59e7a8a](https://github.com/openclaw/openclaw/commit/272bd59e7a8a4d6ef6e03b9bdc1068a02986d56e)

## Risks / Open Questions

- This only establishes that DingTalk belongs on the community-plugin path for OpenClaw core cleanup; it does not verify the third-party DingTalk plugin's maintenance status or feature completeness.

## Close Comment

Closing after Codex review. Current main treats this as a community-plugin integration rather than a bundled OpenClaw core feature: `VISION.md:54` says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high, and `docs/plugins/community.md:52` now lists a DingTalk plugin (`@largezhou/ddingtalk`) with the install command `openclaw plugins install @largezhou/ddingtalk`. On that basis, this belongs on the plugin ecosystem path rather than staying open as a core-feature request. If there is still a concrete blocker in the DingTalk plugin or in the plugin SDK, please open a focused issue with repro details for that specific gap.
