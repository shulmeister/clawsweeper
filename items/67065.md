## Summary

Keep open. The related media-ref problem from #66827/#66910 is separately addressed in current hook mappers, but this issue asks for a second capability: a supported session-scoped way for a managed media flow to own the immediate follow-up turn before stock prompt hooks and the normal agent run. Current main has current-turn `before_agent_reply` and persistent plugin-owned conversation binding machinery, but not a documented one-shot pending-workflow/next-turn claim primitive for `before_agent_reply` or managed hooks.

## Best Possible Solution

Keep the issue open as a plugin/API design gap. The likely path is a narrow, documented plugin-facing primitive that lets a current turn set session/conversation-scoped pending ownership for the next N inbound replies, delivers those replies to the owning workflow before stock prompt hooks and the main agent run, and provides TTL/cleanup/fallback semantics. If maintainers prefer existing plugin conversation bindings, document a supported media-clarifier recipe and ensure `before_agent_reply`/managed-hook authors can bind and release the current Telegram DM without relying on session-key parsing.
