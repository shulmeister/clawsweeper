## Summary

Keep open. Current `main` still routes isolated cron `announce` replies straight to outbound delivery without the TTS transform that normal reply dispatch uses, so `[[tts]]` tags would remain raw text instead of becoming audio payloads.

## Evidence

- **Cron announce path sends raw payloads directly:** `deliverViaDirect()` builds `payloadsForDelivery` and passes them straight into `deliverOutboundPayloads(...)` with no `maybeApplyTtsToPayload(...)` step in between.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:601](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/isolated-agent/delivery-dispatch.ts#L601)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Text-only announce finalization still only produces plain text payloads:** For text-only cron announce delivery, `finalizeTextDelivery()` rewrites the reply to `deliveryPayloads = [{ text: finalReply }]` and then calls the same direct-delivery path; there is still no TTS processing seam on this branch.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:725](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/isolated-agent/delivery-dispatch.ts#L725)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Normal reply dispatch still applies TTS explicitly:** The non-cron reply paths continue to call `maybeApplyTtsToPayload(...)` before delivery, which is the behavior the issue says cron announce is missing.
  - file: [src/auto-reply/reply/dispatch-acp-delivery.ts:113](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/auto-reply/reply/dispatch-acp-delivery.ts#L113)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Config-driven reply dispatch also uses the TTS transform:** `maybeApplyTtsToReplyPayload()` in the standard auto-reply path loads `../../tts/tts.runtime.js` and returns `maybeApplyTtsToPayload(params)` when TTS should run.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:109](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/auto-reply/reply/dispatch-from-config.ts#L109)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **TTS transform is the code that strips tags and attaches audio:** `maybeApplyTtsToPayload()` parses TTS directives, cleans the visible text, and on success returns a payload with `mediaUrl` and `audioAsVoice`, which is exactly the behavior missing from the cron announce path.
  - file: [extensions/speech-core/src/tts.ts:1033](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/speech-core/src/tts.ts#L1033)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Existing cron announce tests still assert plain text passthrough:** Current cron direct-delivery coverage checks that text-only announce calls the channel adapter with the literal text string, and there are no cron test references to TTS handling.
  - file: [src/cron/isolated-agent.direct-delivery-core-channels.test.ts:271](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/cron/isolated-agent.direct-delivery-core-channels.test.ts#L271)
  - command: `rg -n "tts|audioAsVoice|voiceCompatible|\[\[tts" src/cron -g '*.test.ts'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
