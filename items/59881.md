## Summary

Current main still builds plugin hook ctx.channelId from params.messageChannel with a provider fallback, while the run params keep the actual current channel identifier in a separate currentChannelId field. That is too consistent with the reported bug to close conservatively.

## Evidence

- **Hook context still uses messageChannel/provider for channelId:** runEmbeddedPiAgent builds hookCtx with messageProvider: params.messageProvider and channelId: params.messageChannel ?? params.messageProvider ?? undefined. That means a provider-like messageChannel or fallback can still surface as ctx.channelId.
  - file: [src/agents/pi-embedded-runner/run.ts:2850](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/src/agents/pi-embedded-runner/run.ts#L2850)
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **Run params expose a separate current channel id:** RunEmbeddedPiAgentParams defines messageChannel/messageProvider separately from currentChannelId. currentChannelId is documented as the current channel ID, but it is not used in the hookCtx channelId assignment found in run.ts.
  - file: [src/agents/pi-embedded-runner/run/params.ts:575](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/src/agents/pi-embedded-runner/run/params.ts#L575)
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **Issue report provides concrete plugin impact:** The report names OpenClaw 2026.4.1, Discord, @vectorize-io/hindsight-openclaw@0.4.19, and a specific session key where ctx.channelId is observed as "discord" instead of the Discord channel snowflake, causing per-channel memory buckets to merge.
