## Summary

Current main still leaves the reported builtin/default memory_search freshness problem open. The gateway/tool path reuses a long-lived builtin MemoryIndexManager, file watches mark the index dirty, but search only starts dirty sync in the background and immediately queries the existing index. QMD has an awaited dirty-search path, but builtin remains the default backend and the fallback path, so the observable issue is not implemented on main.

## Best Possible Solution

Keep this issue open and fix the builtin memory manager/tool path. The likely maintainer path is to make MemoryIndexManager.search() await pending or newly triggered sync when dirty or sessionsDirty is true for agent-facing memory_search, or add an explicit freshness mode used by the tool, then add regression coverage for a long-lived builtin manager where the first search after creating, deleting, or renaming a memory file observes the updated index and returns no nonexistent paths.
