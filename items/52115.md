## Summary

Keep open. Current `main` still allows stale first-search results in the builtin `memory_search` path for long-running gateway processes. The manager is cached across requests, search-triggered refresh runs in the background instead of being awaited, and stale file rows are only pruned during a later sync pass. That means the reported stale/ghost-path behavior remains plausible on current `main`, so there is not strong evidence for closure.

## Evidence

- **Gateway/runtime uses a long-lived cached manager:** `extensions/memory-core/src/memory/manager.ts:66-67` defines the singleton `INDEX_CACHE`, and `extensions/memory-core/src/memory/manager.ts:155-183` returns cached `MemoryIndexManager` instances keyed by agent/workspace/settings. `extensions/memory-core/src/runtime-provider.ts:5-18` exposes that same manager to runtime callers.
  - file: [extensions/memory-core/src/memory/manager.ts:66](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/memory-core/src/memory/manager.ts#L66)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '1,220p'; nl -ba extensions/memory-core/src/runtime-provider.ts | sed -n '1,80p'`
- **Search-triggered sync is explicitly asynchronous:** `extensions/memory-core/src/memory/manager.ts:320-329` calls `startAsyncSearchSync(...)` before running the query, and `extensions/memory-core/src/memory/manager-async-state.ts:1-14` shows that helper fires `sync({ reason: "search" })` without awaiting it.
  - file: [extensions/memory-core/src/memory/manager.ts:320](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/memory-core/src/memory/manager.ts#L320)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '289,329p'; nl -ba extensions/memory-core/src/memory/manager-async-state.ts | sed -n '1,40p'`
- **Current tests codify non-blocking stale-first-search semantics:** `extensions/memory-core/src/memory/manager.async-search.test.ts:5-27` has an explicit test named `does not await sync when searching`, confirming current intended behavior is to let search proceed before sync finishes.
  - file: [extensions/memory-core/src/memory/manager.async-search.test.ts:5](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/memory-core/src/memory/manager.async-search.test.ts#L5)

  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:682](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/memory-core/src/memory/manager-sync-ops.ts#L682)

Deleted files can therefore surface until sync pruning catches up.
  - file: [extensions/memory-core/src/memory/manager-search.ts:147](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/extensions/memory-core/src/memory/manager-search.ts#L147)
