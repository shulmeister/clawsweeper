## Summary

Close as superseded. Current main handles the documented canonical Ollama path through the bundled native Ollama provider and runtime stream registration, while this PR's proposed `api: "ollama"` OpenAI-completions alias conflicts with the current native `/api/chat` docs and tests. Remaining arbitrary/custom Ollama provider-id behavior is better tracked by the narrower open issues #34252 and #62735.

## Best Possible Solution

Close this PR as superseded. Keep the shipped native Ollama provider path for `api: "ollama"`, keep `api: "openai-completions"` as the explicit documented legacy/proxy mode, and handle arbitrary/multiple Ollama provider ids in the existing narrower issues rather than merging a generic OpenAI-completions alias for native Ollama.
