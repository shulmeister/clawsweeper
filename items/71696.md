## Summary

Keep #71696 open. Current main still does not implement Telegram chat-type-based reply mode or the related reply-threading plumbing the PR adds: Telegram config/schema/docs expose only the global `replyToMode`, Telegram runtime and native commands still carry one global resolved mode, Telegram delivery cannot resolve `[[reply_to_current]]` from the triggering Telegram message in the outbound plan, and plugin command context still lacks provider message IDs. The PR is recent, focused, supersedes closed #70579, and the provided PR context shows it added follow-up commits for the bot-review concerns rather than going stale.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main, preserve runtime `replyToMode` fallback precedence, add Telegram `replyToModeByChatType` consistently across schema/types/generated metadata/docs, resolve reply mode per direct/group for message and native-command delivery, pass the triggering Telegram message ID into outbound reply planning/delivery, expose provider message IDs to plugin command handlers, and run the changed Telegram/config/plugin/outbound checks.
