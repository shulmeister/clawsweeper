## Summary

Close #67085 as duplicate/superseded by open #13364. Current main still confirms the same remaining product/API gap: managed HOOK.md handlers are loaded into the internal hook registry, while tool lifecycle hooks run through the typed plugin hook runner. The detailed repro and security-impact notes in #67085 are useful, but the remaining fix or explicit plugin-only policy decision should live on the canonical #13364 tracker.

## Best Possible Solution

Keep #13364 as the single canonical tracker. The best fix is either to add a typed managed/internal HOOK.md tool-call contract with blocking, parameter rewrite, after-call observation, docs, and regression tests, or to make the plugin-only boundary explicit by rejecting/surfacing unsupported tool lifecycle events in HOOK.md loading and hooks info output.
