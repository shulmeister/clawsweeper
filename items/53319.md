## Summary

Keep this issue open. Current main still treats distinct ACP sessions as independently initialized and runnable in parallel, and the recent ACPX/runtime work does not provide source, test, or release-history proof that the reported rapid two-spawn first-agent launch failure is fixed.

## Best Possible Solution

Keep this open until current main is verified with the bundled acpx 0.6.1 or newer using two rapid `sessions_spawn(runtime:"acp", agentId:"claude", mode:"run")` calls. If it still reproduces, add focused regression coverage around distinct ACP session spawn/init dispatch and fix the ACPX runtime/control-plane launch path, likely with a backend-level init guard, health check, or retry. If it no longer reproduces, close later with live proof and the exact fixing commit/release.

## Evidence

That is a specific launch/init failure, not just missing parent completion relay.
- **Current spawn path still covers the reported surface:** `spawnAcpDirect` creates a fresh ACP child session key, initializes the ACP runtime, then dispatches the first task through the gateway `agent` method on the subagent lane and returns `accepted`.
