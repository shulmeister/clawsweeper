## Summary

Keep open. Current `main` has recent ACP hardening around stale/pending session identity, fresh-session retries, and same-session turn serialization, but I did not find strong evidence that it specifically fixes or obsoletes #53319’s report about two rapid `sessions_spawn runtime:"acp"` launches where the first child never starts its harness process.

## Evidence

- **Only same-session concurrency is covered:** The ACP manager test coverage I found serializes concurrent turns for one existing ACP session key; it does not cover two distinct child ACP sessions spawned in rapid succession from the same parent.
