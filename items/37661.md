## Summary

Keep #37661 open. Current main still has tool-call loop detection and several exact duplicate-suppression paths, but I did not find an assistant text repetition detector, n-gram/sentence guard, generation abort path, or outbound per-turn duplicate/message cap that would stop the reported pure text repetition flood. The April 11 issue comment is important evidence: it says the bug still reproduced on 2026.4.10 during normal main-session conversation with a custom OpenAI-compatible endpoint, and explicitly distinguishes this from tool-call loop issues such as #34574.

## Best Possible Solution

Keep this issue open and implement an output-level guard in the assistant streaming/final reply pipeline, plus an outbound per-turn duplicate/message cap in delivery planning. Add focused tests for repeated deltas, repeated sentence/ngram fragments in one assistant message, repeated block-stream payloads, and repeated channel sends. Keep #34574 separate because it tracks tool-call loops; #37661 is about pure text generation and delivery flooding.

## Risks / Open Questions

- The exact trigger is provider/model-specific and intermittent, so a fix should include synthetic regression tests instead of relying on the custom endpoint reproduction.
- A repetition guard can false-positive on legitimate repeated text, logs, code, or lists; it should be conservative, configurable, and scoped to clear runaway patterns.
- Current exact duplicate suppression and block/preview streaming behavior are adjacent mitigations; a new guard should preserve those paths while adding output-level protection.
