## Summary

Keep #37661 open. Current main still has tool-call loop detection and exact duplicate event suppression, but I found no assistant text repetition/ngram guard and no shared per-turn outbound flood cap that would stop repeated pure text or repeated channel sends. The April 11 follow-up remains important because it reports the same symptom on 2026.4.10 during normal conversation, with a custom OpenAI-compatible endpoint, and explicitly distinguishes this from #34574's tool-call loop scope.

## Best Possible Solution

Keep this issue open. Implement a conservative output-level repetition guard in the assistant streaming/final reply pipeline, plus a shared per-turn outbound duplicate/message cap before channel delivery. Add focused tests for repeated deltas, repeated sentence/ngram fragments within one assistant message, block streaming/final replay, and repeated channel sends. Keep #34574 separate because it tracks tool-call loops, while #37661 tracks pure assistant text/output flooding.

## Risks / Open Questions

- The exact trigger is provider/model-specific and intermittent, so regression coverage should use synthetic repeated-delta, repeated final-text, and repeated outbound-send fixtures rather than relying on the reporter's endpoint.
- A text repetition guard could false-positive on legitimate repeated logs, code blocks, tables, or quoted text; thresholds should be conservative and configurable.
