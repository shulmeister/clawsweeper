## Summary

Keep #70266 open. Current main still renders the macOS Talk Mode overlay as a generic `TalkOrbView` with circle-based visuals, and the native Talk config path only carries Talk provider fields plus `ui.seamColor`. The existing assistant/avatar identity work is present for Control UI and the `agent.identity.get` gateway method, but that identity is not wired into the macOS Talk Mode overlay, matching the reporter's follow-up that this is a distinct native overlay surface from #22199.

## Best Possible Solution

Implement this in the macOS Talk Mode path rather than closing it against Control UI identity work: fetch the active assistant identity/avatar through a stable gateway contract such as `agent.identity.get` or an additive Talk config field, resolve supported local/remote/data avatar sources safely, store the resolved avatar in the native Talk overlay state, and render it as a circular cropped image inside `TalkOrbView` while preserving the existing phase rings, pulse, and fallback orb. Add parser/unit coverage for the identity payload and a macOS overlay smoke/regression test that proves the avatar path is passed into the overlay.
