## Summary

Keep open. Current `main` partially mitigates the issue by aborting restart before action when config validation fails, but it still does not implement an atomic restart/rollback that preserves the previous healthy gateway until the replacement is proven healthy. Service-manager restart paths remain stop/kill-then-start, and the in-process restart path closes the server first; if restart startup then fails, the process stays alive but the gateway stays down.

## Evidence

- **Invalid-config restarts are now blocked before restart action:** `runServiceRestart()` validates config before any restart action, and the paired test asserts restart is aborted and `service.restart` is not called when config is invalid. This addresses the issue's config-error example, but only for invalid config, not for broader restart failures.

- **Managed restart paths are not atomic swaps:** The service backends still use direct restart semantics: systemd runs action `restart`, launchd uses `launchctl kickstart -k`, and Windows scheduled-task restart explicitly ends the task, kills listeners, waits for port release, then starts again. None of these paths keep the old listener serving until the new one is verified healthy.

That is the disconnect scenario the issue is asking to avoid.
