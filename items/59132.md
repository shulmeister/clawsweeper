## Summary

Current `main` has multiple landed protections against replaying the same Telegram inbound message: early webhook acknowledgment, Telegram update-id/message-id dedupe, shared inbound message-id dedupe before dispatch, and followup queue redelivery dedupe. That is strong enough to treat this cleanup item as implemented on `main`.

## Evidence

- **Telegram webhook returns early instead of holding the request open:** Webhook handling is configured with `onTimeout: "return"` and a 5 second callback timeout, which is explicitly intended to stay below Telegram/proxy read timeouts so the handler can continue without Telegram retrying the same update.

  - sha: [78a431ed224b](https://github.com/openclaw/openclaw/commit/78a431ed224b6f5b9d58672346551a5b411ad200)

  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1924](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/telegram/src/bot-handlers.runtime.ts#L1924)
