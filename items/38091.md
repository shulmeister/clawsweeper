## Summary

Keep #38091 open. Current main still clears WebChat active-run state on reconnect and has no replay/resubscribe contract to recover missed chat terminal events. The server has partial chat.send idempotency, but it only returns in_flight for an already-running clientRunId and does not reattach the reconnecting Control UI stream or restore the cleared local run state. Related #38090 is a closed duplicate, while #45952 tracks adjacent outbound message loss and does not supersede active-run recovery.

## Best Possible Solution

Keep #38091 open as the canonical Control UI/WebChat reconnect-safe active-run recovery issue. A proper fix should preserve or reconcile chatRunId and streaming state across reconnect, provide server/client replay or resubscribe for missed chat terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for WebSocket close/reconnect during an active chat.send run. Keep #45952 separate for outbound queue/ACK delivery loss.
