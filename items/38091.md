## Summary

Keep open. Current main still does not preserve an in-flight Control UI/WebChat run across a WebSocket reconnect: the browser reconnect path rejects pending RPCs, the server drops the old connection's session subscriptions, and the UI explicitly clears `chatRunId`/stream state after reconnect. Related fixes reduce adjacent failure modes, but they do not implement the reported expectation that an active session survives reconnect. #38090 is an open duplicate-looking earlier issue, but #38091 contains the richer follow-up reproduction data, so this item should remain canonical unless maintainers explicitly consolidate.

## Best Possible Solution

Keep #38091 open as the active tracking issue. The best fix is to make Control UI reconnect reattach or reconcile active `chat.send` runs instead of clearing local run state unconditionally: preserve the active run identity through reconnect, resubscribe to session events, recover missed terminal state with a gateway-backed reconciliation path such as `agent.wait`/history reload, and add a regression test that simulates a WebSocket reconnect during an active WebChat response. Maintainers can separately close the thinner duplicate-looking #38090 against this richer thread.
