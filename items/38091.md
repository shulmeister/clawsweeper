## Summary

Keep open. Current main still treats a Control UI reconnect as a cleanup boundary for an active WebChat run: reconnect hello clears `chatRunId`/`chatStream`, the browser WebSocket client rejects pending RPCs on close, and server-side `chat.send` idempotency/active-run tracking does not reattach the UI to an in-flight run. The issue has concrete logs plus related user comments, and the closed same-title #38090 was closed as the duplicate of this richer canonical issue, not as a fix confirmation.

## Best Possible Solution

Keep #38091 open as the canonical tracker for Control UI/WebChat reconnect-safe active run recovery. A good fix should preserve or reconcile active `chatRunId`/stream state across reconnect, resubscribe/replay or reload missed terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for reconnect during an active `chat.send` run. If maintainers decide stream resumption is intentionally out of scope, document that product behavior before closing.
