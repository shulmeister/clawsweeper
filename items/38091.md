## Summary

Keep open. The issue has concrete WSL2 logs plus a macOS follow-up showing the broader `terminated` symptom, and current `main` still does not implement the requested behavior of preserving an in-flight Control UI/WebChat run across a WebSocket reconnect. Related hardening exists, but it is partial: lifecycle errors are surfaced, Stop-button aborts are queued across reconnects, and `terminated` transport failures are classified for failover; the reconnect path itself still clears active chat state.

## Best Possible Solution

Keep tracking this until Control UI and Gateway can reconcile active WebChat runs after reconnect. A solid fix should preserve or recover the active run state, resubscribe/replay missed terminal state from server-side run or transcript state, avoid clearing `chatRunId`/`chatStream` until terminal state is known, and add UI plus gateway regression coverage for reconnect during an active `chat.send` run. If maintainers decide stream resumption is out of scope, document that product behavior explicitly before closing.
