## Summary

Keep #38091 open. Current main still does not preserve or reattach an in-flight WebChat/Control UI chat run across a WebSocket reconnect. The UI explicitly clears active run state on reconnect hello, the browser client rejects pending RPCs on socket close, and the available server idempotency path only reports an active duplicate as `in_flight`; it does not replay missed chat deltas/finals or restore the local active-run state.

## Best Possible Solution

Keep #38091 open as the canonical Control UI/WebChat reconnect-safe active-run recovery issue. A proper fix should preserve or reconcile `chatRunId` and stream state across reconnect, add a server/client replay or resubscribe contract for missed chat terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for WebSocket close/reconnect during an active `chat.send`. Keep #45952 separate for outbound queue/ACK delivery loss.
