## Summary

Current main still matches the reported behavior: ACP `sessions_spawn` initializes a child ACP session and immediately dispatches its first turn through the normal ACP turn gate, where `acp.dispatch.enabled=false` is treated as a hard dispatch error. The item is recent, has no maintainer/protected-label blocker, no linked implementation PR in the provided context, and the related ClawSweeper reports cover adjacent ACP session behavior rather than this policy split.

## Best Possible Solution

Keep this issue open and implement a focused ACP policy split. The likely fix belongs around `src/agents/acp-spawn.ts`, `src/agents/agent-command.ts`, and `src/auto-reply/reply/dispatch-acp.ts`: mark explicit `sessions_spawn(runtime="acp")` first turns as manual/parent-owned dispatch, allow them when `acp.enabled=true` and the backend/agent policies pass, while preserving `acp.dispatch.enabled=false` for automatic bound conversation follow-ups and background topic binding. Add tests that prove manual ACP spawn succeeds with dispatch disabled and normal bound ACP message dispatch still returns `ACP_DISPATCH_DISABLED`.
