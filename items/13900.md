## Summary

Current main already provides the requested fresh-per-run cron-session behavior via `sessionTarget: "isolated"`, with retained per-run records that are later pruned by `cron.sessionRetention`.

## Risks / Open Questions

- There is no dedicated `ephemeral: true` field; this closure assumes the shipped `sessionTarget: "isolated"` plus run-session retention/pruning satisfies the underlying request.
- The local checkout only exposes the `v2026.4.22` tag, so an earlier shipping release could not be proven from local history.

## Close Comment

Closing from a Codex review: current `main` already covers this behavior through isolated cron sessions. `docs/automation/cron-jobs.md:89` documents that isolated jobs run with a fresh session, the runtime enforces that with `forceNew` in `src/cron/isolated-agent/run.ts:481-494`, and `src/cron/isolated-agent.session-identity.test.ts:127-149` verifies each cron run gets a distinct `sessionId`/`sessionKey`. OpenClaw also keeps per-run `...:run:<uuid>` records and prunes them with `cron.sessionRetention` (`src/cron/isolated-agent/run-session-state.ts:33-40`, `src/cron/session-reaper.ts:1-36`, `docs/cli/cron.md:78-81`), which gives temporary logs without long-lived context buildup. Based on that, this request is already implemented on `main` and shipped at least in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
