## Close Comment

Closing from a Codex review. This no longer looks actionable on current `main`: the reporter confirmed the CLI/gateway failure stopped after upgrading to OpenClaw `2026.3.23-2`, and later reported `2026.3.28` was still working normally. Current `main` also has explicit coverage for this auth/token family: `src/commands/onboard-non-interactive/local/gateway-config.ts:62` preserves persisted gateway tokens during re-onboard, `src/commands/onboard-non-interactive/local/gateway-config.test.ts:93` covers that regression, `src/gateway/config-reload.test.ts:321` requires a gateway restart when `gateway.auth.token` changes, and `src/gateway/server.reload.test.ts:847` verifies stale websocket auth is rejected while refreshed auth succeeds after reload. `docs/cli/gateway.md:224` also now documents missing `operator.read` as degraded diagnostics rather than a dead gateway. Based on that evidence, I’m closing this as implemented on main. If this still reproduces on a current release, please open a new issue with `openclaw -v`, `openclaw gateway probe --json`, and the exact daemon/onboard path used.
