## Summary

Keep open. Current main still has the reported starvation shape: followup drains run until the followup queue is empty, each followup invokes a full embedded agent run on the same per-session lane, and the command queue is FIFO with no user-message priority or preemption. Existing mitigations for heartbeat dropping and background context maintenance reduce some contributors, but they do not solve fair scheduling for fresh inbound messages behind accumulated followup work.

## Best Possible Solution

Keep this issue open and fix it in the core reply/session scheduler. The best implementation is a bounded or fair followup drain, explicit priority or preemption for fresh user messages over accumulated followup/system-event work on the same session key, a phase-aware aggregate watchdog for queue/preflight/provider/post-turn time, and regression tests that enqueue multiple followups followed by a later inbound user message on the same session lane.
