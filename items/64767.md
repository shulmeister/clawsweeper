## Summary

Keep open. Current main still has the reported failure shape: session transcript readers materialize the whole JSONL file into one string and split it with a regex, and gateway chat history calls that reader before applying response-size caps. Session maintenance covers `sessions.json` rotation and optional whole-directory disk cleanup, but it does not provide a per-transcript size cap or chunked full-history reader. The linked maintainer PR #64858 targeted this issue directly but is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open and land a targeted gateway/session fix: replace full-file transcript reads used by `readSessionMessages`, `readLatestSessionUsageFromTranscript`, and idempotency/history paths with bounded or streaming line iteration, add regression coverage for transcripts larger than the internal chunk size and CRLF/boundary cases, and separately consider doctor/session-maintenance warnings or transcript rotation for active `.jsonl` growth. PR #64858 is useful prior art but needs review/rework because it is closed and unmerged.
