## Summary

Keep open. Current main still wires `write`, `edit`, sandboxed write/edit, and `apply_patch` without an instruction-file write guard, and I found no equivalent implementation, tests, docs, changelog entry, or history on main. The PR remains a plausible focused security hardening item, but it needs maintainer security/product review because it would change documented workflows that currently allow agent-managed `SOUL.md`/`MEMORY.md` edits.

## Best Possible Solution

Keep the PR open for maintainer security/product review. The best path is to either land or port a core guard that covers all mutation surfaces (`write`, `edit`, sandboxed variants, and `apply_patch`), normalizes/canonicalizes alias paths before allowing writes, adds regression tests for bypasses, and updates user-facing docs for the new approval/override behavior; otherwise a maintainer should explicitly reject or replace the approach.
