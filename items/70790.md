## Summary

Keep PR #70790 open. Current main has shipped adjacent toolsAllow forwarding and a /btw-specific empty-tools payload patch, but it has not implemented this PR's embedded-runner stream-resolution guard. The current OpenAI-compatible payload builders can still serialize empty tool arrays from an empty tool context or replayed tool history, and the embedded stream resolver still returns the session fallback/custom/boundary-aware streams unwrapped.

## Best Possible Solution

Keep the PR or a successor open, but rework it against current main: add a focused embedded-runner payload guard that strips tools: [] and orphaned tool_choice/parallel_tool_calls only at appropriate OpenAI-compatible egress paths, preserve existing provider/custom stream semantics, and add regression tests covering boundary-aware, custom, and session-http-fallback paths. Also reconcile it with the shipped toolsAllow forwarding so it does not claim to solve already-fixed allowlist propagation as its primary root cause.
