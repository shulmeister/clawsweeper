## Summary

Keep open. Current main does not show the claimed fix: startup can still load with `preferSetupRuntimeForChannelPlugins: true`, while the deferred reload path has no way to preserve or forward that preference, and the plugin loader still includes the preference in the cache key. That leaves the reported double-load/different-cache-key failure mode plausibly unresolved on main.

## Evidence

- **Startup load still sets the preference when deferred channel plugins exist:** `loadGatewayStartupPlugins` is called with `preferSetupRuntimeForChannelPlugins: deferredConfiguredChannelPluginIds.length > 0`, so startup can use the `prefer-setup` cache-key mode described in the issue.
