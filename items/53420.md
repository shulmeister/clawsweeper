## Summary

Keep open. Current `main` does not contain the redirect-path chmod normalization from PR #53420, and the current redirect test still asserts the umask-reduced mode. The item is therefore not `implemented_on_main` or `cannot_reproduce`. There is also a real scope question still open: `writeSavedMediaBuffer` uses the same umask-affected `fs.writeFile(..., { mode: MEDIA_FILE_MODE })` pattern for local/inbound media paths, so the PR is not obviously complete enough to close out conservatively.

## Evidence

- **Redirect save path still returns immediately after rename:** On current main, `saveMediaSource()` downloads to a temp file, computes the final name, calls `fs.rename(tempDest, finalDest)`, and immediately returns. There is no `fs.chmod(finalDest, MEDIA_FILE_MODE)` normalization in the redirect branch.

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current test suite expects umask-reduced permissions for redirects:** `src/media/store.redirect.test.ts` still expects the saved file mode to be `0o644 & ~process.umask()` (or `0o666` on Windows). That codifies the pre-fix behavior rather than the PR's proposed normalized `0o644` behavior.
  - file: [src/media/store.redirect.test.ts:86](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media/store.redirect.test.ts#L86)

## Risks / Open Questions

- Closing this PR would discard a still-unmerged bug fix; current main and the latest release `v2026.4.22` still lack the redirect-path chmod normalization.
- If the PR is revived as-is, maintainers should decide whether to extend it to `writeSavedMediaBuffer()` or explicitly scope it, because inbound/local media paths still appear exposed to the same umask behavior.
