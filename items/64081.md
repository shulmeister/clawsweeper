## Summary

Keep open. Current main still defaults session-backed task records to `done_only` unless a caller explicitly passes `notifyPolicy`; ACP and subagent creation sites do not pass a config-derived policy. Docs still advertise ACP/subagent defaults as `done_only` and only document per-task `openclaw tasks notify`. The related reports are adjacent notification-routing/cron/exec items, not a canonical implementation of this config-level default request.

## Best Possible Solution

Keep this issue open and implement a core config-backed task notify default. The likely path is adding a typed/schema-backed `agents.defaults.tasks` config section, resolving per-runtime defaults before ACP/subagent/taskflow task records are created, preserving current defaults when unset, then updating docs and focused tests around ACP/subagent creation, cron silence, and the existing `tasks notify` override.

## Risks / Open Questions

- A fix should preserve the current `done_only` default unless users opt into a config override, because current docs and tests encode that behavior.
- The config shape needs owner judgment: a single global default is simpler, while a runtime-specific map avoids changing cron/CLI/media semantics accidentally.
