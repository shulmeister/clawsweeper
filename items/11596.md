## Summary

Current main already formats gateway/subsystem console timestamps in local time, with system-timezone fallback and regression coverage; the changelog records the fix in release 2026.2.27.

## Evidence

- **Console timestamp formatter uses local time:** `formatConsoleTimestamp()` returns local `HH:MM:SS` for pretty logs and local ISO-with-offset for compact/json logs via `formatTimestamp()` / `formatLocalIsoWithOffset()`, not UTC `toISOString()`.
  - file: [src/logging/console.ts:179](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/logging/console.ts#L179)
- **Gateway/subsystem logs call the shared local-time formatter:** Subsystem console rendering routes both JSON and pretty/timestamp-prefix output through `formatConsoleTimestamp(...)`.
  - file: [src/logging/subsystem.ts:217](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/logging/subsystem.ts#L217)
- **Timezone resolution falls back to system timezone:** When no explicit timezone is passed, logging timestamp formatting uses `process.env.TZ` if valid, otherwise `Intl.DateTimeFormat().resolvedOptions().timeZone`, which matches the feature request to use local/system time.
  - file: [src/logging/timestamps.ts:17](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/logging/timestamps.ts#L17)
- **Regression tests cover local offsets including Asia/Shanghai:** Tests assert `Asia/Shanghai` formats as `+08:00`, and console timestamp tests assert local-time output for compact/json/pretty styles.
  - file: [src/logging/timestamps.test.ts:14](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/src/logging/timestamps.test.ts#L14)
- **Changelog records the shipped fix:** Release notes say subsystem console timestamps were changed to use local timezone formatting instead of inline UTC `toISOString()` paths.
  - file: [CHANGELOG.md:3728](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/CHANGELOG.md#L3728)

## Close Comment

Closing as implemented. This was a Codex review against current `main`, and gateway/subsystem console timestamps already use local-time formatting rather than UTC `toISOString()` paths: `src/logging/console.ts:179` formats pretty logs as local `HH:MM:SS` and compact/json logs as local ISO timestamps with offset, `src/logging/subsystem.ts:217` routes subsystem output through that helper, and `src/logging/timestamps.ts:17` falls back to the system timezone when no explicit timezone is provided. There is regression coverage for `Asia/Shanghai` in `src/logging/timestamps.test.ts:14`, and the changelog records this fix in release `2026.2.27` at `CHANGELOG.md:3728`. If you still see UTC timestamps on a current build, please open a fresh bug with the exact log surface and platform details.
