## Summary

Keep #72278 open. Current main implements useful low-level SecretRef batching, but it still only dedupes exact refs and batches by configured `source/provider`; the public docs still show 1Password/Vault as one exec provider command per secret with `id: "value"`, which is exactly the shape the issue says prevents cross-secret backend memoization. Related #17311 is adjacent and open, but it does not fully subsume this generic external-backend batching/memoization contract.

## Best Possible Solution

Keep the issue open and solve it in the core SecretRef integration model: define a backend-scoped external secret provider pattern (or extend exec/configure/docs to make that the default), use SecretRef ids as backend secret paths, batch all active refs per backend provider during startup/reload, and add pass-scoped in-memory memoization plus regression coverage. Coordinate with open #17311 for first-party provider work and #56217 for crash-loop/rate-limit behavior, but do not close #72278 as already implemented or purely plugin-scoped.
