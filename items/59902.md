## Summary

Keep open. The issue is a recent, plausible UI enhancement request, the author is not protected but the item is not stale, and current main still appears to make the History action only select/load the existing runs panel rather than scroll, focus, or open a dedicated history surface.

## Evidence

- **Issue asks for visible History feedback:** Issue #59902 says the /cron History button is easy to mistake as broken because it updates a lower run-history panel that may be outside the viewport, and proposes scrolling/focusing that section or opening a dedicated panel/modal.
- **Current cron view keeps history inline:** Current main renders the run history as an inline card below the job list, with the subtitle changing based on the selected job and the scope filter offering all jobs vs selected job.
  - file: [ui/src/ui/views/cron.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/ui/src/ui/views/cron.ts)
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **History button only loads runs:** In renderJob, the History button stops propagation and calls props.onLoadRuns(job.id); no scrollIntoView, focus, modal, drawer, or navigation behavior is evident there. The selected job gets a list-item-selected class, but that does not address the reported off-screen feedback problem with high confidence.
  - file: [ui/src/ui/views/cron.ts](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/ui/src/ui/views/cron.ts)
