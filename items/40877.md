## Summary

Keep #40877 open. Current main still contains the iOS permission snapshot behavior the PR targets: `currentPermissions()` synchronously calls `SFSpeechRecognizer.authorizationStatus()`, constructs a throwaway `CLLocationManager()`, and calls `CLLocationManager.locationServicesEnabled()` on the main-actor controller path. The PR is an active contributor bug-fix with recent author follow-up and review discussion, and related PR #51433 is narrower because it only covers the CoreLocation portion, not the speech-recognition permission path or the reconnect-race follow-ups introduced by making permission snapshots async.

## Best Possible Solution

Keep #40877 open for maintainer review. The best path is to keep the off-main speech permission snapshot and CoreLocation cleanup if validated, resolve the remaining stale async refresh guard by revalidating the full active gateway config including TLS pin state, keep/manual-connect reconnect semantics covered by focused iOS tests, and coordinate with #51433 as a narrower CoreLocation-only overlap rather than treating it as a superseding canonical PR.
