## Summary

Keep open. Current main supports only configured/static MCP HTTP headers. The remote MCP config, schema, transport resolver, and runtime client creation do not expose a per-request header resolver, headersProvider, templating against session context, or plugin/API seam that can generate headers before each MCP request. Related #65590 covers static streamable-http header forwarding and is distinct; related #62776 tracks OAuth-style auth flow gaps, not this broader per-request header capability.

## Best Possible Solution

Keep the issue open and design this as a core/runtime MCP capability, or as a public plugin API extension if maintainers want plugins to own header resolution. A good implementation would add a typed request-scoped header resolver for remote MCP transports, thread session/runtime context into the MCP request path, merge dynamic headers after static config with clear precedence and secret redaction, cover both streamable-http and SSE, and update docs/tests around tenant/user/token refresh behavior.
