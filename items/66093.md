## Summary

Keep open. Current main still has the reported auto-reply fallback contract gap: provider-scoped auth drops the authProfileId when fallback switches providers, the CLI fallback path passes that dropped value into runCliAgent, and CLI session reuse still invalidates stored/current authProfileId mismatches. The google-gemini-cli backend does not define a defaultAuthProfileId to repair the omission, and current tests still assert the profile-dropping behavior rather than covering google-gemini-cli fallback reuse or stale cliSessionBindings.

## Best Possible Solution

Keep this issue open. The maintainer fix should make auto-reply fallback auth planning resolve a provider-appropriate auth profile for CLI fallback providers without reintroducing #1156 cross-provider auth leakage, likely by sharing the newer runtime auth-plan semantics with the auto-reply path. Add regression coverage for google-gemini-cli fallback session reuse and stale cliSessionBindings where authProfileId is missing on one side.
