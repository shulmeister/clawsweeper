## Summary

Keep open. Current main still does not implement the PR’s central behavior: `resolveSessionAuthProfileOverride` rotates and persists raw `authProfileOverride` ids from the resolved order, with no canonicalization of legacy OAuth `provider:default` overrides to safer non-default OAuth aliases. The existing legacy-default helper is present, but current main only uses it for config/profile repair, not session override resolution. The PR discussion also shows concrete reviewer concerns were followed up with targeted fixes and regression tests, so this is still a plausible focused bug-fix PR rather than obsolete cleanup.

## Best Possible Solution

Keep PR #42585 open for maintainer review. The best path is to land or adapt its `session-override.ts` canonicalization on current main, using the existing legacy-default helper where appropriate, while preserving the follow-up safety constraints already discussed: OAuth-only remap, no cross-identity fallback, full agreement when both email and accountId are present, cooldown-aware selection, and rotation progress that does not loop back to the same canonical credential. Regression coverage should include the positive alias cases plus negative non-OAuth, mismatched identity, partial-conflict identity, and rotation-order cases.
