## Summary

Current main still has the reported behavior: `retryAsync` applies symmetric jitter after choosing a `retryAfterMs` base delay, then clamps only to `minDelayMs`/`maxDelayMs`, so adversarial jitter can still schedule below the server-supplied wait. The linked fix PR #68543 is still open, and the alternate PR #68600 was closed unmerged, so this issue should remain open until a fix lands.

## Best Possible Solution

Land a focused fix equivalent to PR #68543 or #68600: preserve symmetric jitter for plain exponential backoff, but when `retryAfterMs` is present, ensure the final scheduled delay cannot fall below `Math.max(retryAfterMs, minDelayMs)` while still respecting `maxDelayMs`. Add a deterministic regression test that mocks `generateSecureFraction()` to the lower-bound case and asserts `delayMs >= retryAfterMs`.
