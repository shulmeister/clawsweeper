## Summary

Current main has explicit Telegram duplicate-prevention logic and regression coverage for the main duplicate paths: finalizing preview replies in place instead of sending a second final message, avoiding duplicate fallback sends after ambiguous edit timeouts, and skipping replayed Telegram updates via persisted update offsets. The issue discussion also ended with a contributor unable to reproduce, and the report predates several Telegram duplicate-delivery fixes now present on main.
