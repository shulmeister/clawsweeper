## Summary

Current main still intentionally disables the Chrome extension relay, rejects legacy driver profiles, migrates old relay config to Chrome MCP existing-session, and documents replacements that do not satisfy the reported one-click cross-machine Chrome extension workflow. This should stay open for maintainer product/security judgment rather than closing as implemented, ClawHub-only, or duplicate.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The likely path is either restoring an opt-in relay inside the browser plugin with schema/docs/tests/security review, or first adding a narrow public browser transport/profile-driver capability so the linked ClawHosters relay or a ClawHub plugin can integrate with the existing browser tool without private browser internals.

## Risks / Open Questions

- Restoring an extension relay would reintroduce remote browser-control WebSocket behavior, so any fix needs explicit auth, origin, token-handling, exposure, and operator-consent review.
- A third-party plugin can register routes or a separate tool today, but without a typed browser profile-driver/transport API it would not cleanly wire into the existing browser tool profile contract.
- The related closed relay issues and PRs in the provided context cover old relay bugs, hardening attempts, or docs replacement; they do not provide an open canonical item or a shipped replacement for this cross-machine workflow.
