## Summary

Keep open. Current main still contains the reported Gemini CLI OAuth client-config extraction path: it searches local Gemini CLI install layouts, reads oauth2.js/bundled JS content, regex-extracts the client ID and GOCSPX secret, caches the result, and uses it as the fallback when explicit OAuth env vars are absent. The current onboarding warning asks for consent to use the unofficial Gemini CLI OAuth integration, but it does not specifically disclose or gate filesystem extraction of another installation's OAuth client config. The protect-mcp comment is useful broader credential-isolation input, but it does not replace a product/security decision for this bundled Google plugin behavior.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely code path to change is the bundled Google plugin: either remove automatic Gemini CLI source/bundle credential extraction by default, or require a narrowly worded explicit opt-in before any filesystem extraction. A fix should update extensions/google/oauth.credentials.ts, the Gemini OAuth tests, and the public Google/Gemini provider docs so the behavior and consent model are clear.
