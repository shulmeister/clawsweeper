## Summary

Keep open. The report is recent, has concrete reproduction details for OpenClaw 2026.3.31, and current documentation makes the observed unauthenticated curl response expected but does not prove the node-side `canvas.a2ui.*` failure is fixed on main.

## Evidence

- **Issue has actionable reproduction:** Issue #59022 reports a specific command, error code (`A2UI_HOST_UNAVAILABLE`), gateway version, macOS node, bind/auth setup, and curl result. It is not stale and has no maintainer resolution or duplicate link in the discussion.
- **Current docs explain auth behavior, not a fix:** The current configuration docs say `/__openclaw__/a2ui/` is served on the Gateway port, non-loopback canvas routes require Gateway auth, and paired nodes receive scoped capability URLs because node WebViews typically do not send auth headers. That explains the issue's `Unauthorized` curl result, but it leaves the reported node command failure plausible.
  - file: [docs/gateway/configuration-reference.md](https://github.com/openclaw/openclaw/blob/7bbd47349ec37f01a8f2b1dc219bf950615f361b/docs/gateway/configuration-reference.md)
  - sha: [7bbd47349ec3](https://github.com/openclaw/openclaw/commit/7bbd47349ec37f01a8f2b1dc219bf950615f361b)
- **Current Android README treats the symptom as a real operational failure mode:** The Android integration notes still list `A2UI host not reachable` / `A2UI_HOST_NOT_CONFIGURED` troubleshooting steps and require the app Screen tab/canvas readiness for A2UI checks, so the symptom is not obsolete enough to auto-close without reproduction proof.
