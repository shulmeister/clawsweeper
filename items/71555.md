## Summary

Keep this issue open. Current main still pins @mariozechner/pi-coding-agent 0.70.2 and still opens embedded transcripts through SessionManager.open before createAgentSession builds the session context. The pinned SDK source still fully reads/parses the JSONL in loadEntriesFromFile, calls that from SessionManager.open to inspect the header, and then loads the same file again from setSessionFile during construction. The newer maxActiveTranscriptBytes work is related, but it is opt-in, requires truncateAfterCompaction, and its compaction path also opens the large session through the same SDK path, so it does not fully resolve the reported default crash class.

## Best Possible Solution

Keep this issue open and fix it at the embedded Pi/session boundary. The best path is to upgrade or patch @mariozechner/pi-coding-agent so SessionManager.open reads only the header or reuses parsed entries, or add a narrow OpenClaw adapter that avoids the duplicate full-file parse while preserving Pi session semantics. Add focused regression or stress coverage around a large JSONL transcript with long tool results proving embedded startup/compaction returns a controlled result instead of exiting the gateway.
