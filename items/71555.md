## Summary

Keep open. Current main still sends embedded attempts through `SessionManager.open(params.sessionFile)` before any preflight truncation, remains pinned to `@mariozechner/pi-coding-agent` 0.70.2 with no local patch, and only prewarms a small file chunk rather than reusing a parsed SessionManager. The related stale-lock cleanup reports explain the `dead-pid` recovery symptom, but they do not fix the large-transcript crash path described in #71555.

## Best Possible Solution

Keep #71555 open and fix the embedded session load path. The best path is to avoid double full-file parsing for large transcripts: either upstream a `SessionManager.open`/header-only load improvement in pi-coding-agent and bump the pinned package, or add an OpenClaw-owned large-transcript path that reuses one parsed manager and performs preflight truncation/compaction before building the full LLM context. Add a regression with a synthetic large JSONL containing oversized tool results, and update docs/changelog if behavior or limits change.
