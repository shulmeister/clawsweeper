## Summary

Current `main` does not implement the requested `pairing list-approved` / `pairing revoke` CLI, and the existing channel status output still surfaces config-backed `allowFrom` rather than the merged pairing-store approvals the issue asks for.

## Evidence

- **Pairing CLI only exposes pending-list and approve:** `registerPairingCli` defines `pairing list` and `pairing approve` subcommands only; there is no `list-approved` or `revoke` command path on current `main`.
  - file: [src/cli/pairing-cli.ts:45](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cli/pairing-cli.ts#L45)
  - command: `nl -ba src/cli/pairing-cli.ts | sed -n '1,220p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Public docs match the limited CLI surface:** The user-facing pairing CLI docs list only `pairing list` and `pairing approve`, with no approved-sender listing or revoke command documented.
  - file: [docs/cli/pairing.md:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/cli/pairing.md#L16)
  - command: `nl -ba docs/cli/pairing.md | sed -n '1,140p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Core has store primitives but no CLI wiring:** The pairing store already supports reading merged approved entries and removing an allowlist entry (`readChannelAllowFromStore`, `removeChannelAllowFromStoreEntry`), which confirms the requested cleanup surface is not blocked by missing storage primitives but is also not exposed through the `pairing` CLI today.
  - file: [src/pairing/pairing-store.ts:375](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/pairing/pairing-store.ts#L375)

- **Status output still keys off `account.allowFrom`:** `channels status --probe` prints `allow:` from `account.allowFrom`, and common channel config adapters resolve that field from channel config (`account.allowFrom` / `account.config.allowFrom`) rather than directly from the pairing allow-store requested in the issue.
  - file: [src/commands/channels/status.ts:88](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/commands/channels/status.ts#L88)

## Risks / Open Questions

- I did not execute live pairing/status commands against a real user config because this cleanup task is read-only and should not mutate local pairing state.
- Some channel-specific status hooks may add extra snapshot fields, but the generic `channels status` formatter still only renders `account.allowFrom`, so the requested merged approved-sender visibility is not implemented as a general CLI feature.
