## Summary

Keep open. Current `main` still does not apply trusted env-proxy mode to memory remote fetches, so this is not implemented on main. The PR’s current approach is also too broad for conservative cleanup because it keys off `hasProxyEnvConfigured()`, while current main now uses narrower HTTP(S)-proxy and `NO_PROXY` guards elsewhere to avoid bypassing pinned-DNS/SSRF protections.

## Evidence

- **Memory remote fetch still uses strict default on main:** `withRemoteHttpResponse()` on current `main` still calls `fetchWithSsrFGuard({...})` without setting `mode`, so memory remote requests do not opt into trusted env-proxy mode. See `packages/memory-host-sdk/src/host/remote-http.ts:29` and mirrored source `src/memory-host-sdk/host/remote-http.ts:14`.
  - file: [packages/memory-host-sdk/src/host/remote-http.ts:29](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/packages/memory-host-sdk/src/host/remote-http.ts#L29)

  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)

See `src/infra/net/fetch-guard.ts:357-374`. That means the memory helper would need to set `mode` to get the intended behavior, and current main does not.
