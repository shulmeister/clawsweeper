## Summary

Keep open. Current main still applies symmetric jitter after selecting a server-supplied `retryAfterMs`, so a low jitter sample can schedule below the Retry-After lower bound. Existing retry-after tests only cover `jitter: 0`, while real channel/Discord retry runners pass Retry-After values through retry helpers that use nonzero jitter defaults. The related issue #68541 remains open and related PR #68600 is closed unmerged, so #68543 is not obsolete.

## Best Possible Solution

Keep #68543 open for maintainer review. The best product/code path is to land this PR or an equivalent core fix in `src/infra/retry.ts` that preserves server-supplied Retry-After as a lower bound when it is satisfiable, keeps spread for unsatisfiable capped cases, and adds focused regression tests for jitter undercut, cap equality, `retryAfterMs > maxDelayMs`, and non-integer Retry-After values. Then close #68541 as fixed.
