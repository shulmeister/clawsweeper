## Summary

Keep open. Current main still does monolithic, size-proportional `sessions.json` updates on the active session path, and the fields called out by the report remain inline per session. Existing maintenance knobs mitigate stale/oversized stores but do not solve repeated inline `skillsSnapshot` data, per-update full-file parse/stringify/write, sharding, SQLite, or off-critical-path persistence.

## Best Possible Solution

Keep this issue open and fix the core session metadata storage path. A narrow first step is to deduplicate or content-address `skillsSnapshot`; the more complete fix is per-session metadata sharding or SQLite-backed metadata plus moving expensive writes off the run-critical path. Add regression coverage for a large active `sessions.json` under `maxEntries` with repeated snapshots so updating one session no longer scales with whole-store size or consumes the agent wait/run budget.
