## Summary

Keep #71227 open. Current main still stores `skillsSnapshot` inline in each session entry and the write path still re-reads/parses and serializes the whole `sessions.json` for updates. Existing session maintenance is a useful mitigation, but it is retention/count/disk-budget based and does not implement the requested structural fix for large active stores or repeated snapshot payloads. Related #69451 remains open and broader; it does not supersede this latency-specific report.

## Best Possible Solution

Keep the issue open and address it in core session persistence. The strongest product/code direction is to stop putting repeated large `skillsSnapshot` blobs inline, preferably with content-addressed snapshot files referenced from session entries, and/or shard session metadata so one session update does not parse and rewrite unrelated entries. Preserve the existing maintenance controls as fallback cleanup, add migration/backward-compatibility coverage for existing inline stores, and add a regression/performance test around a large `sessions.json` with repeated snapshots.

## Risks / Open Questions

- A retention-only fix can still leave large active stores slow when many recent sessions share repeated skills snapshots.
- Changing session persistence needs migration/backward-compatibility care because `skillsSnapshot` is part of persisted session entries and older stores must keep loading.
