## Summary

Keep open. Current main has substantial built-in protection for the reported unsafe config writes, including full post-change config validation, SecretRef preflight, and an agent-facing config mutation allowlist, but it still does not implement the requested synchronous `config:pre-apply` hook or equivalent plugin API that receives current and pending config and can block direct `config.apply` / `config.patch` writes. The issue discussion also now contains an explicit maintainer/Codex review comment keeping this open, and no protected-label or author-association rule requires closure.

## Best Possible Solution

Keep #9993 open as a plugin API/product-direction issue. The best implementation would add a documented, synchronous config-write interception point invoked by `config.apply` and `config.patch` after the pending config is computed and before persistence/restart scheduling, with current config, pending config, method/patch metadata, and a typed reject result. It should cover direct gateway RPC writes rather than only the model-facing gateway tool, include hook/API docs, and add focused tests proving a plugin can reject a bad config write without changing the active config. If maintainers decide the built-in guard is sufficient, the issue should be explicitly closed or retitled by maintainer judgment rather than automated cleanup.
