## Summary

Keep #10872 open. Current main still only handles iMessage reply context when imsg already provides resolved `reply_to_*` fields; it does not model or resolve `thread_originator_guid` / `reply_to_guid` into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`. The issue discussion and open PR #42683 identify this exact remaining Layer 3 gap, so the request is not implemented or obsolete.

## Best Possible Solution

Keep the issue open until the bundled iMessage plugin resolves imsg reply GUID metadata such as `thread_originator_guid` or `reply_to_guid` into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The likely maintainer path is in `extensions/imessage/src/monitor/*`: add a narrow resolver via documented imsg RPC support or a scoped read-only chat.db lookup when `dbPath` is configured, then add parser/monitor regression tests for GUID-only inline replies. PR #42683 can be used as the starting point or superseded by a maintainer-approved replacement.
