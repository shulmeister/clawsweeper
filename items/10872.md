## Summary

Keep #10872 open. Current main still only forwards iMessage reply context when the imsg payload already contains resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`; it does not model or resolve `thread_originator_guid` / `reply_to_guid`, and there is no chat.db fallback before dispatch. The relevant iMessage monitor surface is unchanged from the latest release tag, while the discussion and related PR #42683 show the remaining work is real but not shipped.

## Best Possible Solution

Keep the issue open until the bundled legacy iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The likely path is in `extensions/imessage/src/monitor/*`: add a narrow resolver using supported imsg metadata/RPC or a scoped read-only chat.db lookup when `dbPath` is configured, guard remote/SSH-wrapper cases, and add regression tests for GUID-only inline replies. PR #42683 is a useful starting point but still needs refresh/maintainer review before it can close this issue.
