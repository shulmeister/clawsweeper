## Summary

Keep #10872 open. Current main still only forwards iMessage reply context when `imsg` has already supplied resolved `reply_to_*` fields; it does not model or resolve the `thread_originator_guid` field that the issue discussion identifies as the remaining imsg v0.5.0 path. BlueBubbles has a GUID fallback and is recommended for new iMessage setups, but that is not the same as fixing the legacy imsg channel requested here.

## Best Possible Solution

Keep the issue open until `extensions/imessage` models `thread_originator_guid` and resolves it into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` through a documented imsg RPC or a carefully scoped chat.db lookup, with parser and inbound-processing regression tests. If maintainers decide legacy imsg should not gain this behavior, that should be an explicit product decision in favor of the BlueBubbles path rather than an automated cleanup close.
