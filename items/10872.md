## Summary

Keep open. Current main still only consumes already-resolved iMessage `reply_to_*` fields and has no handling for the `thread_originator_guid` field that the issue/comment identify as the available upstream signal. The linked general transform-hook issue does not supersede this channel-specific gap.

## Evidence

- **Current reply-context extraction requires resolved quote text:** `describeReplyContext()` returns null unless `message.reply_to_text` is present, then copies only `reply_to_id` and `reply_to_sender`; there is no fallback path from a GUID to the replied-to message.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:60](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/imessage/src/monitor/inbound-processing.ts#L60)
  - command: `nl -ba extensions/imessage/src/monitor/inbound-processing.ts | sed -n '50,80p;370,390p;548,565p;626,638p'`
  - sha: [1dfa52d0711d](https://github.com/openclaw/openclaw/commit/1dfa52d0711dfc86e1c04538fe46e3d43961766f)
- **Inbound parser only recognizes reply_to fields:** The notification parser validates and normalizes `reply_to_id`, `reply_to_text`, and `reply_to_sender`, but does not declare or normalize `thread_originator_guid`. A targeted search for `thread_originator_guid|threadOriginator` under `extensions/imessage`, `src/channels`, and iMessage docs returned no matches.
  - file: [extensions/imessage/src/monitor/parse-notification.ts:68](https://github.com/openclaw/openclaw/blob/1dfa52d0711dfc86e1c04538fe46e3d43961766f/extensions/imessage/src/monitor/parse-notification.ts#L68)

- **Payload type lacks the upstream GUID field:** `IMessagePayload` includes `reply_to_id`, `reply_to_text`, and `reply_to_sender`, but no `thread_originator_guid`, so the imsg v0.5.0 signal is not modeled in the channel payload type.
