## Summary

Keep open. Current `main` still only propagates iMessage reply context when the inbound payload already includes resolved `reply_to_id` / `reply_to_text` / `reply_to_sender` fields. There is no `thread_originator_guid` field in the iMessage payload type, no parser support for it, and no fallback lookup path to resolve a replied-to message from `chat.db` or an RPC helper.

## Evidence

- **Inbound iMessage payload shape lacks `thread_originator_guid`:** `IMessagePayload` only declares `reply_to_id`, `reply_to_text`, and `reply_to_sender` for reply metadata; there is no field for `thread_originator_guid` or any other reply GUID.
  - file: [extensions/imessage/src/monitor/types.ts:10](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/monitor/types.ts#L10)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Notification parser only accepts resolved reply fields:** `parseIMessageNotification()` validates and normalizes `reply_to_id`, `reply_to_text`, and `reply_to_sender`. It does not read or preserve `thread_originator_guid`, so current `main` cannot use the upstream imsg v0.5.0 reply GUID field even if it arrives.
  - file: [extensions/imessage/src/monitor/parse-notification.ts:60](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/monitor/parse-notification.ts#L60)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Reply context extraction still hard-requires `reply_to_text`:** `describeReplyContext()` returns `null` unless `message.reply_to_text` is present. Downstream inbound context formatting and `ReplyTo*` payload fields are populated only from that derived `replyContext`, so a payload that has only a reply GUID still produces no thread context.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:60](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/monitor/inbound-processing.ts#L60)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **Current tests cover only resolved reply fields:** The iMessage monitor tests verify reply context behavior by constructing messages with `reply_to_id`, `reply_to_text`, and `reply_to_sender`. There is no coverage for `thread_originator_guid` or any fallback resolution path.
  - file: [extensions/imessage/src/monitor.gating.test.ts:162](https://github.com/openclaw/openclaw/blob/86f69ba5a06ae256906c3a26308801e0b84e0091/extensions/imessage/src/monitor.gating.test.ts#L162)
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)
- **No lookup implementation exists in the iMessage extension:** A repository search for `thread_originator_guid`, `message.get`, or SQLite-based reply resolution under `extensions/imessage/src` found no implementation that could resolve reply context from a GUID or directly query `chat.db`.
  - command: `rg -n "thread_originator_guid|message\.get|sqlite3|sqlite" extensions/imessage/src`
  - sha: [86f69ba5a06a](https://github.com/openclaw/openclaw/commit/86f69ba5a06ae256906c3a26308801e0b84e0091)

## Risks / Open Questions

- This was a code inspection only; I did not run a live macOS/imsg reproduction in the read-only review environment.
- If upstream imsg now emits fully resolved `reply_to_*` fields in some configurations, OpenClaw would handle those, but current `main` still has no explicit fallback for the GUID-only case described in the issue comment.
