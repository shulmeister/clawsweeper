## Summary

Keep open. Current main still does not show a fix for the reported macOS app launch-at-login persistence/update drift. The UI state is derived from live launchd loaded status, the app has no separate persisted intended launch-at-login setting, and the Sparkle updater path does not re-register or reconcile the desktop LaunchAgent after updates. The related #40213 fix only removed `KeepAlive` for manual quit behavior, and #28423 was a different gateway LaunchAgent entrypoint issue.

## Best Possible Solution

Keep #39588 open and fix the macOS desktop launch-at-login contract directly. The likely product/code path is to persist the user's intended launch-at-login preference separately from live launchd loaded status, distinguish disabled from installed-but-unloaded/stale LaunchAgent states, reconcile or re-bootstrap the desktop app LaunchAgent after Sparkle updates and app startup, and add focused tests for stale/unloaded plist state so the UI cannot turn an update-time launchd failure into a silent disable.
