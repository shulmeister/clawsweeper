## Summary

Keep open. Current main has relevant MiniMax payload and WSL2/Undici transport mitigations, but they do not directly prove that the reported MiniMax CN 10+ second per-message preprocessing delay is fixed. The issue is recent, provider/model-specific, and lacks either a current-main non-repro or a latency regression test tied to this symptom.

## Best Possible Solution

Keep the issue open until a maintainer can reproduce or disprove it on current main with MiniMax CN API-key and OAuth paths on WSL2. The useful next step is to capture timings for `chat.send` receipt, context/history assembly, lifecycle start, model-call payload send, first response bytes, and first visible assistant text; then fix the implicated MiniMax wrapper/provider/transport path and add a focused regression test or diagnostic doc. If current main no longer reproduces with that timing trace, close it then as implemented with the non-repro evidence.

There are no issue comments clarifying root cause or confirming a later fix.
