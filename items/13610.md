## Summary

Keep #13610 open. Current main has the SecretRef baseline for env/file/exec providers, CLI audit/configure/apply/reload support, and docs for wrapping tools such as Vault through exec, but it still does not implement native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager provider sources. The issue discussion also explicitly preserved this as the umbrella after #26155 shipped the generic baseline.

## Best Possible Solution

Keep #13610 open as the native secrets-manager provider umbrella. The maintainer path is to either add first-class AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support across SecretRef source types, provider config schema, resolver dispatch, CLI configure/audit/apply flows, docs, and tests, or make an explicit product/API decision that exec wrappers plus a public secret-provider plugin/ClawHub API are the supported route and link narrower follow-up issues for that work.

## Risks / Open Questions

- Exec providers can operationally wrap Vault, 1Password, SOPS, cloud CLIs, or custom resolvers, so some deployments have a workaround today; that still does not provide native provider source types, provider-specific schema, SDK dispatch, caching/audit semantics, CLI UX, and tests for AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager.
