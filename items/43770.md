## Summary

Keep open. Current main does not implement the PR's passthrough behavior: plugin entries remain strict, and plugin runtime/config validation only consumes `plugins.entries.<id>.config`. The PR also should not be merged as-is, because accepting arbitrary top-level keys would make those keys validate but still be ignored by normalization/loading, matching the existing review concern.

## Best Possible Solution

Keep the PR open for maintainer action, but require revision before merge. The clean path is to preserve the documented `plugins.entries.<id>.config` contract and improve diagnostics/migration for top-level plugin keys, or explicitly design entry-level aliases by updating the Zod/schema contract plus normalization and loader behavior so accepted keys reach `api.pluginConfig`, with regression tests that prove both validation and runtime retention.
