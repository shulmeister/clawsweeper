## Summary

Keep PR #38687 open. Current main still resolves and injects an env-proxy fetch before the media backend baseUrl is known, so NO_PROXY and loopback targets are not bypassed at the runner/proxy-fetch boundary this PR is trying to fix. Main has a reusable NO_PROXY matcher now, but it is used by guarded-fetch mode selection, not by resolveProxyFetchFromEnv or runProviderEntry's fetchFn injection.

## Best Possible Solution

Keep this open and implement the target-aware behavior on current main by resolving the provider baseUrl before choosing fetchFn, then reusing shouldUseEnvHttpProxyForUrl/matchesNoProxy or adding a targetUrl parameter to resolveProxyFetchFromEnv. The final patch should avoid passing an EnvHttpProxyAgent fetch wrapper for loopback and NO_PROXY-covered media backends, while preserving proxy routing for public/non-excluded backends, with focused runner/proxy-fetch regression tests.
