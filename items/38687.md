## Summary

Keep open. Current main does not clearly implement this PR’s requested behavior of skipping the proxy-aware fetch for NO_PROXY-covered internal audio backends. The media runner still injects `resolveProxyFetchFromEnv()` unconditionally when proxy env vars are present, and the current tests do not cover an audio backend whose `baseUrl` is excluded by `NO_PROXY`.

## Evidence

- **Audio runner still injects env-proxy fetch unconditionally:** `runProviderEntry` resolves `fetchFn = resolveProxyFetchFromEnv()` before the audio/video capability branches and passes that fetch into audio transcription calls without considering `baseUrl` or `NO_PROXY` at this layer.
  - file: [src/media-understanding/runner.entries.ts:573](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/media-understanding/runner.entries.ts#L573)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Proxy helper is env-based, not target-aware:** `resolveProxyFetchFromEnv` only accepts an env object and always builds an `EnvHttpProxyAgent` when HTTP(S) proxy env is configured; it has no target URL parameter and no media-backend-specific bypass decision.
  - file: [src/infra/net/proxy-fetch.ts:55](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/infra/net/proxy-fetch.ts#L55)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Current runner regression tests do not cover NO_PROXY audio bypass:** The media runner proxy tests verify that audio/video providers receive a proxy fetch when `HTTPS_PROXY` is set and `undefined` when proxy env vars are blank, but there is no current-main test for an audio backend whose configured `baseUrl` is covered by `NO_PROXY`.
  - file: [src/media-understanding/runner.proxy.test.ts:104](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/media-understanding/runner.proxy.test.ts#L104)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **NO_PROXY handling exists elsewhere, but on a different path:** `matchesNoProxy` and the guarded-fetch tests show main has NO_PROXY-aware logic for the shared provider HTTP helper auto-upgrade path, which is related context but not proof that the media runner’s explicit `fetchFn` injection path already solves this PR’s reported multipart regression.
  - file: [src/media-understanding/shared.test.ts:593](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/src/media-understanding/shared.test.ts#L593)
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)

## Risks / Open Questions

- Closing this PR would assume undici `EnvHttpProxyAgent` fully resolves the reported multipart-audio regression for NO_PROXY-covered backends, but current main has no focused media transcription regression test proving that behavior.
- A nearby shipped fix exists for NO_PROXY-aware trusted env-proxy auto-upgrade in shared provider HTTP helpers (`CHANGELOG.md:664`), but that is not the same code path as the unconditional `resolveProxyFetchFromEnv()` injection in `src/media-understanding/runner.entries.ts`.
