## Summary

Current main still does not bridge tool execution into the internal HOOK.md handler system. Tool-call observation remains implemented and documented as a plugin hook surface, while internal hooks only expose command/session/agent/gateway/message events. The PR is not obsolete: its discussion shows substantive follow-up on the earlier syntax, duplicate-emission, adjusted-params, and sessionKey review issues, and related maintainer hook-boundary work is still open rather than shipped.

## Best Possible Solution

Keep this PR open until maintainers either merge a corrected internal tool-hook bridge or explicitly decide that operator-installed internal hooks should stay command/session/agent/gateway/message-only. If implemented, the best path is a narrow typed tool:after_call internal event emitted once from handleToolExecutionEnd using the stable sessionKey and the same adjusted params used by the plugin after_tool_call event, with docs and focused regression coverage.
