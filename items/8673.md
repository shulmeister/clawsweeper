## Summary

Keep #8673 open. Current main still does not implement bounded retry for ordinary transient OAuth refresh failures: the shared refresh path calls the provider/plugin refresh once, the manager adds locking/timeouts/adoption fallbacks but not generic retry, and the only retry-like branch is specific to refresh_token_reused recovery. The related retry PRs in the provided context are closed unmerged, and the latest maintainer review comment also kept this open.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the OAuth refresh operation. Prefer reusing src/infra/retry.ts with a provider-safe transient-error classifier; preserve current lock/adoption/fallback/reused-token behavior; add focused auth-profile tests for success after a transient refresh failure, retry exhaustion preserving the final error, and non-retryable OAuth failures such as invalid_grant.

## Risks / Open Questions

- A retry implementation needs a narrow transient-error classifier so invalid_grant, revoked refresh tokens, and other permanent OAuth failures are not retried as temporary failures.
- Retry placement must preserve the current global refresh locks, hard refresh timeout, external/main-store credential adoption, provider-plugin refresh hooks, and refresh_token_reused recovery.
- Provider plugins can own custom refresh behavior via refreshOAuth, so a shared policy should avoid surprising double-retry semantics for providers that already retry internally.
