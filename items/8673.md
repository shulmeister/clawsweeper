## Summary

Keep #8673 open. Current main still performs the built-in OAuth token refresh as a single call with timeout/locking and narrow `refresh_token_reused` recovery, but no general retry/backoff for transient network, timeout, 429, or 5xx refresh failures. The related fix PR #45975 is still open/unmerged in the provided context, and the older PR #8677 was closed unmerged.

## Best Possible Solution

Implement a bounded, tested retry policy around the actual OAuth refresh operation on the current auth path, preferably reusing `src/infra/retry.ts`. It should retry clearly transient network, timeout, rate-limit, and 5xx failures; preserve immediate handling for permanent OAuth failures such as invalid grants; keep the existing `refresh_token_reused` adoption path intact; and add focused tests under `src/agents/auth-profiles/` for success after transient failure, retry exhaustion preserving the final error, and non-retryable auth failures.
