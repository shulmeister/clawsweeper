## Summary

Keep #8673 open. Current main still lacks bounded retry for ordinary transient OAuth refresh failures: the shared OAuth path calls the upstream refresh API once, the manager adds locking/timeouts/adoption paths but not generic retry, and the only second-attempt path is the refresh_token_reused recovery. The two linked retry PRs, #8677 and #45975, were closed unmerged, and the latest issue discussion also concluded the item remains relevant.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the existing OAuth refresh operation. Prefer reusing src/infra/retry.ts, add a transient-error classifier, avoid retrying permanent auth failures, preserve current lock/adoption/reused-token recovery behavior, and add focused tests under src/agents/auth-profiles for success after a transient failure, retry exhaustion preserving the final error, and non-retryable OAuth failures.
