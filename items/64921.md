## Summary

Keep open. Current main still routes plugin text-transform resolution through the indirect createRequire-based active runtime registry helper, with no direct getActivePluginRegistry path in src/plugins/text-transforms.runtime.ts. Plugin registration itself is present, and docs promise prompt/message rewriting, but existing tests only cover registration/merge behavior rather than main-process application. The related PR #64924 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep this issue open and review or land a current implementation candidate such as #65331, or an equivalent patch, that makes runtime text-transform resolution read the active plugin registry through a direct main-process-safe seam while preserving external-plugin compatibility. Add regression coverage that registers a plugin textTransform and proves it is applied through the main CLI/provider prompt path.
