## Summary

Keep open. Current `main` still intentionally treats `payload.model` for isolated cron jobs as subject to the `agents.defaults.models` allowlist, and both docs and tests encode that behavior. The PR would change product semantics, not just fix an implementation gap, and the review discussion already identified an unaddressed regression in the proposed catalog check.

## Risks / Open Questions

- Merging this PR as-is would bypass the documented allowlist contract specifically for cron jobs, creating inconsistent behavior across `/model`, session overrides, and cron execution.
- The reviewed `endsWith` catalog check can misclassify cross-provider model ids and move failures from early validation to inference-time runtime errors.
- Because current docs and tests both encode the existing behavior, this needs explicit maintainer/product judgment rather than conservative auto-close or auto-land handling.
