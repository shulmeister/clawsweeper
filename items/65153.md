## Summary

Keep open. Current `main` still intentionally blocks Telegram media downloads when `api.telegram.org` resolves to a private/internal/special-use address unless the operator explicitly enables the Telegram-specific bypass. The repo now documents that workaround, but I did not find evidence that the reported default behavior was changed or that the referenced follow-up UX fix landed on `main`.

## Evidence

- **SSRF guard still throws the same generic resolved-IP block:** The core SSRF path still raises `Blocked: resolves to private/internal/special-use IP address` when a resolved address is classified as private/special-use; no Telegram-specific remediation is attached in this code path.
  - file: [src/infra/net/ssrf.ts:202](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/src/infra/net/ssrf.ts#L202)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '180,260p'`
  - sha: [a5db42862d76](https://github.com/openclaw/openclaw/commit/a5db42862d76fce18e5d7fa57f3907a543e61137)
- **Telegram media fetch only allows private-network targets via explicit opt-in:** `buildTelegramMediaSsrfPolicy` adds `allowPrivateNetwork: true` only when `dangerouslyAllowPrivateNetwork` is set. Otherwise Telegram media downloads keep the default resolved-IP SSRF enforcement.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:47](https://github.com/openclaw/openclaw/blob/a5db42862d76fce18e5d7fa57f3907a543e61137/extensions/telegram/src/bot/delivery.resolve-media.ts#L47)
