## Summary

Keep #69546 open. Current main still appears to have the reported QQBot stale-binding path: QQBot resolves inbound routes using the `cfg` captured when the channel account starts, while top-level `bindings` changes are explicitly treated as dynamic/no-restart config. The generic router would correctly prefer the peer binding if it received the updated config, but the QQBot gateway does not appear to refresh that config for each inbound message.

## Best Possible Solution

Keep the issue open and fix QQBot inbound routing to evaluate against the current runtime config when handling each message, or otherwise give channel plugins a refreshed config getter through the supported runtime surface. Add a regression test that starts QQBot with an account binding, updates top-level `bindings` with a peer-specific route, and verifies the next inbound C2C route matches `binding.peer` without a gateway restart.
