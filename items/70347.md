## Summary

Keep open. Current main still has the cron outer timeout path abort and reject from `executeJobCoreWithTimeout()` without emitting a terminal lifecycle event or carrying the active run session identity into that timeout callback. The gateway/session persistence path exists, but it is only driven by lifecycle start/end/error events that arrive on the agent event bus, so the specific synchronous finalization gap described in #70347 remains plausible and unaddressed.

## Best Possible Solution

Keep #70347 open and fix it in the cron isolated-run timeout path. The likely maintainer path is to carry the active cron run session identity into the timeout/finalization layer, emit a terminal lifecycle event with timeout metadata (`phase: error` or an explicitly timed-out end event, `errorKind`/`stopReason: timeout`), and add regression coverage that the matching run session row is finalized without waiting for gateway maintenance/watchdog repair. This should remain distinct from #43141's queue-abort work.
