## Summary

Keep open. Current `main` still caches bundle-MCP runtimes per OpenClaw session, routes SSE MCP tool calls through the cached client without stale-session recovery, and only recreates the runtime after explicit disposal or config/workspace changes. That matches the reported failure mode after an SSE-backed MCP server restart.

## Evidence

- **SSE MCP configs still use the SDK SSE client transport:** `resolveMcpTransport()` maps `transportType: "sse"` to `new SSEClientTransport(...)` with no OpenClaw-side reconnect wrapper around it.
  - file: [src/agents/mcp-transport.ts:117](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/mcp-transport.ts#L117)

- **Per-session MCP runtime is reused until explicitly disposed:** `getOrCreate()` returns the existing runtime whenever `sessionId`, `workspaceDir`, and config fingerprint match. It does not check downstream server liveness before reusing the cached runtime.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:367](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/pi-bundle-mcp-runtime.ts#L367)

- **Tool execution forwards directly to the cached MCP client:** `callTool()` looks up the cached session and directly calls `session.client.callTool(...)`. There is no catch/dispose/reconnect path for stale SSE sessions or `Session not found` responses.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:325](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/pi-bundle-mcp-runtime.ts#L325)

  - file: [src/agents/pi-bundle-mcp-materialize.ts:106](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/pi-bundle-mcp-materialize.ts#L106)
