## Summary

Keep open. Current main still appears to reuse a cached session-scoped MCP Client/transport for unchanged `mcp.servers` config, and tool execution forwards directly to that cached client without detecting a downstream SSE stale-session error such as `Session not found`, disposing the cached runtime, reconnecting, and retrying. Related ClawSweeper title matches are different stale-session surfaces and do not provide a canonical duplicate for this SSE-backed MCP restart failure.

## Best Possible Solution

Implement bounded stale-session recovery in the session-scoped bundle-MCP runtime: detect downstream SSE/HTTP stale-session failures such as `Session not found` or failed POST-to-endpoint responses, dispose the affected cached server session/runtime, recreate the MCP client/catalog for that session, and retry the failed tool call once. Add a focused regression test that simulates an established SSE-backed MCP session failing after server restart and verifies the next call reconnects instead of requiring a gateway restart.
