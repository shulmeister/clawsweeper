## Summary

Keep open. Current main still does not implement the PR’s client-side stale chat-run recovery or an equivalent `agent.wait` watchdog. The Control UI still treats any `chatRunId` as busy, queues later sends behind it, and only clears that state from terminal chat events or reconnect cleanup. The linked bug #42213 therefore remains plausibly unfixed on main, while the PR is a focused UI fix with useful tests but review comments that should be addressed before merge.

## Best Possible Solution

Keep #42258 open and have the author or maintainer land a refined version of the watchdog or an equivalent Control UI recovery path. The implementation should track chat-run activity, schedule an idle `agent.wait` probe for the active `chatRunId`, clear/reload/flush only when the gateway reports a terminal result, retry without user-visible noise on timeout/transient probe failure, clean up timers on disconnect/reconnect, and keep focused regression coverage in the Control UI chat tests.
