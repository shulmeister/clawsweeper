## Summary

Keep open. Current main still lets failed tool/toolResult text become the subagent completion fallback: the extractor copies tool output without checking top-level `isError`, block-level `is_error`, or `details.status` error/timeout markers, then stores that raw text as completion output and can send it through requester-visible completion fallback delivery. The direct fix PR #39367 was closed unmerged; the adjacent merged hardening in #59649 covers task/status/followup sanitization surfaces, not this extractor branch.

## Best Possible Solution

Keep the issue open and implement a narrow extractor fix in `src/agents/subagent-announce-output.ts`: detect failed tool results before populating `latestRawText`, reuse or mirror `isToolResultError` semantics plus `isError`/`is_error` handling, and add focused regression tests proving failed tool results are ignored while successful tool-only completion fallback still works.

## Risks / Open Questions

- A fix must preserve the intentional successful tool-result fallback while dropping only failed tool results.
- The guard should cover marker shapes present across the codebase: top-level `isError`, content/block-level `is_error`, and status-based failures such as `details.status: "error"` or `"timeout"`.
- If a subagent emits debugging as normal assistant text rather than failed tool output, that is a broader final-output separation problem and may need a separate follow-up.
