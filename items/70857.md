## Summary

Keep this issue open. Current main still contains the reported lock-held synchronous session-store reread path, and the exact watchdog warning remains active. The related open PR #68554 is the direct implementation candidate; #71903 addresses separate session-lock orphan hardening and should not close this active long-held store-lock report by itself.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes redundant skipCache disk rereads from session-store mutation helpers while preserving cache coherency and serialized writes. Add focused regression or performance coverage where feasible, then verify startup and RPC responsiveness on current main with a large session store before closing. Treat #71903 as related lock hardening, not the closure path for this report.
