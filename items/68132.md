## Summary

Keep open. Current `main` still falls back from missing workflow paths to pipeline parsing, so the underlying bug from #68101 is not implemented on `main`, and this PR still has an unresolved review concern about regressing valid inline pipelines that end with workflow-like filenames.

## Evidence

- **Current main still swallows workflow resolution errors:** `detectWorkflowFile()` on current `main` catches every `resolveWorkflowFile()` failure and returns `null`, which matches the reported fallback behavior instead of surfacing `ENOENT` or other path errors.
  - file: [extensions/lobster/src/lobster-runner.ts:199](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/lobster/src/lobster-runner.ts#L199)
  - command: `nl -ba extensions/lobster/src/lobster-runner.ts | sed -n '170,250p'`
  - sha: [c2139635ff82](https://github.com/openclaw/openclaw/commit/c2139635ff82b367b734ea052487258b1b33f060)
- **Current main still falls through to pipeline execution:** When `detectWorkflowFile()` returns `null`, the runner immediately calls `runToolRequest({ pipeline, ctx })`, so a missing path like `lobster/... .lobster` is still treated as an inline pipeline on current `main`.
  - file: [extensions/lobster/src/lobster-runner.ts:283](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/extensions/lobster/src/lobster-runner.ts#L283)

- **Issue report matches current code path:** Issue #68101 reports that a relative workflow path such as `lobster/haiku-ppc64-hidden-alias-autofix-once.lobster` is misreported as `Unknown command ...` after fallback to pipeline parsing. That is consistent with the current `detectWorkflowFile()` plus `runToolRequest({ pipeline })` flow on `main`.
