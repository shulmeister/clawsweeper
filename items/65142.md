## Summary

Current `main` no longer shows a gateway-startup path that synchronously scans all skills before serving requests. Gateway HTTP binds before post-attach sidecars finish, skills RPCs are not startup-blocked, and skill discovery happens on demand during agent runs or explicit `skills.*` calls.

## Evidence

- **Gateway serves before sidecars finish:** `startListening()` happens before post-attach startup work, and the code explicitly documents that HTTP is live while startup sidecars are still settling.
  - file: [src/gateway/server.impl.ts:812](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/server.impl.ts#L812)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '808,860p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Skills methods are not startup-gated:** Only `chat.history` and `models.list` are marked unavailable during startup. The skills RPCs remain in the method list, so they are not blocked behind startup sidecars.
  - file: [src/gateway/server-startup-unavailable-methods.ts:1](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/server-startup-unavailable-methods.ts#L1)
  - command: `nl -ba src/gateway/server-startup-unavailable-methods.ts && nl -ba src/gateway/server-methods-list.ts | sed -n '60,75p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Skill snapshot builds on demand during agent runs:** Skill snapshot generation is triggered from `agent-command` only when a session is new or its cached snapshot is stale; this is not part of gateway boot.
  - file: [src/agents/agent-command.ts:578](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/agents/agent-command.ts#L578)
  - command: `nl -ba src/agents/agent-command.ts | sed -n '576,615p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Embedded runner only loads skill entries when needed:** The embedded runner loads workspace skill entries only if no resolved skill snapshot is already present, which confirms lazy runtime loading instead of unconditional startup scanning.
  - file: [src/agents/pi-embedded-runner/skills-runtime.ts:14](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/agents/pi-embedded-runner/skills-runtime.ts#L14)
  - command: `nl -ba src/agents/pi-embedded-runner/skills-runtime.ts | sed -n '1,80p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Explicit skills scans occur inside request handlers:** `skills.status`/`skills.bins` derive skill data when those handlers run. The scan entrypoint is `loadWorkspaceSkillEntries`, not gateway startup.
  - file: [src/gateway/server-methods/skills.ts:95](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/gateway/server-methods/skills.ts#L95)
  - command: `nl -ba src/gateway/server-methods/skills.ts | sed -n '90,140p' && nl -ba src/agents/skills/workspace.ts | sed -n '836,853p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)

## Risks / Open Questions

- No live Windows reproduction was run; this conclusion is based on current `main` code paths because the review was read-only.
- If a remaining slowdown exists, it is more likely in on-demand skill enumeration or a separate startup sidecar/plugin path than in a mandatory pre-response gateway-wide skill scan.
