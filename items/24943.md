## Summary

Keep #24943 open. Current main still does not strip general Markdown before iMessage delivery; it converts Markdown tables and removes internal directive/scaffolding text, but ordinary Markdown markers such as bold, headings, blockquotes, and inline code still pass through. The related broad fix PR #56283 is still a separate unmerged head, and the prior comment linking steipete/imsg#40 does not make this unactionable in OpenClaw because the OpenClaw iMessage send path already performs outbound text preprocessing before calling imsg.

## Best Possible Solution

Keep #24943 open until PR #56283 or an equivalent fix lands. The likely maintainer path is to strip or render Markdown to plain text for iMessage/plain-text outbound delivery after existing HTML/table/internal-scaffolding handling and before final send, with focused regression tests for direct sends, monitor reply delivery, media captions, and empty-after-strip edge cases.
