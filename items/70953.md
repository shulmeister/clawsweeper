## Summary

Keep #70953 open. Current main still returns `approval-pending` to the agent immediately for foreground gateway/node exec approvals; the approval decision is handled in a detached follow-up path, and docs/system prompt text explicitly describe that immediate-return behavior.

## Best Possible Solution

Implement foreground exec approvals as a true await point: after two-phase registration, await `exec.approval.waitDecision` with timeout and abort handling, apply allow/deny/fallback policy, then return the final exec or denial result to the same tool call. Preserve the current nonblocking follow-up model only for explicit background/detached executions, and update docs/tests/system prompt text to match the new foreground semantics.

## Risks / Open Questions

- Changing this is a behavior/API shift: docs, system prompt guidance, and existing approval follow-up events currently assume immediate `approval-pending` returns.
- Background/explicit detached exec behavior needs a separate path so long-running or intentionally backgrounded commands do not block agent turns indefinitely.
