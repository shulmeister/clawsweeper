## Summary

Current `main` already scopes Telegram exec approvals to the matching Telegram account/session and rejects ambiguous multi-account fanout, which addresses the reported broadcast-to-all-bots behavior.

## Evidence

- **Telegram approval routing is account-aware and rejects ambiguous multi-account delivery:** `matchesTelegramRequestAccount` binds approvals to the stored Telegram account when possible and otherwise refuses foreign-channel delivery if more than one Telegram account is eligible, preventing broadcast across all connected Telegram bots.
  - file: [extensions/telegram/src/exec-approvals.ts:109](https://github.com/openclaw/openclaw/blob/0a3da5cd8a6e712de6418008e01b11fb2afd7aea/extensions/telegram/src/exec-approvals.ts#L109)
  - command: `nl -ba extensions/telegram/src/exec-approvals.ts | sed -n '109,135p'`
  - sha: [0a3da5cd8a6e](https://github.com/openclaw/openclaw/commit/0a3da5cd8a6e712de6418008e01b11fb2afd7aea)
- **Regression tests cover the exact multi-account cases:** Tests verify that a foreign-channel approval bound to the `ops` Telegram account is only handled there, and that unbound foreign-channel approvals are rejected entirely when multiple Telegram accounts could handle them.
  - file: [extensions/telegram/src/exec-approvals.test.ts:192](https://github.com/openclaw/openclaw/blob/0a3da5cd8a6e712de6418008e01b11fb2afd7aea/extensions/telegram/src/exec-approvals.test.ts#L192)

- **Telegram native approval fallback is limited to the initiating Telegram account/channel:** The Telegram native approval capability requires the triggering channel to match and uses the request/target account id for suppression/routing, instead of treating all Telegram accounts as interchangeable.
  - file: [extensions/telegram/src/approval-native.ts:112](https://github.com/openclaw/openclaw/blob/0a3da5cd8a6e712de6418008e01b11fb2afd7aea/extensions/telegram/src/approval-native.ts#L112)
