## Summary

Keep open. Current `main` still bypasses `message_sending` hooks in the Discord reply-delivery path, while Telegram and the shared outbound path invoke them as documented.

## Evidence

- **Discord reply delivery sends directly without hook runner:** `deliverDiscordReply` builds send helpers and routes text chunks into `sendDiscordChunkWithFallback`, which sends via webhook, `sendDiscordText`, or `sendMessageDiscord`. This file does not fetch `getGlobalHookRunner()` or call `runMessageSending()` anywhere in the delivery path.
  - file: [extensions/discord/src/monitor/reply-delivery.ts:146](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/discord/src/monitor/reply-delivery.ts#L146)

- **Telegram reply delivery already wires `message_sending`:** Telegram reply delivery gets the global hook runner, checks `hasHooks("message_sending")`, and invokes `runMessageSending(...)` before sending the reply, including reply-path metadata.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:626](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/extensions/telegram/src/bot/delivery.replies.ts#L626)
