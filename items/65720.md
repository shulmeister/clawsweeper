## Summary

Keep open. Current `main` still serves the Control UI from the Gateway process and same HTTP server, there is no built-in crash-fallback/static-takeover option in the Gateway config or Control UI path, and the request is a coherent core resilience feature rather than a ClawHub/plugin item.

## Evidence

- **Web UI is explicitly coupled to the Gateway server:** Docs say the browser Control UI is served from the same port as the Gateway WebSocket, so a Gateway process exit would take down that surface too.
  - file: [docs/web/index.md:9](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/web/index.md#L9)

- **HTTP routing mounts Control UI handlers inside the Gateway server request pipeline:** `server-http.ts` only pushes the Control UI handlers when the Gateway HTTP server is running, with no separate fallback listener or takeover path.
  - file: [src/gateway/server-http.ts:1107](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/gateway/server-http.ts#L1107)
