## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway process itself on the Gateway HTTP server; existing fallback behavior is only the normal SPA index.html fallback while that process is alive. The repo has service-manager restart guidance and stability diagnostics for crashes, but no same-port static maintenance page or `gateway.serveStaticOnCrash`-style configuration. The provided related items are only broad title-overlap reports and do not supersede this gateway resilience request.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The likely implementation belongs in core Gateway lifecycle/service management, not ClawHub: either define a small supervised fallback/maintenance server that can take over the configured Gateway port when the Gateway child is down, or explicitly document that managed restart is the supported resilience strategy. If implemented, add typed config/schema/docs plus tests covering crash/restart behavior, same-port ownership, custom fallback content, and auth/origin safety.

## Risks / Open Questions

- A same-port fallback after process exit cannot be implemented as a simple in-process toggle; it likely needs a separate supervisor/helper process or service-manager integration that owns the port lifecycle.
- Any static maintenance page must preserve Gateway security boundaries, especially Control UI auth, allowed origins, Tailscale/reverse-proxy deployments, and information disclosure around service health.
