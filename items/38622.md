## Summary

Keep #38622 open. Current main still routes standard workspace bootstrap files through `openBoundaryFile` with the workspace directory as the boundary, rejects final symlinks whose targets resolve outside that boundary, and converts the failed guarded read into a missing bootstrap record. The related fix attempts (#38650 and #40230 open, #38653 closed unmerged) are useful context but are not shipped on main.

## Best Possible Solution

Keep the issue open and land a focused core fix in `src/agents/workspace.ts`: when a standard workspace bootstrap file is explicitly present as a symlink, follow and read the target for injection while preserving `MAX_WORKSPACE_BOOTSTRAP_FILE_BYTES`, missing behavior for dangling symlinks, hardlink rejection, and existing boundary protections elsewhere. Use #38650, #38653, and #40230 as prior-art context, then add regression coverage before closing.

## Risks / Open Questions

- A broad fix in `openBoundaryFile` could weaken boundary protections for unrelated read surfaces; the safer path is a targeted bootstrap-file fallback.
- Following outside-workspace symlinks intentionally expands read scope, so a fix needs explicit tests for relative and absolute symlinks, dangling symlinks, hardlinks, and the bootstrap size limit.
