## Summary

Current main still does not implement multimodal audio support for the `read` tool. The OpenClaw read wrapper and the pinned upstream `@mariozechner/pi-coding-agent` read contract are image-only; audio support found in current main is for inbound media understanding, transcription, TTS, realtime voice, or memory indexing, not arbitrary `read("*.m4a")` tool results. Related #46601/#46607 are narrower Gemini headless `@file` work and do not supersede this broader read-tool request.

## Best Possible Solution

Keep #35835 open. The best implementation path is to extend the read-tool contract or OpenClaw wrapper to detect supported audio files, return a provider-transportable audio content block with sniffed MIME and size/path guards, teach provider transports to forward audio only to models that support it, return a clear unsupported-model result otherwise, and add focused read/tool-result transport tests. Coordinate with #46607 only for the narrower Gemini headless `@file` surface.
