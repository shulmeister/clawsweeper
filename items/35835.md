## Summary

Current main still does not implement audio-file multimodal support in the read tool. The read wrapper and upstream dependency contract are image-only, and non-image binaries still fall through to UTF-8 text handling. Existing audio support is in the inbound media-understanding pipeline, not arbitrary local read("*.m4a") tool calls. Related #46601/#46607 is adjacent but described in the provided context as open/unmerged, so it is not a shipped superseding fix.

## Best Possible Solution

Keep #35835 open until the read tool itself can detect supported audio files and return a real multimodal audio attachment, or until maintainers explicitly choose a different product path. The likely fix belongs in the OpenClaw read-tool integration and/or the upstream pi-coding-agent read contract: add an audio MIME hook/content block or local OpenClaw wrapper, return a clear unsupported-model message where needed, and add focused read-tool tests plus docs. Coordinate with the adjacent #46601/#46607 work, but do not close this issue until that behavior lands on main.
