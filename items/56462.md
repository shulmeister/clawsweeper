## Summary

Keep open. Current `main` still routes the exec tool through child-process paths that decode Windows output as UTF-8 / default `Buffer.toString()`, while the only codepage-aware decoding I found is in `src/node-host/invoke.ts`. The user-facing FAQ still documents the PowerShell `chcp 65001` workaround, so there is not strong evidence that #56462 is fixed on `main` or in the latest shipped release.

## Evidence

- **Exec tool still uses supervisor child mode:** The exec runtime sends non-PTY exec sessions through `supervisor.spawn({ mode: "child", ... })`, so Windows exec output still flows through the child adapter path rather than the node-host decoder.
  - file: [src/agents/bash-tools.exec-runtime.ts:765](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/bash-tools.exec-runtime.ts#L765)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Supervisor child adapter still decodes chunks with Buffer.toString:** `createChildAdapter` forwards stdout/stderr with `listener(chunk.toString())` for both streams, which is not codepage-aware on Windows.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/process/supervisor/adapters/child.ts#L111)

- **Shared exec helper still forces UTF-8 / default string decoding:** `runExec` passes `encoding: "utf8"` to `execFile`, and `runCommandWithTimeout` appends `d.toString()` from child stdout/stderr. That matches the reported mojibake failure mode on non-UTF-8 Windows code pages.
  - file: [src/process/exec.ts:138](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/process/exec.ts#L138)
