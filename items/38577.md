## Summary

Current main already supports the requested split: slash-command authorization can be broader than normal chat authorization, command-only messages from authorized users bypass the model, and skill/plugin slash commands can dispatch directly to tools without an AI run.

## Close Comment

Thanks for the report. I re-checked current `main` in a Codex review, and this is already supported by the existing command/skill routing surfaces.

OpenClaw now has a separate slash-command allowlist via `commands.allowFrom` (`docs/tools/slash-commands.md:19`, `docs/tools/slash-commands.md:73`, `src/config/types.messages.ts:169`), while normal WhatsApp chat access remains gated by `channels.whatsapp.dmPolicy` and `allowFrom` (`docs/channels/whatsapp.md:151`). For authorized senders, command-only messages already take a fast path that bypasses the model (`docs/tools/slash-commands.md:197`; verified in `src/gateway/server.chat.gateway-server-chat.test.ts:612`). On top of that, skill/plugin slash commands can dispatch directly to a tool with `command-dispatch: tool`, so deterministic non-AI command handling is already available without adding a new core router middleware (`src/agents/skills/command-specs.ts:119`, `src/auto-reply/reply/get-reply-inline-actions.ts:261`).

Because the “plugin-only users vs full-access users” split is already achievable on current `main`, I’m closing this as implemented. I also verified the same behavior is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
