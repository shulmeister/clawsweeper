## Summary

Current `main` already carries the Discord voice dependency, resolution, DAVE receive-recovery, and account-config wiring needed for this report; the issue was filed against `2026.3.28`, but the relevant fixes are now present on `main` and documented.

## Evidence

- **Discord plugin now owns and stages its voice runtime dependencies:** `@discordjs/voice` and `opusscript` are direct plugin dependencies, and the plugin manifest sets `openclaw.bundle.stageRuntimeDependencies=true`, which addresses the report's fresh-install/runtime-resolution concern instead of requiring top-level copies.
  - file: [extensions/discord/package.json:6](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/package.json#L6)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Voice SDK resolution is plugin-local:** `loadDiscordVoiceSdk()` uses `createRequire(import.meta.url)` and then requires `@discordjs/voice`, so resolution is anchored to the Discord plugin package rather than the repo root/runtime bundle cwd described in the issue.
  - file: [extensions/discord/src/voice/sdk-runtime.ts:7](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/extensions/discord/src/voice/sdk-runtime.ts#L7)
  - sha: [fc5920fb5134](https://github.com/openclaw/openclaw/commit/fc5920fb51346a682b269fdbeeec0971ae4ae723)
- **Current voice manager proactively handles the DAVE receive failure class from the report:** The manager passes through DAVE join options, enables DAVE receive passthrough immediately after join and again on speaker start, and on repeated decrypt failures logs the condition and performs controlled leave/rejoin recovery.
  - file: [extensions/discord/src/voice/manager.ts:576](https://github.com/openclaw/openclaw/blob/fc5920fb51346a682b269fdbeeec0971ae4ae723/extensions/discord/src/voice/manager.ts#L576)
  - sha: [fc5920fb5134](https://github.com/openclaw/openclaw/commit/fc5920fb51346a682b269fdbeeec0971ae4ae723)
- **Regression coverage exists for passthrough and decrypt-recovery behavior:** The Discord voice E2E test asserts passthrough is enabled after join, and the receive-recovery unit test classifies `DecryptionFailed(UnencryptedWhenPassthroughDisabled)` as a recoverable decrypt failure with thresholded recovery behavior.
  - file: [extensions/discord/src/voice/manager.e2e.test.ts:371](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/voice/manager.e2e.test.ts#L371)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Per-account voice config is supported on current main:** Discord account config is merged from top-level plus `channels.discord.accounts.<id>` and the resolved account config is what gets passed into `DiscordVoiceManager`, so `voice.autoJoin` is read from the selected account config rather than top-level only.
  - file: [extensions/discord/src/accounts.ts:32](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/accounts.ts#L32)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **User-facing docs now describe the exact failure mode and current recovery path:** The Discord docs say current OpenClaw watches repeated decrypt failures, auto-recovers by rejoining, and note that the bundled `@discordjs/voice` line includes the upstream padding fix discussed in the issue's `discord.js#11419` / `#11449` references.
  - file: [docs/channels/discord.md:1015](https://github.com/openclaw/openclaw/blob/6eec38ad5a18fbd23001d19aac19c54a60a43127/docs/channels/discord.md#L1015)
  - sha: [6eec38ad5a18](https://github.com/openclaw/openclaw/commit/6eec38ad5a18fbd23001d19aac19c54a60a43127)
- **No tagged release could be tied to these exact main-branch changes yet:** The checkout's newest tag is `v2026.4.23`, while the current Discord plugin manifest on `main` is `2026.4.25`, so I could not attribute this close to a shipped release tag with high confidence.
