## Summary

Current `main` already supports Telegram delivery of new device-pairing requests through the bundled `device-pair` plugin, so the requested workflow exists even though it shipped as `/pair notify ...` rather than a `gateway.devices.notifyChannel` config key.

## Close Comment

Closing from a Codex review: current `main` already implements the requested Telegram pairing-notification workflow via the bundled `device-pair` plugin. The `/pair` command now includes `/pair notify on|off|once` (`extensions/device-pair/index.ts:549`, `extensions/device-pair/notify.ts:411`), the notifier service polls pending device-pair requests and sends Telegram messages with approve instructions (`extensions/device-pair/notify.ts:314`, `extensions/device-pair/notify.ts:494`), and `/pair qr` on Telegram auto-arms a one-shot notify path for the next pairing request (`extensions/device-pair/index.ts:633`). User-facing docs also list the notify subcommand in the pairing flow (`docs/tools/slash-commands.md:153`). This is on current `main` at `86f69ba5a06ae256906c3a26308801e0b84e0091`, so the underlying request is already satisfied, even though the shipped interface is a `/pair notify ...` command instead of the exact proposed `gateway.devices.notifyChannel` config key.
