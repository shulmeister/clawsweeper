## Summary

Keep open. Current main still captures the original Feishu/Lark platform message id at ingress and stores it on the queued run, but collect-mode batching drops `FollowupRun.messageId` when it synthesizes the batched followup, the followup runtime does not pass a queued id as `currentMessageId`, and followup delivery does not give reply-threading resolution a current message id. The reporter also confirmed on 2026-04-17 that this is still affecting production Feishu/Lark usage, and related PR #36235 targeted these exact missing pieces but was closed unmerged.

## Best Possible Solution

Keep #36212 open and implement the narrowly scoped queue continuity fix: normalize queued `messageId`, carry trusted per-item `message_id` metadata into collect prompts, preserve an appropriate batched `FollowupRun.messageId`, pass it into `runEmbeddedPiAgent.currentMessageId`, and pass it through followup delivery reply-threading. Add regression coverage for collect-mode prompt metadata, summary/individual drains, followup runner `currentMessageId`, and Feishu/Lark native reply behavior where feasible.
