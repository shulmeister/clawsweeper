## Summary

Keep #36212 open. Current main still preserves Feishu/Lark's provider message id at ingress and stores it on queued followup runs, but collect-mode batching drops that id before the followup runtime and final delivery paths. The reporter reaffirmed production impact on 2026-04-17, and the focused fix PR #36235 is closed unmerged.

## Best Possible Solution

Keep the issue open and implement the narrow queue/followup continuity fix: preserve normalized queued messageId through collect and summary drains, include trusted per-item message_id metadata in collect prompts, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, pass it through followup delivery/applyReplyThreading, and add regression coverage for collect, summary/individual drain, followup runtime, and Feishu/Lark reply threading.
