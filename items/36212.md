## Summary

Keep #36212 open. Current main still captures Feishu/Lark's original platform message_id and stores it on the initial queued FollowupRun, but collect/summary drains and followup execution do not carry that id through to per-item queued metadata, runtime currentMessageId, or replyToId for native Feishu replies. The related fix PR #36235 is closed and unmerged, so there is no shipped canonical fix.

## Best Possible Solution

Keep the issue open and implement a narrow queue/followup continuity fix: preserve normalized FollowupRun.messageId through collect and summary drains, expose trusted per-item message_id metadata in collect prompts, pass queued.messageId into runEmbeddedPiAgent as currentMessageId for single-message followups, preserve replyToId through followup delivery, and add regression coverage for collect, summary/individual drain, followup runtime, and Feishu/Lark reply threading. PR #36235 is a useful prior attempt/reference, but it is closed unmerged.
