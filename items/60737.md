## Summary

Keep open. Current main supports static per-topic Telegram routing and manual ACP thread binding, but it does not implement the requested per-DM/per-chat default ACP config that auto-spawns an ACP session on first message in a new topic.

## Evidence

- **No default ACP field in Telegram direct/group config:** TelegramThreadBindingsConfig only exposes spawnSubagentSessions/spawnAcpSessions flags, and TelegramDirectConfig lists per-DM fields such as topics, requireTopic, allowFrom, systemPrompt, error policy, and autoTopicLabel without any defaultAcp/default ACP binding field.
  - file: [src/config/types.telegram.ts:33](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/config/types.telegram.ts#L33)
  - command: `nl -ba src/config/types.telegram.ts | sed -n '1,340p'`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Schema would not accept proposed defaultAcp key:** TelegramDirectSchema is strict and includes only the existing direct config keys; defaultAcp is not part of the accepted object shape.
  - file: [src/config/zod-schema.providers-core.ts:161](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/src/config/zod-schema.providers-core.ts#L161)

- **Routing only consumes explicit per-topic agentId:** Telegram message context resolves group/direct topic config, then passes topicConfig?.agentId into resolveTelegramConversationRoute. There is no branch that reads a chat-level default ACP setting or auto-spawns ACP for an unbound topic.
