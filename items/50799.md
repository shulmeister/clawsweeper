## Summary

Keep #50799 open. Current main still implements the reported macOS GUI behavior: the paperclip flow opens an NSOpenPanel restricted to images, the shared chat view model rejects non-image URLs with an explicit “Only image attachments are supported right now” error, and the gateway chat attachment parser still drops non-image payloads even if a client sends them. This is a real current product limitation, not an already-fixed/stale issue.

## Best Possible Solution

Keep the issue open and implement document attachment support across the shared Chat UI and gateway chat.send path rather than only changing the picker. A robust fix should replace the macOS [.image] panel filter with an explicit supported file type set, rename the UI/help text away from image-only language, generalize OpenClawChatViewModel attachment validation and previews for non-images, and extend or deliberately route gateway parsing for PDFs/text/documents with tests covering .pdf, .txt, and at least one unsupported/oversized file path.
