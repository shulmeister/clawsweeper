## Summary

Current `main` already has first-party MCP and ACP bridges, and the repo vision explicitly pushes new external orchestration/MCP surfaces toward plugins or bundles rather than additional core server layers. This request is better owned as a ClawHub plugin/extension than as new OpenClaw core behavior.

## Evidence

- **Core scope favors plugin-owned MCP surfaces:** The vision document says optional capability should usually ship as plugins, bundle-style plugins are preferred for stable external surfaces such as MCP servers, and core will not merge MCP or heavy orchestration work that duplicates existing MCP, ACPX, plugin, or ClawHub paths without a clear product/security gap.

- **OpenClaw already ships an MCP server surface:** `openclaw mcp serve` is documented as OpenClaw acting as an MCP server. Its current contract is a conversation/approval bridge with tools like `conversations_list`, `messages_read`, `events_wait`, `messages_send`, and `permissions_respond`.
  - file: [docs/cli/mcp.md:10](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/cli/mcp.md#L10)

  - file: [docs/cli/acp.md:26](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/cli/acp.md#L26)

  - file: [docs/tools/plugin.md:369](https://github.com/openclaw/openclaw/blob/78a431ed224b6f5b9d58672346551a5b411ad200/docs/tools/plugin.md#L369)
