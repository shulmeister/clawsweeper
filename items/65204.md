## Summary

Keep #65204 open. The reported config shape still maps the profile to port 80 on current main: `browser.profiles.openclaw.cdpUrl: "http://127.0.0.1"` is parsed as HTTP port 80 and currently overrides the explicit `cdpPort: 18800` for non-stale profile URLs, so Chrome would still be launched with `--remote-debugging-port=80`. Current docs explain the workaround for local managed profiles, but the code does not yet reject or repair this ambiguous config.

## Best Possible Solution

Keep the issue open and fix the config boundary. A conservative fix would either reject local managed profiles that set a loopback `cdpUrl` without an explicit port while also setting `cdpPort`, or prefer the explicit `cdpPort` when `driver: "openclaw"` and `attachOnly: false`. Add a focused regression test using the issue's config shape and a doctor/config validation hint telling users to unset `browser.cdpUrl` and `browser.profiles.openclaw.cdpUrl` for managed local profiles.
