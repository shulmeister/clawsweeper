## Summary

Keep open. Current main has partial tool-governance support, but not enough evidence to close this report: `tools.web.fetch.enabled=false` only disables the `web_fetch` tool, while `curl` is reachable through `exec`/`process` unless those tools are denied or excluded. Current source and tests still show an unresolved adjacent policy gap where configuring `tools.exec` implicitly re-exposes `exec`/`process`, matching the related open #47487 concern. The browser-timeout half also lacks the exact target URL/logs needed for reproduction and is too recent to close as stale.

## Best Possible Solution

Keep #69190 open. Ask the reporter for the exact target URL, sanitized config (`tools.profile`, `tools.allow`, `tools.deny`, `tools.web.fetch.enabled`, `tools.exec`, per-agent overrides, and any runtime `toolsAllow`), the full prompt/instructions, and a tool-call trace showing `curl`. Maintainers should clarify that disabling `web_fetch` does not disable shell `curl`; strict browser-only runs should use an explicit allowlist such as `tools.allow: ["browser"]` or deny `exec`, `process`, and `web_fetch`. If the confirmed remaining bug is implicit `exec` re-exposure, link or consolidate with #47487; if the timeout reproduces on current main, fix it in the browser plugin/runtime with a regression test or documented timeout guidance.
