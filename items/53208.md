## Summary

Keep this issue open. Current main supports explicit WhatsApp outbound sends and same-provider cross-context sends, and the related E.164/new-contact/channel-selection issues are distinct. I still do not find a WhatsApp-owned path that resolves a trusted DM sender's natural-language “tell Jack ...” request to Jack unless the model already supplies Jack's E.164/JID target.

## Best Possible Solution

Keep this issue open. Add a focused WhatsApp regression where an authorized direct sender asks to relay to another approved direct target by name, then implement the smallest WhatsApp-owned target-resolution path. Likely places are WhatsApp directory/alias metadata, session-route guidance, or message-tool prompt context; add a generic core/plugin API seam only if WhatsApp cannot expose the needed contact metadata through existing directory contracts. Maintainers should also clarify whether names should come from explicit config aliases, WhatsApp live contact metadata, previous sessions, or only configured E.164/JID IDs.
