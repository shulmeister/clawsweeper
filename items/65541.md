## Summary

Keep this PR open. Current main still does not implement its central change: disabled block streaming buffers text-only and captioned block replies for final delivery, tests codify that behavior, and docs define `blockStreamingDefault: "off"` as final-reply-only. This is a real behavior/product decision, not an already-implemented or safely obsolete cleanup.

## Best Possible Solution

Keep this PR open for explicit maintainer judgment. The best path is to decide whether `blockStreamingDefault: "off"` should remain final-reply-only or gain a progressive, naturally bounded text delivery mode; if adopted, update `reply-delivery.ts`, disabled-streaming tests, streaming docs, and the direct-send dispatcher/dedupe contract together, while avoiding regressions to captioned `MEDIA:` final delivery and accounting for the same-channel awaitable-delivery work tracked by #32868.

## Risks / Open Questions

- Merging this PR as-is would change the documented meaning of disabled block streaming from final-only to progressive block delivery.
- The direct-send dedupe path still treats queued same-channel block replies as delivered before transport completion, so broadening it to text increases the lost-text blast radius unless dispatcher delivery semantics are handled.
- The PR's all-payload direct-send approach may conflict with the current captioned media buffering behavior that prevents text+MEDIA replies from losing captions when block streaming is off.
