## Summary

Keep open. Current `main` still intentionally buffers text-only and captioned block replies when block streaming is off, and the user-facing docs describe `blockStreamingDefault: "off"` as final-reply-only behavior. This PR is therefore not already implemented on `main`, and it changes product semantics while still relying on the existing direct-send dedupe path that only confirms queueing, not transport success.

## Risks / Open Questions

- The UX complaint may still be valid for users who disable block streaming and also do not rely on preview streaming, so leaving the PR open preserves a plausible behavior change for maintainer judgment.
- The direct-send/final-reply dedupe path is already subtle; closing this as if it were obsolete would overstate certainty because current `main` does not implement the requested behavior.
- Adjacent open work on reply duplication/loss means this area is still moving, so a conservative cleanup should avoid auto-closing the PR without explicit maintainer product direction.
