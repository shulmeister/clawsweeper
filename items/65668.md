## Summary

Keep open. Current `main` still has the reported restart shape for unrecognized supervisors: only `launchd`/`systemd`/`schtasks` are treated as supervised, the fallback still detached-spawns a new gateway process, and the doctor/docs wording still describes `OPENCLAW_NO_RESPAWN=1` as a startup-tuning knob rather than a reliability safeguard for custom supervisors.

## Evidence

- **Fallback respawn still detached on main:** `restartGatewayProcessWithFreshPid()` still returns `disabled` only when `OPENCLAW_NO_RESPAWN` is truthy, returns `supervised` only for recognized supervisors, and otherwise does `spawn(process.execPath, args, { env: process.env, detached: true, stdio: "inherit" })` followed by `child.unref()`. That matches the issue's orphan-process mechanism.

There is no generic custom-supervisor path for shell/Python/container watchdogs, so the Linux fallback remains `null` and drops into detached spawn.
