## Summary

Keep open. Current main still has the sequential per-collection direct QMD search loop that this PR targets, and the proposed work is a focused in-repo memory-core performance fix rather than a duplicate, ClawHub item, or obsolete PR. The PR discussion is substantive: the author supplied latency measurements and regression-test intent, Greptile confirmed the sibling-abort follow-up addressed its P2, and Aisle still raises merge-time backpressure/cleanup risks that need review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The right code direction is still in `extensions/memory-core/src/memory/qmd-manager.ts`: parallelize or otherwise de-serialize direct multi-collection QMD invocations while preserving merge/dedup semantics and adding regression coverage. Before merge, address or explicitly accept bounded-concurrency and abort-cleanup concerns from the PR discussion.
