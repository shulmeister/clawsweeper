## Summary

Current main still appears to have the reported doctor repair gap. Stale plugin repair only scans and mutates `plugins.allow` and `plugins.entries`, while channel stale cleanup only runs `cleanStaleConfig` on resolvable channel doctor adapters; an orphaned third-party `channels.<id>` with no loaded/read-only/bundled plugin is skipped and later remains a validation error.

## Best Possible Solution

Keep this issue open and fix the generic doctor repair path. The likely implementation is to extend stale plugin/channel cleanup so `openclaw doctor --fix` removes or explicitly warns about `channels.<id>` entries that no longer correspond to a built-in channel or a discoverable plugin manifest channel, after confirming plugin discovery is clean. Add a regression test with a stale third-party plugin id such as `openclaw-weixin` present in `plugins.allow`, `plugins.entries`, and `channels.<id>`, then assert doctor repair removes the dangling channel reference or otherwise leaves a non-fatal, actionable state.
