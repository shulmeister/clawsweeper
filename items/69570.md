## Summary

Keep #69570 open. The report is recent, has concrete reproduction detail plus an independent confirmation comment, and current main still contains the core behavior under discussion: approved async exec completions produce follow-up agent turns/system-event wakes, while `/stop` aborts active runs but does not provide a visible task-level cancellation/batching path for pending completion followups. Related #69386 and #72062 are adjacent, and PR #57568 was closed unmerged, so there is no canonical shipped fix that supersedes this issue.

## Best Possible Solution

Keep the issue open and treat it as a core approval/session-control bug. The best path is a small core design that adds task/session-level cancellation for pending exec completion followups, lets `/stop`/chat.abort clear or suppress stale exec-event wakes, batches multiple completion events for the same session/task where safe, and gives user stop/new-input priority over automatic completion processing. Keep #69386 and #72062 linked as related symptoms/mitigations, but do not use them to close this broader cascade report.

- A conservative fix needs regression coverage for multiple quick approved execs, repeated denials, `/stop` during pending completions, and user input arriving while completion followups are queued.
