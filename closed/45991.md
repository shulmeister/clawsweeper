## Summary

Keep open. Current `main` does not implement the proposed cron `metadata` passthrough, and the PR remains a coherent, narrowly scoped core change rather than a ClawHub/plugin-only item.

## Evidence

- **Cron job type on main has no metadata field:** `CronJobBase` ends at `failureAlert` and does not define any opaque metadata property, so current cron jobs have no typed slot for persisted caller context.
  - file: [src/cron/types-shared.ts:1](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/cron/types-shared.ts#L1)
  - command: `nl -ba src/cron/types-shared.ts | sed -n '1,120p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Gateway cron schema does not accept or return metadata:** `CronCommonOptionalFields` and `CronJobSchema` omit `metadata`, with `additionalProperties: false` on the schemas. On current `main`, `cron.add`/`cron.update` cannot formally accept the field and job responses do not expose it.
  - file: [src/gateway/protocol/schema/cron.ts:70](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/gateway/protocol/schema/cron.ts#L70)

- **Cron service does not persist or patch metadata:** `createJob()` copies known fields into the stored job and `applyJobPatch()` handles state, agentId, and sessionKey, but there is no branch for `metadata`. That matches the missing type/schema support and shows the feature is not already implemented on `main`.
