## Summary

Current `main` already implements tool-loop safeguards for repeated identical calls, no-progress polling, ping-pong alternation, and unknown-tool retry loops, with docs and shipped changelog entries covering the feature.

## Risks / Open Questions

- The generic repeat/poll/ping-pong guards are opt-in (`tools.loopDetection.enabled` defaults to `false`), so users who have not enabled them can still encounter similar looping behavior.
- The implemented feature focuses on runtime loop detection/blocking; it does not fully implement the issue's suggested 'explicit task goal/step/exit condition' planning UX.
