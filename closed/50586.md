## Summary

Keep PR #50586 open. Current main still does not include the PR's `chcp 65001` cmd-wrapper behavior or a shared process-level Windows codepage decoder, and the linked bug #50519 remains plausible on the inspected paths. The PR is not merge-ready because review comments identify real scope and behavior concerns, but that is a maintainer review/rework issue rather than a conservative auto-close case.

## Best Possible Solution

Keep the PR open but ask for a focused rework or consolidation with the broader Windows exec encoding efforts such as #64661 and the canonical bug #50519. The best implementation is a shared Windows output decoder used by `runExec`, `runCommandWithTimeout`, and related exec/supervisor paths, with GBK/CP936 and split-chunk regression tests, rather than a blanket `chcp` prefix on every batch wrapper.
