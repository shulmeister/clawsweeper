## Summary

Keep open. Current `main` only applies the DeepSeek V4 `reasoning_content` repair on the native `deepseek` provider path, while Venice is still classified as a generic OpenAI-compatible provider. Because Venice discovery can surface API-returned model IDs that are not in the bundled catalog, there is not strong evidence to close this as already fixed, unreproducible, or incoherent.

## Evidence

- **DeepSeek V4 replay patch is native-provider only:** `createDeepSeekV4ThinkingWrapper` adds blank `reasoning_content` for assistant tool-call turns, but only after gating on `model.provider !== "deepseek" || !isDeepSeekV4ModelId(model.id)`; non-native providers bypass the patch.
  - file: [extensions/deepseek/stream.ts:57](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/deepseek/stream.ts#L57)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **OpenAI compat detection does not classify Venice as DeepSeek:** `isDeepSeek` only becomes true for `deepseek-native` endpoints or default-route provider id `deepseek`, so a Venice-hosted DeepSeek model stays on generic OpenAI compat defaults rather than DeepSeek-specific handling.
  - file: [src/agents/openai-completions-compat.ts:75](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/openai-completions-compat.ts#L75)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Venice model discovery can admit non-catalog model IDs:** `discoverVeniceModels()` builds model definitions for any API-returned model id that is not in the bundled Venice catalog, so a live `venice/deepseek-v4-pro` surface could exist even though it is not listed in the static snapshot.
  - file: [extensions/venice/models.ts:608](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/venice/models.ts#L608)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Bundled Venice docs/catalog do not cover the reported model surface:** Current Venice docs only call out `deepseek-v3.2` as a private model and explicitly say it has no Venice tool support; the published anonymized list does not include a DeepSeek V4 entry. That weakens any "cannot reproduce" close, because the reported surface may depend on live Venice discovery rather than the fallback catalog.
  - file: [docs/providers/venice.md:121](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/providers/venice.md#L121)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **DeepSeek replay support is documented as plugin-owned:** The DeepSeek docs say the stricter `reasoning_content` replay contract is handled inside the DeepSeek plugin for `deepseek/deepseek-v4-flash` and `deepseek/deepseek-v4-pro`, matching the native-only implementation above rather than a provider-agnostic proxy path.
  - file: [docs/providers/deepseek.md:86](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/providers/deepseek.md#L86)
