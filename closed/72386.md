## Summary

Keep this issue open. Current main still matches the reported failure path: Telegram/channel envelope text can become a runtime-context delta, OpenClaw still wraps that delta with the exact human-readable runtime-context preface, queues it as a hidden `openclaw.runtime-context` next-turn custom message, and the pinned Pi session dependency converts custom messages into user-role LLM content. Existing user-facing sanitizers still cover delimited and legacy internal runtime-context formats, not this new header.

## Best Possible Solution

Keep this open for a core embedded-runtime fix. The best path is to stop putting human-readable runtime-context framing into ordinary conversational custom-message content for channel-envelope deltas. Use a structured non-conversational payload, provider-aware/system-level representation, or another model-invisible separation, then add regression coverage across runtime-context prompt assembly, the Pi custom-message conversion path, Telegram/channel-envelope deltas, and outbound/user-facing sanitization for the new header.
