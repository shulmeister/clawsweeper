## Summary

Keep open. Current main only implements image-scoped WebChat/Control UI attachments: the file picker, paste/drop handling, client serialization, and Gateway parser all treat attachments as images and drop or reject non-image payloads. The issue explicitly asks for generic files/documents, and a maintainer comment already separates this from the image-only #51045 work.

## Best Possible Solution

Keep #41985 open as the canonical WebChat/Control UI generic file upload track. The implementation should extend the composer beyond `image/*`, carry filename/MIME/type metadata through `chat.send`, update Gateway attachment parsing/storage so PDFs/text/Office-like documents are preserved or routed when supported, and add targeted UI/Gateway tests for non-image uploads plus visible unsupported-file errors.

## Risks / Open Questions

- Image-only upload work can make this look superficially implemented, but accepting generic files in the UI without changing the Gateway parser would still silently drop or reject documents.
- A complete fix needs MIME/capability policy across Control UI, `chat.send`, transcript display, media storage, and provider support so unsupported files fail visibly instead of disappearing.
