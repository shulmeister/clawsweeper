## Summary

Current main already prevents the compaction transcript data loss described here. When `agents.defaults.compaction.truncateAfterCompaction` is enabled, OpenClaw now writes a compacted successor transcript, preserves the prior transcript/checkpoint instead of rewriting it in place, stores compaction checkpoint metadata, and updates the active session entry to the successor file. Reset paths also archive old transcripts with existing retention cleanup. The linked PR #63161 identified the same problem but was closed unmerged; main now has a different shipped-on-main implementation.

## Best Possible Solution

Close this broad issue as implemented on main. Keep the current successor-transcript/checkpoint design, and track any future request for a separate configurable compaction archive directory or day-based retention window as a narrower enhancement.
