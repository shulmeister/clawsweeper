## Summary

Current `main` has new Discord binding/routing code that resolves exact channel peers and parent-thread peers before broader fallbacks, wires those inputs through Discord preflight, and adds regression tests for the multi-channel cases described in #58750. The proving implementation landed on `main` in commit `4f4288e3b5be2e55abfce5213f0bcf182a252f5e`; no released tag containing it was confirmed from this checkout.

## Evidence

- **Routing precedence now matches the reported Discord binding use case:** The current docs and resolver both define routing precedence as exact peer match first, then parent-peer inheritance, then Discord guild/role fallbacks. This is the behavior needed for distinct Discord channel bindings to route to the correct agent instead of collapsing into broader matches.
  - file: [docs/channels/channel-routing.md:62](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/channels/channel-routing.md#L62)
  - command: `nl -ba docs/channels/channel-routing.md | sed -n '60,78p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Route resolution is keyed by concrete peer and parent-peer context:** `resolveAgentRoute` now includes `peer` and `parentPeer` in the route-cache key and evaluates binding tiers in order, with `binding.peer` and `binding.peer.parent` ahead of guild/account/channel fallback. That prevents cross-channel cache reuse and wrong-agent selection across Discord channels/threads.
  - file: [src/routing/resolve-route.ts:564](https://github.com/openclaw/openclaw/blob/4f4288e3b5be2e55abfce5213f0bcf182a252f5e/src/routing/resolve-route.ts#L564)
