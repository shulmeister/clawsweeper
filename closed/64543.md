## Summary

Current `main` and the shipped `v2026.4.23` release already implement the issue's preferred fix: `compat.requiresStringContent` on `openai-completions` models flattens pure-text multipart message content to plain strings before sending the request, with test and docs coverage.

## Evidence

- **Flatten helper exists on current main:** `flattenStringOnlyCompletionContent` joins pure-text content-part arrays into a newline-delimited string and leaves mixed/non-text arrays unchanged.
  - file: [src/agents/openai-completions-string-content.ts:1](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/agents/openai-completions-string-content.ts#L1)

## Risks / Open Questions

- The behavior is opt-in via `compat.requiresStringContent`; misconfigured self-hosted models can still reproduce the old failure until that compat flag is set.
