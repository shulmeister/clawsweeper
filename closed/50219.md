## Summary

Current main has implemented the node-host recovery behavior needed for this report. The Gateway client now creates a fresh WebSocket for each reconnect, keeps a managed ref'd reconnect timer with bounded exponential backoff, schedules reconnects after non-terminal closes, and the node host logs connect/close failures while terminal credential pauses exit for launchd/systemd supervision. A later maintainer fix for the same node-host silent-zombie failure family (#69800) added regression coverage and docs for this behavior.

## Best Possible Solution

Close this issue as implemented on main and keep the current managed reconnect-timer/node-host supervision behavior. If a current post-4ee537 build still reproduces specifically behind Cloudflare 502s, the next best maintainer path is a narrow regression test that simulates an HTTP 502 WebSocket handshake response and verifies a later successful handshake reconnects without `openclaw node restart`.
