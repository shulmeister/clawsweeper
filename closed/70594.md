## Summary

Current `main` already bounds `openclaw channels status` with explicit request timeouts in the CLI, gateway wrapper, gateway client, and channel-probe path, so the reported indefinite hang no longer matches `main`. The WeChat path is also documented as an external plugin surface, with core responsible for the generic status RPC and timeout behavior.

## Evidence

- **CLI status command now passes an explicit timeout:** `channelsStatusCommand` defaults to `10_000ms` (`30_000ms` with `--probe`) and passes that timeout into the `channels.status` gateway call, so the command no longer waits unboundedly.
