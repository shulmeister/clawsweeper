## Summary

Current `main` already covers this RFC’s underlying need with Task Flow: durable orchestration state, persisted progress metadata, child task/subagent linkage, and operator-visible status/recovery surfaces, shipped in `v2026.4.2`.

## Evidence

- **Task Flow is documented as the durable orchestration layer:** `docs/automation/taskflow.md` says Task Flow is the flow orchestration substrate above background tasks, meant for multi-step work with durable progress tracking across gateway restarts and explicit inspection/cancel commands.

  - file: [src/tasks/task-flow-registry.ts:214](https://github.com/openclaw/openclaw/blob/c0a7b6a510758645c5267734b984f4885a605b67/src/tasks/task-flow-registry.ts#L214)

## Risks / Open Questions

- The RFC asked for a dedicated `batch_context` tool and compaction-aware transcript summaries; current `main` solves the same durability/coordination problem through Task Flow state stored outside the transcript rather than that exact API shape.
