## Summary

Keep open. Current `main` still lacks the proposed `record.file` fallback in `resolvePathArg`, so the PR addresses a real remaining gap rather than something already implemented or obsolete.

## Evidence

- **Current main still omits `file` in verbose path resolution:** `resolvePathArg` only checks `record.path`, `record.file_path`, and `record.filePath`; it does not check `record.file` on current `main`. That is the exact code path used to build read/write/edit detail strings.

  - command: `nl -ba src/agents/tool-display-common.ts | sed -n '170,210p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)

For `read`, `write`, and `edit`, that flow reaches `resolveReadDetail()` / `resolveWriteDetail()`, both of which rely on `resolvePathArg()` for the displayed path.
