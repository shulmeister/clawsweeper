## Summary

Current `main` already has Control UI locale selection, navigator-based English defaulting, English fallback for missing translations, and tests that keep shipped locale bundles structurally aligned with English. I did not find hardcoded Spanish strings in `ui/src` outside the generated locale bundles.

## Evidence

- **Navigator locale resolves to English by default:** `resolveNavigatorLocale` maps known non-English prefixes and otherwise returns `DEFAULT_LOCALE`, so `en-US` stays English rather than selecting Spanish.
  - file: [ui/src/i18n/lib/registry.ts:94](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/ui/src/i18n/lib/registry.ts#L94)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Missing translations fall back to English:** `t()` explicitly retries lookups against the English bundle when the active locale does not contain a key, preventing mixed-language gaps from showing a foreign-language fallback.
  - file: [ui/src/i18n/lib/translate.ts:124](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/ui/src/i18n/lib/translate.ts#L124)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Control UI has a language selector:** The overview screen renders a language `<select>` and applies the chosen locale via `i18n.setLocale(v)`, giving users a direct way to force English or another supported locale from the UI.
  - file: [ui/src/ui/views/overview.ts:350](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/ui/src/ui/views/overview.ts#L350)

  - file: [ui/src/ui/storage.ts:266](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/ui/src/ui/storage.ts#L266)
