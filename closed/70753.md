## Summary

Current main and v2026.4.23 already remove the 2026.4.22 short undici stream-timeout regression that matches the reported `HTTP/2: stream timeout after 15000` hosted SSE MCP failure. The bundled SSE MCP transport still sends configured headers on both SDK fetch paths, and the remaining application-level MCP connection timeout is configurable per server.

## Best Possible Solution

Keep the shipped v2026.4.23/current-main timeout floor. If AgentMail or another hosted SSE MCP still reproduces on v2026.4.23 or newer, track it as a fresh transport-specific bug with updated logs because the reported 2026.4.22 short undici stream-timeout path is already fixed.

## Evidence

- **2026.4.22-to-2026.4.23 timeout fix:** The reported 2026.4.22 release passed the embedded attempt timeout directly into `ensureGlobalUndiciStreamTimeouts`; v2026.4.23/current main changed `configureEmbeddedAttemptHttpRuntime` to floor that value at `DEFAULT_UNDICI_STREAM_TIMEOUT_MS`, preventing short run-attempt budgets from becoming process-wide undici stream/header timeouts before bundle-MCP connects.
