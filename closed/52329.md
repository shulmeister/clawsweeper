## Summary

Keep open. Current main still undercounts `/status` cost when usage includes `cacheRead`/`cacheWrite`: the status renderer displays cache usage but estimates the visible cost from input/output tokens only. The PR is outdated because `buildStatusMessage` now lives in `src/status/status-message.ts`, but the underlying bug remains real and the PR discussion contains useful fixes/tests to port.

## Best Possible Solution

Keep the bug-fix work open. Port the useful parts to `src/status/status-message.ts`: include `cacheRead`/`cacheWrite` in the `/status` cost calculation, decide whether the displayed total should be current-turn or stored session cost, preserve the no-pricing behavior so unknown pricing does not render `$0.0000`, and add focused regression coverage in `src/auto-reply/status.test.ts`. Coordinate with #65381/#65109 so only one refreshed implementation lands.
