## Summary

Keep open. Current main has addressed adjacent fallout around live model switching and generic HTTP 404 failover classification, and the Ollama-related comment appears mitigated by an Ollama-owned provider policy artifact. The central Z.AI Coding Plan bug still has a plausible current source path: Z.AI keeps the correct `/api/coding/paas/v4` base URL, but provider config normalization can still broadcast unmatched `normalizeConfig` hooks, and LM Studio's hook still appends `/v1` without checking the provider.

## Best Possible Solution

Keep #67532 open and fix the provider config normalization boundary. The best core fix is to stop applying arbitrary provider `normalizeConfig` hooks to unmatched providers, or make every provider-owned `normalizeConfig` hook explicitly guard on `ctx.provider`. Add a regression test around `planOpenClawModelsJson`/`normalizeProviders` proving `models.providers.zai.baseUrl = https://api.z.ai/api/coding/paas/v4` remains unchanged in generated `models.json`, with a companion guard that LM Studio normalization still only affects `lmstudio`.
