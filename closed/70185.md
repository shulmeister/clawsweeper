## Summary

Keep open. Current main still lacks the memory publicArtifacts registration in the memory-core CLI metadata entry, and the CLI-metadata loader still only wires registerCli, so registerMemoryCapability would be a no-op in that mode. The destructive-prune part appears mitigated by the current bridge guard, but the 0-artifacts CLI bridge problem is not clearly fixed on main. The PR may still need adjustment because actual plugin CLI command execution uses the full command registry scoped to the primary wiki command, so regression coverage should prove the real `openclaw wiki bridge import` path sees the active memory plugin.

## Best Possible Solution

Keep this PR open, but require a maintainer-quality fix and regression test for the real CLI bridge import path. The safest direction is to ensure the `wiki` CLI command process loads the selected memory slot's public artifact provider, with the same dual-kind memory-slot guard and memory-state rollback used by full plugin registration. Add focused coverage for `openclaw wiki bridge import` or the plugin CLI loader proving memory-core artifacts are visible and absent capability does not prune existing bridge pages.
