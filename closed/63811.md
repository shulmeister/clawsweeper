## Summary

The issue aligns with a 2026.4.9 Telegram config-shape regression around legacy `streaming` keys. Current `main` and the shipped `v2026.4.23` release both include runtime compatibility for legacy Telegram streaming values, `openclaw doctor --fix` migration for the old keys, regression tests for that migration, and public docs stating the legacy values are auto-mapped.

## Evidence

- **Issue discussion points at legacy streaming config as the trigger:** The strongest signal in the thread is the follow-up that Telegram worked again after downgrading and manually removing the `streaming` entries from config. That matches the in-thread hypothesis that the 2026.4.9 regression was tied to the legacy Telegram streaming shape rather than a persistent transport failure.
- **Current runtime accepts legacy Telegram streaming values:** `resolveChannelPreviewStreamMode` still accepts nested `streaming.mode`, legacy scalar `streaming`, legacy `streamMode`, and boolean `streaming`, so current runtime behavior is compatible with old Telegram configs instead of depending only on the nested object shape.
