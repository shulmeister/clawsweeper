## Summary

Issue #61279 is a real Docker named-volume permissions bug, but the same remaining work is already tracked by open PR #63959, which contains the targeted Dockerfile fix and changelog entry. Current main has not implemented the fix yet, so this should close only as superseded by that canonical PR, not as fixed-on-main.

## Evidence

- **Current main still lacks the Dockerfile fix:** At current main 02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54, the runtime Dockerfile chowns /app, sets NODE_ENV, and then switches to USER node. There is no pre-created /home/node/.openclaw directory with node ownership before USER node, and git grep found no install/chown line for that path in Dockerfile.
  - file: [Dockerfile:171](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/Dockerfile#L171)
  - command: `nl -ba Dockerfile | sed -n '160,275p'; git grep -n "install -d -o node -g node -m 0755 /home/node/.openclaw\|mkdir -p /home/node/.openclaw\|chown -R node:node /home/node/.openclaw" -- Dockerfile`
  - sha: [02f3e9cfa25f](https://github.com/openclaw/openclaw/commit/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54)
- **Docker docs support the proposed fix mechanism:** Docker documents that an empty volume mounted over a container directory is populated from the image path by default, and Compose creates named volumes on first use; pre-creating the image directory with correct ownership is therefore the right Docker-level mechanism for new named volumes. ([docs.docker.com](https://docs.docker.com/engine/storage/volumes/))
- **Canonical PR already tracks the remaining work:** PR #63959 is open, targets main, references #61279, and its diff adds RUN install -d -o node -g node -m 0755 /home/node/.openclaw plus a changelog entry crediting @jeanibarz. ([github.com](https://github.com/openclaw/openclaw/pull/63959/files))
  - file: [Dockerfile:344](https://github.com/openclaw/openclaw/blob/8976aba5e9336e369ff28fffb7d6d557d42aec78/Dockerfile#L344)
  - command: `Provided GitHub context; web open https://github.com/openclaw/openclaw/pull/63959/files`
  - sha: [8976aba5e933](https://github.com/openclaw/openclaw/commit/8976aba5e9336e369ff28fffb7d6d557d42aec78)
- **Official setup has related bind-mount mitigation, not the raw named-volume fix:** scripts/docker/setup.sh runs a prestart root container to chown /home/node/.openclaw for setup-managed mounts, while docs/install/docker.md still documents EACCES handling for host bind mounts. That does not cover the reporter's raw Compose named volume mounted directly at /home/node/.openclaw.
  - file: [scripts/docker/setup.sh:491](https://github.com/openclaw/openclaw/blob/02f3e9cfa25f26826e8dfd79d4bf174f2f9d8f54/scripts/docker/setup.sh#L491)
  - command: `nl -ba scripts/docker/setup.sh | sed -n '491,505p'; nl -ba docs/install/docker.md | sed -n '239,245p'`

## Risks / Open Questions

- The fix is not on main yet; this close relies on open PR #63959 remaining the canonical tracker for the bug.
- Existing Docker volumes that were already created with root-owned contents will still require a manual chown even after the image fix lands.
