## Summary

Current `main` already treats assistant commentary and final-answer text separately in the shared visible-text extraction path used by webchat/history surfaces, so startup/system scaffolding no longer renders as chat text. Webchat also now sanitizes assistant-visible text to strip memory scaffolding blocks before display.

## Evidence

- **Webchat assistant rendering now uses shared visible-text extraction:** `extractText()` routes assistant messages through `extractAssistantVisibleText()` before UI rendering, instead of joining raw assistant text blocks directly. That is the key path that prevents commentary/system-startup text from surfacing in webchat.
  - file: [ui/src/ui/chat/message-extract.ts:20](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/ui/src/ui/chat/message-extract.ts#L20)

- **Shared assistant extraction prefers `final_answer` and drops commentary-only text:** The shared extractor explicitly prefers `final_answer` blocks and returns `undefined` for commentary-only assistant content once phased text exists, which matches the reported leak shape for `/new` startup instructions.
