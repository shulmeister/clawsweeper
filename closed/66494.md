## Summary

Keep #66494 open. Current main still collapses Kilocode discovery failures to the one-entry static catalog, and the models.json merge path can replace an existing discovered Kilocode model list with that new one-entry provider entry. The generic configured-provider fallback mitigates some explicitly configured providers, but I did not find a source-backed fix for the reported transient-discovery/static-catalog failure mode.

## Best Possible Solution

Keep #66494 open as the canonical Kilocode transient-discovery bug. The best fix is to preserve a previous non-empty discovered Kilocode catalog when live discovery fails or returns empty data, or otherwise make Kilocode-specific model ids pass through without depending on a stale one-entry allowlist; add regression coverage for HTTP 503/timeout after a prior successful catalog and for fallback chains using `kilocode/<vendor>/<model>` ids.
