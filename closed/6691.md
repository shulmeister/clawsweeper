## Summary

Current main already implements progressive, context-aware skill disclosure with bounded prompt cost: the model scans a lightweight `available_skills` catalog and reads only the matching `SKILL.md` when needed, with compact/truncated catalog fallback and mid-session snapshot refreshes for watcher/allowlist changes. That addresses the token-cost request even though the shipped design differs from the issue's trigger-frontmatter sketch.

## Evidence

## Risks / Open Questions

- The shipped approach is progressive disclosure via model selection, not the exact `auto-load-triggers` or `/load` `/unload` API proposed in the issue.
- Skills are still session-snapshotted by default; refreshes are tied to watcher/version/allowlist changes rather than explicit per-turn unload semantics.
