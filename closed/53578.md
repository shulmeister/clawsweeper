## Summary

Keep open. Current main still routes `google-gemini-cli` through the local `gemini` CLI process, so a fixed per-turn startup/auth delay remains an actionable OpenClaw-facing issue. The implementation has session reuse and cached credential fingerprinting for session invalidation, but I did not find a shipped fix that bypasses Gemini CLI Keychain/FileKeychain initialization or otherwise addresses the reported macOS OAuth latency. The related ClawSweeper items cover different Gemini CLI OAuth failures or profile-routing work and do not supersede this specific slow-per-turn/keychain symptom.

## Best Possible Solution

Keep the issue open and reproduce on macOS with timing split between OpenClaw preparation, process spawn, first stdout, and direct `gemini` startup. If the delay is upstream/local Keychain state, document a focused Gemini CLI OAuth troubleshooting path and workaround. If OpenClaw can avoid the repeated auth startup, implement it in the Google plugin/CLI backend surface, likely through a plugin-owned `prepareExecution` or auth-profile handoff rather than core special-casing.
