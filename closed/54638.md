## Summary

Keep open. Current main has useful opt-in steering and interrupt paths, including `messages.queue.mode=steer`, `/queue steer`, `/stop`, and `sessions.steer`, but ordinary inbound Telegram messages still default to `collect`, so the reported “send a clarification mid-loop and have the agent pivot” behavior is not implemented by default.

## Best Possible Solution

Keep the issue open and scope it to the remaining product/runtime decision: whether normal Telegram inbound follow-ups should auto-steer or interrupt active runs by default, whether Telegram should expose an explicit steer/cancel affordance, and what docs/tests should cover for `messages.queue.byChannel.telegram`. Any implementation should preserve the existing opt-in `steer`, `interrupt`, `/queue`, `/stop`, and `sessions.steer` paths while adding regression coverage in the auto-reply queue and Telegram surfaces.

## Risks / Open Questions

- Changing the default from `collect` to `steer` could alter channel semantics and may create duplicate-looking responses on streaming surfaces, so this likely needs explicit maintainer product judgment.
- Current Pi steering behavior is model-boundary based rather than a hard interruption at every tool-call decision point, so a full fix may need runtime-level constraints documented clearly.
