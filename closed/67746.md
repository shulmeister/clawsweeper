## Summary

Keep open. Current `main` still has the narrow bug called out in the issue discussion: the CLI `openclaw channels logout` path clears local WhatsApp auth state but does not coordinate with the live gateway session, so a running WhatsApp channel can stay active until the gateway handles a separate live logout/stop path. The original report's `channels remove` expectation is partly overstated because `channels remove` without `--delete` is disable-only by design, but that does not resolve the remaining logout/runtime mismatch.

## Evidence

- **CLI logout is still local-only:** `runChannelLogout` resolves the plugin and directly calls `plugin.gateway.logoutAccount(...)`. It does not call the gateway RPC `channels.logout`, and the in-file comment says `Auth-only flow: resolve account + clear session state only.`
  - file: [src/cli/channel-auth.ts:246](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cli/channel-auth.ts#L246)

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Gateway logout is the path that stops the live channel first:** `logoutChannelAccount` calls `context.stopChannel(...)` before `plugin.gateway.logoutAccount(...)`, then marks the channel logged out. That live-stop behavior exists in the gateway RPC path, not in the CLI local logout path above.
  - file: [src/gateway/server-methods/channels.ts:83](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/channels.ts#L83)
