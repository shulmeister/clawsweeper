## Summary

Current main already fixes the WhatsApp QR rotation bug by preserving active login state, propagating refreshed QR payloads through the gateway wait flow, and replacing the displayed QR in Control UI. The fix is present on main via commit 245451b6a9c529464e177afa65013adca0d63eb5, but it is newer than the latest tagged release v2026.4.23.

## Evidence

- **WhatsApp login state now tracks rotated QR payloads:** `startWebLoginWithQr` keeps an active login per account, updates QR state on each `onQr`, and `waitForWebLogin` returns a refreshed `qrDataUrl` plus a non-silent message when the current QR has changed.
  - file: [extensions/whatsapp/src/login-qr.ts:300](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/whatsapp/src/login-qr.ts#L300)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Gateway wait RPC forwards current QR metadata:** `web.login.wait` now passes `currentQrDataUrl` through to the channel plugin wait handler, which is the contract needed to detect QR rotation instead of polling only for `connected`.
  - file: [src/gateway/server-methods/web.ts:133](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/gateway/server-methods/web.ts#L133)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Control UI replaces the displayed QR when wait returns a newer one:** The browser dashboard/controller sends the currently shown QR in `web.login.wait` and overwrites `state.whatsappLoginQrDataUrl` when a refreshed QR comes back.
  - file: [ui/src/ui/controllers/channels.ts:64](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/ui/src/ui/controllers/channels.ts#L64)
