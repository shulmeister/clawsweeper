## Summary

Keep open. Current main still persists unpinned ClawHub plugin installs as exact resolved specs, and the updater reuses that stored spec, so a later `openclaw plugins update <id>` can re-resolve the already-installed version instead of latest. The maintainer comment on 2026-04-26 also confirms this remains real and identifies the same source path.

## Best Possible Solution

Fix the CLI ClawHub install persistence path to preserve the user-requested selector in the durable install record: `clawhub:name` for unpinned installs, `clawhub:name@tag-or-version` only when explicitly requested, while keeping resolved `version`, `integrity`, `resolvedAt`, `clawhubPackage`, and related metadata separate. Add a regression that exercises install -> update, not only update with a hand-built unpinned record; consider the mirrored auto-reply plugin install path while making the change.
