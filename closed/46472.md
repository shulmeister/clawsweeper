## Summary

Keep PR #46472 open. Current main still routes Feishu persistent connections through createFeishuWSClient without any OpenClaw-owned WebSocket liveness timeout or watchdog, so the reported product problem is not proven obsolete. The PR is not merge-ready as written, though: current main uses @larksuiteoapi/node-sdk 1.61.1, whose WSClient constructor type and implementation do not accept a wsConfig/PingTimeout option, and the SDK already has its own server-supplied ping loop. The useful prior local idle verification and Feishu-specific scope mean this should get maintainer/plugin-owner follow-up rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open, but do not merge the current wsConfig shape without maintainer/plugin-owner changes. The best path is to verify current Lark SDK heartbeat behavior against Feishu, then either add a supported Feishu WebSocket liveness watchdog/reconnect wrapper in extensions/feishu, upgrade/use a documented SDK option if one exists, and add a focused regression test for the constructor/monitor behavior.
