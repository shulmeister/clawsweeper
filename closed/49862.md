## Summary

Current `main` no longer resolves Telegram reply context by searching session transcripts. Reply metadata is taken directly from the inbound Telegram message payload and forum topics are kept on topic-scoped session keys, which addresses the cross-topic stale reply context described in #49862.

## Evidence

- **Reply context now comes from Telegram payload:** `describeReplyTarget` builds reply context from `msg.reply_to_message`, `external_reply`, and `msg.quote`, extracting the replied message body directly from the inbound Telegram payload instead of any session-history lookup.
  - file: [extensions/telegram/src/bot/helpers.ts:382](https://github.com/openclaw/openclaw/blob/9f8b4006307d8c840814610e0d664155178aab12/extensions/telegram/src/bot/helpers.ts#L382)
  - command: `nl -ba extensions/telegram/src/bot/helpers.ts | sed -n '382,440p'`
  - sha: [9f8b4006307d](https://github.com/openclaw/openclaw/commit/9f8b4006307d8c840814610e0d664155178aab12)
- **Inbound context stores direct reply fields:** `buildTelegramInboundContextPayload` uses `describeReplyTarget(msg)` and writes `ReplyToId`, `ReplyToBody`, and `ReplyToSender` from that resolved payload into the finalized context. There is no cross-session message-id search in this path.
  - file: [extensions/telegram/src/bot-message-context.session.ts:157](https://github.com/openclaw/openclaw/blob/9f8b4006307d8c840814610e0d664155178aab12/extensions/telegram/src/bot-message-context.session.ts#L157)
