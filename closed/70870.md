## Summary

Current main treats Lossless Claw / LCM as a separate community context-engine plugin, not an OpenClaw core subsystem. The requested `openclaw lcm repair/status` and LCM-specific invalidation/recovery behavior fit the plugin/ClawHub path, and main already documents plugin-owned context-engine storage/lifecycle plus plugin-owned CLI extension points for this kind of feature.

## Evidence

- **Vision is plugin-first for optional capability:** The repo vision says core should stay lean, optional capability should usually ship as plugins, plugin promotion belongs in ClawHub, and plugin APIs should be extended instead of adding one-off core behavior.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/VISION.md#L54)

- **Context engines are installed and configured as plugins:** The context-engine docs show Lossless Claw installed with `openclaw plugins install @martian-engineering/lossless-claw`, selected through `plugins.slots.contextEngine`, and explicitly note that plugin-specific config lives in the plugin's docs.
  - file: [docs/concepts/context-engine.md:28](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/concepts/context-engine.md#L28)

That places LCM DB repair/invalidation in plugin-owned behavior rather than core-owned behavior.

- **Lossless Claw is already listed as a separate community plugin:** The community plugins page lists 'Lossless Claw (LCM)' as `@martian-engineering/lossless-claw` with its own repository, which is direct evidence that this surface already lives outside OpenClaw core.
  - file: [docs/plugins/community.md:64](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/plugins/community.md#L64)
