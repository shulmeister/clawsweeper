## Summary

Close as duplicate/superseded. Current main still does not implement `tools.deny: ["gateway:config.get"]`: the agent-facing `gateway` surface is one tool with an `action` parameter, and policy matching still checks only tool names. The remaining action-level policy work is already tracked by the open general issue #13948, with the open gateway-specific variant #46684 also covering the same core gap. The `config.get` context-explosion motivation has a separate shipped mitigation through live tool-result truncation, so this issue does not need to remain as another tracker.

## Best Possible Solution

Keep the shipped tool-result truncation guard as the context-safety mitigation, and track any core action-level allow/deny design in #13948/#46684. If implemented, it should extend the typed tool policy contract and enforcement path to evaluate tool name plus action before execution, not add another one-off gateway workaround.
