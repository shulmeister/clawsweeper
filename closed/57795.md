## Summary

Current main still preserves the reported stuck WebChat state path. `handleChatEvent` appends a non-silent `final` message from a different run and returns before clearing `chatRunId`, `chatStream`, or `chatStreamStartedAt`; the colocated test suite explicitly asserts that this cross-run final does not clear the active stream. The UI still derives visible busy/abort/streaming state from those fields. Related PR #57887 is an open, unmerged candidate fix for this issue, not shipped behavior on main.

## Best Possible Solution

Keep #57795 open and land PR #57887 or an equivalent fix in `ui/src/ui/controllers/chat.ts`. The fix should clear `chatRunId`, `chatStream`, and `chatStreamStartedAt` when the UI accepts a non-silent cross-run final assistant message, with tests updated to assert the cleared state while preserving intentional side-result/NO_REPLY behavior.
