## Summary

Current main no longer has a separate failing `openclaw logs` connection path: `openclaw logs` now calls the same `logs.tail` Gateway RPC path as `openclaw gateway call logs.tail`, with test coverage for the loopback pairing/scope-upgrade fallback. The reported two-path disagreement does not match current main or the latest release tree for this surface.

## Evidence

- **`openclaw logs` uses `logs.tail` RPC directly:** `fetchLogs()` invokes `callGatewayFromCli("logs.tail", ...)` and only falls back to the local file on recognized loopback pairing/scope-upgrade errors.
  - file: [src/cli/logs-cli.ts:62](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/logs-cli.ts#L62)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Logs CLI helper uses the standard CLI Gateway RPC client:** `callGatewayFromCliRuntime()` forwards to `callGateway()` with `clientName: CLI` and `mode: CLI`.
  - file: [src/cli/gateway-rpc.runtime.ts:6](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/gateway-rpc.runtime.ts#L6)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **`openclaw gateway call` uses the same underlying `callGateway()` CLI path:** `callGatewayCli()` also forwards to `callGateway()` with the same CLI client identity/mode, so the issue's claimed disagreement between `openclaw logs` and `openclaw gateway call logs.tail` no longer matches the code.
  - file: [src/cli/gateway-cli/call.ts:26](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/gateway-cli/call.ts#L26)

  - file: [src/cli/logs-cli.test.ts:136](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/logs-cli.test.ts#L136)
