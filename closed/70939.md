## Summary

Current `main` already prevents the reported exec-only scheduled cron leak by skipping bundled MCP runtime creation when `toolsAllow` only names core tools; the latest release `v2026.4.23` still lacked that guard, so this appears fixed on `main` but not yet shipped in a release.

## Evidence

- **Cron scheduled path still forwards `toolsAllow`:** Scheduled cron runs pass `params.agentPayload?.toolsAllow` into `runEmbeddedPiAgent`, so the allowlist is present on the embedded attempt path used by cron jobs.
  - file: [src/cron/isolated-agent/run-executor.ts:199](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/cron/isolated-agent/run-executor.ts#L199)
  - command: `nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '160,230p'`
  - sha: [bc21f500d4c1](https://github.com/openclaw/openclaw/commit/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2)
- **Current main gates bundled MCP creation on bundled tool ids:** `shouldCreateBundleMcpRuntimeForAttempt()` returns `false` unless tools are enabled and either no allowlist is present or an allowed tool name contains the bundled-tool separator. In `runEmbeddedAttempt()`, `bundleMcpEnabled` is computed from that helper before any bundled MCP runtime is created.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:469](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/src/agents/pi-embedded-runner/run/attempt.ts#L469)

- **Current main only materializes bundled MCP when the guard passes:** The embedded runner now computes `bundleMcpEnabled` from `toolsAllow` and only calls `getOrCreateSessionMcpRuntime()` / `materializeBundleMcpToolsForRun()` when that guard is true. An exec-only allowlist therefore no longer appends bundled MCP tools.

## Risks / Open Questions

- The current fix is narrowly proven for the reported bundled MCP leak on scheduled exec-only cron runs; this review did not execute a live cron reproduction in a write-safe environment.
- `createBundleLspToolRuntime()` is still instantiated separately from the new MCP guard, so any broader claim about all possible bundled LSP allowlist interactions would need a separate verification path.
