## Summary

Keep PR #57733 open. Current main still lacks the requested `agents.defaults.memorySearch.remote.nonBatchConcurrency` config surface and runtime behavior: non-batch indexing still uses the fixed default concurrency of 4, while config types, resolved config, strict runtime schema, help labels, and memory docs do not expose the key. Related shipped Ollama work serializes the Ollama provider's internal embedding batch calls and extends timeouts, but it does not provide outer non-batch indexing concurrency control across files/sessions. The PR is still useful, though it needs the existing review gap fixed by adding the key to the runtime Zod schema and covering schema/runtime behavior before landing.

## Best Possible Solution

Keep the PR open and ask for a focused rebase/update: add `nonBatchConcurrency` to `MemorySearchConfig`, `ResolvedMemorySearchConfig`, config merge, `MemorySearchSchema.remote`, generated schema/help/labels/docs, and tests that prove schema acceptance, default-to-agent merge propagation, clamping, and `getIndexConcurrency()` behavior for batch, configured non-batch, Ollama fallback, and default non-batch paths. Preserve the current shipped Ollama provider serialization/timeout mitigation, but do not treat it as a substitute for the requested outer non-batch indexing control.

## Risks / Open Questions

- The PR as submitted appears incomplete because the strict runtime Zod schema still needs `remote.nonBatchConcurrency`; without that, user config following the new docs would be rejected.
- The proposed Ollama default of non-batch concurrency 1 should be documented or logged clearly because it is conservative for small hosts but can reduce indexing throughput on larger Ollama machines.
