## Summary

Keep #59431 open. Current main still returns plain Content-Disposition filenames verbatim on the Feishu inbound download path, then passes that value into saveMediaBuffer, so the linked Chinese-filename mojibake bug is not implemented on main. The PR is a plausible focused fix for a real bundled Feishu plugin bug, though it needs changes before merge because review comments correctly identify lossy repair of JSON-derived filenames and missing/narrow test coverage.

## Best Possible Solution

Keep the PR open and ask for a narrowed implementation: repair only plain Content-Disposition header filenames that are likely UTF-8 bytes decoded as Latin-1, avoid applying that heuristic to JSON/API file_name fields without Feishu SDK proof, and add focused Feishu media tests for the mojibake header case and non-rewritten valid names.
