## Summary

Keep open. Current main still does not fully solve the reported startup path: the runtime dotenv loader reads the state `.env` and `~/.config/openclaw/gateway.env`, not `~/.openclaw/secrets/.env`; active Telegram `botToken` SecretRefs are still collected for startup activation; and missing env SecretRefs still throw. The Telegram plugin has partial read/inspection degradation for unavailable tokens, but docs and tests still codify startup fail-fast/no partial activation, so the core user problem remains.

## Best Possible Solution

Keep #42518 open. The best fix is a focused product/security decision and regression-covered implementation: either load the credential-broker dotenv path before config SecretRef activation, or add a Telegram/account-level optional SecretRef degradation path that disables only the unresolved account while preserving fail-closed behavior for truly required runtime secrets. Add tests for multi-account Telegram gateway startup with one unresolved env-template token and other valid accounts continuing to start.
