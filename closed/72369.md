## Summary

Keep this issue open. Current main still builds the bare `/new` or `/reset` startup prompt as the reset turn body, carries that same text into the transcript-visible prompt, and the WebChat history/UI filtering paths do not recognize a startup marker or suppress the reported text. The prior ClawSweeper comment is still accurate against d3fd275aa5fc8b8eff8ca9b1148ad86da8abc2ed, and related #58821 covered a broader/different WebChat context leak rather than this bare reset startup transcript path.

## Best Possible Solution

Keep this issue open and fix the owner path. Preserve the startup instruction for the agent runtime, but stop treating it as an ordinary user-visible transcript message. A conservative fix would either move the startup instruction into runtime-only context with an empty or neutral `transcriptCommandBody`, or persist a marker such as `__openclaw: { kind: "startup" }` and teach `chat.history`/WebChat to hide it or render a neutral session divider. Add focused regression coverage around `get-reply-run`, the embedded `transcriptPrompt` path, `chat.history`, and `ui/src/ui/chat/build-chat-items.ts` or `ui/src/ui/controllers/chat.ts`.
