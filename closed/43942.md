## Summary

Keep open. Current main still does not solve the reported Hunter Alpha failure mode where OpenRouter returns the whole assistant reply, including `[[reply_to_current]]`, through a reasoning field that OpenClaw stores as a `thinking` block. The current extractor and message-end delivery path only parse reply directives from visible text blocks, while OpenAI-compatible `reasoning`, `reasoning_content`, and `reasoning_text` deltas are still mapped to hidden thinking content.

## Best Possible Solution

Keep the issue open and fix it with a targeted OpenRouter/Hunter compatibility path plus regression coverage. The safest product direction is not a generic "promote all thinking to visible text" fallback, because that can leak real reasoning. Instead, either disable OpenRouter reasoning injection for `openrouter/hunter-alpha` when it causes final answers to arrive only in `reasoning`, or add a narrowly-scoped, documented provider/model compatibility transform that can distinguish Hunter's final-answer payload from actual hidden reasoning. Add a test that simulates an OpenAI-compatible stream with empty `content` and `delta.reasoning` containing `[[reply_to_current]] ...`, then verifies webchat/block-reply delivery emits visible text with `replyToCurrent: true` and strips the directive.
