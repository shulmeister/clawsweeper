## Summary

Current main now addresses the reported failure mode well enough to close: fresh gateway diagnostics no longer create a first-time read-only paired-device baseline that can force later Control UI admin reconnects into a scope-upgrade loop, direct loopback backend calls avoid stale paired-device scope baselines, and the Control UI preserves local optimistic chat messages when a history reload temporarily returns empty. The manual scope-upgrade approval flow remains intentional, and the current devices CLI/docs make the exact request-ID approval path explicit.

## Best Possible Solution

Close this as implemented on current main, keep the manual scope-upgrade approval model, and ship the current probe/backend/Control UI fixes in the next release. If someone can still reproduce on a build including 8bbb143a or later, the useful follow-up is a new issue with `openclaw devices list` output before and after approval, the exact request ID, Control UI URL/auth mode, browser storage state, and gateway logs around the reconnect.

## Evidence
