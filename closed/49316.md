## Summary

Current main does not contain a shipped core fix for mlx-vlm's non-standard usage field names: the generic OpenAI-completions transport still parses standard OpenAI prompt/completion usage fields. The issue discussion narrowed the problem to mlx-vlm specifically and accepted the custom-provider direction. OpenClaw's plugin/provider surface is sufficient for a ClawHub mlx-vlm provider to own this compatibility mapping without a new core API.

## Best Possible Solution

Move mlx-vlm support into a ClawHub/provider plugin that registers mlx-vlm as its own provider and owns the usage-field normalization through a custom stream function or wrapper. Keep generic OpenAI-compatible core behavior focused on standard OpenAI usage fields unless maintainers make a broader compatibility-policy decision.
