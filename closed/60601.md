## Summary

Keep open. Current main still only executes OpenAI-compatible tool calls when the provider emits structured `tool_calls`; it does not promote bare JSON or `<tools>...</tools>` text content into tool calls, and the default proxy/local behavior only sends `tool_choice: "auto"`, while this report says Qwen 2.5 Coder 32B via llama.cpp needs `required` or a fallback parser.

## Best Possible Solution

Keep #60601 open. The conservative fix is a documented per-model/provider compatibility setting that maps to `tool_choice: "required"` for OpenAI-compatible local endpoints that need it, plus regression coverage for llama.cpp/Qwen-style streamed content. If maintainers also want content promotion, implement it as an opt-in local-model compatibility path that validates tool name and arguments against the active tool schema before execution.

## Risks / Open Questions

- A content-based fallback would need strict validation against the active tool set and probably opt-in behavior so arbitrary assistant prose containing JSON is not executed as a tool call.
- Advanced `params.extra_body` can overwrite OpenAI-compatible payload keys, so users may have a manual workaround for `tool_choice`, but current main does not expose or document this as a model-level compatibility fix for this report.
