## Summary

Keep open. Current `main` intentionally enforces paired-device scope baselines on loopback/shared-token reconnects, so the reporter's updated finding (existing device paired with only `operator.read`, subagent requesting broader operator scopes) still maps to current code and tests. This is not implemented-on-main cleanup; it remains an unresolved behavior/design question, with open RFC #69066 pointing at a broader service-identity solution.

## Risks / Open Questions

- If maintainers consider subagents an internal service path that should bypass user-paired scope baselines, the needed change is product/security design work, not conservative cleanup.
- The related cross-reference to RFC #69066 suggests this may be better resolved by separating internal service identity from user auth rather than silently auto-upgrading paired device scopes.
