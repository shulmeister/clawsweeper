## Summary

Keep #71834 open. The 2026.4.22 generated thinking bundle problem appears to have moved on in current source and was reported as passing in 2026.4.24, but the remaining Telegram webhook startup failure is still present on current main: a transient `setWebhook` failure after the local listener binds closes that listener, rethrows, and leaves recovery to the generic channel restart backoff.

## Best Possible Solution

Keep the issue open and fix the Telegram webhook startup path in `extensions/telegram`: once the local listener binds, keep it available for health/webhook ingress and retry Telegram `setWebhook` registration with bounded backoff for recoverable network failures. Add a regression test that forces `setWebhook` to fail once and asserts the listener remains up or is brought up before retry backoff. Treat release-bundle syntax validation as a separate publish/check concern because the current remaining user impact is the webhook startup teardown.
