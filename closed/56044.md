## Summary

Keep open. Current main partially improves the /stop path, but it does not fully resolve the issue: steer-mode user messages still depend on the active run being marked streaming, and the embedded queue path still rejects non-streaming active runs, which matches the reported tool-execution gap failure.

## Best Possible Solution

Keep the issue open, but narrow the maintainer path: acknowledge that /stop is now better handled out-of-band, then fix or explicitly redesign the remaining steer/control-plane semantics. The likely code direction is to replace the isStreaming-only gate with an active-run/tool-boundary-aware steer mechanism, ensure urgent commands such as /status, /tasks, /queue, /stop, and /kill do not sit behind data-plane work, and add regression coverage for Telegram/direct-channel messages during tool execution gaps. Cross-link #54488 and #47975 as related lane/subagent symptoms rather than closing this report against them unless a canonical issue is created for the exact control-plane priority and steer gap.
