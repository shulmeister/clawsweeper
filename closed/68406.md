## Summary

Current `main` already routes configured MCP stdio servers through a transport that tears down the whole process tree on close, and the shared session MCP runtime is explicitly retired on run end/session cleanup. The relevant code, docs, and regression tests were added after this issue was filed, so this is implemented on `main` but not yet in the latest tagged release.

## Evidence

- **Configured MCP servers use the stdio transport path:** `resolveMcpTransportConfig()` accepts `mcp.servers` entries with `command`/`args`/`env`/`cwd` and returns a `kind: "stdio"` transport, matching the issue's reported config shape.
  - file: [src/agents/mcp-transport-config.ts:95](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/mcp-transport-config.ts#L95)
  - command: `nl -ba src/agents/mcp-transport-config.ts | sed -n '95,118p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Stdio MCP transport now kills the full process tree on close:** `OpenClawStdioClientTransport` starts POSIX children detached, then `close()` waits briefly and falls back to `killProcessTree(processToClose.pid)` if the server does not exit cleanly. That is the direct fix for orphaned stdio descendants surviving after the client exits.
