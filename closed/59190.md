## Summary

Current main still rejects trusted-proxy identity-header requests from loopback sources with `trusted_proxy_loopback_source`, while PR #59190 proposes a still-unshipped same-host reverse-proxy carveout with additional authenticity/config work. The linked bug #59167 and RFC #69066 remain open, and the RFC treats #59190 as one of the in-flight Phase 0 options rather than as obsolete or already handled.

## Best Possible Solution

Keep the PR open for maintainer review alongside #59167 and RFC #69066. The likely path is either to merge a carefully constrained same-host trusted-proxy fix with explicit authenticity material and schema/docs/tests, or to supersede it only after maintainers pick a canonical Phase 0/Phase 1 gateway-auth design in #69066.
