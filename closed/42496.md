## Summary

Keep open. Current main still does not have the PR's runtime-port discovery path for `openclaw tui`; the linked bug remains plausible and the PR is not implemented or superseded, but it needs rebase/current-main cleanup before merge.

## Best Possible Solution

Keep the PR open, rebase it onto current main, and either land it or implement the same fix directly: persist the resolved Gateway runtime port in a validated lock payload, read only a live/current lock before resolving the TUI local URL, avoid reviving `CLAWDBOT_GATEWAY_PORT`, and add a regression test proving `openclaw tui` targets `ws://127.0.0.1:48789` when the active gateway was started on that port.
