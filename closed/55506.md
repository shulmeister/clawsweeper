## Summary

Keep #55506 open. Related reports show adjacent fixes for Telegram stale sessions and bootstrap reinjection, but current main still has a session-key-scoped bootstrap snapshot cache. Telegram DMs use stable per-user session keys, and ordinary same-session runs resolve bootstrap context through that cache, so I do not have high-confidence evidence that updated/deleted workspace files are always reflected in Telegram DM context after the reported workflow.

## Best Possible Solution

Keep the issue open and add a focused regression around the current Telegram DM path plus bootstrap context cache: start a Telegram direct session with BOOTSTRAP.md present, update IDENTITY.md/USER.md/AGENTS.md and delete BOOTSTRAP.md, then assert the next DM run and `/context`/systemPromptReport reflect live workspace files. The implementation should either reload bootstrap files every run, revalidate the session snapshot by file identity/mtime/deletion, or clear the snapshot when workspace/bootstrap state changes and on relevant gateway/session lifecycle boundaries.
