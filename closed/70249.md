## Summary

Current main already supports distinct Ollama provider ids that share `api: "ollama"`, and the shipped `v2026.4.23` release contains regression coverage for that behavior.

## Evidence

- **Ollama runtime uses the selected provider id to resolve baseUrl:** The Ollama plugin's `createStreamFn` passes the active provider id into `resolveConfiguredOllamaProviderConfig`, then uses that provider entry's `baseUrl` for the stream function. That is the decoupled behavior requested in the issue.

## Risks / Open Questions

- The public Ollama docs shown here still use `models.providers.ollama` as the example key rather than an explicit multi-host example, so users may not realize multi-host custom ids are already supported.
