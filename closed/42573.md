## Summary

Keep open. Current main still contains two separate macOS LaunchAgent tracks: the app autostart plist `ai.openclaw.mac` and the gateway plist `ai.openclaw.gateway`. The macOS app still auto-enables the gateway LaunchAgent in Local mode, and doctor’s extra-service scan still only ignores the canonical gateway label, so an existing `ai.openclaw.mac` plist can still be reported as an “other gateway-like” OpenClaw service. That means the central split-service/doctor-confusion report is still plausible on current main.

## Best Possible Solution

Keep the issue open for maintainer product judgment. The likely fix is either to make the two-LaunchAgent model explicit and adjust doctor/service scanning so `ai.openclaw.mac` is treated as the app autostart service rather than an extra gateway-like runtime, or to change onboarding/local-mode behavior so creating `ai.openclaw.gateway` requires clearer user intent when the app LaunchAgent already exists. The minimum actionable path is to update launchd ownership/classification and tests around `findExtraGatewayServices()` plus macOS app local-mode startup behavior.
