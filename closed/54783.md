## Summary

Current main already addresses the host-side split described in #54783: Gateway startup builds the advertised methods and WS dispatch table from the final gateway plugin registry after deferred plugin reloads, and later tool/provider/discovery paths either reuse the compatible gateway-bindable registry or perform non-activating snapshot loads instead of replacing the live Gateway registry. The same implementation is present in release v2026.4.24.

## Best Possible Solution

Close this issue as implemented on main and keep the shipped gateway-bindable registry, deferred-reload method refresh, non-activating provider/tool snapshot behavior, and channel registry pinning. Any future bncr-specific failure should be reopened as a fresh issue with current-version logs showing a method advertised by the same live Gateway hello/features payload but rejected by that same process.
