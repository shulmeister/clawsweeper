## Summary

Current `main` already implements atomic WhatsApp `creds.json` persistence and the later save-queue drain during socket bootstrap; the behavior is covered by tests and documented in the changelog, so this issue can be closed as fixed on shipped releases.

## Evidence

- **Atomic creds write on current main:** `writeCredsJsonAtomically()` writes WhatsApp creds to a unique `.tmp` file, fsyncs it, renames it into place as `creds.json`, reapplies `0o600`, and syncs the parent directory.
  - file: [extensions/whatsapp/src/creds-persistence.ts:32](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/whatsapp/src/creds-persistence.ts#L32)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Socket path uses atomic writer and drains queue first:** `createWaSocket()` waits for the per-auth creds save queue before restore/bootstrap, then handles `creds.update` by enqueueing saves whose writer is `writeCredsJsonAtomically(authDir, state.creds)`.
  - file: [extensions/whatsapp/src/session.ts:136](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/whatsapp/src/session.ts#L136)
