## Summary

Keep this issue open. Current main still has plausible Windows console-popup paths in bundled plugin runtime dependency staging: Telegram’s grammY dependencies and the default-enabled Anthropic provider still use npm staging, and the relevant npm child-process calls do not set `windowsHide`. The recent active-plugin scoping change may reduce unnecessary staging, but it does not address visible Windows npm child windows.

## Best Possible Solution

Keep this issue open and fix the bundled plugin runtime dependency npm staging paths. The likely maintainer fix is to add `windowsHide: true` to the npm `spawn`/`spawnSync` options in `src/plugins/bundled-runtime-deps.ts`, `scripts/postinstall-bundled-plugins.mjs`, and `scripts/stage-bundled-plugin-runtime-deps.mjs`, add focused Windows-option regression assertions, and verify on Windows 11 with Telegram configured and a provider route matching the report. Preserve the active-plugin scoping behavior from `eed7b13b62d6d46123a95790145fe6f39682ab17`; it is useful but insufficient by itself.
