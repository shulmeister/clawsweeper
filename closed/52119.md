## Summary

Current main does not support the issue's claimed OpenClaw gateway parsing failure: `chat.send` accepts `message` as a string, keeps the inbound text as string fields, and forwards a prompt string into the embedded agent runtime. The reported log line is emitted after an embedded run starts and reflects a downstream model/provider error, matching astroclaw's comment that this is likely Jan/local OpenAI-compatible backend compatibility rather than an OpenClaw message parser bug. Current docs already point users at local-backend compatibility knobs such as `compat.requiresStringContent`, `localModelLean`, and `compat.supportsTools: false`, so there is no concrete OpenClaw source change actionable from this report as filed.
