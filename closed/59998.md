## Summary

Keep open. Current main still hardcodes exec shell selection through `getShellConfig()` and does not expose a `tools.exec.shell` config field in the config type, runtime schema, or docs. The requested Windows default-shell override is a core exec behavior gap, not a ClawHub/plugin item.

## Best Possible Solution

Keep the issue open and design a core exec configuration contract. A likely path is to add a typed `tools.exec.shell` or richer shell executable/args setting to `ExecToolConfig`, zod schema, generated schema/help/docs, and wire it through `createExecTool` into `runExecProcess`/`getShellConfig`, with explicit Windows tests for Git Bash-like shells and approval-policy tests covering PowerShell/cmd/bash semantic differences. Decide separately whether node-host exec should inherit this setting or require node-local configuration.

## Risks / Open Questions

- Changing the default shell is security-sensitive because approval parsing, shell-wrapper detection, safeBins, and Windows PowerShell/cmd/bash semantics differ.
- Scope needs product design: gateway exec, node-host exec, sandbox exec, PTY fallback, and per-agent/global inheritance may not all want the same shell contract.
