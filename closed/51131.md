## Summary

Current `main` already makes session-memory slug generation timeout configurable by reading `agents.defaults.timeoutSeconds` inside the slug generator itself, which addresses this PR's underlying timeout problem and supersedes the proposed hook-local `slugTimeoutMs` wiring.

## Evidence

- **Current main resolves slug timeout from config:** `generateSlugViaLLM()` now computes the timeout from `cfg.agents.defaults.timeoutSeconds`, falling back to the old 15s default only when no override is configured.
  - file: [src/hooks/llm-slug-generator.ts:25](https://github.com/openclaw/openclaw/blob/bcdacfa1b3df000ce04e3ce26bbe3051e56da3e8/src/hooks/llm-slug-generator.ts#L25)
  - command: `nl -ba src/hooks/llm-slug-generator.ts | sed -n '1,140p'`
  - sha: [bcdacfa1b3df](https://github.com/openclaw/openclaw/commit/bcdacfa1b3df000ce04e3ce26bbe3051e56da3e8)
- **Resolved timeout is passed into the embedded agent call:** The slug generator passes the resolved `timeoutMs` into `runEmbeddedPiAgent()`, so slow providers no longer have a hardcoded 15s cap.
  - file: [src/hooks/llm-slug-generator.ts:63](https://github.com/openclaw/openclaw/blob/bcdacfa1b3df000ce04e3ce26bbe3051e56da3e8/src/hooks/llm-slug-generator.ts#L63)
