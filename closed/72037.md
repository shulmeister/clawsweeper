## Summary

Keep #72037 open. Current main has bumped the bundled ACPX plugin to acpx 0.6.1 and adds an OpenClaw-generated Codex ACP wrapper, but that wrapper still defaults to spawning `npm exec --yes --package @zed-industries/codex-acp@^0.11.1 -- codex-acp` or `npx`; the package and lockfile still only stage `acpx`, not `@zed-industries/codex-acp` or its platform binary packages. The linked #60693 section is related but broad, and #71549 was closed unmerged, so this focused report still tracks real remaining work.

## Best Possible Solution

Keep the issue open and fix it in the bundled `acpx` plugin/runtime-dependency path: preserve the isolated `CODEX_HOME` wrapper if needed, but make the default Codex ACP command resolve a staged/bundled `@zed-industries/codex-acp` binary or platform package instead of invoking npm/npx on every session. Update `extensions/acpx/package.json`, lock/runtime-dependency staging, wrapper tests, and ACP docs so fresh global installs can start Codex ACP without npm cache/network work.
