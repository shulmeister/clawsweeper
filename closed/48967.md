## Summary

Keep open. Current main has routing mitigations for bound ACP conversations, but it still does not implement this PR's store-level behavior: copying ACP metadata from a base ACP session onto derived thread/topic session entries during inbound session recording. The PR is not merge-ready as-is because review comments identified real edge cases, especially canonical parent-session lookup, but the remaining store fallback is concrete and not superseded by current main.

## Best Possible Solution

Keep the PR open for maintainer/author follow-up. If the store fallback is still desired, rebase it onto current main, derive base thread/topic keys through the existing session-thread parser, resolve the parent with resolveSessionStoreEntry, avoid overwriting existing child acp metadata, and keep/add regression tests for :thread:, :topic:, existing child-without-acp, missing parent, and legacy canonical parent-key cases. If maintainers decide routing-only is the intended product boundary, close with that explicit product decision rather than automated cleanup.
