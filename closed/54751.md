## Best Possible Solution

Close #54751 as superseded by #66865 and keep the remaining work centralized there, with explicit coverage for Windows npm-global EBUSY behavior, restart availability, non-ok update.run responses, and post-restart version verification before merge.

## Evidence

- **Issue scope:** #54751 reports Windows npm-global updates failing from the Control UI Update button and update.run with npm EBUSY rename errors, and asks for a stop gateway -> npm global update -> restart sequence.
- **Current update.run restarts only after update success:** The server handler calls runGatewayUpdate first, then schedules a SIGUSR1 restart only when result.status === "ok". That means current main does not stop the running gateway before the package-manager update runs.
  - file: [src/gateway/server-methods/update.ts:51](https://github.com/openclaw/openclaw/blob/b565e6e96310c03e81b32caf82894ce89f32fec5/src/gateway/server-methods/update.ts#L51)

  - sha: [b565e6e96310](https://github.com/openclaw/openclaw/commit/b565e6e96310c03e81b32caf82894ce89f32fec5)
- **Global package update still runs in the current process path:** The global install branch detects npm/pnpm/bun, cleans stale rename dirs, then executes globalInstallArgs as the "global update" step before verification. There is no pre-stop or supervisor handoff before this install step.
  - file: [src/infra/update-runner.ts:1284](https://github.com/openclaw/openclaw/blob/b565e6e96310c03e81b32caf82894ce89f32fec5/src/infra/update-runner.ts#L1284)
