## Summary

Current main already fixes the reported Feishu `/stop` failure. Feishu abort text is routed onto an out-of-band control lane instead of the normal per-chat sequential lane, Feishu preserves the clean command text for core command handling, and core fast-abort cancels the active reply run before invoking the normal agent resolver. The 2026.4.10 changelog also records the Feishu `/stop` out-of-band lane fix. `/new` remains documented as a reset command rather than the abort command; `/stop` is the documented current-run abort surface.

## Best Possible Solution

Close this issue as fixed for the documented abort command. Keep the shipped Feishu control-lane plus core fast-abort implementation. If maintainers want `/new` itself to become an out-of-band abort/reset hybrid, track that separately as a product behavior change because current docs define `/new` as session reset and `/stop` as current-run abort.
