## Summary

Current `main` does not contain the reported `sleep sleep` startup bug in the checked-in sandbox image/runtime path. The failure described in #70387 matches a stale locally cached sandbox image with an old `ENTRYPOINT`, and the issue comment already documents that removing/rebuilding that image resolves it.

## Evidence

- **Current sandbox image uses CMD, not ENTRYPOINT:** `Dockerfile.sandbox` on current `main` ends with `CMD ["sleep", "infinity"]`. That file does not hardcode an `ENTRYPOINT` that would prepend another `sleep`.
  - file: [Dockerfile.sandbox:24](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/Dockerfile.sandbox#L24)
  - command: `nl -ba Dockerfile.sandbox | sed -n '1,40p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Current runtime create path passes `sleep infinity` once:** `createSandboxContainer` appends `cfg.image, "sleep", "infinity"` when creating the container. With the current CMD-based image, that is a valid start path; the reported `sleep: invalid time interval 'sleep'` requires an old image whose own ENTRYPOINT already injects `sleep infinity`.
  - file: [src/agents/sandbox/docker.ts:499](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/agents/sandbox/docker.ts#L499)

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Docs already treat image updates as recreate-required:** The sandbox CLI docs explicitly say to run `openclaw sandbox recreate --all` after updating a Docker image, and note that runtimes are recreated automatically on next use.
  - file: [docs/cli/sandbox.md:58](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/docs/cli/sandbox.md#L58)
  - command: `nl -ba docs/cli/sandbox.md | sed -n '56,110p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Issue discussion identified stale local image as the cause:** Comment `#issuecomment-4301196731` reports the conflict came from an old local sandbox image whose `ENTRYPOINT` was still `sleep infinity`, and that deleting the old image and letting OpenClaw rebuild it fixed the problem.

## Risks / Open Questions

- OpenClaw does not appear to auto-detect or invalidate a stale locally tagged sandbox image that still carries an older ENTRYPOINT, so users with cached old images can still hit this operationally until they recreate the sandbox/image.
