## Summary

Current main already addresses the reported startup/readiness race. The Gateway attaches WebSocket handlers before binding/listening, logs `ready` only after that control plane is live, keeps WebSocket RPC available while channel/plugin sidecars start, and returns retryable `UNAVAILABLE` for the few startup-gated RPCs instead of letting them hang. The changelog records the WebSocket RPC startup fix as shipped in `v2026.4.10`, which was already the update available in the reporter's observed environment.

## Best Possible Solution

Close this issue as implemented on main. Keep the shipped startup sequencing: WebSocket handlers attach before bind, WebSocket RPC remains available during sidecar startup, readiness remains explicit via `/readyz`, and startup-dependent RPCs should continue returning retryable `UNAVAILABLE` rather than hanging. Any future report should include logs from a current release if `openclaw cron run` still times out after `v2026.4.10`.

## Evidence

This means the loopback WebSocket control plane is installed before the Gateway can accept CLI RPC connections.
