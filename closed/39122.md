## Summary

Current `main` already covers the Windows gateway-restart `EADDRINUSE` / double-restart scenario with a broader scheduled-task restart and restart-health implementation, and that behavior is already present in released tag `v2026.4.23`. The open PR is therefore redundant, and its review thread also identified unresolved issues in the proposed patch itself.

## Evidence

- **Released Windows scheduled-task restart now clears lingering listeners and waits for port release before rerunning the task:** In released tag `v2026.4.23`, `restartScheduledTask` ends the task, resolves the gateway port, terminates lingering gateway listeners and startup-runtime leftovers, waits for the port to become free, force-terminates remaining busy listeners if needed, and only then reruns the task. That directly addresses the occupied-port / `EADDRINUSE` restart failure described by the PR.
  - file: [src/daemon/schtasks.ts:968](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/daemon/schtasks.ts#L968)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:src/daemon/schtasks.ts | nl -ba | sed -n '968,1006p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current restart-health logic handles Windows runtime-status lag without immediately classifying the new busy port as stale:** `inspectGatewayRestart` probes the gateway when the port is busy but service runtime does not yet report `running`; if the gateway is reachable, it returns `healthy: true` with no stale PIDs. `waitForGatewayHealthyRestart` then waits on that probe-backed snapshot instead of blindly forcing a second restart. This is the same underlying Windows `schtasks` status-lag problem the PR was targeting.
