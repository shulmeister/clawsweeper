## Summary

Keep open. Current main has CLI teardown for memory managers on exit, but it has not implemented this PR's one-shot CLI manager purpose. The affected commands still create default memory managers, and default managers still start chokidar/session/interval hooks before the command can complete.

## Best Possible Solution

Port the idea to the current memory-core plugin: extend the memory runtime contracts to include a `cli` purpose; route `memory status --index`, `memory index`, and `memory search` through it; skip watcher/session/interval startup for one-shot CLI managers while preserving explicit `sync`; isolate or bypass builtin and QMD caches by purpose; add regression tests in the current `extensions/memory-core` CLI and manager/search-manager test files.

## Risks / Open Questions

- The PR patch is stale against current main because memory code now lives under `extensions/memory-core`, not the old `src/memory` and `src/cli/memory-cli.ts` paths.

- The fix needs coverage for `memory status --index`, `memory index`, and `memory search`, plus watcher-skip/cache-isolation behavior.
