## Summary

Close as ClawHub/third-party plugin work. The Camofox integration is an external plugin, and its current source auto-starts a subprocess during plugin registration instead of using OpenClaw's documented runtime lifecycle surfaces. Current OpenClaw core already exposes `registerService` plus `gateway_start`/`gateway_stop` for plugin-owned long-running services, so the core repository does not need a new bundled behavior for this report.

## Best Possible Solution

Leave OpenClaw core as-is and fix the external Camofox plugin/ClawHub package to use the supported plugin lifecycle: avoid subprocess side effects during discovery, start the server only during full runtime activation via `registerService` or `gateway_start`, retain the child handle, and stop it in `stop`/`gateway_stop`.

## Evidence

- **External plugin already exists outside core:** The Camofox Browser plugin is published externally at https://github.com/redf0x1/camofox-browser. Its current `plugin.ts` starts the server from `register(...)` when `autoStart` is true, and `src/utils/launcher.ts` spawns `node` with `stdio: ['ignore', 'pipe', 'pipe']` and `detached: false`; that makes the immediate fix belong in the plugin package/listing rather than OpenClaw core.
- **Project scope favors plugins for optional capability:** OpenClaw's vision says optional capability should usually ship as plugins, and core/plugin API discussions are appropriate only when the capability cannot be built as a plugin yet.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/6b6dcafcee9ca86385b7c1ee2442752314d50257/VISION.md#L54)

- **Plugin discovery must not start subprocesses:** The plugin SDK entrypoint docs state that discovery may evaluate plugin entries and should stay lightweight, with no subprocesses, listeners, background workers, or other live runtime side effects at module/discovery time.
  - file: [docs/plugins/sdk-entrypoints.md:235](https://github.com/openclaw/openclaw/blob/6b6dcafcee9ca86385b7c1ee2442752314d50257/docs/plugins/sdk-entrypoints.md#L235)

- **Supported Gateway lifecycle hooks exist:** The public hooks docs say `gateway_start`/`gateway_stop` are for starting and stopping plugin-owned services with the Gateway, and specifically direct plugin-owned runtime services away from the internal `gateway:startup` hook.
  - file: [docs/plugins/hooks.md:257](https://github.com/openclaw/openclaw/blob/6b6dcafcee9ca86385b7c1ee2442752314d50257/docs/plugins/hooks.md#L257)

- **Core exposes start/stop service API:** The typed plugin API includes `registerService`, and `OpenClawPluginService` has explicit `start(ctx)` and optional `stop(ctx)` methods for long-lived plugin services.
  - file: [src/plugins/types.ts:1981](https://github.com/openclaw/openclaw/blob/6b6dcafcee9ca86385b7c1ee2442752314d50257/src/plugins/types.ts#L1981)

- **Gateway starts and stops plugin services:** Gateway sidecar startup invokes `startPluginServices`, which calls each service `start`, records its `stop`, and Gateway shutdown calls the plugin services handle's `stop`. This is the core lifecycle surface the Camofox plugin should use for a managed child process.
