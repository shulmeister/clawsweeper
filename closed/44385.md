## Summary

Keep open. Current main still lacks the PR's per-candidate same-model retry behavior: `runWithModelFallback` attempts each fallback candidate once, records the failure, and advances. The only adjacent retry on main is a full outer run retry for a narrower transient HTTP set, not the requested 408/429/503 same-model retry. The linked issue #44381 was closed in favor of this PR, while #49800 is narrower same-auth-profile overloaded work and does not supersede this general model-candidate retry PR.

## Best Possible Solution

Keep #44385 open as the canonical tracker for general same-model transient retries before model fallback. The best path is a fresh, narrow rebase against current `main` that isolates the model-fallback retry change, normalizes retry-count inputs, handles or exempts stateful retry-unsafe callers such as memory flush, preserves cooldown-probe invariants, and aligns any config/docs surface with the broader retry discussions in #49376 and the narrower overloaded-profile work in #49800.
