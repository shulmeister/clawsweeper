## Summary

Keep open. Current main still writes the Linux systemd user unit, then performs one `daemon-reload`, one `enable`, and one `restart`; an `enable` failure is surfaced immediately. I did not find a retry, visibility check, or regression coverage for the reported `Unit file openclaw-gateway.service does not exist` race after writing the unit.

## Best Possible Solution

Add targeted systemd install hardening in `src/daemon/systemd.ts`: after writing the unit and running `daemon-reload`, handle an `enable` failure matching `Unit file ... does not exist` by reloading/probing once and retrying `enable`, with a focused unit test that simulates the first enable failure and verifies the second attempt succeeds. Keep the issue open until that behavior ships or a maintainer confirms a different systemd recovery design.
