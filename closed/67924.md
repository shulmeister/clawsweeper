## Summary

Do not auto-close this PR. Current main still has the underlying exec preflight false positive: it scans script contents with a raw `$[A-Z_]...` regex and throws on the first match without checking whether the token is inside a Python or JavaScript string literal. The PR targets a real open bug (#67621), but the submitted patch also has unresolved review feedback about JS comment/regex handling and triple-quoted string escape handling, so it should stay open for maintainer review or revision rather than be closed as obsolete.

## Best Possible Solution

Keep this PR open, but request changes or replace it with a parser/tokenizer-aware fix that skips `$VAR` matches inside Python and JavaScript string/comment contexts while preserving detection for real executable shell-bleed tokens. Add focused regression tests for `$SIM`/`$USD` inside Python docstrings, argparse/help strings, JS strings, and for the reviewed JS regex/comment case where `const value = $DM_JSON;` must still be blocked.
