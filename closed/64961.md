## Summary

Current `main` already re-enters the embedded run with the fallback provider/model and re-normalizes tools inside each attempt, so the issue's stated root cause does not match shipped code on `ae5c6573670b3742b156899bd2b9c36b8677f191`. The later thread analysis pointing to separate schema-error classification behavior is consistent with current source.

## Evidence

- **Fallback reruns the embedded agent with the candidate provider/model:** The embedded failover test passes the `runWithModelFallback` callback straight back into `runEmbeddedPiAgent`, forwarding the candidate `provider` and `model` for each fallback attempt rather than reusing a prebuilt request.

  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Each embedded run resolves provider/model fresh:** `runEmbeddedPiAgent` starts from the invocation's `provider`/`model`, applies hook selection, then resolves the runtime model from that pair before executing the attempt. That means a fallback candidate gets a fresh model-specific run context.
  - file: [src/agents/pi-embedded-runner/run.ts:315](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/agents/pi-embedded-runner/run.ts#L315)
