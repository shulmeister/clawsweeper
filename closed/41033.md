## Summary

Current `main` treats `/agent` as a separate read-only mount in `workspaceAccess: "ro"`, while `tools.fs.workspaceOnly` is explicitly scoped to the sandbox workspace directory. The PR would remap `/agent` only for the guard, but the filesystem bridge still resolves `/agent` to `agentWorkspaceDir`, so the proposed change widens read scope instead of fixing a current-main bug.

## Evidence

- **workspaceOnly contract in docs:** The security docs define `tools.fs.workspaceOnly: true` as restricting file-tool paths to the workspace directory, and separately document `workspaceAccess: "ro"` as mounting the agent workspace read-only at `/agent`.
  - file: [docs/gateway/security/index.md:1052](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/docs/gateway/security/index.md#L1052)
  - command: `nl -ba docs/gateway/security/index.md | sed -n '1046,1055p'; nl -ba docs/gateway/security/index.md | sed -n '1096,1098p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **ro sandbox keeps workspace and /agent distinct:** Sandbox setup uses the sandbox workspace as `workspaceDir` for non-`rw` modes, and mounts `agentWorkspaceDir` separately at `/agent` when it differs from the sandbox workspace.
  - file: [src/agents/sandbox/context.ts:45](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/agents/sandbox/context.ts#L45)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '38,46p'; nl -ba src/agents/sandbox/workspace-mounts.ts | sed -n '21,39p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **filesystem bridge resolves /agent to agentWorkspaceDir:** Sandbox filesystem mounts include `/agent` with host root `sandbox.agentWorkspaceDir`, so `/agent/...` resolves to the real agent workspace rather than the sandbox workspace root.
  - file: [src/agents/sandbox/fs-paths.ts:71](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/agents/sandbox/fs-paths.ts#L71)
  - command: `nl -ba src/agents/sandbox/fs-paths.ts | sed -n '61,80p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **current main intentionally rejects /agent under workspaceOnly:** `createOpenClawCodingTools` still guards sandboxed reads only against `containerWorkdir`, and the regression test on `main` asserts that `/agent/secret.txt` is rejected with `Path escapes sandbox root` when `workspaceOnly` is enabled.
  - file: [src/agents/pi-tools.sandbox-mounted-paths.workspace-only.test.ts:92](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/agents/pi-tools.sandbox-mounted-paths.workspace-only.test.ts#L92)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '457,470p'; nl -ba src/agents/pi-tools.sandbox-mounted-paths.workspace-only.test.ts | sed -n '92,114p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **PR discussion identified the scope expansion risk:** The provided PR discussion includes an Aisle security comment explaining that this PR validates `/agent` against the sandbox root but still executes the original `/agent/...` path, which would allow reads from the separate agent workspace mount. That matches the current bridge/mount implementation above.

## Risks / Open Questions

- This closes the PR because the proposed change conflicts with the current security contract; it does not resolve any broader UX confusion around the two-path `workspaceAccess: "ro"` model.
- If maintainers want `/agent` readable under a stricter policy, that needs a different design that validates the real resolved host path or changes the mount model, not just a guard remap.
