## Summary

Keep open. Current main still has a real Telegram native approval routing gap: the explicit forwarding path preserves target.accountId, and target-recipient auth can match accountId, but the per-account native Telegram shouldHandle path does not consult approvals.exec.targets[].accountId when no turn-source or persisted session account is bound. The related #70568 remains open, and #69936 was closed unmerged, so #69916 should remain the canonical bug until a fix lands.

## Best Possible Solution

Keep #69916 open. The best fix is to add native approval account-target filtering before Telegram delivery, preferably in the shared account resolution path when a single channel-scoped approvals.exec/plugin target accountId is unambiguous, or in Telegram's matchesRequestAccount as a narrowly scoped fallback. Add a multi-account regression where the same approver is allowlisted on multiple Telegram accounts but only the approvals.exec.targets[].accountId account returns true from shouldHandleTelegramExecApprovalRequest. Review #70568 against that exact contract before landing; if it only suppresses ambiguity without honoring the explicit target account, adjust it or use the #69936-style target fallback.
