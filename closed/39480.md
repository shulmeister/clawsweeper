## Summary

Current `main` already implements the requested startup ordering: gateway post-attach startup waits for sidecars by default, and pending delivery recovery now runs only after that phase completes. The same behavior is covered by newer gateway runtime tests, so this PR is superseded by work already on `main`.

## Evidence

- **Post-attach startup waits for sidecars before returning:** `startGatewayPostAttachRuntime` starts sidecars via `sidecarsPromise` and, unless `deferSidecars` is explicitly enabled, awaits that promise before returning. That means normal gateway startup does not leave post-attach startup until sidecars have initialized.

- **Scheduled services activate only after awaited post-attach startup:** `startGatewayServer` awaits `startGatewayPostAttachRuntime(...)` first, then calls `activateGatewayScheduledServices(...)`. Delivery recovery is part of that later activation, so it now happens after sidecars are ready.
