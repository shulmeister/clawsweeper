## Summary

Current `main` already handles native `pnpm` binaries safely for `pnpm build`, and the same fix is present in the shipped `v2026.4.23` release.

## Evidence

- **Runner only uses `node <npm_execpath>` for JS or shebang-backed pnpm entrypoints:** `resolvePnpmRunner()` now checks `npm_execpath` with `isNodeRunnablePnpmExecPath()`. Native binaries without a JS extension or shebang do not take the `node <npm_execpath>` path and instead fall back to invoking plain `pnpm`.
  - file: [scripts/pnpm-runner.mjs:29](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/scripts/pnpm-runner.mjs#L29)
  - command: `nl -ba scripts/pnpm-runner.mjs | sed -n '1,240p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **`pnpm build` goes through the guarded pnpm runner:** The root `build` script is `node scripts/build-all.mjs`, and `resolveBuildAllStep()` uses `resolvePnpmRunner()` for pnpm steps, so the native-binary guard applies to the reported `pnpm build` path.
  - file: [scripts/build-all.mjs:138](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/scripts/build-all.mjs#L138)
