## Summary

Current `main` already enforces cross-agent `sessions_send` through the shared session-visibility/A2A guard, and the shipped `v2026.4.23` release contains the same enforcement. The report also appears to conflate `tools.agentToAgent.allow` (used by `sessions_send`) with `subagents.allowAgents` (used by `sessions_spawn`).

## Evidence

- **`sessions_send` applies the cross-agent access guard before delivery:** After resolving the target session, `sessions_send` builds `createSessionVisibilityGuard(...)` and returns a forbidden result when `access.allowed` is false, before the nested agent run starts.
  - file: [src/agents/tools/sessions-send-tool.ts:241](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/sessions-send-tool.ts#L241)

- **The shared guard rejects cross-agent sends unless A2A is enabled and both agents match the allowlist:** `createAgentToAgentPolicy` uses `cfg.tools?.agentToAgent.allow`, and `createSessionVisibilityGuard` rejects cross-agent access when visibility is not `all`, when A2A is disabled, or when `isAllowed(requesterAgentId, targetAgentId)` fails.
  - file: [src/agents/tools/sessions-access.ts:94](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/tools/sessions-access.ts#L94)
