## Summary

Keep #41151 open. Current main has changed the media delivery path, but it does not implement the PR's requested tracking for tool-result media delivered through `onToolResult`. Final reply media dedupe still depends on `messagingToolSentMediaUrls`, and current main only populates that list for successful messaging-tool sends, not arbitrary tool-result media emitted by `emitToolResultMessage`.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up. The best fix is to update it against current main by recording media URLs that are actually delivered through `onToolResult` for tool-result output, immediately trimming the bounded list, and adding a regression that proves a verbose/full tool media result is not re-sent in the final reply while error results remain excluded.
