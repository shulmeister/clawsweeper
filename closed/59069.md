## Summary

Keep open. Current main already normalizes explicit Telegram `:topic:` targets when the real Telegram plugin is loaded, but this PR is specifically a fallback for delivery resolution when plugin-owned parsing is unavailable. Main still lacks that fallback, and tests explicitly preserve raw `:topic:` targets when the plugin registry is unavailable, so the PR is not obsolete or implemented on main.

## Best Possible Solution

Keep the PR open for maintainer judgment. If the fallback is desired, revise it so an explicit Telegram `:topic:` suffix wins over a session-derived threadId, matching the real Telegram plugin parser semantics, and add the complementary regression test noted in review. If maintainers decide unloaded-parser fallback behavior should remain unsupported, close manually with that product/API rationale rather than as already implemented.
