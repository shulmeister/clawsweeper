## Summary

Keep open. Current main has related safeguards for session model reset, fallback ordering, and native Ollama tool-call payload conversion, but I did not find a source/test/history proof that the specific reported failure is fixed: a long-lived Telegram direct session with prior tool-use history falling from native `ollama/kimi-k2.5:cloud` to `openai-codex/gpt-5.4` while fresh native Ollama runs succeed.

## Best Possible Solution

Keep the issue open and investigate the long-lived Telegram native-Ollama path directly. A maintainer should capture the failing `/api/chat` payload or provider error from a session with prior tool-use history, then add focused coverage in the Ollama stream/history conversion or agent fallback layer proving that a valid native Ollama primary is not bypassed unless the provider actually returns a failover-worthy error.
