## Summary

Current `main` does not bypass memory flush in safeguard mode, and the reported `softThresholdTokens: 400000` expectation conflicts with the implemented/documented threshold semantics. The review also found the same mode-independent wiring in the reported v2026.4.5 source, so this report does not match current or reported-source behavior closely enough to keep open.

## Evidence

- **Current main always runs memory flush before the reply turn:** `runPreflightCompactionIfNeeded(...)` and then `runMemoryFlushIfNeeded(...)` are called in sequence before the normal follow-up turn. This call path does not branch on `compaction.mode`.
  - file: [src/auto-reply/reply/agent-runner.ts:1104](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/agent-runner.ts#L1104)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1080,1145p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Flush gating is threshold-distance based, not absolute-token based:** The threshold is computed as `contextWindowTokens - reserveTokensFloor - softThresholdTokens`, and `shouldRunMemoryFlush(...)` fires when total tokens meet or exceed that value. With a 1,000,000-token context window, default 20,000 reserve floor, and `softThresholdTokens: 400000`, the trigger point is about 580,000 tokens, not 400,000.
  - file: [src/auto-reply/reply/memory-flush.ts:49](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/memory-flush.ts#L49)
  - command: `nl -ba src/auto-reply/reply/memory-flush.ts | sed -n '30,85p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Current docs/help describe `softThresholdTokens` as distance to compaction:** The config help text says `softThresholdTokens` is the 'Threshold distance to compaction (in tokens)' for pre-compaction memory flush execution, matching the implementation rather than an absolute token ceiling.
  - file: [src/config/schema.help.ts:1234](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/config/schema.help.ts#L1234)
  - command: `nl -ba src/config/schema.help.ts | sed -n '1228,1238p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **The missing gateway log evidence is weak because these diagnostics are verbose-only:** `memoryFlush check:` and `memoryFlush triggered:` are emitted through `logVerbose(...)`, and `logVerbose` only writes when verbose/debug logging is enabled. Absence of those strings in a normal log file does not prove the check never ran.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:687](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/auto-reply/reply/agent-runner-memory.ts#L687)

- **The reported v2026.4.5 source already had the same mode-independent wiring:** In the reported `v2026.4.5` source (`3e72c03`), `agent-runner.ts` already called `runMemoryFlushIfNeeded(...)` before the main run, and `memory-flush.ts` already used the same threshold formula. That makes the safeguard-specific theory in the report unsupported by the source for the reported build.
  - file: [src/auto-reply/reply/agent-runner.ts:309](https://github.com/openclaw/openclaw/blob/3e72c03/src/auto-reply/reply/agent-runner.ts#L309)
