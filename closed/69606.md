## Summary

Keep open. Current `main` added a generic `beforeDeliver` seam and helper wiring for `message_sending`, but the concrete reply dispatchers still used by Slack and other channels instantiate dispatchers without that hook, so agent replies can still bypass `message_sending` on current `main`.

## Evidence

- **Reply dispatcher only runs hooks when callers provide `beforeDeliver`:** `src/auto-reply/reply/reply-dispatcher.ts` makes `beforeDeliver` optional and only invokes it inside the queued send path. That means reply-path filtering exists as a seam, not as a universal default.
  - file: [src/auto-reply/reply/reply-dispatcher.ts:81](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/auto-reply/reply/reply-dispatcher.ts#L81)
  - command: `nl -ba src/auto-reply/reply/reply-dispatcher.ts | sed -n '34,38p'; nl -ba src/auto-reply/reply/reply-dispatcher.ts | sed -n '257,264p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Helper wrappers now auto-wire `message_sending`:** `src/auto-reply/dispatch.ts` builds a `message_sending`-backed `beforeDeliver` and injects it in `dispatchInboundMessageWithBufferedDispatcher` / `dispatchInboundMessageWithDispatcher`. This is a partial fix path, not the path all channels use.
  - file: [src/auto-reply/dispatch.ts:66](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/auto-reply/dispatch.ts#L66)
  - command: `nl -ba src/auto-reply/dispatch.ts | sed -n '66,96p'; nl -ba src/auto-reply/dispatch.ts | sed -n '151,188p'`
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Slack reply path still bypasses the helper wiring:** `extensions/slack/src/monitor/message-handler/dispatch.ts` still calls `createReplyDispatcherWithTyping({ ...replyPipeline, deliver: ... })` directly, then passes that dispatcher to `dispatchInboundMessage(...)`. `src/plugin-sdk/channel-reply-pipeline.ts` shows `replyPipeline` only carries prefix/typing/transform data, not `beforeDeliver`, so Slack replies still do not get `message_sending` by default.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:752](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/slack/src/monitor/message-handler/dispatch.ts#L752)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Other direct reply dispatchers remain on the same bypass pattern:** Signal and iMessage also build reply dispatchers directly and call `dispatchInboundMessage(...)` without supplying `beforeDeliver`, so the gap is not Slack-only on current `main`.
  - file: [extensions/signal/src/monitor/event-handler.ts:304](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/signal/src/monitor/event-handler.ts#L304)

- **Issue discussion points to an active, still-open Slack fix PR:** The issue comment from `MD-HA` says they hit this in production on Slack and opened PR #70769 specifically to wire `runMessageSending()` into the Slack reply path. The issue timeline also links older open PR #38340 for Discord. That discussion matches the remaining code gap rather than an obsolete report.

## Risks / Open Questions

- A conservative close would be wrong because the new seam on `main` can look like a full fix even though major reply paths still do not pass `beforeDeliver`.
- The linked Slack fix PR is still open, and closing the issue now would hide an active production report before the concrete channel paths are actually wired.
