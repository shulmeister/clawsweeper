## Summary

Close PR #55617: current main no longer reproduces the PR’s merge-order premise. Skills are alphabetically sorted before prompt truncation, docs only guarantee override precedence on name conflicts, and the proposed tail-slice change is not aligned with the current contract.

## Evidence

- **Prompt list is alphabetically sorted before truncation:** `resolveWorkspaceSkillPromptState()` compacts prompt-facing skill paths, sorts the skills by `name`, and only then passes them into `applySkillsPromptLimits()`. The truncation input is therefore alphabetical, not source merge order.
  - file: [src/agents/skills/workspace.ts:791](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/skills/workspace.ts#L791)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '780,798p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current truncation logic operates on the prefix of that sorted list:** `applySkillsPromptLimits()` still takes `slice(0, maxSkillsInPrompt)` and, when compact mode also overflows, binary-searches the largest prefix that fits. On current main this means truncation follows the already-sorted prompt list, not skill-source precedence.
  - file: [src/agents/skills/workspace.ts:678](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/skills/workspace.ts#L678)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '667,707p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Regression test now codifies alphabetical prompt ordering:** Current main includes a test that explicitly asserts `buildWorkspaceSkillsPrompt()` sorts skills alphabetically regardless of entry insertion order (`fixes #64167`). That directly conflicts with the PR’s assumption that prompt order should reflect low→high source precedence.
  - file: [src/agents/skills/compact-format.test.ts:274](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/skills/compact-format.test.ts#L274)
  - command: `nl -ba src/agents/skills/compact-format.test.ts | sed -n '274,285p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Docs define precedence for overrides, not prompt ordering:** The public skills docs describe precedence only for duplicate-name conflicts: workspace wins over project-agent, personal-agent, managed, bundled, and extra-dir copies. That is an override contract, not a prompt display/truncation ordering contract.
  - file: [docs/tools/skills.md:22](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/tools/skills.md#L22)
  - command: `nl -ba docs/tools/skills.md | sed -n '16,42p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **PR discussion already identified a correctness problem with suffix truncation:** The existing review comment on this PR points out that `Map.set()` preserves original insertion position for overridden names, so a tail/suffix strategy can still drop a higher-precedence override when duplicate skill names exist across sources.

## Risks / Open Questions

- If maintainers still want truncation to prefer higher-precedence sources under budget pressure, that is a separate design change and needs a fresh proposal that reconciles with current alphabetical prompt ordering and duplicate-name override semantics.
