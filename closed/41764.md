## Summary

Keep open. Current main does not implement the requested fix: cron still records `lastError` directly from the run result before/independently of the resolved delivery outcome, and the detached isolated runner path can still propagate `status`, `error`, and `delivered` together. The exact remaining fix is tracked by PR #41869, but that PR is open/unmerged; PR #42086 was closed unmerged.

## Best Possible Solution

Keep #41764 open until a focused cron state fix lands. The likely fix belongs in `src/cron/service/timer.ts`: compute delivery state before assigning error fields, clear `lastError` and `lastErrorReason` when the resolved delivery status is `delivered`, preserve delivery-specific failures in `lastDeliveryError`, and add a regression test for the exact `error` plus `delivered: true` state. PR #41869 can be revived or reworked as the canonical implementation.
