## Summary

Keep #68634 open. Current main appears to have addressed the Telegram native-approvals auto-enable part by requiring explicit `execApprovals.enabled`, but the CLI scope-upgrade problem is not fully gone: `openclaw devices list` still runs as a CLI client without narrowed scopes, falls back to the full CLI operator scope set, and the gateway still creates/rejects scope-upgrade pairing when a paired device is only approved for narrower scopes.

## Best Possible Solution

Keep the issue open and narrow the remaining work to CLI auth/scope behavior. Update CLI RPC helpers such as `devices list` to request method-level least-privilege scopes, or use the loopback shared-token backend path where appropriate, so read/diagnostic commands do not create persistent scope-upgrade pending entries. Add regression coverage for `gateway status`, `devices list`, and config read paths with a paired CLI device approved only for read/approvals, while preserving explicit pairing for real admin actions. Separately, align Telegram docs/config hints with the current explicit `execApprovals.enabled` behavior.
