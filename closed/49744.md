## Summary

Current main already avoids native Jiti loading for source TypeScript plugin/channel runtime entries, which is the failure mode described in #49744 on Node 22.6+.

## Evidence

- **Loader disables native Jiti for source runtime entries:** `src/plugins/loader.ts:445-457` computes `tryNative` per module path and documents that source `.ts` runtime shims with sibling `.js` imports must stay off native Jiti so Jiti rewrites them against the source graph.
  - file: [src/plugins/loader.ts:445](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/plugins/loader.ts#L445)
  - command: `nl -ba src/plugins/loader.ts | sed -n '430,470p'`
  - sha: [320d52a23e41](https://github.com/openclaw/openclaw/commit/320d52a23e41a326287826219a20f1f29f334554)
- **Native Jiti is only preferred for built JavaScript artifacts:** `src/plugins/sdk-alias.ts:705-734` makes `shouldPreferNativeJiti()` return `true` only for `.js/.mjs/.cjs/.json` and `false` for `.ts`, so source `*.runtime.ts` entries no longer take the native-import path that triggers the Node 22 `.js`-specifier crash.
  - file: [src/plugins/sdk-alias.ts:705](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/plugins/sdk-alias.ts#L705)

  - file: [src/plugins/sdk-alias.test.ts:843](https://github.com/openclaw/openclaw/blob/320d52a23e41a326287826219a20f1f29f334554/src/plugins/sdk-alias.test.ts#L843)
