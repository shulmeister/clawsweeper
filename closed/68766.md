## Summary

Keep open. Current main still lacks an OpenClaw-owned Feishu WebSocket reconnect supervisor for the runtime failure described in #68766: WebSocket mode creates one Lark WSClient, starts it once, and then waits only for abort or a synchronous start throw. Related PR #68865 is open and unmerged, and review comments on that PR identify that its proposed loop still would not settle on runtime disconnect. The older #52618 tracks the same general missing-supervisor problem, but #68766 adds a concrete tenant_access_token timeout reproduction plus the doctor/status liveness gap, so it should not be auto-closed as implemented or cannot-reproduce.

## Best Possible Solution

Keep #68766 open until Feishu WebSocket runtime recovery is implemented or explicitly consolidated. The likely fix belongs in extensions/feishu/src/monitor.transport.ts: wrap the WSClient lifecycle in an abort-aware application-level reconnect loop, observe wsClient.start() settlement or add a reliable runtime disconnect/stall signal, recreate the client with bounded exponential backoff, reset state on successful reconnect, and add focused tests for token-refresh/start failure, SDK retry exhaustion, abort cleanup, and status/doctor liveness reporting. If maintainers want one canonical tracker, consolidate #68766 into #52618 only after copying the tenant_access_token timeout and doctor-liveness details there and updating PR #68865 to target the canonical issue.
