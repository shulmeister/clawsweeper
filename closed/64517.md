## Summary

Current `main` already contains the fix for multi-agent approval routing: approval filters now fall back to the agent id embedded in `sessionKey`, the shared exec-approval forwarder uses that fallback for `approvals.exec`, and Telegram's native approval routing also applies the same logic with multi-account account scoping. The change landed in commit `25a02825a5b81f8c5a2db980af9dbb9cc523b208` after the latest tagged release, so it is fixed on `main` but not yet in a released tag.

## Evidence

- **Shared approval filters now parse agent id from session key:** `matchesApprovalRequestFilters` gained `fallbackAgentIdFromSessionKey` and uses `parseAgentSessionKey(...).agentId` when `request.agentId` is absent, which matches the multi-agent Telegram scenario from the report.
  - file: [src/infra/approval-request-filters.ts:23](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/infra/approval-request-filters.ts#L23)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Exec approval forwarding applies the fallback for approvals.exec:** `shouldForwardRoute` passes `fallbackAgentIdFromSessionKey: true` before honoring `agentFilter`/`sessionFilter`, so `approvals.exec.mode: "targets"` no longer drops requests just because only the session key carries the bound agent id.
  - file: [src/infra/exec-approval-forwarder.ts:150](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/infra/exec-approval-forwarder.ts#L150)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Regression test proves old-false/new-true behavior:** The test explicitly shows that an approval request with only `sessionKey: "agent:ops-agent:telegram:group:-1001"` matches `agentFilter: ["ops-agent"]` only when the session-key fallback is enabled.
  - file: [src/infra/approval-request-filters.test.ts:21](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/infra/approval-request-filters.test.ts#L21)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Telegram native approval routing now uses the same fallback and account scoping:** Telegram counts eligible approval accounts with `matchesApprovalRequestFilters(..., fallbackAgentIdFromSessionKey: true)` and then scopes requests to the persisted Telegram account binding, covering multi-bot/account routing instead of only the current chat window.
  - file: [extensions/telegram/src/exec-approvals.ts:81](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/telegram/src/exec-approvals.ts#L81)
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Telegram regression tests cover agent-filter fallback and multi-account routing:** Tests cover the exact missing-agent-id case (`matches agent filters from the Telegram session key when request.agentId is absent`) and the stored-account routing case (`scopes non-telegram turn sources to the stored telegram account`).
  - file: [extensions/telegram/src/exec-approvals.test.ts:169](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/extensions/telegram/src/exec-approvals.test.ts#L169)
