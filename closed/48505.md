## Summary

Current `main` already implements the reported outbound Twilio conversation-mode fix: initial outbound webhooks return streaming TwiML instead of empty TwiML, the initial greeting waits for stream connection, and transient stream disconnects no longer trigger immediate hangup.

## Evidence

  - file: [extensions/voice-call/src/providers/twilio.test.ts:80](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/voice-call/src/providers/twilio.test.ts#L80)

## Close Comment

Codex review against current `main` found the specific regression from this report already implemented. `extensions/voice-call/src/providers/twilio/twiml-policy.ts:56` now returns streaming TwiML for outbound conversation webhooks with `callId`, `extensions/voice-call/src/providers/twilio.test.ts:80` has a regression test for that exact `CallStatus=initiated` case, `extensions/voice-call/src/manager.notify.test.ts:180` verifies the greeting waits for the Twilio stream instead of falling back early, and `extensions/voice-call/src/webhook.ts:390` plus `extensions/voice-call/src/webhook.test.ts:967` add a disconnect grace window so a brief stream flap does not immediately end the call. That matches the original empty-TwiML + 1-second-drop symptom, so I’m closing this as implemented on `main`. If you still see a current-build failure, especially the later silent-audio path rather than the original empty-TwiML drop, please open a new issue with fresh logs and the exact OpenClaw version.
