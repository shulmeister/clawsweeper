## Close Comment

Closing this as implemented after Codex review.

Current main already supports cross-channel DM memory through existing session routing: direct messages share one session by default, and per-user cross-channel sharing is supported with `session.dmScope: "per-peer"` plus `session.identityLinks`. The PR's `crossChannelMemory` / `--share-memory` surface is therefore redundant rather than filling a missing capability.

What I checked:

- **Direct messages already share one session by default:** The docs say direct messages use a shared session by default, and the routing test expects a DM with no special config to resolve to `agent:main:main`. Public docs: [`docs/concepts/session.md`](https://docs.openclaw.ai/concepts/session). ([`docs/concepts/session.md:16`](https://docs.openclaw.ai/concepts/session), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Supported cross-channel per-user sharing already exists:** The documented supported path is `dmScope: "per-peer"`, with `session.identityLinks` for the same person across multiple channels. The session-key builder applies `identityLinks` for non-`main` DM scopes, and tests cover canonical cross-channel identities. ([`src/routing/session-key.ts:142`](https://github.com/openclaw/openclaw/blob/564f820efa2b/src/routing/session-key.ts#L142), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **No missing core gap in current config surface:** `AgentConfig` on main has no `crossChannelMemory` field because session-sharing is handled by the existing top-level session routing model, not a per-agent flag. - **No current CLI gap requiring this PR's flag:** `openclaw agents bind` on main only accepts `--agent`, `--bind`, and `--json`; docs match that surface.
