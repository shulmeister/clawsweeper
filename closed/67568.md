## Summary

Keep open. Current main still writes session-memory filenames, fallback slugs, and Markdown headers from UTC `toISOString()` values, so an Asia/Shanghai reset near local midnight can still be saved under the previous UTC calendar date. Startup-context reading has timezone-aware local-day logic now, but the bundled `session-memory` producer path remains UTC-based.

## Best Possible Solution

Keep the issue open and fix the bundled `session-memory` hook to derive its memory filename date, fallback time slug, and human-readable header from `agents.defaults.userTimezone` via the existing timezone helpers, with host-timezone fallback and UTC retained as explicit metadata. Add regression tests for east-of-UTC and west-of-UTC near-midnight cases, and decide separately whether `.reset.*.Z` archive names should remain stable UTC artifacts or gain separate timezone metadata.
