## Summary

Current `main` already computes cron `nextRunAtMs` from the schedule at job creation, not from a fixed 24-hour fallback, and the same implementation is present in shipped release `v2026.4.23`. The exact reproduction path in the report is also outdated because runtime cron state now lives in `~/.openclaw/cron/jobs-state.json`, not `jobs.json`.

## Evidence

- **Job creation computes next run immediately:** `createJob(...)` sets `job.state.nextRunAtMs = computeJobNextRunAtMs(job, now)` during `cron add`, so initial scheduling is derived from the job spec at creation time.
