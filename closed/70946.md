## Summary

Current main already handles final chat events without forcing the racey full history reload when a usable assistant message is present, so this PR is now redundant even though the original diagnosis was useful.

## Evidence

- **Final assistant payload is appended inline:** Current main normalizes final assistant messages with optional role support and appends the normalized final payload directly to state.chatMessages instead of relying on chat.history for that assistant turn.
  - file: [ui/src/ui/controllers/chat.ts:1747](https://github.com/openclaw/openclaw/blob/67506ac2a9c51999e3254fbc95ab09aab530e2c4/ui/src/ui/controllers/chat.ts#L1747)
  - sha: [67506ac2a9c5](https://github.com/openclaw/openclaw/commit/67506ac2a9c51999e3254fbc95ab09aab530e2c4)
- **Final event handler skips reload when payload is usable:** handleChatEvent appends finalMessage for same-run final events and returns the payload state; the only fallbacks are stream text append or non-usable final payload handling, matching the PR's requested behavior to trust assistant final payloads.
  - file: [ui/src/ui/controllers/chat.ts:2004](https://github.com/openclaw/openclaw/blob/67506ac2a9c51999e3254fbc95ab09aab530e2c4/ui/src/ui/controllers/chat.ts#L2004)
  - sha: [67506ac2a9c5](https://github.com/openclaw/openclaw/commit/67506ac2a9c51999e3254fbc95ab09aab530e2c4)
- **Related PR evidence supports the fix shape:** PR #70946 and the Greptile review both identify the same race: unconditional chat.history reload after a final event can overwrite an inline assistant append with stale persisted history. Current main has removed that behavior on the final assistant payload path.
  - file: [ui/src/ui/controllers/chat.ts:2052](https://github.com/openclaw/openclaw/blob/67506ac2a9c51999e3254fbc95ab09aab530e2c4/ui/src/ui/controllers/chat.ts#L2052)
