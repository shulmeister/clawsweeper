## Summary

Keep open. Current `main` still has the health refresh race this PR is trying to address: an in-flight `probe:false` refresh is reused for a later `probe:true` caller, so the later caller's live-probe intent can be dropped. The broader runtimeSnapshot plumbing from the stacked/related work is not on `main`, so this PR needs rebase/scope reconciliation rather than conservative cleanup closure.

## Best Possible Solution

Keep the item open, but ask for a rebased/narrowed PR against current `main`: first fix the in-flight probe intent race in `src/gateway/server/health-state.ts` with a focused regression test; fold the `getRuntimeSnapshot` guard/runtimeSnapshot overlay only into the canonical runtime-backed health-state work if maintainers still want that behavior.
