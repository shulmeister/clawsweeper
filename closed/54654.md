## Summary

Current main already fixes the linked Feishu proactive-send failure from #54430 by making Feishu send/thread-reply cards come only from the shared `presentation` contract, not from a native `params.card` object. Empty native card input can no longer select `sendCardFeishu` on the send path; text sends fall through to `sendMessageFeishu`. The latest release tag `v2026.4.24` contains the same send-path behavior, so this PR is obsolete for its stated bug even though the PR discussion raised useful broader native-card hardening ideas.

## Best Possible Solution

Close PR #54654 as already implemented for the linked Feishu empty-card send bug. Keep the shipped presentation-based Feishu send behavior; track any remaining native edit-card validation or card-action hardening as a separate, maintainer-scoped issue/PR.
