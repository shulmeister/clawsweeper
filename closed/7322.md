## Summary

Current `main` no longer requires hardcoded OpenRouter catalog entries for new refs. The OpenRouter plugin now resolves arbitrary model IDs dynamically, the embedded runner calls that path before surfacing `Unknown model`, and OpenClaw now ships an OpenRouter free-model scan workflow.

## Evidence

- **OpenRouter plugin resolves arbitrary model IDs:** The bundled OpenRouter provider builds runtime models from the requested `ctx.modelId` and registers both `resolveDynamicModel` and `prepareDynamicModel`, so new OpenRouter slugs do not need a bundled catalog row first.
  - file: [extensions/openrouter/index.ts:49](https://github.com/openclaw/openclaw/blob/dfe75db09d2d85e23f7d25d7afcbcac4bb30e16e/extensions/openrouter/index.ts#L49)
  - command: `nl -ba extensions/openrouter/index.ts | sed -n '49,123p'`
  - sha: [dfe75db09d2d](https://github.com/openclaw/openclaw/commit/dfe75db09d2d85e23f7d25d7afcbcac4bb30e16e)
- **Core resolver checks provider dynamic hooks before `Unknown model`:** `resolveModelWithRegistry` tries plugin dynamic resolution before failing, and the async resolver primes `prepareProviderDynamicModel` first, which is the path used for previously unknown OpenRouter refs.
  - file: [src/agents/pi-embedded-runner/model.ts:673](https://github.com/openclaw/openclaw/blob/a2221d6b47094cd34c12025ce809aba02c5e1c1c/src/agents/pi-embedded-runner/model.ts#L673)
