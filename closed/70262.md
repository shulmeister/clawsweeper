## Summary

Issue #70262 proposes an OB1-backed shared memory integration, but current main explicitly treats memory as a plugin-owned slot and routes optional/MCP-adjacent integrations to community plugins and ClawHub rather than OpenClaw core.

## Evidence

- **Project vision keeps optional capability out of core:** The vision doc says core should stay lean, optional capability should usually ship as plugins, plugin promotion belongs in ClawHub, and the bar for adding optional plugins to core is intentionally high.

  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Memory is already a dedicated plugin slot:** Main documents memory as a special plugin slot where only one memory plugin is active at a time, which matches an OB1 integration living as its own memory plugin rather than a core feature.
  - file: [VISION.md:75](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/VISION.md#L75)
  - command: `nl -ba VISION.md | sed -n '45,95p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Vision rejects duplicate MCP/plugin/ClawHub paths:** The roadmap guardrails explicitly say OpenClaw will not merge MCP work that duplicates existing MCP, plugin, or ClawHub paths without a clear product or security gap.

## Risks / Open Questions

- If an OB1 plugin cannot be built cleanly with the current memory-plugin seams, the follow-up should be a narrowly scoped plugin-API issue rather than a core OB1 integration request.
- The issue body mentions a working external reference implementation, but no published OpenClaw plugin/ClawHub package is linked yet.
