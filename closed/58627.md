## Summary

Current `main` already separates `/reasoning on` reasoning blocks from the final answer and has regression coverage that preserves the final assistant reply across block-reply delivery paths.

## Evidence

- **Current runtime emits reasoning and final answer separately:** In the message-end handler, `/reasoning on` with `onBlockReply` now emits reasoning via `isReasoning` and independently emits the final assistant text as a block reply, rather than relying on `assistantTexts` alone for delivery.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:720](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/pi-embedded-subscribe.handlers.messages.ts#L720)
- **Direct regression test for reasoning-on message_end delivery:** This test configures `reasoningMode: "on"`, `onBlockReply`, and `blockReplyBreak: "message_end"`, then asserts two payloads are emitted: the reasoning block first and `Final answer` second.
  - file: [src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.emits-reasoning-as-separate-message-enabled.test.ts:15](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.emits-reasoning-as-separate-message-enabled.test.ts#L15)
- **Adjacent regression coverage for commentary-first turns:** These tests cover the failure family where commentary is streamed first and confirm the runtime still emits the final answer at `text_end` or `message_end` instead of losing it.
  - file: [src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.emits-block-replies-text-end-does-not.test.ts:190](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.emits-block-replies-text-end-does-not.test.ts#L190)
- **Final payload assembly still falls back to the final assistant answer:** `buildEmbeddedRunPayloads` has explicit fallback behavior for the final assistant text when streamed text is unavailable, and the payload test asserts `Done.` is returned from the final assistant message.
  - file: [src/agents/pi-embedded-runner/run/payloads.test.ts:37](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/pi-embedded-runner/run/payloads.test.ts#L37)
- **Shipped release note for reply-delivery buffering until final_answer:** The changelog entry for release `2026.4.5` documents that reply delivery now keeps commentary buffered until `final_answer` for embedded replies, matching the runtime behavior seen in current `main`.

## Close Comment

Closing this as implemented after Codex review.

Current `main` already separates `/reasoning on` reasoning blocks from the final answer and has regression coverage that preserves the final assistant reply across block-reply delivery paths.

What I checked:

- **Current runtime emits reasoning and final answer separately:** In the message-end handler, `/reasoning on` with `onBlockReply` now emits reasoning via `isReasoning` and independently emits the final assistant text as a block reply, rather than relying on `assistantTexts` alone for delivery. (`src/agents/pi-embedded-subscribe.handlers.messages.ts:720`)
- **Direct regression test for reasoning-on message_end delivery:** This test configures `reasoningMode: "on"`, `onBlockReply`, and `blockReplyBreak: "message_end"`, then asserts two payloads are emitted: the reasoning block first and `Final answer` second. So I’m closing this as already implemented rather than keeping a duplicate issue open.
