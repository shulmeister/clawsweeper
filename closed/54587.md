## Summary

Close #54587 as superseded by the already-linked open PR #54591. Current main has not shipped the escape hatch: sandboxed requester sessions still hit the hard sandbox-to-unsandboxed spawn guard, the sandbox runtime status/config types do not expose `dangerouslyAllowUnsandboxedSubagentSpawn`, and tests/docs still describe the guard. The remaining work is therefore best tracked in PR #54591, which is explicitly titled for this flag and says it closes #54587.

## Best Possible Solution

Close the issue as superseded and keep product/security/test review concentrated on PR #54591. If maintainers want the escape hatch, the best implementation path is to finish that PR with direct spawn-guard coverage and schema/docs updates; if they reject the security model, close or revise the PR rather than tracking the same request in two places.

## Evidence

There is no config escape-hatch check in this path.
