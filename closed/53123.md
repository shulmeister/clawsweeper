## Close Comment

Closing this as implemented after Codex review.

Current main already blocks unauthorized WhatsApp group traffic before reply generation, so this PR's delivery-time guard is redundant and weaker than the shipped path.

What I checked:

- **Inbound sender/group policy blocks early:** `checkInboundAccessControl` rejects group traffic that fails `groupPolicy` / `groupAllowFrom` before any downstream processing. (`extensions/whatsapp/src/inbound/access-control.ts:66`)
- **Blocked messages never reach debouncer/onMessage:** Inbound normalization calls `checkInboundAccessControl(...)` and returns `null` when `access.allowed` is false, so unauthorized group messages do not enter the batching or reply pipeline. (`extensions/whatsapp/src/inbound/monitor.ts:405`)
- **Conversation allowlist is enforced before processMessage:** `createWebOnMessageHandler` runs `applyGroupGating` and returns immediately when `shouldProcess` is false; `applyGroupGating` rejects groups outside the conversation allowlist. (`extensions/whatsapp/src/auto-reply/monitor/on-message.ts:132`)
- **Ack/reply side effects are downstream of gating:** `maybeSendAckReaction` is only called inside `processMessage`, after the inbound and group gates above have already passed. (`extensions/whatsapp/src/auto-reply/monitor/process-message.ts:166`)

So I’m closing this as already implemented rather than keeping a duplicate issue open.
