## Summary

Current `main` already routes Feishu DM replies from each inbound event's `chat_id`, not from volatile in-memory DM peer state, and the same behavior is present in shipped release `v2026.4.23`.

## Evidence

- **Inbound Feishu DM parsing reads sender identity from each message event:** `parseFeishuMessageEvent` derives `senderOpenId` directly from `event.sender.sender_id.open_id` (with `user_id` fallback), so sender identity is rebuilt from the fresh webhook/WebSocket payload after a restart.
  - file: [extensions/feishu/src/bot.ts:114](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/feishu/src/bot.ts#L114)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '100,160p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Reply dispatcher is created with the inbound DM chat id:** Single-agent Feishu dispatch passes `chatId: ctx.chatId` into `createFeishuReplyDispatcher`, so reply delivery targets the current conversation id from the inbound event rather than cached DM session metadata.
  - file: [extensions/feishu/src/bot.ts:1201](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/feishu/src/bot.ts#L1201)

- **Outbound Feishu replies send to `chatId`:** The reply dispatcher calls `sendMessageFeishu` with `to: chatId`; it does not substitute the bot's own `open_id` as the receive target on the normal DM reply path.
  - file: [extensions/feishu/src/reply-dispatcher.ts:455](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/feishu/src/reply-dispatcher.ts#L455)
