## Summary

Current main already implements the Control UI/Webchat NO_REPLY suppression requested in #54255. The gateway now suppresses control-token lead fragments before broadcast, including the exact NO_ prefix described in the bug, and the Control UI controller also filters NO_REPLY in deltas, finals, abort fallback, and history reloads. The reporter's follow-up that the bubble disappeared after a few minutes matches the old history-refresh cleanup path; current main prevents both the transient stream leak and the persisted history bubble. The implementation is present in the v2026.4.24 release tag and documented in that release changelog.

## Best Possible Solution

Close #54255 as implemented on main. Keep the shipped defense-in-depth approach: gateway stream suppression prevents transient NO_/NO_REPLY leaks, Control UI filtering protects event/history rendering, and docs/release notes describe chat.history silent-token omission. Any future recurrence should be filed as a new regression with a fresh Control UI recording/logs and the affected release version.
