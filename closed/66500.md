## Summary

Keep open. Current `main` still uses the Feishu download header filename as the preferred inbound name, and the parser only RFC5987-decodes `filename*=` while returning plain `filename=` values unchanged. That leaves the reported UTF-8-to-Latin-1 mojibake path plausible and unfixed; the core media store already preserves Unicode once it receives a correct filename.

## Evidence

- **Feishu inbound filename parser still returns plain `filename=` values verbatim:** `decodeDispositionFileName()` only percent-decodes `filename*=UTF-8''...`; for ordinary `filename="..."` it returns the raw match with no Latin-1-to-UTF-8 repair or fallback normalization.
  - file: [extensions/feishu/src/media.ts:127](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/feishu/src/media.ts#L127)
