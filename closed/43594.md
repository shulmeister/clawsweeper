## Summary

Keep #43594 open. Current main does not implement the proposed `reuseSession` cron field or CLI flags: isolated cron runs still always call `resolveCronSession` with `forceNew: input.job.sessionTarget === "isolated"`, the cron job type/schema omit `reuseSession`, and current docs explicitly describe isolated cron as fresh-per-run while directing persistent context to `current` or `session:<id>`. The PR is stale and broad, with unrelated test/runtime seam churn, but the core feature is still a real OpenClaw code/docs/API change rather than something already shipped or externally actionable.

## Best Possible Solution

Keep the PR open only if maintainers want this product behavior, then rebase and narrow it to the cron contract: add `reuseSession` to cron types/schema/normalization/service validation, CLI add/edit, isolated runner `forceNew` handling, protocol clients/docs, and focused tests for default fresh behavior, opt-in reuse, invalid main-session reuse, and patch rollback. If maintainers prefer the current product boundary, close manually with a product comment explaining that persistent cron context should use `current` or `session:<id>` instead of making isolated sessions reusable.
