## Summary

Current `main` already covers the practical failure described here. The gateway channel manager does not start a second in-process Telegram account task while the old one is still live, including the stop-timeout path used by hot reload, and Telegram `getUpdates` 409 conflicts now trigger webhook-reset + fresh transport rebuild with regression coverage. The proposed per-token in-process map would not detect a separate external process anyway.

## Evidence

## Risks / Open Questions

- A separate external process polling the same bot token can still cause Telegram-side conflicts until that external poller stops; OpenClaw now recovers/retries from 409s, but an in-process token map would not have solved cross-process contention.

## Close Comment

Closing from a Codex review because current `main` already covers this well enough. Hot-reload does `stopChannel()` then `startChannel()`, but the channel manager keeps the old account task reserved and refuses a new start while that task is still live, including the timeout path (`src/gateway/server-channels.ts:306`, `src/gateway/server-channels.ts:570`, `src/gateway/server-reload-handlers.ts:175`). On the Telegram side, `getUpdates` 409 conflicts now reset/retry on a fresh transport instead of reusing the terminated socket (`extensions/telegram/src/polling-session.ts:366`), with regression coverage in `extensions/telegram/src/polling-session.test.ts:917` and `extensions/telegram/src/monitor.test.ts:790`. This shipped in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`; changelog at `CHANGELOG.md:225`). The proposed per-token in-process map also would not detect a separate external process polling the same bot token, so the current recovery-based behavior is the relevant fix on `main`.
