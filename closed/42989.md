## Close Comment

Closing this as better suited for [ClawHub](https://clawhub.ai/)/community plugin work after Codex review.

Current `main` has moved new provider integrations behind the plugin/provider-plugin system rather than adding more provider-specific wiring in core. This PR implements Model Hub by editing legacy core registries (`src/agents/*`, `src/commands/*`, `src/media-understanding/*`, `src/providers/*`), but current maintainer guidance says provider auth/onboarding/catalog behavior should live in a provider plugin or external plugin package instead.

What I checked:

- **Vision says optional capability should ship as plugins:** `VISION.md:52-59` says core should stay lean and optional capability should usually ship as plugins, with external plugin distribution preferred and a high bar for adding optional plugins to core. ([`VISION.md:52`](https://github.com/openclaw/openclaw/blob/564f820efa2b/VISION.md#L52), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Extension boundary keeps provider behavior out of core:** `extensions/AGENTS.md:39-41` explicitly says provider auth, onboarding, catalog selection, and vendor-only product behavior should stay local to the provider plugin and not be moved into core. ([`extensions/AGENTS.md:39`](https://github.com/openclaw/openclaw/blob/564f820efa2b/extensions/AGENTS.md#L39), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Official docs tell contributors to build new providers as provider plugins:** `docs/plugins/sdk-provider-plugins.md:11-13` describes adding a model provider via a provider plugin, and `docs/plugins/sdk-provider-plugins.md:31-95` shows manifest-driven auth/env-var/provider setup intended for ClawHub or external plugin publication. Public docs: [`docs/plugins/sdk-provider-plugins.md`](https://docs.openclaw.ai/plugins/sdk-provider-plugins). ([`docs/plugins/sdk-provider-plugins.md:11`](https://docs.openclaw.ai/plugins/sdk-provider-plugins), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Current onboarding/auth-choice flow is manifest-driven:** `src/commands/auth-choice-options.ts:26-56` now resolves provider setup contributions dynamically from provider flows instead of requiring each provider to patch a static core auth-choice table. ([`src/commands/auth-choice-options.ts:26`](https://github.com/openclaw/openclaw/blob/564f820efa2b/src/commands/auth-choice-options.ts#L26), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Current env-var discovery is plugin-manifest driven:** `src/secrets/provider-env-vars.ts:76-117` builds provider auth env-var candidates from the plugin manifest registry, and `src/agents/model-auth-env-vars.ts:7-16` is just a thin wrapper over that manifest-based resolver. ([`src/secrets/provider-env-vars.ts:76`](https://github.com/openclaw/openclaw/blob/564f820efa2b/src/secrets/provider-env-vars.ts#L76), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Bundled provider example shows the intended shape:** `extensions/kilocode/openclaw.plugin.json:2-23` declares provider auth metadata in the manifest, and `extensions/kilocode/index.ts:10-33` registers the provider/auth/catalog through `defineSingleProviderPluginEntry(...)` rather than core edits. Model Hub is the same class of integration: an OpenAI-compatible routed provider with API-key auth and model discovery. ([`extensions/kilocode/index.ts:10`](https://github.com/openclaw/openclaw/blob/564f820efa2b/extensions/kilocode/index.ts#L10), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))

So I’m closing this as a scope-fit item for the plugin/community path rather than keeping it open as an OpenClaw core request.
