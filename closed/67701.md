## Summary

Keep open. Current `main` has partial session-accounting mitigations through `promptTokens`/`lastCallUsage` and a regression test that preserves an existing `totalTokens`, but it still does not persist `compactResult.result.tokensAfter` from overflow auto-compaction when provider usage is unavailable and the session starts with no token total. The PR is not cleanly landable as-is because it bundles unrelated changes and has a review concern, but the core bug fix is not implemented on `main`.

## Best Possible Solution

Keep the PR open, but ask for a focused rebase or maintainer cherry-pick: thread overflow/timeout auto-compaction `tokensAfter` into current session accounting, use it when usage/prompt snapshots are unavailable, preserve the existing `promptTokens`/`lastCallUsage` behavior, and add regression coverage for a session with no prior `totalTokens`. Split unrelated ACP, cron delivery, and Telegram changes into separate PRs.
