## Summary

Keep #71744 open. Current main has structured model-fallback logs and FallbackSummaryError attempt summaries, but the official runtime trajectory sidecar does not record per-fallback-step events, and the normal operator-facing failure path can still suppress mixed fallback-chain details. The linked #62684 is related observability context, not a shipped fix or canonical duplicate for fallback-step runtime events.

## Best Possible Solution

Keep the issue open and implement a narrow core observability seam: pass a sanitized fallback-step observer from the embedded runner into runWithModelFallback, emit trajectory events for candidate_failed, skip/probe, candidate_succeeded, and chain_exhausted with from/to model, reason/detail, chain position, and final outcome, then add focused tests covering primary failure followed by fallback failure and the exported trajectory event list. Preserve the existing model_fallback_decision logs as secondary diagnostics.

## Risks / Open Questions

- Implementation should reuse the existing trajectory redaction and event-size limits so provider error strings do not leak secrets or oversized payloads.
- The fallback loop is generic and hot; adding runtime events should avoid coupling it directly to session-specific recorder state unless passed through a narrow callback or observer seam.
