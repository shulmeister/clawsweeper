## Summary

Keep #49736 open. Current main has adjacent improvements for custom provider SecretRefs and loopback no-key custom providers, but the reporter's confirmed failure mode still appears valid: `apiKey: "ollama-local"` on a generated custom provider such as `custom-192-168-0-222-11434` is treated as a non-secret auth marker rather than as a usable literal placeholder, so auth resolution can still throw before the model call.

## Best Possible Solution

Keep the issue open and fix the custom-provider auth resolver so plugin-owned non-secret markers are scoped to the owning provider, or so custom provider literal placeholders like `ollama-local` are accepted when explicitly configured. Add a regression test for a generated custom provider ID with `apiKey: "ollama-local"`, and preserve the existing behavior for real Ollama plugin marker auth and managed SecretRefs.

## Evidence
