## Summary

Current main already advances model fallback for the reported HTTP 400 provider rejection shape. It may classify the exact `model_not_supported` sample as `format` rather than `model_not_found`, but that is still failover-worthy and solves the reported no-fallback behavior. The same relevant fallback paths are present in v2026.4.24.

## Best Possible Solution

Close this issue as shipped. Keep the current broad failover behavior; a separate narrow enhancement could add a `model_not_supported` alias to the `model_not_found` telemetry bucket, but it is not needed to fix the reported fallback failure.
