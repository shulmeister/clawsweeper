## Summary

Current `main` already carries the loader-side fix for this regression: cached plugin registry restores now include `memoryCapability`, and read-only `activate:false` snapshot loads explicitly preserve the previously registered memory capability so `memory-wiki` bridge/status paths no longer lose `publicArtifacts`. The fix is covered by regression tests and documented in the `v2026.4.20` changelog.

## Evidence

- **Cache restores include memory capability:** Active cache hits restore `cached.memoryCapability` back into memory plugin state, instead of restoring only the split legacy fields.
  - file: [src/plugins/loader.ts:1996](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/plugins/loader.ts#L1996)

  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Snapshot loads preserve prior capability:** Before each plugin register call, the loader snapshots `previousMemoryCapability`, and for non-activating snapshot loads it restores that prior capability after registration. This directly addresses the reported clobbering path during repeated read-only restores.
