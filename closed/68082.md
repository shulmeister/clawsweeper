## Summary

Current `main` added real mitigations for the reported downstream effects: chat/history and memory/dreaming readers now strip OpenClaw internal runtime context before exposing or flattening transcript content. But the underlying parent-session persistence path still sends `subagent_announce` through the normal gateway `agent` flow and still persists the injected completion envelope as a `role: "user"` message with inter-session provenance rather than an internal/system role. That means the raw session JSONL contamination described in #68082 is not fully fixed on `main`, even though some consumers now sanitize it away.

## Evidence

- **Subagent announce still enters the parent via normal agent ingress with `sourceTool: "subagent_announce"`:** `deliverSubagentAnnouncement` still calls gateway `agent` with the announce text and `inputProvenance.kind = "inter_session"`, `sourceTool = "subagent_announce"`; this is the same persistence path the issue is about.
  - file: [src/agents/subagent-announce-delivery.ts:365](https://github.com/openclaw/openclaw/blob/c2139635ff82b367b734ea052487258b1b33f060/src/agents/subagent-announce-delivery.ts#L365)

- **Persistence layer still keeps these messages as `role: "user"` and only adds provenance:** The wrapper applies `applyInputProvenanceToUserMessage()` before persistence, and that helper explicitly only annotates user-role messages without changing their role. ## Risks / Open Questions

- The user-visible and memory/dreaming symptoms are partly mitigated, which may hide the fact that raw parent-session JSONL still stores internal completion envelopes as ordinary user-role messages.
- Any transcript consumer that reads JSONL directly or bypasses the newer sanitizers can still ingest or surface the leaked internal context.
- The mitigation commit is on current `main` but is not contained in the provided latest release tag `v2026.4.23`, so release users still need an actual shipped fix as well.
