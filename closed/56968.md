## Summary

Keep #56968 open. Current main still has the reported core status failure path: `openclaw status` resolves memory through the built-in `agents.*.memorySearch` config and returns a null memory snapshot before it ever asks the active memory plugin for status. With `memorySearch.enabled: false`, that null snapshot is rendered as `enabled (plugin <slot>) · unavailable`, matching the report and the confirming `memory-cms` comment. Related #60177/#65066 track a memory-lancedb runtime-registration gap, but that does not fully supersede this generic status-gating bug.

## Best Possible Solution

Keep #56968 open and fix the core status resolver. The likely maintainer path is to make `resolveSharedMemoryStatusSnapshot()` treat non-`memory-core` memory-slot plugins as plugin-owned: query the active memory runtime/status path before applying built-in `agents.*.memorySearch` and default-store gates, or add an explicit plugin status hook if the runtime manager contract is insufficient. Add regression coverage for a third-party memory slot with `agents.defaults.memorySearch.enabled = false` whose manager returns healthy status, and coordinate separately with #60177/#65066 for memory-lancedb runtime registration.
