## Summary

Keep open. Current main has partial silent-output defenses, but it still runs the pre-compaction memory flush synchronously before the foreground user turn, and the memory-flush prompt still travels as a normal embedded run prompt rather than a clearly hidden/non-transcript internal operation. The related PR #58975 is still an open, unmerged attempt, and adjacent ClawSweeper reviews keep the blocking and persisted-prompt leak items open.

## Best Possible Solution

Keep #58956 open. The best maintainer path is to make pre-compaction memory flush a truly internal operation: either move it to a separate/background lane that does not preempt the foreground user turn, or explicitly surface a queued/deferrable state while preserving the user's active request. The flush prompt should be stored with hidden/internal provenance, a separate non-visible `transcriptPrompt`, or a chat.history sanitizer backed by structural metadata, with regression tests covering WebChat history, streaming/final UI updates, session JSONL, and active-turn ordering. PR #58975 can be used as context, but current main does not yet solve the central issue.
