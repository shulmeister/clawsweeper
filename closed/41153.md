## Summary

Current `main` already hardens Telegram polling stall detection well enough: it uses a 120s default watchdog threshold, exposes a configurable `channels.telegram.pollingStallThresholdMs` override with per-account support, wires that value into the polling session, documents the behavior, and covers it with regression tests. That makes this PR obsolete even though `main` chose a config-driven implementation instead of the PR’s exact timeout-derived formula.

## Evidence

- **Telegram polling now uses a 120s default stall threshold and configurable override:** `TelegramPollingSession` defines `DEFAULT_POLL_STALL_THRESHOLD_MS = 120_000`, clamps custom values between 30s and 10m, stores the resolved threshold on construction, and feeds it into stall detection.

## Risks / Open Questions

- `main` does not implement this PR's exact approach: the threshold is config-driven rather than derived from the runner fetch timeout, and the watchdog interval remains 30s instead of 10s.
- If someone specifically wants automatic threshold scaling from custom grammY fetch timeouts, that would need a separate follow-up; it is not required to close this PR because the reported false-positive stall behavior is already addressed on `main`.
