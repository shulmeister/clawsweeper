## Summary

Close as `clawhub`: current main already keeps full session transcripts server-side on the gateway, and exporting/mirroring them to an arbitrary extra path fits the documented plugin/ClawHub extension model rather than OpenClaw core.

## Evidence

- **Vision scope says optional capability should ship as plugins:** The project vision explicitly says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/VISION.md#L54)

  - file: [docs/concepts/session.md:74](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/concepts/session.md#L74)

- **Raw transcript access is intentionally file-based, not the agent-facing history tool:** Session tool docs say `sessions_history` is bounded/safety-filtered and that exact byte-for-byte transcript access should use the transcript file on disk instead.
  - file: [docs/concepts/session-tool.md:36](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/concepts/session-tool.md#L36)

Hook docs list `before_message_write`, `session_start`, `session_end`, and `gateway_start/gateway_stop`, which are enough to mirror/export transcripts without making this a core feature.
  - file: [docs/plugins/hooks.md:10](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/plugins/hooks.md#L10)

## Risks / Open Questions
