## Summary

Keep open. Current main still preserves the reported behavior: `openclaw update` refreshes an installed gateway service by rerunning `gateway install --force` with the caller environment, gateway service construction still copies HTTP(S)/proxy variables into the managed service environment, and systemd still serializes those entries as `Environment=` lines. The Feishu side of the failure also remains plausible on main because WebSocket client creation still injects an ambient proxy agent whenever HTTP(S) proxy env is present, with no Feishu-specific opt-out or target-domain bypass in the Feishu config surface.

## Best Possible Solution

Keep the issue open for a maintainer fix. The likely fix is to separate durable, explicit service proxy configuration from transient caller shell or launchd ambient proxy state: `openclaw update` service refresh should not bake temporary `HTTP_PROXY`/`HTTPS_PROXY` into systemd/launchd by default, while preserving deliberate service env sources such as config or state-dir `.env`. Add regression coverage for update-driven service refresh. Separately, Feishu should honor `NO_PROXY`/domain bypass before creating the ambient proxy agent or expose a Feishu channel/account opt-out for ambient proxy use.
