## Summary

Keep open. Current main still matches the reported Webchat add-attachment path: selected, pasted, and dropped files are read as data URLs, stored in ChatAttachment objects, and assigned directly into Lit reactive chatAttachments state. The related shipped fixes cover later chat.send normalization, not the immediate browser-side image-add failure described here.

## Best Possible Solution

Keep this as a Control UI attachment-add bug. Add a focused browser/UI regression for selecting, pasting, and dropping a large image through OpenClawApp, then either close with runtime evidence if current main no longer reproduces or move image payloads out of chatAttachments reactive state, for example by storing Blob/object URLs or id-keyed raw payloads and converting to base64 only when chat.send runs.
