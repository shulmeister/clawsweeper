## Summary

Current `main` already routes WebChat image uploads through gateway attachment parsing that validates base64, sniffs the actual image MIME, and offloads large images to managed media instead of passing them inline. That behavior is also shipped in `v2026.4.23`, so the reported large-image WebChat failure is already covered on the maintained path.

## Evidence

- **`chat.send` uses the attachment parser for WebChat uploads:** The gateway normalizes incoming RPC attachments and, when attachments are present, calls `parseMessageWithAttachments(...)` before continuing the chat send flow.
  - file: [src/gateway/server-methods/chat.ts:2204](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/gateway/server-methods/chat.ts#L2204)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2160,2235p'`
  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **Large images are intercepted and offloaded instead of sent inline:** `parseMessageWithAttachments` estimates decoded size, sniffs the real MIME, and offloads images larger than `OFFLOAD_THRESHOLD_BYTES = 2_000_000` to managed media storage via `saveMediaBuffer(...)`, appending a `media://inbound/...` ref rather than keeping the full inline payload.
  - file: [src/gateway/chat-attachments.ts:84](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/gateway/chat-attachments.ts#L84)

  - sha: [fd65caf4b0ba](https://github.com/openclaw/openclaw/commit/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc)
- **MIME mismatch is handled in the gateway:** The parser compares provided and sniffed MIME types and uses the sniffed type when they differ, which directly covers the report's 'JPEG data with .png extension' case.
  - file: [src/gateway/chat-attachments.ts:344](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/gateway/chat-attachments.ts#L344)
