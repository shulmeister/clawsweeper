## Summary

PR #44351 is superseded by PR #44412, which was opened specifically to carry the same Telegram direct-thread cron inference fix plus the review-requested hardening for casing, optional Telegram account IDs, and using the thread-suffix chat ID. Current main still has the original cron inference behavior, so the right cleanup is not to mark this implemented, but to close this narrower/unhardened PR in favor of the canonical follow-up.

## Best Possible Solution

Close PR #44351 as superseded and continue the fix through PR #44412, with maintainer review focused on both the hardened session-key parser and whether cron should preserve Telegram threadId when delivery is inferred from currentDeliveryContext.
