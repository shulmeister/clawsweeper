## Summary

Keep open. Current main still does not implement the PR’s in-process fallback dedupe for clobbered config snapshots when config-health persistence fails, and the PR itself has unresolved review concerns about suppressing later incidents and retries. The related user issue may be mitigated by newer last-known-good recovery in common backup-backed cases, but that is not strong enough to close this focused PR as implemented or obsolete.

## Best Possible Solution

Keep the PR open, but do not merge it as-is. The best maintainer path is a focused config I/O hardening fix that dedupes repeated identical suspicious observations within a process only after durable snapshot/audit handling succeeds, clears or narrows the dedupe key when config returns healthy or the suspicious sequence changes, covers both async and sync observe paths, and adds regression tests that simulate config-health write failure.
