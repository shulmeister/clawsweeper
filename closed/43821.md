## Summary

Keep #43821 open. Current main has meaningful partial HTTP(S) env-proxy support, including Gateway startup bootstrapping, provider/media/memory paths, and some channel-specific fixes. It still does not implement the requested comprehensive global behavior: untrusted `web_fetch` is intentionally kept on strict DNS-pinned dispatch even when `HTTP_PROXY` is set, `ALL_PROXY` is explicitly not treated as an EnvHttpProxyAgent-backed proxy, and the issue discussion itself identifies #58034 as only one remaining `web_fetch` slice rather than a complete fix for the global request.

## Best Possible Solution

Keep #43821 open as the umbrella for comprehensive env-proxy behavior. The best path is to turn the current partial implementation into an explicit product contract: document which env vars/protocols are supported, add or reject a trusted `web_fetch` env-proxy opt-in, audit remaining direct `fetch` and third-party client/WebSocket call sites, and close narrower fixed slices only when their behavior is verified in code, tests, and docs.
