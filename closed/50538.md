## Summary

Current `main` already implements the plugin-sdk alias-resolution fix this PR proposes: plugin-sdk roots are resolved from the module being loaded, not from `loader.ts`, and cached Jiti loaders are keyed by the resolved alias map so mixed source/dist roots do not share the wrong loader. The regression coverage for those cases is also already present, and the relevant implementation is present in release `v2026.4.22`.

## Evidence

- **Module-root lookup follows the loaded plugin module:** `resolveLoaderPluginSdkPackageRoot(...)` derives the OpenClaw/package root from the module path plus trusted hints, and `listPluginSdkAliasCandidates(...)` uses that root for source/dist plugin-sdk alias selection rather than anchoring on `loader.ts`.
  - file: [src/plugins/sdk-alias.ts:143](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/sdk-alias.ts#L143)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Alias map is built from `modulePath`:** `buildPluginLoaderAliasMap(...)` passes the loaded module path into both `resolvePluginSdkAliasFile(...)` and `resolvePluginSdkScopedAliasMap(...)`, which is the behavior this PR was trying to restore for source checkout plugins.
  - file: [src/plugins/sdk-alias.ts:600](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/sdk-alias.ts#L600)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Cached Jiti loaders are keyed by resolved aliases:** `createPluginLoaderJitiCacheKey(...)` serializes the sorted alias map, and `getCachedPluginJitiLoader(...)` uses that cache key when memoizing loaders, so different source/dist alias maps do not reuse the same Jiti instance.
  - file: [src/plugins/sdk-alias.ts:737](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/sdk-alias.ts#L737)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Loader cache uses the computed config key:** `getCachedPluginJitiLoader(...)` resolves the per-module alias map/config and memoizes by `${jitiFilename}::${cacheKey}`; this is stronger than the PR’s proposed boolean/source-dist split because the full alias map participates in the cache identity.
  - file: [src/plugins/jiti-loader-cache.ts:13](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/jiti-loader-cache.ts#L13)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Regression tests already cover the reported behavior:** Committed tests assert that plugin-sdk aliases come from the module being loaded, reject unsafe cwd fallback, and keep Jiti cache keys stable/scoped by resolved alias configuration.
  - file: [src/plugins/sdk-alias.test.ts:618](https://github.com/openclaw/openclaw/blob/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f/src/plugins/sdk-alias.test.ts#L618)
  - sha: [9b7f6250f43b](https://github.com/openclaw/openclaw/commit/9b7f6250f43bbc1816cdaf26e8c07116dfa4975f)
- **Shipped in the latest release commit:** The same `src/plugins/sdk-alias.ts` and `src/plugins/jiti-loader-cache.ts` implementation is present in release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`), so this was not only added on `main` but has already shipped.
