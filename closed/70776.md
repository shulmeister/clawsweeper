## Summary

Current `main` does not match the reported root cause. Provider-prefixed config refs are parsed into separate `provider`/`model` fields before reply handling and thinking validation, and the Anthropic plugin still exposes `xhigh` for bare `claude-opus-4-7` in both runtime code and tests.

## Evidence

- **Configured model refs are split before runtime use:** `resolveConfiguredModelRef()` parses `agents.defaults.model.primary` through `resolveModelRefFromString()` and returns `resolved.ref`, so a config value like `anthropic/claude-opus-4-7` becomes separate `provider` and bare `model` fields before reply flow starts.
  - file: [src/agents/model-selection-shared.ts:426](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/agents/model-selection-shared.ts#L426)
  - command: `nl -ba src/agents/model-selection-shared.ts | sed -n '420,500p'`
  - sha: [bf0221c5b3de](https://github.com/openclaw/openclaw/commit/bf0221c5b3de3f290a934a314814426c05b59b84)
- **Reply flow carries split provider/model fields:** `get-reply.ts` resolves `{ defaultProvider, defaultModel }` from `resolveDefaultModel()` and initializes `provider`/`model` from those split fields. Heartbeat and channel overrides also go through `resolveModelRefFromString()` before assignment.
  - file: [src/auto-reply/reply/get-reply.ts:193](https://github.com/openclaw/openclaw/blob/bf0221c5b3de3f290a934a314814426c05b59b84/src/auto-reply/reply/get-reply.ts#L193)

That path uses the normalized bare model field, not the original provider-prefixed config string.

- **Anthropic Opus 4.7 profile still includes xhigh:** The Anthropic runtime plugin adds `xhigh`, `adaptive`, and `max` when `isAnthropicOpus47Model(modelId)` matches, and uses that same matcher for the default level logic.
