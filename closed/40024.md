## Summary

Keep #40024 open. Current main still does not satisfy the reporter's baseUrl-only local OpenAI-compatible proxy case: a configured fallback provider without an explicit `api` still resolves to `openai-responses`, the embedded transport then uses the Responses client instead of `/v1/chat/completions`, and private/loopback provider fetches are blocked unless `request.allowPrivateNetwork` is set.

## Best Possible Solution

Keep the issue open. Fix either by inferring/selecting `openai-completions` for trusted custom OpenAI-compatible `/v1` providers that do not support Responses, or by failing before the embedded run with an actionable message requiring `models.providers.<id>.api: "openai-completions"` and `models.providers.<id>.request.allowPrivateNetwork: true` for local/private endpoints. Add a regression test for a `local-agent-proxy` provider at `http://127.0.0.1:3000/v1` that verifies transport selection and error behavior.
