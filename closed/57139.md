## Summary

Keep #57139 open. Current main still exposes `sessions_history` only through store-indexed session keys/session IDs and does not list or select `.jsonl.reset.*` archives. Reset rollover archives the old transcript after replacing the session store entry with a new `sessionId`, so the old archive is no longer addressable through `sessions_list`/`sessions_history`. The bundled `session-logs` skill is a useful workaround for non-sandboxed filesystem access, but it is not the requested Gateway-mediated tool surface for sandboxed agents. Related PR #60409 is still open and only proposes a partial fallback, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open and implement a core/Gateway session archive surface. The likely product path is to extend the session tooling/protocol with an explicit archived transcript list/read contract scoped by current agent and existing session visibility rules, then wire `sessions_history` or a new `sessions_list_archived`/archive read tool to that contract with tests for manual `/new`, `/reset`, daily reset, sandboxed agents, retention pruning, and multiple archives for the same prior session. PR #60409 can inform a narrower fallback, but it does not fully satisfy this issue unless it also makes archived sessions discoverable and selectable.
