## Summary

Current `main` already prefers the manifest `id` over the npm package name when deriving plugin discovery `idHint`s, and the shipped test suite covers the mismatched-name case this PR addresses.

## Evidence

- **Discovery now prefers manifest ids:** `deriveIdHint` accepts an optional `manifestId` and returns it before falling back to the package name. The discovery flow loads `openclaw.plugin.json` via `resolveIdHintManifestId(...)` and passes that value into `deriveIdHint(...)` for package plugins.
  - file: [src/plugins/discovery.ts:422](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/discovery.ts#L422)
  - command: `nl -ba src/plugins/discovery.ts | sed -n '422,457p'; nl -ba src/plugins/discovery.ts | sed -n '883,917p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Mismatched package-name case is covered by tests:** The discovery tests create a package named `@example/local-provider` with manifest/plugin id `local`, then assert discovery includes `local` and excludes `local-provider`. That is the same behavioral fix this PR proposes.
  - file: [src/plugins/discovery.test.ts:594](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/discovery.test.ts#L594)
