## Summary

Current main already fixes the cron task-audit timestamp problem with broader registry-level timestamp normalization. The cron path still flows through that registry creation path, regression tests cover skewed create/update/restore cases, and the original PR diff is obsolete and unsafe because it removed task-flow exports that current callers still need.

## Best Possible Solution

Close #66133 as implemented on main. Keep the broader registry-level normalization that landed with #71871, and do not revive this PR's narrow diff because it removed task-flow executor APIs still used by production callers. Any remaining timestamp-audit problem should be filed as a fresh, narrow regression against current main with current audit output and task records.
