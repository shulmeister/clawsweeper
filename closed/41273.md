## Summary

Current `main` does not reproduce the PR's claimed production leak path. `startChannelHealthMonitor` still supports an optional `abortSignal`, but the only production caller does not pass one, so reload/shutdown `.stop()` calls do not leave an abort listener behind on current `main`.

## Evidence

- **Current stop implementation has no listener cleanup:** `stop()` clears the interval but does not call `removeEventListener`; the abort listener is only registered when `abortSignal` is present.
  - file: [src/gateway/channel-health-monitor.ts:181](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/gateway/channel-health-monitor.ts#L181)

  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Only production caller does not pass abortSignal:** `startGatewayChannelHealthMonitor` calls `startChannelHealthMonitor` with `channelManager`, interval, stale threshold, and restart limits only. No `abortSignal` is forwarded.
  - file: [src/gateway/server-runtime-services.ts:38](https://github.com/openclaw/openclaw/blob/e94c0bf51542fe518e49ad4e2ee29882891b5603/src/gateway/server-runtime-services.ts#L38)

  - sha: [e94c0bf51542](https://github.com/openclaw/openclaw/commit/e94c0bf51542fe518e49ad4e2ee29882891b5603)
- **Reload path only calls stop on the existing monitor:** Config reload stops the current health monitor and creates a new one, but since the production monitor was created without an `abortSignal`, this path does not retain an abort-listener closure.
