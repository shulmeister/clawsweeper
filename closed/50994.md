## Summary

Keep PR #50994 open for maintainer judgment. Current main does not implement the PR’s stated empty-`expr` preservation behavior, and the PR is not safely obsolete: linked #50942 was closed because canonical cron object schedules now work, but this PR’s remaining behavior is a narrower validation/error-shape change with unresolved direct-service safety concerns. It should not be auto-closed as implemented or superseded, and it should not be merged as-is without a maintainer decision on blank cron expression handling.

## Best Possible Solution

Keep the PR open for maintainer review, but do not merge it as-is. The best path is to either rework normalization to reject blank cron expressions before returning normalized schedules, with tests for gateway validation and direct `CronService.add`/`update` callers, or have a maintainer explicitly close the PR if the current missing-`expr` error is the accepted behavior.
