## Summary

Current `main` and the shipped `v2026.4.23` code already support direct/root-level LINE credentials and compute status warnings from resolved account snapshots, not from a raw `accounts[]` config shape. The report's stated mismatch does not match the current implementation or its regression tests.

## Evidence

- **Root-level LINE credentials are supported:** `resolveLineAccount` reads `channels.line.channelAccessToken` / `channelSecret` for the default account, and `listLineAccountIds` adds the default account when a root-level token is present. That means the direct config format described in the issue is a supported path, not an unsupported alternative to `accounts`.
  - file: [extensions/line/src/accounts.ts:38](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/line/src/accounts.ts#L38)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Status warning depends on computed snapshot state:** The LINE status adapter marks the account `configured` when both credentials are present and passes `tokenSource` into the generic dependent-credential collector. The collector only emits `LINE channel access token not configured` when `configured === false` and `tokenSource` is effectively `none`.
  - file: [extensions/line/src/status.ts:20](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/line/src/status.ts#L20)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **`openclaw status` uses resolved account snapshots, not raw config objects:** The gateway status path resolves each plugin account, builds a `ChannelAccountSnapshot`, and stores `channelAccounts[channelId]` as an array of those snapshots. The issue's claimed `collectStatusIssues` mismatch against raw config shape does not match how status payloads are built on current `main`.
  - file: [src/gateway/server-methods/channels.ts:263](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/src/gateway/server-methods/channels.ts#L263)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Regression test covers the no-warning case:** The LINE status test explicitly asserts that a configured sanitized snapshot returns no issues. This matches the expected behavior from the issue and contradicts the reported current behavior.
  - file: [extensions/line/src/channel.status.test.ts:13](https://github.com/openclaw/openclaw/blob/938b53698ec17934a5d1a6cffb279c584f64d9f3/extensions/line/src/channel.status.test.ts#L13)
  - sha: [938b53698ec1](https://github.com/openclaw/openclaw/commit/938b53698ec17934a5d1a6cffb279c584f64d9f3)
- **Same logic is already in the latest release:** There is no diff between `v2026.4.23` and current `main` for the inspected LINE status/account files and shared status helper, so the shipped release already contains the same status logic reviewed here.
  - file: [src/plugin-sdk/status-helpers.ts:352](https://github.com/openclaw/openclaw/blob/899e927dab9ef2d6164004d34a30410b3506b6f5/src/plugin-sdk/status-helpers.ts#L352)
