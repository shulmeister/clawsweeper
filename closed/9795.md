## Summary

Keep open. Current `main` still has the reported unanchored `sanitizeMimeType()` regex, and the surrounding flow can still collapse malformed MIME strings to an allowed prefix before enforcement. The latest release `v2026.4.22` also still contains the same code, so there is no strong evidence this was already fixed on `main` or shipped.

## Evidence

- **Current main still uses unanchored regex:** `sanitizeMimeType()` lowercases and trims, then matches only a prefix with `^...` and no end anchor, so trailing garbage is accepted and truncated.
  - file: [src/media-understanding/apply.ts:82](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/media-understanding/apply.ts#L82)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '70,110p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Input can reach sanitizer with trailing garbage intact:** `normalizeMimeType()` only splits on `;` and lowercases; other suffix garbage remains. Example class: `text/plain<script` would stay non-empty and reach `sanitizeMimeType()`.
  - file: [src/media/input-files.ts:135](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/media/input-files.ts#L135)
