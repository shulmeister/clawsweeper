## Summary

Keep open. Current main has a native/generated Control UI i18n system and many localized strings, but the issue asks for more complete admin-interface page coverage and that remains incomplete: detailed Config, Quick Settings, and Chat surfaces still render hardcoded English body labels, placeholders, buttons, and status copy. This is not a safe implemented-on-main, duplicate, or ClawHub close.

## Best Possible Solution

Keep #66267 open. The best path is a focused Control UI localization pass that extends `ui/src/i18n/locales/en.ts`, replaces remaining hardcoded user-facing strings in `ui/src/ui/**` with `t(...)`, regenerates locale bundles through the existing `scripts/control-ui-i18n.ts` pipeline, and adds a raw-string/i18n coverage guard where practical. Maintainers can separately decide whether translate.js is acceptable, but the core product need is still incomplete page-body localization.
