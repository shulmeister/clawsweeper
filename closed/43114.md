## Summary

AINFT is not implemented on current `main`, but this PR targets core seams that have been replaced by the provider-plugin architecture. Current OpenClaw expects new OpenAI-compatible providers to ship as plugins with manifest-owned auth/catalog/onboarding metadata, and the docs explicitly route provider plugins to ClawHub publishing rather than more hardcoded core wiring.

## Evidence

- **Provider plugins are the documented path for OpenAI-compatible proxies:** The provider-plugin guide explicitly says to use this path when adding an OpenAI-compatible proxy or custom LLM, and shows the resulting `openclaw onboard --...-api-key` flow.
  - file: [docs/plugins/sdk-provider-plugins.md:5](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/plugins/sdk-provider-plugins.md#L5)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Provider plugins are meant to be published via ClawHub:** The same guide has a dedicated 'Publish to ClawHub' section for provider plugins, which is the repository's documented distribution path for external provider integrations.
  - file: [docs/plugins/sdk-provider-plugins.md:679](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/docs/plugins/sdk-provider-plugins.md#L679)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Onboard provider auth flags now come from plugin manifests:** `register.onboard.ts` builds `ONBOARD_AUTH_FLAGS` from `resolveManifestProviderOnboardAuthFlags()`, and `provider-auth-choices.ts` resolves those flags from plugin manifest metadata instead of hardcoded core lists.
  - file: [src/cli/program/register.onboard.ts:50](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/cli/program/register.onboard.ts#L50)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Onboarding auth choices are plugin-owned by contract:** `OnboardOptions` now reserves dynamic provider option keys for manifest `providerAuthChoices[]`, and static core auth choices only include the generic custom-provider path.
  - file: [src/commands/onboard-types.ts:27](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/commands/onboard-types.ts#L27)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **Provider catalogs are loaded from bundled plugin artifacts, not hardcoded static provider builders:** `models-config.providers.static.ts` now resolves bundled provider catalog artifacts from plugin metadata and loads them dynamically, which is incompatible with this PR's older core-builder approach.
  - file: [src/agents/models-config.providers.static.ts:24](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/models-config.providers.static.ts#L24)
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)
- **AINFT is absent from current main:** A repo-wide search for `AINFT|ainft` returned no matches in the checked-out `main`, so this was not independently implemented elsewhere in core.
  - command: `rg -n "AINFT|ainft" -S .`
  - sha: [41c5ffc5d5f4](https://github.com/openclaw/openclaw/commit/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00)

## Risks / Open Questions

- AINFT remains unsupported out of the box until someone ships a provider plugin.
- If maintainers want AINFT bundled rather than external, it still needs a fresh plugin-architecture PR under `extensions/` instead of reviving this core-wiring PR.
