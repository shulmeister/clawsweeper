## Summary

Current `main` already covers the underlying request with file-backed `SecretRef` support for channel credentials, including Discord account tokens, so a separate literal `tokenFile` field is not required for the reported use case.

## Evidence

- **Discord account tokens accept secret inputs:** `DiscordAccountConfig.token` is typed as `SecretInput`, so account tokens can be plaintext or SecretRef-backed rather than inline-only strings.
  - file: [src/config/types.discord.ts:239](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/src/config/types.discord.ts#L239)

## Risks / Open Questions

- There is no literal `tokenFile` key for Discord; the supported mechanism is `SecretRef` with `source: "file"`, so existing requests/examples using `tokenFile` need translation.
- The issue body overreaches to channels that do not use static token/password credentials in the same way (for example WhatsApp runtime creds), and those remain intentionally outside the generic SecretRef surface.
