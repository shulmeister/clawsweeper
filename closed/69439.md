## Summary

Current main already supports the requested per-agent control plane: per-agent tool allow/deny policy, per-agent exec security defaults, per-agent sandbox settings, per-agent auth stores, and per-agent exec allowlists. The exact proposed `tools.exec.policy`/`tools.exec.deny` spelling is not the supported schema, but the documented `agents.list[].tools.deny`, `agents.list[].tools.exec.security`, sandbox, and exec-approvals allowlist controls solve the stated family-agent restriction use case well enough.

## Best Possible Solution

Keep the shipped schema and docs as the supported path: configure restricted agents with `agents.list[].tools.deny` or `agents.list[].tools.exec.security: "deny"`, use exec approvals allowlists for agents allowed to run limited commands, and enable per-agent sandboxing/workspace access for filesystem isolation. For hostile multi-tenant boundaries, continue documenting separate OS users/hosts/gateways rather than presenting one Gateway as a hard auth boundary.
