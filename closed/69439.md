## Summary

Keep open. Current `main` already supports per-agent tool policy, per-agent `tools.exec` config, and sandbox/workspace scoping, so the issue body is partly outdated. But the requested command-variant blocking is still explicitly unsupported: `gateway.nodes.denyCommands` is exact command-name matching only and does not inspect shell text. The multi-tenant isolation story also remains documented as shared-user hardening rather than strong isolation for mutually untrusted operators.

## Evidence

That means current `main` does recognize per-agent exec policy keys, but not the exact `policy`/`deny` shape proposed in the issue body.

## Risks / Open Questions

- The issue's proposed config shape is still mismatched with current schema, so a reporter following the body literally could still hit validation errors even though equivalent per-agent controls now exist under different keys.
- The 'block dangerous command variants inside other strings' part is still unimplemented on main, so closing as implemented would overstate current behavior.
- OpenClaw's own security docs still frame shared-user setups as cooperative hardening and recommend separate gateways or separate OS users/hosts for mutually untrusted operators, so strong multi-tenant isolation is not fully solved in core.
