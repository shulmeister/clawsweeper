## Summary

Current `main` does not match the bug this PR targets. The `/v1/responses` handler now validates flat top-level tool definitions, explicitly rejects wrapped `function` tools in parity tests, and already converts accepted request tools into the internal wrapped shape after validation. The PR's normalization path is obsolete against current code.

## Evidence

- **Current schema expects flat top-level tool fields:** `ToolDefinitionSchema` defines function tools with top-level `type`, `name`, `description`, `parameters`, and `strict`; there is no required top-level `function` wrapper in the current request schema.
  - file: [src/gateway/open-responses.schema.ts:165](https://github.com/openclaw/openclaw/blob/8acc92c881abc1cc02ff5695e15b75b94e294b07/src/gateway/open-responses.schema.ts#L165)
  - command: `nl -ba src/gateway/open-responses.schema.ts | sed -n '150,220p'`
  - sha: [8acc92c881ab](https://github.com/openclaw/openclaw/commit/8acc92c881abc1cc02ff5695e15b75b94e294b07)
- **Parity tests accept flat format and reject wrapped format:** The parity suite has one test that accepts `{ type: "function", name, ... }` and another that rejects `{ type: "function", function: {...} }`, which is the opposite of the PR's premise.
  - file: [src/gateway/openresponses-parity.test.ts:114](https://github.com/openclaw/openclaw/blob/8acc92c881abc1cc02ff5695e15b75b94e294b07/src/gateway/openresponses-parity.test.ts#L114)
  - command: `nl -ba src/gateway/openresponses-parity.test.ts | sed -n '110,260p'`
  - sha: [8acc92c881ab](https://github.com/openclaw/openclaw/commit/8acc92c881abc1cc02ff5695e15b75b94e294b07)
- **HTTP layer already normalizes accepted request tools into internal wrapped tools:** `extractClientTools()` converts validated request tools from the flat request shape into the internal `ClientToolDefinition` wrapper used downstream, so current main already has the needed flat-to-internal translation in the request path.
  - file: [src/gateway/openresponses-http.ts:264](https://github.com/openclaw/openclaw/blob/8acc92c881abc1cc02ff5695e15b75b94e294b07/src/gateway/openresponses-http.ts#L264)
  - command: `nl -ba src/gateway/openresponses-http.ts | sed -n '240,310p'`
  - sha: [8acc92c881ab](https://github.com/openclaw/openclaw/commit/8acc92c881abc1cc02ff5695e15b75b94e294b07)
- **Current parse path cannot emit the reported `tools.0.function` validation failure:** `handleOpenResponsesHttpRequest()` parses `handled.body` directly with `CreateResponseBodySchema`; because that schema no longer defines a wrapped `function` field for tools, the specific validation error described in the PR no longer matches current main.
  - file: [src/gateway/openresponses-http.ts:462](https://github.com/openclaw/openclaw/blob/8acc92c881abc1cc02ff5695e15b75b94e294b07/src/gateway/openresponses-http.ts#L462)

  - file: [src/agents/pi-embedded-runner/anthropic-family-tool-payload-compat.ts:62](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/agents/pi-embedded-runner/anthropic-family-tool-payload-compat.ts#L62)
