## Summary

Keep open. Current main still has the reported behavior in the WebSocket payload builder: `toolChoice` is copied into `tool_choice` even when `tools` is empty, and the current test suite explicitly expects that behavior. The HTTP/completions path also still applies explicit `toolChoice` whenever `context.tools` is present, without checking that the converted tools array is non-empty. Related PR #70790 is relevant and may be the better canonical fix for embedded-runner empty tools, but it is still open and does not prove this PR's WebSocket surface is covered.

## Best Possible Solution

Keep this PR open or explicitly supersede it only after a canonical PR lands. The best fix is a current-main payload egress guard that removes empty `tools` arrays and orphaned `tool_choice`/`parallel_tool_calls` across the embedded HTTP/completions paths, while also updating the WebSocket response-create builder and tests so `tool_choice` is only sent with non-empty tools.
