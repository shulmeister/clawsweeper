## Summary

Keep this PR open. Current main still has memory-only WebChat queued-message state and draft state: queued messages are appended, removed, and drained from `host.chatQueue` without persistence, session switches clear queue/draft state, and reconnect only flushes an already in-memory queue after a seq-gap. The related bug #51549 remains open, the older broad PR #56542 is closed, and #71328 is the separate session-store rotation work. The discussion also shows a maintainer asked for a focused WebChat persistence fix and the author updated this PR in that direction, so this is an active implementation candidate rather than conservative cleanup material.

## Best Possible Solution

Keep this PR open as the active focused implementation candidate for the WebChat queued-message persistence part of #51549. Review it for correct UI-layer persistence wiring, restore only after a real gateway connection/session selection is established, persist cleanup on every queue mutation, and add targeted UI tests for restore, TTL, dedupe, empty-queue cleanup, reconnect drain, and session-switch restore. Track draft persistence explicitly in this PR or as a separate follow-up before closing #51549.
