## Summary

Keep open. Current main has the generic managed web_search/SearXNG registration path and later local-model mitigations, but it still does not solve the reported llama-server behavior where tool calls arrive as assistant text instead of structured OpenAI tool_calls. Related reports cover pieces of the same local-model/tool-call problem, but none cleanly supersedes this web_search + llama-server + prompt-pressure report.

## Best Possible Solution

Keep this open as a local OpenAI-compatible transport compatibility bug. A conservative fix would add an explicit per-model/provider compatibility mode for affected llama.cpp/Qwen-style backends: optionally force an appropriate tool_choice, optionally parse tightly validated JSON/text tool calls only when enabled, validate tool names and arguments against the active tool registry before execution, and document the safe local-model configuration path. Prompt-pressure mitigations such as localModelLean and compaction reserve caps should remain separate supporting work, not be treated as the full fix.
