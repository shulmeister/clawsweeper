## Summary

Current `main` already fixes the reported WhatsApp outbound-media drop by normalizing reply media paths before delivery for both model-initiated replies and CLI delivery, and the WhatsApp sender now consumes the normalized media list directly. The fix appears to be present on `main` after `v2026.4.23`, so this is implemented on `main` but not tied to a shipped release from the local checkout.

## Evidence

- **Shared reply-media normalizer now stages local reply attachments:** `createReplyMediaPathNormalizer(...)` resolves relative/sandbox/absolute local media through `resolveOutboundAttachmentFromUrl(...)`, then writes the staged results back into `mediaUrl`/`mediaUrls`; if all media drops, it adds `⚠️ Media failed.` instead of silently sending text only.
  - file: [src/auto-reply/reply/reply-media-paths.ts:67](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/reply-media-paths.ts#L67)

- **Auto-reply routing now normalizes media before outbound delivery:** The routed reply path creates the normalizer and applies it whenever `resolveSendableOutboundReplyParts(payload).hasMedia` is true, covering normal auto-reply delivery.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:386](https://github.com/openclaw/openclaw/blob/fd65caf4b0bad1b0ac27a82d349c22e92167e7bc/src/auto-reply/reply/dispatch-from-config.ts#L386)
