## Summary

Current `main` routes bound thread follow-ups to the ACP session key stored in the conversation binding, not to a derived `agent:...:acp:...:thread:...` session entry. The reported missing-`acp` metadata path does not match the current routing/binding implementation.

## Evidence

- **Bound conversations retarget to the binding target session:** `initSessionState` resolves the current conversation binding and rewrites `SessionKey` to `binding.targetSessionKey` before session state is loaded, so follow-up thread messages use the bound ACP session key.
  - file: [src/auto-reply/reply/session.ts:192](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/auto-reply/reply/session.ts#L192)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '180,270p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **ACP thread binds store the ACP session key itself:** Thread binding creation calls `getSessionBindingService().bind` with `targetSessionKey: params.sessionKey`; it does not create a separate thread-suffixed ACP target key.
  - file: [src/agents/acp-spawn.ts:871](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/agents/acp-spawn.ts#L871)

  - file: [src/auto-reply/reply/commands-acp.test.ts:627](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/auto-reply/reply/commands-acp.test.ts#L627)
