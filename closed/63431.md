## Summary

Close as `cannot_reproduce`. Current `main` intentionally derives per-agent workspaces under `agents.defaults.workspace`, and the issue’s own `~/.openclaw/agents/local/sessions/...` evidence points to runs for agent `local`, so `/workspace/local` being bootstrapped and recreated matches expected agent-scoped workspace behavior rather than a canonicalization bug in the default workspace.

## Evidence

- **Workspace resolver intentionally appends non-default agent IDs:** `resolveAgentWorkspaceDir()` returns `agents.defaults.workspace` for the default agent, but for non-default agents it returns `path.join(resolveUserPath(fallback, env), id)`. On current `main`, an agent id of `local` therefore resolves to `<configured-workspace>/local` by design.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/agent-scope-config.ts#L149)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '149,171p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Regression test covers nested per-agent workspace shape:** There is an explicit test proving that when `agents.defaults.workspace` is shared and a different agent is default, another agent resolves to a nested path (`/shared-ws/main`). That matches the reported `/workspace/local` pattern.
  - file: [src/agents/agent-scope.test.ts:486](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/agent-scope.test.ts#L486)
  - command: `nl -ba src/agents/agent-scope.test.ts | sed -n '486,504p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Agent creation flow treats derived workspace paths as intended defaults:** `openclaw agents add` pre-fills the new agent workspace from `resolveAgentWorkspaceDir()`, so derived per-agent workspace paths are part of the supported flow, not an accidental fallback.
  - file: [src/commands/agents.commands.add.ts:227](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/commands/agents.commands.add.ts#L227)
  - command: `nl -ba src/commands/agents.commands.add.ts | sed -n '227,235p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Workspace bootstrap recreates deleted derived workspaces on next use:** `ensureAgentWorkspace()` creates the directory, seeds bootstrap files and workspace state, and initializes `.git` for a brand-new workspace. Deleting an agent workspace and then using that agent again will recreate it.
  - file: [src/agents/workspace.ts:325](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/agents/workspace.ts#L325)
  - command: `nl -ba src/agents/workspace.ts | sed -n '325,462p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Issue discussion points to agent-scoped `local` sessions:** The April 18 issue comment says the affected transcript lived under `~/.openclaw/agents/local/sessions/...jsonl`. OpenClaw documents session transcripts under `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`, which matches runs for agent `local` rather than reuse of the default agent workspace.
  - file: [docs/concepts/agent.md:76](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/concepts/agent.md#L76)

## Risks / Open Questions

- Docs currently describe default per-agent workspaces in more than one way (`workspace-<agentId>` in some pages vs derived subdirectories from `agents.defaults.workspace` in current code), which likely contributed to the report’s interpretation.
- If there is still a path that creates `agent:local:*` sessions without the user intentionally selecting/configuring that agent, that would be a separate routing bug; this issue does not provide enough concrete session-key/config evidence to prove that on current `main`.
