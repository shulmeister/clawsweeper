## Summary

#49823 is superseded by the same-author implementation PR #49840. Current main still does not ship a native `maxCostPerRun` cap, so the remaining work should be reviewed and tracked on that PR rather than keeping a duplicate issue open.

## Best Possible Solution

Close #49823 as superseded and make #49840 the canonical review thread. The best product path is to land or revise #49840 so the scheduler/embedded-run dispatch layer has a documented, schema-backed `maxCostPerRun` cap covering heartbeat and cron-style embedded runs, with tests for skipped dispatch and explicit parent/session signaling.
