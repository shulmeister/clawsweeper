## Summary

Keep open. Current `main` still appears to have the reported overwrite path for isolated cron jobs targeting `agent:main:main`, and the proposed PR change has a credible unresolved regression risk from review. This is not implemented on `main`, not obsolete, and not safe to auto-close.

## Evidence

- **Isolated runs still force a fresh session, then reuse the caller key unless it is a cron-prefixed key:** `prepareCronRunContext` calls `resolveCronSession(... forceNew: input.job.sessionTarget === "isolated")`, then sets `runSessionKey = agentSessionKey` unless `baseSessionKey.startsWith("cron:")`. For an isolated job aimed at `agent:main:main`, both keys stay `agent:main:main`.
  - file: [src/cron/isolated-agent/run.ts:482](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/cron/isolated-agent/run.ts#L482)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '470,540p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Persist path still overwrites the agent session key directly:** `createPersistCronSessionEntry` writes `params.cronSession.sessionEntry` straight into `params.agentSessionKey` in memory and on disk. When `runSessionKey === agentSessionKey`, there is no separate run-session record to protect a parent session entry.
  - file: [src/cron/isolated-agent/run-session-state.ts:37](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/cron/isolated-agent/run-session-state.ts#L37)

- **Fresh isolated cron session entries always stamp a new activity time:** `resolveCronSession` sets `sessionEntry.updatedAt = params.nowMs` for the returned entry. Combined with the direct write above, an isolated cron run aimed at the main session key still bumps that key's activity timestamp on current `main`.
