## Summary

Keep open. Current main has newer Claude CLI transcript checks that reduce the later `--resume` crash path, but the reported parser-level bug remains: CLI output parsing still accepts any non-empty `session_id`/`sessionId`/conversation field and the session store still persists that value for CLI providers. There is no backend-aware UUID/session-id format contract or regression test rejecting `rate-limited`, and the linked fix PRs in the provided context were all closed unmerged.

## Best Possible Solution

Add a backend-aware session-id validation contract in the CLI backend parser path, with defaults that keep opaque IDs working for non-Claude backends and require UUID-shaped IDs or explicit validation for Claude-style `--resume` backends. Cover `parseCliJson`, `parseCliJsonl`, streaming JSONL session capture, and session-store persistence with regression tests for `rate-limited`, valid UUIDs, and opaque valid backend IDs.

## Risks / Open Questions

- A fix must preserve opaque non-UUID session IDs for backends such as Codex/Gemini where those IDs are legitimate; a blanket UUID-only parser would regress generic CLI backend support.
- The bundled Claude CLI transcript-missing guard should remain as defense in depth, but it is not a substitute for rejecting invalid parsed session IDs before persistence.
