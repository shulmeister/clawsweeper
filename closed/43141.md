## Summary

Keep #43141 open. Current main still does not make queued command-lane entries abort-aware: cron timeout signals are created and passed down to the embedded runner, but the queue API has no `signal` option and `runEmbeddedPiAgent` does not thread `params.abortSignal` into either session-lane or global-lane enqueue. The linked fix PR #43243 remains open/unmerged in the provided context, and #46174 was closed unmerged, so the bug is not shipped on main.

## Best Possible Solution

Keep #43141 open until main implements abort-aware queued-entry removal in `src/process/command-queue.ts`, extends the enqueue option type to accept a signal, threads `params.abortSignal` through both `runEmbeddedPiAgent` session/global lane enqueue calls, and adds focused regression coverage for abort-before-dequeue behavior.

## Evidence

- **Queue API has no abort option:** `enqueueCommandInLane` accepts only `warnAfterMs` and `onWait`, pushes a `QueueEntry` into `state.queue`, and only removes entries later via `shift()` in `drainLane`. There is no `AbortSignal` listener or queued-entry removal path.
