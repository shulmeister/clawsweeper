## Summary

Current main already implements the observable request: profile CLI invocations set profile-specific config/state before command routing, RPC callers derive the loopback target from the loaded profile config, and gateway service status/probing reads the profile-specific service metadata and configured port instead of defaulting to the main gateway. The issue comment’s suspected failure modes are covered by the current startup, connection, and service-status paths, even though named profiles do not set OPENCLAW_GATEWAY_PORT in applyCliProfileEnv itself.

## Best Possible Solution

Close this issue and keep the current profile-aware startup, RPC connection, and service-status implementation. Future work, if any, should be narrower regression coverage for an exact macOS multi-profile pairing/status scenario rather than a core routing redesign.
