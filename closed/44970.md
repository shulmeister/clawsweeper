## Summary

Keep open. Current main partially supersedes the PR's mechanism, but it does not fully implement the requested stale-config fallback behavior, and the PR head itself now needs rework after restoring the assistant-side fallbackConfigured gate to avoid single-candidate payload loss.

## Best Possible Solution

Keep the PR open for maintainer/user follow-up, but rework it around the current main architecture: pass authoritative outer fallback availability into the embedded failover policy or classify recoverable returned error payloads when the outer loop has remaining candidates, while preserving single-candidate payload delivery. Add focused regression coverage for prompt-side, auth-profile exhaustion, and assistant timeout/rate-limit behavior under a stale embedded config snapshot.
