## Summary

Keep open. Current `main` still has the original broad 80+ char base64 warning heuristic, so the linked false-positive report is not already fixed. This PR also is not close-ready: its proposed `[+/=]` requirement would stop warning on valid alphanumeric-only base64 payloads, and that warning path is the only signal emitted before sandbox and skill env injection.

## Evidence

- **Current main still matches long plain alphanumeric values:** `validateEnvVarValue()` on `main` still warns on any value matching `/^[A-Za-z0-9+/=]{80,}$/`, so long non-base64 alphanumeric payloads still trigger the warning described in the linked issue.

  - command: `node -e 'const v="a1B2c3D4e5F6g7H8i9J0k1L2m3N4o5P6q7R8s9T0u1V2w3X4y5Z6a7B8c9D0e1F2g3H4i5J6k7L8m9N0o1P2q3R4s5"; console.log(v.length); console.log(/^[A-Za-z0-9+/=]{80,}$/.test(v));'`

  - file: [src/agents/sandbox/sanitize-env-vars.test.ts:31](https://github.com/openclaw/openclaw/blob/33d5ebbff7bceb782b314de14ebd9a213ba31869/src/agents/sandbox/sanitize-env-vars.test.ts#L31)
