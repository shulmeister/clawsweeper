## Summary

Keep open. Current main still has a plausible macOS LaunchAgent false-positive path: the launchd extra-service scan treats any non-canonical plist whose contents contain `openclaw` or `clawdbot` as gateway-like, without verifying the gateway binary, configured port, or a service marker+kind contract. The prior ClawAid workaround comment is useful operationally, but it does not supersede an OpenClaw core fix for the diagnostic/cleanup warning path.

## Best Possible Solution

Keep the issue open and tighten `src/daemon/inspect.ts` for launchd: require the managed service marker+kind, a canonical/recognized OpenClaw gateway label with gateway command evidence, or a clearly legacy clawdbot gateway before reporting an extra service. Add Darwin-focused tests for unrelated `com.*` LaunchAgents that mention OpenClaw paths or user automation names, and avoid generic cleanup hints unless the detected service is a confirmed gateway conflict.
