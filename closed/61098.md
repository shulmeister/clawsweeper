## Summary

Current main already resolves `memory_search` from the active config on each request and keys cached memory managers by resolved memory settings, so changing `agents.defaults.memorySearch.provider` no longer reuses the stale embedding backend after restart.

## Evidence

- **Memory tool registration is request-scoped:** The memory plugin registers `memory_search` as a factory that receives `ctx.config` and `ctx.sessionKey` when tools are resolved, rather than constructing one long-lived tool instance at plugin startup.
  - file: [extensions/memory-core/index.ts:42](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/memory-core/index.ts#L42)
  - command: `nl -ba extensions/memory-core/index.ts | sed -n '24,58p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Gateway rebuilds tools from the current config:** Gateway tool resolution calls `createOpenClawTools` with the current `cfg`, and `createOpenClawTools` passes that config into plugin tool resolution each time the tool surface is built.
  - file: [src/gateway/tool-resolution.ts:83](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/gateway/tool-resolution.ts#L83)
  - command: `nl -ba src/gateway/tool-resolution.ts | sed -n '29,145p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Builtin memory manager cache key includes resolved settings:** `MemoryIndexManager.get` resolves memory settings from the provided config and uses `JSON.stringify(settings)` in the cache key, so changing `agents.defaults.memorySearch.provider` produces a different key and forces a fresh manager instead of reusing the old backend.
  - file: [extensions/memory-core/src/memory/manager.ts:161](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/extensions/memory-core/src/memory/manager.ts#L161)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '155,183p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Gateway restart is a fresh-process restart by default:** The restart path prefers a full process respawn with a fresh PID, and the gateway run loop exits after spawning/supervisor handoff. That clears any old in-memory tool or manager state across normal restarts.
  - file: [src/infra/process-respawn.ts:20](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/infra/process-respawn.ts#L20)
  - command: `nl -ba src/infra/process-respawn.ts | sed -n '20,64p'; nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '82,103p'`
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)

## Risks / Open Questions
