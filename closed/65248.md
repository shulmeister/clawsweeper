## Summary

Keep open. Current `main` still has the bug this PR fixes: ACP-bound slash commands like `/acp cancel`, `/acp close`, and `/unfocus` do not bypass ACP dispatch to local handlers, despite current docs and command tests showing those local handlers should own the behavior.

## Evidence

- **Current bypass logic still rejects slash command bypass:** `shouldBypassAcpDispatchForCommand()` only checks `maybeResolveTextAlias()` behind `!normalized.startsWith("/")`, then immediately returns `false` for any non-`!` slash command. On current `main`, that means recognized slash commands in an ACP-bound conversation do not escape to local handlers.
  - file: [src/auto-reply/reply/dispatch-acp-command-bypass.ts:44](https://github.com/openclaw/openclaw/blob/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19/src/auto-reply/reply/dispatch-acp-command-bypass.ts#L44)

  - sha: [1bdf5307d99d](https://github.com/openclaw/openclaw/commit/1bdf5307d99d9c1edcbe099d7c2fc3947a9b0a19)
- **Alias resolver requires a leading slash:** `maybeResolveTextAlias()` returns `null` unless the trimmed command starts with `/`, so the guard in `dispatch-acp-command-bypass.ts` is contradictory for slash commands and cannot preserve `/acp ...` or `/unfocus` on the local path.
