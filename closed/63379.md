## Summary

Keep this PR open. Current main still explicitly rejects trusted-proxy loopback sources, does not define the proposed `allowLoopback` / `loopbackUser` config, and docs describe the current fail-closed behavior. The PR is a concrete implementation candidate for a real gateway/auth gap, but its branch also has unresolved review concerns that need maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open as an implementation candidate for trusted-proxy internal self-connections. Maintainers should either review and land a tightened version with schema/docs/tests and preserved required-header hardening, or make an explicit product decision under #69066 to supersede it with first-class internal service identity.
