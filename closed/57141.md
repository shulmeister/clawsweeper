## Summary

Current main fixes the observable #57141 failure by retargeting normal reply dispatch through the active conversation binding before ACP initialization. A Telegram DM-topic follow-up can still carry a thread-scoped message context, but ACP dispatch now resolves the current conversation binding (`chat:topic:thread`) and uses the bound base ACP session key, so the missing-metadata `:thread:` row is no longer the ACP session that gets initialized.

## Best Possible Solution

Close #57141 as implemented on main. Keep the current dispatch-retarget behavior: for bound ACP conversations, route follow-up turns through the binding's base ACP session key rather than requiring ACP metadata on any channel-specific `:thread:` alias row. Any future residual issue should be filed narrowly with a fresh failing path if a channel bypasses the generic conversation-binding retarget.
