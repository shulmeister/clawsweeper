## Summary

Keep open. Current main still does not inject non-image WebChat attachments into agent context: the shipped Control UI only accepts image MIME types, the WebChat serializer labels accepted attachments as images, and the gateway parser still logs and drops sniffed non-image payloads. Related PR #66613 addressed silent-drop error behavior in a separate branch but explicitly did not add PDF/document support, and newer related issues/PRs show this remains active unresolved work.

## Best Possible Solution

Keep this open or explicitly consolidate it into the canonical WebChat document-upload tracker. The product fix should broaden Control UI attachment acceptance/preview, preserve filename and MIME metadata through `chat.send`, store supported non-image uploads in inbound media, inject an agent-visible `media://inbound/<id>` or bounded inline text reference, and add UI/gateway/parser tests covering PDF/text/document behavior and unsupported-file feedback.
