## Summary

Keep open. Current `main` still streams OpenResponses assistant deltas to HTTP SSE clients without checking `phase`, so the commentary-leak bug is not fixed on `main`. The PR discussion also documents unresolved regressions in the proposed patch, so there is not strong evidence to close this PR conservatively.

## Evidence

- **Current SSE path still emits every assistant delta immediately:** In `handleOpenResponsesHttpRequest`, the streaming handler processes `evt.stream === "assistant"`, appends the content to `accumulatedText`, and always writes a `response.output_text.delta` event. There is no `phase` gate or buffering before emission, which matches the reported leak on tool-call turns.
  - file: [src/gateway/openresponses-http.ts:931](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/gateway/openresponses-http.ts#L931)
  - command: `nl -ba src/gateway/openresponses-http.ts | sed -n '923,952p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Main producer for assistant deltas does not attach phase metadata:** `emitAcpAssistantDelta` emits only `{ text, delta }` on the `assistant` stream. That means the common OpenResponses streaming path still receives unphased assistant deltas on `main`, and any close-as-fixed decision would be unsupported.
