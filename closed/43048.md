## Summary

The PR's core premise is obsolete on current main. Session-targeted nested work is no longer funneled through one shared `nested` lane, so adding `agents.defaults.nestedMaxConcurrent` for `CommandLane.Nested` would not address the current code path.

## Evidence

- **Nested work is session-scoped on current main:** `resolveNestedAgentLaneForSession()` returns `nested:<sessionKey>` for non-empty session keys, with the unscoped `nested` lane used only as a fallback when there is no session key.
  - file: [src/agents/lanes.ts:18](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/lanes.ts#L18)

- **`sessions_send` uses the session-scoped nested lane:** The `sessions_send` tool dispatches nested work with `lane: resolveNestedAgentLaneForSession(resolvedKey)`, so inter-session sends are already sharded by target session instead of sharing one global lane.
  - file: [src/agents/tools/sessions-send-tool.ts:281](https://github.com/openclaw/openclaw/blob/41c5ffc5d5f4dc0f1df6f2c9a7efd8ce6a627f00/src/agents/tools/sessions-send-tool.ts#L281)
