## Summary

Keep open. Current main still has the behavior this PR is trying to change: manual iOS gateway TLS resolution forces TLS for non-loopback hosts even when the caller supplies `useTLS: false`, and the current regression test still asserts that behavior. The Share Extension `bootstrapToken: nil` build fix is already present on main, but the central manual/setup-code `tls=false` behavior is not. The unresolved bot review also identifies a real implementation concern in the PR as written: a safe fix should distinguish an explicit false TLS choice from an unset `UserDefaults.bool(forKey:)` value.

## Best Possible Solution

Keep the PR open, but require a revised iOS fix. The better direction is to preserve secure default TLS behavior for missing/legacy manual TLS preferences while honoring an explicit `tls=false` from setup code or manual UI, update `GatewayConnectionSecurityTests` to cover both explicit false and unset/default cases, and rebase away the already-shipped Share Extension `bootstrapToken: nil` change.
