## Summary

Current `main` and released tag `v2026.4.23` already route dispatcher-backed Slack media downloads through the runtime Undici fetch path instead of ambient `globalThis.fetch`, which covers the Node.js v25 failure described in #63393.

## Evidence

- **Slack media download path avoids ambient fetch when a dispatcher is present:** `createSlackMediaFetch()` checks for `dispatcher` in the guarded request init and switches to `fetchWithRuntimeDispatcher` instead of `globalThis.fetch`, then performs the Slack file request with `redirect: "manual"`.
  - file: [extensions/slack/src/monitor/media.ts:88](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/slack/src/monitor/media.ts#L88)
  - command: `nl -ba extensions/slack/src/monitor/media.ts | sed -n '88,100p'`
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Shared guarded fetch already falls back to runtime Undici for dispatcher-backed requests:** `fetchWithSsrFGuard()` computes `shouldUseRuntimeFetch` when a dispatcher is attached and the caller is not using an explicit mocked/stub fetch, then calls `fetchWithRuntimeDispatcher(...)` rather than the ambient global fetch.
  - file: [src/infra/net/fetch-guard.ts:393](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/src/infra/net/fetch-guard.ts#L393)

- **Regression test covers the exact Slack dispatcher scenario:** `media.test.ts` asserts that dispatcher-backed Slack media requests do not hit `globalThis.fetch`, do call `fetchWithRuntimeDispatcher`, and still carry both `redirect: "manual"` and the Slack `Authorization` header.
  - file: [extensions/slack/src/monitor/media.test.ts:601](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/slack/src/monitor/media.test.ts#L601)
