## Summary

Keep open. Current main partially mitigates the original disappearance symptom by rendering an optimistic user message and showing a visible queued-follow-up panel, and hard WebSocket closes do switch the Control UI into a disconnected state. The stale-open transport case described here is still not fully handled: the browser Gateway client has no tick watchdog or request timeout, so a `chat.send` request can remain pending while `WebSocket.readyState` is still OPEN, with no degraded/reconnecting UI state.

## Best Possible Solution

Keep the issue open and narrow it to Control UI stale-open transport handling: add a browser-side gateway liveness/request-timeout state using the existing tick policy, surface a reconnecting/degraded banner or composer state, keep optimistic/pending message affordances visible, and force reconnect or fail pending sends when ticks or `chat.send` acknowledgements exceed a bounded timeout. If backend stale SSH-channel telemetry can be surfaced to the WebSocket layer, closing the affected browser socket would make the existing reconnect UI path work.

## Evidence

There is no request timeout in this browser client path, so a stale tunnel that does not close the WebSocket can leave `chat.send` pending indefinitely.
