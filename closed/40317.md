## Summary

Keep #40317 open. Current main has strict write validation and last-known-good/doctor recovery paths, but it still does not implement a configurable strict-vs-tolerant validation mode, and normal read validation still rejects unknown keys rather than stripping them and continuing. The related tolerant-load PR #46213 is closed and unmerged, while #40268 remains the open narrow bug report for tolerant read/startup behavior.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A complete solution would define an explicit config validation policy, wire strict defaults through write/control-plane paths, add a tolerant read/startup path that safely strips or reports unknown root and nested keys, document the policy, and add regression coverage for `loadConfig`, startup recovery, `config.patch`, and `config.apply`. If maintainers decide the remaining ask should be narrowed, #40268 is the best canonical follow-up for the read/startup tolerant behavior.
