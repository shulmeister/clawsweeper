## Summary

Keep #42680 open. Current main still starts the cron timeout before `executeJobCore` runs, so the isolated agent path can still spend timeout budget before the agent runner actually gets a lane. The PR is a focused core cron fix, but it needs rebasing/adaptation to the current `run-executor` split and must address the unresolved review concern that the CLI-provider path would skip the new outer cron timeout arming.

## Best Possible Solution

Keep the PR open, but require an updated fix against current main. The best path is to start the cron timeout at a shared, well-defined execution-start boundary after cron/agent lane acquisition for embedded runs, preserve equivalent timeout enforcement for CLI-provider runs, and add regression coverage for embedded queue wait plus CLI, manual `cron.run`, and startup catch-up behavior.
