## Summary

Current `main` already supports the requested depth-1 orchestrator pattern via configurable `agents.defaults.subagents.maxSpawnDepth`; depth-1 subagents can receive `sessions_spawn` when `maxSpawnDepth >= 2`, while depth-2 children remain terminal.

## Evidence

- **User-facing docs describe the exact orchestrator pattern:** The sub-agent docs explicitly say nested sub-agents are enabled with `maxSpawnDepth: 2`, describe the `main -> orchestrator -> worker` flow, and state that depth-1 orchestrators get `sessions_spawn`, `subagents`, `sessions_list`, and `sessions_history` while depth-2 workers cannot spawn further.
  - file: [docs/tools/subagents.md:173](https://github.com/openclaw/openclaw/blob/14934f0b7c874e6af4ed93a1c307a17d222ef739/docs/tools/subagents.md#L173)
  - command: `nl -ba docs/tools/subagents.md | sed -n '170,240p'`
  - sha: [14934f0b7c87](https://github.com/openclaw/openclaw/commit/14934f0b7c874e6af4ed93a1c307a17d222ef739)
- **Runtime gate now keys off configurable max spawn depth:** `spawnSubagent` computes `callerDepth`, reads `cfg.agents?.defaults?.subagents?.maxSpawnDepth`, rejects only when `callerDepth >= maxSpawnDepth`, then assigns child capabilities from the incremented depth. With `maxSpawnDepth: 2`, a depth-1 caller is allowed to spawn a depth-2 child.
  - file: [src/agents/subagent-spawn.ts:619](https://github.com/openclaw/openclaw/blob/14934f0b7c874e6af4ed93a1c307a17d222ef739/src/agents/subagent-spawn.ts#L619)

- **Tool policy grants sessions tools to orchestrator subagents:** The subagent tool policy distinguishes leaf vs orchestrator roles. For depth 1 when `maxSpawnDepth >= 2`, `sessions_spawn`, `subagents`, `sessions_list`, and `sessions_history` are intentionally allowed; only leaf sessions deny them.
  - file: [src/agents/pi-tools.policy.ts:45](https://github.com/openclaw/openclaw/blob/14934f0b7c874e6af4ed93a1c307a17d222ef739/src/agents/pi-tools.policy.ts#L45)

- **Regression tests cover the requested case directly:** The depth-limit tests assert the old default restriction at depth 1 when `maxSpawnDepth` is `1`, then explicitly verify that a depth-1 caller is accepted when `maxSpawnDepth` is `2`, and that depth-2 callers are still rejected.
