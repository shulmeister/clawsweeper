## Summary

Keep open. Current `main` does not provide strong evidence for a conservative close: Active Memory still documents `timeoutMs` as a hard timeout, but the embedded runner intentionally extends timeouts during compaction and budgets extra grace time. I did not run a live repro in this read-only review, but the code and tests point toward the report remaining plausible rather than already fixed.

## Evidence

- **Docs still describe a hard deadline:** The Active Memory docs on current `main` still describe `plugins.entries.active-memory.config.timeoutMs` as a "Hard timeout for the blocking memory sub-agent," which matches the reporter's expectation.
  - file: [docs/concepts/active-memory.md:542](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/concepts/active-memory.md#L542)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Active Memory forwards the configured timeout directly to the embedded run:** The plugin passes `timeoutMs: params.config.timeoutMs` into `runEmbeddedPiAgent(...)` and also arms its own abort controller at exactly that timeout, so the plugin itself expects the embedded run to honor that deadline.
  - file: [extensions/active-memory/index.ts:1678](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/active-memory/index.ts#L1678)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Runner timeout logic explicitly allows extension during compaction:** `resolveRunTimeoutDuringCompaction()` returns `"extend"` on the first timeout reached while compaction is pending/in flight, and `resolveRunTimeoutWithCompactionGraceMs()` adds one full compaction grace window to the run timeout budget.
  - file: [src/agents/pi-embedded-runner/run/compaction-timeout.ts:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/compaction-timeout.ts#L16)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **The main attempt loop actually reschedules the deadline:** When the abort timer fires during compaction, the runner logs that it is extending the deadline and reschedules another timer for `compactionTimeoutMs` instead of aborting immediately.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2087](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/attempt.ts#L2087)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Tests codify the non-hard-timeout behavior:** Current tests explicitly assert both that the first timeout reached during compaction is extended and that one compaction grace window is added to the run timeout budget, which is inconsistent with closing this as already fixed or unreproducible.
  - file: [src/agents/pi-embedded-runner/run/compaction-timeout.test.ts:60](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/compaction-timeout.test.ts#L60)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **No relevant fix is visible between the latest release and current main:** `git diff --stat a9797214338ba31c52c796adbb75afb16e0684a9..95b7a85f0607784d8e33b4ce7e42617174196490 -- extensions/active-memory/index.ts extensions/active-memory/index.test.ts docs/concepts/active-memory.md src/agents/pi-embedded-runner.ts src/agents/pi-embedded-runner/run.ts src/agents/embedded-runner.ts` shows only Active Memory auth/profile prompt tweaks and broader runner changes, not a hard-deadline fix for this timeout contract.
