## Summary

Current main already keeps the normal successful announce-delivery path consistent: successful isolated cron delivery resolves to `lastRunStatus: "ok"`, `lastDeliveryStatus: "delivered"`, and resets `consecutiveErrors`. Delivery outcome is intentionally tracked separately from execution outcome, so a delivered error payload can still remain an execution error. The Feishu sender also uses Feishu-specific error prefixes; the hardcoded Telegram message is confined to the Telegram plugin path.

## Best Possible Solution

Close this issue and keep the current split between execution status and delivery status. If users still see false `consecutiveErrors` from host sleep or connectivity failures, track that through the narrower remaining cron false-failure work such as #49124, or reopen with run history showing a normal non-error payload being classified as `error`.
