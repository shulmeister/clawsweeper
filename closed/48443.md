## Summary

Current `main` already formats web UI model labels from provider-qualified model refs and catalog provider keys, with regression coverage for duplicate-provider cases. The implementation is also present in release `v2026.4.22`.

## Evidence

- **Provider-qualified model display logic:** `formatCatalogChatModelDisplayFromLookup` falls back to `formatChatModelDisplay`, and the catalog lookup is keyed by `entry.provider`, so labels are derived from the actual provider key rather than a transport/API type.
  - file: [ui/src/ui/chat-model-ref.ts:186](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/ui/src/ui/chat-model-ref.ts#L186)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Web UI model selector uses provider-aware display state:** The selector state resolves `activeRow.model` plus `activeRow.modelProvider`, then builds `defaultDisplay` and option labels through the provider-aware catalog display lookup used by the chat header controls.
  - file: [ui/src/ui/chat-model-select-state.ts:47](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/ui/src/ui/chat-model-select-state.ts#L47)
