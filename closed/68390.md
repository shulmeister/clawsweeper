## Summary

Current `main` resolves `openclaw models set <alias>` through the configured alias index before any default-provider fallback, and the same alias-first logic is present in shipped release `v2026.4.23`. Combined with the issue discussion reporting the exact repro passes on both `v2026.4.15` and `main`, there is not enough evidence of a current reproducible core bug.

## Evidence

- **Current `models set` path uses alias-aware resolver:** `modelsSetCommand()` calls `applyDefaultModelPrimaryUpdate()`, which calls `resolveModelTarget()` before writing `agents.defaults.model.primary`. `resolveModelTarget()` builds the alias index and calls `resolveModelRefFromString()` with it.
  - file: [src/commands/models/set.ts:6](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/commands/models/set.ts#L6)
  - command: `nl -ba src/commands/models/set.ts | sed -n '1,40p'; nl -ba src/commands/models/shared.ts | sed -n '70,215p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Alias index reads the exact config shape from the report:** `buildModelAliasIndex()` iterates `cfg.agents?.defaults?.models` and reads each entry's top-level `alias` field (`{ "anthropic/claude-sonnet-4-6": { alias: "sonnet" } }`), adding it to `byAlias`. The config schema also defines `models.*.alias` as an optional string.
  - file: [src/agents/model-selection-shared.ts:273](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/model-selection-shared.ts#L273)
  - command: `nl -ba src/agents/model-selection-shared.ts | sed -n '270,305p'; nl -ba src/config/zod-schema.agent-defaults.ts | sed -n '45,75p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Alias lookup happens before default-provider parsing:** `resolveModelRefFromString()` first checks `aliasIndex.byAlias` for bare values without `/`; only if that misses does it fall through to `parseModelRefWithCompatAlias()`, which is the path that would infer a default provider like `openai/sonnet`. With the reported config shape, `sonnet` should resolve to the configured fully qualified model id instead of creating a bogus new one.
  - file: [src/agents/model-selection-shared.ts:390](https://github.com/openclaw/openclaw/blob/b84e57fca38deacfa9834a82f7e270b32562ae1b/src/agents/model-selection-shared.ts#L390)
  - command: `nl -ba src/agents/model-selection-shared.ts | sed -n '390,460p'`
  - sha: [b84e57fca38d](https://github.com/openclaw/openclaw/commit/b84e57fca38deacfa9834a82f7e270b32562ae1b)
- **Same alias-first behavior is already in shipped release `v2026.4.23`:** The release tag `v2026.4.23` (`899e927dab9ef2d6164004d34a30410b3506b6f5`) contains the same `buildModelAliasIndex()` and `resolveModelRefFromString()` logic for alias resolution before fallback parsing, so this is not a recent post-release change on `main`.
  - file: [src/agents/model-selection-shared.ts:231](https://github.com/openclaw/openclaw/blob/899e927dab9ef2d6164004d34a30410b3506b6f5/src/agents/model-selection-shared.ts#L231)
