## Summary

Current `main` already fixes this class of bug: bundled channels configured under `channels.<id>` are treated as explicit activation, can bypass restrictive `plugins.allow`, and configured bundled channels are auto-materialized into the allowlist before plugin loading. Telegram-specific regression coverage exists.

## Evidence

- **Bundled channel config is treated as an explicit activation cause:** `resolveExplicitPluginSelectionShared` marks bundled channels enabled via `channels.<id>.enabled` as `bundled-channel-enabled-in-config`, instead of treating them like an ordinary plugin that must already be in `plugins.allow`.
  - file: [src/plugins/config-activation-shared.ts:116](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/config-activation-shared.ts#L116)

  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **That explicit bundled-channel cause bypasses the generic allowlist rejection:** `resolvePluginActivationDecisionShared` returns enabled/activated for `bundled-channel-enabled-in-config` before the later `not-in-allowlist` branch runs.
  - file: [src/plugins/config-activation-shared.ts:221](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/config-activation-shared.ts#L221)
