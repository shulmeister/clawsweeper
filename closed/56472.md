## Summary

Current `main` already contains Discord gateway lifecycle changes that prevent the reported late `Max reconnect attempts (0)` path from crashing OpenClaw, while still configuring Discord gateway reconnects with `maxAttempts: 50`. The fix appears to be present on `main` but unreleased.

## Evidence

- **Discord gateway still configures 50 reconnect attempts:** `createDiscordGatewayPlugin()` constructs the Carbon gateway with `reconnect: { maxAttempts: 50 }`.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:377](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-plugin.ts#L377)
  - command: `nl -ba extensions/discord/src/monitor/gateway-plugin.ts | sed -n '360,381p'`
  - sha: [7d1891e6e63d](https://github.com/openclaw/openclaw/commit/7d1891e6e63dc35dd9dcb688ba308049e68fe695)
- **Late reconnect-exhausted errors are suppressed after dispose/teardown:** The gateway supervisor classifies `Max reconnect attempts ...` as `reconnect-exhausted` and logs/suppresses those events in `disposed` and `teardown` phases instead of letting them surface as uncaught process crashes.
  - file: [extensions/discord/src/monitor/gateway-supervisor.ts:89](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/gateway-supervisor.ts#L89)

- **Startup cleanup now disconnects before dispose and then disposes the supervisor:** Provider startup cleanup disconnects the lifecycle gateway first and only then disposes the supervisor, matching the new late-error suppression path.
  - file: [extensions/discord/src/monitor/provider.ts:1143](https://github.com/openclaw/openclaw/blob/7d1891e6e63dc35dd9dcb688ba308049e68fe695/extensions/discord/src/monitor/provider.ts#L1143)
