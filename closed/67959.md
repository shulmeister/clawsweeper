## Summary

Keep open. Current main partially supersedes the PR by adding bounded concurrent channel/account status work, but it does not fully implement the PR’s user-visible fix: the CLI still injects a 10000ms timeout for omitted `channels status --probe`, so the command-layer 30000ms probe fallback is not reached through the normal CLI path. The PR discussion is useful: its body identifies the same root cause and Greptile found the scoped PR safe, but the work should be rebased/adapted rather than closed.

## Best Possible Solution

Keep the PR open and adapt it to current main: preserve or intentionally revise the existing bounded `runTasksWithConcurrency` implementation, remove the Commander-level default for `channels status --timeout`, add/restore CLI and command-flow tests proving omitted probe timeout becomes 30000ms while explicit `--timeout` still wins, and add a focused gateway test for probe concurrency plus stable account ordering under the current concurrency policy.
