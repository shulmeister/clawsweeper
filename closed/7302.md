## Summary

Current main already solves the observable WhatsApp voice-note metadata problem through the pinned Baileys send path. OpenClaw sends WhatsApp audio as a PTT Baileys audio payload, current main normalizes outbound audio to Ogg/Opus for voice-note compatibility, and Baileys 7.0.0-rc.9 computes missing audio `seconds` and `waveform` metadata internally before sending.

## Best Possible Solution

Close this issue as implemented. Keep the current lean OpenClaw PTT payload and Baileys-managed metadata generation rather than copying Discord-specific helpers into WhatsApp; add a narrower regression issue only if a live WhatsApp/Baileys send on current versions still produces `0:00` or lacks transcription.
