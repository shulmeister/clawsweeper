## Summary

Keep open. Current main has substantial macOS LaunchAgent hardening for narrower restart/bootstrap races, but #67335’s broader persistence and split-brain report is not fully covered: current restart behavior can still prefer an unmanaged live gateway listener over re-bootstrapping the LaunchAgent, and status diagnostics only run the stale-listener reconciliation path when the LaunchAgent is loaded.

## Best Possible Solution

Keep #67335 open. The best product/code path is to add a macOS reconciliation path for `LaunchAgent plist exists + launchd job not loaded + gateway listener/RPC still alive`: classify supervised vs orphaned runtime, repair/bootstrap the LaunchAgent when safe, terminate or hand off stale listeners when needed to avoid port conflicts, make `gateway status --deep` report one coherent recovery action for that split-brain state, and add targeted tests for not-loaded/RPC-ok plus degraded `launchctl stop`/bootout recovery.

## Evidence

- **Issue field reports describe a live current failure mode:** The issue body reports OpenClaw 2026.4.14/macOS 26.4.1 cases where `ai.openclaw.gateway` is removed from the GUI launchd domain and later needs doctor/re-bootstrap; the 2026-04-20 follow-up adds a split-brain state where `status --deep` says LaunchAgent not loaded while RPC and `127.0.0.1:18789` are still OK.
- **Current stop path can intentionally leave LaunchAgent unloaded on degraded launchctl failures:** `stopLaunchAgent` disables the service before stop, but if `disable`, `stop`, or stop-state confirmation fails it calls a bootout fallback with warnings that explicitly say it left the service unloaded. That is a legitimate stop fallback, but it is still the kind of unloaded LaunchAgent state #67335 asks to make safer/idempotent across lifecycle flows.
