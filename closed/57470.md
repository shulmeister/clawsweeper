## Summary

Keep open. Current main still implements, tests, documents, and changelogs the opposite behavior: a cron `payload.model` that is not allowlisted falls back to the job's agent/default model with a warning. The PR addresses a concrete linked bug report (#57367), and the provided review discussion only flags minor redundancy, not a reason to auto-close. A later similar PR (#65179) exists, but it is also open and not clearly a better canonical replacement.

## Best Possible Solution

Keep this PR open for maintainer decision on cron allowlist semantics. If explicit cron `payload.model` should bypass `agents.defaults.models`, implement that deliberately in the cron model-selection path, add focused regression tests for configured-provider-but-not-allowlisted models, update cron/model docs, and then reconcile #57367/#65179. If the current allowlist contract is intended, close manually with that rationale and direct users to add the cron model to `agents.defaults.models` or clear the allowlist.
