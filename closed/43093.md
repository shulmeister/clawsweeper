## Summary

Keep open. Current main has a partial fallback for `logs.tail`, but it does not implement this PR's active-logger-file fix. `readConfiguredLogTail()` still starts from `getResolvedLoggerSettings().file`, which recomputes the current day's rolling path, and `resolveLogFile()` only falls back to another rolling file when that recomputed file is missing. That leaves the linked #42875 scenario unresolved when today's file exists but the long-running gateway is still writing to yesterday's file.

## Best Possible Solution

Keep the PR open or replace it with an updated patch on current main: expose an active logger file path from `src/logging/logger.ts` or otherwise make `readConfiguredLogTail()` prefer the cached active file used by the running gateway, then add a regression test where the active cached file is the previous-day rolling file and the recomputed current-day file also exists.
