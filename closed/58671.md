## Summary

Current `main` already handles transient model-provider failures with bounded retry/failover behavior: short `429`/`5xx` responses stay on SDK retry paths, long `Retry-After` windows are surfaced so OpenClaw can rotate auth profiles or fallback models, and overloaded/rate-limit errors are classified and kept in the fallback loop instead of immediately failing the session.
