## Summary

Keep this issue open. Current main delays `gateway:startup` until after `startChannels()` and the Telegram send path now validates a real Bot API `message_id`, but the channel startup lifecycle still hands off Telegram startup before the adapter is necessarily connected/ready. The April 24 reproduction on 2026.4.21 is recent, concrete, and not superseded by the closed restart-notification feature request.

## Best Possible Solution

Keep the issue open and fix it in the gateway/channel lifecycle. The strongest path is to distinguish channel task handoff from adapter readiness, expose a per-channel ready/degraded signal for startup, and delay `gateway:startup` or make startup outbound sends fail/retry until the selected channel account is actually ready. Add a regression test with a mocked channel whose `startAccount` readiness resolves after `startChannels()` to prove startup-hook message sends are not silently accepted before delivery is possible.

## Evidence

- **Recent reproduction:** The issue body reports silent Telegram delivery loss from a `gateway:startup` hook on 2026.3.11, and the April 24 comment independently reproduces the same symptom on 2026.4.21 with Telegram polling and `boot-md`. That makes this a current behavior bug report, not stale or insufficient.
- **Startup hook ordering:** `startGatewaySidecars()` awaits `params.startChannels()` before scheduling the `gateway:startup` internal hook, so main has an ordering barrier around channel start handoff.
