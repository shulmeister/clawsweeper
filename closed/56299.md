## Summary

Keep open. Current main already has browser WebSocket retry/backoff and restart-status handling, but the Control UI still switches the whole app to the login gate whenever `connected` is false, so the disruptive bounce away from the active chat/dashboard is not fixed well enough to close #56299.

## Best Possible Solution

Keep the issue open and implement a dedicated recoverable-disconnect UI state for the browser Control UI: when the socket drops due to shutdown/1012/1006 or other recoverable transport failures, keep the current tab/chat/session visible with a reconnecting status and let `GatewayBrowserClient` retry in the background. Only render the login gate for initial connection, explicit user disconnect, or non-recoverable auth/session errors. Add focused tests that simulate a restart close and assert the active chat/dashboard remains rendered until `hello` restores live data.
