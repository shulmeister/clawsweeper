## Summary

Current main implements the reported startup fix by moving the relevant cold startup/status paths away from broad process-local plugin discovery and onto a persisted plugin registry/index. `openclaw status` and `openclaw gateway status` are route-first with plugin loading disabled, while plugin/provider/channel planning reads the persisted registry before deriving from discovery. The #69925 Windows native-jiti exclusion remains true, but it is no longer the controlling path for the reported `status`/startup discovery bottleneck.

## Best Possible Solution

Keep the shipped direction: route status/gateway status before plugin registration, keep plugin/provider/channel planning on the persisted cold registry, and use `openclaw plugins registry --refresh` or `openclaw doctor --fix` as repair paths. Any remaining Windows slowdown should be tracked as a new current-version issue with timing logs for the specific command and registry state, not as the old process-local discovery-cache startup blocker.
