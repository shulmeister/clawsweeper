## Summary

Keep #34252 open. Current main still has only partial custom-Ollama plumbing: the Ollama stream function can use a custom provider id when called directly, but normal provider ownership, runtime hook activation, native `api: "ollama"` stream registration, synthetic auth, docs, and memory embedding configuration are still centered on the canonical `ollama` provider.

## Best Possible Solution

Keep #34252 open as the canonical tracker. The best fix is a provider-family or activation contract for configured providers whose native API is Ollama, so custom ids such as `ollama2` or `ollama-jetson` can be owned by the Ollama plugin without hard-coding every id. That should cover runtime stream hooks, synthetic/local auth and auth-profile persistence, provider-qualified model selection for heartbeat/defaults, memory-search embedding routing, tests for custom ids, and docs for the supported multi-host config shape.
