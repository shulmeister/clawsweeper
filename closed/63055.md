## Summary

Current main already implements the QMD startup fixes for this report: explicit `memory.qmd.command` paths are preserved, the agent workspace is created before the availability probe so spawn does not fail with misleading ENOENT from a missing cwd, and QMD subprocesses receive a PATH that includes the current Node binary directory. The changelog records this under 2026.4.12.

## Best Possible Solution

Keep the shipped QMD subprocess startup behavior. If users still hit this on a current release, collect a fresh report with current OpenClaw/QMD versions, service environment PATH, configured `memory.qmd.command`, cwd/workspace existence, and `openclaw memory status --deep` logs, preferably on the broader related issue #64001.
