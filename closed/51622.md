## Summary

Current `main` no longer matches the PR’s premise that nested `sessions_send` calls all queue behind one shared `CommandLane.Nested`. Nested session work is now lane-scoped per target session key, so different target sessions can run concurrently without adding `agents.defaults.nested.maxConcurrent`.

## Close Comment

Closing from Codex review because the serialized nested-queue bottleneck described here no longer matches current `main`. Nested session work is now routed onto per-target lanes in `src/agents/lanes.ts:18`, and both the main `sessions_send` path and the A2A follow-up path use those scoped lanes in `src/agents/tools/sessions-send-tool.ts:281` and `src/agents/tools/sessions-send-tool.a2a.ts:95`. The new tests in `src/agents/lanes.test.ts:25` also assert distinct lanes for distinct target sessions. That means different `sessions_send` targets no longer queue behind one shared `CommandLane.Nested`, so the timeout-through-queue failure mode this PR was addressing is obsolete on current `main` (behavior present in `c1b9aa7d9630efe79f366b98564854aa935d00f4`). If maintainers still want a user-visible nested fan-out tuning knob, that should come back as a fresh proposal against the new per-session lane model.
