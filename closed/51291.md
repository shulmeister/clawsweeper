## Summary

Current main does not route poll-bearing `send` requests through Telegram `sendMessage` anymore on the supported path. Instead, core rejects poll fields on `send` and requires the separate `poll` action, which Telegram already implements and tests.

## Evidence

- **Core send path rejects poll params:** `runMessageAction` now throws `Poll fields require action "poll"; use action "poll" instead of "send".` before dispatching any send action, so the reported silent `sendMessage` fallback is not reproducible through the supported outbound path.

- **Regression test covers rejected send-with-poll usage:** Current tests explicitly assert that `send` requests containing `pollQuestion`/`pollOption` or other poll-only fields are rejected with `use action "poll" instead of "send"`.
  - file: [src/infra/outbound/message-action-runner.send-validation.test.ts:87](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/infra/outbound/message-action-runner.send-validation.test.ts#L87)
