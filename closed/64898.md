## Summary

Current main already implements Discord lifecycle status reactions plus preview-based tool progress, and the same behavior is present in the shipped `v2026.4.23` release. That covers the issue’s requested distinction between in-progress vs. failure states and avoids leaking raw command/system progress into separate chat messages when preview tool progress is enabled.

## Evidence

- **Lifecycle status reactions exist in core:** The shared status reaction controller defines separate queued, thinking, tool, done, error, and stall states, with debounced intermediate transitions and explicit stall warnings rather than treating all long work as errors.
  - file: [src/channels/status-reactions.ts:19](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/channels/status-reactions.ts#L19)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Discord wires in-progress, tool, compaction, success, and error states:** Discord inbound processing creates a status reaction controller, switches to thinking/tool/compacting during the run, and only marks error on real failure; otherwise it marks done and then clears/restores per config.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:229](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/discord/src/monitor/message-handler.process.ts#L229)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Discord translates background progress into a single preview message:** When draft preview streaming is active, Discord builds a `Working…` preview with summarized progress lines and sets `suppressDefaultToolProgressMessages`, so tool/command/approval progress is reused in the preview instead of surfacing as raw separate system chatter.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:627](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/discord/src/monitor/message-handler.process.ts#L627)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Automated tests cover the exact UX states:** Tests verify short runs jump from ack to done without flashing intermediate states, long no-progress runs show stall reactions, config overrides apply, and compacting resumes thinking instead of ending in an error state.
  - file: [extensions/discord/src/monitor/message-handler.process.test.ts:466](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/discord/src/monitor/message-handler.process.test.ts#L466)
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Docs describe the shipped Discord behavior:** The Discord docs say preview streaming can reuse the preview message for tool/progress updates, and the gateway config docs say lifecycle status reactions are enabled on Slack and Discord when ack reactions are active.
  - file: [docs/channels/discord.md:537](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/docs/channels/discord.md#L537)
