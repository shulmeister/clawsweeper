## Summary

Current `main` already normalizes Ollama `baseUrl` values back to the native API root before discovery/models.json generation, the exact remote-host `/v1` case is covered by tests, and the fix was already shipped in `v2026.4.23`.

## Evidence

- **Native Ollama base URL normalization:** `resolveOllamaApiBase()` trims trailing slashes and removes a trailing `/v1`, so plugin code uses the native Ollama API root instead of persisting the OpenAI-compatible path.
  - file: [extensions/ollama/src/provider-models.ts:60](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/extensions/ollama/src/provider-models.ts#L60)

- **Discovery returns normalized provider config:** When explicit Ollama config is present, discovery rewrites `explicit.baseUrl` through `resolveOllamaApiBase(...)` before returning the provider config that feeds `models.json` generation.
  - file: [extensions/ollama/src/discovery-shared.ts:116](https://github.com/openclaw/openclaw/blob/734748d4f4fef7436def57f7a7ae3350bba9a091/extensions/ollama/src/discovery-shared.ts#L116)
