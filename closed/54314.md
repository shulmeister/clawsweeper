## Summary

Keep PR #54314 open. Current main still lacks the PR's dedicated `cron-nested` lane and still routes cron-owned embedded agent work through the shared `nested` lane, while gateway startup and hot reload only apply `cron.maxConcurrentRuns` to the outer `cron` lane. The PR discussion shows the contributor already addressed the main review concern by moving away from widening shared `nested` behavior, so this remains useful unresolved work.

## Best Possible Solution

Keep #54314 open as the strongest active vehicle for this bug. The best path is to rebase or rework it onto current main while preserving the dedicated `CommandLane.CronNested` design: add the lane, route cron-owned embedded work through it, apply `cron.maxConcurrentRuns` to both `Cron` and `CronNested` during startup and hot reload, and keep regression coverage for lane resolution plus realistic reload behavior. Older shared-`Nested` fixes should defer to this shape.
