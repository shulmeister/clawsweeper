## Summary

Current `main` already fixes #70352: stale-socket recovery now keys off provider transport activity instead of inbound Discord app events, with regression tests for quiet-channel behavior and a shipped changelog entry under `v2026.4.22` (`src/gateway/channel-health-policy.ts:115`, `extensions/discord/src/monitor/provider.ts:1073`, `CHANGELOG.md:403`, `CHANGELOG.md:567`).

## Evidence

- **Health policy no longer uses app-event freshness as socket liveness:** `evaluateChannelHealth` only applies `stale-socket` when `lastTransportActivityAt` exists on a connected channel, and the inline comment explicitly says app-level events are not socket liveness for quiet Slack/Discord workspaces.
  - file: [src/gateway/channel-health-policy.ts:115](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/gateway/channel-health-policy.ts#L115)

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Discord still tracks inbound events separately from transport liveness:** Discord provider status updates only set `lastEventAt`/`lastInboundAt`, with an inline note that Carbon does not expose stable heartbeat ACK events. Combined with the new health policy, quiet guilds no longer get restarted just because no app events arrived.
