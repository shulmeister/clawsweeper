## Summary

Current main already carries built-in/core tools through `before_tool_call` with `agentId` context before `splitSdkTools()` runs, so the PR’s stated gap is already covered by the existing wrapper-based path rather than by threading context through `toToolDefinitions()`.

## Evidence

- **Embedded attempt builds tools with agent/session/run context before split:** `runEmbeddedAttempt()` calls `createOpenClawCodingTools()` with `agentId`, `sessionKey`, `sessionId`, and `runId` before those tools are later passed into `splitSdkTools()`. That means core tools already receive hook context upstream of the adapter path this PR edits.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:674](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/attempt.ts#L674)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Core tool factory wraps every normalized tool with before-tool hook context:** `createOpenClawCodingTools()` maps all normalized tools through `wrapToolWithBeforeToolCallHook(...)`, passing `agentId`, `sessionKey`, `sessionId`, `runId`, and loop-detection config. This is the existing mechanism that injects hook context for built-in/core tools.
  - file: [src/agents/pi-tools.ts:707](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-tools.ts#L707)

For the built-in/core tools produced by `createOpenClawCodingTools()`, the hook has already run with context and is not lost at `splitSdkTools()`.
