## Summary

## Close Comment

Closing this as implemented after Codex review.

Current `main` already resolves the default cron store path lazily at call time, so the reported import-time `OPENCLAW_STATE_DIR` bug is no longer present. The same implementation is also present in release `v2026.4.22`, which makes this PR redundant on top of current `main`.

What I checked:

- **Cron store path is resolved lazily at runtime:** `resolveCronStorePath()` falls back to `resolveDefaultCronStorePath()` only when called, and `resolveDefaultCronStorePath()` computes the path through `resolveDefaultCronDir()` instead of reading a module-level constant. ([`src/cron/store.ts:26`](https://github.com/openclaw/openclaw/blob/564f820efa2b/src/cron/store.ts#L26), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **State-dir override is read dynamically:** `resolveConfigDir()` reads `env.OPENCLAW_STATE_DIR` inside the function body, so a later `loadDotEnv()` update to `process.env` is honored when the cron path is resolved. ([`src/utils.ts:136`](https://github.com/openclaw/openclaw/blob/564f820efa2b/src/utils.ts#L136), [564f820efa2b](https://github.com/openclaw/openclaw/commit/564f820efa2b))
- **Latest release already contains the lazy resolution:** The `v2026.4.22` tag shows the same `resolveDefaultCronDir()` / `resolveDefaultCronStorePath()` runtime path resolution in [`src/cron/store.ts`](https://github.com/openclaw/openclaw/blob/a94763ee224e/src/cron/store.ts), so this behavior has already shipped. ([`src/cron/store.ts:26`](https://github.com/openclaw/openclaw/blob/a94763ee224e/src/cron/store.ts#L26), [a94763ee224e](https://github.com/openclaw/openclaw/commit/a94763ee224e))

So I’m closing this as already implemented rather than keeping a duplicate issue open.
