## Summary

Keep open. Current main still returns plain Feishu Content-Disposition filename values unchanged and prefers that header-derived filename over the Feishu payload filename, so the linked Chinese filename mojibake bug is not implemented on main. The PR is a plausible focused fix with tests, but its current heuristic has unresolved review risk because it can silently rewrite legitimate Latin-1 filenames.

## Best Possible Solution

Keep the PR open and have the Feishu maintainer or author narrow the fix: recover only when there is a strong mojibake/CJK signal or prefer a trusted Feishu payload filename over a malformed header where available, then add regression coverage for both Chinese mojibake recovery and valid Latin-1 no-rewrite cases.
