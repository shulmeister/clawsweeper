## Summary

Current main already changes cron-triggered embedded runs so they do not inherit the default LLM idle watchdog when no timeout is configured, and that behavior is regression-tested. The changelog records this timeout fix in v2026.4.11, which postdates the reported failures on 2026.4.2/2026.4.5.

## Evidence

- **Cron-specific idle-timeout fix on main:** `resolveLlmIdleTimeoutMs` now returns `0` for `trigger === "cron"` when no explicit timeout is configured, preventing the default cron-only LLM idle timeout path that matches the reported `LLM request timed out` / `FallbackSummaryError` failure.

## Risks / Open Questions

- An explicitly configured `agents.defaults.llm.idleTimeoutSeconds`, `agents.defaults.timeoutSeconds`, or per-job `timeoutSeconds` that is too low can still produce real timeout failures on slow providers.
- `sessionTarget: "current"` binds the job to a persistent session; very large carried-over history can still cause provider-side slowness even though the cron default idle-timeout bug is fixed.
