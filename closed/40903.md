## Summary

Keep #40903 open. Current main still does not implement deferred Telegram media-group offset completion: offsets are persisted when an update is accepted, while media groups are buffered and flushed later. The PR is not merge-ready as-is, but it is a focused behavior change that needs maintainer judgment rather than conservative closure.

## Best Possible Solution

Keep the PR open for maintainer product judgment. If OpenClaw wants deferred media-group completion, port the approach to `extensions/telegram`, carry the original `update_id` into `MediaGroupEntry` before synthetic context creation, add a small hold/release tracker or equivalent, and cover both restart-skip prevention and fatal media-group no-loop behavior in tests. If maintainers prefer the current at-most-once accepted-offset contract, close explicitly as superseded/rejected after documenting that tradeoff.

## Risks / Open Questions

- A naive deferred-offset implementation could reintroduce Telegram polling replay or infinite-loop behavior for fatal media-group errors, the same area addressed by current accepted-offset persistence.
- The PR was authored against the older `src/telegram` layout, while current main uses `extensions/telegram`; any useful work needs a rebase and reconciliation with the current offset contract.
