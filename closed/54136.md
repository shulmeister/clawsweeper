## Summary

The reported Fake-IP failure model does not match current main. Link understanding extracts literal URL text from the inbound message, parses it with the platform URL parser, and passes only `parsed.hostname` to `isBlockedHostnameOrIp`; there is no DNS lookup in the detector or runner before URLs are returned. The shared SSRF helper also only classifies the supplied hostname/IP string at this call site; DNS-resolution checks are in the guarded fetch path. Current tests assert ordinary public hostname URLs are extracted while private/special-use literal hosts are filtered, so adding `allowRfc2544BenchmarkRange` here would only allow raw `198.18.x.x` URL literals, not fix a fake-IP DNS-resolution block for normal hostnames.

## Best Possible Solution

Close this issue as not reproducible against current main and keep the detector's fail-closed filtering for literal private/special-use URL hosts. If users still see link understanding skipped under a fake-IP proxy, the best follow-up is a new issue with the exact inbound message text seen by OpenClaw, affected channel/plugin, proxy mode, config, and logs showing whether the failure occurs during detection or during the configured link-understanding command/fetch path.
