## Summary

Keep open. Current main has adjacent bundled-runtime-deps hardening, but the reported npm update/runtime-deps loop is fresh, active, and not fully fixed or safely superseded. The postinstall update path still treats an existing dependency package.json as enough, so stale installed versions can still be skipped during npm updates; linked PR #72084 remains the focused fix rather than proof this issue is resolved on main.

## Best Possible Solution

Keep #72058 open until maintainers land or replace #72084 with a regression-tested stale-version postinstall repair, then re-evaluate whether #72114 should remain the canonical tracker for staging-timeout/shutdown-race failures. The best code path is to make postinstall dependency detection version-aware, keep runtime/doctor repair safeguards, and preserve the concrete user workarounds and multi-user reproduction details in the linked discussion.
