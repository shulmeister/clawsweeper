## Summary

Keep #51332 open. Current main has useful adjacent mitigations for Ollama setup and visible transcript hygiene, but it does not yet implement the requested diagnostic/log warning when a local model emits tool-call-shaped assistant text without a structured executable tool invocation. The related ClawSweeper items on tool cards and audit logging are adjacent, not canonical replacements for this local-model fake tool-call diagnostic request.

## Best Possible Solution

Keep the issue open. The best path is a targeted runtime/provider diagnostic: when final assistant text contains or is sanitized from tool-call-shaped content and the run has no corresponding structured tool-call event, emit a redacted diagnostic/log entry with provider, model, session/run id, and guidance. Pair that with Ollama/local-backend troubleshooting docs that explain native /api/chat, compat.supportsTools: false, and the strong orchestrator plus local text workers pattern. Add unit tests around the assistant visible-text sanitizer or run-finalization path and a gateway/chat-history assertion that real tool calls remain distinguishable from pseudo tool text.
