## Summary

Keep open. The issue is recent, concrete, non-maintainer-authored, and reports current beta/main-era Control UI and gateway stalls with specific slow RPCs. Current main has relevant mitigations such as optimistic local chat messages and an early `chat.send` ack, but I did not find source, test, changelog, or release evidence that fixes the broader gateway/control-plane unresponsiveness under chat plus expensive Control UI endpoint load.

## Best Possible Solution

Keep #72365 open for focused maintainer investigation. Reproduce the reported Docker/WSL2 plugin set, capture event-loop delay and RPC timings during chat sends and Control UI refreshes, then bound/defer/cache expensive Control UI endpoints or move CPU-heavy work off the gateway event loop. Add regression coverage proving `chat.send`, `chat.history`, WebSocket connect, and lightweight control requests remain responsive while synthetic slow `tools.effective`, usage, model catalog, memory/wiki, and node/device work is in flight.
