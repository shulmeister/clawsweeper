## Summary

Current `main` already routes normal heartbeat polls through the real bound channel context for prompt/tool assembly, so the synthetic-`heartbeat` prompt/tool divergence described in #70417 is no longer how the code behaves.

## Evidence

- **Heartbeat keeps the real channel in request context:** `runHeartbeatOnce()` now sets `OriginatingChannel` and `OriginatingTo` from the resolved delivery target, while keeping `Provider: "heartbeat"` only for system-event bookkeeping. That preserves the session's actual channel identity for downstream prompt/tool routing.

For heartbeat runs with `OriginatingChannel: telegram`, downstream code sees `telegram`, not `heartbeat`.
