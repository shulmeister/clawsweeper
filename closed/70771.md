## Summary

Useful integration, but this PR is a one-off optional Render/VPS monitoring plugin that current OpenClaw vision and plugin docs route to an external ClawHub/npm plugin rather than a new bundled repo-owned plugin, and it also reaches into core `/status` for extension-specific behavior.

## Evidence

- **Repo vision routes optional capability to external plugins:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, the preferred path is npm plus local extension loading, plugin authors should host and maintain plugins in their own repository, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '48,86p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Public plugin docs say plugins do not need to be added to this repo:** `docs/plugins/building-plugins.md` tells authors to publish plugins to ClawHub or npm and install them with `openclaw plugins install ...`; it explicitly presents external publish/install as the normal path.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/docs/plugins/building-plugins.md#L16)
  - command: `nl -ba docs/plugins/building-plugins.md | sed -n '12,22p;120,138p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **SDK already supports the exact runtime shape externally:** OpenClaw already exposes plugin background services in the public SDK and documents `api.registerService(...)` as the place for heavy runtime-only registrations, so a Render/VPS monitor can live as an external code plugin without a bundled-core merge.
  - file: [src/plugins/types.ts:1968](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/plugins/types.ts#L1968)

  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Core/plugin boundary disfavors extension-specific core hooks:** `docs/plugins/architecture.md` says core should not import channel- or plugin-specific convenience behavior directly and should instead consume a narrow generic capability if core truly needs it. The PR's Render-specific `/status` addition is the opposite direction.
  - file: [docs/plugins/architecture.md:239](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/docs/plugins/architecture.md#L239)

- **Current main already has plugin inspection/status surfaces for plugin-owned services:** `src/plugins/status.ts` and `src/plugins/inspect-shape.ts` already report plugin commands/services/gateway methods and shape metadata, so bundled extension-specific `/status` wiring is not the documented path for exposing plugin state.
  - file: [src/plugins/status.ts:51](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/plugins/status.ts#L51)

That reinforces that this should not be merged as-is into the bundled tree.
