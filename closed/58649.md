## Summary

Current main already handles follow-up messages for completed subagent sessions by reactivating the prior session/run state, and the current docs/schema also clarify that `timeoutSeconds` is a run-timeout control rather than an indefinite-session setting while `idleTimeoutMs` is not a `sessions_spawn` parameter.

## Evidence

- **Completed subagent sessions are explicitly reactivated:** `reactivateCompletedSubagentSession()` looks up the latest ended run for the child session and replaces it with the new run id, preserving the session for follow-up work.
