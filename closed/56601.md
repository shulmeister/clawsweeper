## Summary

Keep #56601 open. Current main still has the Web Chat speaker button wired to browser-native SpeechSynthesis, with no path to the configured `messages.tts` Microsoft/Edge provider voice. The backend TTS provider path does honor configured Microsoft voice settings, so the remaining gap is specifically the Web Chat read-aloud control bypassing that configured TTS stack.

## Best Possible Solution

Keep the issue open and fix the Web Chat read-aloud path. The likely product fix is to make the speaker button call gateway `tts.convert`/configured TTS and play the returned audio, with browser SpeechSynthesis only as an explicit fallback or separate local-browser option. Add a UI/controller test proving a configured Microsoft/Edge voice request uses the gateway TTS path rather than `SpeechSynthesisUtterance`.
