## Summary

Current main does register `cron` and `gateway`, but intentionally hides them from non-owner senders. The report’s expectation that `commands.restart: true` should make `gateway` appear for any enabled session does not match current main’s policy-filtered tool list behavior.

## Evidence

- **Built-in tools are still registered:** `createOpenClawTools()` always adds `createCronTool()` and `createGatewayTool()` on non-embedded runs, so current main has not lost registration for either tool.
  - file: [src/agents/openclaw-tools.ts:236](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/openclaw-tools.ts#L236)
  - command: `nl -ba src/agents/openclaw-tools.ts | sed -n '232,260p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Both tools are explicitly owner-only:** `createCronTool()` and `createGatewayTool()` mark themselves `ownerOnly`, so they are not expected to appear for non-owner senders.
  - file: [src/agents/tools/cron-tool.ts:414](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/tools/cron-tool.ts#L414)
  - command: `nl -ba src/agents/tools/cron-tool.ts | sed -n '400,450p'; nl -ba src/agents/tools/gateway-tool.ts | sed -n '345,380p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Owner-only tools are removed from the available tool list:** `createOpenClawCodingTools()` applies `applyOwnerOnlyToolPolicy()` before building the final runtime tool set; non-owners lose `cron`/`gateway` there.
  - file: [src/agents/pi-tools.ts:671](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/pi-tools.ts#L671)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '650,710p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Owner status is separate from basic command authorization:** `resolveCommandAuthorization()` only sets `senderIsOwner` from explicit owner identity/admin scope; being allowed to chat is not enough. The paired test file verifies that direct/group senders are not treated as owners when `commands.ownerAllowFrom` is unset.
  - file: [src/auto-reply/command-auth.ts:689](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/auto-reply/command-auth.ts#L689)
  - command: `nl -ba src/auto-reply/command-auth.ts | sed -n '660,710p'; nl -ba src/auto-reply/command-auth.owner-default.test.ts | sed -n '1,120p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **The system-prompt tool list comes from the same filtered runtime tool set:** `resolveCommandsSystemPromptBundle()` calls `createOpenClawCodingTools()` with `senderIsOwner`, so the prompt tool list is derived from the effective runtime inventory rather than a separate stale registry.
  - file: [src/auto-reply/reply/commands-system-prompt.ts:80](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/auto-reply/reply/commands-system-prompt.ts#L80)
