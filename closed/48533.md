## Summary

Current `main` already implements the reported fix path: unknown `github-copilot/*` model IDs are synthesized dynamically, and Claude Copilot model IDs are routed through Anthropic Messages instead of OpenAI Responses, which addresses the `claude-opus-4.6-fast` mismatch described in #48533.

## Evidence

- **Dynamic Copilot model fallback now accepts unknown model IDs:** `resolveCopilotForwardCompatModel()` synthesizes runtime models for arbitrary unknown `github-copilot/*` model IDs, so new Copilot model IDs no longer require a bundled catalog update first.
  - file: [extensions/github-copilot/models.ts:23](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/github-copilot/models.ts#L23)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Claude Copilot models now use Anthropic Messages:** `resolveCopilotTransportApi()` returns `anthropic-messages` for Copilot model IDs containing `claude`, directly addressing the old `does not support Responses API` failure mode for Claude-family Copilot models.
  - file: [extensions/github-copilot/models.ts:15](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/github-copilot/models.ts#L15)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Provider runtime wires in the dynamic resolver:** The bundled `github-copilot` provider registers `resolveDynamicModel`, so the forward-compat + transport logic is active on the current main runtime path.
  - file: [extensions/github-copilot/index.ts:151](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/github-copilot/index.ts#L151)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Current stream wrapper supports both Copilot transports:** The provider wrapper has separate handling for `anthropic-messages` and `openai-responses`, confirming the Claude-vs-Responses split is intentional in current main.
  - file: [extensions/github-copilot/stream.ts:26](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/github-copilot/stream.ts#L26)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Release notes show both parts of the fix shipped:** `CHANGELOG.md` records forward-compatible GitHub Copilot dynamic model IDs in `2026.3.22` and the Claude transport fix in `2026.4.5` ('route GitHub Copilot Claude models through Anthropic Messages instead of OpenAI Responses').
  - file: [CHANGELOG.md:1185](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/CHANGELOG.md#L1185)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Docs now describe the corrected transport behavior:** The GitHub Copilot docs explicitly say Claude model IDs use the Anthropic Messages transport automatically.
  - file: [docs/providers/github-copilot.md:87](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/providers/github-copilot.md#L87)
