## Summary

Keep open. Current main has a Control UI mitigation that preserves optimistic local user/assistant messages when `chat.history` returns stale or empty data, but the durable CLI/WebChat transcript problem described in #68611 is not fully implemented. The CLI-backed `chat.send` path still emits user transcript updates as events, and the agent-run-start branch does not append the assistant reply to the session transcript before the final history reload path. That leaves `chat.history` dependent on stale/empty local transcript state for non-Claude CLI providers such as `google-gemini-cli`.

## Best Possible Solution

Keep the issue open and fix the gateway/WebChat CLI transcript contract directly: persist both the accepted user turn and final assistant reply for CLI-backed `chat.send` runs via the existing `SessionManager.appendMessage`/transcript helper path before relying on `chat.history`. Add a regression test that a non-Claude CLI provider turn produces alternating user and assistant entries in `chat.history` after reload, not only optimistic in-memory UI preservation.
