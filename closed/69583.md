## Summary

Keep open. Current main still executes multi-collection QMD searches with a sequential await-in-for loop, and the PR's parallel fan-out plus AbortSignal plumbing are not implemented on main. The PR remains a plausible focused performance fix, with useful measurements and regression-test intent, but it still needs maintainer review around bounded concurrency and abort cleanup concerns raised in the discussion.

## Best Possible Solution

Keep PR #69583 open for maintainer review/rework. The best path is to land a bounded parallel multi-collection QMD search implementation in `extensions/memory-core/src/memory/qmd-manager.ts`, preserve the existing merge/dedup and query-fallback semantics, thread robust abort cleanup through `src/memory-host-sdk/host/qmd-process.ts` if fail-fast fan-out is used, and keep the focused regression tests proving parallel invocation and fallback behavior.

## Risks / Open Questions

- The proposed `Promise.all` approach may need a per-request or global concurrency cap before merge to avoid large configured collection counts multiplying subprocess fan-out.
- If abort support is accepted, `runCliCommand` should use unified cleanup/settlement semantics so abort, timeout, error, and close paths cannot leave timers or listeners around.
