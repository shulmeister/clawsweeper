## Summary

Keep open. Current main does not implement the proposed trusted-proxy password fallback: trusted-proxy failures still return before shared-secret auth, credential planning still excludes trusted-proxy passwords, and tests/docs intentionally assert fail-closed loopback behavior. The PR addresses a real reported gap, but it also conflicts with current documented security posture and has active product/security discussion around service identity, so it needs maintainer judgment rather than automated cleanup closure.

## Best Possible Solution

Keep #64122 open for maintainer triage. The best path is either a narrowly rebased gateway/auth PR that addresses the active review concerns and intentionally updates the fail-closed docs/tests, or an explicit maintainer decision to supersede it with #69066's service-identity design once that RFC is accepted.
