## Summary

Current main covers the reported 2026.4.24 mismatch. The top-level `openclaw health --json` still reaches the gateway health RPC, but the gateway call path was changed after v2026.4.24 so direct-local backend `gateway-client` shared-auth calls no longer depend on paired device identity. The server also has a matching direct-local backend self-pairing bypass, and regression coverage asserts the default health call uses backend metadata with `deviceIdentity: null` while explicit CLI calls keep CLI identity. The existing issue comment that OpenClaw 2026.4.26 succeeds matches the current code state.

## Best Possible Solution

Close this issue as implemented on main. Keep the current direct-local backend shared-auth behavior and regression coverage; leave the separate Telegram/channel health JSON state mismatches tracked by #46494 and #59287 rather than conflating them with this CLI wrapper connectivity bug.
