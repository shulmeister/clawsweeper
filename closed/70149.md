## Summary

Keep #70149 open. Current main still sends the WhatsApp Baileys WebSocket through `resolveAmbientNodeProxyAgent()`, which constructs `proxy-agent`; dependency source shows that `proxy-agent` asks `proxy-from-env` for the `wss:` URL, and `proxy-from-env@2.1.0` uses `WSS_PROXY` or `ALL_PROXY` for `wss:`, not `HTTPS_PROXY`/`HTTP_PROXY`. That leaves the exact reporter case unresolved despite the broader #61825 ambient-proxy closure.

## Best Possible Solution

Keep the issue open and fix the WhatsApp plugin narrowly. Resolve the configured HTTP(S) proxy URL into a Node `HttpsProxyAgent` for Baileys' WebSocket `agent` path, while keeping undici dispatchers for fetch/media paths. Add focused regression coverage proving that `HTTPS_PROXY`/`HTTP_PROXY` are honored for the WhatsApp `wss:` login handshake and that the reported `ProxyAgent(env)` path is not used for this case. If `channels.whatsapp.proxy` is added, keep it root/per-account, document it, and split any usable work from broad PR #46356 into a clean WhatsApp-only change.
