## Summary

Keep #42854 open. Current main still normalizes `gemini-3.1-pro` to `gemini-3.1-pro-preview` and still resolves Gemini 3.1 Pro forward-compatible rows from the older `gemini-3-pro-preview` template, so the reported alias/template behavior is not implemented away. The related PRs #42916 and #43046 targeted this issue but were closed unmerged; #70583 addresses a separate Telegram catalog-cache stale-data bug, not this Gemini template mapping.

## Best Possible Solution

Keep the issue open for maintainer/product decision and a focused Google provider fix: decide the canonical Gemini 3.1 Pro id, then update the Google model-id normalizer and forward-compat template precedence together, with regression tests covering config input `google/gemini-3.1-pro`, picker/catalog display, and template selection when both `gemini-3.1-pro-preview` and `gemini-3-pro-preview` metadata exist.
