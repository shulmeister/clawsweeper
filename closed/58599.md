## Summary

Current `main` does not match the reported root cause: edit-tool validation requires `edits[]`, the host edit wrapper reads nested `edits[]`, checked-in tests use `{ path, edits: [...] }` successfully, and the latest release commit contains the same behavior.

## Evidence

- **Validation requires `edits[]`, not top-level `oldText`/`newText`:** `REQUIRED_PARAM_GROUPS.edit` now requires `path` plus an `edits` array validated by `hasValidEditReplacements`; it does not ask for flat top-level replacement fields.

  - sha: [11ad1919edc7](https://github.com/openclaw/openclaw/commit/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41)
- **Host edit parsing reads nested replacements:** `readEditReplacements()` iterates `record.edits` and extracts `oldText`/`newText` from each entry, and `readEditToolParams()` returns those nested edits.
  - file: [src/agents/pi-tools.host-edit.ts:41](https://github.com/openclaw/openclaw/blob/11ad1919edc7278f9d3e36c87c56bd3cb52c6e41/src/agents/pi-tools.host-edit.ts#L41)
