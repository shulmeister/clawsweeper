## Summary

Keep #65997 open. Current main still has the mismatch reported for restrictive `plugins.allow`: `openclaw help` is a Commander framework help command, but OpenClaw only checks `program.commands`, so it can still fall through to the plugin allowlist fallback. Config validation also still treats `help` and `plugin` as missing plugin IDs, so adding them remains a stale-entry warning rather than a clean fix.

## Best Possible Solution

Keep #65997 open. The likely fix belongs in the CLI startup/gating path: teach the primary-command guard or missing-command fallback to recognize Commander framework commands such as `help`, avoid suggesting invalid pseudo-plugin IDs for unknown/core-framework commands, and align config validation so users are not told to add entries that validation later reports as stale. Add focused regression coverage for `openclaw help` and the singular `openclaw plugin`/unknown-command path under restrictive `plugins.allow`.
