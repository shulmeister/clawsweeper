## Summary

Keep open. Current main still does not expose Slack Socket Mode ping/pong timeout configuration, and the linked reports describe a concrete Slack Socket Mode timeout/reconnect problem. The PR’s central capability remains useful, but it should be rebased/reworked for current main because @slack/bolt@4.7.1 now exposes these options on SocketModeReceiver directly, so OpenClaw should not need to mutate internal receiver.client fields.

## Best Possible Solution

Keep the PR open or ask for a maintainer/author update. The best fix is a focused Slack plugin change that adds channels.slack.socketMode to TS config types, strict Zod validation, UI help/labels, and Slack docs, then passes clientPingTimeout, serverPingTimeout, and pingPongLoggingEnabled through createSlackBoltApp into new SocketModeReceiver(...) using the current public @slack/bolt@4.7.1 options. Add regression tests for schema acceptance and receiver args. If this PR is abandoned, keep #14248 as the canonical issue and close only after an equivalent implementation lands.
