## Summary

Current `main` already exposes plugin-based pre-tool-call policy hooks that can inspect tool inputs, block execution, rewrite params, and require approval across HTTP, Codex-native, and MCP tool-entry surfaces. Given `VISION.md`'s scope guidance that optional capability should usually ship as plugins and ClawHub listings rather than new core subsystems, this PR is a better fit as an external plugin/integration than as a new core `guardrails` runtime/config surface.

## Risks / Open Questions

- A plugin-based implementation would need its own packaging/config UX; current main does not provide a dedicated core `guardrails:` config surface.
- Plugin priority gives strong pre-tool control, but it is not a distinct always-before-all-plugins core phase if maintainers later decide that exact ordering is required.
