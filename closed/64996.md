## Summary

Current `main` already covers the WhatsApp group-listener problem through the newer group-access model and regression coverage. This PR's patch targets an older interpretation of `groupAllowFrom` and no longer matches the documented/runtime contract on `main`.

## Evidence

- **PR discussion narrowed the real root cause:** The April 16, 2026 maintainer comment on this PR says the patch is only for group-JID matching in `groupAllowFrom`, while the broader beta blocker from #54613 was a separate WhatsApp config-merge / `groupPolicy` inheritance bug.
- **Current docs split group admission from sender authorization:** The WhatsApp docs now define `channels.whatsapp.groups` as the group membership allowlist, and `channels.whatsapp.groupPolicy` + `groupAllowFrom` as the sender policy. That means group JIDs belong in `groups`, not in `groupAllowFrom`.
  - file: [docs/channels/whatsapp.md:218](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/docs/channels/whatsapp.md#L218)

  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Current runtime enforces that two-layer model:** `resolveChannelGroupPolicy` first resolves allowed groups from `groups`; only when there are no explicit groups and sender filtering is configured does it allow the conversation through for sender-level checks. This is the current mainline implementation of WhatsApp group admission.
