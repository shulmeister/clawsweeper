## Summary

Current main already supports loading `memory-core` alongside another selected memory plugin for dreaming, and the behavior was shipped in `v2026.4.23`. The report appears to be based on older behavior and/or configuring dreaming under `memory-core` instead of the active memory slot plugin entry.

## Evidence

- **Dreaming config follows the active memory slot plugin:** `resolveMemoryDreamingPluginId` returns `plugins.slots.memory` when set, and `resolveMemoryDreamingPluginConfig` then reads config from that selected plugin entry rather than always from `memory-core`.
  - file: [src/memory-host-sdk/dreaming.ts:321](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/memory-host-sdk/dreaming.ts#L321)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:src/memory-host-sdk/dreaming.ts | nl -ba | sed -n '321,343p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

  - file: [src/plugins/gateway-startup-plugin-ids.ts:45](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/gateway-startup-plugin-ids.ts#L45)

## Risks / Open Questions

- Public docs still describe dreaming under `plugins.entries.memory-core.config.dreaming`, which can mislead users even though the runtime resolves dreaming config from the active memory-slot plugin entry.
- This review was read-only and relied on shipped code plus tests rather than a live repro run in the local environment.
