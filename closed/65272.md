## Summary

Keep open. Current main still runs preflight compaction before the silent memory-flush turn, matching the reporter's call-flow analysis. The post-compaction refresh prompt and silent-delivery handling do not prove that the `Pre-compaction memory flush` block is guaranteed to be first in rebuilt context.

## Best Possible Solution

Keep the issue open and fix it in the core auto-reply/embedded-runner path. The likely fix should make the memory flush truly pre-compaction or move it to a hidden/internal transcript lane, then add regression coverage for stale-token preflight compaction showing the flush/user-intent state cannot land mid-context after compaction.
