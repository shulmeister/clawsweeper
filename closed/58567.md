## Summary

Keep open. Current main has broad Anthropic transcript hygiene, and failed/aborted assistant turns are skipped in the pinned pi-ai converter, but I did not find a guard that prevents a normal persisted assistant tail from being sent as prefill to Claude Opus 4.6. Official Anthropic docs now explicitly say prefill requests with Opus 4.6 return 400, so the reported central failure mode still needs a targeted fix.

## Best Possible Solution

Add an Anthropic-owned replay/request guard for models that do not support assistant prefill, probably through the provider replay policy or a narrow model capability hook. Before the LLM call, if the effective Anthropic request would end with a normal assistant message for Opus 4.6/Sonnet 4.6/Opus 4.7-class models, append a non-empty synthetic user continuation marker or otherwise convert the tail into a resumable user turn. Cover it with a regression test using `anthropic/claude-opus-4-6`, a persisted assistant tail, and a blank/runtime resume path, plus a control proving real user follow-up remains unchanged.

## Risks / Open Questions

- A fix should avoid changing byte-stable transcript history unnecessarily; prefer request-time replay shaping or a narrowly marked synthetic user turn over mutating stored user-visible history by default.
- The model capability should be provider/model-specific because some Anthropic-family and non-Anthropic providers may still intentionally support assistant prefill.
